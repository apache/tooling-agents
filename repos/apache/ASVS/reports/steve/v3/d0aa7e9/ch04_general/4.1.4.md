## Findings

### HIGH SEVERITY

#### ASVS-414-HIGH-001: State-Changing Election Operations Exposed via GET Method

**Severity:** HIGH

**File:** `v3/server/pages.py`

**Locations:**
- `do_open_endpoint()` at line ~510
- `do_close_endpoint()` at line ~530

**Related Functions Checked:**
- `do_set_open_at_endpoint()` (line ~420) — correctly uses POST
- `do_set_close_at_endpoint()` (line ~450) — correctly uses POST
- `do_vote_endpoint()` (line ~550) — correctly uses POST
- `do_create_election()` (line ~600) — correctly uses POST
- `do_add_issue()` (line ~650) — correctly uses POST
- `do_edit_issue()` (line ~700) — correctly uses POST
- `do_delete_issue()` (line ~750) — correctly uses POST

**Gap Type:** Type A — Entry point with incorrect HTTP method (no POST restriction on state-changing operations)

**Description:**

Two critical election state management endpoints use GET instead of POST for state-changing operations. This violates the fundamental HTTP principle that GET requests should be safe (read-only) and idempotent. The endpoints `do_open_endpoint()` and `do_close_endpoint()` permanently modify election state but are accessible via simple URL navigation.

**Vulnerable Code:**

```python
# Line ~510
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)  # STATE-CHANGING OPERATION via GET
    _LOGGER.info(f'User[U:{result.uid}] opened election[E:{election.eid}]')
    title = election.get_metadata().title
    await flash_success(f'Opened election: {title}')
    return quart.redirect(f'/manage/{election.eid}', code=303)
```

```python
# Line ~530
@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz
    election.close()  # STATE-CHANGING OPERATION via GET
    _LOGGER.info(f'User[U:{result.uid}] closed election[E:{election.eid}]')
    title = election.get_metadata().title
    await flash_success(f'Closed election: {title}')
    return quart.redirect(f'/manage/{election.eid}', code=303)
```

**Data Flow:**

```
Browser GET request → /do-open/<eid> or /do-close/<eid>
  → Authentication check (passes for any committer)
  → election.open() or election.close() executed
  → Election state permanently changed
```

**Proof of Concept:**

An attacker can trigger election open/close for any authenticated committer via:

1. **Image tag injection** (e.g., in any HTML context the victim views):
   ```html
   <img src="https://steve.example.org/do-open/TARGET_EID" width="0" height="0">
   ```

2. **Link prefetching** — modern browsers may prefetch `<a>` targets:
   ```html
   <link rel="prefetch" href="https://steve.example.org/do-close/TARGET_EID">
   ```

3. **Browser bookmark/history replay** — navigating to a bookmarked or cached URL re-executes the operation.

4. **Proxy/CDN caching** — GET responses may be cached and replayed by intermediaries.

**Impact:**

- An election could be opened prematurely (before issues are finalized), exposing incomplete ballot items to voters
- An election could be closed prematurely, preventing legitimate voters from casting ballots
- Combined with the placeholder CSRF token (acknowledged technical debt), any cross-origin page can trigger these operations on behalf of an authenticated user with zero interaction beyond visiting the page
- The application already demonstrates awareness that state-changing operations require POST (all other `do-*` state-changing endpoints use `@APP.post()`), making this an inconsistency

**Note on related patterns:** Five other state-changing endpoints (`do-set-open_at`, `do-set-close_at`, `do-vote`, `do-add-issue`, `do-edit-issue`, `do-delete-issue`, `do-create-election`) correctly use `@APP.post()`, confirming this is an oversight rather than a design decision.

**Remediation:**

Change both endpoints from `@APP.get()` to `@APP.post()` and update any frontend links to use form submissions or JavaScript POST requests:

```python
@APP.post('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    # ... (rest unchanged)

@APP.post('/do-close/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    # ... (rest unchanged)
```

Update the templates that link to these endpoints to use forms:
```html
<form method="POST" action="/do-open/{{ eid }}">
    <!-- Include CSRF token when implemented -->
    <button type="submit" class="btn btn-primary">Open Election</button>
</form>
```

## Positive Security Patterns

| Pattern | Location | Assessment |
|---------|----------|------------|
| **Per-route method registration** | All routes in `pages.py` | Quart's `@APP.get()`/`@APP.post()` ensures framework returns 405 for unregistered methods on each route |
| **POST for most state changes** | `do-vote`, `do-create-election`, `do-add-issue`, `do-edit-issue`, `do-delete-issue`, `do-set-open_at`, `do-set-close_at` | 7 of 9 state-changing endpoints correctly use POST |
| **POST-Redirect-GET pattern** | All POST endpoints | State-changing POST handlers redirect with 303, preventing resubmission on refresh |
| **No catch-all route** | `pages.py` | No wildcard route that would accept all methods on unintended paths |
| **Authentication on all sensitive routes** | `pages.py` throughout | `@asfquart.auth.require` applied before method handlers execute |

## Architecture Observations

1. **Framework-level method blocking is sound:** Quart (like Flask) inherently returns `405 Method Not Allowed` for HTTP methods not registered on a given route. This means methods like PUT, DELETE, PATCH, TRACE, and CONNECT are rejected by the framework on all defined routes unless explicitly registered.

2. **No application-level TRACE/CONNECT blocking middleware:** While the framework handles this per-route, there is no explicit global middleware blocking dangerous methods like TRACE. In production, this should be handled at the reverse proxy layer (e.g., Apache httpd, nginx). This is noted as an observation rather than a finding since the framework already provides adequate protection.

3. **OPTIONS handling:** Quart provides automatic OPTIONS response with `Allow` headers for defined routes. There is no explicit CORS configuration, which is appropriate for this server-rendered web application that doesn't appear to serve cross-origin API consumers.

4. **Consistent pattern with exceptions:** The application demonstrates strong awareness of proper HTTP method usage across 7 of 9 state-changing operations, suggesting the two GET-based endpoints are oversights rather than architectural decisions.
