## Findings

### Medium Severity

#### Finding: ASVS-441-SEV-001

| Field | Value |
|---|---|
| **Severity** | **MEDIUM** |
| **Finding ID** | ASVS-441-SEV-001 |
| **Category** | Gap Type A — Missing enforcement control for transport security |
| **File** | `v3/server/main.py`, lines 79–83; `v3/server/config.yaml.example`, lines 27–31 |
| **Function** | `run_standalone()` |

**Description:** 

TLS is optional, not enforced. If WebSocket endpoints exist in unprovided files (`pages.py`, `api.py`), they would operate over plaintext WS when TLS is not configured. The server explicitly supports running without TLS based on configuration.

**Vulnerable Code:**

`main.py`, lines ~79-83:
```python
kwargs = {}
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
    extra_files.update((kwargs['certfile'], kwargs['keyfile']))

# Spool up the app!
app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Supporting Configuration:**

`config.yaml.example`, lines ~27-31:
```yaml
# Specify the .pem files to serve using TLS, or leave these two
# fields blank for plain HTTP. These files are relative to the
# "certs" subdirectory of main.py (the app), or they can be absolute
# paths to their installed location.
certfile: localhost.apache.org+3.pem
keyfile: localhost.apache.org+3-key.pem
```

**Data Flow:**
- **Source:** `config.yaml` → `app.cfg.server.certfile`
- **Sink:** `app.runx(**kwargs)` — server starts with or without TLS
- **Missing Control:** No enforcement that TLS must be configured; no startup failure or warning if certfile is blank; no programmatic guarantee that any WebSocket endpoint (current or future) will only accept WSS connections.

**ASGI Mode Gap:**

In `run_asgi()` (lines ~94-110), TLS is not configured at the application level at all — it depends entirely on external Hypercorn or reverse proxy configuration, with no application-level validation that the deployment is actually using TLS:

```python
def run_asgi():
    # ...
    global steve_app
    steve_app = create_app()
    # No TLS enforcement or verification
```

**Related Functions Checked:**
- `run_standalone()` — TLS optional
- `run_asgi()` — TLS not configured
- `create_app()` — No transport security validation

**Proof of Concept:**
1. Set `certfile:` and `keyfile:` to empty strings in `config.yaml`
2. Start the server: `uv run main.py`
3. Server starts on `http://localhost:58383` (plain HTTP)
4. Any WebSocket endpoint (if defined in `pages.py`/`api.py`) would accept `ws://` connections, not `wss://`

**Impact:**

For a voting system handling election data:
- WebSocket traffic (if any) could be intercepted or modified by network attackers
- Session tokens transmitted over WebSocket could be captured
- Vote data or election state changes communicated via WebSocket would lack confidentiality and integrity protections
- Violates ASVS 4.4.1 requirement for WSS on all WebSocket connections

**Remediation:**

**Option 1 — Enforce TLS at startup (fail-closed):**
```python
def run_standalone():
    # ...
    app = create_app()
    
    if not app.cfg.server.certfile or not app.cfg.server.keyfile:
        _LOGGER.critical('TLS certificates must be configured. '
                         'Set server.certfile and server.keyfile in config.yaml.')
        sys.exit(1)
    
    kwargs = {
        'certfile': CERTS_DIR / app.cfg.server.certfile,
        'keyfile': CERTS_DIR / app.cfg.server.keyfile,
    }
    app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Option 2 — If plain HTTP must be supported for development, add WebSocket-specific middleware:**
```python
@app.before_websocket
async def enforce_wss():
    """Reject non-TLS WebSocket connections."""
    from quart import websocket
    if websocket.scheme != 'wss':
        await websocket.close(1008, 'WSS required')
```

## Positive Security Patterns

1. **Default config includes TLS certificates**: The example configuration ships with `certfile` and `keyfile` populated, making TLS the default development experience.

2. **Certificate file path safety**: Uses `pathlib.Path` for certificate path construction (`CERTS_DIR / app.cfg.server.certfile`), preventing path traversal.

3. **OAuth over HTTPS**: OAuth URLs are hardcoded with `https://` scheme (lines 38-41), ensuring authentication flows use TLS regardless of server configuration.

4. **ASGI architecture**: Production mode via Hypercorn allows TLS to be configured at the infrastructure level with mature tooling.

5. **Development documentation**: `quickstart.md` instructs developers to generate certificates before running the server.

---

## Architecture Observations

1. **No WebSocket endpoints identified**: The ARCHITECTURE.md describes a standard HTTP request/response model. No real-time communication channels (WebSocket, SSE) are documented. The Quart framework supports WebSockets natively, but no usage was found.

2. **Two-mode deployment**: The application runs in standalone mode (development, TLS configured in `main.py`) or ASGI mode (production, TLS delegated to Hypercorn/reverse proxy). Neither mode validates that TLS is actually active.

3. **Proxy assumption**: The config comment "Typical usage is that a proxy sits in front of this server" suggests production deployments use a reverse proxy for TLS termination. This is a valid pattern but creates a dependency on correct proxy configuration that the application cannot verify.

4. **Incomplete audit scope**: The route-defining files (`pages.py`, `api.py`) were not provided. These files could contain WebSocket endpoints (Quart's `@app.websocket('/path')` decorator) that would be directly relevant to ASVS 4.4.1.
