## Findings

### Critical Severity

#### Finding ASVS-443-SEV-001: Missing Authorization (Ownership) Verification on All Election Management Endpoints

**Severity:** CRITICAL  
**Gap Type:** Type B — Control EXISTS (session identity + `owner_pid`) but NOT CALLED  
**CWE:** CWE-862 (Missing Authorization)

**Location:**
- **File:** `v3/server/pages.py`
- **Functions Affected:** 9 management endpoints
- **Lines:** 153-168 (`load_election` decorator), 229, 248, 270, 277, 323, 340, 357, 381, 400

**Related Functions Checked:**
- `election.py:create()` — stores `owner_pid`
- `election.py:get_metadata()` — retrieves `owner_pid`
- `pages.py:basic_info()` — retrieves authenticated user identity (`result.uid`)
- `pages.py:load_election()` — decorator that loads elections but never checks ownership

**Description:**

The session management correctly identifies the authenticated user (`result.uid`), and each election stores its owner (`owner_pid`). However, at no management endpoint is the authenticated user verified to be the election owner or otherwise authorized to manage that specific election. Any authenticated committer can manage, open, close, or modify **any** election in the system.

**Affected Endpoints:**

| Endpoint | Method | File:Line | Impact |
|----------|--------|-----------|--------|
| `/manage/<eid>` | GET | `pages.py:229` | View any election's management interface |
| `/manage-stv/<eid>/<iid>` | GET | `pages.py:248` | View any STV issue management |
| `/do-set-open_at/<eid>` | POST | `pages.py:270` | Set open date on any election |
| `/do-set-close_at/<eid>` | POST | `pages.py:277` | Set close date on any election |
| `/do-open/<eid>` | GET | `pages.py:323` | **Open any election for voting** |
| `/do-close/<eid>` | GET | `pages.py:340` | **Close any election** |
| `/do-add-issue/<eid>` | POST | `pages.py:357` | Add issues to any election |
| `/do-edit-issue/<eid>/<iid>` | POST | `pages.py:381` | Edit issues in any election |
| `/do-delete-issue/<eid>/<iid>` | POST | `pages.py:400` | Delete issues from any election |

**Vulnerable Code:**

The `owner_pid` is stored and retrievable:
```python
# election.py — owner_pid is available in metadata
def get_metadata(self):
    md = self._all_metadata()
    return edict(
        ...
        owner_pid=md.owner_pid,  # <-- EXISTS
        ...
    )
```

The user identity is available from the session:
```python
# pages.py — result.uid is the authenticated user
result = await basic_info()  # result.uid = session user ID
```

But at every management endpoint, the `### check authz` comment marks where the comparison was **planned but never implemented**:

```python
# pages.py:323-335 — do_open_endpoint
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    result = await basic_info()

    ### check authz    # <-- NEVER IMPLEMENTED

    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)  # <-- ANY committer can open ANY election
```

Similarly in `load_election` (the shared decorator):
```python
# pages.py:153-168 — load_election decorator
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            ...
        ### check authz    # <-- NEVER IMPLEMENTED
        return await func(e)
    return loader
```

**Data Flow:**
```
Attacker (authenticated as any committer)
  → GET /do-open/TARGET_EID
  → @asfquart.auth.require({R.committer}) → PASSES (is a committer) ✓
  → @load_election → loads election by EID (no ownership check) ✓
  → election.open(pdb) → ELECTION OPENED by unauthorized user ✗
```

**Proof of Concept:**
```http
# Attacker is committer "mallory", election is owned by "alice"
GET /do-open/a1b2c3d4e5 HTTP/1.1
Host: voting.example.org
Cookie: session=<mallory's valid session cookie>

# Result: Election a1b2c3d4e5 is opened for voting by mallory
# Similarly: GET /do-close/a1b2c3d4e5 closes alice's election
```

**Impact:**
- Any authenticated committer can open or close any election, disrupting voting processes
- Any committer can add, edit, or delete issues in elections they don't own
- Any committer can modify election dates, potentially invalidating scheduled votes
- Undermines the entire election integrity model since the owner field exists but is never enforced
- Complete horizontal privilege escalation across all election management functions

**Remediation:**

```python
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            result = await basic_info()
            result.title = 'Unknown Election'
            result.eid = eid
            raise_404(T_BAD_EID, result)

        _LOGGER.debug(f'Loaded: {e}')

        # Verify the authenticated user is the election owner
        s = await asfquart.session.read()
        md = e.get_metadata()
        if md.owner_pid != s['uid']:
            _LOGGER.warning(
                f'User[U:{s["uid"]}] unauthorized access to election[E:{eid}]'
                f' owned by [U:{md.owner_pid}]'
            )
            quart.abort(403)

        return await func(e)
    return loader
```

Consider creating separate decorators for "can view" vs "can manage" vs "can vote" access patterns.

### Medium Severity

#### Finding ASVS-443-SEV-002: State-Changing Operations via GET Requests Bypass Session Security

**Severity:** MEDIUM  
**Gap Type:** Type A — Sensitive operations exposed via method that cannot carry anti-CSRF tokens  
**CWE:** CWE-352 (Cross-Site Request Forgery)

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `do_open_endpoint`, `do_close_endpoint`
- **Lines:** 323, 340

**Related Functions Checked:**
- `pages.py:basic_info()` — returns placeholder CSRF token
- `election.py:open()` — state-changing operation
- `election.py:close()` — state-changing operation

**Description:**

The `/do-open/<eid>` and `/do-close/<eid>` endpoints perform critical state-changing operations (opening and closing elections) via HTTP GET requests. When combined with cookie-based session management, GET requests are inherently vulnerable to cross-site request forgery through simple link injection, image tags, or browser prefetching — even without form-based CSRF. These endpoints also cannot carry request body tokens, making them structurally impossible to protect with CSRF tokens.

**Vulnerable Code:**

```python
# pages.py:323 — state-changing GET endpoint
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)  # Irreversible state change via GET
    ...

# pages.py:340 — state-changing GET endpoint
@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz
    election.close()  # Irreversible state change via GET
    ...
```

**Data Flow:**
```
Attacker crafts malicious page
  → Victim (authenticated committer) visits page
  → Page contains: <img src="https://voting.example.org/do-open/TARGET_EID">
  → Browser automatically sends GET request with session cookie
  → Election state changed without user knowledge or consent
```

**Proof of Concept:**
```html
<!-- Injected into any page viewed by an authenticated committer -->
<!-- Opens election without user knowledge -->
<img src="https://voting.example.org/do-open/TARGET_EID" style="display:none">

<!-- Closes election without user knowledge -->
<img src="https://voting.example.org/do-close/TARGET_EID" style="display:none">
```

**Impact:**
- An attacker who knows an election ID can open or close elections by tricking an authenticated committer into loading a page with the injected URLs
- Election state transitions (EDITABLE → OPEN, OPEN → CLOSED) are irreversible
- Browser prefetching, link preloading, or extensions may trigger these URLs automatically
- Combined with Finding ASVS-443-SEV-001, any committer's session can be used against any election
- No user interaction required beyond visiting a malicious page

**Remediation:**

```python
# Change to POST with CSRF token validation
@APP.post('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    
    # Validate CSRF token from form/header
    form = await quart.request.form
    if not validate_csrf_token(form.get('csrf_token')):
        quart.abort(403)
    
    # Verify ownership (see Finding 1 remediation)
    md = election.get_metadata()
    if md.owner_pid != result.uid:
        quart.abort(403)
    
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    ...
```

## Positive Security Patterns

1. **Strong Cryptographic Primitives:** Argon2 (Type D, 64MB memory), HKDF-SHA256, Fernet encryption, `secrets` module throughout (`v3/steve/crypto.py`)

2. **Per-Voter Salts:** Each mayvote entry gets a unique cryptographic salt via `crypto.gen_salt()` ensuring vote privacy even with identical votes

3. **Anti-Tamper Protection:** `gather_election_data()` creates a deterministic hash of all election data for integrity verification

4. **Cryptographic Shuffling:** `crypto.shuffle()` uses `secrets.randbelow()` for Fisher-Yates shuffle, preventing bias in candidate ordering

5. **Session-Based Authentication:** All state-changing routes require authentication via `@asfquart.auth.require` decorator

6. **Election State Machine:** Business logic enforces state transitions (EDITABLE → OPEN → CLOSED) with assertions in `election.py`

7. **Parameterized Queries:** Database access via `asfpy.db.DB` with parameterized queries preventing SQL injection

8. **Document Access Control:** `/docs/<iid>/<docname>` verifies voter eligibility before serving documents

9. **Vote Encryption:** Each vote is encrypted with a unique key derived from voter identity, issue, and per-voter salt using authenticated encryption (Fernet)

10. **Collision-Resistant ID Generation:** `create_id()` with retry on `IntegrityError` ensures unique election identifiers

## Architecture Observations

1. **No WebSocket Implementation:** The provided code contains no WebSocket endpoints. All communication uses standard HTTP request/response with Quart's async framework. ASVS 4.4.3 (WebSocket session management) is therefore not directly applicable, but the broader API session security concerns apply.

2. **Session Management Model:** Standard cookie-based sessions via `asfquart.session`. The `asfquart.auth.require` decorator provides role-based access control at the authentication level but has no mechanism for resource-level authorization.

3. **Two-Tier Authorization Gap:** The system has authentication (who are you?) but lacks authorization (what can you do to this specific resource?). The `owner_pid` field exists in the data model but is never consulted during request handling.

4. **`random.shuffle()` vs `crypto.shuffle()`:** Line 226 in `pages.py` uses `random.shuffle(issue.candidates)` for candidate ordering, while the codebase has a cryptographically-secure `crypto.shuffle()`. While candidate order bias in a UI is low-severity, using the secure variant is trivial.

5. **Placeholder CSRF Tokens:** The `basic_info()` function returns `'placeholder'` for CSRF tokens (line ~93), indicating incomplete implementation of CSRF protection across all POST endpoints.

6. **State Machine Relies on Assertions:** While effective, using Python `assert` statements for critical state checks can be bypassed with `python -O` flag. The `_all_metadata(required_state)` pattern is more robust.

### Long-Term Actions (Strategic Improvements)

1. **Authorization Framework:** Implement a comprehensive authorization layer with decorators for different access patterns:
   - `@require_election_owner` — for management operations
   - `@require_voter_eligibility` — for voting operations
   - `@require_election_state(state)` — for state-dependent operations

2. **Audit Logging:** Add comprehensive audit logging for all election management operations, including:
   - Who performed the action
   - What election was affected
   - Timestamp and IP address
   - Success/failure status

3. **Rate Limiting:** Implement rate limiting on state-changing endpoints to prevent abuse

4. **Security Testing:** Establish automated security testing including:
   - Authorization bypass attempts
   - CSRF token validation
   - Session management security
   - State machine integrity
