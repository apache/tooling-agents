## Findings

### Medium Severity

#### ASVS-425-SEV-001: Unbounded User Input in Flash Messages Creates Potential for Oversized Cookie Header DoS

**Severity:** MEDIUM

**File:** `v3/server/pages.py`  
**Functions:** `do_vote_endpoint` (line 369), `do_create_endpoint` (line 410), `do_add_issue_endpoint` (line 467), `do_edit_issue_endpoint` (line 489)

**Gap Type:** Type A – Entry points with NO length control

**Description:**

Multiple endpoints incorporate unsanitized, unbounded user input into session flash messages via `quart.flash()`. If the session uses cookie-based storage (the default for Quart/Flask frameworks), the resulting `Set-Cookie` response header can exceed the browser's cookie size limit (~4KB) or the server's incoming header size limit (~8–16KB for most ASGI servers). When the browser sends back the oversized cookie on subsequent requests, the server rejects every request before reaching application code, resulting in a persistent DoS for that user's session.

**Vulnerable Code – Most Direct Path (do_vote_endpoint, lines 385–395):**

```python
# pages.py, lines 385-395
votes = {}
for key, value in form.items():
    if key.startswith('vote-'):
        iid = key.split('-', 1)[1]  # ← Unbounded length from form field name
        votes[iid] = value

for iid, votestring in votes.items():
    if iid not in issue_dict:
        await flash_danger(f'Invalid issue ID: {iid}')  # ← Entire unbounded iid stored in session
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
```

**Vulnerable Code – Election/Issue Title Paths:**

```python
# pages.py, line 424 (do_create_endpoint)
await flash_success(f'Created election: {form.title}')  # ← form.title has no length limit

# pages.py, line 485 (do_add_issue_endpoint)
await flash_success(f'Issue "{form.title}" has been added.')  # ← form.title has no length limit

# pages.py, line 505 (do_edit_issue_endpoint)
await flash_success(f'Issue "{form.title}" has been updated.')  # ← form.title has no length limit
```

**Data Flow:**

```
Source: HTTP POST form field name (vote-<arbitrary_data>) or form body field (title)
    → Extraction: key.split('-', 1)[1] or form.title (NO length check)
    → Sink: quart.flash() → Session storage → Set-Cookie response header
    → Impact: Browser sends oversized Cookie header → Server rejects → Persistent 431 DoS
```

**Proof of Concept:**

```http
POST /do-vote/abc123def456 HTTP/1.1
Host: voting.example.org
Cookie: session=<valid_session>
Content-Type: application/x-www-form-urlencoded
Content-Length: 102410

vote-AAAAAAA[...100KB of 'A'...]=y
```

The server extracts `iid = "AAAAAAA..."` (100KB), finds it's not a valid issue, and calls `flash_danger(f'Invalid issue ID: {iid}')`, storing ~100KB in the session. The `Set-Cookie` header on the 303 response would contain the serialized session with this message. On subsequent requests, the browser sends back the oversized cookie, causing the server's ASGI layer to reject the request before `get_flashed_messages()` can consume and clear it.

**Related Functions Checked:**
- `do_vote_endpoint` (pages.py:369)
- `do_create_endpoint` (pages.py:410)
- `do_add_issue_endpoint` (pages.py:467)
- `do_edit_issue_endpoint` (pages.py:489)
- `flash_danger` / `flash_success` (session flash mechanism)

**Impact:**
- Individual user permanent session lockout until cookies are manually cleared
- The affected user cannot access any authenticated page
- In combination with acknowledged CSRF weakness, this could be triggered against other authenticated users via cross-site form submission

**Remediation:**

Apply length limits at two levels:

1. **Truncate user input before including in flash messages:**

```python
MAX_FLASH_INPUT_LEN = 200

async def do_vote_endpoint(election):
    ...
    for iid, votestring in votes.items():
        if iid not in issue_dict:
            safe_iid = iid[:MAX_FLASH_INPUT_LEN]
            await flash_danger(f'Invalid issue ID: {safe_iid}')
            return quart.redirect(f'/vote-on/{election.eid}', code=303)
    ...

async def do_create_endpoint():
    ...
    title = form.title[:MAX_FLASH_INPUT_LEN]
    await flash_success(f'Created election: {title}')
    ...
```

2. **Enforce maximum request body size via Quart configuration:**

```python
APP.config['MAX_CONTENT_LENGTH'] = 64 * 1024  # 64KB max request body
```

3. **Add server-side input length validation for form fields:**

```python
MAX_TITLE_LEN = 500
MAX_DESCRIPTION_LEN = 5000

async def do_create_endpoint():
    form = edict(await quart.request.form)
    if len(form.get('title', '')) > MAX_TITLE_LEN:
        quart.abort(400, 'Title too long')
    ...
```

## Positive Security Patterns

1. **Fixed-Length Cryptographic IDs in URIs:** All redirect URIs and form actions use `election.eid` and `issue.iid` values generated by `crypto.create_id()`, producing fixed-length identifiers that cannot grow unbounded. (pages.py: all redirect calls)

2. **Hardcoded URI Path Components in Frontend JS:** The `saveDate()` function in manage.ezt uses hardcoded field names (`'open_at'`, `'close_at'`) in the URL, preventing user manipulation of the URI path. (manage.ezt:244–262)

3. **Date Input Validation:** The `_set_election_date` function validates the date string format using `datetime.fromisoformat()` before processing, which implicitly bounds the input length to valid ISO date format. (pages.py:99–103)

4. **Subresource Integrity on External Scripts:** The SortableJS library is loaded with an `integrity` attribute, preventing tampering with the external resource. (vote-on.ezt:227–229)

5. **HTML Escaping in Client-Side DOM Manipulation:** The `escapeHtml()` utility in vote-on.ezt properly escapes `&`, `<`, `>`, and `"` when inserting candidate names into the DOM. (vote-on.ezt, JS `escapeHtml` function)

6. **EZT Format Escaping:** Template uses `[format "js,html"]` for values inserted into JavaScript onclick handlers, preventing injection through template-rendered values. (manage.ezt:204–218)

## Architecture Observations

1. **Session Implementation Unknown:** The session mechanism (`asfquart.session`) is not included in the audit scope. Whether sessions are cookie-based or server-side significantly affects the exploitability of the flash message finding. Cookie-based sessions make it exploitable; server-side sessions with a short session ID cookie would mitigate it.

2. **No Visible MAX_CONTENT_LENGTH Configuration:** The Quart application does not appear to set `MAX_CONTENT_LENGTH`, meaning the framework's default (unlimited or very large) applies. This allows arbitrarily large request bodies to be processed.

3. **Comment-Only Authorization Checks:** Multiple endpoints contain `### check authz` comments without implementation (pages.py: lines 92, 182, 211, 372, 432). While not directly related to ASVS 4.2.5, this means any authenticated committer can exercise all vote/management endpoints, broadening the attack surface for DoS via oversized inputs.
