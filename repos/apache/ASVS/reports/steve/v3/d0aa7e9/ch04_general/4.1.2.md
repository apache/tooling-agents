## Findings

### Medium Severity

#### ASVS-412-MED-001
**Finding: Application lacks any mechanism to differentiate transport security handling between browser-facing pages and action/API endpoints**

**Gap Type:** Type A — No transport security differentiation control exists for any endpoint

**Location:**
- **File:** `v3/server/main.py`
- **Lines:** 76-82
- **Related Files:** `v3/server/pages.py` (all route definitions), `v3/server/config.yaml.example` (lines 24-30)

**Related Functions Checked:**
- `main.py`: TLS configuration loading
- `pages.py`: All route handlers (`/do-vote/<eid>`, `/do-create-election`, `/do-add-issue/<eid>`, `/do-edit-issue/<eid>/<iid>`, `/do-delete-issue/<eid>/<iid>`, `/do-set-open_at/<eid>`, `/do-set-close_at/<eid>`)

**Description:**

The application does not implement any mechanism to differentiate transport security requirements between user-facing browser endpoints and action/API endpoints. All endpoints are treated identically with respect to HTTP/HTTPS handling, creating a vulnerability where action endpoints may silently accept HTTP requests that get redirected to HTTPS by a reverse proxy, masking plaintext data transmission.

**Vulnerable Configuration** (`config.yaml.example`, lines 24-30):
```yaml
    # Specify the .pem files to serve using TLS, or leave these two
    # fields blank for plain HTTP. These files are relative to the
    # "certs" subdirectory of main.py (the app), or they can be absolute
    # paths to their installed location.
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
```

**Vulnerable Code** (`main.py`, lines 76-82):
```python
    kwargs = {}
    if app.cfg.server.certfile:
        kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
        kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
        extra_files.update((kwargs['certfile'], kwargs['keyfile']))

    # Spool up the app!
    app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Data Flow:**
1. Configuration explicitly documents TLS as optional: `"leave these two fields blank for plain HTTP"`
2. Application architecture comment confirms reverse proxy deployment: `"Typical usage is that a proxy sits in front of this server"` (config.yaml.example, line 20)
3. No application-level control distinguishes browser endpoints from action endpoints
4. All POST action endpoints (e.g., `/do-vote/<eid>` at pages.py line ~482) and all GET page endpoints are treated identically for transport security
5. When a reverse proxy implements blanket HTTP→HTTPS redirect, action endpoints like `/do-vote/<eid>` are silently redirected instead of rejected

**Proof of Concept:**

A misconfigured API client or automated script sends vote data over HTTP to the action endpoint:
```http
POST http://steve.example.org/do-vote/abc123 HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: session=...

vote-issue1=yes&vote-issue2=no
```
With a blanket proxy redirect, this becomes a `302` to `https://steve.example.org/do-vote/abc123`. The vote payload was already transmitted in plaintext over HTTP. The redirect silently completes the operation over HTTPS, and the client assumes the interaction was secure — the plaintext leak goes undetected.

**Impact:**
- Vote data, session cookies, and election management commands may be transmitted in plaintext without detection
- For a voting/election system, this undermines vote secrecy and integrity
- A blanket proxy redirect creates false confidence that all communication is encrypted
- Per ASVS 4.1.2, action endpoints should fail loudly on HTTP rather than silently redirect

**Remediation:**

Implement middleware that enforces HTTPS on action/API endpoints and only redirects on browser-facing GET endpoints:

```python
from quart import request, abort, redirect

# Define which routes are user-facing browser pages (safe to redirect)
BROWSER_PREFIXES = {'/', '/voter', '/admin', '/about', '/profile', 
                    '/settings', '/vote-on/', '/manage/', '/manage-stv/',
                    '/static/', '/favicon.ico'}

@APP.before_request
async def enforce_transport_security():
    # Check X-Forwarded-Proto when behind reverse proxy
    proto = request.headers.get('X-Forwarded-Proto', request.scheme)
    
    if proto == 'https':
        return  # Already secure, continue
    
    path = request.path
    
    # Browser-facing GET endpoints: redirect to HTTPS
    if request.method == 'GET' and any(
        path == p or path.startswith(p) 
        for p in BROWSER_PREFIXES
    ):
        return redirect(request.url.replace('http://', 'https://', 1), code=301)
    
    # Action/API endpoints (POST, or state-changing GET like /do-*): 
    # reject with error, do NOT redirect
    abort(403, description='HTTPS required for this endpoint. '
          'Do not rely on HTTP-to-HTTPS redirects for action endpoints.')
```

Additionally, set HSTS headers for browser clients:
```python
@APP.after_request
async def set_security_headers(response):
    response.headers['Strict-Transport-Security'] = 'max-age=31536000; includeSubDomains'
    return response
```

#### ASVS-412-MED-002
**Finding: State-changing operations use GET method, compounding transport security risk**

**Gap Type:** Type A — State-changing operations exposed as GET endpoints are especially vulnerable to plaintext leakage via redirect

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `/do-open/<eid>`, `/do-close/<eid>`

**Related Functions Checked:**
- `do_open_endpoint(election)` 
- `do_close_endpoint(election)`

**Description:**

State-changing operations for opening and closing elections are exposed as GET endpoints rather than POST endpoints. This architectural choice compounds the transport security risk because GET requests are more likely to be logged, cached, and automatically redirected by intermediaries, increasing the attack surface for plaintext credential leakage.

**Vulnerable Code** (`pages.py`, `/do-open/<eid>`):
```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    result = await basic_info()

    ### check authz

    ### should open/keep a PersonDB instance in the APP
    pdb = steve.persondb.PersonDB.open(DB_FNAME)

    # Open the Election.
    election.open(pdb)
```

**Vulnerable Code** (`pages.py`, `/do-close/<eid>`):
```python
@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_close_endpoint(election):
    result = await basic_info()

    ### check authz

    # Close the Election.
    election.close()
```

**Data Flow:**
1. Election open/close operations are GET endpoints
2. GET requests are especially prone to being logged by proxies, browsers, and intermediaries
3. Session cookies and election IDs are exposed in the URL and headers
4. A blanket HTTP→HTTPS proxy redirect for GET requests is the most common redirect pattern
5. The state-changing operation may execute after redirect, but authentication cookies were already sent in plaintext on the initial HTTP request

**Proof of Concept:**
```http
GET http://steve.example.org/do-open/election-abc123 HTTP/1.1
Cookie: session=sensitive-session-token
```
The proxy redirects to HTTPS, but the session cookie was already transmitted in plaintext. An attacker on the network captures the session and can impersonate the election administrator.

**Impact:**
- Session tokens leaked in plaintext allow election administration hijacking
- Election state can be manipulated (open/close) by attacker with stolen session
- GET-based state changes are cached and logged more aggressively than POST

**Remediation:**

Convert state-changing operations to POST and add transport security enforcement:
```python
@APP.post('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    # HTTPS enforcement handled by before_request middleware
    result = await basic_info()
    # ... rest of handler
```

Similarly for `/do-close/<eid>`:
```python
@APP.post('/do-close/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    # HTTPS enforcement handled by before_request middleware
    result = await basic_info()
    # ... rest of handler
```

## Architecture Observations

1. **Proxy-dependent security model**: The configuration comments (`config.yaml.example` line 20) indicate the application is designed to run behind a reverse proxy. However, no application-level controls exist to enforce or verify proxy-provided security guarantees (e.g., `X-Forwarded-Proto` checking).

2. **No endpoint classification system**: All routes are defined with identical transport security characteristics. There is no framework-level or application-level distinction between browser pages, form action endpoints, and potential future API endpoints.

3. **Empty API module**: `api.py` currently contains no endpoints, but when API endpoints are added, they will inherit the same lack of transport security differentiation, making this finding more critical over time.

4. **Session cookies at risk**: Without HTTPS enforcement or `Secure` cookie flag verification at the application level, session cookies sent over HTTP can be intercepted, regardless of proxy-level redirects.
