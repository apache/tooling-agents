## Findings

### Medium Severity

#### ASVS-129-MED-001: Stored XSS via Unescaped Regex Match Output in HTML Construction

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `rewrite_description()`
- **Lines:** 38–46
- **Template:** `v3/server/templates/vote-on.ezt`

**Related Functions Checked:**
- `do_add_issue_endpoint()` (pages.py:~450) — accepts unvalidated `form.description`
- `do_edit_issue_endpoint()` (pages.py:~474) — accepts unvalidated `form.description`
- `vote_on_page()` (pages.py:~258) — calls `rewrite_description()` and renders output

**Description:**

The `rewrite_description()` function processes issue descriptions by applying a regex pattern to convert `doc:filename` references into HTML anchor tags. While the regex pattern itself is static and safe from injection, the captured filename from the regex match is inserted directly into HTML without escaping. Since issue descriptions originate from user input (authenticated committers), this creates a stored XSS vulnerability.

**Vulnerable Code:**

```python
def rewrite_description(issue):
    """Rewrite issue description: wrap in <pre> and convert doc:filename to links."""
    import re
    desc = issue.description
    # Replace doc:filename with <a> link
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'  # ← No escaping
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

**Template Rendering (vote-on.ezt):**

```html
<div id="description-[issues.iid]" class="description mt-2 text-muted small">
    [issues.description]
</div>
```

**Data Flow:**

1. **Input Source:** Authenticated committer submits issue description via:
   - `POST /api/add-issue` → `do_add_issue_endpoint()` (line ~450)
   - `POST /api/edit-issue` → `do_edit_issue_endpoint()` (line ~474)
   - Both accept `form.description` without sanitization or validation

2. **Storage:** Description stored in database as-is

3. **Processing:** When rendering ballot page:
   - `GET /vote-on/<eid>` → `vote_on_page()` (line ~258)
   - Calls `rewrite_description(issue)` for each issue
   - Regex `r'doc:([^\s]+)'` captures filename portion
   - Captured content inserted into HTML `<a>` tag without escaping

4. **Output:** EZT template renders `[issues.description]` without auto-escaping

5. **Missing Control:** No `html.escape()`, `markupsafe.escape()`, or equivalent applied to regex-captured content

**Gap Classification:** Type A — Entry point with no security control (regex match output → HTML construction without escaping)

**Proof of Concept:**

An authenticated committer creates an issue with the following description:

```
See doc:"><img/src=x onerror=alert(document.cookie)><a href=" for details
```

**Attack Flow:**

1. The regex pattern `r'doc:([^\s]+)'` matches: `doc:"><img/src=x onerror=alert(document.cookie)><a`
2. Captured group 1: `"><img/src=x onerror=alert(document.cookie)><a`
3. Replacement generates:

```html
<a href="/docs/ISSUEID/"><img/src=x onerror=alert(document.cookie)><a">"><img/src=x onerror=alert(document.cookie)><a</a>
```

4. Wrapped in `<pre>` tags and stored as `issue.description`
5. Rendered to all voters viewing the ballot page via `vote_on_page()`
6. Browser executes: `<img/src=x onerror=alert(document.cookie)>`

**Impact:**

- **Stored XSS:** Malicious script persists in database and executes for all users viewing the ballot
- **Affected Users:** All voters (authenticated committers) viewing the election
- **Attack Vector:** Any authenticated committer can inject scripts (not limited to election owner)
- **Potential Exploitation:**
  - Session token theft via `document.cookie`
  - Client-side vote manipulation by modifying form data
  - Phishing attacks via injected content
  - Privilege escalation if admin sessions are compromised

**ASVS 1.2.9 Relationship:**

While the requirement focuses on escaping special characters **in** regex patterns, this finding demonstrates that output encoding of regex **match results** is equally critical. The regex pattern itself is safe (static, no user input), but the captured content becomes an injection vector when used to construct HTML without proper escaping.

**Remediation:**

**Option 1: Escape entire description before regex processing (Recommended)**

```python
import html

def rewrite_description(issue):
    """Rewrite issue description: wrap in <pre> and convert doc:filename to links."""
    import re
    # Escape entire description first to neutralize any HTML/JS
    desc = html.escape(issue.description)
    
    def repl(match):
        filename = match.group(1)
        # filename is already HTML-escaped from the desc escaping above
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

**Option 2: Escape only regex-captured portions**

```python
import html

def rewrite_description(issue):
    """Rewrite issue description: wrap in <pre> and convert doc:filename to links."""
    import re
    desc = issue.description
    
    def repl(match):
        filename = html.escape(match.group(1))  # Escape captured content
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    
    # Escape entire description to protect non-matched portions
    desc = html.escape(desc)
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

**Option 3: Use templating with auto-escaping**

If migrating to a modern template engine (Jinja2, Django templates):

```python
from markupsafe import Markup, escape

def rewrite_description(issue):
    import re
    desc = escape(issue.description)
    
    def repl(match):
        filename = escape(match.group(1))
        return Markup(f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>')
    
    desc = re.sub(r'doc:([^\s]+)', repl, str(desc))
    return Markup(f'<pre>{desc}</pre>')
```

## Positive Security Patterns

The following security controls are effectively implemented and should be maintained:

### 1. Static Regex Patterns (✅ Excellent)

**Implementation:** The codebase uses exactly one regex pattern (`r'doc:([^\s]+)'` in `pages.py:43`), which is hardcoded and static.

**Security Benefit:**
- **Prevents Regex Injection:** No user input is ever incorporated into regex patterns, eliminating the risk of attackers injecting malicious regex metacharacters
- **Prevents ReDoS:** Static patterns can be analyzed for catastrophic backtracking vulnerabilities during code review
- **ASVS 1.2.9 Compliance:** Fully satisfies the requirement that special characters in regex patterns must be escaped (no dynamic patterns exist)

**Example:**
```python
# Safe: Pattern is hardcoded
desc = re.sub(r'doc:([^\s]+)', repl, desc)

# Would be unsafe (not present in codebase):
# pattern = f"doc:{user_input}"  # ❌ Never done
# desc = re.sub(pattern, repl, desc)
```

### 2. Simple Non-Backtracking Regex Design (✅ Excellent)

**Implementation:** The pattern `[^\s]+` uses a negated character class without nested quantifiers or alternation.

**Security Benefit:**
- **ReDoS Immunity:** Cannot trigger catastrophic backtracking regardless of input
- **Predictable Performance:** Linear time complexity O(n) for any input
- **Maintainability:** Simple pattern is easy to audit and understand

### 3. Parameterized SQL Queries (✅ Excellent)

**Implementation:** All database queries in `v3/queries.yaml` use `?` parameter placeholders exclusively.

**Security Benefit:**
- **SQL Injection Prevention:** User input is never concatenated into SQL strings
- **Type Safety:** Database driver enforces parameter type checking
- **100% Coverage:** No raw SQL construction found anywhere in codebase

**Example from queries.yaml:**
```yaml
get_election: SELECT * FROM elections WHERE eid = ?
add_issue: INSERT INTO issues (eid, title, description) VALUES (?, ?, ?)
```

### 4. Hardcoded LDAP Filters (✅ Excellent)

**Implementation:** LDAP search in `asf-load-ldap.py:43` uses static filter `'uid=*'`.

**Security Benefit:**
- **LDAP Injection Prevention:** No user input in filter string
- **Predictable Behavior:** Filter scope is fixed and auditable

### 5. EZT JS/HTML Escaping in JavaScript Context (✅ Good)

**Implementation:** Templates `manage.ezt` and `manage-stv.ezt` use `[format "js,html"]...[end]` when embedding data into JavaScript onclick handlers.

**Security Benefit:**
- **Prevents XSS in JS Context:** Properly encodes for JavaScript string literals
- **Context-Aware Escaping:** Uses appropriate encoding for the output context

**Example from manage.ezt:**
```html
<button onclick="editIssue([format "js,html"][issues.iid][end], ...)">
```

### 6. Database-Level Parameter Binding via asfpy.db (✅ Excellent)

**Implementation:** The `asfpy.db` wrapper ensures all query parameters are bound safely through SQLite's parameterized query interface.

**Security Benefit:**
- **Defense in Depth:** Even if queries.yaml were modified, the DB layer enforces parameter binding
- **Framework-Level Protection:** Reduces risk of developer error

## Architecture Observations

### 1. Minimal Regex Surface Area

**Observation:** The codebase uses regex in exactly one location (`rewrite_description` function), representing approximately 0.01% of the codebase.

**Security Implication:**
- **Positive:** Dramatically reduces attack surface for regex-related vulnerabilities
- **Risk Concentration:** The single regex usage point becomes a critical control point
- **Maintainability:** Easy to audit and monitor for changes

**Recommendation:** Maintain this minimalist approach. If future features require pattern matching, consider alternatives (string methods, SQL LIKE queries) before adding regex.

### 2. EZT Template Engine Does Not Auto-Escape

**Observation:** EZT (Easy Template) outputs all variables as-is by default. The `[format "js,html"]` directive is available but must be explicitly applied.

**Security Implication:**
- **Broad XSS Risk:** Every template variable is a potential XSS vector unless manually escaped
- **Developer Burden:** Requires security awareness for every template variable
- **Inconsistent Application:** Only 2 of 13 templates use `[format]` directives

**Affected Templates:**
- ✅ `manage.ezt`, `manage-stv.ezt` — Use `[format "js,html"]` for JS context
- ❌ `vote-on.ezt`, `home.ezt`, `voter.ezt`, etc. — No escaping applied

**Recommendation:** Consider migration to a modern template engine with auto-escaping (Jinja2, Django templates) or implement a global EZT wrapper that escapes by default.

### 3. Admin-to-Voter Trust Boundary

**Observation:** The `rewrite_description` function processes descriptions set by admin users (committers with election management rights) and renders them to voter users (all committers viewing the ballot).

**Security Implication:**
- **Cross-User Rendering:** Even though both roles are authenticated committers, this represents a trust boundary
- **Privilege Assumptions:** Code assumes admin-created content is safe, but any committer can be an admin
- **Stored XSS Impact:** Malicious content persists and affects multiple users over time

**Trust Model:**
```
Committer A (Admin) → Creates Issue → Stored in DB → Rendered to Committer B (Voter)
                                                    ↓
                                            Potential XSS if not escaped
```

**Recommendation:** Apply defense-in-depth: escape all user-generated content regardless of privilege level.

### 4. No `re.escape()` Needed (Current State)

**Observation:** Since no user input flows into regex patterns, there is no need for `re.escape()` in the current codebase.

**Security Implication:**
- **Current State:** Fully compliant with ASVS 1.2.9 by design (no dynamic patterns)
- **Future Risk:** If search features or dynamic pattern matching are added, developers may not be aware of `re.escape()` requirement

**Recommendation:** Document a coding standard for future development:

```python
# SECURITY POLICY: If user input must be used in regex patterns, use re.escape()

import re

# ❌ NEVER DO THIS:
# pattern = f"filename_{user_input}"
# re.search(pattern, text)

# ✅ CORRECT APPROACH:
# pattern = f"filename_{re.escape(user_input)}"
# re.search(pattern, text)
```

### 5. Consistent Use of Parameterized Queries

**Observation:** 100% of database interactions use parameterized queries via `queries.yaml` with `?` placeholders.

**Security Implication:**
- **Exemplary Practice:** No SQL injection vulnerabilities found
- **Architectural Strength:** Centralized query definitions in YAML make auditing straightforward
- **Maintainability:** Clear separation between query structure and data

**Example Pattern:**
```python
# queries.yaml
get_issues: SELECT * FROM issues WHERE eid = ?

# pages.py
issues = db.fetch("get_issues", (eid,))
```

### Recommended Remediation Code

**For ASVS-129-MED-001 (Immediate Priority):**

```python
import html

def rewrite_description(issue):
    """
    Rewrite issue description: wrap in <pre> and convert doc:filename to links.
    
    Security: HTML-escapes all content before processing to prevent XSS.
    """
    import re
    
    # SECURITY: Escape entire description to neutralize HTML/JS injection
    desc = html.escape(issue.description)
    
    # Replace doc:filename with <a> link
    # Note: filename is already escaped from the html.escape() above
    def repl(match):
        filename = match.group(1)  # Already HTML-escaped
        # Use html.escape on URL path component for defense-in-depth
        safe_iid = html.escape(str(issue.iid))
        return f'<a href="/docs/{safe_iid}/{filename}">{filename}</a>'
    
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
    
    return issue
```

import unittest
from unittest.mock import Mock

class TestRewriteDescription(unittest.TestCase):
    
    def test_xss_prevention_in_doc_link(self):
