### High Severity

#### ASVS-212-HIGH-001: Vote Content Not Validated Against Issue Vote Type

**Severity:** HIGH  
**Gap Type:** Type B (Control recognized but NOT implemented)

**Location:** `v3/steve/election.py`, function `add_vote()`, lines 218-230

**Related Functions Checked:**
- `v3/server/pages.py:do_vote_endpoint()` L389-426
- `v3/steve/election.py:add_vote()` L218-230

**Vulnerable Code:**
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting    # <-- TODO, never implemented

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)

    # Pass MAYVOTE.SALT for PBKDF.
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)

    self.c_add_vote.perform(vote_token, ciphertext)
```

**Also missing in caller:** `v3/server/pages.py`, function `do_vote_endpoint()`, lines 389-426
```python
for iid, votestring in votes.items():
    if iid not in issue_dict:
        await flash_danger(f'Invalid issue ID: {iid}')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
    # No validation of votestring format against issue_dict[iid].vtype
    try:
        election.add_vote(result.uid, iid, votestring)
```

**Data Flow:**
```
HTTP POST form['vote-{iid}'] 
  → do_vote_endpoint() (no format check) 
  → election.add_vote() (TODO comment, no check) 
  → crypto.create_vote() (encrypts raw) 
  → database
```

**Proof of Concept:**
```http
POST /do-vote/abc123 HTTP/1.1
Content-Type: application/x-www-form-urlencoded

vote-issue1=INVALID_GARBAGE&vote-issue2=zzz,yyy,xxx_nonexistent_labels
```
For a YNA issue, `INVALID_GARBAGE` is accepted instead of `y`, `n`, or `a`. For an STV issue, non-existent candidate labels are accepted without validation against the issue's `kv.labelmap`.

**Impact:** Invalid vote data is encrypted and stored. During tallying, `vtypes.vtype_module(issue.type).tally()` receives corrupted data, potentially causing crashes, incorrect tallies, or silent data loss. This is the combined data consistency check (votestring + issue.vtype + issue.kv) that ASVS 2.1.2 requires documentation and implementation for.

**Remediation:**
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)

    # Validate votestring against the issue's vote type
    issue = self.q_get_issue.first_row(iid)
    if not issue:
        raise IssueNotFound(iid)
    
    m = vtypes.vtype_module(issue.type)
    m.validate(votestring, self.json2kv(issue.kv))  # raises ValueError on invalid

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    if not mayvote:
        raise VoterNotEligible(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

#### ASVS-212-HIGH-002: Election Owner Authorization Tracked But Never Enforced

**Severity:** HIGH  
**Gap Type:** Type B (Control EXISTS in data model but NOT CALLED at entry points)

**Location:** `v3/server/pages.py`, multiple endpoints

**Control Definition:** `v3/queries.yaml` stores `owner_pid` and `v3/steve/election.py:create()` records the owner.

**Entry Points Missing Enforcement:**

| Endpoint | Function | Line | Comment in Code |
|----------|----------|------|-----------------|
| `GET /manage/<eid>` | `manage_page` | 331 | `### check authz` |
| `POST /do-add-issue/<eid>` | `do_add_issue_endpoint` | 486 | `### check authz` |
| `POST /do-edit-issue/<eid>/<iid>` | `do_edit_issue_endpoint` | 510 | `### check authz` |
| `POST /do-delete-issue/<eid>/<iid>` | `do_delete_issue_endpoint` | 533 | `### check authz` |
| `GET /do-open/<eid>` | `do_open_endpoint` | 451 | `### check authz` |
| `GET /do-close/<eid>` | `do_close_endpoint` | 468 | `### check authz` |
| `POST /do-set-open_at/<eid>` | `do_set_open_at_endpoint` | 375 | via `_set_election_date` |
| `POST /do-set-close_at/<eid>` | `do_set_close_at_endpoint` | 382 | via `_set_election_date` |

**Vulnerable Code (representative):**
```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  # Any committer can access
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz                     # <-- Recognized but not implemented
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)                  # Irreversible operation proceeds
```

**Data Flow:**
```
Authenticated committer (any) 
  → do_open_endpoint(eid) 
  → load_election(eid) (loads any valid EID, no owner check) 
  → election.open() (irreversible)
```

**Proof of Concept:** Any authenticated ASF committer can open, close, or modify elections owned by other users by navigating to `/do-open/{victim_eid}` or `POST /do-add-issue/{victim_eid}`.

**Impact:** The combined data item (requesting user's PID + election.owner_pid) is never validated for consistency. Any committer can perform irreversible operations (open, close) on elections they don't own, and modify election content (add/edit/delete issues) arbitrarily.

**Remediation:**
```python
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            ...
        
        # Enforce owner authorization
        result = await basic_info()
        md = e.get_metadata()
        if md.owner_pid != result.uid:
            quart.abort(403, 'Not the election owner')
        
        return await func(e)
    return loader
```

#### ASVS-212-HIGH-003: Election State Guards Use `assert` — Bypassed with Python Optimization

**Severity:** HIGH  
**Gap Type:** Type C (Control CALLED but result can be silently ignored)

**Location:** `v3/steve/election.py`, multiple methods

**Affected Methods:**

| Method | Line | Assert Statement | Consequence If Bypassed |
|--------|------|-----------------|------------------------|
| `open()` | 78 | `assert self.is_editable()` | Re-open election, reset salt/key, invalidate all votes |
| `close()` | 107 | `assert self.is_open()` | Close editable election without opening |
| `delete()` | 65 | `assert self.is_editable()` | Delete open/closed election, destroy all vote data |
| `add_salts()` | 115 | `assert self.is_editable()` | Regenerate salts on open election, corrupt vote tokens |
| `add_issue()` | 179-180 | `assert self.is_editable()` + `assert vtype in vtypes.TYPES` | Add issues to open election + bypass vtype whitelist |
| `edit_issue()` | 190-191 | `assert self.is_editable()` + `assert vtype in vtypes.TYPES` | Edit issues on open election |
| `delete_issue()` | 201 | `assert self.is_editable()` | Delete issues from open election |
| `add_voter()` | 213 | `assert self.is_editable()` | Add voters to open/closed election |

**Vulnerable Code (representative):**
```python
def open(self, pdb):
    # Double-check the Election is in the editing state.
    assert self.is_editable()       # <-- Removed by python -O
    
    self.add_salts()                # Proceeds unconditionally
    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)
    self.c_open.perform(salt, opened_key, self.eid)  # Overwrites existing salt/key
```

**Contrast with properly implemented check in `add_vote()`:**
```python
def add_vote(self, pid, iid, votestring):
    md = self._all_metadata(self.S_OPEN)  # Raises ElectionBadState, not assert
```

**Impact:** If the application is deployed with `PYTHONOPTIMIZE=1` or `python -O` (common in production), all election state machine guards are silently removed. Combined with the missing owner authorization (ASVS-212-HIGH-002), any committer could re-open a closed election, invalidating all previous votes by regenerating the salt and opened_key.

**Remediation:**
```python
def open(self, pdb):
    # Use _all_metadata with required_state instead of assert
    self._all_metadata(required_state=self.S_EDITABLE)
    
    self.add_salts()
    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)
    self.c_open.perform(salt, opened_key, self.eid)

def add_issue(self, title, description, vtype, kv):
    self._all_metadata(required_state=self.S_EDITABLE)
    if vtype not in vtypes.TYPES:
        raise ValueError(f'Invalid vote type: {vtype}')
    ...
```

### Medium Severity

#### ASVS-212-MEDIUM-001: No Cross-Field Date Consistency Validation

**Severity:** MEDIUM

**Location:** `v3/server/pages.py`, function `_set_election_date()`, lines 79-100

**Related Functions Checked:**
- `v3/server/pages.py:_set_election_date()` L79-100
- `v3/server/pages.py:do_set_open_at_endpoint()` L375
- `v3/server/pages.py:do_set_close_at_endpoint()` L382

**Vulnerable Code:**
```python
async def _set_election_date(election, field):
    result = await basic_info()
    ### check authz
    data = await quart.request.get_json()
    date_str = data.get('date')
    if not date_str:
        quart.abort(400, 'Missing date')

    try:
        dt = datetime.datetime.fromisoformat(date_str).date()
    except ValueError:
        quart.abort(400, 'Invalid date format')

    # No cross-field validation: open_at vs close_at
    if field == 'open_at':
        election.set_open_at(dt)        # Proceeds without checking close_at
    elif field == 'close_at':
        election.set_close_at(dt)       # Proceeds without checking open_at
```

**Data Flow:**
```
JSON body {date: "2020-01-01"} 
  → _set_election_date() (format-only validation) 
  → election.set_open_at() 
  → database
```

**Proof of Concept:**
```http
POST /do-set-close_at/abc123 HTTP/1.1
Content-Type: application/json

{"date": "2020-01-01"}
```
Sets close_at to a date in the past, or before the open_at date.

**Impact:** The combined data items (open_at + close_at) are never validated for contextual consistency. An election could have close_at before open_at, or dates in the past, creating logically inconsistent election metadata.

**Remediation:**
```python
async def _set_election_date(election, field):
    ...
    md = election.get_metadata()
    
    if field == 'open_at':
        if md.close_at and dt.timestamp() >= md.close_at:
            quart.abort(400, 'Open date must be before close date')
        election.set_open_at(int(datetime.datetime.combine(dt, datetime.time()).timestamp()))
    elif field == 'close_at':
        if md.open_at and int(datetime.datetime.combine(dt, datetime.time()).timestamp()) <= md.open_at:
            quart.abort(400, 'Close date must be after open date')
        election.set_close_at(int(datetime.datetime.combine(dt, datetime.time()).timestamp()))
```

#### ASVS-212-MEDIUM-002: State-Changing Operations Use HTTP GET

**Severity:** MEDIUM

**Location:** `v3/server/pages.py`, lines 447 and 464

**Related Functions Checked:**
- `v3/server/pages.py:do_open_endpoint()` L447
- `v3/server/pages.py:do_close_endpoint()` L464
- `v3/server/templates/manage.ezt` L~267

**Vulnerable Code:**
```python
@APP.get('/do-open/<eid>')                    # GET for irreversible state change
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    ...
    election.open(pdb)                         # Irreversible

@APP.get('/do-close/<eid>')                   # GET for irreversible state change
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    ...
    election.close()                           # Irreversible
```

**Also triggered from templates:** `v3/server/templates/manage.ezt`, line ~267:
```javascript
confirmOpen.addEventListener('click', () => {
    openModal.hide();
    window.location.href = '/do-open/[eid]';   // GET request
});
```

**Proof of Concept:**
```html
<img src="https://target.example.org/do-open/known_election_id" width="0" height="0">
```

**Impact:** Opening and closing elections are irreversible, permanent state changes. Using GET violates HTTP semantics and makes these operations trivially exploitable via CSRF without JavaScript (image tags, link prefetch, browser preloading). Combined with the acknowledged CSRF token placeholder (`basic.csrf_token = 'placeholder'`), this creates a practical attack vector. The combination (HTTP method + operation criticality) is contextually inconsistent with safe web practices.

**Remediation:**
```python
@APP.post('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    # Validate CSRF token
    ...
    election.open(pdb)
```

## Positive Security Patterns

1. **Parameterized SQL Queries:** All queries in `queries.yaml` use `?` placeholders, properly preventing SQL injection via `asfpy.db`.

2. **LDAP Connection Security:** `asf-load-ldap.py` uses `ldaps://` (LDAP over TLS) and data comes from the trusted LDAP directory, not user input. The `uid=*` search filter is a constant, preventing LDAP injection.

3. **Sensitive Data Exclusion:** `get_metadata()` and `get_issue()` explicitly exclude `salt` and `opened_key` from returned data, with comments like `# NEVER return person.salt`.

4. **Cryptographic Vote Anonymization:** Vote tokens derived from `(opened_key, pid, iid, salt)` provide strong separation between voter identity and vote content.

5. **Tamper Detection:** `is_tampered()` recomputes the `opened_key` from all election data to detect unauthorized modifications.

6. **ID Collision Handling:** `create()` and `add_issue()` retry with new IDs on `sqlite3.IntegrityError`, preventing ID collision issues.

7. **EZT Template Escaping:** Templates use `[format "js,html"]...[end]` for proper context-aware escaping when injecting values into JavaScript string literals.

8. **STV Candidate Shuffling:** `random.shuffle(issue.candidates)` prevents ordering bias in ballot presentation.

9. **Proper State Check in Voting Path:** `add_vote()` uses `_all_metadata(self.S_OPEN)` which raises an exception rather than using an `assert`, making the voting path state check robust.

---

## Architecture Observations

1. **State Machine Implementation Split:** The election state machine is defined in `election.py` but enforcement is split between `assert` statements (management operations) and `_all_metadata(required_state=)` (voting operations). This inconsistency means the most critical path (voting) is properly protected, but management operations are fragile.

2. **Authorization Layer Absence:** The codebase has a clear data model for authorization (`owner_pid`, `authz`, `mayvote`) but lacks an authorization enforcement layer between web endpoints and business logic. The `### check authz` comments appear 10+ times, indicating this is a known architectural gap.

3. **Validation Responsibility Gap:** Input validation is not clearly assigned to either the web layer (`pages.py`) or the business logic layer (`election.py`). The web layer validates some format aspects (date format, IID existence) while the business logic layer handles state checks. Vote format validation falls through the gap entirely.
