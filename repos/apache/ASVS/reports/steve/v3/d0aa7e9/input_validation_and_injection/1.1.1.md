## Findings

### Critical Severity

#### ASVS-111-CRI-001
**Title:** EZT output encoding control (`[format]`) exists but is not applied at the majority of output points, enabling reflected and stored XSS

**Severity:** CRITICAL (Type B — Control exists but not applied)

**Location:**
- **Files:** Multiple templates
  - `v3/server/templates/e_bad_eid.ezt:8`
  - `v3/server/templates/e_bad_iid.ezt:8`
  - `v3/server/templates/e_bad_pid.ezt:8`
  - `v3/server/templates/manage.ezt:176,180`
  - `v3/server/templates/manage-stv.ezt:134`
  - `v3/server/templates/vote-on.ezt:108-109`
  - `v3/server/templates/voter.ezt:35,88`

**Related Functions Checked:**
- `raise_404()` in `pages.py` (error template rendering)
- `load_election()` decorator in `pages.py:198+`
- `load_issue()` decorator in `pages.py`
- EZT template rendering engine

**Description:**

The EZT templating engine provides the `[format "html"]` directive for HTML encoding and `[format "js,html"]` for JavaScript+HTML encoding. These controls are correctly used in a few JavaScript onclick handlers in `manage.ezt` and `manage-stv.ezt`. However, **no encoding is applied** in any HTML body context across the entire template set. User-controlled values — including URL-derived parameters, issue titles, descriptions, election titles, and owner names — are rendered directly as `[variable]` without any encoding.

**Vulnerable Code:**

Error template (`e_bad_eid.ezt:8`):
```html
The Election ID ([eid]) does not exist,
```

Error template (`e_bad_iid.ezt:8`):
```html
The Issue ID ([iid]) does not exist,
```

Issue display in HTML body (`manage.ezt:176`):
```html
<strong>[issues.title]</strong>
```

**Contrast: Where Control IS Applied (same codebase):**

JavaScript onclick handler (`manage.ezt:180`):
```html
onclick="openEditIssueModal('[issues.iid]',
                            '[format "js,html"][issues.title][end]',
                            '[format "js,html"][issues.description][end]')"
```

**Data Flow (Reflected XSS via error template):**

1. **Input:** Attacker crafts URL: `https://target/vote-on/%3Cimg%20src%3Dx%20onerror%3Dalert(document.cookie)%3E`
2. **Decode:** Quart URL-decodes path parameter once → `<img src=x onerror=alert(document.cookie)>`
3. **Processing:** `load_election()` in `pages.py:198` sets `result.eid = eid` (raw decoded value)
4. **Error Handling:** `raise_404(T_BAD_EID, result)` renders template
5. **Output:** `e_bad_eid.ezt` outputs `[eid]` without encoding → **XSS executes**

**Source:** URL path parameter (`eid`, `iid`) → `pages.py` load decorators  
**Sink:** EZT template `[variable]` output → HTML response body  
**Missing Control:** `[format "html"]` not applied

**Proof of Concept:**

```http
GET /vote-on/%3Cimg%20src%3Dx%20onerror%3Dalert(document.cookie)%3E HTTP/1.1
Host: target
Cookie: [valid session]
```

**Response body includes:**
```html
The Election ID (<img src=x onerror=alert(document.cookie)>) does not exist,
```

**Impact:**
- **Reflected XSS:** Session hijacking, credential theft, phishing attacks
- **Stored XSS:** Admin-created issue/election titles affect all voters viewing a ballot
- **Attack Vector:** Direct URL manipulation, malicious links, stored content
- **Scope:** All authenticated users, including administrators

**Remediation:**

Apply `[format "html"]` to all user-derived template variables in HTML body contexts:

```html
<!-- Error templates -->
The Election ID ([format "html"][eid][end]) does not exist,

<!-- Issue display -->
<strong>[format "html"][issues.title][end]</strong>

<!-- Election titles in voter.ezt -->
<h5 class="card-title mb-3">[format "html"][open_elections.title][end]</h5>

<!-- Issue descriptions -->
<p>[format "html"][issues.description][end]</p>
```

**Alternative (Recommended):** Migrate to a template engine with auto-escaping by default (e.g., Jinja2 with `autoescape=True`) to eliminate this entire class of vulnerabilities architecturally.

### High Severity

#### ASVS-111-HIG-002
**Title:** `rewrite_description()` constructs HTML from unencoded user input, violating encoding-before-processing order

**Severity:** HIGH

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `rewrite_description()`
- **Lines:** 46–54

**Related Functions Checked:**
- Template rendering in `vote-on.ezt:108`
- Issue creation in `do_add_issue_endpoint()`
- Database storage operations

**Vulnerable Code:**

```python
def rewrite_description(issue):
    import re
    desc = issue.description
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

**Description:**

This function wraps a raw, unencoded issue description in `<pre>` tags and converts `doc:filename` patterns into `<a>` links. The user-controlled description content and the regex-captured `filename` are inserted into HTML markup without any HTML encoding. The resulting HTML string is then rendered in templates without encoding (per Finding ASVS-111-CRI-001).

**This violates ASVS 1.1.1's requirement that encoding should occur before further processing** — here, HTML construction (processing) occurs on raw input without prior encoding of user-controlled parts.

**Data Flow:**

1. **Input:** Admin creates issue with description: `<script>alert(1)</script> doc:test"onmouseover="alert(2)`
2. **Storage:** Description stored raw in DB (correct per ASVS 1.1.1 storage guidance)
3. **Processing:** `rewrite_description()` wraps in `<pre>`, regex creates `<a>` link from capture group
4. **Result:** `<pre><script>alert(1)</script> <a href="/docs/IID/test"onmouseover="alert(2)">test"onmouseover="alert(2)</a></pre>`
5. **Output:** Template renders without encoding → **both XSS payloads execute**

**Source:** Issue description from database (admin-controlled input)  
**Sink:** HTML output via `issue.description` in `vote-on.ezt:108`  
**Missing Control:** HTML encoding of user content before HTML construction

**Proof of Concept:**

Create an issue with description:
```
Review this: doc:x"onclick="alert(document.cookie)" data-x="
```

**This produces:**
```html
<pre>Review this: <a href="/docs/IID/x"onclick="alert(document.cookie)" data-x="">x"onclick="alert(document.cookie)" data-x=""</a></pre>
```

**Impact:**
- **Stored XSS:** Admin-created content executes JavaScript in all voters' browsers when viewing the ballot
- **Privilege Escalation:** Lower-privilege admins can attack higher-privilege administrators
- **Persistent Attack:** Affects all users viewing the election until description is modified

**Remediation:**

```python
import html
from urllib.parse import quote

def rewrite_description(issue):
    import re
    # Encode user content FIRST (before HTML construction)
    desc = html.escape(issue.description)
    
    def repl(match):
        filename = match.group(1)  # Already HTML-encoded by escape above
        # URL-encode filename for href, use HTML-encoded version for display
        return f'<a href="/docs/{html.escape(issue.iid)}/{quote(filename)}">{filename}</a>'
    
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

#### ASVS-111-HIG-003
**Title:** STV candidate data embedded in JavaScript string literals without JS encoding, despite `[format "js,html"]` being used elsewhere in the same template

**Severity:** HIGH (Type B — Control exists but not applied)

**Location:**
- **File:** `v3/server/templates/vote-on.ezt`
- **Lines:** Within the `<script>` block (STV_CANDIDATES object definition)

**Related Functions Checked:**
- STV issue creation in `election.py`
- Candidate data storage in `kv` JSON column
- Template rendering in `pages.py`

**Vulnerable Code:**

```javascript
const STV_CANDIDATES = {
[for issues][is issues.vtype "stv"]    "[issues.iid]": {
      seats: [issues.seats],
      title: "[issues.title]",
      candidates: [
[for issues.candidates]        { label: "[issues.candidates.label]", name: "[issues.candidates.name]" },
[end]      ]
    },
[end][end]  };
```

**Contrast: Where Same Template Uses Encoding (different section):**

```html
onclick="openSTVModal('[issues.iid]')"
```

*(Note: Even this onclick doesn't use `[format]`, but `manage.ezt` does for similar patterns)*

**Description:**

STV candidate names and labels from the database are embedded directly into JavaScript string literals using bare `[issues.candidates.name]` without `[format "js"]`. If a candidate name contains a double quote (`"`), backslash, or `</script>`, it will break out of the JavaScript string or script block. 

**The `[format "js,html"]` control exists and is used in the management templates for identical scenarios** (user text in JS strings), but it is not applied here.

**Data Flow:**

1. **Input:** Election admin configures STV issue with candidate name containing `"` or `</script>`
2. **Storage:** Candidate data stored raw in DB (in `kv` JSON column)
3. **Processing:** `vote-on.ezt` embeds candidate name in JS string literal without encoding
4. **Output:** JavaScript parse error or **XSS** when voter loads ballot page

**Source:** STV candidate names from `issue.kv.labelmap` (admin-controlled)  
**Sink:** JavaScript string literal in `vote-on.ezt` template  
**Missing Control:** `[format "js"]` or `[format "js,html"]`

**Proof of Concept:**

Create STV issue with a candidate named:
```
Alice"; alert(document.cookie);//
```

**Rendered JavaScript:**
```javascript
{ label: "a", name: "Alice"; alert(document.cookie);//" },
```

**Impact:**
- **Stored XSS:** Executes in every voter's browser when viewing the STV ballot
- **Vote Manipulation:** Attacker could inject JavaScript to manipulate voting UI or steal votes
- **Session Hijacking:** Cookie theft affects all voters viewing the ballot

**Remediation:**

```javascript
const STV_CANDIDATES = {
[for issues][is issues.vtype "stv"]    "[format "js"][issues.iid][end]": {
      seats: [issues.seats],
      title: "[format "js"][issues.title][end]",
      candidates: [
[for issues.candidates]        { label: "[format "js"][issues.candidates.label][end]", 
          name: "[format "js"][issues.candidates.name][end]" },
[end]      ]
    },
[end][end]  };
```

#### ASVS-111-HIG-004
**Title:** User-controlled input interpolated into flash messages then rendered without HTML encoding

**Severity:** HIGH

**Location:**
- **Files:** 
  - `v3/server/pages.py` (multiple endpoints)
  - `v3/server/templates/flashes.ezt:3`
- **Functions:**
  - `do_vote_endpoint()` at line 504
  - `do_create_endpoint()` at line 535
  - `do_add_issue_endpoint()` at line 598

**Related Functions Checked:**
- `flash_danger()` helper function
- `flash_success()` helper function
- Session management for flash storage
- Template rendering in `flashes.ezt`

**Vulnerable Code in Template:**

`flashes.ezt:3`:
```html
[flashes.message]
```

**Vulnerable Code in Python (examples):**

`pages.py` line 504 (`do_vote_endpoint`):
```python
await flash_danger(f'Invalid issue ID: {iid}')
```

`pages.py` line 535 (`do_create_endpoint`):
```python
await flash_success(f'Created election: {form.title}')
```

`pages.py` line 598 (`do_add_issue_endpoint`):
```python
await flash_success(f'Issue "{form.title}" has been added.')
```

**Description:**

Multiple route handlers interpolate user-controlled values into flash messages using Python f-strings. The `iid` in `do_vote_endpoint` is extracted from form field names (`vote-<iid>`), making it directly controllable by the requester. The `form.title` comes from form input. These interpolated messages are stored in the session and rendered in `flashes.ezt` without any `[format "html"]` encoding. 

**The processing order violation is:** user input is mixed with HTML-destined strings (processing) without prior encoding.

**Data Flow (Reflected XSS via iid):**

1. **Input:** Attacker submits POST to `/do-vote/<valid-eid>` with form field `vote-<img/src=x onerror=alert(1)>=y`
2. **Extraction:** `do_vote_endpoint` extracts iid as `<img/src=x onerror=alert(1)>`
3. **Processing:** iid not in issue_dict → `flash_danger(f'Invalid issue ID: {iid}')`
4. **Storage:** Flash stored in session: `Invalid issue ID: <img/src=x onerror=alert(1)>`
5. **Redirect:** Redirect to `/vote-on/<eid>`
6. **Output:** `flashes.ezt` renders message without encoding → **XSS executes**

**Source:** Form field names and values from `quart.request.form`  
**Sink:** `flashes.ezt` template `[flashes.message]` output  
**Missing Control:** `[format "html"]` in template OR `html.escape()` in Python before interpolation

**Proof of Concept:**

```http
POST /do-vote/VALID_EID HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: [valid session]

vote-%3Cimg+src%3Dx+onerror%3Dalert(1)%3E=y
```

**Impact:**
- **Reflected XSS via flash message:** Session hijacking, credential theft
- **Exploitability increased** by the acknowledged lack of CSRF protection (placeholder token)
- **Attack Vector:** Form field manipulation, malicious form submissions
- **Scope:** All authenticated users

**Remediation (Template-side fix):**

```html
[for flashes]
    <div class="alert alert-[flashes.category] alert-dismissible fade show" role="alert">
        [format "html"][flashes.message][end]
        <button type="button" class="btn-close" data-bs-dismiss="alert" aria-label="Close"></button>
    </div>
[end]
```

**Remediation (Python-side defense-in-depth):**

```python
import html

await flash_danger(f'Invalid issue ID: {html.escape(iid)}')
await flash_success(f'Created election: {html.escape(form.title)}')
await flash_success(f'Issue "{html.escape(form.title)}" has been added.')
```

## Architecture Observations

### 1. EZT Template Engine Design

**Observation:** EZT has no auto-escaping capability. Unlike Jinja2 with `autoescape=True`, EZT outputs all variables raw unless explicitly wrapped in `[format]` directives.

**Implication:** This design requires developers to remember to encode at every single output point — a pattern known to produce gaps in security coverage. The current codebase demonstrates this: encoding is applied in approximately **5% of output locations**.

**Risk:** High likelihood of introducing XSS vulnerabilities when adding new features or modifying templates.

**Recommendation:** Consider migrating to a template engine with auto-escaping by default (e.g., Jinja2, Mako with default filters) to eliminate this entire class of vulnerabilities architecturally.

---

### 2. Processing Order in `rewrite_description()`

**Observation:** The function constructs HTML from raw content, meaning the output is a mix of trusted structure (`<pre>`, `<a>` tags) and untrusted content (user descriptions).

**ASVS 1.1.1 Violation:** This makes it impossible to apply a simple `[format "html"]` at the template level (it would encode the intentional HTML tags). The fix must happen inside the function — **encode the user content before wrapping in HTML** — which is the exact order ASVS 1.1.1 prescribes.

**Current Flow (Incorrect):**
```
Raw Input → HTML Construction → Output (no encoding possible)
```

**Required Flow (ASVS 1.1.1 Compliant):**
```
Raw Input → Encode User Content → HTML Construction → Output
```

---

### 3. Flash Message Architecture

**Observation:** User input is mixed with message text in Python f-strings before the template layer has any opportunity to encode.

**Current Implementation:**
```python
await flash_danger(f'Invalid issue ID: {iid}')  # iid is user-controlled
```

**Issue:** While template-side encoding (`[format "html"]`) would still work (encoding the entire message), a defense-in-depth approach should encode user values before interpolation.

**Best Practice:** Implement encoding at both layers:
1. **Python layer:** Escape user values before string interpolation
2. **Template layer:** Apply `[format "html"]` to the entire message

---

### 4. Input Canonicalization (Compliant)

**Observation:** Quart performs URL decoding, form parsing, and JSON parsing exactly once. The application does not perform any additional decoding. Data is stored in its canonical form without encoding.

**Assessment:** ✅ **This aspect of ASVS 1.1.1 is fully satisfied.**

**Evidence:**
- URL parameters decoded once by Quart framework
- Form data parsed once via `request.form`
- JSON parsed once via `request.get_json()`
- No manual decoding operations in application code
- Database storage uses raw canonical values

---

### 5. Inconsistent Security Control Application

**Observation:** The codebase demonstrates knowledge of proper encoding techniques (evidenced by correct usage in `manage.ezt` onclick handlers) but fails to apply these controls consistently across the application.

**Pattern:**
- ✅ Encoding applied: JavaScript onclick handlers in management templates
- ❌ Encoding missing: HTML body content, error messages, flash messages, STV candidate data

**Root Cause:** Lack of:
1. Secure coding standards documentation
2. Code review checklist for output encoding
3. Automated security testing (SAST/DAST)
4. Template security linting
