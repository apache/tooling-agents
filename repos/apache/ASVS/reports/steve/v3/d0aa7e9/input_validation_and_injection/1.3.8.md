## Findings

### Medium Severity

#### ASVS-1308-MED-001: Missing Vote String Sanitization Before Encryption and Storage

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with NO control (acknowledged via TODO comment)

**Location:**
- **File:** `v3/steve/election.py`
- **Method:** `add_vote`
- **Lines:** ~244–255

**Related Entry Point:**
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint`
- **Lines:** ~463–497

**Related Functions Checked:**
- `election.py`: `add_issue()`, `edit_issue()`, `add_vote()`
- `pages.py`: `do_vote_endpoint()`
- `vtypes` module: type validation infrastructure

**Description:**

The `add_vote()` method accepts arbitrary votestring values without validating them against the issue's vote type (YNA, STV, etc.). A TODO comment at line 249 acknowledges this gap: `### validate VOTESTRING for ISSUE.TYPE voting`. The votestring is encrypted and stored without format checking, allowing invalid votes to be cryptographically sealed in the database.

**Vulnerable Code (`election.py`):**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting    # <-- TODO: control never implemented

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)

    # Pass MAYVOTE.SALT for PBKDF.
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)

    self.c_add_vote.perform(vote_token, ciphertext)
```

**Vulnerable Code (`pages.py`, caller):**

```python
for iid, votestring in votes.items():
    if iid not in issue_dict:
        await flash_danger(f'Invalid issue ID: {iid}')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    try:
        election.add_vote(result.uid, iid, votestring)
```

**Data Flow:**

```
User HTTP POST form field (vote-{iid}=<arbitrary_value>)
  → do_vote_endpoint() extracts votestring from form
    → IID validated against issue_dict ✅
    → votestring NOT validated against issue vote type ❌
      → election.add_vote() encrypts and stores raw votestring
        → c_add_vote.perform(vote_token, ciphertext)
```

**Evidence the Control Was Intended:**

The `vtypes` module is imported and used for type validation during `add_issue` (`assert vtype in vtypes.TYPES`) and during tallying (`vtypes.vtype_module(issue.type).tally(votes, ...)`), but the validation infrastructure is never invoked during vote submission.

**Proof of Concept:**

```http
POST /do-vote/VALID_EID HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: [valid_session]

vote-VALID_IID_YNA=INVALID_NOT_Y_N_A&vote-VALID_IID_STV=nonexistent_label1,nonexistent_label2,,,
```

Both values would be accepted, encrypted, and stored without any format check.

**Impact:**

- **YNA issues:** Arbitrary strings (not `y`/`n`/`a`) are accepted and stored; tallying behavior with unexpected values is undefined
- **STV issues:** Non-existent candidate labels, duplicate labels, or malformed comma-separated strings are accepted; could produce incorrect election results or tallying exceptions
- **No size limit:** Arbitrarily large votestrings are encrypted and stored, enabling potential storage abuse
- Election integrity is compromised because invalid votes are cryptographically sealed, making post-hoc detection difficult

**Remediation:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    md = self._all_metadata(self.S_OPEN)

    # Validate votestring format against issue type
    issue = self.q_get_issue.first_row(iid)
    if not issue:
        raise IssueNotFound(iid)
    
    vtype_mod = vtypes.vtype_module(issue.type)
    vtype_mod.validate_vote(votestring, self.json2kv(issue.kv))
    # validate_vote should raise ValueError for:
    #   YNA: value not in {'y', 'n', 'a'}
    #   STV: labels not in kv['labelmap'], duplicates, empty string

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

Each vtype module should implement `validate_vote(votestring, kv)`:

```python
# vtypes/yna.py
VALID_VOTES = {'y', 'n', 'a'}

def validate_vote(votestring, kv=None):
    if votestring not in VALID_VOTES:
        raise ValueError(f'Invalid YNA vote: must be one of {VALID_VOTES}')

# vtypes/stv.py
def validate_vote(votestring, kv):
    valid_labels = set(kv.get('labelmap', {}).keys())
    labels = [l.strip() for l in votestring.split(',') if l.strip()]
    if len(labels) != len(set(labels)):
        raise ValueError('Duplicate candidates in STV ranking')
    for label in labels:
        if label not in valid_labels:
            raise ValueError(f'Invalid candidate label: {label}')
```

## Positive Security Patterns

1. **Complete SQL Parameter Binding:** Every query in `queries.yaml` uses `?` placeholders. The `asfpy.db` wrapper enforces binding through its cursor API (`perform()`, `first_row()`). No string concatenation or f-string interpolation constructs SQL. This is an exemplary pattern.

2. **LDAP Security:** The LDAP search in `asf-load-ldap.py` uses a hardcoded filter (`'uid=*'`), a hardcoded base DN, and secure transport (`ldaps://`). No user-controlled data enters any LDAP query.

3. **ID Generation via `crypto.create_id()`:** Election IDs and Issue IDs are system-generated, not user-provided. The collision-retry loop with `sqlite3.IntegrityError` ensures uniqueness without exposing auto-increment patterns.

4. **EZT Context-Specific Encoding:** Templates use `[format "js,html"]` when embedding dynamic values in JavaScript contexts (e.g., `openEditIssueModal` calls in `manage.ezt` and `manage-stv.ezt`).

5. **Database-Validated Route Parameters:** The `load_election` and `load_election_issue` decorators validate EID/IID parameters against the database before processing, preventing operations on non-existent resources.

6. **Sensitive Column Protection:** Both `get_metadata()` and `get_person()` explicitly exclude SALT and OPENED_KEY from returned data. The `q_open_to_me` query comment documents this: "SALT and OPENED_KEY are never returned for security."

7. **State Machine Enforcement:** Election operations use assertions (`assert self.is_editable()`, `assert self.is_open()`) to enforce the editable → open → closed state transitions, preventing out-of-order operations.

8. **No Subprocess Calls:** Neither CLI scripts nor web handlers use `subprocess`, `os.system()`, or `os.popen()`, eliminating command injection risk.

## Architecture Observations

1. **JNDI Not Applicable:** This is a Python application; JNDI is a Java technology. The equivalent directory service concern (LDAP) is well-handled.

2. **LDAP is Read-Only Batch:** LDAP usage is confined to a single CLI script (`asf-load-ldap.py`) that performs a one-time bulk import. No LDAP operations occur during web request handling, further reducing the attack surface.

3. **Unused LDAP Constant:** `LDAP_ATTR = 'memberUid'` is defined but never referenced. The search uses `attrlist=None` instead of restricting to needed attributes (`uid`, `cn`, `asf-committer-email`). While not a vulnerability, restricting the attribute list would follow the principle of least privilege.

4. **Vote Validation Gap Is Pre-Encryption:** Because votestrings are encrypted before storage, any malformed data becomes cryptographically sealed and cannot be identified or corrected without decryption. This makes the missing validation more impactful than typical input validation gaps.
