## Findings

### High Severity Findings

#### Finding ASVS-311-HIGH-001

**Title:** No Content-Security-Policy Header Configured on Any Response  

**Location:**  
- File: `v3/server/main.py`
- Function: `create_app()`
- Lines: 32–42
- Additional Location: `v3/server/pages.py` — all route handlers

**Related Functions Checked:**
- `create_app()` in `main.py`
- `rewrite_description()` in `pages.py` (lines 53-61)
- All 18+ HTML-returning route handlers in `pages.py`

**Description:**  
The application serves HTML pages through 18+ route handlers but never sets a `Content-Security-Policy` header on responses. There is no `@APP.after_request` hook, no middleware registration, and no per-response header injection anywhere in the codebase. This is a **Type A gap** — the control does not exist at all.

This is especially critical because `rewrite_description()` in `pages.py` (line 53-61) programmatically injects raw HTML into issue descriptions.

**Vulnerable Code:**
```python
def rewrite_description(issue):
    desc = issue.description
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

**Data Flow:**  
User-provided description → `rewrite_description()` → HTML injection into template → browser renders without CSP restrictions.

**Entry Points Affected:**  
All 18+ HTML-returning routes including `/`, `/voter`, `/vote-on/<eid>`, `/manage/<eid>`, `/admin`, `/about`, `/profile`, `/settings`, `/manage-stv/<eid>/<iid>`.

**Proof of Concept:**  
Without CSP, any reflected or stored XSS vulnerability will execute arbitrary JavaScript in users' browsers without restriction.

**Impact:**  
Without CSP, any reflected or stored XSS vulnerability will execute arbitrary JavaScript in users' browsers. In an election system, this could allow vote manipulation or session hijacking.

**Remediation:**
```python
# In main.py create_app(), add after_request handler:
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    @app.after_request
    async def set_security_headers(response):
        response.headers['Content-Security-Policy'] = (
            "default-src 'self'; "
            "script-src 'self'; "
            "style-src 'self'; "
            "img-src 'self'; "
            "frame-ancestors 'none'; "
            "form-action 'self'; "
            "base-uri 'self'"
        )
        response.headers['X-Content-Type-Options'] = 'nosniff'
        response.headers['X-Frame-Options'] = 'DENY'
        response.headers['Referrer-Policy'] = 'strict-origin-when-cross-origin'
        response.headers['Permissions-Policy'] = (
            'camera=(), microphone=(), geolocation=()'
        )
        return response

    import pages
    import api
    return app
```

#### Finding ASVS-311-HIGH-002

**Title:** No HTTP Strict Transport Security (HSTS) Header Despite TLS Support  

**Location:**  
- File: `v3/server/main.py`
- Function: `run_standalone()`
- Lines: 79–85

**Related Functions Checked:**
- `run_standalone()` in `main.py` (lines 79-85)
- `run_asgi()` in `main.py` (line 96)
- `create_app()` in `main.py` (lines 32-42)

**Description:**  
The application supports TLS configuration but never sets the `Strict-Transport-Security` header. This is a **Type A gap** — TLS is available but HSTS enforcement does not exist. Even when TLS is configured:
1. No HSTS header is sent to instruct browsers to always use HTTPS.
2. No HTTP→HTTPS redirect is configured.
3. No mechanism ensures the application behaves correctly (warns or blocks) when accessed over plain HTTP.
4. In ASGI mode (`run_asgi()`, line 96), TLS is delegated entirely to the reverse proxy with no application-level verification.

**Vulnerable Code:**
```python
kwargs = {}
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
    extra_files.update((kwargs['certfile'], kwargs['keyfile']))

app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Data Flow:**  
User connects over HTTP → No redirect to HTTPS → Authentication cookies and session data transmitted in plaintext → Network-level attacker intercepts credentials.

**Entry Points Affected:**  
All application endpoints when accessed over HTTP.

**Proof of Concept:**  
Users connecting over HTTP (e.g., first visit, downgrade attack, misconfigured proxy) will transmit authentication cookies and session data in plaintext.

**Impact:**  
Users connecting over HTTP (e.g., first visit, downgrade attack, misconfigured proxy) transmit authentication cookies and session data in plaintext. Election data and voter identity are exposed to network-level attackers.

**Remediation:**
```python
@app.after_request
async def set_hsts(response):
    # Only set HSTS when served over HTTPS
    if quart.request.is_secure or quart.request.headers.get('X-Forwarded-Proto') == 'https':
        response.headers['Strict-Transport-Security'] = (
            'max-age=31536000; includeSubDomains'
        )
    return response

# Optionally redirect HTTP to HTTPS
@app.before_request
async def enforce_https():
    if not quart.request.is_secure and not app.debug:
        url = quart.request.url.replace('http://', 'https://', 1)
        return quart.redirect(url, code=301)
```

#### Finding ASVS-311-MEDIUM-003

**Title:** No Browser Security Feature Documentation or Degradation Behavior  

**Location:**  
- File: `v3/server/main.py` — entire file
- File: `v3/server/pages.py` — entire file
- Function: `create_app()`
- Lines: 32-42

**Related Functions Checked:**
- `create_app()` in `main.py`
- All application initialization code
- Configuration loading mechanisms

**Description:**  
ASVS 3.1.1 explicitly requires that application documentation states:
1. Expected security features browsers must support (HTTPS, HSTS, CSP, etc.).
2. How the application behaves when features are unavailable (warning, blocking, graceful degradation).

Neither the application code nor any referenced configuration contains such documentation. Specifically:
- No `SECURITY.md`, security section in README, or inline documentation of browser requirements.
- No runtime checks for browser security feature support.
- No warning mechanism for users on non-conforming browsers.
- No `@app.before_request` handler that validates request security properties.

**Vulnerable Code:**
```python
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    import pages
    import api
    return app
```

**Data Flow:**  
Application deployment → No security requirements documentation → Operations teams cannot verify security configuration → Users not warned about missing security features.

**Entry Points Affected:**  
All application endpoints (no documentation means no enforcement or guidance).

**Proof of Concept:**  
Deployment teams have no guidance on required proxy/CDN security configurations. Operations teams cannot verify that the application is served with appropriate security headers.

**Impact:**  
Without documented browser security requirements, deployment teams cannot verify that the application is served with appropriate security headers. Operations teams have no guidance on required proxy/CDN security configurations. Users are not warned when their browser lacks required security features.

**Remediation:**

1. Create `SECURITY.md` documenting required browser security features:

```markdown
# Browser Security Requirements

## Required Browser Features
- **HTTPS**: All connections MUST use TLS 1.2+
- **HSTS**: Browsers must honor Strict-Transport-Security
- **CSP**: Content-Security-Policy Level 2 support required
- **SameSite Cookies**: Browsers must support SameSite=Lax/Strict

## Degradation Behavior
- HTTP connections: Redirected to HTTPS (301)
- Missing CSP support: Application functions but logs warning
- JavaScript disabled: Critical voting forms require JS; warning displayed
- Unsupported browsers: Banner displayed recommending upgrade

## Deployment Requirements
- Reverse proxy MUST set HSTS with max-age >= 31536000
- CSP header MUST be set (see security_headers.py for values)
- X-Frame-Options: DENY must be set
```

2. Add runtime enforcement in `create_app()`:

```python
REQUIRED_SECURITY_FEATURES = {
    'Content-Security-Policy': "default-src 'self'; script-src 'self'; ...",
    'X-Frame-Options': 'DENY',
    'X-Content-Type-Options': 'nosniff',
    'Strict-Transport-Security': 'max-age=31536000; includeSubDomains',
    'Referrer-Policy': 'strict-origin-when-cross-origin',
    'Permissions-Policy': 'camera=(), microphone=(), geolocation=()',
}

@app.after_request
async def apply_documented_security_headers(response):
    for header, value in REQUIRED_SECURITY_FEATURES.items():
        response.headers[header] = value
    return response
```

#### Finding ASVS-311-MEDIUM-004

**Title:** No X-Frame-Options or frame-ancestors CSP Directive — Clickjacking Unmitigated  

**Location:**  
- File: `v3/server/pages.py`
- Function: All HTML-returning route handlers
- Lines: 203, 315, 448, 468 (critical state-changing endpoints)

**Related Functions Checked:**
- `/vote-on/<eid>` route handler (line 203)
- `/manage/<eid>` route handler (line 315)
- `/do-open/<eid>` route handler (line 448)
- `/do-close/<eid>` route handler (line 468)
- All 18+ HTML-rendering endpoints

**Description:**  
No route handler or application-level middleware sets `X-Frame-Options` or a `Content-Security-Policy` `frame-ancestors` directive. This is a **Type A gap**. All 18+ HTML-rendering endpoints can be embedded in attacker-controlled iframes.

**Entry Points Affected:**  
All HTML endpoints, but most critical are state-changing pages that could be clickjacked:
- `/vote-on/<eid>` — voting form (line 203)
- `/manage/<eid>` — election management (line 315)
- `/do-open/<eid>` — election opening (line 448, GET request — doubly vulnerable)
- `/do-close/<eid>` — election closing (line 468, GET request)

**Vulnerable Code:**
```python
# No X-Frame-Options or CSP frame-ancestors set on any response
# Example vulnerable endpoint:
@APP.get('/do-open/<eid>')
async def do_open(eid: str):
    # State-changing GET request with no clickjacking protection
    ...
```

**Data Flow:**  
Attacker creates malicious page with iframe → Embeds vulnerable endpoint → Overlays deceptive UI → Authenticated user clicks → State-changing action executes.

**Proof of Concept:**
```html
<!-- Attacker page -->
<iframe src="https://steve.apache.org/do-open/TARGET_EID" 
        style="opacity:0; position:absolute; top:0; left:0; width:100%; height:100%;">
</iframe>
<button style="position:relative; z-index:-1;">Click here for a prize!</button>
```

Since `/do-open/<eid>` and `/do-close/<eid>` are **GET requests that perform state changes**, a simple iframe load (without even requiring a click on a form button) could open or close an election.

**Impact:**  
An attacker can trick an authenticated election administrator into opening/closing elections or submitting votes by framing the application page and overlaying deceptive UI elements.

**Remediation:**
```python
@app.after_request
async def prevent_clickjacking(response):
    response.headers['X-Frame-Options'] = 'DENY'
    # Also set via CSP for modern browsers:
    csp = response.headers.get('Content-Security-Policy', '')
    if 'frame-ancestors' not in csp:
        response.headers['Content-Security-Policy'] = csp + "; frame-ancestors 'none'"
    return response
```

Additionally, convert state-changing GET endpoints to POST:
```python
# Change from GET to POST
@APP.post('/do-open/<eid>')  # was @APP.get
@APP.post('/do-close/<eid>')  # was @APP.get
```

## Architecture Observations

1. **Framework Opacity:** The `asfquart` framework is not provided for review. It's possible that `asfquart.construct()` configures some security headers, but there is no evidence of this in the application code, and the application does not configure, override, or verify any security header behavior.

2. **State-Changing GET Requests:** `/do-open/<eid>` and `/do-close/<eid>` use GET methods for state-changing operations. Combined with missing clickjacking protection and CSRF (acknowledged tech debt), this creates compounded risk.

3. **No Security Header Constants:** There are no constants, configuration entries, or references to security header values anywhere in the codebase, suggesting security headers are not a design consideration rather than simply missing from these files.

4. **Template Engine Escaping:** The EZT template engine's auto-escaping behavior is not visible in this code. Combined with `rewrite_description()` injecting raw HTML, CSP is an essential defense-in-depth layer.
