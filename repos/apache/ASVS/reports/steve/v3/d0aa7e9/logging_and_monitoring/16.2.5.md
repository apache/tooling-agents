## Findings

### ASVS-1625-HIGH-001

**Severity: HIGH**

**Title:** Vote submission error handler logs unfiltered exception messages from cryptographic operations handling ballot content

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint`
- **Lines:** 399–403

**Related Functions Checked:**
- `election.add_vote()` — handles vote content and cryptographic operations
- `crypto.gen_vote_token()` — processes opened_key
- `crypto.create_vote()` — processes raw vote content
- `self.c_add_vote.perform()` — processes vote_token

**Vulnerable Code:**
```python
except Exception as e:
    _LOGGER.error(
        f'Error adding vote for user[U:{result.uid}] on issue[I:{iid}]: {e}'
    )
```

**Data Flow:**
1. **Source:** User submits vote via POST `/do-vote/<eid>` → `votestring` extracted from form
2. **Processing:** `election.add_vote(result.uid, iid, votestring)` passes votestring through crypto operations:
   - `crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)` — handles `opened_key`
   - `crypto.create_vote(vote_token, mayvote.salt, votestring)` — handles raw vote content
   - `self.c_add_vote.perform(vote_token, ciphertext)` — handles `vote_token`
3. **Sink:** Any exception from these operations is caught by `except Exception as e` and logged as `_LOGGER.error(f'...{e}')` with **no sanitization**

**Gap Classification:** Type A — No sanitization control exists for exception messages before logging in sensitive operations.

**Description:**

The `add_vote()` method simultaneously handles the system's most sensitive data: the `opened_key` (election master secret), `vote_token` (links voter to vote), `salt`, and `votestring` (actual vote content — the core of ballot secrecy). A broad `except Exception as e` catches all exception types including:
- `sqlite3.IntegrityError` which may include bound parameter values (`vote_token`)
- Cryptographic library exceptions that may reference input data
- `AttributeError` if `mayvote` is None (would expose context)

**Proof of Concept:**

If `crypto.create_vote()` receives malformed input and raises:
```python
# Hypothetical crypto error including parameter info
ValueError("Cannot encrypt value: 'yes' with token b'abc123...'")
```
The log would contain:
```
[ERROR|pages] Error adding vote for user[U:jsmith] on issue[I:issue-42]: Cannot encrypt value: 'yes' with token b'abc123...'
```
This links voter `jsmith` to their vote `yes` on `issue-42`, destroying ballot secrecy. Even if current crypto implementations don't include parameter values, there is **no control** preventing future code changes from causing this exposure.

**Impact:** 

Ballot secrecy violation — vote content and/or cryptographic material (vote_token, opened_key) could be persisted in log files, creating a permanent record linking voters to specific votes. Log files may be accessible to system administrators, backup systems, and log aggregation tools, greatly expanding the exposure surface.

**Remediation:**

```python
except Exception:
    _LOGGER.error(
        f'Error adding vote for user[U:{result.uid}] on issue[I:{iid}]'
        f' in election[E:{election.eid}]: vote processing failed'
    )
    # Log the exception type only, not the message or traceback
    _LOGGER.debug(
        f'Exception type: {type(e).__name__} (details suppressed for ballot secrecy)'
    )
    await flash_danger(f'Error submitting vote for issue {iid}.')
    return quart.redirect(f'/vote-on/{election.eid}', code=303)
```

Or implement a centralized sanitization utility:
```python
def safe_log_exception(e: Exception) -> str:
    """Return exception type name only, suppressing message content."""
    return type(e).__name__

_LOGGER.error(
    f'Error adding vote for user[U:{result.uid}] on issue[I:{iid}]: '
    f'{safe_log_exception(e)}'
)
```

### ASVS-1625-MEDIUM-001

**Severity: MEDIUM**

**Title:** Debug `print()` statements output raw form data to stdout in production request handlers, bypassing logging framework and data classification

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_add_issue_endpoint`, line 482
- **Function:** `do_edit_issue_endpoint`, line 499

**Related Functions Checked:**
- `do_vote_endpoint` — correctly does NOT use `print('FORM:', form)`

**Vulnerable Code:**
```python
# In do_add_issue_endpoint (~line 482):
form = edict(await quart.request.form)
print('FORM:', form)

# In do_edit_issue_endpoint (~line 499):
form = edict(await quart.request.form)
print('FORM:', form)
```

**Data Flow:**
1. **Source:** Admin submits form data via POST `/do-add-issue/<eid>` or `/do-edit-issue/<eid>/<iid>`
2. **Sink:** `print()` writes complete form contents to stdout → captured in process logs (systemd journal, Docker logs, Hypercorn output, etc.)
3. **Missing Control:** No data classification, no masking, bypasses logging framework entirely

**Gap Classification:** Type A — No data classification control applied. `print()` bypasses the logging framework, meaning log level, format, and output destination controls do not apply.

**Description:**

Notably, the `do_vote_endpoint` handler does NOT use `print('FORM:', form)`, correctly avoiding exposure of vote content. This inconsistency demonstrates that the pattern was likely introduced during development and not cleaned up, but also shows no systematic control prevents it.

`print()` bypasses the `logging` framework — output cannot be filtered by log level, formatted consistently, or routed to appropriate destinations. In ASGI mode (Hypercorn), stdout is captured as process output and often persisted in container/system logs.

While current form fields (`title`, `description`) are not highly sensitive, this pattern:
- Creates a precedent for logging unsanitized user input
- Could capture sensitive data if fields are added later
- Violates the ASVS 16.2.5 principle of enforcing logging based on data protection level

**Impact:** 

Uncontrolled data exposure through stdout. Form data persisted in system logs without classification or access controls applied to the logging framework.

**Remediation:**

```python
# Remove print() statements entirely, or replace with appropriate logging:
_LOGGER.debug(f'Issue form submitted for election[E:{election.eid}]')
# Do NOT log form contents; log only metadata about the action
```

### ASVS-1625-MEDIUM-002

**Severity: MEDIUM**

**Title:** Tally error handler prints unfiltered exception messages during vote decryption operations

**Location:**
- **File:** `v3/server/bin/tally.py`
- **Function:** `tally_election`
- **Lines:** 115–118

**Related Functions Checked:**
- `election.tally_issue()` — decrypts votes
- `crypto.gen_vote_token()` — processes opened_key
- `crypto.decrypt_votestring()` — processes vote_token, salt, and ciphertext

**Vulnerable Code:**
```python
except Exception as e:
    print(f'Error tallying issue {issue.iid}: {e}')
    raise  # Fail hard
```

**Data Flow:**
1. **Source:** `election.tally_issue(issue.iid)` decrypts votes:
   - Retrieves `opened_key` from election metadata
   - Computes `vote_token` for each voter using `crypto.gen_vote_token(opened_key, pid, iid, salt)`
   - Decrypts votes using `crypto.decrypt_votestring(vote_token, salt, ciphertext)`
2. **Sink:** Any exception during decryption is printed with `{e}` — could include cryptographic material
3. **Re-raise:** `raise` causes Python to print the full traceback, showing source lines in crypto functions handling sensitive data

**Gap Classification:** Type A — No sanitization exists for exception messages from vote decryption.

**Description:**

During tallying, the following sensitive values are in-scope and could appear in exception messages:
- `opened_key` — the election master secret
- `vote_token` — links voter to encrypted vote
- `salt` — per-voter cryptographic salt
- Decrypted `votestring` — actual vote content

The `raise` after `print()` causes Python's default exception handler to output the full traceback. While the traceback shows source code lines (not variable values by default), custom exception handlers or logging configurations could capture locals.

**Impact:** 

Cryptographic secrets or partial vote data could be exposed via stdout captured to log files. Even in a CLI context, stdout may be piped, redirected, or captured by process supervision systems.

**Remediation:**

```python
except Exception as e:
    # Log only the exception type, not the message
    print(f'Error tallying issue {issue.iid}: {type(e).__name__}')
    _LOGGER.error(
        f'Tally failed for issue[I:{issue.iid}]: {type(e).__name__}'
        ' (details suppressed to protect vote data)'
    )
    raise  # Re-raise preserves traceback for authorized admin debugging
```

## Positive Security Patterns

1. **Vote submission logging correctly excludes vote content** (`pages.py`, line ~396):
   ```python
   _LOGGER.info(
       f'User[U:{result.uid}] voted on issue[I:{iid}] in election[E:{election.eid}]'
   )
   ```
   Logs voter ID and issue ID but never the `votestring`. This correctly maintains ballot secrecy in the normal (non-error) path.

2. **`get_metadata()` explicitly filters sensitive fields** (`election.py`, line ~149):
   ```python
   # NOTE: do not return the SALT or OPENED_KEY
   return edict(eid=md.eid, title=md.title, ...)
   ```
   Consciously prevents cryptographic secrets from leaking to callers.

3. **`get_issue()` explicitly excludes salt** (`election.py`, line ~165):
   ```python
   # NEVER return issue.salt
   return (issue.title, issue.description, issue.type, self.json2kv(issue.kv))
   ```
   Comment documents the security intent clearly.

4. **Commented-out debug prints for crypto material** (`election.py`, lines ~80, ~336):
   ```python
   #print('SALT:', salt)
   #print('KEY:', opened_key)
   ```
   These were correctly disabled. Evidence of developer awareness, though they should be removed entirely.

5. **Structured log format with metadata** (`main.py`, line ~53):
   ```python
   format='[{asctime}|{levelname}|{name}] {message}'
   ```
   Consistent structured format with timestamps, level, and source.

6. **Election lifecycle logging** — Opening (`line ~445`), closing (`line ~459`), creation (`line ~425`), and issue management all log admin identity and action without sensitive election configuration data.

7. **`do_vote_endpoint` does NOT have `print('FORM:', form)`** — Unlike the issue management endpoints, the vote handler correctly avoids printing the form data that contains vote content. This shows awareness even if inconsistently applied.

## Architecture Observations

1. **No Centralized Log Sanitization Layer:** The application lacks a utility to sanitize exception messages before logging. Each handler independently decides what to log. This means every new handler or error path requires developers to manually consider data classification — a fragile approach that has already failed in the identified findings.

2. **`print()` vs `logging` Inconsistency:** The codebase mixes `print()` (tally.py, debug statements in pages.py) with `_LOGGER` (structured logging). `print()` bypasses all logging controls (level, format, destination, filtering). This creates two uncontrolled output streams.

3. **Error Path vs. Normal Path Asymmetry:** The normal logging paths are well-designed (vote content never logged, crypto secrets excluded from API responses). However, error paths lack equivalent protection. This is a common pattern where developers focus on the happy path for security controls.

4. **Commented-Out Debug Prints as Latent Risk:** Several `#print('SALT:',` and `#print('KEY:',` lines exist in `election.py`. While currently inactive, they represent a one-character change (`#` removal) from exposing critical secrets. These should be removed entirely rather than commented out.

## Recommendations Summary

### Immediate Actions (Critical/High Priority)

1. **Sanitize exception messages in `do_vote_endpoint` error handler** — Log only exception type, not `{e}` (Effort: Low, Impact: Prevents ballot secrecy violation)
2. **Remove `print('FORM:', form)` debug statements** from `do_add_issue_endpoint` and `do_edit_issue_endpoint` (Effort: Trivial, Impact: Eliminates uncontrolled data output)

### Short-term Actions (Medium Priority)

3. **Sanitize exception messages in `tally.py` `tally_election` error handler** (Effort: Low, Impact: Prevents crypto material exposure)
4. **Create a `safe_log_exception()` utility** that returns only exception type name (Effort: Low, Impact: Provides reusable control)
5. **Remove all commented-out `print('SALT:', ...)` / `print('KEY:', ...)` lines** from `election.py` (Effort: Trivial, Impact: Eliminates latent risk)

### Long-term Actions (Strategic Improvements)

6. **Establish a logging policy:** Replace all `print()` with `_LOGGER` calls at appropriate levels (Effort: Medium, Impact: Unified log control)
7. **Add a log sanitization middleware** that scrubs patterns matching cryptographic material (hex strings, base64) (Effort: Medium, Impact: Defense in depth)
