## Findings

### 🔴 ASVS-1634-CRT-001: Election Tampering Detection Bypasses Logging System

**Severity:** CRITICAL  
**Gap Type:** B (Control EXISTS but logging NOT CALLED)  
**CWE:** CWE-778 (Insufficient Logging)

**Location:**
- **File:** `v3/server/bin/tally.py`
- **Function:** `main()`
- **Lines:** 158-160

**Related Functions Checked:**
- `steve.election.Election.is_tampered()` (`v3/steve/election.py` lines 379-396)
- `steve.crypto.gen_opened_key()` (referenced but not provided)

**Description:**

The election tampering detection mechanism—the most critical security control in an election system—outputs its findings exclusively to stdout via `print()` rather than the application's structured logging system. When `election.is_tampered()` detects that election data has been modified after opening (by comparing computed and stored `opened_key` values), the event is printed to the console but never recorded in security logs.

**Vulnerable Code:**

```python
# v3/server/bin/tally.py, lines 158-160
# Check for tampering
pdb = steve.persondb.PersonDB.open(db_fname)
if election.is_tampered(pdb):
    print(f'Error: Election {election_id} has been tampered with. Cannot proceed.')
    sys.exit(1)
```

```python
# v3/steve/election.py, lines 379-396
def is_tampered(self, pdb):
    """Check whether the election has been tampered with"""
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)
    tampered = opened_key != md.opened_key
    return tampered  # No logging of tampering detection
```

**Data Flow:**

1. Tally script calls `election.is_tampered(pdb)`
2. `is_tampered()` computes integrity hash (`opened_key`) from current election data
3. Compares computed hash with stored hash from election opening
4. Returns `True` if mismatch detected (tampering)
5. `main()` receives `True` → calls `print()` → outputs to stdout only
6. `_LOGGER` (defined at line 35) is never invoked
7. Structured security logs contain no record of tampering detection

**Proof of Concept:**

**Scenario:** An attacker with database access modifies election data after opening:

```python
# Attacker modifies voter list after election opens
db = sqlite3.connect('steve.db')
db.execute("INSERT INTO person_election VALUES ('attacker@example.org', 'election123')")
db.commit()

# Later, when tally runs:
# $ python tally.py election123
# Error: Election election123 has been tampered with. Cannot proceed.
# [Script exits]

# Result: 
# - Message appears in stdout (if captured)
# - NO entry in application logs
# - NO SIEM alert triggered
# - NO audit trail for forensic investigation
```

**Impact:**

1. **Audit Trail Failure:** The most severe security event in an election system leaves no trace in structured logs, violating ASVS 16.3.4's core requirement
2. **Detection Evasion:** If an attacker tampers with an election and then restores the data before a second tally attempt, there is no persistent record of the tampering
3. **SIEM Blindness:** Security monitoring systems that ingest structured logs will never see tampering events
4. **Incident Response Failure:** Post-incident investigations have no forensic evidence of when tampering was detected or by whom
5. **Compliance Violation:** Election systems typically require comprehensive audit logs for legal and regulatory purposes

**Remediation:**

```python
# v3/server/bin/tally.py, lines 158-165 (modified)
# Check for tampering
pdb = steve.persondb.PersonDB.open(db_fname)
if election.is_tampered(pdb):
    _LOGGER.critical(
        f'TAMPERING DETECTED: Election[E:{election_id}] data integrity check failed. '
        f'Computed opened_key does not match stored value. Tally aborted.'
    )
    print(f'Error: Election {election_id} has been tampered with. Cannot proceed.')
    sys.exit(1)
```

```python
# v3/steve/election.py, lines 379-403 (modified)
def is_tampered(self, pdb):
    """Check whether the election has been tampered with"""
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)
    tampered = opened_key != md.opened_key
    
    if tampered:
        _LOGGER.critical(
            f'Integrity check FAILED for election[E:{self.eid}]: '
            f'opened_key mismatch detected. Expected: {md.opened_key[:16]}..., '
            f'Computed: {opened_key[:16]}...'
        )
    else:
        _LOGGER.debug(f'Integrity check PASSED for election[E:{self.eid}]')
    
    return tampered
```

**Verification:**

After remediation, confirm:
1. Tampering detection events appear in structured logs with `CRITICAL` level
2. Log entries include election ID and timestamp
3. SIEM systems can parse and alert on tampering events
4. Both the `is_tampered()` method and calling code log the event (defense in depth)

### 🟠 ASVS-1634-HGH-002: State-Changing Endpoints Missing Error and Security Failure Logging

**Severity:** HIGH  
**Gap Type:** B (Control EXISTS in `do_vote_endpoint` but NOT CALLED in other endpoints)  
**CWE:** CWE-778 (Insufficient Logging)

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** 8 state-changing endpoints
- **Lines:** 461-481, 484-502, 505-521, 524-541, 544-566, 569-588, 591-608, 611-623

**Related Functions Checked:**
- `steve.election.Election.create()` (line 465)
- `steve.election.Election.open()` (line 494)
- `steve.election.Election.close()` (line 513)
- `steve.election.Election.add_issue()` (line 537)
- `steve.election.Election.edit_issue()` (line 561)
- `steve.election.Election.delete_issue()` (line 583)
- State validation via `assert` statements in `v3/steve/election.py`

**Description:**

Eight of nine state-changing endpoints in the web application do not implement error logging, despite the presence of a correct reference implementation in `do_vote_endpoint`. When operations fail due to state validation errors (e.g., attempting to open an already-open election), database errors, or cryptographic failures, the exceptions propagate to the framework without being logged with user context. Only `do_vote_endpoint` correctly implements the try/except/log pattern.

1. User (or attacker) sends `GET /do-open/<already-open-eid>`
2. `@load_election` decorator loads election object
3. `do_open_endpoint()` calls `election.open(pdb)`
4. `election.open()` executes `assert self.is_editable()` (line 73 of `election.py`)
5. Assertion fails because election state is 'open', not 'editable'
6. `AssertionError` raised and propagates to Quart framework
7. Framework returns 500 error to user
8. **NO `_LOGGER.error()` called** — security control failure is unlogged
9. User identity, election ID, and attempted action are lost

**Impact:**

1. **Attack Detection Failure:** Repeated attempts to manipulate election state (common in reconnaissance or privilege escalation attacks) leave no audit trail

5. **Compliance Risk:** Election systems typically require logging of all administrative actions, both successful and failed

**Remediation:**

**Apply the `do_vote_endpoint` pattern to all state-changing endpoints:**

```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    try:
        ### check authz
        pdb = steve.persondb.PersonDB.open(DB_FNAME)
        election.open(pdb)
        _LOGGER.info(f'User[U:{result.uid}] opened election[E:{election.eid}]')
    except Exception as e:
        _LOGGER.error(
            f'Failed to open election[E:{election.eid}] by User[U:{result.uid}]: '
            f'{type(e).__name__}: {e}'
        )
        await flash_danger('Failed to open election.')
        return quart.redirect(f'/manage/{election.eid}', code=303)
    
    title = election.get_metadata().title
    await flash_success(f'Opened election: {title}')
    return quart.redirect(f'/manage/{election.eid}', code=303)
```

**Alternative: Create a reusable decorator:**

```python
def log_errors(action_description):
    """Decorator to log errors in state-changing operations"""
    def decorator(func):
        @functools.wraps(func)
        async def wrapper(*args, **kwargs):
            try:
                return await func(*args, **kwargs)
            except Exception as e:
                result = await basic_info()
                _LOGGER.error(
                    f'{action_description} failed for User[U:{result.uid}]: '
                    f'{type(e).__name__}: {e}',
                    exc_info=True
                )
                raise
        return wrapper
    return decorator

# Usage:
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
@log_errors('Election open')
async def do_open_endpoint(election):
    # ... implementation
```

3. Verify log entries include exception type and message
4. Test that user-facing error messages remain generic (no sensitive data leakage)

### 🟠 ASVS-1634-HGH-003: Tally Operation Errors Bypass Logging System

**Severity:** HIGH  
**Gap Type:** B (Logger EXISTS but `print()` used instead)  
**CWE:** CWE-778 (Insufficient Logging)

**Location:**
- **File:** `v3/server/bin/tally.py`
- **Function:** `tally_election()`
- **Lines:** 123-126

**Related Functions Checked:**
- `steve.election.Election.tally_issue()` (referenced, implementation not provided)
- `steve.crypto` decryption functions (referenced, implementation not provided)

**Description:**

The `tally_election()` function handles errors during the critical vote tallying process by printing to stdout rather than using the configured `_LOGGER` instance. When `election.tally_issue()` raises exceptions—potentially due to decryption failures, database errors, or corrupted vote data—these security-relevant errors are output to stdout only and do not appear in structured application logs.

**Vulnerable Code:**

```python
# v3/server/bin/tally.py, lines 123-126
for issue in issues:
    try:
        tally_result, issue_voters = election.tally_issue(issue.iid)
        # ... process results
    except Exception as e:
        print(f'Error tallying issue {issue.iid}: {e}')  # ❌ Uses print() not _LOGGER
        raise  # Fail hard
```

**Context - Inconsistent Logging Pattern:**

The same file correctly uses `_LOGGER` for other error conditions:

```python
# Line 112 - CORRECT usage
if not issues:
    _LOGGER.error(f'No issues found for election[E:{election.eid}]')
    return None

# Line 115-116 - CORRECT usage
if not issue:
    _LOGGER.error(f'Issue[I:{iid}] not found in election[E:{election.eid}]')
    return None

# Line 117 - CORRECT usage for success
_LOGGER.info(f'Tallying issue[I:{issue.iid}] in election[E:{election.eid}]')

# Line 125 - INCORRECT usage ❌
print(f'Error tallying issue {issue.iid}: {e}')
```

**Data Flow:**

1. Tally script iterates through election issues
2. Calls `election.tally_issue(issue.iid)` to decrypt and count votes
3. Decryption failure occurs (corrupted ciphertext, wrong key, etc.)
4. `tally_issue()` raises exception (e.g., `CryptoError`, `ValueError`)
5. Exception caught in except block
6. `print()` outputs error to stdout
7. `_LOGGER` (defined at line 35) is never invoked
8. Exception is re-raised, script terminates
9. **Structured logs contain no record of the tally failure**

**Proof of Concept:**

**Scenario:** Corrupted vote ciphertext causes decryption failure:

```python
# Simulate corrupted vote data in database
import sqlite3
db = sqlite3.connect('steve.db')
db.execute("""
    UPDATE votes 
    SET encrypted_vote = 'corrupted_data_xxx' 
    WHERE election_id = 'election123' AND issue_id = 'issue1'
""")
db.commit()

# Run tally
# $ python tally.py election123
# Error tallying issue issue1: Invalid ciphertext format
# Traceback (most recent call last):
#   ...
# [Script exits with error]

# Check application logs:
$ grep "issue1" application.log
# [No results - error not logged]

# Check stdout capture (if configured):
$ grep "issue1" stdout.log
# Error tallying issue issue1: Invalid ciphertext format
```

**Impact:**

1. **Cryptographic Failure Invisibility:** Decryption errors—which could indicate key compromise, data corruption, or active attacks—are not captured in security logs
2. **Operational Blindness:** If stdout is not captured to persistent storage (common in containerized/cloud environments), tally errors are completely lost
3. **Forensic Investigation Failure:** No audit trail exists to determine when tally failures occurred or which issues were affected

    - User identity (if authenticated)
    - Request method and path
    - Exception type and message

### 5. 🔍 No Log Rotation or Retention Policy

**Observation:**

The provided code does not configure log rotation, retention, or archival policies.

**Implications:**

1. **Disk Space Exhaustion:** Logs grow indefinitely, potentially filling disk
2. **Performance Degradation:** Large log files slow down log analysis tools
3. **Compliance Risk:** Many regulations require specific log retention periods
4. **Forensic Investigation Limits:** Very old logs may be lost before being needed

**Recommendation:**

Implement log rotation using Python's `RotatingFileHandler` or `TimedRotatingFileHandler`:

```python
# In main.py
from logging.handlers import TimedRotatingFileHandler

# Configure file handler with rotation
file_handler = TimedRotatingFileHandler(
    filename='logs/steve.log',
    when='midnight',  # Rotate daily
    interval=1,
    backupCount=90,  # Keep 90 days of logs
    encoding='utf-8'
)
file_handler.setFormatter(logging.Formatter(
    '%(asctime)s - %(name)s - %(levelname)s - %(message)s'
))

# Add to root logger
logging.getLogger().addHandler(file_handler)
```
