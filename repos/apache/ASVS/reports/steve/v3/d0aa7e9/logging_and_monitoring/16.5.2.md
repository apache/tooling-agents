## Findings

### High Severity

#### ASVS-1652-HIGH-001 — Multi-Issue Vote Submission Lacks Atomicity; Partial Failure Creates Inconsistent State

**Severity:** HIGH

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint`
- **Lines:** ~349-378

**Related Functions Checked:**
- `election.add_vote()` (called in loop)
- `flash_danger()` (error notification)

**Description:**

The vote submission endpoint processes multiple issue votes in a loop, with each vote committed individually. If a failure occurs mid-loop (database lock, crypto failure, disk full), votes processed before the failure are permanently recorded while subsequent votes are lost. The voter receives only a generic error message and cannot determine which votes were successfully recorded.

**Vulnerable Code:**

```python
for iid, votestring in votes.items():
    if iid not in issue_dict:
        await flash_danger(f'Invalid issue ID: {iid}')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    try:
        election.add_vote(result.uid, iid, votestring)
        _LOGGER.info(
            f'User[U:{result.uid}] voted on issue[I:{iid}] in election[E:{election.eid}]'
        )
    except Exception as e:
        _LOGGER.error(
            f'Error adding vote for user[U:{result.uid}] on issue[I:{iid}]: {e}'
        )
        await flash_danger(f'Error submitting vote for issue {iid}.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
```

**Data Flow:**

1. User submits votes for N issues via POST form
2. Loop calls `election.add_vote()` per issue (each auto-committed individually)
3. If issue K fails (database lock, crypto failure, disk full), issues 1..K-1 are permanently recorded
4. User sees generic "error" flash
5. No rollback of partial votes
6. Voter may not know which votes were recorded

**Proof of Concept:**

1. Voter submits ballot with 5 YNA votes simultaneously
2. Database experiences a transient lock on the 3rd `add_vote()` call
3. Votes 1-2 are recorded permanently; votes 3-5 are lost
4. User sees "Error submitting vote for issue {iid}" and is redirected
5. User re-submits, now recording 5 new votes BUT votes 1-2 are overwritten (if update) or duplicated
6. Audit log shows 2 successful votes and 1 error, but no indication that the batch was incomplete

**Impact:**

Election integrity violation. Partial vote recording without voter awareness could alter election outcomes. The voter cannot verify which issues were voted on successfully, and re-submission may produce unintended results. The system fails to maintain transactional consistency for the most critical operation.

**Remediation:**

```python
async def do_vote_endpoint(election):
    result = await basic_info()
    form = edict(await quart.request.form)
    if not form:
        await flash_danger('No vote data provided.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    votes = {}
    for key, value in form.items():
        if key.startswith('vote-'):
            iid = key.split('-', 1)[1]
            votes[iid] = value

    issues = election.list_issues()
    issue_dict = {i.iid: i for i in issues}

    # Validate all votes before committing any
    for iid in votes:
        if iid not in issue_dict:
            await flash_danger(f'Invalid issue ID: {iid}')
            return quart.redirect(f'/vote-on/{election.eid}', code=303)

    # Submit all votes atomically
    try:
        election.add_votes_batch(result.uid, votes)  # New atomic method
        for iid in votes:
            _LOGGER.info(
                f'User[U:{result.uid}] voted on issue[I:{iid}] in election[E:{election.eid}]'
            )
        await flash_success('Votes submitted successfully!')
    except Exception as e:
        _LOGGER.error(
            f'Batch vote failed for user[U:{result.uid}] in election[E:{election.eid}]: {type(e).__name__}'
        )
        await flash_danger('Error submitting votes. No votes were recorded. Please try again.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    return quart.redirect('/voter', code=303)
```

And in `election.py`, add a batch method wrapping in a transaction:

```python
def add_votes_batch(self, pid, votes_dict):
    """Atomically add all votes for a user. All succeed or none are committed."""
    md = self._all_metadata(self.S_OPEN)
    self.db.conn.execute('BEGIN TRANSACTION')
    try:
        for iid, votestring in votes_dict.items():
            mayvote = self.q_get_mayvote.first_row(pid, iid)
            if mayvote is None:
                raise VoterNotEligible(pid, iid)
            vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
            ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
            self.c_add_vote.perform(vote_token, ciphertext)
        self.db.conn.execute('COMMIT')
    except Exception:
        self.db.conn.execute('ROLLBACK')
        raise
```

#### ASVS-1652-HIGH-002 — add_vote Crashes on Missing Voter Eligibility Record Instead of Failing Securely

**Severity:** HIGH

**Location:**
- **File:** `v3/steve/election.py`
- **Function:** `add_vote`
- **Lines:** ~207-218

**Related Functions Checked:**
- `q_get_mayvote.first_row()` (database lookup)
- `crypto.gen_vote_token()` (called with null)

**Description:**

The `add_vote` method retrieves voter eligibility records from the database but does not check for null results. When a voter attempts to vote on an issue they're not eligible for, the database query returns `None`, and the subsequent access to `mayvote.salt` raises an `AttributeError` instead of a proper authorization failure.

**Vulnerable Code:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
```

**Data Flow:**

1. User POST `/do-vote/<eid>` with crafted issue IDs
2. `add_vote(pid, iid, votestring)` called
3. `q_get_mayvote.first_row(pid, iid)` returns `None` (voter not eligible for this specific issue)
4. `mayvote.salt` raises `AttributeError: 'NoneType' object has no attribute 'salt'`
5. Exception propagates to caller
6. Generic error logged with cryptic message
7. No authorization-failure audit entry

**Proof of Concept:**

```
POST /do-vote/abc123 HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: [valid_session]

vote-ISSUE_ID_NOT_ELIGIBLE=yes
```

Where `ISSUE_ID_NOT_ELIGIBLE` is a valid issue ID in the election but the authenticated user lacks a `mayvote` record for it.

**Impact:**

1. Authorization check fails insecurely — instead of a clear "not eligible" denial, produces an implementation error
2. Security audit trail is polluted with `AttributeError` instead of recording an authorization failure event
3. Could mask an attack where a user attempts to vote on issues they're not authorized for

**Remediation:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    if mayvote is None:
        _LOGGER.warning(
            f'Authorization failure: User[U:{pid}] not eligible for issue[I:{iid}]'
        )
        raise VoterNotEligible(pid, iid)

    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

### Medium Severity

#### ASVS-1652-MEDIUM-003 — Election State-Change Operations Lack Error Handling and Recovery

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `do_open_endpoint` (line ~399) and `do_close_endpoint` (line ~419)

**Related Functions Checked:**
- `PersonDB.open()` (external resource access)
- `election.open()` (multi-step state change)
- `election.add_salts()` (called within open)
- `election.gather_election_data()` (called within open)

**Description:**

The election opening and closing endpoints lack error handling for external resource access failures. The multi-step `election.open()` operation can fail partway through, leaving the election in an inconsistent state with no rollback mechanism. Database and cryptographic operation failures are not caught, and no audit trail is created for failures.

**Vulnerable Code (do_open_endpoint):**

```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    _LOGGER.info(f'User[U:{result.uid}] opened election[E:{election.eid}]')
    title = election.get_metadata().title
    await flash_success(f'Opened election: {title}')
    return quart.redirect(f'/manage/{election.eid}', code=303)
```

**And the underlying `election.open()` in `v3/steve/election.py` (line ~70):**

```python
def open(self, pdb):
    assert self.is_editable()
    self.add_salts()          # Step 1: commits own transaction
    edata = self.gather_election_data(pdb)  # Step 2: read-only
    salt = crypto.gen_salt()                # Step 3: crypto op
    opened_key = crypto.gen_opened_key(edata, salt)  # Step 4: crypto op
    self.c_open.perform(salt, opened_key, self.eid)   # Step 5: autocommit
```

**Data Flow:**

1. Admin clicks "Open Election"
2. `PersonDB.open()` called (can fail if DB locked/corrupted)
3. `election.open()` called
4. `add_salts()` commits salts to mayvote rows
5. If `gather_election_data()` or crypto op fails, salts are permanently written but election remains "editable"
6. Election in inconsistent intermediate state
7. No error page, no audit log for failure

**Impact:**

- If `PersonDB.open()` fails: unhandled exception, no audit trail, user sees raw 500 error
- If failure occurs after `add_salts()` but before `c_open.perform()`: election has salts applied but is still "editable," creating an inconsistent state where a retry would re-salt (overwriting the first set)
- The admin receives no actionable feedback about what went wrong or what state the election is in
- No circuit breaker or retry mechanism for the multi-step state transition

**Remediation:**

```python
async def do_open_endpoint(election):
    result = await basic_info()
    try:
        pdb = steve.persondb.PersonDB.open(DB_FNAME)
    except Exception:
        _LOGGER.error(
            f'PersonDB unavailable when User[U:{result.uid}] tried to open election[E:{election.eid}]'
        )
        await flash_danger('System error: unable to access person database. Please try again later.')
        return quart.redirect(f'/manage/{election.eid}', code=303)

    try:
        election.open(pdb)
    except Exception:
        _LOGGER.error(
            f'Failed to open election[E:{election.eid}] by User[U:{result.uid}]',
            exc_info=True
        )
        await flash_danger('Failed to open election. The election state may need manual review.')
        return quart.redirect(f'/manage/{election.eid}', code=303)

    _LOGGER.info(f'User[U:{result.uid}] opened election[E:{election.eid}]')
    await flash_success(f'Opened election: {election.get_metadata().title}')
    return quart.redirect(f'/manage/{election.eid}', code=303)
```

And make `election.open()` atomic:

```python
def open(self, pdb):
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)

    self.db.conn.execute('BEGIN TRANSACTION')
    try:
        self._add_salts_in_txn()  # Don't use separate transaction
        edata = self.gather_election_data(pdb)
        salt = crypto.gen_salt()
        opened_key = crypto.gen_opened_key(edata, salt)
        self.c_open.perform(salt, opened_key, self.eid)
        self.db.conn.execute('COMMIT')
    except Exception:
        self.db.conn.execute('ROLLBACK')
        raise
```

#### ASVS-1652-MEDIUM-004 — Database Transactions Lack Rollback on Failure

**Severity:** MEDIUM

**Location:**
- **File:** `v3/steve/election.py`
- **Functions:** `add_salts` (line ~117) and `delete` (line ~50)

**Related Functions Checked:**
- `crypto.gen_salt()` (can fail on entropy exhaustion)
- `c_salt_mayvote.perform()` (database write)
- `c_delete_mayvote.perform()` (database write)
- `c_delete_issues.perform()` (database write)
- `c_delete_election.perform()` (database write)

**Description:**

Multiple methods explicitly begin database transactions but fail to include rollback logic in exception handlers. If any operation within the transaction fails (crypto operation, database write, disk full), the transaction is neither committed nor rolled back, leaving the database connection in an undefined state.

**Vulnerable Code (`add_salts`):**

```python
def add_salts(self):
    assert self.is_editable()
    self.db.conn.execute('BEGIN TRANSACTION')
    self.q_all_issues.perform(self.eid)
    for mayvote in self.q_all_issues.fetchall():
        salt = crypto.gen_salt()
        self.c_salt_mayvote.perform(salt, mayvote.rowid)
    self.db.conn.execute('COMMIT')
```

**Vulnerable Code (`delete`):**

```python
def delete(self):
    assert self.is_editable()
    self.db.conn.execute('BEGIN TRANSACTION')
    self.c_delete_mayvote.perform(self.eid)
    self.c_delete_issues.perform(self.eid)
    self.c_delete_election.perform(self.eid)
    self.db.conn.execute('COMMIT')
```

**Data Flow:**

1. Operation begins transaction
2. Performs multiple database writes
3. If any `crypto.gen_salt()` call fails (entropy exhaustion) or any `perform()` raises an exception (disk full, DB corruption)
4. Transaction is never committed AND never rolled back
5. SQLite connection remains in an open transaction
6. Subsequent operations on the same connection may fail or see stale data

**Impact:**

- Database connection left in undefined transactional state after failure
- In `add_salts`: partial salt assignment means some voters have salts and some don't, breaking the election opening process
- In `delete`: partial deletion could leave orphaned records (e.g., issues without an election) that violate referential integrity
- SQLite's rollback journal may hold a lock, blocking other connections

**Remediation:**

```python
def add_salts(self):
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)

    self.db.conn.execute('BEGIN TRANSACTION')
    try:
        self.q_all_issues.perform(self.eid)
        for mayvote in self.q_all_issues.fetchall():
            salt = crypto.gen_salt()
            self.c_salt_mayvote.perform(salt, mayvote.rowid)
        self.db.conn.execute('COMMIT')
    except Exception:
        self.db.conn.execute('ROLLBACK')
        _LOGGER.error(f'Failed to add salts for election[E:{self.eid}], rolled back')
        raise
```

Apply the same pattern to `delete()` and any other methods using explicit transactions.

## Positive Security Patterns

1. **PersonNotFound graceful handling**: `vote_on_page` and `admin_page` properly handle `PersonNotFound` with fallback values or 404 pages.

2. **ElectionNotFound/IssueNotFound decorators**: The `load_election` and `load_election_issue` decorators properly catch not-found exceptions and return user-friendly 404 pages.

3. **Tamper detection before tally**: `tally.py` checks `is_tampered()` before proceeding and exits cleanly with a clear message.

4. **Vote shuffling**: `tally_issue()` shuffles decrypted votes to prevent order correlation — defense in depth for ballot secrecy.

5. **Generic user-facing error messages**: `do_vote_endpoint` shows generic flash messages while logging details server-side.

6. **ID collision retry**: `create()` and `add_issue()` retry with new random IDs on `IntegrityError`.

---

## Architecture Observations

1. **`assert` for security enforcement**: All election state checks (`is_editable()`, `is_open()`) use Python `assert` statements. These are entirely stripped when Python runs with the `-O` optimization flag, silently removing all state enforcement. While not directly an "external resource failure," this represents a fragile security mechanism that can fail based on runtime configuration.

2. **No global error handler**: The application does not define `@APP.errorhandler(500)` or similar global exception handlers. Unhandled exceptions fall through to Quart's defaults, which may expose stack traces depending on configuration.

3. **Single-connection database pattern**: Each request opens a new database connection through the `Election` class. There's no connection pooling or health-check mechanism. A corrupted or locked database file would cause all requests to fail without degradation.

4. **PersonDB opened per-request**: `PersonDB.open()` is called ad-hoc in individual handlers rather than managed as an application-level resource with health monitoring.

## Recommendations Summary

### Immediate Actions (Critical/High Priority)

1. **Add null checks after all database lookups** in `add_vote` and `tally_issue` with proper authorization-failure logging (Addresses ASVS-1652-HIGH-002)

2. **Wrap multi-issue vote submission in a database transaction with rollback** to ensure atomic vote recording across all issues (Addresses ASVS-1652-HIGH-001)

3. **Add try/except with rollback to all explicit `BEGIN TRANSACTION` blocks** in `add_salts()`, `delete()`, and similar methods (Addresses ASVS-1652-MEDIUM-004)

### Short-term Actions (Medium Priority)

4. **Add error handling to state-change endpoints** including `do_open_endpoint`, `do_close_endpoint`, and other administrative handlers (Addresses ASVS-1652-MEDIUM-003)

5. **Make `election.open()` atomic** by wrapping the entire multi-step process (salts + state change) in a single transaction (Addresses ASVS-1652-MEDIUM-003)

6. **Replace all security-critical `assert` statements** with explicit `if/raise` patterns that cannot be optimized away

### Long-term Actions (Architectural Improvements)

7. **Add a global `@APP.errorhandler(Exception)`** that returns generic error pages and logs details server-side

8. **Implement connection health checks or retry logic** for database operations with circuit breaker patterns for external resource access

9. **Centralize database connection management** with connection pooling and health monitoring

10. **Implement structured error types** (e.g., `VoterNotEligible`, `ElectionBadState`) for better error handling and audit logging
