## Findings

### 🔴 CRITICAL

#### ASVS-1621-CRITICAL-001: Tampering Detection Event Bypasses Structured Logging

**Location:** `v3/server/bin/tally.py`, function `main()`, lines 153-155

**Related Functions Checked:**
- `main()` (lines 145-171) — tally entry point
- `tally_election()` (lines 88-142) — tally execution
- `steve.election.Election.is_tampered()` — tamper detection logic (called but result not logged)

**Description:**

The most critical security event in the entire voting system — **detection of election tampering** — bypasses the configured structured logging framework and outputs only to stdout via `print()`. This represents a **Type B control gap**: the logging control (`_LOGGER`) is imported, configured, and used elsewhere in the same file for less critical events, but is NOT invoked for the highest-severity security event.

**Vulnerable Code:**
```python
# Check for tampering
pdb = steve.persondb.PersonDB.open(db_fname)
if election.is_tampered(pdb):
    print(f'Error: Election {election_id} has been tampered with. Cannot proceed.')
    sys.exit(1)
```

**Context:** The same file demonstrates proper logging usage:
```python
# Line 35: Logger is configured
_LOGGER = logging.getLogger(__name__)

# Line 163: Logging framework is initialized
logging.basicConfig(level=logging.INFO)

# Line 118: Less critical event uses structured logging
_LOGGER.info(f'Tallying {len(issues)} issues ...')
```

**Data Flow:**
```
Election data → is_tampered() check → TRUE
  → print() to stdout (unstructured)
  → sys.exit(1)
  → NO structured log entry created
  → SIEM/log aggregation may never receive alert
```

**Missing ASVS 16.2.1 Metadata:**
- ❌ **When:** No structured timestamp (stdout not guaranteed to be timestamped by log collection)
- ❌ **Where:** No system/hostname context for distributed deployments
- ❌ **Who:** No operator identity (who initiated the tally that detected tampering)
- ⚠️ **What:** Partial (election ID present, but no structured event type for alerting rules)
- ❌ **Outcome:** Not structured (plain text message, not machine-parseable)

**Impact:**

Election tampering is the **highest-severity security event** in a voting system. Using `print()` instead of structured logging means:

1. **Alert Loss Risk:** Stdout may not be captured by log aggregation systems, especially in daemon/cron/systemd deployments
2. **No Forensic Timeline:** Without timestamp and operator identity, investigators cannot reconstruct when tampering was detected or who discovered it
3. **False Security Confidence:** Security team believes logging covers all events when the most critical one is excluded
4. **No SIEM Correlation:** Cannot correlate tampering detection with other security events (failed login attempts, unusual access patterns, etc.)
5. **Compliance Failure:** Election systems typically require complete audit trails; this gap would fail most election security audits

**Proof of Concept:**

```bash
# Attacker tampers with election database
sqlite3 steve.db "UPDATE elections SET opened_key = 'malicious' WHERE eid = 'board2026'"

# Admin runs tally - tampering detected
python3 tally.py --election board2026 --db steve.db

# Output goes to stdout only:
# "Error: Election board2026 has been tampered with. Cannot proceed."

# Security logs contain NO entry for this critical event
grep -i tamper /var/log/steve/*.log
# (no results)

# SIEM dashboard shows no tampering alerts
# Security team has no visibility into the incident
```

**Remediation:**

```python
# Check for tampering
pdb = steve.persondb.PersonDB.open(db_fname)
if election.is_tampered(pdb):
    _LOGGER.critical(
        f'TAMPERING_DETECTED: election[E:{election_id}] has been tampered with. '
        f'Tally aborted. db_path={db_fname} spy_on_open={spy_on_open}'
    )
    # Still print for CLI user feedback
    print(f'Error: Election {election_id} has been tampered with. Cannot proceed.')
    sys.exit(1)
```

**Verification:**
```python
# Add unit test to ensure critical events are logged
def test_tampering_detection_logged(caplog):
    with caplog.at_level(logging.CRITICAL):
        # Trigger tampering detection
        result = subprocess.run(['python3', 'tally.py', '--election', 'tampered_election'])
        assert 'TAMPERING_DETECTED' in caplog.text
        assert 'election[E:tampered_election]' in caplog.text
```

#### ASVS-1621-MEDIUM-002: Source IP Address Missing From All Web Security Log Entries

**Location:** `v3/server/pages.py`, all state-changing endpoint handlers (lines 116, 438, 468, 490, 505, 529, 547, 565)

**Related Functions Checked:**
- `do_vote_endpoint()` (lines 410-443)
- `do_create_endpoint()` (lines 455-473)
- `do_open_endpoint()` (lines 476-493)
- `do_close_endpoint()` (lines 495-509)
- `do_add_issue_endpoint()` (lines 511-532)
- `do_edit_issue_endpoint()` (lines 534-555)
- `do_delete_issue_endpoint()` (lines 557-575)
- `_set_election_date()` (line ~116)

**Description:**

ASVS 16.2.1 requires "where" metadata for detailed investigation. For web applications, the **source IP address** is essential context that is **completely absent** from all security log entries. Every state-changing operation logs user identity and action details, but never records the IP address from which the request originated.

**Vulnerable Code (representative example from `do_vote_endpoint`):**
```python
_LOGGER.info(
    f'User[U:{result.uid}] voted on issue[I:{iid}] in election[E:{election.eid}]'
)
```

**All Affected Log Statements:**

| Function | Line | Current Log Format | Missing Data |
|----------|------|-------------------|--------------|
| `_set_election_date` | ~116 | `User[U:{uid}] set {field} for election[E:{eid}]` | Source IP |
| `do_vote_endpoint` | ~438 | `User[U:{uid}] voted on issue[I:{iid}]` | Source IP |
| `do_create_endpoint` | ~468 | `User[U:{uid}] created election[E:{eid}]` | Source IP |
| `do_open_endpoint` | ~490 | `User[U:{uid}] opened election[E:{eid}]` | Source IP |
| `do_close_endpoint` | ~505 | `User[U:{uid}] closed election[E:{eid}]` | Source IP |
| `do_add_issue_endpoint` | ~529 | `User[U:{uid}] added issue[I:{iid}]` | Source IP |
| `do_edit_issue_endpoint` | ~547 | `User[U:{uid}] edited issue[I:{iid}]` | Source IP |
| `do_delete_issue_endpoint` | ~565 | `User[U:{uid}] deleted issue[I:{iid}]` | Source IP |

**Data Flow:**
```
HTTP Request from IP 203.0.113.42
  → Quart framework (request.remote_addr available)
  → Endpoint handler
  → _LOGGER.info(...) ← IP address NOT included
  → Log entry: "User[U:alice] voted on issue[I:5]"
  → IP address lost forever
```

**Missing ASVS 16.2.1 Metadata:**
- ❌ **Where:** Source IP address (`quart.request.remote_addr`)
- ⚠️ **Where (enhanced):** Request ID for correlation across log entries
- ⚠️ **Where (enhanced):** User-Agent for device fingerprinting

**Impact:**

Without source IP addresses, security teams cannot:

1. **Detect Compromised Accounts:** Cannot identify votes/actions from unexpected geolocations
   - Example: Alice normally votes from California (IP range 192.0.2.0/24), but suddenly votes from Eastern Europe (203.0.113.0/24) — no way to detect this anomaly

2. **Correlate Multi-Account Attacks:** Cannot identify single attacker using multiple compromised accounts
   - Example: 10 different user accounts all voting from the same IP address — pattern invisible without IP logging

3. **Investigate Incidents:** During incident response, cannot determine which requests were malicious
   - Example: "User[U:bob] deleted 50 issues" — was this Bob from his home IP, or an attacker who compromised Bob's credentials?

4. **Enforce Rate Limiting:** Cannot implement IP-based rate limiting or abuse prevention
   - Example: Cannot detect a single IP attempting to vote in 100 different elections in rapid succession

5. **Meet Compliance Requirements:** Many election security standards require IP address logging for audit trails

**Proof of Concept:**

```python

# Attacker (from IP 203.0.113.66) votes as Alice in 20 elections

# [2026-04-22 10:15:32|INFO|pages] User[U:alice] voted on issue[I:1] in election[E:board2026]
# [2026-04-22 10:15:35|INFO|pages] User[U:alice] voted on issue[I:2] in election[E:infra2026]
# ... (18 more entries)

# Security team investigates:
# Q: "Are these votes from Alice's usual location?"
# A: Unknown - no IP address logged

# Q: "Are multiple accounts being accessed from the same IP?"
# A: Unknown - no IP address logged

# Q: "When did the compromise occur?"
# A: Unknown - cannot correlate with failed login attempts (no IP to correlate)

# Meanwhile, Alice's actual IP (192.0.2.50) shows no activity
# But this cannot be determined from logs
```

**Remediation:**

```python
# Create a centralized security logging helper in pages.py
async def log_security_event(
    action: str,
    details: str,
    level: int = logging.INFO
) -> None:
    """
    Log security-relevant events with complete ASVS 16.2.1 metadata.
    
    Args:
        action: Event type (e.g., 'vote_submitted', 'election_opened')
        details: Structured details (e.g., 'issue[I:5] election[E:board2026]')
        level: Logging level (default INFO)
    """
    result = await basic_info()
    source_ip = quart.request.remote_addr or 'unknown'
    request_id = quart.request.headers.get('X-Request-ID', 'none')
    user_agent = quart.request.headers.get('User-Agent', 'unknown')[:100]  # Truncate
    
    _LOGGER.log(
        level,
        f'[ip:{source_ip}] [req:{request_id}] User[U:{result.uid}] '
        f'action={action} {details} user_agent="{user_agent}"'
    )

# Refactor endpoint logging to use helper:
async def do_vote_endpoint():
    # ... existing code ...
    
    # OLD:
    # _LOGGER.info(
    #     f'User[U:{result.uid}] voted on issue[I:{iid}] in election[E:{election.eid}]'
    # )
    
    # NEW:
    await log_security_event(
        'vote_submitted',
        f'issue[I:{iid}] election[E:{election.eid}]'
    )
    
    # ... existing code ...
```

#### ASVS-1621-MEDIUM-003: Authorization Failures Not Logged for Vote Access and Document Access

**Location:** `v3/server/pages.py`
- `vote_on_page()` function, lines 294-299
- `serve_doc()` function, lines 607-611

**Related Functions Checked:**
- `vote_on_page()` (lines 274-354) — vote form rendering with eligibility check
- `serve_doc()` (lines 589-625) — document serving with eligibility check
- `steve.election.Election.q_find_issues` — database query for eligibility
- `steve.election.ElectionDB.q_get_mayvote` — database query for document access

**Description:**

Authorization failures are **key security signals** that indicate potential attacks or misconfigurations. The application performs authorization checks before allowing users to vote or access election documents, but **silently returns 404 errors without creating any log entry** when access is denied. This represents a **Type A control gap** — no logging control exists at these authorization decision points.

**Vulnerable Code #1 — Vote Access Authorization:**
```python
# v3/server/pages.py, lines 294-299
# Check if user can vote in this election (has mayvote entries)
election.q_find_issues.perform(result.uid, election.eid)
if not election.q_find_issues.fetchall():
    result = await basic_info()
    result.title = 'Access Denied'
    result.eid = election.eid
    raise_404(T_BAD_EID, result)  # Reuse template for consistency
    # ← NO LOG ENTRY CREATED
```

**Vulnerable Code #2 — Document Access Authorization:**
```python
# v3/server/pages.py, lines 607-611
row = db.q_get_mayvote.first_row(result.uid, iid)
if not row:
    # Nothing fancy. Just pretend the file does not exist
    quart.abort(404)
    # NOTREACHED
    # ← NO LOG ENTRY CREATED
```

**Data Flow:**
```
Authenticated user (alice) requests /vote-on/board2026
  → Authorization check: q_find_issues.perform('alice', 'board2026')
  → Result: No mayvote entries found
  → Return 404 error to user
  → NO log entry created
  → Security team has zero visibility

Authenticated user (bob) requests /docs/5/candidate_statement.pdf
  → Authorization check: q_get_mayvote.first_row('bob', 5)
  → Result: No mayvote entry found
  → Return 404 error to user
  → NO log entry created
  → Security team has zero visibility
```

**Missing ASVS 16.2.1 Metadata:**
- ❌ **When:** No timestamp (event not logged at all)
- ❌ **Where:** No source IP
- ❌ **Who:** User identity available but not logged
- ❌ **What:** Resource being accessed (election ID, issue ID) not logged
- ❌ **Outcome:** Authorization denial not recorded

**Impact:**

Without authorization failure logging, the system cannot:

1. **Detect Reconnaissance Attacks:**
   - Attacker probes `/vote-on/<eid>` for all possible election IDs to discover active elections
   - Attacker attempts to access `/docs/<iid>/*` to enumerate issues they're not authorized for
   - All probing attempts are invisible to security monitoring

2. **Identify Compromised Accounts:**
   - Legitimate user's credentials are stolen
   - Attacker attempts to vote in elections the user isn't eligible for
   - Failed attempts generate no alerts — compromise goes undetected

3. **Detect Privilege Escalation Attempts:**
   - User attempts to access administrative elections they shouldn't have access to
   - Pattern of repeated authorization failures (indicating attack) is invisible

4. **Meet Audit Requirements:**
   - Election security audits typically require logging of ALL access control decisions
   - Current implementation only logs successful operations, creating incomplete audit trail

5. **Investigate Incidents:**
   - Post-incident: "Did the attacker attempt to access other elections?"
   - Answer: Unknown — no log data exists

**Proof of Concept:**

# pages.py - centralized security logging

from enum import Enum
from typing import Optional
import logging

class SecurityEventType(Enum):
    """Enumeration of security-relevant event types."""
    VOTE_SUBMITTED = "vote_submitted"
    ELECTION_CREATED = "election_created"
    ELECTION_OPENED = "election_opened"
    ELECTION_CLOSED = "election_closed"
    ISSUE_ADDED = "issue_added"
    ISSUE_EDITED = "issue_edited"
    ISSUE_DELETED = "issue_deleted"
    AUTHZ_DENIED = "authz_denied"
    ELECTION_DATE_SET = "election_date_set"

async def log_security_event(
    event_type: SecurityEventType,
    election_id: Optional[str] = None,
    issue_id: Optional[str] = None,
    level: int = logging.INFO,
    **extra_context
) -> None:
    """
    Log security-relevant events with complete ASVS 16.2.1 metadata.
    
    This function ensures ALL security events include:
    - WHEN: Timestamp (from logging framework)
    - WHERE: Source IP, request ID
    - WHO: Authenticated user ID
    - WHAT: Event type, resource IDs
    - OUTCOME: Implicit from log level or explicit in extra_context
    
    Args:
        event_type: Type of security event (from SecurityEventType enum)
        election_id: Election identifier (if applicable)
        issue_id: Issue identifier (if applicable)
        level: Logging level (default INFO)
        **extra_context: Additional context (e.g., error messages, outcomes)
    """
    result = await basic_info()
    
    # Required metadata per ASVS 16.2.1
    metadata = {
        'event': event_type.value,
        'user_id': result.uid,
        'source_ip': quart.request.remote_addr or 'unknown',
        'request_id': quart.request.headers.get('X-Request-ID', 'none'),
        'user_agent': quart.request.headers.get('User-Agent', 'unknown')[:100],
    }
    
    # Optional resource identifiers
    if election_id:
        metadata['election_id'] = election_id
    if issue_id:
        metadata['issue_id'] = issue_id
    
    # Additional context
    metadata.update(extra_context)
    
    # Format log message
    msg_parts = [f"{k}={v}" for k, v in metadata.items()]
    message = ' '.join(msg_parts)
    
    _LOGGER.log(level, message)

# Usage in endpoints:
async def do_vote_endpoint():
    # ... existing code ...
    
    await log_security_event(
        SecurityEventType.VOTE_SUBMITTED,
        election_id=election.eid,
        issue_id=iid
    )
    
    # ... existing code ...

async def vote_on_page():
    # ... existing code ...

            SecurityEventType.AUTHZ_DENIED,
            election_id=election.eid,
            level=logging.WARNING,
            reason='no_mayvote_entries'
        )

### 5. No Request Correlation Mechanism

**Observation:**
Web requests that span multiple log entries have no correlation ID to link them together.

**Example Scenario:**
```
User Alice votes on 3 issues in rapid succession:
[2026-04-22 10:15:32|INFO|pages] User[U:alice] voted on issue[I:5]
[2026-04-22 10:15:33|INFO|pages] User[U:alice] voted on issue[I:6]
[2026-04-22 10:15:34|INFO|pages] User[U:alice] voted on issue[I:7]

Question: Were these 3 votes from the same session or 3 different sessions?
Answer: Cannot determine - no session/request correlation
```

**Impact:**
- Cannot track user actions within a single session
- Cannot correlate multi-step operations (e.g., view election → vote → view results)
- Difficult to detect attack patterns that span multiple requests
- Cannot distinguish legitimate rapid voting from automated attacks
