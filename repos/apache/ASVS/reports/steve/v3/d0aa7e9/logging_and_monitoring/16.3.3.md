## Findings

### CRITICAL Severity

#### **ASVS-1633-SEV-001: Tamper Detection Events Logged via print() Instead of Security Logger**

**Severity:** CRITICAL  
**Gap Type:** Type B — Control EXISTS but logging NOT properly CALLED  
**File:** `v3/server/bin/tally.py`  
**Function:** `main()`  
**Lines:** 133-136  

**Related Functions Checked:**
- `tally_election()` (lines 76-113) — calls `main()` tamper check
- `election.is_tampered()` in `v3/steve/election.py` (lines 150-161) — performs actual integrity check

**Description:**  
The application implements a critical tamper detection control that verifies election data integrity before tallying votes. However, when tampering is detected, the event is output using `print()` to stdout instead of the structured security logger. This means the most critical security event in an election system — evidence of data tampering — is not recorded in the application's security log.

**Vulnerable Code:**
```python
# v3/server/bin/tally.py:133-136
if election.is_tampered(pdb):
    print(f'Error: Election {election_id} has been tampered with. Cannot proceed.')
    sys.exit(1)
```

**Data Flow:**
1. `election.is_tampered(pdb)` detects tampering by comparing computed `opened_key` hash against stored value
2. `print()` outputs message to stdout
3. No structured log entry created via `_LOGGER`
4. Security event lost in production environments where:
   - ASGI servers don't capture stdout
   - Log aggregators only ingest structured logs
   - SIEMs monitor log files, not console output
   - Stdout may be redirected to `/dev/null`

**Proof of Concept:**
1. Attacker modifies election data after the election is opened (e.g., adds voters, modifies issues)
2. Administrator runs `tally.py --election-id ELEC123`
3. Tampering is detected and operation aborts with console message
4. Check application log file: **no entry exists**
5. Check SIEM dashboard: **no alert triggered**
6. If deployment redirects stdout (common in systemd services): **event is silently lost**

Result: Election fraud is detected but not recorded, eliminating post-incident investigation capability.

**Impact:**  
- **Election Integrity:** The single most critical security event in an election system is not auditable
- **Incident Response:** No evidence trail for forensic investigation of tampering incidents
- **Compliance:** Fails ASVS 16.3.3 requirement to log security control bypass attempts
- **Detection:** Automated monitoring systems (SIEM, alerting) cannot trigger on this critical event
- **Accountability:** No record of when tampering was detected, which election was affected, or who ran the tally

**Remediation:**
```python
def main(spy_on_open, election_id, issue_id, db_fname, output_format):
    # ... existing code ...
    
    # Check for tampering
    pdb = steve.persondb.PersonDB.open(db_fname)
    if election.is_tampered(pdb):
        _LOGGER.critical(
            'TAMPER_DETECTED: Election[E:%s] has been tampered with. '
            'Tally aborted. Computed opened_key does not match stored value.',
            election_id
        )
        print(f'Error: Election {election_id} has been tampered with. Cannot proceed.')
        sys.exit(1)
    else:
        _LOGGER.info(
            'TAMPER_CHECK_PASSED: Election[E:%s] integrity verified before tally.',
            election_id
        )
    
    # Proceed with tally
    tally_election(election, issue_id, output_format)
```

**Additional Recommendations:**
- Log both tamper detection failures AND successful integrity checks
- Include computed vs. expected hash values in tamper detection logs (for forensics)
- Consider adding alerting configuration examples for this critical event

### HIGH Severity

#### **ASVS-1633-SEV-002: Tally Operations Create No Audit Trail**

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO security event logging  
**File:** `v3/server/bin/tally.py`  
**Functions:** `main()` (lines 116-142), `tally_election()` (lines 76-113)  
**Lines:** 116-142, 76-113  

**Related Functions Checked:**
- `election.tally_issue()` in `v3/steve/election.py` (line 286) — performs vote decryption
- `election.list_issues()` in `v3/steve/election.py` — retrieves issues to tally
- `election.get_state()` in `v3/steve/election.py` — checks election state

**Description:**  
Tallying is a highly privileged operation that decrypts all votes in an election and computes results. The tally tool provides no audit trail of these operations, including: who initiated the tally, which election was tallied, whether the `--spy-on-open-elections` flag was used (allowing premature result access), when the operation occurred, or a summary of results. This directly contradicts the domain requirement that "tally operations create audit trails."

**Vulnerable Code:**
```python
# v3/server/bin/tally.py:76-113
def tally_election(election, issue_id, output_format):
    issues = election.list_issues()
    if not issues:
        _LOGGER.error('No issues to tally in this election.')
        return
    # ... tallies all issues ...
    # NO LOGGING: who tallied, what election, when, results summary
```

```python
# v3/server/bin/tally.py:116-142
def main(spy_on_open, election_id, issue_id, db_fname, output_format):
    # ... loads election, checks tampering ...
    tally_election(election, issue_id, output_format)
    # NO LOGGING: tally initiation or completion
```

**Data Flow:**
1. CLI user runs `tally.py --election-id ABC123 --spy-on-open-elections`
2. Election data loaded, tampering check performed
3. `tally_election()` decrypts all votes
4. Results computed and output to console/file
5. **No audit log entry created at any step**

**Proof of Concept:**
```bash
# Administrator runs tally on open election (sensitive operation)
$ python tally.py --election-id ELEC123 --spy-on-open-elections --output json

# Check application logs
$ grep "ELEC123" /var/log/steve/app.log
# Result: No entries

# Questions that cannot be answered:
# - Who accessed the results?
# - When was the tally performed?
# - Was --spy-on-open-elections used?
# - How many votes were tallied?
# - What were the results?
```

**Impact:**
- **Accountability:** No record of who performed privileged tally operations
- **Audit Trail:** Cannot reconstruct timeline of result access for investigations
- **Insider Threat:** Administrators can access results without detection
- **Compliance:** Fails requirement for audit trails on privileged operations
- **Premature Access:** Use of `--spy-on-open-elections` flag is completely invisible
- **Result Integrity:** No verification that tallies were performed at appropriate times

**Remediation:**

#### **ASVS-1633-SEV-003: Authorization Failures Not Logged**

**Severity:** HIGH  
**Gap Type:** Type A — Entry points with NO security event logging for access denial  
**File:** `v3/server/pages.py`  
**Functions:** `vote_on_page()` (lines 246-251), `serve_doc()` (lines 610-614)  
**Lines:** 246-251, 610-614  

**Related Functions Checked:**
- `election.q_find_issues.perform()` (line 247) — queries mayvote authorization
- `db.q_get_mayvote.first_row()` (line 610) — checks document access authorization
- `raise_404()` (line 251) — returns 404 response
- `quart.abort(404)` (line 613) — returns 404 response

**Description:**  
The application performs authorization checks before allowing users to vote in elections or access issue documents. When these checks fail, the application returns a 404 response but does not log the authorization failure. This makes unauthorized access attempts invisible to security monitoring, allowing attackers to probe for accessible resources without detection.

**Vulnerable Code:**

**Location 1** — `vote_on_page()`, lines 246-251:
```python
# Check if user can vote in this election (has mayvote entries)
election.q_find_issues.perform(result.uid, election.eid)
if not election.q_find_issues.fetchall():
    result = await basic_info()
    result.title = 'Access Denied'
    result.eid = election.eid
    raise_404(T_BAD_EID, result)  # Reuse template for consistency
    # NO LOGGING OF THIS AUTHORIZATION FAILURE
```

**Location 2** — `serve_doc()`, lines 610-614:
```python
row = db.q_get_mayvote.first_row(result.uid, iid)
if not row:
    # Nothing fancy. Just pretend the file does not exist
    quart.abort(404)
    # NOTREACHED
    # NO LOGGING OF THIS AUTHORIZATION FAILURE
```

**Data Flow:**
1. Authenticated user requests protected resource (election vote page or issue document)
2. Authorization query checks `mayvote` table for user's access rights
3. Query returns no results (user not authorized)
4. 404 response returned to user
5. **No security log entry created**

**Proof of Concept:**

**Scenario 1: Election Access Enumeration**
```bash

# 404 - not authorized (not logged)

# ... repeat for 100 election IDs ...

curl -H "Cookie: session=..." https://steve.apache.org/vote-on/ELEC099
# 200 - authorized (logged as normal access)

# Result: Attacker successfully mapped accessible elections with no alerts
```

**Scenario 2: Document Access Probing**
```bash

curl https://steve.apache.org/docs/ISSUE123/financial-report.pdf
# 404 - not authorized (not logged)

curl https://steve.apache.org/docs/ISSUE123/meeting-notes.pdf
# 404 - not authorized (not logged)

# Insider threat: Employee attempts to access elections outside their department
# No detection, no investigation triggered
```

**Impact:**
- **Attack Detection:** Unauthorized access attempts are invisible to security monitoring
- **Insider Threat:** Employees probing for unauthorized access cannot be detected
- **Reconnaissance:** Attackers can enumerate resources without triggering alerts
- **Compliance:** Fails ASVS 16.3.3 requirement to log security control bypass attempts
- **Incident Response:** No forensic evidence of access attempts during investigations
- **Pattern Analysis:** Cannot identify users repeatedly attempting unauthorized access

#### **ASVS-1633-SEV-005: Log Injection via Unsanitized User Input in Log Messages**

**Severity:** MEDIUM  
**Gap Type:** Type C — Control (logging) CALLED but input NOT sanitized  
**File:** `v3/server/pages.py`  
**Function:** `do_create_endpoint()`  
**Lines:** 459  

**Related Functions Checked:**
- `do_create_endpoint()` form processing (lines 443-462)
- `_LOGGER.info()` call (line 459)
- Log format configuration in `main.py:59`

**Description:**  
The application logs election creation events including the user-provided election title. The title is inserted into the log message using f-string interpolation without sanitization, allowing attackers to inject newline characters and forge log entries. This can be used to create false security events, hide malicious activity, or corrupt log parsing by SIEM systems.

**Vulnerable Code:**
```python
# v3/server/pages.py:459
_LOGGER.info(
    f'User[U:{result.uid}] created election[E:{election.eid}];'
    f' title: "{form.title}"'
)
```

**Data Flow:**
1. User submits election creation form with malicious title containing newlines
2. `form.title` (user-controlled) inserted directly into f-string
3. `_LOGGER.info()` called with unsanitized message
4. Log formatter outputs message with injected content
5. Forged log entries appear in log files

**Proof of Concept:**
```python
# Attacker creates election with malicious title
POST /do-create-election
Content-Type: application/x-www-form-urlencoded

title=Legit Election\n[2026-04-22 12:00:00|CRITICAL|steve.election] TAMPER_DETECTED: Election[E:REAL_EID] has been tampered with&monitors=admin@example.org

# Resulting log output:
[2026-04-22 11:30:00|INFO|steve.pages] User[U:attacker@example.org] created election[E:ELEC999]; title: "Legit Election
[2026-04-22 12:00:00|CRITICAL|steve.election] TAMPER_DETECTED: Election[E:REAL_EID] has been tampered with"

# The forged CRITICAL log entry appears legitimate and may trigger false alerts
```

**Impact:**
- **Log Integrity:** Attackers can forge arbitrary log entries
- **False Alerts:** Injected CRITICAL/ERROR entries trigger false SIEM alerts
- **Evidence Tampering:** Malicious log entries can obscure real security events
- **Log Parsing:** SIEM/log aggregators may misparse logs with injected newlines
- **Forensics:** Investigators cannot trust log data for incident analysis
- **Compliance:** Compromised log integrity fails audit requirements

**Remediation:**
```python
def sanitize_for_log(value: str) -> str:
    """
    Remove or encode control characters to prevent log injection.
    
    Args:
        value: User-controlled string to sanitize
        
    Returns:
        Sanitized string safe for log output
    """
    if value is None:
        return 'None'
    # Replace control characters with escaped versions
    return (value
            .replace('\n', '\\n')
            .replace('\r', '\\r')
            .replace('\t', '\\t')
            .replace('\x00', '\\x00'))

# Apply to all user-controlled values in logs:
_LOGGER.info(
    'User[U:%s] created election[E:%s]; title: "%s"',
    result.uid, 
    election.eid, 
    sanitize_for_log(form.title)
)
```

**Additional Recommendations:**
- **Use `%s` formatting:** Python's logging module supports lazy evaluation with `%s`, which is more efficient than f-strings
- **Sanitize all user input in logs:** Review all log statements for user-controlled data (titles, descriptions, usernames, etc.)
- **Truncate long values:** Consider limiting logged string lengths to prevent log flooding
- **Structured logging:** Consider JSON-formatted logs which naturally handle special characters
- **SIEM configuration:** Configure log parsers to detect anomalous log patterns

**Example of comprehensive sanitization:**
```python
def sanitize_for_log(value: str, max_length: int = 200) -> str:
    """Sanitize and truncate user input for safe log output."""
    if value is None:
        return 'None'
    
    # Remove/escape control characters
    sanitized = (value
                 .replace('\n', '\\n')
                 .replace('\r', '\\r')
                 .replace('\t', '\\t')
                 .replace('\x00', '\\x00'))
    
    # Truncate if too long
    if len(sanitized) > max_length:
        sanitized = sanitized[:max_length] + '...[truncated]'
    
    return sanitized

# Usage throughout codebase:
_LOGGER.info(
    'User[U:%s] created election[E:%s]; title: "%s", monitors: "%s"',
    result.uid,
    election.eid,
    sanitize_for_log(form.title),
    sanitize_for_log(form.monitors)
)
```

## Architecture Observations

### 1. Logging Framework Inconsistency

The application demonstrates a split in logging practices between the web server and CLI tools:

- **Web Server (`pages.py`, `main.py`):** Consistently uses Python's `logging` module with structured log messages
- **CLI Tool (`tally.py`):** Reverts to `print()` statements for critical security events

This suggests the tally tool was developed with less emphasis on operational logging, or was created before logging standards were established. The CLI context may have led developers to favor console output over structured logs, not considering production deployment scenarios.

**Recommendation:** Establish logging standards that apply to all components (web, CLI, background jobs) and enforce them through code review.

### 2. Assert-Based Security Controls

Multiple security-critical state checks use Python's `assert` statement:

```python
assert self.is_editable()  # Can be disabled with -O flag
assert self.is_open()       # Produces no log output
```

This pattern has several problems:
- **Optimization Flag:** Python's `-O` flag disables all `assert` statements, completely removing these security controls
- **No Logging:** `AssertionError` exceptions provide no security context
- **Generic Handling:** Callers catch generic exceptions, losing security-specific information

This suggests the `election.py` module may have been developed with a focus on correctness verification (where `assert` is appropriate) rather than security enforcement (where explicit checks with logging are required).

**Recommendation:** Distinguish between development-time assertions (correctness checks) and runtime security controls (state validation). Use explicit checks with logging for the latter.

### 3. Missing API Module

The `main.py` file imports an `api` module that was not provided for audit:

```python
import api  # noqa: F401
```

This module likely contains additional endpoints that may have their own security event logging gaps. The audit's findings should be considered incomplete until the API module is reviewed.

**Recommendation:** Extend this audit to cover the `api` module, applying the same ASVS 16.3.3 requirements to API endpoints.

### 4. Authentication Framework Dependency

Authentication success/failure logging depends on the `asfquart.auth` framework, which is outside the scope of this audit. The domain context explicitly requires authentication events to be logged, but this cannot be verified without reviewing the framework code.

**Recommendation:** Conduct a separate audit of the `asfquart.auth` framework to verify it logs:
- Successful authentications (with user identity and source IP)
- Failed authentication attempts
- Account lockouts
- Session creation/termination

### 5. Separation of Concerns

The application demonstrates good separation between:
- **Presentation layer** (`pages.py`) — handles HTTP requests, performs authorization
- **Business logic** (`election.py`) — implements election rules, state management
- **Data access** (database queries) — encapsulated in query objects

However, security event logging is scattered across these layers without a clear ownership model. Some security events are logged in the presentation layer, while others should be logged in the business logic layer but aren't.

**Recommendation:** Define clear logging ownership:
- **Presentation layer:** Authorization failures, input validation failures
- **Business logic:** State violations, integrity checks, privileged operations
- **Data access:** Consider logging sensitive data access patterns

### 6. CLI Tool Privilege Model

The `tally.py` CLI tool performs highly privileged operations (decrypting all votes) but has no authentication or authorization mechanism. It relies on filesystem permissions to control access. This makes audit logging even more critical, as it's the only control for accountability.

**Recommendation:** Consider adding authentication to CLI tools, or at minimum, enhance logging to capture operator identity, command-line arguments, and operational context.
