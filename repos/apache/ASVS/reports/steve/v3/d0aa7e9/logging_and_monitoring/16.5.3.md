### Key Findings

This audit identified three significant fail-open conditions that violate ASVS 16.5.3:

1. **Critical state enforcement relies on Python `assert` statements** that are disabled with optimization flags, creating catastrophic fail-open conditions for all election management operations
2. **Vote content validation is completely missing**, allowing arbitrary invalid data to be stored and corrupt election results
3. **Multi-vote submission lacks transactional integrity**, creating inconsistent state when partial failures occur

## Findings

### Critical Severity

#### ASVS-1653-SEV-001: Security-Critical State Enforcement Uses `assert` — Fail-Open Under Python Optimization

**Severity:** CRITICAL  
**Gap Type:** Type B — Control EXISTS but can be NOT CALLED  
**File:** `v3/steve/election.py`  
**Lines:** 61, 82, 128, 135, 196, 197, 215, 216, 228, 248

**Description:**

All election state machine enforcement for management operations uses Python `assert` statements. Python's `-O` and `-OO` flags strip `assert` statements entirely at compile time, silently removing these security controls. This creates a catastrophic fail-open condition where:

- Issues can be added, edited, or deleted in open/closed elections
- An already-open election can be "opened" again, **overwriting existing voter salts** and rendering all previously cast votes undecryptable
- Elections can be closed regardless of current state
- Invalid vote types can be used

**Vulnerable Code:**

```python
# election.py:61 — guards election deletion
def delete(self):
    assert self.is_editable()  # REMOVED by python -O

# election.py:80-82 — guards election opening  
def open(self, pdb):
    assert self.is_editable()  # REMOVED by python -O
    self.add_salts()           # OVERWRITES existing salts if election already open

# election.py:132-135 — guards salt generation
def add_salts(self):
    assert self.is_editable()  # REMOVED by python -O

# election.py:194-197 — guards issue creation
def add_issue(self, title, description, vtype, kv):
    assert self.is_editable()  # REMOVED by python -O
    assert vtype in vtypes.TYPES  # REMOVED by python -O

# election.py:125-128 — guards election closing
def close(self):
    assert self.is_open()  # REMOVED by python -O
```

**Data Flow:**

Web request → `do_open_endpoint` (pages.py:477) → `election.open(pdb)` (election.py:80) → `assert self.is_editable()` [SKIPPED with -O] → `add_salts()` overwrites voter salts → all existing votes become undecryptable

**Proof of Concept:**

1. Deploy application with `python -O -m hypercorn main:steve_app`
2. Open an election normally (votes are cast by voters)
3. As any committer, navigate to `GET /do-open/<eid>` for the already-open election
4. The `assert` is stripped; `open()` proceeds:
   - `add_salts()` regenerates ALL salts in the mayvote table
   - `gen_opened_key()` computes a new key with new salts
   - All previously encrypted votes are now unrecoverable

**Impact:**

- Complete corruption of an active election's vote data
- Ability to modify election structure (add/edit/delete issues) after voting has started
- Tampering with election integrity without detection (tampering detection compares keys, but re-opening regenerates them)
- Vote type validation bypass allowing arbitrary issue types

**Note:** The `add_vote()` and `tally_issue()` methods correctly use `_all_metadata(required_state)` which raises `ElectionBadState` — a proper exception that works regardless of optimization flags. This inconsistency confirms the assert usage is an oversight, not a design pattern.

**Remediation:**

Replace all security-relevant `assert` statements with explicit conditional checks that raise appropriate exceptions:

```python
def open(self, pdb):
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
    self.add_salts()
    # ...

def close(self):
    if not self.is_open():
        raise ElectionBadState(self.eid, self.get_state(), self.S_OPEN)
    self.c_close.perform(self.eid)

def add_issue(self, title, description, vtype, kv):
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
    if vtype not in vtypes.TYPES:
        raise ValueError(f'Invalid vote type: {vtype}')
    # ...

def delete(self):
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
    # ...
```

Additionally, wrap calls in `pages.py` with try/except to return user-friendly errors:

```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    try:
        pdb = steve.persondb.PersonDB.open(DB_FNAME)
        election.open(pdb)
    except steve.election.ElectionBadState as e:
        _LOGGER.warning(f'User[U:{result.uid}] attempted invalid state change: {e}')
        await flash_danger('This election cannot be opened in its current state.')
        return quart.redirect(f'/manage/{election.eid}', code=303)
    # ...
```

### High Severity

#### ASVS-1653-SEV-002: Missing Vote Content Validation — Invalid Votes Stored Without Validation

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control  
**File:** `v3/steve/election.py`, line 260  
**Called from:** `v3/server/pages.py`, line 437

**Description:**

The `add_vote()` method contains a TODO comment where vote content validation should occur but has no implementation. Any arbitrary string is accepted, encrypted, and stored as a vote regardless of the issue's vote type (`yna` or `stv`). This is a fail-open condition where the validation step is absent, and the transaction (vote storage) proceeds unconditionally.

**Vulnerable Code:**

```python
# election.py:255-271
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting   # <-- TODO: NOT IMPLEMENTED

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)  # Invalid vote stored
```

```python
# pages.py:425-440 — caller does not validate vote content either
for iid, votestring in votes.items():
    if iid not in issue_dict:
        await flash_danger(f'Invalid issue ID: {iid}')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
    try:
        election.add_vote(result.uid, iid, votestring)  # No validation of votestring
```

**Data Flow:**

POST `/do-vote/<eid>` → form field `vote-<iid>=<arbitrary_string>` → `election.add_vote(uid, iid, arbitrary_string)` → encrypted and stored → corrupts tally results

**Proof of Concept:**

```http
POST /do-vote/abc123 HTTP/1.1
Content-Type: application/x-www-form-urlencoded

vote-issue1=INVALID_GARBAGE&vote-issue2=zzz&vote-issue3=999
```

For a YNA issue, `INVALID_GARBAGE` is stored instead of `y`, `n`, or `a`. For an STV issue, `999` is stored instead of a valid ranking string. During tallying, the vtype module receives these invalid strings, potentially producing incorrect results or crashing.

**Impact:**

- Invalid votes corrupt election tallying results
- For YNA: non-standard vote strings may be counted or cause tally errors
- For STV: malformed ranking data could crash the STV algorithm or produce incorrect seat allocations
- A malicious voter could deliberately submit invalid votes to disrupt election outcomes

**Remediation:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."
    md = self._all_metadata(self.S_OPEN)

    # Validate votestring against issue type
    issue = self.q_get_issue.first_row(iid)
    if not issue:
        raise IssueNotFound(iid)
    
    m = vtypes.vtype_module(issue.type)
    if not m.validate(votestring, self.json2kv(issue.kv)):
        raise InvalidVote(iid, votestring)

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    if not mayvote:
        raise VoterNotAuthorized(pid, iid)
    
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

Each vtype module should implement a `validate(votestring, kv)` function:

```python
# In vtypes/yna.py
def validate(votestring, kv):
    return votestring in ('y', 'n', 'a')

# In vtypes/stv.py
def validate(votestring, kv):
    # Validate ranking format, candidate labels exist, etc.
    ...
```

### Medium Severity

#### ASVS-1653-SEV-003: Non-Atomic Multi-Vote Submission — Partial Failure Creates Inconsistent State

**Severity:** MEDIUM  
**Gap Type:** Type A — Missing transactional control  
**File:** `v3/server/pages.py`, lines 425-444  
**Related:** `v3/steve/election.py`, `add_vote()` auto-commits each vote

**Description:**

The `do_vote_endpoint` processes multiple votes in a sequential loop. Each call to `election.add_vote()` auto-commits to the database individually. If an error occurs on vote N, votes 1 through N-1 are already permanently stored with no rollback mechanism. This violates the ASVS 16.5.3 requirement that transactions should not be processed despite errors — the overall vote submission transaction partially completes despite failure.

**Vulnerable Code:**

```python
# pages.py:425-444
for iid, votestring in votes.items():
    if iid not in issue_dict:
        await flash_danger(f'Invalid issue ID: {iid}')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    try:
        election.add_vote(result.uid, iid, votestring)  # Auto-commits
        _LOGGER.info(
            f'User[U:{result.uid}] voted on issue[I:{iid}] in election[E:{election.eid}]'
        )
    except Exception as e:
        _LOGGER.error(
            f'Error adding vote for user[U:{result.uid}] on issue[I:{iid}]: {e}'
        )
        await flash_danger(f'Error submitting vote for issue {iid}.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
        # Votes 1..N-1 already committed — NO ROLLBACK
```

**Data Flow:**

POST `/do-vote/<eid>` with votes for issues A, B, C → Vote A committed successfully → Vote B committed successfully → Vote C fails (e.g., database error, crypto error) → redirect with error → Votes A and B remain committed → User sees "Error submitting vote" but has partially voted

**Proof of Concept:**

A voter submits a form with votes for 5 issues. If the 3rd vote fails (e.g., due to a transient database lock, corrupted salt, or crypto failure), the voter sees an error message and is redirected. However, their first 2 votes are already stored. The voter may re-submit all 5 votes, but the re-submission for issues 1 and 2 creates additional vote records (the system uses "most recent vote" semantics, which mitigates some impact but not the inconsistency).

**Impact:**

- Voter experiences inconsistent voting state — some votes recorded, others not
- User-facing error message doesn't indicate which votes succeeded
- If the error is persistent (e.g., corrupted salt for one issue), the voter may be unable to complete voting
- Audit trail shows partial vote submissions without clear indication of the incomplete batch

**Remediation:**

Wrap the entire vote submission loop in a database transaction:

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    form = edict(await quart.request.form)
    if not form:
        await flash_danger('No vote data provided.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    votes = {}
    for key, value in form.items():
        if key.startswith('vote-'):
            iid = key.split('-', 1)[1]
            votes[iid] = value

    issues = election.list_issues()
    issue_dict = {i.iid: i for i in issues}

    try:
        election.db.conn.execute('BEGIN TRANSACTION')
        for iid, votestring in votes.items():
            if iid not in issue_dict:
                raise ValueError(f'Invalid issue ID: {iid}')
            election.add_vote(result.uid, iid, votestring)
            _LOGGER.info(
                f'User[U:{result.uid}] voted on issue[I:{iid}] in election[E:{election.eid}]'
            )
        election.db.conn.execute('COMMIT')
    except Exception as e:
        election.db.conn.execute('ROLLBACK')
        _LOGGER.error(
            f'Error submitting votes for user[U:{result.uid}] in election[E:{election.eid}]: {e}'
        )
        await flash_danger('Error submitting your votes. No votes were recorded. Please try again.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    await flash_success('Votes submitted successfully!')
    return quart.redirect('/voter', code=303)
```

## Architecture Observations

1. **Inconsistent State Enforcement Strategy**: The codebase uses two different patterns for state checks — `assert` for management operations and `_all_metadata(required_state)` for voting/tallying operations. The latter is the correct approach; the former creates a systemic vulnerability.

2. **Validation Separation Concern**: Vote content validation is noted as needed (`### validate VOTESTRING`) but left unimplemented. Both the web layer (`pages.py`) and the model layer (`election.py`) skip this validation, meaning there is no defense-in-depth.

3. **Web Endpoint Error Handling Gap**: Management endpoints (`do_open_endpoint`, `do_close_endpoint`, etc.) lack try/except blocks. While current `assert` failures produce 500 errors (fail-closed), replacing asserts with proper exceptions (as recommended) would still result in unhandled exceptions without try/except wrappers.
