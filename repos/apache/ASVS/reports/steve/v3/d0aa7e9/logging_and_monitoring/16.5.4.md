**Key Findings:**
The audit reveals **critical non-compliance** with ASVS 16.5.4. No global exception handler is registered on the Quart web application, and the CLI tally tool lacks comprehensive top-level error handling. This creates both information disclosure risk and potential loss of audit-critical error details. Additionally, security-critical state validation relies on `assert` statements which can be silently removed under Python optimization flags.

## Findings

### HIGH SEVERITY

#### ASVS-1654-HIGH-001: No Global "Last Resort" Exception Handler on Web Application

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control  
**File:** `v3/server/main.py` (lines 34–47) and `v3/server/pages.py` (entire file)

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `create_app()`
- **Lines:** 34-47

**Related Functions Checked:**
- `pages.py`: All route handlers
- `api.py`: All API endpoints (imported but not registered with error handlers)

**Description:**

ASVS 16.5.4 mandates a global "last resort" error handler to ensure all unhandled exceptions are logged and do not crash the application. The audit reveals that **no such handler exists** in the Steve application. The `create_app()` function constructs the Quart application and imports route modules but never registers a `@app.errorhandler(Exception)` decorator.

**Vulnerable Code:**

```python
# main.py lines 34-47
def create_app():
    "Create the asfquart app and its endpoints."
    # ...
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    import pages  # noqa: F401
    import api  # noqa: F401

    return app  # also available as asfquart.APP now
    # ← No @app.errorhandler(Exception) registered
    # ← No @app.errorhandler(500) registered
```

**Data Flow:**

```
Any route handler 
  → Unhandled exception (e.g., AssertionError, sqlite3.OperationalError, TypeError, KeyError)
  → Quart's built-in default handler
  → Potential stack trace exposure to client
  → Error details NOT logged in application's structured format
```

**Proof of Concept:**

Multiple routes call into `election.py` which uses `assert` statements for state validation:

```python
# election.py line 69 (called from pages.py do_open_endpoint, line 495)
def open(self, pdb):
    assert self.is_editable()  # Raises AssertionError if election already opened
```

An authenticated user hitting `GET /do-open/<eid>` for an already-opened election triggers:

```
GET /do-open/abc123def456
  → election.open(pdb)
  → assert self.is_editable()  # FAILS
  → AssertionError propagates unhandled
  → Quart default 500 response (may include traceback in debug mode)
  → Error NOT logged with structured security format (user ID, action, outcome)
```

Similarly, `add_vote()` in `election.py` line 261:

```python
def add_vote(self, pid, iid, votestring):
    md = self._all_metadata(self.S_OPEN)  # Raises ElectionBadState if not open
```

`ElectionBadState` is not caught by any decorator or handler in `pages.py`.

**Affected Entry Points:**

| Route | Handler | Unhandled Exception Risk |
|-------|---------|--------------------------|
| `GET /do-open/<eid>` | `do_open_endpoint` | `AssertionError`, `ElectionBadState` |
| `GET /do-close/<eid>` | `do_close_endpoint` | `AssertionError`, `ElectionBadState` |
| `POST /do-vote/<eid>` | `do_vote_endpoint` | `ElectionBadState` (has partial catch but re-raises generic) |
| `POST /do-add-issue/<eid>` | `do_add_issue_endpoint` | `AssertionError` from `is_editable()` |
| `POST /do-edit-issue/<eid>/<iid>` | `do_edit_issue_endpoint` | `AssertionError` from `is_editable()` |
| `POST /do-delete-issue/<eid>/<iid>` | `do_delete_issue_endpoint` | `AssertionError` from `is_editable()` |
| `POST /do-set-open_at/<eid>` | `do_set_open_at_endpoint` | `sqlite3.OperationalError`, `AttributeError` |
| `POST /do-set-close_at/<eid>` | `do_set_close_at_endpoint` | `sqlite3.OperationalError`, `AttributeError` |
| `GET /voter` | `voter_page` | `sqlite3.OperationalError` |
| `GET /admin` | `admin_page` | `PersonNotFound` (only caught for lookup, not other paths) |
| `GET /docs/<iid>/<docname>` | `serve_doc` | `sqlite3.OperationalError`, filesystem errors |
| ALL routes | Any handler | `json.JSONDecodeError`, `UnicodeDecodeError`, unexpected `None` |

**Impact:**

1. **Error detail loss**: Unhandled exceptions default to Quart's handler which does not use the application's structured log format (`User[U:...] action...`). Security-relevant errors (e.g., state violation attempts, database corruption) may not appear in security logs.

2. **Information disclosure**: Without a custom handler, Quart may return detailed tracebacks to clients, exposing internal file paths, database schema, query details, and election state internals.

3. **Availability risk**: While Quart generally keeps the process alive per-request, certain exceptions in middleware or startup paths could crash the process. Additionally, `assert` statements are **silently removed** under `python -O`, converting security checks into no-ops rather than caught errors.

**Remediation:**

```python
# In main.py create_app(), after constructing the app:

def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    import pages
    import api

    # Register last-resort error handler (ASVS 16.5.4)
    @app.errorhandler(Exception)
    async def handle_unhandled_exception(error):
        """Last resort handler: log all details, return generic message."""
        import traceback
        request = quart.request

        # Log full error details for audit/debugging
        _LOGGER.error(
            'Unhandled exception: method=%s path=%s error=%s',
            request.method,
            request.path,
            str(error),
            exc_info=True,
        )

        # Return generic error to client — no internal details
        return quart.jsonify({'error': 'An internal error occurred'}), 500

    @app.errorhandler(404)
    async def handle_not_found(error):
        return quart.jsonify({'error': 'Not found'}), 404

    return app
```

Additionally, replace `assert` with proper exceptions in `election.py`:

```python
# Instead of: assert self.is_editable()
# Use:
if not self.is_editable():
    raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
```

#### ASVS-1654-HIGH-002: Assert Statements Used for Security-Critical State Enforcement

**Severity:** HIGH  
**Gap Type:** Type C — Control called but result can be silently ignored (under `-O`)  
**File:** `v3/steve/election.py` (multiple locations)

**Location:**
- **File:** `v3/steve/election.py`
- **Functions:** `open()`, `close()`, `add_salts()`, `add_issue()`, `edit_issue()`, `delete_issue()`, `delete()`, `add_voter()`
- **Lines:** 51, 69, 119, 128, 187-188, 206-207, 221, 251

**Related Functions Checked:**
- `is_editable()` — state validation method
- `is_open()` — state validation method
- `get_state()` — state retrieval method
- All route handlers in `pages.py` that invoke these methods

**Description:**

Eight security-critical methods in `election.py` use Python `assert` statements to enforce election state machine rules. This violates ASVS 16.5.4 in two ways:

1. **Under optimization mode** (`python -O`), all assertions are compiled out, completely removing security checks
2. **Under normal execution**, `AssertionError` exceptions are not caught by any global handler, resulting in unstructured error responses and incomplete logging

**Vulnerable Code Locations:**

```python
# Line 69
def open(self, pdb):
    assert self.is_editable()

# Line 119
def close(self):
    assert self.is_open()

# Line 128
def add_salts(self):
    assert self.is_editable()

# Line 187-188
def add_issue(self, title, description, vtype, kv):
    assert self.is_editable()
    assert vtype in vtypes.TYPES

# Line 206-207
def edit_issue(self, iid, title, description, vtype, kv):
    assert self.is_editable()
    assert vtype in vtypes.TYPES

# Line 221
def delete_issue(self, iid):
    assert self.is_editable()

# Line 251
def add_voter(self, pid, iid=None):
    assert self.is_editable()
```

**Data Flow:**

```
Web request 
  → route handler 
  → election.method() 
  → assert check 
  → If Python run with -O flag, assertion is completely removed by compiler
  → State check bypassed 
  → Sensitive operation proceeds without validation
```

Even without `-O`, when assertions fire:

```
assert self.is_editable() 
  → AssertionError() (no message) 
  → No global handler 
  → Quart default 500 
  → Error logged without context (no election ID, no user ID, no action attempted)
```

**Impact:**

1. **Under optimization**: All state machine enforcement disappears. An attacker could add issues to open elections, modify votes, or manipulate closed elections.

2. **Under normal execution**: `AssertionError` produces no useful error message, making debugging and audit trail reconstruction difficult. The generic error reveals that an assertion failed but not *which* assertion or *why*.

3. The errors are not caught by any global handler (per Finding HIGH-001), so they propagate as unstructured 500 errors.

**Remediation:**

Replace all security-critical `assert` statements with proper exception raising:

```python
def open(self, pdb):
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
    # ...

def add_issue(self, title, description, vtype, kv):
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
    if vtype not in vtypes.TYPES:
        raise ValueError(f'Invalid vote type: {vtype}')
    # ...
```

Apply this pattern to all 8 affected methods listed above.

### MEDIUM SEVERITY

#### ASVS-1654-MED-001: CLI Tally Tool Lacks Top-Level Exception Handler

**Severity:** MEDIUM  
**Gap Type:** Type A — No control at entry point  
**File:** `v3/server/bin/tally.py` (lines 172–185)

**Location:**
- **File:** `v3/server/bin/tally.py`
- **Function:** `__main__` block and `tally_election()`
- **Lines:** 172-185, 125-126

**Related Functions Checked:**
- `main()` — primary CLI entry point
- `tally_election()` — election processing function
- `tally_issue()` — issue tallying function (in `election.py`)

**Description:**

The CLI tally tool, which processes election results and is likely run as a scheduled job or manual administrative task, lacks any top-level exception handling. Additionally, error handling within `tally_election()` uses `print()` instead of the configured logger, bypassing structured logging.

**Vulnerable Code:**

```python
# Lines 172-185
if __name__ == '__main__':
    logging.basicConfig(level=logging.INFO)

    parser = argparse.ArgumentParser(...)
    # ... argument parsing ...
    args = parser.parse_args()

    if args.election_id and args.issue_id:
        _LOGGER.error('ISSUE_ID implies an ELECTION_ID; do not set both.')
        sys.exit(1)

    main(args.spy_on_open_elections, args.election_id, args.issue_id, args.db_path, args.output)
    # ← No try/except around main()
```

And within `tally_election()` (lines 125-126):

```python
        except Exception as e:
            print(f'Error tallying issue {issue.iid}: {e}')  # Uses print(), not _LOGGER
            raise  # Fail hard — no graceful handling
```

**Data Flow:**

```
CLI invocation 
  → main() 
  → various election/database operations 
  → Unhandled exception (e.g., database corruption, crypto error, permission denied)
  → Process crash with traceback on stderr 
  → Error NOT captured in log files
```

**Proof of Concept:**

```bash
# Database file is corrupted or locked
$ python tally.py --election-id abc123 --db-path /path/to/corrupted.db

# Traceback shows:
#   sqlite3.DatabaseError: file is not a database
# Error goes to stderr only, NOT to log file
# Process exits with non-zero code, no structured logging
```

**Impact:**

1. Tallying errors during election processing are lost if stderr is not captured by the deployment environment
2. Error details critical for audit trails (which election, what went wrong) are not recorded in structured log format
3. The `print()` call on line 125 specifically bypasses the configured `_LOGGER`, meaning tally errors won't reach any log aggregation system

**Remediation:**

```python
if __name__ == '__main__':
    logging.basicConfig(level=logging.INFO)
    
    parser = argparse.ArgumentParser(...)
    args = parser.parse_args()

    if args.election_id and args.issue_id:
        _LOGGER.error('ISSUE_ID implies an ELECTION_ID; do not set both.')
        sys.exit(1)

    try:
        main(args.spy_on_open_elections, args.election_id, args.issue_id, 
             args.db_path, args.output)
    except steve.election.ElectionNotFound as e:
        _LOGGER.error('Election not found: %s', e)
        sys.exit(2)
    except Exception:
        _LOGGER.critical('Unexpected error during tally', exc_info=True)
        sys.exit(99)
```

And fix `tally_election()`:

```python
        except Exception as e:
            _LOGGER.error('Error tallying issue[I:%s]: %s', issue.iid, e, exc_info=True)
            raise
```

## Positive Security Patterns

1. **Structured logging in route handlers**: `pages.py` consistently uses `_LOGGER.info(f'User[U:{result.uid}] action...')` format for successful operations — good structured audit trail.

2. **`_all_metadata(required_state)` pattern**: The `add_vote()` and `tally_issue()` methods properly use `_all_metadata(self.S_OPEN)` / `_all_metadata(self.S_CLOSED)` which raises `ElectionBadState` — a proper exception rather than an assert. This pattern should be extended to all state-dependent methods.

3. **Domain-specific exceptions defined**: `ElectionNotFound`, `ElectionBadState`, and `IssueNotFound` are well-defined with meaningful messages — these provide the foundation for proper error handling.

4. **Decorator-based exception handling**: `load_election()` and `load_election_issue()` properly catch domain exceptions and convert them to user-friendly 404 responses.

5. **Logging configuration present**: Both `run_standalone()` and `run_asgi()` configure `logging.basicConfig()` with structured format, providing the infrastructure for a global handler to log to.

---

## Architecture Observations

1. **Inconsistent state validation pattern**: Some methods use `assert` (8 methods), while others use `_all_metadata(required_state)` (2 methods). The latter pattern is correct and should be the standard.

2. **No middleware error handling layer**: The application has no middleware between the framework and route handlers to catch and log exceptions. This is the architectural gap that ASVS 16.5.4 specifically addresses.

3. **Debug logging in production paths**: `run_standalone()` sets `level=logging.DEBUG`, which with a global error handler could log excessive detail. However, without a handler, even basic error information may be lost.

4. **CLI and web share `election.py`**: The same `assert`-based code is used by both the web server and CLI tools, meaning the assert issue affects both deployment contexts.
