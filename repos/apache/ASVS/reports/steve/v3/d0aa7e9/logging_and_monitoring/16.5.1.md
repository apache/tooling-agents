The audit identified **2 findings** related to ASVS 16.5.1 (generic error messages on unexpected errors). The primary concern is an inconsistently applied error handling pattern: one endpoint correctly catches exceptions and returns generic messages, but five other state-changing endpoints lack any error handling, and no global error handler exists as a safety net. This creates a **Type B gap** (control exists but not consistently applied), which is the most dangerous category since it creates false confidence in the security posture.

### ASVS-1651-CRITICAL-001

**Title:** Error handling pattern exists in `do_vote_endpoint` but is NOT applied to five other state-changing endpoints

**Severity:** Critical (Type B Gap)

**Location:**
- **File:** `v3/server/pages.py`
- **Affected Endpoints:**
  - `do_open_endpoint` (line 498)
  - `do_close_endpoint` (line 520)
  - `do_add_issue_endpoint` (line 538)
  - `do_edit_issue_endpoint` (line 563)
  - `do_delete_issue_endpoint` (line 586)

**Related Functions Checked:**
- `election.open()` (`election.py:75-89`)
- `election.close()` (`election.py:122-128`)
- `election.add_issue()` (`election.py:190-207`)
- `election.edit_issue()` (`election.py:209-220`)
- `election.delete_issue()` (`election.py:222-233`)

**Description:**

A secure error handling pattern exists in `do_vote_endpoint` that catches exceptions, logs details server-side, and returns generic error messages to users. However, this pattern is not applied to five other state-changing endpoints that perform security-critical operations. These unprotected endpoints call business logic methods that use `assert` statements for state validation, which will raise unhandled `AssertionError` exceptions when violated. Additionally, these endpoints perform cryptographic operations and database transactions that can raise various exceptions.

**Control Defined At:** `pages.py:454-467`

```python

        f'User[U:{result.uid}] voted on issue[I:{iid}] in election[E:{election.eid}]'
    )
except Exception as e:
    _LOGGER.error(
        f'Error adding vote for user[U:{result.uid}] on issue[I:{iid}]: {e}'

    await flash_danger(f'Error submitting vote for issue {iid}.')
```

### ASVS-1651-HIGH-001

**Title:** No global error handler defined ã unhandled exceptions across ALL endpoints fall to framework defaults

**Severity:** High (Type A Gap)

**Location:**
- **File:** `v3/server/pages.py` (entire file) and `v3/server/main.py` (line 38-44, `create_app`)

**Related Functions Checked:**
- `create_app()` (`main.py:38-44`)
- `_set_election_date()` (`pages.py:95-117`)
- All endpoint functions in `pages.py`

**Description:**

The application does not define a global error handler to catch unhandled exceptions. This means that any exception not explicitly caught by individual endpoint handlers will be processed by the framework's default error handling mechanism. Without an explicit global handler, the application has no defense-in-depth protection against information disclosure through error messages.

**Vulnerable Code (absence of control):**

```python

    import api    # noqa: F401
    return app    # No @app.errorhandler registered anywhere
```

**Data Flow:**

```
Any endpoint ã unhandled exception (KeyError, AttributeError, DB error, etc.) ã
  ã [MISSING: @APP.errorhandler(Exception)] ã
  ã Quart/Flask default error handler ã
  ã Production: "Internal Server Error" (minimal) ã
  ã Debug: Full traceback with local variables, source code, file paths
```

**Proof of Concept:**

```bash

  -d "not json" \
  'https://target/do-set-open_at/VALID_EID'

# If debug mode: full traceback with internal file paths, function signatures
# If production: likely "Internal Server Error" but no guarantee across all frameworks
```

**Impact:**

- Without explicit error handling, the application relies entirely on framework defaults for security-sensitive error suppression
- If the application is ever deployed in debug mode (the `run_standalone()` function uses `logging.basicConfig(level=logging.DEBUG)`, suggesting development-oriented configuration), full tracebacks are exposed
- No defense-in-depth: a single missing try-except in any handler (current or future) directly exposes internals
- Crypto key material (`opened_key`, `salt`), database paths, SQL query structures, and internal module names could be exposed in traceback local variables

**Remediation:**

```python
# In main.py create_app() or pages.py ã register a global error handler

@APP.errorhandler(Exception)
async def handle_unexpected_error(error):
    """Catch-all handler ensuring generic messages for unexpected errors."""
    # Log the full error for debugging (server-side only)
    _LOGGER.error(f'Unhandled exception: {error}', exc_info=True)
    
    # Return generic message to user ã NEVER include exception details
    if isinstance(error, quart.exceptions.HTTPException):
        # Preserve intentional HTTP errors (404, 400, etc.)
        return error.get_response()
    
    # For all unexpected errors, return a generic 500
    return quart.Response(
        'An unexpected error occurred. Please try again later.',
        status=500,
        mimetype='text/plain'
    )

@APP.errorhandler(500)
async def handle_500(error):
    """Explicit 500 handler ã generic message only."""
    _LOGGER.error(f'Internal server error: {error}', exc_info=True)
    return quart.Response(
        'An unexpected error occurred. Please try again later.',
        status=500,
        mimetype='text/plain'
    )
```

## Architecture Observations

1. **Assert statements used for security-critical state validation**: The `election.py` module uses Python `assert` statements for enforcing election state machine transitions. These serve double duty as security controls (preventing operations in wrong state) and as the source of unhandled exceptions in web contexts. Additionally, `assert` statements are completely stripped when Python runs with the `-O` (optimize) flag, silently removing these security checks. These should be replaced with explicit `if` checks that raise typed exceptions.

2. **Debug `print()` statements in production code**: `do_add_issue_endpoint` (line 548) and `do_edit_issue_endpoint` (line 567) contain `print('FORM:', form)` statements that output raw form data to stdout/stderr. While not directly exposed to users, this data may appear in server logs and represents a code hygiene issue.

3. **Error handling pattern awareness demonstrated**: The `do_vote_endpoint` shows the team is aware of the need for generic error messages (catching exceptions, logging details, showing generic flash). This makes the absence of the pattern in other endpoints a more significant oversight.
