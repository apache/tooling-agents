## Findings

### Medium Severity

#### ASVS-1125-MED-001: Non-Constant-Time Comparison of Cryptographic Hash in Tamper Detection

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with no control (constant-time comparison never used)  
**File:** `v3/steve/election.py`  
**Function:** `is_tampered()` (approximately line 329-342)  
**ASVS:** 11.2.5 — Cryptographic modules fail securely, errors handled without enabling vulnerabilities

##### Vulnerable Code

```python
def is_tampered(self, pdb):
    md = self._all_metadata()  # no required state
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)

    # The computed key should be unchanged.
    return opened_key != md.opened_key  # <-- Non-constant-time comparison
```

##### Data Flow

1. **Source:** `md.opened_key` retrieved from database (stored 32-byte Argon2 hash)
2. **Computed:** `opened_key` freshly derived via `gen_opened_key(edata, salt)` → BLAKE2b → Argon2d
3. **Comparison:** Python `!=` on two `bytes` objects — CPython short-circuits on first differing byte
4. **Missing control:** `hmac.compare_digest()` for constant-time comparison

##### Related Functions Checked

- `_all_metadata()` — Retrieves stored `opened_key` from database
- `gather_election_data()` — Collects election data for hash computation
- `crypto.gen_opened_key()` — Derives fresh key from election data

##### Proof of Concept

An attacker with database write access and network-level timing observation could:
1. Modify election data (e.g., change an issue title by one character)
2. Trigger tamper check (if exposed via a web endpoint or observe CLI timing)
3. Measure response time — faster rejection means the first byte of the recomputed hash already differs
4. Iteratively modify election data to match successive bytes of `opened_key`, eventually forging tamper-undetectable modifications

While the Argon2 computation (~50ms) adds noise, the comparison timing difference (first byte mismatch ~10ns vs. full 32-byte comparison ~100ns) is detectable with statistical aggregation over repeated observations.

##### Impact

Timing side-channel in the anti-tamper mechanism. An attacker who can modify database records and observe response timing could craft election data modifications that pass tamper detection. This undermines the integrity guarantee of the `opened_key` scheme.

##### Remediation

```python
import hmac

def is_tampered(self, pdb):
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)
    
    # Use constant-time comparison to prevent timing side-channels
    return not hmac.compare_digest(opened_key, md.opened_key)
```

Consider adding `hmac.compare_digest` as a utility in `crypto.py` for reuse:
```python
import hmac

def secure_compare(a: bytes, b: bytes) -> bool:
    """Constant-time comparison of two byte strings."""
    return hmac.compare_digest(a, b)
```

**Severity:** MEDIUM  
**Gap Type:** Type A — No error handling control exists for cryptographic failures  
**File:** `v3/steve/crypto.py` (line ~75), `v3/steve/election.py` (line ~290, ~250)  
**Functions:** `decrypt_votestring()`, `tally_issue()`, `add_vote()`  
**ASVS:** 11.2.5 — Cryptographic modules fail securely

##### Vulnerable Code

**crypto.py — No exception handling:**

def decrypt_votestring(vote_token: bytes, salt: bytes, ciphertext: bytes) -> str:

**tally.py — prints error details before failing:**

    print(f'Error tallying issue {issue.iid}: {e}')
    raise  # Fail hard

##### Data Flow

1. **Source:** Ciphertext from database `vote.ciphertext` or vote_token computation
2. **Sink:** `Fernet.decrypt()` / `Fernet.encrypt()` / `argon2.low_level.hash_secret_raw()`
3. **Missing Control:** No try/except wrapping of cryptographic operations; no application-level crypto error class
4. **Propagation:** Raw `cryptography.fernet.InvalidToken`, `argon2.exceptions.*`, or `ValueError` propagate through `election.py` to the transport layer (CLI stdout or web response)

##### Related Functions Checked

- `crypto.create_vote()` — Encryption function without error handling
- `crypto._b64_vote_key()` — HKDF key derivation without error handling
- `crypto.gen_vote_token()` — Argon2 hashing without error handling
- `election.py:tally_issue()` — Decryption caller without error handling
- `election.py:add_vote()` — Encryption caller without error handling

##### Proof of Concept

**Scenario 1 — Information leakage via stack trace:**
If the web framework returns unhandled exceptions as 500 responses with tracebacks in development/misconfigured mode:
```
cryptography.fernet.InvalidToken
  File "steve/crypto.py", line 77, in decrypt_votestring
    return f.decrypt(ciphertext).decode()
  File "steve/election.py", line 292, in tally_issue
    votestring = crypto.decrypt_votestring(vote_token, mayvote.salt, row.ciphertext)
```
This confirms the encryption library, algorithm (Fernet), and internal function structure.

**Scenario 2 — Single corrupted vote prevents all tallying:**
If an attacker with database write access corrupts one `vote.ciphertext` value, the entire `tally_issue` call fails, and `tally.py` exits with code 1 after printing the error. No other issues are tallied.

##### Impact

- **Information disclosure:** Raw exception types and stack traces from cryptographic operations can confirm encryption library, algorithm choices, and internal architecture to an attacker
- **Availability:** A single corrupted ciphertext prevents tallying of an entire election — no graceful degradation or skip-and-log capability
- **Note:** Fernet's encrypt-then-MAC design inherently prevents padding oracle attacks (HMAC verified before decryption), so the classical padding oracle is not exploitable here. The finding is about the broader fail-secure principle.

##### Remediation

**1. Add a dedicated crypto error class in `crypto.py`:**
```python
class CryptoError(Exception):
    """Raised when a cryptographic operation fails.
    
    This wraps all internal crypto exceptions to prevent
    leaking implementation details to callers.
    """
    pass

def decrypt_votestring(vote_token: bytes, salt: bytes, ciphertext: bytes) -> str:
    """Decrypt CIPHERTEXT into a VOTESTRING."""
    try:
        b64key = _b64_vote_key(vote_token, salt)
        f = cryptography.fernet.Fernet(b64key)
        return f.decrypt(ciphertext).decode()
    except (cryptography.fernet.InvalidToken, Exception) as e:
        # Log internally at DEBUG level, raise sanitized error
        _LOGGER.debug('Decryption failed: %s', type(e).__name__)
        raise CryptoError('Vote decryption failed') from None

def create_vote(vote_token: bytes, salt: bytes, votestring: str) -> bytes:
    """Encrypt VOTESTRING using the VOTE_TOKEN and SALT."""
    try:
        b64key = _b64_vote_key(vote_token, salt)
        f = cryptography.fernet.Fernet(b64key)
        return f.encrypt(votestring.encode())
    except Exception as e:
        _LOGGER.debug('Encryption failed: %s', type(e).__name__)
        raise CryptoError('Vote encryption failed') from None
```

**2. Handle decryption failures gracefully in `tally_issue`:**
```python
try:
    votestring = crypto.decrypt_votestring(
        vote_token, mayvote.salt, row.ciphertext,
    )
    votes.append(votestring)
except crypto.CryptoError:
    _LOGGER.error(
        'Failed to decrypt vote for issue %s (vote_token hash: %s). '
        'Skipping corrupted vote.',
        iid, hashlib.sha256(vote_token).hexdigest()[:8]
    )
    # Continue tallying other votes; flag issue for audit
    continue
```

## Architecture Observations

### 1. HKDF Info Parameter Mismatch
The `_b64_vote_key` function uses `info=b'xchacha20_key'` in HKDF, but the actual encryption algorithm is Fernet (AES-128-CBC + HMAC-SHA256). The comment confirms this is transitional: `"### still using Fernet now, but will switch soon."` During migration to XChaCha20-Poly1305, if the same HKDF parameters are retained, old Fernet keys and new XChaCha20 keys will be derived identically. **Recommendation:** When migrating, change the `info` parameter (e.g., `b'fernet_key'` now, `b'xchacha20_key'` for new algorithm) to ensure algorithm-specific key derivation, or implement a key version indicator.

### 2. Vote Token Input Concatenation Without Domain Separation
`gen_vote_token` concatenates `opened_key + pid.encode() + iid.encode()` without delimiters or length-prefixing. While the fixed 10-character hex format of IID (enforced by schema CHECK constraint) prevents practical collisions, this deviates from cryptographic best practice. **Recommendation:** Use length-prefixed encoding or a structured serialization:
```python
data = opened_key + len(pid).to_bytes(2, 'big') + pid.encode() + iid.encode()
```

### 3. Argon2d vs. Argon2id
The production `_hash` uses `Type.D` (Argon2d) while the benchmark function uses `Type.ID` (Argon2id). Argon2d has data-dependent memory access patterns, making it theoretically vulnerable to side-channel attacks in shared-hosting environments. Argon2id (the recommended variant per RFC 9106) provides hybrid protection. For server-side key derivation, this is low risk but worth aligning with current recommendations.
