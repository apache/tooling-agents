### Summary

This audit identified a critical ASVS 11.3.2 violation: the application uses Fernet (AES-128-CBC + HMAC-SHA256) for vote encryption instead of an approved AEAD cipher mode such as AES-GCM or ChaCha20-Poly1305. Evidence indicates an incomplete migration—the key derivation function is explicitly configured for XChaCha20-Poly1305 (HKDF with `info=b'xchacha20_key'`, 32-byte key length), but the actual encryption/decryption operations were never updated from Fernet. This represents a Type B gap where the control exists but is not applied, creating false confidence that an approved cipher is in use.

## Findings

### Critical Severity

#### ASVS-1132-CRI-001 — AES-128-CBC (Fernet) Used Instead of Approved AEAD Cipher; Incomplete Migration to XChaCha20-Poly1305

**Severity:** CRITICAL  
**Gap Type:** Type B — Control EXISTS (HKDF configured for XChaCha20-Poly1305) but NOT APPLIED (Fernet/AES-128-CBC used for actual encryption)

**Location:**
- **File:** `v3/steve/crypto.py`
- **Functions:** `_b64_vote_key()` (line 63), `create_vote()` (line 77), `decrypt_votestring()` (line 84)
- **Related Functions Checked:** `election.py:add_vote()` L236, `election.py:tally_issue()` L271

**Description:**

The application uses Fernet (AES-128-CBC + HMAC-SHA256) for vote encryption, which violates ASVS 11.3.2's requirement for approved AEAD cipher modes such as AES-GCM or ChaCha20-Poly1305. Evidence of an incomplete cryptographic migration exists: the key derivation function is explicitly configured for XChaCha20-Poly1305, but the actual encryption operations still use Fernet.

**Vulnerable Code:**

Key derivation is explicitly configured for XChaCha20-Poly1305:
```python
# Line 63-75 — _b64_vote_key()
### still using Fernet now, but will switch soon. Leaving comments.
keymaker = hkdf.HKDF(
    algorithm=hashes.SHA256(),
    length=32,  # 32-byte key for XChaCha20-Poly1305
    salt=salt,
    info=b'xchacha20_key',  # <-- context label references XChaCha20
)
vote_key = keymaker.derive(vote_token)
return base64.urlsafe_b64encode(vote_key)
```

But actual encryption uses Fernet (AES-128-CBC):
```python
# Line 77-80 — create_vote()
def create_vote(vote_token: bytes, salt: bytes, votestring: str) -> bytes:
    b64key = _b64_vote_key(vote_token, salt)
    f = cryptography.fernet.Fernet(b64key)       # <-- AES-128-CBC + HMAC-SHA256
    return f.encrypt(votestring.encode())

# Line 84-88 — decrypt_votestring()
def decrypt_votestring(vote_token: bytes, salt: bytes, ciphertext: bytes) -> str:
    b64key = _b64_vote_key(vote_token, salt)
    f = cryptography.fernet.Fernet(b64key)       # <-- AES-128-CBC + HMAC-SHA256
    return f.decrypt(ciphertext).decode()
```

**Data Flow:**
```
vote_token + salt
  → HKDF-SHA256 (info=b'xchacha20_key', length=32)  ← configured for XChaCha20-Poly1305
  → base64 encode (44 chars)
  → Fernet(key)                                       ← actually uses AES-128-CBC + HMAC-SHA256
  → f.encrypt(votestring)                             ← CBC mode, not approved AEAD
  → ciphertext stored in vote.ciphertext
```

**Why This Violates ASVS 11.3.2:**

1. **Unapproved cipher mode:** Fernet uses AES-128-**CBC**, not AES-**GCM** or another approved AEAD mode. The ASVS section description states: *"Authenticated encryption algorithms built on AES and CHACHA20 form the backbone of modern cryptographic practice"* — referring to native AEAD constructions (GCM, CCM, ChaCha20-Poly1305), not composite encrypt-then-MAC schemes.

2. **AES-128 key size:** Fernet splits its 32-byte key into 16 bytes for HMAC-SHA256 signing and 16 bytes for AES-128 encryption. The HKDF derives 32 bytes intended for XChaCha20-Poly1305 (which uses the full 32 bytes), but Fernet only uses 128 bits for the encryption key.

3. **Type B gap (false confidence):** The HKDF `info` parameter (`b'xchacha20_key'`) and the `length=32` comment both reference XChaCha20-Poly1305 as the intended cipher. The control for an approved algorithm was designed and partially implemented, but the actual encryption call was never updated. A reviewer examining only `_b64_vote_key()` would conclude XChaCha20-Poly1305 is in use.

4. **CBC mode risks:** While Fernet's encrypt-then-MAC construction mitigates classic padding oracle attacks, CBC mode remains vulnerable to implementation-level side channels. AEAD modes like GCM provide authentication intrinsically within the mode, reducing the attack surface.

**Proof of Concept:**

The vulnerability is demonstrable by inspecting the cipher construction:
```python
from cryptography.fernet import Fernet
import base64

# Fernet key (32 bytes, base64-encoded)
key = Fernet.generate_key()
raw = base64.urlsafe_b64decode(key)
print(f"Total key: {len(raw)} bytes")
print(f"Signing key (HMAC): {len(raw[:16])} bytes = AES-128 NOT AES-256")
print(f"Encryption key (AES-CBC): {len(raw[16:])} bytes = 128-bit")

# Encrypted output contains CBC IV, not GCM nonce/tag
f = Fernet(key)
token = f.encrypt(b"vote:yes")
decoded = base64.urlsafe_b64decode(token)
# Fernet format: version(1) || timestamp(8) || IV(16) || ciphertext(n) || HMAC(32)
print(f"Mode: AES-128-CBC (IV present at bytes 9-24, separate HMAC at end)")
```

**Impact:**
- All vote ciphertext stored in the `vote` table uses an unapproved cipher mode
- The effective encryption strength is AES-128 (not AES-256), below modern recommendations for high-sensitivity data
- The HKDF key derivation context (`info=b'xchacha20_key'`) creates a cryptographic binding to the wrong algorithm, which could complicate future migration or cause interoperability issues
- A voting system protecting ballot secrecy should use the strongest approved primitives available

**Remediation:**

Complete the migration indicated by the code comments. Replace Fernet with XChaCha20-Poly1305 (as the HKDF is already configured for):

```python
from cryptography.hazmat.primitives.ciphers.aead import ChaCha20Poly1305
# or for XChaCha20: use a library like pynacl
import nacl.secret
import nacl.utils

def _derive_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    """Derive a 32-byte key for XChaCha20-Poly1305."""
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,
        salt=salt,
        info=b'xchacha20_key',
    )
    return keymaker.derive(vote_token)

def create_vote(vote_token: bytes, salt: bytes, votestring: str) -> bytes:
    """Encrypt VOTESTRING using XChaCha20-Poly1305."""
    key = _derive_vote_key(vote_token, salt)
    box = nacl.secret.SecretBox(key)
    return box.encrypt(votestring.encode())  # nonce auto-generated

def decrypt_votestring(vote_token: bytes, salt: bytes, ciphertext: bytes) -> str:
    """Decrypt CIPHERTEXT using XChaCha20-Poly1305."""
    key = _derive_vote_key(vote_token, salt)
    box = nacl.secret.SecretBox(key)
    return box.decrypt(ciphertext).decode()
```

Alternatively, using AES-256-GCM from the `cryptography` library:

```python
from cryptography.hazmat.primitives.ciphers.aead import AESGCM
import os

def _derive_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,  # 256-bit key for AES-256-GCM
        salt=salt,
        info=b'aesgcm_vote_key',
    )
    return keymaker.derive(vote_token)

def create_vote(vote_token: bytes, salt: bytes, votestring: str) -> bytes:
    key = _derive_vote_key(vote_token, salt)
    aesgcm = AESGCM(key)
    nonce = os.urandom(12)  # 96-bit nonce for GCM
    ct = aesgcm.encrypt(nonce, votestring.encode(), None)
    return nonce + ct  # prepend nonce for decryption

def decrypt_votestring(vote_token: bytes, salt: bytes, ciphertext: bytes) -> str:
    key = _derive_vote_key(vote_token, salt)
    aesgcm = AESGCM(key)
    nonce, ct = ciphertext[:12], ciphertext[12:]
    return aesgcm.decrypt(nonce, ct, None).decode()
```

**Note:** Migration requires a re-encryption strategy for existing vote data or a version-aware decryption path.

## Architecture Observations

1. **Incomplete cryptographic migration:** The codebase shows clear evidence of a planned migration from Fernet to XChaCha20-Poly1305 (comment on line 65, HKDF parameters). The migration was started in the key derivation layer but never completed in the encryption/decryption layer.

2. **Argon2d vs Argon2id inconsistency:** The production `_hash()` function uses `Type.D` (Argon2d), while the `benchmark_argon2()` function uses `Type.ID` (Argon2id). Argon2id is the OWASP-recommended variant combining side-channel resistance with GPU resistance. While not an encryption cipher/mode issue per se (and thus not a finding under 11.3.2 at CRITICAL), this should be addressed alongside the cipher migration.

3. **Well-structured key derivation chain:** The multi-layer derivation (election data → BLAKE2b → Argon2 → opened_key → Argon2 → vote_token → HKDF → vote_key) provides strong cryptographic isolation between elections, issues, and voters.
