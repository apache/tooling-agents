## Findings

### Medium Severity

#### ASVS-1123-MED-001: Argon2d Variant Used Instead of NIST/OWASP-Recommended Argon2id

**Severity:** MEDIUM

**Location:**
- **File:** `v3/steve/crypto.py`
- **Function:** `_hash()`
- **Line:** ~88

**Related Functions Checked:**
- `gen_opened_key()` (calls `_hash()` for election anti-tamper key)
- `gen_vote_token()` (calls `_hash()` for per-voter encryption token derivation)
- `benchmark_argon2()` (inconsistently uses Argon2id at line ~130)

**Description:**

The `_hash()` function uses Argon2d instead of the recommended Argon2id variant. Argon2d uses data-dependent memory access patterns, making it vulnerable to cache-timing side-channel attacks (CWE-208). RFC 9106 §4 states: *"If you do not know the difference between them or you consider side-channel attacks as a viable threat, pick Argon2id."*

**Vulnerable Code:**
```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2 to hash the data, with default tuning parameters."

    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,  # <-- Argon2d: vulnerable to side-channel
    )
```

**Inconsistency in same file** (`benchmark_argon2()`, line ~130):
```python
                argon2.low_level.hash_secret_raw(
                    ...
                    type=argon2.low_level.Type.ID,  # <-- Argon2id used in benchmark
                )
```

**Data Flow:**
1. `_hash()` is called by `gen_opened_key()` (election anti-tamper key) and `gen_vote_token()` (per-voter encryption token derivation)
2. `gen_vote_token()` → used in `add_vote()`, `tally_issue()`, `has_voted_upon()` — all vote-path critical operations
3. The vote token is the foundation of vote encryption key derivation (`_b64_vote_key()`)

**Impact:**

In server environments (especially cloud/containerized deployments), a co-located attacker observing cache access patterns could recover hash outputs, bypassing the computational cost of Argon2 and reducing the effective security of the vote token derivation chain. This directly impacts the `opened_key` → `vote_token` → `vote_key` derivation hierarchy that protects all encrypted votes.

The inconsistency with the benchmark function (which uses `Type.ID`) suggests the developer may have intended to use Argon2id but used Argon2d in production. This also means benchmark results do not accurately reflect production hash performance, potentially leading to under-tuned parameters.

**Remediation:**

```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2id to hash the data, with default tuning parameters."

    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.ID,  # Argon2id per RFC 9106 recommendation
    )
```

**Note:** Changing the Argon2 variant changes the hash output. This migration requires re-computing all `opened_key` values and `vote_token` values, which is only possible for elections in the `editable` state. Open or closed elections will retain Argon2d-derived keys. A versioning mechanism for the hash function variant should be considered.

#### ASVS-1123-MED-002: Non-Constant-Time Comparison of Cryptographic Integrity Token

**Severity:** MEDIUM

**Location:**
- **File:** `v3/steve/election.py`
- **Function:** `is_tampered()`
- **Line:** ~309

**Related Functions Checked:**
- `gather_election_data()` (collects data for integrity check)
- `gen_opened_key()` (computes integrity token)
- `tally.py:main()` (calls `is_tampered()` before tallying at line ~135)

**Description:**

The `is_tampered()` function uses Python's standard `!=` operator to compare cryptographic integrity tokens, which is not constant-time and can leak information through timing side-channels.

**Vulnerable Code:**
```python
def is_tampered(self, pdb):
    md = self._all_metadata()  # no required state

    # Compute an opened_key based on the current data.
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)

    # The computed key should be unchanged.
    return opened_key != md.opened_key  # <-- Non-constant-time bytes comparison
```

**Data Flow:**
1. `opened_key` (32 bytes) is computed from current election state
2. `md.opened_key` (32 bytes) is read from database (stored at election open time)
3. Python's `!=` on `bytes` objects uses `memcmp`, which short-circuits on first differing byte
4. Called in `tally.py:main()` (line ~135) before tallying proceeds

**Impact:**

The standard Python `!=` operator for bytes comparison is not constant-time — it returns as soon as it finds a differing byte. While currently only called from the CLI tally tool (limiting exploitability to actors with server access), this function is part of the `Election` class API and could be called from web handlers in future development. An attacker who can trigger repeated tamper checks with controlled modifications to election data could use timing differences to reconstruct the stored `opened_key` byte-by-byte. Knowledge of the `opened_key` combined with the per-voter salts from the `mayvote` table would allow computing `vote_token` values and decrypting individual votes, breaking voter anonymity.

**Remediation:**

```python
import hmac

def is_tampered(self, pdb):
    md = self._all_metadata()

    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)

    # Use constant-time comparison to prevent timing side-channels
    return not hmac.compare_digest(opened_key, md.opened_key)
```

## Positive Security Patterns

1. **Strong Salt Generation:** `crypto.gen_salt()` uses `secrets.token_bytes(16)` providing 128-bit cryptographically secure random salts. Schema constraints (`CHECK length(salt) = 16`) enforce this at the database level.

2. **Multi-Layer Key Derivation Hierarchy:** The chain `election_data → BLAKE2b → Argon2 → opened_key → Argon2 → vote_token → HKDF → vote_key` provides defense-in-depth. Each layer uses approved primitives with adequate key sizes.

3. **Per-Voter Salt Isolation:** Each `(pid, iid)` pair in the `mayvote` table receives a unique 16-byte salt generated with `secrets.token_bytes()`. This ensures that even identical votes by different voters produce different ciphertexts and vote tokens, preventing correlation.

4. **HKDF for Key Derivation:** `_b64_vote_key()` correctly uses HKDF-SHA256 with a per-voter salt for key stretching, following NIST SP 800-56C Rev. 2 recommendations for key derivation.

5. **Fernet Encrypt-then-MAC:** Fernet provides AES-128-CBC encryption with HMAC-SHA256 authentication, following the encrypt-then-MAC paradigm that prevents padding oracle attacks.

6. **Cryptographic Shuffle:** The `shuffle()` function implements Fisher-Yates with `secrets.randbelow()` for cryptographically secure vote ordering randomization before tallying.

7. **Schema-Enforced Cryptographic Field Sizes:** The database schema uses `CHECK` constraints to enforce exact lengths for all cryptographic fields (`salt` = 16 bytes, `opened_key` = 32 bytes, `vote_token` = 32 bytes), preventing accidental truncation or misuse.

8. **Anti-Tamper Integrity Check:** The `opened_key` mechanism hashes the entire election definition (metadata + issues + voters) at open time, detecting any post-open modifications before tallying proceeds.

9. **CSPRNG for All Random Values:** The `secrets` module is used consistently for all security-sensitive random generation: salts, IDs, shuffling. No use of `random` module found.

10. **Argon2 Memory-Hard KDF:** Using Argon2 with 64MB memory cost provides resistance against GPU/ASIC-based brute-force attacks on vote tokens and opened keys.

## Architecture Observations

1. **Planned Cipher Migration:** The comment `### still using Fernet now, but will switch soon` and the HKDF parameters (`info=b'xchacha20_key'`, `length=32`) indicate a planned migration from Fernet (AES-128-CBC) to XChaCha20-Poly1305 (256-bit key). The HKDF is already configured for the target cipher. This migration would increase encryption security from 128 bits to 256 bits.

2. **Key Derivation Chain Strength:** The weakest link in the chain is Fernet's AES-128-CBC at exactly 128 bits. All upstream derivations (Argon2: 256-bit, HKDF: 256-bit) provide headroom. The planned XChaCha20-Poly1305 migration would make the entire chain 256-bit.

3. **HKDF Info Parameter Mismatch:** `_b64_vote_key()` uses `info=b'xchacha20_key'` but currently derives keys for Fernet. While this doesn't affect security (the info parameter is domain separation, not key material), it should be updated either to match current usage or documented as forward-compatibility. During migration, if both Fernet and XChaCha20 coexist, the same `info` value for both would violate key separation principles.

4. **Vote Token Concatenation Lacks Explicit Domain Separation:** `gen_vote_token()` concatenates `opened_key + pid.encode() + iid.encode()` without length prefixes or delimiters. While the unique per-pair salt prevents practical collisions, adding explicit separation (e.g., length-prefixed encoding) would strengthen the construction against theoretical second-preimage scenarios.

5. **Argon2 Parameters Are Reasonable:** The Argon2 parameters (time_cost=2, memory_cost=64MB, parallelism=4) align with OWASP's minimum recommendations and passlib defaults. These should be re-evaluated periodically using the included `benchmark_argon2()` function, though the benchmark must first be corrected to use the same Argon2 variant as production (see Finding ASVS-1123-MED-001).

## Recommendations Summary

### Immediate Actions
1. **Replace non-constant-time comparison** (ASVS-1123-MED-002): Change `is_tampered()` to use `hmac.compare_digest()` instead of `!=` operator. This is a trivial one-line change with no data migration required and should be implemented immediately to prevent timing side-channel vulnerabilities.

### Short-term Actions
2. **Migrate to Argon2id** (ASVS-1123-MED-001): Change `_hash()` to use `argon2.low_level.Type.ID` and align `benchmark_argon2()` to match. This requires planning for key re-derivation for existing elections and implementing a versioning mechanism for hash function variants.

3. **Update HKDF info parameter**: Modify the `info` parameter in `_b64_vote_key()` to accurately reflect the current cipher in use, or document it as forward-compatibility for the planned XChaCha20-Poly1305 migration.

### Long-term Actions
4. **Complete XChaCha20-Poly1305 migration**: Finalize the planned migration from Fernet (AES-128-CBC) to XChaCha20-Poly1305 to achieve 256-bit encryption security across the entire cryptographic chain.

5. **Add explicit domain separation**: Implement length-prefix encoding in `gen_vote_token()` input concatenation to strengthen the construction against theoretical attacks.
