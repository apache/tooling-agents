### Assessment Overview

The codebase demonstrates strong vote-submission encryption (immediate encryption via Fernet in `add_vote`) and thoughtful cryptographic layering, but exhibits weaknesses in how decrypted data and cryptographic key material are handled during the tallying process, and uses a side-channel-vulnerable Argon2 variant for in-use cryptographic operations.

## Findings

### Medium Severity

#### ASVS-1172-MED-001 — Bulk Vote Decryption Retains All Plaintext in Memory Without Cleanup

**Severity:** MEDIUM  
**Gap Type:** Type A (No control for in-memory plaintext minimization)

**Location:**
- **File:** `v3/steve/election.py`
- **Function:** `tally_issue()`
- **Lines:** ~238-280
- **Also affects:** `v3/server/bin/tally.py`, function `tally_election()`, lines ~96-125

**Related Functions Checked:**
- `crypto.decrypt_votestring()` — Returns plaintext without cleanup
- `crypto.shuffle()` — Operates on plaintext list
- `vtypes.vtype_module().tally()` — Receives all plaintext votes

**Description:**

The tallying process decrypts all votes for an issue into a single in-memory list before processing. ALL plaintext votes exist simultaneously in process memory during tallying, persisting beyond the tally computation until Python's garbage collector reclaims them (with no guaranteed timing).

**Vulnerable Code:**

```python
# election.py - tally_issue()
votes = []

self.q_tally.perform(iid)
for mayvote in self.q_tally.fetchall():
    vote_token = crypto.gen_vote_token(
        md.opened_key, mayvote.pid, iid, mayvote.salt,
    )
    row = self.q_recent_vote.first_row(vote_token)
    if row is None:
        continue
    voters.add(mayvote.pid)

    # Each vote decrypted and accumulated as plaintext
    votestring = crypto.decrypt_votestring(
        vote_token, mayvote.salt, row.ciphertext,
    )
    votes.append(votestring)  # All plaintext votes held simultaneously

# Plaintext votes persist through shuffle and tally computation
crypto.shuffle(votes)
m = vtypes.vtype_module(issue.type)
return m.tally(votes, self.json2kv(issue.kv)), voters
```

**Data Flow:**

```
Database (ciphertext) → decrypt_votestring() → plaintext string → votes[] list
→ crypto.shuffle(votes) → m.tally(votes, ...) → results dict → stdout
                                                ↓
                              votes[] still in memory (no cleanup)
```

**Impact:**

- ALL plaintext votes for an issue exist simultaneously in process memory during tallying
- Plaintext votes persist beyond the tally computation until Python's garbage collector reclaims them, with no guaranteed timing
- A process memory dump (via crash, core dump, swap to disk, or memory forensics on the host) would expose every voter's individual ballot content for the current issue
- Python's string interning means plaintext vote strings may persist longer than the `votes` list scope
- For an election with 1000 voters, all 1000 plaintext votes are simultaneously resident

**Proof of Concept:**

An attacker with read access to the server's swap file or core dumps during tallying could recover all individual vote content. For an election with 1000 voters, all 1000 plaintext votes are simultaneously resident.

**Remediation:**

Implement streaming/incremental tallying where each vote is decrypted, contributed to the tally accumulator, and discarded immediately. Modify vtype modules to support incremental input:

```python
def tally_issue(self, iid):
    md = self._all_metadata(self.S_CLOSED)
    issue = self.q_get_issue.first_row(iid)
    voters = set()
    
    m = vtypes.vtype_module(issue.type)
    accumulator = m.create_accumulator(self.json2kv(issue.kv))

    # Collect and decrypt votes, using temporary list for shuffle
    vote_data = []
    self.q_tally.perform(iid)
    for mayvote in self.q_tally.fetchall():
        vote_token = crypto.gen_vote_token(
            md.opened_key, mayvote.pid, iid, mayvote.salt,
        )
        row = self.q_recent_vote.first_row(vote_token)
        if row is None:
            continue
        voters.add(mayvote.pid)
        vote_data.append((vote_token, mayvote.salt, row.ciphertext))

    # Shuffle encrypted references, then decrypt one-at-a-time
    crypto.shuffle(vote_data)
    for vt, salt, ct in vote_data:
        votestring = crypto.decrypt_votestring(vt, salt, ct)
        accumulator.add(votestring)
        del votestring  # Explicit hint to GC
    
    return accumulator.result(), voters
```

#### ASVS-1172-MED-002 — Cryptographic Key Material Not Cleared From Memory After Use

**Severity:** MEDIUM  
**Gap Type:** Type A (No control for key material cleanup)

**Location:**
- **File:** `v3/steve/crypto.py`
- **Functions:** `create_vote()` (line ~65), `decrypt_votestring()` (line ~73), `_b64_vote_key()` (line ~51)
- **Also affects:** `v3/steve/election.py`, functions `add_vote()` (line ~224), `tally_issue()` (line ~238)

**Related Functions Checked:**
- `crypto.gen_vote_token()` — Returns key material without cleanup
- `crypto._b64_vote_key()` — Derives keys without cleanup
- `election._all_metadata()` — Returns `opened_key` without cleanup

**Description:**

Per-voter encryption keys, the election opened_key, and derived key material remain in process memory beyond their operational need. During tallying, key material for every voter/issue pair accumulates across loop iterations. No cleanup mechanism exists for sensitive cryptographic material after use.

**Vulnerable Code:**

```python
# crypto.py - create_vote()
def create_vote(vote_token: bytes, salt: bytes, votestring: str) -> bytes:
    b64key = _b64_vote_key(vote_token, salt)  # Derived key persists
    f = cryptography.fernet.Fernet(b64key)     # Key stored in Fernet._signing_key and ._encryption_key
    return f.encrypt(votestring.encode())       # b64key and f never zeroed

# crypto.py - decrypt_votestring()
def decrypt_votestring(vote_token: bytes, salt: bytes, ciphertext: bytes) -> str:
    b64key = _b64_vote_key(vote_token, salt)  # Same issue
    f = cryptography.fernet.Fernet(b64key)
    return f.decrypt(ciphertext).decode()      # Key material abandoned

# crypto.py - _b64_vote_key()
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,
        salt=salt,
        info=b'xchacha20_key',
    )
    vote_key = keymaker.derive(vote_token)  # 32-byte key never zeroed
    return base64.urlsafe_b64encode(vote_key)  # vote_key abandoned

# election.py - add_vote()
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    # md.opened_key (election master key) persists in local scope
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    # vote_token (32 bytes) persists in local scope
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
    # md, vote_token, votestring all persist until GC
```

**Data Flow:**

```
opened_key (from DB) → gen_vote_token() → vote_token (32 bytes, in memory)
vote_token → _b64_vote_key() → vote_key (32 bytes, in memory)
                              → b64key (44 bytes, in memory)
                              → Fernet instance (internal key copies)
All ↑ remain in memory with no guaranteed cleanup timing
```

**Impact:**

- Per-voter encryption keys, the election opened_key, and derived key material remain in process memory beyond their operational need
- During tallying, key material for every voter/issue pair accumulates across loop iterations
- Memory disclosure vulnerabilities (e.g., via `/proc/<pid>/mem`, heap inspection, or swap) would expose these keys, allowing decryption of any intercepted ciphertexts
- The `opened_key` (election master key) remaining in the `md` variable is particularly critical as it enables derivation of all vote tokens

**Proof of Concept:**

An attacker with memory read access (through swap file, core dump, or `/proc` access) can recover key material and decrypt intercepted vote ciphertexts.

**Remediation:**

While Python doesn't natively support secure memory erasure for immutable types, use `bytearray` for mutable key storage and explicit zeroing. Consider using `ctypes.memset` for critical material:

```python
import ctypes

def _secure_zero(buf):
    """Best-effort zeroing of a bytearray."""
    if isinstance(buf, bytearray):
        ctypes.memset((ctypes.c_char * len(buf)).from_buffer(buf), 0, len(buf))

def create_vote(vote_token: bytes, salt: bytes, votestring: str) -> bytes:
    b64key = bytearray(_b64_vote_key(vote_token, salt))
    try:
        f = cryptography.fernet.Fernet(bytes(b64key))
        return f.encrypt(votestring.encode())
    finally:
        _secure_zero(b64key)

def decrypt_votestring(vote_token: bytes, salt: bytes, ciphertext: bytes) -> str:
    b64key = bytearray(_b64_vote_key(vote_token, salt))
    try:
        f = cryptography.fernet.Fernet(bytes(b64key))
        return f.decrypt(ciphertext).decode()
    finally:
        _secure_zero(b64key)
```

#### ASVS-1172-MED-003 — Argon2d Variant Used Instead of Side-Channel-Resistant Argon2id

**Severity:** MEDIUM  
**Gap Type:** Type B (Awareness exists in benchmark code but production uses weaker variant)

**Location:**
- **File:** `v3/steve/crypto.py`
- **Function:** `_hash()`
- **Line:** ~81

**Related Functions Checked:**
- `crypto.gen_vote_token()` — Calls `_hash()` for every vote operation
- `election.add_vote()` — Triggers `_hash()` during vote submission
- `election.tally_issue()` — Triggers `_hash()` for every voter during tallying
- `election.has_voted_upon()` — Triggers `_hash()` for vote status checks

**Description:**

The production code uses Argon2d (Type.D) which employs data-dependent memory access patterns, making it vulnerable to cache-timing side-channel attacks. RFC 9106 recommends Argon2id for general-purpose applications. Notably, the benchmark code in the same file already uses the recommended Type.ID variant, indicating awareness of the distinction.

**Vulnerable Code:**

```python
# Production code uses Type.D (data-dependent memory access)
def _hash(data: bytes, salt: bytes) -> bytes:
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,  # ← Argon2d: side-channel vulnerable
    )
```

**Contrast with benchmark (same file, line ~117):**

```python
# Benchmark already uses the recommended Type.ID
def benchmark_argon2():
    ...
    argon2.low_level.hash_secret_raw(
        ...
        type=argon2.low_level.Type.ID,  # ← Argon2id: side-channel resistant
    )
```

**Data Flow:**

```
opened_key + pid + iid → _hash(Type.D) → vote_token
                         ↑ data-dependent memory access pattern
                         ↑ observable via cache-timing side channels

This function is called for every voter during:
- Vote submission (add_vote → gen_vote_token → _hash)
- Tallying (tally_issue → gen_vote_token → _hash)
- Vote status check (has_voted_upon → gen_vote_token → _hash)
```

**Impact:**

- Argon2d uses data-dependent memory access patterns, meaning the memory addresses accessed during hashing depend on the secret input
- A co-located attacker (e.g., on the same cloud VM host) can observe cache-timing patterns via Flush+Reload or Prime+Probe attacks to recover information about the `opened_key` or the `pid`/`iid` input
- The `opened_key` is the election master key — if recovered via side-channel during vote submission processing, all vote tokens can be computed, enabling vote correlation or decryption of captured ciphertexts
- RFC 9106 (Argon2) recommends Argon2id for password hashing and key derivation in general-purpose applications

**Proof of Concept:**

On a shared-hosting environment, an attacker VM on the same physical host monitors L3 cache patterns during `add_vote` processing. The data-dependent memory accesses in Argon2d allow recovery of partial information about `opened_key + pid + iid`. Over multiple vote submissions (one per voter), enough information accumulates to reconstruct the `opened_key`.

**Remediation:**

Change `Type.D` to `Type.ID` in the production `_hash` function:

```python
def _hash(data: bytes, salt: bytes) -> bytes:
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.ID,  # Argon2id: recommended by RFC 9106
    )
```

**Note:** This is a breaking change — existing `opened_key` and `vote_token` values will not match after switching. Migration must be coordinated with re-opening elections or a version flag.

## Positive Security Patterns

1. **Immediate vote encryption on submission:** `add_vote()` encrypts the votestring via `crypto.create_vote()` immediately upon receipt, minimizing the plaintext window during the critical vote-casting path.

2. **Metadata field stripping in public API:** `get_metadata()` explicitly excludes `salt` and `opened_key` from its return value, with a clear comment documenting the intent. This is proper data minimization for the public interface.

3. **Vote table contains no PID/IID:** The `vote` table schema stores only `vote_token` (hashed) and `ciphertext` — no direct person or issue identifiers. This is structural data minimization at the database level.

4. **Cryptographically secure shuffle before result output:** `crypto.shuffle()` uses `secrets.randbelow()` (Fisher-Yates) to prevent database-order leakage of votes during tallying.

5. **CSPRNG for all random operations:** All random generation uses the `secrets` module (`token_bytes`, `token_hex`, `randbelow`), ensuring unpredictable salts, IDs, and shuffle operations.

6. **Salt uniqueness per voter/issue pair:** Each `mayvote` row receives an independent 16-byte salt, ensuring cryptographic separation between voters even within the same election.

7. **Multi-layer key derivation:** The hierarchy `election_data → BLAKE2b → Argon2 → opened_key → Argon2 → vote_token → HKDF → encryption_key` provides strong cryptographic separation with each layer adding entropy and computational cost.

---

## Architecture Observations

1. **Asymmetric protection profile:** The vote submission path (`add_vote`) is well-protected with immediate encryption, while the tallying path (`tally_issue`) accumulates all plaintext votes in memory. This asymmetry reflects different threat models: submission faces online attackers while tallying is an offline privileged operation.

2. **Python's memory management limitations:** Python's immutable `bytes` and `str` types and non-deterministic garbage collection make secure memory clearing fundamentally difficult. The findings about key material cleanup acknowledge this platform limitation. For ASVS Level 3 compliance, consider whether a compiled-language crypto module (or `ctypes`-based wrappers) is warranted for the most sensitive operations.

3. **HKDF info parameter mismatch:** `_b64_vote_key()` uses `info=b'xchacha20_key'` with a comment noting planned migration from Fernet. The info string provides domain separation but currently misnames the actual algorithm. This doesn't affect security but complicates future algorithm migration auditing.

4. **Tamper detection integrity:** The `is_tampered()` check uses Argon2 hashing of election data, verified before tallying in `tally.py:main()`. This check uses non-constant-time comparison (`!=`), but since it compares integrity hashes (not secrets), the practical risk is negligible.
