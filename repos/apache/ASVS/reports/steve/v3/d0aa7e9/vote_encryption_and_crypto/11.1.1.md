### Key Findings

The audit reveals a **systematic absence of cryptographic key lifecycle management** across the election system. While cryptographic primitives are generally well-chosen and keys are generated using secure randomness, there is no evidence of a documented key management policy, no mechanisms for key rotation/expiration/revocation/destruction, and several implementation inconsistencies that indicate the lack of a governing standard.

## Findings

### High Severity

#### ASVS-1111-HIGH-001: No Cryptographic Key Lifecycle Management Implementation

**Severity:** HIGH  
**Gap Type:** Type A — Missing control entirely  
**ASVS:** 11.1.1  
**Location:** `v3/steve/crypto.py` (all key generation), `v3/steve/election.py` (key storage/usage), `v3/schema.sql` (key persistence)

**Description:**

NIST SP 800-57 defines a key lifecycle with states: Pre-Activation → Active → Deactivated → Compromised → Destroyed. The codebase implements only the first two states (Pre-Activation as NULL columns, Active when populated) and has no mechanism for any subsequent state.

**Evidence of Missing Lifecycle Stages:**

1. **No Key Rotation:** Once `election.salt`, `opened_key`, and per-voter `mayvote.salt` values are generated, they are permanent. No function exists to rotate any key material.

2. **No Key Expiration:** Schema has no TTL or expiration field on any cryptographic material:
```sql
-- schema.sql: election table
salt  BLOB  CHECK (salt IS NULL OR length(salt) = 16),
opened_key  BLOB  CHECK (opened_key IS NULL OR length(opened_key) = 32),
-- No expiration, no created_at timestamp for key material
```

3. **No Key Destruction:** After election closure, all key material (election salt, opened_key, per-voter salts, vote tokens) remains in the database indefinitely. No `ZEROIZE` or `DELETE` operation exists:
```python
# election.py: close() method - only sets a flag, keys persist
def close(self):
    assert self.is_open()
    self.c_close.perform(self.eid)
    # No key material cleanup
```

4. **No Key Revocation/Compromise Handling:** No mechanism to mark keys as compromised or to re-encrypt data under new keys. If `opened_key` is leaked, there is no recovery path.

5. **No Documented Policy:** No configuration file, code comment, or documentation file establishes which cryptographic keys exist, their intended lifecycle, authorized accessors, or destruction timelines.

**Key Material Inventory Without Lifecycle:**

| Key Material | Generated In | Stored In | Lifecycle End | Gap |
|---|---|---|---|---|
| `election.salt` (16B) | `crypto.gen_salt()` → `election.open()` | `election` table | Never | No expiration/destruction |
| `election.opened_key` (32B) | `crypto.gen_opened_key()` → `election.open()` | `election` table | Never | No expiration/destruction |
| `mayvote.salt` (16B per voter-issue pair) | `crypto.gen_salt()` → `election.add_salts()` | `mayvote` table | Never | No expiration/destruction |
| Derived `vote_token` (32B) | `crypto.gen_vote_token()` | `vote` table | Never | No expiration/destruction |
| Derived vote encryption key (32B) | `crypto._b64_vote_key()` | Memory only | Process exit | Not explicitly zeroized |

**Related Functions Checked:**
- `election.open()` (line 71–82)
- `election.close()` (line 111–116)
- `election.add_salts()` (line 119–135)
- `crypto.gen_salt()` (line 27–29)
- `crypto.gen_opened_key()` (line 32–42)
- `crypto.gen_vote_token()` (line 44–48)

**Data Flow:**
1. Election created → `open()` called → `gen_salt()` + `gen_opened_key()` generate key material
2. Key material stored in database with no expiration metadata
3. Election proceeds through states: EDITABLE → OPEN → CLOSED
4. `close()` method sets status flag but leaves all cryptographic material intact
5. Keys persist indefinitely in database with no destruction mechanism

**Proof of Concept:**

```python
# Demonstrate key material persistence after election closure
import sqlite3
from steve.election import Election

# Create and close an election
db = sqlite3.connect('test.db')
election = Election.create(db, 'Test Election', 'Test Description')
election.open()
election.close()

# Keys still present in database after closure
cursor = db.execute('SELECT salt, opened_key FROM election WHERE eid = ?', (election.eid,))
row = cursor.fetchone()
assert row[0] is not None  # salt still present
assert row[1] is not None  # opened_key still present
# No mechanism exists to destroy these keys
```

**Impact:**
- Cryptographic key material accumulates indefinitely, increasing the window for extraction
- A compromised database backup from any point in time contains all key material needed to decrypt all votes from all elections
- No mechanism exists to limit the blast radius of a key compromise
- Non-compliance with NIST SP 800-57 Part 1, Sections 8.2–8.5
- Violates the core requirement of ASVS 11.1.1 for documented key lifecycle management

**Remediation:**

```python
# election.py - Add key destruction after tally completion
def archive_and_destroy_keys(self):
    """Destroy election key material after tallying is complete and verified."""
    assert self.is_closed()
    
    # 1. Verify tally has been exported/signed
    # 2. Destroy election-level keys
    self.c_destroy_election_keys.perform(self.eid)  # SET salt=NULL, opened_key=NULL
    
    # 3. Destroy per-voter salts
    self.c_destroy_voter_salts.perform(self.eid)    # SET mayvote.salt=NULL
    
    _LOGGER.info(f'Key material destroyed for election[E:{self.eid}]')

# schema.sql - Add key creation timestamp
# ALTER TABLE election ADD COLUMN keys_created_at INTEGER;
# ALTER TABLE election ADD COLUMN keys_destroyed_at INTEGER;
```

Additionally, create a `CRYPTO_KEY_MANAGEMENT.md` document specifying:
- Complete inventory of all cryptographic keys and their purposes
- Authorized entities for each key type
- Maximum key lifetime per election state
- Key destruction procedures post-tallying
- Compromise response procedures

### Medium Severity

#### ASVS-1111-MED-002: Production Argon2 Uses Type.D; Benchmark Uses Type.ID

**Severity:** MEDIUM  
**Gap Type:** Type B — Control exists (benchmark uses ID) but not applied (production uses D)  
**Location:** `v3/steve/crypto.py`, `_hash()` at ~line 80, `benchmark_argon2()` at ~line 116

**Description:**

The production key derivation function `_hash()` uses Argon2**d** while the benchmark function uses Argon2**id**. Argon2d is vulnerable to side-channel attacks (timing/cache attacks) and is not recommended by OWASP or RFC 9106 for server-side key derivation. Argon2id is the recommended variant.

**Vulnerable Code:**

```python
# crypto.py ~line 80 - PRODUCTION code
def _hash(data: bytes, salt: bytes) -> bytes:
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,  # <-- Argon2d: side-channel vulnerable
    )
```

```python
# crypto.py ~line 116 - BENCHMARK code
def benchmark_argon2():
    ...
    argon2.low_level.hash_secret_raw(
        ...
        type=argon2.low_level.Type.ID,  # <-- Argon2id: recommended variant
    )
```

**Related Functions Checked:**
- `_hash()` (line 80–92)
- `gen_opened_key()` (line 32–42) — calls `_hash()`
- `gen_vote_token()` (line 44–48) — calls `_hash()`
- `benchmark_argon2()` (line 116–146)

**Data Flow:**
- `_hash()` is called by `gen_opened_key()` and `gen_vote_token()`
- `gen_opened_key()` → produces election integrity key
- `gen_vote_token()` → produces vote encryption token
- All election key material is derived through the Argon2d path

**Proof of Concept:**

```python
# Demonstrate timing difference in Argon2d vs Argon2id
import time
import argon2.low_level

salt = b'0' * 16
secret1 = b'A' * 32  # All same bytes
secret2 = b'ABCD' * 8  # Variable bytes

# Argon2d shows timing variance based on data
times_d = []
for secret in [secret1, secret2]:
    start = time.perf_counter()
    argon2.low_level.hash_secret_raw(secret, salt, 2, 65536, 4, 32, type=argon2.low_level.Type.D)
    times_d.append(time.perf_counter() - start)

# In shared hosting, this timing difference can leak information
print(f"Timing variance: {abs(times_d[0] - times_d[1])} seconds")
```

**Impact:**
- In shared hosting or cloud environments, an attacker with co-tenant access could use cache timing attacks to extract data-dependent memory access patterns from Argon2d
- The inconsistency between benchmark (Type.ID) and production (Type.D) suggests this was unintentional
- Violates NIST SP 800-57 guidance on algorithm selection for key management

**Remediation:**

```python
def _hash(data: bytes, salt: bytes) -> bytes:
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.ID,  # Argon2id per RFC 9106 / OWASP
    )
```

**Note:** Changing the Argon2 type will change all derived key values. This must be treated as a key rotation event — it cannot be applied to elections that have already been opened.

#### ASVS-1111-MED-003: Non-Constant-Time Comparison of Cryptographic Key Material

**Severity:** MEDIUM  
**Gap Type:** Type C — Control called (comparison) but result mechanism is insecure  
**Location:** `v3/steve/election.py`, `is_tampered()` at ~line 344

**Description:**

The tamper detection mechanism compares a recomputed `opened_key` against the stored value using Python's standard `!=` operator, which short-circuits on the first differing byte. This leaks information about the stored key through timing differences.

**Vulnerable Code:**

```python
# election.py ~line 335
def is_tampered(self, pdb):
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)
    
    # Non-constant-time comparison of 32-byte cryptographic values
    return opened_key != md.opened_key  # <-- timing oracle
```

**Related Functions Checked:**
- `is_tampered()` (line 335–349)
- `_all_metadata()` (line 58–69)
- `gather_election_data()` (line 312–333)
- `crypto.gen_opened_key()` (line 32–42)

**Data Flow:**
- `md.opened_key` → stored 32-byte key from database
- `opened_key` → freshly computed 32-byte key
- `!=` → Python's byte comparison, short-circuits on first difference
- Called from `tally.py` main() before tallying
- Could be called from web handlers if tamper checking is added to the web interface

**Proof of Concept:**

```python
# Demonstrate timing oracle in non-constant-time comparison
import time

stored_key = b'A' * 32
test_key_1 = b'B' + b'A' * 31  # Differs in first byte
test_key_2 = b'A' * 31 + b'B'  # Differs in last byte

# First byte difference returns faster
start = time.perf_counter()
result1 = test_key_1 != stored_key
time1 = time.perf_counter() - start

# Last byte difference takes longer
start = time.perf_counter()
result2 = test_key_2 != stored_key
time2 = time.perf_counter() - start

# Timing difference reveals position of first differing byte
print(f"First byte diff: {time1}, Last byte diff: {time2}")
```

**Impact:**
- An attacker who can trigger tamper checks with controlled election data modifications and observe response timing could gradually reconstruct the `opened_key` value
- While currently CLI-only, the method is a public API on the `Election` class that web handlers could invoke
- Per NIST SP 800-57, key material should be protected with constant-time operations during comparison

**Remediation:**

```python
import hmac

def is_tampered(self, pdb):
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)
    
    # Constant-time comparison prevents timing oracle
    return not hmac.compare_digest(opened_key, md.opened_key)
```

#### ASVS-1111-MED-004: HKDF Domain Separation Label Misidentifies Encryption Algorithm

**Severity:** MEDIUM  
**Gap Type:** Type B — Domain separation control exists but is incorrectly applied  
**Location:** `v3/steve/crypto.py`, `_b64_vote_key()` at ~line 51

**Description:**

The HKDF `info` parameter, which provides cryptographic domain separation, identifies the derived key as `xchacha20_key` while the actual encryption uses Fernet (AES-128-CBC + HMAC-SHA256). This violates the principle of accurate domain separation in key derivation and creates a latent key reuse vulnerability.

**Vulnerable Code:**

```python
# crypto.py ~line 51
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    ### still using Fernet now, but will switch soon. Leaving comments.
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,  # 32-byte key for XChaCha20-Poly1305  <-- comment wrong
        salt=salt,
        info=b'xchacha20_key',  # <-- labels key for wrong algorithm
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

```python
# crypto.py ~line 64 - Actual usage is Fernet, not XChaCha20
def create_vote(vote_token: bytes, salt: bytes, votestring: str) -> bytes:
    b64key = _b64_vote_key(vote_token, salt)
    f = cryptography.fernet.Fernet(b64key)  # <-- AES-128-CBC, not XChaCha20
    return f.encrypt(votestring.encode())
```

**Related Functions Checked:**
- `_b64_vote_key()` (line 51–62)
- `create_vote()` (line 64–69)
- `decrypt_votestring()` (line 72–78)

**Data Flow:**
1. `vote_token` (32 bytes) derived from Argon2
2. `_b64_vote_key()` applies HKDF with `info=b'xchacha20_key'`
3. Resulting key used with `Fernet` (AES-128-CBC + HMAC-SHA256), not XChaCha20
4. Mismatch between label and actual algorithm use

**Proof of Concept:**

```python
# Demonstrate potential key reuse if XChaCha20 is added alongside Fernet
from cryptography.hazmat.primitives import hashes
from cryptography.hazmat.primitives.kdf.hkdf import HKDF
import cryptography.fernet

vote_token = b'0' * 32
salt = b'1' * 16

# Current implementation (Fernet with wrong label)
keymaker = HKDF(algorithm=hashes.SHA256(), length=32, salt=salt, info=b'xchacha20_key')
key_current = keymaker.derive(vote_token)

# If XChaCha20 is added with same label, same key material is used for both algorithms
# This violates key separation principles
```

**Impact:**
- **Domain separation failure:** If `XChaCha20-Poly1305` is later added alongside Fernet (as the comment suggests), both would derive keys with `info=b'xchacha20_key'`, meaning the same key material feeds two different algorithms — a key reuse violation per NIST SP 800-57 §5.2
- **Audit confusion:** The mismatch between code labels and actual behavior makes cryptographic inventory inaccurate, directly contradicting ASVS 11.1.1's requirement for accurate key documentation
- **Indicates no governing key management policy** that would catch algorithm/label mismatches during code review

**Remediation:**

```python
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    """Key-stretch the vote_token for Fernet encryption."""
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,  # 32-byte key: 16B signing + 16B encryption for Fernet
        salt=salt,
        info=b'fernet_vote_key_v1',  # Accurate domain separation
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

**Note:** Changing the `info` parameter changes all derived keys. This requires coordinated migration (similar to Finding 002).

## Positive Security Patterns

1. **CSPRNG Throughout:** All randomness uses Python's `secrets` module — salt generation, ID creation, vote shuffling. No use of `random` module.

2. **Per-Voter Salt Separation:** Each (person, issue) pair gets a unique 16-byte salt, preventing vote correlation even with database access.

3. **Multi-Layer Key Derivation:** The chain `election_data → BLAKE2b → Argon2 → opened_key → Argon2(+pid+iid+salt) → vote_token → HKDF → encryption_key` provides strong cryptographic separation between layers.

4. **State Machine Enforcement:** Cryptographic operations are gated by election state (`assert self.is_editable()`, `_all_metadata(self.S_OPEN)`, etc.), preventing key generation or vote encryption in wrong states.

5. **Schema-Level BLOB Length Checks:** SQLite CHECK constraints enforce exact byte lengths for all cryptographic material, preventing truncation or oversized values.

6. **Vote Shuffling Before Tallying:** `crypto.shuffle()` uses Fisher-Yates with `secrets.randbelow()` to remove database ordering from decrypted votes.

7. **Metadata Filtering:** `get_metadata()` explicitly excludes `salt` and `opened_key` from returned data, limiting key exposure through the API.

8. **Transactional Salt Assignment:** `add_salts()` uses explicit `BEGIN TRANSACTION` / `COMMIT` to ensure atomicity of per-voter salt generation.

---

## Architecture Observations

1. **Key Hierarchy is Sound in Design:** The derivation chain (election data → opened_key → vote_token → encryption key) follows good key management principles for hierarchical key derivation, though the hierarchy is not documented as a policy.

2. **Cryptographic Inventory Gap:** The domain context mentions "config.yaml and the asfquart secrets system" for key management, but these are not visible in the provided code. The inventory of where key material flows between systems is undocumented.

3. **Offline Tallying Reduces Attack Surface:** Vote decryption only occurs through the CLI tool (`tally.py`), which requires server access and explicitly checks for tampering before proceeding.

4. **No Forward Secrecy:** Compromise of the `opened_key` (from any database backup) combined with per-voter salts enables decryption of all votes for that election in perpetuity. Key destruction after tallying would mitigate this.

5. **Algorithm Migration Path Unclear:** The comment "still using Fernet now, but will switch soon" in `_b64_vote_key()` indicates a planned cryptographic migration with no documented migration strategy, versioning scheme, or backwards compatibility plan.
