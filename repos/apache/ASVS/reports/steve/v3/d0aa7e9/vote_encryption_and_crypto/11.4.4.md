### Summary

The cryptographic architecture implements a layered key derivation chain: Argon2 for key stretching → HKDF-SHA256 for key derivation → Fernet for authenticated encryption. The overall design is sound, with Argon2 providing the key stretching and HKDF appropriately used for deriving keys from the high-entropy Argon2 output. Two medium-severity findings relate to (1) the Argon2 variant selection where production code diverges from both the project's own benchmark and industry recommendations, and (2) the lack of documented application-specific parameter tuning.

## Findings

### Medium Severity

#### ASVS-1144-MED-001: Production Code Uses Argon2d Instead of Recommended Argon2id

**Location:**
- **File:** `v3/steve/crypto.py`
- **Function:** `_hash()` (line ~88)
- **Related Functions:** `gen_opened_key()`, `gen_vote_token()`

**Gap Type:** Type B — Correct variant known but not applied in production

**Vulnerable Code:**

```python
# crypto.py, _hash() function, line ~88
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2 to hash the data, with default tuning parameters."

    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,  # <-- Argon2d: side-channel vulnerable
    )
```

**Inconsistency Evidence — Benchmark uses correct variant:**

```python
# crypto.py, benchmark_argon2() function, line ~127
                argon2.low_level.hash_secret_raw(
                    ...
                    type=argon2.low_level.Type.ID,  # <-- Argon2id: recommended variant
                )
```

**Data Flow:**

The `_hash()` function is the sole Argon2 entry point, called by:
1. `gen_opened_key(edata, salt)` → produces the tamper-detection key stored in `election.opened_key`
2. `gen_vote_token(opened_key, pid, iid, salt)` → produces per-voter tokens for vote encryption/decryption

Both paths protect election integrity and ballot secrecy — the system's two most critical security properties.

**Description:**

The production code uses **Argon2d** (`Type.D`) while the project's own benchmark function uses **Argon2id** (`Type.ID`), which is the recommended variant per RFC 9106, OWASP Password Storage Cheat Sheet, and NIST guidance.

**Argon2d** uses data-dependent memory access patterns, making it vulnerable to side-channel attacks (cache-timing, speculative execution). While it provides the strongest resistance against GPU-based brute-force attacks, it is NOT the recommended general-purpose variant.

**Argon2id** is a hybrid that provides Argon2i's side-channel resistance in the first pass and Argon2d's GPU resistance in subsequent passes. It is recommended by:
- [RFC 9106, Section 4](https://www.rfc-editor.org/rfc/rfc9106#section-4): *"If you do not know the difference between them or you consider side-channel attacks as a viable threat, pick Argon2id."*
- [OWASP Password Storage Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html): Lists Argon2id as the first recommended algorithm.
- NIST SP 800-63B references Argon2 with a preference for the `id` variant.

**Impact:**

In a shared hosting or multi-tenant environment, an attacker with co-residency on the same physical server could exploit data-dependent memory access patterns through cache-timing side channels to extract information about the Argon2 input. Since `gen_opened_key()` and `gen_vote_token()` protect election integrity and voter anonymity respectively, this could theoretically compromise both properties.

The risk is mitigated by the fact that this is likely deployed in a controlled server environment, but the fix is trivial and the benchmark code already demonstrates the correct approach.

**Proof of Concept:**

N/A — Side-channel exploitation requires co-location on the same physical hardware.

**Remediation:**

```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2id to hash the data, with tuned parameters."

    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.ID,  # Argon2id per RFC 9106, OWASP
    )
```

> **⚠️ Migration Note:** Changing the variant will produce different hash outputs. Any existing elections with stored `opened_key` values (open or closed elections) will fail tamper detection. A migration strategy is required — either re-derive keys for open elections, or store the variant alongside the hash.

#### ASVS-1144-MED-002: Argon2 Parameters Adopted from Passlib Defaults Without Application-Specific Tuning

**Location:**
- **File:** `v3/steve/crypto.py`
- **Function:** `_hash()` (line ~78)
- **Related Functions:** `benchmark_argon2()`, `gen_opened_key()`, `gen_vote_token()`

**Gap Type:** Type A — Missing documented tuning rationale

**Vulnerable Code:**

```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2 to hash the data, with default tuning parameters."

    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,       # Passlib default
        memory_cost=65536,  # Passlib default
        parallelism=4,      # Passlib default
        hash_len=32,
        type=argon2.low_level.Type.D,
    )
```

**Description:**

The parameters are explicitly annotated as "Passlib default" with no evidence of application-specific tuning. ASVS 11.4.4 requires parameters that "balance security and performance to prevent brute-force attacks." While the values are within a reasonable range, the concern is:

1. **Parallelism of 4** — Higher parallelism allows faster computation for both the server and attackers with multi-core hardware. OWASP's recommended configurations use `p=1`:
   - Configuration 1: `m=47104, t=1, p=1`
   - Configuration 2: `m=19456, t=2, p=1`

2. **No documented tuning rationale** — The `benchmark_argon2()` function exists specifically for parameter tuning, showing the developers built the tooling but the production parameters still use untuned defaults. The benchmark tests 8 different configurations but the production code doesn't reflect any benchmark-informed decision.

3. **Parameter adequacy for threat model** — For this system, Argon2 protects:
   - `gen_opened_key()`: election integrity (tamper detection) — input is high-entropy election data, so Argon2 stretching is less critical here.
   - `gen_vote_token()`: voter anonymity — input includes the 32-byte `opened_key` plus identifiers, so entropy is sufficient.

**Data Flow:**

The `_hash()` function with untuned parameters is called by:
- `gen_opened_key()` → used in `election.py:open()` and `election.py:is_tampered()`
- `gen_vote_token()` → used in `election.py:add_vote()`, `tally_issue()`, and `has_voted_upon()`

**Impact:**

The untuned defaults are not inherently weak for this use case (high-entropy inputs), but they represent a process gap — the ASVS requirement expects deliberate parameter selection based on the application's performance budget and threat model. If the input entropy profile changes (e.g., if identifiers become shorter or more predictable), the current parameters may not provide sufficient protection.

**Proof of Concept:**

N/A — This is a process/documentation gap rather than an exploitable vulnerability.

**Remediation:**

1. Run the existing `benchmark_argon2()` on the production hardware.
2. Select parameters that target 100-500ms computation time per derivation.
3. Document the tuning rationale alongside the parameters:

```python
def _hash(data: bytes, salt: bytes) -> bytes:
    """Use Argon2id to hash the data.
    
    Parameters tuned on [hardware description] targeting ~200ms per derivation.
    Benchmarked [date] using benchmark_argon2(). Review annually or on hardware change.
    See: OWASP Password Storage Cheat Sheet, RFC 9106 Section 4.
    """
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=3,        # Tuned: increased from default 2
        memory_cost=65536,   # 64 MiB - above OWASP minimum of 46 MiB
        parallelism=1,       # Reduced from 4 to match OWASP recommendation
        hash_len=32,
        type=argon2.low_level.Type.ID,
    )
```

## Positive Security Patterns

1. **Layered key derivation architecture**: The chain `BLAKE2b → Argon2 → HKDF → Fernet` properly separates concerns — compression, stretching, derivation, and encryption are handled by purpose-built primitives.

2. **Per-voter cryptographic isolation**: Each `(pid, iid)` pair gets a unique 16-byte CSPRNG salt in the `mayvote` table, ensuring vote tokens and encryption keys are unique per voter per issue. This prevents correlation even by database administrators.

3. **CSPRNG usage throughout**: All random values use `secrets` module — `token_bytes()` for salts, `token_hex()` for IDs, `randbelow()` for shuffling. No use of `random` module for security operations.

4. **Authenticated encryption**: Fernet provides AES-128-CBC + HMAC-SHA256, preventing both confidentiality and integrity attacks on stored votes.

5. **Salt length enforcement at schema level**: Database CHECK constraints enforce exact 16-byte salt and 32-byte key/token lengths, providing defense-in-depth against programming errors.

6. **Cryptographic shuffle for vote ordering**: `crypto.shuffle()` uses Fisher-Yates with `secrets.randbelow()`, preventing any information leakage through vote ordering during tally.

7. **Tamper detection before tally**: `tally.py:main()` calls `election.is_tampered()` before processing any votes, preventing tallying of modified elections.

8. **Transaction safety for salt generation**: `add_salts()` wraps all salt insertions in a transaction, preventing partial salt assignment that could leave some voters without encryption keys.

9. **HKDF context separation**: The `info` parameter in HKDF (`b'xchacha20_key'`) provides domain separation for the derived key, even though the label is currently mismatched with the actual algorithm.

## Architecture Observations

1. **Key derivation chain integrity**: The full chain provides appropriate security for the threat model:
   ```
   election_data + salt → BLAKE2b(64B) → Argon2(32B) → opened_key
   opened_key + pid + iid + salt → Argon2(32B) → vote_token
   vote_token + salt → HKDF-SHA256(32B) → encryption_key
   encryption_key → Fernet → ciphertext
   ```
   Each step uses an appropriate algorithm for its purpose. The Argon2 steps provide the key stretching required by ASVS 11.4.4; the HKDF step correctly handles derivation from already-strong material.

2. **HKDF info/comment mismatch**: The `_b64_vote_key()` function's docstring says "Key-stretch" and references "PBKDF," and the HKDF `info` parameter says `xchacha20_key`, but the actual encryption uses Fernet (AES-128). The `###` comment acknowledges a planned migration. While not a security vulnerability (the key derivation is cryptographically sound), this creates maintenance risk if the migration introduces a second derivation path with the same info label.

3. **Non-constant-time comparison in `is_tampered()`**: The comparison `opened_key != md.opened_key` uses Python's standard `!=` which short-circuits. In practice, this is not exploitable because (a) Argon2 computation time (~100ms+) dominates any comparison timing difference, and (b) the method is called from CLI tooling, not web endpoints. However, using `hmac.compare_digest()` would be best practice for defense-in-depth.

4. **Election key lifecycle**: The `opened_key` serves dual purpose — tamper detection AND as input to vote token generation. Compromise of the opened_key (stored in the election table) would allow an attacker with database read access to derive all vote tokens and decrypt all votes. This is acknowledged in the threat model per the known false positive patterns.
