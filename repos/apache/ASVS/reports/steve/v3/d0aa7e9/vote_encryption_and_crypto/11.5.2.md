## Positive Security Patterns

### 1. Exclusive Use of `secrets` Module
**Location:** `crypto.py` (all random generation)

**Detail:** The entire codebase uses Python's `secrets` module for all cryptographic random number generation. This is the recommended CSPRNG module since Python 3.6, backed by `os.urandom()` which uses the kernel's CSPRNG. On modern Linux, this uses `getrandom(2)` which:
- Blocks only until initial entropy pool is seeded (one-time at boot)
- Never depletes or degrades under heavy demand
- Is thread-safe

### 2. No Weak RNG for Security Operations
**Location:** All files

**Detail:** There is no `import random` in any security-critical code. The `shuffle()` function explicitly implements Fisher-Yates using `secrets.randbelow()` rather than using `random.shuffle()`, demonstrating awareness of the distinction between cryptographic and statistical randomness.

### 3. Fail-Closed on RNG Failure
**Location:** `crypto.py:gen_salt()`, `create_id()`, `shuffle()`

**Detail:** If the underlying `os.urandom()` were to fail (extremely rare), `secrets` functions propagate the `OSError` rather than falling back to a weaker source. This means operations abort rather than proceeding with weak randomness, maintaining security guarantees even in failure scenarios.

### 4. Proper Salt Length
**Location:** `crypto.py:28`, `schema.sql:salt` columns

**Detail:** 16-byte (128-bit) salts are used consistently, enforced both in code (`SALT_LEN = 16`) and at the database level (`CHECK (length(salt) = 16)`). This provides adequate collision resistance (2^64 birthday bound) for the application's scale.

### 5. Per-Voter Salt Generation in Transaction
**Location:** `election.py:add_salts()` (~line 118)

**Detail:** Salt generation for all mayvote rows is wrapped in a transaction (`BEGIN TRANSACTION`/`COMMIT`), ensuring atomicity. If salt generation fails partway through, the transaction rolls back, preventing partial salt assignment and maintaining database consistency.

### 6. Independent Fernet Instances
**Location:** `crypto.py:create_vote()` and `decrypt_votestring()`

**Detail:** A new `Fernet` instance is created for each encryption/decryption operation. This avoids shared state between concurrent operations and ensures each IV is independently generated, preventing any potential state-based vulnerabilities.

## Architecture Observations

### RNG Demand Profile

The system's heaviest RNG demand occurs during two operations:

1. **Election Opening (`add_salts()`):** Generates one 16-byte salt per (voter × issue) pair. For an election with 1,000 voters and 10 issues, this is 10,000 calls to `secrets.token_bytes(16)` in rapid succession. On modern Linux, `os.urandom()` serves these in microseconds each — well within safe operating parameters.

2. **Concurrent Vote Submission:** Each `add_vote()` triggers `Fernet.encrypt()` which internally generates a 16-byte random IV via `os.urandom()`. For a closed-community voting system with authenticated users (per the domain context), concurrent demand is bounded and well within `os.urandom()` capacity.

### Platform Dependency Analysis

The security of the `secrets` module depends on the underlying OS CSPRNG:
- **Linux ≥ 3.17:** Uses `getrandom(2)`, safe under all post-boot conditions
- **macOS:** Uses `getentropy(2)`, similarly safe
- **Windows:** Uses `BCryptGenRandom()`, cryptographically secure
- **Containerized/VM environments:** Modern kernels provide adequate entropy even in minimal environments, though initial boot entropy should be verified for cloud deployments

This is a deployment consideration rather than a code deficiency. The code correctly delegates to OS-level entropy sources as recommended by cryptographic best practices.

## Conclusion

The random number generation mechanism in this codebase is well-designed to work securely under heavy demand. All cryptographic randomness is sourced exclusively from Python's `secrets` module (backed by `os.urandom()`), which is a CSPRNG designed for exactly this purpose. There are no fallback mechanisms to weaker RNG sources, no manual seeding, no use of the non-cryptographic `random` module, and all RNG outputs are properly consumed. The system fails closed if RNG becomes unavailable.

**ASVS 11.5.2 is fully satisfied with zero findings.**
