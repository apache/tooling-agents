### Audit Scope

The codebase implements a multi-layered cryptographic hash architecture using BLAKE2b, Argon2, HKDF-SHA256, and Fernet (HMAC-SHA256) for data integrity, tamper detection, and vote authentication. ## Findings

### Medium Severity

#### ASVS-1143-MED-001: Production Argon2 Uses Type.D (Argon2d) While Benchmark Uses Type.ID (Argon2id) — Inconsistency Indicates Potential Misconfiguration

**Severity:** MEDIUM

**Location:**
- **File:** `v3/steve/crypto.py`
- **Function:** `_hash()` (lines 76-84) vs `benchmark_argon2()` (lines 111-128)
- **Lines:** 83, 122

**Related Functions Checked:**
- `gen_opened_key()` (lines 40-47) — calls `_hash()` with Type.D
- `gen_vote_token()` (lines 50-54) — calls `_hash()` with Type.D
- `benchmark_argon2()` (lines 111-128) — uses Type.ID

**Description:**

The production hash function `_hash()` uses Argon2d (Type.D) while the benchmark function uses Argon2id (Type.ID). This inconsistency suggests a configuration error that degrades security below industry best practices.

**Vulnerable Code:**

Production implementation (line 83):
```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2 to hash the data, with default tuning parameters."
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,  # <-- Argon2d
    )
```

Benchmark code (line 122):
```python
argon2.low_level.hash_secret_raw(
    secret=intermediate,
    salt=salt,
    time_cost=r,
    memory_cost=m,
    parallelism=p,
    hash_len=32,
    type=argon2.low_level.Type.ID,  # <-- Argon2id
)
```

**Data Flow:**

1. Election tamper detection flow:
   ```
   election data → gen_opened_key() → _hash(digest, salt) with Type.D → opened_key
   ```

2. Vote token generation flow:
   ```
   opened_key + pid + iid → gen_vote_token() → _hash(data, salt) with Type.D → vote_token
   ```

**Impact:**

1. **Side-channel vulnerability:** Argon2d's memory access pattern is data-dependent, making it vulnerable to cache-timing side-channel attacks. An attacker with co-located process access could observe memory access patterns during hash computation, potentially recovering hash inputs. This is particularly relevant for the tamper detection mechanism (`gen_opened_key`) and vote token generation (`gen_vote_token`).

2. **Benchmark invalidity:** The `benchmark_argon2()` function tests performance using Argon2id, but production uses Argon2d. Performance characteristics differ between variants, meaning tuning decisions based on benchmark results may not accurately reflect production behavior.

3. **Non-compliance with recommendations:** RFC 9106 (Section 4) explicitly recommends Argon2id as the primary variant: *"If you do not know the difference between them or you consider side-channel attacks as a viable threat, choose Argon2id."* OWASP Password Storage Cheat Sheet also recommends Argon2id.

**Why this matters for ASVS 11.4.3:** While Argon2d IS collision resistant with 256-bit output (meeting the literal bit-length requirement), the ASVS chapter objective states systems should be "aligned with industry best practices." The discrepancy between production and benchmark code strongly suggests this is an inadvertent configuration error rather than an intentional design decision. Using the non-recommended variant for integrity-critical hashing weakens the overall cryptographic posture.

**Proof of Concept:**

No external exploit vector — this requires co-located process or physical access for side-channel observation. The primary concern is the production/benchmark inconsistency indicating a configuration error that degrades the security margin of the hash function below best-practice levels.

**Remediation:**

```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2id to hash the data, with default tuning parameters."
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.ID,  # Argon2id per RFC 9106
    )
```

> **Migration Note:** Changing the Argon2 type will change the output of all hash operations. This means:
> - All existing `opened_key` values will no longer match (triggering false tamper detection)
> - All existing `vote_token` values will be unreproducible (breaking vote lookups for existing elections)
> - Migration must be coordinated: apply only to new elections, or re-derive all hashes during a maintenance window.

## Positive Security Patterns

### 1. Hash Output Bit-Lengths Meet ASVS Requirements

All hash functions producing integrity-critical outputs use ≥256-bit outputs:
- `_hash()` → 32 bytes (256 bits) for collision resistance ✅
- BLAKE2b → 64 bytes (512 bits) as intermediate ✅
- HKDF-SHA256 → 32 bytes (256 bits) for key material ✅
- Fernet HMAC-SHA256 → 256-bit hash output for authentication ✅

### 2. BLAKE2b Pre-Hashing Before Argon2

```python
# crypto.py line 42
digest = hashlib.blake2b(edata).digest()
```

Election data of arbitrary length is pre-hashed through BLAKE2b (512-bit output, collision resistant) before being fed to Argon2. This correctly handles the Argon2 input size limitation while maintaining collision resistance throughout the chain.

### 3. Cryptographically Secure Random Generation

All randomness sources use the `secrets` module:
```python
secrets.token_bytes(SALT_LEN)  # Salt generation (line 35)
secrets.token_hex(5)           # ID generation (line 101)
secrets.randbelow(i + 1)       # Shuffle (line 97)
```

### 4. Per-Voter Salt Prevents Hash Correlation

Each `(pid, iid)` pair in the `mayvote` table receives a unique 16-byte cryptographic salt, ensuring vote tokens cannot be correlated across issues or voters even with identical inputs.

### 5. Database-Level Hash Length Enforcement

The schema enforces correct hash output lengths at the database layer:
```sql
vote_token BLOB NOT NULL CHECK (length(vote_token) = 32)   -- 256 bits
opened_key BLOB CHECK (opened_key IS NULL OR length(opened_key) = 32) -- 256 bits
```

This provides defense-in-depth against implementation errors producing truncated or incorrect hash values.

### 6. Tamper Detection Check Before Tallying

```python
# tally.py line 147
if election.is_tampered(pdb):
    print(f'Error: Election {election_id} has been tampered with. Cannot proceed.')
    sys.exit(1)
```

The tally process validates election integrity via hash recomputation before proceeding with any decryption, preventing tallying of tampered elections.

### 7. Vote Anonymization via Cryptographic Shuffle

```python
# election.py line 242
crypto.shuffle(votes)  # Fisher-Yates with secrets.randbelow()
```

Decrypted votes are shuffled using a cryptographically secure Fisher-Yates implementation before tallying, preventing correlation with database insertion order.

## Architecture Observations

### 1. HKDF Info Parameter Mismatch (Informational)

The `_b64_vote_key()` function uses `info=b'xchacha20_key'` despite deriving keys for Fernet (AES-128-CBC + HMAC-SHA256). The comment confirms a planned migration to XChaCha20-Poly1305. When migration occurs, care must be taken to either:
- Use a different `info` value for the new algorithm to maintain cryptographic domain separation
- Or ensure all existing ciphertexts are re-encrypted

### 2. Hash Chain Dependencies

The cryptographic hash chain is:
```
election_data → BLAKE2b(512-bit) → Argon2d(256-bit) → opened_key
opened_key + pid + iid → Argon2d(256-bit) → vote_token
vote_token → HKDF-SHA256(256-bit) → Fernet key → AES-128-CBC + HMAC-SHA256
```

Each step maintains ≥256-bit security level. The chain is well-structured with clear separation of concerns.

### 3. Argon2 Parameters Are Production-Appropriate

- `time_cost=2`: Provides adequate computational cost for server-side operations
- `memory_cost=65536` (64 MB): Provides memory-hardness against GPU/ASIC attacks
- `parallelism=4`: Reasonable for typical server hardware
- `hash_len=32`: 256-bit output meeting ASVS requirements
