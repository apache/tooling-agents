## Findings

### Medium Severity

#### ASVS-1133-MED-001: Argon2d Variant Used for Key Derivation Instead of Recommended Argon2id

**Severity:** MEDIUM

**Location:**
- **File:** `v3/steve/crypto.py`
- **Function:** `_hash()` (lines 89-99)
- **Called by:** `gen_opened_key()` (line 48), `gen_vote_token()` (line 56)

**Related Functions Checked:**
- `gen_opened_key()` — Uses `_hash()` to derive master tamper detection key
- `gen_vote_token()` — Uses `_hash()` to derive per-voter vote tokens
- `benchmark_argon2()` — Correctly uses `Type.ID` in benchmark code (line ~153)

**Description:**

The production `_hash()` function uses Argon2d (`Type.D`) instead of the RFC 9106-recommended Argon2id variant. Argon2d uses data-dependent memory access patterns, making it vulnerable to cache-timing side-channel attacks. This is particularly concerning because the benchmark function in the same file correctly demonstrates Argon2id usage, creating false confidence that the correct algorithm is deployed.

**Vulnerable Code:**
```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2 to hash the data, with default tuning parameters."
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,  # <-- Argon2d: data-dependent memory access
    )
```

**Evidence of Awareness (same file, line ~153 in `benchmark_argon2()`):**
```python
type=argon2.low_level.Type.ID,  # <-- Benchmark correctly uses Argon2id
```

**Data Flow:**
1. `election.py:add_vote()` calls `crypto.gen_vote_token(opened_key, pid, iid, salt)` → `_hash()` with Argon2**d**
2. The resulting `vote_token` feeds into `_b64_vote_key()` → HKDF → Fernet key
3. Fernet encrypts vote with derived key
4. Side-channel attack on Argon2d could recover `vote_token`, enabling key derivation and forging of valid authenticated ciphertexts

**Proof of Concept:**

An attacker with co-located access (cloud/shared hosting environment) could:
1. Monitor cache access patterns during `gen_vote_token()` execution
2. Use cache-timing analysis to recover the Argon2d output
3. Derive the per-voter Fernet encryption key using the recovered token
4. Decrypt existing votes or create forged authenticated ciphertexts

**Impact:**

Argon2d uses data-dependent memory access patterns, making it vulnerable to cache-timing side-channel attacks (RFC 9106, Section 7.1). An attacker with co-located access (cloud/shared hosting) could observe memory access patterns during `gen_vote_token()` or `gen_opened_key()` execution to recover the hash output. Since these outputs directly feed the encryption key derivation chain, recovery enables:
- Derivation of per-voter Fernet keys
- Decryption of encrypted votes (vote secrecy breach)
- Creation of forged, validly-authenticated ciphertexts (undermining ASVS 11.3.3)

RFC 9106 recommends Argon2id for most use cases as it combines Argon2i's side-channel resistance (first pass) with Argon2d's GPU resistance (subsequent passes).

**Remediation:**
```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2 to hash the data, with default tuning parameters."
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.ID,  # Argon2id per RFC 9106
    )
```

> **Note:** Changing the Argon2 variant will change all derived outputs (opened_key, vote_tokens). This is a breaking change requiring migration of any open or closed elections. Plan migration during a period with no active elections.

#### ASVS-1133-MED-002: Non-Constant-Time Comparison of Cryptographic Key Material in Tamper Detection

**Severity:** MEDIUM

**Location:**
- **File:** `v3/steve/election.py`
- **Function:** `is_tampered()` (lines 345-363)

**Related Functions Checked:**
- `tally.py:main()` — Calls `is_tampered()` before tallying
- `gather_election_data()` — Provides data for tamper key recomputation
- `gen_opened_key()` — Generates comparison key

**Description:**

The `is_tampered()` method uses Python's non-constant-time `!=` operator to compare 32-byte cryptographic key material (`opened_key`). This comparison short-circuits on the first differing byte, creating a timing side-channel that could allow remote extraction of the master key through repeated timing measurements.

**Vulnerable Code:**
```python
def is_tampered(self, pdb):
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)

    # The computed key should be unchanged.
    return opened_key != md.opened_key  # <-- Non-constant-time bytes comparison
```

**Data Flow:**
1. `md.opened_key` (32 bytes) is loaded from the database — this is the stored master key used to derive all vote encryption keys via `gen_vote_token()`
2. `opened_key` is freshly computed from current election data
3. Python's `!=` operator on `bytes` objects uses `memcmp`, which **short-circuits on the first differing byte**
4. Timing differences reveal byte-by-byte information about `md.opened_key`

**Proof of Concept:**

If `is_tampered()` is reachable from a network-accessible path:
1. Attacker modifies election data to produce a guessed `opened_key`
2. Measures response time for `is_tampered()` call
3. Timing variance indicates which byte position first differs
4. Iterates through all 256 values for each byte position
5. After ~256×32 = 8,192 timed requests, extracts full 32-byte `opened_key`
6. Uses recovered key to derive all vote encryption keys

**Impact:**

If reachable from a network-accessible path, an attacker could extract `md.opened_key` through ~256×32 = 8,192 timed requests. The `opened_key` combined with known PIDs and IIDs (voter identifiers) and database-stored per-voter salts enables derivation of all vote encryption keys via `gen_vote_token()` → `_b64_vote_key()` → Fernet key. With the Fernet keys, an attacker could:
- Decrypt all encrypted votes for the election
- Forge new encrypted votes with valid HMAC authentication (defeating ASVS 11.3.3's unauthorized modification protection)

**Entry Point Analysis:**
- Currently called from `tally.py:main()` (CLI) — limited exploitability
- `is_tampered()` is a **public method** on the `Election` class — if called from any web route during state transitions (e.g., checking election integrity on page load), it becomes remotely exploitable

**Remediation:**
```python
import hmac

def is_tampered(self, pdb):
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)

    # Use constant-time comparison to prevent timing side-channels
    return not hmac.compare_digest(opened_key, md.opened_key)
```

#### ASVS-1133-MED-003: HKDF Domain Separation Label Mismatches Actual Encryption Algorithm

**Severity:** MEDIUM

**Location:**
- **File:** `v3/steve/crypto.py`
- **Function:** `_b64_vote_key()` (lines 59-70)

**Related Functions Checked:**
- `create_vote()` — Uses Fernet (AES-128-CBC + HMAC-SHA256), not XChaCha20
- `decrypt_votestring()` — Also uses Fernet
- `gen_vote_token()` — Provides input to `_b64_vote_key()`

**Description:**

The HKDF key derivation function uses `info=b'xchacha20_key'` as its domain separation label, but the actual encryption algorithm in use is Fernet (AES-128-CBC + HMAC-SHA256). This mismatch violates cryptographic domain separation principles and creates a latent vulnerability during the planned migration to XChaCha20-Poly1305.

**Vulnerable Code:**
```python
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    "Key-stretch the vote_token. (ref: PBKDF)"

    ### still using Fernet now, but will switch soon. Leaving comments.
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,  # 32-byte key for XChaCha20-Poly1305  <-- Comment references wrong algorithm
        salt=salt,
        info=b'xchacha20_key',  # <-- Domain separation label for WRONG algorithm
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

**Actual Usage (same file, lines 73-78):**
```python
def create_vote(vote_token: bytes, salt: bytes, votestring: str) -> bytes:
    b64key = _b64_vote_key(vote_token, salt)
    f = cryptography.fernet.Fernet(b64key)  # <-- Fernet (AES-128-CBC + HMAC-SHA256), NOT XChaCha20
    return f.encrypt(votestring.encode())
```

**Data Flow:**
1. `gen_vote_token()` produces a vote-specific token
2. `_b64_vote_key()` derives encryption key using HKDF with incorrect `info` label
3. `create_vote()` uses derived key with Fernet, not XChaCha20
4. During planned migration, both algorithms would derive identical keys from same inputs

**Proof of Concept:**

During algorithm migration, if both Fernet and XChaCha20-Poly1305 are deployed simultaneously:
1. Same `(vote_token, salt)` pair used for both algorithms
2. Both derive identical 32-byte keys due to shared `info=b'xchacha20_key'`
3. Attacker could exploit cross-protocol attacks where a ciphertext valid under one scheme is reinterpreted under another
4. Violates fundamental cryptographic principle that different protocols should use different keys

**Impact:**

The HKDF `info` parameter provides cryptographic domain separation (NIST SP 800-56C Rev. 2, Section 4). Using `b'xchacha20_key'` when the actual algorithm is Fernet/AES-128-CBC means:

1. **During algorithm migration:** If XChaCha20-Poly1305 is introduced alongside Fernet (as the TODO comments indicate), both algorithms would derive **identical keys** from the same `(vote_token, salt)` input since they share the same `info` label. This violates the fundamental purpose of HKDF's `info` parameter and could allow cross-protocol attacks where a ciphertext valid under one scheme is reinterpreted under another.

2. **Audit confusion:** The mismatch between labeled intent and actual implementation creates ambiguity in cryptographic inventory, making it difficult to verify which algorithms are actually in use — a core concern of the ASVS Cryptography chapter's control objectives.

3. **Current direct risk is limited** since only Fernet is currently used, but the explicit TODO comment (`### still using Fernet now, but will switch soon`) indicates migration is planned, making this a latent vulnerability.

**Remediation:**
```python
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    "Key-stretch the vote_token for Fernet encryption."
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,  # Fernet requires 32 bytes (16 signing + 16 encryption)
        salt=salt,
        info=b'fernet_vote_key_v1',  # Correct domain separation for current algorithm
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

> **Note:** Changing the `info` parameter changes derived keys. This is a breaking change requiring re-encryption of existing votes or a versioned migration strategy.

## Architecture Observations

1. **Sound Authenticated Encryption Foundation:** The core vote encryption uses Fernet, which provides authenticated encryption via Encrypt-then-MAC. This correctly protects encrypted votes against unauthorized modification, satisfying the primary intent of ASVS 11.3.3.

2. **Defense-in-Depth Key Derivation:** The key chain (`election_salt → opened_key → vote_token → HKDF → Fernet key`) provides strong separation between elections and between voters. Compromise of one voter's key does not expose other voters' votes.

3. **Algorithm Migration in Progress:** Comments and code artifacts indicate planned migration from Fernet to XChaCha20-Poly1305, which would be an improvement (true AEAD vs composite Encrypt-then-MAC). The `info` parameter mismatch suggests the migration was partially started in the key derivation but not completed in the encryption layer.

4. **Tamper Detection Gap:** While `is_tampered()` is called before tallying in `tally.py`, there is no evidence in the provided code that it is called before vote submission or other web-accessible operations. If election data is tampered with during the open period, votes may be encrypted with keys derived from modified data, causing silent data loss at tally time.
