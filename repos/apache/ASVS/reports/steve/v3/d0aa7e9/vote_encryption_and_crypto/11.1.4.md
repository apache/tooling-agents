## Findings

### HIGH SEVERITY

#### ASVS-1114-HIGH-001: Absence of Formal Cryptographic Inventory and Post-Quantum Migration Plan

**Severity:** HIGH  
**Gap Type:** Type B — Cryptographic awareness EXISTS in code but is NOT FORMALIZED into an inventory or plan

**Location:**
- **Files:** Repository-wide (no `CRYPTO_INVENTORY.md` or equivalent document exists)
- **Primary evidence:** `v3/steve/crypto.py:60-61`
- **Related functions checked:**
  - `crypto.py`: `gen_opened_key()`, `gen_vote_token()`, `_b64_vote_key()`, `create_vote()`, `decrypt_votestring()`, `_hash()`
  - `election.py`: `open()`, `add_salts()`, `add_vote()`, `tally_issue()`, `is_tampered()`
  - `schema.sql`: All cryptographic column definitions

**Description:**

The codebase uses six distinct cryptographic primitives (BLAKE2b, Argon2d, HKDF-SHA256, Fernet/AES-128-CBC, HMAC-SHA256, CSPRNG) across multiple files, but no formal cryptographic inventory document exists that catalogs these primitives, their configuration parameters, approved usage contexts, key lifecycle management, or deprecation timelines.

ASVS 11.1.4 explicitly requires: (a) a maintained cryptographic inventory, and (b) a documented plan outlining migration to new standards including post-quantum cryptography.

**Evidence of awareness without formalization:**

File: `v3/steve/crypto.py`, lines 60-61:
```python
### still using Fernet now, but will switch soon. Leaving comments.
keymaker = hkdf.HKDF(
```

This inline comment demonstrates that migration from Fernet was considered but never formalized into a documented plan. The HKDF parameters were pre-configured for a future algorithm, but no timeline, compatibility strategy, or migration procedure exists.

**Vulnerable Code:**
```python
# crypto.py:60-61 - Evidence of undocumented migration intent
### still using Fernet now, but will switch soon. Leaving comments.
keymaker = hkdf.HKDF(
    algorithm=hashes.SHA256(),
    length=32,  # 32-byte key for XChaCha20-Poly1305
    salt=salt,
    info=b'xchacha20_key',  # References future algorithm not yet implemented
)
```

**Data Flow Analysis:**

The cryptographic architecture spans multiple layers without documentation:

```
Election Creation (election.py:open)
    ↓
Salt Generation (crypto.py:gen_salt) → secrets.token_bytes(16)
    ↓
Tamper Detection Key (crypto.py:gen_opened_key)
    ↓
BLAKE2b(election_data) → Argon2d(hash, salt) → opened_key
    ↓
Vote Submission (election.py:add_vote)
    ↓
Vote Token (crypto.py:gen_vote_token) → Argon2d(voter_id + issue_id, salt)
    ↓
Vote Key Derivation (crypto.py:_b64_vote_key) → HKDF-SHA256(vote_token, salt)
    ↓
Vote Encryption (crypto.py:create_vote) → Fernet(vote_key).encrypt(vote_data)
    ↓
Storage (schema.sql) → SQLite BLOB columns
    ↓
Tallying (election.py:tally_issue)
    ↓
Decryption (crypto.py:decrypt_votestring) → Fernet(vote_key).decrypt(ciphertext)
    ↓
Shuffle (crypto.py:shuffle) → Fisher-Yates with secrets.randbelow()
```

**Specific gaps in what an inventory would cover:**

1. **No documented algorithm catalog** — An auditor must reverse-engineer the full cryptographic posture by reading source code across `crypto.py`, `election.py`, and `schema.sql`
2. **No PQC migration path** — Fernet (AES-128-CBC) and HKDF-SHA256 are quantum-vulnerable for key exchange contexts; BLAKE2b and Argon2 have different quantum resistance profiles; no analysis exists documenting which primitives need migration
3. **No key lifecycle documentation** — Election keys are generated at open time and used at tally time, but retention, rotation, and destruction policies are undocumented
4. **No parameter justification** — Argon2 parameters (t=2, m=64MB, p=4) are set to "Passlib defaults" per comments but have no documented security analysis for this specific threat model

**Proof of Concept:**

To demonstrate the absence of inventory, an auditor attempting to answer basic questions must examine multiple files:

```bash

```

**Impact:**

Without a cryptographic inventory, the system cannot systematically respond to:

1. **Algorithm deprecations** — If AES-128-CBC is deprecated due to practical attacks, there is no documented procedure to identify all affected components
2. **PQC migration requirements** — When NIST PQC standards (ML-KEM, ML-DSA) become mandatory, there is no roadmap for implementation
3. **Compliance audits** — Election systems often require cryptographic certification; absence of inventory creates audit friction
4. **Incident response** — If a cryptographic vulnerability is discovered (e.g., in Fernet implementation), there is no quick reference to assess blast radius
5. **Developer onboarding** — New developers cannot quickly understand the cryptographic architecture without reading all source files

For an election system protecting ballot secrecy, this represents a significant governance gap that could delay response to cryptographic threats.

## Post-Quantum Cryptography Migration Plan

### Risk Assessment

| Algorithm | Quantum Threat | Timeline | Priority |
|-----------|---------------|----------|----------|
| Fernet (AES-128) | Grover's algorithm reduces security to ~64-bit | 2030+ (conservative) | Medium |
| HMAC-SHA256 | Reduced to ~128-bit by Grover's | 2030+ | Low |
| Argon2 | Memory-hardness preserved | N/A | Low |
| HKDF-SHA256 | Hash function security reduced | 2030+ | Low |

### Migration Phases

| Phase | Timeline | Action | Algorithms Affected | Breaking Change |
|-------|----------|--------|---------------------|-----------------|
| 1 | Q2 2026 | Complete this inventory document | All | No |
| 2 | Q3 2026 | Migrate Argon2d → Argon2id | C2 | Yes (re-key required) |
| 3 | Q4 2026 | Migrate Fernet → XChaCha20-Poly1305 (256-bit) | C1, C3 | Yes (re-encrypt required) |
| 4 | Q1 2027 | Implement algorithm versioning in schema | All | No (enables future migrations) |
| 5 | Q2 2027 | Evaluate NIST PQC standards (ML-KEM-768, ML-DSA-65) | All | TBD |
| 6 | Q4 2027 | Proof-of-concept hybrid classical+PQC KDF | C2, C3 | No (parallel implementation) |
| 7 | 2028+ | Production PQC deployment (if threat materializes) | C1, C2, C3 | Yes (versioned migration) |

### Breaking Change Management

All cryptographic algorithm changes that affect key derivation or encryption require data migration:

1. **Argon2d → Argon2id:** All `opened_key` and `vote_token` values change
   - **Migration:** Re-open closed elections with new algorithm (not retroactive)
   - **Compatibility:** Old elections remain readable with legacy code path
   
2. **Fernet → XChaCha20:** All encrypted votes become undecryptable
   - **Migration:** Only affects new elections after deployment
   - **Compatibility:** Schema versioning required (see Phase 4)

3. **HKDF info parameter update:** Changes derived keys
   - **Migration:** Bundled with Fernet → XChaCha20 migration
   - **Compatibility:** New `info=b'xchacha20_poly1305_v1'`

## Parameter Justification

### Argon2 Parameters (t=2, m=65536, p=4)
- **Source:** Passlib library defaults
- **Analysis:** 
  - Memory cost of 64MB provides resistance against GPU attacks
  - Time cost of 2 iterations balances security vs. server performance
  - Parallelism of 4 matches typical server CPU core count
- **Threat Model:** Server-side key derivation; timing attacks not primary concern (but see Argon2id migration)
- **Review Date:** 2026-04-22
- **Next Review:** 2027-04-22

### HKDF-SHA256 Parameters
- **Salt:** 16-byte random per voter-issue (from `mayvote.salt`)
- **Info:** Currently `b'xchacha20_key'` (mislabeled; should be `b'fernet_vote_key_v1'`)
- **Length:** 32 bytes (256 bits)
- **Justification:** NIST SP 800-108 compliant; info parameter provides domain separation

### Fernet Parameters
- **Key Size:** 128-bit AES + 128-bit HMAC (derived from 256-bit HKDF output)
- **Mode:** AES-128-CBC with PKCS7 padding
- **MAC:** HMAC-SHA256
- **Justification:** Industry-standard authenticated encryption; cryptography.io library provides timing-safe implementation

## Positive Security Patterns

1. **Strong CSPRNG usage throughout** — All random generation uses Python's `secrets` module, never `random`
2. **Per-voter-issue salt uniqueness** — Each `mayvote` entry gets its own 16-byte random salt, preventing vote correlation
3. **Authenticated encryption** — Fernet provides both confidentiality (AES-CBC) and integrity (HMAC-SHA256)
4. **Tamper detection** — Election data is fingerprinted with BLAKE2b → Argon2, stored as `opened_key`, and verified before tallying
5. **Vote shuffling before tallying** — `crypto.shuffle()` with Fisher-Yates prevents database-order leakage
6. **Public API hides secrets** — `get_metadata()` explicitly excludes `salt` and `opened_key`; `get_issue()` excludes salt
7. **Transaction safety for salt generation** — `add_salts()` uses explicit `BEGIN TRANSACTION`/`COMMIT`
8. **State machine enforcement** — Assertions prevent cryptographic operations in wrong states (e.g., can't vote on editable election)
9. **Computational hardness** — Argon2 with 64MB memory cost makes brute-force derivation of vote tokens expensive
10. **Centralized crypto module** — All cryptographic operations concentrated in `crypto.py` for auditability

---

## Architecture Observations

1. **Centralized crypto module** — All cryptographic operations are concentrated in `crypto.py`, which is good for auditability but currently serves as the de facto (implicit) inventory rather than being documented as such

2. **Two-layer key derivation** — `opened_key` (election-level) → `vote_token` (voter-level) → `vote_key` (encryption-level) provides good cryptographic separation, but this architecture is not documented outside of code

3. **Offline tallying model** — Decryption keys are derivable from database contents + election key, which means database confidentiality is critical; this architectural decision should be documented in the crypto inventory

4. **No key versioning** — There is no mechanism to version cryptographic parameters, meaning any algorithm change (e.g., Argon2d → Argon2id, Fernet → XChaCha20) would be a breaking change requiring data migration

5. **Code comments indicate migration intent** — The presence of `### still using Fernet now, but will switch soon` suggests cryptographic evolution is considered, but without formal process

6. **Schema enforces cryptographic constraints** — `CHECK` constraints on BLOB lengths provide defense-in-depth against malformed cryptographic data
