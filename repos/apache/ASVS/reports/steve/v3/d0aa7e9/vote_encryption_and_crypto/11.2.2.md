# ASVS 11.2.2 Crypto Agility Audit Report

## Executive Summary

- **Repository:** apache/steve
- **Audit Date:** Apr 22, 2026
- **Auditor:** Tooling Agents
- **ASVS Requirement:** 11.2.2 — Verify that the application is designed with crypto agility such that random number, authenticated encryption, MAC, or hashing algorithms, key lengths, rounds, ciphers and modes can be reconfigured, upgraded, or swapped at any time, to protect against cryptographic breaks. ### Summary

The application lacks fundamental crypto agility architecture. All cryptographic algorithms, parameters, and modes are hardcoded directly in `crypto.py` without any abstraction layer, configuration mechanism, or algorithm versioning in stored data. This makes it impossible to reconfigure, upgrade, or swap cryptographic primitives without code changes and makes post-quantum migration infeasible without a complete redesign of the cryptographic storage layer.

## Findings

### High Severity

#### ASVS-1122-HIGH-001: Absence of Cryptographic Abstraction Layer Prevents Algorithm Agility

**Severity:** HIGH  
**Gap Type:** Type A — Required control (crypto agility architecture) does not exist

**Location:**
- **File:** `v3/steve/crypto.py`
- **Functions:** `create_vote` (line ~62), `decrypt_votestring` (line ~69), `_b64_vote_key` (line ~53), `_hash` (line ~77), `gen_opened_key` (line ~38)

**Related Functions Checked:**
- `election.open()` (caller)
- `election.add_vote()` (caller)
- `election.tally_issue()` (caller)

**Description:**

All cryptographic algorithms are directly instantiated without any abstraction, configuration, or strategy pattern. The application lacks a cryptographic provider layer that would enable algorithm substitution without code changes.

**Vulnerable Code:**

```python
# _b64_vote_key (line 53-61) - Hardcoded HKDF-SHA256
keymaker = hkdf.HKDF(
    algorithm=hashes.SHA256(),
    length=32,
    salt=salt,
    info=b'xchacha20_key',
)

# create_vote (line 62-66) - Hardcoded Fernet
f = cryptography.fernet.Fernet(b64key)
return f.encrypt(votestring.encode())

# decrypt_votestring (line 69-73) - Hardcoded Fernet
f = cryptography.fernet.Fernet(b64key)
return f.decrypt(ciphertext).decode()

# _hash (line 77-87) - Hardcoded Argon2 with fixed parameters
return argon2.low_level.hash_secret_raw(
    secret=data, salt=salt,
    time_cost=2, memory_cost=65536, parallelism=4,
    hash_len=32, type=argon2.low_level.Type.D,
)

# gen_opened_key (line 38-47) - Hardcoded BLAKE2b
digest = hashlib.blake2b(edata).digest()
```

**Data Flow:**

```
election.open() / election.add_vote() / election.tally_issue()
    ↓
crypto.* functions
    ↓
hardcoded algorithm instantiation
    ↓
no configuration or swap mechanism
```

**Impact:**

- Swapping from Fernet (AES-128-CBC) to AES-256-GCM or XChaCha20-Poly1305 requires modifying `create_vote`, `decrypt_votestring`, and `_b64_vote_key` at minimum
- Upgrading Argon2 parameters requires modifying `_hash` and ensuring all existing hashes remain verifiable — no mechanism exists for this
- Migrating to post-quantum cryptography (PQC) algorithms requires a complete rewrite of the crypto module with no backward-compatibility path
- All callers in `election.py` are tightly coupled to the current implementations

**Remediation:**

Introduce a crypto provider abstraction with configuration-driven algorithm selection:

```python
# crypto_provider.py
import yaml
from dataclasses import dataclass

@dataclass
class CryptoConfig:
    encryption_algorithm: str = "fernet"  # or "xchacha20-poly1305", "aes-256-gcm"
    kdf_algorithm: str = "argon2id"
    kdf_time_cost: int = 2
    kdf_memory_cost: int = 65536
    kdf_parallelism: int = 4
    hash_algorithm: str = "blake2b"
    key_derivation: str = "hkdf-sha256"
    algorithm_version: int = 1

    @classmethod
    def from_config(cls, config_path: str) -> 'CryptoConfig':
        with open(config_path) as f:
            return cls(**yaml.safe_load(f).get('crypto', {}))

class CryptoProvider:
    """Abstraction layer for cryptographic operations."""

    ENCRYPTION_REGISTRY = {
        'fernet': FernetEncryptor,
        'aes-256-gcm': AES256GCMEncryptor,
        # Future: 'pqc-kem': PQCEncryptor,
    }

    def __init__(self, config: CryptoConfig):
        self.config = config
        self._encryptor = self.ENCRYPTION_REGISTRY[config.encryption_algorithm]()

    def create_vote(self, vote_token, salt, votestring):
        return self._encryptor.encrypt(
            self._derive_key(vote_token, salt),
            votestring.encode(),
            algorithm_version=self.config.algorithm_version,
        )

    def decrypt_votestring(self, vote_token, salt, ciphertext):
        version = self._extract_version(ciphertext)
        encryptor = self._get_encryptor_for_version(version)
        return encryptor.decrypt(
            self._derive_key(vote_token, salt), ciphertext
        ).decode()
```

#### ASVS-1122-HIGH-002: No Algorithm Versioning in Stored Cryptographic Data Prevents Migration

**Severity:** HIGH  
**Gap Type:** Type A — Required control (algorithm versioning) does not exist

**Location:**
- **File:** `v3/schema.sql`
- **Tables:** `vote` (line ~157), `election` (line ~54), `mayvote` (line ~137)

**Related Functions Checked:**
- `election.add_vote()` (writes to `vote` table)
- `election.tally_issue()` (reads from `vote` table)
- `election.open()` (writes to `election` table)

**Description:**

The database schema contains no algorithm version fields to track which cryptographic algorithms were used to produce stored data. This makes phased migration impossible and prevents the application from knowing which algorithm to use when decrypting or verifying existing data.

**Vulnerable Code:**

```sql
-- vote table: no algorithm version field
CREATE TABLE vote (
    vid  INTEGER PRIMARY KEY AUTOINCREMENT,
    vote_token  BLOB NOT NULL  CHECK (length(vote_token) = 32),
    ciphertext  BLOB NOT NULL
) STRICT;

-- election table: fixed-size opened_key with no version
opened_key  BLOB  CHECK (opened_key IS NULL OR length(opened_key) = 32),

-- mayvote table: salt with no derivation parameter version
salt  BLOB  CHECK (salt IS NULL OR length(salt) = 16),
```

**Data Flow:**

```
Cryptographic operation
    ↓
Data stored in database
    ↓
No version metadata stored
    ↓
Future decryption cannot determine which algorithm was used
```

**Impact:**

- During any algorithm migration, there is no way to determine which algorithm produced a given ciphertext, hash, or key
- A phased migration (new algorithm for new data, old algorithm for existing data) is impossible — all or nothing
- The CHECK constraint `length(vote_token) = 32` and `length(opened_key) = 32` enforce fixed output sizes, preventing algorithms with different output lengths
- Re-encryption of existing votes requires knowing the algorithm used, which is not stored
- This directly blocks the "seamless upgrades to post-quantum cryptography" required by ASVS 11.2.2

**Remediation:**

Add algorithm version fields and relax fixed-length constraints:

```sql
CREATE TABLE vote (
    vid  INTEGER PRIMARY KEY AUTOINCREMENT,
    vote_token  BLOB NOT NULL,
    ciphertext  BLOB NOT NULL,
    -- Algorithm version used for encryption; enables phased migration
    crypto_version  INTEGER NOT NULL DEFAULT 1
) STRICT;

CREATE TABLE election (
    -- ... existing fields ...
    salt  BLOB  CHECK (salt IS NULL OR length(salt) >= 16),
    opened_key  BLOB,
    -- Version of crypto parameters used when opening this election
    crypto_version  INTEGER,
    -- ... rest of fields ...
) STRICT;

CREATE TABLE mayvote (
    -- ... existing fields ...
    salt  BLOB  CHECK (salt IS NULL OR length(salt) >= 16),
    crypto_version  INTEGER,
    -- ... rest of fields ...
) STRICT;
```

## Positive Security Patterns

1. **Strong CSPRNG Usage**: All randomness correctly sourced from Python's `secrets` module — `gen_salt()`, `shuffle()`, `create_id()` all use cryptographically secure randomness.

2. **Multi-Layer Key Derivation**: The key derivation pipeline (BLAKE2b → Argon2 → HKDF → Fernet) provides defense-in-depth with a memory-hard KDF as the central component.

3. **Per-Voter Salt Isolation**: Each `(person, issue)` pair in `mayvote` receives a unique 16-byte salt, preventing cross-voter correlation even if the database is compromised.

4. **Authenticated Encryption**: Fernet provides both confidentiality (AES-128-CBC) and integrity (HMAC-SHA256), preventing ciphertext manipulation.

5. **Cryptographic Shuffling**: The Fisher-Yates shuffle with `secrets.randbelow()` properly breaks database insertion order to protect voter anonymity during tallying.

6. **Tamper Detection**: The `opened_key` mechanism cryptographically binds all election metadata, issues, and voter lists at election-open time, enabling detection of post-opening modifications.

7. **State Machine Enforcement**: Election lifecycle states (editable → open → closed) are enforced through assertions before cryptographic operations, preventing premature key generation or tallying.

8. **Benchmarking Tool**: The `benchmark_argon2()` function demonstrates awareness of parameter tuning needs, even though results cannot currently be applied without code changes.

## Architecture Observations

1. **Tightly Coupled Crypto Layer**: `election.py` directly calls specific functions in `crypto.py` (e.g., `crypto.gen_vote_token()`, `crypto.create_vote()`). There is no interface or abstraction between the business logic and cryptographic operations, making algorithm substitution require changes in both modules.

2. **In-Progress Migration Stalled**: Comments in `_b64_vote_key()` explicitly state `### still using Fernet now, but will switch soon`, and HKDF parameters reference XChaCha20-Poly1305. This indicates awareness of needed migration but no architectural support to accomplish it.

3. **Argon2 Type Inconsistency**: Production code uses `Type.D` (data-dependent, vulnerable to side-channel attacks) while the benchmark uses `Type.ID` (hybrid, recommended by RFC 9106 and OWASP). This should be `Type.ID` for the production `_hash()` function as well.

4. **Schema Rigidity**: CHECK constraints enforce exact byte lengths (`length(salt) = 16`, `length(opened_key) = 32`, `length(vote_token) = 32`), which would prevent adoption of algorithms with different output sizes without schema migration.

## Recommendations Summary

### Immediate Actions (0-30 days)

| Priority | Finding | Remediation Effort | Impact |
|----------|---------|-------------------|--------|
| 1 | ASVS-1122-HIGH-001 | High — Requires new abstraction layer | Enables all future algorithm changes |
| 2 | ASVS-1122-HIGH-002 | Medium — Schema migration + serialization format | Enables phased data migration |

**Additional Immediate Recommendation:** Change `argon2.low_level.Type.D` to `argon2.low_level.Type.ID` in the production `_hash()` function to align with RFC 9106 recommendations for resistance to side-channel attacks.

### Short-term Actions (1-3 months)

1. **Implement Cryptographic Abstraction Layer**: Design and implement a `CryptoProvider` class that encapsulates all cryptographic operations with a registry-based algorithm selection mechanism.

2. **Add Algorithm Versioning to Schema**: Migrate database schema to include `crypto_version` fields in all tables storing cryptographic material. Relax fixed-length CHECK constraints to accommodate future algorithms.

3. **Create Configuration System**: Externalize all cryptographic parameters (algorithm choices, Argon2 parameters, key lengths) to a YAML configuration file with validation.

4. **Fix HKDF Domain Separation**: Update the `info` parameter in `_b64_vote_key()` to correctly reflect the current algorithm (`b'fernet_vote_key_v1'`) and implement version-aware derivation.

### Long-term Actions (3-12 months)

1. **Implement Re-encryption Mechanism**: Build administrative tooling to re-encrypt existing votes when migrating to new algorithms, using the version fields to identify which algorithm was used for each record.

2. **Complete XChaCha20-Poly1305 Migration**: Finish the migration from Fernet to XChaCha20-Poly1305 using the new abstraction layer and versioning system.

3. **Post-Quantum Cryptography Preparation**: Research and prototype PQC algorithms suitable for the voting use case. Design hybrid encryption schemes that combine classical and PQC algorithms during the transition period.

4. **Automated Parameter Tuning**: Integrate the `benchmark_argon2()` results into the configuration system to automatically adjust KDF parameters based on target hardware performance.
