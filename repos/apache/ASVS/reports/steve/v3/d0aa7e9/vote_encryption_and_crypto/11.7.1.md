This audit evaluates the election system's cryptographic implementation against ASVS 11.7.1, which requires full memory encryption to protect sensitive data while in use. The application handles highly sensitive data (vote content, cryptographic keys, voter identity) but implements no memory protection mechanisms at the application level. ## Findings

### High Severity

#### ASVS-1171-HIGH-001: Argon2d Used Instead of Argon2id — Vulnerable to Memory Side-Channel Attacks

**Severity:** HIGH  
**Gap Type:** Type B — Correct library exists but wrong variant selected, creating false confidence in side-channel resistance

**Location:**
- **File:** `v3/steve/crypto.py`
- **Function:** `_hash()`
- **Lines:** 90–100

**Related Functions Checked:**
- `gen_opened_key()` (calls `_hash()`)
- `gen_vote_token()` (calls `_hash()`)
- `benchmark_argon2()` (uses correct `Type.ID`)

**Description:**

The `_hash()` function uses Argon2d (`Type.D`) instead of the recommended Argon2id (`Type.ID`). Argon2d performs data-dependent memory accesses, meaning the pattern of which memory locations are read during key derivation directly correlates with the secret input.

**Vulnerable Code:**
```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2 to hash the data, with default tuning parameters."

    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,  # <-- Data-dependent memory access
    )
```

**Evidence of Awareness:**

The benchmark function at line 153 correctly uses `Type.ID` (Argon2id), demonstrating awareness of the recommended variant while production code uses the vulnerable one:

```python
# In benchmark_argon2():
type=argon2.low_level.Type.ID,  # Correct variant in benchmarks only
```

**Data Flow:**
- `_hash()` ← `gen_opened_key(edata, salt)` ← election opening ceremony (election key derivation)
- `_hash()` ← `gen_vote_token(opened_key, pid, iid, salt)` ← every vote submission AND every tally decryption
- Secret material (election key data, voter identities, vote tokens) flows through Argon2d where memory access patterns are data-dependent

**Proof of Concept:**

An attacker with co-located access (e.g., shared hosting, VM co-residency, cache-timing analysis) can observe memory access patterns during key derivation operations.

**Impact:**

An attacker with co-located access (e.g., shared hosting, VM co-residency, cache-timing analysis) can observe these access patterns to recover:
- The `opened_key` (compromises all vote encryption for the election)
- Individual `vote_token` values (compromises individual vote secrecy)

RFC 9106 §4 explicitly recommends Argon2id for scenarios where side-channel resistance is needed, as it uses data-independent access patterns in the first pass. For an election system where vote secrecy is paramount, this is a significant weakness.

**Remediation:**

```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2id to hash the data, with default tuning parameters."

    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.ID,  # Argon2id: side-channel resistant
    )
```

> **Note:** Changing the Argon2 type will alter all derived keys. This requires a migration strategy: re-derive `opened_key` values for open elections or implement a version field to support both variants during transition.

#### ASVS-1171-HIGH-002: No Application-Level Memory Protection for Sensitive Cryptographic Material

**Severity:** HIGH  
**Gap Type:** Type A — No memory protection control exists anywhere in the codebase

**Affected Files and Functions:**

| Location | Sensitive Data in Memory | Duration |
|----------|--------------------------|----------|
| `crypto.py:_b64_vote_key()` L60–71 | 32-byte AES/HMAC key (`vote_key`) | Until GC |
| `crypto.py:create_vote()` L74–79 | Fernet key, plaintext votestring | Until GC |
| `crypto.py:decrypt_votestring()` L82–87 | Fernet key, decrypted vote | Until GC |
| `crypto.py:gen_opened_key()` L40–50 | BLAKE2b digest, Argon2 output | Until GC |
| `election.py:tally_issue()` L262–320 | **ALL** decrypted votes for an issue, all vote tokens, opened_key | Until function returns |
| `election.py:add_vote()` L247–260 | vote_token, plaintext votestring, opened_key | Until function returns |
| `tally.py:tally_election()` L103–145 | Accumulated results of all issues | Until function returns |

**Description:**

The application handles highly sensitive cryptographic material (encryption keys, plaintext votes, voter tokens) but implements no memory protection mechanisms. Python's immutable `bytes` and `str` objects cannot be overwritten, and no memory locking or zeroing is performed.

**Vulnerable Pattern (most critical instance):**

```python
# election.py:tally_issue() - ALL decrypted votes held simultaneously
votes = []
for mayvote in self.q_tally.fetchall():
    vote_token = crypto.gen_vote_token(md.opened_key, mayvote.pid, iid, mayvote.salt)
    row = self.q_recent_vote.first_row(vote_token)
    if row is None:
        continue
    votestring = crypto.decrypt_votestring(vote_token, mayvote.salt, row.ciphertext)
    votes.append(votestring)  # Plaintext accumulates, never zeroed
crypto.shuffle(votes)  # Still in memory, just reordered
```

**Specific Concerns:**

1. **Immutable `bytes` for keys:** Python `bytes` objects cannot be overwritten in place. Key material in `_b64_vote_key`, `gen_vote_token`, and `gen_opened_key` persists until garbage collected, with no guaranteed zeroing.
2. **Immutable `str` for plaintext votes:** Decrypted votestrings are Python `str` objects, which are also immutable and cannot be zeroed.
3. **No `mlock()`:** Sensitive memory pages can be swapped to disk, persisting key material and plaintext votes in swap files.
4. **Bulk accumulation during tally:** The entire election's decrypted votes exist in memory simultaneously, maximizing the exposure window.

**Data Flow:**

All sensitive cryptographic operations flow through functions that hold keys and plaintext in memory without protection, with the most critical exposure occurring during the tallying phase when all votes are decrypted simultaneously.

**Proof of Concept:**

A memory dump (via core dump, swap analysis, cold boot attack, or co-located process) during vote submission or tallying could recover plaintext votes, cryptographic keys, and voter-to-vote mappings.

**Impact:**

A memory dump (via core dump, swap analysis, cold boot attack, or co-located process) during vote submission or tallying could recover plaintext votes, cryptographic keys, and voter-to-vote mappings. For an election system operating at ASVS Level 3, this violates the requirement for sensitive data protection while in use.

**Remediation:**

```python
import ctypes
import mmap
import os

def secure_zero(data: bytearray):
    """Securely zero a mutable byte buffer."""
    ctypes.memset(ctypes.addressof((ctypes.c_char * len(data)).from_buffer(data)), 0, len(data))

def _b64_vote_key(vote_token: bytes, salt: bytes) -> tuple[bytes, bytearray]:
    """Key-stretch the vote_token, returning key and a cleanup handle."""
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,
        salt=salt,
        info=b'fernet_vote_key',
    )
    # Use mutable bytearray for key material
    vote_key = bytearray(keymaker.derive(vote_token))
    b64key = base64.urlsafe_b64encode(bytes(vote_key))
    secure_zero(vote_key)  # Zero the raw key immediately
    return b64key  # b64key still needed for Fernet, but has shorter lifetime

# For tallying, process and aggregate incrementally rather than accumulating all plaintext:
def tally_issue_secure(self, iid):
    """Process votes one at a time, zeroing each after processing."""
    # ... setup code ...
    tally_accumulator = vtypes.vtype_module(issue.type).create_accumulator()
    for mayvote in self.q_tally.fetchall():
        vote_token = crypto.gen_vote_token(md.opened_key, mayvote.pid, iid, mayvote.salt)
        row = self.q_recent_vote.first_row(vote_token)
        if row is None:
            continue
        votestring = crypto.decrypt_votestring(vote_token, mayvote.salt, row.ciphertext)
        tally_accumulator.add(votestring)
        del votestring  # Hint to GC (not guaranteed but reduces window)
    return tally_accumulator.result()
```

Additionally, deploy with:
- OS-level memory encryption (Intel TME, AMD SME/SEV)
- `mlockall(MCL_CURRENT | MCL_FUTURE)` to prevent swapping
- Disable core dumps for the application process

### Medium Severity

#### ASVS-1171-MED-003: Non-Constant-Time Comparison of Cryptographic Hash in Tamper Detection

**Severity:** MEDIUM  
**Gap Type:** Type C — Cryptographic comparison is performed but result quality is undermined by timing leakage

**Location:**
- **File:** `v3/steve/election.py`
- **Function:** `is_tampered()`
- **Line:** 363

**Related Functions Checked:**
- `tally.py:main()` (calls `is_tampered()` at line 167)
- `gen_opened_key()` (generates hash being compared)

**Description:**

The `is_tampered()` function uses Python's `!=` operator to compare cryptographic hashes, which performs a non-constant-time comparison that short-circuits on the first differing byte.

**Vulnerable Code:**

```python
def is_tampered(self, pdb):
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)
    
    # Python's != short-circuits on first differing byte
    return opened_key != md.opened_key
```

**Data Flow:**
- `is_tampered()` ← `tally.py:main()` (pre-tally check, line 167)
- `opened_key` is a 32-byte Argon2 hash of election configuration
- `md.opened_key` is the stored reference hash from election opening
- Python's `!=` on `bytes` objects uses `memcmp`-like comparison that exits on first mismatch

**Attack Scenario:**

If `is_tampered()` is exposed through any web endpoint (not just CLI), an attacker who can modify election data (e.g., via SQL injection or database access) could:
1. Make a targeted modification to election data
2. Observe response timing to determine how many leading bytes of the recomputed hash match the stored hash
3. Iteratively adjust modifications to forge data that produces a matching hash

While the Argon2 computation dominates the timing (making byte-level timing differences harder to observe), the principle of constant-time comparison for cryptographic material is a fundamental requirement.

**Proof of Concept:**

An attacker with database write access could iteratively modify election data and measure response times to determine hash prefix matches, potentially forging valid election configurations.

**Impact:**

Potential timing oracle for the tamper detection mechanism. Practical exploitation requires database write access and timing observation, but the fix is trivial and eliminates the risk entirely.

**Remediation:**

```python
import hmac

def is_tampered(self, pdb):
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)
    
    # Constant-time comparison prevents timing side-channels
    return not hmac.compare_digest(opened_key, md.opened_key)
```

## Positive Security Patterns

1. **CSPRNG Throughout:** All random generation (`gen_salt`, `create_id`, `shuffle`) uses the `secrets` module — the strongest available PRNG in Python.

2. **Multi-Layer Key Derivation:** The key derivation chain (BLAKE2b → Argon2 → HKDF → Fernet) provides defense-in-depth with each layer serving a specific purpose (size reduction, memory-hardness, domain separation, authenticated encryption).

3. **Per-Voter Salt Isolation:** Each `(pid, iid)` pair in `mayvote` has a unique 16-byte salt, preventing cross-voter correlation even with database access.

4. **Authenticated Encryption:** Fernet provides AES-CBC + HMAC-SHA256, ensuring both confidentiality and integrity of encrypted votes.

5. **State Machine Enforcement:** Election state transitions are enforced through assertions (`is_editable`, `is_open`, `is_closed`) before sensitive operations.

6. **Tamper Detection:** The `opened_key` mechanism provides integrity verification of election configuration before tallying, preventing post-opening manipulation.

7. **Vote Order Anonymization:** Cryptographically-secure Fisher-Yates shuffle of decrypted votes prevents ordering-based correlation.

8. **Transactional Integrity:** Bulk operations (salt generation, deletion) use explicit transactions.

## Architecture Observations

1. **Python Limitations for Memory Security:** Python's memory model (immutable `bytes`/`str`, non-deterministic garbage collection, no native `mlock` support) makes application-level memory protection inherently difficult. For ASVS Level 3 compliance, this may necessitate wrapping critical crypto operations in a C extension or using `cffi`/`ctypes` to manage sensitive memory directly.

2. **Tally Operation is the Highest-Risk Window:** During `tally_issue()`, the system simultaneously holds the election master key (`opened_key`), all voter tokens, and all decrypted votes in plaintext. This represents the maximum attack surface for memory-based exfiltration.

3. **Argon2 Variant Inconsistency:** The benchmark function uses `Type.ID` (correct) while production uses `Type.D` (vulnerable), suggesting this was either an oversight or a premature optimization for GPU resistance without considering side-channel risk.

4. **HKDF Info Label Mismatch:** `_b64_vote_key` uses `info=b'xchacha20_key'` but the actual cipher is Fernet/AES-128-CBC. While not a vulnerability, this indicates a planned but incomplete migration that could cause key reuse issues if XChaCha20 is introduced without re-derivation.
