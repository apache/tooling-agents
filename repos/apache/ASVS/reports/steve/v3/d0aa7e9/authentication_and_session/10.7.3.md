## Findings

### Medium Severity

#### ASVS-1073-MED-001: Complete Absence of Consent Management Functionality

**Severity:** MEDIUM

**Location:** 
- **File:** `v3/server/pages.py`
- **Functions:** `profile_page()` (lines ~554-560), `settings_page()` (lines ~563-569)
- **Related Functions:** `basic_info()` (session data retrieval)

**Description:**

The application provides no mechanism for users to review, modify, or revoke OAuth consents granted through the authorization server. While the application integrates with `oauth.apache.org` as an OAuth client, it lacks any consent management interface required by ASVS 10.7.3.

**Vulnerable Code:**

```python
# v3/server/pages.py, lines ~554-560
@APP.get('/profile')
@asfquart.auth.require  # Bare decorator means just require a valid session
@APP.use_template(TEMPLATES / 'profile.ezt')
async def profile_page():
    result = await basic_info()
    result.title = 'Profile'
    return result


# v3/server/pages.py, lines ~563-569
@APP.get('/settings')
@asfquart.auth.require  # Bare decorator means just require a valid session
@APP.use_template(TEMPLATES / 'settings.ezt')
async def settings_page():
    result = await basic_info()
    result.title = 'Settings'
    return result
```

**Supporting Code Context:**

```python
# basic_info() function showing available session data
async def basic_info():
    # ...
    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
    # ...
    basic.csrf_token = 'placeholder'
    return basic
```

**Data Flow:**

```
1. User authenticates via oauth.apache.org
2. Application receives OAuth token
3. Session created with uid/fullname/email
4. User visits /profile or /settings
5. Only basic_info() returned (uid, name, email)
6. NO consent data shown
7. NO revocation mechanism available
```

**Gap Classification:** **Type A** — Entry points exist (`/profile`, `/settings`) but have no consent management controls implemented.

**Proof of Concept:**

1. User authenticates via `oauth.apache.org` and grants consent to the STeVe application
2. User navigates to `/profile` — sees only their name/email, no consent information
3. User navigates to `/settings` — sees only basic info, no way to revoke consent
4. User searches all available routes — no `/revoke`, `/disconnect`, `/manage-consent`, or similar endpoint exists
5. User has no visibility into:
   - What scopes/permissions they have granted
   - When consent was given
   - How to revoke the application's access

**Impact:**

- **User Control Violation:** Users cannot exercise control over delegated authorization
- **Privacy Risk:** Users cannot review what data the application accesses on their behalf
- **Compliance Failure:** Direct violation of ASVS 10.7.3 Level 2 requirement
- **Trust Degradation:** No transparency into OAuth consent lifecycle
- **Revocation Gap:** Users cannot revoke application access without visiting the authorization server directly

**Remediation:**

**Immediate Actions:**

1. **Implement Consent Review Page:**

```python
@APP.get('/consents')
@asfquart.auth.require
@APP.use_template(TEMPLATES / 'consents.ezt')
async def consents_page():
    """Display active OAuth consents and allow revocation."""
    result = await basic_info()
    result.title = 'Manage Consents'

    s = await asfquart.session.read()
    if s:
        # Show what the application has been granted
        result.consents = [{
            'application': 'Apache STeVe Voting',
            'granted_at': s.get('auth_time'),
            'scopes': s.get('scopes', ['profile', 'email']),
            'authorization_server': 'oauth.apache.org',
        }]
    else:
        result.consents = []

    return result
```

2. **Implement Consent Revocation Endpoint:**

```python
@APP.post('/revoke-consent')
@asfquart.auth.require
async def revoke_consent():
    """Revoke OAuth consent and clear local session."""
    result = await basic_info()

    # 1. Call the AS token revocation endpoint (RFC 7009)
    s = await asfquart.session.read()
    if s and s.get('access_token'):
        async with httpx.AsyncClient() as client:
            await client.post(
                'https://oauth.apache.org/revoke',
                data={
                    'token': s['access_token'],
                    'token_type_hint': 'access_token',
                }
            )

    # 2. Clear the local session
    await asfquart.session.clear()

    _LOGGER.info(f'User[U:{result.uid}] revoked OAuth consent')
    await flash_success('Consent revoked. You have been logged out.')
    return quart.redirect('/', code=303)
```

3. **Store Consent Metadata at Authentication:**

```python
# During OAuth callback processing
session_data = {
    'uid': user_id,
    'fullname': user_name,
    'email': user_email,
    'access_token': access_token,
    'scopes': granted_scopes,
    'auth_time': datetime.utcnow().isoformat(),
    'authorization_server': 'oauth.apache.org'
}
```

## Positive Security Patterns

1. **TLS for OAuth Communication:** OAuth URLs use `https://` for both authorization and token endpoints (`main.py` lines 38-41), ensuring secure credential transmission

2. **Authentication Enforcement:** Routes consistently use `@asfquart.auth.require` decorators with role requirements, preventing unauthorized access

3. **Session-Based User Identification:** `basic_info()` properly reads session data and handles absent-session cases gracefully

4. **Audit Logging:** User actions are consistently logged with user identification (e.g., `f'User[U:{result.uid}]...'`), supporting security monitoring

5. **LDAPS Usage:** LDAP connections use `ldaps://` (TLS) for directory operations, protecting credential transmission

6. **Scaffolded Architecture:** Profile and settings pages exist as proper entry points, making consent management integration straightforward

---

## Architecture Observations

### OAuth Integration Architecture

1. **External Authorization Server:** The application delegates authentication entirely to `oauth.apache.org`. Consent is granted at the AS, but the application provides no post-authentication interface to manage it.

2. **Session Management:** User sessions store basic identity information (uid, fullname, email) but no consent-related metadata (scopes, grant time, token references).

3. **Missing Token Management:** The codebase shows no mechanism for storing or managing OAuth access/refresh tokens beyond the session, indicating token revocation infrastructure is absent at the framework integration level.

### Code Structure

4. **Profile/Settings Scaffolding:** Both `/profile` and `/settings` are clearly scaffolded routes ready for future functionality — they currently return only template title and session basics, making them natural candidates for consent management features.

5. **Missing API Module:** The `main.py` imports an `api` module which was not provided for review. While unlikely based on the application's architecture, consent management endpoints could potentially exist there.

### Security Implications

6. **Consent Lifecycle Gap:** No visibility into the consent lifecycle means users cannot exercise the principle of least privilege or revoke access when no longer needed.

7. **Compliance Risk:** The absence of consent management creates a direct compliance gap for ASVS Level 2 applications requiring OAuth/OIDC.
