### HIGH Severity

#### Finding ASVS-10413-HIGH-001: Authorization Code Grant Without Pushed Authorization Requests (PAR)

**Vulnerable Code**

```python
# main.py lines 37-42
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Description**

The application uses the OAuth authorization code grant type (confirmed by the `code=%s` parameter in `OAUTH_URL_CALLBACK`) but constructs authorization requests using the traditional approach of passing parameters directly in URL query strings. This violates ASVS 10.4.13 Level 3 requirement that the authorization code grant type must always be used together with Pushed Authorization Requests (PAR).

Per [RFC 9126 (PAR)](https://datatracker.ietf.org/doc/html/rfc9126), the correct PAR-based flow requires:

1. Client POSTs authorization parameters to the AS's PAR endpoint
2. AS validates parameters server-side and returns a `request_uri`
3. Client redirects user with only `request_uri` and `client_id`

The current implementation bypasses this security mechanism entirely.

**Data Flow Analysis**

```
User initiates login
  ↓
asfquart framework constructs authorization URL using OAUTH_URL_INIT template
  ↓
Browser redirected to: https://oauth.apache.org/auth?state=<STATE>&redirect_uri=<URI>
  ↓
Authorization parameters EXPOSED in URL query string (no PAR)
  ↓
Parameters logged in browser history, server logs, referrer headers
  ↓
Authorization code returned to: OAUTH_URL_CALLBACK with code parameter
  ↓
Token exchange completes
```

**Attack Vectors**

1. **Authorization Request Tampering**: Attacker with ability to observe/modify authorization request can:
   - Manipulate `redirect_uri` parameter (if AS validation is insufficient)
   - Inject additional authorization parameters
   - Exploit authorization request replay

2. **Parameter Exposure**: Authorization parameters transmitted via URL expose sensitive data through:
   - Browser history (persistent on user device)
   - Server access logs (on proxies, load balancers, AS)
   - Referrer headers (leaked to third-party sites)
   - Shoulder surfing (visible in address bar)

3. **Mix-Up Attacks**: Without server-side pre-validation of authorization requests, the client is more vulnerable to AS mix-up attacks where an attacker tricks the client into using a malicious AS.

**Proof of Concept**

```python
# Current (vulnerable) flow:
# Step 1: User clicks login
# Step 2: Application redirects to:
"https://oauth.apache.org/auth?state=abc123&redirect_uri=https://steve.apache.org/callback"

# Attacker scenarios:
# Scenario A: Browser history extraction
#   - Attacker with device access retrieves full authorization URL from history
#   - State parameter and redirect_uri exposed

# Scenario B: Open redirect chain exploitation
#   - Attacker crafts malicious redirect_uri (if validation weak)
#   - Authorization code delivered to attacker-controlled endpoint

# Scenario C: Server log analysis
#   - Attacker compromises proxy/load balancer
#   - Extracts authorization parameters from access logs
```

3. **Implement PAR Flow**: Add PAR request handling to OAuth flow:

```python
import httpx
import secrets
import hashlib
import base64

def generate_s256_challenge(verifier: str) -> str:
    """Generate S256 code challenge from verifier."""
    digest = hashlib.sha256(verifier.encode('utf-8')).digest()
    return base64.urlsafe_b64encode(digest).decode('utf-8').rstrip('=')

async def initiate_authorization(state: str, redirect_uri: str, code_verifier: str):
    """Push authorization request to PAR endpoint, then redirect.
    
    Implements RFC 9126 (PAR) flow:
    1. POST authorization parameters to PAR endpoint
    2. Receive request_uri from AS
    3. Redirect user with only client_id and request_uri
    """
    
    code_challenge = generate_s256_challenge(code_verifier)
    
    # Step 1: POST parameters to PAR endpoint (server-to-server)
    async with httpx.AsyncClient() as client:
        par_response = await client.post(
            OAUTH_PAR_ENDPOINT,
            data={
                'client_id': CLIENT_ID,
                'redirect_uri': redirect_uri,
                'state': state,
                'response_type': 'code',
                'code_challenge': code_challenge,
                'code_challenge_method': 'S256',
                'scope': 'openid profile email',
            },
            # Client authentication (mTLS or private_key_jwt recommended)
            auth=(CLIENT_ID, CLIENT_SECRET),
            timeout=10.0,
        )
    
    if par_response.status_code != 201:
        raise OAuthError(f'PAR request failed: {par_response.status_code}')
    
    par_data = par_response.json()
    request_uri = par_data['request_uri']
    expires_in = par_data.get('expires_in', 60)
    
    # Store request_uri with expiration for validation
    await store_par_request(request_uri, expires_in, state, code_verifier)
    
    # Step 2: Redirect with only client_id and request_uri (minimal exposure)
    auth_url = f'{OAUTH_AUTH_ENDPOINT}?client_id={CLIENT_ID}&request_uri={request_uri}'
    return redirect(auth_url)
```

## Architecture Observations

### OAuth Role Analysis

**Current Architecture:**

```
┌─────────────────┐         ┌──────────────────────┐         ┌─────────────────┐
│   End User      │         │   steve Application  │         │ oauth.apache.org│
│  (Resource      │◄───────►│   (OAuth Client/RP)  │◄───────►│  (Authorization │
│   Owner)        │         │                      │         │     Server)     │
└─────────────────┘         └──────────────────────┘         └─────────────────┘
                                      │
                                      ▼
                            ┌──────────────────────┐
                            │  asfquart Framework  │
                            │  (OAuth Abstraction) │
                            └──────────────────────┘
```

**Key Observations:**

1. **OAuth Client Role**: This application is an OAuth client (relying party in OIDC terminology), NOT an authorization server. The AS is external (`oauth.apache.org`). ASVS 10.4.13 compliance requires the **client** to implement PAR on its end, and the AS to support and enforce it.

2. **Framework Abstraction Layer**: The actual OAuth flow mechanics are abstracted within the `asfquart` framework (not provided for review). The application only configures:
   - `OAUTH_URL_INIT`: Authorization endpoint URL template
   - `OAUTH_URL_CALLBACK`: Token endpoint URL template
   
   This limited configuration surface suggests the framework may not currently support PAR, requiring framework-level changes.

3. **Configuration Limitations**: Only two OAuth-related configuration values are exposed. Missing configuration options include:
   - PAR endpoint URL
   - PKCE parameters (`code_challenge_method`)
   - Client authentication method
   - Token validation settings
   - AS metadata endpoint

4. **Dependency on External AS**: Compliance depends on coordination with `oauth.apache.org` operators to:
   - Enable PAR endpoint
   - Configure client to require PAR
   - Reject non-PAR authorization requests for this client

- **Validation Timing**: Authorization parameters validated by AS only after user interaction, not before
