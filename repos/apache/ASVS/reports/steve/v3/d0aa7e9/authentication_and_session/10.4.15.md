## Findings

### Medium Severity

#### ASVS-10415-MED-001: Server-Side OAuth Client Does Not Use PAR or JAR for Authorization Requests

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/main.py`
- **Lines:** 38–42
- **Function:** `create_app()`

**Related Functions Checked:**
- `asfquart.generics.OAUTH_URL_INIT` (configuration)
- `asfquart.generics.OAUTH_URL_CALLBACK` (configuration)
- OAuth flow implementation in `asfquart` framework (not visible in audit scope)

**Description:**

This application is a **server-side client** (runs on the ASF server, not on end-user devices). The OAuth authorization flow uses a simple redirect pattern where all authorization parameters are passed through the user's browser as URL query parameters. ASVS 10.4.15 requires that for server-side clients, the authorization server must be able to ensure that authorization request parameters (particularly `authorization_details`) originate from the client backend without user tampering, typically through PAR or JAR mechanisms.

The current implementation does not use either PAR (RFC 9126) or JAR (RFC 9101), meaning:
- All authorization parameters travel through the browser without integrity protection
- The authorization server cannot cryptographically verify parameter origin
- Users could potentially intercept and modify authorization request parameters
- If `authorization_details` (RFC 9396) were added for fine-grained authorization, they would be vulnerable to tampering

**Vulnerable Code:**

```python
# v3/server/main.py, lines 38-42
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**

```
1. Client Backend (main.py)
   ↓ [builds URL with parameters]
2. HTTP Redirect to Browser
   ↓ [state=%s&redirect_uri=%s in URL]
3. User's Browser
   ↓ [user can intercept/modify here]
4. Authorization Server (oauth.apache.org)
   ↓ [cannot verify parameter integrity]
```

**Missing Control:** No PAR or JAR mechanism to ensure authorization request parameters are transmitted directly from client backend to authorization server with integrity protection.

**Proof of Concept:**

An authenticated user could intercept the redirect to `oauth.apache.org` using browser developer tools or a proxy and modify query parameters:

```http
# Original redirect URL constructed by server:
https://oauth.apache.org/auth?state=abc123&redirect_uri=https://steve.apache.org/callback

# User-modified URL (redirect_uri changed):
https://oauth.apache.org/auth?state=abc123&redirect_uri=https://attacker.example.com/steal

# If authorization_details were added in the future:
# Original:
https://oauth.apache.org/auth?state=abc123&redirect_uri=...&authorization_details={"type":"election_admin","actions":["manage","tally"]}

# User-modified:
https://oauth.apache.org/auth?state=abc123&redirect_uri=...&authorization_details={"type":"election_admin","actions":["manage","tally","delete_all"]}
```

**Impact:**

- **Integrity Risk:** The authorization server cannot distinguish between authorization parameters originating from the client backend vs. those modified by the user during the browser redirect
- **Future Risk:** If fine-grained authorization requests (`authorization_details`) are introduced for election management features, they would be vulnerable to user tampering
- **Current Exposure:** Even current OAuth flow parameters (`redirect_uri`, `state`) are exposed to manipulation in transit through the browser, though AS-side validation should mitigate most risks
- **Compliance:** Does not meet ASVS 10.4.15 Level 3 requirements for server-side clients

**Remediation:**

**Option 1: Implement Pushed Authorization Requests (PAR) per RFC 9126:**

```python
# main.py - Updated OAuth configuration with PAR support
import httpx

# PAR endpoint for the authorization server
OAUTH_PAR_ENDPOINT = 'https://oauth.apache.org/par'

async def initiate_authorization(state, redirect_uri, client_id, client_secret):
    """Push authorization request directly to AS, get request_uri."""
    async with httpx.AsyncClient() as client:
        par_response = await client.post(
            OAUTH_PAR_ENDPOINT,
            data={
                'client_id': client_id,
                'client_secret': client_secret,
                'state': state,
                'redirect_uri': redirect_uri,
                'response_type': 'code',
                # authorization_details would go here if needed
            },
            # Or use mTLS client authentication
        )
        par_data = par_response.json()
        request_uri = par_data['request_uri']

    # Only the opaque request_uri goes through the browser
    return f'https://oauth.apache.org/auth?client_id={client_id}&request_uri={request_uri}'
```

**Option 2: Implement JAR (RFC 9101) by signing authorization request parameters:**

```python
import jwt

def build_jar_request(params, client_private_key, client_id):
    """Build a JWT-Secured Authorization Request."""
    signed_request = jwt.encode(
        {**params, 'iss': client_id, 'aud': 'https://oauth.apache.org'},
        client_private_key,
        algorithm='RS256',
    )
    return f'https://oauth.apache.org/auth?client_id={client_id}&request={signed_request}'
```

**Prerequisites:**
- Coordinate with Apache Infrastructure team to confirm `oauth.apache.org` supports PAR or JAR
- Obtain client credentials or signing keys as appropriate
- Update `asfquart` framework to support the chosen mechanism

## Positive Security Patterns

1. **HTTPS for OAuth URLs**: Both `OAUTH_URL_INIT` and `OAUTH_URL_CALLBACK` use `https://` scheme, protecting parameters in transit from network-level attackers.

2. **State Parameter Usage**: The `OAUTH_URL_INIT` template includes `state=%s`, indicating CSRF protection is at least structurally present in the OAuth flow (implementation in `asfquart` framework).

3. **Server-side Token Exchange**: The `OAUTH_URL_CALLBACK` pattern (`token?code=%s`) suggests the authorization code is exchanged server-to-server, not in the browser, which is correct for a confidential client.

4. **Role-based Access Control**: The `asfquart.auth.require` decorator with `R.committer` and `R.pmc_member` requirements shows differentiated authorization levels derived from OAuth-provided identity.

5. **Secure LDAP**: `asf-load-ldap.py` uses `ldaps://` (LDAP over TLS) for the connection to the directory server, protecting credential transmission.

6. **Framework-based OAuth**: Delegating OAuth implementation to the `asfquart` framework reduces the risk of custom implementation errors, assuming the framework follows OAuth security best practices.

---

## Architecture Observations

1. **OAuth Client, Not Authorization Server**: This application is an OAuth relying party/client, not an authorization server. The AS is the external `oauth.apache.org`. ASVS 10.4.15 describes AS responsibilities, but the client's architecture determines whether the AS *can* enforce those requirements.

2. **Framework Abstraction**: The actual OAuth flow implementation (redirect construction, callback handling, state validation, token exchange) is encapsulated in the `asfquart` framework, which is not included in the source files under review. A complete audit would require examining `asfquart.generics` and the OAuth callback handler.

3. **Simple OAuth Flow**: The URL template pattern (`state=%s&redirect_uri=%s`) suggests a basic OAuth 2.0 Authorization Code flow without advanced security features (PAR, JAR, PKCE not visible in configuration).

4. **Trust Boundary**: The browser redirect between client and AS is the critical trust boundary for this requirement. All authorization request parameters currently cross this boundary without cryptographic integrity protection from the client.

5. **External Authorization Server**: Remediation depends on capabilities of the external AS (`oauth.apache.org`). Implementation requires coordination with Apache Infrastructure team.

## Recommendations Summary

### Immediate Actions

1. **Coordinate with Apache Infrastructure** to determine if `oauth.apache.org` supports PAR (RFC 9126) or JAR (RFC 9101) endpoints
2. **Document current OAuth flow security controls** implemented in the `asfquart` framework (state validation, redirect_uri validation, PKCE)
3. **Assess whether `authorization_details`** (RFC 9396) is planned for future use in election management authorization

### Short-term Actions (1-3 months)

1. **Audit `asfquart` framework** OAuth implementation for compliance with OAuth 2.0 Security Best Current Practice (BCP 212)
2. **Implement PAR** (RFC 9126) if supported by `oauth.apache.org`, ensuring all authorization parameters are transmitted backend-to-backend
3. **Update OAuth configuration** in `main.py` to use PAR or JAR mechanisms once AS support is confirmed

### Long-term Actions (3-6 months)

1. **Implement JAR** (RFC 9101) as fallback if PAR is not available, providing cryptographic integrity for authorization requests
2. **Document OAuth security architecture** including trust boundaries, parameter validation points, and security controls
3. **Establish security review process** for any future additions of `authorization_details` or other authorization parameters
