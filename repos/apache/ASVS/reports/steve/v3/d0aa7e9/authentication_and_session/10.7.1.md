### Context

This application is an **OAuth client** delegating authentication to an external authorization server (`oauth.apache.org`). The audit focuses on how the client configures and participates in the OAuth flow with respect to consent enforcement. One significant finding was identified where the OAuth configuration deliberately bypasses OIDC (which provides standardized consent mechanisms) and omits all consent-enforcing parameters, making it impossible to verify or guarantee that the external AS prompts for user consent on each authorization.

## Findings

### Medium Severity

#### ASVS-1071-MED-001: Missing Consent Enforcement Parameters in OAuth Authorization Flow

| Attribute | Detail |
|-----------|--------|
| **Severity** | **MEDIUM** |
| **Gap Type** | Type A — No consent-enforcing control exists in the OAuth flow |
| **File** | `v3/server/main.py` |
| **Function** | `create_app()` |
| **Lines** | 36–42 |
| **Related Functions** | OAuth initialization in `asfquart.generics` |

**Description:**

The OAuth authorization URL configuration omits all consent-enforcing parameters and explicitly disables OIDC support. This makes it impossible to verify or guarantee that the external authorization server prompts users for consent on each authorization request.

**Vulnerable Code:**

```python
# Avoid OIDC
import asfquart.generics

asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**

1. `create_app()` configures OAuth authorization URL → no `prompt`, `consent_prompt`, or `scope` parameters included
2. Comment explicitly states `# Avoid OIDC` — removes access to the standardized `prompt=consent` mechanism defined in OpenID Connect Core §3.1.2.1
3. When `asfquart` redirects a user to the AS for authorization, the AS receives no instruction to prompt for consent
4. The AS (`oauth.apache.org`) may silently issue tokens for returning users without displaying a consent screen
5. No application-side verification that consent was actually obtained during the callback

**Why This Matters:**

ASVS 10.7.1 requires consent for each authorization request. The second clause states: *"If the identity of the client cannot be assured, the authorization server must always explicitly prompt the user for consent."* Since the OAuth URL template contains no `client_id` and no `prompt` parameter, the client relies entirely on the AS's default behavior. Without OIDC's `prompt=consent`, there is no standardized mechanism to:

- Request the AS to display a consent screen
- Verify via the ID token or token response that consent was freshly granted
- Distinguish between a first-time authorization and a silently re-issued token

For a voting system, silent re-authorization without explicit consent could enable scenarios where a user's browser is redirected through the OAuth flow (e.g., via CSRF on the login initiation endpoint) and obtains a valid session without the user's awareness.

**Proof of Concept:**

If an attacker can trigger a navigation to the OAuth init URL for a user who previously authorized the application:

```
https://oauth.apache.org/auth?state=ATTACKER_STATE&redirect_uri=https://steve.apache.org/callback
```

The AS may complete the flow silently (no consent prompt), issuing a code that gets exchanged for a session. The user may not realize a new session was established.

**Impact:**

- Users may be authorized without being aware of individual authorization events
- No audit trail distinguishing user-initiated vs. silently-completed OAuth flows
- Reduced user agency over their delegated authorization
- Potential for unauthorized session establishment in a voting system

**Remediation:**

Switch to OIDC (or add consent parameters if the AS supports them in plain OAuth):

```python
def create_app():
    import asfquart.generics

    # Use OIDC with explicit consent prompting
    asfquart.generics.OAUTH_URL_INIT = (
        'https://oauth.apache.org/auth?'
        'response_type=code&'
        'scope=openid%20profile%20email&'
        'prompt=consent&'
        'state=%s&'
        'redirect_uri=%s'
    )
    asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
    
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    import pages
    import api
    return app
```

If OIDC adoption is not feasible, coordinate with the `oauth.apache.org` operators to confirm that consent is **always** prompted for the STeVe client registration, and document this as a compensating control.

## Positive Security Patterns

1. **State parameter is included**: The OAuth init URL includes `state=%s`, which (assuming the framework validates it) protects against CSRF on the authorization callback. This is a prerequisite for meaningful consent.

2. **TLS enforced on LDAP**: `ldaps://ldap-us.apache.org/` uses LDAP over TLS, protecting credential transmission.

3. **Authentication decorators consistently applied**: All state-changing endpoints use `@asfquart.auth.require`, ensuring a valid session (and thus a completed OAuth flow) exists before operations proceed.

4. **External AS delegation**: Using Apache's centralized OAuth service (`oauth.apache.org`) rather than a custom AS is a strong architectural choice — the consent behavior can be configured centrally for all ASF applications.

---

## Architecture Observations

1. **Client vs. AS boundary**: This application is an OAuth **client**, not an authorization server. ASVS 10.7.1 primarily governs AS behavior. The finding reported above addresses the client's responsibility to **request** consent and to **verify** that it was obtained, which is the extent of what a client can do.

2. **OIDC avoidance is a conscious decision**: The `# Avoid OIDC` comment suggests an intentional architectural choice. However, this removes access to standardized consent mechanisms (`prompt` parameter, `auth_time` claim for verifying freshness, `acr` for authentication context). The rationale for this decision should be documented.

3. **First-party application context**: As an ASF application authenticating ASF members against ASF infrastructure, the consent model is arguably simpler than third-party OAuth. The AS operator and the application operator are the same organization. This reduces (but does not eliminate) the risk of unauthorized authorization.

4. **Session longevity**: Session expiration configuration is not visible in the provided code. Long-lived sessions effectively bypass re-consent since users won't be redirected through the OAuth flow again. The session management configuration in `asfquart` should be reviewed to ensure session lifetimes are appropriate for a voting system.

## Recommendations Summary

### Immediate Actions (Priority 1)

| Recommendation | Effort | Impact |
|----------------|--------|--------|
| Coordinate with `oauth.apache.org` operators to confirm consent is always prompted for the STeVe client, and document this as a compensating control | Low | High |

### Short-term Actions (Priority 2-3)

| Recommendation | Effort | Impact |
|----------------|--------|--------|
| Add `prompt=login` or `prompt=consent` to the OAuth authorization URL (if supported by the AS) to ensure active user participation in each authorization | Low | High |
| Evaluate migrating from plain OAuth to OIDC to gain access to standardized consent mechanisms, `auth_time` verification, and `prompt` parameter support | Medium | High |
| Add `scope` parameter to the authorization URL so the consent screen (if displayed) can show the user what permissions are being requested | Low | Medium |

### Long-term Actions (Priority 4-5)

| Recommendation | Effort | Impact |
|----------------|--------|--------|
| In the OAuth callback handler (within `asfquart`), log whether the authorization was freshly consented vs. silently completed, for audit trail purposes | Medium | Medium |
