### Audit Scope

This audit evaluates ASVS 10.7.2 compliance for the STeVe voting application's OAuth integration. The application acts as an OAuth **client**, delegating authentication to the ASF's OAuth authorization server at `oauth.apache.org`. The audit focuses on whether the client's OAuth requests provide sufficient information for the authorization server to present meaningful consent to users, and whether the application's own authorization model is reflected in the OAuth flow.

## Findings

### Medium Severity

#### ASVS-1072-MED-001: OAuth Authorization Request Missing Scope Parameter Prevents Meaningful User Consent

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `create_app`
- **Lines:** 37–41

**Related Functions Checked:**
- `asfquart.session.read()` in `pages.py:79-93` (consumes OAuth session data)
- `basic_info()` in `pages.py:84-87` (accesses uid, fullname, email)

**Description:**

The OAuth authorization request URL template includes only `state` and `redirect_uri` parameters. It does **not** include a `scope` parameter (e.g., `openid`, `profile`, `email`), nor a `client_id` parameter in the visible URL template. Without scopes, the authorization server at `oauth.apache.org` cannot present the user with information about **what data or permissions** the STeVe application is requesting.

ASVS 10.7.2 requires that the consent prompt presents "the nature of the requested authorizations (typically based on scope)." By omitting scopes entirely, the authorization server cannot fulfill this requirement regardless of its own implementation quality.

**Vulnerable Code:**

```python
# v3/server/main.py, lines 37-41
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**

1. User clicks login → `asfquart` framework formats `OAUTH_URL_INIT` with state and redirect_uri only
2. Browser redirects to `oauth.apache.org/auth?state=...&redirect_uri=...`
3. Authorization server has no `scope` parameter to determine what to present in consent
4. User consents without understanding that `uid`, `fullname`, and `email` will be shared (as consumed at `pages.py:84-87`)

**Evidence of Data Consumed Post-Consent:**

```python
# pages.py lines 84-87 — personal data accessed from OAuth session
basic.update(
    uid=s['uid'],
    name=s['fullname'],
    email=s['email'],
)
```

**Proof of Concept:**

1. Navigate to STeVe login page
2. Intercept OAuth redirect to `oauth.apache.org/auth`
3. Observe URL parameters: only `state` and `redirect_uri` present
4. Authorization server displays generic consent (or no consent) without specifying data access
5. After authentication, application accesses `uid`, `fullname`, `email` from session without these being declared in scopes

**Impact:**

- Users cannot make informed consent decisions about personal data sharing (uid, name, email)
- The authorization server's consent screen cannot distinguish between minimal authentication and the full profile data the application actually consumes
- Violates the principle that consent should reflect "the nature of the requested authorizations"
- Non-compliance with ASVS 10.7.2 requirement for "sufficient and clear information about what is being consented to"

**Remediation:**

Specify explicit OAuth scopes in the authorization URL to communicate the client's data requirements to the authorization server:

```python
# main.py - create_app()
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?'
    'response_type=code&'
    'client_id=steve-voting&'
    'scope=openid+profile+email&'
    'state=%s&'
    'redirect_uri=%s'
)
```

If the ASF OAuth server does not support standard scopes, coordinate with the OAuth server administrators to implement scope-based consent presentation per RFC 6749 §3.3.

#### ASVS-1072-MED-002: Deliberate OIDC Avoidance Eliminates Standardized Consent and Identity Claims

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `create_app`
- **Lines:** 35–36

**Related Functions Checked:**
- Default `asfquart` framework OAuth/OIDC configuration (overridden)
- `OAUTH_URL_INIT` and `OAUTH_URL_CALLBACK` configuration (lines 37-41)

**Description:**

The application deliberately overrides the framework's default OAuth/OIDC URLs to "avoid OIDC," replacing them with a custom ASF OAuth endpoint. OIDC provides standardized consent mechanisms including well-defined scopes (`openid`, `profile`, `email`), standardized claims, and the `prompt=consent` parameter that gives the authorization server explicit signals to display consent screens with structured claim information.

By bypassing OIDC in favor of a custom OAuth flow, the application loses:

1. Standardized `scope` values that map to well-defined user data categories
2. The `prompt=consent` mechanism to force consent re-display
3. ID Token claims that document what data was authorized
4. The client identification via registered `client_id` that the AS presents to users ("STeVe Voting System is requesting access to...")

**Vulnerable Code:**

```python
# v3/server/main.py, lines 35-36
### is this really needed right now?
# Avoid OIDC
import asfquart.generics
```

**Data Flow:**

1. Default `asfquart` framework likely supports OIDC with proper scopes → Overridden in `create_app()`
2. Custom OAuth URLs bypass OIDC standardized consent flow
3. Authorization server receives non-OIDC request → falls back to potentially minimal or absent consent UI
4. No `id_token` returned → no cryptographic proof of what was consented to

**Proof of Concept:**

1. Review `asfquart` framework default OIDC configuration
2. Observe override in `main.py:35-36` with comment "Avoid OIDC"
3. Compare OAuth flow to OIDC standard flow:
   - Missing: `response_type=code`, `client_id`, `scope`, `prompt=consent`
   - Missing: ID Token in response
   - Missing: Standardized claims structure
4. Authorization server cannot leverage OIDC consent presentation features

**Impact:**

- The authorization server cannot leverage OIDC-standard consent presentation (identity of the application, requested scopes with descriptions, authorization lifetime)
- No standardized mechanism to request or verify that consent was actually displayed
- Users may be authenticated without any consent prompt, or with a generic "allow this application" prompt that doesn't specify the STeVe application name or data access
- No cryptographic proof (ID Token) of what was consented to
- Non-compliance with ASVS 10.7.2 requirement for presenting "the identity of the authorized application"

**Remediation:**

Re-evaluate the OIDC bypass decision. If the ASF's OAuth server supports OIDC:

```python
def create_app():
    # Use standard OIDC flow for proper consent management
    # Do NOT override asfquart.generics.OAUTH_URL_INIT
    # Let the framework use its default OIDC endpoints
    
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    # If custom endpoints are needed, ensure OIDC parameters are preserved:
    # asfquart.generics.OAUTH_URL_INIT = (
    #     'https://oauth.apache.org/auth?'
    #     'response_type=code&'
    #     'client_id=steve&'
    #     'scope=openid+profile+email&'
    #     'prompt=consent&'
    #     'state=%s&redirect_uri=%s'
    # )
    
    import pages
    import api
    return app
```

#### ASVS-1072-MED-003: Authorization Tiers Not Reflected in OAuth Consent — Election Management Privileges Granted Without Specific Consent

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** Multiple authorization-protected endpoints
- **Lines:** 518, 540, 561, 580, 632, 476

**Affected Entry Points:**

| Route | Auth Level | Privilege | Lines |
|-------|-----------|-----------|-------|
| `/do-create-election` | `R.pmc_member` | Create elections | 518 |
| `/do-open/<eid>` | `R.committer` | Open elections | 540 |
| `/do-close/<eid>` | `R.committer` | Close elections | 561 |
| `/do-add-issue/<eid>` | `R.committer` | Add ballot issues | 580 |
| `/do-delete-issue/<eid>/<iid>` | `R.committer` | Delete ballot issues | 632 |
| `/do-vote/<eid>` | `R.committer` | Submit votes | 476 |

**Related Functions Checked:**
- `OAUTH_URL_INIT` configuration in `main.py:37-39` (single OAuth flow for all privilege tiers)
- `@asfquart.auth.require({R.committer})` decorators (Tier 1 authorization)
- `@asfquart.auth.require({R.pmc_member})` decorators (Tier 2 authorization)

**Description:**

The application enforces a two-tiered authorization model internally:

- **Tier 1:** `R.committer` — voting, election viewing, and election management
- **Tier 2:** `R.pmc_member` — election creation (higher privilege)

However, the OAuth consent flow is identical for all users regardless of their eventual privilege tier. A user who logs in solely to view elections goes through the same consent flow as an election administrator who will create and manage elections affecting other voters. ASVS 10.7.2 requires that consent presents "the nature of the requested authorizations."

The single OAuth flow means users are never informed during consent that:

- Their ASF membership/group data will determine election management privileges
- The application will query LDAP group membership to determine PMC membership
- Authentication grants potential access to election administration functions

**Vulnerable Code:**

```python
# main.py line 37-39 — single OAuth configuration for all users
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
```

```python
# pages.py — two different authorization levels, same OAuth flow
@asfquart.auth.require({R.committer})  # Tier 1
async def do_vote_endpoint(election): ...

@asfquart.auth.require({R.pmc_member})  # Tier 2 — different privilege, same consent
async def do_create_endpoint(): ...
```

**Data Flow:**

1. User initiates login (regardless of intended privilege level)
2. Redirected to `oauth.apache.org/auth` with no scope differentiation
3. Authorization server presents generic consent (if any)
4. User authenticates and returns to application
5. Application queries LDAP to determine privilege tier
6. User granted committer or PMC member privileges without tier-specific consent

**Proof of Concept:**

1. Log in as committer user (Tier 1)
2. Log in as PMC member user (Tier 2)
3. Compare OAuth authorization requests: identical URLs, no scope differentiation
4. PMC member can access `/do-create-election` without having consented to elevated privileges
5. Authorization server consent screen (if displayed) does not distinguish between basic voter and election administrator

**Impact:**

- Users are not informed that their LDAP group membership determines administrative privileges in the application
- A PMC member who logs in is not specifically consented to the elevated election management capabilities
- No scope differentiation means the AS consent screen cannot distinguish between basic voter access and full administrative access
- Non-compliance with ASVS 10.7.2 requirement for presenting "the nature of the requested authorizations"
- Violates principle of least privilege in consent: users consent to maximum possible privileges rather than actual needed privileges

**Remediation:**

Define distinct OAuth scopes or Rich Authorization Request (RAR) details that map to application privilege tiers, and request them contextually:

```python
# Define scope sets for different authorization contexts
SCOPE_VOTER = 'openid profile email steve:vote'
SCOPE_ADMIN = 'openid profile email steve:vote steve:manage'

# When redirecting to admin functions, use elevated scopes
# Or implement step-up consent for management operations
```

If the ASF OAuth server doesn't support custom scopes, implement an application-level consent screen before granting elevated privileges:

```python
@APP.get('/admin')
@asfquart.auth.require({R.committer})
async def admin_page():
    result = await basic_info()
    # Check if user has consented to admin-level access
    s = await asfquart.session.read()
    if not s.get('admin_consent_granted'):
        return quart.redirect('/consent/admin-access')
    # ... proceed with admin page
```

Alternatively, implement explicit authorization lifetime disclosure:

```python
# Display consent screen showing:
# - Specific privileges being granted (vote vs. manage elections)
# - Session duration
# - Data access (uid, name, email, group membership)
```

## Positive Security Patterns

The following security controls are properly implemented and demonstrate good security practices:

### 1. HTTPS for OAuth Endpoints

**Location:** `main.py:37-41`

Both `OAUTH_URL_INIT` and `OAUTH_URL_CALLBACK` use `https://` URLs, ensuring OAuth exchanges are encrypted in transit and protecting against interception attacks.

```python
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

### 2. State Parameter Included

**Location:** `main.py:37-39`

The OAuth init URL includes `state=%s` parameter, which (when properly validated) prevents CSRF attacks on the OAuth flow by binding the authorization request to the user's session.

### 3. Session-Based Identity

**Location:** `pages.py:79-93`

User identity is read from server-side session (`asfquart.session.read()`), not from client-supplied tokens, preventing token manipulation attacks.

```python
async def basic_info():
    s = await asfquart.session.read()
    basic = {
        'uid': s.get('uid', 'anonymous'),
        'name': s.get('fullname', 'Anonymous'),
        'email': s.get('email', 'none'),
        # ...
    }
```

### 4. Authorization Decorators Consistently Applied

**Location:** `pages.py` (multiple routes)

All state-changing routes have `@asfquart.auth.require` decorators — the authorization framework exists and is consistently applied to protect sensitive operations.

### 5. LDAPS for Credential Source

**Location:** `asf-load-ldap.py:35`

The LDAP connection uses `ldaps://` protocol, protecting voter eligibility data and group membership information in transit.

## Architecture Observations

### 1. First-Party OAuth Usage

This is a first-party ASF application using ASF's own OAuth server. The ASVS chapter acknowledges that "OAuth and OIDC were initially designed for third-party applications" and that first-party usage "may introduce new security challenges." The consent requirements still apply but may be addressed differently in first-party contexts where the authorization server and resource server are operated by the same organization.

**Implication:** While consent is still required per ASVS 10.7.2, the implementation approach may differ from third-party OAuth scenarios. The ASF may choose to implement consent at the authorization server level with organizational context.

### 2. Separation of Authentication and Authorization

Authentication is delegated to OAuth (`oauth.apache.org`), while authorization is handled internally via LDAP groups and database tables. This separation means the OAuth consent screen cannot fully represent the application's authorization model without custom scopes or RAR.

**Implication:** The OAuth layer handles "who you are" (authentication), but "what you can do" (authorization) is determined post-authentication via LDAP queries. This architectural split requires either custom OAuth scopes or application-level consent to satisfy ASVS 10.7.2.

### 3. No Authorization Lifetime Information

The OAuth flow doesn't communicate session/authorization lifetime to the user. Session duration is handled entirely server-side with no user visibility during consent.

**Implication:** ASVS 10.7.2 requires presenting "the lifetime of these authorizations" when applicable. The current implementation provides no mechanism for users to understand how long their authorization will remain valid.

### 4. LDAP as Source of Truth for Privileges

**Location:** `asf-load-ldap.py`

The application uses LDAP to determine voter eligibility and PMC membership, which then drives internal authorization decisions. This external dependency means privilege changes in LDAP are reflected in the application without re-consent.

**Implication:** Users are not informed during consent that their LDAP group membership will determine their application privileges, and changes to LDAP groups can alter their privileges without new consent.
