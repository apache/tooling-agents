## Findings

### Applicability Assessment

**ASVS 10.4.1 Requirement Not Applicable**

After comprehensive analysis of all provided source files, the application under audit does not implement an OAuth Authorization Server component. The codebase exclusively operates as an **OAuth client/relying party**, delegating all authorization server responsibilities to external ASF OAuth infrastructure via the `asfquart.auth` framework.

#### Evidence

**OAuth Client Pattern in `pages.py`:**
```python
# pages.py - Client-side auth consumption only
from asfquart.auth import Requirements as R

@APP.post('/do-create-election')
@asfquart.auth.require({R.pmc_member})  # OAuth client decorator
async def do_create_endpoint():
    ...
```

**Absence of Authorization Server Components:**
- No OAuth client registration tables
- No redirect URI allowlist storage
- No authorization code generation/validation
- No token issuance endpoints
- No `/authorize` or `/token` endpoints

**Internal Application Redirects Only:**
```python
# All redirects are to hardcoded paths or path templates with DB-sourced IDs:
return quart.redirect(f'/manage/{election.eid}', code=303)  # eid from DB
return quart.redirect('/voter', code=303)                    # hardcoded
return quart.redirect(f'/vote-on/{election.eid}', code=303)  # eid from DB
```

No user-controllable redirect targets exist. No `?redirect=`, `?next=`, or `redirect_uri` parameter processing is present.

## Positive Security Patterns

### 1. Appropriate Separation of Concerns
The OAuth Authorization Server responsibility is correctly delegated to the ASF infrastructure rather than reimplemented. This follows security best practices for first-party applications by leveraging centralized, specialized authentication infrastructure.

### 2. No Open Redirect Vulnerabilities
All `quart.redirect()` calls use either:
- Hardcoded internal paths (`'/voter'`, `'/manage'`)
- Database-sourced election IDs constrained by schema CHECK constraints to `[0-9a-f]{10}` format

This prevents redirect manipulation attacks.

### 3. Consistent Authentication Enforcement
All sensitive routes use `@asfquart.auth.require` decorators, ensuring OAuth-authenticated sessions are enforced at the framework level.

### 4. Schema-Level Security Constraints
```sql
-- Election and issue IDs are constrained to prevent injection
CHECK (eid ~ '^[0-9a-f]{10}$')
```

This prevents injection attacks when IDs are used in redirect path construction.

## Architecture Observations

### OAuth Architecture Pattern

The application implements a **client-only OAuth architecture**:

```
┌─────────────────┐         ┌──────────────────┐
│  Apache Steve   │ ──────> │  ASF OAuth AS    │
│  (OAuth Client) │ <────── │  (External)      │
└─────────────────┘         └──────────────────┘
        │
        │ Uses asfquart.auth
        │
        v
  Protected Routes
```

**Key Architectural Points:**

1. **External Authorization Server**: The ASF OAuth infrastructure handles all AS responsibilities including redirect URI validation
2. **Framework Abstraction**: The `asfquart` package encapsulates OAuth client flow, callback handling, and token exchange
3. **Out-of-Scope Dependencies**: ASVS 10.4.1 compliance depends entirely on the external ASF OAuth AS and `asfquart` framework implementation

### Security Boundary Analysis

**In Scope (This Application):**
- OAuth client flow consumption
- Session management post-authentication
- Internal application redirects

**Out of Scope (External Dependencies):**
- OAuth authorization endpoint
- Redirect URI validation logic
- Client registration and allowlist management
- Token issuance and validation

## Recommendations Summary

### Immediate Actions (0-30 days)

1. **Document OAuth Client Registration**
   - Verify and document the exact `redirect_uri` registered with the ASF OAuth AS
   - Ensure no wildcards or pattern matching are used in registration
   - Confirm the registered URI matches the application's callback endpoint exactly

### Short-term Actions (1-3 months)

2. **Verify External AS Compliance**
   - Coordinate with the ASF infrastructure team to confirm the external OAuth Authorization Server validates redirect URIs using exact string comparison against pre-registered client-specific allowlists
   - Request documentation or attestation of compliance with RFC 6749 Section 3.1.2.2
   - Verify compliance with ASVS 10.4.1 requirements

3. **Audit `asfquart` Framework**
   - Conduct independent security audit of the `asfquart.auth` module
   - Verify client-side redirect URI validation
   - Review state parameter generation and validation
   - Confirm PKCE implementation if applicable for public clients

### Long-term Actions (3-12 months)

4. **Establish Dependency Security Review Process**
   - Create periodic review schedule for external OAuth dependencies
   - Monitor security advisories for `asfquart` and ASF OAuth infrastructure
   - Document security assumptions and trust boundaries

5. **Security Configuration Documentation**
   - Maintain comprehensive documentation of OAuth client configuration
   - Document all registered redirect URIs and their purposes
   - Create runbook for OAuth configuration updates and security validation

## Conclusion

ASVS 10.4.1 requires OAuth Authorization Servers to validate redirect URIs using exact string comparison against client-specific allowlists. This requirement **does not apply** to the apache/steve application as it contains no Authorization Server implementation.

The application correctly delegates OAuth AS responsibilities to external ASF infrastructure. To achieve full ASVS compliance, the organization should verify that the external ASF OAuth Authorization Server implements the required redirect URI validation controls.
