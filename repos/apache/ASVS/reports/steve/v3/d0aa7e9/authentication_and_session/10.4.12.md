## Findings

### Medium Severity

#### ASVS-10412-MED-001

**Finding:** OAuth Client Authorization Request Does Not Explicitly Specify `response_mode`, Relying Entirely on External AS Enforcement

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `create_app()`
- **Lines:** 39-43

**Related Functions Checked:**
- `create_app()` — OAuth URL configuration
- `asfquart.generics.OAUTH_URL_INIT` — Authorization endpoint template
- `asfquart.generics.OAUTH_URL_CALLBACK` — Token exchange endpoint template

**Vulnerable Code:**
```python
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**
1. `create_app()` sets `OAUTH_URL_INIT` as a URL template with only `state` and `redirect_uri` parameters
2. The `asfquart` framework formats this URL and redirects users to the external AS
3. No `response_mode` parameter is specified in the authorization request
4. No `response_type` parameter is visible in the template either
5. The client has no mechanism (PAR/JAR) to ensure the AS enforces the intended `response_mode`

**Description:**

The authorization URL template omits `response_mode` (and `response_type`). While `asfquart` may add parameters internally, the visible configuration shows no defense-in-depth from the client side. ASVS 10.4.12 requires that the AS only allow the `response_mode` appropriate for the client, and suggests PAR or JAR as enforcement mechanisms — neither is used.

Without explicit `response_mode=query` in the authorization request:
- An attacker who can manipulate the authorization request (e.g., via open redirect or parameter injection) could append `response_mode=fragment`, causing the authorization code to be returned in the URL fragment
- Fragment-based responses are not sent to the server and can be intercepted by client-side scripts or leaked via the Referer header
- The comment `# Avoid OIDC` (line 38) suggests this is a deliberate departure from OIDC defaults, which may also remove OIDC-specific `response_mode` protections

**Proof of Concept:**

If the `redirect_uri` or `state` value is not properly URL-encoded by `asfquart`, an attacker could craft:
```
https://oauth.apache.org/auth?state=xyz&redirect_uri=https://steve.example.org/callback&response_mode=fragment
```
This would instruct the AS to return the authorization code in the fragment if the AS does not restrict `response_mode` per-client.

**Impact:**

If the external AS (`oauth.apache.org`) does not enforce per-client `response_mode` restrictions, the authorization code could be delivered via an unintended mechanism (fragment), potentially enabling token interception. The risk is mitigated if `oauth.apache.org` independently enforces `response_mode` restrictions, but this cannot be verified from the client-side code.

**Remediation:**

**Option 1** — Explicitly specify `response_mode` and `response_type` in the authorization request:
```python
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?response_type=code&response_mode=query&state=%s&redirect_uri=%s'
)
```

**Option 2** — Use Pushed Authorization Requests (PAR) per RFC 9126, where the authorization request parameters are sent server-to-server:
```python
# Pseudocode for PAR flow
par_response = httpx.post('https://oauth.apache.org/par', data={
    'client_id': CLIENT_ID,
    'redirect_uri': REDIRECT_URI,
    'response_type': 'code',
    'response_mode': 'query',
    'state': state_value,
    'code_challenge': pkce_challenge,
    'code_challenge_method': 'S256',
})
request_uri = par_response.json()['request_uri']
# Redirect user to: https://oauth.apache.org/auth?request_uri={request_uri}&client_id={CLIENT_ID}
```

**Option 3** — Use JWT-Secured Authorization Request (JAR) per RFC 9101, where authorization parameters are signed by the client.

## Positive Security Patterns

1. **HTTPS for OAuth endpoints**: Both `OAUTH_URL_INIT` and `OAUTH_URL_CALLBACK` use `https://` for communication with the AS, ensuring encrypted communication channels.

2. **State parameter present**: The authorization URL template includes `state=%s`, which is essential for CSRF protection in the OAuth flow and prevents authorization code injection attacks.

3. **Server-side code exchange**: The callback URL uses `code=%s`, indicating a server-side token exchange (not implicit flow), which is the recommended pattern and prevents token exposure to the browser.

4. **Authentication decorators**: All state-changing endpoints use `@asfquart.auth.require()` decorators consistently, ensuring proper access control enforcement.

5. **Authorization code flow**: The use of `code` in the callback confirms the authorization code flow rather than the less secure implicit flow, aligning with current OAuth security best practices.

---

## Architecture Observations

1. **External AS dependency**: The application delegates all OAuth AS responsibilities to `oauth.apache.org`. ASVS 10.4.12 enforcement depends entirely on that external server's configuration, which is outside this codebase's control.

2. **Framework opacity**: The `asfquart` framework handles the actual OAuth flow mechanics. Without its source code, it is impossible to verify whether it adds `response_mode`, validates the response, or implements any client-side `response_mode` checks.

3. **OIDC bypass**: The explicit comment `# Avoid OIDC` and URL override suggests a deliberate decision to use plain OAuth instead of OIDC. OIDC specifications provide more explicit `response_mode` handling (including `response_mode=jwt` from JARM), which would strengthen alignment with ASVS 10.4.12.
