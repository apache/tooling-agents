## Findings

### HIGH SEVERITY

#### Finding ASVS-1035-SEV-001: No Sender-Constrained Access Token Implementation

| Attribute | Details |
|-----------|---------|
| **Finding ID** | ASVS-1035-SEV-001 |
| **Severity** | **HIGH** |
| **Gap Type** | **Type A** — Entry point with NO control |
| **ASVS Requirement** | 10.3.5 — Sender-constrained access tokens (mTLS or DPoP) |
| **CWE** | CWE-294: Authentication Bypass by Capture-replay |
| **Primary Location** | `v3/server/main.py` lines 37–41 |
| **Affected Locations** | `v3/server/pages.py` (all 21 `@asfquart.auth.require` endpoints) |

**Description:**

The application implements OAuth 2.0 authentication through Apache's OAuth infrastructure but provides **no mechanism to bind access tokens to the presenting client**. Neither Mutual TLS (RFC 8705) nor Demonstration of Proof-of-Possession (DPoP, RFC 9449) is implemented. This allows stolen access tokens or session tokens to be replayed from any network location by any attacker who obtains them.

**Vulnerable Code:**

*OAuth Configuration (`main.py:37-41`):*
```python
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

*Resource Server Endpoint Examples (`pages.py`):*
```python
@APP.get('/vote-on/<eid>')
@asfquart.auth.require({R.committer})  # Validates identity but NOT token binding
@load_election
@APP.use_template(TEMPLATES / 'vote-on.ezt')
async def vote_on_page(election):
    ...

@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})  # No DPoP/mTLS verification
@load_election
async def do_vote_endpoint(election):
    ...
```

**Data Flow:**

```
User → oauth.apache.org (Authorization Code Grant)
     → Token endpoint returns plain bearer token (no cnf claim, no DPoP binding)
     → asfquart exchanges code for token (OAUTH_URL_CALLBACK, no client cert)
     → Session established from token claims
     → All resource server endpoints validate session identity only
     → NO verification that the presenting party is the original token recipient
```

**Evidence of Absence:**

1. **No mTLS configuration for OAuth token exchange:** The `OAUTH_URL_CALLBACK` format string (`'https://oauth.apache.org/token?code=%s'`) passes only the authorization code. There is no client certificate configuration, no `certfile`/`keyfile` for mutual TLS in the OAuth client context. The TLS certificates in `run_standalone()` (lines 77-80) are server-side only:

```python
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
```

2. **No DPoP header processing:** Across all 21 route handlers in `pages.py`, no endpoint inspects a `DPoP` HTTP header, validates a DPoP proof JWT, or checks a `cnf` (confirmation) claim with a `jkt` (JWK thumbprint) value.

3. **No token-binding middleware:** The `asfquart.auth.require` decorator is applied at every protected endpoint but performs identity verification only (committer role, PMC membership, or valid session), with no sender-constraining check.

**Related Functions Checked:**

- `main.py`: OAuth configuration (lines 37-41), TLS setup (lines 77-80)
- `pages.py`: All 21 protected endpoints using `@asfquart.auth.require`
- `asfquart` framework: No DPoP or mTLS support detected in usage patterns

**Proof of Concept:**

*Scenario 1: Session Token Replay*

An attacker who intercepts or exfiltrates a valid session token (e.g., via XSS, network sniffing on a misconfigured proxy, or session store compromise) can replay it from any client:

```bash
# Stolen session cookie replayed from attacker's machine
curl -b "session=<stolen_session_token>" \
     https://steve.apache.org/do-vote/ELECTION_123 \
     -d "vote-ISSUE_1=yes"
# Vote is accepted — no proof that the sender is the legitimate token holder
```

*Scenario 2: OAuth Token Replay*

For the OAuth token exchange phase:
```bash
# Stolen authorization code replayed from different machine
curl "https://oauth.apache.org/token?code=<stolen_auth_code>"
# Token issued without verifying client certificate or DPoP proof
```

**Impact:**

- **Authentication Bypass:** A stolen access token or session token can be replayed from **any network location** by **any party**
- **Vote Manipulation:** For a voting system, this enables unauthorized vote submission through session hijacking
- **No Cryptographic Binding:** No cryptographic binding exists between the token and the legitimate client's key material
- **ASVS L3 Non-Compliance:** Violates the core protection model of ASVS Level 3 for OAuth resource servers
- **Lateral Movement:** Compromised tokens can be used across multiple sessions and endpoints without detection

**Remediation:**

Implement DPoP (RFC 9449), which is the most practical sender-constraining mechanism for browser-based OAuth flows:

**Option A — DPoP Implementation (Recommended):**

```python

import jwt
from cryptography.hazmat.primitives.asymmetric import ec

# During token request, include DPoP proof header
asfquart.generics.OAUTH_DPOP_ENABLED = True

# pages.py — Validate DPoP proof at resource server
import hashlib
import base64
import time
import functools

def validate_dpop_proof(request, access_token):
    """Validate DPoP proof JWT per RFC 9449."""
    dpop_header = request.headers.get('DPoP')
    if not dpop_header:
        quart.abort(401, 'Missing DPoP proof')

    try:
        # Decode without verification first to get JWK
        unverified = jwt.decode(dpop_header, options={"verify_signature": False})
        header = jwt.get_unverified_header(dpop_header)

        # Verify the DPoP proof signature using embedded JWK
        jwk = header['jwk']
        proof = jwt.decode(dpop_header, jwk, algorithms=['ES256'])

        # Validate claims
        assert proof['htm'] == request.method
        assert proof['htu'] == str(request.url)
        assert abs(time.time() - proof['iat']) < 300  # 5 min window

        # Verify access token hash binding (ath claim)
        token_hash = base64.urlsafe_b64encode(
            hashlib.sha256(access_token.encode()).digest()
        ).rstrip(b'=').decode()
        assert proof.get('ath') == token_hash

        # Verify JWK thumbprint matches token's cnf.jkt claim
        # (validated during token introspection)

    except Exception as e:
        quart.abort(401, f'Invalid DPoP proof: {e}')

# Apply to all protected endpoints
def dpop_required(func):
```

**Option B — Mutual TLS (RFC 8705):**

```python
# main.py — Configure mTLS for token endpoint
# Requires infrastructure-level changes:
# 1. Configure reverse proxy to pass client certificate
# 2. Bind certificate thumbprint to token at AS

# Validate certificate-bound token at RS
def validate_mtls_binding(request, token_claims):
    """Verify client cert thumbprint matches token's cnf.x5t#S256."""
    client_cert = request.headers.get('X-Client-Cert')
    if not client_cert:
        quart.abort(401, 'Missing client certificate')

    cert_thumbprint = compute_cert_thumbprint(client_cert)
    expected = token_claims.get('cnf', {}).get('x5t#S256')
    if cert_thumbprint != expected:
        quart.abort(401, 'Certificate-bound token mismatch')

def compute_cert_thumbprint(cert_pem):
    """Compute SHA-256 thumbprint of certificate."""
    from cryptography import x509
    from cryptography.hazmat.backends import default_backend
    import hashlib
    import base64
    
    cert = x509.load_pem_x509_certificate(cert_pem.encode(), default_backend())
    digest = hashlib.sha256(cert.public_bytes(encoding=serialization.Encoding.DER)).digest()
    return base64.urlsafe_b64encode(digest).rstrip(b'=').decode()
```

## Architecture Observations

### System Architecture Analysis

1. **OAuth Delegation Model:**
   - The application delegates OAuth entirely to the `asfquart` framework and Apache's OAuth infrastructure (`oauth.apache.org`)
   - Sender-constraining would need to be implemented both at the framework level (DPoP proof generation during token exchange) and at the resource server level (DPoP proof validation on each request)
   - This architectural dependency means remediation requires coordination across multiple components

2. **Session-Based Architecture:**
   - After OAuth authentication, the application operates on session cookies rather than passing OAuth access tokens per-request
   - This means sender-constraining could alternatively be achieved via session-bound mechanisms (e.g., binding session to client TLS certificate)
   - However, this approach is not currently implemented

3. **Framework Dependency:**
   - The `asfquart` framework is a shared component across ASF projects
   - Implementing DPoP/mTLS support likely requires coordination with the framework maintainers rather than application-level changes alone
   - This may extend the remediation timeline but would benefit all ASF projects using the framework

4. **Resource Server Pattern:**
   - The application correctly implements the resource server pattern with consistent authentication checks
   - The foundation for adding sender-constraining is solid; only the binding mechanism is missing
   - All 21 protected endpoints use the same decorator pattern, making it straightforward to add DPoP validation uniformly

### Security Implications

- **Current State:** The authentication layer is well-structured but incomplete for ASVS L3 requirements
- **Risk Profile:** High risk for token replay attacks, especially in the context of a voting system where vote integrity is critical
- **Remediation Path:** Clear upgrade path exists through DPoP implementation without requiring architectural changes
