# ASVS 10.3.2 Security Audit Report

## Executive Summary

**Repository:** apache/steve

**Audit Date:** Apr 21, 2026

**Auditor:** Tooling Agents

**ASVS Requirement:** 10.3.2 — Verify that the resource server enforces authorization decisions based on claims from the access token that define delegated authorization. If claims such as 'sub', 'scope', and 'authorization_details' are present, they must be part of the decision.

**Files Analyzed:** 3 relevant out of 3 total, 0 skipped

**Finding Summary:**

| Severity | Count |
|----------|-------|

| Medium | 0 |
| Low | 0 |
| **Total** | **3** |

**Key Findings:** The application uses ASF's OAuth provider for authentication but has critical gaps in enforcing authorization decisions based on token-derived claims. While the `uid` (equivalent to the `sub` claim) is extracted from the session, it is not consistently used for authorization decisions on sensitive operations. OAuth scopes are neither requested nor enforced. Multiple endpoints have explicit `### check authz` comments marking where authorization should exist but doesn't — classic Type B (false confidence) gaps.

## Findings

### Critical Severity

#### ASVS-1032-CRI-001: Vote Submission Endpoint Lacks Voter Eligibility Authorization Check (Type B)

**Severity:** CRITICAL

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint`
- **Lines:** 424–467

**Related Functions Checked:**
- `vote_on_page` (line 268) — correctly implements eligibility check
- `load_election` decorator (line 195) — missing authorization stub
- `q_find_issues.perform()` — eligibility query exists but not used in POST handler

**Description:**

The vote submission endpoint fails to verify that the authenticated user (`sub` claim from OAuth token, stored as `uid` in session) is eligible to vote in the target election. While the GET handler (`vote_on_page`) correctly checks voter eligibility using `election.q_find_issues.perform(result.uid, election.eid)`, the POST handler that actually records votes performs no such check. The endpoint has an explicit `### check authz` comment stub at line 426, indicating the developers intended to implement this check but never did.

**Vulnerable Code:**
```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    ### check authz                          # <-- STUB: never implemented

    # Parse the form data
    form = edict(await quart.request.form)
    ...
    for iid, votestring in votes.items():
        ...
        election.add_vote(result.uid, iid, votestring)
```

**Control Exists But Not Applied:**

The GET handler (`vote_on_page`, line 268) correctly checks voter eligibility:
```python
election.q_find_issues.perform(result.uid, election.eid)
if not election.q_find_issues.fetchall():
    ...raise_404...
```
But the POST handler that **actually records the vote** performs no such check.

**Data Flow:**
1. Attacker authenticates as any ASF committer (valid `sub`/`uid` claim in session)
2. Sends `POST /do-vote/<eid>` with fabricated vote data
3. `@asfquart.auth.require({R.committer})` passes — attacker is a committer
4. `load_election` loads the election — no ownership/eligibility check
5. `election.add_vote(result.uid, iid, votestring)` records unauthorized vote

**Proof of Concept:**
```http
POST /do-vote/election-uuid-here HTTP/1.1
Cookie: <valid_committer_session>
Content-Type: application/x-www-form-urlencoded

vote-issue1=yes&vote-issue2=no
```
Any authenticated committer can vote in any election, even those they are not eligible for.

**Impact:** 

Integrity of election results compromised. Unauthorized voters can cast votes, potentially changing outcomes of ASF governance decisions. This directly violates ASVS 10.3.2 as the `sub` claim (stored as `uid`) is present in the token/session but not used for authorization decisions.

**Remediation:**
```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    # Verify voter is eligible for this election
    election.q_find_issues.perform(result.uid, election.eid)
    if not election.q_find_issues.fetchall():
        await flash_danger('You are not authorized to vote in this election.')
        return quart.redirect('/voter', code=303)

    form = edict(await quart.request.form)
    ...
```

#### ASVS-1032-CRI-002: Election Management Endpoints Missing Ownership Authorization (Type B)

**Severity:** CRITICAL

**Location:**
- **File:** `v3/server/pages.py`
- **Multiple Functions:** See table below

**Affected Endpoints:**

| Endpoint | Function | Line | `### check authz` Stub |
|----------|----------|------|----------------------|
| `GET /do-open/<eid>` | `do_open_endpoint` | 493 | Line 498 |
| `GET /do-close/<eid>` | `do_close_endpoint` | 515 | Line 520 |
| `POST /do-set-open_at/<eid>` | `do_set_open_at_endpoint` | 410 | Line 98 (via helper) |
| `POST /do-set-close_at/<eid>` | `do_set_close_at_endpoint` | 417 | Line 98 (via helper) |
| `POST /do-add-issue/<eid>` | `do_add_issue_endpoint` | 534 | Line 539 |
| `POST /do-edit-issue/<eid>/<iid>` | `do_edit_issue_endpoint` | 559 | Line 564 |
| `POST /do-delete-issue/<eid>/<iid>` | `do_delete_issue_endpoint` | 583 | Line 588 |
| `GET /manage/<eid>` | `manage_page` | 355 | Line 195 (via decorator) |

**Related Functions Checked:**
- `load_election` decorator (line 195) — contains authorization stub but no enforcement
- `admin_page` (line 318) — uses `Election.owned_elections()` for display only
- `Election.owned_elections()` — query exists but not used for enforcement
- `Election.get_metadata()` — contains owner_pid field but not validated

**Description:**

All election management endpoints fail to verify that the authenticated user (identified by the `sub` claim from the OAuth token, stored as `uid` in the session) owns the election being modified. The `uid` claim is available throughout the application but is never compared against election ownership. The `Election.owned_elections(DB_FNAME, result.uid)` query exists and is used in `admin_page` for **display purposes**, but is never used as an **enforcement gate** for state-changing operations.

**Vulnerable Code (representative — `do_open_endpoint`):**
```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    result = await basic_info()

    ### check authz                          # <-- STUB: never implemented

    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)                       # <-- State change with no ownership check
```

**And in the `load_election` decorator itself (line 195):**
```python
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            ...

        ### check authz                      # <-- STUB: never implemented

        return await func(e)
    return loader
```

**Data Flow:**
1. Attacker authenticates as any ASF committer (`sub`/`uid` claim present in session)
2. The `uid` from the session (mapped from OAuth `sub` claim) is available but **not compared against election ownership**
3. `Election.owned_elections(DB_FNAME, result.uid)` exists in `admin_page` for **display** but is never used as an **enforcement gate**
4. Attacker sends `GET /do-close/target-election-uuid` — election is closed regardless of ownership

**Proof of Concept:**
```http
GET /do-open/target-election-id HTTP/1.1
Cookie: <any_valid_committer_session>
```
Any committer can open, close, or modify any other committer's election.

**Impact:** 

Any authenticated committer can tamper with elections they don't own — opening elections prematurely, closing them early to suppress votes, deleting issues, or modifying election content. This directly violates ASVS 10.3.2 as the `sub` claim is present but not used to enforce resource-level authorization decisions.

**Remediation:**
```python
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            ...

        # Enforce ownership: verify the authenticated user's uid matches
        # the election owner (using 'sub' claim from token/session)
        s = await asfquart.session.read()
        metadata = e.get_metadata()
        if metadata.owner_pid != s['uid']:
            quart.abort(403)

        return await func(e)
    return loader
```

### High Severity

#### ASVS-1032-HIG-003: OAuth Scope Neither Requested Nor Enforced in Authorization Decisions

**Severity:** HIGH

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `create_app`
- **Lines:** 37–41

**Related Functions Checked:**
- `asfquart.generics.OAUTH_URL_INIT` configuration
- All endpoint handlers in `pages.py` — none check scope claims
- `asfquart.session.read()` — scope not stored in session
- Token callback handler (not visible in provided code)

**Description:**

The OAuth authorization URL contains no `scope` parameter. Per ASVS 10.3.2, if the OAuth flow supports scopes, the resource server must enforce authorization decisions based on them. The application has two distinct privilege levels (voter vs. election manager) that map naturally to OAuth scopes, but all authorization is done through a single flat role check (`R.committer`). The application:

1. **Does not request scopes** in the OAuth authorization URL — no `scope=` parameter
2. **Does not validate scopes** at any endpoint — no endpoint inspects token scope claims
3. **Has no scope differentiation** between voting operations and election management operations

**Vulnerable Code:**
```python
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
```

**Data Flow:**
1. OAuth token issued without explicit scope constraints
2. Token exchanged and session created with `uid`, `fullname`, `email` — no scope stored
3. All endpoints check `R.committer` role only — no scope-level authorization
4. User with a token intended only for "read" access could perform "write" operations

**Proof of Concept:**

A user obtains an OAuth token through a legitimate flow. Even if the authorization server intended to limit the token to read-only operations (viewing election status), the application would accept it for write operations (voting, managing elections) because no scope validation occurs.

**Impact:** 

The delegated authorization model cannot express or enforce least-privilege access. A token obtained for viewing election status carries the same authorization as one for managing elections. This violates the principle of scope-based access control required by OAuth best practices and ASVS 10.3.2, which mandates that scope claims must be part of authorization decisions when present.

**Remediation:**
```python
# In main.py - Request appropriate scopes
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s&scope=openid+profile+email+steve:vote+steve:manage'
)

# In pages.py - Enforce scope on management endpoints
async def require_scope(required_scope):
    s = await asfquart.session.read()
    token_scopes = s.get('scope', '').split()
    if required_scope not in token_scopes:
        quart.abort(403, f'Missing required scope: {required_scope}')

# Apply to management endpoints
@APP.post('/do-add-issue/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_add_issue_endpoint(election):
    await require_scope('steve:manage')
    ...
```

## Positive Security Patterns

1. **Authentication decorators consistently applied**: All state-changing endpoints have `@asfquart.auth.require` decorators — the authentication layer is present.

2. **Voter eligibility check exists** in `vote_on_page` (GET) — the concept and query are implemented, just not applied to the POST handler.

3. **Election ownership query exists** in `admin_page` — `Election.owned_elections()` is defined and used for display.

4. **LDAPS used for directory access**: `LDAP_URL = 'ldaps://ldap-us.apache.org/'` uses encrypted connection.

5. **Comprehensive audit logging**: All state-changing operations log the user ID, election ID, and action.

6. **PMC member role separation**: Election creation correctly requires `R.pmc_member`, showing awareness of privilege differentiation.

7. **State-changing GET endpoints use POST for votes**: Vote submission correctly uses POST method.

---

## Architecture Observations

1. **Authorization model is authentication-only**: The current model treats all committers as equally authorized for all elections. The `sub` claim equivalent (`uid`) is captured but not used for resource-level authorization decisions.

2. **10 explicit `### check authz` stubs**: The developers clearly intended to implement resource-level authorization but left it as TODO. This creates a dangerous false confidence gap — code reviewers may assume authorization exists because the comments suggest it's planned.

3. **Display vs. Enforcement disconnect**: The admin page shows only owned elections, creating a UI-level illusion of access control. But API endpoints accept any election ID without ownership verification.

4. **OAuth integration is opaque**: The actual token validation, claim extraction, and session creation happen inside `asfquart` framework code (not provided). This makes it impossible to fully verify whether token claims are properly validated before session creation.

5. **No defense in depth**: Authorization checks are missing at multiple layers — no validation in the decorator, no validation in the endpoint handler, and no validation in the business logic layer.

## Recommendations Summary

### Immediate Actions (P0)

1. **Implement voter eligibility check in `do_vote_endpoint`** (ASVS-1032-CRI-001)
   - Add `election.q_find_issues.perform(result.uid, election.eid)` check before recording votes
   - Deploy immediately to prevent unauthorized vote manipulation

2. **Implement ownership verification in `load_election` decorator** (ASVS-1032-CRI-002)
   - Compare `metadata.owner_pid` against session `uid` before allowing state changes
   - Protects all 8 management endpoints with a single fix

### Short-term Actions (P1)

3. **Add OAuth scope parameter to authorization URL** (ASVS-1032-HIG-003)
   - Request `openid profile email steve:vote steve:manage` scopes
   - Store scope in session during token exchange

4. **Enforce scope-based authorization at endpoints**
   - Create `require_scope()` helper function
   - Apply to all management endpoints requiring `steve:manage` scope
   - Apply to vote submission requiring `steve:vote` scope

5. **Systematically implement all 10 `### check authz` stubs**
   - Audit each stub location
   - Implement appropriate authorization check (ownership, eligibility, or scope)
   - Remove stub comments after implementation

### Long-term Actions (P2)

6. **Centralize authorization logic**
   - Create authorization middleware layer
   - Move authorization checks out of individual endpoint handlers
   - Implement policy-based access control (PBAC) framework

7. **Add authorization unit tests**
   - Test each endpoint with unauthorized users
   - Test cross-election access attempts
   - Test scope enforcement

8. **Implement audit logging for authorization failures**
   - Log all 403 responses with user ID, resource ID, and attempted action
   - Monitor for potential attack patterns
