### Key Observations

ASVS 10.4.4 requires that for a given client, the authorization server only allows the usage of grants that the client needs, and that the `token` (Implicit) and `password` (ROPC) grants must not be used. The provided codebase implements an **OAuth client/relying party** (election management application), not an OAuth Authorization Server. The OAuth AS functionality is handled externally by the ASF infrastructure via the `asfquart` framework. Consequently, grant type configuration for the AS is not present in the audited code. No CRITICAL findings directly attributable to ASVS 10.4.4 were identified in the provided source files.

## Findings

**No findings identified for ASVS 10.4.4.**

### Analysis Rationale

1. **Application Role**: The application is an OAuth **client** (relying party), not an OAuth Authorization Server. It delegates authentication to an external ASF OAuth Authorization Server via the `asfquart` framework. The codebase contains no OAuth AS implementation — no `/authorize` endpoint, no `/token` endpoint, no client registration, and no grant type configuration.

2. **No Prohibited Grant Usage Detected**: There is no code in the provided files that initiates or accepts an Implicit flow (`response_type=token`) or Resource Owner Password Credentials flow (`grant_type=password`). The authentication pattern (`asfquart.auth.require` → `asfquart.session.read()` returning server-side session data with `uid`, `fullname`, `email`) is consistent with an Authorization Code flow, which is the recommended approach.

3. **Grant Restriction Is External**: The responsibility for restricting which grant types the client may use lies with the ASF OAuth Authorization Server's client registration configuration. This configuration is not within the scope of the provided files.

4. **Audit Scope Limitation**: ASVS 10.4.4 applies to OAuth Authorization Servers, not OAuth clients. Since this codebase implements only the client side, full compliance verification requires auditing the external ASF OAuth Authorization Server's client registration and configuration.

## Architecture Observations

### OAuth Integration Architecture

1. **External OAuth Authorization Server**: The ASF OAuth Authorization Server is a separate infrastructure component not included in this codebase. The application acts solely as an OAuth client/relying party.

2. **Framework-Mediated OAuth Flow**: All OAuth flow logic is encapsulated in the `asfquart.auth` and `asfquart.session` modules. The application code does not directly interact with OAuth endpoints or tokens.

3. **Session-Based Authentication**: The application uses server-side sessions rather than client-side token storage, which is consistent with Authorization Code flow best practices.

### Compliance Verification Limitations

To fully verify ASVS 10.4.4 compliance, the following external components must be audited:

- **ASF OAuth Authorization Server client registration** to confirm:
  - Only `authorization_code` grant type is permitted for this client
  - `token` (Implicit) and `password` (ROPC) grants are explicitly disabled
  - PKCE is required for this client (if applicable)
  
- **`asfquart` framework OAuth implementation** to confirm:
  - Proper OAuth client configuration
  - No fallback to prohibited grant types
  - Proper validation of AS responses

### Authorization Implementation Gaps (Out of Scope)

Multiple endpoints contain `### check authz` comments without implementation (e.g., `pages.py` for `load_election`, `do_open_endpoint`, `do_close_endpoint`, `do_add_issue_endpoint`, `do_edit_issue_endpoint`, `do_delete_issue_endpoint`). While this represents a significant authorization concern, it relates to application-level access control (ASVS Chapter 4) rather than OAuth grant type restrictions (ASVS 10.4.4).
