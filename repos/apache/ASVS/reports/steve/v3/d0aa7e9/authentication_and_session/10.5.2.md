## Findings

### HIGH Severity

#### ASVS-1052-SEV-001: No ID Token Handling - Custom OAuth Bypasses OIDC

**Severity:** HIGH  
**ASVS Requirement:** 10.5.2 — Client uniquely identifies user from ID Token claims (`sub` claim)  
**File:** `v3/server/main.py`  
**Function:** `create_app()`  
**Lines:** 38–43  
**Gap Type:** Type A — No ID Token handling exists at all

**Description:**

The application explicitly configures custom OAuth endpoints and comments `# Avoid OIDC`. This means no ID Token is issued or consumed, and the `sub` claim (which OIDC guarantees to be a locally unique, never-reassigned identifier) is not used.

**Vulnerable Code:**

```python
### is this really needed right now?
# Avoid OIDC
import asfquart.generics

asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**

```
OAuth callback → asfquart (opaque token exchange with oauth.apache.org) 
→ session populated with uid/fullname/email 
→ uid used as user identity throughout application
```

At no point in this flow is an OIDC ID Token parsed, its signature verified, or its `sub` claim extracted.

**Related Functions Checked:**
- `create_app()` in `main.py`
- `basic_info()` in `pages.py` (consumes the session data)
- OAuth callback handler (within `asfquart` framework, not visible in provided code)

**Analysis:**

Without OIDC ID Token processing, the following critical validations are absent:
1. Cryptographic signature verification of identity assertions
2. `iss` (issuer) validation
3. `aud` (audience) validation ensuring the token was intended for this client
4. `exp`/`iat` temporal validity checks on the identity assertion
5. `nonce` validation for replay protection

**Impact:**

The user identity used for all authorization decisions (voting eligibility, election management, audit logging) is obtained through a custom OAuth flow without the security guarantees that OIDC ID Token validation provides. If the OAuth token exchange or the proprietary user-info mechanism is compromised or spoofed, user identity could be manipulated. In a voting system, this could allow vote impersonation or unauthorized election management.

**Proof of Concept:**

Without ID Token signature verification, if an attacker can intercept or manipulate the response from `https://oauth.apache.org/token?code=%s`, they could inject arbitrary `uid` values into the session. Standard OIDC ID Token validation (signature check against the OP's published keys) would prevent this.

**Remediation:**

Migrate from the custom OAuth flow to standard OIDC, consuming and validating the ID Token:

```python
# main.py - Use standard OIDC discovery instead of custom OAuth
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    # Configure OIDC with proper ID Token validation
    # Use OIDC discovery endpoint for automatic key/endpoint configuration
    app.config['OIDC_ISSUER'] = 'https://oauth.apache.org'
    app.config['OIDC_CLIENT_ID'] = '<client_id>'
    # ... standard OIDC configuration
    
    import pages
    import api
    return app
```

```python
# In session establishment (within asfquart or callback handler):
def establish_session_from_id_token(id_token_claims):
    # Use 'sub' claim as the unique, non-reassignable user identifier
    user_id = id_token_claims['sub']  # NOT email, NOT preferred_username
    
    # Verify issuer matches expected OP
    assert id_token_claims['iss'] == 'https://oauth.apache.org'
    
    # Verify audience includes this client
    assert '<client_id>' in id_token_claims['aud']
    
    session['uid'] = user_id  # sub claim is guaranteed non-reassignable per issuer
```

### MEDIUM Severity

#### ASVS-1052-SEV-002: No Technical Enforcement of Identifier Immutability

**Severity:** MEDIUM  
**ASVS Requirement:** 10.5.2 — User identifier cannot be reassigned to other users  
**File:** `v3/server/pages.py`  
**Function:** `basic_info()`  
**Lines:** 77–88  
**Gap Type:** Type A — No technical enforcement of identifier immutability

**Description:**

The application uses `s['uid']` from the session as the sole user identifier for all security decisions. While Apache LDAP UIDs are operationally stable (not reassigned), there is no **technical enforcement** in this codebase that the `uid` originates from a claim that is contractually non-reassignable (like OIDC `sub`).

**Vulnerable Code:**

```python
s = await asfquart.session.read()
if s:
    basic.update(
        uid=s['uid'],
        name=s['fullname'],
        email=s['email'],
    )
```

**Related Functions Checked:**
- `basic_info()` in `pages.py:77-88`
- LDAP loading in `asf-load-ldap.py:55-59`
- All endpoint handlers that consume `result.uid`

**Analysis:**

The `uid` is populated by the `asfquart` framework during the OAuth callback, drawing from whatever the Apache OAuth provider returns. There is no technical enforcement that:

1. The `uid` originates from a claim that is contractually non-reassignable (like OIDC `sub`)
2. The `uid` has not been modified between the identity provider and the session
3. The `uid` is bound to a single identity provider (no `iss` + `sub` compound key)

The LDAP loading script confirms `uid` comes from LDAP's `uid` attribute:

```python
uid = entry.uid[0].decode('utf-8')
pdb.add_person(uid, visname, email)
```

If the OAuth provider were to ever be changed or federated (multiple identity providers), using `uid` alone without an issuer qualifier could lead to identifier collisions across providers.

**Impact:**

In the current single-IdP deployment, the risk is lower because Apache LDAP UIDs are administratively non-reassignable. However, the code provides no technical guarantee of this property. If the identity provider changes, or if the system is deployed in a different context, user identity confusion could occur, potentially allowing one user to vote as another.

**Proof of Concept:**

If the system were deployed with a federated identity setup:
1. Provider A has user with uid="jdoe"
2. Provider B has different user with uid="jdoe"
3. Without issuer qualification, both would appear as the same user in the voting system

**Remediation:**

Use a compound identifier (`iss` + `sub`) or validate that the identifier source guarantees non-reassignment:

```python
async def basic_info():
    basic = edict()
    s = await asfquart.session.read()
    if s:
        # Use sub claim from ID Token, qualified by issuer
        # This ensures uniqueness even across federated IdPs
        basic.update(
            uid=s['sub'],           # From ID Token sub claim
            issuer=s['iss'],        # From ID Token iss claim  
            name=s['fullname'],
            email=s['email'],
        )
    # ...
```

## Positive Security Patterns

1. **Consistent use of `uid` as the user identifier** — All authorization decisions throughout `pages.py` use `result.uid` from the session. No endpoint uses `email` or `fullname` for identity decisions. This prevents mutable-attribute confusion.

2. **LDAP `uid` as the canonical identifier** — The `asf-load-ldap.py` script correctly uses the LDAP `uid` attribute (which is an immutable login name in Apache's LDAP) rather than `cn` or `email` for the person database primary key.

3. **Authorization decorators applied consistently** — Every sensitive endpoint has an `@asfquart.auth.require` decorator, ensuring session validation occurs before `basic_info()` extracts the `uid`.

4. **LDAPS connection** — The LDAP script uses `ldaps://` (line 33), ensuring encrypted communication for the identity data source.

---

## Architecture Observations

1. **OAuth-for-Authentication Anti-Pattern**: The system uses OAuth (which is a *delegation* protocol) for *authentication*, explicitly bypassing OIDC which was designed for this purpose. The comment `# Avoid OIDC` suggests a deliberate choice, but this removes the security properties that OIDC ID Tokens provide (signed identity assertions, audience restriction, temporal validity).

2. **Opaque Framework Dependency**: The `asfquart` framework handles the OAuth token exchange and session population internally. The security of user identification depends entirely on the correctness of this framework, which cannot be audited from the provided code. The application has no defense-in-depth for identity verification.

3. **Single Identity Provider Assumption**: The codebase assumes a single identity provider (`oauth.apache.org`). There is no issuer qualification on the `uid`, meaning the architecture would break if federated identity were introduced.
