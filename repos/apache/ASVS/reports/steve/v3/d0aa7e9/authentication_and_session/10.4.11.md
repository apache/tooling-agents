## Findings

### High Severity

#### ASVS-10411-HIGH-001 — Missing Election Ownership Authorization on All Management Endpoints

**Severity:** HIGH  
**Gap Type:** Type B (control planned but never implemented — false confidence)

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** Multiple endpoints
- **Lines:** 97, 193, 217, 227, 487, 508, 527, 554, 581

**Affected Endpoints:**

| Endpoint | Route | Line |
|----------|-------|------|
| `do_open_endpoint` | `GET /do-open/<eid>` | ~487 |
| `do_close_endpoint` | `GET /do-close/<eid>` | ~508 |
| `do_add_issue_endpoint` | `POST /do-add-issue/<eid>` | ~527 |
| `do_edit_issue_endpoint` | `POST /do-edit-issue/<eid>/<iid>` | ~554 |
| `do_delete_issue_endpoint` | `POST /do-delete-issue/<eid>/<iid>` | ~581 |
| `do_set_open_at_endpoint` | `POST /do-set-open_at/<eid>` | ~97 (via `_set_election_date`) |
| `do_set_close_at_endpoint` | `POST /do-set-close_at/<eid>` | ~97 (via `_set_election_date`) |
| `load_election` decorator | N/A | ~193 |
| `load_election_issue` decorator | N/A | ~217, ~227 |

**Related Functions Checked:**
- `load_election()` — decorator at line ~193
- `load_election_issue()` — decorator at lines ~217, ~227
- `do_open_endpoint()` — line ~487
- `do_close_endpoint()` — line ~508
- `do_add_issue_endpoint()` — line ~527
- `do_edit_issue_endpoint()` — line ~554
- `do_delete_issue_endpoint()` — line ~581
- `_set_election_date()` — line ~97

**Description:**

All election management endpoints contain a `### check authz` comment indicating planned authorization checks, but these checks are never implemented. Any authenticated ASF committer can open, close, or modify any election in the system, including adding, editing, or deleting ballot issues, regardless of election ownership.

**Vulnerable Code:**

```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    result = await basic_info()

    ### check authz       # <-- NEVER IMPLEMENTED

    # Open the Election.
    election.open(pdb)    # <-- Any committer can open any election
```

**Data Flow:**

```
Authenticated committer 
  → GET /do-open/<any_eid> 
  → R.committer passes 
  → load_election loads election by EID (no ownership check) 
  → election.open() executes 
  → election owned by another user is opened/closed/modified
```

**Proof of Concept:**

```http
# Attacker is any authenticated ASF committer, not the election owner
GET /do-open/election-uuid-owned-by-someone-else HTTP/1.1
Cookie: [valid_session_cookie]
```

**Impact:**

Any authenticated ASF committer can open, close, or modify any election in the system, including adding, editing, or deleting ballot issues. This completely undermines election integrity since the OAuth-assigned scope (committer role) is far broader than the required scope (election owner). This is a direct violation of ASVS 10.4.11's principle that only required scopes should be assigned to operations.

**Remediation:**

```python
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            # ... existing 404 handling
            pass

        # AUTHORIZATION CHECK: verify current user is the election owner
        result = await basic_info()
        metadata = e.get_metadata()
        if metadata.owner_pid != result.uid:
            quart.abort(403, 'Not authorized to manage this election')

        return await func(e)
    return loader
```

#### ASVS-10411-HIGH-002 — Vote Submission Endpoint Missing Voter Eligibility Check

**Severity:** HIGH  
**Gap Type:** Type B (control exists in GET handler but not applied to POST handler)

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint()`
- **Lines:** ~411-456

**Related Functions Checked:**
- `vote_on_page()` — line ~262 (contains proper eligibility check)
- `do_vote_endpoint()` — lines ~411-456 (missing eligibility check)

**Description:**

The voter eligibility check (mayvote verification) exists in the `vote_on_page` GET handler but is not applied to the `do_vote_endpoint` POST handler. Any authenticated ASF committer can submit votes on any open election, regardless of whether they are listed as an eligible voter.

**Control EXISTS in `vote_on_page` (GET /vote-on/<eid>), lines ~262-267:**

```python
# Check if user can vote in this election (has mayvote entries)
election.q_find_issues.perform(result.uid, election.eid)
if not election.q_find_issues.fetchall():
    result = await basic_info()
    result.title = 'Access Denied'
    result.eid = election.eid
    raise_404(T_BAD_EID, result)
```

**Control NOT APPLIED in `do_vote_endpoint` (POST /do-vote/<eid>), lines ~411-456:**

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    ### check authz       # <-- NEVER IMPLEMENTED — no mayvote check

    form = edict(await quart.request.form)
    # ... proceeds directly to:
    election.add_vote(result.uid, iid, votestring)  # Accepts vote from ineligible voter
```

**Data Flow:**

```
Ineligible committer 
  → POST /do-vote/<eid> with crafted form data 
  → R.committer passes 
  → no mayvote check 
  → election.add_vote() stores vote 
  → ineligible user has voted
```

**Proof of Concept:**

```http
POST /do-vote/target-election-eid HTTP/1.1
Cookie: [valid_session_cookie_for_non_eligible_committer]
Content-Type: application/x-www-form-urlencoded

vote-issue1=yes&vote-issue2=no
```

**Impact:**

Any authenticated ASF committer can submit votes on any open election, regardless of whether they are listed as an eligible voter (in the mayvote table). The eligibility check exists only on the display page (GET), which an attacker bypasses entirely by POSTing directly. This is a scope enforcement failure: the committer scope should not grant voting rights on elections where the user is not a registered voter.

**Remediation:**

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    # Verify voter eligibility before accepting votes
    election.q_find_issues.perform(result.uid, election.eid)
    if not election.q_find_issues.fetchall():
        quart.abort(403, 'Not authorized to vote in this election')

    # ... rest of vote processing
```

### Medium Severity

#### ASVS-10411-MEDIUM-001 — OAuth Authorization Request Omits Scope Parameter

**Severity:** MEDIUM  
**Gap Type:** Type A (no control)

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `create_app()`
- **Lines:** 38–42

**Related Functions Checked:**
- `create_app()` — lines 38-42 (OAuth configuration)

**Description:**

The OAuth authorization URL template only includes `state` and `redirect_uri` parameters. No `scope` parameter is specified. Per RFC 6749 §3.3, when the client omits the scope parameter, the authorization server either uses a pre-configured default or fails. Without an explicit `scope` value, the application relies entirely on server-side defaults at `oauth.apache.org`, violating the principle of least privilege mandated by ASVS 10.4.11.

**Vulnerable Code:**

```python
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
```

**Data Flow:**

```
create_app() 
  → sets OAuth init URL 
  → user redirected to AS 
  → no scope parameter 
  → AS grants default/full scopes 
  → application receives token with potentially excessive permissions
```

**Analysis:**

The application requires two distinct authorization levels (committer operations and PMC-member operations), yet makes no attempt to request differentiated scopes from the OAuth AS. This means the issued token may carry broader permissions than any individual operation requires.

**Impact:**

The OAuth client may receive access tokens with scopes exceeding what's needed for the user's intended operation. If the AS grants broad default scopes, any compromised token grants full application access rather than limited access.

**Remediation:**

```python
# Explicitly request only needed scopes
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s&scope=openid%20profile%20email'
)
```

Additionally, validate returned scopes in the token response to confirm the AS only granted what was requested.

## Positive Security Patterns

1. **LDAPS for directory access** (`asf-load-ldap.py`): Uses `ldaps://` protocol for LDAP communication.
2. **OAuth uses HTTPS endpoints**: Both init and callback URLs use `https://`.
3. **Two-tier role distinction**: `R.pmc_member` for election creation vs `R.committer` for operations shows intent for scope separation.
4. **Voter eligibility check on display page**: `vote_on_page` correctly checks mayvote before showing the ballot.
5. **Document serving authorization**: `serve_doc` checks mayvote before serving files.
6. **Audit logging**: All state-changing operations log the user ID and action.

---

## Architecture Observations

1. **OAuth is used purely for authentication**, not for fine-grained authorization. The application derives roles from LDAP group membership rather than OAuth scopes. This means ASVS 10.4.11's scope minimization principle must be enforced at both the OAuth level (request minimal scopes) and the application level (enforce ownership and eligibility).

2. **The authorization model has a structural gap**: Authentication (OAuth) and coarse authorization (committer/pmc_member) are implemented, but fine-grained authorization (election ownership, voter eligibility) is systematically missing from state-changing endpoints despite being documented as needed (`### check authz`).

3. **GET methods used for state-changing operations**: `/do-open/<eid>` and `/do-close/<eid>` use GET, which compounds the CSRF risk noted in TODO.md.
