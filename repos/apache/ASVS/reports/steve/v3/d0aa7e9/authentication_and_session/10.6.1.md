## Findings

### Medium Severity

#### ASVS-1061-SEV-001: Missing Explicit `response_type=code` Parameter in OAuth Authorization URL

**Severity:** MEDIUM  
**Gap Type:** Type A — Missing control  
**File:** `v3/server/main.py`  
**Function:** `create_app()`  
**Lines:** 36-41

**Related Functions Checked:**
- `create_app()` in `main.py` (OAuth URL configuration)
- All authentication-decorated endpoints in `pages.py` (session dependency on OAuth flow)

**Description:**

The OAuth authorization URL template does not include the required `response_type=code` parameter. Per RFC 6749 §4.1.1, `response_type` is a **REQUIRED** parameter in authorization requests. While the callback URL pattern (`code=%s`) implies the code flow is expected, the authorization request itself does not enforce this.

**Vulnerable Code:**

```python
# Avoid OIDC
import asfquart.generics

asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**

1. `create_app()` overrides `OAUTH_URL_INIT` with custom template
2. Authorization URL constructed without `response_type` parameter
3. URL sent to browser for OAuth authorization
4. Browser redirects to `oauth.apache.org`
5. OP interprets request **without explicit `response_type=code`**
6. OP behavior depends on default configuration (undefined per spec)

**Proof of Concept:**

The constructed authorization URL would be:
```
https://oauth.apache.org/auth?state=<random>&redirect_uri=<callback_url>
```

**Missing:** `&response_type=code`

An OP that defaults to or allows `response_type=token` (implicit flow) when the parameter is not specified would return access tokens in the URL fragment instead of authorization codes. This would result in:

```
https://<callback_url>#access_token=<token>&token_type=bearer&expires_in=3600
```

Instead of the expected:
```
https://<callback_url>?code=<authorization_code>&state=<state>
```

**Impact:**

Without an explicit `response_type=code` parameter:

1. **Reliance on Undefined Behavior:** The RP relies entirely on the external OP's default behavior, which is not guaranteed by the OAuth specification
2. **Implicit Flow Risk:** If the OP defaults to or supports `response_type=token`, access tokens could be returned in the URL fragment
3. **Token Leakage Vectors:** Tokens in URL fragments are susceptible to:
   - Browser history exposure
   - Referrer header leakage
   - JavaScript access by third-party scripts
   - Server logs (if fragment is inadvertently captured)
4. **ASVS 10.6.1 Intent Violation:** This directly contradicts the requirement's prohibition of `token` (implicit flow)
5. **Bypassed Security Controls:** Authorization code exchange provides critical security properties (client binding, token rotation) that implicit flow lacks

**Remediation:**

Explicitly include `response_type=code` in the authorization URL template:

```python
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?response_type=code&state=%s&redirect_uri=%s'
)
```

**Additional Recommendations:**

1. **Verify Framework Behavior:** Confirm whether the `asfquart` framework adds `response_type` internally. If so, document this behavior.
2. **Consider PKCE:** For enhanced security, add PKCE parameters (`code_challenge`, `code_challenge_method`) to prevent authorization code interception attacks.
3. **Validate Callback Response:** Implement defense-in-depth by validating that the callback contains a `code` parameter and not token parameters.
4. **Re-evaluate OIDC Avoidance:** Consider whether the intentional bypass of OIDC (comment: `# Avoid OIDC`) is justified given the standardized security properties OIDC provides.

## Positive Security Patterns

1. **Authorization Code Flow Implied**  
   The callback URL template uses `code=%s` (line 41), confirming the application expects authorization codes rather than tokens directly. This aligns with the spirit of ASVS 10.6.1's preference for the `code` response type over implicit flows.

2. **HTTPS Enforced for OAuth Endpoints**  
   Both `OAUTH_URL_INIT` and `OAUTH_URL_CALLBACK` use `https://` URLs (lines 38-41), preventing token interception during transit via man-in-the-middle attacks.

3. **LDAPS for Directory Access**  
   The LDAP connection in `asf-load-ldap.py` uses `ldaps://` protocol (line 33), ensuring encrypted directory queries and preventing credential exposure.

4. **State Parameter Present**  
   The authorization URL includes `state=%s` (line 38-40), which is critical for CSRF protection in the OAuth flow, aligning with ASVS 10.3.x requirements.

5. **Consistent Authentication Enforcement**  
   All state-changing endpoints in `pages.py` have authentication decorators (`@asfquart.auth.require()`), ensuring the session established via OAuth is validated before processing sensitive operations.

6. **Session Validation**  
   The application consistently uses `asfquart.session.read()` (line 97 and throughout `pages.py`) to validate authenticated sessions before granting access to protected resources.

## Architecture Observations

1. **Relying Party Role**  
   This application functions as a Relying Party (RP) consuming tokens from `oauth.apache.org`, not as an OpenID Provider. ASVS 10.6.1 targets the OpenID Provider's configuration, which is managed externally. The OP's `response_type` restrictions cannot be verified from this codebase alone.

2. **Intentional OIDC Avoidance**  
   The explicit comment `# Avoid OIDC` (line 36) and override of `asfquart.generics` suggest a deliberate architectural decision to use plain OAuth rather than OpenID Connect. This means:
   - No ID Tokens are issued or consumed
   - OIDC-specific aspects of 10.6.1 (e.g., `id_token`, `id_token code` response types) are not applicable
   - The rationale for this decision is not documented in the code

3. **Framework Dependency**  
   The actual OAuth flow implementation resides within the `asfquart` framework, which was not provided for audit. The URL template override may or may not represent the complete picture of how authorization requests are constructed. The framework may:
   - Add `response_type` internally
   - Provide additional security controls
   - Handle token validation and session management

4. **External Trust Boundary**  
   The application places significant trust in the external OAuth provider (`oauth.apache.org`) to:
   - Implement secure defaults
   - Reject implicit flow requests
   - Properly validate redirect URIs
   - Issue secure authorization codes

5. **Session Management Model**  
   The application appears to use server-side sessions (via `asfquart.session`) backed by OAuth tokens, rather than client-side JWT sessions. This provides better security control but requires proper session storage and invalidation mechanisms.
