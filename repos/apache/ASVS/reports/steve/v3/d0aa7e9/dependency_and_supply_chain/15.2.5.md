This audit evaluates whether the STeVe voting application implements additional architectural protections (sandboxing, encapsulation, containerization, network isolation) around its documented dangerous functionality and risky third-party components. Three findings of MEDIUM or higher severity were identified, centered on the lack of process isolation for vote decryption, undermined encapsulation of database operations, and use of a side-channel-vulnerable Argon2 variant without compensating controls.

## Findings

### HIGH Severity

#### Finding ASVS-1525-SEV-001

**Severity:** HIGH  
**Finding ID:** ASVS-1525-SEV-001  
**Gap Type:** Type A — Dangerous functionality with NO isolation control  

**Title:** Vote Decryption/Tallying Functionality Lacks Process Isolation from Web Attack Surface

**Location:**  
- File: `v3/steve/election.py`
- Method: `tally_issue()`
- Lines: 284–349
- Related: `decrypt_votestring()` in `v3/steve/crypto.py`, lines 82–87

**Related Functions Checked:**
- `tally_issue()` — vote decryption orchestration
- `decrypt_votestring()` — actual decryption operation
- `gen_vote_token()` — key derivation for decryption
- `_all_metadata()` — key material retrieval

**Vulnerable Pattern:**

```python
# election.py - tally_issue() runs in the same process as web handlers
def tally_issue(self, iid):
    # ...
    md = self._all_metadata(self.S_CLOSED)
    # ...
    for mayvote in self.q_tally.fetchall():
        vote_token = crypto.gen_vote_token(
            md.opened_key,      # <-- Key material in web process memory
            mayvote.pid, iid, mayvote.salt,
        )
        row = self.q_recent_vote.first_row(vote_token)
        if row is None:
            continue
        votestring = crypto.decrypt_votestring(  # <-- Decrypts votes in-process
            vote_token, mayvote.salt, row.ciphertext,
        )
        votes.append(votestring)
```

**Data Flow:**
```
Web request → Election instance (same process) → tally_issue() → crypto.decrypt_votestring() → plaintext votes in memory
```

**Description:**

The `tally_issue()` method, which decrypts **all** encrypted votes for a given issue, resides in the same `Election` class and runs in the same process as web-facing request handlers. The `opened_key` (the master key material that, combined with per-voter salts, can decrypt every vote) is loaded into the web server's process memory during tallying. There is no:

- **Process isolation**: Tallying runs in the same Python process as HTTP handlers
- **Privilege separation**: No separate privilege level for decryption operations  
- **Sandboxing**: No OS-level sandbox (seccomp, AppArmor) constraints
- **Network isolation**: No evidence that tallying runs on a separate network segment

A vulnerability in any web handler (e.g., SSRF, template injection, deserialization flaw) could allow an attacker to invoke `tally_issue()` or access `opened_key` in process memory, compromising all vote secrecy.

**Impact:**

Complete loss of vote confidentiality. An attacker who compromises the web layer can decrypt all cast votes across all issues in an election.

**Proof of Concept:**

```python
# Hypothetical exploit via web handler vulnerability
# If attacker gains code execution in web process:
from steve.election import Election

# Attacker can directly call tallying from web context
election = Election("steve.db", "election-2024")
results = election.tally_issue("issue-1")  # Decrypts all votes
# Or access key material directly:
metadata = election._all_metadata(election.S_CLOSED)
master_key = metadata.opened_key  # All votes now decryptable
```

**Remediation:**

Implement process-level separation for tallying operations:

```python
# Option A: Separate tallying service (recommended for L3 compliance)
# tallying_service.py - runs as a separate process/container
import multiprocessing

def isolated_tally(db_fname, eid, iid, result_queue):
    """Run tallying in isolated subprocess with restricted privileges."""
    import os
    # Drop capabilities after opening DB
    election = Election(db_fname, eid)
    result = election.tally_issue(iid)
    result_queue.put(result)
    # Key material is destroyed when subprocess exits

# Option B: At minimum, restrict Election class API surface
class Election:
    # Remove __getattr__ proxy, use explicit method delegation
    # Separate TallyElection subclass for privileged operations
    pass

class TallyElection(Election):
    """Only instantiable from CLI/privileged context."""
    def tally_issue(self, iid):
        # ... decryption logic here only
        pass
```

### MEDIUM Severity

#### Finding ASVS-1525-SEV-002

**Severity:** MEDIUM  
**Finding ID:** ASVS-1525-SEV-002  
**Gap Type:** Type B — Encapsulation control EXISTS (state assertions) but NOT APPLIED to proxied cursors  

**Title:** `__getattr__` Proxy Undermines Encapsulation of Dangerous Database Operations

**Location:**  
- File: `v3/steve/election.py`
- Method: `__getattr__()`
- Line: 56

**Related Functions Checked:**
- `delete()` — uses `assert is_editable()`
- `open()` — uses `assert is_editable()`
- `close()` — uses `assert is_open()`
- `add_vote()` — uses `_all_metadata(S_OPEN)`
- All `c_*` cursors in `queries.yaml` — accessible via proxy

**Vulnerable Code:**

```python
class Election:
    def __getattr__(self, name):
        "Proxy the cursors."
        return self.__dict__.get(name, getattr(self.db, name))
```

**Description:**

The `Election` class defines explicit public methods with state-machine assertions to guard dangerous operations:

```python
def delete(self):
    assert self.is_editable()  # State check before destructive op
    # ... then performs c_delete_election, c_delete_issues, etc.
```

However, the `__getattr__` proxy exposes **all** database cursors defined in `queries.yaml` to any code holding an `Election` instance. This completely bypasses the state-machine protections:

```python
# Through public API - state assertion enforced:
election.delete()  # asserts is_editable()

# Through proxied cursor - NO state assertion:
election.c_delete_election.perform(eid)   # Direct deletion, no state check
election.c_open.perform(salt, key, eid)   # Open election, no state check
election.c_close.perform(eid)             # Close election, no state check
election.c_add_vote.perform(token, ct)    # Add vote, no state check
```

This means a programming error in any API handler that creates an `Election` instance could inadvertently invoke destructive or state-bypassing database operations without the intended safety checks.

**Data Flow:**

```
API handler → Election instance → __getattr__("c_delete_election") → db.c_delete_election → Direct DB operation (no state check)
```

**Impact:**

State-machine bypass allowing election deletion when open, vote insertion when closed, or direct manipulation of opened_key/salt values. Undermines the security architecture's reliance on state-based access control.

**Proof of Concept:**

```python
# In any API handler with Election access:
election = Election("steve.db", "election-2024")

# Intended safe path:
# election.delete()  # Would fail with assertion if not editable

# Bypass via proxy:
election.c_delete_election.perform(election.eid)  # Deletes regardless of state
election.c_open.perform(b"fake_salt", b"fake_key", election.eid)  # Opens closed election
```

**Remediation:**

Replace the open proxy with explicit, controlled delegation:

```python
class Election:
    # Remove __getattr__ proxy entirely
    # Instead, define explicit private properties for needed cursors:
    
    @property
    def _q_metadata(self):
        return self.db.q_metadata
    
    @property  
    def _c_delete_election(self):
        return self.db.c_delete_election
    
    # Or use __getattr__ with an allowlist:
    _ALLOWED_ATTRS = frozenset({
        'q_metadata', 'q_issues', 'q_check_election',
        # ... explicitly list each allowed cursor
    })
    
    def __getattr__(self, name):
        if name not in self._ALLOWED_ATTRS:
            raise AttributeError(f"Access to '{name}' is not permitted")
        return getattr(self.db, name)
```

#### Finding ASVS-1525-SEV-003

**Severity:** MEDIUM  
**Finding ID:** ASVS-1525-SEV-003  
**Gap Type:** Type A — Risky component (`argon2.low_level` with Type.D) with NO compensating controls  

**Title:** Argon2 Type.D Used for Critical Key Derivation Without Side-Channel Mitigations

**Location:**  
- File: `v3/steve/crypto.py`
- Function: `_hash()`
- Lines: 90–101

**Related Functions Checked:**
- `gen_opened_key()` — calls `_hash()` for master key
- `gen_vote_token()` — calls `_hash()` for per-voter keys
- `benchmark_argon2()` — uses Type.ID (correct variant)

**Vulnerable Code:**

```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2 to hash the data, with default tuning parameters."
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,  # <-- Side-channel vulnerable variant
    )
```

**Data Flow:**

```
Election opening → gen_opened_key() → _hash(passphrase, salt, Type.D) → master key
Vote casting → gen_vote_token() → _hash(opened_key + pid + iid, salt, Type.D) → encryption key
```

**Description:**

The application uses `argon2.low_level` (an explicitly low-level, risky API) with **Argon2d** (`Type.D`) for all critical key derivation: generating `opened_key` (election master key) and `vote_token` (per-voter encryption key). Argon2d is vulnerable to side-channel timing attacks because its memory access pattern is data-dependent.

This is notable because:

1. The **benchmark function** in the same file uses `Type.ID` (Argon2id), demonstrating awareness of the recommended variant:
   ```python
   # benchmark_argon2() at line 153:
   type=argon2.low_level.Type.ID,  # Uses Argon2id!
   ```

2. The application uses `argon2.low_level` directly instead of the high-level `argon2` API (which defaults to Argon2id and provides safe parameter choices).

3. No compensating controls exist (process isolation, sandboxing, memory protection) to mitigate the side-channel risk inherent in the `argon2.low_level` + Type.D combination.

Per OWASP and RFC 9106, **Argon2id** is the recommended variant for all general use cases, combining Argon2d's GPU resistance with Argon2i's side-channel resistance.

**Impact:**

In shared hosting or cloud environments where co-located processes could observe memory access patterns, an attacker could potentially recover `opened_key` or `vote_token` values, enabling vote decryption.

**Proof of Concept:**

```python
# Theoretical side-channel attack scenario:
# 1. Attacker co-locates process on same physical host
# 2. Monitors cache timing during _hash() execution
# 3. Observes data-dependent memory access patterns of Argon2d
# 4. Recovers portions of secret input (passphrase or opened_key)
# 5. Uses recovered key material to decrypt votes
```

**Remediation:**

```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2id to hash the data."
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.ID,  # Argon2id - recommended variant
    )

# Or better yet, use the high-level API:
from argon2 import PasswordHasher
ph = PasswordHasher(time_cost=2, memory_cost=65536, parallelism=4)
```

> **Note:** Changing the Argon2 type will alter derived keys, making existing encrypted votes unrecoverable. This change must be coordinated with a migration plan for any elections with existing votes.

## Positive Security Patterns

1. **Cryptographic randomness**: Consistent use of `secrets` module for all random values — salts (`secrets.token_bytes`), IDs (`secrets.token_hex`), and shuffle (`secrets.randbelow`). No use of `random` module.

2. **Parameterized queries**: All database operations use `asfpy.db` with YAML-defined queries, preventing SQL injection. No string concatenation for SQL construction found anywhere.

3. **State machine model**: Election state transitions (`editable → open → closed`) are enforced with assertions in all public methods, providing defense against invalid operations.

4. **Per-voter cryptographic separation**: Each voter/issue pair has a unique salt, and vote tokens are derived from `opened_key + pid + iid + salt`, ensuring votes cannot be correlated even if the database is compromised.

5. **Sensitive data exclusion**: `get_metadata()` explicitly excludes `salt` and `opened_key` from returned data. `get_issue()` similarly excludes `issue.salt`.

6. **Tamper detection**: `is_tampered()` recomputes the election hash from current data and compares against the stored `opened_key`, detecting modifications to election configuration.

7. **Transactional integrity**: Destructive operations (`delete`, `add_salts`) use explicit `BEGIN TRANSACTION`/`COMMIT` blocks.

8. **Cryptographic shuffle**: Vote order is randomized using Fisher-Yates with `secrets.randbelow()` before tallying results, preventing database-order correlation.

## Architecture Observations

1. **Single-process architecture**: The entire application — web framework, API handlers, cryptographic operations, and tallying — runs in a single Python process. This means a vulnerability in any web handler gives access to all functionality, including vote decryption.

2. **No containerization evidence**: No Dockerfile, docker-compose, or container orchestration configuration is visible in the provided files.

3. **No privilege separation**: The web server process has the same privileges as the tallying process. There is no capability dropping or privilege de-escalation after startup.

4. **Hazmat usage is module-contained but process-shared**: The `crypto.py` module properly centralizes all cryptographic operations, but since it runs in the same process, this is logical encapsulation only — not a security boundary.

5. **HKDF info parameter mismatch**: `_b64_vote_key()` uses `info=b'xchacha20_key'` but encrypts with Fernet (AES-CBC). This is functionally harmless but indicates planned migration that will require careful key derivation management.

## Recommendations Summary

### Immediate Actions (Critical/High Priority)

| Priority | Finding | Effort | Impact | Timeline |
|----------|---------|--------|--------|----------|
| 1 | **ASVS-1525-SEV-001**: Isolate tallying into a separate process/service | High | Prevents web compromise from enabling vote decryption | Before next production election |

**Rationale:** This HIGH severity finding represents the most significant architectural gap. Vote decryption running in the same process as web handlers violates the principle of least privilege and creates a single point of compromise for all vote confidentiality.

### Short-term Actions (Medium Priority, 1-3 months)

| Priority | Finding | Effort | Impact | Timeline |
|----------|---------|--------|--------|----------|
| 2 | **ASVS-1525-SEV-002**: Replace `__getattr__` proxy with explicit allowlisted delegation | Low | Restores state-machine encapsulation guarantees | Next release cycle |
| 3 | **ASVS-1525-SEV-003**: Switch from Argon2d to Argon2id | Low (new elections) / Medium (migration) | Eliminates side-channel attack vector in key derivation | Next major version |

**Rationale:** Both findings undermine existing security controls (state machine and cryptographic best practices). The fixes are relatively straightforward but require careful testing and, for SEV-003, a migration strategy.

### Long-term Actions (Architectural Improvements, 3-12 months)

| Priority | Recommendation | Effort | Impact |
|----------|---------------|--------|--------|
| 4 | Add containerization with separate containers for web and tallying | Medium | Provides OS-level isolation boundary |
| 5 | Implement network segmentation so tallying container cannot receive inbound connections | Medium | Prevents lateral movement from web to tallying |
| 6 | Add seccomp/AppArmor profiles restricting syscalls available to web process | Medium | Limits blast radius of web compromise |
| 7 | Implement privilege separation: web process runs as unprivileged user, tallying requires elevated privileges | High | Enforces least-privilege at OS level |

**Rationale:** These changes transform the application from a monolithic single-process design to a defense-in-depth architecture with multiple isolation boundaries, fully satisfying ASVS 15.2.5 Level 3 requirements.
