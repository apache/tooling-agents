## Findings

### HIGH Severity Findings

#### ASVS-1515-SEV-001: Security-Critical State Machine Enforcement Uses Python `assert` Statements (Undocumented Dangerous Pattern)

**Severity:** HIGH

**Location:** `v3/steve/election.py`
- `delete()` — Line 58
- `open()` — Line 77
- `close()` — Line 121
- `add_salts()` — Line 129
- `add_issue()` — Lines 190-191
- `edit_issue()` — Lines 210-211
- `delete_issue()` — Line 225
- `add_voter()` — Line 250

**Related Functions Checked:**
- `is_editable()`
- `is_open()`
- `get_state()`

**Description:**

The entire election state machine (EDITABLE → OPEN → CLOSED) is enforced through 8 `assert` statements. Python's `-O` flag (commonly used in production deployments for performance) strips all assertions at bytecode compilation. This means the state machine checks are eliminated entirely when Python runs with optimization enabled.

**Vulnerable Code:**

```python
# election.py:58
def delete(self):
    assert self.is_editable()   # <-- disabled with python -O

# election.py:77
def open(self, pdb):
    assert self.is_editable()   # <-- disabled with python -O

# election.py:121
def close(self):
    assert self.is_open()       # <-- disabled with python -O

# election.py:190-191
def add_issue(self, title, description, vtype, kv):
    assert self.is_editable()   # <-- disabled with python -O
    assert vtype in vtypes.TYPES # <-- disabled with python -O

# election.py:250
def add_voter(self, pid, iid=None):
    assert self.is_editable()   # <-- disabled with python -O
```

**Data Flow:**

Python runtime flag (`-O` or `PYTHONOPTIMIZE=1`) → all `assert` statements removed at bytecode compilation → state machine checks eliminated → election lifecycle protections bypassed entirely.

**Why This Is Dangerous Functionality:**

This is internally-developed dangerous functionality that could silently fail in production. When assertions are disabled:
- Issues can be added to open/closed elections
- Voters can be added after election opening
- Open elections can be deleted
- Elections can be opened/closed out of sequence

**Why Documentation Is Required:**

No documentation warns that:
1. The state machine depends on assertions
2. Running with `-O` disables all election integrity checks
3. Deployment configurations must never use `PYTHONOPTIMIZE`

**Proof of Concept:**

```bash
# Normal execution - assertion fires
python3 -c "from steve.election import Election; e = Election(...); e.open(...); e.delete()"
# AssertionError

# Optimized execution - no protection
python3 -O -c "from steve.election import Election; e = Election(...); e.open(...); e.delete()"
# Succeeds, corrupting election state
```

**Impact:**

Complete bypass of election lifecycle integrity if Python runs with optimization. An authenticated election owner could modify election structure after it has been opened, potentially invalidating cryptographic tamper detection.

**Remediation:**

```python
# Replace all security-critical assertions with explicit checks
def delete(self):
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
    # ... rest of method

def add_issue(self, title, description, vtype, kv):
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
    if vtype not in vtypes.TYPES:
        raise ValueError(f"Invalid vote type: {vtype}")
    # ... rest of method
```

Additionally, document this pattern in architecture documentation as a dangerous area requiring explicit runtime checks.

#### ASVS-1515-SEV-002: Argon2d (Type.D) Used in Production While Benchmark Uses Argon2id (Type.ID) — Undocumented Cryptographic Choice

**Severity:** HIGH

**Location:** `v3/steve/crypto.py`
- `_hash()` — Line 101 (production code using Type.D)
- `benchmark_argon2()` — Line 155 (benchmark code using Type.ID)

**Related Functions Checked:**
- `gen_opened_key()` — Uses `_hash()` for election tamper detection
- `gen_vote_token()` — Uses `_hash()` for per-voter-per-issue vote tokens

**Description:**

The `_hash()` function uses Argon2d (Type.D) while the benchmark function uses Argon2id (Type.ID). Argon2d is vulnerable to side-channel attacks (cache-timing) in shared hosting/cloud environments. OWASP Password Storage Cheat Sheet explicitly recommends Argon2id. The inconsistency with the benchmark function suggests this may be an oversight rather than a deliberate choice.

**Vulnerable Code:**

```python
# Production code (line 101) - uses Type.D
def _hash(data: bytes, salt: bytes) -> bytes:
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,  # <-- Argon2d: side-channel vulnerable
    )

# Benchmark code (line 155) - uses Type.ID
argon2.low_level.hash_secret_raw(
    ...
    type=argon2.low_level.Type.ID,  # <-- Argon2id: OWASP recommended
)
```

**Data Flow:**

Vote token / election ID → `_hash()` with Argon2d → derived keys → used for Fernet encryption and tamper detection

**Why This Is Dangerous Functionality:**

The `_hash()` function is the cryptographic foundation for:
- `gen_opened_key()` — election tamper detection key
- `gen_vote_token()` — per-voter-per-issue vote tokens used for encryption key derivation

This is a security-critical cryptographic decision that:
1. Affects all vote token generation and tamper detection
2. Contradicts the project's own benchmark code
3. Contradicts OWASP recommendations
4. Is not documented or justified anywhere in the code

**Why Documentation Is Required:**

No documentation explains or justifies this cryptographic algorithm choice, which contradicts industry best practices.

**Proof of Concept:**

In shared hosting or cloud environments with co-located attacker access, cache-timing analysis could potentially recover Argon2d outputs through side-channel observation.

**Impact:**

In shared hosting or cloud environments, an attacker with co-located access could potentially recover Argon2d outputs through cache-timing analysis, compromising vote tokens and election integrity keys.

**Remediation:**

```python
# Option 1: Switch to Argon2id (recommended)
def _hash(data: bytes, salt: bytes) -> bytes:
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.ID,  # Argon2id: side-channel resistant
    )

# Option 2: If Type.D is intentional, document extensively
# In SECURITY.md or architecture doc:
# "Argon2d is used intentionally because [justification].
#  This requires deployment in environments without shared memory/cache."
```

#### ASVS-1515-SEV-003: `cryptography.hazmat` and `argon2.low_level` API Usage Not Documented as Dangerous Functionality

**Severity:** MEDIUM

**Location:** `v3/steve/crypto.py`
- Lines 25-26 — `cryptography.hazmat` imports
- Line 23 — `argon2.low_level` import
- Line 62 — `_b64_vote_key()` using hazmat HKDF
- Lines 92-103 — `_hash()` using low-level Argon2

**Related Functions Checked:**
- `_b64_vote_key()` — HKDF key derivation
- `_hash()` — Argon2 hashing
- `create_vote()` — Fernet encryption
- `decrypt_votestring()` — Fernet decryption

**Description:**

The codebase uses two explicitly dangerous low-level cryptographic APIs without formal documentation:
- `cryptography.hazmat` module (explicitly named "hazardous materials" by maintainers)
- `argon2.low_level` module (bypasses high-level safety features)

**Vulnerable Code:**

```python
# Line 25-26: hazmat imports (explicitly labeled "hazardous materials" by the library)
from cryptography.hazmat.primitives import hashes
from cryptography.hazmat.primitives.kdf import hkdf

# Line 23: low-level Argon2 (bypasses high-level safety checks)
import argon2.low_level
```

**Why This Is Dangerous Functionality:**

- The `cryptography` library's `hazmat` module is **explicitly named** "hazardous materials" by its maintainers to signal that misuse can lead to severe vulnerabilities. The library's own documentation states: *"This is a 'Hazardous Materials' module. You should ONLY use it if you're 100% absolutely sure that you know what you're doing."*
- The `argon2.low_level` module bypasses the high-level `argon2-cffi` API's safety features (parameter validation, automatic encoding, type selection).

**Why Documentation Is Required:**

Per ASVS 15.1.5, application documentation must highlight where dangerous functionality is used. These two modules are the canonical examples of third-party components with inherently dangerous functionality. The code contains only brief inline comments (e.g., `### still using Fernet now, but will switch soon`) but no formal documentation that:
1. Inventories all hazmat/low-level crypto usage
2. Explains why high-level APIs were insufficient
3. Documents the security review status of these usages
4. Identifies the specific risks of each operation

**Impact:**

Without documentation, future developers or auditors may not recognize these areas require elevated scrutiny, potentially introducing subtle cryptographic vulnerabilities during maintenance.

**Remediation:**

Create a `SECURITY.md` or architecture document section that inventories dangerous functionality:

```markdown
## Dangerous Functionality Inventory

### cryptography.hazmat (crypto.py)
- **HKDF-SHA256** (`_b64_vote_key`): Used for key stretching of vote tokens.
  Low-level API required because Fernet needs specific key format.
  Risk: Incorrect parameter selection could weaken encryption keys.
  
### argon2.low_level (crypto.py)  
- **Argon2 hashing** (`_hash`): Used for opened_key generation and vote tokens.
  Low-level API required for raw byte output (high-level returns encoded string).
  Risk: Incorrect parameter tuning could weaken brute-force resistance.
  Parameters: time_cost=2, memory_cost=64MB, parallelism=4, Type=D
```

#### ASVS-1515-SEV-004: HKDF Info Parameter Labels XChaCha20 While Actual Encryption Uses Fernet (AES-128-CBC) — Undocumented Technical Debt

**Severity:** MEDIUM

**Location:** `v3/steve/crypto.py`
- `_b64_vote_key()` — Lines 65-73 (HKDF with mismatched info parameter)
- `create_vote()` — Uses derived key with Fernet encryption
- `decrypt_votestring()` — Decrypts with Fernet

**Related Functions Checked:**
- `_b64_vote_key()` — Key derivation
- `create_vote()` — Vote encryption
- `decrypt_votestring()` — Vote decryption

**Description:**

HKDF's `info` parameter provides domain separation — it ensures keys derived for different purposes produce different outputs. The `info` value `b'xchacha20_key'` creates a cryptographic binding to a specific algorithm that isn't actually being used.

**Vulnerable Code:**

```python
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    "Key-stretch the vote_token. (ref: PBKDF)"

    ### still using Fernet now, but will switch soon. Leaving comments.
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,           # 32-byte key for XChaCha20-Poly1305
        salt=salt,
        info=b'xchacha20_key',  # <-- Labels output as XChaCha20 key
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

The derived key is then used for Fernet encryption:
```python
def create_vote(vote_token: bytes, salt: bytes, votestring: str) -> bytes:
    b64key = _b64_vote_key(vote_token, salt)
    f = cryptography.fernet.Fernet(b64key)    # <-- Actually AES-128-CBC
    return f.encrypt(votestring.encode())
```

**Data Flow:**

Vote token → HKDF (info=`xchacha20_key`) → Fernet key → AES-128-CBC encryption (not XChaCha20)

**Why This Is Dangerous Functionality:**

When the planned migration to XChaCha20-Poly1305 occurs:
1. The `info` parameter is already set for XChaCha20, meaning the same derived key material would be reused
2. Key length (32 bytes) happens to work for both algorithms, masking the mismatch
3. Existing encrypted votes (under Fernet) would use keys with the same `info` as new votes (under XChaCha20), creating potential confusion during migration

**Why Documentation Is Required:**

The inline `###` comment acknowledges planned migration but no formal documentation:
1. Tracks this as a known dangerous technical debt item
2. Defines the migration plan to prevent key reuse
3. Explains why the current mismatch exists

**Impact:**

During the cryptographic migration, the shared `info` parameter could lead to key confusion between old (Fernet) and new (XChaCha20) ciphertexts, potentially causing decryption failures or, in the worst case, using the wrong key for the wrong algorithm.

**Remediation:**

```python
# Fix the info parameter to match actual algorithm
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,
        salt=salt,
        info=b'fernet_vote_key_v1',  # Matches actual algorithm + version
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

Document the migration plan:
```markdown
## Planned Cryptographic Migration: Fernet → XChaCha20-Poly1305
- Current: Fernet (AES-128-CBC + HMAC-SHA256), HKDF info=b'fernet_vote_key_v1'
- Target: XChaCha20-Poly1305, HKDF info=b'xchacha20_vote_key_v1'
- Migration requires re-encryption of active election votes
- HKDF info MUST change to ensure domain separation
```

## Positive Security Patterns

1. **`secrets` module used consistently** — All random generation (`gen_salt()`, `create_id()`, `shuffle()`) correctly uses `secrets` rather than `random`, ensuring cryptographic quality randomness.

2. **Per-voter-per-issue salts** — Each mayvote row gets a unique salt, preventing correlation between voters or issues. Salt generation uses `secrets.token_bytes(16)`.

3. **Sensitive fields excluded from public APIs** — `get_metadata()` explicitly excludes `salt` and `opened_key`. `get_issue()` comments "NEVER return issue.salt".

4. **Cryptographic shuffling** — Vote ordering is randomized using Fisher-Yates with `secrets.randbelow()`, preventing information leakage from database ordering during tallying.

5. **Transaction-based database operations** — `delete()` and `add_salts()` use explicit `BEGIN TRANSACTION` / `COMMIT` for data integrity.

6. **Tamper detection** — `is_tampered()` recomputes the election hash and compares against the stored `opened_key`, detecting unauthorized modifications.

7. **Collision-resistant ID generation** — Both `create()` and `add_issue()` retry with new random IDs on `IntegrityError`.

## Architecture Observations

1. **Cryptographic layering is well-designed**: The chain `BLAKE2b → Argon2 → HKDF → Fernet` provides defense-in-depth with each layer serving a distinct purpose (compression, key strengthening, key derivation, authenticated encryption).

2. **State machine model is sound but implementation is fragile**: The EDITABLE → OPEN → CLOSED lifecycle with tamper detection is a good security design, but reliance on `assert` for enforcement undermines it.

3. **No formal dangerous functionality inventory exists**: The codebase contains significant cryptographic complexity spread across `crypto.py` and `election.py` with no single document cataloging these areas for security review.

4. **Third-party dependency documentation gap**: While the code uses `argon2-cffi`, `cryptography`, `asfpy`, and `easydict`, there is no visible SBOM, dependency review documentation, or version pinning verification in the provided files.
