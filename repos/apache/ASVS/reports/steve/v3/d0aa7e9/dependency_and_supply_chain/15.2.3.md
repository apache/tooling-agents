## Findings

### MEDIUM Severity

#### Finding ASVS-1523-SEV-001

| Attribute | Detail |
|---|---|
| **Severity** | **MEDIUM** |
| **File** | `v3/steve/crypto.py`, lines 129–158 |
| **Function** | `benchmark_argon2()` |
| **Gap Type** | Type A — Development/test function present in production module |

**Vulnerable Code:**

```python
def benchmark_argon2():
    # Test Data
    dummy_data = b'This is a sample datum representing one of your 1000 entries.'
    salt = b'16_byte_salt_123'  # 16 bytes
    intermediate = hashlib.blake2b(dummy_data).digest()

    # Define Low/High levels for the 3 parameters
    options = {'rounds': [2, 4], 'memory': [65536, 131072], 'parallelism': [4, 8]}

    print(f'{"Rounds":<8} | {"Memory (MB)":<12} | {"Threads":<8} | {"Time (s)":<10}')
    print('-' * 50)

    for r in options['rounds']:
        for m in options['memory']:
            for p in options['parallelism']:
                start = time.perf_counter()
                argon2.low_level.hash_secret_raw(
                    secret=intermediate, salt=salt, time_cost=r,
                    memory_cost=m, parallelism=p, hash_len=32,
                    type=argon2.low_level.Type.ID,
                )
                duration = time.perf_counter() - start
                print(f'{r:<8} | {m // 1024:<12} | {p:<8} | {duration:.4f}s')

if __name__ == '__main__':
    print('--- Argon2 Parameter Benchmarking ---')
    benchmark_argon2()
```

**Data Flow:** `import crypto` → `crypto.benchmark_argon2()` available at runtime → executes 8 CPU/memory-intensive Argon2 operations with up to 128MB memory each.

**Impact:**
1. **Resource exhaustion**: The function is importable from any code path. If reachable through any server-side codepath (e.g., an overly permissive import or debugging route), calling `benchmark_argon2()` executes 8 Argon2 iterations consuming up to 128MB memory each, creating a denial-of-service vector.
2. **Information disclosure**: It uses hardcoded test salts and `print()` statements that write to stdout/logs, potentially exposing Argon2 tuning parameters and timing information to an attacker analyzing logs.
3. **Algorithm mismatch**: The benchmark uses `argon2.Type.ID` while production `_hash()` uses `argon2.Type.D`, meaning the benchmark doesn't even represent production behavior — indicating it is purely development tooling.

**Proof of Concept:** Any Python code running in the same process can call:
```python
from steve import crypto
crypto.benchmark_argon2()  # ~1GB memory spike, blocking CPU
```

**Remediation:**
Move the benchmark to a separate development-only script (e.g., `tools/benchmark_argon2.py`) excluded from the production deployment package.

```python
# Remove from crypto.py:
# - benchmark_argon2() function (lines 129-158)
# - if __name__ == '__main__' block (lines 160-162)  
# - import time (line 26, if unused elsewhere)

# Create separate file: tools/benchmark_argon2.py
#!/usr/bin/env python3
"""Argon2 parameter benchmarking tool - NOT for production deployment."""
import time, hashlib
import argon2.low_level
# ... benchmark code here ...
```

#### Finding ASVS-1523-SEV-002

| Attribute | Detail |
|---|---|
| **Severity** | **MEDIUM** |
| **File** | `v3/server/main.py`, lines 85–96 |
| **Function** | `run_asgi()` |
| **Gap Type** | Type A — Development-level logging configuration in production deployment path |

**Vulnerable Code:**

```python
def run_asgi():
    "Run as an ASGI process; eg. Hypercorn"

    # NOTE: no-op if Hypercorn has set up the root logger.
    logging.basicConfig(
        level=logging.DEBUG,       # <-- DEBUG in production
        style='{',
        format='[{asctime}|{levelname}|{name}] {message}',
        datefmt=DATE_FORMAT,
    )

    # ...
    _LOGGER.setLevel(logging.DEBUG)  # <-- Explicit DEBUG on app logger
```

**Data Flow:** Hypercorn starts → `run_asgi()` called → DEBUG level set → all application-level debug messages (including cryptographic operations, database queries, election state transitions) are written to production logs.

**Impact:**
The `run_asgi()` function is the **production** code path (triggered when the module is imported by Hypercorn, not run directly). Setting `_LOGGER.setLevel(logging.DEBUG)` on line 96 is unconditional and always takes effect. Within `election.py`, debug-level log messages include election IDs and operation details:
```python
_LOGGER.debug(f'{op} election ID "{eid}"')  # election.py:41
_LOGGER.debug('EID conflict(!!) ... trying again.')  # election.py:423
_LOGGER.debug('IID conflict(!!) ... trying again.')  # election.py:193
```

While these specific messages are relatively benign, the DEBUG level setting means **any future debug logging added anywhere in the application** will automatically be exposed in production — creating a latent information disclosure risk. This is characteristic of development configuration that was not hardened for production.

**Remediation:**

```python
def run_asgi():
    "Run as an ASGI process; eg. Hypercorn"

    logging.basicConfig(
        level=logging.INFO,  # Production-appropriate level
        style='{',
        format='[{asctime}|{levelname}|{name}] {message}',
        datefmt=DATE_FORMAT,
    )

    # Use INFO for production; override with environment variable if needed
    import os
    log_level = os.environ.get('STEVE_LOG_LEVEL', 'INFO').upper()
    _LOGGER.setLevel(getattr(logging, log_level, logging.INFO))

    _LOGGER.info(' ** Run-mode: ASGI')

    global steve_app
    steve_app = create_app()
```

## Positive Security Patterns

1. **Proper entry point separation** (`main.py:108-118`): The `if __name__ == '__main__'` gate correctly separates standalone development mode from ASGI production mode.

2. **No use of dangerous functions**: No instances of `eval()`, `exec()`, `pickle.loads()`, or `subprocess` with `shell=True` were found in any of the reviewed files.

3. **Cryptographic module is well-structured** (`crypto.py:34-126`): Production cryptographic functions use appropriate algorithms (`secrets`, Argon2, HKDF, Fernet) with proper parameterization. The benchmark function is the only extraneous element.

4. **State machine enforcement** (`election.py`): All sensitive operations (`delete`, `open`, `close`, `add_vote`, `tally_issue`) include state assertions that prevent operations in wrong states.

5. **Sensitive data exclusion in public APIs** (`election.py:155-169`): `get_metadata()` explicitly excludes `SALT` and `OPENED_KEY` from returned data. Similarly, `get_issue()` notes "NEVER return issue.salt."

6. **No test endpoints visible**: The `create_app()` function imports `pages` and `api` modules without any visible test or debug route registration in the reviewed code.

---

## Architecture Observations

1. **Development-to-production transition artifacts**: Multiple `###` comments throughout the codebase indicate active development decisions that have not been cleaned up for production. The HKDF `info=b'xchacha20_key'` parameter combined with the comment "still using Fernet now, but will switch soon" indicates the cryptographic layer is in a transitional state that should be resolved before production deployment.

2. **Commented-out key printing**: In `election.py`, at least 4 commented-out `print()` statements would expose cryptographic keys (`opened_key`, `salt`) if accidentally uncommented during debugging. These should be removed entirely rather than left as comments — use proper `_LOGGER.debug()` calls with a named logger that can be filtered in production.

3. **Missing `pages` and `api` modules**: These dynamically imported modules define the actual HTTP endpoints but were not provided for review. A complete ASVS 15.2.3 audit requires verifying these modules do not expose debug endpoints, test routes, or development-only functionality.
