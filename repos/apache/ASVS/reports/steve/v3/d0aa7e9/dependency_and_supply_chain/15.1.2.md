## Findings

### High Severity

#### ASVS-1512-HIGH-001: No SBOM or Dependency Manifest Exists

**Severity:** HIGH

**Location:** Project root (missing file — no `requirements.txt`, `setup.py`, `pyproject.toml`, `uv.lock`, or SBOM document in CycloneDX/SPDX format)

**Related Functions Checked:**
- All import statements in `main.py`, `crypto.py`, `election.py`
- Project root for manifest files

**Description:**

The codebase has no formal dependency manifest or SBOM documentation. The `main.py` shebang line indicates `uv` as a potential dependency manager, but the required PEP 723 inline metadata block is absent. Dependencies are only discoverable through manual import tracing across multiple files.

**Vulnerable Code:**

`v3/server/main.py` (line 1):
```python
#!/usr/bin/env -S uv run --script
```

The `uv run --script` command expects inline dependency metadata in a `# /// script` block (PEP 723), which is completely absent from the file.

`v3/steve/crypto.py` (lines 21-24):
```python
import argon2.low_level
import cryptography.fernet
from cryptography.hazmat.primitives import hashes
from cryptography.hazmat.primitives.kdf import hkdf
```

`v3/steve/election.py` (lines 24-25):
```python
import asfpy.db
from easydict import EasyDict as edict
```

`v3/server/main.py` (lines 29, 37-38):
```python
import asfquart.generics
import pages  # noqa: F401
import api  # noqa: F401
```

**Data Flow:**

Developer installs dependencies ad-hoc → No version locked → Production environment could have different/vulnerable versions than development → Supply chain compromise possible → No audit trail of what was deployed

**Proof of Concept:**

Without a manifest file, attempting to use automated security tools fails:
```bash
$ pip-audit
ERROR: No dependency manifest found
$ osv-scanner .
WARNING: No lockfile found
$ trivy fs .
WARN: No dependency files found
```

**Impact:**

- **No Reproducible Builds:** Dependency versions may drift across environments
- **Vulnerability Scanning Impossible:** Tools like Dependabot, Snyk, and Trivy cannot operate without a manifest
- **Incident Response Degraded:** Cannot quickly identify whether a newly-disclosed CVE (e.g., in `cryptography`) affects this deployment
- **Compliance Failure:** SOC 2, ISO 27001, and other frameworks requiring supply chain documentation cannot be satisfied
- **Transitive Dependencies Invisible:** Dependencies like `cffi` via `argon2-cffi`, `werkzeug` via `quart`, `pyyaml` via `asfpy` are completely undocumented

**Remediation:**

1. Create a `pyproject.toml` with pinned dependencies:
```toml
[project]
name = "steve-voting"
version = "3.0.0"
requires-python = ">=3.11"
dependencies = [
    "asfquart>=0.x.x",
    "asfpy>=0.x.x",
    "cryptography>=43.0.0",
    "argon2-cffi>=23.1.0",
    "easydict>=1.13",
]
```

2. Generate and maintain a lock file:
```bash
uv lock
# or
pip-compile --generate-hashes requirements.in > requirements.txt
```

3. Generate a machine-readable SBOM:
```bash
# CycloneDX format
cyclonedx-py environment -o sbom.json --format json
# or SPDX format
syft . -o spdx-json > sbom.spdx.json
```

4. Commit SBOM to version control and regenerate on every dependency change

#### ASVS-1512-HIGH-002: No Version Pinning for Security-Critical Cryptographic Libraries

**Severity:** HIGH

**Location:** All import statements across `crypto.py` and `election.py`

**Related Functions Checked:**
- `crypto.py`: `_b64_vote_key()`, `encrypt_vote()`, `decrypt_vote()`, `integrity_hash()`
- All cryptographic operations depending on `cryptography` and `argon2-cffi`

**Description:**

The `cryptography` library is used for core vote encryption/decryption operations, and `argon2-cffi` is used for key derivation and integrity checking. Neither has any version constraint specified anywhere in the provided codebase. This creates significant security and stability risks.

**Vulnerable Code:**

`v3/steve/crypto.py` (lines 21-24):
```python
import argon2.low_level
import cryptography.fernet
from cryptography.hazmat.primitives import hashes
from cryptography.hazmat.primitives.kdf import hkdf
```

These imports have no version constraints, allowing any version to be installed.

`v3/steve/crypto.py` (line 87) — Low-level argon2 usage:
```python
argon2.low_level.hash_secret_raw(
    secret=secret,
    salt=salt,
    time_cost=time_cost,
    memory_cost=memory_cost,
    parallelism=parallelism,
    hash_len=hash_len,
    type=argon2.low_level.Type.ID,
)
```

**Data Flow:**

Vote data → `encrypt_vote()` → `_b64_vote_key()` (uses HKDF from `cryptography`) → Fernet encryption (uses `cryptography`) → Encrypted ballot stored

Election key + voter salt → `integrity_hash()` → `argon2.low_level.hash_secret_raw()` → Integrity token

**Impact:**

- **Known CVEs:** The `cryptography` library has had multiple CVEs:
  - CVE-2023-49083 affecting versions <41.0.6
  - CVE-2024-26130 affecting versions <42.0.4
  - Without version pinning, a deployment could inadvertently use a vulnerable version
- **Breaking API Changes:** Major version bumps could cause silent behavioral changes in cryptographic operations, potentially compromising vote integrity
- **Low-Level API Sensitivity:** The `argon2.low_level` API used directly at line 87 is particularly sensitive to version changes that might alter parameter validation or hashing behavior
- **Supply Chain Attacks:** Without version pinning, a compromised package update could be automatically installed

**Remediation:**

1. Pin to minimum safe versions with upper bounds on major versions:
```toml
[project]
dependencies = [
    "cryptography>=43.0.0,<44",
    "argon2-cffi>=23.1.0,<24",
]
```

2. Use hash verification for critical packages:
```txt
cryptography==43.0.0 \
    --hash=sha256:abc123...
argon2-cffi==23.1.0 \
    --hash=sha256:def456...
```

3. Implement automated dependency update checking (e.g., Dependabot or Renovate) with mandatory security review for cryptographic library updates

4. Document the minimum required versions based on security requirements and test compatibility before updates

## Positive Security Patterns

1. **Trusted Library Sources:** All identified third-party libraries (`cryptography`, `argon2-cffi`, `asfpy`, `asfquart`) are from well-known, reputable sources — PyCA, ASF, and mainstream PyPI packages. No evidence of libraries from unknown or untrusted repositories.

2. **Standard Library Preference for Core Operations:** The code appropriately uses Python's `secrets` module (not `random`) for cryptographic randomness, `hashlib` for BLAKE2b, and `json` for serialization — reducing third-party dependency surface for security-critical operations.

3. **No Dangerous Function Usage:** No instances of `eval()`, `exec()`, `pickle.loads()`, or `subprocess` with `shell=True` were found in any of the audited files.

4. **Parameterized Database Queries:** Database access through `asfpy.db` with YAML-defined queries and `.perform()` / `.first_row()` methods indicates parameterized query patterns, preventing SQL injection regardless of the database wrapper version.

5. **Cryptographic Library Quality:** The `cryptography` library (maintained by PyCA with funded security audits) and `argon2-cffi` (reference Argon2 implementation bindings) are industry-standard choices for their respective operations.

6. **Minimal Dependency Footprint:** The codebase uses only five direct third-party dependencies, limiting the attack surface compared to projects with dozens of dependencies.

## Architecture Observations

1. **`uv` Script Mode Without Metadata:** The `#!/usr/bin/env -S uv run --script` shebang in `main.py` suggests the project intends to use `uv` for dependency management, but the PEP 723 inline metadata block (`# /// script`) that `uv run --script` requires is absent. This means the script runner cannot resolve dependencies automatically, indicating either an incomplete migration to `uv` or dependencies being installed system-wide without management.

2. **HKDF Info String / Algorithm Mismatch:** The `_b64_vote_key` function (crypto.py:53-64) uses `info=b'xchacha20_key'` but the actual encryption uses Fernet (AES-128-CBC with HMAC). While not a security vulnerability, this documentation-code divergence complicates future SBOM maintenance when the planned XChaCha20-Poly1305 migration occurs and the `cryptography` dependency requirements change.

3. **ASF Internal Dependencies:** Both `asfquart` and `asfpy` are ASF-internal libraries. While trusted within the ASF ecosystem, their maintenance lifecycle may not follow the same public release cadence as mainstream PyPI packages, making version tracking and vulnerability monitoring more challenging without a formal inventory.

4. **Transitive Dependencies Completely Undocumented:** The direct dependencies create a significant transitive dependency tree that is not documented anywhere:

| Direct Dependency | Known Transitive Dependencies |
|---|---|
| `asfquart` | `quart`, `hypercorn`, `werkzeug`, `jinja2`, `markupsafe`, `click` |
| `asfpy` | `pyyaml`, `requests`, `urllib3`, `certifi` |
| `cryptography` | `cffi`, `pycparser`, OpenSSL bindings |
| `argon2-cffi` | `argon2-cffi-bindings`, `cffi` |
| `easydict` | (none) |

The `queries.yaml` file referenced at `election.py` line 34 is loaded by `asfpy.db`, which implies a YAML parsing dependency (`pyyaml`) that is completely invisible in the direct imports.

5. **`easydict` as Potentially Risky Component:** The `easydict` library used throughout `election.py` (lines 155-161, 220-226) has infrequent releases and minimal security-focused development. Per ASVS section description, this qualifies as a "risky component" that is "poorly maintained." It could be replaced by Python standard library alternatives (`types.SimpleNamespace`, `dataclasses`, or `typing.TypedDict`).
