### Key Observations

The application makes extensive use of Argon2 hashing (64 MiB memory, 4 threads per invocation) as a core cryptographic primitive for vote token generation. While the cryptographic design is sound, several code paths allow these expensive operations to be triggered repeatedly or at scale without adequate resource bounds, throttling, or caching mechanisms. This creates exploitable vectors for availability degradation by authenticated users.

### Medium Severity

#### ASVS-1522-MED-001: Repeated Vote Submissions Trigger Unbounded Argon2 Computation Without Throttling

**Severity:** MEDIUM

**Location:**
- **File:** `v3/steve/election.py`
- **Function:** `add_vote()`
- **Lines:** 266–280

**Related Functions Checked:**
- `crypto.gen_vote_token()` (calls Argon2)
- `crypto.create_vote()` (HKDF + Fernet encryption)
- `c_add_vote.perform()` (database INSERT)

**Description:**

The `add_vote()` method allows authenticated eligible voters to submit votes without any rate limiting or throttling mechanism. Each vote submission triggers an expensive Argon2 computation (64 MiB memory, 4 CPU threads, ~100ms) before validation or deduplication checks. The code includes a TODO comment acknowledging missing votestring validation, and there is no mechanism to prevent rapid repeated submissions.

**Vulnerable Code:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting   # <-- TODO: no validation

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)

    # Pass MAYVOTE.SALT for PBKDF.
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)

    self.c_add_vote.perform(vote_token, ciphertext)
```

**Data Flow:**

1. Authenticated user → web endpoint (api.py, not provided) → `add_vote()`
2. Each call triggers `crypto.gen_vote_token()` → `_hash()` → Argon2 (64 MiB, ~100ms, 4 threads)
3. Then `crypto.create_vote()` → HKDF + Fernet encryption
4. Then `c_add_vote` INSERT (multiple vote records are stored, with latest-wins semantics)
5. **No throttling, rate limiting, or deduplication before the expensive Argon2 computation**

**Proof of Concept:**

An authenticated eligible voter could script rapid repeated POST requests to the vote submission endpoint. Each request forces:
- 1× Argon2 computation (64 MiB memory allocation, 4 CPU threads, ~100ms)
- 1× HKDF + Fernet encryption
- 1× database INSERT

At 10 concurrent requests/second, this consumes ~640 MiB peak memory and saturates 40 CPU threads, degrading service for all other users.

**Impact:**

Authenticated user can degrade server availability by saturating CPU and memory with Argon2 computations through rapid vote resubmission. This affects all concurrent users of the election system.

**Remediation:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    
    # 1. Validate votestring before expensive operations
    issue = self.q_get_issue.first_row(iid)
    if not issue:
        raise IssueNotFound(iid)
    vtypes.validate_votestring(issue.type, votestring, self.json2kv(issue.kv))
    
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    if not mayvote:
        raise VoterNotEligible(pid, iid)
    
    # 2. Consider short-circuit: check if identical vote already exists
    #    before computing expensive token
    
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

Additionally, implement rate limiting at the web layer:

```python
from quart_rate_limiter import RateLimiter, rate_limit

rate_limiter = RateLimiter(app)

@app.route('/api/vote', methods=['POST'])
@rate_limit(5, timedelta(minutes=1))  # 5 votes per minute per user
async def submit_vote():
    ...
```

#### ASVS-1522-MED-002: `has_voted_upon()` Performs O(N) Argon2 Operations Per Request Without Caching or Bounds

**Severity:** MEDIUM

**Location:**
- **File:** `v3/steve/election.py`
- **Function:** `has_voted_upon()`
- **Lines:** 350–375

**Related Functions Checked:**
- `q_find_issues.perform()` (database query)
- `crypto.gen_vote_token()` (Argon2 per issue)
- `q_has_voted.first_row()` (database query per issue)

**Description:**

The `has_voted_upon()` method iterates over all issues a voter is eligible for and computes an Argon2 hash for each one to generate vote tokens. This operation scales linearly with the number of issues (O(N)) and is likely called on every page load when voters view their election dashboard. There is no caching of computed vote tokens between requests and no upper bound on the iteration count.

**Vulnerable Code:**

```python
def has_voted_upon(self, pid):
    "Return {ISSUE-ID: BOOL} stating what has been voted upon."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    voted_upon = {}

    self.q_find_issues.perform(pid, self.eid)
    for row in self.q_find_issues.fetchall():
        vote_token = crypto.gen_vote_token(   # Argon2 per issue!
            md.opened_key,
            pid,
            row.iid,
            row.salt,
        )
        voted = self.q_has_voted.first_row(vote_token)
        voted_upon[row.iid] = voted is not None

    return voted_upon
```

**Data Flow:**

1. Authenticated voter → web endpoint (likely election status/dashboard page) → `has_voted_upon(pid)`
2. Iterates over **all issues** voter is eligible for in the election
3. Each iteration computes `gen_vote_token()` → Argon2 (64 MiB, ~100ms, 4 threads)
4. No caching of computed vote tokens between requests
5. No upper bound on iteration count

**Resource Impact Calculation:**

| Issues per Election | Argon2 Calls | Approx. CPU Time | Peak Memory |
|---|---|---|---|
| 10 | 10 | ~1.0s | 64 MiB |
| 25 | 25 | ~2.5s | 64 MiB |
| 50 | 50 | ~5.0s | 64 MiB |

Each page load for a voter viewing their status triggers this entire computation. With concurrent users refreshing the page, server CPU is rapidly saturated.

**Impact:**

Legitimate voters refreshing their election dashboard cause O(issues) Argon2 operations per request, with no caching. Concurrent access during active elections can degrade availability for all users.

**Remediation:**

```python
import functools
import time

# Cache vote tokens per session/request to avoid recomputation
def has_voted_upon(self, pid):
    md = self._all_metadata(self.S_OPEN)
    voted_upon = {}

    self.q_find_issues.perform(pid, self.eid)
    issues = self.q_find_issues.fetchall()
    
    # Bound the number of issues processed per request
    MAX_ISSUES_PER_CHECK = 100
    if len(issues) > MAX_ISSUES_PER_CHECK:
        raise TooManyIssues(self.eid, len(issues))
    
    for row in issues:
        vote_token = crypto.gen_vote_token(
            md.opened_key, pid, row.iid, row.salt,
        )
        voted = self.q_has_voted.first_row(vote_token)
        voted_upon[row.iid] = voted is not None

    return voted_upon
```

Consider a time-limited cache for vote status at the web layer to avoid re-computation on page refreshes.

#### ASVS-1522-MED-003: `tally_issue()` Computes Argon2 for Every Eligible Voter Without Resource Bounds or Timeout

**Severity:** MEDIUM

**Location:**
- **File:** `v3/steve/election.py`
- **Function:** `tally_issue()`
- **Lines:** 282–348

**Related Functions Checked:**
- `q_tally.perform()` (queries all eligible voters)
- `crypto.gen_vote_token()` (Argon2 per voter)
- `crypto.decrypt_votestring()` (HKDF + Fernet per actual vote)
- `crypto.shuffle()` (anonymization)
- `vtypes.vtype_module().tally()` (vote counting)

**Description:**

The `tally_issue()` method queries all eligible voters for an issue and computes an Argon2 hash for each one to derive their vote token, regardless of whether they actually voted. This scales linearly with the number of eligible voters (O(N)) and can result in extremely long-running operations for large elections. While tallying is documented as a privileged CLI operation, the method itself has no enforcement of this restriction and would monopolize server resources if called during normal operations.

**Vulnerable Code:**

```python
def tally_issue(self, iid):
    md = self._all_metadata(self.S_CLOSED)
    issue = self.q_get_issue.first_row(iid)
    voters = set()
    votes = []

    self.q_tally.perform(iid)
    for mayvote in self.q_tally.fetchall():          # ALL eligible voters
        vote_token = crypto.gen_vote_token(           # Argon2 per voter!
            md.opened_key, mayvote.pid, iid, mayvote.salt,
        )
        row = self.q_recent_vote.first_row(vote_token)
        if row is None:
            continue
        voters.add(mayvote.pid)
        votestring = crypto.decrypt_votestring(       # HKDF + Fernet per vote
            vote_token, mayvote.salt, row.ciphertext,
        )
        votes.append(votestring)

    crypto.shuffle(votes)
    m = vtypes.vtype_module(issue.type)
    return m.tally(votes, self.json2kv(issue.kv)), voters
```

**Data Flow:**

1. Tally request → `tally_issue(iid)`
2. Queries **all eligible voters** for the issue (not just those who voted)
3. Computes Argon2 for **every eligible voter** to derive their vote token
4. Then queries DB with the token to check if they voted
5. If they voted, additionally performs HKDF + Fernet decryption

**Resource Impact Calculation:**

| Eligible Voters | Argon2 Calls | Approx. CPU Time | Peak Memory |
|---|---|---|---|
| 100 | 100 | ~10s | 64 MiB |
| 500 | 500 | ~50s | 64 MiB |
| 1,000 | 1,000 | ~100s | 64 MiB |

**Impact:**

Tallying monopolizes server CPU for extended periods proportional to voter count. On shared infrastructure, this degrades web application availability during tallying. Even as a CLI operation, running on the same server as the web application will compete for the same CPU and memory resources.

**Remediation:**

```python
import asyncio

def tally_issue(self, iid, progress_callback=None):
    md = self._all_metadata(self.S_CLOSED)
    issue = self.q_get_issue.first_row(iid)
    
    self.q_tally.perform(iid)
    all_mayvote = self.q_tally.fetchall()
    
    # Log expected resource consumption
    _LOGGER.info(f'Tallying issue {iid}: {len(all_mayvote)} eligible voters '
                 f'(estimated {len(all_mayvote) * 0.1:.0f}s CPU time)')
    
    voters = set()
    votes = []
    
    for idx, mayvote in enumerate(all_mayvote):
        vote_token = crypto.gen_vote_token(
            md.opened_key, mayvote.pid, iid, mayvote.salt,
        )
        # ... rest of processing ...
        
        # Optional: yield control periodically
        if progress_callback and idx % 50 == 0:
            progress_callback(idx, len(all_mayvote))
```

Consider running tally operations in a separate process or with CPU affinity to isolate resource impact from the web server.

#### ASVS-1522-MED-004: Missing Votestring Validation Before Resource-Intensive Cryptographic Operations

**Severity:** MEDIUM

**Location:**
- **File:** `v3/steve/election.py`
- **Function:** `add_vote()`
- **Line:** 273

**Related Functions Checked:**
- `crypto.create_vote()` (Fernet encryption of arbitrary-length input)
- `c_add_vote.perform()` (database storage)

**Description:**

The `add_vote()` method contains an explicit TODO comment indicating that votestring validation is missing. No validation of votestring length, format, or content against the issue's vote type occurs before passing the data to expensive cryptographic operations. This allows voters to submit arbitrarily large or malformed votestrings that consume disproportionate resources during encryption, storage, and later decryption during tallying.

**Vulnerable Code:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting    # <-- Explicit TODO

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

**Data Flow:**

1. Authenticated voter submits arbitrary `votestring` content → `add_vote()`
2. **No validation** of votestring length, format, or content against the issue's vote type
3. Votestring passed directly to `crypto.create_vote()` which Fernet-encrypts it
4. Encrypted (expanded) ciphertext stored in database

**Proof of Concept:**

A voter submits a votestring of 10 MiB (or whatever the web framework's request size limit allows). This bypasses all vote-type validation (since none exists) and:
- Fernet encrypts the full 10 MiB payload (CPU + memory)
- Stores the encrypted blob in SQLite (storage consumption)
- During tallying, must decrypt the full blob

Repeated submissions of oversized votestrings compound storage and processing costs.

**Impact:**

Lack of input validation before expensive operations allows disproportionate resource consumption per request. The explicit TODO comment confirms this is acknowledged but unimplemented. This can lead to storage exhaustion and increased processing time during tallying operations.

**Remediation:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)

    # Validate votestring BEFORE expensive crypto operations
    issue = self.q_get_issue.first_row(iid)
    if not issue:
        raise IssueNotFound(iid)
    
    # Hard limit on votestring size
    MAX_VOTESTRING_LEN = 4096  # generous for any vote type
    if len(votestring) > MAX_VOTESTRING_LEN:
        raise ValueError(f'Votestring exceeds maximum length ({MAX_VOTESTRING_LEN})')
    
    # Type-specific validation
    m = vtypes.vtype_module(issue.type)
    m.validate(votestring, self.json2kv(issue.kv))
    
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

## Positive Security Patterns

1. **Cryptographically secure randomness throughout**: All random generation uses `secrets` module (`token_bytes`, `token_hex`, `randbelow`). No use of `random` module.

2. **Parameterized database queries**: All SQL access goes through `asfpy.db` with YAML-defined queries and `.perform()` parameter binding, preventing SQL injection.

3. **Transaction management**: Critical multi-step operations (`delete()`, `add_salts()`) use explicit `BEGIN TRANSACTION` / `COMMIT` for atomicity.

4. **State machine enforcement**: Election state transitions are validated via `_all_metadata(required_state=...)` with proper exceptions (not just assertions) for the most sensitive operations (`add_vote`, `tally_issue`, `has_voted_upon`).

5. **Metadata sanitization**: `get_metadata()` explicitly excludes `salt` and `opened_key` from returned data.

6. **Collision-safe ID generation**: `create()` and `add_issue()` handle `IntegrityError` with retry loops for the (unlikely) case of ID collision.

7. **Anti-tamper mechanism**: `is_tampered()` recomputes the election hash to detect unauthorized modifications, providing a consistency check.

8. **No dangerous functions**: No use of `eval()`, `exec()`, `pickle.loads()`, or `subprocess` with `shell=True` in any of the provided files.

## Architecture Observations

### 1. Assert Statements for Security-Critical State Checks

Several methods use Python `assert` for state validation:

```python
assert self.is_editable()  # in delete(), add_issue(), edit_issue(), etc.
assert vtype in vtypes.TYPES  # in add_issue(), edit_issue()
```

These are **stripped when Python runs with `-O` or `-OO` flags**. The more critical operations (`add_vote`, `tally_issue`, `has_voted_upon`) correctly use `_all_metadata(required_state=...)` with exceptions. The deployment configuration should ensure Python optimization flags are never used.

### 2. Argon2 Type Selection

The code uses `Type.D` (data-dependent), while the benchmark uses `Type.ID` (hybrid). Type.D is faster but less resistant to side-channel attacks. For a server application, `Type.ID` is the recommended choice per the Argon2 RFC. This is a cryptographic design consideration rather than an availability issue.

### 3. No Documented Resource Management Strategy

Per ASVS 15.2.2, defenses should be "based on the documented security decisions and strategies." No documentation of resource management strategies is visible in the codebase for handling the inherently expensive Argon2 operations at scale.

## Recommendations Summary

### Immediate Actions (Priority 1)

1. **Implement rate limiting on vote submission endpoint** (addresses ASVS-1522-MED-001)
   - **Effort:** Low
   - **Impact:** High — prevents the most easily exploitable vector
   - Deploy Quart rate limiter with conservative limits (e.g., 5 votes per minute per user)

2. **Add votestring validation before crypto operations** (addresses ASVS-1522-MED-004)
   - **Effort:** Low
   - **Impact:** Medium — eliminates oversized input processing
   - Implement maximum length check and vote-type-specific validation

### Short-term Actions (Priority 2)

3. **Add resource bounds/caching for `has_voted_upon()`** (addresses ASVS-1522-MED-002)
   - **Effort:** Medium
   - **Impact:** Medium — improves normal-use performance and resilience
   - Implement maximum issue count check and consider session-level caching

4. **Configure explicit request timeouts and body size limits**
   - **Effort:** Low
   - **Impact:** Defense-in-depth
   - Set conservative limits in server/ASGI configuration

5. **Isolate tally operations from web server resources** (addresses ASVS-1522-MED-003)
   - **Effort:** Medium
   - **Impact:** Medium — prevents tally from degrading web availability

6. **Document resource management strategy for Argon2-heavy operations**
   - **Effort:** Low
   - **Impact:** Compliance — satisfies ASVS 15.2.2 documentation requirement
   - Create architectural documentation describing resource consumption patterns and mitigation strategies
