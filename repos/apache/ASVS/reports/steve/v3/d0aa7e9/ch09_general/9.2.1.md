## Findings

### Medium Severity

#### Finding ASVS-921-SEV-001

**Title:** Election time-based validity constraints (`open_at`/`close_at`) are stored but never enforced during vote acceptance or state computation

**Location:**
- **File:** `v3/steve/election.py`
- **Functions:** `_compute_state()` (line ~306), `add_vote()` (line ~211)
- **Related Functions:** `set_open_at()` (L367), `set_close_at()` (L371), `postprocess_election()` in `pages.py` (L590-600)

**Description:**

The election system stores `open_at` and `close_at` timestamp fields in the database and displays them to users in the UI, creating an expectation that voting is only permitted within the specified time window. However, these time constraints are never validated when accepting votes or computing election state. The `_compute_state()` method only checks the manual `closed` flag and the presence of cryptographic keys, ignoring the time-based validity fields entirely.

**Vulnerable Code:**

```python
# election.py lines 306-318 — _compute_state ignores open_at/close_at entirely
@classmethod
def _compute_state(cls, md):
    "Compute Election state, given all metadata."
    if md.closed == 1:
        assert md.salt is not None and md.opened_key is not None
        return cls.S_CLOSED
    assert md.closed in (None, 0)
    if md.salt is None:
        assert md.opened_key is None
        return cls.S_EDITABLE
    assert md.opened_key is not None
    return cls.S_OPEN  # ← Returns OPEN regardless of close_at being in the past
```

```python
# election.py lines 211-222 — add_vote checks state but not time
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."
    md = self._all_metadata(self.S_OPEN)  # ← Only checks state is OPEN, not time
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

Time fields are set via dedicated endpoints:

```python
# pages.py lines 402-412
@APP.post('/do-set-open_at/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_set_open_at_endpoint(election):
    return await _set_election_date(election, 'open_at')

@APP.post('/do-set-close_at/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_set_close_at_endpoint(election):
    return await _set_election_date(election, 'close_at')
```

And displayed in the UI:

```python
# pages.py lines 590-600
dt_open = e.open_at and datetime.datetime.fromtimestamp(e.open_at)
e.fmt_open_at = format_datetime(dt_open)
dt_close = e.close_at and datetime.datetime.fromtimestamp(e.close_at)
e.fmt_close_at = format_datetime(dt_close)
```

**Data Flow:**

1. Admin sets `close_at` via `POST /do-set-close_at/<eid>` → stored in database
2. UI displays formatted deadline to voters ("closes on Jan 15")
3. `close_at` timestamp passes (Jan 15 arrives)
4. Voter submits vote via `POST /do-vote/<eid>` → `add_vote()` called
5. `_all_metadata(self.S_OPEN)` checks state → state is still `S_OPEN` (admin hasn't manually closed)
6. `_compute_state()` never reads `close_at` → vote is **accepted after deadline**

**Gap Classification:** Type B — Control EXISTS (time fields stored and displayed) but NOT APPLIED (never checked during vote acceptance or state computation)

**Proof of Concept:**

```
1. Admin creates election, sets close_at to "2025-01-15"
2. Admin opens election manually on 2025-01-10
3. Voters see "Closes on Jan 15" in the UI
4. Jan 15 passes — election remains OPEN (no automated transition)
5. On Jan 20, voter submits:
   POST /do-vote/<eid>
   Content-Type: application/x-www-form-urlencoded
   vote-abc123=yes
   → Vote accepted successfully (state is still S_OPEN)
```

**Impact:**

- Votes can be accepted after the displayed deadline, undermining election integrity
- Voters who relied on the deadline and chose not to vote early lose influence against late voters
- The time constraints create a **false expectation of enforcement** — the UI communicates a deadline that the backend does not respect
- This directly mirrors the ASVS 9.2.1 concept: time-based validity data exists in the data structure but is not validated when making security-sensitive decisions (accepting votes)

**Remediation:**

**Option 1: Enforce time constraints in `_compute_state()`**

```python
# election.py — Enhanced _compute_state with time-based checks
import time as time_mod

@classmethod
def _compute_state(cls, md):
    "Compute Election state, given all metadata."
    if md.closed == 1:
        assert md.salt is not None and md.opened_key is not None
        return cls.S_CLOSED
    assert md.closed in (None, 0)
    
    if md.salt is None:
        assert md.opened_key is None
        # Even if editable, check if open_at has not yet arrived
        return cls.S_EDITABLE
    assert md.opened_key is not None
    
    # Enforce close_at: if the deadline has passed, treat as closed
    now = time_mod.time()
    if md.close_at and now > md.close_at:
        return cls.S_CLOSED
    
    # Optionally enforce open_at: election shouldn't accept votes before open_at
    if md.open_at and now < md.open_at:
        return cls.S_EDITABLE  # or a new state like S_SCHEDULED
    
    return cls.S_OPEN
```

**Option 2: Add explicit time checks in `add_vote()`**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    
    # Enforce time-based validity
    import time as time_mod
    now = time_mod.time()
    if md.close_at and now > md.close_at:
        raise ElectionBadState(self.eid, 'past_deadline', self.S_OPEN)
    if md.open_at and now < md.open_at:
        raise ElectionBadState(self.eid, 'before_opening', self.S_OPEN)
    
    # ... rest of vote logic
```

## Architecture Observations

1. **No Application-Layer Self-Contained Tokens**: The application does not create, sign, or validate JWTs, SAML assertions, or similar self-contained tokens. Authentication relies entirely on the `asfquart` framework's session mechanism. This means ASVS 9.2.1 compliance for authentication tokens depends on the framework implementation, which is outside the auditable scope.

2. **Fernet Token Usage is Appropriate**: The Fernet encryption in `crypto.py` is used for vote confidentiality (encryption at rest), not as bearer tokens. The lack of TTL validation on `decrypt()` is correct for this use case — votes must remain decryptable for tallying regardless of age.

3. **Manual vs. Automated State Transitions**: The election lifecycle is entirely manual (admin opens/closes). The `open_at`/`close_at` fields serve as advisory metadata displayed in the UI but do not trigger automated state transitions. This creates a disconnect between what users see and what the system enforces.

4. **State Machine is Sound but Incomplete**: The three-state model (EDITABLE → OPEN → CLOSED) with assertion-based guards is architecturally sound. The gap is that time-based conditions are not incorporated into state transitions.

## Recommendations Summary

### Immediate Actions (Critical Priority)

| Priority | Recommendation | Rationale |
|----------|---------------|-----------|
| 1 | **Enforce `open_at`/`close_at` in `_compute_state()` and/or `add_vote()`** | Time constraints exist in the data model and UI but are not enforced, creating false confidence in deadline enforcement (ASVS-921-SEV-001) |

### Short-term Actions (High Priority)

| Priority | Recommendation | Rationale |
|----------|---------------|-----------|
| 2 | **Audit `asfquart` framework for session token time validation** | Session management is delegated to the framework; ASVS 9.2.1 compliance for authentication tokens cannot be verified from application code alone |
| 3 | **Document the security model for time constraints** | Clarify whether `open_at`/`close_at` are advisory (for display only) or mandatory (must be enforced), and implement accordingly |

### Long-term Actions (Medium Priority)

| Priority | Recommendation | Rationale |
|----------|---------------|-----------|
| 4 | **Consider automated election close via background task** | Even with enforcement in `_compute_state()`, a background scheduler that closes elections at `close_at` would provide defense-in-depth and clear audit trail |
