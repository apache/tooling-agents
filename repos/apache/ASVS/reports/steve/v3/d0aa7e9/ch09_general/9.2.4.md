### Audit Conclusion

After thorough analysis of the provided codebase, the system **does not use self-contained tokens** (no JWTs, SAML assertions, or similar claim-bearing token formats). The cryptographic "tokens" in this system are opaque Argon2 hashes used as derived keys and database lookup identifiers, not parseable claim-bearing tokens. **No findings at MEDIUM severity or above were identified for this specific ASVS requirement.**

## Positive Security Patterns

### 1. Inherent Audience Scoping in Token Derivation

**Location:** `crypto.py:54-58`

```python
def gen_vote_token(opened_key: bytes, pid: str, iid: str, salt: bytes) -> bytes:
    return _hash(opened_key + pid.encode() + iid.encode(), salt)
```

The vote_token is cryptographically bound to the specific election (`opened_key`), voter (`pid`), issue (`iid`), and a unique per-voter-per-issue `salt`. This provides equivalent protection to a formal audience restriction — a token from one context cannot be valid in another.

### 2. Per-Election Key Isolation

**Location:** `election.py:80-84`

```python
def open(self, pdb):
    ...
    salt = crypto.gen_salt()  # Random per-election
    opened_key = crypto.gen_opened_key(edata, salt)  # edata includes EID
```

Each election has a unique `opened_key` derived from election-specific data plus a random salt. This ensures no shared "private key" exists across elections, which is the precondition that triggers ASVS 9.2.4.

### 3. Anti-Tamper Integrity Check

**Location:** `election.py:382-397`

```python
def is_tampered(self, pdb):
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)
    return opened_key != md.opened_key
```

The `opened_key` doubles as a tamper-detection mechanism, ensuring the election data hasn't been modified after opening.

### 4. Metadata Filtering

**Location:** `election.py:138-148`

```python
def get_metadata(self):
    md = self._all_metadata()
    # NOTE: do not return the SALT or OPENED_KEY
    return edict(eid=md.eid, title=md.title, ...)
```

Sensitive key material (`salt`, `opened_key`) is explicitly filtered from public API responses.

### 5. Strong Cryptographic Primitives

- **CSPRNG**: `secrets.token_bytes()` / `secrets.token_hex()` for all random values
- **KDF**: Argon2 (memory-hard) for token derivation
- **Key stretching**: HKDF-SHA256 for encryption key derivation
- **Encryption**: Fernet (AES-128-CBC + HMAC-SHA256)

## Architecture Observations

### Why ASVS 9.2.4 Does Not Directly Apply

1. **No self-contained tokens**: The system uses opaque Argon2 hashes, not JWTs or SAML assertions. These tokens carry no parseable claims — they are raw 32-byte hash outputs used as database keys and encryption key inputs.

2. **No shared signing key**: The ASVS requirement triggers "if a token issuer uses the same private key for issuing tokens to different audiences." Each election has a unique `opened_key`, so the precondition is not met.

3. **Inherent audience binding**: Unlike JWTs where audience must be explicitly validated after parsing, the vote_token derivation chain makes cross-context reuse cryptographically infeasible by design.

### Design Considerations (Informational, Not Findings)

| Observation | Detail | Risk Level |
|-------------|--------|------------|
| Argon2d vs Argon2id | `Type.D` used instead of recommended `Type.ID`; Argon2d is vulnerable to side-channel attacks in shared-hosting environments | Low (not applicable to ASVS 9.2.4) |
| Concatenation without separator | `opened_key + pid.encode() + iid.encode()` — theoretically ambiguous for certain pid/iid combinations (e.g., pid="ab", iid="cd" vs pid="abc", iid="d") | Negligible — mitigated by unique per-voter-per-issue salt |
| HKDF info constant | `info=b'xchacha20_key'` doesn't vary by context | Negligible — input `vote_token` is already fully scoped |
| Comment-code mismatch | HKDF comment says "XChaCha20-Poly1305" but Fernet (AES) is used | Cosmetic |

## Recommendations Summary

Since no findings at MEDIUM or above were identified for ASVS 9.2.4, these are **hardening recommendations** (not findings):

### Immediate Actions

None required. The system's current design provides equivalent protection to explicit audience restrictions through cryptographic derivation.

### Short-term Improvements

1. **Consider Argon2id over Argon2d** (`crypto.py:99`): Change `Type.D` to `Type.ID` for resistance to side-channel attacks. This is the OWASP-recommended variant.

2. **Add domain separator to token derivation** (`crypto.py:57`): Use a structured input format to prevent theoretical concatenation ambiguity:
   ```python
   def gen_vote_token(opened_key, pid, iid, salt):
       data = opened_key + len(pid).to_bytes(4, 'big') + pid.encode() + iid.encode()
       return _hash(data, salt)
   ```

### Long-term Enhancements

3. **Document the audience-scoping model**: Since the system relies on cryptographic derivation rather than explicit audience claims, this design decision should be documented for future auditors and maintainers to understand the security model.
