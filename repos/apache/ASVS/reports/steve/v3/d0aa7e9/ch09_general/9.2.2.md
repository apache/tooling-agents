## Positive Security Patterns

### 1. Purpose-Bound Cryptographic Derivation
**Location:** `crypto.py`, `gen_vote_token()`, lines 50-55

The vote token derivation incorporates election-specific (`opened_key`), person-specific (`pid`), and issue-specific (`iid`) inputs, ensuring tokens cannot be confused across contexts:

```python
def gen_vote_token(opened_key: bytes, pid: str, iid: str, salt: bytes) -> bytes:
    return _hash(opened_key + pid.encode() + iid.encode(), salt)
```

**Impact:** This design inherently prevents cross-context token confusion, aligning with the spirit of ASVS 9.2.2's purpose validation requirement.

### 2. State Machine Enforcement
**Location:** `election.py`, `_all_metadata()`, lines 152-194

Operations are gated by election state, preventing cryptographic material generation/use in inappropriate states:

```python
async def _all_metadata(eid: str, required_state: str | None = None) -> dict:
    # ... state validation logic ...
    if required_state and edata['state'] != required_state:
        raise ValueError(f"Election {eid} is not in required state {required_state}")
```

**Impact:** Provides contextual validation for internal cryptographic operations, ensuring keys are only used during appropriate lifecycle phases.

### 3. Sensitive Data Isolation
**Location:** `election.py`, `get_metadata()`, line 195

The public metadata endpoint explicitly excludes cryptographic secrets:

```python
async def get_metadata(eid: str) -> dict:
    return {k: v for k, v in (await _all_metadata(eid)).items() 
            if k not in ('salt', 'opened_key')}
```

**Impact:** Prevents accidental exposure of cryptographic material through API responses.

### 4. Separation of Authentication and Authorization
**Location:** `pages.py`, decorator pattern throughout

Authentication (`@asfquart.auth.require`) is cleanly separated from application logic:

```python
@pages.post("/do-vote/<eid>")
@asfquart.auth.require({R.committer})
async def do_vote(eid: str):
    # Application logic operates on already-authenticated session
```

**Impact:** Follows principle of least privilege and single responsibility, reducing risk of authentication bypass.

### 5. Cryptographic Domain Separation
**Location:** `crypto.py`, multiple functions

Different purposes use distinct derivation chains:
- Election integrity: BLAKE2b → Argon2 (`gen_opened_key`)
- Vote tokens: Argon2 with purpose-specific inputs (`gen_vote_token`)
- Encryption keys: HKDF from vote tokens (referenced in usage)

**Impact:** Prevents key reuse across cryptographic contexts, a best practice in key management.

## Architecture Observations

### 1. No Self-Contained Tokens in Application Layer
The application does not import JWT, JOSE, or SAML libraries. No token parsing, claim extraction, or signature verification occurs in the provided code. All authentication token processing is delegated to the `asfquart` framework.

**Implication:** ASVS 9.2.2 compliance for authentication tokens depends entirely on the framework's implementation, which is outside the audit scope.

### 2. Internal Token Design Aligns with ASVS 9.2.2 Principles
While `vote_token` and `opened_key` are not self-contained tokens with explicit type/purpose claims, their derivation incorporates:
- **Context binding:** Election ID, person ID, issue ID
- **Purpose separation:** Different derivation functions for different uses
- **State-based validity:** Keys only generated/used during appropriate election states

**Implication:** The internal cryptographic architecture demonstrates security-conscious design aligned with the intent of ASVS 9.2.2, even though the requirement technically does not apply to these constructs.

### 3. Framework Dependency Creates Audit Gap
**Location:** `pages.py`, lines 30-31
```python
import asfquart.session
from asfquart.auth import Requirements as R
```

The application relies on `asfquart` for:
- Token parsing and validation (if OAuth2/OIDC is used)
- Session management
- Role-based access control

**Implication:** A complete ASVS 9.2.2 audit requires reviewing the `asfquart` framework's source code to verify:
- Token type validation (access token vs. ID token)
- Audience claim checking
- Issuer validation
- Token binding to the voting application

### 4. Election State Machine as Access Control Context
The three-state model (`S_EDITABLE` → `S_OPEN` → `S_CLOSED`) provides temporal context for cryptographic operations:
- `opened_key` generated only during `S_EDITABLE` → `S_OPEN` transition
- Vote tokens generated only when election is in `S_OPEN` state
- Integrity checks performed only when election is in `S_CLOSED` state

**Implication:** This provides a form of purpose validation for internal cryptographic material, ensuring keys are not misused across election lifecycle phases.

#### 1. Audit the `asfquart` Framework
**Rationale:** ASVS 9.2.2 compliance for authentication tokens cannot be verified without reviewing the framework's token validation logic.

**Action Items:**
- Request source code or documentation for `asfquart.auth` and `asfquart.session`
- Verify token type validation if OAuth2/OIDC is used:
  - Access tokens used for authorization decisions
  - ID tokens used only for authentication
  - Audience (`aud`) claim checked
  - Token type (`typ` header) validated
- Verify session tokens are bound to the voting application context

**Acceptance Criteria:**
- Framework demonstrates explicit token type checking before security decisions
- Framework validates audience and issuer claims
- Framework prevents cross-service token reuse

**Rationale:** Explicit documentation enables future auditors to verify ASVS 9.2.2 compliance without ambiguity.

  - Internal cryptographic construct purposes (`vote_token`, `opened_key`)
  - State machine constraints on cryptographic operations
- Include data flow diagrams showing token/key lifecycle
- Document assumptions about framework behavior

**Deliverable:** `docs/security/token-architecture.md` in repository

**Action Items:**

### Long-Term Actions (Priority: Low)

**Action Items:**

  - Explicit token type validation before use
  - Audience and issuer claim checking
  - Token binding to prevent replay attacks
- Maintain framework integration as fallback during transition

**Trade-offs:**
- **Pro:** Direct control over token validation logic
- **Con:** Increased maintenance burden and attack surface
- **Recommendation:** Only pursue if framework proves inadequate

## Audit Methodology

This audit followed a risk-based approach focused on ASVS 9.2.2 requirements:

1. **Scope Definition:** Identified all code paths handling authentication, authorization, or cryptographic tokens
2. **Token Inventory:** Cataloged all token-like constructs (session cookies, vote tokens, cryptographic keys)
3. **Type Classification:** Determined whether each construct qualifies as a "self-contained token" per ASVS 9.2.2
4. **Validation Review:** For self-contained tokens, verified type and purpose validation before security decisions
5. **Framework Analysis:** Documented authentication delegation to external framework
6. **Gap Analysis:** Identified areas where compliance cannot be verified due to missing framework source

**Limitations:**
- `asfquart` framework source code not available for review
- Production configuration and deployment architecture not examined
- Runtime behavior and logging not assessed
- Integration with Apache Software Foundation identity infrastructure not verified
