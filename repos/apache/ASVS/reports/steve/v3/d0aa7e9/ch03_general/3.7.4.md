## 4. Findings

### 4.1 High Severity Findings

#### Finding ASVS-374-HIGH-001: Complete Absence of HSTS Configuration and Preload List Submission

| Attribute | Value |
|-----------|-------|
| **Finding ID** | ASVS-374-HIGH-001 |
| **Severity** | HIGH |
| **Gap Type** | Type A — Entry point with NO control |
| **File** | `v3/server/main.py` (entire application) |
| **Function** | All response handlers |
| **Lines** | N/A (absence of control) |
| **ASVS Requirement** | 3.7.4 — HSTS Preload List |

**Description**

The application does not set the `Strict-Transport-Security` response header on any response, and there is no evidence the domain has been submitted to the HSTS preload list at [hstspreload.org](https://hstspreload.org). The HSTS header with `preload`, `includeSubDomains`, and a `max-age` of at least 31536000 (1 year) is a **prerequisite** for preload list submission.

**Vulnerable Code (absence of control)**

In `main.py`, TLS is optionally configured but no HSTS policy is set:

```python
# main.py lines 77-80
kwargs = {}
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
```

There is no `after_request` handler, no middleware, and no framework configuration anywhere in the codebase that adds `Strict-Transport-Security` to responses.

In `config.yaml.example`, there is no HSTS configuration option:

```yaml
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
db: steve.db
```

**Exhaustive Search Results**

- Searched all provided files for: `Strict-Transport-Security`, `hsts`, `preload`, `max-age`, `includeSubDomains`, `after_request`, `after_serving`, `response.headers` — **zero matches**.
- `ARCHITECTURE.md` states: *"HTTPS: Enforced via TLS certificates in certs/"* — confirming HSTS is not part of the security model.

**Related Functions Checked**

- `create_app()` in `main.py` — no security header middleware
- All route handlers in `pages.py` — no HSTS header set
- Configuration loading in `main.py` — no HSTS options

**Data Flow**

```
Browser → HTTP/HTTPS Request → Quart Application → Response (missing Strict-Transport-Security header)
                                                          ↓
                                              No HSTS policy cached in browser
                                                          ↓
                                              Domain NOT on preload list
                                                          ↓
                                              Vulnerable to SSL stripping on every first visit
```

**Impact**

- Without HSTS preloading, browsers have no built-in knowledge that this domain requires TLS
- First-time visitors are vulnerable to SSL stripping / downgrade attacks (e.g., via a MITM on public WiFi)
- Even returning visitors are vulnerable if the HSTS header was never set and cached
- For a voting system handling election data, this could enable interception or manipulation of authentication tokens and votes in transit

**Proof of Concept**

An attacker on the same network (e.g., conference WiFi where ASF committers might access the voting system) can intercept the initial HTTP request before TLS is established:

```bash
# Using sslstrip or similar tool
$ sslstrip -l 8080
# ARP spoof the victim
$ arpspoof -i eth0 -t <victim_ip> <gateway_ip>
# Victim's browser connects to http://steve.apache.org (no preload protection)
# Attacker downgrades to plaintext, captures session cookies and votes
```

**Remediation**

**Step 1: Add HSTS header via Quart `after_request` hook** in `main.py`:

```python
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    @app.after_request
    async def set_security_headers(response):
        # HSTS with preload prerequisites:
        # - max-age >= 31536000 (1 year)
        # - includeSubDomains required
        # - preload directive required
        response.headers['Strict-Transport-Security'] = (
            'max-age=63072000; includeSubDomains; preload'
        )
        return response

    import pages
    import api

    return app
```

**Step 2: Add HSTS configuration to `config.yaml`**:

```yaml
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
    hsts:
        enabled: true
        max_age: 63072000  # 2 years
        include_subdomains: true
        preload: true
```

**Step 3: Submit the production domain to the HSTS preload list:**

- Visit https://hstspreload.org
- Submit the top-level domain (e.g., `apache.org` or the specific subdomain if applicable)
- Ensure the domain meets all preload requirements before submission

**Step 4: Add HTTP → HTTPS redirect** (if the application can receive HTTP requests):

```python
@app.before_request
async def redirect_to_https():
    if quart.request.scheme == 'http':
        url = quart.request.url.replace('http://', 'https://', 1)
        return quart.redirect(url, code=301)
```

## 6. Architecture Observations

1. **TLS is optional, not enforced**: The `certfile` check in `main.py` line 77 (`if app.cfg.server.certfile:`) means the application can run without TLS entirely. There is no fail-safe requiring TLS in production.

2. **No security headers framework**: The application has no centralized security header management. Beyond HSTS, other security headers (CSP, X-Content-Type-Options, X-Frame-Options, etc.) are also absent from the reviewed code. This suggests a systemic gap in browser security controls.

3. **ASGI deployment gap**: When running under Hypercorn (`run_asgi()`), TLS configuration is delegated entirely to the ASGI server. The application code has no mechanism to ensure HSTS headers are added regardless of deployment mode.

4. **Documentation inconsistency**: `ARCHITECTURE.md` claims "HTTPS: Enforced via TLS certificates" but enforcement only means TLS is *available* — there's no HSTS to ensure browsers *require* it.
