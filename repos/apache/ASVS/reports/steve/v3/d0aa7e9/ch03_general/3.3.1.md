## Findings

### High Severity

#### ASVS-331-HIGH-001: Session Cookies Lack `Secure` Attribute Configuration

| Field | Value |
|-------|-------|
| **Severity** | HIGH |
| **Finding ID** | ASVS-331-HIGH-001 |
| **Gap Type** | Type A — Entry point with NO control |
| **File** | `v3/server/main.py`, lines 30-44 (`create_app()`) |
| **Also relevant** | `v3/server/pages.py`, line 86 (`asfquart.session.read()`) |

**Vulnerable Code (`main.py:30-44`):**
```python
def create_app():
    "Create the asfquart app and its endpoints."
    # ...
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    # No SESSION_COOKIE_SECURE configuration
    import pages  # noqa: F401
    import api  # noqa: F401
    return app
```

**Data Flow:**
1. `asfquart.construct()` creates the Quart application (source)
2. Session cookies are created by the framework when `asfquart.session.read()` is called across all authenticated endpoints (sink)
3. No `SESSION_COOKIE_SECURE = True` is set anywhere in the application configuration (missing control)

**Evidence that TLS is not guaranteed (`main.py:77-80`):**
```python
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
```
TLS is conditionally configured only when `certfile` is present, meaning the application can run over plain HTTP. Without `Secure` attribute, session cookies would be transmitted in cleartext.

**Affected Entry Points (all endpoints using sessions):**
- All 17+ authenticated endpoints in `pages.py`
- Session-based flash messages in all POST handlers
- OAuth callback flows configured in `main.py:35-38`

**Impact:** Session cookies can be intercepted over unencrypted connections (e.g., network sniffing, MITM). An attacker on the same network as a voter could steal session cookies and impersonate authenticated users, potentially casting fraudulent votes or manipulating elections.

**Proof of Concept:**
```
# Attacker on same network captures HTTP traffic
# Session cookie is transmitted without Secure flag:
Cookie: session=eyJhbGci...

# Attacker replays cookie over HTTPS to authenticated endpoint:
GET /voter HTTP/1.1
Host: steve.example.org
Cookie: session=eyJhbGci...
```

**Remediation:**
```python
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    # Enforce secure cookie attributes (ASVS 3.3.1)
    app.config['SESSION_COOKIE_SECURE'] = True
    app.config['SESSION_COOKIE_HTTPONLY'] = True  # bonus: ASVS 3.4.2
    app.config['SESSION_COOKIE_SAMESITE'] = 'Lax'  # bonus: ASVS 3.4.3
    
    import pages
    import api
    return app
```

### Medium Severity

#### ASVS-331-MED-001: Session Cookie Name Missing `__Host-` or `__Secure-` Prefix

| Field | Value |
|-------|-------|
| **Severity** | MEDIUM |
| **Finding ID** | ASVS-331-MED-001 |
| **Gap Type** | Type A — Entry point with NO control |
| **File** | `v3/server/main.py`, lines 30-44 (`create_app()`) |

**Vulnerable Code (`main.py:36-38`):**
```python
app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
# No SESSION_COOKIE_NAME configuration with __Host- or __Secure- prefix
```

**Analysis:**
Quart (and Flask) default the session cookie name to `session`. ASVS 3.3.1 requires that if the `__Host-` prefix is not used, the `__Secure-` prefix **must** be used. Neither prefix is configured anywhere in the provided application code.

The `__Secure-` prefix instructs browsers to only send the cookie over HTTPS and requires the `Secure` attribute. The `__Host-` prefix additionally restricts the cookie to the exact host and root path, preventing subdomain attacks.

**Impact:** Without the `__Secure-` or `__Host-` prefix, the browser does not enforce prefix-based cookie protections. Combined with the missing `Secure` attribute (Finding 1), this means:
- No browser-enforced HTTPS-only transmission
- Potential for subdomain cookie injection attacks
- Cookies could be overwritten by a less-secure subdomain

**Remediation:**
```python
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    # Use __Host- prefix for maximum cookie security (ASVS 3.3.1)
    # __Host- requires: Secure, Path=/, no Domain attribute
    app.config['SESSION_COOKIE_NAME'] = '__Host-steve_session'
    app.config['SESSION_COOKIE_SECURE'] = True
    app.config['SESSION_COOKIE_PATH'] = '/'
    # Do NOT set SESSION_COOKIE_DOMAIN (required for __Host- prefix)
    
    # Alternative: Use __Secure- prefix (less restrictive)
    # app.config['SESSION_COOKIE_NAME'] = '__Secure-steve_session'
    # app.config['SESSION_COOKIE_SECURE'] = True
    
    import pages
    import api
    return app
```

## Positive Security Patterns

1. **Authentication enforcement**: Nearly all endpoints use `@asfquart.auth.require({R.committer})` or `@asfquart.auth.require({R.pmc_member})`, providing consistent authentication gating.
2. **Session abstraction**: Session management is abstracted through `asfquart.session`, meaning a fix to cookie configuration in one place (`create_app()`) will propagate to all endpoints.
3. **Logging of security-relevant actions**: All state-changing operations log the user ID and action, providing an audit trail.
4. **Input validation**: The `_set_election_date()` function validates date input with proper error handling.
5. **OAuth over HTTPS**: OAuth endpoints hardcoded to `https://oauth.apache.org/...`, ensuring token exchange is encrypted.

---

## Architecture Observations

1. **Single configuration point**: The `create_app()` function in `main.py` is the correct and only place to configure cookie security attributes. A fix here covers all session cookies application-wide.
2. **Framework dependency**: Cookie security depends entirely on `asfquart`/Quart framework defaults. Quart inherits from Flask, where `SESSION_COOKIE_SECURE` defaults to `False` and `SESSION_COOKIE_NAME` defaults to `"session"` — neither meeting ASVS 3.3.1.
3. **Deployment ambiguity**: The conditional TLS configuration (`if app.cfg.server.certfile`) suggests the application may be deployed behind a reverse proxy that terminates TLS. Even so, the application must set `SESSION_COOKIE_SECURE = True` to ensure browsers handle the cookie correctly regardless of proxy configuration.
