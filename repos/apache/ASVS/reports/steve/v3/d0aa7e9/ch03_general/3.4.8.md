### Overview

The application **does not set the `Cross-Origin-Opener-Policy` (COOP) header** on any HTTP response that renders HTML content. A thorough review of all route handlers, middleware configuration, response construction functions, and the application factory reveals no mechanism—either global or per-route—for adding this header. This leaves all document-rendering responses vulnerable to cross-origin window handle attacks such as tabnabbing and frame counting.

#### Finding ASVS-348-MED-001

**Title:** Missing `Cross-Origin-Opener-Policy` header on all HTML responses

**Gap Type:** Type A — Entry points with NO control

**Location:**

**Description:**

The application does not set the `Cross-Origin-Opener-Policy` (COOP) header on any HTTP response that renders HTML content. This leaves all document-rendering responses vulnerable to cross-origin window handle attacks such as tabnabbing and frame counting.

**Vulnerable Code:**

In `v3/server/main.py`, lines 32–47 — Application factory registers no security header middleware:
```python
def create_app():
    "Create the asfquart app and its endpoints."

    # ...
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    # Now that we have an APP, import modules that will add page
    # and API endpoints into the APP.
    import pages  # noqa: F401
    import api  # noqa: F401

    return app  # also available as asfquart.APP now
    # ← No @app.after_request for security headers
```

In `v3/server/pages.py`, line 659 — Manual HTML response with no COOP header:
```python
def raise_404(template, data):
    content = asfquart.utils.render(template, data)
    quart.abort(quart.Response(content, status=404, mimetype='text/html'))
    # ← No Cross-Origin-Opener-Policy header set
```

In `v3/server/pages.py`, all template-decorated endpoints (e.g., line ~125):
```python
@APP.get('/')
@APP.use_template(TEMPLATES / 'home.ezt')
async def home_page():
    result = await basic_info()
    result.title = 'Home'
    return result
    # ← Template rendering returns HTML with no COOP header
```

**Data Flow:**
```
Client Request → Quart Router → Handler Function → @APP.use_template decorator
    → EZT template rendering → HTTP Response (Content-Type: text/html)
    → Missing: Cross-Origin-Opener-Policy header
    → Browser allows cross-origin window handle access
```

**Proof of Concept:**

An attacker page opened via a link from the voting application (e.g., in an election description) can retain a reference to the opener window:

```html
<!-- Attacker's page, linked from an election issue description -->
<script>
  // Without COOP: same-origin, the opener reference is accessible
  // from cross-origin pages opened via window.open or target="_blank"
  
  // Tabnabbing: redirect the voting page to a phishing page
  if (window.opener) {
    window.opener.location = 'https://evil.example.com/fake-vote-page';
  }
  
  // Frame counting: enumerate open windows/tabs
  // to infer voting activity patterns
</script>
```

A malicious link in an election's issue description (rendered via `rewrite_description()` at line ~52 which converts `doc:filename` patterns to `<a>` links) or in any user-controlled content could enable this attack.

**Impact:**
- **Tabnabbing:** An attacker-controlled page opened from the voting application can navigate the original tab to a phishing page mimicking the voting UI, potentially capturing credentials or manipulating vote submissions.
- **Frame Counting:** Cross-origin pages can enumerate browsing contexts to infer voting behavior, undermining the system's anonymity goals.
- **Window Reference Leakage:** Without COOP, the `window.opener` property leaks a reference across origins, enabling cross-origin state inspection.

**Remediation:**

Add a global `after_request` hook in the application factory to set the `Cross-Origin-Opener-Policy` header on all HTML responses:

```python
# In v3/server/main.py, inside create_app():
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    @app.after_request
    async def set_security_headers(response):
        # Set COOP on all document-rendering responses
        content_type = response.content_type or ''
        if 'text/html' in content_type:
            response.headers['Cross-Origin-Opener-Policy'] = 'same-origin'
        return response

    import pages
    import api

    return app
```

Also update the `raise_404` function in `v3/server/pages.py` to include the header on manual responses:

```python
def raise_404(template, data):
    content = asfquart.utils.render(template, data)
    response = quart.Response(content, status=404, mimetype='text/html')
    response.headers['Cross-Origin-Opener-Policy'] = 'same-origin'
    quart.abort(response)
```

> **Note:** Use `same-origin` as the default directive. If the application requires popup interactions (e.g., OAuth flows using popups), use `same-origin-allow-popups` instead. Given the ASF OAuth flow appears to use redirects rather than popups (see `main.py` lines 37-40), `same-origin` is the appropriate choice.

## Architecture Observations

1. **Centralized template rendering is an advantage:** The `@APP.use_template` decorator pattern means a single `after_request` hook can address all 9 HTML-rendering endpoints simultaneously, making remediation straightforward.

2. **Missing security header strategy entirely:** The absence of COOP is likely symptomatic of a broader gap — no security header middleware exists for any browser security headers (e.g., `X-Content-Type-Options`, `X-Frame-Options`, `Content-Security-Policy`, `Cross-Origin-Embedder-Policy`). A comprehensive `after_request` hook should address all of these together.

3. **The `raise_404` function bypasses any future middleware:** Since it constructs a `quart.Response` directly and passes it to `quart.abort()`, it may bypass `after_request` hooks depending on Quart's abort handling. This needs explicit verification after adding the global hook.

4. **Document serving (`/docs/<iid>/<docname>`) could serve HTML:** The `serve_doc` endpoint at line 575 serves arbitrary files from a directory. If any served document is HTML, it would also need the COOP header. `send_from_directory` responses should be covered by the `after_request` hook, but only if their `Content-Type` is correctly detected.
