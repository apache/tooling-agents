## Positive Security Patterns

### 1. Modern Framework Selection

**Location:** `v3/server/bin/fetch-thirdparty.sh`, Lines 23-33

All client-side dependencies are current-generation libraries from reputable, actively maintained projects:

```bash
VERSION="5.3.1"        # Bootstrap 5.x — current major version line
I_VERSION="1.13.1"     # Bootstrap Icons — actively maintained
S_VERSION="1.15.7"     # SortableJS — actively maintained
```

**Security Benefit:** Ensures access to security patches and eliminates risks associated with unmaintained or deprecated technologies explicitly prohibited by ASVS 3.7.1 (Flash, Shockwave, ActiveX, Silverlight, NACL, Java applets, NSAPI plugins).

---

### 2. Subresource Integrity (SRI) Hash Generation

**Location:** `v3/server/bin/fetch-thirdparty.sh`, Lines 54-57, 93-95

The script generates SHA-384 hashes for SRI, enabling the application to verify that served files have not been tampered with:

```bash
echo -n "sha384-" ; openssl dgst -sha384 -binary "${STATIC_DIR}/css/bootstrap.min.css" | openssl base64 -A
echo -n "sha384-" ; openssl dgst -sha384 -binary "${STATIC_DIR}/js/bootstrap.bundle.min.js" | openssl base64 -A
echo -n "sha384-" ; openssl dgst -sha384 -binary "${STATIC_DIR}/js/Sortable.min.js" | openssl base64 -A
```

**Security Benefit:** This is a strong defense-in-depth measure that complements ASVS 3.7.1 by ensuring the integrity of the chosen client-side technologies at runtime. Even if static files are compromised post-build, browsers will reject tampered resources.

---

### 3. HTTPS for All Downloads

**Location:** `v3/server/bin/fetch-thirdparty.sh`, Lines 25, 29, 33

All source URLs use HTTPS, preventing man-in-the-middle injection during the build process:

```bash
B_URL="https://github.com/twbs/bootstrap/releases/download/v${VERSION}/${DIST}.zip"
I_URL="https://github.com/twbs/icons/releases/download/v${I_VERSION}/${I_DIST}.zip"
S_URL="https://github.com/SortableJS/Sortable/archive/refs/tags/${S_VERSION}.zip"
```

**Security Benefit:** Protects the supply chain by ensuring dependencies cannot be intercepted and modified during download, maintaining the integrity of the supported client-side technologies.

---

### 4. Pinned Versions

**Location:** `v3/server/bin/fetch-thirdparty.sh`, Lines 23, 27, 31

Dependencies are pinned to exact versions rather than using `latest` or floating tags:

```bash
VERSION="5.3.1"
I_VERSION="1.13.1"
S_VERSION="1.15.7"
```

**Security Benefit:** Prevents unexpected changes that could introduce deprecated or insecure technologies. Provides predictable behavior and enables deliberate security review before version updates.

---

### 5. Self-Hosted Dependencies

**Location:** Entire script architecture

The script downloads and self-hosts all client-side assets rather than loading them from CDNs at runtime:

```bash
# Downloads to local static directory
unzip -q -o "${ZIPFILE}" -d "${STATIC_DIR}"
```

**Security Benefit:** Reduces third-party risk, eliminates runtime dependency on external services, and provides full control over the client-side technologies served to users. Enhances compliance with ASVS 3.7.1 by ensuring only vetted, supported technologies are served.

---

### 6. Standards-Based Technology Stack

**Location:** Entire codebase (negative finding — no plugin-based technologies detected)

The application uses only standards-based HTML/CSS/JS with no reliance on browser plugins.

**Security Benefit:** Complete alignment with ASVS 3.7.1's prohibition of deprecated plugin technologies. Modern web standards are actively maintained by browser vendors and standards bodies, ensuring long-term security support.

## Architecture Observations

### 1. Build-Time Dependency Management
Client-side libraries are fetched at build/setup time and served as static files, which is a sound architecture choice that avoids runtime CDN dependencies. This approach provides better control over the client-side technology stack and simplifies compliance verification for ASVS 3.7.1.

### 2. Minimal Client-Side Footprint
The application uses only three client-side libraries:
- **Bootstrap** for UI components
- **Bootstrap Icons** for iconography
- **SortableJS** for drag-and-drop functionality

This lean frontend approach minimizes attack surface and reduces the risk of incorporating deprecated or insecure technologies.

### 3. No Plugin-Based Technologies
The technology stack is purely standards-based (HTML/CSS/JS) with no reliance on browser plugins. This architecture is fully aligned with ASVS 3.7.1 and represents current best practices for web application security.

### 4. SRI as Complementary Control
While not directly part of ASVS 3.7.1, the SRI hash generation creates a verification chain:
```
Known good versions → Integrity hashes → Runtime validation
```
This ensures that even if static files are tampered with post-build, the browser will reject them, providing defense-in-depth for the supported client-side technology requirement.

### 5. Supply Chain Security Posture
The combination of HTTPS downloads, version pinning, and self-hosting demonstrates a mature approach to supply chain security for client-side dependencies, reducing the risk of incorporating compromised or deprecated technologies.

---

#### 2. Add Pre-Extraction Checksum Verification
**Current State:** SRI hashes protect at runtime, but no build-time checksum verification exists

**Recommendation:** Add build-time checksum verification to complete the integrity chain:
```bash
# Example implementation
EXPECTED_SHA256="<known-good-hash>"
ACTUAL_SHA256=$(sha256sum "${ZIPFILE}" | cut -d' ' -f1)
if [ "$ACTUAL_SHA256" != "$EXPECTED_SHA256" ]; then
    echo "ERROR: Checksum mismatch for Bootstrap download"
    exit 1
fi
```

**Benefit:** Provides additional supply chain protection by verifying downloads before extraction, complementing the existing SRI runtime verification.

**Priority:** Low (defense-in-depth enhancement)
