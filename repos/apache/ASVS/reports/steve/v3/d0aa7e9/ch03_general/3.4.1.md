## Findings

### Medium Severity

#### ASVS-341-MED-001: Missing Strict-Transport-Security Header on All Responses

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with NO control  
**ASVS Compliance:** ❌ Fails ASVS 3.4.1 (Level 1, Level 2)

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `create_app()`
- **Lines:** 31-47
- **Affected Routes:** All 21 application endpoints (see Route Inventory above)

**Related Functions Checked:**
- `create_app()` in `main.py` (lines 31-47) — No `after_request` hook
- All route handlers in `pages.py` — No response header manipulation
- Configuration in `config.yaml.example` — No HSTS configuration option
- Architecture documentation in `ARCHITECTURE.md` — No HSTS mention

**Description:**

No `Strict-Transport-Security` header is set on any HTTP response. The entire codebase—`main.py`, `pages.py`, configuration, and architecture documentation—contains zero references to HSTS, `Strict-Transport-Security`, or any security header middleware. While TLS certificates are configured for the server, browsers are never instructed to enforce HTTPS-only connections.

**Exact File Locations Searched:**

| File | Evidence |
|------|----------|
| `v3/server/main.py` | No `after_request`, no response header manipulation, no HSTS config |
| `v3/server/pages.py` | No response header manipulation across 20+ route handlers |
| `v3/server/config.yaml.example` | No HSTS configuration option |
| `v3/ARCHITECTURE.md` | States "HTTPS: Enforced via TLS certificates" — no HSTS mention |

**Vulnerable Code:**

`main.py` lines 31-47:
```python
def create_app():
    "Create the asfquart app and its endpoints."
    # ...
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    # Now that we have an APP, import modules that will add page
    # and API endpoints into the APP.
    import pages  # noqa: F401
    import api  # noqa: F401

    return app  # also available as asfquart.APP now
    # ← No after_request hook registered for security headers
```

**Supporting Evidence:**

CSRF implementation in `pages.py` line 84 shows that the `asfquart` framework does **not** automatically inject security headers:
```python
### generate a real token and store in the session
basic.csrf_token = 'placeholder'
```

This confirms that security controls like CSRF and HSTS must be explicitly implemented by the application.

**Data Flow:**

```
Browser → HTTPS request → Quart app → Response (no STS header) → Browser
                                                                    ↓
                                            Browser does NOT pin HTTPS-only
                                                                    ↓
                                            Subsequent requests vulnerable to
                                            SSL stripping / downgrade attacks
```

**Proof of Concept:**

```bash
# 1. User visits site over HTTPS - response has no STS header:
$ curl -sI https://steve.apache.org/voter
HTTP/2 200
content-type: text/html
# ← Missing: strict-transport-security: max-age=31536000; includeSubDomains

# 2. Attacker on same network performs SSL strip (e.g., via mitmproxy):
#    - Intercepts HTTP redirect or user typing http://steve.apache.org
#    - Browser has no HSTS pin, so it willingly connects over HTTP
#    - Session cookies, votes, and authentication tokens exposed in cleartext
```

**Impact:**

- Browsers will not enforce HTTPS-only connections to the application
- Users are vulnerable to SSL stripping / downgrade attacks on untrusted networks
- Session cookies (containing authentication tokens) can be intercepted in transit
- For a **voting system**, this could allow interception or modification of votes and voter authentication sessions
- Fails ASVS 3.4.1 at all levels:
  - **L1:** Missing header entirely
  - **L2:** Missing `includeSubDomains` directive

**Remediation:**

Register an `after_request` hook in `create_app()` to add HSTS (and other security headers) to every response:

```python
# In main.py, within create_app():
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    @app.after_request
    async def set_security_headers(response):
        # ASVS 3.4.1: HSTS with 1 year max-age + includeSubDomains (L2)
        response.headers['Strict-Transport-Security'] = (
            'max-age=31536000; includeSubDomains'
        )
        # Additional recommended headers:
        response.headers['X-Content-Type-Options'] = 'nosniff'
        response.headers['X-Frame-Options'] = 'DENY'
        response.headers['Referrer-Policy'] = 'strict-origin-when-cross-origin'
        return response

    import pages
    import api

    return app
```

For production deployments behind a reverse proxy, **also** configure the proxy (e.g., Apache `httpd`, Nginx) to add the header, ensuring defense in depth:

```nginx
# Nginx example
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
```

## Architecture Observations

### 1. No Security Header Framework

The absence of HSTS is symptomatic of a broader pattern — there is no centralized security header middleware. Other headers like `X-Content-Type-Options`, `X-Frame-Options`, `Content-Security-Policy`, and `Referrer-Policy` are also absent from the reviewed code.

### 2. Proxy Deployment Model

The `config.yaml.example` notes "Typical usage is that a proxy sits in front of this server." While HSTS could theoretically be added at the proxy layer, the application should set it as defense-in-depth, and the current code doesn't document or enforce this requirement.

### 3. TLS is Optional

In `main.py` lines 78-82, TLS certificates are only loaded if `app.cfg.server.certfile` is set. If unset, the server runs plain HTTP with no warning, making HSTS enforcement even more critical at the proxy layer.
