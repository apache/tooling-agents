## Positive Security Patterns

| Pattern | Description | Location | Security Benefit |
|---------|-------------|----------|------------------|
| **No open redirect parameters** | No `?next=`, `?redirect=`, `?url=`, or similar query parameters are read for redirect targets | Entire file | Eliminates most common open redirect attack vector |
| **All redirects use absolute paths** | Every `quart.redirect()` call uses a path starting with `/`, preventing protocol-relative URL injection | All 11 redirect sites | Prevents `//evil.com` style attacks that bypass naive validation |
| **Database validation before redirect** | Dynamic redirect components (`election.eid`) are always validated via database lookup before use | `load_election`, `load_election_issue` decorators | Ensures only legitimate, existing election IDs can be used in redirect paths |
| **303 See Other status code** | POST handlers correctly use `code=303` for POST-redirect-GET pattern | All POST endpoint redirects | Prevents form resubmission issues and follows HTTP best practices |
| **Decorator-based validation** | Security-critical validation logic centralized in reusable decorators | `load_election`, `load_election_issue` | Reduces code duplication and ensures consistent validation across endpoints |

**Key Security Implementation:**

The `load_election` decorator (line ~198) provides robust validation:

```python
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            # Raises 404, redirect is NEVER reached
            raise_404(T_BAD_EID, result)
        return await func(e)
    return loader
```

This ensures that any `election.eid` value used in a redirect has been verified to exist in the database, preventing injection of arbitrary values.

## Architecture Observations

### Current Architecture Strengths

1. **Zero external redirects** — The application currently performs no redirects to external hostnames or domains, eliminating the primary risk addressed by ASVS 3.7.2.

2. **Consistent redirect pattern** — All redirects follow a uniform pattern using either:
   - Hardcoded internal paths (`'/voter'`)
   - Database-validated identifiers in path construction (`f'/manage/{election.eid}'`)

3. **Decorator-based security** — Validation logic is centralized in decorators, making security controls easier to audit and maintain.

### Potential Architecture Concerns

1. **No redirect allowlist mechanism exists** — While none is currently needed since zero redirects target external hostnames, if future features add external redirects (e.g., OAuth callback URLs, third-party integrations, federated authentication), an allowlist mechanism should be implemented proactively.

2. **Framework-level redirects not audited** — The `asfquart.auth.require` decorator and any underlying OAuth flow may perform redirects to external identity providers. Those components should be audited separately against ASVS 3.7.2 to ensure complete coverage.

3. **Protocol-relative URL injection not possible** — Even if `election.eid` hypothetically contained `//evil.com`, the redirect would produce `/manage///evil.com` — still an absolute path on the same origin. However, this protection is incidental rather than by design.

## Recommendations Summary

### Immediate Actions
*No immediate actions required.* The current implementation is compliant with ASVS 3.7.2.

### Short-term Recommendations

1. **[INFORMATIONAL]** **Audit authentication framework redirects** — If `asfquart.auth` performs redirects to external OAuth providers or uses post-login redirect parameters, verify those against an allowlist. This is the most likely location for ASVS 3.7.2 violations but is outside the provided source code.
   - **Priority:** Medium
   - **Effort:** 2-4 hours
   - **Owner:** Security team

### Long-term Recommendations

2. **[DEFENSE-IN-DEPTH]** **Implement centralized redirect validation** — Consider adding a centralized `safe_redirect()` helper that validates all redirect targets are relative paths or on an explicit allowlist, to protect against future regressions:

```python
from urllib.parse import urlparse

ALLOWED_REDIRECT_HOSTS = set()  # empty = same-origin only

def safe_redirect(url, code=303):
    """
    Safely redirect to a URL, ensuring it's either a relative path
    or on the allowlist of permitted external hosts.
    """
    parsed = urlparse(url)
    if parsed.netloc and parsed.netloc not in ALLOWED_REDIRECT_HOSTS:
        _LOGGER.warning(f'Blocked redirect to disallowed host: {parsed.netloc}')
        return quart.redirect('/', code=code)
    return quart.redirect(url, code=code)
```

   - **Priority:** Low (defense-in-depth)
   - **Effort:** 4-8 hours (implementation + refactoring)
   - **Owner:** Development team
   - **Benefit:** Provides explicit protection against future code changes that might introduce open redirect vulnerabilities

3. **[PROACTIVE]** **Document redirect security policy** — Create documentation that explicitly states:
   - All redirects must use relative paths unless explicitly approved
   - External redirects require security review and allowlist entry
   - The `safe_redirect()` helper must be used for all new redirect code
   
   - **Priority:** Low
   - **Effort:** 2 hours
   - **Owner:** Security team
