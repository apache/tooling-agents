This audit examines the application for Time-of-Check to Time-of-Use (TOCTOU) race conditions in critical operations. The application uses Quart's async model with SQLite in autocommit mode. Multiple critical operations perform state/permission checks as separate database operations from the actions that depend on those checks, creating exploitable race windows. The most significant findings involve vote submission, election state transitions, and election deletion where state verification and dependent actions are not wrapped in atomic transactions.

### High Severity

#### ASVS-1542-HIGH-001: Vote Submission Eligibility Check and Insert Not in Same Transaction (TOCTOU)

**Location:**  
- **File:** `v3/steve/election.py`
- **Function:** `add_vote()`
- **Lines:** ~227-238

**Related Functions Checked:**
- `_all_metadata()` (state verification)
- `q_get_mayvote.first_row()` (eligibility verification)
- `c_add_vote.perform()` (vote insertion)

**Description:**

The `add_vote()` function performs two critical checks (election state and voter eligibility) as separate database queries before inserting the vote. These three operations are not wrapped in a single atomic transaction, creating a TOCTOU race condition where the election state can change between the checks and the vote insertion.

**Vulnerable Code:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)          # CHECK 1: state (separate query)

    ### validate VOTESTRING for ISSUE.TYPE voting

    mayvote = self.q_get_mayvote.first_row(pid, iid)  # CHECK 2: eligibility (separate query)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)

    # Pass MAYVOTE.SALT for PBKDF.
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)

    self.c_add_vote.perform(vote_token, ciphertext)  # USE: insert vote (separate autocommit)
```

**Data Flow:**

1. **Source:** User vote submission via `POST /do-vote/<eid>`
2. **Check 1 (state):** `_all_metadata(self.S_OPEN)` → SELECT from election table → verifies election is open
3. **Check 2 (eligibility):** `q_get_mayvote.first_row(pid, iid)` → SELECT from mayvote table → verifies voter is eligible
4. **Action:** `c_add_vote.perform(vote_token, ciphertext)` → INSERT into vote table
5. **Missing Control:** No transaction wrapping all three operations

**Race Window:**

Between the state check (Check 1) and the vote INSERT (Action), the election can be closed by a concurrent admin request. Between the eligibility check (Check 2) and the INSERT, the state can also change.

**Proof of Concept:**

```
Timeline:
T1: Voter A → POST /do-vote/abc123     → add_vote() checks _all_metadata(S_OPEN) → PASS
T2: Admin B → GET /do-close/abc123     → close() checks is_open() → PASS, c_close.perform() → election now CLOSED
T3: Voter A → add_vote() continues     → q_get_mayvote succeeds, c_add_vote.perform() → Vote recorded on CLOSED election
```

In Quart's async model, `await quart.request.form` in `do_vote_endpoint` yields control, allowing the close request to be processed before the vote insertion completes.

**Impact:**

- Votes can be recorded on elections after they are closed, violating the election integrity model
- The election state machine (Editable → Open → Closed) can be circumvented
- Tallying results could include votes cast after the official close, undermining election validity

**Remediation:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # Wrap the entire check-and-act in a single transaction
    self.db.conn.execute('BEGIN IMMEDIATE')
    try:
        # CHECK: Election is open (within transaction, holds read lock)
        md = self._all_metadata(self.S_OPEN)

        # CHECK: Voter is eligible (within same transaction)
        mayvote = self.q_get_mayvote.first_row(pid, iid)
        if not mayvote:
            raise VoterNotEligible(pid, iid)

        vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
        ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)

        # USE: Insert vote (within same transaction)
        self.c_add_vote.perform(vote_token, ciphertext)
        
        self.db.conn.execute('COMMIT')
    except Exception:
        self.db.conn.execute('ROLLBACK')
        raise
```

Using `BEGIN IMMEDIATE` acquires a reserved lock, preventing concurrent writes (like election close) from completing until this transaction finishes.

#### ASVS-1542-HIGH-002: Election Open Operation — State Check and Multi-Step Transition Not Atomic (TOCTOU)

**Location:**  
- **File:** `v3/steve/election.py`
- **Function:** `open()`
- **Lines:** ~68-77

**Related Functions Checked:**
- `is_editable()` (state assertion)
- `add_salts()` (salt generation)
- `c_open.perform()` (state transition)

**Description:**

The `open()` function checks election state, generates per-voter salts, and transitions the election to open state across multiple separate operations. The state check occurs outside any transaction, and `add_salts()` uses its own transaction, creating multiple race windows where concurrent open requests can corrupt the election's cryptographic integrity.

**Vulnerable Code:**

```python
def open(self, pdb):
    # Double-check the Election is in the editing state.
    assert self.is_editable()          # CHECK: reads state from DB (separate query)

    # Add salts first. If this is gonna fail, then make sure it
    # happens before we move to the "opened" state.
    self.add_salts()                   # USE 1: modifies mayvote rows (own transaction)

    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)

    self.c_open.perform(salt, opened_key, self.eid)  # USE 2: marks election as open (autocommit)
```

And `add_salts()`:

```python
def add_salts(self):
    assert self.is_editable()          # CHECK: second separate state read
    self.db.conn.execute('BEGIN TRANSACTION')
    self.q_all_issues.perform(self.eid)
    for mayvote in self.q_all_issues.fetchall():
        salt = crypto.gen_salt()
        self.c_salt_mayvote.perform(salt, mayvote.rowid)
    self.db.conn.execute('COMMIT')
```

**Data Flow:**

1. **Source:** Admin request via `GET /do-open/<eid>`
2. **Check:** `is_editable()` → reads election metadata → verifies `salt IS NULL AND opened_key IS NULL`
3. **Action 1:** `add_salts()` → generates and stores per-voter salts (in its own transaction)
4. **Action 2:** `c_open.perform()` → sets `salt` and `opened_key` on election (separate autocommit)
5. **Missing Control:** No single transaction wrapping the state check, salt generation, and state transition

**Race Window:**

Two concurrent admin requests to open the same election both pass `is_editable()`, then both call `add_salts()`. The second invocation overwrites the salts written by the first. The `opened_key` computed by the first admin uses the first set of salts, but the database now contains the second set. This creates a mismatch between `opened_key` and the actual salt values, corrupting the anti-tamper mechanism and potentially making all votes undecryptable.

**Proof of Concept:**

```
Timeline:
T1: Admin A → GET /do-open/abc123 → open() asserts is_editable() → PASS
T2: Admin B → GET /do-open/abc123 → open() asserts is_editable() → PASS (still editable!)
T3: Admin A → add_salts() → BEGIN, writes salts S1, COMMIT
T4: Admin B → add_salts() → BEGIN, writes salts S2 (OVERWRITING S1), COMMIT
T5: Admin A → gather_election_data() → reads data with salts S2 (or mixed!)
T6: Admin A → c_open.perform(salt_A, key_A) → election "opened" with key_A
T7: Admin B → c_open.perform(salt_B, key_B) → election "opened" with key_B (overwrites A)
Result: opened_key may not match the actual election data, is_tampered() returns True
```

**Impact:**

- Election integrity check (`is_tampered()`) may always return True, preventing valid tallying
- Per-voter encryption salts could be overwritten, making previously-cast votes undecryptable if any were cast between the two opens
- Election could be left in an inconsistent state where `opened_key` doesn't correspond to actual election data

**Remediation:**

```python
def open(self, pdb):
    # Use a single IMMEDIATE transaction for the entire operation
    self.db.conn.execute('BEGIN IMMEDIATE')
    try:
        # Atomic state check within the transaction
        md = self._all_metadata(self.S_EDITABLE)
        
        # Generate per-voter salts
        self.q_all_issues.perform(self.eid)
        for mayvote in self.q_all_issues.fetchall():
            salt = crypto.gen_salt()
            self.c_salt_mayvote.perform(salt, mayvote.rowid)

        edata = self._gather_election_data_internal(pdb)
        salt = crypto.gen_salt()
        opened_key = crypto.gen_opened_key(edata, salt)

        # Transition state atomically
        # Ideally c_open SQL: UPDATE election SET salt=?, opened_key=? 
        #   WHERE eid=? AND salt IS NULL AND opened_key IS NULL
        self.c_open.perform(salt, opened_key, self.eid)
        
        self.db.conn.execute('COMMIT')
    except Exception:
        self.db.conn.execute('ROLLBACK')
        raise
```

Additionally, the `c_open` SQL should include state conditions in its WHERE clause:

```sql
UPDATE election SET salt = ?, opened_key = ? 
WHERE eid = ? AND salt IS NULL AND opened_key IS NULL
```

#### ASVS-1542-HIGH-003: Election Delete — State Assertion Before Transaction Creates Race Window (TOCTOU)

**Location:**  
- **File:** `v3/steve/election.py`
- **Function:** `delete()`
- **Lines:** ~48-65

**Related Functions Checked:**
- `is_editable()` (state assertion)
- `c_delete_mayvote.perform()` (mayvote deletion)
- `c_delete_issues.perform()` (issues deletion)
- `c_delete_election.perform()` (election deletion)

**Description:**

The `delete()` function asserts that the election is editable before beginning a transaction to delete the election and its related data. This state check occurs outside the transaction boundary, allowing a concurrent request to open the election after the check passes but before the transaction begins, resulting in deletion of an active election.

**Vulnerable Code:**

```python
def delete(self):
    "Delete this Election and its Issues and Person/Issue pairs."

    # Can't delete if it has been opened (even if later closed).
    assert self.is_editable()                       # CHECK: outside transaction

    # Normally, we are in auto-commit mode. Switch to transactional.
    self.db.conn.execute('BEGIN TRANSACTION')        # Transaction starts AFTER check

    # Order these things because of referential integrity.
    self.c_delete_mayvote.perform(self.eid)
    self.c_delete_issues.perform(self.eid)
    self.c_delete_election.perform(self.eid)

    self.db.conn.execute('COMMIT')
```

**Data Flow:**

1. **Source:** Admin action (called from `delete_by_eid()` or directly)
2. **Check:** `is_editable()` → reads election state → verifies no salt/opened_key
3. **Action:** `BEGIN TRANSACTION` → delete mayvote → delete issues → delete election → `COMMIT`
4. **Missing Control:** State assertion is OUTSIDE the transaction boundary

**Race Window:**

Between `assert self.is_editable()` passing and `BEGIN TRANSACTION` executing, a concurrent request could open the election via `open()`. The delete then proceeds on an election that is now open, destroying an active election with salts and voter data.

**Proof of Concept:**

```
Timeline:
T1: Admin A → delete(eid) → is_editable() → PASS
T2: Admin B → open(eid) → is_editable() → PASS, add_salts(), c_open.perform() → Election now OPEN
T3: Admin A → BEGIN TRANSACTION → deletes mayvote (with salts!), issues, election → COMMIT
Result: Active election with generated salts deleted; voters who received ballot links find election gone
```

**Impact:**

- An active, opened election could be deleted, destroying all voter eligibility data and per-voter salts
- Any votes already cast become orphaned (vote_tokens reference deleted mayvote salts)
- Election integrity is destroyed with no recovery path

**Remediation:**

```python
def delete(self):
    "Delete this Election and its Issues and Person/Issue pairs."

    self.db.conn.execute('BEGIN IMMEDIATE')
    try:
        # Check state INSIDE the transaction
        md = self._all_metadata(self.S_EDITABLE)

        self.c_delete_mayvote.perform(self.eid)
        self.c_delete_issues.perform(self.eid)
        self.c_delete_election.perform(self.eid)

        self.db.conn.execute('COMMIT')
    except Exception:
        self.db.conn.execute('ROLLBACK')
        raise

    self.db.conn.close()
    self.db = None
```

#### ASVS-1542-MEDIUM-004: Election Close — State Check and Close Update Not Atomic (TOCTOU)

**Location:**  
- **File:** `v3/steve/election.py`
- **Function:** `close()`
- **Lines:** ~108-113

**Related Functions Checked:**
- `is_open()` (state assertion)
- `c_close.perform()` (close update)

**Description:**

The `close()` function checks that the election is open before updating it to closed state. These operations are not wrapped in a transaction, and the UPDATE likely lacks state conditions in its WHERE clause, creating a race condition with concurrent vote submissions and potentially allowing double-close scenarios.

**Vulnerable Code:**

```python
def close(self):
    "Close an election."

    # The Election should be open.
    assert self.is_open()               # CHECK: reads state from DB
    
    # Simple tweak of the metadata to close the Election.
    self.c_close.perform(self.eid)      # USE: updates closed flag
```

**Data Flow:**

1. **Source:** Admin request via `GET /do-close/<eid>`
2. **Check:** `is_open()` → `get_state()` → `_all_metadata()` → SELECT from election
3. **Action:** `c_close.perform(self.eid)` → UPDATE election SET closed=1
4. **Missing Control:** No transaction wrapping check and update; UPDATE likely lacks state condition in WHERE clause

**Race Condition with Vote Submission:**

The more significant race is between a voter's `add_vote()` (which checks `_all_metadata(S_OPEN)`) and an admin's `close()`. Since neither operation locks the election row, a vote can be recorded after close is initiated but before the update completes, or vice versa.

**Proof of Concept:**

```
Timeline:
T1: Voter → add_vote() → _all_metadata(S_OPEN) → PASS (election is open)
T2: Admin → close() → is_open() → PASS, c_close.perform() → election now CLOSED
T3: Voter → add_vote() continues → c_add_vote.perform() → Vote recorded on CLOSED election
```

**Impact:**

- Votes may be accepted on closed elections, undermining the finality of election closure
- Tally results could include votes cast after official close time
- While close itself is idempotent, the interaction with concurrent vote submission violates the state machine invariant

**Remediation:**

```python
def close(self):
    "Close an election."
    self.db.conn.execute('BEGIN IMMEDIATE')
    try:
        self._all_metadata(self.S_OPEN)  # Verify state within transaction
        self.c_close.perform(self.eid)
        self.db.conn.execute('COMMIT')
    except Exception:
        self.db.conn.execute('ROLLBACK')
        raise
```

The `c_close` SQL should also verify state:

```sql
UPDATE election SET closed = 1 WHERE eid = ? AND closed IS NOT 1 AND salt IS NOT NULL
```

#### ASVS-1542-MEDIUM-005: Batch Vote Submission Not Atomic — Partial Vote Recording on Failure

**Location:**  
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint()`
- **Lines:** ~373-410

**Related Functions Checked:**
- `election.list_issues()` (issue retrieval)
- `election.add_vote()` (per-issue vote submission)

**Description:**

The `do_vote_endpoint()` function processes multi-issue ballot submissions by calling `add_vote()` individually for each issue. Each call is a separate autocommit operation. If a failure occurs mid-loop (e.g., election closes), some votes are recorded while others are not, with no ability to rollback the partial submission.

**Vulnerable Code:**

```python
async def do_vote_endpoint(election):
    result = await basic_info()
    form = edict(await quart.request.form)
    # ...
    votes = {}
    for key, value in form.items():
        if key.startswith('vote-'):
            iid = key.split('-', 1)[1]
            votes[iid] = value

    issues = election.list_issues()
    issue_dict = {i.iid: i for i in issues}

    # Process each vote individually — no wrapping transaction
    for iid, votestring in votes.items():
        if iid not in issue_dict:
            await flash_danger(f'Invalid issue ID: {iid}')
            return quart.redirect(f'/vote-on/{election.eid}', code=303)

        try:
            election.add_vote(result.uid, iid, votestring)   # Each is independent autocommit
        except Exception as e:
            # ...
            await flash_danger(f'Error submitting vote for issue {iid}.')
            return quart.redirect(f'/vote-on/{election.eid}', code=303)
```

**Data Flow:**

1. **Source:** Voter form submission via `POST /do-vote/<eid>` with multiple `vote-<iid>` fields
2. **Issue:** Each `add_vote()` call is a separate autocommit operation. If the 3rd of 5 votes fails (e.g., election is closed mid-loop), votes 1-2 are recorded but votes 3-5 are not
3. **Missing Control:** No transaction wrapping all votes in the batch

**Race Condition:**

Between processing individual votes in the loop, the election could be closed or the async event loop could yield control to process other requests. Since each `add_vote()` contains an `await`-free synchronous database call, the individual calls are sequential but separated by Python-level operations.

**Impact:**

- A voter submitting a multi-issue ballot may have only some votes recorded
- The voter receives an error message but has no way to know which votes succeeded
- Re-voting may be possible (by design), but the voter may not realize partial submission occurred
- The election state check in `add_vote()` could pass for early votes and fail for later ones if the election is concurrently closed

**Remediation:**

```python
async def do_vote_endpoint(election):
    result = await basic_info()
    form = edict(await quart.request.form)
    # ... extract votes dict ...

    # Wrap all votes in a single transaction
    election.db.conn.execute('BEGIN IMMEDIATE')
    try:
        for iid, votestring in votes.items():
            if iid not in issue_dict:
                raise ValueError(f'Invalid issue ID: {iid}')
            election.add_vote_in_txn(result.uid, iid, votestring)
        election.db.conn.execute('COMMIT')
    except Exception as e:
        election.db.conn.execute('ROLLBACK')
        await flash_danger(f'Error submitting votes: {e}')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    await flash_success('Votes submitted successfully!')
    return quart.redirect('/voter', code=303)
```

## Positive Security Patterns

1. **Three-State Election Model:** The schema and code enforce a clear Editable → Open → Closed lifecycle with well-defined invariants (salt/opened_key NULL in editable, NOT NULL in open, closed=1 in closed).

2. **Anti-Tamper Detection:** `is_tampered()` recomputes the `opened_key` from current election data and compares it, providing integrity verification even if race conditions occur.

3. **Per-Voter Cryptographic Salts:** Each voter-issue pair gets a unique salt, preventing vote correlation even if the database is compromised.

4. **Transaction Use for Multi-Row Operations:** `delete()` and `add_salts()` correctly use `BEGIN TRANSACTION...COMMIT` for their multi-row operations (the issue is that state checks occur outside these transactions).

5. **Re-Voting by Design:** The `vote` table allows duplicate `vote_token` values with autoincrement ordering, making the system tolerant of double-submissions (latest wins). This reduces the severity of vote submission races.

6. **Referential Integrity Constraints:** Foreign keys with `ON DELETE RESTRICT` prevent accidental cascade deletions.

7. **Schema-Level Constraints:** STRICT tables, CHECK constraints on field lengths, and GLOB patterns on IDs provide defense-in-depth at the data layer.

8. **Trigger for Closed-State Protection:** `prevent_open_close_update` prevents modification of date fields on closed elections.

---

## Architecture Observations

1. **SQLite Autocommit Default:** The application operates in autocommit mode by default, meaning each SQL statement is its own transaction. This means all check-then-act patterns are inherently non-atomic unless explicitly wrapped in transactions. The `BEGIN IMMEDIATE` variant would be ideal as it acquires a reserved lock, serializing concurrent writes.

2. **Assertion-Based State Guards:** The code uses Python `assert` statements for state validation (`assert self.is_editable()`). These can be disabled with `python -O` and provide no database-level guarantee. State conditions should be encoded in SQL WHERE clauses for atomic enforcement.

3. **Async Framework + Synchronous DB:** Quart's async handlers can interleave request processing. While SQLite operations themselves are synchronous (blocking), the time between the async handler receiving a request and the synchronous DB operations executing creates race windows, especially around `await` points like `await quart.request.form` and `await basic_info()`.

4. **Missing `queries.yaml`:** The actual SQL statements are in `queries.yaml` (not provided). Atomic WHERE clause conditions (e.g., `WHERE eid = ? AND salt IS NULL`) could partially mitigate some TOCTOU issues at the SQL level. However, the Python code pattern of assert-then-perform strongly suggests the SQL doesn't include such conditions, as checking both Python-side and SQL-side would be redundant if the SQL conditions existed.

## Recommendations Summary

### Immediate Actions (Priority 1 — Critical Path)

1. **Wrap `add_vote()` in a single `BEGIN IMMEDIATE` transaction** including the state check (`_all_metadata`), eligibility check (`q_get_mayvote`), and vote insertion (`c_add_vote`). This is the highest-traffic security-critical operation and the one explicitly called out in the domain context.

2. **Move state assertions inside transactions for `open()` and `delete()`** — both can leave the election in an inconsistent state under concurrent access. Use `BEGIN IMMEDIATE` to serialize these state transitions.

### Short-term Actions (Priority 2 — Next Sprint)

3. **Add state conditions to SQL WHERE clauses** in `c_open`, `c_close`, and `c_delete_election` queries in `queries.yaml`. For example:
   - `c_open`: `UPDATE election SET salt=?, opened_key=? WHERE eid=? AND salt IS NULL`
   - `c_close`: `UPDATE election SET closed=1 WHERE eid=? AND closed IS NOT 1 AND salt IS NOT NULL`
   - Then check `rowcount` to detect the race condition and fail safely.

4. **Wrap batch vote submission** in `do_vote_endpoint()` in a single transaction to ensure all-or-nothing vote recording.
