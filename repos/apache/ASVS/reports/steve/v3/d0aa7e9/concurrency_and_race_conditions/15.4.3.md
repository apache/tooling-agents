### Summary

This audit examines the election system's concurrency controls against ASVS 15.4.3 requirements for consistent lock usage, prevention of deadlocks/livelocks, and encapsulation of locking logic. The application relies on SQLite's file-level locking and transactional semantics as its primary concurrency control, supplemented by state machine checks. Several significant gaps exist where state checks and data modifications are not atomic, creating time-of-check-to-time-of-use (TOCTOU) vulnerabilities, and where the state enforcement mechanism (`assert`) can be completely disabled by a Python runtime flag.

## Findings

### HIGH Severity Findings

#### Finding 1: TOCTOU in Vote Submission — Eligibility Check and Vote Insert Not Atomic

**Finding ID:** ASVS-1543-SEV-001  
**Severity:** HIGH  
**Gap Type:** Type A (Entry point with no concurrency control)

**Location:**
- **File:** `v3/steve/election.py`
- **Method:** `add_vote()`
- **Lines:** 266–283

**Related Functions Checked:**
- `_all_metadata()` (state check query)
- `q_get_mayvote.first_row()` (eligibility check query)
- `c_add_vote.perform()` (vote insert operation)
- `do_vote_endpoint()` in `pages.py:307` (HTTP entry point)

**Description:**

The `add_vote()` method performs three separate, non-atomic database operations: (1) checking election state, (2) verifying voter eligibility, and (3) inserting the vote. These operations are not wrapped in a transaction, creating a time-of-check-to-time-of-use (TOCTOU) vulnerability where the election state can change between the state check and vote insertion.

**Vulnerable Code:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)          # STEP 1: READ state (separate query)

    ### validate VOTESTRING for ISSUE.TYPE voting

    mayvote = self.q_get_mayvote.first_row(pid, iid)  # STEP 2: READ eligibility (separate query)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)

    # Pass MAYVOTE.SALT for PBKDF.
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)

    self.c_add_vote.perform(vote_token, ciphertext)    # STEP 3: WRITE vote (separate operation)
```

**Data Flow:**

HTTP POST `/do-vote/<eid>` → `do_vote_endpoint()` (pages.py:307) → `election.add_vote()` → three separate non-transactional database operations

**Race Condition Scenario:**

1. **Request A** (voter): Calls `_all_metadata(S_OPEN)` — election is open ✓
2. **Request B** (admin): Calls `close()` — election transitions to closed
3. **Request A** (voter): Calls `c_add_vote.perform()` — vote inserted into a **closed** election

**Proof of Concept:**

```bash
# Two concurrent requests:
# Terminal 1 (admin closes election):
curl -b session=admin_cookie https://example.com/do-close/a1b2c3d4e5

# Terminal 2 (voter submits vote, overlapping with close):
curl -X POST -b session=voter_cookie \
  -d "vote-f6g7h8i9j0=yes" \
  https://example.com/do-vote/a1b2c3d4e5
```

**Impact:**

Votes can be recorded after an election has been officially closed, undermining election integrity. The closed election's tally would include votes cast after the intended cutoff. This also means the `opened_key` tamper-detection mechanism could miss the inconsistency since it was computed before closure.

**Remediation:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    self.db.conn.execute('BEGIN IMMEDIATE')  # Acquire write lock immediately
    try:
        # Check state WITHIN the transaction
        md = self._all_metadata(self.S_OPEN)

        mayvote = self.q_get_mayvote.first_row(pid, iid)
        if not mayvote:
            raise ValueError(f'Voter {pid} not eligible for issue {iid}')

        vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
        ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
        self.c_add_vote.perform(vote_token, ciphertext)

        self.db.conn.execute('COMMIT')
    except Exception:
        self.db.conn.execute('ROLLBACK')
        raise
```

#### Finding 2: Non-Atomic Election `open()` — Multi-Step State Transition Without Overarching Transaction

**Finding ID:** ASVS-1543-SEV-002  
**Severity:** HIGH  
**Gap Type:** Type B (Transaction control EXISTS in `add_salts()` but NOT wrapping the full operation)

**Location:**
- **File:** `v3/steve/election.py`
- **Method:** `open()`
- **Lines:** 75–91

**Related Functions Checked:**
- `is_editable()` (state check)
- `add_salts()` (has its own BEGIN/COMMIT)
- `gather_election_data()` (multiple reads)
- `c_open.perform()` (state change operation)
- `do_open_endpoint()` in `pages.py:378` (HTTP entry point)

**Description:**

The `open()` method performs a multi-step state transition consisting of: (1) state verification via assert, (2) salt generation in its own transaction, (3) election data gathering, and (4) final state change. These steps are not wrapped in a single overarching transaction, allowing race conditions where multiple concurrent `open()` calls can interleave, corrupting cryptographic keys and salts.

**Vulnerable Code:**

```python
def open(self, pdb):
    # Double-check the Election is in the editing state.
    assert self.is_editable()                  # STEP 1: State check (assert, separate query)

    # Add salts first.
    self.add_salts()                           # STEP 2: Own BEGIN/COMMIT transaction

    edata = self.gather_election_data(pdb)     # STEP 3: Multiple reads
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)

    self.c_open.perform(salt, opened_key, self.eid)  # STEP 4: State change (autocommit)
```

**Data Flow:**

HTTP GET `/do-open/<eid>` → `do_open_endpoint()` (pages.py:378) → `election.open()` → four separate operations with `add_salts()` having its own transaction

**Race Condition Scenario:**

1. **Admin A**: `assert self.is_editable()` passes ✓, enters `add_salts()`
2. **Admin B**: `assert self.is_editable()` passes ✓ (salts transaction from A committed, but election still editable), enters `add_salts()` — **overwrites A's salts**
3. **Admin A**: `c_open.perform()` with `opened_key` computed from original data
4. **Admin B**: `c_open.perform()` with different `opened_key` computed from same data but different salts — **overwrites A's opened_key**

Result: `opened_key` corresponds to B's salts, but some mayvote entries may have A's salts (depending on timing), leading to vote tokens that cannot decrypt votes correctly.

**Crash Scenario:**

If the process crashes after `add_salts()` (step 2) but before `c_open.perform()` (step 4), the mayvote table has salts but the election remains in "editable" state. A subsequent `open()` call would overwrite all salts, but any voter who received a ballot link between the crash and re-open would have stale tokens.

**Impact:**

Election state corruption. Voters may be unable to cast valid votes if their salt was overwritten. The opened_key integrity check would fail, potentially flagging the election as tampered when it was actually a race condition.

**Remediation:**

```python
def open(self, pdb):
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)

    self.db.conn.execute('BEGIN IMMEDIATE')
    try:
        # Re-verify state within the write lock
        md = self.q_metadata.first_row(self.eid)
        if self._compute_state(md) != self.S_EDITABLE:
            self.db.conn.execute('ROLLBACK')
            raise ElectionBadState(self.eid, self._compute_state(md), self.S_EDITABLE)

        # Add salts within this transaction (remove inner BEGIN/COMMIT from add_salts)
        self._add_salts_inner()

        edata = self.gather_election_data(pdb)
        salt = crypto.gen_salt()
        opened_key = crypto.gen_opened_key(edata, salt)

        self.c_open.perform(salt, opened_key, self.eid)
        self.db.conn.execute('COMMIT')
    except Exception:
        self.db.conn.execute('ROLLBACK')
        raise
```

#### Finding 3: State Machine Enforcement via `assert` — Removable in Production

**Finding ID:** ASVS-1543-SEV-003  
**Severity:** HIGH  
**Gap Type:** Type B (Control exists but not reliably applied — `assert` disabled by `-O` flag)

**Location:**
- **File:** `v3/steve/election.py`
- **Multiple methods** (see table below)

**Related Functions Checked:**

All state-checking methods across the Election class

**Vulnerable Code Locations:**

| Method | Line (approx) | Assert Statement |
|--------|------|-----------------|
| `delete()` | 56 | `assert self.is_editable()` |
| `open()` | 78 | `assert self.is_editable()` |
| `close()` | 124 | `assert self.is_open()` |
| `add_salts()` | 133 | `assert self.is_editable()` |
| `add_issue()` | 193 | `assert self.is_editable()` |
| `edit_issue()` | 211 | `assert self.is_editable()` |
| `delete_issue()` | 227 | `assert self.is_editable()` |
| `add_voter()` | 257 | `assert self.is_editable()` |
| `add_issue()` | 194 | `assert vtype in vtypes.TYPES` |
| `edit_issue()` | 212 | `assert vtype in vtypes.TYPES` |

**Description:**

The application uses Python `assert` statements for critical state machine enforcement in admin operations. When Python runs with optimization flags (`-O` or `-OO`), all assert statements are removed from the bytecode, completely disabling state machine enforcement. This violates ASVS 15.4.3's requirement that "locking logic stays within the code responsible for managing the resource to ensure locks cannot be inadvertently... modified."

**Proof of Concept:**

```bash
# Running with optimization removes all assert statements:
python -O -m quart run
# or
PYTHONOPTIMIZE=1 gunicorn ...

# Now all state checks are bypassed:
# - Issues can be added to OPEN elections
# - Elections can be "opened" twice (overwriting salts/keys)
# - Elections can be "closed" when already closed
# - Voters can be added to OPEN elections
```

**Impact:**

When Python runs with `-O` or `-OO` flags (common in production deployments), **all** state machine enforcement for admin operations is silently removed. This means:
- Issues can be added/modified in open elections, invalidating the `opened_key` tamper check
- `close()` can be called on already-closed elections
- `open()` can be called on already-open elections, overwriting cryptographic keys and corrupting all existing votes
- `delete()` can be called on open/closed elections, destroying active elections

This directly violates ASVS 15.4.3's requirement that "locking logic stays within the code responsible for managing the resource to ensure locks cannot be inadvertently... modified."

**Remediation:**

```python
# Replace all assert-based state checks with proper exceptions:
def close(self):
    "Close an election."
    if not self.is_open():
        raise ElectionBadState(self.eid, self.get_state(), self.S_OPEN)
    self.c_close.perform(self.eid)

def delete(self):
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
    # ... rest of method

# Apply the same pattern to: open(), add_salts(), add_issue(),
# edit_issue(), delete_issue(), add_voter()
```

### MEDIUM Severity Findings

#### Finding 4: Non-Atomic Election `close()` — Check-Then-Act Without Transaction

**Finding ID:** ASVS-1543-SEV-004  
**Severity:** MEDIUM  
**Gap Type:** Type A (No transaction wrapping state check + state change)

**Location:**
- **File:** `v3/steve/election.py`
- **Method:** `close()`
- **Lines:** 121–128

**Related Functions Checked:**
- `is_open()` (state check)
- `c_close.perform()` (state change operation)
- `do_close_endpoint()` in `pages.py` (HTTP entry point)

**Description:**

The `close()` method performs a state check followed by a state change operation without wrapping them in a transaction. This creates a check-then-act race condition where multiple administrators can simultaneously close an election, or where the state can change between the check and the action.

**Vulnerable Code:**

```python
def close(self):
    "Close an election."

    # The Election should be open.
    assert self.is_open()                  # State check (separate query)

    # Simple tweak of the metadata to close the Election.
    self.c_close.perform(self.eid)         # State change (separate operation)
```

**Race Condition Scenario:**

Two administrators simultaneously close the same election. Both `assert self.is_open()` pass. Both `c_close.perform()` execute. While the end result (closed=1) is the same, this demonstrates the lack of atomic state transitions. More critically, if the `c_close` query doesn't include a WHERE clause checking current state, it could set `closed=1` on an election that was never properly opened.

**Impact:**

While double-close has limited direct impact (idempotent), the pattern demonstrates that state transitions lack atomicity. Combined with Finding 3 (assert removal), this allows arbitrary state transitions. The `c_close` query parameters (`self.eid` only) suggest no conditional WHERE clause checking current state, meaning the SQL layer provides no protection.

**Remediation:**

```python
def close(self):
    "Close an election."
    if not self.is_open():
        raise ElectionBadState(self.eid, self.get_state(), self.S_OPEN)

    # Use conditional UPDATE that checks current state atomically
    # In queries.yaml, c_close should be:
    # UPDATE election SET closed = 1
    #   WHERE eid = ? AND salt IS NOT NULL AND opened_key IS NOT NULL
    #   AND (closed IS NULL OR closed = 0)
    self.c_close.perform(self.eid)
    if self.c_close.rowcount == 0:
        raise ElectionBadState(self.eid, self.get_state(), self.S_OPEN)
```

#### Finding 5: Batch Vote Submission Without Wrapping Transaction

**Finding ID:** ASVS-1543-SEV-005  
**Severity:** MEDIUM  
**Gap Type:** Type A (No transaction for multi-vote atomic submission)

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint()`
- **Lines:** 307–353

**Related Functions Checked:**
- `election.add_vote()` (called in loop)
- Form parsing and validation logic
- Flash message handlers

**Description:**

The vote submission endpoint processes multiple votes (one per issue) in a loop without wrapping them in a transaction. Each `add_vote()` call is independent, allowing partial ballot submission if an error occurs mid-loop or if the election state changes during processing.

**Vulnerable Code:**

```python
# Process each vote
for iid, votestring in votes.items():
    if iid not in issue_dict:
        await flash_danger(f'Invalid issue ID: {iid}')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    try:
        election.add_vote(result.uid, iid, votestring)  # Each is independent
        _LOGGER.info(...)
    except Exception as e:
        _LOGGER.error(...)
        await flash_danger(f'Error submitting vote for issue {iid}.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
```

**Race Condition Scenario:**

A voter submits votes for 5 issues simultaneously. After 3 votes are inserted, the election is closed by an admin. The remaining 2 votes fail. The voter's ballot is now partially submitted — 3 of 5 issues have votes, with no indication to the voter which succeeded.

**Impact:**

Voters can end up with partially recorded ballots. In a YNA election with multiple issues, this could change the outcome (e.g., a voter intended to vote "no" on all 5, but only 3 "no" votes were recorded). The voter receives an error message but may not know which votes succeeded.

**Remediation:**

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    form = edict(await quart.request.form)
    if not form:
        await flash_danger('No vote data provided.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    votes = {}
    for key, value in form.items():
        if key.startswith('vote-'):
            iid = key.split('-', 1)[1]
            votes[iid] = value

    issues = election.list_issues()
    issue_dict = {i.iid: i for i in issues}

    # Validate all votes first
    for iid in votes:
        if iid not in issue_dict:
            await flash_danger(f'Invalid issue ID: {iid}')
            return quart.redirect(f'/vote-on/{election.eid}', code=303)

    # Submit all votes atomically
    try:
        election.add_votes_batch(result.uid, votes)  # New method wrapping in transaction
    except Exception as e:
        _LOGGER.error(f'Error submitting votes for user[U:{result.uid}]: {e}')
        await flash_danger('Error submitting votes. No votes were recorded.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    await flash_success('Votes submitted successfully!')
    return quart.redirect('/voter', code=303)
```

## Positive Security Patterns

1. **Transaction usage in `delete()` and `add_salts()`**: These methods correctly use `BEGIN TRANSACTION` / `COMMIT` to wrap multi-step operations, demonstrating awareness of atomicity requirements.

2. **Non-sequential IDs**: Using random hex IDs for elections and issues prevents enumeration attacks that could amplify race conditions.

3. **Re-voting by design**: The vote table allows duplicate `vote_token` entries with `AUTOINCREMENT` ordering. This means that a race condition causing duplicate vote submissions does not corrupt data — the latest vote is used during tallying.

4. **`_all_metadata(required_state)` pattern**: For voting and tallying operations, proper exception-based state checking is implemented (unlike the assert-based admin operations).

5. **`opened_key` tamper detection**: The cryptographic integrity check provides a post-hoc detection mechanism for state corruption, even if concurrency controls fail.

6. **Trigger-based protection**: The `prevent_open_close_update` trigger provides database-level enforcement that cannot be bypassed by application code or Python optimization flags.

7. **Per-request `Election` instances**: Each HTTP handler creates its own `Election` object with a dedicated database connection, avoiding shared mutable state at the application level.

---

## Architecture Observations

1. **SQLite as sole concurrency mechanism**: The application relies entirely on SQLite's file-level locking for concurrency control. While SQLite provides serializable transactions via `BEGIN IMMEDIATE`, this is only effective when transactions are actually used. Most critical operations do not use transactions.

2. **Synchronous DB in async framework**: The `Election` class uses synchronous SQLite operations within Quart's async handlers. This means each database call blocks the event loop, providing accidental serialization within a single worker. However, this protection disappears with multiple workers (e.g., Hypercorn with `--workers > 1`).

3. **No application-level locks**: The codebase has zero instances of `asyncio.Lock`, `threading.Lock`, or any application-level locking mechanism. All concurrency control is delegated to the database layer, which is only effective within properly-scoped transactions.

4. **State machine enforcement split between application and database**: The election state machine is checked in Python (`assert`/`_all_metadata`) but not enforced in SQL queries. Ideally, state-changing SQL queries should include WHERE clauses that verify current state, providing database-level enforcement.

## Recommendations Summary

### Immediate Actions (Priority 1 — Critical Concurrency Fixes)

**Addressing Findings 1, 2, 3:**

1. **Replace all `assert` statements with proper exceptions** (Finding 3): This is the lowest-effort, highest-impact change. Every `assert self.is_editable()` and `assert self.is_open()` should become an `if not ...: raise ElectionBadState(...)`. This ensures state checks cannot be silently removed.

2. **Wrap `add_vote()` in a transaction** (Finding 1): Use `BEGIN IMMEDIATE` to acquire a write lock before checking state and eligibility, then insert the vote within the same transaction. This is the most critical data integrity fix.

3. **Wrap `open()` in a single transaction** (Finding 2): Combine `add_salts()` and `c_open.perform()` into a single transaction. Remove the inner `BEGIN`/`COMMIT` from `add_salts()` and wrap the entire `open()` operation.

### Short-term Actions (Priority 2 — Defensive SQL Queries)

**Addressing Finding 4:**

4. **Add state-checking WHERE clauses to state-changing SQL queries**: Modify `c_open`, `c_close`, and similar queries in `queries.yaml` to include conditions like `WHERE eid = ? AND salt IS NULL` (for open) or `WHERE eid = ? AND closed IS NULL OR closed = 0` (for close). Check `rowcount` after execution to detect concurrent modifications.

### Long-term Actions (Priority 3 — Batch Vote Atomicity)

**Addressing Finding 5:**

5. **Create an `add_votes_batch()` method** that wraps multiple vote inserts in a single transaction, ensuring all-or-nothing ballot submission.
