### ASVS-812-SEV-001
**Classification: CRITICAL AUTHORIZATION GAP**  
**Title:** Election management endpoints lack ownership authorization — any committer can manage any election

**Gap Type:** Type B — Authorization concept exists (`owner_pid`, `authz` fields, `### check authz` comments) but is **never checked**.

**File:** `v3/server/pages.py`

**All affected endpoints with explicit `### check authz` TODO markers:**

| Endpoint | Function | Line | Comment |
|---|---|---|---|
| `POST /do-vote/<eid>` | `do_vote_endpoint` | ~379 | `### check authz` |
| `POST /do-create-election` | `do_create_endpoint` | ~419 | `### check authz` |
| `GET /do-open/<eid>` | `do_open_endpoint` | ~436 | `### check authz` |
| `GET /do-close/<eid>` | `do_close_endpoint` | ~453 | `### check authz` |
| `POST /do-add-issue/<eid>` | `do_add_issue_endpoint` | ~469 | `### check authz` |
| `POST /do-edit-issue/<eid>/<iid>` | `do_edit_issue_endpoint` | ~489 | `### check authz` |
| `POST /do-delete-issue/<eid>/<iid>` | `do_delete_issue_endpoint` | ~511 | `### check authz` |
| `POST /do-set-open_at/<eid>` | `_set_election_date` | ~85 | `### check authz` |
| Decorator `load_election` | `loader` | ~140 | `### check authz` |
| Decorator `load_election_issue` | `loader` | ~164 | `### check authz` |

**Related Functions Checked:**
- `load_election` decorator (line ~140)
- `load_election_issue` decorator (line ~164)
- `do_open_endpoint` (line ~436)
- `do_close_endpoint` (line ~453)
- `do_add_issue_endpoint` (line ~469)
- `do_edit_issue_endpoint` (line ~489)
- `do_delete_issue_endpoint` (line ~511)

**Description:**

The application defines authorization concepts in the database schema (`owner_pid` and `authz` fields) and documents their intended purpose, but never enforces these restrictions in any endpoint handler. All election management endpoints contain explicit `### check authz` TODO comments acknowledging this gap. Any authenticated ASF committer can perform administrative operations on any election, regardless of ownership or group membership.

**Vulnerable Code (representative — `do_open_endpoint`):**

```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    result = await basic_info()

    ### check authz          # <--- TODO: never implemented

    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)       # <--- ANY committer can open ANY election
    ...
```

**Vulnerable Code (`load_election` decorator):**

```python
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            ...
        ### check authz      # <--- TODO: never implemented
        return await func(e)
    return loader
```

**Data Flow:**

1. Attacker authenticates as any ASF committer
2. Attacker discovers/guesses election EID (10-hex-char)
3. Attacker sends `GET /do-open/{eid}` or `GET /do-close/{eid}`
4. `@asfquart.auth.require({R.committer})` passes (user is a committer)
5. `load_election` loads the election without checking `owner_pid` or `authz`
6. Election state is irreversibly changed

**Proof of Concept:**

```http
GET /do-close/a1b2c3d4e5 HTTP/1.1
Cookie: <any-committer-session>
```

Any committer can close any open election, permanently ending voting.

```http
POST /do-delete-issue/a1b2c3d4e5/f6g7h8i9j0 HTTP/1.1
Cookie: <any-committer-session>
Content-Type: application/x-www-form-urlencoded
```

Any committer can delete issues from any editable election.

**Impact:**

- Any of the ~800+ ASF committers can open, close, or modify ANY election
- Election integrity can be compromised by adding/editing/deleting issues before opening
- Closing an election prematurely is irreversible
- Opening an election prematurely locks its structure permanently

**Remediation:**

```python
async def check_election_authz(election, uid):
    """Verify uid is authorized to manage this election."""
    md = election.get_metadata()
    
    # Owner always has access
    if md.owner_pid == uid:
        return True
    
    # Check authz group if defined
    if md.authz:
        s = await asfquart.session.read()
        if md.authz in s.get('groups', []):
            return True
    
    quart.abort(403, 'Not authorized to manage this election')

# Apply in load_election decorator:
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            ...
        await check_election_authz(e, (await basic_info()).uid)
        return await func(e)
    return loader
```

### ASVS-812-SEV-002
**Classification: CRITICAL AUTHORIZATION GAP**  
**Title:** `authz` field defined in schema and documented but never evaluated in any access control decision

**Gap Type:** Type B — Authorization field exists in database schema and documentation but is never checked.

**File:** `v3/schema.sql` (definition), `v3/docs/schema.md` (documentation), `v3/server/pages.py` (missing enforcement)

**Related Functions Checked:**
- `get_metadata()` in `election.py` (line ~143)
- All management endpoint handlers in `pages.py`
- Template rendering in `voter.ezt`

**Description:**

The database schema defines an `authz` field for group-based election editing permissions, and the schema documentation explicitly describes its access control purpose. However, this field is only retrieved for display purposes in templates and is never evaluated in any authorization decision. This creates a false sense of security for anyone reviewing the schema documentation.

**Schema definition (`schema.sql` line ~52):**

```sql
/* What authz group is allowed to edit this election? If NULL,
   then only the OWNER_PID can edit.  */
/* ### contents/format is TBD; think "which PMC" or "Foundation"  */
authz  TEXT,
```

**Documentation (`schema.md`):**

> `authz` (TEXT): Specifies an LDAP group (e.g., group name or identifier) allowed to edit the election. If `NULL`, only `owner_pid` can edit.

**Vulnerable Code — `authz` is read but only for display, never for access control:**

In `get_metadata()` (`election.py` line ~143):

```python
def get_metadata(self):
    md = self._all_metadata()
    return edict(
        ...
        authz=md.authz,   # Passed through for display only
        ...
    )
```

In templates (e.g., `voter.ezt`):

```html
[if-any open_elections.authz]
    <span>[open_elections.authz] can vote</span>
[end]
```

**Data Flow:**

1. Election is created with `authz` field set to a specific LDAP group
2. Schema documentation indicates this should restrict editing to group members
3. User not in the specified group authenticates as any committer
4. User accesses management endpoints
5. `authz` field is retrieved and displayed in UI
6. No authorization check evaluates group membership
7. Unauthorized user successfully modifies election

**Impact:**

- The schema documentation describes `authz` as an access control mechanism ("allowed to edit"), creating false confidence in security architecture
- Authorization rules for group-based editing are documented in schema but completely absent from implementation
- Security auditors reviewing only the schema documentation would incorrectly assume group-based authorization is enforced

**Remediation:**

Create an authorization policy document and implement `authz` group checks in the `load_election` decorator (see ASVS-812-SEV-001 remediation). Also create a dedicated `AUTHORIZATION.md` document defining all access rules.

### ASVS-812-SEV-003
**Classification: HIGH AUTHORIZATION GAP**  
**Title:** Vote submission endpoint lacks explicit voter eligibility authorization check

**Gap Type:** Type B — Eligibility check exists on the GET page but is missing on the POST endpoint.

**File:** `v3/server/pages.py`

**Related Functions Checked:**
- `vote_on_page` (GET endpoint, line ~257)
- `do_vote_endpoint` (POST endpoint, line ~376)
- `election.add_vote()` in `election.py`

**Description:**

The GET endpoint for the voting page (`vote_on_page`) correctly verifies voter eligibility by checking for mayvote entries. However, the POST endpoint (`do_vote_endpoint`) that actually submits votes lacks this explicit check. While `election.add_vote()` will fail if the user is not eligible, this failure is caught as a generic exception with no distinction between authorization failures and other errors.

**GET endpoint — checks eligibility (line ~257):**

```python
@APP.get('/vote-on/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def vote_on_page(election):
    result = await basic_info()
    ...
    # Check if user can vote in this election (has mayvote entries)
    election.q_find_issues.perform(result.uid, election.eid)
    if not election.q_find_issues.fetchall():
        result = await basic_info()
        result.title = 'Access Denied'
        raise_404(T_BAD_EID, result)
    ...
```

**POST endpoint — NO eligibility check (line ~376):**

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    ### check authz          # <--- TODO: never implemented

    form = edict(await quart.request.form)
    ...
    for iid, votestring in votes.items():
        ...
        try:
            election.add_vote(result.uid, iid, votestring)
        except Exception as e:          # Catches auth failure as generic error
            await flash_danger(f'Error submitting vote for issue {iid}.')
            ...
```

**Data Flow:**

1. User authenticates as a committer NOT in the election's voter roll
2. User crafts a POST request directly to `/do-vote/<eid>`
3. The eligibility check from the GET page is bypassed
4. `election.add_vote()` will fail with `AttributeError` (mayvote returns None), caught as generic error
5. No audit log distinguishes authorization failure from other errors

**Proof of Concept:**

```http
POST /do-vote/a1b2c3d4e5 HTTP/1.1
Cookie: <committer-not-in-voter-roll>
Content-Type: application/x-www-form-urlencoded

vote-f6g7h8i9j0=y
```

**Impact:**

- Authorization failure is masked as a generic error
- No audit trail for unauthorized vote attempts
- If `add_vote` implementation changes (e.g., to handle None mayvote gracefully), the implicit protection disappears
- Defense-in-depth principle is violated

**Remediation:**

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    # Verify voter eligibility
    election.q_find_issues.perform(result.uid, election.eid)
    if not election.q_find_issues.fetchall():
        _LOGGER.warning(
            f'Unauthorized vote attempt by User[U:{result.uid}] '
            f'on election[E:{election.eid}]'
        )
        quart.abort(403, 'Not eligible to vote in this election')
    ...
```

### ASVS-812-SEV-004
**Classification: HIGH AUTHORIZATION GAP**  
**Title:** State-dependent write restrictions enforced via Python `assert` statements, which can be disabled at runtime

**Gap Type:** Type C — Control is called but its enforcement can be bypassed.

**File:** `v3/steve/election.py`

**Related Functions Checked:**
- `delete()` (line ~53)
- `open()` (line ~69)
- `add_salts()` (line ~120)
- `close()` (line ~111)
- `add_issue()` (line ~168)
- `edit_issue()` (line ~186)
- `delete_issue()` (line ~197)
- `add_voter()` (line ~226)

**Description:**

Multiple state-dependent write operations use Python `assert` statements to enforce election state requirements. Python `assert` statements can be globally disabled with the `-O` or `-OO` command-line flags, which removes all assertions from the bytecode. This makes state-based authorization controls bypassable through deployment configuration rather than code modification.

**Vulnerable code (multiple locations):**

```python
def delete(self):
    assert self.is_editable()           # Line ~53 - Disabled with python -O
    ...

def open(self, pdb):
    assert self.is_editable()           # Line ~69

def add_salts(self):
    assert self.is_editable()           # Line ~120

def close(self):
    assert self.is_open()               # Line ~111

def add_issue(self, title, description, vtype, kv):
    assert self.is_editable()           # Line ~168
    assert vtype in vtypes.TYPES        # Line ~169

def edit_issue(self, iid, title, description, vtype, kv):
    assert self.is_editable()           # Line ~186

def delete_issue(self, iid):
    assert self.is_editable()           # Line ~197

def add_voter(self, pid, iid):
    assert self.is_editable()           # Line ~226
```

**Data Flow:**

1. Application is deployed with Python optimization flags (`python -O` or `python -OO`)
2. All `assert` statements are removed from bytecode during compilation
3. State checks for `is_editable()` and `is_open()` are silently bypassed
4. Users can modify elections in any state (edit closed elections, close editable elections, etc.)

**Impact:**

- Running Python with `-O` or `-OO` flags disables all `assert` statements
- State-based write restrictions (e.g., cannot edit issues after election opens) would be silently removed
- The election state machine's integrity depends entirely on these assertions
- Per Python documentation: "assert should not be used for data validation because it can be globally disabled"

**Remediation:**

```python
def _require_state(self, required_state):
    """Enforce election state requirement. Raises ElectionBadState."""
    current = self.get_state()
    if current != required_state:
        raise ElectionBadState(self.eid, current, required_state)

def add_issue(self, title, description, vtype, kv):
    self._require_state(self.S_EDITABLE)
    if vtype not in vtypes.TYPES:
        raise ValueError(f'Invalid vote type: {vtype}')
    ...
```

### ASVS-812-SEV-005
**Classification: MEDIUM DOCUMENTATION GAP**  
**Title:** No formal authorization documentation defining field-level read/write access rules per consumer role and resource state

**Gap Type:** Documentation — Required by ASVS 8.1.2

**Files Affected:** 
- Missing: `AUTHORIZATION.md` or equivalent policy document
- Partial documentation in: `v3/schema.md`, `v3/schema.sql`, `v3/server/pages.py` (comments)

**Related Functions Checked:**
- All endpoint handlers in `pages.py`
- All state-dependent methods in `election.py`
- Schema documentation in `schema.md`

**Description:**

ASVS 8.1.2 specifically requires that authorization documentation defines rules for field-level access restrictions based on consumer permissions and resource attributes. While the codebase contains scattered comments and schema documentation, there is no comprehensive authorization policy document.

**What exists:**

- `schema.md` describes database fields and election lifecycle
- `schema.sql` has comments about `authz` and `owner_pid` semantics
- Code has `### check authz` and `### need general solution` comments

**What is missing:**

No document defines:

1. Which roles (unauthenticated, authenticated, committer, PMC member, owner, authz group) can read/write which fields
2. How election state (editable/open/closed) affects which fields can be modified and by whom
3. Field-level access for:
   - Election metadata (title, open_at, close_at) — who can write?
   - Issues (title, description, type, kv) — who can add/edit/delete?
   - Votes — who can submit, who can tally?
   - Person records — who can view PID/name/email?
4. State-transition authorization (who can open/close an election?)

**Evidence of gap:**

- `pages.py` line ~19 comment: `### NOTE: the voting handlers require login for ASF committers only. Obviously, this is not a general purpose solution.`
- `schema.sql` line ~52: `/* ### contents/format is TBD */` for `authz`
- Multiple `### need general solution` annotations on auth decorators

**Impact:**

- Developers cannot consistently implement authorization without a specification
- Testers cannot verify authorization without documented expected behavior
- The current state where any committer has full administrative access is likely unintended but undetectable without a policy document

**Remediation:**

Create an `AUTHORIZATION.md` document specifying:

```markdown
# Authorization Policy

## Roles
| Role | Description |
|------|------------|
| anonymous | Unauthenticated user |
| authenticated | Valid session |
| committer | ASF committer |
| pmc_member | PMC member |
| election_owner | owner_pid matches session UID |
| authz_group | User is member of election.authz group |

## Election Field Access Matrix
| Field | Read | Write (Editable) | Write (Open) | Write (Closed) |
|-------|------|-------------------|--------------|----------------|
| title | committer+eligible | owner/authz_group | DENIED | DENIED |
| open_at | committer+eligible | owner/authz_group | DENIED | DENIED |
| close_at | committer+eligible | owner/authz_group | owner/authz_group | DENIED |
| issues | committer+eligible | owner/authz_group | DENIED | DENIED |
| votes | N/A | N/A | eligible voter | N/A |
| tally | N/A | N/A | N/A | owner/authz_group |

## State Transition Rules
| Transition | Authorized Roles |
|------------|-----------------|
| editable → open | election_owner, authz_group |
| open → closed | election_owner, authz_group |
```

## Positive Security Patterns

1. **Parameterized SQL queries** (`queries.yaml`): All queries use `?` placeholders — no SQL injection vectors.

2. **Non-sequential IDs**: 10-char hex EIDs/IIDs prevent enumeration.

3. **Document access control**: `serve_doc()` correctly checks mayvote eligibility before serving files.

4. **Scoped voter views**: `voter_page` and `admin_page` correctly scope queries to the authenticated user's PID.

5. **`_all_metadata()` with `required_state`**: `add_vote()` and `tally_issue()` use proper exception-based state checking via `ElectionBadState`.

6. **Tamper detection**: `is_tampered()` in tally flow verifies election integrity before producing results.

7. **Vote anonymization**: `crypto.shuffle(votes)` removes database ordering from tally results.

8. **Database schema constraints**: STRICT mode, CHECK constraints, and foreign keys enforce data integrity at the storage layer.

9. **Trigger protection**: `prevent_open_close_update` prevents timestamp modification after election closure.

10. **PMC-level auth for election creation**: `do_create_endpoint` uses `R.pmc_member` rather than `R.committer`.

---

## Architecture Observations

1. **Authorization model is designed but unimplemented**: The database schema includes both `owner_pid` and `authz` fields specifically for authorization. Schema documentation describes their intended semantics. The code contains 12+ `### check authz` markers. This is not a design oversight but acknowledged technical debt.

2. **Two layers of state enforcement with different robustness**: The `election.py` model uses both `assert` (for editing operations) and `_all_metadata(required_state=...)` (for voting/tallying). The latter is robust; the former is not.

3. **Authentication ≠ Authorization**: The system correctly authenticates users (committer/PMC roles) but treats authentication as sufficient for authorization. The `### need general solution` comments on decorators confirm this is a known gap.

4. **GET methods used for state-changing operations**: `/do-open/<eid>` and `/do-close/<eid>` use GET requests for irreversible state transitions, compounding the authorization gap (link sharing or CSRF could trigger these).

## Recommendations Summary

### Immediate Actions (P0)

1. **Implement ownership + authz group checks in `load_election` decorator**
   - Addresses: ASVS-812-SEV-001, ASVS-812-SEV-002
   - Create `check_election_authz()` function that verifies `owner_pid` or `authz` group membership
   - Apply in `load_election` and `load_election_issue` decorators
   - Impact: Prevents horizontal privilege escalation across all management endpoints

2. **Create formal `AUTHORIZATION.md` defining field-level access rules per role and state**
   - Addresses: ASVS-812-SEV-005
   - Document role definitions (anonymous, committer, owner, authz_group, etc.)
   - Create field access matrix showing read/write permissions by state
   - Define state transition authorization rules
   - Impact: Provides specification for consistent authorization implementation and testing

### Short-term Actions (P1)

3. **Add explicit voter eligibility check in `do_vote_endpoint`**
   - Addresses: ASVS-812-SEV-003
   - Duplicate the mayvote check from GET endpoint to POST endpoint
   - Add authorization failure logging distinct from generic errors
   - Impact: Explicit defense-in-depth and improved audit trail

4. **Replace all `assert` state checks with `_require_state()` raising exceptions**
   - Addresses: ASVS-812-SEV-004
   - Create `_require_state()` method that raises `ElectionBadState`
   - Replace all `assert self.is_editable()` and `assert self.is_open()` calls
   - Impact: State-based authorization cannot be disabled via deployment flags

### Long-term Actions (P2-P3)

5. **Change `/do-open/<eid>` and `/do-close/<eid>` from GET to POST methods**
   - Priority: P2
   - Impact: Defense-in-depth against CSRF and accidental link sharing

6. **Add authorization failure audit logging distinct from generic errors**
