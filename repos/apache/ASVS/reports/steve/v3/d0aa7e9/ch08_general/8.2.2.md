### Key Findings

The audit reveals **critical authorization gaps** in the election management system. While authentication controls are consistently applied, **object-level authorization is fundamentally missing** on all election management endpoints. Authorization data (`owner_pid`, `authz`) exists in the database schema and is displayed in the UI, but is **never evaluated for access control decisions** — a textbook Type B gap (control exists but is not applied).

### ASVS-822-CRT-001: Missing Object-Level Authorization on All Election Management Endpoints (Type B Gap)

**Severity:** CRITICAL

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `load_election` decorator (~line 150), `load_election_issue` decorator (~line 175)
- **Affected Endpoints:** 9 routes (see table below)

**Related Functions Checked:**
- `do_open_endpoint` (~line 435)
- `do_close_endpoint` (~line 455)
- `do_add_issue_endpoint` (~line 473)
- `do_edit_issue_endpoint` (~line 495)
- `do_delete_issue_endpoint` (~line 515)
- `manage_page` (~line 305)
- `manage_stv_page` (~line 330)
- `do_set_open_at_endpoint` (~line 360)
- `do_set_close_at_endpoint` (~line 367)

**Description:**

The `load_election` and `load_election_issue` decorators load any election by EID from the database but never verify that the authenticated user is the election's owner (`owner_pid`) or a member of the election's authorization group (`authz`). These fields exist in the database, are populated during election creation, and are even displayed in UI templates — but are never evaluated for access control. Every management endpoint has an explicit `### check authz` TODO comment confirming the developers intended to add this check but never did.

**Control Definition:**
- `election.owner_pid` — `schema.sql` line 72-73
- `election.authz` — `schema.sql` line 77-79
- `q_metadata` returns both fields — `queries.yaml`

**Where It Should Be Applied (9 endpoints):**

|----------|------|------|-----------|

| `GET /manage-stv/<eid>/<iid>` | `pages.py` | ~330 | ❌ |
| `POST /do-set-open_at/<eid>` | `pages.py` | ~360 | ❌ |
| `POST /do-set-close_at/<eid>` | `pages.py` | ~367 | ❌ |

| `POST /do-edit-issue/<eid>/<iid>` | `pages.py` | ~495 | ❌ |

- Issues can be **added, modified, or deleted** in other users' elections
- Election integrity is fundamentally compromised
- The admin page creates **false confidence** by showing only owned elections, while the API enforces no such restriction

**Remediation:**

Add ownership verification to the `load_election` decorator:

```python
def load_election(func):
    """Decorator to load/pass-argument an Election from EID."""
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            result = await basic_info()
            result.title = 'Unknown Election'
            result.eid = eid
            raise_404(T_BAD_EID, result)

        _LOGGER.debug(f'Loaded: {e}')

        # Verify the authenticated user is authorized to manage this election
        result = await basic_info()
        md = e.get_metadata()
        
        if md.owner_pid != result.uid:
            # Check authz group membership if authz is set
            if md.authz:
                # TODO: Implement authz group check (e.g., LDAP group membership)
                if not is_member_of_authz_group(result.uid, md.authz):
                    _LOGGER.warning(
                        f'User[U:{result.uid}] denied access to election[E:{eid}]'
                        f' (owner: {md.owner_pid}, authz: {md.authz})'
                    )
                    quart.abort(403)
            else:
                _LOGGER.warning(
                    f'User[U:{result.uid}] denied access to election[E:{eid}]'
                    f' (owner: {md.owner_pid})'
                )
                quart.abort(403)

        return await func(e)
    return loader
```

Apply the same pattern to `load_election_issue`. For the `do_vote_endpoint`, use a separate decorator that checks voter eligibility (see Finding ASVS-822-HIG-002).

### ASVS-822-HIG-002: Missing Explicit Voter Eligibility Check on Vote Submission Endpoint

**Severity:** HIGH

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint` (~line 374)
- **Line:** ~378 (`### check authz` comment)

**Related Functions Checked:**
- `vote_on_page` (~line 230) — has proper eligibility check
- `election.add_vote` in `v3/steve/election.py` (~line 210) — has implicit check via crash

**Description:**

The `vote_on_page` (GET) properly verifies voter eligibility via the `mayvote` table before rendering the ballot. However, the `do_vote_endpoint` (POST) — which actually records votes — has a `### check authz` TODO and performs no explicit eligibility check. The underlying `add_vote()` method will crash with an `AttributeError` when accessing `mayvote.salt` on a `None` result, and this crash is caught by a generic `except Exception` handler that returns a vague error message. This is a Type B gap: the authorization data and check mechanism exist but are not applied at the critical entry point.

**Control Definition:**
- `vote_on_page` eligibility check — `pages.py` ~line 230
- `mayvote` table — `schema.sql`
- `q_find_issues` query — `queries.yaml`

**Where Control IS Applied:**
```python
# vote_on_page (GET /vote-on/<eid>) — PROPER CHECK
election.q_find_issues.perform(result.uid, election.eid)
if not election.q_find_issues.fetchall():
    result = await basic_info()
    result.title = 'Access Denied'
    result.eid = election.eid
    raise_404(T_BAD_EID, result)  # Returns 404
```

**Where Control is NOT Applied:**

`do_vote_endpoint` (`pages.py` ~line 374):
```python

@asfquart.auth.require({R.committer})
@load_election

    form = edict(await quart.request.form)

            ...
        try:
            election.add_vote(result.uid, iid, votestring)  # Crashes on unauthorized

            _LOGGER.error(

            )
            await flash_danger(f'Error submitting vote for issue {iid}.')
            return quart.redirect(f'/vote-on/{election.eid}', code=303)
```

**Implicit (Fragile) Check:**

`election.py` `add_vote` (~line 210):
```python

    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    # If mayvote is None → AttributeError on next line (implicit denial)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
```

**Data Flow:**
```
Attacker (committer, not eligible voter) → POST /do-vote/<eid>
  → @asfquart.auth.require({R.committer}) → PASS
  → @load_election → loads election (no ownership check needed for voting)
  → do_vote_endpoint → NO eligibility check
  → election.add_vote(uid, iid, votestring)
  → q_get_mayvote returns None → AttributeError → caught by except Exception
  → Generic error flash: "Error submitting vote for issue {iid}"
```

**Proof of Concept:**
```
# Attacker is a committer who is NOT in the mayvote table for this election
POST /do-vote/a1b2c3d4e5
Cookie: session=<attacker_session>
Content-Type: application/x-www-form-urlencoded
vote-f1a2b3c4d5=yes

→ 303 Redirect with flash "Error submitting vote for issue f1a2b3c4d5."
# Vote is NOT recorded (crashes before INSERT), but:
# 1. No proper 403 response
# 2. Error log doesn't clearly indicate authorization failure
# 3. Information leak: confirms issue ID exists in the election
# 4. If add_vote() is refactored to handle None gracefully, authz breaks entirely
```

**Impact:**
- Currently, unauthorized votes are not recorded (due to accidental crash)
- The implicit authorization is **fragile** — any refactoring of `add_vote()` that handles `None` mayvote could eliminate the only authorization check
- Error logs conflate authorization failures with system errors
- Information disclosure: an attacker can probe which IIDs belong to which elections
- No audit trail distinguishing unauthorized attempts from genuine errors

**Remediation:**

Add explicit eligibility check before processing votes:

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    # Verify voter is eligible for this election
    election.q_find_issues.perform(result.uid, election.eid)
    eligible_issues = {row.iid for row in election.q_find_issues.fetchall()}
    if not eligible_issues:
        _LOGGER.warning(
            f'User[U:{result.uid}] attempted to vote in election[E:{election.eid}]'
            f' without eligibility'
        )
        quart.abort(403)

    form = edict(await quart.request.form)
    ...
    for iid, votestring in votes.items():
        if iid not in eligible_issues:
            _LOGGER.warning(
                f'User[U:{result.uid}] attempted to vote on ineligible'
                f' issue[I:{iid}] in election[E:{election.eid}]'
            )
            await flash_danger(f'You are not eligible to vote on issue {iid}.')
            return quart.redirect(f'/vote-on/{election.eid}', code=303)
        ...
```

Also add an explicit `None` check in `add_vote()`:
```python
def add_vote(self, pid, iid, votestring):
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    if mayvote is None:
        raise VoterNotEligible(pid, iid)  # New explicit exception
    ...
```

### ASVS-822-MED-003: Issue Operations Not Scoped to Parent Election (Cross-Election IDOR)

**Severity:** MEDIUM

**Location:**
- **File:** `v3/queries.yaml`
- **Queries:** `q_get_issue`, `c_edit_issue`, `c_delete_issue`
- **File:** `v3/steve/election.py`
- **Functions:** `get_issue` (~line 145), `edit_issue` (~line 160), `delete_issue` (~line 170)

**Related Functions Checked:**
- `do_edit_issue_endpoint` in `pages.py` (~line 495)
- `do_delete_issue_endpoint` in `pages.py` (~line 515)
- `load_election_issue` decorator in `pages.py` (~line 175)

**Description:**

The `q_get_issue` query retrieves issues by IID alone without verifying the issue belongs to the election specified in the URL. Combined with Finding ASVS-822-CRT-001, this allows cross-election issue manipulation: an attacker can route an issue edit/delete through a different election's URL, bypassing that issue's actual election's state protections (e.g., editing an issue in an open election by routing through an editable election).

**Vulnerable Query (`queries.yaml`):**
```yaml
q_get_issue: SELECT * FROM issue WHERE iid = ?
```

**Vulnerable Edit Command (`queries.yaml`):**
```yaml
c_edit_issue: |
    UPDATE issue
    SET title = ?, description = ?, type = ?, kv = ?
    WHERE iid = ?
```

**Vulnerable Delete Command (`queries.yaml`):**
```yaml
c_delete_issue: DELETE FROM issue WHERE iid = ?
```

**Attack Scenario:**
```
Election A: state=editable, eid=aaaa000001 (attacker's or any committer's)
Election B: state=open,     eid=bbbb000002, has issue iid=cccc000003

# Attacker edits election B's issue through election A's URL
POST /do-edit-issue/aaaa000001/cccc000003
Cookie: session=<attacker_session>
Content-Type: application/x-www-form-urlencoded
title=Tampered+Issue&description=Modified+in+open+election

Flow:
1. load_election_issue loads election A (editable) ← state check passes
2. get_issue(cccc000003) loads issue from election B ← no EID filter
3. election.edit_issue(cccc000003, ...) checks self.is_editable() on election A ← passes
4. c_edit_issue updates issue cccc000003 ← belongs to election B (open)
```

**Vulnerable Code in `load_election_issue` (`pages.py` ~line 175):**
```python
async def loader(eid, iid):
    try:
        e = steve.election.Election(DB_FNAME, eid)  # Loads election by eid
    except steve.election.ElectionNotFound:
        ...
    try:
        i = e.get_issue(iid)  # Loads issue by iid ONLY — no eid filter
    except steve.election.IssueNotFound:
        ...
```

**Vulnerable Code in `election.py` `edit_issue` (~line 160):**
```python
def edit_issue(self, iid, title, description, vtype, kv):
    assert self.is_editable()  # Checks THIS election's state, not the issue's
    self.c_edit_issue.perform(title, description, vtype, self.kv2json(kv), iid)
```

**Data Flow:**
```
Attacker → POST /do-edit-issue/<controlled_eid>/<victim_iid>
  → load_election_issue loads controlled election (state=editable)
  → get_issue(victim_iid) loads issue from different election (no EID check)
  → edit_issue checks controlled election's state (passes)
  → UPDATE issue WHERE iid=victim_iid (no EID filter)
  → Victim's issue in open/closed election is modified
```

**Proof of Concept:**
```
# Prerequisites:
# - Attacker controls election aaaa000001 (state=editable)
# - Victim has issue cccc000003 in election bbbb000002 (state=open)

POST /do-edit-issue/aaaa000001/cccc000003
Cookie: session=<attacker_session>
Content-Type: application/x-www-form-urlencoded
title=Tampered+Title&description=Modified+through+cross-election+IDOR

→ 303 Redirect (issue cccc000003 updated despite belonging to different election)
```

**Impact:**
- Bypasses election state machine: issues in open/closed elections can be edited/deleted by routing through an editable election
- Combined with ASVS-822-CRT-001, any committer can perform this attack
- Practical exploitability limited by 10-hex-char random IIDs (brute-force infeasible), but IIDs may leak via URL sharing, logs, or templates

**Remediation:**

Scope issue queries and operations to their parent election:

```yaml
# queries.yaml
q_get_issue: SELECT * FROM issue WHERE iid = ? AND eid = ?
c_edit_issue: |
    UPDATE issue
    SET title = ?, description = ?, type = ?, kv = ?
    WHERE iid = ? AND eid = ?
c_delete_issue: DELETE FROM issue WHERE iid = ? AND eid = ?
```

```python
# election.py
def get_issue(self, iid):
    issue = self.q_get_issue.first_row(iid, self.eid)  # Add EID filter
    if not issue:
        raise IssueNotFound(iid)
    return (issue.title, issue.description, issue.type, self.json2kv(issue.kv))

def edit_issue(self, iid, title, description, vtype, kv):
    assert self.is_editable()
    assert vtype in vtypes.TYPES
    self.c_edit_issue.perform(title, description, vtype, self.kv2json(kv), iid, self.eid)
    if self.c_edit_issue.rowcount == 0:
        raise IssueNotFound(iid)

def delete_issue(self, iid):
    assert self.is_editable()
    self.c_delete_issue.perform(iid, self.eid)
    if self.c_delete_issue.rowcount == 0:
        raise IssueNotFound(iid)
```

## Positive Security Patterns

1. **Per-document authorization in `serve_doc()`** — Checks `q_get_mayvote(uid, iid)` before serving files. Returns 404 (not 403) to prevent information disclosure.

2. **Session-sourced PID for voting** — `do_vote_endpoint` uses `result.uid` from the authenticated session, preventing horizontal IDOR on vote submission (can't vote as another user).

3. **Data-filtered listing queries** — `open_to_pid()`, `owned_elections()`, `upcoming_to_pid()` all filter by the authenticated user's PID at the database query level, preventing data leakage in listing endpoints.

4. **Parameterized queries throughout** — All SQL uses `?` placeholders via `asfpy.db`, preventing SQL injection as the mechanism for IDOR bypass.

5. **Non-predictable object identifiers** — EIDs and IIDs are 10-hex-character random values generated by `crypto.create_id()`, making enumeration difficult (though not a substitute for proper authorization).

6. **Election state machine enforcement** — `assert self.is_editable()` / `is_open()` / `is_closed()` prevents invalid state transitions in the data layer, even though the web layer doesn't enforce ownership.

7. **Vote token indirection** — Votes are stored with a derived `vote_token` instead of direct PID/IID references, preventing correlation through the vote table.

---

## Architecture Observations

1. **Authorization is split between layers with gaps**: The database has ownership fields (`owner_pid`, `authz`), the admin UI shows only owned elections, but the API layer enforces no ownership. This creates a dangerous illusion of security.

2. **The `### check authz` comments serve as a roadmap**: The developers clearly intended to implement object-level authorization. The comments appear in `load_election` (line ~165), `load_election_issue` (line ~195), `do_vote_endpoint` (line ~378), `do_open_endpoint` (line ~439), `do_close_endpoint` (line ~459), `do_add_issue_endpoint` (line ~477), `do_edit_issue_endpoint` (line ~499), `do_delete_issue_endpoint` (line ~519), and `do_create_endpoint` (line ~419).

3. **Defense in depth is partially present**: The `election.py` state machine (`assert is_editable()`) provides data-layer protection, but this only validates state, not identity. An unauthorized user who knows an EID can perform any state-valid operation.

4. **The `authz` field is dead code for authorization**: Stored, queried, displayed, but never evaluated. This field was designed to support group-based management authorization but remains unimplemented.

## Recommendations Summary

### Immediate Actions (Priority 1 — CRITICAL)

**Implement ownership/authz verification in `load_election` and `load_election_issue` decorators.**

This is the single highest-impact fix, closing 9 BOLA vulnerabilities at once. The authorization data already exists in the database — it just needs to be checked. This addresses **ASVS-822-CRT-001**.

**Implementation Steps:**
1. Add ownership check in `load_election` decorator comparing `election.owner_pid` to `result.uid`
2. Implement `authz` group membership check (LDAP or equivalent)
3. Return HTTP 403 for unauthorized access attempts
4. Add security logging for all authorization failures
5. Apply same pattern to `load_election_issue` decorator

**Estimated Effort:** 4-8 hours  
**Risk if Delayed:** Any authenticated committer can manipulate any election

### Short-term Actions (Priority 2 — HIGH)

**Add explicit voter eligibility check to `do_vote_endpoint`.**

Replicate the `q_find_issues` check from `vote_on_page` into the submission handler. Add a `None` guard in `add_vote()` with a descriptive exception. This addresses **ASVS-822-HIG-002**.

**Implementation Steps:**
1. Add `q_find_issues` eligibility check at start of `do_vote_endpoint`
2. Create `VoterNotEligible` exception class
3. Add explicit `None` check in `add_vote()` method
4. Return HTTP 403 for ineligible voters
5. Add security logging for unauthorized vote attempts

**Estimated Effort:** 2-4 hours  
**Risk if Delayed:** Fragile implicit authorization may break during refactoring

### Medium-term Actions (Priority 3 — MEDIUM)

**Scope issue queries and commands to their parent election.**

Add `AND eid = ?` to `q_get_issue`, `c_edit_issue`, and `c_delete_issue`. This prevents cross-election manipulation and strengthens the state machine enforcement. This addresses **ASVS-822-MED-003**.

**Implementation Steps:**
1. Update `q_get_issue`, `c_edit_issue`, `c_delete_issue` queries to include EID parameter
2. Modify `get_issue()`, `edit_issue()`, `delete_issue()` to pass `self.eid`
3. Add rowcount checks to detect cross-election attempts
4. Add test cases for cross-election IDOR scenarios

**Estimated Effort:** 2-3 hours  
**Risk if Delayed:** State machine bypass possible via cross-election routing

### Long-term Improvements (Priority 4)

**Centralize authorization in reusable functions or middleware.**

Rather than checking in each handler, create `verify_election_management_access(election, uid)` and `verify_voter_eligibility(election, uid)` functions that are called from the decorators, ensuring consistent enforcement.

**Implementation Steps:**
1. Create centralized authorization module
2. Implement `verify_election_management_access()` function
3. Implement `verify_voter_eligibility()` function
4. Refactor decorators to use centralized functions
5. Add comprehensive unit tests for authorization logic
6. Document authorization architecture and decision matrix

**Estimated Effort:** 8-16 hours  
**Benefits:** Consistent enforcement, easier auditing, reduced code duplication
