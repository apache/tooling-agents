### Summary

This audit evaluates the election management system against ASVS 8.4.1, treating each **election as a tenant**. The system has fundamental cross-tenant isolation failures: authorization controls are **defined** (schema fields `owner_pid`, `authz`) but **never enforced** at any endpoint, and critical issue-level operations use queries without tenant (election) scoping, allowing cross-election data manipulation.

## Findings

### ASVS-841-CRI-001: Missing Election Ownership Authorization on All Management Endpoints (Type B Gap)

**Severity:** CRITICAL

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `load_election` decorator (line ~157)
- **Affected Endpoints:** Lines 268, 310, 317, 324, 391, 406, 420, 441, 457

**Related Functions Checked:**
- `load_election()` (pages.py line 157)
- `basic_info()` (pages.py)
- All management endpoints using `@load_election` decorator

**Control Exists But Not Called:**
- `election.owner_pid` is stored for every election (`schema.sql` line 57)
- `election.authz` field designed for group-level authorization (`schema.sql` line 62)
- Neither is ever checked against the authenticated user at any endpoint

**Description:**

The system defines election ownership and authorization controls in the database schema but never enforces them at the application layer. Every management endpoint has a placeholder comment `### check authz` indicating this was planned but never implemented. Any authenticated committer can perform privileged operations on any election regardless of ownership.

**Vulnerable Code — `load_election` decorator (pages.py lines 157-172):**

```python
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            ...
        _LOGGER.debug(f'Loaded: {e}')
        ### check authz          # <-- PLACEHOLDER, NEVER IMPLEMENTED
        return await func(e)
    return loader
```

**Affected Endpoints:**

| Endpoint | Line | Impact |
|----------|------|--------|
| `GET /do-open/<eid>` | 391 | Any committer can **open** any election |
| `GET /do-close/<eid>` | 406 | Any committer can **close** any election |
| `POST /do-add-issue/<eid>` | 420 | Any committer can **add issues** to any election |
| `POST /do-edit-issue/<eid>/<iid>` | 441 | Any committer can **edit issues** in any election |
| `POST /do-delete-issue/<eid>/<iid>` | 457 | Any committer can **delete issues** from any election |
| `POST /do-set-open_at/<eid>` | 310 | Any committer can **set dates** on any election |
| `POST /do-set-close_at/<eid>` | 317 | Any committer can **set dates** on any election |
| `GET /manage/<eid>` | 268 | Any committer can **view** any election's management page |
| `POST /do-vote/<eid>` | 324 | Any committer can **attempt votes** in any election |

**Data Flow:**

```
Authenticated committer → GET /do-close/<any_eid> → load_election(eid) 
→ [NO owner_pid check] → [NO authz check] → election.close() → Election closed
```

**Proof of Concept:**

```http
GET /do-close/a1b2c3d4e5 HTTP/1.1
Host: voting.apache.org
Cookie: session=<mallory_session>

# User "mallory" (a committer but NOT election owner) closes election owned by "alice"
# Result: Election a1b2c3d4e5 is permanently closed, votes can no longer be cast
```

**Impact:**

Any authenticated committer can open, close, or modify ANY election in the system, regardless of ownership. This violates the fundamental tenant isolation requirement: election owners lose control over their elections. An attacker can:
- Prematurely close active elections, preventing legitimate voters from participating
- Reopen closed elections, allowing vote manipulation after results are announced
- Modify election metadata and scheduling
- Access sensitive election management data

**Remediation:**

```python
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            ...
        
        result = await basic_info()
        md = e.get_metadata()
        
        # Enforce ownership or authz group membership
        if md.owner_pid != result.uid:
            if not md.authz or not user_in_authz_group(result.uid, md.authz):
                quart.abort(403, 'Not authorized to manage this election')
        
        return await func(e)
    return loader
```

### ASVS-841-CRI-002: Cross-Election Issue Data Access and Modification via Unscoped Queries

**Severity:** CRITICAL

**Location:**
- **File:** `v3/queries.yaml` — `q_get_issue`, `c_edit_issue`, `c_delete_issue`
- **File:** `v3/steve/election.py` — `get_issue()` (line ~151), `edit_issue()` (line ~161), `delete_issue()` (line ~171)
- **File:** `v3/server/pages.py` — `load_election_issue()` (line ~175)

**Related Functions Checked:**
- `load_election_issue()` decorator (pages.py line 175)
- `get_issue()` (election.py line 151)
- `edit_issue()` (election.py line 161)
- `delete_issue()` (election.py line 171)

**Description:**

Issue-level queries filter only by `iid` without constraining to the parent election's `eid`. Combined with the `load_election_issue` decorator not validating issue-election affiliation, operations on Election A can read/modify/delete issues belonging to Election B. This allows an attacker to bypass election state restrictions by routing operations through an editable election.

**Vulnerable Queries (`queries.yaml`):**

```yaml
q_get_issue: SELECT * FROM issue WHERE iid = ?          # No EID filter
c_edit_issue: |
    UPDATE issue
    SET title = ?, description = ?, type = ?, kv = ?
    WHERE iid = ?                                         # No EID filter
c_delete_issue: DELETE FROM issue WHERE iid = ?           # No EID filter
```

**Vulnerable Code — `load_election_issue` (pages.py lines 175-200):**

```python
def load_election_issue(func):
    @functools.wraps(func)
    async def loader(eid, iid):
        e = steve.election.Election(DB_FNAME, eid)  # Loads election E1
        # ...
        i = e.get_issue(iid)  # Loads issue by IID only — could be from ANY election
        # MISSING: verify issue.eid == e.eid
        issue = edict(iid=iid, title=i[0], description=i[1], vtype=i[2], kv=i[3])
        return await func(e, issue)
```

**Vulnerable Code — `get_issue` (election.py lines 151-157):**

```python
def get_issue(self, iid):
    issue = self.q_get_issue.first_row(iid)  # SELECT * FROM issue WHERE iid = ?
    if not issue:
        raise IssueNotFound(iid)
    # issue.eid is available but NEVER compared to self.eid
    return (issue.title, issue.description, issue.type, self.json2kv(issue.kv))
```

**Vulnerable Code — `edit_issue` (election.py lines 161-170):**

```python
def edit_issue(self, iid, title, description, vtype, kv):
    assert self.is_editable()  # Checks THIS election's state, not the issue's election
    assert vtype in vtypes.TYPES
    self.c_edit_issue.perform(title, description, vtype, self.kv2json(kv), iid)
    # UPDATE issue SET ... WHERE iid = ?  — affects issue regardless of election
```

**Data Flow:**

```
Attacker → POST /do-edit-issue/<editable_eid>/<target_iid_from_other_election>
→ load_election(editable_eid) → loads Election A (editable state) → [no authz check]
→ load_election_issue: e.get_issue(target_iid) → SELECT * FROM issue WHERE iid = target_iid
  → Returns issue from Election B (different election!)
→ edit_issue(target_iid, ...) → assert self.is_editable() checks Election A, passes
→ UPDATE issue SET ... WHERE iid = target_iid → Modifies Election B's issue
```

**Proof of Concept:**

```http
POST /do-edit-issue/aaaa000001/bbbb000002 HTTP/1.1
Host: voting.apache.org
Cookie: session=<attacker_session>
Content-Type: application/x-www-form-urlencoded

title=TAMPERED+ISSUE&description=Changed+during+active+voting

# Attacker owns editable election E1 (eid=aaaa000001)
# Victim has open election E2 with issue I2 (iid=bbbb000002)
# E2 is OPEN for voting — normally issues cannot be edited
# Result: Issue bbbb000002 in election E2 is modified, even though E2 is open
# The is_editable() assertion passes because it checks E1, not E2
```

**Impact:**

- **Read** issue titles, descriptions, and vote configurations from other elections (information disclosure)
- **Edit** issues in open/closed elections by routing through an editable election (bypasses state machine)
- **Delete** issues from other elections, destroying voting data and election integrity
- Tamper with active elections during voting periods
- Election integrity is completely undermined across the entire system

**Remediation:**

In `queries.yaml`:

```yaml
q_get_issue: SELECT * FROM issue WHERE iid = ? AND eid = ?
c_edit_issue: |
    UPDATE issue SET title = ?, description = ?, type = ?, kv = ?
    WHERE iid = ? AND eid = ?
c_delete_issue: DELETE FROM issue WHERE iid = ? AND eid = ?
```

In `election.py`:

```python
def get_issue(self, iid):
    issue = self.q_get_issue.first_row(iid, self.eid)  # Scoped to this election
    if not issue:
        raise IssueNotFound(iid)
    return (issue.title, issue.description, issue.type, self.json2kv(issue.kv))

def edit_issue(self, iid, title, description, vtype, kv):
    assert self.is_editable()
    assert vtype in vtypes.TYPES
    self.c_edit_issue.perform(title, description, vtype, self.kv2json(kv), iid, self.eid)
    if self.c_edit_issue.rowcount == 0:
        raise IssueNotFound(iid)

def delete_issue(self, iid):
    assert self.is_editable()
    self.c_delete_issue.perform(iid, self.eid)
    if self.c_delete_issue.rowcount == 0:
        raise IssueNotFound(iid)
```

### ASVS-841-HIG-003: Vote Submission Endpoint Missing Explicit Voter Eligibility Check

**Severity:** HIGH

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint` (line ~324)
- **Lines:** 324-369

**Related Functions Checked:**
- `do_vote_endpoint()` (pages.py line 324)
- `vote_on_page()` (pages.py line 208)
- `add_vote()` (election.py line 213)

**Description:**

The `vote_on_page` (GET, line ~208) explicitly checks voter eligibility via `q_find_issues`, but the `do_vote_endpoint` (POST, line ~324) does NOT repeat this check. The eligibility check only succeeds incidentally when `add_vote` raises `AttributeError` on `None.salt`. This implicit protection is fragile and could be broken by code refactoring.

**Vulnerable Code — `do_vote_endpoint` (pages.py lines 324-369):**

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    ### check authz                           # <-- PLACEHOLDER
    form = edict(await quart.request.form)
    # ...
    issues = election.list_issues()           # Lists ALL issues for election (no voter filter)
    issue_dict = {i.iid: i for i in issues}
    
    for iid, votestring in votes.items():
        if iid not in issue_dict:             # Only checks issue exists in election
            ...
        try:
            election.add_vote(result.uid, iid, votestring)  # Relies on implicit failure
```

**Contrasted with GET page (vote_on_page, line ~226):**

```python
# Check if user can vote in this election (has mayvote entries)
election.q_find_issues.perform(result.uid, election.eid)
if not election.q_find_issues.fetchall():
    result = await basic_info()
    result.title = 'Access Denied'
    raise_404(T_BAD_EID, result)             # Explicit rejection
```

**Implicit Failure in `add_vote` (election.py line ~213):**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    # If mayvote is None (ineligible voter), next line raises AttributeError
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
```

**Data Flow:**

```
Ineligible committer → POST /do-vote/<eid> → [no eligibility check]
→ add_vote(uid, iid, votestring) → q_get_mayvote returns None
→ mayvote.salt → AttributeError → caught by generic except → flash "Error submitting vote"
```

**Proof of Concept:**

```http
POST /do-vote/a1b2c3d4e5 HTTP/1.1
Host: voting.apache.org
Cookie: session=<eve_session>
Content-Type: application/x-www-form-urlencoded

vote-f1f2f3f4f5=yes

# Committer "eve" is NOT in mayvote for election E1 but IS authenticated
# Result: AttributeError caught, generic error message returned
# Vote NOT recorded (implicit protection), but no explicit 403
```

**Impact:**

While the vote is NOT recorded (the implicit failure prevents data corruption), the absence of an explicit authorization check means:
- Ineligible users receive a misleading error message instead of a clear 403 Forbidden
- The authorization boundary is enforced by an accidental `AttributeError`, not by design
- If the `add_vote` code is refactored to handle `None` mayvote gracefully, the implicit protection disappears
- Information leakage: the generic error vs. "Invalid issue ID" response lets an attacker distinguish between "issue exists but I can't vote" vs. "issue doesn't exist"
- Violates security-by-design principles by relying on implementation accidents

**Remediation:**

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    
    # Explicit voter eligibility check (matching GET endpoint)
    election.q_find_issues.perform(result.uid, election.eid)
    eligible_issues = {row.iid for row in election.q_find_issues.fetchall()}
    if not eligible_issues:
        quart.abort(403, 'Not authorized to vote in this election')
    
    form = edict(await quart.request.form)
    votes = {}
    for key, value in form.items():
        if key.startswith('vote-'):
            iid = key.split('-', 1)[1]
            if iid not in eligible_issues:
                quart.abort(403, f'Not authorized to vote on issue {iid}')
            votes[iid] = value
    # ... proceed with add_vote
```

## Positive Security Patterns

1. **Parameterized Queries Throughout:** All SQL queries in `queries.yaml` use `?` placeholders, preventing SQL injection.

2. **Election State Machine:** The `is_editable()`/`is_open()`/`is_closed()` state checks prevent illogical operations (e.g., voting on closed elections).

3. **Tamper Detection:** `is_tampered()` using `opened_key` Argon2 hash validates election data integrity.

4. **Per-Voter Salts:** Unique salts in `mayvote` table prevent vote correlation across voters.

5. **Vote Token Architecture:** The `vote_token` design (derived from `opened_key + pid + iid + salt`) cryptographically ties votes to specific person-issue pairs.

6. **Metadata Sanitization:** `get_metadata()` explicitly excludes `salt` and `opened_key` from returned data.

7. **Document Serving Authorization:** `serve_doc` checks `q_get_mayvote` before serving per-issue documents.

8. **EID Scoping Where Applied:** `list_issues()`, `add_issue()`, `q_find_issues` correctly scope by EID.

9. **Unpredictable IDs:** 10-character hex IDs generated via `crypto.create_id()` prevent enumeration.

10. **Database Trigger Protection:** `prevent_open_close_update` trigger prevents timestamp modification after election closure.

---

## Architecture Observations

1. **Multi-Tenancy Model:** Elections serve as tenants. The schema supports ownership (`owner_pid`) and group authorization (`authz`), but the application layer never enforces these boundaries.

2. **Authentication vs. Authorization Gap:** The system has robust authentication (`asfquart.auth.require`) but virtually no authorization beyond the authentication level. The `### check authz` comments appear at 12+ locations indicating this was planned but never implemented.

3. **State Machine as Sole Guard:** The election state machine (`editable → open → closed`) currently serves as the only authorization-like control. However, it only prevents logically invalid operations (e.g., editing an open election), not unauthorized operations (e.g., a non-owner closing someone else's election).

4. **Inconsistent Scoping Pattern:** Some queries are properly scoped by EID (`q_issues`, `c_add_issue`, `q_find_issues`) while others are not (`q_get_issue`, `c_edit_issue`, `c_delete_issue`). This inconsistency suggests the scoping was applied ad-hoc rather than as a systematic design pattern.

## Recommendations Summary

### Immediate Actions (P0)

1. **Implement election ownership verification** in `load_election` and `load_election_issue` decorators, checking `owner_pid` and `authz` fields against authenticated user.
   - **Addresses:** ASVS-841-CRI-001
   - **Effort:** Medium
   - **Impact:** Prevents unauthorized cross-election management operations

2. **Add election scoping to issue queries** by adding `AND eid = ?` to `q_get_issue`, `c_edit_issue`, `c_delete_issue` queries and passing `self.eid` in all calls.
   - **Addresses:** ASVS-841-CRI-002
   - **Effort:** Low
   - **Impact:** Prevents cross-election issue manipulation

### Short-term Actions (P1)

3. **Add explicit voter eligibility check** to `do_vote_endpoint` before processing votes, matching the check in `vote_on_page`.
   - **Addresses:** ASVS-841-HIG-003
   - **Effort:** Low
   - **Impact:** Ensures authorization is explicit and maintainable

4. **Define and implement `authz` field semantics** including LDAP group lookup and role mapping for shared election management.
   - **Addresses:** ASVS-841-CRI-001
   - **Effort:** High
   - **Impact:** Enables proper multi-owner election management

### Long-term Actions (P2)

5. **Add election scoping to tally queries** by adding `AND eid = ?` to `q_tally` query to prevent cross-election tally leakage.
   - **Addresses:** Defense-in-depth
   - **Effort:** Low
   - **Impact:** Additional protection layer

6. **Create centralized authorization middleware** that checks election access based on operation type (manage vs. vote).
   - **Addresses:** Architecture improvement
   - **Effort:** High
   - **Impact:** Systematic authorization enforcement across all endpoints
