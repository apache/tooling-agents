### Key Findings

The audit reveals **critical authorization gaps** where the system authenticates users by role (committer/pmc_member) but fails to enforce object-level permissions based on the originating subject's relationship to specific resources. The `owner_pid` field exists in the database schema, the `owned_elections` query exists to filter by owner, and `### check authz` placeholder comments appear throughout the code — but **actual authorization enforcement was never implemented** for election management operations. This is a textbook Type B gap (control EXISTS but NOT CALLED), creating false confidence in the authorization model.

### Critical Severity

#### ASVS-8033-CRT-001 — Missing Object-Level Authorization on Election Management Endpoints

**Severity:** CRITICAL  
**Gap Type:** Type B — Control EXISTS (owner_pid, authz fields, owned_elections query) but NOT CALLED  
**ASVS Requirement:** 8.3.3 — Access based on originating subject's permissions  
**Location:** `v3/server/pages.py` (lines 170-216, 326-345, 348-376, 378-387, 453-470, 473-490, 491-515, 517-539, 541-559), `v3/steve/election.py` (lines 157-170), `v3/queries.yaml`

**Related Functions Checked:**
- `load_election` decorator (lines 170-190)
- `do_open_endpoint` (lines 453-470)
- `do_close_endpoint` (lines 473-490)
- `do_add_issue_endpoint` (lines 491-515)
- `do_edit_issue_endpoint` (lines 517-539)
- `do_delete_issue_endpoint` (lines 541-559)
- `_set_election_date` (lines 96-121)
- `manage_page` (lines 326-345)
- `manage_stv_page` (lines 348-376)

**Description:**

All election management endpoints determine access using only the authenticated user's role (`R.committer`) rather than verifying the originating subject has a specific permission relationship (owner) with the target election object. The `owner_pid` field is stored in the `election` table and used for display purposes, but is **never checked** before allowing state-changing operations.

Ten `### check authz` placeholder comments confirm this authorization was planned but never implemented, creating a false sense of security.

**Vulnerable Code — `load_election` decorator (lines 170-190):**

```python

            result.title = 'Unknown Election'
            result.eid = eid
```


**Proof of Concept:**

Attacker (committer user `mallory`) knows or discovers an election EID belonging to user `alice`:

```http
# Mallory opens Alice's election prematurely
GET /do-open/alice-election-eid-12345 HTTP/1.1
Cookie: session=mallory_session_token

# Mallory closes Alice's election early to suppress votes
GET /do-close/alice-election-eid-12345 HTTP/1.1
Cookie: session=mallory_session_token

# Mallory adds a fraudulent issue to Alice's election
POST /do-add-issue/alice-election-eid-12345 HTTP/1.1
```

**Impact:**

- **Election integrity compromise**: Any committer can open, close, or modify any election regardless of ownership
- **Privilege escalation**: Committer-level users gain election-owner-level privileges on all elections
- **Tampering**: Attackers can add, edit, or delete issues in elections they don't own
- **Denial of service**: Premature opening/closing of elections disrupts voting processes
- The tamper detection mechanism (`is_tampered`) would detect post-open modifications, but pre-open manipulation (adding issues, setting dates) would be incorporated into the opened_key hash

**All Affected Endpoints:**

|----------|-----------|--------|

| `GET /do-close/<eid>` | Close election | Critical — ends voting prematurely |

| `POST /do-edit-issue/<eid>/<iid>` | Modify issue | Critical — alters ballot content |

**Remediation:**

Add owner authorization check in the `load_election` decorator (central enforcement point):

```python
def load_election(func, require_owner=False):
    """Decorator to load an Election from EID with authorization."""

    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            result = await basic_info()
            result.title = 'Unknown Election'
            result.eid = eid
            raise_404(T_BAD_EID, result)

        _LOGGER.debug(f'Loaded: {e}')

        # Enforce object-level authorization
        if require_owner:
            s = await asfquart.session.read()
            md = e.get_metadata()
            if not s or s['uid'] != md.owner_pid:
                _LOGGER.warning(
                    f'User[U:{s and s["uid"]}] denied access '
                    f'to election[E:{eid}] (owner: {md.owner_pid})'
                )
                quart.abort(403)

        return await func(e)

    return loader


# Usage: Create separate decorators for view vs manage
def load_election_for_management(func):
    return load_election(func, require_owner=True)
```

Then apply to management endpoints:

```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election_for_management    # <-- Enforces owner check
async def do_open_endpoint(election):
    ...
```

### High Severity

#### ASVS-8033-HGH-002 — Missing Explicit Voter Eligibility Check on Vote Submission Endpoint

**Severity:** HIGH  
**Gap Type:** Type B/D — Eligibility check exists for GET but not for corresponding POST; data-layer exception serves as unintended guard  
**ASVS Requirement:** 8.3.3 — Access based on originating subject's permissions  
**Location:** `v3/server/pages.py` (lines 390-427), `v3/steve/election.py` (lines 254-268)

**Related Functions Checked:**
- `do_vote_endpoint` (lines 390-427)
- `vote_on_page` (lines 253-258)
- `add_vote` in `election.py` (lines 254-268)

**Description:**

The `vote_on_page` (GET) endpoint explicitly verifies voter eligibility by checking the `mayvote` table before rendering the voting form (lines 254-258). However, the corresponding `do_vote_endpoint` (POST) endpoint that actually processes and stores votes has a `### check authz` placeholder but **no explicit eligibility verification**. The data layer (`add_vote`) provides an implicit guard — if the user lacks a `mayvote` entry, `q_get_mayvote.first_row()` returns `None`, and subsequent `mayvote.salt` access raises an `AttributeError` — but this is caught by a generic exception handler that returns a misleading error message and a 303 redirect rather than a proper 403 Forbidden.

**Vulnerable Code — `do_vote_endpoint` (lines 390-427):**

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    ### check authz                    # <-- PLACEHOLDER, NEVER IMPLEMENTED

    form = edict(await quart.request.form)
    ...
    for iid, votestring in votes.items():
        if iid not in issue_dict:
            ...
        try:
            election.add_vote(result.uid, iid, votestring)   # <-- No pre-check
        except Exception as e:                                # <-- Generic catch
            _LOGGER.error(...)
            await flash_danger(f'Error submitting vote for issue {iid}.')  # Misleading
            return quart.redirect(f'/vote-on/{election.eid}', code=303)
```

**Compare with the GET handler (lines 253-258) that DOES check:**

```python
async def vote_on_page(election):
    result = await basic_info()
    ...
    # Check if user can vote in this election (has mayvote entries)
    election.q_find_issues.perform(result.uid, election.eid)
    if not election.q_find_issues.fetchall():
        ...
        raise_404(T_BAD_EID, result)  # Proper denial
```

**Data Flow:**

```
Attacker (committer but NOT in mayvote for this election)
  → POST /do-vote/<eid>
  → @asfquart.auth.require({R.committer})   ← PASSES
  → load_election(eid)                       ← LOADS (no authz check)
  → ### check authz                          ← SKIPPED
  → election.add_vote(attacker_uid, iid, vote)
    → q_get_mayvote.first_row(attacker_uid, iid) → None
    → None.salt → AttributeError             ← IMPLICIT DENIAL (not authorization)
  → except Exception: flash_danger(...)      ← Generic error, no audit trail
```

**Proof of Concept:**

```http
POST /do-vote/some-election-eid HTTP/1.1
Cookie: session=unauthorized_committer_session
Content-Type: application/x-www-form-urlencoded

vote-issue123=yes
```

Result: The vote is NOT stored (correct outcome), but the failure is an unhandled `AttributeError` caught generically, logged as "Error adding vote" without indicating it was an authorization failure.

**Impact:**

- While the data layer prevents unauthorized vote storage, the authorization failure is:
  - Not logged as a security event (logged as a generic "error")
  - Not returned as a proper 403 response
  - Susceptible to future regression if `add_vote` is refactored to handle `None` gracefully
- The inconsistency between GET (explicit check) and POST (implicit via exception) violates defense-in-depth
- If the data layer is modified in the future, the implicit guard could silently disappear

**Remediation:**

Add explicit eligibility verification to the POST handler, matching the GET handler's check:

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    # Verify voter eligibility before processing any votes
    election.q_find_issues.perform(result.uid, election.eid)
    eligible_issues = election.q_find_issues.fetchall()
    if not eligible_issues:
        _LOGGER.warning(
            f'Unauthorized vote attempt by User[U:{result.uid}] '
            f'on election[E:{election.eid}]'
        )
        quart.abort(403)

    eligible_iids = {row.iid for row in eligible_issues}

    form = edict(await quart.request.form)
    ...
    for iid, votestring in votes.items():
        if iid not in eligible_iids:
            _LOGGER.warning(
                f'User[U:{result.uid}] attempted to vote on '
                f'unauthorized issue[I:{iid}]'
            )
            quart.abort(403)
        ...
```

#### ASVS-8033-HGH-003 — `load_election_issue` Decorator Missing Authorization Allowing Cross-Election Issue Access

**Severity:** HIGH  
**Gap Type:** Type B — Authorization placeholder exists but never implemented  
**ASVS Requirement:** 8.3.3 — Access based on originating subject's permissions  
**Location:** `v3/server/pages.py` (lines 193-221)

**Related Functions Checked:**
- `load_election_issue` decorator (lines 193-221)
- `get_issue` in `election.py`
- `q_get_issue` in `queries.yaml`

**Description:**

The `load_election_issue` decorator loads an election and then separately loads an issue by IID, but performs **no authorization check** to verify:
1. The authenticated user owns or has permission to manage the election
2. The requested issue actually belongs to the specified election

The issue is loaded via `e.get_issue(iid)` which queries `SELECT * FROM issue WHERE iid = ?` — this query does **not** filter by `eid`. A malicious user could supply an `iid` belonging to a different election, and the decorator would load it without verifying the relationship.

**Vulnerable Code (lines 193-221):**

```python
def load_election_issue(func):
    @functools.wraps(func)
    async def loader(eid, iid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            ...

        ### check authz           # <-- NEVER IMPLEMENTED

        try:
            i = e.get_issue(iid)   # <-- Queries by IID only, no EID filter
        except steve.election.IssueNotFound:
            ...

        issue = edict(iid=iid, title=i[0], description=i[1], vtype=i[2], kv=i[3])
        return await func(e, issue)
    return loader
```

**Supporting query in `queries.yaml`:**

```yaml
q_get_issue: SELECT * FROM issue WHERE iid = ?    # No EID filter
```

**Data Flow:**

```
Attacker (any committer)
  → POST /do-edit-issue/<attacker_eid>/<victim_iid_from_other_election>
  → load_election_issue loads attacker's election (EID check passes)
  → get_issue(victim_iid) loads issue from DIFFERENT election
  → edit_issue modifies victim's issue content
```

**Proof of Concept:**

```http
# Edit an issue belonging to election-B while specifying election-A in the URL
POST /do-edit-issue/election-a-eid/issue-from-election-b HTTP/1.1
Cookie: session=attacker_session
Content-Type: application/x-www-form-urlencoded

title=Tampered+Title&description=Tampered+Description
```

**Impact:**

- An attacker could edit or delete issues from elections they don't own by specifying a different election's EID in the URL path
- Combined with Finding ASVS-8033-CRT-001, this means any committer can modify any issue in any election

**Remediation:**

Update `q_get_issue` to filter by both IID and EID, and pass the EID to `get_issue`:

```yaml
# queries.yaml
q_get_issue: SELECT * FROM issue WHERE iid = ? AND eid = ?
```

```python
# election.py
def get_issue(self, iid):
    issue = self.q_get_issue.first_row(iid, self.eid)  # Add EID filter
    if not issue:
        raise IssueNotFound(iid)
    return (issue.title, issue.description, issue.type, self.json2kv(issue.kv))
```

## Positive Security Patterns

1. **Parameterized Queries**: All SQL queries in `queries.yaml` use `?` placeholders, preventing SQL injection

2. **Voter Eligibility on View**: `vote_on_page` (GET) properly checks `mayvote` before showing the voting form (lines 254-258)

3. **Document Access Control**: `serve_doc` properly checks `mayvote` before serving per-issue documents (lines 593-596)

4. **Cryptographic Vote Tokens**: `add_vote` uses per-person, per-issue tokens derived from `opened_key`, `pid`, `iid`, and `salt` — ensuring votes are cryptographically bound to the authorized voter

5. **Tamper Detection**: `is_tampered()` computes election data hash to detect post-open modifications

6. **Role Escalation for Creation**: `do_create_endpoint` requires `R.pmc_member` (higher privilege than `R.committer`)

7. **Session-Based Identity**: All endpoints that use `result.uid` derive it from the authenticated session, not from user-supplied input

---

## Architecture Observations

1. **Missing Authorization Layer**: The system has authentication (who are you?) but lacks object-level authorization (what are you allowed to do to this specific resource?). The `owner_pid` and `authz` fields represent an authorization model that was designed but never enforced.

2. **Authorization Debt is Documented**: The 10+ `### check authz` comments and `### need general solution` annotations indicate the developers are aware of this gap. This represents known technical debt rather than an oversight, but it remains a critical vulnerability.

3. **Single-Tier Authorization**: ASVS 8.3.3 specifically addresses multi-service architectures, but the principle applies here: the web layer uses `result.uid` (the subject's identity) but only checks role membership, not the subject's specific permissions for the target object. The authorization decision should use `result.uid == election.owner_pid` for management operations.

4. **Implicit vs Explicit Guards**: The pattern of relying on data-layer exceptions (e.g., `None.salt` in `add_vote`) rather than explicit authorization checks creates fragile security boundaries that could be broken by routine refactoring.

## Recommendations Summary

### Immediate Actions (P0 - Critical)

1. **Implement Owner Authorization Check** (ASVS-8033-CRT-001)
   - Add owner authorization check to `load_election` decorator
   - Create `load_election_for_management` variant that enforces ownership
   - Apply to all 9 management endpoints
   - **Effort:** Medium — centralized fix covers multiple endpoints
   - **Impact:** Prevents any committer from managing elections they don't own

### Short-Term Actions (P1-P2 - High Priority)

2. **Fix Cross-Election Issue Access** (ASVS-8033-HGH-003)
   - Add EID filter to `q_get_issue` query
   - Update `get_issue` method to validate issue belongs to election
   - **Effort:** Low — query change + parameter addition
   - **Impact:** Prevents cross-election issue manipulation

3. **Add Explicit Voter Eligibility Check** (ASVS-8033-HGH-002)
   - Replicate `vote_on_page` eligibility check in `do_vote_endpoint`
   - Return proper 403 responses for unauthorized vote attempts
   - **Effort:** Low — replicate existing check from GET handler
   - **Impact:** Provides explicit authorization boundary for vote submission

### Long-Term Actions (P3-P4)

4. **Replace Assert-Based State Validation**
   - Replace `assert` statements with proper exceptions that cannot be disabled with Python `-O` flag
   - Audit all state validation in `election.py`
   - **Effort:** Medium — requires review of all assertion-based checks
   - **Impact:** Ensures state validation cannot be bypassed in production

5. **Enhance Audit Logging**
   - Log authorization failures as security events distinct from application errors
   - Add structured logging for denied access attempts
   - Include user ID, resource ID, and attempted action
   - **Effort:** Low — add logging to authorization check points
   - **Impact:** Improves security monitoring and incident response
