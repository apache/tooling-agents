## Findings

### ASVS-831-CRIT-001 — Missing Object-Level Authorization on All Election Management Endpoints

**Severity:** CRITICAL  
**Gap Type:** Type B (Control EXISTS but NOT CALLED)

**Location:**
- **File:** `v3/server/pages.py`
- **Affected Functions & Lines:**
  - `load_election` decorator (line ~161): `### check authz` — never implemented
  - `load_election_issue` decorator (line ~187): `### check authz` — never implemented
  - `do_open_endpoint` (line ~445): `### check authz`
  - `do_close_endpoint` (line ~465): `### check authz`
  - `do_add_issue_endpoint` (line ~483): `### check authz`
  - `do_edit_issue_endpoint` (line ~507): `### check authz`
  - `do_delete_issue_endpoint` (line ~530): `### check authz`
  - `_set_election_date` (line ~83): `### check authz`

**Related Functions Checked:**
- `Election.get_metadata()` — confirms `owner_pid` field exists and is populated
- `Election.owned_elections()` — confirms ownership filtering is implemented for UI display
- All management endpoint handlers — none perform ownership verification

**Description:**

The application implements authentication via `@asfquart.auth.require({R.committer})` but completely lacks object-level authorization for election management operations. While the system tracks election ownership via the `owner_pid` field in metadata and filters the admin UI to show only owned elections, the underlying API endpoints accept any valid election ID without verifying the current user is the owner.

The codebase contains 10 instances of `### check authz` comments marking locations where authorization checks were planned but never implemented. The `load_election` decorator (the primary entry point for all management operations) explicitly includes this TODO comment at the exact location where ownership verification should occur.

**Vulnerable Code:**

```python
# v3/server/pages.py, line ~161
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            # ... 404 handling ...

        _LOGGER.debug(f'Loaded: {e}')

        ### check authz    # <-- PLANNED BUT NEVER IMPLEMENTED

        return await func(e)
    return loader
```

**The Control That EXISTS But Is NOT Called:**

```python
# v3/steve/election.py, line ~255
def get_metadata(self):
    md = self._all_metadata()
    return edict(
        ...
        owner_pid=md.owner_pid,  # <-- Ownership data EXISTS
        ...
    )

# v3/server/pages.py, line ~285 (admin_page)
owned = steve.election.Election.owned_elections(DB_FNAME, result.uid)
# Only shows user's own elections in the UI — but API has no such filter
```

**Data Flow:**

```
Source: URL path parameter <eid> (user-controllable)
  ↓
Load: Election loaded from database (any valid EID succeeds)
  ↓
Missing Control: No check that session.uid == election.owner_pid
  ↓
Sink: State-changing operations execute for any authenticated committer
```

**Proof of Concept:**

Alice (PMC member) creates election `abc123`. Bob (committer, not PMC member, not election owner) performs:

```http
# View another user's election management page
GET /manage/abc123 HTTP/1.1
Cookie: session=<bob_session>

→ 200 OK (full management interface rendered)

# Add a malicious issue to Alice's election
POST /do-add-issue/abc123 HTTP/1.1
Cookie: session=<bob_session>
Content-Type: application/x-www-form-urlencoded

title=Tampered+Issue&description=Injected+by+Bob

→ 303 Redirect (issue created successfully)

# Open Alice's election prematurely
GET /do-open/abc123 HTTP/1.1
Cookie: session=<bob_session>

→ 303 Redirect (election opened, irreversible)

# Close Alice's election, ending voting
GET /do-close/abc123 HTTP/1.1
Cookie: session=<bob_session>

→ 303 Redirect (election closed, irreversible)
```

**Impact:**

Any authenticated ASF committer can fully control ANY election in the system:
- Add, edit, or delete issues in elections they don't own
- Open elections prematurely (before proper configuration)
- Close elections early (suppressing legitimate votes)
- Modify scheduled open/close dates
- Access management interfaces for all elections

This completely undermines election integrity because management authorization relies solely on client-side UI filtering (which elections are displayed), not on server-side enforcement. An attacker who knows valid election IDs can manipulate any election in the system.

**Remediation:**

**Immediate (P0):** Implement ownership verification in the `load_election` decorator at the exact location marked by the `### check authz` comment:

```python
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            result = await basic_info()
            result.title = 'Unknown Election'
            result.eid = eid
            raise_404(T_BAD_EID, result)

        return await func(e)
    return loader


def load_election_owned(func):
    """Decorator that loads an Election AND verifies the current user owns it."""
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            result = await basic_info()
            result.title = 'Unknown Election'
            result.eid = eid
            raise_404(T_BAD_EID, result)

        # AUTHORIZATION: verify the current user is the election owner
        result = await basic_info()
        metadata = e.get_metadata()
        if result.uid != metadata.owner_pid:
            _LOGGER.warning(
                f'User[U:{result.uid}] denied access to election[E:{eid}]'
                f' (owner: {metadata.owner_pid})'
            )
            quart.abort(403)

        return await func(e)
    return loader
```

**Apply to all management endpoints:**

### ASVS-831-HIGH-001 — Irreversible State Changes via GET Requests with Client-Side-Only Confirmation

**Severity:** HIGH  
**Gap Type:** Type A (Entry point with NO server-side authorization control beyond authentication)

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** 
  - `do_open_endpoint` (line ~445)
  - `do_close_endpoint` (line ~465)
- **Template File:** `v3/server/templates/manage.ezt` (line ~262)

**Related Functions Checked:**
- `election.open()` — irreversible state transition
- `election.close()` — irreversible state transition
- Client-side confirmation modal implementation

**Description:**

Critical state-changing operations (opening and closing elections) are implemented as GET endpoints with only client-side JavaScript confirmation dialogs. The server-side handlers perform no verification beyond authentication, and the use of GET methods means these operations can be triggered via simple URL navigation, image tags, iframe embeds, or browser prefetch mechanisms — completely bypassing the client-side confirmation.

Combined with the missing ownership check (ASVS-831-CRIT-001), this allows any authenticated committer's browser session to be weaponized to open or close any election through cross-site request forgery or social engineering.

**Vulnerable Code:**

```python
# v3/server/pages.py, line ~445
@APP.get('/do-open/<eid>')           # <-- GET method for state change
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz                   # <-- No authorization check
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)                # <-- Irreversible operation executes
    ...

# v3/server/pages.py, line ~465
@APP.get('/do-close/<eid>')          # <-- GET method for state change
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz                   # <-- No authorization check
    election.close()                  # <-- Irreversible operation executes
    ...
```

**Client-side-only control:**

```javascript
// v3/server/templates/manage.ezt, line ~262
const confirmOpen = document.getElementById('confirm-open');
confirmOpen.addEventListener('click', () => {
    openModal.hide();
    window.location.href = '/do-open/[eid]';  // Simple GET navigation
});
```

**Data Flow:**

```
Source: GET /do-open/<eid> (triggerable via link, img, iframe, prefetch)
  ↓
Authorization: Only @asfquart.auth.require({R.committer}) — no ownership check
  ↓
Client-side "barrier": JavaScript confirmation modal (trivially bypassed)
  ↓
Sink: election.open() — permanently transitions election to open state
```

**Proof of Concept:**

An attacker who knows a committer's session is active can trigger open/close via:

```html
<!-- Embedded in any page the victim committer visits -->
<img src="https://voting.example.org/do-open/abc123" style="display:none">
<!-- or -->
<img src="https://voting.example.org/do-close/abc123" style="display:none">

<!-- Or via link in phishing email -->
<a href="https://voting.example.org/do-open/abc123">
  View Election Results
</a>

<!-- Or via iframe -->
<iframe src="https://voting.example.org/do-close/abc123" 
        style="display:none"></iframe>
```

Because these are GET endpoints, browsers will follow the URL automatically when:
- The user clicks a link
- An image tag loads
- An iframe is rendered
- Browser prefetch is triggered
- Any JavaScript performs navigation

No form submission, CSRF token, or JavaScript confirmation is involved. The server-side handler executes the irreversible state change without any additional verification.

**Impact:**

Elections can be opened prematurely (before issues are properly configured and voter lists are finalized) or closed prematurely (suppressing legitimate votes and preventing eligible voters from participating), all without any server-side authorization beyond authentication. 

The only protection is a client-side JavaScript confirmation dialog, which is completely bypassed when the URL is loaded directly through any of the attack vectors described above.

Combined with ASVS-831-CRIT-001 (no ownership check), this means ANY committer's browser session can be leveraged to open or close ANY election through:
- Social engineering (clicking a malicious link)
- Cross-site request forgery (visiting a page with embedded attack code)
- Browser prefetch mechanisms
- Email-based attacks

**Remediation:**

**Immediate (P0):** Convert to POST endpoints and add ownership verification:

```python
@APP.post('/do-open/<eid>')          # POST, not GET
@asfquart.auth.require({R.pmc_member})
@load_election_owned                  # Ownership check
async def do_open_endpoint(election):
    result = await basic_info()
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    return quart.redirect(f'/manage/{election.eid}')

@APP.post('/do-close/<eid>')         # POST, not GET
@asfquart.auth.require({R.pmc_member})
@load_election_owned                  # Ownership check
async def do_close_endpoint(election):
    result = await basic_info()
    election.close()
    return quart.redirect(f'/manage/{election.eid}')
```

**Update template JavaScript to use form submission:**

```javascript
// v3/server/templates/manage.ezt
const confirmOpen = document.getElementById('confirm-open');
confirmOpen.addEventListener('click', () => {
    openModal.hide();
    fetch('/do-open/[eid]', {

            'Content-Type': 'application/x-www-form-urlencoded',
            'X-CSRFToken': csrfToken  // When CSRF tokens are implemented
        },
    }).then(() => window.location.reload());
});

const confirmClose = document.getElementById('confirm-close');

    closeModal.hide();
```

### ASVS-831-MED-001 — Authorization Tier Inconsistency: Lower Privilege Required for Management Than Creation

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** All management endpoints vs. `do_create_endpoint`

**Related Functions Checked:**
- `do_create_endpoint` (line ~423) — requires `R.pmc_member`
- `do_open_endpoint` (line ~445) — requires only `R.committer`
- `do_close_endpoint` (line ~465) — requires only `R.committer`
- `do_add_issue_endpoint` (line ~483) — requires only `R.committer`
- `do_edit_issue_endpoint` (line ~507) — requires only `R.committer`
- `do_delete_issue_endpoint` (line ~530) — requires only `R.committer`
- `do_set_open_at_endpoint` — requires only `R.committer`
- `do_set_close_at_endpoint` — requires only `R.committer`

**Description:**

The application has an inverted authorization model where creating an election requires higher privileges (`R.pmc_member`) than performing all subsequent management operations (`R.committer`). This means users who lack sufficient privileges to create elections can nonetheless fully manage, modify, open, close, and delete issues from any existing election.

Every management endpoint includes a comment acknowledging this issue: `### need general solution`.

**Vulnerable Code:**

```python
# Election CREATION requires PMC member (line ~423)
@APP.post('/do-create-election')
@asfquart.auth.require({R.pmc_member})   # Higher privilege
async def do_create_endpoint():
    ...

# Election MANAGEMENT requires only committer (lines ~445, ~465, ~483, etc.)
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})    # Lower privilege
@load_election
async def do_open_endpoint(election):
    ...

@APP.post('/do-add-issue/<eid>')
@asfquart.auth.require({R.committer})    # Lower privilege
@load_election
async def do_add_issue_endpoint(election):
    ...
```

**Data Flow:**

```
Source: Any committer-level user (lower tier than PMC member)
  ↓
Cannot create elections (R.pmc_member required)
  ↓
CAN manage/modify/open/close any election (only R.committer required)
  ↓
Authorization tier inversion: more sensitive operations require less privilege
```

**Proof of Concept:**

A committer who is NOT a PMC member:

```http
# Cannot create:
POST /do-create-election HTTP/1.1
Cookie: session=<committer_session>

→ 403 Forbidden (R.pmc_member required)

# But CAN fully manage any existing election:
POST /do-add-issue/abc123 HTTP/1.1
Cookie: session=<committer_session>
Content-Type: application/x-www-form-urlencoded

title=New+Issue&description=Added+by+committer

→ 303 Success (only R.committer needed)

GET /do-open/abc123 HTTP/1.1
Cookie: session=<committer_session>

→ 303 Success

GET /do-close/abc123 HTTP/1.1
Cookie: session=<committer_session>

→ 303 Success

POST /do-delete-issue/abc123/issue1 HTTP/1.1
Cookie: session=<committer_session>

→ 303 Success
```

**Impact:**

The authorization model is inverted: creation of elections (a lower-impact, reversible operation that simply initializes a new election) requires higher privilege than opening/closing elections and modifying issues (higher-impact, irreversible operations that affect election integrity and voter participation).

A committer who should only have voter-level access can perform all administrative operations on any election. This violates the principle of least privilege and creates an inconsistent security model where the barrier to entry (creation) is higher than the barrier to destructive actions (management).

While this is partially mitigated by the fact that committers are generally trusted community members, it still represents a significant authorization flaw, especially when combined with the missing ownership checks.

## Positive Security Patterns

1. **Voter eligibility enforcement on view page** (`vote_on_page`, line ~249): Explicit check using `q_find_issues` before rendering ballot — proper server-side authorization that cannot be bypassed by client manipulation.

2. **Document serving authorization** (`serve_doc`, line ~574): Per-document authorization check via `q_get_mayvote.first_row(result.uid, iid)` before serving files — good defense-in-depth that prevents unauthorized access to election materials.

3. **Vote recording integrity** (`add_vote`, `election.py:226`): The `q_get_mayvote` query acts as a server-side eligibility gate — votes cannot be recorded without a mayvote entry, regardless of client manipulation.

4. **Election tampering detection** (`is_tampered`, `election.py:268`): Integrity verification using Argon2 hash of election data before tallying operations — prevents undetected modification of election results.

5. **Cryptographic vote isolation**: Per-voter salts, vote tokens, and encrypted vote storage prevent correlation even with database access — strong privacy protection for voter anonymity.

6. **SortableJS integrity attribute** (`vote-on.ezt`): Subresource integrity hash on third-party JavaScript dependency — prevents supply chain attacks via CDN compromise.

7. **Principle of least information in API** (`get_metadata`, `election.py:255`): Salt and `opened_key` are excluded from the metadata returned to handlers — reduces information leakage.

8. **Exception-based state enforcement** (`_all_metadata(required_state)`, `election.py:183`): Proper use of exceptions for critical state checks in `add_vote` and `tally_issue` — cannot be stripped by Python optimization flags.

## Architecture Observations

### 1. Authorization Model is Authentication-Only

The system currently has a two-tier authentication model (`committer` vs `pmc_member`) but lacks object-level authorization entirely. The codebase explicitly plans for this (`### check authz` appears 10 times) but it was never implemented. This represents a significant gap between design intent and implementation.

### 2. UI Filtering Creates False Confidence

The admin page only shows elections owned by the current user via `owned_elections()`, but the underlying API endpoints accept any valid election ID. This creates an illusion of access control at the presentation layer that does not exist at the API layer — a classic "security by obscurity" pattern where the UI hides functionality that remains accessible via direct API calls.

### 3. State Enforcement Uses `assert` Statements

Seven of nine state-checking call sites use Python `assert` statements, which are stripped by the `-O` optimization flag. Only `add_vote` and `tally_issue` use proper exception-based state checks via `_all_metadata(required_state)`. This means state enforcement could be accidentally disabled in production if Python is run with optimization enabled.

### 4. Planned CSRF Protection Not Implemented

Templates include `csrfToken: 'placeholder'` throughout, indicating CSRF protection was planned but never implemented. Combined with GET-based state changes, this creates multiple attack vectors for unauthorized operations.

### 5. Data Layer Authorization vs. Presentation Layer Authorization

The system demonstrates strong data-layer authorization for voting operations (voter eligibility is enforced at the database query level via `q_get_mayvote` and `q_find_issues`), but completely lacks presentation-layer authorization for management operations. This inconsistency suggests different parts of the codebase were developed with different security models in mind.
