#### ASVS-832-SEV-001 — `owner_pid` and `authz` Authorization Values Exist But Are Never Consulted

**Gap Type:** Type B — Control EXISTS but NOT CALLED

**Severity:** CRITICAL

**Location:** `v3/server/pages.py` — `load_election` decorator (line ~193), `load_election_issue` decorator (line ~216), and all management endpoints; `v3/steve/election.py` — `get_metadata()` (line ~149)

**Related Functions Checked:**
- `load_election()` — `pages.py` ~line 193
- `load_election_issue()` — `pages.py` ~line 216
- `_set_election_date()` — `pages.py` ~line 99
- `do_vote_endpoint()` — `pages.py` ~line 308
- `do_create_endpoint()` — `pages.py` ~line 341
- `do_open_endpoint()` — `pages.py` ~line 371
- `do_close_endpoint()` — `pages.py` ~line 386
- `do_add_issue_endpoint()` — `pages.py` ~line 401
- `do_edit_issue_endpoint()` — `pages.py` ~line 420
- `do_delete_issue_endpoint()` — `pages.py` ~line 440

**Description:** 

The `owner_pid` and `authz` fields are stored in the `election` table and are clearly intended as authorization decision factors for controlling who may manage an election. However, these values are **never checked** at any endpoint. All management operations only require `R.committer` authentication, meaning any authenticated committer can manage any election. Critically, any change to `owner_pid` or `authz` (the values on which authorization decisions should be made) has **zero effect** on actual authorization — the system never reads these values for access control purposes.

**Authorization Data Flow (broken):**
```
owner_pid stored → Election.create() → c_create query → DB
                                                         ↓
                      [NEVER READ FOR AUTHORIZATION]     ↓
                                                         ↓
authz stored → Election.create() → c_create query → DB  ↓
                                                         ↓
ANY committer → @asfquart.auth.require({R.committer}) → load_election → endpoint
                                        ↑
                     No owner_pid/authz check anywhere
```

```python
def require_election_owner(func):
    """Decorator to verify the current user owns the election."""
    @functools.wraps(func)
    async def checker(election, *args, **kwargs):
        s = await asfquart.session.read()
```

### High Findings

#### ASVS-832-SEV-002 — Voter Eligibility Not Explicitly Verified at Vote Submission Endpoint

**Gap Type:** Type C (partial) — Data-layer lookup result not explicitly checked; authorization denial occurs via unhandled crash

**Severity:** HIGH

**Location:** `v3/server/pages.py` — `do_vote_endpoint()` (line ~308); `v3/steve/election.py` — `add_vote()` (line ~229)

**Related Functions Checked:**
- `do_vote_endpoint()` — `pages.py` ~line 308
- `add_vote()` — `election.py` ~line 229
- `vote_on_page()` — `pages.py` ~line 264 (comparison: has explicit check)
- `q_get_mayvote.first_row()` — invoked in `election.py` ~line 233
- `q_find_issues.perform()` — invoked in `pages.py` ~line 266

**Description:** 

The vote submission endpoint (`POST /do-vote/<eid>`) does not explicitly verify that the submitting user is an eligible voter for each issue before calling `add_vote()`. While the page-load handler (`vote_on_page`) performs an explicit eligibility check via `q_find_issues`, the submission endpoint skips this check entirely. The `add_vote()` method queries `q_get_mayvote` but never checks the result for `None` — if the voter is ineligible, the code crashes with an `AttributeError` when accessing `mayvote.salt`. This crash happens to prevent the unauthorized vote, but it is not a deliberate authorization check.

This violates ASVS 8.3.2 because: if voter eligibility were to change (mayvote entries revoked via direct database modification or future admin feature), the system's response would be an unhandled exception rather than a proper authorization denial with audit logging.

**Data Flow:**
```
POST /do-vote/<eid>
  → @asfquart.auth.require({R.committer})  ← only checks authentication
  → @load_election                          ← only checks election exists
  → do_vote_endpoint()
    → form parsing extracts vote-{iid} pairs
    → checks iid in issue_dict              ← only checks issue exists in election
    → election.add_vote(result.uid, iid, votestring)
      → _all_metadata(self.S_OPEN)          ← checks election is open ✓
      → q_get_mayvote.first_row(pid, iid)   ← returns None if ineligible
      → mayvote.salt                        ← CRASHES with AttributeError if None
    → except Exception as e                 ← caught generically, no authz log
```

**Vulnerable Code — `add_vote()` (`election.py` ~line 229):**

#### ASVS-832-SEV-003 — Election State-Changing Operations Use GET Requests, Undermining Authorization State Integrity

**Gap Type:** Type A — Entry point with insufficient protection for authorization-impacting state change

**Severity:** MEDIUM

**Location:** `v3/server/pages.py` — `do_open_endpoint()` (line ~371), `do_close_endpoint()` (line ~386)

**Related Functions Checked:**
- `do_open_endpoint()` — `pages.py` ~line 371
- `do_close_endpoint()` — `pages.py` ~line 386
- `election.open()` — `election.py` ~line 84
- `election.close()` — `election.py` ~line 126

**Description:** 

Election open and close operations are implemented as GET endpoints. Election state (`editable` → `open` → `closed`) is a critical authorization decision factor — it controls whether voting is accepted, whether issues can be edited, and whether tallying is permitted. Because these state transitions are triggered by GET requests, they can be initiated by embedded image tags, link prefetching, browser extensions, or social engineering (shared link). Combined with the missing owner authorization (Finding ASVS-832-SEV-001), any authenticated committer clicking a crafted link could unknowingly change an election's authorization state.

ASVS 8.3.2 requires that changes to authorization decision values be controlled. Using GET for these operations means the authorization state change is trivially triggerable without the user's explicit intent.

**Proof of Concept:**
```html

<!-- Or sent as a link: "Check out the election results!" -->
```

**Impact:**
- Election state can be changed without explicit user intent
- Closing an open election immediately prevents all further voting (authorization change)
- Opening an editable election locks out all issue editing (authorization change)

## Positive Security Patterns

1. **Real-time state verification at data layer:** `add_vote()` checks `_all_metadata(self.S_OPEN)` which queries the database at execution time, ensuring election state changes are immediately reflected (lines 229-230, `election.py`).

2. **Sensitive data filtering in public APIs:** `get_metadata()` explicitly excludes `SALT` and `OPENED_KEY` (lines 149-163, `election.py`). `get_issue()` never returns the salt. Queries in `queries.yaml` (`q_owned`, `q_open_to_me`) exclude crypto fields.

3. **Parameterized SQL queries throughout:** All queries in `queries.yaml` use `?` placeholders, preventing SQL injection.

4. **Transaction management for multi-step operations:** `delete()` and `add_salts()` use explicit `BEGIN TRANSACTION` / `COMMIT` for atomicity.

5. **Document serving with eligibility check:** `serve_doc()` verifies the requesting user has a `mayvote` entry for the issue before serving documents (lines 498-504, `pages.py`).

6. **Deterministic election integrity verification:** `is_tampered()` recomputes the `opened_key` from current election data and compares with the stored value, providing tamper detection.

7. **Vote anonymization:** Votes are shuffled before tallying (`crypto.shuffle(votes)`), vote tokens are used instead of direct PID/IID pairs, and per-voter salts prevent correlation.

## Architecture Observations

1. **Authorization is two-tier but only one tier is implemented:** Authentication (`R.committer`, `R.pmc_member`) is consistently applied via decorators. However, the second tier — operation-level authorization based on `owner_pid` and `authz` — exists only as database fields and TODO comments. This creates a false sense of security, as the authorization infrastructure appears to exist but provides no enforcement.

2. **State-machine enforcement is solid at the data layer:** The Election class properly enforces state transitions (`editable` → `open` → `closed`) via assertions and database queries. This is the correct pattern for ASVS 8.3.2 compliance regarding election state.

3. **No mechanism to revoke voter eligibility during an open election:** The `add_voter()` method asserts `is_editable()`, and there is no `remove_voter()` method. Once an election is open, voter eligibility is immutable. This is actually a positive for ASVS 8.3.2 — the authorization factors can't change, so immediate application is trivially satisfied for voter eligibility.

4. **Error handling masks authorization failures:** The generic `except Exception` in `do_vote_endpoint` catches both legitimate errors and authorization failures identically, preventing proper audit trails.
