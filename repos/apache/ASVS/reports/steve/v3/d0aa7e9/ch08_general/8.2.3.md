### Critical Severity

#### ASVS-8230-CRI-001: Authorization Fields Defined But Never Enforced in Election Management Decorators

**Severity:** CRITICAL (Type B ŌĆö Control EXISTS but NOT CALLED)

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `load_election` (lines 167-186), `load_election_issue` (lines 189-218)
- **Supporting schema:** `v3/schema.sql` lines 48-59 (`owner_pid`, `authz` columns)

**Related Functions Checked:**
- `do_open_endpoint` (line 440)
- `do_close_endpoint` (line 457)
- `do_add_issue_endpoint` (line 474)
- `do_edit_issue_endpoint` (line 496)
- `do_delete_issue_endpoint` (line 518)
- `do_set_open_at_endpoint` (line 99)
- `do_set_close_at_endpoint` (line 99)
- `manage_page` (line 322)
- `manage_stv_page` (line 189)

**Description:**

The application defines ownership (`owner_pid`) and authorization group (`authz`) fields in the election schema but never enforces them in any management endpoint. All 9 election management handlers contain placeholder comments `### check authz` that were never implemented, allowing any authenticated committer to fully control any election regardless of ownership.

**Vulnerable Code:**

```python
# pages.py lines 167-186
def load_election(func):
    """Decorator to load/pass-argument an Election from EID."""

    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            result = await basic_info()
            result.title = 'Unknown Election'
            result.eid = eid
            raise_404(T_BAD_EID, result)

        _LOGGER.debug(f'Loaded: {e}')

        ### check authz          # <-- PLACEHOLDER, NEVER IMPLEMENTED

        return await func(e)

    return loader
```

The same placeholder exists in `load_election_issue` at line 210 and in each endpoint handler (e.g., `do_open_endpoint` line 447, `do_close_endpoint` line 460, `do_add_issue_endpoint` line 478).

**Authorization Data That Exists But Is Ignored:**

```sql
-- schema.sql lines 48-59
owner_pid  TEXT NOT NULL,       -- who created this election
authz  TEXT,                    -- what authz group can edit (NULL = owner only)
```

These fields are populated at creation time (`election.py` line 295, `pages.py` line 430) and returned by `get_metadata()` (`election.py` lines 145-146), but never used in authorization decisions.

**Data Flow:**

```
User Request ŌåÆ @asfquart.auth.require({R.committer}) ŌåÆ load_election(eid)
    ŌåÆ Election loaded (owner_pid, authz available) ŌåÆ ### check authz [NEVER IMPLEMENTED]
    ŌåÆ Handler executes WITH FULL ACCESS to election properties
```

**Affected Endpoints (9 handlers):**
- `GET /manage/<eid>` ŌĆö exposes all election metadata and issue properties
- `GET /manage-stv/<eid>/<iid>` ŌĆö exposes STV issue configuration (seats, candidates)
- `POST /do-set-open_at/<eid>` ŌĆö modifies election date property
- `POST /do-set-close_at/<eid>` ŌĆö modifies election date property
- `GET /do-open/<eid>` ŌĆö transitions election state (generates crypto keys, irrevocable)
- `GET /do-close/<eid>` ŌĆö transitions election state (irrevocable)
- `POST /do-add-issue/<eid>` ŌĆö creates new issue properties
- `POST /do-edit-issue/<eid>/<iid>` ŌĆö modifies issue title, description, type, kv
- `POST /do-delete-issue/<eid>/<iid>` ŌĆö destroys issue and its voting records

**Proof of Concept:**

Any authenticated committer (User B) can manage an election created by User A:

```http
# User B opens User A's election (irreversible state change)
GET /do-open/a1b2c3d4e5 HTTP/1.1
Host: steve.apache.org
Cookie: session=<user_b_session>

# User B closes User A's election (irreversible)  
GET /do-close/a1b2c3d4e5 HTTP/1.1
Host: steve.apache.org
Cookie: session=<user_b_session>

# User B adds a malicious issue to User A's election
POST /do-add-issue/a1b2c3d4e5 HTTP/1.1
Host: steve.apache.org
Cookie: session=<user_b_session>
Content-Type: application/x-www-form-urlencoded

title=Malicious+Issue&description=Injected+by+unauthorized+user

# User B deletes an issue from User A's election
POST /do-delete-issue/a1b2c3d4e5/f6g7h8i9j0 HTTP/1.1
Host: steve.apache.org
Cookie: session=<user_b_session>
```

**Impact:**
- Any of the ~800+ ASF committers can fully control any election: view sensitive configuration, add/edit/delete issues, prematurely open or close elections, and modify scheduling dates
- Election integrity is compromised since unauthorized users can tamper with issues before or after the tamper-detection key (`opened_key`) is generated
- Opening an election is irrevocable ŌĆö once opened, the election cannot return to the editable state, and unauthorized opening would lock out the legitimate owner from making changes
- Direct violation of ASVS 8.2.3: object properties (election metadata, issue configuration) are accessible and modifiable without field-level authorization

**Remediation:**

Add ownership/authz verification in the `load_election` decorator:

```python
def load_election(func, *, require_owner=False):
    """Decorator to load an Election and verify authorization."""

    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            result = await basic_info()
            result.title = 'Unknown Election'
            result.eid = eid
            raise_404(T_BAD_EID, result)

        # Verify the authenticated user is authorized to manage this election
        result = await basic_info()
        md = e.get_metadata()
        
        if not is_authorized_for_election(result.uid, md):
            quart.abort(403)

        return await func(e)

    return loader


def is_authorized_for_election(uid, election_metadata):
    """Check if UID is authorized to manage the election.
    
    Authorization rules:
    - The election owner always has access
    - If authz is set, members of that group have access
    - Otherwise, only the owner has access
    """
    if uid == election_metadata.owner_pid:
        return True
    if election_metadata.authz:
        # Check group membership (implementation depends on authz format)
        return check_group_membership(uid, election_metadata.authz)
    return False
```

**Severity:** HIGH

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `vote_on_page` (lines 225-272), `do_vote_endpoint` (lines 389-419)

## Positive Security Patterns

1. **Explicit field filtering in `get_metadata()`** (`election.py:141-152`): Constructs return object with only allowed fields, explicitly documented with `# NOTE: do not return the SALT or OPENED_KEY`. This demonstrates proper implementation of field-level access control for sensitive cryptographic material.

2. **Column-specific SQL queries**: `q_owned`, `q_open_to_me`, and `q_upcoming_to_me` queries in `queries.yaml` select specific columns rather than `SELECT *`, preventing sensitive fields (`salt`, `opened_key`) from reaching the presentation layer.

3. **Document-level authorization in `serve_doc`** (`pages.py:478-486`): Checks mayvote entry before serving per-issue documents, ensuring only eligible voters can access supporting materials. This is a correct implementation of resource-level authorization.

4. **Parameterized queries throughout**: All database operations use `?` placeholders via the `asfpy.db` layer, preventing SQL injection.

5. **Voter eligibility gate in `vote_on_page`** (`pages.py:236-241`): Verifies the user has at least one mayvote entry before displaying the voting interface, though this needs enhancement to filter at the per-issue level.

6. **State machine enforcement**: Election state transitions are validated via assertions (`is_editable()`, `is_open()`, `is_closed()`) before state-changing operations in `election.py`, preventing invalid state transitions.

7. **Tamper detection**: The `is_tampered()` method (`election.py:304-319`) verifies election integrity by recomputing the opened_key hash from current data.

8. **`list_issues()` explicit field construction** (`election.py:196-207`): Builds output objects with selected fields rather than passing raw database rows, demonstrating awareness of field-level security.

## Architecture Observations

1. **Authorization architecture gap**: The system has authentication (via `asfquart.auth.require`) and stores authorization data (`owner_pid`, `authz`), but lacks the middleware layer to connect them. The `### check authz` comments (appearing 10+ times) confirm this was a planned feature that was never implemented.

2. **Two-tier authorization model partially implemented**: The system distinguishes between `R.committer` (voter-facing) and `R.pmc_member` (election creation), but doesn't implement the third tier: per-election ownership authorization.

3. **Field-level access at data layer is well-designed**: The `Election` class properly separates internal methods (`_all_metadata`) from external APIs (`get_metadata`, `get_issue`, `list_issues`), with explicit field filtering. The gap is entirely at the controller/decorator layer where object-level authorization should gate access.

4. **Per-issue eligibility model is structurally sound but underutilized**: The `mayvote` table correctly models per-person-per-issue eligibility with unique `(pid, iid)` pairs, but the presentation and submission layers treat eligibility as election-wide.

5. **Defense-in-depth partially present**: While sensitive fields are filtered at the data layer, the lack of object-level authorization at the controller layer means the defense relies entirely on a single control point that can be bypassed.
