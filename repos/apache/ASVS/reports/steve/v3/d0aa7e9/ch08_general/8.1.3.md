## Findings

### HIGH SEVERITY

#### Finding ASVS-813-HIGH-001

**Title:** Authorization decision model documented as "TBD" while application enforces incomplete access control

**Location:**
- **File:** `v3/docs/schema.md` (lines referencing `authz` field)
- **Corroborating Files:** `v3/server/pages.py` (multiple locations)

**Functions Affected:**
- `load_election` decorator (`pages.py:168`)
- `load_election_issue` decorator (`pages.py:197`)
- `do_vote_endpoint` (`pages.py:376`)
- `do_create_endpoint` (`pages.py:424`)
- `do_open_endpoint` (`pages.py:446`)
- `do_close_endpoint` (`pages.py:465`)
- `do_add_issue_endpoint` (`pages.py:484`)
- `do_edit_issue_endpoint` (`pages.py:506`)
- `do_delete_issue_endpoint` (`pages.py:527`)

**Description:**

The documentation states that `authz` defines who can edit an election and that `owner_pid` restricts editing when `authz` is NULL, but neither check is implemented. This creates a **Type B gap** — the control is DEFINED in documentation and schema but NOT CALLED in any endpoint.

**Vulnerable Documentation Quote** (`schema.md`):

```markdown
- `authz` (TEXT): Specifies an LDAP group (e.g., group name or identifier) allowed to edit the election. 
  If `NULL`, only `owner_pid` can edit. The exact format is TBD but will likely map to LDAP-based persons.
```

```markdown
- **Authorization**: The `authz` field's format is TBD but will likely use LDAP group identifiers 
  to define editing permissions. Validation occurs outside the schema (e.g., via LDAP integration).
```

**Code Evidence** (`pages.py`):

```python
# Line 168 (load_election decorator):
        ### check authz

# Line 197 (load_election_issue decorator):
        ### check authz

# Line 376 (do_vote_endpoint):
        ### check authz

# Line 424 (do_create_endpoint):
        ### check authz

# Line 446 (do_open_endpoint):
        ### check authz

# Line 465 (do_close_endpoint):
        ### check authz

# Line 484 (do_add_issue_endpoint):
        ### check authz

# Line 506 (do_edit_issue_endpoint):
        ### check authz

# Line 527 (do_delete_issue_endpoint):
        ### check authz
```

**Data Flow:**

The `authz` column is stored in the `election` table → `get_metadata()` retrieves it → but no code path ever evaluates it → all endpoints rely solely on `R.committer` role check → any committer can manage ANY election.

**Impact:**

Any authenticated committer can:
- Open/close any election (`do_open_endpoint`, `do_close_endpoint`)
- Add/edit/delete issues in any election
- Set open/close dates on any election

The documentation's "TBD" status means developers and testers have no specification to implement or verify against, violating ASVS 8.1.3's requirement that authorization decision-making factors be documented and actionable.

**Proof of Concept:**

Authenticated committer "user-A" who is NOT the `owner_pid` of election `abc1234567`:

```http
POST /do-add-issue/abc1234567
Content-Type: application/x-www-form-urlencoded

title=Injected+Issue&description=Unauthorized+modification
```

This succeeds because `load_election` only verifies the election exists, not that the user owns it.

**Remediation:**

1. **Document the authorization model** — Define in schema.md (or a dedicated security document):

```markdown
## Authorization Model

### Election Management Authorization
The following contextual attributes determine authorization for election management operations:

| Attribute | Type | Description | Used For |
|-----------|------|-------------|----------|
| `owner_pid` | Identity | Person who created the election | Full management access |
| `authz` | Group membership | LDAP group identifier | Delegated edit access |
| `election.state` | Contextual | Current lifecycle state (editable/open/closed) | Operation eligibility |
| User role | Identity | ASF committer or PMC member status | Endpoint access tier |

### Decision Rules
- **Create election**: Requires PMC member role
- **Manage election (add/edit/delete issues, open, close)**: Requires `owner_pid` match OR membership in `authz` group
- **Vote**: Requires committer role AND `mayvote` entry for the specific issue
- **View documents**: Requires valid session AND `mayvote` entry for the issue
```

2. **Implement the documented checks** in `load_election`:

```python
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        result = await basic_info()
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            # ... existing 404 handling ...
            
        md = e.get_metadata()
        # Enforce ownership/authz for management operations
        if md.owner_pid != result.uid:
            if not md.authz or not await check_group_membership(result.uid, md.authz):
                quart.abort(403, 'Not authorized to manage this election')
        
        return await func(e)
    return loader
```

#### Finding ASVS-813-MEDIUM-003

**Title:** Election state as a security-decision attribute is enforced inconsistently relative to documentation

**Location:**
- **Files:** `v3/docs/schema.md` (Election Lifecycle section), `v3/steve/election.py`, `v3/server/pages.py`

**Functions Affected:**
- `add_issue` (`election.py:167-171`)
- `edit_issue` (`election.py:186`)
- `delete_issue` (`election.py:197`)
- `delete` (`election.py:64`)
- `add_vote` (`election.py:208`) — correct implementation

**Description:**

The documentation defines the election state model as a security control (editable state restricts modifications), but the enforcement mechanism uses Python `assert` statements. When Python is run with optimization flags (`-O` or `PYTHONOPTIMIZE=1`), all `assert` statements are stripped.

**Documentation states** (`schema.md`):

```markdown
- **Editable**: The election is created and can be modified (e.g., adding issues, voters).
- **Open**: [...] Eligible voters (in `mayvote`) can vote on issues.
- **Closed**: [...] enabling tallying.
- The owner (`owner_pid`) can monitor and edit the election at all stages
```

**Code enforcement** (`election.py`):

```python
# Lines 167-171 - State checked via assertion
def add_issue(self, title, description, vtype, kv):
    "Add a new issue with a generated unique IID."
    assert self.is_editable()  # Only enforced in debug mode
    
# Line 186 - Same pattern
def edit_issue(self, iid, title, description, vtype, kv):
    assert self.is_editable()

# Line 197
def delete_issue(self, iid):
    assert self.is_editable()

# Line 64
def delete(self):
    assert self.is_editable()
```

**Correct implementation for comparison:**

```python
# Line 208 - Properly raises exception regardless of optimization
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)  # Raises ElectionBadState if not open
```

**Gap Analysis:**

The documentation does not mention this deployment constraint or specify the required enforcement mechanism. This inconsistency is undocumented — some state checks are advisory (assert) while others are mandatory (exception-based).

**Impact:**

If the application is deployed with Python optimizations enabled, state-based authorization for election management operations (add/edit/delete issues, delete election) would be silently bypassed. The documentation does not warn about this deployment requirement or clarify the enforcement guarantee level for each state transition.

**Remediation:**

1. **Document the enforcement model** for state-based decisions:

```markdown
### State-Based Authorization Enforcement
Election state checks MUST be enforced using exception-based validation 
(via `_all_metadata(required_state=...)`) rather than `assert` statements. 
The application MUST NOT be deployed with Python optimization flags (-O).
```

2. **Replace assertions with proper checks** in `election.py`:

```python
def add_issue(self, title, description, vtype, kv):
    "Add a new issue with a generated unique IID."
    self._all_metadata(self.S_EDITABLE)  # Raises ElectionBadState
    if vtype not in vtypes.TYPES:
        raise ValueError(f'Invalid vote type: {vtype}')
```

## Positive Security Patterns

1. **Tamper detection**: The `opened_key` mechanism using Argon2 hashing of election data provides integrity verification, and the tally CLI checks for tampering before proceeding (`tally.py:148`).

2. **Cryptographic vote anonymity**: Per-voter salts in `mayvote`, Argon2-derived vote tokens, and Fernet-encrypted vote storage provide strong vote anonymity even against database compromise.

3. **State machine design**: The election lifecycle (editable → open → closed) provides a well-structured model for operation authorization, even though enforcement is inconsistent.

4. **Audit logging**: Extensive `_LOGGER.info()` calls record user actions with UIDs and election/issue IDs throughout `pages.py`.

5. **Candidate shuffle**: `random.shuffle(issue.candidates)` in `vote_on_page` prevents positional bias in STV elections.

6. **Non-predictable IDs**: 10-character hex IDs for elections and issues prevent URL enumeration.

7. **Document access control**: `serve_doc` checks `mayvote` eligibility before serving per-issue documents.

---

## Architecture Observations

1. **Authentication vs. Authorization Gap**: The application has a clear authentication layer (`asfquart.auth`) with role-based route protection, but lacks a corresponding authorization layer for resource-level access control. The comments throughout the code (`### need general solution`, `### check authz`) indicate this is a known architectural gap.

2. **Documentation-Code Alignment**: The `schema.md` documentation is oriented toward database structure rather than security policy. ASVS 8.1.3 requires a security-focused document that maps contextual attributes to security decisions — this is architecturally different from schema documentation.

3. **CLI Tooling Security Model**: The `tally.py` CLI tool relies on filesystem-level access control (server access required) rather than application-level authentication, which is documented as intentional. However, the `--spy-on-open-elections` flag allows viewing partial results of active elections, and the security implications of this capability are not documented.
