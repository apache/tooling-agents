### Overview

The application configures TLS certificate/key pairs for HTTPS but **completely lacks cipher suite configuration, TLS protocol version restriction, and forward secrecy enforcement** in both standalone and ASGI deployment modes. No `ssl.SSLContext` is created or configured anywhere in the codebase, resulting in Python/system default cipher suites being used—which typically include weak and non-forward-secret ciphers.

## Findings

### High Severity

#### ASVS-1212-HIGH-001: Complete Absence of Cipher Suite Configuration in Standalone TLS Server

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `run_standalone()`
- **Lines:** 79-84

**Related Functions Checked:**
- `create_app()`
- `app.runx()`

**Description:**

The server passes raw certificate/key file paths to the underlying Quart/Hypercorn runtime without creating an `ssl.SSLContext`. This means:
1. **No cipher suite restriction** — All system-default ciphers are enabled, including potentially weak ones (e.g., `RC4`, `3DES`, `NULL`, `EXPORT`, CBC-mode ciphers vulnerable to BEAST/Lucky13).
2. **No cipher preference order** — Server does not enforce strongest-first ordering (`ssl.OP_CIPHER_SERVER_PREFERENCE` is not set).
3. **No forward secrecy enforcement** — Non-ECDHE/DHE cipher suites remain available.
4. **No TLS version pinning** — TLS 1.0 and 1.1 may be negotiated, which support weaker cipher suites.

**Vulnerable Code:**
```python
kwargs = {}
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
    extra_files.update((kwargs['certfile'], kwargs['keyfile']))

# Spool up the app!
app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Data Flow:**
```
config.yaml (certfile/keyfile) 
  → CERTS_DIR path construction 
  → app.runx(certfile=..., keyfile=...) 
  → Missing: SSLContext with cipher suite configuration
```

**Proof of Concept:**
```bash
# A client can negotiate a weak cipher suite against this server:
openssl s_client -connect host:58383 -cipher 'RC4-SHA' -tls1
openssl s_client -connect host:58383 -cipher 'DES-CBC3-SHA'
openssl s_client -connect host:58383 -cipher 'AES128-SHA'  # No forward secrecy

# Enumerate all accepted ciphers:
nmap --script ssl-enum-ciphers -p 58383 host
```

**Impact:**
- Weak cipher suites allow passive decryption by an attacker who compromises the server's private key (no forward secrecy)
- Certain legacy ciphers have known cryptographic weaknesses exploitable by active or passive attackers
- Fails ASVS 12.1.2 L2 (recommended ciphers only) and L3 (forward secrecy requirement)

**Remediation:**
```python
import ssl

def run_standalone():
    # ... existing logging setup ...
    
    app = create_app()
    
    kwargs = {}
    if app.cfg.server.certfile:
        # Create a properly configured SSLContext
        ctx = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
        ctx.minimum_version = ssl.TLSVersion.TLSv1_2
        ctx.maximum_version = ssl.TLSVersion.TLSv1_3
        
        # Only allow recommended cipher suites with forward secrecy
        ctx.set_ciphers(
            'ECDHE+AESGCM:ECDHE+CHACHA20:DHE+AESGCM:DHE+CHACHA20'
            ':!aNULL:!eNULL:!EXPORT:!DES:!RC4:!3DES:!MD5:!PSK'
        )
        
        # Server chooses cipher order (strongest preferred)
        ctx.options |= ssl.OP_CIPHER_SERVER_PREFERENCE
        # Disable compression (CRIME attack)
        ctx.options |= ssl.OP_NO_COMPRESSION
        
        ctx.load_cert_chain(
            certfile=CERTS_DIR / app.cfg.server.certfile,
            keyfile=CERTS_DIR / app.cfg.server.keyfile,
        )
        
        kwargs['ssl'] = ctx

    app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

### Medium Severity

#### ASVS-1212-MED-002: No TLS/Cipher Configuration for ASGI Deployment Mode

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `run_asgi()`
- **Lines:** 94-115

**Related Functions Checked:**
- `create_app()`

**Description:**

The ASGI mode creates the application but provides **no TLS configuration whatsoever**. The inline documentation suggests running via:
```
$ uv run python -m hypercorn main:steve_app
```
However:
- No Hypercorn configuration file (`hypercorn.toml`) is provided in the codebase
- No `--ciphers`, `--certfile`, `--keyfile`, or `--ssl-version` command-line guidance
- No programmatic SSLContext configuration within `run_asgi()`
- Deployments following this pattern will either lack TLS entirely or use Hypercorn's permissive defaults

**Vulnerable Code:**
```python
def run_asgi():
    "Run as an ASGI process; eg. Hypercorn"
    # ... logging setup ...
    
    _LOGGER.info(' ** Run-mode: ASGI')

    global steve_app
    steve_app = create_app()
```

**Data Flow:**
```
run_asgi() → create_app() → No TLS configuration provided to Hypercorn
```

**Impact:**
- Production deployments using ASGI mode have no secure cipher suite baseline
- Operators have no reference configuration for cipher suite hardening
- Cipher suite selection is left entirely to deployment luck

**Remediation:**

Provide a Hypercorn configuration file with hardened TLS settings:

```toml
# hypercorn.toml
bind = "0.0.0.0:58383"
certfile = "certs/server.pem"
keyfile = "certs/server-key.pem"
ciphers = "ECDHE+AESGCM:ECDHE+CHACHA20:DHE+AESGCM:!aNULL:!eNULL:!EXPORT:!DES:!RC4:!3DES"
```

And document the required invocation:
```bash
$ uv run python -m hypercorn --config hypercorn.toml main:steve_app
```

#### ASVS-1212-MED-003: Example Configuration Lacks Cipher Suite and TLS Version Settings

**Location:**
- **File:** `v3/server/config.yaml.example`
- **Lines:** 23-31

**Related Functions Checked:**
- Configuration schema parsing in `main.py`

**Description:**

The example configuration is the primary deployment reference, yet it only includes `certfile` and `keyfile`. It provides **no cipher suite configuration options**, meaning:
1. The configuration schema doesn't support cipher suite settings
2. Operators cannot restrict cipher suites via configuration
3. No secure defaults are documented or enforceable
4. No `tls_version_min` or `ciphers` fields exist

**Vulnerable Code:**
```yaml
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
```

**Data Flow:**
```
config.yaml → Configuration parser → No cipher suite fields available
```

**Impact:**
- Every deployment based on this template inherits Python/system default cipher suites
- No configuration-driven mechanism exists to enforce ASVS 12.1.2 compliance
- Operators must modify source code to achieve compliant cipher suite configuration

**Remediation:**

Extend the configuration schema and example:
```yaml
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
    
    # TLS hardening (ASVS 12.1.2)
    tls_min_version: "1.2"
    ciphers: "ECDHE+AESGCM:ECDHE+CHACHA20:DHE+AESGCM:DHE+CHACHA20:!aNULL:!eNULL:!EXPORT:!DES:!RC4:!3DES:!MD5:!PSK"
    prefer_server_ciphers: true
```

Then consume these settings in `main.py` when constructing the `SSLContext`.

## Architecture Observations

1. **No SSLContext anywhere**: The entire codebase delegates TLS to the framework default without any `ssl.SSLContext` creation. This is the root cause of all findings — without an explicit context, there is no mechanism to configure cipher suites.

2. **Two deployment paths, both uncontrolled**: Standalone mode passes raw file paths; ASGI mode provides no TLS configuration at all. Neither path has a hook for cipher suite control.

3. **Framework dependency**: The `asfquart` framework wrapper abstracts TLS setup, but this abstraction removes the operator's ability to configure cipher suites without modifying the framework or source code.

4. **Outbound connections unconfigured**: OAuth callbacks to `oauth.apache.org` use whatever the HTTP client library's defaults are. While Apache's OAuth server likely enforces strong ciphers server-side, the client should also restrict its acceptable ciphers.
