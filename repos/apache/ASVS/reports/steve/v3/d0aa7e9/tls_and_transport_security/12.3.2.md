## Findings

### Medium Severity

#### ASVS-1232-MED-001: No Explicit TLS Certificate Validation for OAuth Token Exchange

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with NO explicit control  
**File:** `v3/server/main.py`  
**Function:** `create_app()`  
**Lines:** 38–41  

**Description:**

The application makes outbound HTTPS requests to `oauth.apache.org` for authentication token exchange. While the configured URLs correctly use the `https://` scheme, there is no explicit SSL context creation, certificate verification enforcement, or CA trust store configuration anywhere in the application code. The application delegates all TLS client behavior to the `asfquart` framework with no verification that certificate validation is active.

**Vulnerable Code:**

```python
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**

```
User completes OAuth login → Application receives auth code →
Application acts as TLS CLIENT to oauth.apache.org:443 →
NO explicit ssl_context with verify_mode=CERT_REQUIRED →
Relies on implicit library defaults for certificate validation →
Receives OAuth token (authentication credential)
```

**What's Missing:**

No code in the audited files creates an `ssl.SSLContext` with `verify_mode = ssl.CERT_REQUIRED`, configures a CA bundle path, or passes a `verify=True` / `ssl` parameter to the HTTP client making the outbound OAuth request. If the underlying `asfquart` or its HTTP client library were ever configured (or defaulted) to skip verification, the application would silently accept fraudulent certificates, enabling man-in-the-middle attacks on the OAuth token exchange.

**Proof of Concept:**

An attacker performing a network-level MITM (e.g., DNS poisoning or ARP spoofing on the same network segment) could present a self-signed certificate for `oauth.apache.org`. Without explicit certificate validation enforcement, if the underlying library has verification disabled or improperly configured, the application would:

1. Accept the fraudulent certificate
2. Send the OAuth authorization code to the attacker's server
3. The attacker could exchange the code for a valid token or harvest credentials

**Impact:**

If certificate validation is not enforced by the underlying library, an attacker could intercept OAuth token exchanges, hijack authentication sessions, and impersonate legitimate voters in the election system.

**Remediation:**

Create and pass an explicit SSL context for all outbound connections:

```python
import ssl

def create_oauth_ssl_context():
    """Create a strict SSL context for outbound OAuth connections."""
    ctx = ssl.create_default_context()  # Loads system CA certificates
    ctx.minimum_version = ssl.TLSVersion.TLSv1_2
    ctx.verify_mode = ssl.CERT_REQUIRED
    ctx.check_hostname = True
    return ctx

# In create_app() or asfquart configuration:
oauth_ssl_context = create_oauth_ssl_context()
# Pass to HTTP client used for OAuth token exchange, e.g.:
# async with aiohttp.ClientSession() as session:
#     async with session.get(url, ssl=oauth_ssl_context) as resp:
```

Additionally, configure the `asfquart` framework to accept and use an explicit SSL context for its OAuth HTTP client operations.

#### ASVS-1232-MED-002: No Explicit SSL Context for Server-Side TLS Configuration

**Severity:** MEDIUM  
**Gap Type:** Type A — No control present  
**File:** `v3/server/main.py`  
**Function:** `run_standalone()`  
**Lines:** 76–78  

**Description:**

The server-side TLS configuration passes raw `certfile` and `keyfile` paths directly to `app.runx()` without constructing an `ssl.SSLContext`. This means there is no enforcement of minimum TLS protocol version, cipher suite selection, or other security parameters for inbound connections. While this is server-side TLS rather than client certificate validation, the pattern demonstrates the broader absence of explicit TLS configuration discipline, which extends to the client-side finding above.

**Vulnerable Code:**

```python
kwargs = {}
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
    extra_files.update((kwargs['certfile'], kwargs['keyfile']))

# Spool up the app!
app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**What's Missing:**

No `ssl.SSLContext` is created with protocol version restrictions (`minimum_version = TLSv1_2`), cipher suite configuration, or client certificate verification settings. The underlying framework decides all TLS parameters.

**Impact:**

Without explicit SSL context configuration, the server may accept deprecated TLS versions (TLS 1.0, 1.1) or weak cipher suites depending on the framework and Python version defaults. This contributes to a pattern where TLS security is entirely implicit rather than explicitly enforced.

**Remediation:**

```python
import ssl

def create_server_ssl_context(certfile, keyfile):
    ctx = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
    ctx.minimum_version = ssl.TLSVersion.TLSv1_2
    ctx.load_cert_chain(certfile=certfile, keyfile=keyfile)
    # Optionally restrict cipher suites
    ctx.set_ciphers('ECDHE+AESGCM:ECDHE+CHACHA20:DHE+AESGCM:DHE+CHACHA20')
    return ctx

# In run_standalone():
if app.cfg.server.certfile:
    certpath = CERTS_DIR / app.cfg.server.certfile
    keypath = CERTS_DIR / app.cfg.server.keyfile
    ssl_ctx = create_server_ssl_context(certpath, keypath)
    kwargs['ssl'] = ssl_ctx
```

## Positive Security Patterns

1. **HTTPS OAuth URLs**: Both OAuth endpoints are hardcoded to `https://`, preventing accidental use of plain HTTP for authentication flows.
2. **TLS Support Built-in**: The server does support TLS configuration through certificate/key files, showing awareness of transport security.
3. **Certificate Path Resolution**: Certificate files are resolved relative to a `certs/` subdirectory, providing organized key material storage.
4. **Static Folder Disabled**: `static_folder=None` reduces attack surface.

---

## Architecture Observations

1. **Framework Delegation**: The application heavily delegates TLS handling to the `asfquart` framework. The audited code contains no direct HTTP client calls — the actual OAuth HTTP request implementation is within the framework. A complete audit requires reviewing `asfquart`'s HTTP client configuration.

2. **Two Deployment Modes**: The application supports standalone (`app.runx()`) and ASGI (Hypercorn) modes. In ASGI mode, TLS is configured entirely through Hypercorn's external configuration, meaning the application code has zero control over TLS parameters.

3. **No LDAP Code Visible**: The domain context mentions LDAP connections, but no LDAP client code is present in the audited files. If LDAP is used elsewhere, those connections must also validate TLS certificates (LDAPS).

4. **Single External Dependency**: The only outbound TLS connection visible is to `oauth.apache.org`. This is a high-value target for MITM since it handles authentication.
