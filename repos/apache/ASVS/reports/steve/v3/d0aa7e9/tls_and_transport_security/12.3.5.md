**Overview:**

The audit evaluates the STeVe election application against ASVS 12.3.5, which requires strong mutual authentication (e.g., mTLS) for intra-service communications, PKI-based identity verification, and replay resistance. The application's TLS configuration is minimal: it supports optional server-side TLS via certificate files but lacks SSL context hardening, mutual TLS capability, and secure-by-default configuration. Three findings at MEDIUM or above severity are identified.

## Findings

### High Severity

#### ASVS-1235-HIGH-001: No SSL Context Configuration — TLS Version and Cipher Suite Not Enforced

**Severity:** HIGH

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `run_standalone`
- **Lines:** 79-87

**Related Functions Checked:**
- `run_standalone()` — TLS initialization
- `create_app()` — Application setup (no TLS controls)

**Vulnerable Code:**
```python
kwargs = {}
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
    extra_files.update((kwargs['certfile'], kwargs['keyfile']))

# Spool up the app!
app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Gap Type:** Type A — Entry point with NO control.

**Data Flow:**
`config.yaml` certfile/keyfile → `pathlib.Path` construction → raw paths passed as `kwargs` to `app.runx()` → underlying Hypercorn/Quart TLS setup uses library defaults.

**Description:**
No `ssl.SSLContext` is created. The certificate and key paths are passed directly as keyword arguments without any context that would enforce:
- Minimum TLS protocol version (e.g., `ssl.TLSVersion.TLSv1_2`)
- Cipher suite selection (e.g., disabling weak ciphers like RC4, 3DES)
- Client certificate verification (`ssl.CERT_REQUIRED`)
- Certificate revocation checking

The application relies entirely on whatever defaults the underlying ASGI server (Quart/Hypercorn) provides. While modern Python defaults to TLS 1.2+, ASVS Level 3 requires explicit, verifiable configuration — not implicit reliance on runtime defaults.

**Proof of Concept:**
```bash
# Test for TLS 1.0 acceptance (should be rejected):
openssl s_client -connect target:58383 -tls1

# Test for weak ciphers:
openssl s_client -connect target:58383 -cipher 'RC4:3DES:NULL'
```

**Impact:**
The server may negotiate deprecated TLS versions or weak cipher suites depending on the Python version and underlying library, undermining transport security guarantees for an election system. An attacker performing a downgrade attack could exploit weaker protocol negotiations.

**Remediation:**
```python
import ssl

def run_standalone():
    # ... existing logging setup ...
    app = create_app()
    
    kwargs = {}
    if app.cfg.server.certfile:
        ssl_context = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
        ssl_context.minimum_version = ssl.TLSVersion.TLSv1_2
        ssl_context.maximum_version = ssl.TLSVersion.TLSv1_3
        ssl_context.set_ciphers(
            'ECDHE+AESGCM:ECDHE+CHACHA20:DHE+AESGCM:DHE+CHACHA20:!aNULL:!MD5:!DSS:!RC4:!3DES'
        )
        cert_path = CERTS_DIR / app.cfg.server.certfile
        key_path = CERTS_DIR / app.cfg.server.keyfile
        ssl_context.load_cert_chain(certfile=cert_path, keyfile=key_path)
        
        # For mTLS (ASVS 12.3.5):
        # ssl_context.verify_mode = ssl.CERT_REQUIRED
        # ssl_context.load_verify_locations(cafile=CERTS_DIR / 'ca.pem')
        
        kwargs['ssl'] = ssl_context
    
    app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

### Medium Severity

#### ASVS-1235-MED-002: No Mutual TLS (mTLS) Capability for Service-to-Service Authentication

**Severity:** MEDIUM

**Location:**
- **Files:** `v3/server/config.yaml.example` (lines 28-31), `v3/server/main.py` (lines 79-87)
- **Function:** `run_standalone`

**Related Functions Checked:**
- `run_standalone()` — TLS configuration
- Configuration schema in `config.yaml.example`

**Vulnerable Code (config.yaml.example):**
```yaml
server:
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
    # No ca_certs, no verify_mode, no client_cert options
```

**Gap Type:** Type A — No mTLS control exists anywhere in the codebase.

**Description:**
ASVS 12.3.5 explicitly requires: *"Strong authentication methods, such as TLS client authentication, must be employed to ensure identity, using public-key infrastructure."* The configuration schema supports only server-side TLS certificates (`certfile`, `keyfile`). There is no configuration for:
- `ca_certs` or `cafile` for a trusted CA bundle to validate connecting services
- `verify_mode` to require client certificates
- Any client certificate path for outbound connections to internal services

The domain context notes the server sits behind a proxy (`# a proxy sits in front of this server`). Communication between the reverse proxy and this application server is an intra-service channel that ASVS 12.3.5 requires to be mutually authenticated. Without mTLS, any process that can reach port 58383 can impersonate a legitimate proxy.

Additionally, outbound connections to ASF OAuth (`oauth.apache.org`) are configured without any explicit certificate pinning or CA verification settings visible in the audited code.

**Impact:**
Without mutual TLS, a network-adjacent attacker could connect directly to the backend server bypassing the proxy, or a compromised co-located service could impersonate the proxy. Service identity is not cryptographically verified on either inbound or outbound connections.

**Remediation:**

**Configuration (config.yaml):**
```yaml
# config.yaml — add mTLS configuration
server:
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
    # Mutual TLS for intra-service authentication
    ca_certs: ca-chain.pem          # CA bundle for verifying client certs
    verify_client: true             # Require client certificate
```

**Code (main.py):**
```python
# main.py — apply mTLS configuration
if app.cfg.server.certfile:
    ssl_context = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
    ssl_context.load_cert_chain(
        certfile=CERTS_DIR / app.cfg.server.certfile,
        keyfile=CERTS_DIR / app.cfg.server.keyfile,
    )
    if getattr(app.cfg.server, 'ca_certs', None):
        ssl_context.verify_mode = ssl.CERT_REQUIRED
        ssl_context.load_verify_locations(cafile=CERTS_DIR / app.cfg.server.ca_certs)
    kwargs['ssl'] = ssl_context
```

#### ASVS-1235-MED-003: TLS Configuration Allows Plain HTTP as a Valid Deployment Mode

**Severity:** MEDIUM

**Location:**
- **Files:** `v3/server/config.yaml.example` (lines 28-31), `v3/server/main.py` (lines 79-87)
- **Function:** `run_standalone`

**Related Functions Checked:**
- `run_standalone()` — Server initialization
- `create_app()` — Application setup

**Vulnerable Code (config.yaml.example):**
```yaml
    # Specify the .pem files to serve using TLS, or leave these two
    # fields blank for plain HTTP.
```

**Vulnerable Code (main.py):**
```python
kwargs = {}
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
    # ...
# If certfile is blank/None, kwargs has no TLS — starts plain HTTP
app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Gap Type:** Type A — No enforcement control exists.

**Description:**
The configuration explicitly documents plain HTTP as a supported deployment mode by leaving certificate fields blank. The `main.py` conditionally enables TLS only when `certfile` is populated. For an election system processing authentication tokens and votes, running without TLS exposes all traffic (including OAuth tokens and ballot submissions) to network interception.

While the comment mentions a proxy typically sits in front, there is no validation that TLS termination actually occurs somewhere in the chain. A misconfigured deployment with blank cert fields and no TLS-terminating proxy would expose all traffic in cleartext.

**Impact:**
A deployment with blank certificate configuration transmits all data — including OAuth tokens, session cookies, and vote submissions — in plaintext, enabling passive eavesdropping and active tampering by any network-positioned attacker.

**Remediation:**

**Code (main.py):**
```python
def run_standalone():
    # ... setup ...
    app = create_app()
    
    if not app.cfg.server.certfile:
        _LOGGER.critical(
            'TLS is not configured. Set server.certfile and server.keyfile '
            'in config.yaml, or ensure a TLS-terminating proxy is in place.'
        )
        # For production, enforce TLS:
        # sys.exit(1)
    
    # ... rest of setup ...
```

**Configuration (config.yaml):**
```yaml
server:
    require_tls: true  # Set to false ONLY if behind a TLS-terminating proxy
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
```

## Positive Security Patterns

1. **OAuth endpoints use HTTPS**: The hardcoded OAuth URLs at lines 39-42 of `main.py` use `https://oauth.apache.org/...`, ensuring outbound authentication flows default to encrypted transport.

2. **Certificate directory separation**: Certificates are stored in a dedicated `certs/` subdirectory (`CERTS_DIR = THIS_DIR / 'certs'`), providing clear separation from application code.

3. **Certificate file watching**: When TLS is enabled, certificate files are added to `extra_files` for hot-reload monitoring (line 84), which supports certificate rotation without full restarts.

4. **Proxy-aware architecture**: The configuration acknowledges a proxy deployment model, which when properly configured can provide TLS termination and additional security layers.

---

## Architecture Observations

1. **Monolithic architecture**: The application is a single Quart process with file-based SQLite storage. There are no microservice-to-microservice network calls visible in the audited code, which reduces the intra-service attack surface. However, the proxy-to-app channel is an intra-service communication that must be secured.

2. **Dependency on `asfquart` library**: The TLS and OAuth handling delegate significantly to the `asfquart` library (not audited). The security posture of outbound HTTPS connections (certificate validation, pinning) is entirely determined by this dependency.

3. **Dual deployment modes**: Standalone (`main.py` direct) and ASGI (Hypercorn) modes have different TLS configuration paths. The ASGI mode (`run_asgi()`) has **no TLS configuration at all** — it defers entirely to Hypercorn's configuration, which is outside the application's control and not validated.
