The audit of ASVS 12.1.4 reveals that the application's TLS configuration lacks any certificate revocation mechanism. The server TLS setup passes only raw certificate and key file paths without creating a customized `ssl.SSLContext`, meaning OCSP Stapling is never configured for inbound connections. Additionally, no explicit certificate revocation checking is configured for outbound HTTPS connections to OAuth endpoints.

## Findings

### High Severity

#### ASVS-1214-HIGH-001: Missing OCSP Stapling Configuration in Server TLS Setup

| Attribute | Detail |
|-----------|--------|
| **Severity** | **HIGH** |
| **Gap Type** | **Type A** — Entry point with NO control |
| **File** | `v3/server/main.py` |
| **Function** | `run_standalone()` |
| **Lines** | 93–100 |
| **ASVS Requirement** | 12.1.4 — Proper certificate revocation (OCSP Stapling) enabled and configured |

**Related Functions Checked:**
- `run_standalone()` (L93-100)
- `create_app()` (L44-48)

**Description:**

The TLS setup passes only `certfile` and `keyfile` paths directly to `app.runx()` without creating a custom `ssl.SSLContext`. This means:

1. **No OCSP Stapling callback** is registered, so the server cannot provide stapled OCSP responses to connecting clients. Clients must independently query the CA's OCSP responder, which introduces latency and privacy leakage, or (more critically) many clients will skip revocation checking entirely.
2. **No control over revocation behavior** — if the application's own certificate is revoked by the CA, clients relying on default behavior may not detect this.
3. **No SSL context parameters** are set — protocol version minimums, cipher suites, and certificate verification modes all rely on framework defaults which are not auditable from this code.

**Vulnerable Code:**
```python
    kwargs = {}
    if app.cfg.server.certfile:
        kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
        kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
        extra_files.update((kwargs['certfile'], kwargs['keyfile']))

    # Spool up the app!
    app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Data Flow:**
```
config.yaml (certfile/keyfile paths) → main.py run_standalone() → app.runx() → TLS listener
                                        ↑ Missing: ssl.SSLContext with OCSP stapling callback
```

**Proof of Concept:**

If the server's TLS certificate is revoked by the CA, clients connecting to the server will continue to accept the certificate because:
1. No OCSP stapled response is provided by the server
2. Clients may skip OCSP checking due to soft-fail policies
3. The revoked certificate continues to appear valid

**Impact:**

If the server's TLS certificate is compromised and subsequently revoked by the CA, clients connecting to the server will have no reliable mechanism to learn of the revocation. The revoked certificate would continue to be accepted by clients, maintaining a false sense of trust. This directly violates ASVS 12.1.4 Level 3 requirements.

**Remediation:**

Create an explicit `ssl.SSLContext` with OCSP Stapling support and pass it to the server:

```python
import ssl

def _create_ssl_context(certfile: pathlib.Path, keyfile: pathlib.Path) -> ssl.SSLContext:
    """Create a hardened SSL context with OCSP Stapling support."""
    ctx = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
    ctx.minimum_version = ssl.TLSVersion.TLSv1_2
    ctx.load_cert_chain(certfile=str(certfile), keyfile=str(keyfile))

    # Enable OCSP Stapling — requires an OCSP response to be provided
    # In production, use a stapling daemon (e.g., via reverse proxy)
    # or set the OCSP callback:
    ctx.set_ocsp_server_callback(lambda conn, data: True)  # placeholder

    # Harden cipher selection
    ctx.set_ciphers(
        'ECDHE+AESGCM:ECDHE+CHACHA20:DHE+AESGCM:DHE+CHACHA20:!aNULL:!MD5:!DSS'
    )
    return ctx

# In run_standalone():
if app.cfg.server.certfile:
    ssl_context = _create_ssl_context(
        CERTS_DIR / app.cfg.server.certfile,
        CERTS_DIR / app.cfg.server.keyfile,
    )
    kwargs['ssl'] = ssl_context
```

For production deployments, OCSP Stapling is most effectively handled by a reverse proxy (e.g., Nginx with `ssl_stapling on; ssl_stapling_verify on;`). This should be documented as a deployment requirement.

### Medium Severity

#### ASVS-1214-MEDIUM-001: No OCSP Configuration Fields or ASGI TLS Guidance

| Attribute | Detail |
|-----------|--------|
| **Severity** | **MEDIUM** |
| **Gap Type** | **Type A** — Entry point with NO control |
| **File** | `v3/server/config.yaml.example`, `v3/server/main.py` |
| **Function** | `run_asgi()` |
| **Lines** | config.yaml.example L28-30; main.py L103-120 |
| **ASVS Requirement** | 12.1.4 — Proper certificate revocation (OCSP Stapling) enabled and configured |

**Related Functions Checked:**
- `run_asgi()` (L103-120)
- Configuration schema (config.yaml.example L28-30)

**Description:**

1. **No OCSP Stapling configuration exists anywhere** in the example configuration or documentation. The `config.yaml.example` contains only `certfile` and `keyfile` — there are no fields for OCSP responder URL, stapling file path, or any revocation-related settings.

2. **ASGI mode has zero TLS configuration** — when running under Hypercorn, TLS would need to be configured via Hypercorn's config or command-line arguments. The codebase provides no Hypercorn configuration file, no documentation of required TLS parameters, and no programmatic OCSP stapling setup.

3. **No OCSP or CRL configuration fields** are defined in the configuration schema, meaning even operators who want to enable revocation checking have no supported mechanism to do so.

**Vulnerable Code (config):**
```yaml
    # Specify the .pem files to serve using TLS, or leave these two
    # fields blank for plain HTTP. These files are relative to the
    # "certs" subdirectory of main.py (the app), or they can be absolute
    # paths to their installed location.
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
```

**Vulnerable Code (ASGI mode):**
```python
def run_asgi():
    # ...
    global steve_app
    steve_app = create_app()
    # No TLS or OCSP configuration at all
```

**Data Flow:**
```
config.yaml (no OCSP fields) → ASGI mode → Hypercorn (unconfigured) → TLS listener
                                ↑ Missing: OCSP configuration and deployment guidance
```

**Proof of Concept:**

An operator deploying in ASGI mode would:
1. Read the configuration example (no OCSP guidance)
2. Deploy with Hypercorn (no TLS configuration provided)
3. Potentially deploy without any OCSP Stapling, even if they intended to enable it

**Impact:**

Operators deploying this application have no guidance or mechanism to enable OCSP Stapling through the application's configuration. In ASGI mode (the production deployment pattern), TLS configuration is entirely absent from the application, creating a gap where OCSP Stapling could easily be overlooked during deployment.

**Remediation:**

Add OCSP-related configuration to the example config and provide deployment documentation:

```yaml
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
    
    # OCSP Stapling configuration
    # Path to OCSP stapling response file (renewed by cron/certbot)
    ocsp_staple_file: ""
    
    # Minimum TLS version (default: TLSv1.2)
    tls_minimum_version: "1.2"
```

For ASGI deployments, add a Hypercorn configuration template:

```python
# hypercorn_config.py
certfile = "certs/server.pem"
keyfile = "certs/server-key.pem"
ciphers = "ECDHE+AESGCM:ECDHE+CHACHA20:DHE+AESGCM:!aNULL:!MD5"
```

And document that the reverse proxy must be configured with OCSP Stapling:
```nginx
# nginx.conf (deployment requirement)
ssl_stapling on;
ssl_stapling_verify on;
ssl_trusted_certificate /path/to/chain.pem;
resolver 8.8.8.8;
```

#### ASVS-1214-MEDIUM-002: No Certificate Revocation Checking for Outbound OAuth Connections

| Attribute | Detail |
|-----------|--------|
| **Severity** | **MEDIUM** |
| **Gap Type** | **Type A** — Entry point with NO control |
| **File** | `v3/server/main.py` |
| **Function** | `create_app()` |
| **Lines** | 44-48 |
| **ASVS Requirement** | 12.1.4 — Proper certificate revocation enabled for outbound connections |

**Related Functions Checked:**
- `create_app()` (L44-48)
- OAuth URL initialization

**Description:**

The application makes outbound HTTPS connections to the Apache OAuth service for authentication. There is no visible configuration of certificate revocation checking (OCSP or CRL) for these outbound TLS connections. While the `asfquart` framework may handle the actual HTTP requests, the application code provides no mechanism to:

1. Enforce OCSP checking on the OAuth endpoint's certificate
2. Provide a CRL distribution point for validation
3. Configure an SSL context for outbound connections with revocation verification

If the OAuth server's certificate were compromised and revoked, the application could continue to trust and send sensitive authentication tokens to an attacker-controlled endpoint presenting the revoked certificate.

**Vulnerable Code:**
```python
    asfquart.generics.OAUTH_URL_INIT = (
        'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
    )
    asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**
```
User login → create_app() OAuth URL config → asfquart HTTP client → https://oauth.apache.org
                                                                      ↑ No cert revocation check configured
```

**Proof of Concept:**

An attacker who:
1. Compromises the OAuth server's private key
2. Performs a MITM attack on connections to oauth.apache.org
3. Presents the compromised (but now revoked) certificate

Would not be detected by the application, even after the legitimate certificate is revoked by the CA, potentially allowing interception of OAuth authorization codes and tokens.

**Impact:**

An attacker who compromises the OAuth server's private key and performs a MITM attack would not be detected even after the legitimate certificate is revoked, potentially allowing interception of OAuth authorization codes and tokens.

**Remediation:**

Configure outbound HTTPS connections with certificate revocation verification:

```python
import ssl
import certifi

def create_secure_ssl_context():
    """Create SSL context for outbound connections with revocation checking."""
    ctx = ssl.create_default_context(cafile=certifi.where())
    ctx.check_hostname = True
    ctx.verify_mode = ssl.CERT_REQUIRED
    # Enable OCSP checking where supported
    ctx.verify_flags |= ssl.VERIFY_CRL_CHECK_LEAF
    return ctx

# Pass to asfquart or underlying HTTP client
```

## Architecture Observations

1. **Reverse Proxy Assumption**: The config comment ("Typical usage is that a proxy sits in front of this server") suggests production deployments use a reverse proxy. OCSP Stapling is often handled at the proxy layer (Nginx/Apache), but this is **not documented as a requirement**.

2. **Dual Runtime Modes**: The application runs in standalone mode (direct TLS) or ASGI mode (Hypercorn). Neither mode configures OCSP Stapling, creating a gap regardless of deployment choice.

3. **Framework Dependency**: TLS behavior depends heavily on `asfquart` and its underlying server (likely Hypercorn). Without explicit SSL context creation, OCSP behavior is entirely framework-dependent and unverifiable.
