## Findings

### Medium Severity

#### ASVS-1533-MED-001: No Explicit Field Whitelist Enforcement on Form-Handling Endpoints

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `do_create_endpoint()` (line ~493), `do_add_issue_endpoint()` (line ~549), `do_edit_issue_endpoint()` (line ~572), `do_vote_endpoint()` (line ~453)

**Related Functions Checked:**
- `steve.election.Election.create()` (`election.py` line ~370)
- `election.add_issue()` (`election.py` line ~148)
- `election.edit_issue()` (`election.py` line ~166)

**Vulnerable Code:**
```python
# do_create_endpoint (~line 500)
form = edict(await quart.request.form)
election = steve.election.Election.create(DB_FNAME, form.title, result.uid)

# do_add_issue_endpoint (~line 557)
form = edict(await quart.request.form)
vtype = 'yna'
kv = None
iid = election.add_issue(form.title, form.description, vtype, kv)

# do_edit_issue_endpoint (~line 580)
form = edict(await quart.request.form)
election.edit_issue(issue.iid, form.title, form.description, issue.vtype, issue.kv)
```

**Data Flow:**
```
HTTP POST body (all fields) 
  → edict(await quart.request.form) (captures everything) 
  → form.specific_field (selective access, no whitelist rejection)
```

**Description:**

All form-handling POST endpoints capture the complete form submission into an `EasyDict` object without validating or restricting the set of allowed fields. While individual handlers currently extract only specific fields (e.g., `form.title`, `form.description`), unexpected fields are silently accepted rather than rejected. The `EasyDict` class makes any form field accessible as an attribute, meaning any code that accesses `form.attacker_field` will succeed if the attacker included it in the submission.

The code comments explicitly acknowledge that field handling is incomplete:
```python
### do better with these
vtype = 'yna'
kv = None
### for STV, there is a SEATS form parameter. Create empty LABELMAP.
```

This indicates planned expansion of accepted fields without an established whitelisting pattern to govern which fields become accepted.

**Proof of Concept:**
```http
POST /do-create-election HTTP/1.1
Content-Type: application/x-www-form-urlencoded

title=Legitimate+Election&authz=public&open_at=2025-01-01&close_at=2025-12-31
```
Currently, the extra fields (`authz`, `open_at`, `close_at`) are captured in the `form` edict but not used. However, if a developer later changes the handler to pass `form.authz` or `form.open_at` to `Election.create()`, the attacker's values would be used without additional validation—the EasyDict pattern makes this a single-line change with no compile-time or runtime guard.

**Impact:**
- **Current:** No active exploit—extra fields are ignored by current handlers
- **Structural risk:** Any future code accessing `form.<field>` immediately trusts attacker input; no systematic defense prevents mass assignment when handlers evolve
- **ASVS 15.3.3 compliance gap:** The requirement calls for "countermeasures to protect against mass assignment attacks by limiting allowed fields per controller and action"—passive non-use of extra fields is not an active countermeasure

**Remediation:**

Implement explicit field whitelisting per action. Example using a helper function:

```python
ALLOWED_FIELDS = {
    'create_election': {'title'},
    'add_issue': {'title', 'description'},
    'edit_issue': {'title', 'description'},
    'vote': set(),  # Dynamic vote-* fields handled separately
}

def extract_allowed_fields(form_data, action_name):
    """Extract only whitelisted fields from form data, rejecting unexpected fields."""
    allowed = ALLOWED_FIELDS.get(action_name, set())
    unexpected = set(form_data.keys()) - allowed
    if unexpected:
        _LOGGER.warning(f'Unexpected form fields rejected for {action_name}: {unexpected}')
        quart.abort(400, f'Unexpected fields: {", ".join(unexpected)}')
    return {k: form_data[k] for k in allowed if k in form_data}

# Usage in handler:
@APP.post('/do-create-election')
@asfquart.auth.require({R.pmc_member})
async def do_create_endpoint():
    result = await basic_info()
    raw_form = await quart.request.form
    fields = extract_allowed_fields(raw_form, 'create_election')
    election = steve.election.Election.create(DB_FNAME, fields['title'], result.uid)
```

#### ASVS-1533-MED-002: Vote Submission Handler Does Not Restrict Writable Issue IDs to Voter's Eligible Subset

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint()` (line ~453)

**Related Functions Checked:**
- `v3/steve/election.py`, method `add_vote()` (line ~216)
- `election.list_issues()` (line ~130)
- `election.q_get_mayvote.first_row()` (called from `add_vote`)

**Vulnerable Code (Handler):**
```python
# pages.py ~line 468-478
votes = {}
for key, value in form.items():
    if key.startswith('vote-'):
        iid = key.split('-', 1)[1]
        votes[iid] = value

issues = election.list_issues()
issue_dict = {i.iid: i for i in issues}  # ALL issues, not voter-eligible subset

for iid, votestring in votes.items():
    if iid not in issue_dict:  # Checks existence in election, NOT voter eligibility
        await flash_danger(f'Invalid issue ID: {iid}')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
    try:
        election.add_vote(result.uid, iid, votestring)
```

**Vulnerable Code (Model):**
```python
# election.py ~line 216-228
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    ### validate VOTESTRING for ISSUE.TYPE voting    # <-- acknowledged missing validation
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)  # AttributeError if mayvote is None
```

**Data Flow:**
```
Form keys (vote-<attacker_controlled_iid>) 
  → handler validates iid exists in election (not voter-eligible) 
  → add_vote(pid, iid, votestring) 
  → q_get_mayvote.first_row(pid, iid) returns None for ineligible voter 
  → None.salt raises AttributeError 
  → caught by generic except Exception
```

**Description:**

The vote handler accepts `vote-*` form fields where the issue ID portion is entirely user-controlled from the form key name. The handler validates the issue ID against ALL issues in the election (`issue_dict`), not the subset the voter is eligible for. The actual eligibility check is in the model's `add_vote()` method, but it manifests as an `AttributeError` when accessing `.salt` on a `None` mayvote result—not as an explicit authorization decision.

This creates two problems:
1. **Gap Type B (Control exists but relied upon via exception):** The mayvote check exists but isn't called explicitly—eligibility enforcement happens as a side effect of `None` attribute access
2. **Partial batch processing:** The loop processes votes sequentially with early return on error. If an attacker submits legitimate votes first, then an ineligible vote, the legitimate votes are committed but later votes in the batch are not, leaving the voter in a partial state

**Proof of Concept:**
```http
POST /do-vote/abc123 HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: session=<valid_session>

vote-eligible_issue_1=yes&vote-eligible_issue_2=no&vote-INELIGIBLE_issue=yes
```

If `INELIGIBLE_issue` exists in the election but the voter isn't eligible for it:
- `vote-eligible_issue_1=yes` → committed to database ✅
- `vote-eligible_issue_2=no` → committed to database ✅
- `vote-INELIGIBLE_issue=yes` → `add_vote` → `mayvote=None` → `AttributeError` → generic error → redirect

Result: votes 1 and 2 are persisted, vote 3 is rejected, and the user sees a generic "Error submitting vote" message without understanding why.

**Impact:**
- The ineligible vote is NOT recorded (the model-layer check prevents the write)
- Legitimate votes submitted before the failure in the same batch ARE persisted without user awareness
- The controller layer does not limit which issue IDs (fields) are valid per the specific voter's authorization, violating the ASVS 15.3.3 principle of limiting allowed fields per action
- Error handling reveals that the voter is not eligible for a specific issue (information disclosure)

**Remediation:**

Pre-filter eligible issue IDs at the controller level before processing any votes:

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    form = edict(await quart.request.form)
    
    # Pre-load voter's eligible issues
    election.q_find_issues.perform(result.uid, election.eid)
    eligible_iids = {row.iid for row in election.q_find_issues.fetchall()}
    
    votes = {}
    for key, value in form.items():
        if key.startswith('vote-'):
            iid = key.split('-', 1)[1]
            if iid not in eligible_iids:
                _LOGGER.warning(
                    f'User[U:{result.uid}] submitted vote for ineligible issue[I:{iid}]'
                )
                await flash_danger(f'You are not eligible to vote on issue {iid}.')
                return quart.redirect(f'/vote-on/{election.eid}', code=303)
            votes[iid] = value
    
    # Process only validated votes (consider wrapping in transaction)
    for iid, votestring in votes.items():
        election.add_vote(result.uid, iid, votestring)
    
    await flash_success('Votes submitted successfully!')
    return quart.redirect('/voter', code=303)
```

Additionally, add explicit eligibility check in `add_vote`:
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    if mayvote is None:
        raise VoterNotEligible(pid, iid)  # Explicit exception, not AttributeError
    # ... rest of method
```

## Architecture Observations

1. **Two-layer defense pattern**: The application provides mass assignment protection at both the handler layer (explicit field extraction) and the model layer (explicit method parameters). This defense-in-depth is effective but informal.

2. **EasyDict pattern risk**: Using `EasyDict` as the form data container makes any form field accessible as a Python attribute. While currently fields are explicitly selected, this pattern doesn't fail-safe—accessing an attacker-supplied field name succeeds silently if it's in the form data.

3. **Incomplete TODO items**: Multiple `### check authz` comments indicate that authorization checks are not yet implemented in management endpoints. While not mass assignment per se, the absence of ownership verification means any committer can manage any election's issues, expanding the attack surface for field manipulation.

4. **Vote type validation gap**: The `add_vote` method includes the comment `### validate VOTESTRING for ISSUE.TYPE voting`, acknowledging that vote content validation is not yet implemented. While not mass assignment, this means vote fields accept arbitrary string values rather than being constrained to valid vote type values (yes/no/abstain for YNA, valid rankings for STV).
