## Findings

### Medium Severity

#### ASVS-681-MED-001: User Identity Model Lacks IdP Namespacing Despite Multi-IdP Capable Framework

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/main.py`
- **Lines:** 43-47
- **Function:** `create_app()`
- **Related Files:** `v3/server/pages.py` (lines 80-95), `v3/steve/election.py` (throughout)

**Related Functions Checked:**
- `pages.py:basic_info()` (lines 80-95)
- `election.py:add_vote()` (lines 184-196)
- `election.py:create()` (line 295)
- `election.py:open_to_pid()` (lines 308-317)
- `election.py:owned_elections()` (lines 321-330)

**Description:**

The application's identity model uses bare user identifiers (`uid`/`pid`) without IdP namespacing throughout the entire authentication and authorization flow. While currently configured with a single OAuth provider, the underlying `asfquart` framework explicitly supports OIDC multi-IdP authentication, which has been deliberately disabled. This architectural gap means that re-enabling OIDC or adding a second IdP would immediately introduce identity spoofing vulnerabilities with no code-level protection.

**Vulnerable Code:**

`main.py` — Framework OIDC support explicitly overridden:
```python
# Avoid OIDC
import asfquart.generics

asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

`pages.py` — Session identity extraction without IdP namespace:
```python
s = await asfquart.session.read()
if s:
    basic.update(
        uid=s['uid'],       # Bare user ID, no IdP identifier
        name=s['fullname'],
        email=s['email'],
    )
```

`election.py` — Voter identification by bare `pid`:
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)  # pid has no IdP namespace
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
```

**Data Flow:**

```
OAuth session['uid'] (bare string, e.g., "jdoe")
  → pages.py basic_info() → result.uid
    → election.add_vote(result.uid, ...)   # pid = "jdoe" — no IdP context
    → Election.open_to_pid(DB_FNAME, result.uid)  # pid = "jdoe"
    → Election.owned_elections(DB_FNAME, result.uid)  # pid = "jdoe"
    → pdb.get_person(result.uid)  # pid = "jdoe"
```

**Analysis:**

The `asfquart` framework natively supports OIDC (OpenID Connect), which is the standard protocol for federated authentication with multiple identity providers. The comment `# Avoid OIDC` and the explicit URL overrides in `main.py` confirm the framework has multi-IdP capability that has been deliberately disabled in favor of a single Apache OAuth provider.

The entire identity model throughout the application — session handling, PersonDB, election ownership, voter eligibility, and vote recording — uses a bare `uid`/`pid` string with **no IdP identifier or namespace component**. This means:

1. **No IdP namespace in session:** `s['uid']` is extracted and used directly as the user's identity. No `s['idp']` or provider identifier is stored or checked.

2. **No IdP namespace in PersonDB:** The `person` table stores users by `pid` alone. If user `"jdoe"` exists at IdP-A and a different `"jdoe"` exists at IdP-B, they would collide.

3. **No IdP namespace in election system:** `owner_pid`, voter `pid` in `mayvote`, and all authorization checks use bare identifiers.

4. **No IdP namespace in vote tokens:** `crypto.gen_vote_token(opened_key, pid, iid, salt)` uses bare `pid`, meaning vote tokens from different IdPs for the same `uid` would be identical.

While the single-IdP configuration currently prevents exploitation, the ASVS 6.8.1 standard mitigation (combining IdP ID + user ID) is entirely absent from the architecture. If OIDC is re-enabled (as the framework supports) or another IdP is added, identity spoofing becomes immediately possible with no code-level protection.

**Proof of Concept (Hypothetical — requires OIDC re-enablement):**

If OIDC were enabled and a second IdP (e.g., corporate Azure AD) were configured:

1. Attacker registers at IdP-B with username `"jdoe"` (matching a legitimate user at IdP-A)
2. Attacker authenticates via IdP-B, creating session `{uid: "jdoe", ...}`
3. Application treats attacker as the same user: `result.uid = "jdoe"`
4. Attacker can vote in elections assigned to the legitimate `"jdoe"`, manage their elections, etc.

**Impact:**

- **Identity collision** if multiple IdPs are enabled
- **Vote manipulation** (voting as another user)
- **Election management takeover** (managing elections owned by another user with the same `uid`)
- The entire cryptographic vote anonymization scheme relies on `pid` integrity — if `pid` is spoofed, vote tokens are compromised

**Remediation:**

Store and use a composite identity key combining the IdP identifier and user ID:

```python
# In session creation/reading (pages.py basic_info):
s = await asfquart.session.read()
if s:
    idp_id = s.get('idp', 'apache-oauth')  # IdP namespace
    raw_uid = s['uid']
    composite_uid = f"{idp_id}:{raw_uid}"  # Namespaced identity
    basic.update(
        uid=composite_uid,
        name=s['fullname'],
        email=s['email'],
    )
```

```python
# In election.py - ensure pid always includes IdP namespace:
def add_vote(self, pid: str, iid: str, votestring: str):
    # pid should be "idp_id:user_id" format
    assert ':' in pid, "pid must include IdP namespace"
    ...
```

```sql
-- PersonDB schema should include IdP:
CREATE TABLE person (
    idp TEXT NOT NULL,        -- Identity provider identifier
    idp_uid TEXT NOT NULL,    -- User ID within the IdP
    pid TEXT GENERATED ALWAYS AS (idp || ':' || idp_uid) STORED,
    name TEXT,
    email TEXT,
    PRIMARY KEY (idp, idp_uid)
);
```

## Positive Security Patterns

1. **Single IdP Configuration (Current):** The explicit override to a single OAuth provider in `main.py` prevents multi-IdP spoofing *in the current deployment*. This is an effective runtime mitigation even though the code lacks architectural protection.

2. **Parameterized Queries:** All database operations use parameterized queries through `asfpy.db`, preventing SQL injection of `pid` values.

3. **Vote Token Derivation:** `crypto.gen_vote_token(opened_key, pid, iid, salt)` cryptographically binds the voter identity to the vote, which would correctly namespace votes IF `pid` were namespaced.

4. **Tamper Detection:** `is_tampered()` includes voter `pid` values in the election data hash via `gather_election_data()`, providing integrity protection for the voter list.

5. **Role-Based Access Control:** Authentication decorators enforce role requirements (`R.committer`, `R.pmc_member`), adding a layer beyond identity matching.

---

## Architecture Observations

1. **Framework vs. Application Gap:** The `asfquart` framework has OIDC capabilities (multi-IdP by design), but the application identity model is single-IdP only. This creates a dangerous gap where enabling a framework feature would silently introduce identity spoofing.

2. **Flat Identity Model:** The entire system — from session through PersonDB to election/voting — uses a single string `pid`/`uid` as the user identity. There is no concept of identity provenance at any layer.

3. **Comment-Based Security Controls:** Multiple `### check authz` comments throughout `pages.py` indicate authorization logic is incomplete. While not directly related to 6.8.1, the absence of proper ownership verification (e.g., any committer can open/close any election) compounds the identity risk.

---

## Recommendations Summary

### Immediate Actions

1. **Add IdP namespace to user identity model** — Implement composite identity keys (IdP ID + user ID) throughout the application to prevent identity collision across providers. This is the foundational architectural change required for ASVS 6.8.1 compliance.

2. **Store IdP identifier in session and PersonDB** — Ensure identity provenance is tracked at all layers of the authentication and authorization stack.

### Short-term Actions

3. **Update all `pid`/`uid` references to use composite key** — Systematically refactor all identity references in `pages.py`, `election.py`, and database schemas to use namespaced identifiers.

4. **Implement IdP allowlist validation** — Add configuration and validation logic to ensure only approved IdPs can provide identities, preventing unauthorized provider registration.

### Long-term Actions

5. **Add integration test for cross-IdP identity isolation** — Create comprehensive test suite to verify identity separation holds when multiple IdPs are configured, preventing regression when OIDC is re-enabled.
