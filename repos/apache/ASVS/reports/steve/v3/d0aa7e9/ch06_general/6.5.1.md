### `v3/steve/election.py` — Vote Token Reuse Analysis

The `add_vote()` method at **line 223** generates a `vote_token` that is deterministic for a given `(opened_key, pid, iid, salt)` tuple:

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

This token is deliberately **reusable** — voters may update their vote (re-vote), and tallying uses `q_recent_vote` to fetch the latest. This is a voting system design choice, not an authentication token, and is **not in scope for ASVS 6.5.1**.

## Findings

**No findings at MEDIUM severity or above were identified for ASVS 6.5.1.**

### Rationale for No Findings

1. **No MFA mechanisms exist in the audited code.** ASVS 6.5.1 is a conditional requirement — it applies when lookup secrets, TOTP, or OOB codes are present. None are implemented.

2. **Authentication is fully delegated** to the `asfquart` framework (Apache Software Foundation's Quart-based auth), which is not included in the audit scope.

3. **Vote tokens and cryptographic salts** used in the election system are not authentication tokens and are not subject to ASVS 6.5.1 single-use requirements.

4. The `vote_token` reusability in `election.py` is intentional (allows vote updates) and is not an authentication mechanism.

## Architecture Observations

### 1. MFA Delegation to External Framework

Authentication, including any MFA, is handled entirely by `asfquart.auth`. The application has no visibility into or control over whether MFA is enforced. If `asfquart` does not implement MFA, or if its MFA tokens are replayable, the application would be non-compliant with ASVS 6.5.1 despite having no ability to remediate within its own codebase.

**Recommendation:** Verify that the `asfquart` / ASF authentication infrastructure enforces MFA with single-use TOTP/OOB codes. This requires a separate audit of that component.

### 2. Session Integrity

The session read at `pages.py:68-82` trusts whatever `asfquart.session.read()` returns. If the underlying session framework doesn't bind sessions to MFA completion status, a session established without MFA could access all protected routes.

### 3. No Step-Up Authentication for Sensitive Operations

Critical operations like opening/closing elections (`/do-open/<eid>`, `/do-close/<eid>`) and vote submission (`/do-vote/<eid>`) use the same authentication level as viewing pages. There is no step-up authentication or re-authentication for elevated-risk actions.

## Recommendations Summary

### Immediate Actions

| Priority | Recommendation | Rationale |
|----------|----------------|-----------|
| 1 | **Audit the `asfquart` authentication framework** for MFA implementation and single-use token enforcement | ASVS 6.5.1 compliance cannot be determined without examining the auth provider |
| 2 | **Verify MFA completion in session data** — ensure routes check that the session was established with completed MFA, not just valid credentials | Prevents session established via password-only from accessing protected routes |

### Short-term Actions

| Priority | Recommendation | Rationale |
|----------|----------------|-----------|
| 3 | **Consider step-up authentication** for election lifecycle operations (open, close, delete) | These are irreversible high-impact operations that warrant additional verification |

### Long-term Actions

| Priority | Recommendation | Rationale |
|----------|----------------|-----------|
| 4 | **Document the MFA trust boundary** — explicitly state that MFA enforcement is delegated to the ASF auth infrastructure and what guarantees are expected | Ensures the security assumption is visible and testable |
