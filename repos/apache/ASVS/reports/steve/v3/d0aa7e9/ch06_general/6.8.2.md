## Findings

### Medium Severity

#### ASVS-682-MED-001: Authentication Assertion Signature Validation Unverifiable — Entirely Delegated to Unaudited External Library

**Severity:** MEDIUM

**Location:** 
- **File:** `v3/server/pages.py`
- **Functions:** All authenticated routes; `basic_info()` L65-80
- **Related Dependency:** `asfquart` (not provided in audit scope)

**Related Functions Checked:**
- `basic_info()` — Session data retrieval (L65-80)
- `do_vote_endpoint()` — Vote submission using session identity (L302-303)
- All route handlers with `@asfquart.auth.require` decorators

**Description:**

The application delegates 100% of its authentication assertion validation to the `asfquart` framework. No code in the audited codebase validates the presence or integrity of digital signatures on authentication assertions from the ASF Identity Provider. There is no JWT parsing, no SAML signature verification, no JWKS endpoint configuration, and no public key or certificate configuration visible anywhere in the audited files.

**Vulnerable Code:**

```python
# pages.py L65-80 — Session data trusted without any visible signature verification
async def basic_info():
    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],        # Trusted as authenticated identity
            name=s['fullname'],  # Trusted from IdP assertion
            email=s['email'],    # Trusted from IdP assertion
        )
```

All authorization decisions flow from this trusted session data:

```python
# pages.py L302-303 — Vote submission uses uid from unverified-at-app-level session
async def do_vote_endpoint(election):
    result = await basic_info()
    # ...
    election.add_vote(result.uid, iid, votestring)  # uid determines voter identity
```

**Data Flow:**

```
IdP Assertion (JWT/SAML) 
  → asfquart [NOT AUDITED: signature validation assumed here]
    → asfquart.session.read() [opaque session store]
      → basic_info() reads uid/name/email [NO app-level verification]
        → uid used for all authorization: voting, election management, ownership
```

**What's Missing from Auditable Code:**

1. No JWT library imports (`pyjwt`, `python-jose`, etc.)
2. No SAML library imports (`python3-saml`, `pysaml2`, etc.)
3. No JWKS/certificate configuration for IdP signature verification
4. No OAuth token exchange or callback endpoint
5. No assertion claim validation (issuer, audience, expiration, nonce)
6. No check that assertions are signed (reject unsigned assertions)

**Proof of Concept:**

If the `asfquart` framework contains any deficiency in assertion validation, an attacker could exploit it through the following attack vectors:

1. **Algorithm Confusion Attack:** Submit JWT with `alg: none` if `asfquart` doesn't reject unsigned assertions
2. **Signature Bypass:** Submit assertion with invalid signature if verification is not enforced
3. **Issuer Spoofing:** Submit assertion from untrusted issuer if issuer validation is missing
4. **Expired Token Reuse:** Replay old assertions if expiration validation is insufficient

**Impact:**

If the `asfquart` framework contains any deficiency in assertion validation—such as accepting unsigned JWTs, not validating the `alg` header (algorithm confusion attack), not verifying issuer/audience claims, or misconfiguring JWKS—the entire application's authentication and authorization model would be bypassed. An attacker could:

- **Forge voter identity** to cast votes as any eligible person
- **Impersonate election owners** to open/close/modify elections
- **Access any authenticated endpoint** with fabricated identity

The election system's security model fundamentally depends on correct voter identity, making this an especially critical dependency.

**Remediation:**

1. **Include `asfquart` in audit scope:** The `asfquart.auth` and `asfquart.session` modules MUST be audited for ASVS 6.8.2 compliance since they contain the actual assertion validation logic.

2. **Add defense-in-depth assertion claim validation at the application layer:**

```python
async def basic_info():
    s = await asfquart.session.read()
    if s:
        # Defense-in-depth: validate critical session claims
        if not s.get('uid') or not s.get('email'):
            raise AuthenticationError("Incomplete authentication assertion")
        
        # Verify assertion freshness (if timestamp available)
        if 'auth_time' in s:
            max_age = 86400  # 24 hours
            if time.time() - s['auth_time'] > max_age:
                raise AuthenticationError("Authentication assertion expired")
        
        # Verify expected issuer (if available)
        if 'iss' in s and s['iss'] not in TRUSTED_ISSUERS:
            raise AuthenticationError("Untrusted assertion issuer")
```

3. **Document IdP configuration requirements** including JWKS endpoint, expected algorithm, issuer, and audience values so deployments can be verified.

4. **Implement assertion validation logging** to create an audit trail of authentication events and signature verification results.

## Positive Security Patterns

1. **Consistent use of auth decorators:** All state-changing and sensitive routes use `@asfquart.auth.require` with appropriate role requirements (`R.committer`, `R.pmc_member`). No authenticated routes are missing the decorator.

2. **Session-based identity propagation:** The application correctly passes the authenticated `uid` from session data through to business logic rather than accepting user-supplied identity values in request parameters.

3. **Separation of concerns:** Authentication is handled at the framework/middleware layer via decorators, not in business logic code, which is architecturally sound.

4. **Graceful handling of missing sessions:** `basic_info()` properly handles the case where no session exists, defaulting to `None` values and allowing public pages to render without authentication.

5. **Role-based access tiering:** The application distinguishes between session-only access (profile/settings), committer access (voting/management), and PMC member access (election creation).

## Architecture Observations

### Single Point of Trust

The `asfquart` framework is the sole gatekeeper for authentication assertion validation. There is no secondary verification, no assertion caching with signature re-validation, and no application-level claim inspection. This creates a single point of failure for authentication.

### No IdP Metadata Visible

The audited codebase contains no IdP configuration (JWKS endpoints, signing certificates, expected issuers, allowed algorithms). These may be in `asfquart`'s configuration or `config.yaml` (not provided), but their absence from the audit makes compliance verification impossible.

### Election Integrity Model vs. Authentication Model Gap

The election data integrity is carefully protected with Argon2-based tamper detection (`is_tampered()`), but the authentication assertion integrity relies entirely on an external, unaudited library. The security rigor applied to election data should extend to authentication verification.

### Framework Dependency Risk

The application's entire authentication security posture depends on the correct implementation and configuration of `asfquart`. This dependency should be:
- Explicitly documented
- Regularly audited
- Version-pinned with security update monitoring
- Supplemented with application-layer validation
