### Summary

The apache/steve voting system demonstrates strong cryptographic practices overall, with no traditional MFA lookup secrets detected. All stored sensitive values use Argon2 hashing with 128-bit salts, far exceeding ASVS 6.5.2 requirements. However, one medium-severity finding was identified: the production code uses Argon2d (Type.D) instead of the recommended Argon2id (Type.ID) variant, creating potential side-channel vulnerabilities in shared hosting environments.

## Findings

### Medium Severity

#### ASVS-652-MED-001: Production Argon2 Uses Type.D (Argon2d) Instead of Recommended Type.ID (Argon2id)

| Attribute | Value |
|-----------|-------|
| **Severity** | MEDIUM |
| **Gap Type** | Type B — Correct variant exists in benchmark code but is NOT used in production hashing |
| **File** | `v3/steve/crypto.py` |
| **Function** | `_hash()`, line 79–88 |
| **Callers affected** | `gen_opened_key()` (line 40), `gen_vote_token()` (line 49) |

**Vulnerable Code:**
```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2 to hash the data, with default tuning parameters."
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,  # ← Argon2d: side-channel vulnerable
    )
```

**Benchmark code (correct variant, NOT used in production):**
```python
# In benchmark_argon2(), line ~130:
argon2.low_level.hash_secret_raw(
    ...
    type=argon2.low_level.Type.ID,  # ← Argon2id: recommended variant
)
```

**Data Flow:**
- `election.py:open()` → `crypto.gen_opened_key(edata, salt)` → `_hash(digest, salt)` → **Argon2d** → stored as `election.opened_key`
- `election.py:add_vote()` → `crypto.gen_vote_token(opened_key, pid, iid, salt)` → `_hash(...)` → **Argon2d** → stored as `vote.vote_token`
- `election.py:tally_issue()` → same chain for vote lookup/decryption
- `election.py:is_tampered()` → same chain for integrity verification

**Issue:**

ASVS 6.5.2 requires an "approved password storage hashing algorithm." While Argon2 as a family qualifies, the specific variant matters:

- **Argon2d** (Type.D): Data-dependent memory access. Vulnerable to side-channel attacks (cache-timing, memory access pattern analysis). Designed for settings where side-channel attacks are impossible.
- **Argon2id** (Type.ID): Hybrid mode combining Argon2i (first pass, side-channel resistant) and Argon2d (subsequent passes, memory-hard). Universally recommended by OWASP Password Storage Cheat Sheet, RFC 9106, and OWASP ASVS 2.4.4 which states: *"Verify that if argon2id is not available, that bcrypt, scrypt, or PBKDF2 is used."*

The benchmark function (`benchmark_argon2()`) at line ~130 correctly uses `Type.ID`, demonstrating awareness of the recommended variant, but the production `_hash()` function uses `Type.D`.

**Impact:**
- In cloud/shared hosting (typical deployment), a co-located attacker could exploit cache-timing side-channels against Argon2d to recover information about hashed values
- Compromised `vote_token` values could enable vote de-anonymization
- Compromised `opened_key` could undermine election integrity detection
- This is a systemic issue affecting ALL Argon2 hashing operations in the application

**Proof of Concept:**

A co-located process on the same physical host monitors L1/L2 cache access patterns during Argon2d execution. Because Argon2d's memory access pattern depends on the secret data, the attacker can extract bits of the hashed secret through differential timing analysis, unlike Argon2id which uses data-independent access patterns in its first pass.

**Remediation:**
```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2id to hash the data, with default tuning parameters."
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.ID,  # Argon2id: recommended by OWASP/RFC 9106
    )
```

## Positive Security Patterns

1. **Strong Salt Generation**: All salts use `secrets.token_bytes(16)` (128 bits), far exceeding the ASVS 6.5.2 minimum of 32 bits. Salt uniqueness is enforced per voter-issue pair.

2. **Argon2 for All Sensitive Hashing**: Despite the variant concern, the use of Argon2 (a memory-hard function with configurable parameters) for both `opened_key` and `vote_token` derivation is appropriate. Parameters (time_cost=2, memory_cost=64MB, parallelism=4) are reasonable for server-side operation.

3. **Key Hierarchy Design**: The multi-layer key derivation (BLAKE2b → Argon2 → HKDF → Fernet) provides excellent separation of concerns and domain separation. Each layer serves a distinct purpose.

4. **Schema-Level Cryptographic Field Validation**: CHECK constraints in `schema.sql` enforce salt/hash lengths at the database level, preventing storage of improperly sized cryptographic values.

5. **Vote Token as Stored Hash**: The `vote_token` stored in the `vote` table is the Argon2 *output*, not a raw secret. The original identifying information (pid, iid) is not stored alongside the vote, preserving voter anonymity. The input entropy (>256 bits via `opened_key`) far exceeds the 112-bit threshold where a standard hash would suffice.

6. **Transaction Safety for Salt Assignment**: The `add_salts()` method uses explicit `BEGIN TRANSACTION`/`COMMIT` to ensure atomic salt assignment for all voters, preventing partial-state issues.

7. **Deterministic Election Data Assembly**: `gather_election_data()` uses repeatable ordering for hash input construction, ensuring `opened_key` verification is reliable.

## Architecture Observations

1. **No Traditional MFA Lookup Secrets Detected**: The codebase does not implement traditional lookup secrets (recovery codes, TANs, grid codes) as described in ASVS 6.5.2. The closest analogs are vote tokens, which are server-computed Argon2 hashes with >112-bit input entropy, making them compliant even with a standard hash.

2. **Election/Issue IDs as URL Tokens**: `create_id()` produces 40-bit entropy identifiers (10 hex chars). While explicitly designed for non-enumerability rather than authentication, if ballot links rely solely on these IDs for access scoping (even post-LDAP-auth), 40 bits provides ~1 trillion combinations. This is adequate for anti-enumeration but would be insufficient if ever used as standalone authentication tokens. The web layer (not provided) should be audited for how ballot links are constructed and validated.

3. **Salt Reuse Across Algorithms**: The same `mayvote.salt` is used for both Argon2 token generation and HKDF key derivation. Domain separation is provided by different algorithms and HKDF's `info` parameter (`b'xchacha20_key'`), which is cryptographically sound per HKDF specification. No action needed.

4. **Ballot Link Mechanism Not Auditable**: The `get_voters_for_email()` method suggests ballot links are sent to voters, but the link construction and token generation code is not in the provided files. If ballot links contain short-lived one-time tokens, those would fall under ASVS 6.5.2 and should be verified separately.
