## Findings

### MEDIUM Severity

#### Finding ASVS-637-SEV-001: No User Notification When Person Details (Email/Name) Are Modified

| Attribute | Details |
|-----------|---------|
| **Finding ID** | ASVS-637-SEV-001 |
| **Severity** | MEDIUM |
| **Gap Type** | Type A — Entry point with NO control |
| **File** | `v3/steve/persondb.py` |
| **Function** | `add_person` |
| **Lines** | 46–51 |
| **Related Functions** | `v3/steve/election.py:get_voters_for_email` (lines 510–516) |
| **ASVS Requirement** | 6.3.7 (Level 3) |

**Description:**

The `add_person` method performs an upsert operation that can silently modify a user's email address without any notification to either the old or new email address. Email addresses are security-sensitive authentication details used for election-related communications, making silent modifications a security risk.

**Vulnerable Code:**

```python
# v3/steve/persondb.py lines 46-51
def add_person(self, pid, name, email):
    "Add or update a Person designated by PID."

    # If we ADD, then SALT will be NULL. If we UPDATE, then it will not
    # be touched (it should be NULL).
    self.c_add_person.perform(pid, name, email)
```

**Data Flow:**

```
CLI/Admin tool → PersonDB.add_person(pid, name, new_email)
    → c_add_person (upsert query) → DB updated
    → ❌ NO notification to user at old email
    → ❌ NO notification to user at new email
    → ❌ NO audit log entry generated
```

**Technical Analysis:**

1. **Email is a security-sensitive field**: `election.py:get_voters_for_email()` (line 510) uses stored emails for election-related communications
2. **Silent changes enable redirection attacks**: A silently changed email could redirect election notifications to an attacker
3. **No change detection**: The method doesn't check if values actually changed before updating
4. **No audit trail**: No logging exists in `add_person`, preventing administrative oversight

**Proof of Concept:**

```python
pdb = PersonDB.open(db_path)
# Silently change voter's email - NO notification sent
pdb.add_person("victim_pid", "Victim Name", "attacker@evil.com")
# Now get_voters_for_email() returns attacker's email for this voter
# Legitimate user receives NO notification of the change
```

**Impact:**

- ❌ **ASVS 6.3.7 Violation**: Users are not notified when authentication details (email) are modified
- 🔓 **Silent Email Takeover**: Compromised admin could redirect election notifications without user awareness
- 📝 **No Audit Trail**: Person detail modifications leave no log entries for security review
- ⚠️ **Level 3 Non-Compliance**: Fails to meet ASVS Level 3 requirements for authentication detail change notifications

**Remediation:**

```python
# v3/steve/persondb.py
import logging

_LOGGER = logging.getLogger(__name__)

def add_person(self, pid, name, email):
    "Add or update a Person designated by PID."
    
    # Check if this is an update (person already exists)
    existing = self.q_get_person.first_row(pid)
    
    self.c_add_person.perform(pid, name, email)
    
    if existing:
        old_name, old_email = existing.name, existing.email
        changes = []
        if old_email != email:
            changes.append(f'email: {old_email} → {email}')
        if old_name != name:
            changes.append(f'name: {old_name} → {name}')
        
        if changes:
            _LOGGER.info(
                f'Person[P:{pid}] details updated: {", ".join(changes)}'
            )
            # Send notification to BOTH old and new email addresses
            self._notify_detail_change(pid, old_email, email, changes)
    else:
        _LOGGER.info(f'Person[P:{pid}] created: {name}, {email}')

def _notify_detail_change(self, pid, old_email, new_email, changes):
    """Notify user at both old and new email about detail changes.
    
    Per ASVS 6.3.7, users must be notified after updates to 
    authentication details.
    """
    # Implementation depends on email infrastructure
    # Must notify old_email to alert legitimate user
    # Must notify new_email to confirm the change
```

## Architecture Observations

### 1. Delegated Authentication Model

**Authentication is delegated to ASF OAuth** — The application does not manage credentials directly, which reduces the scope of ASVS 6.3.7 for credential resets. However, the email address stored in `persondb` IS used for election communications and can be silently modified, making it fall under "authentication details" for notification purposes.

### 2. Missing Notification Infrastructure

**No email/notification infrastructure exists** — There is no email sending module, no notification queue, and no message templates for security alerts. The `get_voters_for_email()` method in `election.py` suggests email-based communication is planned or in use, but there's no general notification capability for security events.

### 3. CLI Tool Access Pattern

**CLI tools operate without notification** — Direct database access via CLI tools is intentional for administrative purposes. However, the lack of logging in `add_person` means even administrative actions lack an audit trail, which is a gap for security monitoring.

### 4. Profile Management Evolution

**Profile pages exist but are incomplete** — The `/profile` and `/settings` routes indicate planned user-facing profile management. The current GET-only implementation should be extended with notification infrastructure before POST handlers are added.
