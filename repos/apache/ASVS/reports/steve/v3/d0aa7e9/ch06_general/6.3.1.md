## Findings

### High Severity

#### ASVS-631-HIGH-001: No Brute-Force or Credential Stuffing Protection on Authentication Flow

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control  

**Location:**
- **File:** `v3/server/main.py`
- **Lines:** 36–44
- **Function:** `create_app()`
- **Related Files:** `v3/server/pages.py` (entire file, all `@asfquart.auth.require` decorated endpoints)

**Related Functions Checked:**
- `create_app()` in `main.py`
- All authenticated endpoint handlers in `pages.py`
- OAuth callback handlers (framework-level)

**Description:**

The application delegates credential verification to ASF OAuth (`oauth.apache.org`) but implements **zero** local controls against authentication abuse at the application boundary. Specifically:

1. **No rate limiting on OAuth flow initiation**: An attacker can repeatedly trigger the OAuth redirect flow without any throttling. While credential validation happens externally, the application generates `state` parameters and redirects without limit, enabling automated credential stuffing attempts through the application as a proxy.

2. **No monitoring of failed authentication callbacks**: When the OAuth callback returns a failure or an attacker replays/forges callback attempts, there is no detection, logging, or throttling mechanism.

3. **No documentation of brute-force mitigation strategy**: The security documentation (`ARCHITECTURE.md`, lines 116–120) states only:
   ```
   - **Authentication**: External (ASF OAuth); no local auth.
   - **Authorization**: Role-based (committer, PMC member) via decorators.
   ```
   There is no documented strategy for brute-force prevention, no reference to NIST SP 800-63B § 5.2.2 (Rate Limiting/Throttling), and no mention of delegation of this responsibility to the OAuth provider with verification that the provider meets the requirements.

4. **No session creation throttling**: After OAuth callback, sessions are created without any rate limiting, enabling rapid session enumeration or replay attempts.

**Vulnerable Code:**

`main.py`, lines 36–44:
```python
def create_app():
    import asfquart.generics

    asfquart.generics.OAUTH_URL_INIT = (
        'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
    )
    asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'

    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    # No rate limiting middleware configured
    # No brute-force protection configured
```

**Data Flow:**
- **Source:** Unauthenticated HTTP requests to initiate OAuth or authenticated requests to protected endpoints
- **Sink:** OAuth provider redirect / session creation / authenticated action execution
- **Missing Control:** No rate limiting, throttling, account lockout, or failed-attempt monitoring at any layer

**Proof of Concept:**

```bash
# Automated OAuth flow initiation — no rate limiting prevents rapid attempts
for i in $(seq 1 1000); do
  curl -s -o /dev/null -w "%{http_code}" \
    "https://steve.apache.org/oauth/init" &
done
# All 1000 requests processed without throttling or blocking
```

**Impact:**

An attacker can conduct credential stuffing attacks using the application's OAuth flow as a gateway without detection or throttling. Even though credential validation is external, the application serves as an amplification point and provides no defense-in-depth at its boundary. Successful credential stuffing could grant unauthorized access to the voting system.

**NIST Reference:** NIST SP 800-63B § 5.2.2 requires:
> *"The verifier SHALL implement controls to protect against online guessing attacks [...] unless it implements an approved authentication protocol that demonstrates proof of possession of the authenticator output [...] The verifier SHALL limit consecutive failed authentication attempts on a single account to no more than 100."*

**Remediation:**

Implement rate limiting middleware at the application level:

```python
# main.py - Add rate limiting middleware
from quart_rate_limiter import RateLimiter, rate_limit

def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    # Apply global rate limiter
    rate_limiter = RateLimiter(app, default_limits=[
        rate_limit(300, timedelta(minutes=1)),  # Global: 300 req/min per IP
    ])
    
    import pages
    import api
    return app
```

```python
# pages.py - Add specific rate limits to auth-adjacent endpoints
from quart_rate_limiter import rate_limit
from datetime import timedelta

# Rate limit the OAuth callback (if handler is exposed)
@APP.route('/oauth/callback')
@rate_limit(10, timedelta(minutes=1))  # 10 attempts per minute per IP
async def oauth_callback():
    ...
```

Additionally, implement failed authentication attempt logging:
```python
# Log failed authentication attempts for monitoring
@APP.errorhandler(401)
async def handle_unauthorized(error):
    client_ip = quart.request.remote_addr
    _LOGGER.warning(f'Failed auth attempt from IP={client_ip}, path={quart.request.path}')
    # Increment counter; block IP after threshold
    return 'Unauthorized', 401
```

### Medium Severity

#### ASVS-631-MED-001: No Throttling on Vote Submission Endpoint

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with NO control  

**Location:**
- **File:** `v3/server/pages.py`
- **Lines:** 290–323
- **Function:** `do_vote_endpoint()`

**Related Functions Checked:**
- `add_vote()` in `v3/steve/election.py` (line 265)
- `crypto.gen_vote_token()` (called from `add_vote()`)
- `crypto.create_vote()` (called from `add_vote()`)

**Description:**

The vote submission endpoint has no throttling mechanism. While the known false positive pattern states that general per-route rate limiting absence is intentional and directs auditors to specifically check "whether vote submission has appropriate throttling," no such throttling exists.

An authenticated attacker (or compromised account) could:
1. Submit rapid automated vote changes to create timing side-channels
2. Flood the endpoint to cause resource exhaustion (each vote triggers cryptographic operations: `crypto.gen_vote_token()` + `crypto.create_vote()`)
3. Abuse the "last vote wins" behavior to race-condition vote manipulation

The `add_vote()` method in `election.py` (line 265) performs multiple cryptographic operations per call without any throttling:

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

**Vulnerable Code:**

`pages.py`, lines 290–323:
```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    ### check authz   # <-- Authorization check is a placeholder comment

    # Parse the form data
    form = edict(await quart.request.form)
    if not form:
        await flash_danger('No vote data provided.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    # Extract votes from form (keys like 'vote-<iid>')
    votes = {}
    for key, value in form.items():
        if key.startswith('vote-'):
            iid = key.split('-', 1)[1]
            votes[iid] = value

    # Process each vote — no rate limiting, no duplicate submission check
    for iid, votestring in votes.items():
        # ...
        election.add_vote(result.uid, iid, votestring)
```

**Data Flow:**
- **Source:** Authenticated user submitting `POST /do-vote/<eid>` with form data
- **Sink:** `election.add_vote()` → database write → encrypted vote storage
- **Missing Control:** No rate limiting, no submission cooldown, no duplicate detection at the HTTP layer

**Proof of Concept:**

```bash
# Rapid vote submission flooding — no rate limit prevents abuse
SESSION_COOKIE="session=<stolen_or_legitimate_session>"
for i in $(seq 1 500); do
  curl -s -X POST "https://steve.example.org/do-vote/ELECTION_ID" \
    -H "Cookie: $SESSION_COOKIE" \
    -d "vote-ISSUE_ID=yes" &
done
# 500 concurrent vote submissions processed, each triggering crypto operations
```

**Impact:**

Resource exhaustion through cryptographic operation amplification. Each vote submission triggers Argon2 key derivation and Fernet encryption, which are computationally expensive by design. Without throttling, an attacker can consume significant server resources, potentially degrading service for legitimate voters during active elections.

**Remediation:**

Add endpoint-specific rate limiting for the vote submission endpoint:

```python
from quart_rate_limiter import rate_limit
from datetime import timedelta

@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@rate_limit(5, timedelta(minutes=1))  # Max 5 vote submissions per minute per user
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    
    # Add submission cooldown check
    last_vote_time = await get_last_vote_timestamp(result.uid, election.eid)
    if last_vote_time and (datetime.datetime.now() - last_vote_time).seconds < 10:
        await flash_warning('Please wait before resubmitting your vote.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
    
    # ... rest of vote processing
```

## Positive Security Patterns

1. **External Authentication Delegation**: The application correctly delegates credential management to ASF OAuth over HTTPS, avoiding local password storage entirely.

2. **Role-Based Access Control Decorators**: Consistent use of `@asfquart.auth.require({R.committer})` and `@asfquart.auth.require({R.pmc_member})` across all protected endpoints ensures authentication is checked before handler execution.

3. **Parameterized Database Queries**: All SQL operations use prepared statements via `asfpy.db`, eliminating SQL injection vectors.

4. **Cryptographic Vote Protection**: Vote encryption uses per-voter salts with Argon2 key derivation, providing strong protection of vote content.

5. **Election State Machine**: Assertions enforce valid state transitions (`is_editable()`, `is_open()`, `is_closed()`), preventing operations in invalid states.

6. **Tamper Detection**: `is_tampered()` method verifies election integrity by recomputing the opened key from current data and comparing against the stored hash.

7. **Comprehensive Audit Logging**: User actions (vote, create, open, close, edit, delete) are logged with user identifiers for accountability.

## Architecture Observations

1. **Authentication Boundary Gap**: The application treats the external OAuth provider as a complete solution for authentication security but implements no defense-in-depth at its own boundary. NIST SP 800-63B § 5.2.2 requires rate limiting regardless of where credential verification occurs — the relying party (this application) must implement throttling controls.

2. **No Security Documentation for Brute-Force**: The `ARCHITECTURE.md` security section does not address credential stuffing or brute-force prevention strategies. ASVS 6.3.1 explicitly requires controls "according to the application's security documentation," but no such documentation exists for this threat category.

3. **Compute-Expensive Operations Unprotected**: The vote submission path involves Argon2 key derivation per request, making it a high-value target for resource exhaustion attacks. This is particularly concerning given the absence of any rate limiting.

4. **Authorization Gap Amplifies Risk**: Multiple endpoints have `### check authz` placeholder comments (lines 101, 293, 337, 355, 366, 380, 394, 405), meaning even authenticated users may access elections they shouldn't. Combined with no rate limiting, this creates a larger attack surface.
