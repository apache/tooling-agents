# ASVS 6.8.4 Security Audit Report

## Executive Summary

**Repository:** apache/steve  
**Audit Date:** Apr 22, 2026  
**Auditor:** Tooling Agents  
**ASVS Requirement:** 6.8.4 — Verify that, if an application uses a separate Identity Provider (IdP) and expects specific authentication strength, methods, or recentness for specific functions, the application verifies this using the information returned by the IdP. This audit evaluates the Apache STeVe v3 voting system against ASVS 6.8.4, which requires verification of authentication strength, methods, and recentness when using an external Identity Provider (IdP). The application delegates authentication entirely to Apache OAuth but never validates any authentication metadata from the IdP, nor does it implement a documented fallback approach as required when such information is unavailable.

## Findings

### High Severity

#### ASVS-684-HIGH-001: No Verification of IdP Authentication Strength or Method for Any Operation

**Severity:** HIGH

**Location:**
- **File:** `v3/server/main.py`
- **Lines:** 40-44
- **Function:** OAuth configuration
- **Related Files:** `v3/server/pages.py` (lines 56-83)

**Related Functions Checked:**
- `basic_info()` (`pages.py:56-83`)
- `do_vote_endpoint()` (`pages.py:367-407`)
- `do_create_endpoint()` (`pages.py:409-429`)
- `do_open_endpoint()` (`pages.py:433-452`)
- `do_close_endpoint()` (`pages.py:454-472`)

**Description:**

The application uses Apache OAuth as its external Identity Provider, configured with custom (non-OIDC) endpoints. It never requests, receives, or validates any authentication strength metadata (equivalent to OIDC claims `acr`, `amr`, `auth_time`) from the IdP. This applies to **all** endpoints, including highly sensitive voting and election lifecycle operations.

**Vulnerable Code:**

OAuth Configuration (source of authentication):
```python
# main.py lines 40-44
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

Session data consumed (no auth metadata):
```python
# pages.py lines 72-80
s = await asfquart.session.read()
if s:
    basic.update(
        uid=s['uid'],
        name=s['fullname'],
        email=s['email'],
    )
```

**Data Flow:**
```
Apache OAuth IdP → session (uid, fullname, email only) → @asfquart.auth.require (role check only) → sensitive operation executed
                                                          ↑
                                              No acr/amr/auth_time extraction or validation
```

**Affected Endpoints (all authenticated routes):**

| Endpoint | Operation | Auth Decorator | Auth Strength Check |
|----------|-----------|----------------|---------------------|
| `POST /do-vote/<eid>` | Cast vote | `R.committer` | ❌ None |
| `POST /do-create-election` | Create election | `R.pmc_member` | ❌ None |
| `GET /do-open/<eid>` | Open election | `R.committer` | ❌ None |
| `GET /do-close/<eid>` | Close election | `R.committer` | ❌ None |
| `POST /do-delete-issue/<eid>/<iid>` | Delete issue | `R.committer` | ❌ None |
| `POST /do-set-open_at/<eid>` | Set open date | `R.committer` | ❌ None |
| `POST /do-set-close_at/<eid>` | Set close date | `R.committer` | ❌ None |

**Impact:**
- In a voting system, the inability to verify *how* a user authenticated means the application cannot distinguish between a user who authenticated with multi-factor authentication and one who used only a username/password (or a session that was hijacked/replayed).
- An attacker who gains access via a weaker authentication path (e.g., stolen password without MFA enforcement at the IdP) can perform all operations—including casting votes and managing elections—without the application detecting or preventing it.
- Since Apache OAuth (not OIDC) is used, no ID Token with standard claims is available, which per ASVS 6.8.4 specifically requires a **documented fallback approach assuming minimum strength authentication was used**. No such documentation or compensating control exists in the codebase or `ARCHITECTURE.md`.

**Proof of Concept:**
A user authenticates to Apache OAuth using only a username and password (no MFA). The session is created with `uid`, `fullname`, `email`. The user then navigates to `/do-vote/<eid>` and casts a vote. At no point does the application verify whether the authentication was single-factor or multi-factor, recent or stale.

**Remediation:**

**Option A — If the IdP can provide authentication metadata (preferred):**
```python
# In session creation callback or basic_info():
async def basic_info():
    basic = edict()
    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
            # Extract authentication metadata from IdP response
            auth_strength=s.get('acr', 'urn:oasis:names:tc:SAML:2.0:ac:classes:Password'),
            auth_methods=s.get('amr', ['pwd']),
            auth_time=s.get('auth_time'),
        )
    # ...
    return basic

def require_strong_auth(result):
    """Enforce minimum authentication strength for sensitive operations."""
    # Check authentication method
    if 'mfa' not in result.get('auth_methods', []):
        quart.abort(403, 'Multi-factor authentication required for this operation')
    # Check recentness (within 5 minutes for sensitive operations)
    if result.auth_time:
        age = datetime.datetime.now().timestamp() - result.auth_time
        if age > 300:  # 5 minutes
            quart.abort(403, 'Recent authentication required. Please re-authenticate.')
```

**Option B — Documented fallback when IdP cannot provide metadata (ASVS 6.8.4 minimum):**
```python
# Add to ARCHITECTURE.md and implement in code:
"""
## Authentication Strength Fallback Policy

Apache OAuth does not provide acr/amr/auth_time claims. Per ASVS 6.8.4,
the application assumes MINIMUM strength authentication (single-factor,
username/password) for all sessions.

Compensating controls:
1. Session lifetime limited to 30 minutes for voting operations
2. Re-authentication required before casting votes
3. Election management restricted to verified PMC members via separate channel
"""

# Implement session age check as compensating control
async def require_recent_session(max_age_seconds=1800):
    """Require session to have been created within max_age_seconds."""
    s = await asfquart.session.read()
    if not s or 'created_at' not in s:
        quart.abort(401, 'Session expired. Please re-authenticate.')
    session_age = datetime.datetime.now().timestamp() - s['created_at']
    if session_age > max_age_seconds:
        quart.abort(401, 'Session too old for this operation. Please re-authenticate.')
```

### Medium Severity

#### ASVS-684-MEDIUM-001: No Authentication Recentness Check for State-Changing Election Operations

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/pages.py`
- **Lines:** 367, 433, 454
- **Functions:** `do_vote_endpoint()`, `do_open_endpoint()`, `do_close_endpoint()`

**Related Functions Checked:**
- `do_vote_endpoint()` (`pages.py:367-407`)
- `do_open_endpoint()` (`pages.py:433-452`)
- `do_close_endpoint()` (`pages.py:454-472`)
- `basic_info()` (`pages.py:56-83`)

**Description:**

Critical state-changing operations that alter election lifecycle (open, close, vote, create) perform no verification of when the user last authenticated. A session established hours or days earlier can be used to perform irreversible operations like casting votes or closing elections.

**Vulnerable Code:**

```python
@load_election
async def do_vote_endpoint(election):

    # ...
    election.add_vote(result.uid, iid, votestring)
```

```python

@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):

    # No session freshness check before irreversible state change

```

```python
# pages.py line 454 - Close election (irreversible)
@APP.get('/do-close/<eid>')

@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    # No session freshness check before irreversible state change
```

**Data Flow:**
```
Stale session (created hours/days ago) → basic_info() → role check passes → irreversible operation executes
                                          ↑
                                   No freshness verification
```

**Impact:**
- A stale or long-lived session (potentially from a compromised browser, shared workstation, or session replay) can be used to perform irreversible election operations.
- For a voting system, this undermines the assurance that the person casting the vote is the legitimate user and was actively present at the time of voting.

**Proof of Concept:**
1. User authenticates and receives a session
2. User leaves browser open on a shared workstation
3. Hours later, another person uses the still-active session to cast votes or manipulate election state
4. No recentness check prevents this abuse

**Remediation:**
```python
# Add session timestamp at login and verify before sensitive operations
SENSITIVE_OPS_MAX_AGE = 300  # 5 minutes

async def verify_session_freshness(max_age=SENSITIVE_OPS_MAX_AGE):
    """Verify session was established recently enough for sensitive operations."""
    s = await asfquart.session.read()
    if not s:
        quart.abort(401)
    
    auth_time = s.get('auth_time') or s.get('session_created')
    if not auth_time:
        _LOGGER.warning(f'Session for {s["uid"]} has no auth_time - assuming stale')
        quart.abort(401, 'Re-authentication required for this operation')
    
    age = datetime.datetime.now().timestamp() - auth_time
    if age > max_age:
        quart.abort(401, 'Session too old. Please re-authenticate to continue.')

# Apply to sensitive endpoints
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    await verify_session_freshness()  # Require recent auth
    result = await basic_info()
    # ... proceed with vote
```

## Positive Security Patterns

1. **Role-based authorization decorators** — The application consistently applies `@asfquart.auth.require({R.committer})` or `@asfquart.auth.require({R.pmc_member})` to all sensitive endpoints, providing a baseline authorization layer.

2. **Elevated privilege for election creation** — `do_create_endpoint` requires `R.pmc_member` (higher than committer), showing awareness of privilege separation for administrative operations.

3. **Voter eligibility verification** — `vote_on_page()` (line 243-248) checks `mayvote` entries to verify the authenticated user is eligible to vote in the specific election.

4. **State machine enforcement** — `election.py` uses `assert self.is_editable()` / `self.is_open()` guards before state-changing operations, preventing out-of-order lifecycle transitions.

5. **Comprehensive security logging** — All sensitive operations log user identity and action (e.g., `f'User[U:{result.uid}] voted on issue[I:{iid}]'`).

---

## Architecture Observations

1. **Non-OIDC OAuth**: The application uses Apache's custom OAuth endpoints (`oauth.apache.org`), not standard OIDC. This means no ID Token with standard `acr`/`amr`/`auth_time` claims is available. ASVS 6.8.4 explicitly addresses this scenario: a documented fallback approach assuming minimum-strength authentication is required.

2. **No Step-Up Authentication**: The application has no mechanism for requiring stronger authentication for more sensitive operations. All committer-level operations (including irreversible ones like voting and election state changes) use the same authentication gate.

3. **Session contains only identity, not authentication context**: The session (`uid`, `fullname`, `email`) captures *who* the user is but not *how* or *when* they authenticated. This architectural gap makes ASVS 6.8.4 compliance impossible without changes to either the IdP integration or compensating controls.

---

## Recommendations Summary

### Immediate Actions (High Priority)

1. **Document and implement a fallback authentication strength policy** as required by ASVS 6.8.4 when the IdP cannot provide `acr`/`amr`/`auth_time`. At minimum, add to `ARCHITECTURE.md` the assumption that all sessions use single-factor authentication, and implement compensating controls (session lifetime limits, re-authentication for sensitive operations).

2. **Implement session freshness verification** for irreversible operations (voting, opening/closing elections). Store `auth_time` or session creation timestamp and enforce a maximum age before permitting sensitive actions.

### Short-term Actions (Medium Priority)

3. **Investigate whether Apache OAuth can provide authentication metadata** (e.g., whether MFA was used). If available, capture this in the session and enforce minimum authentication levels for sensitive operations. If migrating to OIDC is feasible, standard claims would directly satisfy this requirement.

4. **Consider implementing step-up authentication** for election lifecycle operations (open/close) separate from general committer access, since these are irreversible actions with high impact on election integrity.

### Long-term Actions

5. **Evaluate migration to OIDC-compliant IdP** to enable native support for authentication context claims (`acr`, `amr`, `auth_time`) and streamline compliance with ASVS 6.8.4 and similar security standards.
