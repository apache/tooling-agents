## Findings

### Medium Severity

#### ASVS-638-MED-001: Differential Response in `/admin` Reveals PersonDB Registration Status

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `admin_page`
- **Lines:** ~297-310

**Related Functions Checked:**
- `basic_info()` — Session data retrieval
- `steve.persondb.PersonDB.open()` — Database access
- `pdb.get_person()` — User lookup
- `raise_404()` — Error response handler

**Description:**

When an authenticated ASF committer visits `/admin`, the application checks whether they exist in the PersonDB (the STeVe voter/person database). Two distinct responses are returned:

| Condition | HTTP Status | Template | Page Title | Processing Time |
|-----------|------------|----------|------------|-----------------|
| Person in PersonDB | 200 | `admin.ezt` | "Administration" | Longer (additional queries for owned elections, postprocessing) |
| Person NOT in PersonDB | 404 | `e_bad_pid.ezt` | "Unknown Person" | Shorter (immediate abort) |

This creates three observable differentiators:
1. **HTTP response code**: 200 vs 404
2. **Page content/template**: Full admin page vs. "Unknown Person" error
3. **Timing**: Successful path executes additional DB queries and template processing

**Vulnerable Code:**

```python
pdb = steve.persondb.PersonDB.open(DB_FNAME)
try:
    me = pdb.get_person(result.uid)
except steve.persondb.PersonNotFound:
    ### the authn'd committer is not in "person"

    result = await basic_info()
    result.title = 'Unknown Person'
    result.pid = result.uid
    # Note: result.uid (and friends) are needed for the navbar.
    raise_404(T_BAD_PID, result)
    # NOTREACHED
```

**Data Flow:**

```
Authenticated Session (result.uid)
    ↓
PersonDB.get_person(result.uid)
    ↓
├─ Found → HTTP 200 + admin.ezt template + additional DB queries
└─ Not Found → HTTP 404 + e_bad_pid.ezt template + immediate abort
```

**Gap Classification:** Type A — Entry point with no user-enumeration control

**Proof of Concept:**

```http
# Authenticated committer NOT in PersonDB:
GET /admin HTTP/1.1
Cookie: session=<valid_committer_session>

→ HTTP/1.1 404 Not Found
→ Content: "Unknown Person" page (T_BAD_PID template)

# Authenticated committer IN PersonDB:
GET /admin HTTP/1.1
Cookie: session=<valid_committer_session>

→ HTTP/1.1 200 OK
→ Content: Full admin dashboard (admin.ezt template)
```

**Impact:**

At ASVS Level 3, this differential response reveals whether an authenticated ASF committer is registered in the STeVe PersonDB (voter database). While the impact is limited because:
- It requires prior ASF OAuth authentication
- It only reveals the authenticated user's own registration status
- PersonDB is a secondary database (not the authentication database)

...at Level 3, even self-status information leakage through differential responses violates the principle of consistent error handling. In environments where PersonDB registration status is operationally sensitive (e.g., revealing who has been provisioned as a voter before an election is announced), this could be meaningful.

**Remediation:**

Return a consistent response regardless of PersonDB status. Either show a "setup required" page with the same HTTP 200 code, or handle the missing person case gracefully within the normal admin template:

```python
@APP.get('/admin')
@asfquart.auth.require({R.committer})
@APP.use_template(TEMPLATES / 'admin.ezt')
async def admin_page():
    result = await basic_info()
    result.title = 'Administration'

    with asfpy.stopwatch.Stopwatch():
        election = steve.election.Election.open_to_pid(DB_FNAME, result.uid)
        owned = steve.election.Election.owned_elections(DB_FNAME, result.uid)

    result.owned = [postprocess_election(e) for e in owned]

    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    try:
        me = pdb.get_person(result.uid)
        for e in result.owned:
            e.owner_name = me[0]
        result.person_registered = True
    except steve.persondb.PersonNotFound:
        # Return a consistent 200 response with a "not yet registered" state
        # visible in the template, rather than a 404 error page
        result.person_registered = False

    result.len_election = len(election)
    result.len_owned = len(owned)

    return result
```

## Positive Security Patterns

### 1. Consistent Error Template Reuse in `vote_on_page`

**Location:** `v3/server/pages.py`, line ~257

```python
raise_404(T_BAD_EID, result)  # Reuse template for consistency
```

The developer explicitly noted the intent to reuse the `T_BAD_EID` template for consistency when a user is not eligible, rather than creating a distinct "not eligible" error page. This is a good practice for preventing information leakage, though the different `result.title` value partially undermines it.

### 2. Generic Vote Submission Error Messages

**Location:** `v3/server/pages.py`, lines ~405-408

```python
except Exception as e:
    _LOGGER.error(f'Error adding vote for user[U:{result.uid}] on issue[I:{iid}]: {e}')
    await flash_danger(f'Error submitting vote for issue {iid}.')
```

Detailed error information is logged server-side only, while the user receives a generic error message that doesn't reveal the specific failure reason (e.g., eligibility vs. crypto error vs. database error).

### 3. Document Access Returns Generic 404

**Location:** `v3/server/pages.py`, lines ~578-581

```python
row = db.q_get_mayvote.first_row(result.uid, iid)
if not row:
    # Nothing fancy. Just pretend the file does not exist
    quart.abort(404)
```

The developer deliberately returns 404 (pretending the resource doesn't exist) rather than 403 (revealing the resource exists but access is denied), preventing information leakage about document existence.

### 4. Session-Based User Identification

All user-related checks use `result.uid` from the authenticated session rather than accepting user identifiers as URL parameters or form inputs. This prevents one authenticated user from probing information about another user.

### 5. Election Existence Checks Use Consistent Templates

**Location:** `v3/server/pages.py`, `load_election` and `load_election_issue` decorators

Both decorators use the same `T_BAD_EID`/`T_BAD_IID` template and 404 status code for not-found resources, providing consistent responses.

## Architecture Observations

### 1. External Authentication Dependency

Authentication is entirely delegated to ASF OAuth (`asfquart.auth`). ASVS 6.3.8 compliance for the login flow itself **cannot be verified** from this codebase. The external authentication system must be independently audited for user enumeration protections.

### 2. No Registration or Forgot Password Flows

The application has no user-facing registration or password reset functionality. Voter registration is performed through administrative processes (CLI tools, PersonDB management). This eliminates two of the three enumeration vectors specified in ASVS 6.3.8.

### 3. Two-Layer User Database

The system has two user databases — the external ASF OAuth system (authentication) and the internal PersonDB (authorization/voter management). The differential response in `admin_page` reveals status in the secondary database, not the primary authentication database.

### 4. Session-Only User Identification

All post-authentication user checks derive the user identifier from the server-side session (`result.uid`), never from client-supplied parameters. This architectural decision effectively prevents cross-user enumeration.
