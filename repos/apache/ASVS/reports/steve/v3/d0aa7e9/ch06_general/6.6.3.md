**Key Findings:**

This application delegates authentication entirely to external ASF OAuth and does not implement a traditional code-based out-of-band (OOB) authentication mechanism. However, the system communicates election access URLs to voters via email (an out-of-band channel), and these URLs contain identifiers with only 40 bits of entropy and no rate limiting on lookup endpoints — creating a tangible enumeration risk that falls within the scope of ASVS 6.6.3's protective intent.

## Findings

### Medium Severity

#### Finding ASVS-663-SEV-001

**Title:** Election/Issue Identifiers Have Only 40 Bits of Entropy — Below 64-Bit Recommendation

**Severity:** MEDIUM

**Location:**
- **File:** `v3/steve/crypto.py`
- **Function:** `create_id()`
- **Line:** ~110

**Related Functions Checked:**
- `Election.create()` (line ~367, `election.py`) — generates `eid`
- `Election.add_issue()` (line ~182, `election.py`) — generates `iid`
- `get_voters_for_email()` (line ~430, `election.py`) — sends EIDs via email

**Vulnerable Code:**
```python
def create_id():
    "Create a standard ID value."
    # Use 10 hex characters for the ID
    return secrets.token_hex(5)  # 5 bytes
```

**Data Flow:**
1. `create_id()` generates a 5-byte (40-bit) hex token
2. Used by `Election.create()` (line ~367, `election.py`) to generate `eid`
3. Used by `Election.add_issue()` (line ~182, `election.py`) to generate `iid`
4. EIDs are communicated to voters via email (`get_voters_for_email()`, line ~430, `election.py`)
5. EIDs used in all URL routes: `/vote-on/<eid>`, `/manage/<eid>`, `/do-vote/<eid>`, etc.
6. IIDs used in routes: `/manage-stv/<eid>/<iid>`, `/docs/<iid>/<docname>`, etc.

**Description:**

The `get_voters_for_email()` method and the architecture documentation confirm voters are contacted via email with ballot links containing EIDs. This constitutes an out-of-band communication of access codes. The ASVS requirement explicitly recommends "a code with at least 64 bits of entropy" for codes used in OOB mechanisms. The current 40-bit (5-byte) identifiers provide only ~1.1 trillion possible values.

**Schema Confirmation** (`v3/schema.sql`):
```sql
eid  TEXT PRIMARY KEY NOT NULL
    CHECK (length(eid) = 10
           AND eid GLOB '[0-9a-f]...')
```

**Impact:**

An authenticated attacker (any ASF committer) could attempt to enumerate valid EIDs to discover elections they are not authorized to view. With 40-bit entropy and no rate limiting, a systematic scan could discover valid elections. The `manage_page` endpoint (line ~306, `pages.py`) lacks owner-specific authorization checks (`### check authz` comment present but not implemented), meaning discovered elections could be viewed.

**Proof of Concept:**
```python
import itertools, aiohttp, asyncio

async def enumerate_elections():
    async with aiohttp.ClientSession(cookies=auth_cookies) as session:
        # 40-bit space = ~1.1 trillion, but active elections are sparse
        # Targeted prefix scanning or timing attacks narrow the space
        for candidate in generate_hex_candidates(length=10):
            async with session.get(f'https://target/manage/{candidate}') as resp:
                if resp.status != 404:
                    print(f'Found election: {candidate}')
```

**Remediation:**

Increase identifier entropy to at least 64 bits (8 bytes / 16 hex characters) to align with ASVS 6.6.3 guidance:

```python
def create_id():
    "Create a standard ID value."
    # Use 16 hex characters for 64-bit entropy (ASVS 6.6.3)
    return secrets.token_hex(8)  # 8 bytes = 64 bits
```

Update the schema CHECK constraint accordingly:
```sql
eid TEXT PRIMARY KEY NOT NULL
    CHECK (length(eid) = 16
           AND eid GLOB '[0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f]')
```

#### Finding ASVS-663-SEV-002

**Title:** No Rate Limiting on Resource Identifier Endpoints — Brute Force Enumeration Unprotected

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** All `@load_election` and `@load_election_issue` decorated endpoints

**Affected Endpoints:**

| Endpoint | Line | Decorator |
|----------|------|-----------|
| `GET /vote-on/<eid>` | ~217 | `@load_election` |
| `GET /manage/<eid>` | ~306 | `@load_election` |
| `POST /do-vote/<eid>` | ~362 | `@load_election` |
| `GET /do-open/<eid>` | ~418 | `@load_election` |
| `GET /do-close/<eid>` | ~436 | `@load_election` |
| `GET /docs/<iid>/<docname>` | ~536 | Direct IID lookup |

**Related Functions Checked:**
- `load_election` decorator (line ~161, `pages.py`)
- `load_election_issue` decorator (line ~180, `pages.py`)
- All endpoints utilizing these decorators

**Vulnerable Code:**

`load_election` decorator, line ~161:
```python
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            result = await basic_info()
            result.title = 'Unknown Election'
            result.eid = eid
            raise_404(T_BAD_EID, result)
        # No rate limiting, no delay, no tracking of failed attempts
        return await func(e)
    return loader
```

**Data Flow:**
1. Attacker authenticates via ASF OAuth (valid committer account)
2. Sends rapid requests to `GET /manage/<eid>` with sequential/random EID guesses
3. Server responds 404 for invalid EIDs, 200 for valid ones — providing an oracle
4. No rate limiting, account lockout, or exponential backoff exists
5. With 40-bit EID space, systematic enumeration is tractable

**Description:**

Despite the application requiring authentication for all sensitive endpoints, no brute-force protection mechanism exists anywhere in the codebase. The absence of rate limiting on election/issue lookup endpoints means an authenticated attacker can systematically probe for valid identifiers without restriction. For an L2 application, ASVS 6.6.3 explicitly requires rate limiting as a defense against brute force of out-of-band codes.

**Impact:**

Combined with Finding ASVS-663-SEV-001 (40-bit entropy), the absence of rate limiting means an authenticated attacker can systematically discover valid election IDs. The `manage_page` endpoint contains a `### check authz` comment indicating missing authorization checks, allowing discovered elections to be accessed without proper ownership validation.

**Remediation:**

**Option 1:** Implement per-user rate limiting on election/issue lookup endpoints:

```python
from quart_rate_limiter import RateLimiter, rate_limit

rate_limiter = RateLimiter(APP)

@APP.get('/manage/<eid>')
@asfquart.auth.require({R.committer})
@rate_limit(10, timedelta(minutes=1))  # 10 requests/minute per IP
@load_election
async def manage_page(election):
    ...
```

**Option 2:** Implement custom tracking of failed EID lookups per session with exponential backoff:

```python
async def loader(eid):
    s = await asfquart.session.read()
    uid = s['uid'] if s else 'anonymous'
    
    if await is_rate_limited(uid, 'eid_lookup'):
        quart.abort(429)
    
    try:
        e = steve.election.Election(DB_FNAME, eid)
    except steve.election.ElectionNotFound:
        await record_failed_lookup(uid, 'eid_lookup')
        raise_404(T_BAD_EID, result)
```

## Positive Security Patterns

1. **Strong cryptographic primitives**: Argon2 for key derivation, HKDF-SHA256 for key stretching, Fernet for authenticated encryption — all properly parameterized.

2. **`secrets` module used throughout**: All random values (salts, IDs, shuffling) use cryptographically secure randomness via the `secrets` module, ensuring cryptographic-quality entropy sources.

3. **Vote tokens are 256-bit**: The `gen_vote_token()` output (32 bytes) far exceeds the 64-bit entropy recommendation for vote-level operations, providing strong protection against brute force attacks on individual votes.

4. **Salts are 128-bit**: All `gen_salt()` calls produce 16-byte (128-bit) salts, exceeding the 64-bit minimum and preventing rainbow table attacks.

5. **Tamper detection**: `is_tampered()` recomputes election data hash via Argon2 to detect post-opening modifications, ensuring election integrity.

6. **Parameterized queries**: All SQL uses prepared statements via `asfpy.db`, preventing SQL injection vulnerabilities.

7. **Authenticated encryption**: Vote data uses Fernet (AES-128-CBC + HMAC-SHA256) providing both confidentiality and integrity protection.

## Architecture Observations

1. **Authentication is fully externalized**: ASF OAuth handles all user authentication. The application has no local authentication code, no password storage, and no OOB authentication mechanism of its own. ASVS 6.6.3 is therefore **not directly applicable** in its strictest interpretation.

2. **Election IDs serve as de facto access tokens**: Despite OAuth authentication, EIDs function as "something you know" communicated out-of-band (email). The combination of 40-bit entropy + no rate limiting + incomplete authorization (`### check authz`) creates a meaningful enumeration risk.

3. **The vote submission path has stronger protections**: The `add_vote()` method requires the election to be open and the user to have a `mayvote` entry, providing defense-in-depth even if an EID is guessed. The critical voting operation is better protected than the management operations.

4. **Trust boundary at ASF OAuth**: The application trusts ASF's OAuth implementation completely. All authorization decisions assume authenticated users are legitimate ASF committers, with no additional verification or suspicious activity monitoring.
