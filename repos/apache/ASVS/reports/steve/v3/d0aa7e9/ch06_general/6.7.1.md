**Summary:**  
ASVS 6.7.1 requires that certificates used to verify cryptographic authentication assertions are stored in a way that protects them from modification. The audit analyzed the provided codebase — `main.py` (application entry point), `quickstart.md`, and `schema.md` — with a focus on how certificates and cryptographic authentication materials are stored and loaded.

The application uses OAuth over HTTPS for authentication. TLS certificates protecting this authentication channel are loaded from a regular filesystem directory within the application tree without any integrity verification. The codebase lacks mechanisms to detect certificate modification, enforce restrictive permissions, or validate certificate identity before use.

## Findings

### Medium Severity

#### ASVS-671-MED-001: TLS Certificates Loaded Without Integrity Verification

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with NO control  
**File:** `v3/server/main.py`  
**Function:** `run_standalone()`  
**Lines:** 85-90  

**Related Functions Checked:**
- `run_standalone()` — Certificate loading logic
- Application initialization flow from line 52
- Certificate path resolution at lines 85-86

**Description:**  
The TLS certificate and private key files — which protect the OAuth authentication channel — are loaded directly from the filesystem without any integrity verification. There is no hash comparison, fingerprint validation, or signature check to ensure certificates have not been tampered with. An attacker with write access to the `server/certs/` directory could substitute a rogue certificate and key, enabling man-in-the-middle interception of the OAuth authentication flow.

**Vulnerable Code:**
```python
CERTS_DIR = THIS_DIR / 'certs'  # line 37

# lines 85-90 in run_standalone()
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
    extra_files.update((kwargs['certfile'], kwargs['keyfile']))

app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Data Flow:**  
`filesystem (server/certs/)` → `pathlib join with config value` → `passed to TLS server startup` → **no integrity check at any point**

**Impact:**  
If an attacker gains write access to the `certs/` directory (e.g., through a compromised deployment pipeline, shared hosting, or misconfigured permissions), they can replace the certificate/key pair. This would:
1. Allow MITM attacks against users authenticating via OAuth
2. Enable interception of OAuth tokens during the redirect/callback flow
3. Undermine the entire authentication system without detection

**Proof of Concept:**
```bash
# Attacker with write access to server/certs/ generates a rogue certificate
openssl req -x509 -newkey rsa:2048 -keyout server/certs/server.key \
  -out server/certs/server.crt -days 365 -nodes -subj "/CN=localhost"

# Server restarts (or auto-reloads since certs are in extra_files watch list)
# and now serves attacker-controlled TLS certificate
# OAuth redirect_uri traffic can be intercepted
```

**Note on auto-reload risk:** The certificates are explicitly added to the `extra_files` watch set (line 88), meaning the server will automatically reload when certificate files change on disk. This amplifies the risk — a certificate swap triggers immediate adoption without manual restart.

**Remediation:**
```python
import hashlib

EXPECTED_CERT_FINGERPRINT = "sha256:abc123..."  # Store in separate, protected config

def verify_certificate_integrity(cert_path, expected_fingerprint):
    """Verify certificate file matches expected fingerprint before use."""
    with open(cert_path, 'rb') as f:
        cert_data = f.read()
    actual = "sha256:" + hashlib.sha256(cert_data).hexdigest()
    if actual != expected_fingerprint:
        raise RuntimeError(
            f"Certificate integrity check failed for {cert_path}. "
            f"Expected {expected_fingerprint}, got {actual}"
        )
    return cert_path

# In run_standalone():
if app.cfg.server.certfile:
    cert_path = CERTS_DIR / app.cfg.server.certfile
    key_path = CERTS_DIR / app.cfg.server.keyfile
    verify_certificate_integrity(cert_path, app.cfg.server.cert_fingerprint)
    kwargs['certfile'] = cert_path
    kwargs['keyfile'] = key_path
```

Additionally:
- Enforce restrictive file permissions (`0o400` for key, `0o444` for cert) at startup
- Store certificate fingerprints in a separate, integrity-protected configuration
- Consider removing certificates from `extra_files` to prevent automatic reload on modification

#### ASVS-671-MED-002: Certificate File Paths Accept Unvalidated Configuration Input

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with NO control  
**File:** `v3/server/main.py`  
**Function:** `run_standalone()`  
**Lines:** 85-86  

**Related Functions Checked:**
- `run_standalone()` — Path construction logic
- `pathlib.Path` join operation at lines 85-86
- Configuration loading via `app.cfg.server.*`

**Description:**  
Certificate and key file paths are constructed by joining `CERTS_DIR` with values from `config.yaml` without validating that the resulting paths remain within the intended `certs/` directory. The `pathlib.Path` `/` operator does not sanitize path traversal sequences.

**Vulnerable Code:**
```python
kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile  # line 85
kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile     # line 86
```

**Data Flow:**  
`config.yaml (certfile/keyfile values)` → `pathlib join` → `TLS server initialization` → **no path containment check**

**Proof of Concept:**
```yaml
# Malicious config.yaml
server:
  certfile: "../../../tmp/attacker-cert.pem"
  keyfile: "../../../tmp/attacker-key.pem"
```

```python
# Resolution:
# CERTS_DIR / "../../../tmp/attacker-cert.pem" 
# resolves to /tmp/attacker-cert.pem (outside certs/ directory)
```

**Impact:**  
An attacker who can modify `config.yaml` (but not necessarily the code or certs directory) could redirect certificate loading to an arbitrary filesystem path, causing the server to use an attacker-controlled certificate. While config file modification requires some prior access, defense-in-depth demands path validation.

**Remediation:**
```python
def safe_cert_path(certs_dir, filename):
    """Ensure certificate path stays within the certs directory."""
    resolved = (certs_dir / filename).resolve()
    if not resolved.is_relative_to(certs_dir.resolve()):
        raise ValueError(
            f"Certificate path escapes certs directory: {filename}"
        )
    if not resolved.is_file():
        raise FileNotFoundError(f"Certificate file not found: {resolved}")
    return resolved

# Usage:
kwargs['certfile'] = safe_cert_path(CERTS_DIR, app.cfg.server.certfile)
kwargs['keyfile'] = safe_cert_path(CERTS_DIR, app.cfg.server.keyfile)
```

## Positive Security Patterns

1. **HTTPS-only OAuth:** OAuth URLs are hardcoded with `https://` scheme (lines 43-47), preventing downgrade to HTTP for the authentication callback flow.

2. **Cryptographic salt generation:** Schema documentation specifies `secrets.token_bytes()` for salt generation, which is a cryptographically secure source.

3. **Argon2 for tamper detection:** The `opened_key` field uses Argon2 hashing to detect modification of election data after opening, providing integrity verification for database-stored cryptographic state.

4. **Non-predictable identifiers:** 10-character hex IDs for elections and issues prevent enumeration attacks.

5. **STRICT mode SQLite:** All tables use `STRICT` typing, which enforces type constraints and prevents type confusion.

6. **CHECK constraints on cryptographic fields:** Database-level constraints enforce expected sizes for salts (16 bytes), hashes (32 bytes), preventing truncation or malformed data.

## Architecture Observations

1. **Incomplete Codebase for Full Assessment:** The provided files (`main.py`, two docs) represent the application entry point and documentation only. The actual certificate handling for authentication assertion verification (if any exists beyond TLS) would be in the `pages`, `api`, or `steve.election` modules which were not provided. A complete audit requires reviewing these modules.

2. **Framework Delegation:** Security-critical operations (OAuth verification, TLS handling, session management) are delegated to the `asfquart` framework. The security posture of certificate verification during OAuth callbacks depends entirely on this framework's implementation, which was not provided for review.

3. **No HSM/TPM Integration:** The architecture stores all cryptographic materials (certificates, keys, salts) either in the filesystem or SQLite database. For ASVS Level 3 compliance, hardware-backed key storage should be considered for the server's TLS private key and any signing keys.

4. **Auto-Reload of Certificates:** Adding certificate files to the `extra_files` watch list (line 88) means the server automatically picks up certificate changes. While convenient for development, this creates a window where certificate substitution is immediately effective without operator intervention.
