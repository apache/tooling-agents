### Assessment Summary

ASVS 13.1.3 requires documented resource-management strategies for every external system, including resource-release procedures, timeout settings, failure handling, and retry logic specifications. The audited codebase has **no documented resource-management strategy** for any of the external systems it uses. The implementation itself also lacks the controls that such documentation would mandate: outbound HTTP calls have no configured timeouts, database connections are opened without explicit lifecycle management, and retry loops run without limits or back-off.

## Findings

### HIGH SEVERITY

#### ASVS-1313-HIGH-001: Outbound OAuth HTTP Requests Lack Timeout Configuration and Failure Handling

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control  

**Location:**
- **File:** `v3/server/main.py`
- **Lines:** 32–37
- **Function:** `create_app()`
- **Also Applicable:** `v3/server/config.yaml.example` (missing timeout configuration)

**Related Functions Checked:**
- `asfquart.generics.OAUTH_URL_INIT` assignment
- `asfquart.generics.OAUTH_URL_CALLBACK` assignment
- OAuth callback handler (framework-level)

**Vulnerable Code:**
```python
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Description:**  
The application communicates with `oauth.apache.org` for authentication via OAuth callback. ASVS 13.1.3 explicitly mandates: *"For synchronous HTTP request–response operations it should mandate short timeouts and either disable retries or strictly limit retries to prevent cascading delays and resource exhaustion."*

Neither the application code (`main.py`) nor the configuration documentation (`config.yaml.example`) defines:
- A connection timeout for outbound OAuth requests
- A read/response timeout
- Failure handling behavior (what happens when `oauth.apache.org` is unreachable)
- Whether retries are disabled or limited

**Data Flow:**  
```
User login request 
  → asfquart framework 
    → synchronous HTTP call to oauth.apache.org 
      → **no timeout configured** 
        → potential indefinite hang
```

If the OAuth provider becomes slow or unresponsive, every user authentication attempt will hold a server connection/thread indefinitely, leading to cascading resource exhaustion.

**Proof of Concept:**  
If `oauth.apache.org` experiences a network partition or responds slowly (e.g., TCP connection accepted but no response), each login callback hangs the handling coroutine. With the async server, this consumes event loop capacity; with enough concurrent auth attempts, the server becomes unresponsive.

**Impact:**  
Denial of service via resource exhaustion. All server resources consumed by pending OAuth callbacks, preventing legitimate users from voting.

**Remediation:**  
Add timeout configuration to `config.yaml.example` and enforce it in the OAuth client:

```yaml
# config.yaml
oauth:
  # Short timeouts for synchronous OAuth token exchange
  connect_timeout: 5    # seconds
  read_timeout: 10      # seconds
  retries: 0            # Disable retries for auth callbacks
  # Failure behavior: return 503 to user with retry-after header
```

In application code, ensure the HTTP client used by `asfquart` applies these timeouts:
```python
import httpx

async def oauth_token_exchange(code):
    async with httpx.AsyncClient(
        timeout=httpx.Timeout(connect=5.0, read=10.0, write=5.0, pool=5.0)
    ) as client:
        try:
            response = await client.post(OAUTH_URL_CALLBACK % code)
            response.raise_for_status()
        except httpx.TimeoutException:
            raise OAuthUnavailable("Authentication service timed out")
        except httpx.HTTPStatusError:
            raise OAuthUnavailable("Authentication service error")
```

## Positive Security Patterns

1. **Cryptographic RNG for IDs** — `secrets.token_hex()` and `secrets.token_bytes()` used consistently throughout `crypto.py` for salt generation and ID creation.

2. **Parameterized Queries** — All database queries appear to use `asfpy.db` with YAML-defined query templates and `.perform()` parameterization, avoiding SQL injection.

3. **Transaction Isolation** — `delete()` and `add_salts()` correctly wrap multi-statement operations in `BEGIN TRANSACTION`/`COMMIT`.

4. **State Machine Enforcement** — Election state transitions are enforced with assertions in every mutating method, preventing operations in wrong states.

5. **Sensitive Data Filtering** — `get_metadata()` explicitly excludes `SALT` and `OPENED_KEY` from returned data; `owned_elections()` documentation notes these are omitted.

6. **Vote Shuffling** — `crypto.shuffle()` uses cryptographically secure randomness to prevent vote ordering correlation during tallying.

---

## Architecture Observations

1. **Single External HTTP Dependency** — OAuth to `oauth.apache.org` is the only outbound HTTP call visible in the provided code. This is the highest-risk external dependency for availability and must have timeout/failure handling.

2. **SQLite as Single Data Store** — All data persistence goes through SQLite via `asfpy.db`. While acknowledged as intentional, the connection management pattern (no pooling, no explicit lifecycle) means resource management is entirely dependent on Python garbage collection.

3. **CPU-Intensive Crypto Operations Inline** — Argon2 with 64MB memory cost runs in the request path (`add_vote`, `has_voted_upon`, `tally_issue`). Without concurrency limits, multiple simultaneous requests could exhaust server memory.

4. **No Health Check or Circuit Breaker Pattern** — There is no mechanism to detect when external services (OAuth) are degraded and fail fast, which ASVS 13.1.3 aims to address.

### Immediate Actions (Priority 1)

1. Add OAuth timeout configuration to `config.yaml.example`
2. Implement timeout enforcement in OAuth client code
3. Document failure behavior (return 503 with appropriate headers)
4. Test timeout behavior with simulated OAuth provider delays

### Short-term Actions (1-3 months)

1. **Create Resource Management Documentation** — Develop comprehensive documentation covering all external systems (OAuth, SQLite, TLS certificates) with timeout specifications, retry policies, and failure handling procedures.

2. **Implement Database Connection Lifecycle Management** — Add context manager support to `Election` class; ensure all code paths explicitly close database connections; fix exception path leaks.

3. **Add Retry Limits to ID Generation** — Replace unbounded `while True` loops with bounded retry logic including limits, delays, and proper error handling.

### Long-term Actions (3-6 months)

1. **Implement Circuit Breaker Pattern** — Add health checking and automatic failure detection for OAuth service to fail fast during degradation.

2. **Add Resource Monitoring** — Implement metrics collection for database connection counts, OAuth request latencies, and Argon2 operation concurrency.

3. **Create Operational Runbook** — Document recovery procedures for each failure scenario identified in resource management strategy.
