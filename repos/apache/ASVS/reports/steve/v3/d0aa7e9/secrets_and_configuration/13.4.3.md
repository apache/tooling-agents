## Positive Security Patterns

### 1. Explicit Static File Serving Disabled (PRIMARY CONTROL)

**Location:** `v3/server/main.py:41`

```python
app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
```

**Analysis:** By passing `static_folder=None` to the Quart/asfquart application constructor, the framework will **not** register any static file serving route. This means:

- No `/static/` endpoint exists
- No directory listing is possible through the framework's built-in static handler
- Requests for arbitrary file paths will hit application routes or return 404

This is the **correct and preferred approach** — disabling the capability entirely rather than relying on configuration to restrict it.

**Security Impact:** This single control effectively prevents all framework-level directory listing exposure, providing a strong foundation for ASVS 13.4.3 compliance.

---

### 2. No File-Serving Utility Imports

**Location:** `v3/server/main.py` (entire module)

**Analysis:** The `main.py` module does not import `send_from_directory`, `send_file`, or any file-serving utilities from Quart/Flask. The imported modules (`pages`, `api`) are endpoint modules — while their full source isn't provided, the architecture pattern is consistent with API-only endpoints rather than file serving.

**Security Impact:** Absence of file-serving utilities reduces the attack surface and prevents accidental introduction of directory listing vulnerabilities through custom route handlers.

## Architecture Observations

### Application-Level Directory Listing: MITIGATED

The `static_folder=None` parameter is the definitive control at the application layer. The Quart framework (which asfquart wraps) respects this setting and will not serve any files from the filesystem.

**Technical Details:**
- Framework: Quart (via asfquart wrapper)
- Control mechanism: Application factory parameter
- Scope: All routes and endpoints managed by the application

### Reverse Proxy Configuration: OUT OF SCOPE

The `config.yaml.example` comments indicate a reverse proxy is expected but its configuration is not included in the provided source files. Production hardening of the reverse proxy (e.g., `Options -Indexes` for Apache, `autoindex off` for Nginx) should be verified separately during deployment audit.

**Note:** This is a deployment concern, not an application code vulnerability.

### Sensitive Files Adjacent to Application

The following files exist in or near the application directory and could theoretically be exposed if a reverse proxy is misconfigured:
