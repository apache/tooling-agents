### Medium Severity

#### ASVS-1326-MED-001: No Backend Connection Configuration for Any External or Internal Service

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry points with NO control  
**CWE:** CWE-1188 (Insecure Default Initialization of Resource)

**Location:**
- **File:** `v3/server/config.yaml.example` (entire file)
- **Lines:** 1-5 (complete contents)
- **Related Files:** 
  - `v3/steve/election.py` (all `open_database` calls)
  - `v3/server/main.py` (OAuth URLs, lines 40-44)

**Related Functions Checked:**
- `election.py::open_database()` (line 39)
- `election.py::open_to_pid()` (line 458)
- `election.py::owned_elections()` (line 470)
- `election.py::upcoming_to_pid()` (line 484)
- `election.py::list_closed_election_ids()` (line 496)
- `election.py::create()` (line 436)
- `main.py` OAuth configuration (lines 40-44)

**Description:**

The application's configuration file (`config.yaml.example`) serves as the sole documented configuration surface. It defines only server port, TLS certificate paths, and database path. There is **zero configuration** for any connection management parameter required by ASVS 13.2.6.

**Vulnerable Code:**

```yaml
# config.yaml.example — COMPLETE contents:
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
db: steve.db
# No connection pool settings
# No timeout settings
# No retry strategy settings
# No max connection settings
```

The application connects to at least two backend services:
1. **SQLite Database** — via `asfpy.db.DB()` throughout `election.py`
2. **Apache OAuth Provider** — via `https://oauth.apache.org/` in `main.py`

Neither service has documented or configurable connection management parameters.

**Data Flow:**
1. Application reads `config.yaml` at startup
2. Database path extracted → no connection parameters available
3. OAuth URLs hardcoded → no connection parameters available
4. Runtime connections created with system defaults
5. No operational control over connection behavior

**Proof of Concept:**

Operators attempting to configure production connection behavior have no documented mechanism:

```bash
# Attempt to configure database connection timeout
$ grep -i "timeout\|pool\|retry\|connection" config.yaml.example
# No results — no configuration surface exists

# Attempt to limit concurrent connections
$ grep -i "max.*connection\|concurrent" config.yaml.example
# No results — no configuration surface exists
```

**Impact:**

- **Operational Risk:** Operations teams cannot tune connection behavior for production environments
- **Availability Risk:** Without timeouts, stalled OAuth provider or locked database could cause entire application to hang indefinitely
- **Resource Exhaustion:** Without max-connection limits, resource exhaustion possible under load or attack
- **Unpredictable Behavior:** Connection behavior depends on Python/OS defaults, varies across environments
- **ASVS Non-Compliance:** Direct violation of requirement to follow "documented configuration for each connection"

**Remediation:**

Add a backend connection configuration section to `config.yaml.example`:

```yaml
# Backend service connection configuration
connections:
  database:
    # SQLite-specific: busy_timeout in milliseconds
    busy_timeout: 5000
    # Maximum concurrent Election objects (connection instances)
    max_connections: 10
    # WAL journal mode for concurrent reads
    journal_mode: wal
    
  oauth:
    # HTTP client settings for oauth.apache.org
    connect_timeout: 5      # seconds
    read_timeout: 10        # seconds
    max_retries: 3
    retry_backoff_factor: 0.5
    # Connection pool
    pool_connections: 5
    pool_maxsize: 10
```

Update application initialization to read and apply these settings:

```python
# In main.py create_app()
def create_app():
    app = quart.Quart(__name__)
    app.cfg = yaml.safe_load(open('config.yaml'))
    
    # Apply database connection config
    DB_CONFIG = app.cfg.get('connections', {}).get('database', {})
    app.db_busy_timeout = DB_CONFIG.get('busy_timeout', 5000)
    app.db_max_connections = DB_CONFIG.get('max_connections', 10)
    
    # Apply OAuth connection config
    OAUTH_CONFIG = app.cfg.get('connections', {}).get('oauth', {})
    asfquart.generics.OAUTH_CONNECT_TIMEOUT = OAUTH_CONFIG.get('connect_timeout', 5)
    asfquart.generics.OAUTH_READ_TIMEOUT = OAUTH_CONFIG.get('read_timeout', 10)
    asfquart.generics.OAUTH_MAX_RETRIES = OAUTH_CONFIG.get('max_retries', 3)
```

#### ASVS-1326-MED-002: Database Connections Opened Without Timeouts or Lifecycle Management

**Severity:** MEDIUM  
**Gap Type:** Type A — No control exists  
**CWE:** CWE-404 (Improper Resource Shutdown or Release), CWE-1188 (Insecure Default Initialization)

**Location:**
- **File:** `v3/steve/election.py`
- **Primary Function:** `open_database()` (line 39)
- **Affected Functions:** Lines 42, 436, 458, 470, 484, 496

**Related Functions Checked:**
- `__init__()` (line 42) — stores connection without timeout
- `open_to_pid()` (line 458) — creates connection, never closes
- `owned_elections()` (line 470) — creates connection, never closes
- `upcoming_to_pid()` (line 484) — creates connection, never closes
- `list_closed_election_ids()` (line 496) — creates connection, never closes
- `create()` (line 436) — creates connection, doesn't close in method
- `delete()` (line 73) — ✅ explicitly closes connection
- `_disappeared()` (line 155) — ✅ explicitly closes connection

**Description:**

Every database interaction creates a new `asfpy.db.DB` connection with no timeout, no connection limit, and no pool. Multiple class methods create connections that are **never explicitly closed**, leading to potential connection leaks.

**Vulnerable Code:**

```python
# election.py - open_database (line 39) - no timeout parameter
@staticmethod
def open_database(db_fname):
    return asfpy.db.DB(db_fname, yaml_fname=QUERIES, yaml_section='election')
    # No busy_timeout configured
    # No connection pooling
    # No maximum connection limit
```

```python
# election.py - open_to_pid (line 458) - connection NEVER closed
@classmethod
def open_to_pid(cls, db_fname, pid):
    db = cls.open_database(db_fname)  # new connection
    db.q_open_to_me.perform(pid,)
    return [row for row in db.q_open_to_me.fetchall()]
    # db falls out of scope — relies on GC for cleanup
    # No explicit close()
    # No timeout if database locked
```

```python
# election.py - owned_elections (line 470) - connection NEVER closed
@classmethod
def owned_elections(cls, db_fname, pid):
    db = cls.open_database(db_fname)
    db.q_owned_by_me.perform(pid,)
    return [row for row in db.q_owned_by_me.fetchall()]
    # Same leak pattern
```

```python
# election.py - upcoming_to_pid (line 484) - connection NEVER closed
@classmethod
def upcoming_to_pid(cls, db_fname, pid):
    db = cls.open_database(db_fname)
    db.q_upcoming_for_me.perform(pid,)
    return [row for row in db.q_upcoming_for_me.fetchall()]
    # Same leak pattern
```

```python
# election.py - list_closed_election_ids (line 496) - connection NEVER closed
@classmethod
def list_closed_election_ids(cls, db_fname):
    db = cls.open_database(db_fname)
    db.q_closed_elections.perform()
    return [row['id'] for row in db.q_closed_elections.fetchall()]
    # Same leak pattern
```

```python
# election.py - create (line 436) - connection not closed in method
@classmethod
def create(cls, db_fname, issue, owner, monitors):
    db = cls.open_database(db_fname)
    # ... creates election ...
    return cls(db, eid)  # Returns Election instance with SEPARATE connection
    # Original 'db' connection orphaned
```

**Data Flow:**

```
HTTP Request
    ↓
Class Method (e.g., open_to_pid)
    ↓
open_database(db_fname)
    ↓
asfpy.db.DB() — creates new SQLite connection
    ↓
Query execution
    ↓
Method returns
    ↓
Connection object orphaned (GC-dependent cleanup)
    ↓
[Connection may remain open until GC runs]
```

**Proof of Concept:**

Simulate concurrent requests triggering connection leaks:

```python

import threading
from steve.election import Election

def trigger_leak():
    # Each call creates a connection that's never closed
    elections = Election.open_to_pid('steve.db', 'testuser')
    # Connection leaked here

# Simulate 100 concurrent requests
threads = [threading.Thread(target=trigger_leak) for _ in range(100)]
for t in threads:
    t.start()
for t in threads:
    t.join()

```

Demonstrate lock failure without `busy_timeout`:

```python
# PoC: Immediate failure on concurrent write without busy_timeout
import sqlite3
import threading

def writer(db_path, writer_id):
    conn = sqlite3.connect(db_path)
    # No busy_timeout set — default is 0
    try:
        conn.execute("BEGIN IMMEDIATE")
        time.sleep(1)  # Hold lock
        conn.commit()
        print(f"Writer {writer_id} succeeded")
    except sqlite3.OperationalError as e:
        print(f"Writer {writer_id} failed: {e}")  # "database is locked"
    finally:
        conn.close()

# Start two concurrent writers
t1 = threading.Thread(target=writer, args=('steve.db', 1))
t2 = threading.Thread(target=writer, args=('steve.db', 2))
t1.start()
time.sleep(0.1)  # Ensure t1 acquires lock first
t2.start()
# t2 immediately fails with "database is locked"
```

**Impact:**

- **Connection Leaks:** Under concurrent requests, leaked connections accumulate until garbage collection
- **Resource Exhaustion:** Unclosed connections consume file descriptors and memory
- **Lock Failures:** SQLite without `busy_timeout` immediately raises `sqlite3.OperationalError: database is locked` on concurrent writes instead of waiting
- **Undefined Behavior:** No documented behavior for connection exhaustion scenarios
- **Unpredictable Performance:** Connection cleanup timing depends on Python GC, varies by implementation

**Remediation:**

**Step 1:** Set SQLite `busy_timeout` on every connection:

```python
@staticmethod
def open_database(db_fname):
    db = asfpy.db.DB(db_fname, yaml_fname=QUERIES, yaml_section='election')
    # Configure timeout from app config (default 5 seconds)
    timeout_ms = getattr(app_config, 'database_busy_timeout', 5000)
    db.conn.execute(f'PRAGMA busy_timeout = {int(timeout_ms)}')
    return db
```

**Step 2:** Use context managers for class methods:

```python
@classmethod
def open_to_pid(cls, db_fname, pid):
    db = cls.open_database(db_fname)
    try:
        db.q_open_to_me.perform(pid)
        return [row for row in db.q_open_to_me.fetchall()]
    finally:
        db.conn.close()  # Guaranteed cleanup
```

**Step 3:** Or implement connection pooling:

```python
# Centralized connection pool
from queue import Queue, Full
import threading

class ConnectionPool:
    def __init__(self, db_fname, max_size=10, timeout_ms=5000):
        self.db_fname = db_fname
        self.max_size = max_size
        self.timeout_ms = timeout_ms
        self._pool = Queue(maxsize=max_size)
        self._lock = threading.Lock()
        self._created = 0
    
    def get_connection(self, timeout=5):
        try:
            # Try to get existing connection
            return self._pool.get(timeout=timeout)
        except:
            # Create new if under limit
            with self._lock:
                if self._created < self.max_size:
                    self._created += 1
                    db = asfpy.db.DB(self.db_fname, yaml_fname=QUERIES, yaml_section='election')
                    db.conn.execute(f'PRAGMA busy_timeout = {self.timeout_ms}')
                    return db
            raise RuntimeError(f"Connection pool exhausted (max={self.max_size})")
    
    def return_connection(self, db):
        try:
            self._pool.put_nowait(db)
        except Full:
            db.conn.close()  # Pool full, discard connection

# Usage in Election class
@classmethod
def open_to_pid(cls, db_fname, pid):
    conn = connection_pool.get_connection()
    try:
        conn.q_open_to_me.perform(pid)
        return [row for row in conn.q_open_to_me.fetchall()]
    finally:
        connection_pool.return_connection(conn)
```

**Location:**
- **File:** `v3/server/main.py`

**Related Functions Checked:**
- `main.py::create_app()` (lines 79-82) — app initialization
- `asfquart.generics.OAUTH_URL_INIT` (line 40-42)
- `asfquart.generics.OAUTH_URL_CALLBACK` (line 43-44)

**Description:**

The application connects to an external OAuth service (`oauth.apache.org`) for authentication. The URLs are configured, but there is no associated timeout or retry configuration anywhere in the codebase.

**Vulnerable Code:**

```python
# main.py lines 40-44
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
# No timeout configuration set
# No retry configuration set
# No connection pool configuration set
# No max concurrent request limit
```

The `config.yaml.example` contains no OAuth connection parameters:

```yaml
# config.yaml.example — no OAuth section
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem

# Missing: OAuth connection configuration
```

The `asfquart.generics` module handles the actual HTTP requests, but the application provides no configuration to control:
- Connect timeout (how long to wait for TCP connection)
- Read timeout (how long to wait for OAuth response)

**Data Flow:**

```
User Authentication Request
    ↓
asfquart.generics OAuth handler
    ↓
HTTP request to oauth.apache.org
    ↓
[No timeout configured — waits indefinitely]
    ↓
```

**Proof of Concept:**

Simulate OAuth provider latency causing application hang:


Simulate transient network error with no retry:

```python
# PoC: Transient failure with no retry
import aiohttp

async def oauth_callback_no_retry(code):
```

**Impact:**

- **Availability Risk:** If `oauth.apache.org` experiences latency, every authentication attempt could block indefinitely
- **Resource Exhaustion:** Hanging OAuth requests consume async worker slots/connections; attacker triggering many simultaneous authentication flows against slow OAuth provider could exhaust server resources
- **Poor User Experience:** Transient network errors cause immediate authentication failures with no retry
- **Unpredictable Behavior:** Connection timeout depends on OS TCP defaults (typically 60-120 seconds)
- **No Circuit Breaker:** Repeated failures to OAuth provider will continue attempting connections instead of failing fast

**Remediation:**

**Step 1:** Add OAuth connection configuration to `config.yaml`:

```yaml
connections:
  oauth:
    provider: https://oauth.apache.org
    connect_timeout: 5      # seconds to establish TCP connection
    read_timeout: 10        # seconds to receive response
    max_retries: 3          # retry on transient failures
    retry_backoff_factor: 0.5  # exponential backoff: 0.5, 1.0, 2.0 seconds
    pool_connections: 5     # connection pool size
    pool_maxsize: 10        # maximum pool size
```

**Step 2:** Configure the HTTP client used by `asfquart.generics`:

```python
# In main.py create_app()
def create_app():
    app = quart.Quart(__name__)
    app.cfg = yaml.safe_load(open('config.yaml'))
    
    # Configure OAuth connection parameters
    oauth_cfg = app.cfg.get('connections', {}).get('oauth', {})
    
    asfquart.generics.OAUTH_URL_INIT = (
        f"{oauth_cfg.get('provider', 'https://oauth.apache.org')}/auth?state=%s&redirect_uri=%s"
    )
    asfquart.generics.OAUTH_URL_CALLBACK = (
        f"{oauth_cfg.get('provider', 'https://oauth.apache.org')}/token?code=%s"
    )
    
    # Configure connection parameters
    asfquart.generics.OAUTH_CONNECT_TIMEOUT = oauth_cfg.get('connect_timeout', 5)
    asfquart.generics.OAUTH_READ_TIMEOUT = oauth_cfg.get('read_timeout', 10)
    asfquart.generics.OAUTH_MAX_RETRIES = oauth_cfg.get('max_retries', 3)
    asfquart.generics.OAUTH_RETRY_BACKOFF = oauth_cfg.get('retry_backoff_factor', 0.5)
```

**Step 3:** If framework doesn't support native timeout/retry, implement wrapper:

```python
import asyncio
import aiohttp
from aiohttp import ClientTimeout

class OAuthClient:
    def __init__(self, config):
        self.config = config
        timeout = ClientTimeout(
            connect=config.get('connect_timeout', 5),
            total=config.get('read_timeout', 10)
        )
        self.session = aiohttp.ClientSession(timeout=timeout)
    
    async def get_token(self, code):
        url = f"{self.config['provider']}/token?code={code}"
        max_retries = self.config.get('max_retries', 3)
        backoff = self.config.get('retry_backoff_factor', 0.5)
        
        for attempt in range(max_retries):
            try:
                async with self.session.get(url) as resp:
                    resp.raise_for_status()
                    return await resp.json()
            except (aiohttp.ClientError, asyncio.TimeoutError) as e:
                if attempt == max_retries - 1:
                    _LOGGER.error(f"OAuth failed after {max_retries} attempts: {e}")
                    raise OAuthError("Authentication service unavailable")
                
                wait_time = backoff * (2 ** attempt)
                _LOGGER.warning(f"OAuth attempt {attempt+1} failed, retrying in {wait_time}s")
                await asyncio.sleep(wait_time)

# Usage
oauth_client = OAuthClient(app.cfg['connections']['oauth'])
token = await oauth_client.get_token(auth_code)
```

## Architecture Observations

### 1. No Connection Abstraction Layer

**Observation:** Database connections are created ad-hoc through `asfpy.db.DB()` calls scattered across class methods with no centralized management.

**Impact:** 
- Impossible to enforce consistent connection policies (timeouts, pooling, limits)
- No single point to instrument connection metrics or logging
- Difficult to swap database implementations or add connection middleware

**Recommendation:** Introduce a `ConnectionManager` singleton that all database access routes through.

---

### 2. Mixed Connection Lifecycles

**Observation:** Instance methods hold connections for the object lifetime (cleaned up in `delete`/`_disappeared`), while class methods create ephemeral connections that are never closed.

**Lifecycle Patterns Found:**

| Pattern | Methods | Connection Lifetime | Cleanup |
|---------|---------|---------------------|---------|
| **Long-lived instance connection** | `__init__`, `delete`, `_disappeared` | Duration of Election object | Explicit `close()` in `delete`/`_disappeared` |
| **Ephemeral class method connection** | `open_to_pid`, `owned_elections`, `upcoming_to_pid`, `list_closed_election_ids` | Single query | ❌ No cleanup (GC-dependent) |
| **Hybrid** | `create` | Creates connection, returns new instance with different connection | ❌ Original connection orphaned |

**Impact:** Unpredictable connection behavior; resource leaks under load.

**Recommendation:** Standardize on single lifecycle pattern (preferably connection pooling with context managers).

### 3. External Service Dependency Without Resilience

**Observation:** Application's authentication depends entirely on `oauth.apache.org` availability with no fallback, circuit breaker, or graceful degradation documented.

**Current State:**
- Single OAuth provider (no fallback)
- No timeout configured (hangs indefinitely on provider latency)
- No retry logic (fails immediately on transient errors)
- No circuit breaker (continues attempting connections during outages)
- No cached authentication state (every request requires OAuth round-trip)

**Impact:** OAuth provider outage or latency renders entire application unusable for authentication.

**Recommendation:** Implement resilience patterns:
1. Circuit breaker to fail fast during OAuth outages
2. Retry with exponential backoff for transient failures
3. Consider session caching to reduce OAuth dependency
4. Document degraded-mode behavior when OAuth unavailable
