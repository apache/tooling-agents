## Findings

### Medium Severity

#### ASVS-1345-MED-001: Production ASGI Deployment Path Enables DEBUG-Level Logging

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `run_asgi()` (lines 88–107)
- **Lines:** 95-99, 104

**Related Functions Checked:**
- `run_asgi()` — production ASGI entry point
- `run_standalone()` — development entry point (also uses DEBUG)
- `_LOGGER.debug()` calls throughout `election.py` (lines 46, 189, 403)

**Gap Type:** Type A — Missing control (no production-appropriate log level)

**Description:**

The production deployment entry point `run_asgi()` configures the logging system with DEBUG-level verbosity, which causes extensive operational details to be logged in production environments. This violates the ASVS principle that "Production configurations should be hardened to avoid disclosing unnecessary data."

**Vulnerable Code:**

```python
def run_asgi():
    "Run as an ASGI process; eg. Hypercorn"

    # NOTE: no-op if Hypercorn has set up the root logger.
    logging.basicConfig(
        level=logging.DEBUG,       # <-- DEBUG in production path
        style='{',
        format='[{asctime}|{levelname}|{name}] {message}',
        datefmt=DATE_FORMAT,
    )

    # ...

    # Configure our app's loggers (rather than root logger default).
    _LOGGER.setLevel(logging.DEBUG)  # <-- Explicit DEBUG on app logger
```

**Data Flow:**

1. `run_asgi()` is invoked as the **production deployment entry point** (used with Hypercorn ASGI server)
2. Sets `logging.basicConfig(level=logging.DEBUG)` which, if Hypercorn hasn't pre-configured the root logger, enables DEBUG for **all libraries** including database drivers, HTTP handling, and framework internals
3. Unconditionally sets `_LOGGER.setLevel(logging.DEBUG)` for the application logger
4. Debug log statements throughout `election.py` then emit operational details:
   ```python
   _LOGGER.debug(f'{op} election ID "{eid}"')      # election.py:46
   _LOGGER.debug('EID conflict(!!) ... trying again.')  # election.py:403
   _LOGGER.debug('IID conflict(!!) ... trying again.')  # election.py:189
   ```

**Proof of Concept:**

When deployed with Hypercorn in production:
```bash
hypercorn main:steve_app
```

Production logs will contain:
```
[2026-04-21 10:23:45|DEBUG|steve.election] get election ID "abc123def456"
[2026-04-21 10:23:46|DEBUG|asyncio] Using selector: EpollSelector
[2026-04-21 10:23:47|DEBUG|steve.election] EID conflict(!!) ... trying again.
```

**Impact:**

- Production logs capture DEBUG-level output from all application and library modules
- Internal operation details (election IDs, retry behavior, database interactions) are logged
- Verbose library logging (asyncio, HTTP handling, TLS) may expose connection details, timing information, or internal state
- Facilitates information gathering when combined with log access (e.g., shared logging infrastructure, log aggregation systems)
- Per ASVS section guidance: *"Production configurations should be hardened to avoid disclosing unnecessary data"*

**Remediation:**

```python
def run_asgi():
    "Run as an ASGI process; eg. Hypercorn"

    logging.basicConfig(
        level=logging.WARNING,  # Production-appropriate default
        style='{',
        format='[{asctime}|{levelname}|{name}] {message}',
        datefmt=DATE_FORMAT,
    )

    # Application logger at INFO for operational visibility
    _LOGGER.setLevel(logging.INFO)

    _LOGGER.info(' ** Run-mode: ASGI')

    global steve_app
    steve_app = create_app()
```

Additionally, make log level configurable through `config.yaml`:
```yaml
server:
    port: 58383
    log_level: WARNING  # Override via environment variable
```

#### ASVS-1345-MED-002: No Production Configuration Controls for Endpoint Exposure and Debug Mode

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/config.yaml.example` (entire file)
- **Function:** N/A — configuration template
- **Lines:** 1-12 (entire file)

**Related Functions Checked:**
- `create_app()` in `main.py:33-43` — application factory
- `run_asgi()` in `main.py:88-107` — production entry point
- Framework initialization via `asfquart.construct()`

**Gap Type:** Type A — Missing control (no production hardening configuration)

**Description:**

The production configuration template provides no mechanism to explicitly control debug mode, log levels, or endpoint exposure. This prevents operators from verifying production hardening through configuration review and provides no defense-in-depth mechanism to disable debug features.

**Vulnerable Configuration:**

```yaml
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem

db: steve.db
# No debug mode setting
# No log level setting  
# No endpoint exposure controls
```

**Data Flow:**

1. Configuration file is loaded by `asfquart.construct()` during application initialization
2. No `debug` setting → framework relies on defaults (potentially unsafe)
3. No `log_level` setting → code hardcodes DEBUG (see ASVS-1345-MED-001)
4. No endpoint controls → cannot audit which framework-provided endpoints are active
5. Production deployment has no documented hardening checklist

**Analysis:**

The production configuration template provides no mechanism to:

1. **Explicitly disable debug mode** — No `debug: false` setting. The `asfquart` framework (Quart-based) supports debug mode which enables interactive debuggers, detailed error pages with stack traces, and auto-reloading. Without explicit configuration, production relies entirely on framework defaults.

2. **Control log verbosity** — No `log_level` setting. As shown in ASVS-1345-MED-001, the code hardcodes DEBUG.

3. **Disable framework-provided endpoints** — The `asfquart` framework provides built-in OAuth endpoints (evidenced by `asfquart.generics.OAUTH_URL_INIT` and `OAUTH_URL_CALLBACK` being configured in `main.py:37-39`). There is no configuration to inventory or restrict which framework-provided endpoints are active.

4. **Enable/disable monitoring endpoints** — No way to control whether health checks, status pages, or diagnostic endpoints are exposed.

**Impact:**

- Production deployments cannot be audited for proper hardening through configuration review alone
- No defense-in-depth mechanism to disable debug features even if framework defaults are secure
- Operators have no documented way to verify endpoint exposure aligns with production requirements
- Unclear which endpoints the `asfquart` framework registers by default beyond the explicitly imported `pages` and `api` modules
- Violates ASVS 13.4.5 requirement to verify "documentation (such as for internal APIs) and monitoring endpoints are not exposed unless explicitly intended"

**Remediation:**

Add production hardening settings to `config.yaml.example`:

```yaml
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
    
    # Production hardening
    debug: false              # MUST be false in production
    log_level: WARNING        # Minimum: WARNING for production
    
    # Endpoint exposure control
    enable_health_endpoint: false   # Set true only if monitoring requires it
    
db: steve.db
```

And enforce in `create_app()`:

```python
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    # Enforce production hardening
    if not app.cfg.server.get('debug', False):
        app.debug = False
        app.config['TESTING'] = False
    
    import pages
    import api
    return app
```

## Positive Security Patterns

1. **Static file serving explicitly disabled**: `static_folder=None` in `asfquart.construct()` prevents accidental exposure of server-side files through default static file handling.

2. **Sensitive data filtered from public APIs**: `get_metadata()` explicitly constructs a response object without `salt` or `opened_key`. The `get_issue()` method has a clear comment "NEVER return issue.salt". `owned_elections()` documents returning a "subset of columns."

3. **Benchmark code guarded**: `benchmark_argon2()` in `crypto.py` is protected by `if __name__ == '__main__'` and will never execute in the web server context.

4. **Example configuration file**: `config.yaml.example` uses the `.example` suffix, signaling it should not be deployed as-is (though `.gitignore` status should be verified for the actual `config.yaml`).

5. **Development vs. production paths separated**: `run_standalone()` (direct execution) vs. `run_asgi()` (Hypercorn) provides clear separation, though both currently use DEBUG logging.

---

## Architecture Observations

1. **Incomplete audit scope**: The `pages` and `api` modules (imported in `main.py:42-43`) register all web endpoints but were not provided for audit. **Complete verification of ASVS 13.4.5 is impossible without these files.** These modules could expose documentation endpoints, debug views, or monitoring routes.

2. **Framework opacity**: The `asfquart` framework injects behavior (OAuth endpoints, potentially health checks) that cannot be audited from application code alone. The `asfquart.generics` module modification suggests the framework has its own endpoint registration.

3. **Commented-out debug prints in election.py**: Lines like `# print('EDATA:', edata)` and `#print('SALT:', salt)` and `#print('KEY:', opened_key)` (around lines 80-82, 348-350) indicate cryptographic keys were previously printed during development. While currently commented, they represent a risk if accidentally uncommented.
