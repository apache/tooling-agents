## Findings

### Medium Severity

#### ASVS-1332-MED-001 — State-Checking Operations Unnecessarily Retrieve Full Secret Material

**Severity:** MEDIUM  
**Gap Type:** Type A — Least privilege control absent for secret column access

**Location:**
- **File:** `v3/steve/election.py`
- **Function:** `_all_metadata()` (line ~125)
- **Called by:** `get_state()` (line ~329), `is_editable()` (line ~318), `is_open()` (line ~322), `is_closed()` (line ~326), `get_metadata()` (line ~135), `gather_election_data()` (line ~78)

**Related Functions Checked:**
- `_compute_state()` — only checks NULL/NOT-NULL status
- `get_state()` — state determination logic
- `is_editable()`, `is_open()`, `is_closed()` — state query wrappers
- `get_metadata()` — public metadata accessor
- `gather_election_data()` — election data aggregation

**Description:**

The `_all_metadata()` method retrieves complete election metadata including cryptographic secrets (`salt` and `opened_key`) for every call, even when the calling code only needs to check state or retrieve non-sensitive metadata. Six different methods call `_all_metadata()` when they only need NULL/NOT-NULL checks or non-secret fields.

**Vulnerable Code:**

```python
def _all_metadata(self, required_state=None):
    "INTERNAL ONLY: return all metadata about this Election."
    # NOTE: this returns the SALT and OPENED_KEY columns. This
    # API is not for public use.
    md = self.q_metadata.first_row(self.eid)  # Fetches salt + opened_key EVERY time
    ...
    return md
```

**Callers that do NOT need secret values:**

```python
def get_state(self):
    return self._compute_state(self._all_metadata())  # Only checks if salt/opened_key are None

def get_metadata(self):
    md = self._all_metadata()  # Fetches secrets...
    return edict(            # ...then discards them
        eid=md.eid, title=md.title, ...  # salt and opened_key intentionally omitted
    )

def gather_election_data(self, pdb):
    md = self._all_metadata()  # Fetches secrets...
    mdata = md.eid + md.title  # ...only uses eid and title
```

**Data Flow:**

`is_editable()` → `get_state()` → `_all_metadata()` → `q_metadata` SQL query → returns `salt` + `opened_key` in memory → `_compute_state()` only checks `md.salt is None` / `md.opened_key is not None`

**Impact:**

- The `opened_key` (master election secret enabling vote decryption) is loaded into memory during every state check, metadata request, and election data gathering — operations that have no need for the actual key values.
- `_compute_state()` only checks for NULL/NOT-NULL status but receives actual secret bytes.
- State checks occur on nearly every request path (election display, voting eligibility, metadata), broadening the window where secrets reside in process memory.
- If any exception handler, debugger, or future logging change captures local variables, `md.salt` and `md.opened_key` would be exposed.

**Remediation:**

Create a separate state-only query and method that returns only the information needed for state computation:

```python
# In queries.yaml, add:
# q_state_info: SELECT closed, (salt IS NOT NULL) AS has_salt, 
#               (opened_key IS NOT NULL) AS has_key FROM metadata WHERE eid = ?

def _get_state_info(self):
    """Return state-relevant metadata WITHOUT secret columns."""
    info = self.q_state_info.first_row(self.eid)
    if not info:
        raise self._disappeared()
    return info

def get_state(self):
    info = self._get_state_info()
    return self._compute_state_from_flags(info)

@classmethod
def _compute_state_from_flags(cls, info):
    if info.closed == 1:
        assert info.has_salt and info.has_key
        return cls.S_CLOSED
    if not info.has_salt:
        assert not info.has_key
        return cls.S_EDITABLE
    return cls.S_OPEN
```

#### ASVS-1332-MED-002 — Unrestricted Database Cursor Proxy Bypasses Secret-Filtering Controls

**Severity:** MEDIUM  
**Gap Type:** Type B — Control EXISTS (`get_metadata()`) but NOT ENFORCED (cursor proxy allows bypass)

**Location:**
- **File:** `v3/steve/election.py`
- **Function:** `__getattr__()` (line ~44)

**Related Functions Checked:**
- `get_metadata()` — intended secret-filtering accessor
- `_all_metadata()` — internal secret-returning method
- All cursor proxying via `__getattr__`

**Description:**

The `Election` class implements `__getattr__()` to proxy database cursor access, which allows any code with an `Election` instance to bypass the intentional secret-filtering in `get_metadata()` by directly accessing `q_metadata` cursor.

**Vulnerable Code:**

```python
def __getattr__(self, name):
    "Proxy the cursors."
    return self.__dict__.get(name, getattr(self.db, name))
```

**Control that is bypassed:**

```python
def get_metadata(self):
    md = self._all_metadata()
    # NOTE: do not return the SALT or OPENED_KEY
    return edict(eid=md.eid, title=md.title, ...)  # Intentionally filters secrets
```

**Bypass path:**

```python
# Any code with an Election instance can bypass get_metadata() filtering:
election = Election(db_fname, eid)
md = election.q_metadata.first_row(eid)  # Direct cursor access via __getattr__
print(md.opened_key)  # Full secret material accessible
```

**Data Flow:**

Any code holding `Election` instance → `__getattr__('q_metadata')` → `self.db.q_metadata` → direct SQL execution → returns all columns including `salt`, `opened_key`

**Impact:**

- The developer intentionally created `get_metadata()` to filter `salt` and `opened_key` from public access, but the `__getattr__` proxy makes the underlying secret-returning cursor equally accessible.
- Any module that imports and uses the `Election` class (API handlers, page handlers, CLI tools) can directly access `q_metadata` and retrieve full secrets without going through the filtering method.
- This creates false confidence that secrets are protected by the `get_metadata()` abstraction.

**Remediation:**

Restrict the `__getattr__` proxy to only expose safe cursors, or make secret-accessing operations require explicit method calls:

```python
# Option 1: Allowlist safe cursors
_SAFE_PROXIED_ATTRS = frozenset({
    'q_check_election', 'q_issues', 'q_get_issue', 'q_open_to_me',
    'c_create', 'c_add_issue', 'c_edit_issue', 'c_delete_issue',
    # ... explicitly list safe cursors
})

def __getattr__(self, name):
    "Proxy only safe cursors. Secret-accessing queries require explicit methods."
    if name in self._SAFE_PROXIED_ATTRS:
        return getattr(self.db, name)
    raise AttributeError(f"'{type(self).__name__}' has no attribute '{name}'")

# Option 2: Private method for secret access
def _get_opened_key(self):
    """Explicitly retrieve opened_key - for vote operations only."""
    md = self.q_metadata.first_row(self.eid)
    return md.opened_key, md.salt
```

#### ASVS-1332-MED-003 — Production ASGI Entry Point Hardcodes DEBUG Logging Level

**Severity:** MEDIUM  
**Gap Type:** Type A — No production-safe logging configuration

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `run_asgi()` (line ~83)

**Related Functions Checked:**
- `logging.basicConfig()` — root logger configuration
- `_LOGGER.setLevel()` — application logger configuration

**Description:**

The production ASGI deployment entry point hardcodes `logging.DEBUG` level for both root and application loggers, violating the requirement that debug mode be disabled in production and increasing the risk of inadvertent secret exposure through logging.

**Vulnerable Code:**

```python
def run_asgi():
    "Run as an ASGI process; eg. Hypercorn"

    # NOTE: no-op if Hypercorn has set up the root logger.
    logging.basicConfig(
        level=logging.DEBUG,          # DEBUG in production ASGI path
        style='{',
        format='[{asctime}|{levelname}|{name}] {message}',
        datefmt=DATE_FORMAT,
    )
    # ...
    _LOGGER.setLevel(logging.DEBUG)   # Explicit DEBUG on app logger
```

**Context — evidence that secrets were previously logged at debug level:**

```python
# In election.py, commented-out debug prints show developer debugging patterns:
#print('SALT:', salt)
#print('KEY:', opened_key)
# print('EDATA:', edata)
```

**Impact:**

- The `run_asgi()` function is the production deployment path (used with Hypercorn), yet it hardcodes `logging.DEBUG` for both the root logger and the application logger.
- The domain context explicitly requires: "debug mode is disabled in production config."
- Commented-out `print()` statements for `salt` and `opened_key` in `election.py` demonstrate that developers debug with secret values — converting these to `_LOGGER.debug()` calls would immediately expose secrets in production logs.
- DEBUG logging captures maximum detail including election IDs, state transitions, and database operations, broadening the information available if logs are compromised.
- This violates least privilege for logging: production systems should only log information necessary for operations, not debug-level detail.

**Remediation:**

Set production logging to INFO or WARNING, and make it configurable:

```python
def run_asgi():
    "Run as an ASGI process; eg. Hypercorn"

    log_level = os.environ.get('STEVE_LOG_LEVEL', 'INFO').upper()
    
    logging.basicConfig(
        level=getattr(logging, log_level, logging.INFO),
        style='{',
        format='[{asctime}|{levelname}|{name}] {message}',
        datefmt=DATE_FORMAT,
    )
    
    _LOGGER.setLevel(getattr(logging, log_level, logging.INFO))
    _LOGGER.info(f' ** Run-mode: ASGI (log level: {log_level})')
```

## Positive Security Patterns

1. **Explicit secret filtering in `get_metadata()`** (election.py:135): Creates a new edict with only non-secret fields rather than removing fields from the full result — defense against accidental serialization of the source object.

2. **"NEVER return issue.salt" comment** (election.py:153): Developer awareness and explicit documentation at the return site of `get_issue()`.

3. **Column subset in `owned_elections()`** (election.py:384-389): Explicit comment and SQL query design to exclude `SALT` and `OPENED_KEY`.

4. **Cryptographic RNG throughout** (crypto.py): All random values use `secrets` module — `token_bytes()` for salts, `token_hex()` for IDs, `randbelow()` for shuffling.

5. **Per-voter unique salts** (election.py:105-121): Each person/issue pair gets a unique cryptographic salt via `crypto.gen_salt()`, generated in a transaction.

6. **Strong key derivation chain**: BLAKE2b → Argon2 → HKDF provides defense-in-depth for key material.

7. **Vote shuffling before tallying** (election.py:277): `crypto.shuffle(votes)` prevents database-order leakage.

8. **Anti-tamper verification** (election.py:303-316): `is_tampered()` recomputes the `opened_key` to detect data manipulation.

9. **config.yaml.example pattern**: Example config file with non-real values; actual config should be separate and excluded from version control.

10. **Transactional salt addition** (election.py:111-120): `BEGIN TRANSACTION` / `COMMIT` ensures atomicity of salt assignment.

## Architecture Observations

1. **Missing queries.yaml from audit scope**: The SQL queries that determine which columns are returned (and thus which secrets are accessible) are defined in `queries.yaml`, which was not provided. The `q_open_to_me` and `q_upcoming_to_me` queries lack the explicit "no secrets" comments that `q_owned` has — the actual queries should be verified to ensure they don't return `salt` or `opened_key` columns.

2. **Missing API/pages layer from audit scope**: The `api.py` and `pages.py` modules that handle HTTP requests are not provided. These are the entry points where access control decisions should enforce least privilege before calling Election methods. Without them, we cannot verify that secret-accessing methods are appropriately restricted.

3. **Single-class architecture for all privilege levels**: The `Election` class serves both web-facing (voter) operations and privileged (admin/tallying) operations. There is no architectural separation between contexts that need secret access and those that don't.

4. **No config.yaml file permission enforcement**: The code reads config paths but does not verify file permissions on configuration files or the database file, as noted in the domain context checklist.

5. **Environment variable integration noted but not visible**: The domain context mentions "asfquart's environment variable integration" for secrets, but the provided code only shows file-based configuration in `config.yaml.example`. The mechanism for overriding secrets via environment variables should be verified.

## Recommendations Summary

### Immediate Actions (High Priority, Low Effort)

1. **SEV-003: Configure production logging level**
   - Set `STEVE_LOG_LEVEL` environment variable to `INFO` or `WARNING` for production deployments
   - Effort: Low | Impact: Reduces risk of inadvertent secret exposure via future DEBUG messages

2. **SEV-001: Create state-only database query**
   - Add `q_state_info` query returning only `closed`, `has_salt`, `has_key` flags
   - Refactor state-checking methods to use new query
   - Effort: Low | Impact: Eliminates 6+ unnecessary secret retrievals per request cycle

### Short-term Actions (Medium Priority)

3. **SEV-002: Restrict cursor proxy access**
   - Implement allowlist-based `__getattr__` or make secret access require explicit methods
   - Effort: Medium | Impact: Enforces the secret-filtering pattern as a real control rather than convention

4. **Verify queries.yaml**
   - Audit all SQL queries for unnecessary secret column inclusion
   - Focus on `q_open_to_me`, `q_upcoming_to_me`
   - Effort: Low | Impact: Ensures SQL-level least privilege

### Long-term Actions (Strategic)

5. **Verify API access controls**
   - Audit `api.py` and `pages.py` to ensure secret-accessing methods are only called from appropriately authenticated/authorized endpoints
   - Effort: Medium | Impact: Ensures endpoint-level least privilege

6. **Consider architectural separation**
   - Evaluate splitting `Election` class into public and privileged interfaces
   - Effort: High | Impact: Structural enforcement of least privilege principle
