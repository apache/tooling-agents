### Medium Severity

#### ASVS-1311-MED-001: External OAuth Service Dependency Hardcoded and Undocumented in Configuration

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `create_app()`
- **Lines:** 37-40

**Related Functions Checked:**
- `run_standalone()` - OAuth configuration consumed
- `run_asgi()` - OAuth configuration consumed

**Description:**

The application has a hard runtime dependency on `oauth.apache.org` for authentication, but this external service is not documented in the configuration file. The OAuth endpoints are hardcoded in source code rather than externalized as configuration parameters.

**Vulnerable Code:**
```python
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Corresponding Configuration (`config.yaml.example`):**
```yaml
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
db: steve.db
# ← No OAuth or external service configuration documented
```

**Data Flow:**
1. User initiates login → Application redirects to `oauth.apache.org/auth`
2. OAuth provider authenticates user → Redirects to application callback
3. Application exchanges code at `oauth.apache.org/token`
4. The `redirect_uri` parameter (passed as `%s`) represents a callback URL whose construction is opaque

**Proof of Concept:**

An operator deploying this application would:
1. Review `config.yaml.example` for network requirements
2. Configure firewall rules based on documented communication needs
3. Block outbound HTTPS to unknown destinations
4. Deploy application → **Authentication fails** due to blocked OAuth connections

**Impact:**
- **Operational:** Operators cannot perform accurate network security planning
- **Audit:** Security teams cannot verify all external communication paths from configuration alone
- **Incident Response:** Outbound connection destinations are hidden in code, hampering investigation
- **Disaster Recovery:** No documented procedure for changing OAuth provider
- **Compliance:** Direct violation of ASVS 13.1.1 requirement to document "external services which the application relies upon"

**Remediation:**

Add OAuth configuration to `config.yaml.example`:
```yaml
# config.yaml.example - REMEDIATED
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem

db: steve.db

# --- External Service Dependencies ---
# The application requires outbound HTTPS connectivity to the following:
oauth:
    # OAuth authentication provider - REQUIRED for all user authentication
    # The application initiates outbound connections to this service during login
    auth_url: "https://oauth.apache.org/auth"
    token_url: "https://oauth.apache.org/token"
    # redirect_uri is constructed from the application's own base_url
    # Ensure this matches the registered callback in the OAuth provider
    # allowed_redirect_hosts:
    #   - "your-steve-instance.apache.org"
```

Update `main.py` to use configuration:
```python
app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

# Use documented configuration for OAuth endpoints
asfquart.generics.OAUTH_URL_INIT = (
    f'{app.cfg.oauth.auth_url}?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = f'{app.cfg.oauth.token_url}?code=%s'
```

#### ASVS-1311-MED-002: Absence of Comprehensive Communication Architecture Documentation

**Location:**
- **File:** `v3/server/config.yaml.example`
- **Function:** N/A (configuration file)
- **Lines:** Entire file

**Related Functions Checked:**
- `main.py:create_app()` - OAuth configuration
- `main.py:run_standalone()` - Server binding
- `main.py:run_asgi()` - Production server
- `election.py` - Database access patterns

**Description:**

ASVS 13.1.1 at Level 2 requires all communication needs to be documented. The current `config.yaml.example` serves as the primary configuration documentation but provides incomplete coverage of the application's communication architecture.

**Communication Channels Analysis:**

| Communication Channel | Documented? | Location in Code |
|---|---|---|
| Inbound HTTP/HTTPS (server port) | ✅ Partial | `config.yaml.example` line 7 |
| TLS configuration | ✅ Yes | `config.yaml.example` lines 11-12 |
| SQLite database (local file) | ✅ Yes | `config.yaml.example` line 15 |
| Outbound: OAuth auth endpoint | ❌ No | `main.py` line 38 |
| Outbound: OAuth token endpoint | ❌ No | `main.py` line 40 |
| LDAP backend (per domain context) | ❌ No | Not in provided files |
| Inter-process: CLI tallying tools | ❌ No | Referenced in domain context |
| Inbound: OAuth callback (redirect_uri) | ❌ No | `main.py` line 38 |

**Data Flow:**

The application operates in a multi-component architecture:
1. **Web Server** → SQLite database (read/write)
2. **CLI Tools** → SQLite database (read/write)
3. **Web Server** → OAuth provider (outbound HTTPS)
4. **OAuth Provider** → Web Server (inbound callback)
5. **Web Server** → LDAP server (referenced but not implemented in provided files)

**Proof of Concept:**

A security auditor performing ASVS Level 2 compliance verification would:
1. Request documentation of all communication needs
2. Receive only `config.yaml.example`
3. Identify 3 documented channels out of 8 total channels
4. **Fail the audit** due to incomplete documentation

**Impact:**
- **Threat Modeling:** Incomplete threat model — undocumented channels cannot be risk-assessed
- **Network Security:** Firewall rules and network segmentation policies cannot be comprehensively derived
- **Security Audits:** Auditors cannot verify all communication paths without full source code review
- **Compliance:** Direct gap at ASVS Level 2 compliance

**Remediation:**

Add comprehensive communication section to configuration documentation:

```yaml
# config.yaml.example - Communication Architecture Documentation
#
# === COMMUNICATION OVERVIEW ===
# This application has the following communication needs:
#
# INBOUND:
#   - HTTPS on configured port (default 58383) from end users (browsers)
#   - OAuth callback from oauth.apache.org to /auth/callback
#
# OUTBOUND:
#   - HTTPS to oauth.apache.org (authentication)
#   - LDAPS to configured LDAP server (authorization lookups)
#
# LOCAL:
#   - SQLite database file (configured below)
#   - CLI tools (tally, admin) access the same database file directly
#
# USER-CONTROLLABLE DESTINATIONS:
#   - None. The application does not connect to user-specified URLs.
#   - OAuth redirect_uri is constructed server-side from the configured base_url.
#
# === END COMMUNICATION OVERVIEW ===

server:
    base_url: "https://steve.apache.org"  # Used to construct OAuth redirect_uri
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem

db: steve.db

oauth:
    auth_url: "https://oauth.apache.org/auth"
    token_url: "https://oauth.apache.org/token"

# ldap:
#     server: "ldaps://ldap.apache.org:636"
#     base_dn: "dc=apache,dc=org"
#     # bind_dn and bind_password should be provided via environment variables:
#     #   STEVE_LDAP_BIND_DN, STEVE_LDAP_BIND_PASSWORD
```

#### ASVS-1311-MED-003: Debug Logging Level Enabled by Default in Both Run Modes

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `run_standalone()`, `run_asgi()`
- **Lines:** 50, 91

**Related Functions Checked:**
- `election.py` - Contains commented debug prints of cryptographic material
- `crypto.py` - Cryptographic operations that could be logged

**Description:**

Both the standalone and ASGI (Hypercorn production) code paths configure `logging.DEBUG` as the default level. The `config.yaml.example` has no documented log-level setting. This creates a risk of sensitive data exposure in production logs.

**Vulnerable Code (standalone mode):**
```python
def run_standalone():
    logging.basicConfig(
        level=logging.DEBUG,  # ← DEBUG in production-facing code path
        ...
    )
```

**Vulnerable Code (ASGI/production mode):**
```python
def run_asgi():
    logging.basicConfig(
        level=logging.DEBUG,  # ← DEBUG in production deployment path
        ...
    )
    _LOGGER.setLevel(logging.DEBUG)  # ← Explicit DEBUG for app logger
```

**Evidence of Sensitive Data in Debug Context (`election.py`):**
```python
#print('SALT:', md.salt)
#print('KEY:', opened_key)
```

**Data Flow:**

1. Application starts in production mode (`run_asgi()`)
2. Logging configured to DEBUG level by default
3. Future debug logging statements capture cryptographic material
4. Logs written to disk/stdout
5. Sensitive data persisted in log aggregation systems

**Proof of Concept:**

If a developer adds debug logging during troubleshooting:
```python
# In election.py
_LOGGER.debug(f'Generated vote token: {vote_token}')
_LOGGER.debug(f'Encryption key: {opened_key}')
```

With DEBUG logging enabled by default:
1. These statements execute in production
2. Vote tokens and encryption keys appear in logs
3. Log aggregation systems capture sensitive material
4. Attackers with log access can compromise vote integrity

**Impact:**
- **Data Exposure:** Potential exposure of cryptographic material (salts, keys, tokens) in production logs
- **Configuration Gap:** No documented configuration to control logging verbosity
- **Compliance:** Domain context audit requirement states "debug mode is disabled in production config" — cannot be met with current configuration template
- **Operational Security:** Log aggregation systems may retain sensitive data indefinitely

**Remediation:**

Add logging configuration to `config.yaml.example`:
```yaml
# config.yaml.example
server:
    port: 58383
    # Log level: DEBUG, INFO, WARNING, ERROR, CRITICAL
    # IMPORTANT: Use INFO or higher in production to prevent
    # sensitive data exposure in logs
    log_level: INFO
```

Update `main.py` to use configuration:
```python
def run_asgi():
    log_level = getattr(logging, app.cfg.server.get('log_level', 'INFO').upper())
    logging.basicConfig(level=log_level, ...)
    _LOGGER.setLevel(log_level)
```

Update `run_standalone()` similarly:
```python
def run_standalone():
    log_level = getattr(logging, app.cfg.server.get('log_level', 'DEBUG').upper())
    logging.basicConfig(level=log_level, ...)
```

## Positive Security Patterns

1. **TLS Support Documented**: The `config.yaml.example` properly documents TLS certificate configuration with inline comments explaining proxy deployment model, demonstrating awareness of secure deployment practices.

2. **Strong Cryptographic Primitives**: `crypto.py` uses industry-standard cryptographic libraries:
   - `secrets` module for cryptographically secure random number generation
   - Argon2id for password hashing with appropriate parameters
   - HKDF for key derivation
   - Fernet for authenticated symmetric encryption

3. **Sensitive Field Exclusion**: The codebase demonstrates proper data minimization:
   - `get_metadata()` explicitly excludes SALT and OPENED_KEY from public API responses
   - `get_issue()` explicitly excludes salt from issue data
   - Prevents accidental exposure of cryptographic material through API endpoints

4. **State Machine Enforcement**: All mutation operations validate election state via assertions before proceeding, preventing invalid state transitions that could compromise election integrity.

5. **Transaction Safety**: Database operations that modify multiple tables use explicit `BEGIN TRANSACTION`/`COMMIT` patterns, ensuring atomic updates and preventing partial state corruption.

6. **Collision-Safe ID Generation**: `create_id()` uses `secrets.token_hex()` with retry logic on `IntegrityError`, properly handling the unlikely case of ID collisions.

7. **Vote Anonymization**: `crypto.shuffle()` implements Fisher-Yates shuffle with `secrets.randbelow()` for cryptographically secure randomization before tallying, protecting voter privacy.

## Architecture Observations

1. **Configuration Framework**: The application uses `asfquart.construct()` which integrates `config.yaml` with environment variable overrides. This is a solid pattern for twelve-factor app compliance, but the configuration template needs to document all available settings to be effective.

2. **Two-Mode Deployment**: Both `run_standalone()` and `run_asgi()` execution paths exist, suggesting development and production modes. However, security-critical settings (logging, debug) are identical in both, indicating no production hardening profile exists.

3. **Key Derivation Chain**: The multi-stage cryptographic derivation chain is well-designed:
   ```
   election_data → BLAKE2b → Argon2 → opened_key → vote_token → HKDF → Fernet key
   ```
   This approach provides defense in depth with appropriate parameter choices at each stage.

4. **OAuth Integration Uncertainty**: The comment at line 35 of `main.py` (`### is this really needed right now? # Avoid OIDC`) suggests the OAuth configuration may be provisional or under review, reinforcing the critical need for proper documentation of the authentication external dependency.

5. **Shared Database Architecture**: The domain context indicates CLI tools access the same SQLite database as the web server. This shared-state architecture requires careful consideration of file locking, transaction isolation, and concurrent access patterns.
