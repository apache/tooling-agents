## Findings

**No MEDIUM or higher severity findings identified for ASVS 13.2.3.**

After thorough analysis of all four provided source files, no default credentials for service authentication were found. The rationale:

1. **Database layer**: Uses SQLite, which is file-based and does not use credential-based authentication. (Acknowledged as intentional per the known false-positive patterns — auditing should focus on file permissions and SQL injection prevention instead.)

2. **OAuth integration**: Only OAuth endpoint URLs are configured in the provided code (`main.py:42-45`). The OAuth client secret is not present in these files; it is presumably managed by the `asfquart` framework through its environment variable integration, per the domain context.

3. **Cryptographic keys**: All encryption keys are derived dynamically through the `crypto.py` key derivation chain (`gen_salt()` → `gen_opened_key()` → `gen_vote_token()` → `_b64_vote_key()`). No static/default encryption keys exist.

4. **No hardcoded credentials**: A comprehensive search of all provided files found zero instances of patterns such as `root`, `admin`, `password`, `default`, `changeme`, `secret` as credential values, nor any fallback-to-default logic for service authentication.

---

## Positive Security Patterns

1. **Dynamic key derivation chain** (`crypto.py`): All cryptographic material is derived from `secrets.token_bytes()` salts and election data, with no static default keys anywhere in the derivation pipeline.

2. **No credential storage in config example** (`config.yaml.example`): The example configuration deliberately omits secrets, consistent with the domain context that secrets should be loaded from environment variables or secure vaults.

3. **Conditional TLS** (`main.py:82-84`): Certificate paths are loaded from configuration rather than hardcoded, and the example makes clear these are development certificates to be replaced:
   ```python
   if app.cfg.server.certfile:
       kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
       kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
   ```

4. **Cryptographically secure random generation** (`crypto.py:37`): Uses `secrets.token_bytes()` for all salts and `secrets.token_hex()` for IDs, ensuring no predictable/default values.

5. **Argon2 with reasonable parameters** (`crypto.py:92-102`): Key derivation uses Argon2 with `time_cost=2`, `memory_cost=65536` (64 MiB), `parallelism=4` — no weak default parameters.

## Architecture Observations

1. **Credential management is delegated to asfquart framework**: The `asfquart.construct()` call at `main.py:48` handles configuration loading, including (per domain context) environment variable integration for secrets. The provided code correctly avoids embedding secrets directly.

2. **SQLite's lack of credential auth shifts security to file permissions**: Since SQLite has no built-in authentication, the security of the database depends entirely on OS-level file permissions and access controls — a concern outside the scope of ASVS 13.2.3 but relevant to overall backend security.

3. **OAuth client secret configuration not visible**: The OAuth client secret, mentioned in the domain context, is managed outside the four provided files. Full verification of ASVS 13.2.3 for this credential requires reviewing the asfquart framework configuration and deployment scripts.

## Recommendations Summary

### Immediate Actions
- **Verify OAuth client secret is not a default** (Priority: HIGH — out of scope for this code review but critical for full ASVS 13.2.3 compliance):
  - Review the `asfquart` framework's OAuth client secret handling
  - Confirm it rejects startup if no client secret is configured (fail-closed)
  - Ensure no default/fallback client secret exists in the framework

### Short-term Improvements
- **Add startup validation for required secrets** (Priority: MEDIUM — defense-in-depth):
  ```python
  def create_app():
      app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
      
      # Fail fast if critical secrets are missing or default
      required_secrets = ['oauth_client_secret']  # add others as applicable
      for secret_name in required_secrets:
          value = getattr(app.cfg, secret_name, None)
          if not value:
              raise RuntimeError(f"Required secret '{secret_name}' is not configured")
          if value in ('changeme', 'default', 'secret'):
              raise RuntimeError(f"Secret '{secret_name}' appears to be a default value")
      
      return app
  ```

### Long-term Enhancements
- **Document credential configuration in example** (Priority: LOW):
  Add commented placeholders in `config.yaml.example` indicating which secrets are expected via environment variables:
  ```yaml
  # OAuth client secret: set via STEVE_OAUTH_SECRET environment variable
  # LDAP credentials: set via STEVE_LDAP_PASS environment variable
  # Do NOT place real secrets in this file
  ```
