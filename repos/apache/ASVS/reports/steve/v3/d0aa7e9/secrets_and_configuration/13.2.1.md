## Findings

### Medium Severity

#### ASVS-1321-MED-001

| Field | Value |
|-------|-------|
| **Severity** | MEDIUM |
| **Gap Type** | Type A — Entry point with no control |
| **File** | `v3/steve/election.py` |
| **Function** | `Election.open_database()` (line ~40) and all callers |
| **ASVS Requirement** | 13.2.1 — Data layer communications must be authenticated |

**Vulnerable Code:**

```python
# election.py, line ~40
@staticmethod
def open_database(db_fname):
    return asfpy.db.DB(db_fname, yaml_fname=QUERIES, yaml_section='election')
```

**All Entry Points (6 callers, 0 authenticated):**

| Caller | Line | Authentication | Status |
|--------|------|---------------|--------|
| `Election.__init__()` | ~46 | None | ❌ |
| `Election.create()` | ~365 | None | ❌ |
| `Election.open_to_pid()` | ~381 | None | ❌ |
| `Election.owned_elections()` | ~390 | None | ❌ |
| `Election.upcoming_to_pid()` | ~402 | None | ❌ |
| `Election.list_closed_election_ids()` | ~412 | None | ❌ |

**Data Flow:**
```
config.yaml (db path) → Election.open_database(db_fname) → asfpy.db.DB(db_fname) → SQLite file
                                                              ↑
                                              No authentication, no permission check,
                                              no encryption-at-rest verification
```

**Impact:**

The database stores high-value cryptographic material: `opened_key` (Argon2 hash enabling vote token derivation), per-voter `salt` values (enabling vote decryption when combined with opened_key), and encrypted vote ciphertext. While SQLite is an in-process library without native authentication, ASVS 13.2.1 requires data layer access to be authenticated. No compensating controls exist — the code does not verify file ownership, permissions, or employ database-level encryption (e.g., SQLCipher).

Any process running as the same user can open the database and extract `opened_key` and `salt` values, which are sufficient to derive vote tokens and decrypt all votes.

**Remediation:**

```python
import os
import stat

@staticmethod
def open_database(db_fname):
    db_path = pathlib.Path(db_fname)
    
    if db_path.exists():
        # Verify restrictive file permissions (owner read/write only)
        file_stat = db_path.stat()
        mode = stat.S_IMODE(file_stat.st_mode)
        if mode & (stat.S_IRGRP | stat.S_IWGRP | stat.S_IROTH | stat.S_IWOTH):
            raise PermissionError(
                f"Database {db_fname} has overly permissive mode {oct(mode)}. "
                f"Expected 0o600 or stricter."
            )
    
    db = asfpy.db.DB(db_fname, yaml_fname=QUERIES, yaml_section='election')
    
    # Set restrictive permissions on newly created databases
    if db_path.exists():
        os.chmod(db_path, 0o600)
    
    return db
```

Additionally, consider SQLCipher or application-level database encryption for the `opened_key` and `salt` columns.

#### ASVS-1321-MED-002

| Field | Value |
|-------|-------|
| **Severity** | MEDIUM |
| **Gap Type** | Type A — Backend communication with no visible authentication configuration |
| **File** | `v3/server/main.py` (lines ~39-43), `v3/server/config.yaml.example` |
| **Function** | `create_app()` |
| **ASVS Requirement** | 13.2.1 — Backend API communications must be authenticated, not with unchanging credentials |

**Vulnerable Code:**

```python
# main.py, lines ~39-43
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

```yaml
# config.yaml.example — COMPLETE file contents
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
db: steve.db
# NOTE: No OAuth client_id, client_secret, or any backend credential configuration
```

**Analysis:**

The application performs a backend-to-backend HTTP call to `oauth.apache.org/token` during the OAuth token exchange. This requires OAuth client credentials (`client_id` and `client_secret`). However:

1. **`config.yaml.example`** contains zero credential-related configuration — no OAuth settings, no environment variable references, no vault integration examples
2. **`main.py`** configures OAuth URLs but no visible credential passing
3. The `asfquart` framework presumably manages OAuth client secrets, but this is opaque from the application's perspective
4. No documentation in the configuration template indicates how secrets should be provided (environment variables, vault references, etc.)

**Impact:**

Without visible credential management configuration, it is impossible to verify that:
- OAuth client secrets are not hardcoded in the framework or deployment
- Secrets can be rotated without code changes
- Each deployment environment uses distinct credentials
- Credentials follow ASVS 13.2.1 requirements (not unchanging shared passwords)

This also affects other secrets mentioned in the domain context (LDAP credentials, database encryption keys) that have no visible configuration mechanism.

**Remediation:**

Update `config.yaml.example` to document the expected secret management pattern:

```yaml
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem

db: steve.db

# OAuth backend communication
# Credentials MUST be provided via environment variables:
#   STEVE_OAUTH_CLIENT_ID - OAuth client identifier
#   STEVE_OAUTH_CLIENT_SECRET - OAuth client secret (rotate quarterly)
# Do NOT place actual credentials in this file.
oauth:
    client_id: ${STEVE_OAUTH_CLIENT_ID}
    client_secret: ${STEVE_OAUTH_CLIENT_SECRET}

# All secrets should be loaded from environment variables or a secrets vault.
# See deployment documentation for required environment variables.
```

In application code, verify credential source at startup:

```python
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    # Verify secrets are not hardcoded defaults
    if hasattr(app.cfg, 'oauth'):
        if app.cfg.oauth.client_secret and not app.cfg.oauth.client_secret.startswith('${'):
            _LOGGER.warning("OAuth client_secret appears to be a literal value in config. "
                          "Use environment variables for secrets.")
    
    return app
```

## Positive Security Patterns

1. **Sensitive data stripping in public API** (`election.py:get_metadata()`, line ~157):
   ```python
   # NOTE: do not return the SALT or OPENED_KEY
   return edict(eid=md.eid, title=md.title, ...)
   ```
   The `_all_metadata()` method is correctly marked as internal-only, and `get_metadata()` explicitly excludes `salt` and `opened_key`.

2. **Cryptographically secure random generation** (`crypto.py`):
   All randomness uses `secrets` module (`token_bytes`, `token_hex`, `randbelow`) — no use of `random` module for security-critical operations.

3. **Per-voter unique salts** (`election.py:add_salts()`, line ~120):
   Each voter/issue pair receives a unique `crypto.gen_salt()` value, stored in the `mayvote` table. This prevents vote correlation even if the database is compromised.

4. **Multi-tier key derivation** (`crypto.py`):
   Election data → BLAKE2b → Argon2 → `opened_key` → per-voter `vote_token` → HKDF → `vote_key` → Fernet encryption. Keys are never stored directly; they are derived from salts when needed.

5. **TLS support available** (`main.py:run_standalone()`, line ~73):
   ```python
   if app.cfg.server.certfile:
       kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
       kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
   ```

6. **Transactional integrity for sensitive operations** (`election.py`):
   Salt generation (`add_salts`) and election deletion (`delete`) use explicit `BEGIN TRANSACTION`/`COMMIT` blocks.

7. **Vote shuffling before tally return** (`election.py:tally_issue()`, line ~296):
   `crypto.shuffle(votes)` uses Fisher-Yates with `secrets.randbelow()` to eliminate database-order information leakage.

## Architecture Observations

1. **Assert-based authorization**: State checks use `assert` statements (e.g., `assert self.is_editable()`). While not directly related to ASVS 13.2.1, `assert` statements are disabled when Python runs with `-O` flag. Security-critical state validations should use explicit conditional checks that raise exceptions. This affects authorization controls throughout `election.py`.

2. **Framework opacity**: The `asfquart` framework handles OAuth flow, configuration loading, and session management. Without access to framework source, several ASVS 13.2.1 aspects cannot be fully verified, including how OAuth client credentials are stored and transmitted to the OAuth provider.

3. **Single-tier architecture**: The application, database, and crypto operations all run in the same process. This means "backend communication" is primarily in-process function calls, which reduces the ASVS 13.2.1 attack surface compared to a distributed architecture. The primary external backend communication is the OAuth token exchange.

4. **Debug logging in both run modes**: Both `run_standalone()` and `run_asgi()` configure `logging.DEBUG` level. While current log statements don't expose secrets (commented-out prints for SALT/KEY suggest awareness), debug logging in production increases the risk of future accidental secret exposure during maintenance.
