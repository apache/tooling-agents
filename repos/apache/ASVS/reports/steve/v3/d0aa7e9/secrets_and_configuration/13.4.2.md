## Findings

### Medium Severity

#### ASVS-1342-MED-001
**Title:** DEBUG Logging Level Hardcoded in Production ASGI Deployment Path

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `run_asgi()`
- **Lines:** 90–107

**Related Functions Checked:**
- `run_asgi()` — production ASGI entry point
- `election.py` debug logging calls (lines ~39, ~176, ~398)

**Description:**

The production ASGI deployment path unconditionally sets the root logger and application logger to DEBUG level, with no configuration mechanism to override this behavior. This causes all DEBUG-level messages from the application and its dependencies to be emitted in production logs.

**Vulnerable Code:**
```python
def run_asgi():
    "Run as an ASGI process; eg. Hypercorn"

    # NOTE: no-op if Hypercorn has set up the root logger.
    logging.basicConfig(
        level=logging.DEBUG,          # <-- Root logger set to DEBUG
        style='{',
        format='[{asctime}|{levelname}|{name}] {message}',
        datefmt=DATE_FORMAT,
    )

    # Make some loggers explicitly INFO, rather than default to DEBUG.
    logging.getLogger('watchfiles.main').setLevel(logging.INFO)

    # Configure our app's loggers (rather than root logger default).
    _LOGGER.setLevel(logging.DEBUG)   # <-- App logger unconditionally DEBUG
```

**Data Flow:**

`run_asgi()` (production entry point via Hypercorn) → `logging.basicConfig(level=DEBUG)` + `_LOGGER.setLevel(DEBUG)` → all DEBUG-level messages from application and dependencies emitted in production logs

**Consuming debug messages in `election.py`:**
```python
_LOGGER.debug(f'{op} election ID "{eid}"')          # line ~39 - leaks election IDs
_LOGGER.debug('IID conflict(!!) ... trying again.')  # line ~176 - leaks internal state
_LOGGER.debug('EID conflict(!!) ... trying again.')  # line ~398 - leaks internal state
```

**Gap Type:** Type A — No control exists to disable debug logging in production.

**Impact:**
- Election IDs are logged at DEBUG level during every election operation, providing an attacker with log access a roadmap of election activity.
- All third-party library DEBUG output (asfquart, asfpy.db, cryptography, argon2) is captured, potentially exposing query structures, timing information, and internal framework state.
- DEBUG-level logging from the `pages` and `api` modules (imported but not provided) may log request parameters, headers, or authentication details.
- Verbose logs increase the attack surface if log files are exposed through misconfiguration, log aggregation services, or file inclusion vulnerabilities.

**Proof of Concept:**

When deployed with Hypercorn (`uv run python -m hypercorn main:steve_app`), the `run_asgi()` path is executed. Any request that opens or creates an election produces:
```
[01/15 10:30|DEBUG|steve.election] Opening election ID "a1b2c3d4e5"
```
This information leakage occurs for every election operation without any configuration option to suppress it.

**Remediation:**

Add a configurable logging/debug level to `config.yaml` and use it in `run_asgi()`:

```yaml
# config.yaml
server:
    port: 58383
    debug: false
    log_level: INFO   # DEBUG, INFO, WARNING, ERROR
```

```python
def run_asgi():
    "Run as an ASGI process; eg. Hypercorn"
    
    # Create app first to access configuration
    global steve_app
    steve_app = create_app()
    
    # Use configured log level; default to INFO for production safety
    log_level_name = getattr(steve_app.cfg.server, 'log_level', 'INFO')
    log_level = getattr(logging, log_level_name.upper(), logging.INFO)
    
    logging.basicConfig(
        level=log_level,
        style='{',
        format='[{asctime}|{levelname}|{name}] {message}',
        datefmt=DATE_FORMAT,
    )
    
    _LOGGER.setLevel(log_level)
    _LOGGER.info(' ** Run-mode: ASGI')
```

#### ASVS-1342-MED-002
**Title:** No Configuration Mechanism to Disable Debug Mode — Both Execution Paths Default to DEBUG

**Location:**
- **File:** `v3/server/main.py`
  - **Function:** `run_standalone()`
  - **Lines:** 51–87
  - **Function:** `run_asgi()`
  - **Lines:** 90–107
- **File:** `v3/server/config.yaml.example` (entire file — no debug/logging settings)

**Related Functions Checked:**
- `run_standalone()` — standalone server entry point
- `run_asgi()` — ASGI server entry point
- `create_app()` — application factory

**Description:**

The configuration schema has no mechanism to control debug behavior. Both execution paths (standalone and ASGI) hardcode DEBUG logging with no conditional logic to check environment or configuration settings. The `config.yaml.example` template contains no debug, log_level, or environment settings.

**Vulnerable Code in `run_standalone()`:**
```python
def run_standalone():
    "Run as a standalone server."

    logging.basicConfig(
        level=logging.DEBUG,    # <-- Always DEBUG, no config check
        style='{',
        ...
    )
```

**Vulnerable Code in `config.yaml.example`:**
```yaml
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
db: steve.db
# No debug, log_level, or environment settings exist
```

**Gap Type:** Type A — The configuration schema has no mechanism to control debug behavior; the code has no conditional logic to check an environment or configuration setting before enabling debug logging.

**Impact:**
- Every deployment — whether standalone or ASGI — runs with DEBUG logging by default with no documented or implemented way to change this via configuration.
- Operators deploying the application have no way to harden logging without modifying source code, violating the principle that production configurations should be hardened by default.
- The standalone mode also uses `app.runx()` with `extra_files` for hot-reloading, which is a development feature. If `run_standalone()` is used in production (it's documented as a valid execution path), file-watching and reloading capabilities are active.

**Proof of Concept:**

1. Deploy using either documented method:
   - `./main.py` (standalone) 
   - `uv run python -m hypercorn main:steve_app` (ASGI)
2. Both produce DEBUG-level output with no configuration option to change this behavior.
3. The `config.yaml.example` template, which serves as the basis for production configs, contains no logging or debug settings.

**Remediation:**

Extend configuration schema and implement environment-aware defaults:

```yaml
# config.yaml.example
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
    
    # Set to false in production (default: false)
    debug: false
    
    # Logging level: DEBUG, INFO, WARNING, ERROR (default: INFO)
    log_level: INFO

db: steve.db
```

```python
def _configure_logging(cfg):
    """Configure logging based on application config."""
    level_name = getattr(cfg.server, 'log_level', 'INFO')
    level = getattr(logging, level_name.upper(), logging.INFO)
    
    logging.basicConfig(
        level=level,
        style='{',
        format='[{asctime}|{levelname}|{name}] {message}',
        datefmt=DATE_FORMAT,
    )
    _LOGGER.setLevel(level)
    return level


def run_standalone():
    app = create_app()
    level = _configure_logging(app.cfg)
    
    if getattr(app.cfg.server, 'debug', False):
        _LOGGER.warning('Running in DEBUG mode — not for production use')
        # Only enable extra_files watching in debug mode
        extra_files = {steve.election.QUERIES}
        # ... rest of debug-specific setup
    else:
        extra_files = set()
    
    # ...
```

## Positive Security Patterns

1. **No framework debug mode enabled**: `asfquart.construct()` is called without `debug=True`, meaning the Quart/Flask interactive debugger and stack trace display are not enabled — this is the most critical aspect of debug mode and it's handled correctly.

2. **Sensitive debug prints are commented out**: In `election.py`, lines that would print salts and opened keys (`#print('SALT:', salt)`, `#print('KEY:', opened_key)`) are commented out, showing awareness that these values should not be logged.

3. **`get_metadata()` filters sensitive fields**: The public metadata accessor explicitly excludes `salt` and `opened_key` columns, with a clear comment: `# NOTE: do not return the SALT or OPENED_KEY`.

4. **`owned_elections()` filters sensitive data**: Contains explicit comment `# NOTE: contains subset of columns. We don't want to return the SALT or OPENED_KEY values.`

5. **Crypto benchmark is guarded**: `benchmark_argon2()` in `crypto.py` is behind `if __name__ == '__main__'`, preventing it from executing when imported as a module.

6. **Some loggers are elevated to INFO**: Both `run_standalone()` and `run_asgi()` explicitly raise certain noisy loggers (`asyncio`, `hpack`, `sslproto`, `watchfiles.main`) to INFO, showing awareness of log verbosity — this pattern just needs to be extended to the application loggers.

---

## Architecture Observations

1. **No environment distinction**: The application has no concept of "production" vs "development" environments. Both code paths (`run_standalone` and `run_asgi`) apply the same DEBUG logging. A robust deployment should have environment-aware configuration.

2. **Configuration schema is minimal**: `config.yaml.example` only covers server port, TLS certs, and database path. It lacks settings for logging level, debug mode, environment name, and other operational parameters that affect security posture.

3. **asfquart integration is opaque**: The `asfquart` framework may have its own debug settings loaded via environment variables (mentioned in domain context as "asfquart's environment variable integration"). Without visibility into this framework's debug behavior, there may be additional debug surfaces not covered by this audit.
