## Findings

### Medium Severity

#### ASVS-1325-MED-001: No configurable allowlist for outbound server connections or resource loading

**Gap Type:** Type A — No control exists

**Location:**
- **File:** `v3/server/config.yaml.example` (entire file)
- **File:** `v3/server/main.py` (lines 37-40)
- **Functions Reviewed:** Configuration loading, OAuth URL initialization, file path resolution

**Description:**

The application configuration (`config.yaml`) provides no mechanism to define an allowlist of permitted outbound hosts, services, or resource locations. The server makes outbound connections (OAuth token exchange) and loads files from configurable paths, but none of these are governed by an explicit allowlist.

**Vulnerable Configuration:**

```yaml
# config.yaml.example
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem

db: steve.db
# NOTE: No 'allowed_hosts', 'allowed_resources', or 'outbound_allowlist' section
```

**Outbound Connection Endpoints:**

```python
# main.py:37-40
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**
- **Outbound HTTP**: Server → `oauth.apache.org` (hardcoded, not configurable allowlist)
- **File loading**: Server → filesystem paths from `config.yaml` (no path restrictions)
- **Missing**: No allowlist configuration that would prevent SSRF if new endpoints introduce user-controllable URL parameters, and no restriction on which file paths the server may access

**Impact:**

1. If any future endpoint (in the unshown `pages` or `api` modules) processes user-supplied URLs for fetching or redirecting, there is no application-level SSRF prevention.
2. The `db` path in configuration accepts arbitrary filesystem paths without restriction to an approved directory scope.
3. No defense-in-depth against outbound connection abuse if the OAuth library or another dependency is exploited to make requests to unintended hosts.

**Proof of Concept:**

A configuration like the following would be accepted without restriction:

```yaml
db: /etc/passwd          # No path validation or allowed directory list
# Or if an attacker modifies config:
db: ../../sensitive.db   # Path traversal in db path
```

While `config.yaml` is admin-controlled (reducing direct exploitability), ASVS 13.2.5 explicitly requires a **configured allowlist** regardless of who controls the configuration—the point is to enforce least-privilege for server outbound communication.

**Remediation:**

Add an allowlist section to `config.yaml` and enforce it at application startup:

```yaml
# config.yaml - Add outbound communication allowlist
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem

db: steve.db

# ASVS 13.2.5: Allowlist of permitted outbound hosts and resource paths
allowed_outbound_hosts:
  - oauth.apache.org

allowed_file_paths:
  - /opt/steve/data/    # Database files
  - /opt/steve/certs/   # TLS certificates
```

Enforcement in `main.py`:

```python
import urllib.parse
import pathlib

def validate_outbound_allowlist(app):
    """Verify all configured outbound endpoints are in the allowlist."""
    allowed_hosts = set(app.cfg.get('allowed_outbound_hosts', []))
    
    # Validate OAuth endpoints against allowlist
    for url_template in [
        asfquart.generics.OAUTH_URL_INIT,
        asfquart.generics.OAUTH_URL_CALLBACK,
    ]:
        # Extract host from URL template (handle %s placeholders)
        parsed = urllib.parse.urlparse(url_template.split('?')[0])
        if parsed.hostname not in allowed_hosts:
            raise RuntimeError(
                f"Outbound host '{parsed.hostname}' not in allowed_outbound_hosts"
            )

def validate_file_paths(app):
    """Verify configured file paths are within allowed directories."""
    allowed_paths = [pathlib.Path(p).resolve() for p in app.cfg.get('allowed_file_paths', [])]
    
    db_path = pathlib.Path(app.cfg.db).resolve()
    if not any(db_path.is_relative_to(ap) for ap in allowed_paths):
        raise RuntimeError(f"Database path '{db_path}' not within allowed_file_paths")
```

## Architecture Observations

1. **Limited outbound surface (positive):** The visible codebase has a narrow outbound communication surface—only OAuth token exchange. This reduces the practical risk, but ASVS 13.2.5 still requires an explicit allowlist.

2. **Unreviewed modules:** The `pages` and `api` modules (imported at `main.py:43-44`) are not included in this review. These are the HTTP endpoint handlers and are the most likely location for SSRF vulnerabilities if they process user-supplied URLs. A complete audit requires reviewing these files.

3. **Framework dependency:** The `asfquart` framework handles OAuth flows internally. Its HTTP client behavior and any built-in SSRF protections are not visible in the provided code.

4. **File path handling:** Path construction uses `pathlib.Path` with relative resolution (`CERTS_DIR /`), which provides some implicit containment but is not equivalent to an allowlist.
