## Findings

### High Severity

#### ASVS-1333-HIGH-001: Master Election Key (`opened_key`) Stored in Application Database Co-located with Encrypted Votes

**Severity:** HIGH  
**Gap Type:** Type A — No control exists for key storage isolation

**Location:**
- **File:** `v3/steve/election.py`
- **Functions:** `open()` (line 67-84), `_all_metadata()` (line 118-131)
- **Related Functions:** `add_vote()` (line 222-236), `tally_issue()` (line 257-299), `has_voted_upon()` (line 238-256)

**Vulnerable Code:**

Storage at election open (line 83):
```python
def open(self, pdb):
    # ...
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)
    self.c_open.perform(salt, opened_key, self.eid)  # Stored directly in SQLite
```

Retrieval for every crypto operation (line 125):
```python
def _all_metadata(self, required_state=None):
    md = self.q_metadata.first_row(self.eid)  # opened_key retrieved from DB
    # ...
    return md  # opened_key exposed to all callers
```

Usage in vote operations (lines 229-232):
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)  # opened_key loaded from DB
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
```

**Data Flow:**

```
crypto.gen_opened_key() → opened_key (in memory)
    → c_open.perform() → SQLite metadata table (at rest, unprotected)
    → q_metadata.first_row() → md.opened_key (back to memory)
    → crypto.gen_vote_token() → vote_token → crypto.create_vote() → ciphertext
                                                                      ↓
                                              SQLite votes table ←────┘
                                              SQLite mayvote table (salts)
```

All three components needed for complete de-anonymization (`opened_key`, per-voter `salt`, encrypted `ciphertext`) reside in the **same SQLite database file**. No external secret is required.

**Proof of Concept:**

An attacker with read access to `steve.db` can reconstruct every vote-to-voter mapping:

```python
import sqlite3
from steve.crypto import gen_vote_token, decrypt_votestring

db = sqlite3.connect('steve.db')
# Step 1: Extract opened_key from metadata
row = db.execute("SELECT salt, opened_key FROM metadata WHERE eid=?", (eid,)).fetchone()
opened_key = row[1]

# Step 2: For each voter/issue pair, derive vote_token and decrypt
for pid, iid, salt in db.execute("SELECT pid, iid, salt FROM mayvote WHERE iid IN (SELECT iid FROM issue WHERE eid=?)", (eid,)):
    vote_token = gen_vote_token(opened_key, pid, iid, salt)
    row = db.execute("SELECT ciphertext FROM vote WHERE vote_token=? ORDER BY rowid DESC LIMIT 1", (vote_token,)).fetchone()
    if row:
        votestring = decrypt_votestring(vote_token, salt, row[0])
        print(f"Voter {pid} voted: {votestring}")  # Complete de-anonymization
```

With Argon2 parameters (time_cost=2, memory_cost=64MB), each `gen_vote_token` call takes ~100-200ms. For 500 voters × 5 issues = 2,500 computations ≈ **4-8 minutes** for complete de-anonymization of an entire election. This is not meaningful "cryptographic separation."

**Impact:**

A single database file compromise (backup leak, unauthorized filesystem access, misconfigured permissions) enables an attacker to **decrypt every vote and attribute it to the specific voter who cast it**, completely defeating the stated security goal of voter anonymity. No additional secret or privilege is required beyond read access to the SQLite file. ASVS 13.3.3 requires that key material be managed in an isolated security module specifically to prevent this class of attack — where compromise of the data store also compromises the keys protecting that data.

**Remediation:**

Store the `opened_key` in an external secrets manager, requiring a separate compromise vector. At minimum, split the key derivation so that one component comes from outside the database:

```python
# Option A: Use HashiCorp Vault or cloud KMS
import hvac

def open(self, pdb):
    assert self.is_editable()
    self.add_salts()
    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)

    # Store key in vault, only store a reference in DB
    vault_client = hvac.Client(url=VAULT_URL, token=VAULT_TOKEN)
    vault_client.secrets.kv.v2.create_or_update_secret(
        path=f'elections/{self.eid}',
        secret={'opened_key': base64.b64encode(opened_key).decode()}
    )
    # Store salt in DB (without opened_key)
    self.c_open.perform(salt, None, self.eid)  # opened_key NOT in DB

# Option B: Minimum — XOR opened_key with a master key from environment variable
import os
MASTER_KEY = os.environ['STEVE_MASTER_KEY']  # 32 bytes, hex-encoded

def _protect_key(opened_key: bytes) -> bytes:
    master = bytes.fromhex(MASTER_KEY)
    return bytes(a ^ b for a, b in zip(opened_key, master))
```

#### ASVS-1333-MED-002: No Secret Management System Integration for Application Configuration

**Severity:** MEDIUM  
**Gap Type:** Type A — No control exists

**Location:**
- **File:** `v3/server/config.yaml.example` (entire file)
- **File:** `v3/server/main.py` (line 41)

**Vulnerable Code:**

`config.yaml.example` — no vault references, no environment variable patterns:
```yaml
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
db: steve.db
```

`main.py` — app created from config file with no vault integration:
```python
app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
```

**Data Flow:**

```
config.yaml → asfquart.construct() → app.cfg.* (all config in memory)
                                          ↓
                         No vault/KMS/env-var substitution visible
```

**Impact:**

The domain context identifies that the application manages OAuth client secrets, database encryption keys, LDAP credentials, and Fernet encryption keys. The configuration system shows no integration with any secret management solution (HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, SOPS, etc.). The example configuration template contains no placeholder patterns for environment variable substitution (e.g., `${ENV_VAR}`) or vault references. While `asfquart` may support environment variable overrides at the framework level, no documentation or configuration patterns demonstrate secure secret injection for any of the identified secret categories.

This means that secrets are likely stored in plaintext in the configuration file or passed directly as environment variables without centralized management, rotation support, or access auditing.

**Remediation:**

Integrate a secret management system and document the pattern:

```yaml
# config.yaml with vault references
server:
    port: 58383
    certfile: vault://secret/steve/tls#certfile
    keyfile: vault://secret/steve/tls#keyfile

oauth:
    client_secret: env://STEVE_OAUTH_SECRET  # or vault reference

db: steve.db
# Database encryption key from vault
db_encryption_key: vault://secret/steve/db#key
```

Or at minimum, document environment variable usage:
```yaml
# All secrets MUST be provided via environment variables:
# STEVE_OAUTH_SECRET, STEVE_DB_KEY, STEVE_LDAP_PASS
# See deployment/secrets.md for provisioning instructions
```

#### ASVS-1333-MED-003: Key Material Exposed in Python Exception Tracebacks and Debug Logging

**Severity:** MEDIUM  
**Gap Type:** Type A — No control exists for key material exposure prevention

**Location:**
- **File:** `v3/steve/crypto.py` — lines 38-80 (all crypto functions)
- **File:** `v3/steve/election.py` — lines 222-236 (`add_vote`), 257-299 (`tally_issue`), 238-256 (`has_voted_upon`)
- **File:** `v3/server/main.py` — lines 57, 97 (DEBUG logging level)

**Vulnerable Code:**

Key material passed as plain function parameters (election.py lines 229-232):
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    # If any exception occurs here, traceback shows: opened_key, vote_token, salt, votestring
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

No exception wrapping in crypto.py (lines 69-73):
```python
def create_vote(vote_token: bytes, salt: bytes, votestring: str) -> bytes:
    b64key = _b64_vote_key(vote_token, salt)  # Derived key in local variable
    f = cryptography.fernet.Fernet(b64key)    # Key loaded into Fernet
    return f.encrypt(votestring.encode())      # If encrypt fails, traceback shows b64key
```

Debug logging enabled in production entry points (main.py lines 56-57):
```python
def run_standalone():
    logging.basicConfig(level=logging.DEBUG, ...)
```

```python
def run_asgi():
    logging.basicConfig(level=logging.DEBUG, ...)
    _LOGGER.setLevel(logging.DEBUG)
```

Evidence of historical key material logging (election.py lines 80-81, commented out):
```python
#print('SALT:', salt)
#print('KEY:', opened_key)
```

**Data Flow:**

```
Normal flow: opened_key → vote_token → b64key → Fernet → ciphertext
                                                              ↓
Exception flow:  Any error at any step → Python traceback
                                              ↓
                  Traceback includes ALL local variables and function arguments
                                              ↓
                  Logging framework (at DEBUG level) → log files/stdout
                                              ↓
                  opened_key, vote_token, b64key, plaintext votestring EXPOSED
```

**Impact:**

If any exception occurs during cryptographic operations (database connectivity issues, malformed data, library errors), Python's default exception handling will include all function arguments and local variables in the traceback. With logging set to DEBUG, these tracebacks will be written to logs. The exposed key material includes the master `opened_key`, per-voter `vote_token`, derived Fernet encryption keys (`b64key`), and plaintext vote content (`votestring`). Log aggregation systems, monitoring tools, or log file access could then expose this material.

The commented-out `print` statements for `SALT` and `KEY` demonstrate that this exposure has occurred during development, creating risk of re-enabling during debugging.

**Remediation:**

1. Wrap cryptographic operations in exception handlers that sanitize key material:

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    try:
        vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
        ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
        self.c_add_vote.perform(vote_token, ciphertext)
    except Exception as e:
        _LOGGER.error(f'Vote encryption failed for issue {iid}: {type(e).__name__}')
        raise VoteProcessingError(iid) from None  # "from None" suppresses original traceback
    finally:
        # Clear local key references
        vote_token = None
        ciphertext = None
```

2. Set production logging to INFO or WARNING:

```python
def run_asgi():
    log_level = os.environ.get('STEVE_LOG_LEVEL', 'INFO')
    logging.basicConfig(level=getattr(logging, log_level), ...)
```

3. Remove commented-out key printing statements entirely from the codebase.

## Architecture Observations

1. **Single Security Boundary:** The entire system (application logic, cryptographic operations, key storage, encrypted data) operates within a single security boundary — the application server and its SQLite database file. ASVS 13.3.3 Level 3 expects at least two boundaries: the application and an isolated security module.

2. **Defense-in-Depth Gap:** The cryptographic design is sound in isolation (Argon2 + HKDF + Fernet with per-voter salts), but all defensive layers collapse upon database compromise because the root key (`opened_key`) is co-located with the data it protects. Introducing a single external secret (vault-stored key, environment-provided master key) would make the entire cryptographic chain resistant to database-only compromise.

3. **Operational Logging Risk:** Both server entry points (`run_standalone`, `run_asgi`) default to `DEBUG` logging level. Combined with the lack of exception wrapping around cryptographic operations, this creates a systemic risk of key material appearing in operational logs.

4. **Crypto Module Abstraction:** The `crypto.py` module provides a clean abstraction layer over cryptographic primitives. This is architecturally good — it would be straightforward to introduce a vault/HSM-backed implementation behind the same interface without changing callers.
