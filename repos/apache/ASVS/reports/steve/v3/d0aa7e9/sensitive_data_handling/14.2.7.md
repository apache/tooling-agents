### Summary

ASVS 14.2.7 requires that sensitive information be subject to data retention classification, with outdated or unnecessary data deleted automatically, on schedule, or as the situation requires. **The codebase has no data retention classification, no scheduled cleanup, and no mechanisms to purge obsolete sensitive data.** This is a systemic gap affecting all sensitive data categories in the system.

### HIGH Severity

#### ASVS-1427-HIGH-001: No Data Retention Classification for Any Sensitive Data Category

**Severity:** HIGH

**Location:**
- **Files:** System-wide (schema.sql, election.py, persondb.py, pages.py)
- **Functions:** N/A (systemic gap)
- **Lines:** N/A

**Related Functions Checked:**
- `election.py:delete()` (lines ~180-200)
- `election.py:open()` (lines 64-78)
- `election.py:close()` (lines ~80-90)
- `persondb.py:delete_person()` (lines 51-64)

**Description:**

The system handles multiple categories of sensitive data — encrypted votes, voter PII (names, emails), per-voter cryptographic salts, election keys, and voter-to-issue mappings — but no data retention classification exists. There are no retention period definitions, no expiration timestamps in the schema, and no administrative interfaces or scheduled processes for data lifecycle management.

**Vulnerable Code:**

```sql
-- v3/schema.sql — vote table
CREATE TABLE vote (
    vid  INTEGER PRIMARY KEY AUTOINCREMENT,
    vote_token  BLOB NOT NULL  CHECK (length(vote_token) = 32),
    ciphertext  BLOB NOT NULL
) STRICT;
-- No created_at, no expires_at, no retention_class column
```

```sql
-- v3/schema.sql — person table
CREATE TABLE person (
    pid  TEXT PRIMARY KEY NOT NULL,
    name  TEXT,
    email  TEXT NOT NULL
) STRICT;
-- No created_at, no last_activity, no retention period
```

```python
# v3/steve/election.py — only deletion method is restricted to editable state
def delete(self):
    "Delete this Election and its Issues and Person/Issue pairs."
    # Can't delete if it has been opened (even if later closed).
    assert self.is_editable()
    # ...
```

```python
# v3/server/pages.py — past elections feature is unimplemented
result.past_elections = []  ### TBD
```

**Data Flow:**

Sensitive data enters → is stored → remains indefinitely with no exit path.

**Proof of Concept:**

N/A (systemic architectural gap)

**Impact:**

Violates ASVS 14.2.7 requirement for data retention classification. Over time, the system accumulates an ever-growing corpus of sensitive data including voter PII and cryptographic material, expanding the attack surface and the impact of any database compromise.

**Remediation:**

1. Define a data retention classification document mapping each data type to a retention period:

| Data Type | Classification | Retention Period | Trigger |
|-----------|---------------|-----------------|---------|
| Encrypted votes | Audit-critical | Per-policy (e.g., 2 years post-close) | Election close date |
| Election keys (salt, opened_key) | Cryptographic | Delete after final tally verified | Tally completion |
| Per-voter salts (mayvote.salt) | Cryptographic | Delete after final tally verified | Tally completion |
| Person PII (name, email) | PII | Delete when no active elections reference them | Last election closed + retention |
| Superseded votes | Unnecessary | Immediate | Upon re-vote |

2. Add schema support:
```sql
ALTER TABLE election ADD COLUMN tallied_at INTEGER;  -- when tally was completed
ALTER TABLE vote ADD COLUMN created_at INTEGER NOT NULL DEFAULT (strftime('%s','now'));
```

3. Implement a periodic cleanup process or CLI command.

#### ASVS-1427-HIGH-002: Election Cryptographic Key Material Persisted Indefinitely After Use

**Severity:** HIGH

**Location:**
- **File:** `v3/schema.sql` (election table), `v3/steve/election.py`
- **Functions:** `open()`, `close()`, `tally_issue()`
- **Lines:** 64-78 (`open`), 217-255 (`tally_issue`)

**Related Functions Checked:**
- `election.py:open()` (lines 64-78)
- `election.py:close()` (lines ~80-90)
- `election.py:tally_issue()` (lines 217-255)
- `election.py:add_salts()` (lines ~50-60)

**Description:**

When an election is opened, a 16-byte `salt` and 32-byte `opened_key` are stored in the `election` table. The `opened_key` is derived from the election definition and used to generate `vote_tokens`, which in turn derive per-vote encryption keys. After an election is closed and tallied, these cryptographic values remain in the database forever. There is no mechanism to purge them after they are no longer needed.

**Vulnerable Code:**

```python
# v3/steve/election.py — open()
def open(self, pdb):
    assert self.is_editable()
    self.add_salts()
    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)
    self.c_open.perform(salt, opened_key, self.eid)
    # Keys persist in database indefinitely after this point
```

```python
# v3/steve/election.py — close()
def close(self):
    "Close an election."
    assert self.is_open()
    self.c_close.perform(self.eid)
    # Only sets closed=1, does not purge salt/opened_key
```

```sql
-- v3/schema.sql
CREATE TABLE election (
    -- ...
    salt  BLOB  CHECK (salt IS NULL OR length(salt) = 16),
    opened_key  BLOB  CHECK (opened_key IS NULL OR length(opened_key) = 32),
    -- No mechanism to purge after tallying
) STRICT;
```

```sql
CREATE TABLE mayvote (
    -- ...
    salt  BLOB  CHECK (salt IS NULL OR length(salt) = 16),
    -- No mechanism to purge after tallying
) STRICT;
```

**Data Flow:**

`open()` generates keys → keys stored in DB → `tally_issue()` uses keys to decrypt → keys remain in DB after tally → indefinite window for key compromise.

**Proof of Concept:**

N/A (design issue requiring database inspection)

**Impact:**

The combination of `election.opened_key` + `election.salt` + per-voter `mayvote.salt` values enables decryption of all votes in an election. After tallying is complete, these keys serve no operational purpose, but their continued presence means that a future database compromise would allow retroactive decryption of votes from all past elections, violating the system's ballot secrecy goal.

**Remediation:**

```python
def purge_keys(self):
    """Purge cryptographic material after tallying is verified.
    
    Call after tally results have been confirmed/exported.
    This is irreversible — votes can no longer be decrypted.
    """
    assert self.is_closed()
    
    self.db.conn.execute('BEGIN TRANSACTION')
    # Null out election-level keys
    self.c_purge_election_keys.perform(self.eid)
    # Null out per-voter salts  
    self.c_purge_mayvote_salts.perform(self.eid)
    self.db.conn.execute('COMMIT')
    
    _LOGGER.info(f'Purged cryptographic material for election[E:{self.eid}]')
```

Add queries:
```yaml
c_purge_election_keys: UPDATE election SET salt = NULL, opened_key = NULL WHERE eid = ? AND closed = 1
c_purge_mayvote_salts: UPDATE mayvote SET salt = NULL WHERE iid IN (SELECT iid FROM issue WHERE eid = ?)
```

### MEDIUM Severity

#### ASVS-1427-MED-001: Superseded Votes Retained Indefinitely as Unnecessary Data

**Severity:** MEDIUM

**Location:**
- **File:** `v3/schema.sql` (vote table definition), `v3/steve/election.py`
- **Functions:** `add_vote()`, `tally_issue()`
- **Lines:** 204-215 (`add_vote`), 217-255 (`tally_issue`)

**Related Functions Checked:**
- `election.py:add_vote()` (lines 204-215)
- `election.py:tally_issue()` (lines 217-255)
- Schema query: `q_recent_vote`

**Description:**

When a voter re-votes on an issue, the system creates a new `vote` row with the same `vote_token` but a new auto-incrementing `vid`. Only the most recent vote is used during tallying (`q_recent_vote`). The superseded votes serve no purpose but remain in the database indefinitely.

**Vulnerable Code:**

```python
# v3/steve/election.py — add_vote()
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
    # No cleanup of previous votes with same vote_token
```

```sql
-- v3/schema.sql
/* ... duplicates of (person, issue) may occur (the vote_token will be the
   same), as re-voting is allowed. Only the latest is used.  */
CREATE TABLE vote (
    vid  INTEGER PRIMARY KEY AUTOINCREMENT,
    vote_token  BLOB NOT NULL  CHECK (length(vote_token) = 32),
    ciphertext  BLOB NOT NULL
) STRICT;
```

**Data Flow:**

Voter re-votes → `add_vote()` inserts new row → old row with same `vote_token` remains → `tally_issue()` reads only `q_recent_vote` → old rows are orphaned unnecessary data.

**Proof of Concept:**

A voter who changes their vote 5 times will have 5 encrypted vote rows in the database. An attacker with database access can observe that a specific `vote_token` voted 5 times and potentially correlate timing of row insertions (via `vid` ordering) with other events.

**Impact:**

Unnecessary accumulation of encrypted vote data. For a system whose core goal is ballot secrecy, retaining the history of vote changes for each `vote_token` provides an unnecessary information channel — particularly the count of re-votes per token and their ordering.

**Remediation:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    
    # Delete any previous vote(s) for this token before inserting new one
    self.db.conn.execute('BEGIN TRANSACTION')
    self.c_delete_prior_votes.perform(vote_token)
    self.c_add_vote.perform(vote_token, ciphertext)
    self.db.conn.execute('COMMIT')
```

Add query to `queries.yaml`:
```yaml
c_delete_prior_votes: DELETE FROM vote WHERE vote_token = ?
```

#### ASVS-1427-MED-002: Person PII (Name, Email) Has No Practical Deletion Path

**Severity:** MEDIUM

**Location:**
- **File:** `v3/steve/persondb.py`, `v3/schema.sql`
- **Functions:** `delete_person()`
- **Lines:** 51-64 (`delete_person`)

**Related Functions Checked:**
- `persondb.py:delete_person()` (lines 51-64)
- `persondb.py:add_person()` (lines ~30-40)
- Schema foreign key constraints: `mayvote` → `person`

**Description:**

The `person` table stores PII (name, email) for all voters ever registered. While a `delete_person()` method exists, referential integrity constraints from the `mayvote` table prevent deletion of any person who has been associated with any election. The code comment explicitly acknowledges this limitation with no resolution.

**Vulnerable Code:**

```python
# v3/steve/persondb.py — delete_person()
def delete_person(self, pid):
    "Delete the Person designated by PID."
    # NOTE: if this person has ever been involved in a vote
    #   (ie. the PID exists in a "mayvote" row), then this will
    #   throw a referential integrity error.
    #
    ### maybe we just don't delete a person, ever?
    self.c_delete_person.perform(pid)
```

```sql
-- v3/schema.sql — mayvote table
FOREIGN KEY (pid) REFERENCES person(pid)
ON DELETE RESTRICT
ON UPDATE NO ACTION
```

**Data Flow:**

Person registered → added to mayvote for election → election closes → person PII (name, email) remains forever → no path to remove even after all associated elections are long past.

**Proof of Concept:**

N/A (design issue)

**Impact:**

Voter PII accumulates without any lifecycle management. For a voting system that may serve many elections over years, this creates an ever-growing store of personal data with no ability to honor data subject deletion requests or comply with data minimization principles.

**Remediation:**

```python
def anonymize_person(self, pid):
    """Anonymize a person's PII while preserving referential integrity.
    
    This replaces name and email with anonymized values while
    keeping the PID reference intact for mayvote/vote integrity.
    """
    self.c_anonymize_person.perform(
        f'[redacted-{pid[:4]}]',  # anonymized name
        f'redacted@invalid',       # anonymized email  
        pid
    )
```

Add query:
```yaml
c_anonymize_person: UPDATE person SET name = ?, email = ? WHERE pid = ?
```

## Positive Security Patterns

1. **Vote encryption at rest**: All votes stored as ciphertext with per-voter key derivation.
2. **Metadata exclusion from public API**: `get_metadata()` explicitly excludes `salt` and `opened_key` (election.py line ~132).
3. **PersonDB excludes salt**: `get_person()` has comment "NEVER return person.salt" (persondb.py line ~37).
4. **State machine enforcement**: `assert` guards prevent operations in wrong election states.
5. **Vote shuffling before tally**: `crypto.shuffle(votes)` prevents order-based correlation (election.py line ~254).
6. **Transaction-based critical operations**: `add_salts()` and `delete()` use explicit transactions.
7. **Tamper detection**: `is_tampered()` recomputes `opened_key` to detect unauthorized modifications.

---

## Architecture Observations

1. **No background job infrastructure**: The application is a pure request-response web server with no cron, Celery, or similar scheduled task system. This means automatic data retention enforcement is architecturally impossible without adding new infrastructure.

2. **Closed election immutability**: The `prevent_open_close_update` trigger and state assertions enforce forward-only state transitions, which is good for integrity but currently blocks any post-close cleanup.

3. **Single SQLite database**: All data categories (crypto keys, votes, PII) are in one database file, making selective backup/purge more challenging than a multi-store architecture.
