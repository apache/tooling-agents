## Findings

### Critical Findings

#### ASVS-1431-CRITICAL-001: Complete Absence of Authenticated Data Clearing from Client Storage

**Severity:** CRITICAL  
**Gap Type:** Type A — No control exists for clearing authenticated data from client storage upon session termination  
**ASVS Requirement:** 14.3.1 (Level 1)  
**CWE:** CWE-524 (Use of Cache Containing Sensitive Information)

##### Location

- **Primary Files:** `v3/server/pages.py`
- **Affected Functions:** All 12+ authenticated route handlers
- **Lines:** 85–95 (session data injection), 148+ (voter_page), 186+ (vote_on_page), 528+ (profile_page)

##### Related Functions Checked

- `basic_info()` — Sources authenticated session data to all templates
- `voter_page()` — Exposes election participation list
- `vote_on_page()` — Exposes voting interface with voter identity
- `profile_page()` — Exposes user profile data
- `admin_page()` — Exposes owned elections
- `manage_page()` — Exposes election management interface
- All POST handlers (`do_vote`, `do_create_election`, etc.) — Flash confirmations cached

##### Description

The application completely lacks mechanisms to clear authenticated data from client storage after session termination. Specifically:

1. **No `Clear-Site-Data` HTTP header** is sent on any response
2. **No logout endpoint exists** to trigger session termination and cleanup
3. **No `Cache-Control` headers** prevent browser caching of authenticated pages
4. **No client-side JavaScript** clears DOM/storage when session ends

All 12+ authenticated routes inject voter identity (uid, name, email) and election data into HTML responses via the `basic_info()` function. Without cache-control headers, browsers cache these pages containing sensitive voter information.

##### Proof of Concept

**Scenario: Shared Workstation Voter Privacy Breach**

1. Authenticated voter accesses `/vote-on/a1b2c3d4e5` on a shared workstation
2. Browser caches the response containing:
   - Voter's full name, uid, email address
   - Election issues and candidate list
   - Vote status information
3. Session expires or user closes the browser tab (no logout endpoint available to trigger cleanup)
4. Next user on the same workstation presses Back button or inspects browser cache/history
5. Cached HTML reveals:
   - Voter identity (uid, full name, email)
   - Which elections they were eligible for
   - That they accessed the voting page (confirming participation)
   - Flash messages confirming vote submission

**No mechanism exists to prevent or mitigate this scenario:**
- No `Clear-Site-Data: "cache", "cookies", "storage"` header is sent on any response
- No `Cache-Control: no-store` header prevents browser caching of authenticated pages
- No logout endpoint exists to trigger session termination and cleanup
- No client-side JavaScript clears DOM/storage when the session ends

##### Impact

In the context of a **voting system** handling **voter identities**:

1. **Voter privacy violation**: Browser cache on shared computers exposes who voted and in which elections, violating ballot secrecy principles
2. **Session persistence**: Without logout, session cookies remain active until server-side expiration, enabling session reuse by subsequent users
3. **Cached vote confirmation**: Flash messages confirming vote submission persist in browser cache, proving a voter's participation
4. **Election administration exposure**: Management pages cached with election configuration details accessible through browser history
5. **Compliance risk**: Failure to implement Level 1 ASVS requirement for clearing authenticated data

**Specific risks in voting context:**
- Voter coercion: Cached pages can be used to verify how someone voted or whether they participated
- Privacy breach: Personal information (uid, name, email) exposed through browser cache
- Audit trail contamination: Multiple users on shared workstation create ambiguous access patterns

##### Remediation

**1. Add logout endpoint with `Clear-Site-Data` header:**

```python
@APP.get('/logout')
async def logout():
    # Invalidate server-side session
    await asfquart.session.destroy()
    
    response = quart.redirect('/', code=303)
    # Instruct browser to clear all site data
    response.headers['Clear-Site-Data'] = '"cache", "cookies", "storage"'
    return response
```

**2. Add `Cache-Control` and security headers to all authenticated responses via `after_request`:**

```python
@APP.after_request
async def set_security_headers(response):
    # For authenticated pages, prevent caching of sensitive data
    if quart.request.endpoint and getattr(
        quart.request.endpoint, '_requires_auth', False
    ):
        response.headers['Cache-Control'] = 'no-store, no-cache, must-revalidate, max-age=0'
        response.headers['Pragma'] = 'no-cache'
        response.headers['Expires'] = '0'
    return response
```

**Simpler alternative — apply globally since most routes are authenticated:**

```python
@APP.after_request
async def set_cache_headers(response):
    # Voting system: default to no caching; whitelist static assets
    if not quart.request.path.startswith(('/static/', '/favicon.ico')):
        response.headers['Cache-Control'] = 'no-store, no-cache, must-revalidate, max-age=0'
        response.headers['Pragma'] = 'no-cache'
        response.headers['Expires'] = '0'
    return response
```

**3. Add client-side cleanup as fallback (in base template `header.ezt` or `footer.ezt`):**

```javascript
// Detect session expiration client-side and clean up
window.addEventListener('beforeunload', function() {
    // Clear any sensitive data from sessionStorage
    sessionStorage.clear();
});

// Periodic session check — clear DOM if session is gone
setInterval(async function() {
    try {
        const resp = await fetch('/api/session-check', { credentials: 'same-origin' });
        if (resp.status === 401) {
            // Session expired — clear sensitive DOM content
            document.querySelectorAll('[data-sensitive]').forEach(el => el.textContent = '');
            sessionStorage.clear();
            localStorage.clear();
            window.location.href = '/';
        }
    } catch(e) {
        // Server unreachable — clear sensitive data as precaution
        document.querySelectorAll('[data-sensitive]').forEach(el => el.textContent = '');
        sessionStorage.clear();
    }
}, 60000);
```

**4. Mark sensitive DOM elements in templates for targeted cleanup:**

```html
<!-- In voter.ezt, vote-on.ezt, profile.ezt, etc. -->
<div data-sensitive class="voter-info">
    <span data-sensitive>[basic.name]</span>
    <span data-sensitive>[basic.email]</span>
</div>
```

## Architecture Observations

1. **No HTTP response middleware**: The application has no `after_request` handler or middleware layer for setting security headers. This means each response is sent with framework defaults, which do not include `Cache-Control: no-store` or `Clear-Site-Data`. This is a systemic architectural gap affecting all endpoints.

2. **Session management delegated to framework**: `asfquart.session` handles session reads, but no session destruction or logout mechanism is exposed in the application layer. The framework may provide this capability, but the application does not invoke it anywhere in the codebase.

3. **Template-rendered sensitive data**: The EZT template system renders voter identity (uid, name, email) and election participation data directly into HTML responses. Without cache-control headers, browsers are free to cache these pages per their default behavior (typically caching GET responses).

4. **Flash messages contain confirmation of sensitive actions**: Vote submission confirmation ("Votes submitted successfully!") stored in session and rendered in subsequent responses can reveal voter participation through browser history/cache. Flash messages persist across redirects, increasing exposure window.

5. **No client-side JavaScript framework**: The application appears to use server-side rendering exclusively with minimal client-side JavaScript. This means no existing client-side state management or cleanup mechanisms are present to build upon.

6. **Separation of concerns**: Authentication (`asfquart.auth`), session management (`asfquart.session`), and routing are cleanly separated, which would make adding security middleware straightforward.
