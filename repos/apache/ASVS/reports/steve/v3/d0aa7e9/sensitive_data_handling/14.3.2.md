### Summary

The application **completely lacks** anti-caching HTTP response headers (`Cache-Control: no-store`) across all endpoints that serve sensitive election, voter, and administrative data. This is a systemic **Type A gap** (entry point with NO control) — no cache-control mechanism is defined, configured, or applied anywhere in the codebase.

## Findings

### HIGH Severity

#### ASVS-1432-HIGH-001: No Anti-Caching Headers on Voting Interface Pages

**Severity:** HIGH  
**Gap Type:** Type A — No control exists  

**Location:**
- File: `v3/server/pages.py`
- Functions: `vote_on_page()` (line ~238), `voter_page()` (line ~151)
- Routes: `GET /vote-on/<eid>`, `GET /voter`

**Related Functions Checked:**
- `basic_info()` (pages.py, ~line 67)
- `@APP.use_template()` decorator
- `load_election` decorator

**Vulnerable Code:**

`pages.py`, `vote_on_page()` (~line 238):
```python
@APP.get('/vote-on/<eid>')
@asfquart.auth.require({R.committer})
@load_election
@APP.use_template(TEMPLATES / 'vote-on.ezt')
async def vote_on_page(election):
    result = await basic_info()
    # ... builds ballot data with issues, candidates, vote controls ...
    return result  # No Cache-Control headers set
```

`pages.py`, `voter_page()` (~line 151):
```python
@APP.get('/voter')
@asfquart.auth.require({R.committer})
@APP.use_template(TEMPLATES / 'voter.ezt')
async def voter_page():
    result = await basic_info()
    # ... includes open_elections, upcoming_elections ...
    return result  # No Cache-Control headers set
```

**Data Flow:**
```
User request → vote_on_page() → builds ballot with candidates, issues, CSRF token
  → @APP.use_template renders HTML → HTTP response with NO Cache-Control header
  → Browser caches full ballot page in disk/memory cache
```

**Proof of Concept:**
1. Authenticated voter navigates to `/vote-on/<eid>` on a shared computer
2. Voter submits ballot and signs out
3. Next user presses browser Back button or opens cache (`chrome://cache/`)
4. Full ballot page including election issues, candidate names, and form structure is visible
5. Flash messages confirming vote submission may also be cached, revealing voting activity

**Sensitive Data Exposed via Cache:**
- `vote-on.ezt` embeds STV candidate data in JavaScript: `const STV_CANDIDATES = { "[issues.iid]": { candidates: [...] } }`
- Issue titles and descriptions visible in cached HTML
- CSRF tokens cached (enables replay if session still valid)
- Voter eligibility confirmed by page accessibility
- Per the domain context: "voter-to-issue mappings" and ballot structure are sensitive

**Impact:**  
On shared or public computers, the browser cache retains the complete voting interface, including issue details, candidate lists, and evidence of which elections a user was eligible for. This directly undermines ballot secrecy — a stated core security goal. An attacker with physical access to the browser (or its cache directory) can determine that the voter participated in specific elections and see the ballot structure.

**Remediation:**

Option 1 — Global `after_request` hook (recommended):
```python
@APP.after_request
async def set_security_headers(response):
    # Anti-caching for all authenticated responses
    if await asfquart.session.read():
        response.headers['Cache-Control'] = 'no-store, no-cache, must-revalidate, max-age=0'
        response.headers['Pragma'] = 'no-cache'
        response.headers['Expires'] = '0'
    return response
```

Option 2 — Per-route via Quart `make_response`:
```python
@APP.get('/vote-on/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def vote_on_page(election):
    result = await basic_info()
    # ... build result ...
    content = asfquart.utils.render(APP.load_template(TEMPLATES / 'vote-on.ezt'), result)
    response = await quart.make_response(content)
    response.headers['Cache-Control'] = 'no-store'
    response.headers['Pragma'] = 'no-cache'
    return response
```

#### ASVS-1432-HIGH-002: No Anti-Caching Headers on Administration and Election Management Pages

**Severity:** HIGH  
**Gap Type:** Type A — No control exists

**Location:**
- File: `v3/server/pages.py`
- Functions: `admin_page()` (line ~302), `manage_page()` (line ~328), `manage_stv_page()` (line ~348)
- Routes: `GET /admin`, `GET /manage/<eid>`, `GET /manage-stv/<eid>/<iid>`

**Related Functions Checked:**
- `basic_info()` (pages.py, ~line 67)
- `@APP.use_template()` decorator
- `load_election` decorator

**Vulnerable Code:**

`pages.py`, `admin_page()` (~line 302):
```python
@APP.get('/admin')
@asfquart.auth.require({R.committer})
@APP.use_template(TEMPLATES / 'admin.ezt')
async def admin_page():
    result = await basic_info()
    # ... includes owned elections with owner_name, owner_pid, eid, title ...
    return result  # No Cache-Control headers set
```

`pages.py`, `manage_page()` (~line 328):
```python
@APP.get('/manage/<eid>')
@asfquart.auth.require({R.committer})
@load_election
@APP.use_template(TEMPLATES / 'manage.ezt')
async def manage_page(election):
    result = await basic_info()
    # ... election metadata, issue list, state machine, CSRF tokens ...
    return result  # No Cache-Control headers set
```

**Sensitive Data Exposed via Cache:**
- Election IDs (10 hex chars, meant to be non-guessable)
- Election titles, states, and owner identities
- Issue titles and descriptions (election content before voting opens)
- `manage.ezt` embeds CSRF token: `<input type="hidden" id="csrf-token" value="[csrf_token]">`
- `admin.ezt` shows: `eid: [owned.eid]`, `Created by [owned.owner_name] ([owned.owner_pid])`

**Data Flow:**
```
Admin user → admin_page()/manage_page() → builds admin interface with election metadata
  → @APP.use_template renders HTML → NO Cache-Control header
  → Browser caches administrative data to disk
```

**Proof of Concept:**
1. Election administrator accesses `/admin` or `/manage/<eid>` on a shared workstation
2. Administrator logs out
3. Subsequent user accesses browser cache or history
4. Election IDs, owner identities, and pre-election issue content are exposed

**Impact:**  
Cached management pages reveal election infrastructure details: election IDs enable direct URL construction to election endpoints, issue details before elections open could bias voters, and owner identity exposure has privacy implications. CSRF tokens in cache could be exploited if sessions remain active.

**Remediation:** 
Same global `after_request` hook as ASVS-1432-HIGH-001.

## Architecture Observations

1. **No Response Header Framework**: The application has no centralized mechanism for setting security response headers. There is no `after_request` hook, no response wrapper class, and no middleware layer. This means Cache-Control, X-Content-Type-Options, X-Frame-Options, and other security headers are all absent.

2. **Template Decorator Pattern**: The `@APP.use_template()` decorator abstracts response creation, making it non-obvious where to inject response headers. This architectural choice makes per-route header setting awkward and favors a global approach.

3. **Mixed Response Types**: The application returns template-rendered HTML, `quart.redirect()`, `quart.send_from_directory()`, and raw tuples (`('', 204)`). A comprehensive solution must handle all response types.

4. **Quart Framework**: Quart (async Flask) does not set `Cache-Control: no-store` by default. Without explicit configuration, all responses are cacheable per HTTP semantics.

## Recommendations Summary

### Immediate Actions (Priority 1)

**Implement Global Anti-Caching Hook** — Addresses all findings

Add to `v3/server/pages.py`:

```python
@APP.after_request
async def set_security_headers(response):
    """Set anti-caching and security headers on all authenticated responses."""
    session = await asfquart.session.read()
    if session:
        # Prevent caching of any authenticated content
        response.headers['Cache-Control'] = 'no-store, no-cache, must-revalidate, max-age=0'
        response.headers['Pragma'] = 'no-cache'
        response.headers['Expires'] = '0'
    
    # Additional security headers for all responses
    response.headers['X-Content-Type-Options'] = 'nosniff'
    response.headers['X-Frame-Options'] = 'DENY'
    
    return response
```

This single change addresses both HIGH findings and ensures future routes are automatically protected.

### Short-term Actions (Priority 2)

**Add HTML Meta Fallback in `header.ezt`**

As defense-in-depth for cases where HTTP headers might be stripped by proxies, add to `v3/server/templates/header.ezt`:

```html
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <meta http-equiv="Cache-Control" content="no-store">
    <meta http-equiv="Pragma" content="no-cache">
    <meta http-equiv="Expires" content="0">
    <title>Apache STeVe: [title]</title>
    ...
</head>
```

### Long-term Actions (Priority 3)

**Verify Static Assets Are Exempt**

Ensure the global hook does not apply `no-store` to genuinely static resources (`/static/css/*`, `/static/js/*`, `/favicon.ico`). The session-check condition in Priority 1 handles this since static routes don't read sessions, but verify:

```python
@APP.get('/static/<path:filename>')
async def serve_static(filename):
    response = await quart.send_from_directory(STATICDIR, filename)
    response.headers['Cache-Control'] = 'public, max-age=86400'  # 24h for static assets
    return response
```

**Implement Security Header Testing**

Add automated tests to verify cache-control headers are present on all authenticated endpoints and absent on static resources.
