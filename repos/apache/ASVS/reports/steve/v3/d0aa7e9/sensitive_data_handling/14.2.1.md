## Findings

### Summary

**No findings identified.** All analyzed files comply with ASVS 14.2.1 requirements.

### Applicability Assessment

ASVS 14.2.1 targets the HTTP request/response cycle — specifically preventing sensitive data (API keys, session tokens, credentials) from appearing in URLs and query strings where they would be logged by proxies, browsers, and server access logs.

The audited codebase consists of:

1. **Command-line tools** (`create-election.py`, `mail-voters.py`, `tally.py`) that:
   - Accept input exclusively via **CLI arguments** (argparse) and **local file paths**
   - Make **no outbound HTTP requests** with sensitive data in URLs
   - Interact with **local SQLite databases** via direct file access
   - Send email via `asfpy.messaging.mail()` which uses message body, not URL parameters

2. **Library modules** (`stv.py`, `yna.py`) that perform pure computation with no network I/O

3. **Web application components** that properly handle sensitive data in request bodies and headers

No sensitive data (encryption keys, session tokens, voter credentials, decrypted vote contents) is passed through any URL or query string mechanism.

### CLI Argument Sensitivity Analysis

| Script | CLI Arguments | Contains Secrets? | Risk Assessment |
|--------|--------------|-------------------|-----------------|
| `create-election.py` | `yaml_file` (path) | No — file path only | ✅ Safe — No sensitive data exposure |
| `mail-voters.py` | `--eid` (election ID), `--template` (path) | No — identifiers and paths | ✅ Safe — Public identifiers only |
| `tally.py` | `--election-id`, `--issue-id`, `--db-path`, `--output`, `--spy-on-open-elections` | No — identifiers, paths, flags | ✅ Safe — No credentials or tokens |

**Analysis:** No encryption keys, database credentials, or authentication tokens are passed as CLI arguments (which would appear in `/proc/*/cmdline` and shell history). The sensitive election encryption key is derived internally within `steve.election`, not passed externally.

## Positive Security Patterns

### 1. No HTTP Surface Area
All administrative tools operate locally via filesystem and direct database access, eliminating the URL/query-string attack surface entirely. This architectural choice inherently prevents ASVS 14.2.1 violations.

**Location:** All CLI scripts (`create-election.py`, `mail-voters.py`, `tally.py`)

### 2. Sensitive Data Stays Internal
Decryption keys are derived inside `election.tally_issue()`, never exposed via arguments or URLs. This follows the principle of keeping secrets in memory only.

**Location:** `v3/steve/election.py` — key derivation functions

### 3. Email Body, Not URL
`mail-voters.py` sends voter PII (email, name) in the email message body via `asfpy.messaging.mail()`, not appended to any URL. This prevents credential leakage in mail server logs.

**Location:** `mail-voters.py:main()` — email sending implementation

### 4. YAML File for Election Configuration
`create-election.py` reads sensitive election setup data from a local YAML file rather than passing it as individual command-line arguments, reducing exposure in process listings.

**Location:** `create-election.py` — YAML file parsing

### 5. Tampering Detection Before Data Access
`tally.py` checks `election.is_tampered(pdb)` before decrypting and outputting any vote data, ensuring data integrity before processing sensitive information.

**Location:** `tally.py:tally_election()` — integrity verification

### 6. Proper Separation of Concerns
Administrative tools are separated from web endpoints, with the trust boundary correctly placed at filesystem/OS access control rather than HTTP authentication.

**Location:** Architecture-wide pattern

## Architecture Observations

### Strengths

1. **Clear Trust Boundaries:** The separation of CLI administrative tools from web application endpoints creates distinct trust domains with appropriate access controls for each.

2. **Defense in Depth:** Multiple layers prevent sensitive data exposure:
   - CLI tools use filesystem permissions
   - Web application uses session management in headers/cookies
   - Database encryption keys never leave process memory
   - Email transport abstracted through `asfpy.messaging`

3. **Minimal Attack Surface:** By avoiding HTTP transmission for administrative operations, the codebase eliminates entire classes of web-based attacks (URL tampering, parameter pollution, log-based credential harvesting).

4. **Appropriate Technology Choices:**
   - SQLite for local database access (no network credentials)
   - argparse for CLI input (no URL parsing)
   - YAML files for configuration (filesystem-protected)
   - SMTP library abstraction (no manual URL construction)

### Design Considerations

1. **Operational Security:** CLI tools rely on OS-level access controls. Ensure proper file permissions on:
   - Database files
   - YAML configuration files
   - Output files from `tally.py`

2. **Process Visibility:** While CLI arguments don't contain secrets, consider that process listings (`ps`, `/proc`) expose election/issue IDs. This is acceptable for non-secret identifiers but worth documenting.

3. **Email Transport Security:** The `asfpy.messaging` library handles email sending. Verify that this library properly secures SMTP credentials and uses TLS for transport.
