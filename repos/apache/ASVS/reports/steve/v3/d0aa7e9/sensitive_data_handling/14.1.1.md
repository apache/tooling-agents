**Assessment Summary:**

ASVS 14.1.1 requires that all sensitive data created and processed by the application has been identified and classified into protection levels. This voting system handles multiple categories of highly sensitive data — encrypted votes, cryptographic keying material, per-voter salts, voter PII, and election metadata — where the core security goal is ballot secrecy. While the codebase contains several positive ad-hoc protections (salt exclusion from public APIs, vote encryption), there is no systematic data classification framework, and multiple concrete gaps demonstrate the absence of formal protection levels.

### ASVS-1411-HIGH-001: Absence of Formal Sensitive Data Classification and Protection Levels

**Severity:** HIGH

**Location:**
- **File:** `v3/steve/election.py`
  - **Function:** `get_metadata()` — lines 146-157
  - **Function:** `get_issue()` — line 163 (comment)
- **File:** `v3/steve/persondb.py`
  - **Function:** `get_person()` — line 38 (comment)
- **File:** `v3/server/pages.py`
  - **Function:** `basic_info()` — line 57
  - **Function:** `postprocess_election()` — line 603
- **File:** `v3/schema.sql` — schema comments (various)

**Related Functions Checked:**
- `election.py`: `_all_metadata()`, `open()`, `add_salts()`, `add_vote()`, `tally_issue()`
- `persondb.py`: `list_persons()`
- `pages.py`: `owned_elections()` (line 416-425), `vote_on_page()`, `do_vote_endpoint()`

**Description:**

The system processes at least six distinct categories of sensitive data, each requiring different protection levels, but none are formally classified. Ad-hoc protections exist (salt exclusion in specific functions, vote encryption) but there is no systematic framework to ensure consistent handling across all code paths.

**Sensitive Data Categories Identified:**

| Data Category | Sensitivity Level | Current Ad-Hoc Protection | Formal Classification |
|---------------|-------------------|---------------------------|----------------------|
| Election cryptographic salt/opened_key | CRITICAL | `get_metadata()` excludes them | ❌ None |
| Per-voter salts (`mayvote.salt`) | CRITICAL | `get_issue()` comment: "NEVER return" | ❌ None |
| Vote content (plaintext votestring) | CRITICAL | Encrypted before storage via `add_vote()` | ❌ None |
| Vote tokens (`vote.vote_token`) | HIGH | Internal-only processing | ❌ None |
| Voter PII (`person.email`, `person.name`) | MEDIUM (privacy) | No explicit protection level | ❌ None |
| Election metadata (eid, authz, state) | LOW-MEDIUM | No explicit protection level | ❌ None |

**Vulnerable Code — Inconsistent Ad-Hoc Protections:**

In `election.py`, `get_metadata()` correctly excludes sensitive fields:
```python

```

However, `postprocess_election()` in `pages.py` accepts and processes election data without verifying sensitive fields are absent:
```python
# line 603
def postprocess_election(e):
    "Post-process attributes in an Election, as an EasyDict."
    # NOTE: side effects. This function manipulates the E argument.
    e.closed = ezt.boolean(e.closed == 1)
    # ... processes all fields in `e` without filtering
```

Similarly, `owned_elections()` relies on code comments rather than enforcement:
```python
# lines 416-425

    db = cls.open_database(db_fname)
```

**Data Flow:**

```
_all_metadata() [returns salt/opened_key]
    ↓
Various query results (q_open_to_me, q_owned)
    ↓
postprocess_election() [no sensitive field filter]
    ↓
Template context dictionary
    ↓

```

**Proof of Concept:**

If any upstream query (e.g., `q_open_to_me`, `q_owned`) were modified to include `salt` or `opened_key` columns:

1. The database would return these fields
2. `postprocess_election()` would process them without filtering
3. The template context would contain cryptographic material
4. Depending on template implementation, this could be rendered in HTML

**Impact:**

Without formal classification, there is no systematic way to verify that all sensitive data types are consistently protected across all code paths. The current protections are:

- **Convention-based** — Individual developers must remember to exclude sensitive fields
- **Comment-driven** — Security decisions documented in comments rather than enforced by code
- **Function-specific** — No defense-in-depth at architectural boundaries (e.g., template rendering layer)

**Consequences:**

1. New features or query modifications could inadvertently expose cryptographic material or voter data
2. The ballot secrecy guarantee cannot be verified as complete
3. Compliance with data protection regulations cannot be demonstrated
4. Code reviews cannot systematically verify protection level adherence
5. Refactoring may accidentally remove ad-hoc protections

**Remediation:**

**Step 1: Create formal data classification document**

```python
# data_classification.py

class DataClassification:
    """
    Formal data classification levels for the STeVe voting system.
    All data must be assigned a classification level with specific handling rules.
    """
    
    # CRITICAL: Never exposed outside secure processing context
    CRITICAL = 'critical'
    CRITICAL_FIELDS = {
        'salt',           # Election cryptographic salt
        'opened_key',     # Election decryption key
        'vote_token',     # Per-vote authentication token
        'ciphertext',     # Encrypted vote content
        'votestring',     # Plaintext vote (only in memory during encryption/decryption)
    }
    
    # SENSITIVE: Encrypted in transit and at rest, access-controlled
    SENSITIVE = 'sensitive'
    SENSITIVE_FIELDS = {
        'email',          # Voter PII
        'name',           # Voter PII (less sensitive than email)
    }
    
    # INTERNAL: Not public but not secret, authenticated access only
    INTERNAL = 'internal'
    INTERNAL_FIELDS = {
        'eid',            # Election ID (unpredictable but not secret once known)
        'iid',            # Issue ID
        'pid',            # Person ID
        'authz',          # Authorization level
        'state',          # Election state
        'owner_pid',      # Election owner
    }
    
    # PUBLIC: Freely visible
    PUBLIC = 'public'
    PUBLIC_FIELDS = {
        'title',          # Election title
        'description',    # Issue description
        'open_at',        # Election schedule
        'close_at',       # Election schedule
    }

    @classmethod
    def get_classification(cls, field_name):
        """Return the classification level for a given field name."""
        if field_name in cls.CRITICAL_FIELDS:
            return cls.CRITICAL
        elif field_name in cls.SENSITIVE_FIELDS:
            return cls.SENSITIVE
        elif field_name in cls.INTERNAL_FIELDS:
            return cls.INTERNAL
        elif field_name in cls.PUBLIC_FIELDS:
            return cls.PUBLIC
        else:
            # Default to INTERNAL for unclassified fields (fail-secure)
            return cls.INTERNAL
```

**Step 2: Add defense-in-depth filtering at template boundary**

```python
# pages.py

from data_classification import DataClassification

def sanitize_for_template(data):
    """
    Remove CRITICAL-classified fields before template rendering.
    Defense-in-depth: Even if upstream code accidentally includes sensitive fields,
    prevent them from reaching templates.
    """
    if isinstance(data, dict):
        forbidden = DataClassification.CRITICAL_FIELDS
        for field in forbidden:
            if field in data:
                _LOGGER.warning(
                    'SECURITY: Removed CRITICAL field [%s] at template boundary',
                    field
                )
                data.pop(field)
    elif isinstance(data, (list, tuple)):
        return [sanitize_for_template(item) for item in data]
    return data

# Apply to all template rendering
@APP.use_template(TEMPLATES / 'vote-on.ezt')

    result = await basic_info()
    # ... populate result ...
    return sanitize_for_template(result)  # Add sanitization
```

```python
# logging_config.py

class SensitiveFieldFilter(logging.Filter):
    """
    Filter that removes sensitive fields from log records.
    """
    SENSITIVE_KEYS = {
        'title', 'description', 'votestring', 'salt', 
        'opened_key', 'vote_token', 'email', 'csrf_token'
    }
    
    def filter(self, record):
        # If log record contains a dict, sanitize it
        if hasattr(record, 'form_data'):
            record.form_data = self._sanitize_dict(record.form_data)
        return True
    
    def _sanitize_dict(self, data):
        if not isinstance(data, dict):
            return data
        return {
            k: ('***REDACTED***' if k in self.SENSITIVE_KEYS else v)
            for k, v in data.items()
        }

# Apply filter to all loggers
logging.getLogger().addFilter(SensitiveFieldFilter())
```

```python
# data_classification.py (extend)
```

### ASVS-1411-MED-003: Missing Cache-Control Headers on Pages Processing Sensitive Election and Voter Data

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/pages.py`
  - **Function:** `vote_on_page()` — line ~226
  - **Function:** `voter_page()` — line ~137
  - **Function:** `admin_page()` — line ~285
  - **Function:** `manage_page()` — line ~322

**Related Functions Checked:**
- `basic_info()` — line 57
- `postprocess_election()` — line 603
- All `@APP.use_template()` decorated endpoints

**Description:**

Pages that display sensitive election data — voter eligibility, candidate lists, election structure, and voting interfaces — are served without any `Cache-Control` headers or equivalent meta tags. The domain context explicitly states that "cache-control headers prevent vote caching" as a required control.

**Evidence — `vote_on_page()` Serves Sensitive Data Without Cache Controls:**

**Evidence — `header.ezt` Contains No Cache Directives:**

```html
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Apache STeVe: [title]</title>
    <!-- No <meta http-equiv="Cache-Control" content="no-store"> -->
    <!-- No <meta http-equiv="Pragma" content="no-cache"> -->
    <!-- No <meta http-equiv="Expires" content="0"> -->
```

**Data Exposed on Cached Pages:**

| Page Endpoint | Sensitive Data Cached | Classification Level |
|---------------|----------------------|---------------------|
| `/vote-on/<eid>` | Election title, all candidate names (STV), issue descriptions, voter eligibility proof | INTERNAL + SENSITIVE |
| `/voter` | List of elections user is eligible for, election metadata, voting status | INTERNAL |
| `/admin` | Owned elections, administrative state information | INTERNAL |
| `/manage/<eid>` | Election configuration, issue details, state machine position | INTERNAL |
| `/profile` | User PII (name, email) | SENSITIVE |

**Data Flow — Caching Attack:**

```
User authenticates and visits /vote-on/abc1234567
    ↓
Server renders page with candidate list
    ↓
Browser caches response (no Cache-Control: no-store)
    ↓
User logs out or session expires
    ↓
Attacker with physical access presses "Back" button
    ↓
Browser serves cached page from disk/memory
    ↓
Attacker sees candidate list and election structure
```

**Proof of Concept:**

1. Authenticate as a voter and navigate to `/vote-on/abc1234567`
2. Observe response headers:
   ```http
   HTTP/1.1 200 OK
   Content-Type: text/html; charset=utf-8
   Content-Length: 4523
   
   <!-- No Cache-Control header -->
   ```
3. Log out or close browser
4. Press browser "Back" button
5. **Result:** Cached page is displayed with full election data, despite no active session

**Impact:**

Browser back-button, history, and proxy caches can reveal:

1. **Voter Participation:** Which elections a voter participated in (confirming voter eligibility is itself sensitive metadata in organizational contexts)
2. **Election Structure:** Full candidate lists and election structure before voting is complete
3. **Administrative Access:** Whether the user was viewing the page in an administrative or voting capacity
4. **PII:** User profile information cached on shared computers

**Attack Scenarios:**

| Scenario | Impact |
|----------|--------|
| Shared workstation | Next user presses "Back" and sees previous user's eligible elections |
| Proxy cache (misconfigured) | Corporate proxy caches authenticated pages, serves to wrong users |
| Browser forensics | Attacker with disk access extracts cached election data from browser cache |
| Mobile device theft | Thief accesses cached voting pages without authentication |

**Demonstration that Data is Unclassified:**

The absence of cache-control headers demonstrates that the data displayed on these pages has not been assigned a protection level that would mandate cache prevention. A formal classification would specify:

- **CRITICAL data:** Never cached (not applicable here, as crypto material is not displayed)
- **SENSITIVE data:** `Cache-Control: no-store, no-cache`
- **INTERNAL data:** `Cache-Control: no-store, no-cache` (same as SENSITIVE for authenticated pages)
- **PUBLIC data:** Normal caching allowed

**Remediation:**

**Immediate: Add after_request handler for cache headers**

```python
# pages.py

@APP.after_request
async def set_security_headers(response):
    """
    Add security headers to all responses.
    Cache-Control prevents sensitive data from being cached.
    """
    # Apply to all HTML responses (authenticated pages)
    if response.content_type and 'text/html' in response.content_type:
        response.headers['Cache-Control'] = 'no-store, no-cache, must-revalidate, max-age=0'
```

**Data Flow — Election ID Exposure:**

```
User clicks "Open Election" button
    ↓
JavaScript navigates to /do-open/<eid> via GET
    ↓
Election ID appears in:
  • Browser address bar
  • Browser history (persistent)
  • Server access logs (GET /do-open/abc1234567)
  • Proxy/CDN logs
  • Referer header (if redirect target loads external resources)
    ↓
Election ID persisted in multiple uncontrolled locations
```

**Proof of Concept:**

1. As an election owner, navigate to `/manage/abc1234567`
2. Click "Open Election" button
3. Observe browser address bar: `https://vote.example.org/do-open/abc1234567`
4. Check browser history (Ctrl+H): Entry shows `/do-open/abc1234567`
5. Check server access logs:
   ```
   192.168.1.100 - - [22/Apr/2026:14:23:45 +0000] "GET /do-open/abc1234567 HTTP/1.1" 303 0
   ```

**Impact:**

Election IDs are designed to be unpredictable (10-character hex, schema `CHECK(length(eid) = 10)`) to prevent enumeration attacks. Using GET for state-changing operations undermines this protection by causing election IDs to appear in:

1. **Browser History:**
   - Persisted on user's device indefinitely
   - Visible to anyone with device access (shared workstations, device theft)
   - Survives session logout
   - Synced to cloud in browsers with history sync enabled

2. **Server Access Logs:**
   - `GET /do-open/abc1234567` logged with full URL
   - Typically retained for extended periods (30-90 days or longer)
   - Accessible to operations teams who may not need election access
   - May be aggregated in SIEM systems with broader access

3. **Proxy/CDN Logs:**
   - If any intermediate proxy is present (corporate proxy, CDN, load balancer)
   - Logs may have different retention policies than application logs

4. **Referer Headers:**
   - If the redirect target (`/manage/<eid>`) loads external resources (CDN, analytics)
   - `Referer: https://vote.example.org/do-open/abc1234567` sent to third parties

**Compounding Factor — Irreversible Operations:**

These endpoints perform **irreversible state changes**:

- **`do_open()`:** Generates cryptographic material (`opened_key`), transitions election to open state
- **`do_close()`:** Transitions election to closed state, enables tallying

The combination of:
1. Sensitive operation (state change)
2. Unpredictable identifier (election ID)
3. GET method (causes logging)

...creates a security gap where the data exposure is compounded by the action's sensitivity.

**Demonstration that Election IDs are Unclassified:**

The use of GET for these operations demonstrates that election IDs have not been assigned a protection level with handling rules. A formal classification would specify:

- **Election IDs:** INTERNAL classification
- **Handling rule:** Must not appear in URLs for state-changing operations
- **Technical control:** POST-only for all state changes

## Positive Security Patterns

The following security controls demonstrate good practices and should be maintained:

### 1. Explicit Field Selection in `get_metadata()`

**Location:** `v3/steve/election.py` — lines 146-157

### 2. Vote Encryption Before Storage

**Location:** `v3/steve/election.py` — lines 239-253

**Pattern:**
```python
def add_vote(self, uid, iid, votestring):
    "Add a vote for the given user and issue."
    assert self.is_open(), 'Election is not open'
    
    # ... validation ...

```

**Why This is Good:**
- Plaintext vote strings never reach the database
- Encryption is mandatory (not optional)
- Function signature ensures caller cannot bypass encryption
- Even database administrators cannot read vote content

### 3. State Machine Enforcement

### 5. Per-Voter Salts

**Location:** `v3/steve/election.py` — lines 121-139, schema `mayvote.salt`

**Pattern:**
```python
def add_salts(self, pdb):
    "Add salt values for all persons in the authorization list."
    assert self.is_editable(), 'Election is not editable'
    
    db = self.db
    db.cursor.execute('BEGIN TRANSACTION')

        salt = crypto.create_salt()
        db.q_add_salt.perform(self.eid, pid, salt,)
    
    db.cursor.execute('COMMIT')
```

**Why This is Good:**
- Each voter gets unique cryptographic salt
- Salts are generated atomically within transaction
- Even with database access, vote tokens cannot be correlated across issues without per-voter material
- Strengthens ballot secrecy against database compromise

### 6. Transaction-Based Salt Generation

**Location:** `v3/steve/election.py` — lines 121-139

**Pattern:**
```python
db.cursor.execute('BEGIN TRANSACTION')

for pid, in db.q_pids.fetchall():
    salt = crypto.create_salt()
    db.q_add_salt.perform(self.eid, pid, salt,)

```

**Why This is Good:**
- Ensures atomicity — either all salts are generated or none
- Prevents partial salt assignment if operation is interrupted
- Maintains database consistency
- Follows ACID principles for security-critical operations

## Architecture Observations

### 1. Data Protection Boundary

**Current State:**
The system has an implicit two-tier data model:
- **Public metadata:** titles, dates, state, candidate lists
- **Cryptographic material:** salts, keys, tokens, ciphertext

**Observation:**
This boundary is enforced by convention (individual function implementations) rather than by architecture (separate data access layers or classification-aware middleware).

**Implications:**
- Refactoring may accidentally violate the boundary
- New developers must learn conventions through code reading
- No automated verification that boundary is maintained

**Recommendation:**
Implement architectural enforcement via:
- Separate data access layer with classification-aware queries
- Middleware that validates data classification before template rendering
- Type system annotations for classified data

### 2. Comment-Driven Security

**Current State:**
Multiple critical security decisions are documented only in code comments:
- `### check authz` (pages.py)
- `# NEVER return issue.salt` (election.py line 163)
- `# NOTE: do not return the SALT or OPENED_KEY` (election.py line 148)

**Observation:**
These comments indicate developer awareness of sensitive data, but comments:
- Are not enforced by the runtime
- Can become outdated without detection
- Rely on code review to maintain

**Implications:**
- Security depends on developer discipline
- Automated tools cannot verify comment-based requirements
- Comments may be removed during refactoring

**Recommendation:**
Convert comment-based security decisions to:
- Runtime assertions
- Type system constraints
- Automated tests that verify the security property

### 3. Future Risk in `tally_issue()`

**Current State:**
The function contains a commented-out line:
```python
#md = self._all_metadata()
# TBD: we need a param to 'spy' on Open elections
```

**Observation:**
This suggests planned functionality to view partial results during open elections. If implemented, this would fundamentally undermine ballot secrecy by allowing:
- Correlation of vote timing with vote content
- Influence on later voters based on partial results
- Coercion ("prove you voted for X by showing partial results")

**Implications:**
- The TODO indicates this feature is under consideration
- Without formal data classification, there is no documented policy forbidding this
- Future developer may implement this without understanding ballot secrecy implications

**Recommendation:**
- **Immediate:** Add comment explaining why this is forbidden: `# SECURITY: Viewing open election results violates ballot secrecy. DO NOT IMPLEMENT.`
- **Short-term:** Document in data classification policy that decrypted vote content is CRITICAL and must never be accessible while election is open
- **Long-term:** Add automated test that verifies `tally_issue()` raises exception if called on open election

### 4. PersonDB Email Exposure

**Current State:**
The `get_person()` API returns `(name, email)` as a tuple:
```python
def get_person(self, pid):
    "Return the Person with the given PID."
    # NOTE: NEVER return the salt value.
    for name, email in self.db.q_get_person.fetchall():
        return name, email
```

**Observation:**
Most callers only use `name`:
```python
name, _ = pdb.get_person(uid)  # Email discarded
```

**Implications:**
- Unnecessary PII exposure (email is more sensitive than name)
- Violates principle of least privilege
- A classification-aware API would separate these into different access levels

**Recommendation:**
```python
def get_person_name(self, pid):
    """Return only the name for the given PID (least privilege)."""
    for name, in self.db.q_get_person_name.fetchall():
        return name

def get_person_email(self, pid):
    """Return the email for the given PID (requires justification)."""
    for email, in self.db.q_get_person_email.fetchall():
        return email

def get_person_full(self, pid):
    """Return full person record including PII (requires justification)."""
    for name, email in self.db.q_get_person.fetchall():
        return name, email
```
