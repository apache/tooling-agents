### Critical Severity

#### ASVS-1424-CRI-001: Missing Owner Authorization on Election Management Endpoints

**Severity:** CRITICAL  
**Gap Type:** Type B (Control EXISTS but NOT CALLED)

**Location:**
- **File:** `v3/server/pages.py`
- **Affected Functions:**
  - `do_open_endpoint` (line 460)
  - `do_close_endpoint` (line 481)
  - `do_add_issue_endpoint` (line 499)
  - `do_edit_issue_endpoint` (line 523)
  - `do_delete_issue_endpoint` (line 549)
  - `manage_page` (line 333)
  - `_set_election_date` (line 69)

**Related Functions Checked:**
- `schema.sql` line 65: `owner_pid TEXT NOT NULL` — election owner definition
- `schema.sql` line 71: `authz TEXT` — authorized editing group definition
- `election.py` `get_metadata()` line 212: Returns `owner_pid` and `authz` in metadata

**Description:**

Authorization data for election ownership exists in the database schema (`owner_pid`, `authz`), but no endpoint validates the current user against these fields before allowing management operations. All management endpoints only verify generic `@asfquart.auth.require({R.committer})` authentication, allowing any authenticated committer to manage any election.

**Vulnerable Code:**

```python
# pages.py line 460-478
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    result = await basic_info()

    ### check authz          # <-- COMMENT ONLY, NO ACTUAL CHECK

    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)       # <-- Generates cryptographic keys, irreversible
    ...
```

**All Affected Entry Points:**

| Endpoint | Method | Impact |
|----------|--------|--------|
| `/do-open/<eid>` | GET | Any committer generates election keys, opens voting irreversibly |
| `/do-close/<eid>` | GET | Any committer permanently closes any election |
| `/manage/<eid>` | GET | Any committer views election management data |
| `/do-add-issue/<eid>` | POST | Any committer adds issues to any election |
| `/do-edit-issue/<eid>/<iid>` | POST | Any committer modifies any issue |
| `/do-delete-issue/<eid>/<iid>` | POST | Any committer deletes any issue |
| `/do-set-open_at/<eid>` | POST | Any committer changes advisory dates |
| `/do-set-close_at/<eid>` | POST | Any committer changes advisory dates |

**Data Flow:**

```
Authenticated committer
    ↓
pages.py route handler
    ↓
### check authz (no-op comment)
    ↓
election.open()/close()/add_issue()/etc.
    ↓
Sensitive operation executes without owner verification
```

**Proof of Concept:**

Any authenticated committer (not the election owner) navigates to:
```
GET /do-close/a1b2c3d4e5
```
This closes the election permanently, preventing all remaining voters from casting ballots.

**Impact:**

Complete compromise of election integrity. Any authenticated user can:
- Open elections prematurely (before all voters/issues are configured)
- Close elections to suppress votes
- Add/remove ballot issues
- Modify issue content

This directly undermines ballot secrecy and election integrity as the core security goal.

**Remediation:**

```python
async def check_election_owner(election, uid):
    """Verify the current user owns or is authorized for this election."""
    md = election.get_metadata()
    if md.owner_pid != uid:
        # Future: also check md.authz group membership
        quart.abort(403, 'Not authorized to manage this election')

@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    await check_election_owner(election, result.uid)
    # ... proceed with open
```

### High Severity

#### ASVS-1424-HIG-001: State-Changing Operations via GET Requests

**Severity:** HIGH

**Location:**
- **File:** `v3/server/pages.py`
- **Lines:** 460, 481
- **Functions:** `do_open_endpoint`, `do_close_endpoint`

**Related Functions Checked:**
- `manage.ezt` line ~301: Template JavaScript navigation

**Description:**

Critical state-changing operations (opening and closing elections) are implemented as GET requests rather than POST. This violates HTTP semantics and creates multiple security vulnerabilities related to caching, logging, and unintended request replay.

**Vulnerable Code:**

```python
# pages.py line 460
@APP.get('/do-open/<eid>')        # GET for irreversible state change
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    ...
    election.open(pdb)            # Generates salts, opened_key — irreversible
```

```python
# pages.py line 481
@APP.get('/do-close/<eid>')       # GET for irreversible state change
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    ...
    election.close()              # Permanently ends voting — irreversible
```

**Template Confirmation** (`manage.ezt` line ~301):
```javascript
window.location.href = '/do-open/[eid]';   // Navigates via GET
window.location.href = '/do-close/[eid]';
```

**Data Flow:**

```
Browser GET request
    ↓
URL stored in: browser history, proxy logs, server access logs, Referrer headers
    ↓
Reveals irreversible election state changes in cacheable/loggable form
```

**Proof of Concept:**

Browser prefetch or a `<link rel="prefetch" href="/do-open/a1b2c3d4e5">` injected via XSS on any authenticated page could open an election without user intent.

**Impact:**

1. **Caching:** Browsers and reverse proxies may cache GET responses; a retry/back-button could silently repeat the state change attempt
2. **Browser history/proxy logs:** URLs like `/do-open/a1b2c3d4e5` persist in access logs and history, leaking election IDs and administrative actions
3. **Link prefetching:** Browser link prefetching could trigger the operation unintentionally
4. **Referrer leakage:** If the redirect target includes external resources, the state-changing URL appears in Referrer headers

**Remediation:**

```python
@APP.post('/do-open/<eid>')       # Changed to POST
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    ...
```

```javascript
// In template, submit via form POST:
document.getElementById('openForm').submit();
```

#### ASVS-1424-HIG-002: Missing Cache-Control Headers on Sensitive Pages

**Severity:** HIGH

**Location:**
- **File:** `v3/server/pages.py`
- **Affected Functions:** All page handlers, specifically:
  - `vote_on_page` (line 223)
  - `voter_page` (line 119)
  - `manage_page` (line 333)
  - `admin_page` (line 286)

**Related Functions Checked:**
- Reviewed all response handlers for header setting
- Checked for after-request hooks or middleware

**Description:**

No `Cache-Control`, `Pragma`, or `Expires` headers are set on any response. Quart does not set restrictive cache headers by default for HTML responses. This violates the domain requirement: "cache-control headers prevent vote caching."

**Evidence of Absence:**

No `Cache-Control`, `Pragma`, or `Expires` headers are set anywhere in `pages.py`. No response middleware or after-request hooks set these headers.

**Data Flow:**

```
Sensitive page request
    ↓
Response without cache control headers
    ↓
Browser/proxy cache stores response
    ↓
Cached sensitive data persists on shared devices
```

**Impact:**

- `/vote-on/<eid>`: Voting page with issue lists, vote forms, and election metadata cached in browser
- `/voter`: Lists of elections the user can vote in, with EIDs, owner information
- `/manage/<eid>`: Election management data including state, configuration
- Shared computers or public terminals would retain cached pages showing election data and voting activity
- HTTP proxy caches could serve stale election data to other users

**Remediation:**

```python
@APP.after_request
async def set_security_headers(response):
    # Prevent caching of authenticated pages
    if 'text/html' in response.content_type:
        response.headers['Cache-Control'] = 'no-store, no-cache, must-revalidate, max-age=0'
        response.headers['Pragma'] = 'no-cache'
        response.headers['Expires'] = '0'
    return response
```

#### ASVS-1424-MED-001: Debug Print Statements Leak Form Data to Stdout

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/pages.py`
- **Lines:** 509, 533
- **Functions:** `do_add_issue_endpoint`, `do_edit_issue_endpoint`

**Related Functions Checked:**
- Form data handling in both functions
- Logging configuration

**Description:**

Debug `print()` statements dump all form data to stdout without redaction, causing user-submitted election issue data to be written to uncontrolled log destinations.

**Vulnerable Code:**

```python
# pages.py line 509 (do_add_issue_endpoint)
form = edict(await quart.request.form)
print('FORM:', form)                       # Dumps all form data to stdout

# pages.py line 533 (do_edit_issue_endpoint)
form = edict(await quart.request.form)
print('FORM:', form)                       # Dumps all form data to stdout
```

**Data Flow:**

```
User form submission (title, description, csrf_token)
    ↓
print() statement
    ↓
stdout
    ↓
Container logs / systemd journal / log aggregator
    ↓
Uncontrolled retention of user-submitted data
```

**Impact:**

Form data including election issue titles and descriptions is written to stdout without redaction. In containerized deployments, stdout feeds into centralized logging systems that may have different retention policies and access controls than the application database. This violates the principle of controlling sensitive data logging pathways.

**Remediation:**

```python
# Remove print statements; use structured logging if debug info is needed
_LOGGER.debug('Adding issue to election[E:%s]', election.eid)
```

#### ASVS-1424-MED-002: Missing Vote Content Validation Before Encryption

**Severity:** MEDIUM

**Location:**
- **File:** `v3/steve/election.py`
- **Line:** 221
- **Function:** `add_vote`

**Related Functions Checked:**
- `do_vote_endpoint` in `pages.py` (caller)
- `tally_issue` (vote processor)
- `vtypes` module usage

**Description:**

Arbitrary strings are accepted as vote content and encrypted without validation against the issue's vote type. Invalid votes cannot be detected until decryption during tallying, when correction is impossible.

**Vulnerable Code:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting    # <-- NOT IMPLEMENTED

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

**Data Flow:**

```
User form input (arbitrary string)
    ↓
do_vote_endpoint
    ↓
election.add_vote()
    ↓
### validate (no-op comment)
    ↓
crypto.create_vote()
    ↓
Invalid vote encrypted and stored
```

**Impact:**

Arbitrary strings are accepted, encrypted, and stored as votes. During tallying (`tally_issue`), the `vtypes` module processes decrypted votes. Invalid vote content (e.g., `"xyz"` for a YNA vote, or `"a,a,a,b"` with duplicates for STV) would either produce incorrect tallies or cause tally-time errors. Since votes are encrypted, invalid content cannot be detected until the offline tallying process, when correction is impossible (the election is closed). This undermines data integrity verification at the point of collection.

**Remediation:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)

    # Validate votestring against the issue's vote type
    issue = self.q_get_issue.first_row(iid)
    m = vtypes.vtype_module(issue.type)
    if not m.validate(votestring, self.json2kv(issue.kv)):
        raise ValueError(f'Invalid vote format for {issue.type} issue')

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    ...
```

#### ASVS-1424-MED-003: Voter-Issue Correlation Recorded in Application Logs

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/pages.py`
- **Lines:** 425-427
- **Function:** `do_vote_endpoint`

**Related Functions Checked:**
- All logging statements in voting flow
- `election.add_vote()` logging behavior

**Description:**

Per-issue logging creates a record of exactly which issues each voter voted on, creating an audit trail that could reveal voting participation patterns beyond what's necessary for operational purposes.

**Vulnerable Code:**

```python
# pages.py line 425-427
for iid, votestring in votes.items():
    ...
    election.add_vote(result.uid, iid, votestring)
    _LOGGER.info(
        f'User[U:{result.uid}] voted on issue[I:{iid}] in election[E:{election.eid}]'
    )
```

**Data Flow:**

```
Voter submits ballot
    ↓
Per-issue loop
    ↓
_LOGGER.info() records (voter_pid, issue_id)
    ↓
Log files contain voter-to-issue mapping
```

**Domain Requirement Analysis:**

"Check if logs avoid recording vote contents or voter-vote correlations."

Vote content (`votestring`) is correctly excluded from logs. However, the per-issue logging creates a record of exactly which issues each voter voted on. While voter eligibility (`mayvote` table) already records who *may* vote, these logs additionally reveal who *did* vote and on which specific issues.

**Impact:**

Log files maintained for operational purposes contain voter-to-issue correlations that persist beyond the election lifecycle. Access to logs (which may have broader access controls than the election database) could reveal voting participation patterns. In elections where some issues are more sensitive than others, or where abstention patterns are meaningful, this creates an audit trail that could be used to infer voting behavior.

**Remediation:**

```python
# Log at election level only, not per-issue
_LOGGER.info(
    f'User[U:{result.uid}] submitted ballot for election[E:{election.eid}]'
    f' ({len(votes)} issues)'
)
```

## Positive Security Patterns

1. **Sensitive field exclusion in `get_metadata()`** (`election.py` line 212-223): Explicitly constructs return dict without `salt` or `opened_key`, with code comment documenting the intent.

2. **Per-voter salt generation** (`election.py` `add_salts()` line 107-120): Wrapped in database transaction ensuring atomicity; each mayvote entry gets a unique 16-byte salt.

3. **Vote shuffling before tally** (`election.py` `tally_issue()` line 313): `crypto.shuffle(votes)` ensures vote ordering in results doesn't correlate with database insertion order.

4. **Tamper detection** (`election.py` `is_tampered()` line 331-342): Recomputes `opened_key` from current election data and compares against stored value, detecting post-open modifications.

5. **Vote content excluded from logs**: The `votestring` parameter is never passed to any logging call. Only election/issue identifiers appear in log messages.

6. **Document serving authorization** (`pages.py` `serve_doc()` line 596-611): Per-document authorization via `q_get_mayvote` check before serving, ensuring only eligible voters access issue documents.

7. **State assertions before operations** (`election.py`): All mutating operations (`add_issue`, `edit_issue`, `open`, `close`, `add_vote`) assert required election state, preventing out-of-order operations.

## Architecture Observations

1. **queries.yaml not available for audit**: Several security-critical queries (`q_owned`, `q_open_to_me`, `q_metadata`) are defined in `queries.yaml` which was not provided. The comment in `owned_elections` states columns exclude `salt`/`opened_key`, but this cannot be independently verified.

2. **Foreign key enforcement dependency**: The schema requires `PRAGMA foreign_keys = ON` for referential integrity, but this must be set by `asfpy.db.DB` at connection time. If not enabled, the `ON DELETE RESTRICT` constraints that prevent orphaned records would be silently ignored.

3. **Cryptographic module not provided**: `steve.crypto` implements all cryptographic operations (key derivation, encryption, vote tokens) but was not available for review. The known false positive notes confirm Fernet encryption and Argon2 hashing are used.

4. **EZT template encoding**: Templates use `[format "js,html"]` for JavaScript context encoding in `manage.ezt` and `manage-stv.ezt`, which is a positive pattern for preventing XSS in JavaScript string literals.
