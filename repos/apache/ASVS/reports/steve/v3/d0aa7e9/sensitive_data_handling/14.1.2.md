## Findings

### ASVS-1412-SEV-001

**Severity:** CRITICAL  
**Type:** Control EXISTS in comments but NOT CALLED  
**Title:** Missing Owner-Level Authorization on All Election Management Endpoints

**Location:**  
- **File:** `v3/server/pages.py`
- **Primary Function:** `load_election` decorator (line ~135)
- **Affected Endpoints:** 10 functions (see table below)

**Related Functions Checked:**
- `do_open_endpoint` (~line 460)
- `do_close_endpoint` (~line 477)
- `do_set_open_at_endpoint` (~line 492)
- `do_set_close_at_endpoint` (~line 507)
- `do_add_issue_endpoint` (~line 500)
- `do_edit_issue_endpoint` (~line 520)
- `do_delete_issue_endpoint` (~line 545)
- `do_create_endpoint` (~line 280)
- `election.open()` in `v3/steve/election.py` (~line 180)
- `election.close()` in `v3/steve/election.py` (~line 197)

**Description:**

Multiple endpoints contain `### check authz` comments indicating authorization was planned as a protection requirement but was never implemented. Any authenticated committer can perform election management operations on elections they do not own.

**Vulnerable Code:**

`load_election` decorator (`v3/server/pages.py`, ~line 135):
```python
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            ...
        ### check authz    # <-- NEVER IMPLEMENTED
        return await func(e)
    return loader
```

`do_open_endpoint` (`v3/server/pages.py`, ~line 460):
```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()

    ### check authz    # <-- NEVER IMPLEMENTED

    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)  # Proceeds without verifying ownership
```

**All Affected Endpoints:**

| Endpoint | Method | Function | Line | Sensitive Operation |
|----------|--------|----------|------|---------------------|
| `/do-open/<eid>` | GET | `do_open_endpoint` | ~460 | Opens election, generates cryptographic keys |
| `/do-close/<eid>` | GET | `do_close_endpoint` | ~477 | Permanently closes election |
| `/do-set-open_at/<eid>` | POST | `do_set_open_at_endpoint` | ~492 | Modifies election schedule |
| `/do-set-close_at/<eid>` | POST | `do_set_close_at_endpoint` | ~507 | Modifies election schedule |
| `/do-add-issue/<eid>` | POST | `do_add_issue_endpoint` | ~500 | Adds ballot items |
| `/do-edit-issue/<eid>/<iid>` | POST | `do_edit_issue_endpoint` | ~520 | Modifies ballot items |
| `/do-delete-issue/<eid>/<iid>` | POST | `do_delete_issue_endpoint` | ~545 | Removes ballot items |
| `/do-vote/<eid>` | POST | `do_vote_endpoint` | ~410 | Submits votes |
| `/do-create-election` | POST | `do_create_endpoint` | ~280 | Creates elections |
| `/manage/<eid>` | GET | `manage_page` | ~300 | Displays management interface |

**Verification in Model Layer** (`v3/steve/election.py`):

```python
def open(self, pdb):
    # Double-check the Election is in the editing state.
    assert self.is_editable()       # Only checks STATE, not OWNER
    self.add_salts()
    ...

def close(self):
    assert self.is_open()           # Only checks STATE, not OWNER
    self.c_close.perform(self.eid)
```

The `election` table schema includes an `owner_pid` column that is never checked:
```sql
CREATE TABLE election (
    eid  TEXT PRIMARY KEY CHECK (length(eid) = 10 AND eid GLOB '[0-9a-f]*'),
    owner_pid  TEXT NOT NULL,
    ...
) STRICT;
```

**Data Flow:**
1. **Source:** Any authenticated committer (HTTP request)
2. **Path:** Route handler → `load_election` decorator → `election.open()` / `election.close()` / etc.
3. **Missing Control:** No comparison of `result.uid` against `election.owner_pid` or `election.authz` group membership
4. **Sink:** State-changing database operations execute without authorization

**Proof of Concept:**

Any authenticated committer (not the election owner) can navigate to:
```http
GET /do-open/a1b2c3d4e5
```

This will:
1. Authenticate the committer successfully
2. Load the election without checking ownership
3. Execute `election.open(pdb)`, triggering:
   - Generation of per-voter salts via `crypto.gen_salt()`
   - Generation of `opened_key` via `crypto.gen_opened_key()`
   - Permanent state transition to "open"

The same pattern applies to:
- Closing elections prematurely
- Adding/removing/modifying ballot issues
- Changing election schedules
- Deleting voter eligibility records

**Impact:**

- **Election Disruption:** Any committer can prematurely close elections owned by others, preventing legitimate voters from participating
- **Ballot Tampering:** Any committer can add/remove/modify ballot issues in elections they don't own, compromising election integrity
- **Premature Key Generation:** Any committer can open elections before they're ready, triggering irreversible cryptographic key generation
- **Anti-Tamper Bypass:** The `opened_key` anti-tamper mechanism is bypassed because unauthorized changes are made BEFORE the election is opened (when the key is generated)
- **ASVS 14.1.2 Violation:** Directly violates the requirement for "access controls around sensitive data" — no documented or implemented owner-level authorization exists

**Remediation:**

Implement ownership verification in the `load_election` decorator:

```python
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            await flash_danger(f'Election {eid} not found.')
            return quart.redirect('/', code=303)

        # IMPLEMENT ownership/authz check
        result = await basic_info()
        md = e.get_metadata()
        
        # Check if user is the owner
        if md.owner_pid != result.uid:
            # Check if user is in authorized group (if specified)
            if not md.authz or not await check_group_membership(result.uid, md.authz):
                await flash_danger('You do not have permission to manage this election.')
                return quart.redirect('/', code=403)

        return await func(e)
    return loader
```

Implement the group membership check:
```python
async def check_group_membership(uid: str, authz_group: str) -> bool:
    """Check if user is member of authorization group."""
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    person = pdb.get_person(uid)
    return authz_group in person.groups  # Assumes groups field exists
```

### ASVS-1412-SEV-002

**Severity:** HIGH  
**Title:** Potential Sensitive Data Leakage Through Exception Logging During Vote Processing

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint`
- **Lines:** ~425-432

**Related Functions Checked:**
- `election.add_vote()` in `v3/steve/election.py` (~line 207)
- `crypto.create_vote()` (referenced, implementation not provided)
- `crypto.gen_vote_token()` (referenced, implementation not provided)

**Description:**

Exception messages during vote processing are logged without sanitization, potentially exposing plaintext vote content, cryptographic tokens, or per-voter salts in application logs.

**Vulnerable Code:**

`do_vote_endpoint` (`v3/server/pages.py`, lines ~425-432):
```python
try:
    election.add_vote(result.uid, iid, votestring)
    _LOGGER.info(
        f'User[U:{result.uid}] voted on issue[I:{iid}] in election[E:{election.eid}]'
    )
except Exception as e:
    _LOGGER.error(
        f'Error adding vote for user[U:{result.uid}] on issue[I:{iid}]: {e}'
    )
```

**Data Flow:**

1. **Source:** `election.add_vote()` processes sensitive data:
   - `votestring` (plaintext vote content)
   - `vote_token` (32-byte cryptographic token)
   - `mayvote.salt` (16-byte per-voter salt)

2. **Trace through `election.add_vote()`** (`v3/steve/election.py`, line ~207):
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

3. **Missing Control:** No sanitization of exception content before logging

4. **Sink:** `_LOGGER.error()` writes exception message `{e}` to application log files

**Potential Exception Scenarios:**

If `crypto.create_vote()` or `crypto.gen_vote_token()` raises an exception, the error message may include:
- Function arguments in the traceback (`votestring`, `vote_token`, `salt`)
- Representation of sensitive objects (e.g., `TypeError: expected bytes, got 'yes' for vote_token`)
- Stack traces revealing cryptographic implementation details

**Impact:**

- **Ballot Secrecy Violation:** Plaintext vote content could appear in application logs, violating fundamental ballot secrecy
- **Cryptographic Material Exposure:** Vote tokens and salts in logs enable vote de-anonymization by correlating logged tokens with database records
- **ASVS 14.1.2 Violation:** Directly violates the requirement for controlling "how the data is to be logged" and "access controls around sensitive data in logs"
- **Compliance Risk:** Exposure of voting choices may violate election law requirements for ballot secrecy

**Proof of Concept:**

Trigger an exception during vote processing:
1. Modify `crypto.create_vote()` to raise `ValueError(f"Invalid votestring: {votestring}")`
2. Submit a vote via `/do-vote/<eid>`
3. Observe application log contains: `Error adding vote for user[U:alice] on issue[I:issue123]: Invalid votestring: yes`

**Remediation:**

Remove exception details from logging and never include sensitive data:

```python
except Exception:
    _LOGGER.error(
        f'Error adding vote for user on issue[I:{iid}] '
        f'in election[E:{election.eid}]'
        # Never log exception details that may contain vote content
    )
    await flash_danger(f'Error submitting vote for issue {iid}.')
    return quart.redirect(f'/vote-on/{election.eid}', code=303)
```

For debugging, log only non-sensitive metadata:
```python
except Exception:
    _LOGGER.error(
        f'Vote submission failed: election[E:{election.eid}] issue[I:{iid}]',
        extra={'election_state': election.state}
    )
    # Detailed exceptions should only go to secure debug logs with restricted access
```

### ASVS-1412-SEV-003

**Severity:** HIGH  
**Title:** State-Changing Election Operations Use GET Method, Exposing Sensitive Operations in Access Logs

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `do_open_endpoint` (line ~459), `do_close_endpoint` (line ~477)
- **Template:** `v3/server/templates/manage.ezt`

**Related Functions Checked:**
- `election.open()` in `v3/steve/election.py` (~line 180)
- `election.close()` in `v3/steve/election.py` (~line 197)

**Description:**

Critical state-changing operations (opening/closing elections) use HTTP GET method, causing election IDs and management operations to be logged in web server access logs, browser history, and potentially leaked via Referrer headers.

**Vulnerable Code:**

Route handlers (`v3/server/pages.py`):
```python
@APP.get('/do-open/<eid>')          # GET for state change
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)              # Generates cryptographic keys
    await flash_success(f'Election {election.eid} has been opened.')
    return quart.redirect(f'/manage/{election.eid}', code=303)

@APP.get('/do-close/<eid>')         # GET for state change
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz
    election.close()                # Permanently closes election
    await flash_success(f'Election {election.eid} has been closed.')
    return quart.redirect(f'/manage/{election.eid}', code=303)
```

Triggered from template (`manage.ezt`):
```javascript
confirmOpen.addEventListener('click', () => {
    openModal.hide();
    window.location.href = '/do-open/[eid]';   // GET request with EID in URL
});
```

**Data Flow:**

1. **Source:** Election ID (EID) embedded in GET URL path
2. **Path:** 
   - User clicks "Open Election" button
   - JavaScript navigates to `/do-open/a1b2c3d4e5` via `window.location.href`
   - Web server logs: `GET /do-open/a1b2c3d4e5 200`
3. **Sinks:**
   - HTTP server access logs (Apache, nginx, etc.)
   - Browser history
   - Proxy server logs
   - Referrer headers to subsequent requests
4. **Missing Control:** POST method with request body

**Impact:**

- **Access Log Exposure:** Election IDs and management operations appear in web server access logs (e.g., `192.168.1.100 - - [22/Apr/2026:10:15:30] "GET /do-open/a1b2c3d4e5 HTTP/1.1" 200`)
- **Browser History:** Operations recorded in browser history, making them bookmarkable and replayable
- **Referrer Leakage:** If user navigates to external site after operation, Referrer header may leak the URL
- **Prefetch Risk:** Browser prefetch/prerender mechanisms could inadvertently trigger state changes
- **ASVS 14.1.2 Violation:** Directly violates requirement for documented "how the data is to be logged" — sensitive election management operations are logged in HTTP access logs by default without documented protection requirements

**Proof of Concept:**

1. As authenticated committer, navigate to `/manage/a1b2c3d4e5`
2. Click "Open Election"
3. Check web server access log:
```
192.168.1.100 - alice [22/Apr/2026:10:15:30 +0000] "GET /do-open/a1b2c3d4e5 HTTP/1.1" 303 - "https://steve.apache.org/manage/a1b2c3d4e5" "Mozilla/5.0..."
```
4. Check browser history — URL is saved and can be replayed

**Remediation:**

Change to POST method with CSRF protection:

```python
@APP.post('/do-open/<eid>')  # Change to POST
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    
    # Validate CSRF token
    form = await quart.request.form
    if not validate_csrf(form.get('csrf_token')):
        await flash_danger('Invalid security token.')
        return quart.redirect(f'/manage/{election.eid}', code=303)
    
    ### check authz
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    await flash_success(f'Election {election.eid} has been opened.')
    return quart.redirect(f'/manage/{election.eid}', code=303)
```

Update template (`manage.ezt`):
```javascript
confirmOpen.addEventListener('click', async () => {
    openModal.hide();
    
    // Create and submit POST form
    const form = document.createElement('form');
    form.method = 'POST';
    form.action = '/do-open/[eid]';
    
    const csrf = document.createElement('input');
    csrf.type = 'hidden';
    csrf.name = 'csrf_token';
    csrf.value = '[csrf_token]';  // From template context
    form.appendChild(csrf);
    
    document.body.appendChild(form);
    form.submit();
});
```

Apply same pattern to `/do-close/<eid>` and all other state-changing GET endpoints.

### ASVS-1412-SEV-004

**Severity:** MEDIUM  
**Title:** Debug Print Statements Write Unfiltered Form Data to Process Output

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `do_add_issue_endpoint` (line ~501), `do_edit_issue_endpoint` (line ~521)

**Related Functions Checked:**
- Form processing in both functions
- Template rendering in `manage.ezt`

**Description:**

Debug `print()` statements output unfiltered form data to stdout, which may be captured by container logs, systemd journal, or process monitoring systems without appropriate access controls.

**Vulnerable Code:**

`do_add_issue_endpoint` (`v3/server/pages.py`, ~line 501):
```python

    result = await basic_info()
    ### check authz

    form = edict(await quart.request.form)
    print('FORM:', form)           # <-- Uncontrolled output to stdout
    
    title = form.get('title', '').strip()
    description = form.get('description', '').strip()
    ...
```

`do_edit_issue_endpoint` (`v3/server/pages.py`, ~line 521):
```python

@load_election_issue

    print('FORM:', form)           # <-- Uncontrolled output to stdout
    
    title = form.get('title', '').strip()
    description = form.get('description', '').strip()
    ...
```

**Data Flow:**

1. **Source:** HTTP POST form data containing:
   - `title` (issue title)
   - `description` (issue description)
   - `csrf_token` (when properly implemented)
   - `type` (voting method)

2. **Path:** 
   - User submits form → `await quart.request.form`
   - Converted to dict → `edict()`
   - Printed to stdout → `print('FORM:', form)`

3. **Sinks:**
   - Process stdout (captured by systemd, Docker, Kubernetes)
   - Container logs (potentially aggregated to centralized logging)
   - Terminal output (in development)
   - Log files (if stdout redirected)

4. **Missing Control:** Structured logging with appropriate level and data filtering

**Impact:**

- **Issue Content Exposure:** Ballot item titles and descriptions written to uncontrolled output destinations

### ASVS-1412-SEV-005

**Severity:** MEDIUM  
**Title:** Voter-Issue Timing Correlation Recorded in Application Logs

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint`
- **Lines:** ~427-429

**Related Functions Checked:**
- `election.add_vote()` in `v3/steve/election.py` (~line 207)
- Vote table schema in `v3/schema.sql`

**Description:**

Per-issue vote logging creates a timing side channel that, combined with the `vote` table's auto-incrementing `vid` column, enables voter-vote correlation through timing analysis.

**Vulnerable Code:**

`do_vote_endpoint` (`v3/server/pages.py`, lines ~427-429):
```python
for iid, votestring in votes.items():
    try:
        election.add_vote(result.uid, iid, votestring)
        _LOGGER.info(
            f'User[U:{result.uid}] voted on issue[I:{iid}] in election[E:{election.eid}]'
        )
    except Exception as e:
        ...
```

**Data Flow:**

1. **Source:** Authenticated user identity (`result.uid`) and issue ID (`iid`)
2. **Path:** Each vote submission triggers separate log entry with timestamp
3. **Sink:** Application log file via `_LOGGER.info()`
4. **Missing Control:** Aggregated or anonymized logging that prevents per-issue correlation

**Schema Analysis:**

Vote table (`v3/schema.sql`):
```sql
CREATE TABLE vote (
    vid  INTEGER PRIMARY KEY AUTOINCREMENT,  -- Insert-order record
    vote_token  BLOB NOT NULL,
    ciphertext  BLOB NOT NULL
) STRICT;
```

The `vid` column creates a temporal ordering of vote insertions. Combined with per-issue logging:

**Attack Scenario:**

Attacker with access to both application logs and database:

1. **Log Analysis:**
```
2026-04-22 10:15:30 - INFO - User[U:alice] voted on issue[I:issue1] in election[E:elec1]
2026-04-22 10:15:31 - INFO - User[U:alice] voted on issue[I:issue2] in election[E:elec1]
2026-04-22 10:15:45 - INFO - User[U:bob] voted on issue[I:issue1] in election[E:elec1]
```

2. **Database Query:**
```sql
SELECT vid, vote_token FROM vote 
WHERE vote_token IN (
    SELECT vote_token FROM vote 
    WHERE vid BETWEEN 1000 AND 1010  -- Time window for alice's votes
);
```

3. **Correlation:** Narrow down which `vote_token` values belong to alice by:
   - Identifying insertion time window from logs
   - Mapping `vid` ranges to log timestamps
   - Reducing candidate set for cryptographic brute-force

**Impact:**

- **Side Channel Creation:** Timing correlation enables voter-vote linkage despite cryptographic separation
- **Undermines Cryptographic Protection:** The system's cryptographic design (per-voter salts, vote tokens) is intended to prevent voter-vote correlation, but logging creates a bypass
- **ASVS 14.1.2 Violation:** Domain context explicitly requires checking if "logs avoid recording vote contents or **voter-vote correlations**"
- **Ballot Secrecy Risk:** Combined with database access, enables partial de-anonymization of votes

**Proof of Concept:**

1. Enable application logging with timestamps
2. Submit votes as User A at time T1
3. Submit votes as User B at time T2
4. Query database:
```sql
SELECT vid, created_at FROM vote ORDER BY vid;
```
5. Correlate `vid` ranges with log timestamps to identify which votes belong to which user

**Remediation:**

Log only aggregated ballot submission, not per-issue:

```python
# Before loop
vote_count = len(votes)

for iid, votestring in votes.items():
    try:
        election.add_vote(result.uid, iid, votestring)
        # REMOVE per-issue logging
    except Exception:
        _LOGGER.error(
            f'Vote submission failed for election[E:{election.eid}]'
        )
        await flash_danger(f'Error submitting vote.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

# After loop - single aggregated log entry
_LOGGER.info(
    f'User[U:{result.uid}] submitted ballot for election[E:{election.eid}] '
    f'({vote_count} issue(s))'
)
```

This prevents timing correlation while maintaining audit capability.

## Positive Security Patterns

The codebase demonstrates several strong security practices worth highlighting:

### 1. Cryptographic Vote Separation

**Location:** `v3/steve/election.py`, `add_vote()` method (~line 207)

Votes are encrypted using per-voter salts with derived keys, ensuring that even with database access, vote content cannot be correlated without the `opened_key`:

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

Schema enforces salt length (`v3/schema.sql`):
```sql
CHECK (salt IS NULL OR length(salt) = 16)
```

**Security Value:** Provides cryptographic separation between voter identity and vote content, protecting ballot secrecy even if database is compromised.

### 2. Anti-Tamper Mechanism

**Location:** `v3/steve/election.py`, `gather_election_data()` (~line 95) and `is_tampered()` (~line 222)

Creates deterministic hash of election metadata, issues, and voters. The `opened_key` derived from this data detects post-opening modifications:

```python
def gather_election_data(self):
    """Gather election data for hashing to detect tampering."""
    md = self.get_metadata()
    issues = self.get_issues()
    voters = self.get_voters()
    
    # Create deterministic representation
    data = {
        'eid': md.eid,
        'title': md.title,
        'issues': [{'iid': i.iid, 'title': i.title} for i in issues],
        'voters': [v.pid for v in voters]
    }
    return json.dumps(data, sort_keys=True).encode()

def is_tampered(self):
    """Check if election structure has been modified after opening."""
    current_hash = hashlib.sha256(self.gather_election_data()).digest()
    return current_hash != self.expected_hash
```

**Security Value:** Detects unauthorized modifications to election structure after cryptographic keys are generated, preventing ballot stuffing or voter list manipulation.

### 3. State Machine Enforcement

**Location:** `v3/steve/election.py`, multiple methods; `v3/schema.sql`, triggers

Election lifecycle (editable → open → closed) is enforced at both application and database layers:

```python
def open(self, pdb):
    assert self.is_editable()  # Prevents opening already-open elections
    ...

def close(self):
    assert self.is_open()  # Prevents closing non-open elections
    ...
```

Database trigger (`v3/schema.sql`):
```sql
CREATE TRIGGER prevent_open_close_update
BEFORE UPDATE ON election
FOR EACH ROW
WHEN OLD.state IN ('open', 'closed') AND NEW.state != OLD.state
BEGIN
    SELECT RAISE(ABORT, 'Cannot change state of open or closed election');
END;
```

**Security Value:** Prevents invalid state transitions that could compromise election integrity (e.g., reopening closed election, voting in closed election).

### 4. Vote Anonymization During Tally

**Location:** `v3/steve/election.py`, `tally_issue()` method (~line 240)

Votes are shuffled before tallying to prevent correlation through database insertion order:

```python
def tally_issue(self, iid):
    votes = self.q_get_votes_for_issue.all_rows(iid)
    
    # Decrypt votes
    decrypted = [crypto.decrypt_vote(v.ciphertext, v.vote_token) for v in votes]
    
    # Shuffle to prevent insertion-order correlation
    crypto.shuffle(decrypted)
    
    # Tally
    results = self._count_votes(decrypted)
    return results
```

**Security Value:** Breaks correlation between vote submission order and tally results, enhancing ballot secrecy.

### 5. Metadata Exclusion

**Location:** `v3/steve/election.py`, `get_metadata()` (~line 130); `v3/steve/persondb.py`, `get_person()` (~line 85)

Sensitive fields are explicitly excluded from returned metadata with inline documentation:

### 6. ID Unpredictability

**Location:** `v3/steve/election.py`, `add_issue()` (~line 145); `v3/schema.sql`, constraints

Election and issue IDs are generated via `crypto.create_id()` (10 hex characters) rather than sequential integers:

```python
def add_issue(self, title, description, type_):
    # Generate unpredictable ID
    while True:
        iid = crypto.create_id()
        try:
            self.c_add_issue.perform(self.eid, iid, title, description, type_)
            break
        except sqlite3.IntegrityError:
            continue  # Retry on collision
    return iid
```

Schema enforces format (`v3/schema.sql`):
```sql
CHECK (length(eid) = 10 AND eid GLOB '[0-9a-f]*')
```

**Security Value:** Prevents URL enumeration attacks and makes election/issue IDs non-guessable.

### 8. Referential Integrity

Comprehensive foreign key constraints prevent orphaned records:

## Architecture Observations

### 1. No Formal Data Classification Document

**Observation:** Sensitive data categories are not formally classified into protection levels. Protection decisions are embedded in code comments rather than referenced in a security document.

**Evidence:**
- Comments like `# NOTE: do not return the SALT or OPENED_KEY` exist in code
- No `SECURITY.md`, `DATA_PROTECTION.md`, or equivalent documentation
- ASVS 14.1.2 explicitly requires "documented set of protection requirements"

**Data Categories Identified (Undocumented):**
- **Level 1 (Highest):** Encrypted vote content (`ciphertext`), vote tokens
- **Level 2:** Cryptographic keys (`opened_key`), per-voter salts
- **Level 3:** Voter identities (`pid`), election ownership (`owner_pid`)
- **Level 4:** Election metadata (titles, descriptions)

**Recommendation:** Create `docs/DATA_PROTECTION.md` documenting:
- Data classification levels
- Required controls per level (encryption, logging, retention, access)
- Rationale for protection decisions

### 2. No Data Retention Policy

**Observation:** The schema provides indefinite storage with auto-incrementing vote IDs. No mechanism exists for data purging, archival, or retention-based access restrictions.

**Evidence:**
```sql
CREATE TABLE vote (
    vid  INTEGER PRIMARY KEY AUTOINCREMENT,  -- No expiration
    ...
);
```

**Domain Context Note:** "Votes are stored indefinitely for audit purposes" — but no formal retention schedule exists for:
- Person data (voter identities)
- Election metadata
- Audit logs
- Cryptographic keys

**Recommendation:** Document retention policy in `DATA_PROTECTION.md`:
```
- Votes: Indefinite (audit requirement)
- Person data: Retain while ASF member, purge 90 days after membership ends
- Election metadata: Indefinite (historical record)
- Audit logs: 7 years (compliance requirement)
- Cryptographic keys: Purge 90 days after election close
```

### 3. Database Encryption Decision Undocumented

**Observation:** The system design assumes database compromise (per known false positive guidance), but the risk acceptance and rationale for not using SQLCipher or equivalent at-rest encryption is not formally documented.

**Evidence:**
- Application-level encryption (`crypto.create_vote`) protects vote content
- Database-level encryption not implemented
- No documented risk acceptance or compensating controls

**Current Protection:**
- ✅ Vote content encrypted at application layer
- ❌ Cryptographic keys (`opened_key`, salts) stored in plaintext in database
- ❌ Voter identities (`pid`, `email`) stored in plaintext

**Recommendation:** Document in `SECURITY.md`:
```
## Database Encryption

**Decision:** Database-level encryption (SQLCipher) not implemented.

**Rationale:** 
- Vote content protected via application-level encryption
- Database compromise assumed in threat model
- Cryptographic keys rotated per-election (limited exposure window)

**Compensating Controls:**
- File system encryption (LUKS/dm-crypt) on database host
- Database file permissions restricted to application user
- Regular security audits of database access logs

**Risk Acceptance:** Cryptographic keys and voter identities exposed in database compromise scenario. Accepted based on defense-in-depth at infrastructure layer.
```

**Template for #7** (`docs/DATA_PROTECTION.md`):
```markdown
# Data Protection Requirements

## Data Classification

### Level 1: Highly Sensitive
- **Data:** Vote content (ciphertext), vote tokens
- **Encryption:** AES-256 application-level encryption (required)
- **Logging:** Never log vote content or tokens
- **Retention:** Indefinite (audit requirement)
- **Access:** Decryption only during tally by election owner

### Level 2: Sensitive Cryptographic Material
- **Data:** opened_key, per-voter salts
- **Encryption:** Database file-system encryption (recommended)
- **Logging:** Never log keys or salts
- **Retention:** Purge 90 days after election close
- **Access:** Restricted to election owner

### Level 3: Personal Identifiable Information
- **Data:** Voter identities (pid, email)
- **Encryption:** Database file-system encryption (recommended)
- **Logging:** Log only aggregated metrics, not individual PIDs
- **Retention:** Retain while ASF member, purge 90 days after membership ends
- **Access:** Election owner + administrators

### Level 4: Election Metadata
- **Data:** Titles, descriptions, issue content
- **Encryption:** None required (public after election opens)
- **Logging:** Standard application logging permitted
- **Retention:** Indefinite (historical record)
- **Access:** Public (authenticated users)
```
