## Findings

### Medium Severity

#### Finding ASVS-511-SEV-001

| Field | Detail |
|-------|--------|
| **Severity** | **MEDIUM** |
| **Finding ID** | ASVS-511-SEV-001 |
| **Title** | Complete Absence of File Handling Documentation for Document Serving Feature |
| **Location** | `v3/docs/schema.md`, `v3/ARCHITECTURE.md`, `v3/server/pages.py:562-580` |
| **Gap Type** | Type A — Feature with no documentation or controls |

**Description:**

The application has an active document-serving feature with two components:
1. A route `GET /docs/<iid>/<docname>` that serves files from the `DOCSDIR / iid` directory
2. A `rewrite_description()` function that converts `doc:filename` tokens in issue descriptions into clickable download links

Neither the `schema.md`, `ARCHITECTURE.md`, nor any other provided documentation defines:
- **Permitted file types** for documents associated with issues
- **Expected file extensions** (e.g., `.pdf`, `.txt`, `.md`)
- **Maximum file size** (including unpacked size for archives)
- **How files are made safe** for end-user download and processing (Content-Disposition, Content-Type validation, anti-virus scanning)
- **Behavior when a malicious file is detected**

**Relevant Code — Serving endpoint (`pages.py:562-580`):**
```python
@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require  # fine-grained per-doc authz within the handler
async def serve_doc(iid, docname):
    result = await basic_info()  # get PID

    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)

    ### verify the propriety of DOCNAME.       # <-- Acknowledged but not implemented

    return await quart.send_from_directory(DOCSDIR / iid, docname)
```

**Relevant Code — Link generation (`pages.py:52-58`):**
```python
def rewrite_description(issue):
    import re
    desc = issue.description
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

**Relevant Documentation Reference (`ARCHITECTURE.md:18`):**
```
│   ├── docs/                  # Per-issue documentation storage
```

This single line is the only documentation reference to the documents feature. No security specifications are provided.

**Impact:**

Without documented file handling requirements, developers have no specification to implement or test against. This has directly led to the missing validation in `serve_doc()`. An attacker who can place files in the docs directory (or exploit any future upload feature) could serve:
- HTML files with embedded JavaScript (stored XSS via Content-Type sniffing)
- Executable files disguised as documents
- Excessively large files causing storage exhaustion

**Remediation:**

Create a file handling specification document and reference it from `ARCHITECTURE.md`:

```markdown
## File Handling Specification

### Per-Issue Documents (`/docs/<iid>/`)

**Permitted file types:** PDF, plain text, Markdown
**Expected extensions:** `.pdf`, `.txt`, `.md`
**Maximum file size:** 10 MB per file, 50 MB per issue
**Maximum unpacked size:** N/A (archives not accepted)

**Safety measures:**
- File extension MUST be validated against allowlist before serving
- Content-Type header MUST be set explicitly based on extension mapping
- Content-Disposition: attachment MUST be set for all non-text files
- X-Content-Type-Options: nosniff MUST be set on all responses
- Files with unrecognized extensions MUST be rejected with 403

**Malicious file behavior:**
- Files failing extension validation are logged and return HTTP 403
- Upload mechanisms (when implemented) MUST validate MIME type matches extension
- Server logs MUST record denied file access attempts with user ID and filename
```

#### Finding ASVS-511-SEV-003

| Field | Detail |
|-------|--------|
| **Severity** | **MEDIUM** |
| **Finding ID** | ASVS-511-SEV-003 |
| **Title** | Issue Description Doc-Link Rewriting Generates Unvalidated File References |
| **Location** | `v3/server/pages.py:52-58`, function `rewrite_description()` |
| **Gap Type** | Type A — No validation control exists |

**Description:**

The `rewrite_description()` function parses issue descriptions and converts `doc:filename` patterns into HTML anchor tags pointing to `/docs/{iid}/{filename}`. The filename extracted from the description is not validated against any allowlist of permitted file types or extensions before being embedded in the HTML link:

```python
def rewrite_description(issue):
    import re
    desc = issue.description
    def repl(match):
        filename = match.group(1)  # No validation of filename
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

The regex `r'doc:([^\s]+)'` captures any non-whitespace sequence, meaning filenames like `../../../etc/passwd`, `evil.html`, or `payload.exe` would be turned into clickable links. While the `serve_doc` endpoint's `send_from_directory` provides basic path traversal protection, the absence of documented permitted file types means there is no basis for validation at either the link-generation or file-serving layer.

**Data Flow:**
```
Issue description (DB) → regex match `doc:filename` → unvalidated filename → HTML <a> tag → User clicks → serve_doc(iid, filename) → unvalidated file served
```

**Impact:**

- Generates links to file types that should not be served (executables, HTML, etc.)
- Social engineering vector: attackers with issue-editing privileges can embed links to dangerous file types
- The lack of documentation means there is no definition of what `doc:filename` references should be valid

**Remediation:**

Validate the filename in `rewrite_description()` against the documented allowlist:

```python
ALLOWED_DOC_EXTENSIONS = {'.pdf', '.txt', '.md'}

def rewrite_description(issue):
    import re
    desc = issue.description
    def repl(match):
        filename = match.group(1)
        ext = pathlib.Path(filename).suffix.lower()
        if ext not in ALLOWED_DOC_EXTENSIONS or '/' in filename or '\\' in filename:
            return f'[invalid document reference: {filename}]'
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

### High Severity

#### Finding ASVS-511-SEV-002

| Field | Detail |
|-------|--------|
| **Severity** | **HIGH** |
| **Finding ID** | ASVS-511-SEV-002 |
| **Title** | Document Serving Endpoint Lacks Filename Validation and Safe-Download Controls (Type B Gap) |
| **Location** | `v3/server/pages.py:576-580`, function `serve_doc()` |
| **Gap Type** | Type B — Control need identified (TODO comment) but not implemented |

**Description:**

The `serve_doc()` function serves arbitrary files from the `DOCSDIR / iid` directory without any filename validation, extension allowlisting, Content-Type enforcement, or safe-download headers. The developers explicitly acknowledged this gap with a TODO comment but never implemented the control:

```python
### verify the propriety of DOCNAME.

# Return per-issue document.
return await quart.send_from_directory(DOCSDIR / iid, docname)
```

This is a **Type B gap**: the need for a security control was identified, but the control was never implemented, creating false confidence that the issue is tracked.

**Data Flow:**
```
User request → URL parameter `docname` → NO validation → send_from_directory(DOCSDIR/iid, docname) → File served to browser
```

**Proof of Concept:**

Assuming a malicious file `evil.html` containing JavaScript exists in `DOCSDIR/<valid-iid>/`:

```http
GET /docs/a1b2c3d4e5/evil.html HTTP/1.1
Host: voting.example.org
Cookie: <valid_session>
```

The file would be served with a `text/html` Content-Type, executing any embedded JavaScript in the user's browser context (stored XSS).

Similarly, serving unexpected file types:
```http
GET /docs/a1b2c3d4e5/payload.exe HTTP/1.1
```

**Impact:**

- **Stored XSS** if HTML/SVG files exist in the docs directory
- **Malware distribution** if executable files can be placed in docs
- **Content-Type confusion** attacks via browser MIME sniffing
- Without documented file handling requirements (Finding 001), there is no specification for what constitutes a valid document, making this impossible to properly remediate

**Remediation:**

First, create the documentation (Finding 001). Then implement the specified controls:

```python
import mimetypes

# Define allowlist based on documentation specification
ALLOWED_DOC_EXTENSIONS = {'.pdf', '.txt', '.md'}
SAFE_CONTENT_TYPES = {
    '.pdf': 'application/pdf',
    '.txt': 'text/plain; charset=utf-8',
    '.md': 'text/plain; charset=utf-8',
}

@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require
async def serve_doc(iid, docname):
    result = await basic_info()

    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)

    # Validate filename: alphanumeric, hyphens, underscores, single dot for extension
    import re
    if not re.match(r'^[a-zA-Z0-9_-]+\.[a-zA-Z0-9]+$', docname):
        _LOGGER.warning(f'Invalid docname "{docname}" requested by {result.uid}')
        quart.abort(403)

    # Validate extension against allowlist
    ext = pathlib.Path(docname).suffix.lower()
    if ext not in ALLOWED_DOC_EXTENSIONS:
        _LOGGER.warning(f'Disallowed extension "{ext}" requested by {result.uid}')
        quart.abort(403)

    # Serve with explicit Content-Type and safe headers
    response = await quart.send_from_directory(DOCSDIR / iid, docname)
    response.headers['Content-Type'] = SAFE_CONTENT_TYPES.get(ext, 'application/octet-stream')
    response.headers['Content-Disposition'] = f'attachment; filename="{docname}"'
    response.headers['X-Content-Type-Options'] = 'nosniff'
    return response
```

## Architecture Observations

1. **Document management is partially implemented**: The serving infrastructure exists (`serve_doc`, `rewrite_description`, `DOCSDIR`), but no upload mechanism is visible in the provided code. This suggests documents are placed manually or through CLI tools, making documentation of acceptable file types even more critical since there's no programmatic enforcement at upload time.

2. **Two-layer file access**: Documents require both authentication (`@asfquart.auth.require`) and authorization (mayvote table check). This is a positive pattern, but the lack of file-type validation at the serving layer undermines the security posture.

3. **The `iid` in the path creates per-issue isolation**: Documents are stored in `DOCSDIR / iid /`, which naturally isolates documents per issue. Combined with the mayvote authorization check, this limits the blast radius of any single compromised document.

4. **No upload endpoint in provided code**: The absence of a visible upload endpoint means either (a) uploads happen through a mechanism not shown, or (b) files are placed manually. Either way, documentation is the primary control that should define what files are acceptable.
