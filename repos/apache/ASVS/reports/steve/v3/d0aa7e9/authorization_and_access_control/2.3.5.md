## Executive Summary

**Repository:** apache/steve

**Auditor:** Tooling Agents

**ASVS Requirement:** 2.3.5 — Verify that high-value business logic flows require multi-user approval to prevent unauthorized or accidental actions. This could include but is not limited to large monetary transfers, contract approvals, access to classified information, or safety overrides in manufacturing.

**Summary:** The election system contains critical and high-severity gaps in authorization enforcement and completely lacks multi-user approval workflows for irreversible, high-value election operations. The database schema defines ownership and authorization fields (`owner_pid`, `authz`) but these are never checked in any web handler — a Type B control gap. Furthermore, no multi-user approval mechanism exists for any election lifecycle operation (opening, closing, deletion), all of which are irreversible and have significant consequences for the integrity of the election process.

### Critical Severity

#### ASVS-235-CRT-001: Missing Election Ownership Authorization — Any Committer Can Open/Close/Modify Any Election (Type B Gap)

**Severity:** CRITICAL

**File:** `v3/server/pages.py` (multiple functions) and `v3/steve/election.py`

**Gap Type:** Type B — Control EXISTS (schema fields `owner_pid` and `authz`) but is NOT CALLED at any entry point.

**Control Definition:**
```sql
-- v3/schema.sql lines 68-75
owner_pid  TEXT NOT NULL,
/* What authz group is allowed to edit this election? If NULL,
   then only the OWNER_PID can edit. */
authz  TEXT,
```

**Vulnerable Code (Opening an election — any committer):**
```python
# v3/server/pages.py lines 479-496
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    result = await basic_info()

    ### check authz       # <--- NEVER IMPLEMENTED

    pdb = steve.persondb.PersonDB.open(DB_FNAME)

    # Open the Election — IRREVERSIBLE, generates crypto keys
    election.open(pdb)
    ...
```

**Data Flow:**
```
Authenticated committer (any) → GET /do-open/<eid> → load_election(eid) 
→ [NO owner_pid check] → election.open(pdb) → IRREVERSIBLE state change
```

**Proof of Concept:**
Any authenticated ASF committer, even one with no relation to the election, can:
```
GET /do-open/a1b2c3d4e5   → Opens any editable election
GET /do-close/a1b2c3d4e5  → Closes any open election
POST /do-add-issue/a1b2c3d4e5  → Adds issues to any election
POST /do-delete-issue/a1b2c3d4e5/f6g7h8i9j0  → Deletes issues from any election
```

**Impact:** Any authenticated committer can irreversibly open or close elections they do not own, add or remove voting issues from others' elections, and effectively sabotage any election in the system. This provides false confidence because the schema documents an ownership model that is never enforced.

**Remediation:**
```python
# Add to load_election decorator or create a separate check
async def verify_election_owner(election, uid):
    """Verify the current user is the election owner or in the authz group."""
    md = election.get_metadata()
    if md.owner_pid == uid:
        return True
    if md.authz:
        # Check LDAP group membership for authz group
        session = await asfquart.session.read()
        if md.authz in session.get('groups', []):
            return True
    quart.abort(403, 'You are not authorized to manage this election')

# Apply in each management endpoint:
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    await verify_election_owner(election, result.uid)  # ENFORCE
    ...
```

### High Severity

#### ASVS-235-HGH-001: No Multi-User Approval for Irreversible Election State Transitions

**Severity:** HIGH

**Files:** `v3/server/pages.py` lines 479-515, `v3/steve/election.py` lines 70-120

**Description:** Opening and closing elections are the highest-value operations in this system. Opening an election is explicitly irreversible (generates cryptographic salt and opened_key, sets per-voter salts), and closing permanently terminates voting. Neither operation requires approval from a second authorized user.

**Irreversibility evidence in election.py:**
```python
# v3/steve/election.py lines 70-83
def open(self, pdb):
    # Double-check the Election is in the editing state.
    assert self.is_editable()
    
    # Add salts first — generates per-voter cryptographic material
    self.add_salts()
    
    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)  # Anti-tamper hash
    
    self.c_open.perform(salt, opened_key, self.eid)
    # NO UNDO PATH EXISTS — state machine prevents reversal
```

```python
# v3/steve/election.py lines 114-120
def close(self):
    "Close an election."
    assert self.is_open()
    # Simple tweak — but irreversible via state machine
    self.c_close.perform(self.eid)
```

**Single-user invocation (pages.py):**
```python
# v3/server/pages.py lines 479-496
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)  # Single user, immediate execution, no approval
    ...
```

**High-Value Operations Without Multi-User Approval:**

| Operation | Endpoint | Reversible | Multi-User Approval |
|-----------|----------|------------|---------------------|
| Open election | `GET /do-open/<eid>` | ❌ No | ❌ None |
| Close election | `GET /do-close/<eid>` | ❌ No | ❌ None |
| Delete election | `election.delete()` (CLI) | ❌ No | ❌ None |
| Tally votes | `election.tally_issue()` (CLI) | N/A (read) | ❌ None |

**Impact:** A single user (or an attacker who compromises a single committer account) can unilaterally open an election prematurely (before all issues/voters are configured), close an election early (disenfranchising voters), or trigger tallying. In an election system where integrity and fairness are paramount, the lack of multi-user approval for irreversible operations creates significant risk of both accidental and intentional abuse.

**Proof of Concept:**
```
# Attacker with any committer credentials prematurely opens an election
# that is still being configured, locking it from further edits:
GET /do-open/a1b2c3d4e5

# Or closes a running election before intended close date:
GET /do-close/a1b2c3d4e5
```

**Remediation:**
```python
# 1. Add approval tracking to the schema
# schema.sql addition:
"""
CREATE TABLE approval_request (
    arid INTEGER PRIMARY KEY AUTOINCREMENT,
    eid TEXT NOT NULL,
    action TEXT NOT NULL CHECK (action IN ('open', 'close')),
    requested_by TEXT NOT NULL,
    requested_at INTEGER NOT NULL,
    approved_by TEXT,
    approved_at INTEGER,
    status TEXT NOT NULL DEFAULT 'pending' 
        CHECK (status IN ('pending', 'approved', 'rejected')),
    FOREIGN KEY (eid) REFERENCES election(eid),
    FOREIGN KEY (requested_by) REFERENCES person(pid),
    FOREIGN KEY (approved_by) REFERENCES person(pid),
    CHECK (requested_by != approved_by)  -- Different users required
) STRICT;
"""

# 2. Two-phase endpoint pattern:
@APP.post('/do-request-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def request_open_endpoint(election):
    result = await basic_info()
    await verify_election_owner(election, result.uid)
    # Record pending approval request
    create_approval_request(election.eid, 'open', result.uid)
    await flash_info('Open request submitted. A second authorized user must approve.')
    return quart.redirect(f'/manage/{election.eid}', code=303)

@APP.post('/do-approve-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def approve_open_endpoint(election):
    result = await basic_info()
    await verify_election_owner(election, result.uid)
    request = get_pending_approval(election.eid, 'open')
    if request.requested_by == result.uid:
        quart.abort(403, 'Cannot approve your own request')
    # Now execute the open
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    mark_approved(request, result.uid)
    ...
```

#### ASVS-235-HGH-002: Irreversible State-Changing Operations Use GET Method Enabling Accidental Triggering

**Severity:** HIGH

**File:** `v3/server/pages.py` lines 479-515

**Description:** The most consequential operations in the system — opening and closing elections — are implemented as GET endpoints. This violates HTTP semantics (GET should be safe/idempotent) and makes these irreversible operations trivially triggerable by link clicks, browser prefetching, crawlers, or embedded resources, directly contravening ASVS 2.3.5's goal of preventing "unauthorized or accidental actions."

**Vulnerable Code:**
```python
# v3/server/pages.py lines 479-480
@APP.get('/do-open/<eid>')      # GET for irreversible state change!
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    ...
    election.open(pdb)           # Irreversible
    ...

# v3/server/pages.py lines 499-500
@APP.get('/do-close/<eid>')     # GET for irreversible state change!
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    ...
    election.close()             # Irreversible
    ...
```

**Attack Vector:**
```html
<!-- Embedded in any page visible to an authenticated committer -->
<img src="https://steve.example.org/do-close/a1b2c3d4e5" width="0" height="0">

<!-- Or link that appears legitimate -->
<a href="/do-open/a1b2c3d4e5">View election details</a>
```

**Impact:** Combined with the missing authorization checks (ASVS-235-CRT-001), this means an attacker can embed a URL that will irreversibly open or close any election when visited by any authenticated committer. Even without malicious intent, browser prefetching or link previewing tools could accidentally trigger these operations.

**Remediation:**
```python
# Change to POST with confirmation
@APP.post('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    await verify_election_owner(election, result.uid)
    # Verify CSRF token
    form = await quart.request.form
    if form.get('csrf_token') != result.csrf_token:
        quart.abort(403)
    election.open(pdb)
    ...

@APP.post('/do-close/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    await verify_election_owner(election, result.uid)
    form = await quart.request.form
    if form.get('csrf_token') != result.csrf_token:
        quart.abort(403)
    election.close()
    ...
```

## Positive Security Patterns

1. **Robust State Machine:** The `_compute_state()` method and assertion-based state checks in `election.py` effectively prevent invalid state transitions (e.g., voting on an editable election, closing an already-closed election).

2. **Anti-Tamper Detection:** The `is_tampered()` method and `opened_key` mechanism provide integrity verification — changes to election data, issues, or voter lists after opening would be detected.

3. **Cryptographic Vote Privacy:** Per-voter salts, encrypted vote storage, and vote shuffling before tallying demonstrate strong consideration for voter anonymity.

4. **Non-Sequential IDs:** 10-character hex IDs with collision retry prevent election/issue enumeration.

5. **Parameterized Queries:** Use of `asfpy.db` with YAML-defined parameterized queries prevents SQL injection.

6. **Voter Eligibility Enforcement:** The `mayvote` table with `(pid, iid)` primary key provides database-level enforcement of voter eligibility, and this is checked in `add_vote()` before accepting votes.

7. **Transactional Operations:** Multi-step destructive operations (`delete()`, `add_salts()`) use explicit `BEGIN TRANSACTION`/`COMMIT`.

## Architecture Observations

1. **Two-Layer Authorization Gap:** The system has authentication at the web layer (`asfquart.auth.require`) but zero authorization at either the web layer (pages.py) or business logic layer (election.py). The `owner_pid` and `authz` fields create a documented authorization model that is never implemented.

2. **Insufficient Separation of Privileges:** Election creation requires `R.pmc_member` (elevated privilege), but all subsequent management operations (including opening/closing) only require `R.committer`. This privilege downgrade means the stricter creation check provides no real protection.

3. **CLI Access as Implicit Multi-User Bypass:** The CLI tools (`tally.py`, etc.) have direct database access with no approval workflow, meaning server administrators can unilaterally perform all high-value operations.

4. **State Machine as Partial Control:** The state machine prevents certain classes of invalid operations (e.g., double-opening), but it doesn't address who should be authorized to trigger valid transitions.
