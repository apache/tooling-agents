**Executive Assessment:**

The application implements several business logic controls in its design (election state machine, owner authorization, vote type validation, tamper detection) but critical gaps exist where these controls are **defined but never called** at key entry points. The most severe finding is that election ownership authorization is completely unimplemented across all management endpoints, despite placeholders indicating it was intended.

#### ASVS-232-HIG-001: Inconsistent State Machine Enforcement Using Bypassable `assert` Statements

**Severity:** HIGH  
**Gap Type:** Type B (Proper control mechanism EXISTS but NOT USED for most operations)

**Location:**
- **File:** `v3/steve/election.py`
- **Functions:** `open()` (line 70), `close()` (line 110), `add_issue()` (line 176), `edit_issue()` (line 193), `delete_issue()` (line 205), `add_voter()` (line 227), `delete()` (line 54), `add_salts()` (line 119), `_compute_state()` (lines 349-355)
- **Related Functions:** `_all_metadata()` (proper control mechanism), `add_vote()` (line 236, correct usage), `tally_issue()` (line 252, correct usage)

**Proper Control Defined:**
```python
# election.py - _all_metadata with required_state parameter
def _all_metadata(self, required_state=None):
    md = self.q_metadata.first_row(self.eid)
    if not md:
        raise self._disappeared()
    if required_state:
        state = self._compute_state(md)
        if state != required_state:
            raise ElectionBadState(self.eid, state, required_state)
    return md
```

**Where Proper Control IS Used (2 methods):**
- `add_vote()` at `election.py:236`: `md = self._all_metadata(self.S_OPEN)` ✅
- `tally_issue()` at `election.py:252`: `md = self._all_metadata(self.S_CLOSED)` ✅

| `close()` | `election.py:110` | `assert self.is_open()` |
| `add_issue()` | `election.py:176` | `assert self.is_editable()` |
| `edit_issue()` | `election.py:193` | `assert self.is_editable()` |
| `delete_issue()` | `election.py:205` | `assert self.is_editable()` |
| `add_voter()` | `election.py:227` | `assert self.is_editable()` |
| `delete()` | `election.py:54` | `assert self.is_editable()` |
| `add_salts()` | `election.py:119` | `assert self.is_editable()` |
| `_compute_state()` | `election.py:349-355` | Multiple `assert` for state invariants |

**Vulnerable Code:**
```python
# election.py:70
def open(self, pdb):
    # Double-check the Election is in the editing state.
    assert self.is_editable()    # BYPASSED with python -O
    self.add_salts()             # Regenerates all salts, invalidating existing votes
    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)
    self.c_open.perform(salt, opened_key, self.eid)

# election.py:110
def close(self):
    assert self.is_open()        # BYPASSED with python -O
    self.c_close.perform(self.eid)
```

**Data Flow:**
1. Python interpreter runs with `-O` or `-OO` flags (or `PYTHONOPTIMIZE` env var)
2. All `assert` statements are compiled out
3. State checks are silently removed
4. Any state transition can occur regardless of current state

**Proof of Concept:**
```bash
# Run application with optimization
PYTHONOPTIMIZE=1 python server.py

# Or
python -O server.py

# Now all assert-based state checks are bypassed
# An already-open election can be "re-opened":
GET /do-open/already_open_election_id
# This regenerates all salts, invalidating every existing encrypted vote
```

**Impact:**
- If the Python interpreter runs with optimization (`-O` or `-OO` flags, or `PYTHONOPTIMIZE` env var), **all `assert` statements are silently removed**
- An already-open election could be "re-opened", regenerating all salts and `opened_key`, **invalidating every existing encrypted vote** (votes become undecryptable because vote tokens change)
- A closed election could be re-opened or issues could be modified after election closure
- The proper enforcement mechanism (`_all_metadata(required_state)`) already exists but is only used for 2 of 11 state-sensitive methods

**Remediation:**
```python
def open(self, pdb):
    # Use the proper state validation mechanism
    self._all_metadata(required_state=self.S_EDITABLE)
    self.add_salts()
    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)
    self.c_open.perform(salt, opened_key, self.eid)

def close(self):
    self._all_metadata(required_state=self.S_OPEN)
    self.c_close.perform(self.eid)

def add_issue(self, title, description, vtype, kv):
```

**Severity:** MEDIUM  
**Gap Type:** Type B (Control EXISTS but NOT CALLED)

**Location:**
- **File:** `v3/steve/election.py`
- **Function:** `is_tampered()` (lines 316-331) — defined but never called
- **Related Functions:** `add_vote()` (line 236), `tally_issue()` (line 252), `vote_on_page()` in `pages.py`
- **Related Endpoint:** `do_vote_endpoint()` in `pages.py` (line 336)

**Control Defined:**
```python
# election.py:316-331
def is_tampered(self, pdb):
    # The Election should be open (don't allow voting when TAMPERED)
    # or it may be closed (don't bother tallying, if TAMPERED).
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, salt)
    return opened_key != md.opened_key
```

**Where It Should Be Called But Isn't:**

| Operation | Endpoint | File:Line | Why Needed |
|-----------|----------|-----------|------------|
| Vote submission | `do_vote_endpoint()` → `add_vote()` | `pages.py:336`, `election.py:236` | Method comment says "don't allow voting when TAMPERED" |
| Election tallying | `tally_issue()` | `election.py:252` | Method comment says "don't bother tallying, if TAMPERED" |
| View voting page | `vote_on_page()` | `pages.py` | Users should be warned if election integrity is compromised |

**Evidence the control is intended to be used:**
The method's own docstring states: *"The Election should be open (don't allow voting when TAMPERED)"* and *"it may be closed (don't bother tallying, if TAMPERED)"*, but neither `add_vote()` nor `tally_issue()` calls `is_tampered()`.

**Data Flow:**
1. Election data (issues, voters) is tampered with after opening
2. Votes are submitted via `do_vote_endpoint()` → `add_vote()`
3. No tamper check is performed
4. Votes are encrypted and stored against corrupted data
5. Tallying proceeds on tampered election data

**Proof of Concept:**
```sql

UPDATE issue SET description = 'Tampered description' WHERE eid = 'a1b2c3d4e5';

-- Or manipulate voter list
DELETE FROM mayvote WHERE eid = 'a1b2c3d4e5' AND pid = 'legitimate_voter';
```

```http
-- Votes still accepted despite tampering
POST /do-vote/a1b2c3d4e5 HTTP/1.1
Cookie: [valid session]
Content-Type: application/x-www-form-urlencoded

vote-issue1=y&vote-issue2=n
```

**Impact:**
- If election data is tampered with (issues modified, voters altered) after opening, the system will silently accept votes and produce tallies against corrupted data
- The anti-tamper mechanism provides no protection because it is never executed during normal operation flow
- The `opened_key` cryptographic hash is computed but never verified, rendering the integrity protection useless

```python
    # Validate votestring against issue type
    issue = self.q_get_issue.first_row(iid)
    if not issue:
        raise IssueNotFound(iid)
    
    m = vtypes.vtype_module(issue.type)
    if not m.validate(votestring, self.json2kv(issue.kv)):
        raise InvalidVote(iid, votestring)

    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

#### ASVS-232-MED-003: Irreversible State-Changing Operations Use GET Method

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `do_open_endpoint()` (line 404), `do_close_endpoint()` (line 422)
- **Related Functions:** Correctly designed POST endpoints at lines 336, 388, 439, 461, 481

**Vulnerable Endpoints:**
```python
# pages.py:404
@APP.get('/do-open/<eid>')            # GET for irreversible state change!
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    # ...
    election.open(pdb)                 # Irreversible: generates salts and opened_key

# pages.py:422
@APP.get('/do-close/<eid>')           # GET for irreversible state change!
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    # ...
    election.close()                   # Irreversible: prevents further voting
```

**Contrast with correctly-designed endpoints:**
```python
@APP.post('/do-vote/<eid>')           # POST - correct
@APP.post('/do-create-election')      # POST - correct  
@APP.post('/do-add-issue/<eid>')      # POST - correct
@APP.post('/do-edit-issue/<eid>/<iid>')  # POST - correct
@APP.post('/do-delete-issue/<eid>/<iid>')  # POST - correct
```

**Data Flow:**
1. Attacker embeds resource in page visited by authenticated committer
2. Browser automatically sends GET request with authentication cookies
3. Irreversible state change occurs without user intent

**Proof of Concept:**
```html
<!-- Embedded in any page a committer might visit -->
<img src="https://steve.example.org/do-open/a1b2c3d4e5" width="0" height="0">
<img src="https://steve.example.org/do-close/a1b2c3d4e5" width="0" height="0">

<!-- Or via prefetch -->
<link rel="prefetch" href="https://steve.example.org/do-open/a1b2c3d4e5">

<!-- Or via redirect -->
<meta http-equiv="refresh" content="0;url=https://steve.example.org/do-open/a1b2c3d4e5">
```

**Impact:**
- Opening an election is **irreversible** — it generates cryptographic salts for all voter/issue pairs and computes the tamper-detection key. Using GET means this can be triggered by:
  - Browser link prefetching (`<link rel="prefetch">`)
  - Image tags: `<img src="/do-open/a1b2c3d4e5">`
  - Crawlers, browser extensions, or proxy pre-fetching
  - Appearing in browser history, referer headers, and access logs
- Combined with the missing owner authorization (ASVS-232-CRI-001), any page visited by a committer could silently trigger election state transitions via embedded resources
- Election IDs appear in URLs and could be leaked via Referer headers

**Remediation:**
```python
@APP.post('/do-open/<eid>')     # Change to POST
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    await flash_info(f'Election opened: {election.eid}')
    return quart.redirect('/manager', code=303)
    
@APP.post('/do-close/<eid>')    # Change to POST
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz
    election.close()
    await flash_info(f'Election closed: {election.eid}')
    return quart.redirect('/manager', code=303)
```

## Positive Security Patterns

1. **Parameterized queries via `asfpy.db`**: All SQL operations use a query YAML system with parameterized inputs, mitigating SQL injection.

2. **Non-sequential IDs**: Election and issue IDs use 10 hex characters generated by `crypto.create_id()`, preventing enumeration.

3. **Per-voter cryptographic salts**: Each voter/issue pair has a unique salt for vote token generation and encryption key derivation.

4. **Proper state checking for votes and tallying**: `add_vote()` and `tally_issue()` use `_all_metadata(required_state)` which properly raises `ElectionBadState`.

5. **Schema-level constraints**: `CHECK` constraints, `STRICT` mode, foreign keys with `ON DELETE RESTRICT`, and the `prevent_open_close_update` trigger provide defense-in-depth.

6. **Transactional operations**: Multi-step operations (`delete()`, `add_salts()`) use explicit transactions.

7. **Vote-on page eligibility check**: `vote_on_page()` verifies mayvote entries before displaying the voting interface.

8. **Document serving authorization**: `serve_doc()` checks mayvote eligibility before serving per-issue documents.

9. **Election creation requires higher privilege**: `R.pmc_member` for creation vs `R.committer` for viewing.

10. **Vote shuffling before tallying**: `crypto.shuffle(votes)` prevents database-order leakage.

## Architecture Observations

1. **Authorization model is designed but unimplemented**: The schema has `owner_pid` and `authz` fields, and code has `### check authz` placeholders everywhere. The architecture clearly intended owner-based authorization, but it was never connected.

2. **Two state enforcement mechanisms coexist**: The codebase has both `assert` (ad-hoc, bypassable) and `_all_metadata(required_state)` (proper, exception-based). The proper mechanism was added later for vote/tally operations but not retrofitted to earlier methods.

3. **Implicit vs explicit eligibility checking**: Vote eligibility is enforced implicitly by a NoneType crash in `add_vote()` rather than an explicit check with a proper error message and response code.

4. **Defense-in-depth at schema level**: The database schema includes multiple protective layers (triggers, foreign keys, CHECK constraints) that would catch some logic errors even if application code fails.

5. **Cryptographic integrity unused**: The `opened_key` tamper-detection mechanism is computed and stored but never verified in any operational code path, representing wasted cryptographic effort.
