### Key Findings

The application delegates authentication entirely to an external SSO provider (ASF OAuth) but maintains its own sessions via `asfquart.session`. There is **no mechanism** in the codebase to terminate active sessions after authentication factor changes, nor is there any user-facing option to terminate other sessions. This creates a gap where compromised sessions persist even after a user changes their credentials at the SSO provider.

## Findings

### High Severity

#### ASVS-743-HIGH-001: No Mechanism to Terminate Sessions After Authentication Factor Changes

**Severity:** HIGH

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `basic_info()` (Lines 63-76), `profile_page()` (Lines 506-520), `settings_page()` (Lines 514-520)
- **Related Files:** `v3/ARCHITECTURE.md`

**Related Functions Checked:**
- `basic_info()` — Session reading only, no termination capability
- `profile_page()` — Read-only profile display
- `settings_page()` — Read-only settings display
- All authenticated routes — No session termination logic

**Description:**

The application provides no functionality to terminate all other active sessions after a change or removal of any authentication factor. Since authentication is delegated to an external SSO provider (ASF OAuth), but the application maintains its own independent sessions via `asfquart.session`, there is no integration point to invalidate application-side sessions when a user's credentials are changed at the SSO provider.

**Vulnerable Code:**

*Session Management (read-only, no termination):*

```python

        email=s['email'],

else:
    # No session.
    basic.update(
        uid=None,
        name=None,
        email=None,
    )
```

*Profile and Settings pages (read-only with no session management):*

**Architecture Evidence:**

```markdown
# File: v3/ARCHITECTURE.md
- **Authentication**: External (ASF OAuth); no local auth.
```

**Data Flow:**

1. User authenticates via ASF OAuth (external SSO provider)
2. Application creates session via `asfquart.session`
3. User changes password/credentials at SSO provider
4. **Gap:** Application has no mechanism to be notified or react to credential change
5. Existing application sessions remain valid indefinitely
6. Compromised sessions persist even after credential rotation

**Missing Capabilities:**

| Missing Capability | Impact |
|--------------------|--------|
| No "terminate all sessions" endpoint | Users cannot invalidate compromised sessions |
| No SSO backchannel logout / event listener | Application is unaware when credentials change at the SSO provider |
| No user-facing session listing | Users cannot see or manage active sessions |
| No logout endpoint visible in code | No clear way to terminate even the current session |
| No session invalidation logic anywhere | Once created, sessions appear to persist until natural expiration |

**Proof of Concept:**

1. Attacker obtains a valid session cookie (via XSS, session fixation, network sniffing, or shared device)
2. Victim detects unauthorized access and changes their password at the ASF SSO provider
3. Attacker's existing application session **remains valid** because the application has no mechanism to be notified of or react to the credential change
4. Attacker continues to operate as the victim in the voting application, potentially casting votes or managing elections

**Impact:**

- **Compliance Violation:** Direct violation of ASVS 7.4.3 Level 2 requirement
- **Session Persistence After Credential Change:** Compromised sessions survive credential rotation, undermining the purpose of password changes
- **Voting Integrity Risk:** For a voting application, this means an attacker could continue casting votes or modifying elections even after the legitimate user has attempted remediation
- **Standards Violation:** Violates NIST SP 800-63B Section 7.1 guidance on session re-authentication after credential changes
- **User Trust:** Users have no visibility into or control over their active sessions

**Gap Classification:** Type A — Entry point (authentication factor change event from SSO) with **NO control** implemented on the application side.

**Remediation:**

**1. Implement a "Terminate All Sessions" endpoint:**

```python
@APP.post('/do-terminate-sessions')
@asfquart.auth.require
async def do_terminate_sessions():
    s = await asfquart.session.read()
    current_session_id = s.get('session_id')
    
    # Invalidate all sessions for this user except current
    await asfquart.session.terminate_all_for_user(
        uid=s['uid'],
        except_session=current_session_id,
    )
    
    await flash_success('All other sessions have been terminated.')
    return quart.redirect('/settings', code=303)
```

**2. Integrate with SSO backchannel logout (if supported):**

```python
@APP.post('/backchannel-logout')
async def sso_backchannel_logout():
    """Handle SSO provider notification of credential change."""
    # Validate the logout token from the SSO provider
    token = await quart.request.get_data()
    uid = validate_sso_logout_token(token)
    if uid:
        await asfquart.session.terminate_all_for_user(uid=uid)
    return '', 200
```

**3. Add session management UI to the `/settings` page:**

```python
@APP.get('/settings')
@asfquart.auth.require
@APP.use_template(TEMPLATES / 'settings.ezt')
async def settings_page():
    result = await basic_info()
    result.title = 'Settings'

    s = await asfquart.session.read()
    result.active_sessions = await asfquart.session.list_for_user(s['uid'])

    return result
```

## Positive Security Patterns

1. **Consistent Authentication Enforcement:** All sensitive routes use `@asfquart.auth.require` with appropriate role requirements (`R.committer`, `R.pmc_member`), ensuring proper access control throughout the application.

2. **Server-Side Session Data Reading:** `asfquart.session.read()` extracts user identity server-side, not from client-controlled input, preventing session manipulation attacks.

3. **Comprehensive Security Event Logging:** All state-changing operations log the UID performing the action (Lines 101, 382, 404, 425, 446, 473, 497, 518), providing an audit trail for security investigations.

4. **SQL Injection Prevention:** The `Election` class uses prepared statements via `asfpy.db` to prevent SQL injection attacks.

5. **Proper SSO Integration:** Authentication is correctly delegated to the ASF SSO provider rather than implementing custom authentication logic, reducing the attack surface.

---

## Architecture Observations

1. **Decoupled Authentication Creates Session Termination Blind Spot:** The application correctly delegates authentication to the ASF SSO provider but does not subscribe to any events from the provider, creating a gap where the application-side session state diverges from the SSO authentication state. This architectural decision requires additional integration work to maintain session consistency.

2. **No Session Store Abstraction Visible:** The `asfquart.session` module is used only for reading. There is no visible API for session enumeration, invalidation, or bulk termination — suggesting the underlying framework may need extension to support ASVS 7.4.3 requirements.

3. **Settings Page as Natural Integration Point:** The `/settings` route (Line 514) exists but renders a read-only page. This is the logical location for session management controls and would provide a familiar user experience pattern.

4. **Session Lifecycle Management Gap:** While session creation and reading are implemented, there is no evidence of session termination, timeout, or renewal logic in the reviewed code, suggesting a broader session lifecycle management gap beyond just the ASVS 7.4.3 requirement.
