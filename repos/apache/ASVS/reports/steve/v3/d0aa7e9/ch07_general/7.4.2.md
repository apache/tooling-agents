### Key Findings

The application **fails to meet ASVS 7.4.2 requirements**. When user accounts are deleted via `PersonDB.delete_person()`, no mechanism exists to terminate active sessions. Users retain full application access until their session naturally expires. Additionally, no account disable/deactivation mechanism exists, and per-request validation of user account status is absent.

## Findings

### High Severity

#### ASVS-742-HIGH-001: No Session Termination When User Account Is Deleted

**Severity:** HIGH

**Location:** `v3/steve/persondb.py`, `PersonDB.delete_person()`, Lines 51-61

**Gap Type:** Type A — Entry point with NO control

**Vulnerable Code:**
```python
def delete_person(self, pid):
    "Delete the Person designated by PID."

    # NOTE: if this person has ever been involved in a vote
    #   (ie. the PID exists in a "mayvote" row), then this will
    #   throw a referential integrity error.
    #
    ### maybe we just don't delete a person, ever?

    self.c_delete_person.perform(pid)

    # If the Person didn't exist, we deleted nothing.
    if self.c_delete_person.rowcount == 0:
        raise PersonNotFound(pid)
    # else .rowcount == 1
```

**Data Flow:**
1. Admin invokes `delete_person(pid)` to remove a departing user
2. User record is deleted from the `person` table in SQLite
3. **No session store is consulted or modified**
4. Deleted user's active session (managed by `asfquart.session`) remains valid
5. Deleted user continues accessing the application on subsequent requests

**Proof of Concept:**
1. User `alice` authenticates via SSO, obtaining a session cookie
2. Administrator deletes Alice's account: `pdb.delete_person('alice')`
3. Alice's browser still holds a valid session cookie
4. Alice accesses `/voter` — the `@asfquart.auth.require({R.committer})` decorator validates the existing session token (SSO-issued), not Alice's PersonDB status
5. `basic_info()` reads `uid='alice'` from the session without verifying Alice still exists
6. Alice retains full access until the session expires naturally or the SSO provider independently revokes credentials

**Impact:**
Departing employees, revoked committers, or removed voters retain access to the application after account deletion. They can view election data, potentially vote if their `mayvote` entries were not yet cleaned up, and access administrative functions. This directly violates the requirement that all active sessions are terminated when a user account is disabled or deleted.

**Remediation:**
```python
# In persondb.py - delete_person should accept a session manager
def delete_person(self, pid, session_manager=None):
    "Delete the Person designated by PID and terminate their sessions."
    
    self.c_delete_person.perform(pid)
    
    if self.c_delete_person.rowcount == 0:
        raise PersonNotFound(pid)
    
    # Terminate all active sessions for this user
    if session_manager:
        session_manager.revoke_all_sessions_for_user(pid)

# In a new session management layer:
class SessionManager:
    def revoke_all_sessions_for_user(self, uid):
        """Invalidate all sessions associated with the given user ID."""
        # For server-side sessions: delete from session store by uid
        # For SSO: call SSO provider's session revocation API
        # For self-contained tokens: add uid to a deny-list checked on each request
        pass
```

#### ASVS-742-HIGH-002: No Per-Request Revalidation of User Account Status

**Severity:** HIGH

**Location:** `v3/server/pages.py`, `basic_info()`, Lines 78-92

**Gap Type:** Type B — Control EXISTS (session read) but account status NOT CHECKED (false confidence)

**Vulnerable Code:**
```python
async def basic_info():
    """Return base-level EZT template data."""
    basic = edict()
    # ...
    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
    else:
        basic.update(uid=None, name=None, email=None)
    # ...
    return basic
```

**Data Flow:**
1. `basic_info()` is called at the start of every authenticated handler
2. Session data is read from the session store — `uid`, `fullname`, `email`
3. These values are trusted and used directly for authorization decisions (e.g., `election.q_find_issues.perform(result.uid, election.eid)`)
4. **At no point is PersonDB consulted** to verify the user still exists or is active
5. A deleted user's `uid` from the session is used as if the account is fully active

**Entry Points Affected:**
All 16+ authenticated endpoints use `basic_info()` as the identity source:
- `/voter` (voter_page)
- `/vote-on/<eid>` (vote_on_page)
- `/admin` (admin_page)
- `/manage/<eid>` (manage_page)
- `/do-vote/<eid>` (do_vote_endpoint)
- `/do-create-election` (do_create_endpoint)
- `/do-open/<eid>`, `/do-close/<eid>`
- All `/do-add-issue/`, `/do-edit-issue/`, `/do-delete-issue/` endpoints
- `/profile`, `/settings`
- `/docs/<iid>/<docname>`

**Impact:**
Even if an external mechanism (SSO provider, admin tool) deletes or disables a user, the application itself never cross-references the session identity against the current user database. This creates a window during which a deleted/disabled user retains full application access. The window's duration depends entirely on the session/token expiry policy, which may be hours or days.

**Remediation:**
```python
async def basic_info():
    """Return base-level EZT template data."""
    basic = edict()
    # ...
    s = await asfquart.session.read()
    if s:
        # Verify user still exists and is active
        pdb = steve.persondb.PersonDB.open(DB_FNAME)
        try:
            person = pdb.get_person(s['uid'])
        except steve.persondb.PersonNotFound:
            # User no longer exists - destroy session immediately
            await asfquart.session.destroy()
            basic.update(uid=None, name=None, email=None)
            return basic
        
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
    else:
        basic.update(uid=None, name=None, email=None)
    # ...
    return basic
```

For performance, consider caching the user-existence check with a short TTL (e.g., 60 seconds) rather than hitting the database on every request.

### Medium Severity

#### ASVS-742-MEDIUM-003: No Account Disable/Deactivation Mechanism Exists

**Severity:** MEDIUM

**Location:** `v3/steve/persondb.py`, entire `PersonDB` class, Lines 28-73

**Gap Type:** Type A — No control exists

**Analysis:**
The `PersonDB` class supports only two account lifecycle operations:
- `add_person()` — create/update a user
- `delete_person()` — permanently remove a user (with referential integrity constraints)

There is no `disable_person()` or `deactivate_person()` method, no `is_active` field in the person schema, and no mechanism to temporarily or reversibly revoke access. The `delete_person()` method itself acknowledges this limitation:

```python
### maybe we just don't delete a person, ever?
```

The comment suggests the developers recognize that deletion is problematic due to referential integrity (a person with `mayvote` entries cannot be deleted), leaving **no mechanism at all** to remove access for users who have been associated with elections.

**Impact:**
If a user has participated in any election (has entries in the `mayvote` table), their account **cannot be deleted** due to foreign key constraints, and there is **no disable mechanism**. This means the application has no way to comply with ASVS 7.4.2 for users who have ever been eligible voters — their sessions cannot be terminated through any application-level account lifecycle operation.

**Remediation:**
```python
# Add an is_active field to the person schema
# ALTER TABLE person ADD COLUMN is_active INTEGER DEFAULT 1;

def disable_person(self, pid):
    """Disable a person's account and terminate all their sessions."""
    self.c_disable_person.perform(pid)  # SET is_active = 0 WHERE pid = ?
    if self.c_disable_person.rowcount == 0:
        raise PersonNotFound(pid)
    # Terminate all active sessions for this user
    self._terminate_sessions(pid)

# Then in basic_info() or auth middleware, check is_active
```

## Architecture Observations

1. **SSO Dependency:** The application relies on `asfquart.auth` (ASF SSO) for authentication. Session lifecycle management is split between the SSO provider and the application, but neither side implements session termination on account deletion within this codebase.

2. **No Session Store Access:** The provided code shows no mechanism to enumerate or invalidate sessions by user ID. `asfquart.session.read()` reads the current request's session, but there is no `asfquart.session.revoke_by_uid()` or equivalent API visible.

3. **PersonDB vs. SSO Identity Gap:** The `PersonDB` tracks persons eligible for elections, while authentication is handled externally. A user can be deleted from PersonDB but retain SSO credentials, or vice versa. There is no synchronization between these systems for session lifecycle.

## Recommendations Summary

### Immediate Actions (0-30 days)

1. **Add per-request user status validation** in `basic_info()` — check PersonDB on each request (with short-duration cache) and destroy session if user not found/disabled
   - **Effort:** Medium
   - **Impact:** Closes both ASVS-742-HIGH-001 and ASVS-742-HIGH-002

### Short-term Actions (1-3 months)

2. **Add `is_active` field to person schema** with a `disable_person()` method that sets it to `0` and triggers session revocation
   - **Effort:** Medium
   - **Impact:** Enables account deactivation without deletion; resolves referential integrity blocker (ASVS-742-MEDIUM-003)

3. **Implement session revocation by user ID** — either via server-side session store cleanup or by maintaining a revocation list checked during `basic_info()`
   - **Effort:** Medium-High
   - **Impact:** Provides the underlying mechanism needed for immediate session termination

### Long-term Actions (3-6 months)

4. **Coordinate with SSO provider** — if `asfquart.auth` supports session revocation callbacks or backchannel logout (e.g., OIDC Back-Channel Logout), integrate this for SSO-level session termination
   - **Effort:** High
   - **Impact:** Ensures sessions are terminated even at the SSO layer
