### HIGH Severity

#### ASVS-732-HIGH-001: No Absolute Maximum Session Lifetime

| Field | Value |
|-------|-------|
| **Severity** | **HIGH** |
| **Finding ID** | ASVS-732-HIGH-001 |
| **Gap Type** | Type A — Entry points with NO control |
| **File** | `v3/server/pages.py` |
| **Function** | `basic_info()` (lines 45–68), all authenticated route handlers |
| **Lines** | 45-68, 127, 307, 342, 362, 380 |

**Description:**

The application does not implement any absolute maximum session lifetime mechanism. Sessions remain valid indefinitely as long as they are not explicitly destroyed, creating an unbounded window of opportunity for session hijacking attacks. This is particularly critical in an election/voting system where session integrity directly impacts election outcomes.

**Vulnerable Code:**

```python

    ### generate a real token and store in the session
```

**Data Flow:**

```
Session Cookie (no expiry check) 
  → asfquart.session.read() [pages.py:60]
    → basic_info() returns uid [pages.py:62]
      → uid used as authenticated identity in vote submission [pages.py:307-330]
      → uid used as authenticated identity in election management [pages.py:342-425]
      → NO absolute lifetime check anywhere in this chain
```

**Proof of Concept:**

1. Attacker obtains a session token (via XSS, network sniffing, shared computer, etc.)
2. Days/weeks/months later, the session token remains valid
3. Attacker uses the stale session to:
   - Cast votes on behalf of the victim: `POST /do-vote/<eid>` with stolen session cookie
   - Create elections as the victim: `POST /do-create-election`
   - Open/close elections: `GET /do-open/<eid>`, `GET /do-close/<eid>`

**Impact:**

- **Vote integrity compromise:** A hijacked session allows impersonation for vote casting. Since this is an election system, a single fraudulent vote can alter election outcomes.
- **Extended attack window:** Without absolute session lifetime, the window of opportunity for session hijacking is unbounded. Even if an idle timeout exists in the framework, a persistent attacker can keep the session alive through periodic requests.
- **Election manipulation:** Administrative operations (open/close elections, add/edit/delete issues) can be performed using stale sessions belonging to PMC members.
- **No forced re-authentication:** Users who change passwords or whose accounts are compromised have no mechanism to force session invalidation after a maximum period.

**Remediation:**

```python
# Option 1: Application-layer absolute session lifetime check
import time

MAX_SESSION_LIFETIME = 8 * 60 * 60  # 8 hours absolute maximum

async def basic_info():
    """Return base-level EZT template data."""
    basic = edict()
    basic.flashes = [
        edict(message=f[1], category=f[0])
        for f in quart.get_flashed_messages(with_categories=True)
    ]

    s = await asfquart.session.read()
    if s:
        # Enforce absolute maximum session lifetime
        created_at = s.get('created_at')
        if created_at is None or (time.time() - created_at) > MAX_SESSION_LIFETIME:
            # Session has exceeded maximum lifetime — force re-authentication
            await asfquart.session.destroy()
            quart.abort(quart.redirect('/login?reason=session_expired'))
            
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
    else:
        basic.update(uid=None, name=None, email=None)

    basic.csrf_token = generate_csrf_token()  # Also fix the placeholder
    return basic


# At session creation time (in authentication handler), store timestamp:
async def on_successful_login(user_info):
    session_data = {
        'uid': user_info['uid'],
        'fullname': user_info['fullname'],
        'email': user_info['email'],
        'created_at': time.time(),  # Absolute session start time
    }
    await asfquart.session.write(session_data)
```

## Architecture Observations

1. **Session management is fully delegated to `asfquart`:** The application contains no session configuration, creation, or lifetime management code. All session handling is through `asfquart.session.read()` with no visible configuration of the underlying session store or lifetime parameters.

2. **`basic_info()` is the de facto session gate:** Nearly every authenticated handler calls `basic_info()` to obtain the user identity. This function is the ideal location for implementing session lifetime checks since it's already the centralized entry point.

3. **No session invalidation on election state transitions:** When an election closes or opens, existing sessions for voters/admins are not affected. A stale session from before an election opened could be used after it opens.

4. **Authentication and session lifetime are conflated:** The `@asfquart.auth.require` decorator checks authentication status but not session age. These are distinct security concerns that should be independently enforced.

## Recommendations Summary

### Immediate Actions (HIGH Priority)

| Priority | Recommendation | Effort | Impact |
|----------|---------------|--------|--------|
| **1** | Implement absolute maximum session lifetime of ≤8 hours in `basic_info()` or framework configuration | Medium | Directly addresses ASVS 7.3.2; bounds attack window for session hijacking |
| **2** | Store `created_at` timestamp in session at authentication time | Low | Required foundation for lifetime enforcement |

### Short-term Actions (MEDIUM Priority)

| Priority | Recommendation | Effort | Impact |
|----------|---------------|--------|--------|
| **3** | Implement step-up re-authentication for irreversible operations (`do-open`, `do-close`, `do-vote`) | Medium | Reduces impact of session hijacking on critical operations |
| **4** | Document session lifetime decisions with risk analysis per ASVS requirement | Low | Satisfies the "documented security decisions" aspect of ASVS 7.3.2 |
| **5** | Add server-side session invalidation capability (e.g., session revocation list) | Medium | Enables forced logout on password change or account compromise |

### Long-term Actions

- Implement differentiated session lifetimes based on operation risk level (vote casting: 4h, admin operations: 2h, browsing: 8h)
- Add session monitoring and anomaly detection for suspicious session usage patterns
- Implement session binding to additional factors (IP address, user agent) with appropriate privacy considerations
