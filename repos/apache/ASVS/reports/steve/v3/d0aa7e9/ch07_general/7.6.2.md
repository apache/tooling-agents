**Overview**

The application uses federated authentication via Apache OAuth (IdP). Analysis reveals that the OAuth integration and session creation flow do not enforce explicit user consent or action before creating new application sessions. When a user's application session expires but their IdP session remains active, visiting any protected endpoint triggers an automatic redirect chain that silently re-establishes an application session without user consent.

## Findings

### Medium Severity

#### Finding ASVS-762-MED-001: Session Creation Without User Consent

| Field | Value |
|-------|-------|
| **Severity** | **MEDIUM** |
| **Gap Type** | **Type A** — No control exists to require explicit user consent before session creation |
| **File** | `v3/server/main.py` lines 37-40, `v3/server/pages.py` (all `@asfquart.auth.require` routes) |
| **ASVS Requirement** | 7.6.2 — Session creation must require user consent or explicit action |

**Vulnerable Code:**

In `main.py:create_app()`:
```python
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

In `pages.py`, every protected route auto-triggers the OAuth flow:
```python
@APP.get('/voter')
@asfquart.auth.require({R.committer})  # Redirects to IdP if no session
async def voter_page():
    ...

@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  # Redirects to IdP if no session
@load_election
async def do_open_endpoint(election):
    ...
```

**Related Functions Checked:**
- `main.py:create_app()` (L37-40) — OAuth URL configuration
- `pages.py:voter_page()` (L136-165) — Protected route with auto-redirect
- `pages.py:do_open_endpoint()` (L437-453) — State-changing GET with auto-redirect
- `pages.py:do_close_endpoint()` (L456-472) — State-changing GET with auto-redirect
- `pages.py:basic_info()` (L65-78) — Session reading (no creation)

**Description:**

The application does not enforce explicit user consent or action before creating new application sessions. When a user's application session expires but their IdP session remains active, visiting any protected endpoint triggers an automatic redirect chain that silently re-establishes an application session without user interaction.

**Data Flow:**

```
User (no app session) → navigates to ANY protected endpoint
  → @asfquart.auth.require detects no session
  → HTTP 302 Redirect to https://oauth.apache.org/auth?state=X&redirect_uri=Y
     (NO prompt=login, NO prompt=consent, NO max_age parameter)
  → IdP has active session → IdP issues assertion silently (passive authentication)
  → HTTP 302 Redirect back to application callback
  → Application creates new session WITHOUT explicit user consent
  → Original protected endpoint executes
```

**Proof of Concept:**

1. User authenticates with the application, creating both an IdP session and an RP (application) session
2. Application session expires (timeout)
3. User clicks any link to a protected page — or an attacker crafts a link to a state-changing GET endpoint like `/do-open/<eid>`
4. The `@asfquart.auth.require` decorator redirects to the IdP
5. Since the IdP session is still active, the IdP silently authenticates and redirects back
6. A new application session is created without the user ever seeing a login screen or consent prompt
7. The original request (including state-changing GET operations) executes immediately

This is particularly dangerous with the state-changing GET endpoints:
```
Attacker sends email with: <a href="https://steve.example.org/do-close/TARGET_EID">Click here</a>
→ Victim clicks link while IdP session active but app session expired
→ Silent re-authentication creates new app session
→ Election gets closed as a side effect
```

**Impact:**

- Violates NIST SP 800-63C guidance on passive authentication at the IdP: sessions can be silently established without the subscriber's explicit awareness or consent
- Application session timeout policies become ineffective — any expired session is automatically re-created through the redirect chain
- Enables attacks where third-party content (links, image tags for GET endpoints) can trigger session creation and state-changing operations in a single redirect chain
- Users have no opportunity to refuse session creation after their previous session expired

**Remediation:**

1. **Add `prompt=login` or `prompt=consent` to the OAuth initiation URL** to force explicit user interaction at the IdP:

```python
# main.py - Force explicit user interaction during re-authentication
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s&prompt=login'
)
```

2. **Implement an interstitial login page** instead of auto-redirecting to the IdP. When `@asfquart.auth.require` detects no session, render a page with a "Sign In" button rather than auto-redirecting:

```python
# Instead of auto-redirect in the auth decorator:
@APP.get('/login')
@APP.use_template(TEMPLATES / 'login.ezt')
async def login_page():
    """Explicit login page requiring user action to initiate OAuth flow."""
    result = await basic_info()
    result.title = 'Sign In'
    result.return_url = quart.request.args.get('next', '/voter')
    return result

# In login.ezt template:
# <form method="POST" action="/auth/begin">
#     <input type="hidden" name="next" value="[return_url]">
#     <button type="submit" class="btn btn-primary">Sign in with Apache OAuth</button>
# </form>
```

3. **Add `max_age` parameter** to limit how recently the user must have authenticated at the IdP:

```python
# Force re-authentication if IdP session is older than 300 seconds
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s&max_age=300'
)
```

## Architecture Observations

1. **Federated Authentication via Apache OAuth**: The application relies on `asfquart` framework for OAuth integration, meaning session creation logic is largely in the framework. The application only configures OAuth URLs and applies auth decorators.

2. **No Explicit Login Endpoint**: The codebase contains no dedicated `/login` route. Authentication is triggered implicitly by the `@asfquart.auth.require` decorator on any protected route, meaning there is no explicit "sign in" action boundary.

3. **State-Changing GET Endpoints**: `/do-open/<eid>` and `/do-close/<eid>` perform state changes via GET requests. Combined with automatic session creation through the OAuth redirect chain, these endpoints are particularly susceptible to attacks where a crafted link triggers both session creation and the state change.

4. **Framework Dependency**: The session creation, OAuth flow management, and callback handling are all within the `asfquart` framework (not provided). The audit can only assess the configuration surface exposed in `main.py`.
