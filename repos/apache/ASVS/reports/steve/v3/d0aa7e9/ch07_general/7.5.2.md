## Findings

### HIGH Severity

#### ASVS-752-HIGH-001: Complete Absence of Active Session Viewing Capability

| Attribute | Detail |
|-----------|--------|
| **Severity** | HIGH |
| **Gap Type** | Type A — Entry point with NO control |
| **File** | `v3/server/pages.py` |
| **Function** | `profile_page()`, `settings_page()` (lines ~537-549) |
| **Related Functions** | `basic_info()` (lines ~68-78) |
| **ASVS Requirement** | 7.5.2 — Users must be able to view all currently active sessions |

**Description:**

The application defines two user-facing account pages (`/profile` and `/settings`) that could host session management functionality, but neither provides the ability to view active sessions. Users cannot see a list of their currently active sessions, including device information, IP addresses, last activity times, or creation timestamps.

**Vulnerable Code:**

```python
# v3/server/pages.py, lines ~537-549
@APP.get('/profile')
@asfquart.auth.require  # Bare decorator means just require a valid session
@APP.use_template(TEMPLATES / 'profile.ezt')
async def profile_page():
    result = await basic_info()
    result.title = 'Profile'
    return result


@APP.get('/settings')
@asfquart.auth.require  # Bare decorator means just require a valid session
@APP.use_template(TEMPLATES / 'settings.ezt')
async def settings_page():
    result = await basic_info()
    result.title = 'Settings'
    return result
```

The only session interaction is reading the **current** session for display purposes:

```python
# v3/server/pages.py, lines ~68-78
async def basic_info():
    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
```

**Data Flow:**

1. User accesses `/profile` or `/settings`
2. `basic_info()` reads only the current session data
3. No retrieval of other active sessions for the user
4. No session list displayed to user

**Proof of Concept:**

A comprehensive audit of every route in `pages.py` confirms no session listing capability exists anywhere in the application. No route provides a list of active sessions with metadata such as device, IP address, last activity, or creation time.

**Impact:**

Users cannot detect if an attacker has hijacked or is actively abusing one of their sessions. In a voting system, this means a compromised session could be used to cast unauthorized votes without the legitimate user's knowledge. This represents a complete failure to meet ASVS 7.5.2's viewing requirement.

**Remediation:**

Add a session listing endpoint that shows all active sessions for the authenticated user:

```python
@APP.get('/sessions')
@asfquart.auth.require
@APP.use_template(TEMPLATES / 'sessions.ezt')
async def sessions_page():
    result = await basic_info()
    result.title = 'Active Sessions'
    
    # Retrieve all sessions for the current user
    sessions = await session_store.get_all_sessions_for_user(result.uid)
    result.sessions = [
        edict(
            session_id=s.id,
            created_at=s.created_at,
            last_active=s.last_active,
            ip_address=s.ip_address,
            user_agent=s.user_agent,
            is_current=s.id == current_session_id,
        )
        for s in sessions
    ]
    return result
```

This requires implementing a server-side session store that tracks sessions per user (not just opaque cookies).

#### ASVS-752-HIGH-002: Complete Absence of Session Termination Capability

| Attribute | Detail |
|-----------|--------|
| **Severity** | HIGH |
| **Gap Type** | Type A — Entry point with NO control |
| **File** | `v3/server/pages.py` |
| **Function** | N/A (functionality does not exist) |
| **Related Functions** | All routes examined |
| **ASVS Requirement** | 7.5.2 — Users must be able to terminate any or all active sessions |

**Description:**

A full text search of all routes in `pages.py` reveals no endpoint for terminating sessions. There is no capability to:
- Terminate a specific session by ID
- Terminate all sessions except the current one
- Log out from the current session

**Vulnerable Code:**

No logout endpoint exists in the entire application. The complete route table from `pages.py` shows 21 defined routes, none of which provide session termination:

```python
# All defined routes in pages.py - NO logout, session terminate, or session revoke endpoint exists
@APP.get('/')                              # home
@APP.get('/voter')                         # voter dashboard
@APP.get('/vote-on/<eid>')                 # vote page
@APP.get('/admin')                         # admin dashboard
@APP.get('/manage/<eid>')                  # manage election
@APP.get('/manage-stv/<eid>/<iid>')        # manage STV issue
@APP.post('/do-set-open_at/<eid>')         # set open date
@APP.post('/do-set-close_at/<eid>')        # set close date
@APP.post('/do-vote/<eid>')                # submit vote
@APP.post('/do-create-election')           # create election
@APP.get('/do-open/<eid>')                 # open election
@APP.get('/do-close/<eid>')                # close election
@APP.post('/do-add-issue/<eid>')           # add issue
@APP.post('/do-edit-issue/<eid>/<iid>')    # edit issue
@APP.post('/do-delete-issue/<eid>/<iid>')  # delete issue
@APP.get('/profile')                       # profile page
@APP.get('/settings')                      # settings page
@APP.get('/about')                         # about page
@APP.get('/docs/<iid>/<docname>')          # document serving
@APP.get('/static/<path:filename>')        # static files
@APP.get('/favicon.ico')                   # favicon
```

**Data Flow:**

Once a session is established, there is no user-initiated way to end it. The only session expiration would be server-side timeout (if configured in the framework).

**Proof of Concept:**

1. User authenticates and receives session cookie
2. Attacker obtains session cookie (e.g., shared computer, session fixation, XSS)
3. User notices suspicious activity but has no way to invalidate the stolen session
4. Attacker uses the session to vote on the user's behalf via `POST /do-vote/<eid>`
5. User cannot terminate the compromised session or log out their own session

**Impact:**

If a user's session token is stolen (e.g., through XSS, network interception, or shared device), the user has no mechanism to:
1. Discover the compromised session exists
2. Revoke the compromised session
3. Revoke all sessions as a defensive measure
4. Even log out from their own current session

In a voting system, this is particularly dangerous as an attacker could maintain persistent access to cast or modify votes. This represents a complete failure to meet ASVS 7.5.2's termination requirement.

**Remediation:**

Implement session termination endpoints with proper re-authentication:

```python
@APP.post('/sessions/terminate/<session_id>')
@asfquart.auth.require
async def terminate_session(session_id):
    result = await basic_info()
    
    # Re-authenticate the user (at least one factor)
    # This could be a password confirmation or OAuth re-auth
    form = await quart.request.form
    if not await verify_reauthentication(result.uid, form.get('password')):
        await flash_danger('Re-authentication required to terminate sessions.')
        return quart.redirect('/sessions', code=303)
    
    # Terminate the specified session
    await session_store.revoke_session(result.uid, session_id)
    _LOGGER.info(f'User[U:{result.uid}] terminated session {session_id}')
    await flash_success('Session terminated.')
    return quart.redirect('/sessions', code=303)


@APP.post('/sessions/terminate-all')
@asfquart.auth.require
async def terminate_all_sessions():
    result = await basic_info()
    
    # Re-authenticate
    form = await quart.request.form
    if not await verify_reauthentication(result.uid, form.get('password')):
        await flash_danger('Re-authentication required.')
        return quart.redirect('/sessions', code=303)
    
    current_session = await asfquart.session.read()
    await session_store.revoke_all_sessions(result.uid, except_session=current_session.id)
    _LOGGER.info(f'User[U:{result.uid}] terminated all other sessions')
    await flash_success('All other sessions terminated.')
    return quart.redirect('/sessions', code=303)


@APP.post('/logout')
@asfquart.auth.require
async def logout():
    await asfquart.session.destroy()
    return quart.redirect('/', code=303)
```

### MEDIUM Severity

#### ASVS-752-MEDIUM-003: No Re-authentication Mechanism for Sensitive Session Operations

| Attribute | Detail |
|-----------|--------|
| **Severity** | MEDIUM |
| **Gap Type** | Type A — Entry point with NO control |
| **File** | `v3/server/pages.py` |
| **Function** | All authenticated routes |
| **Related Functions** | `@asfquart.auth.require` decorator usage throughout |
| **ASVS Requirement** | 7.5.2 — Re-authentication with at least one factor before session termination |

**Description:**

ASVS 7.5.2 explicitly requires that users re-authenticate with at least one factor before terminating sessions. The codebase has no re-authentication mechanism at all for sensitive operations.

**Vulnerable Code:**

The only authentication checks are binary "is authenticated" gates:

```python
# The only auth checks are binary "is authenticated" gates:
@asfquart.auth.require({R.committer})  # initial login only
@asfquart.auth.require                 # bare session check only
```

There is no implementation of:
- Password/credential verification endpoint
- Step-up authentication flow
- OAuth re-authentication redirect
- Session-age check requiring recent authentication

**Data Flow:**

Authentication is entirely delegated to `asfquart.auth` which only provides initial login gates. Once authenticated, there is no mechanism to require re-authentication for sensitive operations.

**Proof of Concept:**

Even if session termination were implemented, without re-authentication:
1. Attacker obtains a stolen session token
2. Attacker accesses hypothetical session management page
3. Attacker terminates all of the legitimate user's other sessions
4. Legitimate user is locked out while attacker maintains access
5. Security control is inverted and works against the legitimate user

**Impact:**

Without re-authentication, an attacker who has a stolen session could use it to terminate the legitimate user's other sessions (locking the real user out), which inverts the security control. This prevents secure implementation of ASVS 7.5.2's termination requirement.

**Remediation:**

Implement a re-authentication flow that must be passed before session management operations are permitted:

```python
async def verify_reauthentication(uid, password):
    """Verify user's password for step-up authentication."""
    # Option 1: Verify password directly
    if password:
        return await verify_user_password(uid, password)
    
    # Option 2: Check if user authenticated recently (e.g., within last 5 minutes)
    session = await asfquart.session.read()
    if session.get('last_auth_time'):
        time_since_auth = time.time() - session['last_auth_time']
        if time_since_auth < 300:  # 5 minutes
            return True
    
    return False


async def require_recent_auth(max_age_seconds=300):
    """Decorator to require recent authentication."""
    session = await asfquart.session.read()
    last_auth = session.get('last_auth_time', 0)
    if time.time() - last_auth > max_age_seconds:
        # Redirect to re-authentication page
        return quart.redirect('/reauth?return_to=' + quart.request.path)
    return None
```

Apply this to all session management endpoints before allowing termination operations.

## Architecture Observations

1. **Session architecture is opaque**: The session system (`asfquart.session`) appears to be a thin wrapper providing read-only access to session data via `asfquart.session.read()` (line ~68). There is no visible session store that tracks multiple sessions per user, which is a prerequisite for implementing ASVS 7.5.2. The architecture would need to be extended to support server-side session enumeration and management.

2. **External authentication dependency**: Authentication is delegated to ASF OAuth (`asfquart.auth`). Session termination and listing would need to be implemented at the application level since the external system only handles initial authentication. This creates an architectural gap where session lifecycle management is not addressed.

3. **No logout anywhere**: The absence of even a basic logout endpoint suggests session lifecycle management was not part of the original design scope. This is a fundamental gap in the application's security architecture.

4. **Voting system criticality**: As a voting application, session security is particularly critical. Compromised sessions could lead to unauthorized vote manipulation, making the absence of session management capabilities a significant risk to election integrity.
