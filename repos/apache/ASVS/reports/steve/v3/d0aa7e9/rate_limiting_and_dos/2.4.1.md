## Findings

### High Severity

#### ASVS-241-HIGH-001: No Throttling or Timing Enforcement on Vote Submission Endpoint

**Severity:** HIGH

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint`
- **Lines:** 426–470

**Related Functions Checked:**
- `do_vote_endpoint` (lines 426–470)
- `election.add_vote()` (called but not visible in provided files)

**Description:**

The vote submission endpoint has no rate limiting, timing checks, or cooldown periods. A compromised authenticated account or malicious insider can submit automated votes at machine speed with no human-interaction timing verification.

**Vulnerable Code:**
```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    ### check authz

    # Parse the form data
    form = edict(await quart.request.form)
    if not form:
        await flash_danger('No vote data provided.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    # Extract votes from form (keys like 'vote-<iid>')
    votes = {}
    for key, value in form.items():
        if key.startswith('vote-'):
            iid = key.split('-', 1)[1]
            votes[iid] = value

    # Process each vote
    for iid, votestring in votes.items():
        # ...
        election.add_vote(result.uid, iid, votestring)
```

**Data Flow:**
Authenticated user → `POST /do-vote/<eid>` → direct call to `election.add_vote()` → No rate limit, no timing check, no cooldown

**Gap Type:** Type A — No anti-automation control exists for this critical endpoint.

**Proof of Concept:**
```python
import requests

session = requests.Session()
# After authenticating once:
for i in range(10000):
    session.post(
        "https://target/do-vote/ELECTION_ID",
        data={"vote-ISSUE1": "yes", "vote-ISSUE2": "no"},
        cookies=auth_cookies
    )
    # No delay needed — no throttling or timing check exists
```

**Impact:**
- A compromised authenticated account (or malicious insider) can submit automated votes at machine speed with no human-interaction timing verification.
- Enables rapid vote-change cycling that could interfere with tallying if done during a race condition window.
- Generates excessive database write operations (one per issue per request), creating denial-of-service conditions on the SQLite database through write lock contention.
- The domain context explicitly calls for "timing requirements enforce human-like interaction speeds for voting operations" and "rate limiting on vote submission endpoints (per user, per IP)" — both are absent.

**Remediation:**
```python
import time
from collections import defaultdict
from functools import wraps

# Simple per-user rate limiter
_vote_timestamps = defaultdict(list)
VOTE_RATE_LIMIT = 5         # max submissions
VOTE_RATE_WINDOW = 60       # per 60 seconds
VOTE_MIN_DELAY_SECS = 3     # minimum time between page load and vote

def rate_limit_votes(func):
    @wraps(func)
    async def wrapper(*args, **kwargs):
        s = await asfquart.session.read()
        uid = s['uid']
        now = time.monotonic()

        # Clean old entries
        _vote_timestamps[uid] = [
            t for t in _vote_timestamps[uid]
            if now - t < VOTE_RATE_WINDOW
        ]

        if len(_vote_timestamps[uid]) >= VOTE_RATE_LIMIT:
            _LOGGER.warning(f'Rate limit exceeded for User[U:{uid}] on vote submission')
            quart.abort(429, 'Too many vote submissions. Please wait.')

        _vote_timestamps[uid].append(now)
        return await func(*args, **kwargs)
    return wrapper

@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@rate_limit_votes          # <-- Add rate limiting
@load_election
async def do_vote_endpoint(election):
    # ... existing logic
```

### Medium Severity

#### ASVS-241-MEDIUM-002: No Rate Limiting on Election Creation Endpoint

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_create_endpoint`
- **Lines:** 473–490

**Related Functions Checked:**
- `do_create_endpoint` (lines 473–490)
- `Election.create()` (called but not visible in provided files)

**Description:**

The election creation endpoint lacks rate limiting and quota controls. A compromised PMC member account can create unbounded elections, causing database bloat, garbage-data creation, and quota exhaustion.

**Vulnerable Code:**
```python
@APP.post('/do-create-election')
@asfquart.auth.require({R.pmc_member})  ### need general solution
async def do_create_endpoint():
    "Create a new Election."

    result = await basic_info()

    ### check authz

    form = edict(await quart.request.form)

    # Create the Election.
    election = steve.election.Election.create(DB_FNAME, form.title, result.uid)
```

**Data Flow:**
Authenticated PMC member → `POST /do-create-election` → `Election.create()` writes to database → No rate limit or quota

**Gap Type:** Type A — No control exists.

**Proof of Concept:**
```python
for i in range(100000):
    session.post(
        "https://target/do-create-election",
        data={"title": f"Spam Election {i}"},
        cookies=auth_cookies
    )
```

**Impact:**
- A compromised PMC member account can create unbounded elections, causing database bloat, garbage-data creation, and quota exhaustion.
- Each election creates cryptographic keys (per `steve.crypto`), consuming CPU resources for key derivation.
- The SQLite database has no inherent size limits — unchecked creation leads to disk exhaustion on the server.

**Remediation:**
```python
# Add a per-user creation quota
MAX_ELECTIONS_PER_USER = 50  # configurable

@APP.post('/do-create-election')
@asfquart.auth.require({R.pmc_member})
async def do_create_endpoint():
    result = await basic_info()

    # Check quota
    owned = steve.election.Election.owned_elections(DB_FNAME, result.uid)
    if len(owned) >= MAX_ELECTIONS_PER_USER:
        await flash_danger(f'Election quota reached ({MAX_ELECTIONS_PER_USER}).')
        return quart.redirect('/admin', code=303)

    form = edict(await quart.request.form)
    election = steve.election.Election.create(DB_FNAME, form.title, result.uid)
    # ...
```

#### ASVS-241-MEDIUM-003: No Limits on Election Size (Issues per Election)

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_add_issue_endpoint`
- **Lines:** 523–545

**Related Functions Checked:**
- `do_add_issue_endpoint` (lines 523–545)
- `election.add_issue()` (called but not visible in provided files)
- `election.list_issues()` (referenced in impact analysis)

**Description:**

The issue creation endpoint has no limits on the number of issues per election and no rate limiting. An election with unbounded issues causes resource exhaustion during voting page load, tallying operations, and vote submission.

**Vulnerable Code:**
```python
@APP.post('/do-add-issue/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_add_issue_endpoint(election):
    result = await basic_info()

    ### check authz

    form = edict(await quart.request.form)
    print('FORM:', form)

    ### do better with these
    vtype = 'yna'
    kv = None

    iid = election.add_issue(form.title, form.description, vtype, kv)
```

**Data Flow:**
Authenticated user → `POST /do-add-issue/<eid>` → `election.add_issue()` → No limit on issue count per election, no rate limiting

**Gap Type:** Type A — No control exists.

**Proof of Concept:**
```python
for i in range(1000000):
    session.post(
        f"https://target/do-add-issue/{election_id}",
        data={"title": f"Issue {i}", "description": f"Desc {i}"},
        cookies=auth_cookies
    )
```

**Impact:**
- An election with unbounded issues causes resource exhaustion during:
  - **Voting page load** (`vote_on_page`): `election.list_issues()` fetches all issues, `random.shuffle()` runs per STV issue, and each issue is processed for template rendering.
  - **Tallying operations**: Each issue requires vote decryption and counting — a million-issue election would make tallying computationally infeasible.
  - **Vote submission**: The `do_vote_endpoint` iterates over all submitted votes with database writes per issue, causing extended write locks.
- The domain context explicitly asks to "Check if there are limits on election size (number of issues, candidates, voters) to prevent resource exhaustion."

**Remediation:**
```python
MAX_ISSUES_PER_ELECTION = 100  # configurable per election type

@APP.post('/do-add-issue/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_add_issue_endpoint(election):
    result = await basic_info()

    # Enforce issue count limit
    current_issues = election.list_issues()
    if len(current_issues) >= MAX_ISSUES_PER_ELECTION:
        await flash_danger(
            f'Maximum issues ({MAX_ISSUES_PER_ELECTION}) reached for this election.'
        )
        return quart.redirect(f'/manage/{election.eid}', code=303)

    form = edict(await quart.request.form)
    # ... rest of logic
```

## Positive Security Patterns

1. **Authentication on all state-changing endpoints**: Every mutating endpoint uses `@asfquart.auth.require` with appropriate requirement levels (`R.committer` or `R.pmc_member`).

2. **Elevated privilege for election creation**: `POST /do-create-election` requires `R.pmc_member`, a higher privilege level than general committer access, limiting the attack surface for election spam.

3. **Fine-grained authorization for document access**: `serve_doc()` checks `mayvote` entries before serving documents, implementing per-resource authorization beyond just authentication.

4. **Candidate randomization**: `random.shuffle(issue.candidates)` in `vote_on_page` prevents ballot ordering bias, showing attention to election integrity.

5. **Structured logging of sensitive operations**: All state-changing operations log the user ID, action, and target resource, providing an audit trail.

## Architecture Observations

1. **No middleware layer for cross-cutting concerns**: The `create_app()` function in `main.py` creates the app and imports endpoint modules, but never registers any middleware for rate limiting, request counting, or abuse detection. This is the root cause — anti-automation should be a cross-cutting concern.

2. **Quart/ASGI architecture**: The application uses Quart (async Flask) with optional Hypercorn. Neither framework provides built-in rate limiting. Libraries like `quart-rate-limiter` or reverse proxy configuration (nginx `limit_req`) would be needed.

3. **SQLite backend amplifies DoS risk**: SQLite's single-writer model means that rapid automated write requests (vote submission, election/issue creation) will serialize at the database level, creating a bottleneck that amplifies denial-of-service impact for all users.

4. **OAuth delegation for authentication**: Authentication is delegated to `oauth.apache.org`, which likely provides its own brute-force protection. However, session token replay after initial auth would bypass any OAuth-level throttling.
