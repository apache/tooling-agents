## Findings

### HIGH SEVERITY

#### ASVS-242-HIGH-001: Vote Submission Endpoint Has No Human Timing Enforcement

**Location**
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint`
- **Lines:** ~412–460

**Related Functions Checked**
- `do_vote_endpoint` (vote submission handler)
- `vote_on_page` (ballot display handler)
- `basic_info` (user context retrieval)

**Description**

The vote submission endpoint processes votes immediately upon receiving POST requests with no rate limiting, cooldown period, or minimum dwell-time enforcement. An authenticated user can submit or modify votes at machine speed, enabling automated vote manipulation that violates the ASVS 2.4.2 requirement for realistic human timing in business logic flows.

**Vulnerable Code**

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    ### check authz

    # Parse the form data
    form = edict(await quart.request.form)
    if not form:
        await flash_danger('No vote data provided.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    # Extract votes from form (keys like 'vote-<iid>')
    votes = {}
    for key, value in form.items():
        if key.startswith('vote-'):
            iid = key.split('-', 1)[1]
            votes[iid] = value

    # Process each vote
    for iid, votestring in votes.items():
        # ...
        election.add_vote(result.uid, iid, votestring)
```

**Data Flow**

```
HTTP POST /do-vote/<eid>
    ↓
Form data parsed
    ↓
election.add_vote() called immediately
    ↓
No timing check, no cooldown, no rate limit
```

**Proof of Concept**

```python
import httpx, time

# Automated script: vote, change vote, vote again - all in < 1 second
session = httpx.Client(cookies=stolen_or_valid_session_cookie)

for i in range(100):
    # Submit or change votes as fast as possible
    session.post(f"https://target/do-vote/{eid}", data={
        "vote-issue1": "yes",
        "vote-issue2": "no",
        "vote-issue3": "1,2,3",  # STV ranking
    })
    # ~10ms per request = 100 vote changes in 1 second
```

An automated script could:
1. Load `/vote-on/<eid>` to scrape issue IDs (no dwell-time check)
2. Immediately POST votes for all issues in a single request
3. Repeat to change votes hundreds of times per second
4. In an STV election, rapidly cycle through different ranking permutations to probe or manipulate outcomes

**Impact**

- Enables automated vote manipulation at machine speed
- An authenticated but malicious user (or compromised session) can submit/change votes faster than any human could read the ballot
- In time-sensitive elections, an attacker could repeatedly change their vote right up to closing time, potentially exploiting race conditions in the close logic
- Violates the ASVS 2.4.2 requirement that business logic flows require realistic human timing

**Remediation**

```python
import time
from quart import session as quart_session

# Minimum seconds between vote submissions per user
VOTE_COOLDOWN_SECONDS = 10
# Minimum seconds between page load and first vote
MIN_BALLOT_DWELL_SECONDS = 5

@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    now = time.monotonic()
    
    # Check cooldown since last vote submission
    last_vote_time = quart_session.get('last_vote_time', 0)
    if now - last_vote_time < VOTE_COOLDOWN_SECONDS:
        await flash_warning('Please wait before submitting again.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
    
    # Check dwell time (ballot must have been viewed for minimum time)
    ballot_loaded_at = quart_session.get(f'ballot_loaded_{election.eid}', 0)
    if now - ballot_loaded_at < MIN_BALLOT_DWELL_SECONDS:
        await flash_warning('Please take time to review the ballot.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
    
    # ... existing vote processing ...
    
    # Record submission time
    quart_session['last_vote_time'] = now
```

Also add timestamp tracking on ballot load:
```python
@APP.get('/vote-on/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def vote_on_page(election):
    # ... existing logic ...
    quart_session[f'ballot_loaded_{election.eid}'] = time.monotonic()
```

### MEDIUM SEVERITY

#### ASVS-242-MED-001: Election Creation Endpoint Has No Rate Limiting or Timing Control

**Location**
- **File:** `v3/server/pages.py`
- **Function:** `do_create_endpoint`
- **Lines:** ~463–482

**Related Functions Checked**
- `do_create_endpoint` (election creation handler)
- `Election.create` (election creation method)
- `basic_info` (user context retrieval)

**Description**

The election creation endpoint processes requests immediately without any rate limiting, cooldown period, or maximum count restrictions. A compromised PMC member account could flood the system with elections, causing database bloat and resource exhaustion.

**Vulnerable Code**

```python
@APP.post('/do-create-election')
@asfquart.auth.require({R.pmc_member})  ### need general solution
async def do_create_endpoint():
    "Create a new Election."
    result = await basic_info()
    form = edict(await quart.request.form)
    # Create the Election.
    election = steve.election.Election.create(DB_FNAME, form.title, result.uid)
    # ... redirect ...
```

**Data Flow**

```
HTTP POST /do-create-election
    ↓
form.title extracted
    ↓
Election.create() called immediately
    ↓
No rate limit, no cooldown, no maximum count check
```

**Proof of Concept**

```python
import httpx

session = httpx.Client(cookies=pmc_member_session)
# Create 1000 elections in seconds — resource exhaustion
for i in range(1000):
    session.post("https://target/do-create-election", data={
        "title": f"Spam Election {i}"
    })
```

**Impact**

- A compromised PMC member account could flood the system with elections
- Database bloat and resource exhaustion (SQLite write contention)
- Degrades the voter experience by polluting the election list
- Domain context explicitly lists "election creation spam" as a DoS risk

**Remediation**

```python
MAX_ELECTIONS_PER_USER_PER_DAY = 5
ELECTION_CREATE_COOLDOWN = 30  # seconds

@APP.post('/do-create-election')
@asfquart.auth.require({R.pmc_member})
async def do_create_endpoint():
    result = await basic_info()
    
    # Enforce creation cooldown
    now = time.monotonic()
    last_create = quart_session.get('last_election_create', 0)
    if now - last_create < ELECTION_CREATE_COOLDOWN:
        await flash_warning('Please wait before creating another election.')
        return quart.redirect('/admin', code=303)
    
    # Enforce daily limit
    daily_count = steve.election.Election.count_created_today(DB_FNAME, result.uid)
    if daily_count >= MAX_ELECTIONS_PER_USER_PER_DAY:
        await flash_danger('Daily election creation limit reached.')
        return quart.redirect('/admin', code=303)
    
    # ... existing creation logic ...
    quart_session['last_election_create'] = now
```

#### ASVS-242-MED-002: Election State-Change Endpoints Lack Timing Controls and Use GET for Mutations

**Location**
- **File:** `v3/server/pages.py`
- **Functions:** `do_open_endpoint`, `do_close_endpoint`
- **Lines:** ~485–504 (open), ~507–523 (close)

**Related Functions Checked**
- `do_open_endpoint` (election open handler)
- `do_close_endpoint` (election close handler)
- `election.open` (state change method)
- `election.close` (state change method)

**Description**

The election state-change endpoints (open/close) execute immediately upon GET requests with no timing controls, confirmation steps, or cooldowns. An election could be rapidly toggled between open and closed states, disrupting active voters. Additionally, the use of GET methods for state-changing operations violates HTTP semantics.

**Vulnerable Code**

```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    # ...

@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz
    election.close()
    # ...
```

**Data Flow**

```
HTTP GET /do-open/<eid> or /do-close/<eid>
    ↓
State change executed immediately
    ↓
No timing control, no confirmation step, no cooldown
```

**Proof of Concept**

```python
# Rapidly toggle election open/close state
import httpx
session = httpx.Client(cookies=valid_session)
for _ in range(50):
    session.get(f"https://target/do-open/{eid}")
    session.get(f"https://target/do-close/{eid}")
# 100 state transitions in seconds — disrupts active election
```

**Impact**

- An election could be rapidly opened and closed, disrupting voters mid-ballot
- Combined with the lack of owner-only authorization (`### check authz` is commented out), any authenticated committer can toggle any election's state at machine speed
- No human-paced interaction is required for critical election lifecycle operations

**Remediation**

```python
# 1. Change to POST methods
# 2. Add confirmation step / cooldown
# 3. Add owner authorization check

ELECTION_STATE_COOLDOWN = 60  # seconds

@APP.post('/do-open/<eid>')  # Changed from GET to POST
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    
    # Verify ownership
    metadata = election.get_metadata()
    if metadata.owner_pid != result.uid:
        quart.abort(403, 'Only the election owner may open it.')
    
    # Enforce cooldown on state changes
    cooldown_key = f'election_state_{election.eid}'
    last_change = quart_session.get(cooldown_key, 0)
    if time.monotonic() - last_change < ELECTION_STATE_COOLDOWN:
        await flash_warning('Please wait before changing election state again.')
        return quart.redirect(f'/manage/{election.eid}', code=303)
    
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    quart_session[cooldown_key] = time.monotonic()
```

## Positive Security Patterns

1. **Authentication is consistently applied**: All state-changing endpoints use `@asfquart.auth.require` with appropriate role requirements (`R.committer`, `R.pmc_member`).

2. **Comprehensive logging**: All sensitive operations log the acting user ID, election ID, and action taken — valuable for post-incident investigation.

3. **Voter eligibility check**: The vote-on page checks `q_find_issues` to verify the user has `mayvote` entries before rendering the ballot.

4. **STV candidate randomization**: `random.shuffle(issue.candidates)` prevents position bias in ballot presentation.

---

## Architecture Observations

1. **No anti-automation framework exists at any layer.** The Quart application has no rate-limiting middleware, no per-route throttling decorators, and no session-based timing state. This is a systemic gap, not an oversight on individual endpoints.

2. **The `basic_info()` function is called in every handler** but does not record or check request timing — this would be a natural integration point for timing enforcement.

3. **Session infrastructure exists** (`asfquart.session.read()`) and could support timing state storage (e.g., `last_vote_time`, `ballot_loaded_at`), but this capability is unused.

4. **The `### check authz` comments** throughout the codebase indicate that authorization beyond authentication is not yet implemented, compounding the anti-automation gap — any committer can access any election's state-change endpoints at any speed.
