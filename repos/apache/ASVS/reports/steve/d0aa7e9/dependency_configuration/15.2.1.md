## Findings

### Critical Severity

#### ASVS-1521-CRI-001: Complete Absence of SBOM and Dependency Version Tracking

**Severity:** CRITICAL  
**Gap Type:** Type A — No control exists for a foundational requirement  
**CWE:** CWE-1104 (Use of Unmaintained Third Party Components)

**Location:**
- **File:** Project-wide
- **Evidence:** No `pyproject.toml`, `requirements.txt`, `uv.lock`, `Pipfile.lock`, or SBOM document found
- **Related Files:** 
  - `v3/server/main.py` (line 1)
  - `v3/steve/crypto.py` (lines 1–10)
  - `v3/steve/election.py` (lines 1–10)

**Related Functions Checked:**
- All import statements across `main.py`, `crypto.py`, `election.py`
- Package manager configuration (not found)
- CI/CD dependency scanning (not found)

**Description:**

The application lacks any mechanism to track, version, or audit third-party dependencies. While `main.py` declares a `uv`-based script runner, no dependency manifest with version constraints exists.

**Vulnerable Code:**

```python
# main.py (line 1)
#!/usr/bin/env -S uv run --script
# No inline script metadata block (PEP 723) present
# No pyproject.toml or requirements.txt found

# crypto.py (lines 1-10)
import argon2.low_level                          # No version constraint
import cryptography.fernet                        # No version constraint
from cryptography.hazmat.primitives.kdf import hkdf  # No version constraint

# election.py (lines 1-10)
import asfpy.db                                   # No version constraint
from easydict import EasyDict as edict            # No version constraint

# main.py (dependencies)
import asfquart                                   # No version constraint
```

**Data Flow:**
1. **Source:** Unversioned dependency declarations across all modules
2. **Resolution:** `uv run --script` resolves dependencies at install time without constraints
3. **Sink:** Production runtime environment with unknown component versions
4. **Missing Control:** No SBOM, no version pinning, no lock file, no documented remediation timeframes

**Why CRITICAL:**

1. **Cryptographic library currency is unverifiable.** The application's entire security model depends on `cryptography` (Fernet encryption of votes) and `argon2-cffi` (key derivation). Historical CVEs in the `cryptography` package (e.g., CVE-2023-49083, CVE-2024-26130) demonstrate that running outdated versions can introduce exploitable vulnerabilities. Without pinned versions, there is no way to verify the deployed version is patched.

2. **ASVS 15.2.1 is rendered completely unauditable.** The requirement asks to verify components "have not breached the documented update and remediation time frames." With no documented timeframes and no recorded component versions, this verification is impossible — the control is entirely absent.

3. **Non-reproducible builds.** Without a lock file, each deployment may resolve to different dependency versions, including ones with known vulnerabilities. The `uv run --script` invocation without a lock file will resolve dependencies at install time.

4. **Supply chain attack surface.** No protection against dependency confusion, typosquatting, or malicious package updates without version pinning.

**Proof of Concept:**

```bash
# Scenario: Vulnerable cryptography version deployed unknowingly

# Deployment 1 (January 2024)
$ uv run main.py
# Resolves to cryptography==41.0.0 (contains CVE-2023-49083)

# Deployment 2 (March 2024) 
$ uv run main.py
# Resolves to cryptography==42.0.5 (patched)

# No way to verify which version is running in production
# No audit trail of when dependencies were reviewed
# No documented policy for remediation timeframes
```

**Impact:**

- **Confidentiality:** Vulnerable versions of `cryptography` could compromise vote encryption/decryption integrity
- **Integrity:** Outdated `argon2-cffi` may allow password hash attacks
- **Availability:** Unpatched dependencies may contain DoS vulnerabilities
- **Compliance:** ASVS Level 1 requirement failure — this is the minimum baseline
- **Supply Chain:** No mitigation for dependency confusion attacks
- **Audit Trail:** No evidence of when dependencies were last reviewed or updated

**Remediation:**

**Immediate Actions:**

1. **Create a dependency manifest with pinned versions:**

```toml
# pyproject.toml
[project]
name = "steve"
version = "3.0.0"
requires-python = ">=3.11"
dependencies = [
    "asfquart>=0.x.x",
    "asfpy>=0.x.x",
    "cryptography>=44.0.0",
    "argon2-cffi>=23.1.0",
    "easydict>=1.13",
]

[tool.uv]
# Generate and commit uv.lock for reproducible builds
```

2. **Generate and commit a lock file:**

```bash
uv lock
git add uv.lock pyproject.toml
git commit -m "Add dependency version tracking"
```

3. **Generate a formal SBOM (CycloneDX or SPDX format):**

```bash
pip install cyclonedx-bom
cyclonedx-py environment -o sbom.json --format json
git add sbom.json
```

4. **Document update and remediation timeframes in `SECURITY.md`:**

```markdown
## Dependency Update Policy

### Remediation Timeframes
- **Critical CVEs:** Patch within 48 hours of disclosure
- **High CVEs:** Patch within 7 days
- **Medium CVEs:** Patch within 30 days
- **Routine updates:** Monthly dependency review cycle
- **SBOM refresh:** On every release and quarterly audit

### Process
1. Automated vulnerability scanning runs daily via pip-audit
2. Security team reviews findings within 24 hours
3. Patches applied per timeframes above
4. SBOM regenerated after each dependency update
```

5. **Integrate automated vulnerability scanning:**

```bash
# Install scanning tool
uv pip install pip-audit

# Add to CI/CD pipeline
pip-audit --requirement requirements.txt --format json

# Or use osv-scanner
osv-scanner --lockfile=uv.lock
```

6. **Enable automated dependency updates:**

```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: "pip"
    directory: "/"
    schedule:
      interval: "weekly"
    open-pull-requests-limit: 10
```

## Architecture Observations

### Dangerous Functionality Identification

`crypto.py` correctly centralizes all cryptographic operations (encryption, decryption, key derivation, hashing). Per the domain context, this module contains "dangerous functionality" and its dependencies (`cryptography`, `argon2-cffi`) are the highest-priority components for version tracking — making the absence of version management especially impactful.

**Risk:** The most security-critical dependencies are completely untracked, creating maximum exposure to known CVEs.

### `uv` Tooling Partial Adoption

The `#!/usr/bin/env -S uv run --script` shebang in `main.py` indicates the project uses `uv` as its package manager. `uv` supports `uv.lock` files for reproducible resolution. The tooling infrastructure exists but the lock file and version constraints are not present in the provided codebase.

**Opportunity:** Minimal work required to implement proper dependency management — the tooling is already in place.

### No Evidence of Dependency Auditing Pipeline

No CI/CD configuration, pre-commit hooks, or scanning tool configuration was found that would automatically flag outdated or vulnerable dependencies.

**Gap:** No automated enforcement of the dependency management policy that needs to be established.
