## Positive Security Patterns

### 1. **Architectural Enforcement of Parameterization**

The `asfpy.db` wrapper enforces parameterized queries by design. Query SQL is defined in external YAML files, and parameters are bound at runtime via `.perform()` and `.first_row()` methods. This architectural pattern separates code from data at a structural level, making SQL injection virtually impossible.

**Example from `election.py:201`:**
```python
self.c_add_issue.perform(iid, self.eid, title, description, vtype, kv_json)
```

### 2. **Proxy Pattern Prevents Ad-Hoc Queries**

`Election.__getattr__` (line ~47) and `PersonDB.__getattr__` (line ~29) proxy only to pre-defined cursors from the YAML configuration. This design makes it impossible to accidentally construct queries from user input, as there is no mechanism to execute arbitrary SQL.

**Implementation:**
```python
def __getattr__(self, name):
    return getattr(self.db, name)
```

### 3. **Vote Data Encryption Before Storage**

In `add_vote()` (line ~274), `votestring` is encrypted via `crypto.create_vote()` before being passed to `c_add_vote.perform()`. This adds defense-in-depth: even if parameterization were somehow bypassed (which it cannot be with the current architecture), the encrypted data would not be interpretable as SQL.

### 4. **Type Validation Before Database Writes**

Functions like `add_issue()` and `edit_issue()` validate `vtype in vtypes.TYPES` before database operations, ensuring that only expected data types reach the database layer.

**Example from `election.py:201`:**
```python
assert vtype in vtypes.TYPES
self.c_add_issue.perform(iid, self.eid, title, description, vtype, kv_json)
```

### 5. **Transaction Boundaries for Multi-Step Operations**

Functions like `delete()` (line ~58) and `add_salts()` (line ~140) properly use `BEGIN TRANSACTION` / `COMMIT` for referential integrity, ensuring atomic operations without introducing injection risks.

### 6. **ID Generation from Cryptographic Source**

`crypto.create_id()` generates IDs that are then used in parameterized queries, preventing any injection through ID values. IDs are never derived from user input directly.

### 7. **Single Database Access Pattern**

The entire codebase uses exactly one mechanism for database access (`asfpy.db.DB` with `.perform()` / `.first_row()`). There are no alternative paths, helper functions, or legacy code that bypass parameterization.

### 8. **Clean Separation of Concerns**

Web handlers (`pages.py`) never construct SQL — all database logic lives in `election.py` and `persondb.py`, which consistently use the parameterized wrapper. This separation makes security review straightforward and reduces the attack surface.

## Architecture Observations

### 1. **Query Definition Externalization**

SQL lives in `queries.yaml`, not in Python code. This makes query review and parameterization verification straightforward. All queries can be audited in a single location, and the YAML format enforces a consistent structure.

**Security Benefit:** Impossible to accidentally introduce string concatenation or dynamic query construction, as the query text is completely separated from the parameter binding logic.

### 2. **Consistent Wrapper Usage**

100% of database operations use the `asfpy.db.DB` wrapper. No direct SQLite API calls with user data were found. This consistency eliminates the risk of developers bypassing security controls.

### 3. **SQLite Strict Mode**

Per domain context, the database uses `strict mode`, which provides additional type safety at the database level. This complements parameterization by ensuring that even if data were somehow mishandled, type mismatches would be caught.

### 4. **Minimal Database API Surface**

Only two methods are used for database interaction: `.perform()` for mutations and `.first_row()` for queries. This minimal API reduces complexity and makes security review efficient.

### 5. **No ORM Complexity**

While the system doesn't use a traditional ORM, the `asfpy.db` wrapper provides ORM-like benefits (parameterization, query abstraction) without the complexity and potential vulnerabilities of full ORM frameworks.

## Recommendations Summary

While no SQL injection vulnerabilities were found, the following defense-in-depth improvements are recommended:

### Immediate Actions (0-30 days)

**None required.** The current implementation fully satisfies ASVS 1.2.4.

### Short-term Enhancements (1-3 months)

1. **Add Input Validation for `votestring`**
   - **Location:** `election.py:270` (comment: `### validate VOTESTRING for ISSUE.TYPE voting`)
   - **Rationale:** While parameterization prevents SQL injection, validating vote content against expected patterns (e.g., `y`/`n`/`a` for YNA, comma-separated labels for STV) would catch application logic errors early
   - **Implementation:** Add type-specific validation functions in `vtypes.py`
   - **Benefit:** Defense-in-depth, improved error messages for users

2. **Validate `docname` in `serve_doc()`**
   - **Location:** `pages.py:545` (comment: `### verify the propriety of DOCNAME`)
   - **Rationale:** While `send_from_directory` provides path traversal protection, explicit validation of document names against expected patterns would add defense-in-depth
   - **Implementation:** Whitelist expected document name patterns
   - **Benefit:** Reduced attack surface for path traversal attempts

3. **Document Parameterization Guarantee**
   - **Location:** Developer documentation / code comments
   - **Rationale:** Make the security property explicit for future maintainers
   - **Implementation:** Add comment confirming that `asfpy.db.DB.perform()` uses SQLite parameter binding (`?` placeholders)
   - **Benefit:** Knowledge transfer, easier onboarding for security-conscious developers

### Long-term Improvements (3-12 months)

1. **Automated Security Testing**
   - Add integration tests that attempt SQL injection patterns to verify parameterization holds under all code paths
   - Include tests in CI/CD pipeline

2. **Query Performance Monitoring**
   - Add logging for slow queries to detect potential denial-of-service through complex query patterns
   - While not directly related to injection, complements overall database security posture

3. **Periodic Security Audits**
   - Schedule annual reviews of database interaction patterns
   - Review `queries.yaml` for any new queries that might introduce risks
