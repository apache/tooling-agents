## Findings

### Critical Severity

#### ASVS-222-SEV-001: Server-Side Vote Content Validation Missing — Client-Side Controls Relied Upon as Sole Security Control (Type B Gap)

**Severity:** CRITICAL

**Gap Type:** Type B — Control acknowledged as needed (`### validate VOTESTRING for ISSUE.TYPE voting`) but **never implemented**. Client-side form controls create false confidence that votes are constrained.

**Location:**
- **Entry point:** `v3/server/pages.py`, `do_vote_endpoint()`, lines 387–428
- **Service layer:** `v3/steve/election.py`, `add_vote()`, lines 208–219

**Related Functions Checked:**
- `v3/server/pages.py`: `do_vote_endpoint()` (lines 387-428)
- `v3/steve/election.py`: `add_vote()` (lines 208-219)
- `v3/steve/election.py`: `list_issues()` (provides vote type information)
- `v3/steve/vtypes`: Vote type modules (consumers of unvalidated data)

**Vulnerable Code:**

`v3/steve/election.py` lines 208-219:
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting       # <── CONTROL IDENTIFIED BUT NOT IMPLEMENTED

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)

    # Pass MAYVOTE.SALT for PBKDF.
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)

    self.c_add_vote.perform(vote_token, ciphertext)     # <── STORED WITHOUT VALIDATION
```

**Data Flow:**
```
HTTP POST /do-vote/<eid>
  → form["vote-<iid>"] (user-controlled string)
    → do_vote_endpoint: votes[iid] = value (NO validation)
      → election.add_vote(uid, iid, votestring) (NO validation, just a TODO comment)
        → crypto.create_vote(..., votestring) → encrypted and stored
          → Later: tally_issue() decrypts → vtypes module processes garbage data
```

**Client-side controls relied upon (NOT server-enforced):**

For **YNA** votes (`vote-on.ezt`): HTML radio buttons constrain value to `y`, `n`, `a`:
```html
<input type="radio" name="vote-[issues.iid]" value="y">
<input type="radio" name="vote-[issues.iid]" value="n">
<input type="radio" name="vote-[issues.iid]" value="a">
```

For **STV** votes (`vote-on.ezt`): JavaScript modal constructs comma-separated candidate labels from a drag-and-drop UI, stored in a hidden input. All constraints are purely client-side.

**Proof of Concept:**

YNA vote corruption — submit arbitrary string instead of `y`/`n`/`a`:
```bash
curl -X POST https://steve.example.org/do-vote/a1b2c3d4e5 \
  -H "Cookie: session=<valid_session>" \
  -d "vote-f6g7h8i9j0=MALICIOUS_STRING&csrf_token=placeholder"
```

STV vote corruption — submit invalid/duplicate candidate labels:
```bash
curl -X POST https://steve.example.org/do-vote/a1b2c3d4e5 \
  -H "Cookie: session=<valid_session>" \
  -d "vote-k1l2m3n4o5=FAKE_LABEL,FAKE_LABEL,FAKE_LABEL&csrf_token=placeholder"
```

**Impact:**
- **Vote integrity corruption:** Invalid votestrings are encrypted and stored. During tally (`tally_issue()`), the decrypted garbage is passed to `vtypes.vtype_module(issue.type).tally(votes, ...)`, which may silently miscalculate results, crash, or produce undefined behavior.
- **Election outcome manipulation:** An authenticated voter can submit strategically crafted invalid votes to disrupt tallying of a legitimate election.
- **Silent failure:** Because votes are encrypted at submission time, invalid content is undetectable until decryption during tally — long after the election closes.

**Remediation:**

Add server-side votestring validation in **both** the handler and the service layer:

In `v3/server/pages.py` (`do_vote_endpoint`):
```python
for iid, votestring in votes.items():
    if iid not in issue_dict:
        await flash_danger(f'Invalid issue ID: {iid}')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    issue = issue_dict[iid]
    # Server-side vote content validation
    if issue.vtype == 'yna' and votestring not in ('y', 'n', 'a'):
        await flash_danger(f'Invalid vote value for issue {iid}.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
    elif issue.vtype == 'stv':
        labels = set(issue.kv.get('labelmap', {}).keys())
        submitted = [s.strip() for s in votestring.split(',') if s.strip()]
        if not submitted or len(submitted) != len(set(submitted)):
            await flash_danger(f'Invalid ranking for issue {iid}.')
            return quart.redirect(f'/vote-on/{election.eid}', code=303)
        if not all(s in labels for s in submitted):
            await flash_danger(f'Unknown candidate in ranking for issue {iid}.')
            return quart.redirect(f'/vote-on/{election.eid}', code=303)
```

In `v3/steve/election.py` (`add_vote`) — defense-in-depth:
```python
def add_vote(self, pid: str, iid: str, votestring: str):
```

## Architecture Observations

1. **Validation architecture is partially established but incomplete.** The codebase demonstrates a clear pattern of server-side validation for structural identifiers (EID, IID, state) via decorators and assertions, but this pattern was not extended to content validation of business-critical data (vote content).

2. **The `issue_dict` with vote type information is constructed at the entry point** (`do_vote_endpoint` line 402) but is only used for IID existence checking, not for votestring content validation. All necessary information for validation is available but unused.

3. **Encrypted storage masks validation failures.** The vote encryption architecture means invalid votestrings become undetectable immediately after submission. This makes the missing validation particularly dangerous — there is no secondary check before the data causes harm during tally.

---

## Recommendations Summary

### Immediate (Critical Priority)

1. **Implement server-side votestring validation** in both `do_vote_endpoint()` and `add_vote()` as described in ASVS-222-SEV-001 remediation. This is the single highest-priority item because it directly affects election integrity and the control was explicitly identified as needed but never implemented.

### Short-term

2. **Add a `validate()` function to each vote type module** (`vtypes`) that can be called from `add_vote()` as a defense-in-depth layer, ensuring the service layer independently enforces business rules regardless of the caller.

### Long-term

3. **Add server-side presence and length validation** for form fields (title, description) in `do_create_endpoint`, `do_add_issue_endpoint`, and `do_edit_issue_endpoint` to match the client-side constraints and complete the server-side validation layer.
