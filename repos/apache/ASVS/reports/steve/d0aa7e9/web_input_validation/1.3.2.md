## Positive Security Patterns

### 1. Complete Avoidance of Dynamic Code Execution Primitives

The codebase completely avoids `eval()`, `exec()`, `compile()`, `pickle`, and similar dangerous functions across all reviewed files. This represents the gold standard for ASVS 1.3.2 compliance.

**Evidence:**
- Grep analysis confirmed zero occurrences of `eval()`, `exec()`, `compile()`
- No imports of `pickle`, `subprocess`, or `os.system`
- All module imports are static

### 2. Safe JSON Handling for Structured Data

`election.py` uses `json.loads()`/`json.dumps()` (lines ~300-305) which safely parses/serializes data without code execution capability. This is the recommended safe alternative to `eval()` or `pickle.loads()` for structured data deserialization.

**Location:** `v3/steve/election.py` ~L300-305

### 3. Hardcoded Dispatch Patterns

Where branching logic is needed (e.g., `_set_election_date()` field routing), explicit if/elif chains with string comparison are used instead of dynamic dispatch via `getattr()` with user input.

**Example:** `v3/server/pages.py` `_set_election_date()` ~L84-107

### 4. Inherently Safe Template Engine

The choice of EZT as the template engine is inherently safe against server-side template injection (SSTI). Unlike Jinja2, Mako, or other full-featured engines, EZT does not support expression evaluation or code execution — it is a pure substitution/iteration engine.

**Impact:** This architectural decision eliminates an entire class of vulnerabilities at the design level, which is more robust than relying on input sanitization.

### 5. Parameterized Database Queries

Database values are never dynamically assembled into executable code; they flow through parameterized query boundaries provided by the `asfpy.db` layer.

**Evidence:** All user input in `do_create_endpoint()`, `do_add_issue_endpoint()`, and similar functions is passed as data parameters, not concatenated into SQL strings.

### 6. Static Import Statements Only

All `import` statements are static, at module level or within functions (e.g., `import re` in `rewrite_description`). No dynamic module loading based on user input.

**Evidence:** Review of all Python files confirmed no usage of `__import__()`, `importlib.import_module()` with user-controlled parameters.

### 7. Context-Appropriate Output Escaping

Where EZT values are embedded in JavaScript contexts (modal function parameters), `[format "js,html"]` encoding is applied.

**Example:** `manage.ezt`, `manage-stv.ezt` templates

## Architecture Observations

### 1. Clean Separation of Concerns

User input flows through a clear pipeline: form/JSON parsing → Python handler → parameterized DB operations. At no point does user data enter a code execution pathway. This architectural pattern makes it structurally difficult to introduce dynamic code execution vulnerabilities.

### 2. Template Engine Choice Eliminates SSTI Risk

EZT's design makes server-side template injection impossible as a code execution vector, which is a stronger guarantee than relying on sanitization of user input before template rendering. This is a defense-in-depth approach at the architectural level.

### 3. `__getattr__` Proxy Pattern

The `Election.__getattr__` and `PersonDB.__getattr__` methods proxy to the `asfpy.db.DB` object. While this is a dynamic dispatch mechanism, the attribute names are always determined by Python source code at call sites (e.g., `self.q_check_election`, `self.c_add_issue`), never by external input. This is a standard ORM proxy pattern and does not present a security risk.

**Locations:**
- `v3/steve/election.py` ~L46
- `v3/steve/persondb.py` ~L30

### 4. Safe Deserialization Pattern

The `kv` field stored in the database is JSON-serialized and deserialized using `json.loads()`. This is the recommended safe alternative to `eval()` or `pickle.loads()` for structured data deserialization, as JSON parsing can only create native Python types (dict, list, str, int, float, bool, None) without code execution capability.

**Location:** `v3/steve/election.py` ~L300-305

## Recommendations Summary

Since no findings were identified for ASVS 1.3.2, recommendations are preventive in nature:

### Immediate Actions

None required — the application fully satisfies ASVS 1.3.2.

### Short-term Recommendations

1. **Add automated linting for dynamic code execution**
   - Implement a pre-commit hook or CI check using `bandit` with rules:
     - `B307` — detects `eval()` usage
     - `B102` — detects `exec()` usage
     - `B301` — detects `pickle` usage
     - `B603`/`B605` — detects subprocess/shell execution
   - This prevents accidental introduction of dynamic code execution primitives in future development

2. **Document template engine security rationale**
   - Create architecture documentation explaining that EZT was selected (in part) for its inherent SSTI protection
   - Document this as a security control in the threat model
   - Include guidance for future developers on maintaining this security boundary

### Long-term Recommendations

1. **Maintain current security patterns**
   - Continue avoiding `eval()`, `exec()`, and dynamic code execution across all future development
   - Ensure new developers are trained on these security principles
   - Include ASVS 1.3.2 compliance in code review checklists

2. **Periodic security audits**
   - Re-audit against ASVS 1.3.2 when significant architectural changes are made
   - Include dynamic code execution checks in regular security assessments

## Conclusion

The application **fully satisfies ASVS 1.3.2** at Level 1. No instances of `eval()`, `exec()`, or other dynamic code execution features were found. Data deserialization uses safe primitives (`json.loads()`), the template engine (EZT) does not support code execution, and user input never enters a dynamic execution pathway.

The codebase demonstrates exemplary security practices in this area, with multiple layers of defense:
- Complete avoidance of dangerous primitives
- Architectural choices that eliminate vulnerability classes
