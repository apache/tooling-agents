## Findings

### Critical Severity

#### ASVS-231-CRT-001: Election Lifecycle State Enforcement Uses `assert` Statements — Control Exists But Is Removable (Type B Gap)

**Severity:** CRITICAL

**File:** `v3/steve/election.py`  
**Functions:** `open()` (line 70), `close()` (line 123), `add_issue()` (line 208), `edit_issue()` (line 228), `delete_issue()` (line 241), `add_voter()` (line 273), `delete()` (line 50)

**Related Functions Checked:**
- `_all_metadata()` (lines 160-177) — proper enforcement mechanism that exists but is not used
- `is_editable()` (lines 407-409) — state check helper used via assert
- `is_open()` (lines 411-413) — state check helper used via assert
- `_compute_state()` (lines 424-437) — state derivation logic

**Description:**

Seven critical election lifecycle management methods rely exclusively on Python `assert` statements for state enforcement. When the application runs with the `-O` optimization flag (a standard production practice), all `assert` statements are removed from bytecode, completely eliminating state transition controls. This allows the election state machine (editable → open → closed) to be traversed in arbitrary order, including backwards transitions and step skipping.

The codebase contains a proper enforcement mechanism (`_all_metadata(required_state)`) that raises `ElectionBadState` exceptions and is used correctly in voting-related methods (`add_vote()`, `tally_issue()`, `has_voted_upon()`), but this pattern was not applied to administrative lifecycle methods.

**Vulnerable Code:**

```python
# election.py line 68-70
def open(self, pdb):
    # Double-check the Election is in the editing state.
    assert self.is_editable()  # <-- REMOVED by python -O

# election.py line 120-123
def close(self):
    "Close an election."
    # The Election should be open.
    assert self.is_open()  # <-- REMOVED by python -O

# election.py line 206-208
def add_issue(self, title, description, vtype, kv):
    "Add a new issue with a generated unique IID."
    assert self.is_editable()  # <-- REMOVED by python -O

# (same pattern for edit_issue, delete_issue, add_voter, delete)
```

**Contrasted with the proper pattern used elsewhere in the same file:**

```python
# election.py line 282-286 — CORRECT implementation
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."
    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)  # <-- Raises ElectionBadState, NOT removable

# election.py line 160-177 — The proper mechanism that EXISTS
def _all_metadata(self, required_state=None):
    md = self.q_metadata.first_row(self.eid)
    if not md:
        raise self._disappeared()
    if required_state:
        state = self._compute_state(md)
        if state != required_state:
            raise ElectionBadState(self.eid, state, required_state)
    return md
```

**Data Flow:**

- **Source:** Web endpoint (e.g., `do_open_endpoint` at `pages.py:447`) receives user request
- **Sink:** State-changing method (e.g., `election.open()` at `election.py:68`) modifies election state
- **Missing Control:** The `assert` statements on lines 50, 70, 123, 208, 228, 241, 273 are the ONLY barriers preventing out-of-order state transitions. Python's `-O` flag removes them entirely.

**Proof of Concept:**

When the application is run with `python -O` (a common production optimization):

**1. Re-open an already-open election (destroys all existing votes):**
```http
GET /do-open/a1b2c3d4e5 HTTP/1.1
Cookie: [valid committer session]
```

`election.open(pdb)` runs without hitting the removed `assert self.is_editable()`. This calls `add_salts()` which regenerates ALL mayvote salts, then generates a new `opened_key`. All previously cast votes become permanently undecryptable because their vote tokens were derived from the old salts.

**2. Add issues to an open election (corrupts election integrity):**
```http
POST /do-add-issue/a1b2c3d4e5 HTTP/1.1
Cookie: [valid committer session]
Content-Type: application/x-www-form-urlencoded

title=Malicious+Issue&description=injected
```

`election.add_issue()` runs without hitting the removed `assert self.is_editable()`. The issue is added to an active election. No mayvote entries with salts exist for it, and the `opened_key` tamper detection is retroactively triggered.

**3. Delete issues from an open election (destroys active voting options):**
```http
POST /do-delete-issue/a1b2c3d4e5/f6g7h8i9j0 HTTP/1.1
Cookie: [valid committer session]
```

**Impact:**

- **Election integrity destruction:** Re-opening an active election regenerates cryptographic salts, making all previously cast votes permanently undecryptable. Voters' choices are silently and irreversibly destroyed.
- **State machine bypass:** The election lifecycle (editable → open → closed) can be traversed in any order, including backwards (open → editable operations) or skipping (editable → closed).
- **False confidence:** The `assert` statements create an appearance of state enforcement in code review while providing zero protection in optimized deployments. The proper mechanism (`_all_metadata(required_state)`) exists in the same file, proving the correct pattern was known but inconsistently applied.

**Remediation:**

Replace all `assert` statements used for state enforcement with the existing `_all_metadata(required_state)` mechanism:

```python
def open(self, pdb):
    # Verify the Election is in the editing state (raises ElectionBadState if not)
    self._all_metadata(required_state=self.S_EDITABLE)
    
    self.add_salts()
    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)
    self.c_open.perform(salt, opened_key, self.eid)

def close(self):
    "Close an election."
    # Verify the Election is open (raises ElectionBadState if not)
    self._all_metadata(required_state=self.S_OPEN)
    self.c_close.perform(self.eid)

def add_issue(self, title, description, vtype, kv):
    "Add a new issue with a generated unique IID."
    self._all_metadata(required_state=self.S_EDITABLE)
    if vtype not in vtypes.TYPES:
        raise ValueError(f'Invalid vote type: {vtype}')
    # ... rest of method

def delete(self):
    "Delete this Election and its Issues and Person/Issue pairs."
    self._all_metadata(required_state=self.S_EDITABLE)
    # ... rest of method

# Apply same pattern to: edit_issue(), delete_issue(), add_voter(), add_salts()
```

#### ASVS-231-CRT-002: Vote Content Validation Step Entirely Absent in Vote Submission Flow (Type A Gap)

**Severity:** CRITICAL

**File:** `v3/steve/election.py`, function `add_vote()`, lines 282-298  
**Called from:** `v3/server/pages.py`, function `do_vote_endpoint()`, lines 383-424

**Related Functions Checked:**
- `crypto.create_vote()` — encrypts votestring without validation
- `crypto.gen_vote_token()` — generates encryption key
- `q_get_mayvote.first_row()` — voter eligibility check (present and working)
- `vtypes.vtype_module()` — vote type system exists but no validation interface

**Description:**

The vote submission flow skips the critical business logic step of validating vote content against the issue's vote type (YNA, STV, etc.) before encryption. The expected sequential steps are: (1) authenticate user, (2) verify election is open, (3) verify voter eligibility, (4) **validate vote content**, (5) encrypt and store vote. Step 4 is completely absent, acknowledged by a TODO comment that was never implemented.

Raw user input travels directly from HTTP form fields to encrypted storage in the database. Invalid, malformed, or malicious vote strings are accepted and stored, with corruption only discovered during tallying when votes are decrypted and processed.

**Expected Sequential Steps for Vote Submission:**

1. ✅ Authenticate user → `@asfquart.auth.require({R.committer})`
2. ✅ Verify election is open → `_all_metadata(self.S_OPEN)`
3. ✅ Verify voter eligibility → `q_get_mayvote.first_row(pid, iid)`
4. ❌ **Validate vote content matches issue type** → COMPLETELY MISSING
5. ✅ Encrypt and store vote → `crypto.create_vote()`

Step 4 is skipped entirely. The `votestring` travels directly from user input to encrypted storage without any domain validation.

**Vulnerable Code:**

```python
# election.py lines 282-298
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting   # <-- TODO: validation NEVER implemented

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)

    # Pass MAYVOTE.SALT for PBKDF.
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)

    self.c_add_vote.perform(vote_token, ciphertext)
```

```python
# pages.py lines 397-415 — caller also performs no validation
for iid, votestring in votes.items():
    if iid not in issue_dict:
        await flash_danger(f'Invalid issue ID: {iid}')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    try:
        election.add_vote(result.uid, iid, votestring)  # votestring is raw user input
```

**Data Flow:**

- **Source:** HTTP form field `vote-<iid>` submitted by user (`pages.py:393-395`)
- **Sink:** `crypto.create_vote()` encrypts and stores the raw string (`election.py:293-295`)
- **Missing Control:** No validation exists between receiving the vote string and encrypting it. The expected business logic step (validate vote against issue type) is acknowledged as missing via the TODO comment but never implemented.

**Proof of Concept:**

**For a YNA (Yes/No/Abstain) issue, submit an invalid vote:**
```http
POST /do-vote/a1b2c3d4e5 HTTP/1.1
Cookie: [valid committer session with mayvote entry]
Content-Type: application/x-www-form-urlencoded

vote-f6g7h8i9j0=DEFINITELY_NOT_A_VALID_YNA_VALUE
```

**For an STV (Single Transferable Vote) issue, submit a corrupted ranking:**
```http
POST /do-vote/a1b2c3d4e5 HTTP/1.1
Cookie: [valid committer session with mayvote entry]
Content-Type: application/x-www-form-urlencoded

vote-f6g7h8i9j0=zzz,zzz,zzz,zzz,zzz
```

Both requests succeed with "Votes submitted successfully!" The invalid votestring is encrypted and stored in the `vote` table.

**Impact:**

- **Election result corruption:** Invalid votes are encrypted and stored. During `tally_issue()`, the decrypted votestrings are passed directly to `vtypes.vtype_module(issue.type).tally(votes, kv)`. Depending on tally implementation, malformed votes could be silently miscounted, crash the tally process, or produce incorrect results.
- **Irreversible data corruption:** Once encrypted and stored, there is no mechanism to distinguish valid from invalid votes without decrypting all of them. The damage is only discovered at tally time.
- **Per-election scope:** A single malicious voter can corrupt results for every issue they're eligible to vote on in an election.

**Remediation:**

Implement the missing validation step using the existing `vtypes` module infrastructure:

```python
# In election.py, add_vote()
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    # Fetch issue to determine vote type
    issue = self.q_get_issue.first_row(iid)
    if not issue:
        raise IssueNotFound(iid)

    # Validate votestring for the issue's vote type
    vtype_mod = vtypes.vtype_module(issue.type)
    if not vtype_mod.validate(votestring, self.json2kv(issue.kv)):
        raise InvalidVoteString(iid, issue.type, votestring)

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

```python
# Each vtype module should implement validate():
# vtypes/yna.py
VALID_VOTES = {'yes', 'no', 'abstain'}
def validate(votestring, kv):
    return votestring.lower().strip() in VALID_VOTES

# vtypes/stv.py  
def validate(votestring, kv):
    candidates = set(kv.get('labelmap', {}).keys())
    rankings = votestring.split(',')
    return all(r.strip() in candidates for r in rankings) and len(rankings) <= len(candidates)
```

## Architecture Observations

1. **Inconsistent enforcement patterns:** The codebase contains both reliable (`_all_metadata(required_state)`) and unreliable (`assert`) state enforcement mechanisms. The reliable mechanism was applied to the voting-specific methods but not to the election lifecycle management methods. This suggests the voting flow was hardened separately from the administrative flow.

2. **Web layer delegates entirely to domain layer:** The `pages.py` handlers perform no independent state validation before calling `election.py` methods. This means the domain layer is the sole point of business logic enforcement. When that enforcement uses `assert`, there is no defense-in-depth.

3. **Election lifecycle is well-modeled:** The three-state model (editable → open → closed) with clear database indicators (salt/opened_key presence for open, closed=1 for closed) is well-designed. The issue is purely in how transitions are guarded, not in the model itself.

4. **Vote type system exists but incomplete:** The `vtypes` module provides an extensible framework for different voting methods, with proper tally implementations. However, the validation interface was never completed, leaving a critical gap in the vote submission pipeline.

## Recommendations Summary

### Immediate (Critical Priority)

| Priority | Finding | Effort | Impact |
|----------|---------|--------|--------|
| **1** | ASVS-231-CRT-001: Replace all 7 `assert` statements with `_all_metadata(required_state)` calls | Low — the replacement mechanism already exists and is proven in `add_vote()`/`tally_issue()` | Eliminates complete state bypass under `-O` flag |
| **2** | ASVS-231-CRT-002: Implement vote content validation in `add_vote()` using per-vtype `validate()` functions | Medium — requires implementing validation logic for each vote type (yna, stv) | Prevents invalid votes from corrupting election results |

### Short-term

- **Add integration tests for `-O` mode:** Create test suite that runs with optimization flags enabled to catch similar issues
- **Implement defense-in-depth in web layer:** Add state checks in `pages.py` endpoints before delegating to `election.py` methods
- **Add pre-tally vote validation:** Implement a validation pass during `tally_issue()` that reports invalid votes rather than silently failing

### Long-term

- **Establish coding standards prohibiting `assert` for business logic:** Document that `assert` must only be used for developer-facing invariants, never for security or business rule enforcement
- **Implement comprehensive state transition logging:** Add audit trail for all election state changes with before/after snapshots
- **Add vote content preview/confirmation:** Allow voters to see their parsed vote before final submission to catch client-side corruption
