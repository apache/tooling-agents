## Findings

### Critical Severity

#### ASVS-1531-CRITICAL-001: Inconsistent Field Filtering — Election List Methods Return Raw Database Rows Without Python-Level Sensitive Field Exclusion

**Severity:** CRITICAL  
**Gap Type:** Type B — Control EXISTS (`get_metadata()`) but NOT CALLED at parallel data return paths  

**Location:**
- **File:** `v3/steve/election.py`
- **Functions:** `open_to_pid()` (lines 407-412), `upcoming_to_pid()` (lines 438-446), `owned_elections()` (lines 420-436)
- **Consumed in:** `v3/server/pages.py` lines 155-162, 320-324

**Related Functions Checked:**
- `get_metadata()` (election.py:145-159) — ✅ Implements proper filtering
- `postprocess_election()` (pages.py:477-519) — ❌ Does not strip sensitive fields
- `voter_page()` (pages.py:155-162) — Consumes unfiltered data
- `admin_page()` (pages.py:320-324) — Consumes unfiltered data

**Description:**

Audit of ASVS 15.3.1 (data minimization in API/data responses) reveals one CRITICAL finding related to inconsistent application of field-level filtering controls for election data objects. The codebase demonstrates awareness of the need to exclude sensitive cryptographic fields through an explicit filtering control in `get_metadata()`, but this control is not applied to three parallel code paths that also return election data to user-facing page templates.

**Control Definition (Where Filtering IS Applied):**

In `election.py`, `get_metadata()` (lines 145-159) explicitly constructs a new object excluding sensitive cryptographic fields:

```python
def get_metadata(self):
    "Return basic metadata about this Election."
    md = self._all_metadata()
    # NOTE: do not return the SALT or OPENED_KEY    <-- Developer awareness
    return edict(
        eid=md.eid,
        title=md.title,
        owner_pid=md.owner_pid,
        authz=md.authz,
        closed=md.closed,
        open_at=md.open_at,
        close_at=md.close_at,
        state=self._compute_state(md),
    )
```

**Data Flow:**

```
Source:  SQLite election table (contains salt BLOB(16), opened_key BLOB(32))
    ↓
Path:   open_to_pid() / upcoming_to_pid() / owned_elections()
        → returns raw EasyDict rows from query (NO Python-level field filtering)
    ↓
Sink:   voter_page() (pages.py:155-162) / admin_page() (pages.py:320-324)
        → postprocess_election() (adds fields, NEVER strips sensitive fields)
        → result.open_elections / result.upcoming_elections / result.owned
        → EZT template rendering context
        → HTTP response to authenticated user
    ↓
Missing: Explicit field construction (as implemented in get_metadata())
```

**`postprocess_election()`** (pages.py lines 477-519) — processes fields but NEVER removes sensitive ones:
```python
def postprocess_election(e):
    # NOTE: side effects. This function manipulates the E argument.
    e.closed = ezt.boolean(e.closed == 1)
    # ... adds formatting fields ...
    # NEVER checks for or removes e.salt or e.opened_key
    return e
```

**Proof of Concept:**

An attacker who is an authenticated committer navigates to `/voter`:

1. If `q_open_to_me` returns `salt` and `opened_key`, these are in the server-side template context
2. Any template rendering error, debug mode, or template injection that exposes the context object reveals the cryptographic materials
3. With `opened_key` and `mayvote.salt` (obtainable if `q_open_to_me` joins mayvote), the attacker can:
   - Compute `vote_token = crypto.gen_vote_token(opened_key, victim_pid, iid, salt)` for any eligible voter
   - Decrypt existing votes: `crypto.decrypt_votestring(vote_token, salt, ciphertext)`
   - Submit forged votes: `crypto.create_vote(vote_token, salt, fake_votestring)`

**Impact:**

If the SQL queries in `queries.yaml` for `q_open_to_me`, `q_upcoming_to_me`, or `q_owned` include `salt` or `opened_key` columns (e.g., via `SELECT *` or `SELECT e.*`), these cryptographic materials flow into the template rendering context for every authenticated user viewing the voter or admin pages:

- **`salt`** (16-byte election-level salt): Used in generating `opened_key`; exposure enables recomputation attacks
- **`opened_key`** (32-byte derived key): Combined with per-voter `mayvote.salt`, enables generation of `vote_token` values, decryption of encrypted votes, and vote forgery

The `queries.yaml` file is not available for verification, but the absence of Python-level filtering at these code paths creates a single-layer defense that violates defense-in-depth. The codebase's own pattern (`get_metadata()`) demonstrates the developer's recognition that these fields must be excluded — making the omission in parallel paths a Type B gap (control exists but not applied, creating false confidence).

**Remediation:**

Apply the same explicit field construction pattern used in `get_metadata()` to all class methods that return election data:

```python
@classmethod
def open_to_pid(cls, db_fname, pid):
    "List of elections that are OPEN for PID to vote upon."
    db = cls.open_database(db_fname)
    db.q_open_to_me.perform(pid,)
    return [cls._safe_election_summary(row) for row in db.q_open_to_me.fetchall()]

@classmethod
def owned_elections(cls, db_fname, pid):
    "List of elections that PID has created."
    db = cls.open_database(db_fname)
    db.q_owned.perform(pid,)
    return [cls._safe_election_summary(row) for row in db.q_owned.fetchall()]

@classmethod
def upcoming_to_pid(cls, db_fname, pid):
    "List of editable elections for PID to vote upon."
    db = cls.open_database(db_fname)
    db.q_upcoming_to_me.perform(pid,)
    return [cls._safe_election_summary(row) for row in db.q_upcoming_to_me.fetchall()]

@staticmethod
def _safe_election_summary(row):
    """Construct a safe election summary, excluding cryptographic fields.
    
    Mirrors the field exclusion pattern in get_metadata().
    NEVER include: salt, opened_key
    """
    result = edict(
        eid=row.eid,
        title=row.title,
        owner_pid=row.owner_pid,
        closed=row.closed,
        open_at=row.open_at,
        close_at=row.close_at,
    )
    # Carry through computed columns if present in the query
    if hasattr(row, 'is_opened'):
        result.is_opened = row.is_opened
    if hasattr(row, 'issue_count'):
        result.issue_count = row.issue_count
    return result
```

## Positive Security Patterns

1. **`get_metadata()` — Explicit Field Exclusion** (election.py:145-159): The canonical example of correct field filtering with clear documentation: `# NOTE: do not return the SALT or OPENED_KEY`. Constructs a new edict with only safe fields.

2. **`list_issues()` / `get_issue()` — Selective Field Extraction** (election.py:161-228): Both methods construct new data objects containing only the fields needed for display, never returning raw database rows.

3. **Vote Data Never Exposed via HTTP**: No web endpoint returns `vote_token`, `ciphertext`, or decrypted vote strings. The `tally_issue()` method exists only for CLI tool use.

4. **Mass Assignment Prevention** (pages.py: all POST handlers): Every POST handler extracts specific named fields from form data rather than passing the entire form to model methods. Non-writable fields (e.g., `vtype`, `kv` in `do_edit_issue_endpoint`) are sourced from the existing database record, not from user input.

5. **`has_voted_upon()` — Boolean Reduction** (election.py:309-334): Uses sensitive crypto materials internally but only returns a `{iid: True/False}` dict, properly minimizing the data returned.

6. **Error Message Minimization** (pages.py: `do_vote_endpoint`): Catches exceptions during vote submission, logs the full error server-side, but only returns a generic flash message to the user: `'Error submitting vote for issue {iid}.'`

7. **Schema-Level Data Typing** (schema.sql): `STRICT` table mode enforces column types, preventing type confusion. CHECK constraints enforce BLOB lengths for cryptographic materials.

## Architecture Observations

1. **Two Distinct Patterns for Data Return**: The codebase uses explicit field construction for single-entity retrieval (`get_metadata`, `get_issue`) but raw query passthrough for list retrieval (`open_to_pid`, `upcoming_to_pid`, `owned_elections`). This inconsistency is the root cause of the finding.

2. **SQL-Level vs. Application-Level Filtering**: The `owned_elections()` comment indicates the developer intended SQL queries to handle column selection. While this can work, it creates a fragile single-layer defense that depends on `queries.yaml` correctness and resists verification at the Python layer.

3. **`postprocess_election()` as a Pass-Through**: This function modifies and adds fields but never strips them, meaning it cannot serve as a defense layer for data minimization. Any sensitive fields in the input survive to the output.

4. **Separation of Web and CLI Concerns**: Vote decryption and tallying are properly isolated to the `tally_issue()` method, which is not called from any web handler. This is a sound architectural decision for data minimization.

## Recommendations Summary

### Immediate Actions (Critical Priority)

1. **Implement `_safe_election_summary()` static method** and apply it in `open_to_pid()`, `upcoming_to_pid()`, and `owned_elections()` as shown in the remediation code above. This ensures all code paths returning election data to templates use explicit field construction.

### Short-term Actions (Defense-in-Depth)

2. **Add a `postprocess_election()` guard** that explicitly deletes known sensitive fields if they exist:
   ```python
   def postprocess_election(e):
       # Defense-in-depth: strip fields that should never reach templates
       for field in ('salt', 'opened_key'):
           e.pop(field, None)
       # ... existing processing ...
   ```

3. **Audit `queries.yaml`** to confirm that `q_open_to_me`, `q_upcoming_to_me`, and `q_owned` queries do NOT select `salt` or `opened_key` columns. Document the expected column sets in comments alongside each query.

### Long-term Actions (Process Improvement)

4. **Establish a coding standard** that ALL methods returning data objects to callers outside the `Election` class MUST use explicit field construction (allowlist pattern), never raw query passthrough. Add this as a code review checklist item.
