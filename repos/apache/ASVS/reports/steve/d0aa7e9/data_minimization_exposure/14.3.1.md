## Findings

### Critical Severity

#### ASVS-1431-SEV-001: Complete Absence of Authenticated Data Clearing from Client Storage

**Severity:** CRITICAL  
**Gap Type:** Type A — No control exists for clearing authenticated data from client storage upon session termination  
**ASVS Requirement:** 14.3.1 (Level 1)

##### Location

- **File:** `v3/server/pages.py`
- **Functions:** `basic_info()` (lines 72–103), all authenticated route handlers
- **Affected Routes:** 12+ authenticated endpoints

##### Related Functions Checked

- `basic_info()` — lines 85–95
- `voter_page()` — line 148
- `vote_on_page()` — line 186
- `profile_page()` — line 528
- All authenticated route handlers in `pages.py`

##### Description

The application completely lacks mechanisms to clear authenticated data from client storage upon session termination. Specifically:

1. **No `Clear-Site-Data` header** is sent on any response
2. **No logout endpoint exists** to trigger session termination and cleanup
3. **No `Cache-Control` headers** prevent browser caching of authenticated pages
4. **No client-side JavaScript** implements cleanup when sessions end or server is unreachable

This is a complete absence of the ASVS 14.3.1 Level 1 requirement, which is mandatory for all applications.

##### Vulnerable Code

**Session data sourced to every authenticated template response (`pages.py`, lines 85–95):**

```python
async def basic_info():

            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
    # ...
    return basic
```

**All 12+ authenticated routes call `basic_info()` and render voter identity into HTML without any cache-control or cleanup mechanism:**

##### Proof of Concept

1. Authenticated voter accesses `/vote-on/a1b2c3d4e5` on a shared workstation
2. Browser caches the response containing the voter's name, uid, election issues, and candidate list
3. Session expires or user closes the browser tab (no logout endpoint available)
4. Next user on the same workstation presses Back button or inspects browser cache/history
5. Cached HTML reveals: voter identity (uid, full name, email), which elections they were eligible for, and that they accessed the voting page (confirming participation)

**No mechanism exists to prevent or mitigate this scenario:**
- No `Clear-Site-Data: "cache", "cookies", "storage"` header is sent on any response
- No `Cache-Control: no-store` header prevents browser caching of authenticated pages
- No logout endpoint exists to trigger session termination and cleanup
- No client-side JavaScript clears DOM/storage when the session ends

##### Impact

In the context of a **voting system** handling **voter identities**:

1. **Voter privacy violation**: Browser cache on shared computers exposes who voted and in which elections
2. **Session persistence**: Without logout, session cookies remain active until server-side expiration, enabling session reuse
3. **Cached vote confirmation**: Flash messages confirming vote submission persist in browser cache, proving a voter's participation
4. **Election administration exposure**: Management pages cached with election configuration details

##### Remediation

**1. Add `Clear-Site-Data` header on logout response:**

```python
@APP.get('/logout')
async def logout():
    # Invalidate server-side session
    await asfquart.session.destroy()
    
    response = quart.redirect('/', code=303)
    # Instruct browser to clear all site data
    response.headers['Clear-Site-Data'] = '"cache", "cookies", "storage"'
    return response
```

**2. Add `Cache-Control` and security headers to all authenticated responses via `after_request`:**

```python
@APP.after_request
async def set_security_headers(response):
    # For authenticated pages, prevent caching of sensitive data
    if quart.request.endpoint and getattr(
        quart.request.endpoint, '_requires_auth', False
    ):
        response.headers['Cache-Control'] = 'no-store, no-cache, must-revalidate, max-age=0'
        response.headers['Pragma'] = 'no-cache'
        response.headers['Expires'] = '0'
    return response
```

**Simpler alternative — apply globally since most routes are authenticated:**

```python
@APP.after_request
async def set_cache_headers(response):
    # Voting system: default to no caching; whitelist static assets
    if not quart.request.path.startswith(('/static/', '/favicon.ico')):
        response.headers['Cache-Control'] = 'no-store, no-cache, must-revalidate, max-age=0'
        response.headers['Pragma'] = 'no-cache'
        response.headers['Expires'] = '0'
    return response
```

**3. Add client-side cleanup as fallback (in base template):**

```javascript
// Detect session expiration client-side and clean up
window.addEventListener('beforeunload', function() {
    // Clear any sensitive data from sessionStorage
    sessionStorage.clear();
});

// Periodic session check — clear DOM if session is gone
setInterval(async function() {
    try {
        const resp = await fetch('/api/session-check', { credentials: 'same-origin' });
        if (resp.status === 401) {
            // Session expired — clear sensitive DOM content
            document.querySelectorAll('[data-sensitive]').forEach(el => el.textContent = '');
            sessionStorage.clear();
            localStorage.clear();
            window.location.href = '/';
        }
    } catch(e) {
        // Server unreachable — clear sensitive data as precaution
        document.querySelectorAll('[data-sensitive]').forEach(el => el.textContent = '');
        sessionStorage.clear();
    }
}, 60000);
```

## Architecture Observations

1. **No HTTP response middleware**: The application has no `after_request` handler or middleware layer for setting security headers. This means each response is sent with framework defaults, which do not include `Cache-Control: no-store` or `Clear-Site-Data`.

2. **Session management delegated to framework**: `asfquart.session` handles session reads, but no session destruction or logout mechanism is exposed in the application layer. The framework may provide this, but the application does not invoke it.

3. **Template-rendered sensitive data**: The EZT template system renders voter identity (uid, name, email) and election participation data directly into HTML responses. Without cache-control headers, browsers are free to cache these pages per their default behavior.

4. **Flash messages contain confirmation of sensitive actions**: Vote submission confirmation ("Votes submitted successfully!") stored in session and rendered in subsequent responses can reveal voter participation through browser history/cache.

---

## Recommendations Summary

### Immediate (P0)

1. **Implement `Cache-Control: no-store` on all authenticated responses** via `after_request` middleware
   - **Effort:** Low
   - **Impact:** Prevents browser caching of voter identity and election data

2. **Create a logout endpoint** that destroys the session and sends `Clear-Site-Data` header
   - **Effort:** Low
   - **Impact:** Enables users to terminate sessions and clear client-side data

### Short-term (P1)

3. **Add client-side session-check with DOM cleanup** as fallback for server-unreachable scenarios
   - **Effort:** Medium
   - **Impact:** Satisfies ASVS 14.3.1 requirement for client-side cleanup

### Long-term (P2)

4. **Mark sensitive DOM elements with `data-sensitive` attribute** in templates for targeted cleanup
   - **Effort:** Medium
   - **Impact:** Enables precise client-side data clearing
