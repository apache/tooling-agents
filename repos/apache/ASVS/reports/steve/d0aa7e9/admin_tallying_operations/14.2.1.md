## Findings

### Summary

**No findings identified.** The ASVS 14.2.1 requirement is not violated by these command-line administrative tools because they do not participate in the HTTP request cycle and do not transmit sensitive data via URLs or query strings.

All five files under audit are command-line tools or library modules that:

1. Accept input exclusively via **CLI arguments** (argparse) and **local file paths**
2. Make **no outbound HTTP requests** with sensitive data in URLs
3. Interact with **local SQLite databases** via direct file access
4. Send email via `asfpy.messaging.mail()` which uses message body, not URL parameters

No sensitive data (encryption keys, session tokens, voter credentials, decrypted vote contents) is passed through any URL or query string mechanism.

---

## Positive Security Patterns

The following security-positive patterns were identified in the codebase:

1. **No HTTP surface area**: All tools operate locally via filesystem and direct database access, eliminating the URL/query-string attack surface entirely. This architectural decision inherently prevents the vulnerability class addressed by ASVS 14.2.1.

2. **Sensitive data stays internal**: Decryption keys are derived inside `election.tally_issue()`, never exposed via arguments or URLs. This follows the principle of minimal exposure for cryptographic material.

3. **Email body, not URL**: `mail-voters.py` sends voter PII (email, name) in the email message body via `asfpy.messaging.mail()`, not appended to any URL. This prevents accidental logging of sensitive voter information in HTTP access logs or browser history.

4. **YAML file for election config**: `create-election.py` reads sensitive election setup data from a local YAML file rather than passing it as individual command-line arguments, reducing exposure in process listings (e.g., `ps`, `/proc/*/cmdline`).

5. **Tampering detection before data access**: `tally.py` checks `election.is_tampered(pdb)` before decrypting and outputting any vote data, ensuring data integrity before sensitive operations.

6. **Structured argument parsing**: All scripts use `argparse` for CLI input handling, providing type safety and validation rather than raw argument processing.

7. **Abstracted transport layer**: Email sending delegates to `asfpy.messaging` which abstracts the transport layer, preventing accidental URL-based credential exposure and ensuring proper message formatting.

## Architecture Observations

### Trust Boundary Placement

The separation of these tools as CLI scripts rather than web endpoints is a sound architectural choice that eliminates the URL/query-string sensitive data leakage risk addressed by ASVS 14.2.1. The trust boundary is correctly placed at filesystem/OS access control rather than HTTP authentication, which is appropriate for administrative tooling.

### Data Flow Architecture

The sensitive data handling (vote decryption, voter PII, election keys) occurs entirely within local process memory and direct database access, with no HTTP-based data transmission pathway. This design pattern inherently mitigates several classes of web application vulnerabilities including:

- URL-based sensitive data leakage (ASVS 14.2.1)
- Browser history exposure
- Proxy/CDN logging of sensitive parameters
- Referer header leakage

### Operational Security Considerations

While the current architecture is sound for ASVS 14.2.1 compliance, operators should be aware that:

- CLI arguments may still appear in shell history files
- Process listings may expose non-sensitive identifiers
- Local filesystem access control becomes the primary security boundary

## Recommendations Summary

### Immediate Actions

None required. The codebase demonstrates full compliance with ASVS 14.2.1 within its applicable scope.

### Short-term Recommendations

1. **Documentation**: Document the architectural decision to use CLI tools rather than web APIs for administrative functions, explicitly noting the security benefits including ASVS 14.2.1 compliance.

2. **Operational guidance**: Provide operator documentation on shell history management and process listing security for environments where CLI argument exposure is a concern.

### Long-term Recommendations

1. **Maintain architectural separation**: If web-based administrative interfaces are added in the future, ensure that sensitive data transmission follows ASVS 14.2.1 requirements (POST body or headers only, never in URLs or query strings).

2. **Security boundary documentation**: Maintain clear documentation of trust boundaries and data flow paths as the system evolves to ensure continued compliance with data protection requirements.
