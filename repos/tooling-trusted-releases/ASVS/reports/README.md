# ASVS Audit Reports

This directory contains the output of ASVS security audit pipeline runs against the ATR codebase.

## Directory structure

Each run is organized under a commit hash corresponding to the version of ATR that was audited:

```
reports/
├── README.md               ← this file
├── da901ba/                ← commit hash of the audited code
│   ├── L1/                 ← individual reports for ASVS Level 1 requirements (pre-triage, not included in `tooling-agents`)
│   │   ├── 1.2.1.md
│   │   ├── 1.2.2.md
│   │   └── ...
│   ├── L2/                 ← individual reports for ASVS Level 2 requirements (pre-triage, not included in `tooling-agents`)
│   │   ├── 1.1.1.md
│   │   ├── 1.2.6.md
│   │   └── ...
│   ├── consolidated-L1-L2.md   ← merged report across L1 and L2
│   └── issues-L1-L2.md         ← GitHub-issue-formatted findings
└── {next-commit}/          ← future runs against newer commits
    └── ...
```

## File types

### Individual reports (`L1/*.md`, `L2/*.md`)

One report per ASVS requirement (e.g., `7.2.1.md`). Each contains: executive summary with metadata (repository, date, auditor, ASVS requirement description), security controls inventory with coverage analysis, findings grouped by severity with code references and remediation, positive security patterns observed, and an appendix listing all files analyzed.

These are generated by the `run_asvs_security_audit` agent and pushed to GitHub by the `orchestrate_asvs_audit_to_github` agent.

### Consolidated report (`consolidated-L1-L2.md`)

A single merged report that reads all individual reports from the L1 and L2 directories, extracts findings, deduplicates across ASVS sections and levels, and produces: report metadata table, executive summary with severity distribution and systemic risk analysis, level coverage summary highlighting L2-only findings, all findings organized by severity with full detail, positive security controls table, ASVS compliance summary table (Pass/Fail/Partial/N/A per section), cross-reference matrix by attack surface, and level coverage analysis.

The filename includes which levels were consolidated (e.g., `consolidated-L1-L2.md` or `consolidated-L1-L2-L3.md`).

### Issues file (`issues-L1-L2.md`)

GitHub-issue-formatted findings, one issue per actionable finding. Each issue includes: descriptive title, labels (`bug`, `security`, `priority:{severity}`, `asvs-level:{level}`), ASVS level(s), summary, technical details with affected files, recommended remediation with code examples, acceptance criteria, and references to source reports and related findings.

Informational findings are excluded from the issues file (they remain in the consolidated report).

The filename matches the consolidated report's level suffix.

## Creating a new run

To audit a new commit:

1. Download the source code at the target commit into the Gofannon data store
2. Run the orchestrator with L1 and/or L2 section lists, pointing `directory` to `security/ASVS/reports/{commit}/L1` (or `L2`)
3. After all individual reports are complete, run the consolidation agent with `directories: security/ASVS/reports/{commit}/L1, security/ASVS/reports/{commit}/L2`

See the [pipeline runbook](../README.md) for detailed step-by-step instructions.

## Notes

- The commit hash in the directory name should be the short hash (7 chars) of the commit being audited, matching the commit in the report metadata
- Individual reports are generated independently and can be re-run without affecting other reports in the same directory
- The consolidation agent skips files starting with `consolidated` or `issues` when reading source reports, so it's safe to re-run consolidation without removing previous output
- The GitHub push agent (`add_markdown_file_to_github_directory`) handles overwrites by fetching the existing SHA, so re-runs update in place
