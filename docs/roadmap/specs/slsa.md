# SLSA Build Levels Integration

## Overview

[SLSA (Supply-chain Levels for Software Artifacts)](https://slsa.dev/) is a framework for ensuring the integrity of software artifacts throughout the supply chain. It defines build levels (1–3) with increasing requirements for build process security, provenance generation, and artifact verification.

SLSA complements our existing tools: the ASVS pipeline checks application code, the GHA review pipeline checks workflow security, and SLSA checks the build process itself — whether the artifact you're distributing was actually built from the source you tagged.

## Where SLSA Fits

| Concern | ASVS Pipeline | GHA Review | ATR | SLSA |
|---|---|---|---|---|
| Code vulnerabilities | ✅ | — | — | — |
| Workflow injection | — | ✅ | — | — |
| Unpinned actions | — | ✅ | — | ✅ (build isolation) |
| Release signing | — | — | ✅ (verifies) | ✅ (provenance signing) |
| Checksums | — | — | ✅ (verifies) | — |
| License compliance | — | — | ✅ (verifies) | — |
| Build provenance | — | — | — | ✅ |
| Reproducible builds | — | — | — | ✅ |
| Source integrity | — | — | — | ✅ |
| Build isolation | — | — | — | ✅ |

SLSA is primarily relevant for projects that publish packages (npm, PyPI, Maven, Docker Hub). Our GHA review already identifies which ASF projects publish — that data feeds directly into SLSA scoping.

## SLSA v1.0 Requirements

### Build Level 1: Provenance Exists

| ID | Requirement |
|---|---|
| SLSA-L1-1 | Build process MUST produce provenance metadata describing how the artifact was built |
| SLSA-L1-2 | Provenance MUST identify the build platform and entry point |
| SLSA-L1-3 | Provenance MUST identify the source repository and commit |

### Build Level 2: Hosted Build, Signed Provenance

| ID | Requirement |
|---|---|
| SLSA-L2-1 | Build MUST run on a hosted build service (not developer laptop) |
| SLSA-L2-2 | Provenance MUST be signed by the build platform |
| SLSA-L2-3 | Build service MUST authenticate provenance generation (not user-controlled) |
| SLSA-L2-4 | Provenance MUST be verifiable by consumers |

### Build Level 3: Hardened Build

| ID | Requirement |
|---|---|
| SLSA-L3-1 | Build environment MUST be ephemeral (fresh for each build) |
| SLSA-L3-2 | Build MUST be isolated from other builds and from the build service itself |
| SLSA-L3-3 | Build inputs MUST be fully declared (no implicit dependencies) |
| SLSA-L3-4 | Build MUST be reproducible (same source → same artifact) |
| SLSA-L3-5 | Provenance MUST be generated by the build service, not a user-controlled step |

## Analysis Approach

SLSA requirements are about the build process, not application code. The audit approach is different from ASVS:

### What to Analyze

| Source | What it tells us |
|---|---|
| CI workflow files (`.github/workflows/`) | Build steps, triggers, artifact generation |
| Build configuration (`Makefile`, `pyproject.toml`, `pom.xml`, `Dockerfile`) | Build inputs, dependencies, output artifacts |
| Release scripts | Signing, checksums, upload steps |
| Package registry configuration | Trusted publishing setup, API tokens |
| Provenance artifacts (if any) | `*.intoto.jsonl`, Sigstore attestations |

### Hybrid: Static + LLM

Like the ASF Baseline, many SLSA requirements can be checked statically:

| Requirement | Method | LLM Needed? |
|---|---|---|
| SLSA-L1-1 (provenance exists) | Check for provenance generation in workflows | Partial |
| SLSA-L2-1 (hosted build) | All GHA builds are hosted | No (always pass for GHA) |
| SLSA-L2-2 (signed provenance) | Check for `slsa-github-generator` or Sigstore | No |
| SLSA-L3-1 (ephemeral build) | GHA runners are ephemeral by default (self-hosted are not) | No |
| SLSA-L3-3 (declared inputs) | Analyze build files for implicit dependencies | Yes |
| SLSA-L3-4 (reproducible build) | Build system analysis | Yes |

## Integration with GHA Review

Our GHA review pipeline already has the data SLSA needs:

| GHA Review Data | SLSA Use |
|---|---|
| `ci-workflows:{owner}` (cached YAML) | Build workflow analysis |
| `ci-classification:{owner}` (publishing classification) | Which projects publish what |
| `ci-security:{owner}` (security findings) | Overlapping checks (unpinned actions, self-hosted runners) |

SLSA analysis could run as an additional agent in the GHA review pipeline rather than the ASVS pipeline, since it operates on workflow files not application code.

### Possible Architecture

```
GHA Review Pipeline (existing):
  pre-fetch → publishing → security → review → brief → JSON export

Extended with SLSA:
  pre-fetch → publishing → security → slsa → review → brief → JSON export
                                        ↑
                              New agent: analyzes build provenance
                              and supply chain for publishing repos
```

The SLSA agent would only run against repos identified as `release_artifact` or `snapshot_artifact` by the publishing agent, keeping scope focused.

## Connection to Trusted Publishing

Our GHA review already identifies "Trusted Publishing Migration Opportunities" — repos using long-lived API tokens where OIDC-based trusted publishing is available. SLSA L2+ requires platform-authenticated provenance, which aligns with trusted publishing:

- Trusted publishing: "the registry trusts the build platform's identity"
- SLSA L2: "provenance is authenticated by the build platform"

These are two sides of the same coin. A project that adopts trusted publishing is halfway to SLSA L2.

## Relationship to ATR and ASF Baseline

ATR, SLSA, and ASF Baseline address artifact integrity at three different layers:

| Layer | Tool | Question it answers |
|---|---|---|
| **Source & CI** | ASF Baseline | "Is the code and CI configured to produce secure releases?" |
| **Build process** | SLSA | "Was this artifact built securely, with provenance?" |
| **Distribution** | ATR | "Is this released artifact signed, checksummed, and license-compliant?" |

They form a chain: ASF Baseline checks that your release workflow has a signing step → SLSA verifies the build produced provenance and ran in an isolated environment → ATR verifies the final artifact is signed and has checksums before it reaches mirrors.

A project with all three has end-to-end assurance: the source is set up correctly (Baseline), the build is tamper-resistant (SLSA), and the artifact is verified at distribution (ATR).
