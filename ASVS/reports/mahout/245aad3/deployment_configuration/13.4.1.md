## Findings

### LOW SEVERITY

#### Finding ASVS-1341-LOW-001: No Build/Packaging Configuration to Exclude Source Control Metadata from Distribution Artifacts

**Severity**: LOW  
**Gap Type**: Type A — No control exists  
**ASVS Requirement**: 13.4.1  
**CWE**: CWE-540 (Inclusion of Sensitive Information in Source Code)

**Location**:
- **Primary Files Affected**:
  - `qdp/qdp-python/qumat_qdp/__init__.py` (Python package entry point)
  - `qumat/qumat.py` (Second Python package)
  - `qdp/qdp-core/src/lib.rs` (Rust crate root)
- **Missing Configuration Files**: MANIFEST.in, pyproject.toml, Cargo.toml (not provided for review)

**Related Functions Checked**:
- Package initialization: `qdp/qdp-python/qumat_qdp/__init__.py` lines 1-50
- Module loading: `qumat/qumat.py` line 67 (`import_module`)
- FFI exports: `qdp/qdp-core/src/lib.rs` (all `#[pyfunction]` decorated functions)

**Description**:

The provided source files contain no mechanism to exclude `.git`, `.svn`, or other source control metadata from distribution packages. The domain context explicitly confirms: *"No guidance exists on removing source control metadata (.git directories) from production deployments."*

For a Python library, the relevant deployment vectors are:

1. **PyPI distribution** (sdist/wheel) — setuptools may include `.git` in sdist if no `MANIFEST.in` or `pyproject.toml` exclusion exists
2. **Vendored deployment** — consumers copying the repository directly into their application (common in scientific computing workflows)
3. **Container images** — building Docker images from the source tree without `.dockerignore`

**Vulnerable Configuration** (Missing):

```ini
# MANIFEST.in (NOT FOUND)
global-exclude .git
global-exclude .gitignore
global-exclude .svn
prune .git
prune .svn
```

```toml
# Cargo.toml (NOT PROVIDED FOR REVIEW)
[package]
exclude = [".git/", ".svn/", ".github/"]
```

**Data Flow**:

```
Source Repository (.git present)
         ↓
    Build Process (no exclusion rules)
         ↓
    Distribution Artifact (sdist/wheel/crate)
         ↓ (potential inclusion)
    .git metadata in package
         ↓
    Installation in shared environment
         ↓
    Co-tenant access to .git/
         ↓
    Information disclosure
```

**Proof of Concept**:

If the library is installed by cloning and the `.git` directory is retained:

```bash
# In a shared HPC environment where the library is installed from source
cat /path/to/installed/qumat_qdp/.git/config
# Reveals: remote origin URLs, branch tracking info

git -C /path/to/installed/qumat_qdp/ log --oneline -20
# Reveals: recent commits, potentially including security fix messages

git -C /path/to/installed/qumat_qdp/ show HEAD:some/internal/file
# Reveals: any file in the repository history
```

**Impact**:

While this is a library (not a web-accessible deployment), `.git` metadata exposure can leak:

- **Complete commit history** — including security-related patches that reveal vulnerability details
- **Developer identities** — names and email addresses from commit metadata
- **Internal branch names** — revealing development process and feature roadmap
- **Pre-release code** — via `git checkout` if `.git/objects` is present
- **Configuration secrets** — if accidentally committed and later removed from working tree

**Risk Context**: For scientific computing libraries that are often deployed in shared HPC environments or Jupyter notebook servers, the `.git` directory could be traversable by co-tenant users.

**Remediation**:

**1. Python Packaging** — Add `MANIFEST.in`:

```ini
# MANIFEST.in
global-exclude .git*
global-exclude .svn*
prune .git
prune .svn
prune .github
```

Or in `pyproject.toml`:

```toml
[tool.setuptools]
packages = { find = { exclude = [".git*", ".svn*"] } }

[tool.hatch.build]
exclude = [
    ".git",
    ".svn", 
    ".git/**",
    ".svn/**",
    ".github/**"
]
```

**2. Rust Packaging** — Add to `Cargo.toml`:

```toml
[package]
exclude = [
    ".git/",
    ".svn/",
    ".github/",
    "**/.git",
    "**/.svn"
]
```

**3. Container Builds** — Add `.dockerignore`:

```
.git
.svn
.github
.gitignore
.gitattributes
**/.git
**/.svn
```

**4. Deployment Documentation** — Add to README.md:

```markdown
## Production Deployment

When deploying this library, ensure source control metadata is excluded:

- **Recommended**: Install from PyPI (`pip install qumat-qdp`) rather than from source
- **If installing from source**: Remove `.git` directories after cloning:
  ```bash
  find /install/path -name .git -type d -exec rm -rf {} +
  ```
- **For containerized deployments**: Use the provided `.dockerignore` to exclude SCM metadata
```

**5. CI/CD Verification** — Add build verification:

```yaml
# .github/workflows/verify-package.yml
- name: Verify no SCM metadata in distribution
  run: |
    python -m build
    unzip -l dist/*.whl | grep -q '\.git' && exit 1 || echo "OK: No .git in wheel"
    tar -tzf dist/*.tar.gz | grep -q '\.git' && exit 1 || echo "OK: No .git in sdist"
```

## Positive Security Patterns

### 1. Constrained Dynamic Module Loading

**Location**: `qumat/qumat.py`, line 67

```python
self.backend_module = import_module(f".{self.backend_name}_backend", package="qumat")
```

**Analysis**: The relative import with `package="qumat"` constrains module loading to the `qumat` package namespace, preventing arbitrary module import. Combined with input validation on `backend_config` (lines 45-60), this limits the attack surface for module injection attacks.

**Security Value**: Prevents path traversal or arbitrary code execution via malicious backend names.

---

### 2. Graceful Backend Degradation

**Location**: `qdp/qdp-python/qumat_qdp/__init__.py`, lines 31-38

```python
_qdp_mod = get_qdp()
if _qdp_mod is not None:
    RustQdpEngine = getattr(_qdp_mod, "QdpEngine")
```

**Analysis**: Uses `getattr()` with safe defaults, preventing import errors from propagating sensitive stack traces that could reveal internal paths or implementation details.

**Security Value**: Reduces information leakage through error messages (adjacent to ASVS 13.4 scope).

---

### 3. CUDA Pointer Validation

**Location**: `qdp/qdp-core/src/lib.rs`, lines 58-95

**Function**: `validate_cuda_input_ptr()`

**Analysis**: Validates null pointers, memory types, and device affinity before any unsafe operations. This is consistently called across all `encode_from_gpu_ptr*` variants.

**Security Value**: Prevents memory corruption and information disclosure through invalid pointer dereferences.

---

### 4. Input Validation Before Unsafe Operations

**Location**: `qdp/qdp-core/src/lib.rs`, multiple entry points

**Analysis**: All GPU pointer entry points validate `input_len == 0` and `num_samples == 0` checks before entering unsafe FFI code, preventing null-length buffer operations.

**Security Value**: Defense-in-depth against buffer overruns and undefined behavior in unsafe code.

## Architecture Observations

### 1. Library vs. Service Deployment Model

This codebase is a computational library, not a network-accessible service. The primary deployment vector is package installation (`pip install`, `cargo build`), which naturally excludes `.git` when using wheels but may include it in sdist or source installations. 

**Implication**: The threat model for 13.4.1 is different than for web applications but still applicable for shared-environment deployments (HPC clusters, multi-tenant Jupyter servers).

---

### 2. Build Configuration Gap

No build configuration files (`Cargo.toml`, `pyproject.toml`, `setup.py`, `MANIFEST.in`) were provided for audit. These are the primary control points for ASVS 13.4.1 compliance in a library context. 

**Implication**: **A complete audit requires reviewing these files.** The current assessment can only confirm the absence of controls in source code, not in the build pipeline.

---

### 3. Error Message Information Leakage

CUDA error messages in `lib.rs` include error codes and descriptions (`cuda_error_to_string(ret)`). While not SCM metadata, these can reveal GPU driver versions and hardware details as noted in the domain context. 

**Assessment**: This is inherent to debugging GPU applications and is LOW risk for a library. Consider adding a release build flag to sanitize error messages if the library is used in security-sensitive contexts.

---

### 4. Conditional Unsafe Code Handling

The `#![allow(unused_unsafe)]` directive in `lib.rs` line 20 suppresses compiler warnings for conditional unsafe blocks. This is documented as intentional for CUDA/non-CUDA build parity.

**Assessment**: Does not affect deployment security. The unsafe blocks are appropriately guarded by input validation.
