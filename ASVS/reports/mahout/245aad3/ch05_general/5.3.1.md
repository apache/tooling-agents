**Audit Conclusion:**

After thorough analysis of all three source files, **ASVS 5.3.1 is Not Directly Applicable to this codebase**. This is a GPU-accelerated scientific computing library (quantum data encoding), not a web application. It contains:

- **No HTTP server** or web framework
- **No file upload mechanism** that stores files in a public folder
- **No web-accessible public directory** where generated/uploaded files could be served
- **No file serving functionality**

The library **reads** files from user-specified paths and produces **in-memory GPU tensors** — it never writes user-influenced content to disk locations that could be served via HTTP.

Despite the non-applicability of the core requirement, this audit documents relevant security controls, positive patterns, and observations related to file handling safety.

## Findings

### No Vulnerabilities Found for ASVS 5.3.1

The ASVS 5.3.1 requirement targets web applications that accept file uploads and store them in publicly-accessible directories. This codebase is a computational library that:

1. **Reads** files from user-specified paths (local or remote)
2. **Processes** them into GPU tensor representations
3. **Returns** in-memory tensor objects — never writes output files to any directory

No file upload-to-public-folder-to-HTTP-execution attack surface exists.

## Positive Security Patterns

### 1. Safe PyTorch Deserialization ✅

**File:** `loader.py` line ~301

```python
raw = torch.load(path, weights_only=True)
```

Uses `weights_only=True` to prevent arbitrary code execution through pickle deserialization. This is a critical defense against malicious `.pt`/`.pth` files.

### 2. Extension-Based Format Whitelisting ✅

**File:** `engine.rs` lines ~358-393

```rust
let ptr = if path.ends_with(".parquet") {
    // ...
} else if path.ends_with(".arrow") || path.ends_with(".feather") {
    // ...
} else {
    return Err(PyRuntimeError::new_err(format!(
        "Unsupported file format. Expected .parquet, .arrow, .feather, .npy, .pt, .pth, or .pb, got: {}",
        path
    )));
};
```

Unrecognized file extensions are explicitly rejected rather than processed through a default handler.

### 3. Remote URL Fragment/Query Rejection ✅

**File:** `loader.py` lines ~127-130

```python
if "://" in path and ("?" in path or "#" in path):
    raise ValueError(
        "Remote URL query/fragment is not supported; use plain scheme://bucket/key path."
    )
```

Prevents potential URL manipulation when accessing remote storage.

### 4. RAII Temporary File Cleanup ✅

**File:** `engine.rs` lines ~724-729

```rust
#[cfg(feature = "remote-io")]
let _resolved = qdp_core::remote::resolve_path(path_str.as_str()).map_err(|e| {
    PyRuntimeError::new_err(format!("Remote path resolution failed: {}", e))
})?;
```

Remote files downloaded to temporary locations are cleaned up via RAII guards (`_resolved`), preventing temporary file accumulation.

### 5. Strict Fallback File Type Restriction ✅

**File:** `loader.py` lines ~37-39

```python
_TORCH_FILE_EXTS = frozenset({".pt", ".pth"})
_NUMPY_FILE_EXTS = frozenset({".npy"})
_FALLBACK_FILE_EXTS = _TORCH_FILE_EXTS | _NUMPY_FILE_EXTS
```

The Python fallback backend restricts loadable formats to a minimal safe set.

### 6. Type Validation After Deserialization ✅

**File:** `loader.py` lines ~302-304

```python
if not isinstance(raw, torch.Tensor):
    raise RuntimeError(
        f"Expected torch.Tensor in {path}, got {type(raw).__name__}"
    )
```

After loading, the deserialized object is type-checked before further processing.

### 7. Streaming Format Restriction ✅

**File:** `loader.py` lines ~131-135

```python
if streaming and not (check_path.lower().endswith(".parquet")):
    raise ValueError(
        "streaming=True supports only .parquet files; use streaming=False for other formats."
    )
```

Streaming mode is restricted to Parquet, the only format with proven chunked-read support, preventing potential issues with formats that don't support partial reads.

## Architecture Observations

1. **No Web Server Component**: The codebase contains no HTTP server, web framework, routing, or file serving logic. Files are exclusively consumed as computational inputs.

2. **Read-Only File Access Pattern**: All file operations are read-only. No code path writes user-influenced content to any directory on the filesystem. Output tensors exist only in GPU memory.

3. **Two-Layer Validation Architecture**: Python-level validation (`loader.py:_validate_loader_args()`, `source_file()`) provides early rejection, while Rust-level validation (`engine.rs:encode_from_file()`, `loader.rs:parse_null_handling()`) provides defense-in-depth.

4. **Remote I/O Isolation**: Remote file resolution (`remote-io` feature) downloads to temporary files managed by RAII guards, not to application directories. The temporary files are automatically cleaned up.

5. **No Generated File Storage**: The library generates `QuantumTensor` objects (GPU memory pointers wrapped in DLPack protocol) — never disk files. There is no write path that could place executable content in a web-accessible directory.

## Recommendations Summary

Since no vulnerabilities were found for ASVS 5.3.1, the following are **defense-in-depth suggestions** for future-proofing:

### Immediate Actions
None required. ASVS 5.3.1 is not applicable to this architecture.

### Short-term Improvements

**LOW — Harmonize Remote URL Validation Across Entry Points**

The URL fragment/query rejection in `loader.py:source_file()` is not replicated in `engine.rs:encode_from_file()` when called directly (not through the Python loader). While this doesn't create a 5.3.1 issue, harmonizing validation across all entry points improves defense-in-depth.

**Current gap:**
```python
# loader.py — validates ✅
if "://" in path and ("?" in path or "#" in path):
    raise ValueError(...)
```
```rust
// engine.rs — no equivalent check for direct encode() calls with remote URLs
fn encode_from_file(&self, path: &str, ...) -> PyResult<QuantumTensor> {
    // Delegates directly to qdp_core::remote::resolve_path()
}
```

**LOW — Verify `qdp_core::encode_from_torch()` Uses Safe Deserialization**

The Python fallback uses `weights_only=True`, but the Rust path delegates `.pt`/`.pth` loading to `qdp_core::encode_from_torch()`. Verify that the core library also uses safe deserialization (this code is outside the audited scope but worth confirming).

### Long-term Considerations

**LOW — Document Non-Applicability for Web Integration**

If this library is ever embedded in a web service that accepts file uploads, the integrating application must implement ASVS 5.3.1 controls (non-executable upload directories, Content-Type enforcement, filename sanitization). Consider adding a security note in integration documentation.

## Conclusion

**ASVS 5.3.1 Status: NOT APPLICABLE**

This codebase is a scientific computing library with no HTTP server, no file upload mechanism, and no public file storage. All file operations are read-only, and outputs exist exclusively as in-memory GPU tensors. The requirement's threat model (uploaded files being executed as server-side code via HTTP) has no attack surface in this architecture.

The codebase demonstrates strong defensive patterns for file handling safety, including extension whitelisting, safe deserialization (`weights_only=True`), RAII cleanup of temporary files, and input validation at multiple layers.
