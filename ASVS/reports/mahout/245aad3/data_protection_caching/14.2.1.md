## Findings

### LOW SEVERITY

#### ASVS-1421-SEV-001: No Data Minimization in Backend Configuration Passing

**Severity:** LOW  
**Gap Type:** Type A (Entry point with no control)

**Location:**
- **File:** `qumat/qumat.py`
- **Functions:** `__init__` (line 58), `execute_circuit` (line 255), `get_final_state_vector` (line 308)

**Related Functions Checked:**
- `initialize_backend()` (line 58)
- `execute_circuit()` (line 224-258)
- `get_final_state_vector()` (line 277-312)

**Description:**

The entire `backend_config` dictionary, including any credentials or sensitive identifiers, is passed to backend modules without data minimization. If a backend module (current or future) constructs HTTP URLs from arbitrary config fields, sensitive data could leak into URLs, violating ASVS 14.2.1.

**Vulnerable Code:**

```python
# Line 58 - Full config dict passed to backend
self.backend = self.backend_module.initialize_backend(self.backend_config)

# Line 253-257 - Config mutated with runtime data, then passed wholesale
self.backend_config["parameter_values"] = bound_parameters
return self.backend_module.execute_circuit(
    self.circuit, self.backend, self.backend_config
)
```

**Data Flow:**

```
User-provided backend_config (may contain credentials, API keys, S3 paths)
  → Stored as self.backend_config (mutable dict)
  → Mutated: parameter_values added at runtime
  → Passed wholesale to backend_module.execute_circuit()
  → Backend module constructs HTTP requests to cloud services
  → No control ensures credentials stay in headers, not URLs
```

**Proof of Concept:**

```python
# A custom or misconfigured backend could extract credentials from config
# and place them in URL parameters:
config = {
    "backend_name": "custom_cloud",
    "backend_options": {
        "api_key": "sk-secret-key-12345",  # Sensitive
        "endpoint": "https://quantum-api.example.com",
        "shots": 1000
    }
}
qm = QuMat(config)
# config["api_key"] passed to backend, which might do:
# requests.get(f"{endpoint}/run?api_key={api_key}&shots={shots}")
```

**Impact:**

This is mitigated by the current use of official SDKs that handle HTTP transport correctly, but creates a fragile dependency on third-party SDK behavior. If a backend module (current or future) constructs HTTP URLs from arbitrary config fields, sensitive data could leak into URLs, violating ASVS 14.2.1.

**Remediation:**

```python
def execute_circuit(self, parameter_values=None):
    self._ensure_circuit_initialized()
    # ... parameter binding logic ...
    
    # Data minimization: only pass execution-relevant fields
    execution_config = {
        "backend_options": {
            k: v for k, v in self.backend_config.get("backend_options", {}).items()
            if k in ("simulator_type", "shots")  # Allowlist non-sensitive fields
        },
        "parameter_values": bound_parameters,
    }
    
    return self.backend_module.execute_circuit(
        self.circuit, self.backend, execution_config
    )
```

#### ASVS-1421-SEV-002: Backend Configuration Mutation Enables Data Accumulation

**Severity:** LOW  
**Gap Type:** Type A (Entry point with no control)

**Location:**
- **File:** `qumat/qumat.py`
- **Functions:** `execute_circuit` (line 253), `get_final_state_vector` (line 306)

**Related Functions Checked:**
- `execute_circuit()` (line 224-258)
- `get_final_state_vector()` (line 277-312)

**Description:**

The `backend_config` dictionary is mutated during execution to add `parameter_values`. Over multiple circuit executions, this object accumulates execution data alongside any credentials. If this object is serialized, logged, cached, or transmitted (e.g., through telemetry as noted in the domain context: "Sensitive data could be sent to external trackers if quantum backends have telemetry enabled"), both credentials and quantum algorithm parameters (which may be IP-sensitive) would be exposed together.

**Vulnerable Code:**

```python
# Line 253 - execute_circuit
self.backend_config["parameter_values"] = bound_parameters

# Line 306 - get_final_state_vector  
self.backend_config["parameter_values"] = bound_parameters
```

**Data Flow:**

```
Initial backend_config (credentials + options)
  → execute_circuit() adds parameter_values (quantum IP)
  → Entire dict now contains credentials + quantum IP
  → If backend has telemetry → both leak to external trackers
  → If serialized to URL for cloud API → violates 14.2.1
```

**Impact:**

If the `backend_config` object is serialized, logged, cached, or transmitted, both credentials and quantum algorithm parameters (which may be IP-sensitive) would be exposed together. This increases the blast radius of any data exposure incident.

**Proof of Concept:**

```python
qm = QuMat({"backend_name": "braket", "backend_options": {"api_key": "secret"}})
qm.execute_circuit([0.5, 1.0])  # Adds parameter_values to backend_config
# Now backend_config contains both api_key and parameter_values
# If logged or sent to telemetry: both leak
```

**Remediation:**

```python
def execute_circuit(self, parameter_values=None):
    # ... validation ...
    
    # Create execution-scoped config copy instead of mutating instance config
    exec_config = dict(self.backend_config)
    exec_config["parameter_values"] = bound_parameters
    
    return self.backend_module.execute_circuit(
        self.circuit, self.backend, exec_config
    )
```

## Architecture Observations

### 1. No HTTP Layer in Provided Code

The provided files are library internals (GPU memory, file I/O, circuit interface). HTTP transport is delegated to third-party quantum backend SDKs (Qiskit, Cirq, Braket). ASVS 14.2.1 compliance depends on those SDKs, which are outside the audit scope but are official SDKs that follow proper HTTP practices.

### 2. Configuration as Credential Transport

The `backend_config` dictionary serves double duty as both configuration and implicit credential transport. There is no type-safe separation between:
- Operational parameters (shots, simulator_type)
- Credentials (API keys, AWS config)
- Runtime data (parameter_values)

This makes it difficult to enforce that credentials never appear in URLs if a web wrapper is added.

### 3. Web Service Wrapping Risk

The domain context notes this could be "wrapped in web service." If a REST API is built on top of `QuMat`:
- Circuit parameters could be passed as URL query parameters (e.g., `/execute?shots=1000&angle=1.57`)
- Backend names could be in URL paths (e.g., `/backend/amazon_braket/execute`)
- State vectors could be returned with full precision rather than minimum necessary data

### 4. Full State Vector Exposure

The domain context notes: "The system returns full quantum state vectors rather than minimum required data." While not a URL-specific issue, `get_final_state_vector()` returns the complete quantum state, which could contain IP-sensitive research data. If exposed via a web API, this data should only travel in response bodies, never cached in URLs.

## Recommendations Summary

### Immediate Actions (Critical/High Priority)

No immediate actions required — no Critical or High severity findings identified.

### Short-term Recommendations (Medium Priority)

| Priority | Recommendation | Rationale |
|----------|----------------|-----------|
| 1 | **Separate credentials from operational config** in `QuMat.__init__`: Create a typed configuration class that isolates credentials (sourced from env vars per AWS best practices) from operational parameters (shots, simulator_type). | Prevents credential leakage if config is serialized, logged, or passed to URL-constructing code |
| 2 | **Stop mutating `self.backend_config`** in `execute_circuit()` and `get_final_state_vector()`: Create execution-scoped copies instead. | Prevents data accumulation and reduces blast radius if config object is exposed |
| 3 | **Document ASVS 14.2.1 requirements for web wrappers**: If the library is wrapped in an HTTP service, provide guidance that credentials and quantum state data must only be transmitted in HTTP bodies/headers, never in URLs. | The library is likely to be wrapped in a web API; proactive documentation prevents violations |

### Long-term Recommendations (Low Priority)

| Priority | Recommendation | Rationale |
|----------|----------------|-----------|
| 4 | **Add anti-caching headers guidance**: If quantum state vectors or measurement results are served over HTTP, ensure `Cache-Control: no-store` and `Pragma: no-cache` headers are documented for web service implementors. | Domain context notes: "No anti-caching headers are set" |
| 5 | **Implement data minimization for state vectors**: Add an option to `get_final_state_vector()` that returns only measurement probabilities or specific qubit states rather than the full state vector. | Reduces the amount of sensitive IP data that could be exposed through any channel |
