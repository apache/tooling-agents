## Findings

### Medium Severity

#### ASVS-231-MED-001: Validation Step Bypassable in QuantumEncoder Trait

| Attribute | Detail |
|-----------|--------|
| **Finding ID** | ASVS-231-MED-001 |
| **Severity** | MEDIUM |
| **Gap Type** | Type B – Control EXISTS but NOT structurally CALLED |
| **File** | `qdp/qdp-core/src/gpu/encodings/mod.rs` |
| **Function** | `QuantumEncoder` trait definition (lines 56–145) |
| **Related Functions** | `validate_input()`, `encode()`, `encode_batch()`, `encode_batch_f32()` |
| **ASVS Requirement** | 2.3.1 – Business logic flows must be processed in expected sequential step order without skipping steps |

**Description:**  
The `QuantumEncoder` trait exposes `validate_input()` and `encode()` as separate, independently callable public methods. The trait contract does not enforce that validation must occur before encoding. A consumer of this API (or a new encoder implementation) can invoke `encode()`, `encode_batch()`, `encode_batch_f32()`, or any GPU-pointer variant directly, bypassing all Rust-level preprocessing validation (qubit count limits, data size checks, NaN/Infinity detection, zero-norm detection).

**Vulnerable Code:**
```rust
// mod.rs lines 56-73
pub trait QuantumEncoder: Send + Sync {
    /// Encode classical data to quantum state on GPU
    fn encode(
        &self,
        device: &Arc<CudaDevice>,
        data: &[f64],
        num_qubits: usize,
    ) -> Result<GpuStateVector>;

    // ... encode_batch, encode_batch_f32, encode_from_gpu_ptr all similarly unguarded

    /// Validate input data before encoding — separate, optional call
    fn validate_input(&self, data: &[f64], num_qubits: usize) -> Result<()> {
        Preprocessor::validate_input(data, num_qubits)
    }
}
```

**Data Flow:**
```
Caller → get_encoder("amplitude") → Box<dyn QuantumEncoder>
  → encoder.encode(device, data, 31)  // SKIP validate_input
  → GPU memory allocation of 2^31 complex numbers (~32 GB)
  → OOM / resource exhaustion
```

**Proof of Concept:**
```rust
let encoder = get_encoder("amplitude")?;
let device = CudaDevice::new(0)?;

// Attacker-controlled inputs:
let malicious_data = vec![1.0; 4]; // small data
let num_qubits = 31; // exceeds MAX_QUBITS (30)

// Validation exists but is not called:
// encoder.validate_input(&malicious_data, num_qubits)?;  // Would catch this

// Directly encode — skips qubit count, data size, NaN/Inf checks:
let state = encoder.encode(&device, &malicious_data, num_qubits)?;
// Attempts to allocate 2^31 * 16 bytes = 32 GB GPU memory
```

**Impact:**
- GPU memory exhaustion (denial of service) by exceeding MAX_QUBITS limit
- Invalid quantum states from NaN/Infinity input propagating to GPU kernels
- Potential GPU kernel hangs or undefined behavior from malformed inputs
- Buffer overflows if data length exceeds state vector size and encoder doesn't check internally

**Remediation – Template Method Pattern:**
```rust
pub trait QuantumEncoder: Send + Sync {
    /// Public entry point — validates then encodes (non-overridable)
    fn encode(
        &self,
        device: &Arc<CudaDevice>,
        data: &[f64],
        num_qubits: usize,
    ) -> Result<GpuStateVector> {
        self.validate_input(data, num_qubits)?;
        self.encode_impl(device, data, num_qubits)
    }

    /// Implementation-specific encoding logic (override this)
    fn encode_impl(
        &self,
        device: &Arc<CudaDevice>,
        data: &[f64],
        num_qubits: usize,
    ) -> Result<GpuStateVector>;

    /// Similarly for batch methods:
    fn encode_batch(
        &self,
        device: &Arc<CudaDevice>,
        batch_data: &[f64],
        num_samples: usize,
        sample_size: usize,
        num_qubits: usize,
    ) -> Result<GpuStateVector> {
        Preprocessor::validate_batch(batch_data, num_samples, sample_size, num_qubits)?;
        self.encode_batch_impl(device, batch_data, num_samples, sample_size, num_qubits)
    }

    fn validate_input(&self, data: &[f64], num_qubits: usize) -> Result<()> {
        Preprocessor::validate_input(data, num_qubits)
    }

    // ...
}
```

#### ASVS-231-MED-002: Missing f32 Batch Validation Path

| Attribute | Detail |
|-----------|--------|
| **Finding ID** | ASVS-231-MED-002 |
| **Severity** | MEDIUM |
| **Gap Type** | Type A – Entry point with NO control |
| **File** | `qdp/qdp-core/src/preprocessing.rs` and `qdp/qdp-core/src/gpu/encodings/mod.rs` |
| **Function** | `Preprocessor::validate_batch` (line 83) vs `QuantumEncoder::encode_batch_f32` (line 118) |
| **Related Functions** | `validate_batch()`, `encode_batch_f32()`, `encode_batch_from_gpu_ptr_f32()` |
| **ASVS Requirement** | 2.3.1 – Business logic flows must not skip steps |

**Description:**  
The Rust preprocessing layer provides `validate_batch()` only for `&[f64]` data. The `QuantumEncoder` trait exposes `encode_batch_f32()` and `encode_batch_from_gpu_ptr_f32()` for `f32` data paths, but there is no corresponding `Preprocessor::validate_batch_f32()` function. This means the expected sequential flow of "validate batch → encode batch" has no Rust-level validation step available for the f32 encoding path. While GPU-level finite-value checking exists in `validation.cu`, the Rust-level checks (batch data length consistency, sample size vs. state vector size, qubit count) have no f32 equivalent.

**Vulnerable Code:**
```rust
// preprocessing.rs:83 — only f64
pub fn validate_batch(
    batch_data: &[f64],  // <-- f64 only
    num_samples: usize,
    sample_size: usize,
    num_qubits: usize,
) -> Result<()> { ... }

// mod.rs:118 — f32 path exists with no corresponding validation
fn encode_batch_f32(
    &self,
    _device: &Arc<CudaDevice>,
    _batch_data: &[f32],  // <-- f32 data, no validate_batch_f32 exists
    _num_samples: usize,
    _sample_size: usize,
    _num_qubits: usize,
) -> Result<GpuStateVector> { ... }
```

**Data Flow:**
```
Caller → encoder.encode_batch_f32(device, f32_data, samples, size, qubits)
  → No Preprocessor::validate_batch_f32 available
  → Rust-level checks (data length, sample_size vs state_len) SKIPPED
  → Only GPU-level finite check available (if implementation calls it)
  → Potential: data length mismatch, oversized samples
```

**Proof of Concept:**
```rust
let encoder = get_encoder("amplitude")?;
let device = CudaDevice::new(0)?;

// Mismatched batch dimensions — would be caught by validate_batch for f64
let batch_f32: Vec<f32> = vec![1.0f32; 100];
let num_samples = 5;
let sample_size = 30;  // 5 * 30 = 150 != 100

// No validate_batch_f32 to call:
// Preprocessor::validate_batch_f32(&batch_f32, 5, 30, 4)?; // doesn't exist

// If encoder implements encode_batch_f32 without own validation:
let state = encoder.encode_batch_f32(&device, &batch_f32, num_samples, sample_size, 4)?;
// Potential out-of-bounds read or incorrect quantum state
```

**Impact:**
- Batch dimension mismatches go undetected at the Rust level
- `sample_size > state_vector_size` check bypassed for f32 data
- Inconsistent validation coverage between f64 and f32 encoding paths creates false confidence
- GPU kernels may process malformed batches leading to incorrect results or resource issues

**Remediation – Add generic or f32 validation:**
```rust
impl Preprocessor {
    /// Validates input constraints for f32 batch processing.
    pub fn validate_batch_f32(
        batch_data: &[f32],
        num_samples: usize,
        sample_size: usize,
        num_qubits: usize,
    ) -> Result<()> {
        if num_samples == 0 {
            return Err(MahoutError::InvalidInput(
                "num_samples must be greater than 0".to_string(),
            ));
        }
        if batch_data.len() != num_samples * sample_size {
            return Err(MahoutError::InvalidInput(format!(
                "Batch data length {} doesn't match num_samples {} * sample_size {}",
                batch_data.len(), num_samples, sample_size
            )));
        }
        validate_qubit_count(num_qubits)?;
        let state_len = 1usize << num_qubits;
        if sample_size > state_len {
            return Err(MahoutError::InvalidInput(format!(
                "Sample size {} exceeds state vector size {}",
                sample_size, state_len
            )));
        }
        Ok(())
    }
}
```

## Positive Security Patterns

### 1. Comprehensive Circuit Lifecycle Enforcement (Python)
`_ensure_circuit_initialized()` is consistently called as the first operation in **every** gate application, execution, and visualization method. This prevents null-circuit operations with 100% coverage across 17 public methods. This is an exemplary implementation of sequential flow enforcement.

### 2. Multi-Parameter Qubit Index Validation
Methods accepting multiple qubit indices (CNOT, Toffoli, SWAP, CSWAP) validate **each** index independently with descriptive parameter names, preventing partial validation where one index is checked but another is not.

### 3. Unbound Parameter Detection Before Execution
Both `execute_circuit()` and `get_final_state_vector()` check for unbound parameters and raise descriptive errors listing exactly which parameters need values. This prevents executing parameterized circuits in an incomplete state.

### 4. Multi-Layer Numerical Validation (Rust + CUDA)
The validation pipeline checks for NaN, Infinity, and zero values at both the Rust preprocessing level (`calculate_l2_norm`, `calculate_batch_l2_norms`) and the CUDA kernel level (`check_finite_batch_kernel_f32`). Defense-in-depth across language boundaries.

### 5. GPU Kernel Sequential Integrity
`launch_check_finite_batch_f32` encapsulates its entire flow in a single C function: null-check → memset → kernel launch → error propagation. Steps cannot be skipped because they're not separately callable.

### 6. Resource Limit Constants
`MAX_QUBITS = 30` and `MAX_GRID_BLOCKS` provide hard caps that prevent resource exhaustion regardless of input values, acting as defense-in-depth backstops.

### 7. Encoder Factory with Allowlist
`get_encoder()` uses an explicit match on known encoder names, rejecting unknown values with a descriptive error. This prevents arbitrary encoder instantiation.

## Architecture Observations

1. **Strategy Pattern vs. Template Method Pattern**: The `QuantumEncoder` trait uses a Strategy pattern where each encoder implements `encode()` independently. This provides flexibility but sacrifices flow enforcement. A Template Method pattern (where a non-overridable `encode()` calls `validate_input()` then `encode_impl()`) would structurally enforce the validate-then-encode sequence. This is the root cause of Finding ASVS-231-MED-001.

2. **Type-Split Validation Paths**: The f64 path has comprehensive Rust-level validation (`validate_input`, `validate_batch`, `calculate_l2_norm`), but the f32 path relies entirely on encoder implementations and the CUDA kernel. This asymmetry means adding new f32 encoders is riskier than f64 encoders.

3. **Python Layer Exemplary, Rust Layer Structural Gap**: The Python `QuMat` class demonstrates excellent sequential flow enforcement with its `_ensure_circuit_initialized` pattern. The Rust `QuantumEncoder` trait would benefit from adopting a similar structural enforcement approach.

4. **Unsafe GPU Pointer Paths**: The `encode_*_from_gpu_ptr` methods necessarily bypass Rust-level validation since data resides on GPU. The CUDA validation kernel provides a backstop for finite-value checking, but dimensional validation (data length, sample size consistency) cannot be enforced at the GPU level. These paths inherently have weaker sequential guarantees.
