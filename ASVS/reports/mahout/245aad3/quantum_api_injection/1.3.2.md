## Findings

### Medium Severity

#### ASVS-132-MED-001: Dynamic Module Import Without Allowlist

| Attribute | Value |
|-----------|-------|
| **Severity** | MEDIUM |
| **Gap Type** | Type A — Entry point with no control |
| **File** | `qumat/qumat.py` |
| **Function** | `QuMat.__init__` |
| **Lines** | 84-87 |
| **Related Functions** | `import_module()` |

**Description**

`importlib.import_module()` is a dynamic code execution feature — importing a Python module executes its top-level code. While the import is constrained to relative imports within the `qumat` package (`package="qumat"`), the `backend_name` value is user-supplied and only validated for *existence* (line 75-79), not for *allowed values*.

This creates two risk vectors:

1. **Unintended module loading:** If any additional `*_backend.py` file exists in the `qumat` package directory (e.g., from a supply-chain compromise, a development artifact, or a namespace package collision), an attacker controlling the configuration can force its import and execution.

2. **Relative import traversal:** A `backend_name` containing dots (e.g., `"..sibling_package.module"`) produces a module path like `"...sibling_package.module_backend"` — the additional dots cause Python's import system to traverse up the package hierarchy, potentially loading modules outside the `qumat` package.

**Vulnerable Code**

```python
self.backend_module: ModuleType = import_module(
    f".{self.backend_name}_backend", package="qumat"
)
```

**Data Flow**

```
Source: backend_config["backend_name"] (user-supplied configuration dict)
  → Interpolated into: f".{self.backend_name}_backend"
  → Sink: importlib.import_module() (dynamic code execution)
  → Missing Control: No allowlist validation of backend_name
```

**Proof of Concept**

```python
# Scenario 1: If qumat/debug_backend.py accidentally exists in the package
config = {
    "backend_name": "debug",  # loads qumat/debug_backend.py
    "backend_options": {"simulator_type": "default", "shots": 1}
}
qm = QuMat(config)  # Executes all top-level code in debug_backend.py

# Scenario 2: Relative import traversal (if qumat is a subpackage)
config = {
    "backend_name": "..malicious_pkg.payload",
    "backend_options": {"simulator_type": "default", "shots": 1}
}
qm = QuMat(config)  # Attempts: import_module("...malicious_pkg.payload_backend", package="qumat")
```

**Impact**

Arbitrary code execution within the Python process if an attacker can both (a) control the `backend_config` dictionary and (b) ensure a target module exists in the reachable import path. In the intended single-user scientific library context, risk is limited but violates defense-in-depth principles.

**Remediation**

```python
ALLOWED_BACKENDS = frozenset({"qiskit", "cirq", "amazon_braket"})

def __init__(self, backend_config: Mapping[str, Any] | None) -> None:
    # ... existing validation ...
    
    self.backend_name: str = self.backend_config["backend_name"]
    
    # Allowlist validation for dynamic module loading
    if self.backend_name not in ALLOWED_BACKENDS:
        raise ValueError(
            f"Unsupported backend '{self.backend_name}'. "
            f"Allowed backends: {sorted(ALLOWED_BACKENDS)}"
        )
    
    self.backend_module: ModuleType = import_module(
        f".{self.backend_name}_backend", package="qumat"
    )
```

## Positive Security Patterns

1. **No `eval()`/`exec()` usage anywhere in the codebase.** All user-supplied string parameters are converted to safe SDK-specific parameter objects (`qiskit.circuit.Parameter`, `sympy.Symbol`, `braket.circuits.FreeParameter`) — none of which evaluate arbitrary code.

2. **Safe parameterized circuit execution.** All three backends use their SDK's built-in parameter binding mechanisms rather than string interpolation into executable code:
   - Qiskit: `assign_parameters()` with validated `Parameter` objects
   - Cirq: `ParamResolver` with symbol-to-value mapping
   - Braket: `inputs` dict with `FreeParameter` names

3. **Circuit immutability protection.** Both `cirq_backend.py` (`copy.deepcopy(circuit)`, line 126) and `qiskit_backend.py` (`circuit.copy()`, line 133) create working copies before mutation, preventing side effects.

4. **Comprehensive input validation in QuMat.** The `_validate_qubit_index()` method (lines 115-135) checks type, range, and bounds. The `_ensure_circuit_initialized()` method prevents operations on uninitialized state. The `bind_parameters()` method validates parameter names exist before binding.

5. **No string formatting in code-generation paths.** Circuit serialization uses `str()` on SDK objects (e.g., `str(circuit)` in `draw_circuit`), not user-controlled format strings. The f-strings used in `print()` statements for unsupported simulator messages are diagnostic, not code-generative.

6. **No regular expressions with user input.** The codebase uses no regex matching, eliminating the ReDoS risk mentioned in the domain context.

---

## Architecture Observations

1. **Backend abstraction pattern is sound but needs a constraint.** The pattern of loading backend modules dynamically is architecturally clean, but ASVS 1.3.2 requires that dynamic code execution features include sanitization (in this case, an allowlist). The fix is a one-line addition.

2. **Parameter flow is well-isolated.** User-supplied parameter names flow through `_handle_parameter()` → backend-specific parameter constructors → SDK parameter binding. At no point are parameter names or values passed to any evaluation function.

3. **No template engines or expression languages.** The domain context mentions SpEL and template engines — none are present. The codebase avoids string-based code generation entirely, delegating circuit representation to backend SDKs.
