This audit evaluates the QuMat unified quantum circuit API against ASVS 1.2.4 (Injection Prevention). The codebase does not use traditional databases but dynamically constructs and executes quantum circuit representations across Qiskit, Cirq, and Amazon Braket backends. Per the domain context, injection risks map to code injection via eval-style patterns, unsafe string interpolation in code generation, and unvalidated input in dynamic code loading paths.

The codebase demonstrates **strong positive patterns** in its use of parameterized SDK APIs for all quantum operations. **One finding** was identified related to unrestricted dynamic module loading based on user-supplied configuration.

## Findings

### Medium Severity

#### ASVS-124-MED-001: Unrestricted Dynamic Module Loading via User-Supplied Backend Name

| Attribute | Detail |
|-----------|--------|
| **Severity** | **MEDIUM** |
| **Gap Type** | **Type A** – Entry point with NO control |
| **File** | `qumat/qumat.py` |
| **Function** | `QuMat.__init__` |
| **Lines** | 67-69 |
| **Related Functions** | None |

**Vulnerable Code:**
```python
self.backend_name: str = self.backend_config["backend_name"]
self.backend_module: ModuleType = import_module(
    f".{self.backend_name}_backend", package="qumat"
)
```

**Data Flow:**
```
Source: backend_config["backend_name"] (user-supplied configuration dict)
  → String interpolation: f".{self.backend_name}_backend"
  → Sink: importlib.import_module(name, package="qumat")
  → Missing control: No whitelist validation of backend_name
```

**Description:**

The `backend_name` value from user-supplied configuration is used directly in `importlib.import_module()` via string interpolation without validation against a whitelist of known backends (`qiskit`, `cirq`, `amazon_braket`). While the relative import constraint (`package="qumat"`) and `_backend` suffix provide partial mitigation, they do not prevent:

1. **Dot-traversal**: A `backend_name` containing dots (e.g., `"..somepackage.module"`) results in `"...somepackage.module_backend"`, which `import_module` interprets as a multi-level relative import that could navigate outside the `qumat` package.
2. **Loading unintended modules**: Any file matching `*_backend.py` within the `qumat` package (or reachable through relative traversal) could be loaded and its `initialize_backend()` function executed.

**Proof of Concept:**
```python
# Attacker controls the config dictionary
malicious_config = {
    "backend_name": "..attacker_package.payload",  # traverses up from qumat
    "backend_options": {"simulator_type": "default"}
}
# If attacker_package/payload_backend.py exists in the parent package:
# import_module("...attacker_package.payload_backend", package="qumat")
# would load and execute initialize_backend() from attacker's module
qm = QuMat(malicious_config)  # Loads arbitrary module
```

**Impact:**

Arbitrary code execution if an attacker can place a `*_backend.py` module in a reachable package, or if a legitimate package in the Python path happens to have a matching module name. In scientific computing environments with broad package installations, this attack surface is non-trivial.

**Remediation:**
```python
ALLOWED_BACKENDS = frozenset({"qiskit", "cirq", "amazon_braket"})

def __init__(self, backend_config: Mapping[str, Any] | None) -> None:
    # ... existing validation ...
    
    self.backend_name: str = self.backend_config["backend_name"]
    
    # Whitelist validation before dynamic import
    if self.backend_name not in ALLOWED_BACKENDS:
        raise ValueError(
            f"Unsupported backend '{self.backend_name}'. "
            f"Allowed backends: {sorted(ALLOWED_BACKENDS)}"
        )
    
    self.backend_module: ModuleType = import_module(
        f".{self.backend_name}_backend", package="qumat"
    )
```

## Positive Security Patterns

### Pattern 1: Parameterized Gate Parameters (All Backends)

All three backends convert user-supplied string parameter names into typed symbolic objects through SDK-safe constructors rather than string interpolation or evaluation:

| Backend | Constructor | File:Line |
|---------|------------|-----------|
| Amazon Braket | `FreeParameter(angle)` | `amazon_braket_backend.py:152,161,170` |
| Cirq | `sympy.Symbol(angle)` | `cirq_backend.py:148,155,162` |
| Qiskit | `qiskit.circuit.Parameter(angle)` | `qiskit_backend.py:180,187,194` |

**Assessment:** ✅ Equivalent to parameterized queries. User strings create named symbolic placeholders—they are never evaluated as code or interpolated into executable strings.

### Pattern 2: Safe Parameter Binding at Execution

Parameter values are bound through each SDK's safe binding mechanism:

- **Braket**: `backend.run(circuit, inputs={param_name: value})` — SDK-managed parameter resolution (`amazon_braket_backend.py:117`)
- **Cirq**: `cirq.ParamResolver(parameter_values)` with `simulator.run_sweep()` — type-safe resolver (`cirq_backend.py:128-131`)
- **Qiskit**: `transpiled_circuit.assign_parameters(parameter_bindings)` — compile-time binding (`qiskit_backend.py:131`)

**Assessment:** ✅ All backends use their SDK's native parameterized execution rather than string substitution.

### Pattern 3: Typed SDK Method Calls for Gate Operations

All gate operations use typed SDK method calls rather than string-based command construction:
```python
# Braket: circuit.rx(qubit_index, angle)
# Cirq: circuit.append(cirq.rx(param).on(qubit))
# Qiskit: circuit.rx(param, qubit_index)
```

No string concatenation, format strings, or template rendering is used to construct gate operations.

### Pattern 4: No eval/exec/compile Usage

The entire codebase contains **zero instances** of `eval()`, `exec()`, `compile()`, or other dynamic code evaluation functions. This eliminates the primary code injection vector identified in the domain context.

### Pattern 5: Circuit Serialization Uses SDK toString

The `draw_circuit` functions use safe SDK-provided serialization:
```python
# amazon_braket_backend.py:147 - return str(circuit)
# cirq_backend.py:143 - return str(circuit)  
# qiskit_backend.py:175 - return circuit.draw()
```

No custom string formatting or template rendering that could introduce injection.

### Pattern 6: Input Validation for Qubit Indices

`QuMat._validate_qubit_index()` (`qumat.py:98-115`) provides comprehensive bounds checking with type validation, negativity checks, and range validation. This is consistently applied across all 13 gate-application methods.

## Architecture Observations

1. **Consistent parameterization across all backends**: The three-backend architecture applies the same safe pattern—string parameters become symbolic objects, numeric values are passed directly to typed API methods. This is the quantum computing equivalent of parameterized queries.

2. **No custom code generation**: The codebase does not generate backend-specific code strings (e.g., QASM, OpenQASM) from user input. All operations go through SDK method calls, eliminating the string interpolation injection vector mentioned in the domain context.

3. **No regular expressions**: The domain context mentioned ReDoS risk from circuit pattern matching, but no regex usage was found in the audited files.

4. **Mutable configuration state**: `self.backend_config["parameter_values"]` is mutated during `execute_circuit()` and `get_final_state_vector()`. While not an injection issue, this side-effect could cause unexpected behavior if config objects are shared.

## Recommendations Summary

### Immediate Actions (High Priority)

| Priority | Recommendation | Effort | Impact |
|----------|---------------|--------|--------|
| 1 | Add `ALLOWED_BACKENDS` whitelist before `import_module()` call | Low | Eliminates arbitrary module loading risk |
| 2 | Validate `backend_name` contains only alphanumeric characters and underscores (defense-in-depth) | Low | Prevents dot-traversal in import paths |

### Short-term Actions

| Priority | Recommendation | Effort | Impact |
|----------|---------------|--------|--------|
| 3 | Consider making `backend_config` immutable after `__init__` (use `types.MappingProxyType`) | Low | Prevents accidental config mutation side-effects |

### Long-term Actions

| Priority | Recommendation | Effort | Impact |
