## Findings

### Medium Severity

#### ASVS-1531-MED-001: Full Quantum State Vector Returned Without Subsetting Capability

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with no control  

**Location:**
- **File:** `qumat/qumat.py`
- **Function:** `get_final_state_vector()`
- **Lines:** 332–368

**Related Functions Checked:**
- `measure_overlap()` (lines 505-547) — demonstrates correct minimal extraction pattern
- `calculate_prob_zero()` (lines 549-565) — demonstrates correct minimal extraction pattern
- Backend module state vector retrieval functions

**Description:**

The method unconditionally returns the **entire** state vector (2^N complex amplitudes). For N=30 qubits, this is ~8GB of data. The API provides no mechanism to:
- Request amplitudes for specific basis states by index
- Request a partial trace (reduced density matrix for a qubit subset)
- Limit the number of returned amplitudes (e.g., top-k by magnitude)
- Return summary statistics instead of raw vector

This directly matches the domain context: *"The application returns full quantum state vectors rather than minimal subsets, potentially causing network/memory exhaustion."*

Notably, the codebase **demonstrates awareness of the data minimization principle** in other methods — `measure_overlap()` extracts a single float from full circuit results, and `calculate_prob_zero()` returns a single probability. This control pattern exists but is **not applied** to `get_final_state_vector()`, making this a **Type B-adjacent gap** (the minimization pattern exists elsewhere but isn't extended here).

**Vulnerable Code:**
```python
def get_final_state_vector(self) -> Any:
    """Return the final state vector of the quantum circuit.

    The complete quantum state vector after circuit execution,
    representing the full quantum state of all qubits.
    ...
    """
    self._ensure_circuit_initialized()
    # ... parameter binding ...
    self.backend_config["parameter_values"] = bound_parameters
    return self.backend_module.get_final_state_vector(
        self.circuit, self.backend, self.backend_config
    )
```

**Data Flow:**
```
User calls get_final_state_vector()
  → Backend computes full 2^N complex amplitude array
  → Entire array returned to caller without subsetting
  → No mechanism to request specific amplitudes, qubit subset, or size limit
```

**Proof of Concept:**
```python
qm = QuMat({"backend_name": "qiskit", "backend_options": {"simulator_type": "statevector", "shots": 1}})
qm.create_empty_circuit(30)  # 2^30 = 1,073,741,824 amplitudes
# Apply gates...
state = qm.get_final_state_vector()  # Returns ~16GB numpy array (8 bytes × 2 complex components × 2^30)
```

**Impact:**
- Memory exhaustion: caller receives exponentially-sized array with no upper bound
- Network exhaustion if state vector is serialized for transmission (e.g., in a notebook or RPC scenario)
- No way for downstream consumers to request only the data they need

**Remediation:**

Add subsetting parameters while preserving backward compatibility:

```python
def get_final_state_vector(
    self,
    indices: list[int] | None = None,
    top_k: int | None = None,
    qubit_subset: list[int] | None = None,
) -> Any:
    """Return the state vector, optionally filtered to a subset.
    
    :param indices: Specific basis state indices to return.
    :param top_k: Return only the top-k amplitudes by magnitude.
    :param qubit_subset: Return reduced state for these qubits only.
    :returns: Full or filtered state vector.
    """
    self._ensure_circuit_initialized()
    # ... existing parameter binding ...
    
    full_state = self.backend_module.get_final_state_vector(
        self.circuit, self.backend, self.backend_config
    )
    
    if indices is not None:
        return full_state[indices]
    
    if top_k is not None:
        import numpy as np
        magnitudes = np.abs(full_state)
        top_indices = np.argpartition(magnitudes, -top_k)[-top_k:]
        return {int(i): full_state[i] for i in top_indices}
    
    # Default: return full vector (backward compatible)
    return full_state
```

### Low Severity

#### ASVS-1531-LOW-001: Full Measurement Results Returned Without Filtering

**Severity:** LOW  
**Gap Type:** Type A — Entry point with no control  

**Location:**
- **File:** `qumat/qumat.py`
- **Function:** `execute_circuit()`
- **Lines:** 265–310

**Related Functions Checked:**
- `measure_overlap()` (lines 505-547) — demonstrates correct minimal extraction from execution results
- Backend module execution functions

**Description:**

The method returns the complete measurement distribution. While measurement results are naturally compressed (only measured states appear), for circuits approaching uniform distributions, the result dictionary can approach 2^N entries. There is no option to:
- Filter results by minimum count threshold
- Request only the top-k most probable states
- Limit the total number of entries returned

This is rated LOW because measurement results are inherently sparser than state vectors, and `measure_overlap()` demonstrates the correct pattern of extracting minimal data from execution results.

**Vulnerable Code:**
```python
def execute_circuit(
    self, parameter_values: Mapping[str, float] | None = None
) -> Any:
    # ...
    self.backend_config["parameter_values"] = bound_parameters
    return self.backend_module.execute_circuit(
        self.circuit, self.backend, self.backend_config
    )
```

**Data Flow:**
```
User calls execute_circuit()
  → Backend returns full measurement result dictionary (all 2^N possible states)
  → Entire results dict returned without filtering
  → No option to threshold by count, filter specific states, or limit entries
```

**Impact:**
- Potential memory overhead for circuits with near-uniform measurement distributions
- Unnecessary data transmission when only top results are needed
- Less severe than state vector exposure due to natural sparsity of measurement results

**Remediation:**

```python
def execute_circuit(
    self,
    parameter_values: Mapping[str, float] | None = None,
    min_count: int | None = None,
    top_k: int | None = None,
) -> Any:
    """Execute circuit with optional result filtering.
    
    :param parameter_values: Parameter values for parameterized circuits.
    :param min_count: Filter results with counts below this threshold.
    :param top_k: Return only the top-k most frequent measurement outcomes.
    :returns: Full or filtered measurement results.
    """
    # ... existing logic ...
    results = self.backend_module.execute_circuit(
        self.circuit, self.backend, self.backend_config
    )
    
    if isinstance(results, dict):
        if min_count is not None:
            results = {k: v for k, v in results.items() if v >= min_count}
        if top_k is not None:
            results = dict(sorted(results.items(), key=lambda x: x[1], reverse=True)[:top_k])
    
    return results
```

## Architecture Observations

1. **Inconsistent data minimization pattern**: The codebase demonstrates the correct pattern in `measure_overlap()` (extracting a single scalar from full quantum circuit results) but does not extend this principle to the primary data retrieval APIs (`get_final_state_vector`, `execute_circuit`). This suggests awareness of the principle but incomplete application.

2. **Rust layer is well-designed**: All pipeline and preprocessing functions in the Rust layer return minimal data — either `Result<()>` for status or single scalar values. The excessive data return problem is isolated to the Python API layer.

3. **Exponential scaling concern**: The quantum state vector size scales as 2^N, making the data minimization requirement increasingly critical as qubit counts grow. The `validate_qubit_count` function limits qubit counts, providing an implicit upper bound on return sizes, but this limit (referenced from `validate_qubit_count` in encodings) is designed for memory safety, not API data minimization.
