**Key Security Code (line ~56):**
```rust
let mut tensor_proto = crate::tf_proto::tensorflow::TensorProto::decode(buffer)
    .map_err(|e| MahoutError::Io(format!("Failed to parse TensorProto: {}", e)))?;
```

**Analysis:** The `prost` decoder applies its default recursion limit of 100. The `TensorProto` schema contains:
- `dtype` (int32) — scalar field
- `tensor_shape` → `TensorShapeProto` → `repeated Dim` — flat, not recursive
- `tensor_content` (bytes) — opaque blob
- `double_val` (repeated double) — flat array

**Conclusion:** No recursive message nesting exists that could cause exponential expansion (the protobuf analog of XXE billion-laughs attack).

## Positive Security Patterns

### 1. Complete Avoidance of XML Attack Surface

**Description:** The architecture correctly avoids XML entirely, using modern binary serialization formats (Parquet, Arrow IPC, NumPy, Protocol Buffers, PyTorch).

**Security Benefit:** This is the strongest possible mitigation for ASVS 1.5.1 — the XXE attack surface simply doesn't exist.

**Locations:** All readers in `qdp/qdp-core/src/readers/`

---

### 2. Post-Parse Validation in TensorFlow Reader

**Location:** `qdp/qdp-core/src/readers/tensorflow.rs`, lines 56-61

**Implementation:**
```rust
const DT_DOUBLE: i32 = 2;
if tensor_proto.dtype != DT_DOUBLE {
    return Err(MahoutError::InvalidInput(format!(
        "Expected DT_DOUBLE (2), got {}", tensor_proto.dtype
    )));
}
```

**Security Benefit:** Type validation occurs after parsing but before any data extraction or unsafe operations, preventing type confusion attacks.

---

### 3. Integer Overflow Protection Throughout

**Locations:**
- `tensorflow.rs`, lines 73-86
- `arrow_ipc.rs`, lines 83-89, 99-105

**Implementation:**
```rust
// tensorflow.rs
let expected_elems = num_samples.checked_mul(sample_size).ok_or_else(|| ...)?;
let expected_bytes = expected_elems.checked_mul(std::mem::size_of::<f64>()).ok_or_else(|| ...)?;

// arrow_ipc.rs
let new_capacity = current_size.checked_mul(batch.num_rows()).ok_or_else(|| ...)?;
```

**Security Benefit:** Prevents integer overflow vulnerabilities during size calculations, which could lead to buffer overflows or memory corruption.

### 4. Schema Validation Before Data Processing

**Locations:** All readers (`parquet.rs`, `arrow_ipc.rs`, `numpy.rs`, `tensorflow.rs`)

**Security Benefit:** All readers validate schema structure (column count, data types, dimensions) before proceeding with data extraction. This defense-in-depth approach prevents type confusion and malformed data attacks.

**Security Benefit:** Prevents state confusion from multiple deserialization passes and ensures predictable reader lifecycle.

## Architecture Observations

### 1. No XML Attack Surface — Requirement Fully Satisfied

The codebase uses five deserialization formats (Parquet, Arrow IPC, NumPy, TensorFlow protobuf, PyTorch), none of which involve XML. The XXE attack vector is completely absent, fully satisfying ASVS 1.5.1 requirements.

### 2. Protobuf Parser Uses Safe Defaults

The `prost` library's default recursion limit (100) is appropriate for the flat `TensorProto` schema. The schema contains no self-referential message fields that could trigger recursive expansion attacks.

### 3. Third-Party Parser Trust Model

The code delegates parsing to well-maintained Rust crates:
- `arrow-rs` (Parquet, Arrow IPC)
- `prost` (Protocol Buffers)
- `ndarray-npy` (NumPy)

These libraries benefit from Rust's memory safety guarantees, reducing the risk of buffer overflow or memory corruption during deserialization — unlike C/C++ XML parsers where XXE and related vulnerabilities historically occur.

### 4. Resource Exhaustion — Outside ASVS 1.5.1 Scope

Multiple TODO comments note missing OOM protection. While this is a valid resource exhaustion concern, it falls under availability (ASVS 10.x / DoS prevention) rather than XXE prevention (ASVS 1.5.1). This represents an intentional design decision for scientific computing workloads.
