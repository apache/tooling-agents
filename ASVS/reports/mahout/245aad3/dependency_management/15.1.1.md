Analysis of the source code reveals an extensive dependency surface — spanning Python packages, Rust crates, native GPU runtimes, and scientific computing libraries — with **no evidence of documented remediation policies, SBOM generation, or vulnerability tracking mechanisms**. ## Findings

### HIGH Severity

#### Finding ASVS-1511-HIGH-001: No Documented Remediation Timeframes for Vulnerable Third-Party Components

**Severity:** HIGH  
**Gap Type:** Type A — Required control absent entirely  
**ASVS Requirement:** 15.1.1  
**Location:** Project-wide (evidenced across both `qdp/qdp-core/src/lib.rs` and `qdp/qdp-python/qumat_qdp/__init__.py`)

**Description:**

The application integrates a broad spectrum of third-party dependencies with no documented policy for remediating vulnerabilities. This includes components performing dangerous operations (deserialization, binary parsing, FFI) and risky components (rapidly evolving quantum SDKs).

**Evidence from Code:**

**Rust (`lib.rs`):**
```rust
// Line 56 — External crate dependency
use cudarc::driver::CudaDevice;

// Lines 245-250, 270-275, 295-300, 320-325 — Format-specific I/O delegates

crate::io::read_numpy_batch(path)?          // NumPy format parser (pickle-adjacent deserialization)
crate::io::read_torch_batch(path)?          // PyTorch format parser (pickle-based deserialization)
crate::io::read_tensorflow_batch(path)?     // TensorFlow protobuf parser (prost crate)
```

**Python (`__init__.py`):**
```python
# Lines 26-30 — Dynamic loading of native extension with no version verification
_qdp_mod = get_qdp()
if _qdp_mod is not None:
    RustQdpEngine = getattr(_qdp_mod, "QdpEngine")
    NativeQuantumTensor = getattr(_qdp_mod, "QuantumTensor")
    run_throughput_pipeline_py = getattr(_qdp_mod, "run_throughput_pipeline_py", None)
```

**Related Functions Checked:**
- `encode_from_parquet` (line 226)
- `encode_from_arrow_ipc` (line 251)
- `encode_from_numpy` (line 270)
- `encode_from_torch` (line 295)
- `encode_from_tensorflow` (line 320)
- `encode_from_gpu_ptr` (line 350)

**Data Flow:**
External libraries → application logic → GPU processing → user output. A vulnerability in any component (e.g., a protobuf parsing vulnerability in `prost`, a memory corruption in `cudarc`, a pickle deserialization exploit in NumPy/PyTorch format readers) could directly compromise the application.

**What's Missing:**
- No `SECURITY.md`, `DEPENDENCY_POLICY.md`, or equivalent documentation defining:
  - Critical vulnerability remediation SLA (e.g., ≤24 hours for CVSS ≥9.0)
  - High vulnerability remediation SLA (e.g., ≤7 days for CVSS 7.0-8.9)
  - Medium vulnerability remediation SLA (e.g., ≤30 days for CVSS 4.0-6.9)
  - Low vulnerability remediation SLA (e.g., next release cycle)
  - General update cadence for non-vulnerable dependencies
- No evidence of `cargo audit`, `pip-audit`, `safety`, or equivalent tooling configuration
- No CI/CD pipeline configuration for automated dependency scanning

**Impact:**

Without defined remediation timeframes, the team has no obligation or guidance to patch vulnerable dependencies within a reasonable window. Given the inclusion of components with "dangerous functionality" (pickle deserializers, protobuf parsers, raw FFI bindings) and "risky components" (rapidly evolving quantum SDKs), the attack window for known CVEs is unbounded.

**Remediation:**

Create a `docs/DEPENDENCY_SECURITY_POLICY.md` with risk-tiered remediation timeframes:

```markdown
# Third-Party Component Security Policy

## Vulnerability Remediation Timeframes

| Risk Tier | CVSS Score | Remediation SLA | Applies To |
|-----------|-----------|-----------------|------------|
| Critical  | 9.0-10.0  | 48 hours        | All components |
| High      | 7.0-8.9   | 7 calendar days | All components |
| Medium    | 4.0-6.9   | 30 calendar days| All components |
| Low       | 0.1-3.9   | Next release    | All components |

## Component Risk Classification

### Dangerous Functionality Components (elevated priority)
- **PyTorch format reader** (pickle deserialization) — Critical tier SLA
- **NumPy format reader** (pickle-adjacent parsing) — Critical tier SLA
- **TensorFlow protobuf parser** (prost/protobuf) — High tier SLA
- **CUDA FFI bindings** (cudarc, qdp-kernels) — Critical tier SLA

### Risky Components (monitored quarterly)
- Qiskit SDK — rapid release cadence, track breaking changes
- Cirq SDK — rapid release cadence, track breaking changes
- arrow-rs ecosystem (50+ transitive crates) — audit transitive tree quarterly

### System-Level Dependencies
- CUDA Runtime / ROCm — vendor-managed; document minimum supported version
- GPU drivers — track NVIDIA/AMD security bulletins monthly

## General Update Policy
- All dependencies reviewed for updates quarterly
- Pin exact versions in Cargo.lock and requirements.txt
- SBOM generated with each release
```

**Related Functions Checked:**

**What's Missing:**

- No `cargo sbom` or `syft` configuration

**Remediation:**

```yaml
# .github/workflows/sbom.yml
name: Generate SBOM
on: [push, release]
jobs:
  sbom:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Generate Rust SBOM
        run: |
          cargo install cargo-cyclonedx
          cd qdp/qdp-core && cargo cyclonedx --format json
      - name: Generate Python SBOM
        run: |
          pip install cyclonedx-bom
          cd qdp/qdp-python && cyclonedx-py requirements
      - name: Upload SBOM artifacts
        uses: actions/upload-artifact@v4
        with:
          name: sbom
          path: |
            qdp/qdp-core/bom.json
            qdp/qdp-python/bom.json
```

## Positive Security Patterns

1. **GPU pointer validation (`validate_cuda_input_ptr`)** — All unsafe GPU pointer entry points call this validation function before use, checking null, memory type, and device affinity. This is a well-implemented defense-in-depth pattern for the FFI boundary.

2. **Input size validation** — Zero-length checks are consistently applied across batch encoding methods (`num_samples == 0`, `sample_size == 0`, `input_len == 0`).

3. **Result-based error propagation** — The Rust code consistently uses `Result<T>` return types with the `?` operator, ensuring CUDA and I/O errors are not silently ignored.

4. **Graceful backend degradation** — The Python `__init__.py` handles absence of the Rust extension gracefully, setting bindings to `None` rather than crashing.

5. **Conditional compilation** — Platform-specific code is gated behind `#[cfg(target_os = "linux")]` and `#[cfg(feature = "remote-io")]`, reducing attack surface on non-applicable platforms.

6. **Encoding method validation** — The `get_encoder(encoding_method)?` call validates encoding strategy selection, rejecting unknown methods.

## Architecture Observations

1. **Broad dependency surface**: The application is a compute engine bridging Python, Rust, CUDA, and multiple scientific file formats. This creates an unusually large third-party component surface that makes ASVS 15.1.1 compliance particularly important.

2. **Multi-language dependency chains**: Vulnerabilities can exist in Python packages, Rust crates, CUDA runtime, and GPU drivers simultaneously. A single remediation policy must span all ecosystems.

3. **System-level dependencies outside application control**: CUDA and ROCm runtimes are installed system-wide. The remediation policy must address who is responsible for updating these and what minimum versions are acceptable.

4. **Transitive dependency opacity**: The arrow-rs ecosystem's 50+ transitive crates represent a significant blind spot. Without SBOM generation, a vulnerability in a deeply nested crate (e.g., a compression codec used by Parquet) could go unnoticed indefinitely.
