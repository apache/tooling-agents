### High Severity

#### ASVS-1521-SEV-002: No Software Bill of Materials (SBOM) Generated or Maintained

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control  
**Location:** Build pipeline gap

**Description:**

Without an SBOM, there is no authoritative inventory of which component versions are deployed. This prevents any assessment of whether components have breached remediation timeframes, even if such timeframes were defined.

**Related Functions/Components Checked:**
- Build system configuration
- Package metadata
- Deployment artifacts
- Runtime version detection in `qdp/qdp-python/qumat_qdp/__init__.py` (lines 29-36)

**Vulnerable Code:**

Evidence from `qdp/qdp-python/qumat_qdp/__init__.py`:

```python
# Lines 29-36: Dynamic loading of native extension with no version tracking
_qdp_mod = get_qdp()
if _qdp_mod is not None:
    RustQdpEngine = getattr(_qdp_mod, "QdpEngine")
    NativeQuantumTensor = getattr(_qdp_mod, "QuantumTensor")
    run_throughput_pipeline_py = getattr(_qdp_mod, "run_throughput_pipeline_py", None)
```

The Python package loads a native Rust extension dynamically. There is no mechanism to verify the version of the loaded extension, its embedded dependencies, or whether those dependencies contain known vulnerabilities.

Evidence from `qdp/qdp-core/src/lib.rs`:

```rust
// Module declarations reference multiple dependency ecosystems
pub mod io;           // Arrow, Parquet, NumPy, PyTorch, TensorFlow readers
pub mod gpu;          // cudarc, CUDA runtime
pub mod tf_proto;     // prost (protobuf)
#[cfg(feature = "remote-io")]
pub mod remote;       // object_store (S3/GCS clients, HTTP stack)
```

**Data Flow:**

External ecosystem (crates.io, PyPI, system packages) → Build system → Binary → Deployment — **no SBOM generated at any stage**.

**Proof of Concept:**

When a CVE is disclosed (e.g., in `arrow-rs`, `prost`, or `object_store`), there is no way to quickly determine whether the deployed build contains the affected version, leading to delayed or missed remediation.

**Impact:**

Without an SBOM:
- Vulnerability impact assessment requires manual code inspection
- Incident response time increases significantly
- Compliance audits cannot verify component versions
- Supply chain risk management is impossible

**Remediation:**

For Rust dependencies, integrate SBOM generation into the build:

```toml
# Cargo.toml - add cargo-cyclonedx to build pipeline
[package.metadata.scripts]
sbom = "cargo cyclonedx --format json --output-file sbom.json"
```

For Python, generate SBOM at packaging time:

```bash
# In CI pipeline
pip install cyclonedx-bom
cyclonedx-py environment -o python-sbom.json --format json
```

For system dependencies (CUDA/ROCm):

```python
# Add to package metadata or runtime check
import subprocess
def get_cuda_version():
    result = subprocess.run(["nvcc", "--version"], capture_output=True, text=True)
    # Parse and include in SBOM
```

#### ASVS-1521-SEV-003: No Automated Vulnerability Scanning for Dependencies

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control  
**Location:** CI/CD pipeline gap

**Description:**

The domain context confirms *"No automated vulnerability scanning or dependency update policy exists."* Without scanning, vulnerabilities in the broad dependency surface are only discovered reactively (if at all).

**Related Functions/Components Checked:**
- CI/CD pipeline configuration
- Pre-commit hooks
- Build-time security checks
- Dependency declarations in `qdp/qdp-core/src/lib.rs` (lines 27-34, 57, 500-508)

**Vulnerable Code:**

Evidence — High-risk dependency surface in `qdp/qdp-core/src/lib.rs`:

The `tf_proto` module (line 33) implies protobuf parsing via `prost`, which has had historical vulnerabilities. The `io` module (line 27) includes readers for formats with known deserialization risks:

```rust
// Line 33: TensorFlow protobuf definitions (prost-generated)
pub mod tf_proto;

// Lines 27-29: File format readers with complex parsing logic
pub mod io;
pub mod reader;
pub mod readers;
```

The `remote` module (line 34) adds an HTTP/cloud client stack when enabled:

```rust
#[cfg(feature = "remote-io")]
pub mod remote;  // object_store crate: S3, GCS, Azure clients + HTTP stack
```

**Data Flow:**

Known CVE published → Security advisory database → **No scanning** → Vulnerable code remains in production

**Proof of Concept:**

Known CVEs in dependencies could be present in production builds without detection. For example:
- A buffer overflow in `arrow-rs` Parquet parsing could enable code execution when processing malicious Parquet files
- A vulnerability in `prost` could be exploited via crafted TensorFlow protobuf files
- An HTTP-level vulnerability in `object_store`'s dependency chain could affect remote I/O

**Impact:**

Without automated scanning:
- Zero-day vulnerabilities may be discovered but not remediated
- Public CVEs remain undetected until manual review
- No audit trail of vulnerability exposure duration
- Compliance with remediation timeframes is impossible to verify

**Remediation:**

Add to CI pipeline:

```yaml
# .github/workflows/security.yml
jobs:
  dependency-audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      # Rust dependency audit
      - name: Install cargo-audit
        run: cargo install cargo-audit
      - name: Audit Rust dependencies
        run: cargo audit --json > rust-audit.json
      
      # Rust dependency advisories (broader coverage)
      - name: Install cargo-deny
        run: cargo install cargo-deny
      - name: Check dependencies
        run: cargo deny check advisories bans licenses sources
      
      # Python dependency audit
      - name: Audit Python dependencies
        run: |
          pip install pip-audit safety
          pip-audit --format json > python-audit.json
          safety check --json > safety-report.json
      
      # Fail build on HIGH/CRITICAL findings
      - name: Check results
        run: |
          python scripts/check_audit_results.py \
            --rust rust-audit.json \
            --python python-audit.json \
            --max-severity medium
```

## Positive Security Patterns

1. **Input validation before unsafe operations:** `validate_cuda_input_ptr` (lines 63-107 in `lib.rs`) performs null checks, memory type verification, and device ownership validation before any pointer dereference — this is a strong pattern for FFI safety.

2. **Graceful backend degradation:** The Python `__init__.py` (lines 29-36) safely handles absence of the native extension without crashing, using `getattr` with `None` fallback — good defensive coding.

3. **Error propagation with context:** All CUDA operations propagate errors with human-readable context via `MahoutError`, including CUDA error code translation — assists in debugging without exposing sensitive internals.

4. **Compile-time feature gating:** `#[cfg(feature = "remote-io")]` (line 34) ensures the remote I/O dependency surface is only compiled when needed, reducing attack surface for builds that don't require cloud connectivity.

5. **Empty input validation:** Consistent zero-length checks across all GPU pointer entry points prevent undefined behavior from empty buffers reaching kernel launches.

## Architecture Observations

1. **Three-ecosystem dependency surface:** The project spans Rust crates (Cargo), Python packages (pip/PyPI), and system-level native libraries (CUDA/ROCm). Each ecosystem has different vulnerability databases and tooling, requiring a multi-pronged scanning approach.

2. **Rapidly evolving quantum SDKs:** Qiskit and Cirq have frequent breaking changes, creating tension between staying current (for security patches) and maintaining stability. A documented update cadence would help manage this tradeoff.

3. **System-level dependencies outside application control:** CUDA and ROCm runtimes are managed at the OS/driver level. The application should at minimum document minimum required versions and check at startup.

4. **Feature-gated attack surface:** The `remote-io` feature adds an HTTP client stack (via `object_store`). Builds without this feature have a meaningfully smaller dependency surface.
