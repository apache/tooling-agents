## ASVS-2.2.2: Server-Side Input Validation

### Findings for 2.2.2

No findings detected.

This codebase is a Python library (not a client-server application). All validation present in `qumat.py` executes within the library itself — there is no client-side/server-side split where client validation could be bypassed. The QDP architecture processes data through Python → PyO3 → Rust → CUDA, with validation at the Rust core layer being the authoritative check, which is the correct trusted service layer.

### Security Controls Inventory for 2.2.2

| Control | Location | Layer | Status |
|---------|----------|-------|--------|
| QuMat input validation | `qumat/qumat.py` | Library (trusted) | ✅ All validation at library level |
| QDP Rust encoder validation | `qdp-core` (per docs) | Native extension (trusted) | ✅ Authoritative validation in Rust |
| CUDA kernel input checks | `qdp-kernels` (per docs) | GPU (trusted) | ✅ Final processing layer |

### Positive Patterns for 2.2.2

- **No client-side-only validation:** All validation in `qumat.py` runs in the same process as the computation — there is no untrusted client layer.
- **Defense-in-depth in QDP:** Validation occurs at both the Python API layer and the Rust core layer, with the Rust layer being authoritative. Even if Python validation is bypassed (e.g., calling `_qdp` directly), the Rust core validates inputs before GPU execution.
- **`_validate_qubit_index`** is called at the library entry point for every gate operation, not deferred to backend modules.

## Cross-cutting Architecture Observations

1. **Validation Asymmetry Between QDP and QuMat:** The QDP (Rust) layer has comprehensive, well-documented input validation (num_qubits 1–30, NaN/Inf rejection, type enforcement, encoding method validation). The QuMat (Python) layer has strong qubit index validation but significant gaps for other parameters (`num_qubits`, rotation angles, configuration structure). This creates a false sense of security — the Python layer appears validated but has exploitable gaps before inputs reach the Rust layer.

2. **Mutable Shared State Pattern:** `self.backend_config` serves as both a configuration store and a runtime communication channel (via injected `parameter_values`). This pattern creates implicit coupling between operations and makes reasoning about state across sequential calls difficult.

3. **Defense-in-Depth Present but Incomplete:** The architecture has validation at multiple layers (Python API → Rust core → CUDA kernels), but the Python layer validation has gaps that could cause confusing failures or resource exhaustion before inputs reach the Rust validation layer.

4. **Library vs. Service Context:** The codebase is a library, not a web service. Many ASVS web-focused requirements (HTTP headers, session management) are not directly applicable. However, input validation and defensive coding requirements remain fully relevant.

5. **Documentation Quality Split:** QDP documentation is thorough with explicit validation rules, supported formats, and error conditions. QuMat documentation covers API usage but lacks formal validation rule specifications.

## Cross-cutting Recommendations

### Immediate (address before next release)
1. **Add `num_qubits` validation to `create_empty_circuit`** — type check (int), non-negative, and reasonable upper bound. This is the highest-impact gap as it guards all subsequent operations. *(ASVS-221-HIGH-001)*
2. **Reset `self.parameters` in `create_empty_circuit`** — prevents stale parameter injection across circuit lifecycles. Single-line fix with high correctness impact. *(ASVS-231-MED-001)*

### Short-term (next sprint)
3. **Add angle/float validation helper** (`_validate_angle`) and apply to all rotation gate methods (`apply_rx/ry/rz_gate`, `apply_u_gate`). Check for type correctness and finiteness (no NaN/Inf). *(ASVS-221-MED-001)*
4. **Validate `backend_options` structure** in constructor — must be a `dict`, optionally validate expected keys. Add `backend_name` allow-list. *(ASVS-221-MED-002)*
5. **Stop mutating `self.backend_config`** — pass execution-specific data via a separate dict or shallow copy to avoid state leakage between calls. *(ASVS-231-LOW-001, ASVS-1531-MED-001)*
6. **Document validation rules** for QuMat class inputs in a structured format matching the QDP documentation standard. *(ASVS-211-MED-001)*

### Long-term (architectural)
7. **Create a `CircuitConfig` or `ExecutionContext` dataclass** that separates initialization config from runtime state, replacing the mutable dict pattern.
8. **Scope backend function signatures** to accept only the fields each operation needs, rather than a full configuration dict.
9. **Consider a validation decorator or mixin** that centralizes input validation for all public methods, reducing the risk of future methods being added without validation.
