## ASVS-15.2.1: Security Architecture and Dependencies — Component Currency

### Findings for 15.2.1

#### [MEDIUM]
**Finding ID:** ASVS-1521-MED-001
**Title:** Unable to verify component currency without documented remediation timeframes

**File location:** All Rust source files (dependency usage); `dev/release.md` (release process)

**Vulnerable code quote:**
```rust
// qdp/qdp-core/src/gpu/memory.rs - uses cudarc
use cudarc::driver::{CudaDevice, CudaSlice, DevicePtr};

// qdp/qdp-core/src/error.rs - uses thiserror
use thiserror::Error;

// qdp/qdp-core/src/gpu/memory.rs - uses qdp_kernels (internal CUDA bindings)
use qdp_kernels::{CuComplex, CuDoubleComplex};
```

**Data flow:** N/A — this is a process/verification gap

**Impact:** Without ASVS 15.1.1 compliance (documented timeframes), compliance with 15.2.1 is structurally impossible to verify. The following risks exist:
- Dependencies may contain known CVEs without a mechanism to detect or track them
- No `Cargo.lock` or `requirements.txt` freeze file was provided for audit, preventing version verification
- The `qdp_kernels` crate (likely internal) contains unsafe CUDA FFI that requires careful version management
- Parquet/Arrow file parsers handle untrusted input and are a common source of vulnerabilities

**Remediation:**
1. Implement the policy from ASVS-1511-MED-001
2. Add automated dependency scanning to CI:

```yaml
# Example GitHub Actions workflow
name: Dependency Audit
on: [push, pull_request]
jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Rust dependency audit
        run: |
          cargo install cargo-audit
          cargo audit
      - name: Python dependency audit
        run: |
          pip install pip-audit
          pip-audit -r requirements.txt
```

3. Include `Cargo.lock` in the repository for reproducible builds and auditability
4. Add a dependency review step to the release process in `dev/release.md`:

```markdown
### 1.0 Pre-Release Dependency Audit
Before cutting an RC:
1. Run `cargo audit` and resolve all findings above the threshold
2. Run `pip-audit` for Python dependencies
3. Verify no dependencies exceed their remediation timeframe
4. Generate SBOM: `cargo sbom > sbom.json`
```

### Positive Patterns for 15.2.1

1. **Minimal dependency surface**: The Rust core uses relatively few direct dependencies (`cudarc`, `thiserror`, `qdp_kernels`), reducing the attack surface compared to projects with large dependency trees.
2. **Conditional compilation**: Platform-specific code is behind `#[cfg(target_os = "linux")]` and feature flags, meaning unused dependencies are not compiled.
3. **Internal kernel crate**: `qdp_kernels` appears to be an internal crate, giving the project full control over its security posture rather than depending on an external FFI binding.

## Cross-cutting Architecture Observations

1. **Library vs. Web Application**: The audited codebase is a GPU computing library (Rust + Python), not a web application. Several ASVS requirements (12.2.2, 14.3.1) have limited applicability. The most impactful findings are in the documentation/process domain (15.1.1, 15.2.1).

2. **Unsafe Code Boundary**: The codebase contains significant `unsafe` Rust code for CUDA FFI operations. While the code demonstrates disciplined unsafe usage with safety documentation, pointer validation, and RAII wrappers, this represents a "dangerous functionality" category per ASVS 15.1.1 that should be formally documented.

3. **File Format Parsers as Attack Surface**: The library accepts untrusted file inputs in multiple formats (Parquet, Arrow IPC, NumPy, PyTorch, TensorFlow protobuf). These parsers process binary data from potentially untrusted sources and represent the highest-risk attack surface. The parsing is delegated to well-known libraries (Arrow, etc.), but vulnerability tracking is essential.

4. **Remote IO Feature**: The `remote-io` feature enables cloud storage access, introducing network-level security requirements (TLS, authentication). The implementation was not provided for audit, creating a significant blind spot.

5. **Credential Management in Release Process**: The `dev/release.md` correctly advises storing `.pypirc` in `~/.pypirc` with `chmod 600` and warns against placing it in the project directory — a solid operational security practice.

## Cross-cutting Recommendations

### Immediate (0–30 days)
1. **Create dependency vulnerability management policy** (ASVS-1511-MED-001): Define remediation timeframes for Critical/High/Medium/Low severity vulnerabilities in a `SECURITY.md` or equivalent document.
2. **Add `cargo audit` to CI pipeline** (ASVS-1521-MED-001): Implement automated Rust dependency vulnerability scanning on every PR and scheduled weekly runs.
3. **Document dangerous components**: Formally identify `qdp_kernels`, `cudarc`, and file format parsers as "dangerous functionality" per ASVS 15.1.1 definitions.

### Short-term (30–90 days)
4. **Audit the `remote` module** (ASVS-1222-MED-001): Verify TLS certificate validation for S3/GCS connections. Ensure publicly trusted certificates are enforced and no bypass exists.
5. **Add deployment hardening documentation** (ASVS-1341-MED-001): If QDP or its documentation is deployed as a web service, document exclusion of `.git` and other VCS metadata.
6. **Add Python dependency auditing**: Implement `pip-audit` or `safety` checks for the Python package surface.

### Long-term (90+ days)
7. **SBOM generation**: Integrate Software Bill of Materials generation into the release process for both Rust (`cargo sbom`) and Python (`cyclonedx-py`) components.
8. **Sanitize paths in error messages** (ASVS-1421-LOW-001): Implement path redaction for remote URLs in error messages to prevent information leakage of internal bucket/key names.
9. **Sandboxing for file parsers**: Consider containerization or process isolation for file format parsing (Parquet, Arrow, TensorFlow protobuf) to limit blast radius from parser vulnerabilities.
