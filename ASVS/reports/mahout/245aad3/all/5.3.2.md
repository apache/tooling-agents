## ASVS-5.3.2: File Storage — Path Traversal Protection

### Findings for 5.3.2

#### [MEDIUM]

**Finding ID:** ASVS-532-MED-001
**Title:** `path_from_py` accepts user-supplied file paths with no validation or sanitization
**Location:** `qdp/qdp-python/src/loader.rs`, function `path_from_py`, lines 109–113

**Vulnerable code:**
```rust
/// Resolve path from Python str or pathlib.Path (__fspath__).
pub fn path_from_py(path: &Bound<'_, PyAny>) -> PyResult<String> {
    path.extract::<String>().or_else(|_| {
        path.call_method0("__fspath__")
            .and_then(|m| m.extract::<String>())
    })
}
```

**Data flow:**
`Python caller (user-controlled str/pathlib.Path)` → `path_from_py()` → raw `String` → downstream `QdpEngine::encode_from_parquet()`, `encode_from_arrow_ipc()`, `encode_from_numpy()`, `encode_from_torch()`, `encode_from_tensorflow()` → OS-level file open

**Missing control:** No validation or sanitization is applied. Specifically absent are:
- Path traversal sequence rejection (`../`, `..\\`, encoded variants)
- Null byte injection checks
- Allowlist of permitted base directories
- Canonicalization (resolve symlinks, normalize `.`/`..`)
- Scheme validation (no check preventing `file://`, `http://`, etc. when `remote-io` is disabled)

**Entry points that call `path_from_py` (or consume its output):**
From the code inventory, the following `QdpEngine` methods accept a `path: &str` originating from Python input:
| Method | File Format |
|--------|-------------|
| `encode_from_parquet` | `.parquet` |
| `encode_from_arrow_ipc` | `.arrow`, `.feather` |
| `encode_from_numpy` | `.npy` |
| `encode_from_torch` | `.pt`, `.pth` |
| `encode_from_tensorflow` | `.pb` |

None of these downstream consumers perform path validation either (**Type A — entry point with NO control**).

**Proof of concept:**
```python
from qumat_qdp import QuantumDataLoader

# Path traversal — read /etc/passwd (if it happened to be a valid format)
loader = QuantumDataLoader(
    path="../../../../etc/shadow",
    num_qubits=4,
    encoding_method="amplitude",
    batch_size=1
)

# Null byte injection (Python 3 blocks this at OS level, but not all
# downstream Rust calls may)
loader = QuantumDataLoader(
    path="/safe/dir/data.parquet\x00../../etc/shadow",
    ...
)
```

**Impact:**
If this library is integrated into a service (API, notebook server, job scheduler) where file paths originate from untrusted user input, an attacker could:
1. **Path traversal:** Read arbitrary files on the server accessible to the process.
2. **SSRF (with `remote-io` feature):** Supply `s3://` or `gs://` URLs targeting internal infrastructure resources.
3. **Information disclosure:** Exfiltrate sensitive configuration, credentials, or data.

**Remediation:**
```rust
use std::path::{Path, PathBuf};

/// Resolve and validate a user-supplied path, constraining it to an
/// allowed base directory and rejecting traversal attempts.
pub fn path_from_py(path: &Bound<'_, PyAny>, allowed_base: &Path) -> PyResult<String> {
    let raw: String = path.extract::<String>().or_else(|_| {
        path.call_method0("__fspath__")
            .and_then(|m| m.extract::<String>())
    })?;

    // Reject null bytes
    if raw.contains('\0') {
        return Err(pyo3::exceptions::PyValueError::new_err(
            "Path must not contain null bytes",
        ));
    }

    // Canonicalize to resolve symlinks and ../ sequences
    let canonical = allowed_base
        .join(&raw)
        .canonicalize()
        .map_err(|e| pyo3::exceptions::PyValueError::new_err(
            format!("Invalid path '{}': {}", raw, e),
        ))?;

    // Ensure resolved path is still under the allowed base
    if !canonical.starts_with(allowed_base) {
        return Err(pyo3::exceptions::PyValueError::new_err(
            format!("Path '{}' escapes allowed directory", raw),
        ));
    }

    canonical
        .to_str()
        .map(|s| s.to_string())
        .ok_or_else(|| pyo3::exceptions::PyValueError::new_err(
            "Path contains non-UTF-8 characters",
        ))
}
```

### Positive Patterns for 5.3.2

1. **Whitelist of null-handling values:** The `parse_null_handling` function (lines 76–84) demonstrates a proper allowlist pattern for string inputs, rejecting unknown values with a descriptive error. This pattern should be extended to file paths.
2. **Rust's type system:** Using `String` (not `CString`) avoids null-byte-in-middle issues at the Rust/OS boundary for most standard library I/O operations.
3. **Documentation lists supported formats:** The API documentation (`api.md`, `getting-started.md`) clearly specifies allowed file extensions and URL schemes, providing a basis for implementing validation.

## Cross-cutting Architecture Observations

1. **Library vs. Application scope:** The audited code is a native Rust extension library (PyO3) for GPU-accelerated quantum computing, not a web application. Five of six ASVS requirements (6.1.1, 6.2.1–6.2.4) relate to authentication and passwords, which are entirely absent from this codebase. These requirements would apply to any service or application that wraps this library and exposes it to end users.

2. **File I/O surface area is the primary risk:** The sole applicable ASVS requirement (5.3.2) has a genuine gap. The library accepts file paths from Python callers and performs I/O operations (reading Parquet, Arrow, NumPy, PyTorch, TensorFlow files) with **no path validation**. The `path_from_py` function is the single chokepoint where validation should be added.

3. **Input validation is inconsistent:** The codebase demonstrates strong validation for numeric/enum inputs (e.g., `parse_null_handling` rejects unknown values, encoding methods are validated, qubit ranges are checked, GPU pointer validity is verified via `cudaPointerGetAttributes`). However, string-based inputs that map to filesystem or network resources (`path`) receive no validation, creating an asymmetry.

4. **Remote I/O amplifies risk:** When the `remote-io` feature flag is enabled, the library can access `s3://` and `gs://` URLs. Combined with the absence of path/URL validation, this creates an SSRF attack surface if an attacker can control the path input.

5. **Documentation quality is high but does not address security boundaries:** The API documentation thoroughly describes input constraints (data types, shapes, encoding rules) but does not specify any security boundaries for file paths (e.g., "paths must be absolute and within a designated data directory").

## Cross-cutting Recommendations

### Immediate (P0 — before next release)

1. **Add path validation to `path_from_py`:** Implement canonicalization, traversal rejection, and optionally a base-directory constraint. This is the single actionable vulnerability in scope.

2. **Add URL scheme validation when `remote-io` is enabled:** If the path starts with a scheme (`s3://`, `gs://`, `http://`, etc.), validate against an allowlist of permitted schemes and reject unexpected ones (especially `file://`, `http://`, `ftp://`).

### Short-term (next 1–2 sprints)

3. **Document security boundaries for file path inputs:** Add a "Security Considerations" section to the API documentation specifying:
   - Which inputs are trusted vs. potentially untrusted
   - How integrators should sanitize paths before passing them to the library
   - The SSRF risk when `remote-io` is enabled

4. **Add file extension validation:** Enforce that the resolved path ends with one of the documented supported extensions (`.parquet`, `.arrow`, `.feather`, `.npy`, `.pt`, `.pth`, `.pb`) to limit the attack surface.

### Long-term (roadmap)

5. **If the library is ever embedded in a service context:** The wrapping application must implement all ASVS authentication controls (6.x). Consider providing integration guidance or a reference architecture that includes rate limiting, authentication, and authorization layers.

6. **Consider a `SafePath` wrapper type:** Create a validated path type that can only be constructed through the validation function, enforcing the "parse, don't validate" pattern and making it impossible to accidentally use an unvalidated path in I/O operations.
