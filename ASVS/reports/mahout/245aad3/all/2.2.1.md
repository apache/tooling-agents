## ASVS-2.2.1: Input Validation

### Findings for 2.2.1

#### HIGH
**Finding ID:** ASVS-221-HIGH-001
**Location:** `qumat/qumat.py`, `create_empty_circuit()` (lines ~82-85)

**Description:** The `num_qubits` parameter is not validated for type, sign, or upper bound. It is stored directly and used in subsequent range checks for qubit indices. An invalid value (float, negative, extremely large, or non-numeric) produces undefined behavior downstream.

```python
def create_empty_circuit(self, num_qubits: int | None = None) -> None:
    self.num_qubits = num_qubits
    self.circuit = self.backend_module.create_empty_circuit(num_qubits)
```

**Data Flow:** User input (`num_qubits`) → stored as `self.num_qubits` → used in `_validate_qubit_index` comparisons → passed to backend `create_empty_circuit`

**Proof of Concept:**
```python
# Extremely large value — potential resource exhaustion in backend
qm = QuMat(config)
qm.create_empty_circuit(10000000)  # Backend may allocate enormous structures

# Float value — breaks comparison logic
qm.create_empty_circuit(3.7)  # qubit index 3 would pass validation (3 < 3.7)

# Negative value — all qubit indices fail validation
qm.create_empty_circuit(-1)
```

**Impact:** Resource exhaustion (DoS) with large values; logic errors with non-integer types; confusing error messages with unsupported types. The QDP documentation specifies 1–30 as the valid range, but `qumat.py` enforces no upper bound.

**Remediation:**
```python
def create_empty_circuit(self, num_qubits: int | None = None) -> None:
    if num_qubits is not None:
        if not isinstance(num_qubits, int):
            raise TypeError(
                f"num_qubits must be an integer, got {type(num_qubits).__name__}"
            )
        if num_qubits < 0:
            raise ValueError(f"num_qubits must be non-negative, got {num_qubits}")
    self.num_qubits = num_qubits
    self.circuit = self.backend_module.create_empty_circuit(num_qubits)
```

#### MEDIUM
**Finding ID:** ASVS-221-MED-001
**Location:** `qumat/qumat.py`, `apply_rx_gate()` (line ~303), `apply_ry_gate()` (line ~321), `apply_rz_gate()` (line ~339), `apply_u_gate()` (line ~356)

**Description:** When rotation angles are provided as floats, no validation is performed for finiteness (NaN, Inf) or type correctness. While `_handle_parameter` registers string parameter names, float values pass through unchecked. The QDP documentation explicitly requires finite values for similar parameters.

```python
def apply_rx_gate(self, qubit_index: int, angle: float | str) -> None:
    self._ensure_circuit_initialized()
    self._validate_qubit_index(qubit_index)
    self._handle_parameter(angle)  # Only registers strings; floats pass through
    self.backend_module.apply_rx_gate(self.circuit, qubit_index, angle)
```

Similarly, `apply_u_gate` accepts `theta`, `phi`, `lambd` with no validation:
```python
def apply_u_gate(self, qubit_index, theta, phi, lambd):
    self._ensure_circuit_initialized()
    self._validate_qubit_index(qubit_index)
    # No type or finiteness checks on theta, phi, lambd
    self.backend_module.apply_u_gate(self.circuit, qubit_index, theta, phi, lambd)
```

**Data Flow:** User input (angle/theta/phi/lambd) → no validation → passed directly to backend module

**Proof of Concept:**
```python
import math
qm = QuMat(config)
qm.create_empty_circuit(2)
qm.apply_rx_gate(0, float('nan'))   # NaN angle — undefined quantum state
qm.apply_rx_gate(0, float('inf'))   # Infinite angle — undefined behavior
qm.apply_u_gate(0, "not_a_number", 1.0, 1.0)  # String where float expected
```

**Impact:** NaN or Inf values produce mathematically undefined quantum states. Backends may silently produce incorrect results rather than raising errors, leading to data integrity issues in quantum computations.

**Remediation:**
```python
def _validate_angle(self, angle: float, param_name: str = "angle") -> None:
    """Validate that a gate angle is a finite number."""
    if not isinstance(angle, (int, float)):
        raise TypeError(f"{param_name} must be a number, got {type(angle).__name__}")
    if math.isnan(angle) or math.isinf(angle):
        raise ValueError(f"{param_name} must be finite, got {angle}")
```

#### MEDIUM
**Finding ID:** ASVS-221-MED-002
**Location:** `qumat/qumat.py`, `__init__()` (lines ~53-75)

**Description:** The `backend_options` value is checked for existence but not validated for type or structure. The `backend_name` is not validated against an allow list of known backends, relying solely on `import_module` to fail for unknown names.

```python
if "backend_options" not in backend_config:
    raise KeyError(...)
# No validation that backend_options is a dict
# No validation that backend_name is in {"qiskit", "cirq", "amazon_braket"}

self.backend_module: ModuleType = import_module(
    f".{self.backend_name}_backend", package="qumat"
)
```

**Data Flow:** `backend_config` → `backend_name` used in `import_module` f-string → module loaded; `backend_options` stored and used later without structure checks

**Proof of Concept:**
```python
# backend_options not a dict — fails later with confusing error
QuMat({"backend_name": "qiskit", "backend_options": 42})

# Unknown backend — ImportError instead of clear validation error
QuMat({"backend_name": "nonexistent", "backend_options": {}})
```

**Impact:** Unclear error messages for misconfigured backends; implicit validation through ImportError rather than explicit business rule check. The relative import (`package="qumat"`) limits the attack surface of the module loading to within the `qumat` package.

**Remediation:**
```python
_ALLOWED_BACKENDS = frozenset({"qiskit", "cirq", "amazon_braket"})

def __init__(self, backend_config):
    # ... existing dict/key checks ...
    
    if not isinstance(backend_config["backend_options"], dict):
        raise TypeError("backend_options must be a dictionary")
    
    if backend_config["backend_name"] not in self._ALLOWED_BACKENDS:
        raise ValueError(
            f"Unknown backend '{backend_config['backend_name']}'. "
            f"Allowed backends: {sorted(self._ALLOWED_BACKENDS)}"
        )
```

### Positive Patterns for 2.2.1

- **`_validate_qubit_index`** is a centralized, well-implemented validation function that checks type (`isinstance`), sign (non-negative), and range (within circuit bounds). It is **consistently called** from every gate method — no gaps in application.
- **`_ensure_circuit_initialized`** provides a reliable prerequisite check before any operation.
- **Constructor validation** properly checks that `backend_config` is a dictionary with required keys, producing clear error messages.
- **QDP Rust core** (per documentation and inventory) validates `num_qubits` (1–30), data types, NaN/Inf, and encoding-specific constraints at the native layer.

## Cross-cutting Architecture Observations

1. **Validation Asymmetry Between QDP and QuMat:** The QDP (Rust) layer has comprehensive, well-documented input validation (num_qubits 1–30, NaN/Inf rejection, type enforcement, encoding method validation). The QuMat (Python) layer has strong qubit index validation but significant gaps for other parameters (`num_qubits`, rotation angles, configuration structure). This creates a false sense of security — the Python layer appears validated but has exploitable gaps before inputs reach the Rust layer.

2. **Mutable Shared State Pattern:** `self.backend_config` serves as both a configuration store and a runtime communication channel (via injected `parameter_values`). This pattern creates implicit coupling between operations and makes reasoning about state across sequential calls difficult.

3. **Defense-in-Depth Present but Incomplete:** The architecture has validation at multiple layers (Python API → Rust core → CUDA kernels), but the Python layer validation has gaps that could cause confusing failures or resource exhaustion before inputs reach the Rust validation layer.

4. **Library vs. Service Context:** The codebase is a library, not a web service. Many ASVS web-focused requirements (HTTP headers, session management) are not directly applicable. However, input validation and defensive coding requirements remain fully relevant.

5. **Documentation Quality Split:** QDP documentation is thorough with explicit validation rules, supported formats, and error conditions. QuMat documentation covers API usage but lacks formal validation rule specifications.

## Cross-cutting Recommendations

### Immediate (address before next release)
1. **Add `num_qubits` validation to `create_empty_circuit`** — type check (int), non-negative, and reasonable upper bound. This is the highest-impact gap as it guards all subsequent operations. *(ASVS-221-HIGH-001)*
2. **Reset `self.parameters` in `create_empty_circuit`** — prevents stale parameter injection across circuit lifecycles. Single-line fix with high correctness impact. *(ASVS-231-MED-001)*

### Short-term (next sprint)
3. **Add angle/float validation helper** (`_validate_angle`) and apply to all rotation gate methods (`apply_rx/ry/rz_gate`, `apply_u_gate`). Check for type correctness and finiteness (no NaN/Inf). *(ASVS-221-MED-001)*
4. **Validate `backend_options` structure** in constructor — must be a `dict`, optionally validate expected keys. Add `backend_name` allow-list. *(ASVS-221-MED-002)*
5. **Stop mutating `self.backend_config`** — pass execution-specific data via a separate dict or shallow copy to avoid state leakage between calls. *(ASVS-231-LOW-001, ASVS-1531-MED-001)*
6. **Document validation rules** for QuMat class inputs in a structured format matching the QDP documentation standard. *(ASVS-211-MED-001)*

### Long-term (architectural)
7. **Create a `CircuitConfig` or `ExecutionContext` dataclass** that separates initialization config from runtime state, replacing the mutable dict pattern.
8. **Scope backend function signatures** to accept only the fields each operation needs, rather than a full configuration dict.
9. **Consider a validation decorator or mixin** that centralizes input validation for all public methods, reducing the risk of future methods being added without validation.
