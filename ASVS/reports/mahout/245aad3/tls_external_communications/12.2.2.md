The application connects to external cloud services (AWS S3, GCS, AWS Braket) for remote object storage and quantum computing operations. TLS with publicly trusted certificates is enforced **by default** through the underlying libraries (`object_store` crate, `boto3`). However, one critical gap exists in the Rust code: the use of `from_env()` builder methods allows environment variables (`AWS_ALLOW_HTTP`, `GOOGLE_ALLOW_HTTP`) to silently downgrade connections to plain HTTP, bypassing TLS entirely — with no programmatic safeguard to prevent this in the production code path.

## Findings

### Medium Severity

#### ASVS-1222-SEV-001: HTTPS Enforcement Bypassable via Environment Variable in S3/GCS Client Construction

**Severity:** MEDIUM  
**Gap Type:** Type B — Control EXISTS (TLS enforcement by default) but NOT EXPLICITLY CALLED (no `with_allow_http(false)` override after `from_env()`)  
**File:** `qdp/qdp-core/src/remote.rs`  
**Function:** `build_store()`  
**Lines:** 69–93

**Related Functions Checked:**
- `download_to_tempfile()` (transitive)
- `resolve_path()` (transitive)
- `parse_url()` (clean)

**Description:**

The `build_store()` function constructs S3 and GCS clients using builder patterns that read configuration from environment variables via `from_env()`. While TLS is enforced by default in the underlying `object_store` crate, the environment variables `AWS_ALLOW_HTTP` and `GOOGLE_ALLOW_HTTP` can silently downgrade connections to plain HTTP. The production code does not explicitly override this behavior with `.with_allow_http(false)`, leaving the application vulnerable to accidental or malicious TLS downgrade through environment misconfiguration.

**Vulnerable Code:**

```rust
// S3 path (line ~71-81)
fn build_store(scheme: &str, bucket: &str) -> Result<Arc<dyn ObjectStore>> {
    match scheme {
        "s3" => {
            let store = object_store::aws::AmazonS3Builder::from_env()  // ← reads AWS_ALLOW_HTTP
                .with_bucket_name(bucket)
                .build()                                                 // ← no .with_allow_http(false)
                // ...
        }
        "gs" => {
            let store = object_store::gcp::GoogleCloudStorageBuilder::from_env()  // ← reads allow_http env var
                .with_bucket_name(bucket)
                .build()                                                           // ← no .with_allow_http(false)
                // ...
        }
    }
}
```

**Data Flow:**

```
Environment variable AWS_ALLOW_HTTP=true
    → AmazonS3Builder::from_env() reads it
    → Internal config sets allow_http = true
    → .build() creates client that accepts HTTP connections
    → download_to_tempfile() streams data over plain HTTP
    → Quantum circuit data / scientific data transmitted unencrypted
```

**Evidence from Test Configuration (lines ~198-199):**

The test code explicitly documents this bypass pattern:
```rust
/// AWS_ENDPOINT=http://localhost:9123 AWS_REGION=us-east-1 AWS_ALLOW_HTTP=true \
/// cargo test -p qdp-core --features remote-io -- test_download_from_minio
```

This demonstrates that the production `build_store()` function — the same code path used by `resolve_path()` and `download_to_tempfile()` — honors `AWS_ALLOW_HTTP=true` to connect over plain HTTP.

**Proof of Concept:**

An environment misconfiguration (inherited from CI/CD, container base image, or operator error) causes silent TLS downgrade:

```bash
# Production deployment accidentally inherits test settings
export AWS_ALLOW_HTTP=true
export AWS_ENDPOINT=http://proxy.internal:8080

# Application silently connects over HTTP — all data unencrypted
# Attacker on network path can:
# 1. Read AWS credentials from Authorization headers
# 2. Read/modify downloaded scientific data
# 3. Intercept quantum circuit intellectual property
```

**Impact:**
- Complete loss of TLS protection for S3/GCS data transfers
- AWS credentials (SigV4 signatures, access keys) exposed in HTTP headers
- Data integrity of downloaded scientific files cannot be guaranteed
- Quantum circuit intellectual property transmitted in cleartext across potentially untrusted networks (per domain context)

**Remediation:**

Explicitly enforce HTTPS after `from_env()` to override any environment variable misconfiguration:

```rust
fn build_store(scheme: &str, bucket: &str) -> Result<Arc<dyn ObjectStore>> {
    match scheme {
        "s3" => {
            let store = object_store::aws::AmazonS3Builder::from_env()
                .with_bucket_name(bucket)
                .with_allow_http(false)  // ← Enforce HTTPS regardless of env vars
                .build()
                .map_err(|e| {
                    MahoutError::Io(format!(
                        "Failed to build S3 client for bucket '{}': {}",
                        bucket, e
                    ))
                })?;
            Ok(Arc::new(store))
        }
        "gs" => {
            let store = object_store::gcp::GoogleCloudStorageBuilder::from_env()
                .with_bucket_name(bucket)
                .with_allow_http(false)  // ← Enforce HTTPS regardless of env vars
                .build()
                .map_err(|e| {
                    MahoutError::Io(format!(
                        "Failed to build GCS client for bucket '{}': {}",
                        bucket, e
                    ))
                })?;
            Ok(Arc::new(store))
        }
        _ => Err(MahoutError::InvalidInput(format!(
            "Unsupported remote scheme '{}://'. Supported: {}",
            scheme,
            REMOTE_SCHEMES.join(", ")
        ))),
    }
}
```

For testing with MinIO, use a separate builder function or a test-only feature flag:

```rust
#[cfg(test)]
fn build_store_for_test(scheme: &str, bucket: &str) -> Result<Arc<dyn ObjectStore>> {
    // Allow HTTP only in test builds
    match scheme {
        "s3" => {
            let store = object_store::aws::AmazonS3Builder::from_env()
                .with_bucket_name(bucket)
                // from_env() handles AWS_ALLOW_HTTP in test context
                .build()?;
            Ok(Arc::new(store))
        }
        // ...
    }
}
```

## Positive Security Patterns

### 1. Credential Management — Environment Variable Sourcing ✅
Both files source cloud credentials exclusively from environment variables via `from_env()` (Rust) and `boto3.Session()` (Python), following provider best practices. No hardcoded credentials anywhere in the codebase.

**Locations:**
- `remote.rs:71` — `AmazonS3Builder::from_env()`
- `remote.rs:84` — `GoogleCloudStorageBuilder::from_env()`
- `amazon_braket_backend.py:37` — `boto3.Session(region_name=region)`

### 2. No Certificate Verification Bypass ✅
Neither file contains `verify=False` (Python), `danger_accept_invalid_certs` (Rust/reqwest), or any other mechanism to disable TLS certificate validation. Certificate validation is active by default.

### 3. URL Input Validation ✅
`parse_url()` in `remote.rs` (lines 39–61) validates cloud URLs by:
- Rejecting query parameters and fragments
- Requiring non-empty bucket and key components
- Validating URL scheme structure

### 4. Boto3 Default Security Configuration ✅
The `amazon_braket_backend.py` file uses `boto3.Session()` without any parameters that would weaken TLS:
- No `verify=False` on any client creation
- No custom SSL context manipulation
- `AwsSession` wraps `boto3.Session` preserving all security defaults

### 5. Streaming Download with Temp File Cleanup ✅
`download_to_tempfile()` streams data in chunks and uses `NamedTempFile` with RAII-based cleanup, preventing both memory exhaustion and temp file leakage.

## Architecture Observations

1. **In-Process FFI Communication**: Python-to-Rust communication occurs via FFI (PyO3), not over network. This eliminates an entire class of TLS requirements for internal service communication — a positive architectural choice.

2. **Library vs. Service**: This is a computational library, not a network service. It has no external-facing TLS endpoints to configure. The ASVS 12.2.2 requirement applies only to its outbound connections to cloud services.

3. **Trust Store Delegation**: Both the `object_store` crate (via `reqwest`) and `boto3` (via `botocore`/`urllib3`) delegate certificate trust to the operating system's trust store or `webpki-roots`. This means publicly trusted certificates from AWS and GCP will be validated correctly without application-level configuration.

4. **No Certificate Revocation Checking**: As noted in the domain context, no OCSP stapling or CRL checking is configured. While this is a defense-in-depth gap, it is common in library code that delegates TLS to HTTP client libraries and is not a direct violation of ASVS 12.2.2.

5. **No Documented External Service Inventory**: The domain context notes "No documentation defines which external services are trusted or required." Maintaining such an inventory would help verify that all external connections use publicly trusted TLS certificates.

## Recommendations Summary

### Immediate Actions (High Priority)
- Add `.with_allow_http(false)` to both `AmazonS3Builder` and `GoogleCloudStorageBuilder` in `build_store()` (Effort: Minimal - 2 lines; Impact: Prevents accidental HTTP downgrade in production)

### Short-term Actions (Medium Priority)
- Create separate test-only builder function or feature flag for MinIO/local testing that allows HTTP (Effort: Low; Impact: Maintains test functionality while hardening production)

### Long-term Actions (Low Priority)
- Document external service inventory (S3, GCS, Braket endpoints) with expected TLS requirements (Effort: Low; Impact: Supports ongoing audit and monitoring)
- Consider adding TLS minimum version enforcement (TLS 1.2+) via `reqwest` client configuration if the `object_store` crate allows custom HTTP clients (Effort: Medium; Impact: Defense-in-depth against protocol downgrade)
