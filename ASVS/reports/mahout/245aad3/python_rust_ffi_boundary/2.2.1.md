**Overall Assessment**

The codebase demonstrates a generally well-designed multi-layer input validation architecture across the Python-Rust FFI boundary. Qubit count validation, tensor shape/dtype/contiguity checks, NaN/Inf detection, and device consistency verification are implemented systematically. However, the audit identified specific gaps where validation controls are incomplete or inconsistently applied, potentially allowing malformed inputs to reach unsafe Rust code or trigger unhandled panics.

## Findings

### Medium Severity

#### ASVS-221-SEV-001: Missing `sample_size > 0` Validation in `validate_batch` Enables Panic

**Severity:** MEDIUM  
**Gap Type:** Type A (Entry point with no control for this specific input characteristic)

**Location:**
- **File:** `qdp/qdp-core/src/preprocessing.rs`
- **Function:** `Preprocessor::validate_batch()`
- **Lines:** 83-112

**Related Functions Checked:**
- `validate_input()` — correctly checks for empty data
- `calculate_batch_l2_norms()` — panics on `par_chunks(0)`

**Description:**

The `validate_batch` function validates that `num_samples > 0` but does not validate that `sample_size > 0`. When `sample_size == 0`, the validation passes for empty batch data (since `num_samples * 0 == 0`), but the subsequent call to `calculate_batch_l2_norms` invokes `par_chunks(0)`, which panics with "chunk size must be non-zero".

**Vulnerable Code:**

```rust
pub fn validate_batch(
    batch_data: &[f64],
    num_samples: usize,
    sample_size: usize,
    num_qubits: usize,
) -> Result<()> {
    if num_samples == 0 {
        return Err(MahoutError::InvalidInput(
            "num_samples must be greater than 0".to_string(),
        ));
    }

    if batch_data.len() != num_samples * sample_size {
        // When sample_size == 0: num_samples * 0 == 0, matches empty batch_data
        return Err(MahoutError::InvalidInput(format!(...)));
    }
    // ... validation passes for sample_size == 0 with empty data
```

**Data Flow:**

```
Python: engine.encode(np.zeros((5, 0)), num_qubits=2)
  → engine.rs:encode_from_numpy() extracts shape=[5,0], num_samples=5, sample_size=0
    → core_engine.encode_batch(&[], 5, 0, 2, "amplitude")
      → Preprocessor::validate_batch(&[], 5, 0, 2) → PASSES
        → Preprocessor::calculate_batch_l2_norms(&[], 5, 0)
          → batch_data.par_chunks(0) → PANIC ("chunk size must be non-zero")
```

**Proof of Concept:**

```python
import numpy as np
from _qdp import QdpEngine

engine = QdpEngine(device_id=0)
# Zero-width 2D array passes validation, causes Rust panic
engine.encode(np.zeros((5, 0), dtype=np.float64), num_qubits=2)
# Results in PanicException from par_chunks(0) instead of clean validation error
```

**Impact:**

A zero-width NumPy array (which is a valid NumPy construct) passes `validate_batch` and triggers an unhandled panic in `calculate_batch_l2_norms`. While PyO3 catches Rust panics as `PanicException`, this bypasses the intended validation error handling path. The `validate_input` function correctly checks `host_data.is_empty()` for single samples, but the batch equivalent (`sample_size == 0`) is missing.

**Remediation:**

```rust
pub fn validate_batch(
    batch_data: &[f64],
    num_samples: usize,
    sample_size: usize,
    num_qubits: usize,
) -> Result<()> {
    if num_samples == 0 {
        return Err(MahoutError::InvalidInput(
            "num_samples must be greater than 0".to_string(),
        ));
    }

    // Add: validate sample_size > 0 (analogous to empty check in validate_input)
    if sample_size == 0 {
        return Err(MahoutError::InvalidInput(
            "sample_size must be greater than 0".to_string(),
        ));
    }

    if batch_data.len() != num_samples * sample_size {
        // ...
    }
    // ...
}
```

### Low Severity

#### ASVS-221-SEV-002: Unchecked Integer Multiplication in `validate_batch` Could Bypass Size Validation

**Severity:** LOW  
**Gap Type:** Type C variant (Control called but implementation has correctness flaw)

**Location:**
- **File:** `qdp/qdp-core/src/preprocessing.rs`
- **Function:** `Preprocessor::validate_batch()`
- **Line:** 97

**Related Functions Checked:**
- Integer overflow behavior in Rust release builds

**Description:**

In Rust release builds, `num_samples * sample_size` wraps on overflow (no panic). If the product overflows `usize::MAX` and the wrapped result happens to equal `batch_data.len()`, the validation check passes incorrectly.

**Vulnerable Code:**

```rust
if batch_data.len() != num_samples * sample_size {
    return Err(MahoutError::InvalidInput(format!(
        "Batch data length {} doesn't match num_samples {} * sample_size {}",
        batch_data.len(),
        num_samples,
        sample_size
    )));
}
```

**Issue:**

The multiplication `num_samples * sample_size` can silently wrap on overflow in release builds, potentially causing the validation to pass when it should fail.

**Practical Exploitability:**

On a 64-bit system, overflow requires `num_samples * sample_size > 2^64`, meaning > 128 exabytes of f64 data—physically impossible with current hardware when `num_samples` and `sample_size` derive from actual array shapes via the Python API. However, this function is `pub` in `qdp-core` and could be called from other Rust code with inconsistent parameters.

**Impact:**

Theoretical validation bypass in release builds. The flawed check undermines confidence in the validation pipeline, even if not exploitable from the current Python interface.

**Remediation:**

```rust
let expected_len = num_samples.checked_mul(sample_size).ok_or_else(|| {
    MahoutError::InvalidInput(format!(
        "num_samples ({}) * sample_size ({}) overflows usize",
        num_samples, sample_size
    ))
})?;

if batch_data.len() != expected_len {
    return Err(MahoutError::InvalidInput(format!(
        "Batch data length {} doesn't match num_samples {} * sample_size {}",
        batch_data.len(), num_samples, sample_size
    )));
}
```

#### ASVS-221-SEV-003: `run_throughput_pipeline_py` Lacks Boundary Input Validation

**Severity:** LOW  
**Gap Type:** Type A (Entry point with no control at boundary)

**Location:**
- **File:** `qdp/qdp-python/src/lib.rs`
- **Function:** `run_throughput_pipeline_py()`
- **Lines:** 30-55

**Related Functions Checked:**
- `QdpEngine::new()` — validates `precision` and `backend` at boundary for comparison

**Description:**

All user-controllable parameters (`num_qubits`, `batch_size`, `total_batches`, `encoding_method`, `warmup_batches`) are passed directly to the core pipeline without any boundary validation. While the core function may validate internally, this creates a defense-in-depth gap and is inconsistent with the validation practices in `QdpEngine::new()`.

**Vulnerable Code:**

```rust
fn run_throughput_pipeline_py(
    py: Python<'_>,
    device_id: usize,
    num_qubits: u32,
    batch_size: usize,
    total_batches: usize,
    encoding_method: String,
    warmup_batches: usize,
    seed: Option<u64>,
    float32_pipeline: bool,
) -> PyResult<(f64, f64, f64)> {
    let config = qdp_core::PipelineConfig {
        device_id,
        num_qubits,
        batch_size,
        total_batches,
        encoding_method,
        // ... all values passed directly, no validation
    };
    let result = py
        .detach(|| qdp_core::run_throughput_pipeline(&config))
        // ...
```

**Data Flow:**

```
Python: _qdp.run_throughput_pipeline_py(0, 4294967295, 0, 0, "invalid_method")
  → PipelineConfig created with unvalidated values
    → qdp_core::run_throughput_pipeline(&config) — behavior depends on core validation
```

**Proof of Concept:**

```python
from _qdp import run_throughput_pipeline_py

# batch_size=0 could cause division-by-zero in throughput calculations
# num_qubits=4294967295 could cause shift overflow in 1 << num_qubits
# encoding_method="invalid" depends on core validation
run_throughput_pipeline_py(0, 4294967295, 0, 0, "invalid")
```

**Impact:**

The lack of boundary validation creates a defense-in-depth gap. The `num_qubits` type is `u32` here but `usize` in `Preprocessor`, introducing a type inconsistency at the boundary. Parameters like `batch_size=0` could cause division-by-zero in throughput calculations if not validated by the core.

**Remediation:**

```rust
fn run_throughput_pipeline_py(
    py: Python<'_>,
    device_id: usize,
    num_qubits: u32,
    batch_size: usize,
    total_batches: usize,
    encoding_method: String,
    warmup_batches: usize,
    seed: Option<u64>,
    float32_pipeline: bool,
) -> PyResult<(f64, f64, f64)> {
    // Validate at boundary before delegating to core
    if batch_size == 0 {
        return Err(PyRuntimeError::new_err("batch_size must be > 0"));
    }
    if total_batches == 0 {
        return Err(PyRuntimeError::new_err("total_batches must be > 0"));
    }
    if warmup_batches > total_batches {
        return Err(PyRuntimeError::new_err(
            "warmup_batches must be <= total_batches"
        ));
    }
    let valid_methods = ["amplitude", "angle", "basis", "iqp", "iqp-z"];
    if !valid_methods.contains(&encoding_method.to_ascii_lowercase().as_str()) {
        return Err(PyRuntimeError::new_err(format!(
            "Unsupported encoding_method '{}'. Supported: {:?}",
            encoding_method, valid_methods
        )));
    }
    // ... proceed to create config
```

#### ASVS-221-SEV-004: `encoding_method` Validation Not Applied at Boundary for Non-CUDA Paths

**Severity:** LOW  
**Gap Type:** Type B (Control exists but not called at all equivalent entry points)

**Location:**
- **File:** `qdp/qdp-python/src/engine.rs`
- **Functions:** `encode_from_list()` (line 386), `encode_from_numpy()` (line 246), `encode_from_pytorch()` CPU path (line 292), `encode_from_file()` (line 400)

**Control Defined At:**

```rust
// pytorch.rs:128 — validate_cuda_tensor_for_encoding()
if !CUDA_ENCODING_METHODS.contains(&method.as_str()) {
    return Err(PyRuntimeError::new_err(format!(
        "CUDA tensor encoding currently only supports {} methods, got '{}'.",
        format_supported_cuda_encoding_methods(), encoding_method
    )));
}
```

**Control Applied At:** Only the CUDA tensor path in `_encode_from_cuda_tensor` (line 478 via `validate_cuda_tensor_for_encoding`).

**Control NOT Applied At:**

```rust
// encode_from_list (line 396) — no encoding_method validation
let ptr = self.core_engine()?.encode(&vec_data, num_qubits, encoding_method)...

// encode_from_numpy 1D (line 268) — no encoding_method validation  
let ptr = self.core_engine()?.encode(data_slice, num_qubits, encoding_method)...

// encode_from_file (line 412) — no encoding_method validation
self.core_engine()?.encode_from_parquet(path, num_qubits, encoding_method)...
```

**Description:**

While the core engine likely validates `encoding_method` internally, the Python boundary only validates it for CUDA tensor paths. This creates an inconsistency where invalid encoding methods produce different error messages and fail at different layers depending on the input type.

**Impact:**

Inconsistent validation behavior across input types. For defense-in-depth, all paths should validate at the boundary to provide consistent error messages and fail-fast behavior.

**Remediation:**

Add a centralized encoding method validation at the start of `encode_with_core`:

```rust
fn encode_with_core(
    &self,
    data: &Bound<'_, PyAny>,
    num_qubits: usize,
    encoding_method: &str,
) -> PyResult<QuantumTensor> {
    // Validate encoding method at boundary for all paths
    let valid_methods = ["amplitude", "angle", "basis", "iqp", "iqp-z"];
    let method_lower = encoding_method.to_ascii_lowercase();
    if !valid_methods.contains(&method_lower.as_str()) {
        return Err(PyRuntimeError::new_err(format!(
            "Unsupported encoding method '{}'. Supported: {:?}",
            encoding_method, valid_methods
        )));
    }
    // ... existing dispatch logic
```

## Positive Security Patterns

### P1: Multi-Layer Validation Architecture

The codebase implements validation at multiple layers (Python type checking → Rust boundary validation → Preprocessor validation → GPU kernel validation). This defense-in-depth approach is sound and provides multiple opportunities to catch invalid inputs before they reach critical code paths.

### P2: Comprehensive NaN/Inf Detection

`calculate_l2_norm` and `calculate_batch_l2_norms` both check computed norms for NaN, Infinity, and zero. Since any NaN/Inf in input data propagates through the L2 norm computation, this indirectly catches all problematic floating-point values.

### P3: CUDA Tensor Validation Chain

`validate_cuda_tensor_for_encoding` performs thorough validation: encoding method allowlist, dtype-per-method checks, contiguity, non-empty, and device ID matching. The additional DLPack device ID cross-check (`pytorch_device_id != dlpack_info.device_id`) is an excellent defense against inconsistent metadata.

### P4: Single-Consume DLPack Semantics

`QuantumTensor.__dlpack__` enforces single consumption with the `consumed` flag, preventing double-free. The `Drop` implementation correctly checks this flag before calling the deleter.

### P5: Null Pointer Guards

Null pointer checks are consistently applied: `QuantumTensor.ptr.is_null()` in `__dlpack__` and `__dlpack_device__`, `data_ptr == 0` in `extract_cuda_tensor_info`, stream pointer validation in `get_torch_cuda_stream_ptr`.

### P6: CUDA Stream Device Consistency Validation

`get_torch_cuda_stream_ptr` validates that the CUDA stream's device type is "cuda" and that the stream device index matches the tensor device index—preventing cross-device stream misuse.

### P7: File Extension Allowlist

`encode_from_file` uses an explicit extension allowlist (`.parquet`, `.arrow`, `.feather`, `.npy`, `.pt`, `.pth`, `.pb`), rejecting all other formats with a clear error message.

### P8: Constructor Parameter Allowlists

`QdpEngine::new` validates both `precision` and `backend` against explicit allowlists, returning descriptive errors for unsupported values.

## Architecture Observations

### 1. Validation Responsibility Split

The Preprocessor in `qdp-core` handles data-level validation (sizes, norms, NaN/Inf), while `qdp-python` handles type-level validation (tensor dtype, contiguity, device). This separation is logical but creates seams where certain checks may be assumed to happen in the other layer.

### 2. `validate_batch` vs `validate_input` Asymmetry

`validate_input` checks for empty data; `validate_batch` checks for `num_samples == 0` but not `sample_size == 0`. This asymmetry suggests the batch validation was modeled on the single-sample validation but didn't fully translate the equivalent constraints.

### 3. `_num_samples` Unused in `calculate_batch_l2_norms`

The function ignores `_num_samples` and derives the count from `par_chunks`. While safe (it processes whatever data exists), this creates a semantic gap if `validate_batch`'s size check was bypassed—the function would silently process fewer samples than expected.

### 4. Consistent Error Handling

All validation failures return typed `MahoutError::InvalidInput` or `PyRuntimeError`, enabling consistent error handling by callers. Error messages include context (dimensions, expected/actual values), aiding debugging.
