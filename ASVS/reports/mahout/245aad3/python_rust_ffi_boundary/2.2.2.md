# ASVS 2.2.2 Security Audit Report: Input Validation at Trusted Service Layer

## Executive Summary

**Repository:** apache/mahout  
**Audit Date:** May 02, 2026  
**Auditor:** Tooling Agents  
**ASVS Requirement:** 2.2.2 — Verify that the application is designed to enforce input validation at a trusted service layer. While client-side validation improves usability and should be encouraged, it must not be relied upon as a security control.  
**Level:** 1  
**Section:** Input Validation  
**Files Analyzed:** 5 relevant out of 5 total, 0 skipped

### Finding Summary

| Severity | Count |
|----------|-------|
| Critical | 0 |

| Low | 0 |
| **Total** | **2** |

The system implements a multi-layered validation architecture across the Python/Rust/GPU trust boundary that largely satisfies ASVS 2.2.2. The Rust `Preprocessor` module serves as the trusted service layer for input validation, with Python-side validation providing defense-in-depth for CUDA tensor paths. Two findings were identified where the trusted service layer validation has coverage gaps that allow specific classes of invalid data to bypass validation.

## Findings

### Medium Severity

#### Finding 1: Missing `sample_size` Zero Check in `validate_batch` Causes Unhandled Panic

**ID:** ASVS-222-SEV-001  
**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with missing control

**Location:**
- **File:** `qdp/qdp-core/src/preprocessing.rs`
- **Function:** `Preprocessor::validate_batch()`
- **Lines:** 82–107

**Related Functions Checked:**
- `Preprocessor::calculate_batch_l2_norms()` (lines 110–140)
- `encode_from_numpy()` in `engine.rs`
- `core_engine.encode_batch()`

**Description:**

The `validate_batch()` function checks that `num_samples > 0` but fails to validate that `sample_size > 0`. When `sample_size` is zero, the validation passes all checks because:
- `batch_data.len() == num_samples * sample_size` evaluates to `0 == num_samples * 0`, which is true for empty data
- `sample_size > state_len` evaluates to `0 > state_len`, which is false and passes

However, the validated data is then passed to `calculate_batch_l2_norms()`, which calls `batch_data.par_chunks(sample_size)`. The `par_chunks()` method panics unconditionally when the chunk size is zero.

**Vulnerable Code:**

```rust
pub fn validate_batch(
    batch_data: &[f64],
    num_samples: usize,
    sample_size: usize,
    num_qubits: usize,
) -> Result<()> {
    if num_samples == 0 {  // ← checks num_samples
        return Err(MahoutError::InvalidInput(
            "num_samples must be greater than 0".to_string(),
        ));
    }
    // ← NO check for sample_size == 0

    if batch_data.len() != num_samples * sample_size {
        // When sample_size=0: num_samples * 0 = 0, and empty batch_data has len() 0 → passes
        ...
    }
    validate_qubit_count(num_qubits)?;
    let state_len = 1 << num_qubits;
    if sample_size > state_len {
        // When sample_size=0: 0 > state_len is false → passes
        ...
    }
    Ok(())
}
```

**Downstream Impact in `calculate_batch_l2_norms` (lines 110–140):**

```rust
pub fn calculate_batch_l2_norms(
    batch_data: &[f64],
    _num_samples: usize,  // ← unused, no cross-check
    sample_size: usize,
) -> Result<Vec<f64>> {
    batch_data
        .par_chunks(sample_size)  // ← par_chunks(0) PANICS unconditionally
        // ...
}
```

**Data Flow:**

```
Python: np.empty((5, 0), dtype=np.float64)
  → engine.rs: encode_from_numpy() extracts shape[1]=0 as sample_size
    → core_engine.encode_batch([], 5, 0, num_qubits, method)
      → Preprocessor::validate_batch([], 5, 0, num_qubits) → Ok(()) [all checks pass]
        → Preprocessor::calculate_batch_l2_norms([], 5, 0) → PANIC at par_chunks(0)
```

**Proof of Concept:**

```python
import numpy as np
from _qdp import QdpEngine

engine = QdpEngine(device_id=0)
# 2D array with zero-width features → sample_size = 0
data = np.empty((5, 0), dtype=np.float64)
engine.encode(data, num_qubits=2, encoding_method="amplitude")
# → RuntimeError from Rust panic: "chunk_size must be non-zero"
```

**Why CUDA tensors are NOT affected:** `validate_cuda_tensor_for_encoding` (pytorch.rs:201-205) checks `numel == 0`, which catches tensors with any zero dimension. NumPy arrays and CPU PyTorch tensors lack this check at the Python layer.

**Impact:**

Unhandled panic in the Rust trusted service layer. While PyO3 converts panics to Python exceptions (no memory safety issue), the validation function falsely reports the input as valid, violating the principle that the trusted service layer enforces input constraints. This could also cause resource leaks if the panic unwinds through code holding non-RAII resources.

**Remediation:**

```rust
pub fn validate_batch(
    batch_data: &[f64],
    num_samples: usize,
    sample_size: usize,
    num_qubits: usize,
) -> Result<()> {
    if num_samples == 0 {
        return Err(MahoutError::InvalidInput(
            "num_samples must be greater than 0".to_string(),
        ));
    }

    // Add: Prevent zero-size samples (causes panic in par_chunks)
    if sample_size == 0 {
        return Err(MahoutError::InvalidInput(
            "sample_size must be greater than 0".to_string(),
        ));
    }

    // Existing checks continue...
    if batch_data.len() != num_samples * sample_size {
        // ...
    }
    // ...
}
```

#### Finding 2: NaN/Inf Validation Missing from Primary Input Validation Function

**ID:** ASVS-222-SEV-002  
**Severity:** MEDIUM  
**Gap Type:** Type B — Control EXISTS (`calculate_l2_norm`) but NOT APPLIED in the primary validation path (`validate_input`)

**Location:**
- **File:** `qdp/qdp-core/src/preprocessing.rs`
- **Function:** `Preprocessor::validate_input()`
- **Lines:** 32–51

**Related Functions Checked:**
- `Preprocessor::calculate_l2_norm()` (lines 56–79)
- `encode_from_list()` in `engine.rs`
- Angle, IQP, and IQP-Z encoding implementations

**Description:**

The `validate_input()` function performs structural validation (qubit count, empty data check, length bounds) but does not check individual elements for NaN or Infinity values. NaN/Inf detection only occurs in `calculate_l2_norm()`, which is called exclusively for amplitude encoding (which requires normalization). Other encoding methods (angle, IQP, IQP-Z) bypass this check entirely, allowing non-finite values to reach GPU kernels.

**Vulnerable Code:**

```rust
pub fn validate_input(host_data: &[f64], num_qubits: usize) -> Result<()> {
    validate_qubit_count(num_qubits)?;

    if host_data.is_empty() {
        return Err(MahoutError::InvalidInput(
            "Input data cannot be empty".to_string(),
        ));
    }

    let state_len = 1 << num_qubits;
    if host_data.len() > state_len {
        return Err(MahoutError::InvalidInput(format!(
            "Input data length {} exceeds state vector size {}",
            host_data.len(), state_len
        )));
    }

    Ok(())
    // ← NO NaN/Inf check on individual elements
}
```

**NaN/Inf detection exists ONLY in L2 norm computation (lines 56–79):**

```rust
pub fn calculate_l2_norm(host_data: &[f64]) -> Result<f64> {
    let norm_sq: f64 = host_data.par_iter().map(|x| x * x).sum();
    let norm = norm_sq.sqrt();
    // ... NaN/Inf checks on computed norm (not individual elements)
    if norm.is_nan() { return Err(...); }
    if norm.is_infinite() { return Err(...); }
    Ok(norm)
}
```

**Data Flow (for non-normalization encoding paths):**

```
Python: [1.0, float('nan'), 3.0]
  → engine.rs: encode_from_list() extracts Vec<f64>
    → core_engine.encode(data, num_qubits, "angle")
      → Preprocessor::validate_input([1.0, NaN, 3.0], num_qubits) → Ok(())
      → [angle encoding does NOT call calculate_l2_norm]
        → GPU kernel receives NaN values → silent corruption
```

**Proof of Concept:**

```python
import math
from _qdp import QdpEngine

engine = QdpEngine(device_id=0)
data = [1.0, float('nan'), 3.0, 4.0]
# Angle encoding doesn't require normalization → no L2 norm check
result = engine.encode(data, num_qubits=2, encoding_method="angle")
# NaN reaches GPU kernel → sin(NaN) = NaN, cos(NaN) = NaN
# → silently corrupted quantum state vector
```

**Affected encoding methods:**
- `angle`: maps values directly to rotation angles → `sin(NaN)`/`cos(NaN)` = NaN
- `iqp` / `iqp-z`: uses values as circuit parameters → NaN propagates through gate operations
- `amplitude`: NOT affected because L2 norm computation catches NaN/Inf

**Impact:**

Silent data corruption in quantum state vectors for non-amplitude encoding methods. In a scientific computing context, this produces incorrect results without any error indication, potentially invalidating downstream quantum computations or machine learning pipelines. The domain context explicitly lists "NaN/Inf checks" as a key Rust preprocessing responsibility.

**Remediation — Option A (add to `validate_input`):**

```rust
pub fn validate_input(host_data: &[f64], num_qubits: usize) -> Result<()> {
    validate_qubit_count(num_qubits)?;

    if host_data.is_empty() {
        return Err(MahoutError::InvalidInput(
            "Input data cannot be empty".to_string(),
        ));
    }

    // Add: Check for NaN/Inf in input data
    if let Some(idx) = host_data.par_iter().position_any(|x| x.is_nan() || x.is_infinite()) {
        return Err(MahoutError::InvalidInput(format!(
            "Input data contains non-finite value at index {} (NaN or Infinity)",
            idx
        )));
    }

    let state_len = 1 << num_qubits;
    if host_data.len() > state_len {
        return Err(MahoutError::InvalidInput(format!(
            "Input data length {} exceeds state vector size {}",
            host_data.len(), state_len
        )));
    }

    Ok(())
}
```

**Remediation — Option B (add dedicated function and call from all encoding paths):**

```rust
/// Validates that all elements are finite (no NaN or Infinity).
pub fn validate_finite(data: &[f64]) -> Result<()> {
    if let Some(idx) = data.par_iter().position_any(|x| !x.is_finite()) {
        return Err(MahoutError::InvalidInput(format!(
            "Input data contains non-finite value at index {}",
            idx
        )));
    }
    Ok(())
}
```

## Positive Security Patterns

### 1. Correct Trust Boundary Architecture

All entry points in `engine.rs` delegate core validation (qubit counts, data sizes) to the Rust trusted service layer via `core_engine.encode()` / `core_engine.encode_batch()`, rather than relying on Python-side checks as security controls. This directly satisfies ASVS 2.2.2.

### 2. Defense-in-Depth for CUDA Tensors

CUDA tensor paths implement comprehensive Python-layer validation (`validate_cuda_tensor_for_encoding`) as a usability/early-error layer, while still relying on core engine validation as the security control:

```rust
// pytorch.rs:145-233 — validates dtype, contiguity, non-empty, device match, encoding method whitelist
validate_cuda_tensor_for_encoding(data, expected_device_id, encoding_method)?;
```

### 3. DLPack Double-Free Prevention

`QuantumTensor` in `tensor.rs` implements single-consume semantics with explicit tracking:

```rust
if self.consumed {
    return Err(PyRuntimeError::new_err("DLPack tensor already consumed"));
}
self.consumed = true;
```

With proper cleanup in `Drop`:

```rust
if !self.consumed && !self.ptr.is_null() {
    if let Some(deleter) = (*self.ptr).deleter {
        deleter(self.ptr);
    }
}
```

### 4. Device ID Cross-Validation

Multiple independent device ID checks prevent device mismatch:
- Engine vs. tensor device (`validate_cuda_tensor_for_encoding`)
- PyTorch vs. DLPack metadata (`encode_from_pytorch`)
- Stream device vs. tensor device (`get_torch_cuda_stream_ptr`)

### 5. Encoding Method Whitelist

`CUDA_ENCODING_METHODS` constant provides a whitelist for CUDA tensor encoding paths, with method-specific dtype validation (pytorch.rs:162-192).

### 6. Comprehensive L2 Norm Validation

For amplitude encoding, `calculate_l2_norm` and `calculate_batch_l2_norms` effectively detect NaN, Infinity, and zero-norm conditions:

```rust
if norm == 0.0 { return Err(...); }
if norm.is_nan() { return Err(...); }
if norm.is_infinite() { return Err(...); }
```

### 7. Null Pointer Guards

Systematic null pointer checks before unsafe operations:
- `tensor.rs:56-58` — DLPack tensor pointer
- `tensor.rs:96-98` — device info access
- `pytorch.rs:250-254` — CUDA tensor data pointer

### 8. File Format Whitelist

Extension-based format detection in `encode_from_file` acts as a whitelist, rejecting unrecognized extensions with a clear error message.

## Architecture Observations

1. **Validation Architecture Aligns with ASVS 2.2.2:** The design correctly places authoritative validation in the Rust core (trusted service layer) while using Python-layer validation for usability and defense-in-depth. No entry point relies solely on Python-side validation for security-critical checks.

2. **Separation of Structural and Content Validation:** The `Preprocessor` separates structural validation (`validate_input`/`validate_batch`) from content validation (`calculate_l2_norm`/`calculate_batch_l2_norms`). This creates a risk that encoding paths may call structural validation without content validation, as demonstrated in Finding 2.

3. **Unused Parameter Anti-Pattern:** `_num_samples` in `calculate_batch_l2_norms` indicates the function doesn't cross-verify chunk count against expected sample count. While not a security vulnerability, this reduces defense-in-depth within the preprocessing pipeline.

4. **`CudaTensorInfo.data_ptr` Type:** The struct always types `data_ptr` as `*const f64` regardless of actual tensor dtype. The pointer is cast to `*const c_void` before use, so functionality is correct, but the intermediate type is misleading for `int64` (basis encoding) and `float32` paths.

## Recommendations Summary

### Immediate Actions (Priority 1)

1. **Add `sample_size > 0` check to `validate_batch()`** — Prevents panic path reachable from Python with zero-width arrays.
2. **Add per-element NaN/Inf validation to `validate_input()`** — Ensures all encoding paths check for non-finite values at the trusted service layer, regardless of whether L2 norm computation occurs.

### Short-term Improvements (Priority 2)

3. **Add corresponding NaN/Inf check to `validate_batch()`** — For consistency with `validate_input`, batch validation should also check for non-finite values.
4. **Use checked arithmetic in `validate_batch`** — Replace `num_samples * sample_size` with `num_samples.checked_mul(sample_size)` to prevent theoretical integer overflow:
   ```rust
   let expected_len = num_samples.checked_mul(sample_size).ok_or_else(|| {
       MahoutError::InvalidInput("num_samples * sample_size overflows usize".to_string())
   })?;
   if batch_data.len() != expected_len { ... }
   ```

### Long-term Enhancements (Priority 3)

5. **Use `num_samples` parameter in `calculate_batch_l2_norms`** — Add assertion `assert_eq!(batch_data.len(), num_samples * sample_size)` or remove the unused parameter to prevent API misuse.
6. **Type `CudaTensorInfo.data_ptr` as `*const c_void`** — More accurately represents the polymorphic nature of the GPU pointer for different dtypes.
