## Positive Security Patterns

### 1. ✅ HTML-Wrapped Data Delivery (Strong XSSI Protection)

**Location:** `v3/server/templates/vote-on.ezt` — lines containing `STV_CANDIDATES`

```javascript
(function () {
  '''use strict''';
  const STV_CANDIDATES = {
    // ... authorized candidate data ...
  };
  // ... all logic in closure ...
})();
```

**Why it'''s positive:**
- Authorized data is embedded within an HTML page (not a standalone `.js` file), making it impossible to load cross-origin via `<script src>`
- Wrapped in an IIFE (Immediately Invoked Function Expression), keeping variables in a closure even if somehow executed out of context
- Uses `'''use strict'''` mode for additional safety

### 2. ✅ No Dynamic JavaScript Endpoints

**Location:** `v3/server/pages.py` — all routes

**Why it'''s positive:** No endpoint in the application generates `application/javascript` responses containing authorized data. All data-bearing responses are HTML, which cannot be executed as JavaScript cross-origin.

### 3. ✅ Static JS Files Contain No Sensitive Data

**Location:** `v3/server/static/js/steve.js`

```javascript
function showModal() { /* ... */ }
function validateRequiredField() { /* ... */ }
function submitFormWithLoading() { /* ... */ }
```

**Why it'''s positive:** The static JavaScript file contains only generic UI utility functions (modal display, field validation, form submission). No election data, user data, or tokens are present in static JS files.

### 4. ✅ Subresource Integrity on Third-Party Scripts

**Location:** `v3/server/templates/vote-on.ezt`

```html
<script src="/static/js/Sortable.min.js"
        integrity="sha384-DgmC6Xe2bSN2WjTDXzWYbUbxyhNP+NNkGDR/g78pCXV7E7rcVTGxVg0uIVCUUcBc"
        crossorigin="anonymous" referrerpolicy="no-referrer"></script>
```

**Why it'''s positive:** SRI hash verification prevents tampered third-party scripts from executing, providing defense-in-depth against supply chain attacks.

### 5. ✅ CSRF Token Delivered via DOM, Not Script

**Location:** `v3/server/templates/manage.ezt`, `admin.ezt`, `vote-on.ezt`

```html
<input type="hidden" id="csrf-token" value="[csrf_token]">
```

```javascript
const csrfToken = document.getElementById('''csrf-token''').value;
```

**Why it'''s positive:** The CSRF token is placed in the HTML DOM and read by JavaScript at runtime, rather than being assigned in a standalone `.js` file. This prevents token extraction via XSSI.

### 6. ✅ State-Changing POST Endpoints Return No Data

**Location:** `v3/server/pages.py` — `_set_election_date()` (line ~100)

```python
return '''''', 204
```

**Why it'''s positive:** POST endpoints that modify state return empty `204 No Content` responses, eliminating any data leakage vector even if an attacker could trigger a cross-origin POST (which would still be blocked by CORS).

## Architecture Observations

### 1. ⚠️ `api.py` Not Provided for Audit

**Observation:** `ARCHITECTURE.md` references `api.py` as containing "API endpoints (currently minimal)." This file was not provided for analysis.

**Implication:** If `api.py` contains JSON endpoints returning authorized data, those would need separate XSSI assessment to ensure they use appropriate countermeasures (e.g., POST-only access, anti-XSSI prefixes like `)]}'''`, or `X-Content-Type-Options: nosniff`).

### 2. ℹ️ Missing `X-Content-Type-Options: nosniff` Header

**Observation:** No explicit setting of this header was observed in the provided code.

**Implication:** While the HTML content type naturally prevents XSSI, adding `X-Content-Type-Options: nosniff` would provide defense-in-depth against content-type confusion attacks in older browsers.

### 3. ℹ️ Document Serving Endpoint (`/docs/<iid>/<docname>`)

**Observation:** The `serve_doc` endpoint serves uploaded files with content types determined by file extension.

**Analysis:** If an attacker could upload a `.js` file as a document, it would be served with `application/javascript`. However:
- The endpoint requires authentication
- Per-issue authorization (mayvote check) is enforced
- Document content would be election-related, not containing dynamic authorized data structures
- **Risk Level:** Low

### 4. ✅ Template Data-Binding Approach is Inherently XSSI-Safe

**Observation:** The EZT template engine renders authorized data directly into HTML pages server-side.

**Analysis:** This "server-side rendering with inline data" approach is inherently safer against XSSI than "client-side fetch of JSON API" approaches, because the data never exists as a standalone script resource that could be loaded cross-origin.

## Recommendations Summary

### Immediate Actions (0-30 days)
*None required for ASVS 3.5.7 compliance — no vulnerabilities identified.*

### Short-Term Improvements (1-3 months)

1. **Add `X-Content-Type-Options: nosniff` header globally**
   - **Priority:** Medium
   - **Effort:** Low
   - **Benefit:** Defense-in-depth against content-type confusion attacks
   - **Implementation:** Apply to all responses in middleware/framework configuration

2. **Audit `api.py` when available**
   - **Priority:** Medium
   - **Effort:** Low-Medium (depends on endpoint count)
   - **Benefit:** Ensure JSON endpoints include XSSI countermeasures
   - **Verification Points:**
     - Check for non-executable prefixes like `)]}'''` on JSON responses
     - Verify POST-only access for sensitive data endpoints
     - Confirm proper `Content-Type` headers

### Long-Term Enhancements (3-6 months)

3. **Implement Content Security Policy (CSP)**
   - **Priority:** Low-Medium
   - **Effort:** Medium
   - **Benefit:** Additional layer of defense against script injection
   - **Recommendation:** CSP header with `script-src '''self'''` and appropriate nonce/hash for inline scripts

4. **Document the data-delivery architecture decision**
   - **Priority:** Low
   - **Effort:** Low
   - **Benefit:** Prevent future developers from inadvertently introducing XSSI vulnerabilities
   - **Action:** Add architectural decision record (ADR) documenting the intentional pattern of embedding authorized data in HTML templates rather than separate JS files or JSON APIs
