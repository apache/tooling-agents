**All authorized data delivery paths serve HTML, not JavaScript.** This means an attacker cannot use `<script src="https://victim.com/vote-on/EID">` to extract data cross-origin — the browser would fail to parse the HTML as JavaScript due to the `<!DOCTYPE html>` preamble causing an immediate syntax error.

## Positive Security Patterns

### 1. HTML-Wrapped Data Delivery (Strong XSSI Protection)

**Location:** `v3/server/templates/vote-on.ezt` — lines containing `STV_CANDIDATES`

```javascript
(function () {
  '''use strict''';
  const STV_CANDIDATES = {
    // ... authorized candidate data ...
  };
  // ... all logic in closure ...
})();
```

**Why it'''s positive:** Authorized data is:
- Embedded within an HTML page (not a standalone `.js` file), making it impossible to load cross-origin via `<script src>`
- Wrapped in an IIFE (Immediately Invoked Function Expression), keeping variables in a closure even if somehow executed out of context
- Using `'''use strict'''` mode for additional safety

### 2. No Dynamic JavaScript Endpoints

**Location:** `v3/server/pages.py` — all routes

**Why it'''s positive:** No endpoint in the application generates `application/javascript` responses containing authorized data. All data-bearing responses are HTML.

### 3. Static JS Files Contain No Sensitive Data

**Location:** `v3/server/static/js/steve.js`

**Why it'''s positive:** The static JavaScript file contains only generic UI utility functions (modal display, field validation, form submission). No election data, user data, or tokens are present in static JS.

### 4. Subresource Integrity on Third-Party Scripts

**Location:** `v3/server/templates/vote-on.ezt`

```html
<script src="/static/js/Sortable.min.js"
        integrity="sha384-DgmC6Xe2bSN2WjTDXzWYbUbxyhNP+NNkGDR/g78pCXV7E7rcVTGxVg0uIVCUUcBc"
        crossorigin="anonymous" referrerpolicy="no-referrer"></script>
```

**Why it'''s positive:** SRI hash verification prevents tampered third-party scripts from executing.

### 5. CSRF Token Delivered via DOM, Not Script

**Location:** `v3/server/templates/manage.ezt`, `admin.ezt`, `vote-on.ezt`

```html
<input type="hidden" id="csrf-token" value="[csrf_token]">
```

```javascript
const csrfToken = document.getElementById('''csrf-token''').value;
```

**Why it'''s positive:** The CSRF token is placed in the HTML DOM and read by JavaScript at runtime, rather than being assigned in a standalone `.js` file. This prevents token extraction via XSSI.

### 6. State-Changing POST Endpoints Return No Data

**Location:** `v3/server/pages.py` — `_set_election_date()` (line ~100)

```python
return '''''', 204
```

**Why it'''s positive:** POST endpoints that modify state return empty `204 No Content` responses, eliminating any data leakage vector.

## Architecture Observations

### 1. `api.py` Not Provided for Audit

`ARCHITECTURE.md` references `api.py` as containing "API endpoints (currently minimal)." This file was not provided. If it contains JSON endpoints returning authorized data, those would need XSSI assessment (e.g., ensuring they use POST, anti-XSSI prefixes like `)]}'''`, or `X-Content-Type-Options: nosniff`).

### 2. Missing `X-Content-Type-Options: nosniff` Header

No explicit setting of this header was observed in the provided code. While the HTML content type naturally prevents XSSI, adding `X-Content-Type-Options: nosniff` would provide defense-in-depth against content-type confusion attacks in older browsers.

### 3. Document Serving Endpoint (`/docs/<iid>/<docname>`)

The `serve_doc` endpoint serves uploaded files with content types determined by the file extension. If an attacker could upload a `.js` file as a document, it would be served with `application/javascript`. However, the endpoint requires authentication and per-issue authorization (mayvote check), and the document content would be election-related, not containing dynamic authorized data structures.

### 4. Template Data-Binding Approach is Inherently XSSI-Safe

The EZT template engine renders authorized data directly into HTML pages server-side. This "server-side rendering with inline data" approach is inherently safer against XSSI than "client-side fetch of JSON API" approaches, because the data never exists as a standalone script resource.

## Recommendations Summary

### Immediate Actions (0-30 days)

1. **Add `X-Content-Type-Options: nosniff` header** — Apply globally to all responses as defense-in-depth. This prevents browsers from MIME-sniffing responses, which could theoretically lead to content-type confusion.

### Short-term Actions (1-3 months)

2. **Audit `api.py` when available** — If API endpoints exist that return JSON with authorized data, verify they include XSSI countermeasures (e.g., non-executable prefix `)]}'''`, POST-only access, proper `Content-Type` headers).

3. **Consider Content Security Policy** — A CSP header with `script-src '''self'''` (and appropriate nonce/hash for inline scripts) would provide an additional layer of defense against script injection that could lead to data exfiltration.

### Long-term Actions (3-12 months)

4. **Document the data-delivery architecture decision** — The choice to embed authorized data in HTML templates rather than separate JS files or JSON APIs is a sound security decision for XSSI prevention. Document this as an intentional pattern to prevent future developers from inadvertently introducing dynamic JS endpoints with authorized data.
