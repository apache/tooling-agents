## Findings

### Medium Severity

#### ASVS-153-MED-001: STV Vote String Parser Inconsistency Between Submission and Tallying

**Severity:** MEDIUM  
**Gap Type:** Type B — Control (validation/normalization) EXISTS conceptually but is NOT CALLED at the submission entry point  

**Location:**
- Submission: `v3/steve/election.py`, `add_vote()`, lines 200-213
- Tallying: `v3/steve/vtypes/stv.py`, `tally()`, lines 46-63

**Related Functions Checked:**
- `add_vote()` — No vote string validation performed
- `stv.tally()` — Comma-split parser with label validation
- `do_vote_endpoint()` — Extracts raw form data without normalization

**Vulnerable Code (submission path — no parsing):**
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting    # <-- TODO: no validation

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

**Parsing code (tally path — custom comma parser with label validation):**
```python
def tally(votestrings, kv):
    trimmed = [ s for vs in votestrings if (s := vs.strip()) ]
    labelmap = kv['''labelmap''']
    votes = [
        [labelmap[l] for label in vs.split(''',''') if (l := label.strip())]
        for vs in trimmed
    ]
```

**Data Flow:**
1. **Source:** Form field value `vote-{iid}` from `do_vote_endpoint` (`pages.py:321`)
2. **Submission parser:** None — raw string is encrypted and stored without parsing
3. **Tally parser:** Comma-split → whitespace strip → label map lookup (`stv.py:60-63`)
4. **Missing control:** No shared normalization/validation at submission time

**Proof of Concept:**
```http
POST /do-vote/ELECTION_ID HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: session=...

vote-ISSUE_ID=x%2Cy%2Cz
```

Where `x`, `y`, `z` are not valid labels in the issue'''s `labelmap`. The vote is accepted and encrypted, but during tallying, `labelmap[l]` raises `KeyError`, crashing the tally for the entire issue.

Additional malicious inputs:
- `vote-ISSUE_ID=,,,,` — Empty segments filtered silently at tally, creating phantom "voted but blank" records
- `vote-ISSUE_ID=%20a%20,%20b%20` — Whitespace stripped at tally (`label.strip()`) but stored differently, creating parsing ambiguity

**Impact:**
- Invalid votes stored in the database can cause `KeyError` exceptions during tallying, preventing results from being computed
- Whitespace-variant vote strings normalize differently at tally time vs. storage time, violating the principle that the same data type should be parsed consistently
- The explicit `### validate VOTESTRING for ISSUE.TYPE voting` TODO comment confirms this is a known gap

**Remediation:**
Add a shared validation/normalization function called at submission time:

```python
# In stv.py or vtypes/__init__.py
def validate_votestring(votestring: str, kv: dict) -> str:
    """Validate and normalize an STV vote string. Raises ValueError if invalid."""
    labelmap = kv.get('''labelmap''', {})
    labels = [l.strip() for l in votestring.split(''',''') if l.strip()]
    for label in labels:
        if label not in labelmap:
            raise ValueError(f'''Invalid label: {label}''')
    # Return normalized form
    return ''','''.join(labels)

# In election.py add_vote():
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    issue = self.q_get_issue.first_row(iid)
    m = vtypes.vtype_module(issue.type)
    kv = self.json2kv(issue.kv)
    normalized = m.validate_votestring(votestring, kv)  # Parse consistently
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, normalized)
    self.c_add_vote.perform(vote_token, ciphertext)
```

#### ASVS-153-MED-002: Election Date Serialization/Deserialization Inconsistency

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point uses incompatible serialization format with no consistency control

**Location:**
- Write path: `v3/server/pages.py`, `_set_election_date()`, lines 105-127
- Read path 1: `v3/server/pages.py`, `postprocess_election()`, lines 489-494
- Read path 2: `v3/server/bin/tally.py`, `list_elections()` → `select_election()`, lines 79-81

**Related Functions Checked:**
- `_set_election_date()` — Stores date as ISO string
- `postprocess_election()` — Reads date as Unix timestamp
- `do_set_open_at_endpoint()` / `do_set_close_at_endpoint()` — Entry points for date setting
- `tally.py:select_election()` — CLI tool expects Unix timestamp

**Vulnerable Code (write path):**
```python
async def _set_election_date(election, field):
    data = await quart.request.get_json()          # JSON parser
    date_str = data.get('''date''')
    dt = datetime.datetime.fromisoformat(date_str).date()  # Produces datetime.date object
    if field == '''open_at''':
        election.set_open_at(dt)  # Stores datetime.date to DB
```

**Incompatible read path:**
```python
def postprocess_election(e):
    dt_open = e.open_at and datetime.datetime.fromtimestamp(e.open_at)  # Expects numeric Unix timestamp
    e.fmt_open_at = format_datetime(dt_open)
```

```python
# tally.py select_election()
close_str = (
    datetime.datetime.fromtimestamp(election.close_at).strftime('''%Y-%m-%d %H:%M''')  # Expects numeric
    if election.close_at else '''N/A'''
)
```

**Data Flow:**
1. **Source:** JSON request body `{"date": "2024-06-15"}` from authenticated admin
2. **Write parser:** `fromisoformat()` → `datetime.date` object → stored via SQLite (as ISO string `"2024-06-15"`)
3. **Read parser:** `fromtimestamp()` expecting numeric Unix timestamp (e.g., `1718409600`)
4. **Missing control:** No shared serialization format for date fields

**Proof of Concept:**
```http
POST /do-set-open_at/ELECTION_ID HTTP/1.1
Content-Type: application/json
Cookie: session=...

{"date": "2024-06-15"}
```

After this request succeeds, any subsequent request to `/manage/ELECTION_ID` or `/voter` that calls `postprocess_election()` will crash with:
```
TypeError: an integer is required (got type str)
```
because `datetime.datetime.fromtimestamp("2024-06-15")` fails.

**Impact:**
- Server returns 500 errors when displaying elections whose dates were set via the API, causing denial of service for election administration
- The tally CLI tool (`tally.py:81`) similarly fails when listing elections, preventing tallying operations
- If date comparison logic is added for auto-open/close (fields `open_at`/`close_at` suggest this is planned), the type mismatch could cause elections to remain in incorrect states

**Remediation:**
Normalize to Unix timestamp at write time to match all read paths:

```python
async def _set_election_date(election, field):
    data = await quart.request.get_json()
    date_str = data.get('''date''')
    if not date_str:
        quart.abort(400, '''Missing date''')
    try:
        dt = datetime.datetime.fromisoformat(date_str)
    except ValueError:
        quart.abort(400, '''Invalid date format''')
    
    # Convert to Unix timestamp to match fromtimestamp() read paths
    timestamp = int(dt.timestamp())
    
    if field == '''open_at''':
        election.set_open_at(timestamp)
    elif field == '''close_at''':
        election.set_close_at(timestamp)
```

#### ASVS-153-MED-003: Document URL Construction/Parsing Inconsistency

**Severity:** MEDIUM  
**Gap Type:** Type A — URL construction uses no encoding; route handler uses URL decoding; no shared normalization

**Location:**
- URL construction: `v3/server/pages.py`, `rewrite_description()`, lines 50-57
- URL route handler: `v3/server/pages.py`, `serve_doc()`, lines 454-465

**Related Functions Checked:**
- `rewrite_description()` — Regex-based URL construction without encoding
- `serve_doc()` — URL route handler with decoded parameters
- `vote_on()` — Calls `rewrite_description()` for issue display
- `do_add_issue_endpoint()` — Entry point for issue creation with description

**Vulnerable Code (URL construction without encoding):**
```python
def rewrite_description(issue):
    import re
    desc = issue.description
    def repl(match):
        filename = match.group(1)
        return f'''<a href="/docs/{issue.iid}/{filename}">{filename}</a>'''  # No URL encoding
    desc = re.sub(r'''doc:([^\s]+)''', repl, desc)
    issue.description = f'''<pre>{desc}</pre>'''
```

**Route handler (URL-decoded parameters, no validation):**
```python
@APP.get('''/docs/<iid>/<docname>''')
@asfquart.auth.require
async def serve_doc(iid, docname):
    result = await basic_info()
    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)
    ### verify the propriety of DOCNAME.              # <-- TODO: no validation
    return await quart.send_from_directory(DOCSDIR / iid, docname)
```

**Data Flow:**
1. **Source:** Issue description containing `doc:filename` patterns (set by admins via `do_add_issue_endpoint`)
2. **URL construction:** Regex extracts filename → directly interpolated into HTML `href` without `urllib.parse.quote()`
3. **URL parsing:** Quart/Werkzeug URL decoder normalizes percent-encoded characters before route matching
4. **File serving:** `send_from_directory(DOCSDIR / iid, docname)` — `iid` used as directory component without path validation
5. **Missing controls:** URL encoding at construction, docname validation at serving

**Proof of Concept:**

An admin creates an issue with description:
```
Please review doc:report%252e%252e%252f%252e%252f
```

The regex captures `report%252e%252e%252f%252e%252f` and creates a link:
```html
<a href="/docs/ISSUE_IID/report%252e%252e%252f%252e%252f">
```

When clicked, the browser sends the URL. The server URL-decodes once to `report%2e%2e%2f%2e%2f`, and depending on ASGI server double-decode behavior, this could resolve to `report../../`. While `send_from_directory` provides path traversal protection for the `docname` parameter, the `iid` parameter is used directly in path construction (`DOCSDIR / iid`) without validation.

Additionally, filenames containing `#` or `?` in the unencoded URL would be truncated by the browser (interpreted as fragment/query), causing the wrong resource to be requested compared to what was intended.

**Impact:**
- URL parsing inconsistency between construction (no encoding) and consumption (with decoding) creates ambiguity for filenames containing percent-encoded sequences
- The acknowledged `### verify the propriety of DOCNAME` TODO confirms the missing validation layer
- While `send_from_directory` provides baseline protection, the `iid` parameter in the directory path (`DOCSDIR / iid`) is not validated against path traversal, relying solely on the authorization check as an indirect guard

**Remediation:**
```python
import urllib.parse
import re

def rewrite_description(issue):
    desc = issue.description
    def repl(match):
        filename = match.group(1)
        safe_filename = urllib.parse.quote(filename, safe='''''')  # URL-encode
        escaped_display = html_escape(filename)  # HTML-escape display text
        return f'''<a href="/docs/{issue.iid}/{safe_filename}">{escaped_display}</a>'''
    desc = re.sub(r'''doc:([^\s]+)''', repl, desc)
    issue.description = f'''<pre>{desc}</pre>'''

@APP.get('''/docs/<iid>/<docname>''')
@asfquart.auth.require
async def serve_doc(iid, docname):
    # Validate iid format (should match crypto.create_id() pattern)
    if not re.fullmatch(r'''[a-zA-Z0-9_-]+''', iid):
        quart.abort(404)
    # Validate docname (no path traversal, only safe filenames)
    if not re.fullmatch(r'''[a-zA-Z0-9._-]+''', docname):
        quart.abort(404)
    
    result = await basic_info()
    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)
    return await quart.send_from_directory(DOCSDIR / iid, docname)
```

## Architecture Observations

1. **Vote string format is essentially a custom DSL** — STV uses comma-separated labels, YNA presumably uses simple strings like "yes"/"no"/"abstain". Neither format has a formal schema or shared validation layer, creating implicit parser coupling between submission and tallying.

2. **Dual input parsing paths** — The application accepts both `application/x-www-form-urlencoded` (most endpoints) and `application/json` (date setting). While these serve different endpoints, the lack of a consistent input parsing layer means new endpoints could inadvertently mix conventions.

3. **`doc:` link rewriting is an ad-hoc URL construction mechanism** — The regex-based filename extraction and raw string interpolation into HTML/URLs bypasses both URL encoding and HTML escaping libraries, creating a fragile link between content authoring and URL routing.

4. **SQLite type flexibility masks inconsistencies** — SQLite'''s dynamic typing means a column can silently accept both a date string and a numeric timestamp. This masks the write/read inconsistency until the Python code attempts to interpret the value.

## Recommendations Summary

### Immediate Actions (Critical Path)

| Priority | Finding | Effort | Impact |
|----------|---------|--------|--------|
| **1** | ASVS-153-MED-001: Add vote string validation/normalization at submission | Medium | Prevents tally crashes and ensures parser consistency |
| **2** | ASVS-153-MED-002: Normalize date storage to Unix timestamps | Low | Prevents 500 errors on election display |
| **3** | ASVS-153-MED-003: Add URL encoding and docname validation | Low | Eliminates URL parsing ambiguity and path traversal risk |

### Short-term Improvements

- Create a shared `VoteFormat` abstraction per vote type with `validate()`, `normalize()`, and `parse()` methods used at both submission and tally
- Add explicit column type annotations or conversion layer for date/timestamp fields
- Implement comprehensive input validation at all entry points using a centralized validation framework

### Long-term Strategic Enhancements

- Establish a formal data serialization policy documenting the canonical format for each data type
- Create automated tests that verify parser consistency by round-tripping data through all read/write paths
- Consider adopting a schema validation library (e.g., Pydantic) to enforce type consistency at API boundaries
