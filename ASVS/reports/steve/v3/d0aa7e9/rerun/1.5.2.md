## Positive Security Patterns

### 1. Exclusive Use of Safe JSON Deserialization

**Location:** `election.py:301-305`

**Pattern:** The application exclusively uses `json.loads()` and `json.dumps()` for structured data serialization. JSON deserialization is inherently safe as it only produces basic Python types (dict, list, str, int, float, bool, None) and cannot instantiate arbitrary objects or execute code.

```python
# election.py:301-305
def kv2json(self, kv: dict) -> str:
    return json.dumps(kv)

def json2kv(self, js: str) -> dict:
    return json.loads(js)
```

**Security Benefit:** Eliminates entire classes of deserialization vulnerabilities (arbitrary code execution, object injection, POP chains).

---

### 2. No Client-Controlled Object Type Negotiation

**Location:** All deserialization points (`pages.py`, `election.py`)

**Pattern:** The application never allows clients to specify object types or class names during deserialization. Form data produces strings only, and JSON produces basic types only.

**Security Benefit:** Prevents attackers from injecting malicious class names or type specifications that could lead to gadget chain exploitation.

---

### 3. Authenticated Encryption Produces Strings Only

**Location:** `crypto.py:79-82`

**Pattern:** `Fernet.decrypt()` returns raw bytes, which are decoded to UTF-8 strings via `.decode()`. No object reconstruction or complex deserialization occurs after decryption.

```python
# crypto.py:79-82 (conceptual)
decrypted_bytes = fernet.decrypt(ciphertext)
vote_string = decrypted_bytes.decode('''utf-8''')
```

**Security Benefit:** Ensures cryptographic operations don'''t introduce deserialization vulnerabilities in the decryption path.

---

### 4. Hardcoded Module Loading Path

**Location:** `stv.py:23-31`

**Pattern:** Dynamic module loading uses a compile-time constant path (`STV_RELPATH = '''../../../monitoring/stv_tool.py'''`), making it equivalent to a standard Python import rather than a user-controllable deserialization vector.

```python
# stv.py:23
STV_RELPATH = '''../../../monitoring/stv_tool.py'''
# ... later ...
# stv.py:28-31
spec = importlib.util.spec_from_file_location("stv_tool", stv_path)
module = importlib.util.module_from_spec(spec)
spec.loader.exec_module(module)
```

**Security Benefit:** Eliminates risk of arbitrary module/code loading via path traversal or injection attacks.

---

### 5. Type-Constrained Vote Types

**Location:** `election.py:176`

**Pattern:** The `add_issue()` method validates `vtype in vtypes.TYPES` before storage, preventing injection of arbitrary module names when `vtypes.vtype_module(issue.type)` is later called during tallying.

**Security Benefit:** Ensures only allowlisted vote type modules can be loaded, preventing arbitrary code execution via malicious vote type names.

---

### 6. Database STRICT Mode

**Location:** `schema.md`

**Pattern:** SQLite tables use `STRICT` mode with explicit type constraints, preventing type confusion when data is read from the database.

**Security Benefit:** Provides defense-in-depth by ensuring database layer enforces type safety, preventing unexpected data types from reaching deserialization code.

#### 1. Add JSON Schema Validation (Priority: LOW)

**Recommendation:** Consider adding JSON schema validation to `json2kv()` to enforce expected structure.

**Location:** `election.py:305`

**Rationale:** While not a deserialization vulnerability, validating that `kv` contains expected keys (e.g., `labelmap` and `seats` for STV issues) would provide defense-in-depth against malformed data.

**Implementation:**
```python
import jsonschema

ISSUE_KV_SCHEMA = {
    "type": "object",
    "properties": {
        "labelmap": {"type": "object"},
        "seats": {"type": "integer", "minimum": 1}
    },
    "required": ["labelmap", "seats"]
}

def json2kv(self, js: str) -> dict:
    kv = json.loads(js)
    jsonschema.validate(kv, ISSUE_KV_SCHEMA)
    return kv
```

**Impact:** Minimal performance overhead; improves data integrity
