## Positive Security Patterns

1. **Minimal API surface**: The `api.py` file is intentionally minimal, reducing unnecessary attack surface and limiting potential vulnerabilities.

2. **Framework-based routing**: Uses Quart's standard HTTP routing rather than introducing additional API paradigms, maintaining simplicity and reducing complexity-related security risks.

3. **Static folder disabled**: `asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)` — explicit disabling of default static file serving reduces exposure to directory traversal and unauthorized file access vulnerabilities.

4. **Clear architectural separation**: Well-documented architecture with distinct layers (web, API, data) promotes maintainable security controls.

### Long-term Considerations

**Advisory Note for Future Development**: If GraphQL is introduced in the future (e.g., for richer client-side querying or third-party API integration), the following controls should be implemented from the start:

1. **Disable introspection in production**: Configure `introspection=False` or equivalent based on the chosen GraphQL library
2. **Implement query depth limits**: Prevent deeply nested queries that could cause performance degradation or DoS
3. **Apply query complexity limits**: Prevent resource exhaustion from computationally expensive queries
4. **Extend existing authentication**: Apply the same authentication/authorization decorators used on existing routes to GraphQL resolvers
5. **Input validation**: Validate and sanitize all GraphQL input variables using the same patterns as current HTTP endpoints
6. **Rate limiting**: Apply appropriate rate limiting to GraphQL endpoints
7. **Query allowlisting**: Consider implementing persisted queries/query allowlisting for production use
