## Findings

### Medium Severity

#### ASVS-413-MED-001: No Trusted Proxy Configuration or X-Forwarded-* Header Sanitization

**Severity:** MEDIUM

**Location:** 
- **File:** `v3/server/main.py`
- **Lines:** 34–53 (app creation), 78–95 (standalone server), 96–113 (ASGI server startup)

**Related Functions Checked:**
- `create_app()` (lines 34–53)
- `run_standalone()` (lines 78–95)
- `run_asgi()` (lines 96–113)

**Description:**

The application, designed to run behind a reverse proxy via Hypercorn (ASGI), lacks any configuration or middleware to sanitize, validate, or restrict intermediary-set HTTP headers (e.g., `X-Forwarded-For`, `X-Forwarded-Proto`, `X-Forwarded-Host`). While the application reads user identity from server-side sessions rather than headers, the underlying Quart framework and OAuth redirect flow may implicitly trust these spoofable headers.

**Vulnerable Code:**

In `main.py`, the application is created and launched with no proxy header handling:

```python
def create_app():
    # ...
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    import pages  # noqa: F401
    import api  # noqa: F401
    return app
```

```python
def run_asgi():
    # ...
    global steve_app
    steve_app = create_app()
    # No trusted proxy configuration
```

```python
def run_standalone():
    # ...
    app = create_app()
    kwargs = {}
    if app.cfg.server.certfile:
        kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
        kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
    app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
    # No proxy middleware applied
```

**Data Flow:**

1. **Source:** End-user HTTP request includes spoofed `X-Forwarded-For: 10.0.0.1`, `X-Forwarded-Proto: https`, or `X-Forwarded-Host: evil.com`
2. **Sink:** Quart framework's `request.remote_addr`, `request.scheme`, `request.host`, and URL generation functions
3. **Missing Control:** No `ProxyFix` middleware, no Hypercorn `--forwarded-allow-ips` restriction, no explicit header stripping

**Why This Matters for This Application:**

The `run_asgi()` path (lines 96–113) explicitly supports Hypercorn deployment, which is canonically placed behind a reverse proxy (nginx, Apache). In this configuration:

1. **OAuth redirect manipulation:** The OAuth flow configured at line 47 (`redirect_uri=%s`) likely derives the redirect URI from `request.host` or `request.url_root`, which Quart constructs from the `Host` header or `X-Forwarded-Host`. An attacker could inject `X-Forwarded-Host: attacker.com` to redirect OAuth callbacks to a malicious domain.

2. **Audit log integrity:** While user identity is correctly sourced from sessions (positive pattern), if any logging or monitoring infrastructure (e.g., WAF, access logs, rate limiting) relies on `request.remote_addr` derived from `X-Forwarded-For`, the end-user can spoof their apparent IP.

3. **Scheme confusion:** Without validated `X-Forwarded-Proto`, the framework may generate HTTP URLs for resources that should be HTTPS-only, potentially leaking session cookies if `Secure` flag depends on detected scheme.

**Proof of Concept:**

```bash
# Attacker spoofs X-Forwarded-Host to manipulate OAuth redirect
curl -H "X-Forwarded-Host: attacker.com" \
     -H "X-Forwarded-Proto: https" \
     -H "X-Forwarded-For: 192.168.1.1" \
     https://legitimate-steve-server/voter

# If the OAuth flow constructs redirect_uri from request.host,
# the redirect could become: https://attacker.com/oauth/callback
```

**Impact:**
- Potential OAuth redirect hijacking if redirect URIs are constructed from framework-derived host
- Spoofed client IP addresses in any downstream logging/monitoring
- Scheme confusion leading to insecure URL generation

**Remediation:**

For Quart/Hypercorn deployment, configure trusted proxy handling at the ASGI server level and/or within the application:

```python
# Option 1: Hypercorn configuration (hypercorn.toml or CLI)
# Only trust forwarded headers from specific proxy IPs
# --forwarded-allow-ips="127.0.0.1,10.0.0.0/8"

# Option 2: Application-level middleware in main.py
from quart import Quart

def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    # Configure trusted proxy handling
    # Quart uses ProxyFixMiddleware similar to Werkzeug
    from quart.middleware import ProxyFixMiddleware
    app.asgi_app = ProxyFixMiddleware(
        app.asgi_app,
        mode="modern",  # Use modern Forwarded header
        trusted_hops=1,  # Trust only the immediate proxy
    )
    
    import pages
    import api
    return app

# Option 3: Strip dangerous headers in a before_request handler
@APP.before_request
async def strip_untrusted_proxy_headers():
    """Remove proxy headers that should only be set by trusted infrastructure."""
    # If not behind a proxy, strip all forwarded headers
    untrusted_headers = [
        'X-Forwarded-For', 'X-Forwarded-Proto', 'X-Forwarded-Host',
        'X-Real-IP', 'X-User-ID', 'Forwarded'
    ]
    for header in untrusted_headers:
        if header in quart.request.headers:
            # Log and strip — actual implementation depends on framework
            _LOGGER.warning(f'Stripped untrusted header: {header}')
```

## Positive Security Patterns

1. **Session-based identity:** All user identity (`uid`, `fullname`, `email`) is sourced from `asfquart.session.read()` — a server-side session store — rather than from any HTTP header. This means even if `X-User-ID` or similar headers were injected, they would not affect authentication decisions.

2. **No IP-based access control:** The application does not use `request.remote_addr` for any authorization decisions, which reduces (but does not eliminate) the impact of `X-Forwarded-For` spoofing.

3. **Centralized authentication:** All protected endpoints use the `@asfquart.auth.require` decorator consistently, and the auth module reads from the session, not from request headers.

---

## Architecture Observations

1. **Deployment model implies proxy:** The `run_asgi()` function in `main.py` (line 96) with Hypercorn support strongly implies deployment behind a reverse proxy. The comment at line 108 explicitly references Hypercorn usage. This makes the absence of proxy header handling a concrete configuration gap rather than a theoretical concern.

2. **OAuth dependency:** The OAuth URLs configured at lines 47–50 of `main.py` use `redirect_uri` which will be constructed from the request context. Without trusted proxy configuration, this redirect URI derivation is vulnerable to host header manipulation.

3. **Framework opacity:** The `asfquart` wrapper is ASF-specific and its source is not provided. It may or may not have internal proxy handling. However, following defense-in-depth, the application should not rely on hidden framework behavior for a security-critical control.
