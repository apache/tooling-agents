## Findings

### Medium Severity

#### Finding ASVS-421-MED-001

**Title:** No explicit HTTP request body size limits configured, enabling denial-of-service via overly long HTTP messages

**Location:**
- **File:** `v3/server/pages.py` (global configuration gap); `v3/server/main.py` (lines 31-44)
- **Function:** `create_app()`
- **Lines:** 31-44

**Related Functions Checked:**
- `create_app()` in `main.py`
- All POST endpoint handlers in `pages.py`

**Description:**

The Quart application does not set `max_content_length` or configure Hypercorn body size limits. The ASVS 4.2.1 parent section explicitly includes "denial of service via overly long HTTP messages" as an attack vector. Multiple POST endpoints accept unbounded request bodies.

**Affected Entry Points:**

| Endpoint | Body Parser | Line |
|----------|------------|------|
| `POST /do-vote/<eid>` | `quart.request.form` | `pages.py:403` |
| `POST /do-set-open_at/<eid>` | `quart.request.get_json()` | `pages.py:96` |
| `POST /do-set-close_at/<eid>` | `quart.request.get_json()` | `pages.py:96` |
| `POST /do-create-election` | `quart.request.form` | `pages.py:440` |
| `POST /do-add-issue/<eid>` | `quart.request.form` | `pages.py:504` |
| `POST /do-edit-issue/<eid>/<iid>` | `quart.request.form` | `pages.py:531` |

**Vulnerable Code:**

```python
# main.py:31-44 — no max_content_length or body size configuration
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    # No app.config['MAX_CONTENT_LENGTH'] set
    import pages
    import api
    return app
```

**Data Flow:**

```
Client HTTP request (arbitrarily large body)
  → Hypercorn (no configured limit)
  → Quart (no max_content_length)
  → request.form / request.get_json() (unbounded parsing)
  → Memory exhaustion
```

**Proof of Concept:**

```bash
# Send a multi-gigabyte form body to the vote endpoint
python3 -c "
import requests
# Generate a 500MB payload
payload = 'vote-fake=' + 'A' * (500 * 1024 * 1024)
requests.post('https://target/do-vote/some-eid',
    data=payload,
    headers={'Content-Type': 'application/x-www-form-urlencoded'},
    cookies={'session': 'valid_session_cookie'})
"
```

**Impact:**

An authenticated attacker (any committer) can submit arbitrarily large HTTP request bodies that are fully buffered by the framework before reaching handler code. This can exhaust server memory and cause denial of service during an active election, potentially disrupting voting.

**Remediation:**

```python
# In main.py create_app():
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    # Limit request body to 1MB (adjust as needed for election data)
    app.config['MAX_CONTENT_LENGTH'] = 1 * 1024 * 1024  # 1 MB
    
    import pages
    import api
    return app
```

Additionally, configure Hypercorn limits in the ASGI deployment:

```toml
# hypercorn.toml
h11_max_incomplete_size = 16384
h2_max_concurrent_streams = 100
h2_max_header_list_size = 65536
```

#### Finding ASVS-421-MED-002

**Title:** State-changing operations implemented as GET requests increase HTTP request smuggling attack surface

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `do_open_endpoint()`, `do_close_endpoint()`
- **Lines:** 453-470, 475-492

**Related Functions Checked:**
- `do_open_endpoint()` at lines 453-470
- `do_close_endpoint()` at lines 475-492
- `election.open()` in `election.py`
- `election.close()` in `election.py`

**Description:**

Two state-changing operations (`/do-open/<eid>` and `/do-close/<eid>`) are implemented as GET requests. In the context of ASVS 4.2.1, this is significant because GET requests have simpler message boundary determination (no body parsing) and are therefore the easiest payloads to smuggle through a misconfigured proxy/server chain. A smuggled GET request requires only a request line and minimal headers, making successful exploitation more likely if any infrastructure component mishandles message boundaries.

**Gap Classification:** Type B — Authorization control stub EXISTS (`### check authz`) but is NOT CALLED, compounding the smuggling risk by removing the ownership check that would limit impact.

**Vulnerable Code:**

```python
# pages.py:453-470
@APP.get('/do-open/<eid>')       # State-changing operation on GET
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz               # <-- Authorization check not implemented
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)            # <-- Irreversible state change
    ...

# pages.py:475-492
@APP.get('/do-close/<eid>')      # State-changing operation on GET
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz               # <-- Authorization check not implemented  
    election.close()              # <-- Irreversible state change
    ...
```

**Data Flow:**

```
Smuggled GET /do-open/<known-eid>
  → Hypercorn
  → Quart routing
  → @asfquart.auth.require({R.committer}) (passes if session matches any committer)
  → ### check authz (not implemented)
  → election.open(pdb) (election opened by unauthorized committer)
```

**Proof of Concept:**

In an HTTP/1.1 request smuggling scenario (CL-TE or TE-CL disagreement between proxy and Hypercorn):

```http
POST / HTTP/1.1
Host: target
Content-Length: 44
Transfer-Encoding: chunked

0

GET /do-close/known-eid HTTP/1.1
Host: target

```

The smuggled GET request opens/closes an election without a request body, making the smuggled payload minimal and more likely to succeed.

**Impact:**

If HTTP request smuggling is achievable at the infrastructure level (reverse proxy ↔ Hypercorn), any authenticated committer's session could be hijacked to open or close elections they don't own. Because these are GET requests, the smuggled payload is trivially small and doesn't require Content-Type or body handling agreement between components.

**Remediation:**

Convert state-changing operations to POST with CSRF protection:

```python
@APP.post('/do-open/<eid>')  # POST, not GET
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    
    # Verify ownership
    md = election.get_metadata()
    if md.owner_pid != result.uid:
        quart.abort(403, 'Not the election owner')
    
    # Validate CSRF token
    form = await quart.request.form
    validate_csrf_token(form.get('csrf_token'))
    
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    ...
```

## Positive Security Patterns

1. **Framework-delegated HTTP parsing**: All request body parsing uses Quart's built-in `request.form` and `request.get_json()`, which rely on the ASGI server (Hypercorn) for HTTP message boundary determination. No custom HTTP parsing is present, eliminating application-level request smuggling vectors.

2. **Parameterized SQL queries**: All database operations in `election.py` use parameterized statements through `asfpy.db`, preventing SQL injection regardless of how HTTP payloads are parsed.

3. **Safe file serving**: `quart.send_from_directory()` is used for static files and documents, providing built-in path traversal protection at the framework level.

4. **State machine enforcement**: The `Election` class enforces state transitions with assertions (`is_editable()`, `is_open()`, `is_closed()`), providing defense-in-depth even if a smuggled request reaches a handler.

5. **Authentication on all sensitive endpoints**: Every state-changing route has an `@asfquart.auth.require` decorator, meaning a smuggled request must still have a valid session to succeed (though session-fixation through smuggling remains a theoretical risk).

---

## Architecture Observations

1. **No reverse proxy configuration visible**: The codebase shows standalone and ASGI deployment modes but no reverse proxy (nginx, Apache httpd) configuration. ASVS 4.2.1 specifically mentions load balancers and firewalls. **If deployed behind a reverse proxy, the proxy and Hypercorn must be validated for consistent HTTP message boundary handling** — this cannot be verified from the application code alone.

2. **Hypercorn supports HTTP/2 natively**: When run as ASGI via Hypercorn, the server supports both HTTP/1.1 and HTTP/2. Hypercorn's HTTP/2 implementation validates Content-Length against DATA frame lengths per RFC 9113 §8.1.1. However, this is a framework-level guarantee, not an application-level configuration.

3. **`api.py` not provided for review**: The `main.py` imports an `api` module that was not included in this audit. API endpoints may have different HTTP message handling characteristics that should be reviewed separately.

## Recommendations Summary

### Immediate Actions

1. **Set `MAX_CONTENT_LENGTH` on the Quart app** to prevent DoS via oversized bodies
   - **Effort:** Low
   - **Impact:** Prevents memory exhaustion attacks

### Short-term Actions

2. **Convert `GET /do-open/<eid>` and `GET /do-close/<eid>` to POST methods** with CSRF tokens
   - **Effort:** Medium
   - **Impact:** Reduces smuggling attack surface and prevents CSRF

3. **Implement the `### check authz` ownership verification stubs** on all management endpoints
   - **Effort:** Medium
   - **Impact:** Prevents any authenticated committer from modifying arbitrary elections

### Long-term Actions

4. **Document and validate the deployment topology** (proxy ↔ Hypercorn) for consistent HTTP parsing
   - **Effort:** Medium
   - **Impact:** Ensures ASVS 4.2.1 compliance at the infrastructure level

5. **Add explicit Hypercorn configuration** for HTTP parsing limits (`h11_max_incomplete_size`, `h2_max_concurrent_streams`)
   - **Effort:** Low
   - **Impact:** Defense-in-depth for HTTP message handling
