**Key Observations**

ASVS 4.1.5 requires per-message digital signatures for highly sensitive requests or transactions that traverse multiple systems. This is a **Level 3** requirement. The application under audit is an election/voting system where vote integrity and authenticity are paramount. The audit reveals a **complete absence of per-message digital signature mechanisms** across all sensitive operations, representing a fundamental gap in the security architecture.

## Findings

### HIGH SEVERITY

#### Finding ASVS-415-HIGH-001: No Per-Message Digital Signatures on Vote Submission

| Field | Value |
|-------|-------|
| **Severity** | **HIGH** |
| **Finding ID** | ASVS-415-HIGH-001 |
| **File** | `v3/server/pages.py` |
| **Function** | `do_vote_endpoint()` |
| **Lines** | 422-469 |
| **Gap Type** | **Type A** — Entry point with NO control |
| **ASVS Compliance** | ❌ Does not satisfy 4.1.5 |

**Related Functions Checked:**
- `election.add_vote()` (`election.py:229-240`)
- `crypto.create_vote()` (`crypto.py:67-72`)
- `crypto.gen_vote_token()` (`crypto.py:44-54`)

**Description:**

Vote submission, the most sensitive operation in an election system, lacks per-message digital signatures. The endpoint relies solely on session cookie authentication over TLS, with no cryptographic binding between the authenticated voter identity and the vote payload at the application layer.

**Vulnerable Code:**

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    ### check authz  # <-- authorization also incomplete

    # Parse the form data
    form = edict(await quart.request.form)
    # ...
    for iid, votestring in votes.items():
        # ...
        election.add_vote(result.uid, iid, votestring)
```

Downstream processing in `election.py:229-240`:

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."
    md = self._all_metadata(self.S_OPEN)
    ### validate VOTESTRING for ISSUE.TYPE voting  # <-- also not done
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

**Data Flow:**

```
HTTP POST body (form data) → quart.request.form → votes dict → election.add_vote() → crypto.create_vote() → database
                              ↑                                                        ↑
                    No signature verification                              Fernet encrypts, but no
                    on the incoming request                                request-level signature
```

**Impact:**

1. **No non-repudiation**: There is no cryptographic proof binding a specific vote to the voter's authenticated identity at the message level. The system relies solely on session cookies (bearer tokens) over TLS.

2. **Intermediary tampering risk**: If a TLS-terminating proxy, load balancer, or middleware component sits between the voter and the application (common in production deployments), vote content could be observed or modified after TLS decryption and before the application processes it.

3. **Replay without detection**: A captured form submission can be replayed without cryptographic detection at the application layer (the system does allow vote updates, so replays would silently overwrite).

4. **No voter-verifiable receipt**: The voter has no cryptographic proof of what they submitted.

5. **Fails ASVS 4.1.5 requirement**: Vote submission is a highly sensitive transaction that should have additional assurance beyond transport protection.

**Proof of Concept:**

An attacker with access to a reverse proxy or TLS-terminating load balancer between the voter and the application server can:

```
1. Intercept the plaintext POST to /do-vote/<eid>
2. Modify form field "vote-<iid>" from "yes" to "no"
3. Forward the modified request
4. The application processes and encrypts the tampered vote
5. No detection mechanism exists at the application layer
```

**Remediation:**

Implement client-side signing of vote payloads. For a web application, this can use the Web Crypto API:

```python
# Server-side: verify per-message signature on vote submission
from cryptography.hazmat.primitives.asymmetric import ed25519
from cryptography.hazmat.primitives import serialization
from cryptography.exceptions import InvalidSignature
import base64
import json

@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    
    data = await quart.request.get_json()
    vote_payload = data.get('vote_payload')  # canonical JSON of votes
    signature = base64.b64decode(data.get('signature'))
    
    # Retrieve voter's registered public key
    voter_pubkey = get_voter_public_key(result.uid)  # Ed25519 public key
    
    try:
        voter_pubkey.verify(signature, vote_payload.encode())
    except InvalidSignature:
        quart.abort(400, 'Invalid vote signature')
    
    # Parse verified payload and process votes
    votes = json.loads(vote_payload)
    # ... process votes with assurance of authenticity
```

Alternatively, implement JWS (JSON Web Signatures) for the vote payload:

```python
import jose.jws

# Verify JWS-signed vote submission
signed_vote = data.get('signed_vote')
try:
    payload = jose.jws.verify(signed_vote, voter_public_key, algorithms=['EdDSA'])
    votes = json.loads(payload)
except jose.JWSError:
    quart.abort(400, 'Vote signature verification failed')
```

**Client-side implementation** (JavaScript with Web Crypto API):

```javascript
// Generate or retrieve voter's key pair
const keyPair = await window.crypto.subtle.generateKey(
  { name: "ECDSA", namedCurve: "P-256" },
  true,
  ["sign", "verify"]
);

// Create canonical vote payload
const votePayload = JSON.stringify(votes, Object.keys(votes).sort());

// Sign the payload
const signature = await window.crypto.subtle.sign(
  { name: "ECDSA", hash: { name: "SHA-256" } },
  keyPair.privateKey,
  new TextEncoder().encode(votePayload)
);

// Submit signed vote
await fetch(`/do-vote/${eid}`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    vote_payload: votePayload,
    signature: btoa(String.fromCharCode(...new Uint8Array(signature)))
  })
});
```

### MEDIUM SEVERITY

#### Finding ASVS-415-MEDIUM-001: No Per-Message Digital Signatures on Election Lifecycle Transitions

| Field | Value |
|-------|-------|
| **Severity** | **MEDIUM** |
| **Finding ID** | ASVS-415-MEDIUM-001 |
| **File** | `v3/server/pages.py` |
| **Functions** | `do_open_endpoint()`, `do_close_endpoint()` |
| **Lines** | 496-517 (open), 520-538 (close) |
| **Gap Type** | **Type A** — Entry point with NO control |
| **ASVS Compliance** | ❌ Does not satisfy 4.1.5 |

**Related Functions Checked:**
- `election.open()` (`election.py:269-282`)
- `election.close()` (`election.py:285-296`)
- `crypto.gen_opened_key()` (`crypto.py:31-41`)

**Description:**

Election open and close operations are irreversible state machine transitions performed without per-message digital signatures. These endpoints use GET methods for state-changing operations and rely only on session cookie authentication, creating multiple security vulnerabilities.

**Vulnerable Code:**

```python
@APP.get('/do-open/<eid>')          # State-changing via GET — compounds the issue
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz                   # Authorization also incomplete
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)               # Irreversible state transition, no request signature
    # ...

@APP.get('/do-close/<eid>')         # State-changing via GET
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz
    election.close()                 # Irreversible state transition, no request signature
    # ...
```

**Impact:**

1. **No cryptographic confirmation of intent**: Election open and close are irreversible state machine transitions (`EDITABLE → OPEN → CLOSED`). Opening an election triggers cryptographic key generation (`opened_key`) and salt assignment. Closing an election permanently ends voting. These operations lack cryptographic proof of the administrator's intent.

2. **GET method vulnerability**: These endpoints use GET requests, making them easily triggerable via link injection, img tags, CSRF attacks, or browser prefetching.

3. **Incomplete authorization**: The `### check authz` comments confirm authorization checking is not implemented — any authenticated committer can trigger these transitions.

4. **No audit trail integrity**: While operations are logged, there is no cryptographic binding between the logged action and the administrator who authorized it.

5. **Fails ASVS 4.1.5 requirement**: These are highly sensitive transactions that traverse the application layer and should have additional assurance beyond transport protection.

**Proof of Concept:**

```html
<!-- Injected into any page visible to an authenticated committer -->
<img src="https://voting.example.org/do-close/a1b2c3d4e5" width="0" height="0">
<!-- This silently closes an election if the victim is authenticated -->
```

Or via CSRF:

```html
<form action="https://voting.example.org/do-open/a1b2c3d4e5" method="GET" id="csrf">
</form>
<script>document.getElementById('csrf').submit();</script>
```

**Remediation:**

1. **Change to POST methods** with signed request bodies
2. **Require confirmation signatures**:

```python
from cryptography.hazmat.primitives.asymmetric import ed25519
from cryptography.exceptions import InvalidSignature
import time
import json
import base64

@APP.post('/do-close/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    
    data = await quart.request.get_json()
    signature = base64.b64decode(data['signature'])
    
    # Verify administrator signed the close intent with timestamp
    close_message = json.dumps({
        'action': 'close_election',
        'eid': election.eid,
        'timestamp': data['timestamp']
    }, sort_keys=True)
    
    admin_pubkey = get_admin_public_key(result.uid)
    try:
        admin_pubkey.verify(signature, close_message.encode())
    except InvalidSignature:
        quart.abort(400, 'Invalid signature on close request')
    
    # Verify timestamp freshness (prevent replay)
    if abs(time.time() - data['timestamp']) > 300:
        quart.abort(400, 'Stale request (timestamp too old)')
    
    # Check nonce to prevent replay within the 5-minute window
    if not check_and_consume_nonce(data.get('nonce')):
        quart.abort(400, 'Request nonce already used')
    
    election.close()
    
    # Log with signature verification confirmation
    logger.info(f"User[U:{result.uid}] closed election[E:{election.eid}] with verified signature")
```

3. **Implement similar protection for `/do-open/<eid>`**:

```python
@APP.post('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    
    data = await quart.request.get_json()
    signature = base64.b64decode(data['signature'])
    
    open_message = json.dumps({
        'action': 'open_election',
        'eid': election.eid,
        'timestamp': data['timestamp'],
        'nonce': data['nonce']
    }, sort_keys=True)
    
    admin_pubkey = get_admin_public_key(result.uid)
    try:
        admin_pubkey.verify(signature, open_message.encode())
    except InvalidSignature:
        quart.abort(400, 'Invalid signature on open request')
    
    if abs(time.time() - data['timestamp']) > 300:
        quart.abort(400, 'Stale request')
    
    if not check_and_consume_nonce(data['nonce']):
        quart.abort(400, 'Request nonce already used')
    
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
```

4. **Add nonce storage** (in `crypto.py` or a new module):

```python
# Simple in-memory nonce store (use Redis/database in production)
_used_nonces = {}
_nonce_lock = threading.Lock()

def check_and_consume_nonce(nonce: str) -> bool:
    """Check if nonce is unused, mark as used if valid."""
    with _nonce_lock:
        # Clean up nonces older than 10 minutes
        cutoff = time.time() - 600
        _used_nonces = {n: t for n, t in _used_nonces.items() if t > cutoff}
        
        if nonce in _used_nonces:
            return False
        
        _used_nonces[nonce] = time.time()
        return True
```

## Positive Security Patterns

The following security practices are well-implemented and should be maintained:

1. **Strong key derivation chain** (`crypto.py:31-72`): The `opened_key → vote_token → vote_key` chain using BLAKE2b → Argon2 → HKDF is well-designed for vote privacy and per-voter cryptographic separation. This demonstrates strong understanding of key management principles.

2. **Election tamper detection** (`election.py:283-296`): The `is_tampered()` method recomputes the `opened_key` from current election data and compares it to the stored value, providing data-at-rest integrity verification of the election definition.

3. **Cryptographic shuffling** (`crypto.py:97-111`): Vote tallying uses `crypto.shuffle()` with `secrets.randbelow()` to prevent information leakage through database ordering, protecting voter privacy during the counting process.

4. **Per-voter salts** (`election.py:141-156`): Each voter/issue pair gets a unique cryptographic salt via `secrets.token_bytes(16)`, preventing cross-voter correlation and enhancing vote privacy.

5. **Comprehensive audit logging** (`pages.py` throughout): All sensitive operations are logged with structured identifiers (`User[U:...] action election[E:...]`), providing an audit trail for forensic analysis.

6. **State machine enforcement** (`election.py:269-282`): Election lifecycle transitions are enforced via assertions checking the current state, preventing invalid state transitions.

7. **Use of `secrets` module**: All random generation consistently uses the cryptographically secure `secrets` module rather than `random`, demonstrating proper understanding of cryptographic randomness requirements.

8. **Fernet for vote encryption**: The use of Fernet (AES-128-CBC + HMAC-SHA256) provides both confidentiality and integrity for stored votes, with proper key derivation.

## Architecture Observations

### Current Architecture Limitations

1. **No PKI or key pair infrastructure**: The `crypto.py` module contains only symmetric cryptography (Fernet, Argon2, HKDF, BLAKE2b). There are no asymmetric primitives imported (`ed25519`, `ec`, `rsa`) and no infrastructure for managing voter or administrator key pairs. This means per-message signatures cannot be added without architectural changes.

2. **Session-cookie-only authentication**: All request authentication relies on `asfquart.auth.require` decorators which check session cookies. There is no mechanism for request-level cryptographic authentication or signing.

3. **Incomplete authorization model**: The repeated `### check authz` comments throughout `pages.py` indicate that fine-grained authorization (e.g., "is this user the election owner?") is planned but not implemented, compounding the lack of per-message signatures.

4. **GET for state changes**: The `/do-open/<eid>` and `/do-close/<eid>` endpoints use GET methods for irreversible state transitions, making them vulnerable to CSRF-style attacks even without the signature concern.

5. **No API versioning or signature standards**: The `api.py` module is an empty stub, suggesting the API layer is not yet developed. This is an opportunity to build signature verification into the API from the ground up.

### Architectural Strengths

1. **Clean separation of concerns**: Cryptographic operations are isolated in `crypto.py`, business logic in `election.py`, and HTTP handling in `pages.py`, making it easier to add signature verification at the appropriate layers.

2. **Database abstraction**: The use of query objects (`q_*`) and command objects (`c_*`) provides a clean abstraction that can be extended to log signature verification results.

3. **Election state machine**: The well-defined state transitions (`EDITABLE → OPEN → CLOSED`) provide clear points where signature verification should be enforced.
