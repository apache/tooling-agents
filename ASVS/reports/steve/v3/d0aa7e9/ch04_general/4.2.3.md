## Findings

### Medium Severity

#### ASVS-423-MED-001: No Application-Level HTTP/2 Connection-Specific Header Validation

| Attribute | Detail |
|-----------|--------|
| **Severity** | **MEDIUM** |
| **Gap Type** | Type A — Entry points with NO control |
| **Location** | `v3/server/main.py` (lines 33–48, 91–110) and all routes in `v3/server/pages.py` |
| **ASVS Requirement** | 4.2.3 — Application must not send/accept HTTP/2 or HTTP/3 messages with connection-specific headers |

**Description:**

The application runs on Hypercorn, which supports HTTP/2 by default when TLS is enabled (via ALPN negotiation) and can support HTTP/3. There is no application-level middleware, Quart extension, or Hypercorn configuration to:

1. **Reject** incoming HTTP/2/HTTP/3 requests containing prohibited connection-specific headers (`Transfer-Encoding`, `Connection`, `Keep-Alive`, `Proxy-Connection`, `Upgrade`, `TE` except for `trailers`).
2. **Prevent** connection-specific headers from being included in outgoing HTTP/2/HTTP/3 responses.
3. **Validate** header integrity during HTTP version conversion (e.g., if deployed behind a reverse proxy that downgrades/upgrades HTTP versions).

**Vulnerable Code — Application Entry Point (`main.py`, lines 91–110):**

```python
def run_asgi():
    "Run as an ASGI process; eg. Hypercorn"
    # ...
    _LOGGER.info(' ** Run-mode: ASGI')

    global steve_app
    steve_app = create_app()
    # No middleware added for HTTP/2 header validation
    # No Hypercorn configuration to restrict HTTP versions or validate headers
```

**Vulnerable Code — App Construction (`main.py`, lines 33–48):**

```python
def create_app():
    "Create the asfquart app and its endpoints."
    # ...
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    import pages  # noqa: F401
    import api  # noqa: F401

```

**Affected Entry Points (all routes in `pages.py`):**

All 22 routes are unprotected, including security-sensitive ones:

| Route | Method | Concern |
|-------|--------|---------|
| `/do-vote/<eid>` | POST | Vote submission reads `request.form` — smuggled requests could manipulate votes |
| `/do-set-open_at/<eid>` | POST | Reads `request.get_json()` — smuggled requests could alter election dates |
| `/do-set-close_at/<eid>` | POST | Same as above |
| `/do-create-election` | POST | Election creation with form data |
| `/do-open/<eid>` | GET | State-changing GET (already risky); smuggling compounds it |
| `/do-close/<eid>` | GET | Same as above |
| `/docs/<iid>/<docname>` | GET | File serving with authorization check that could be bypassed |

**Data Flow:**

```
HTTP/2 Client or Reverse Proxy
    → sends request with Transfer-Encoding header over HTTP/2
    → Hypercorn accepts TCP/TLS connection, negotiates HTTP/2 via ALPN
    → h2 library parses HTTP/2 frames
    → [NO APPLICATION-LEVEL VALIDATION]
    → Request reaches Quart route handler (e.g., do_vote_endpoint)
    → request.form / request.get_json() parses body
    → Potential request smuggling if proxy and server disagree on body boundaries
```

**Proof of Concept:**

In an HTTP/2-to-HTTP/1.1 downgrade proxy scenario, an attacker could craft:

```
# HTTP/2 pseudo-headers + prohibited header
:method: POST
:path: /do-vote/election123
:authority: voting.example.org
transfer-encoding: chunked
content-type: application/x-www-form-urlencoded

0\r\n
\r\n
GET /do-open/election456 HTTP/1.1\r\n
Host: voting.example.org\r\n
\r\n
```

If a frontend proxy forwards this over HTTP/2 to Hypercorn (or vice versa) and the `Transfer-Encoding` header is not stripped, the backend could interpret this as two separate requests — one vote submission and one election state change.

**Impact:**

- **Request Smuggling:** A mismatch in how proxies and the application handle `Transfer-Encoding` in HTTP/2 could allow request smuggling, bypassing authentication/authorization decorators.
- **Response Splitting:** Connection-specific headers in responses could confuse HTTP/2 clients or intermediary proxies.
- **Authorization Bypass:** Smuggled requests could reach state-changing endpoints (`/do-open`, `/do-close`, `/do-vote`) without proper session validation.

**Remediation:**

Add ASGI middleware to validate and strip connection-specific headers for HTTP/2/HTTP/3 requests:

```python
# v3/server/middleware.py
from quart import Quart, Response
import functools

# RFC 9113 Section 8.2.2 - prohibited connection-specific headers in HTTP/2
CONNECTION_SPECIFIC_HEADERS = frozenset({
    'transfer-encoding',
    'connection',
    'keep-alive',
    'proxy-connection',
    'upgrade',
})

class HTTP2HeaderValidationMiddleware:
    """ASGI middleware that rejects HTTP/2+ requests containing
    connection-specific header fields per RFC 9113 Section 8.2.2."""

    def __init__(self, app):
        self.app = app

    async def __call__(self, scope, receive, send):
        if scope['type'] == 'http':
            http_version = scope.get('http_version', '1.1')
            if http_version in ('2', '3', '2.0', '3.0'):
                headers = dict(scope.get('headers', []))
                for hdr in CONNECTION_SPECIFIC_HEADERS:
                    if hdr.encode() in headers:
                        # Reject with 400 Bad Request
                        await send({
                            'type': 'http.response.start',
                            'status': 400,
                            'headers': [
                                [b'content-type', b'text/plain'],
                            ],
                        })
                        await send({
                            'type': 'http.response.body',
                            'body': b'Bad Request: connection-specific '
                                    b'header in HTTP/2+ request',
                        })
                        return
        await self.app(scope, receive, send)
```

Register the middleware in `main.py`:

```python
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    import pages  # noqa: F401
    import api   # noqa: F401

    # Add HTTP/2 header validation middleware
    from middleware import HTTP2HeaderValidationMiddleware
    app.asgi_app = HTTP2HeaderValidationMiddleware(app.asgi_app)

    return app
```

Additionally, add a Quart `after_request` handler to strip connection-specific headers from responses:

```python
@app.after_request
async def strip_connection_headers(response):
    for header in ('Transfer-Encoding', 'Connection', 'Keep-Alive',
                    'Proxy-Connection', 'Upgrade'):
        response.headers.pop(header, None)
    return response
```

## Architecture Observations

1. **Framework Dependency for HTTP/2 Security:** The application entirely delegates HTTP/2 protocol compliance to Hypercorn and its `h2` library. While `h2` implements RFC 9113, there is no application-level defense-in-depth layer. This creates an implicit trust boundary at the ASGI server level.

2. **Deployment Topology Unknown:** The code supports both standalone and ASGI deployment modes (`main.py` lines 51–88 vs 91–110). In production, if deployed behind a reverse proxy (nginx, Apache, HAProxy), HTTP version conversion between proxy and Hypercorn introduces the exact attack surface that ASVS 4.2.3 targets.

3. **State-Changing GET Endpoints:** Routes like `/do-open/<eid>` and `/do-close/<eid>` use GET methods for state-changing operations (`pages.py` lines 499, 520). In a request smuggling scenario, these are especially dangerous since they can be smuggled without a request body.

4. **No Response Header Controls:** The application has no `after_request` hooks or middleware to sanitize outgoing response headers, meaning any headers added by Quart internals or extensions in HTTP/2 mode pass through unvalidated.

## Recommendations Summary

### Immediate Actions (Priority 1)

| Recommendation | Effort | Impact |
|----------------|--------|--------|
| **Implement ASGI middleware** to reject HTTP/2+ requests with connection-specific headers | Low | Directly addresses ASVS 4.2.3 |
| **Add `after_request` hook** to strip connection-specific headers from all responses | Low | Prevents outgoing header leakage |

### Short-term Actions (Priority 2-3)

| Recommendation | Effort | Impact |
|----------------|--------|--------|
| **Configure Hypercorn** explicitly for HTTP version handling (document supported versions, disable HTTP/3 if not needed) | Low | Reduces attack surface |
| **Convert state-changing GETs to POSTs** (`/do-open/<eid>`, `/do-close/<eid>`) | Medium | Reduces request smuggling impact |

### Long-term Actions (Priority 4-5)

| Recommendation | Effort | Impact |
|----------------|--------|--------|
| **Add integration tests** validating that HTTP/2 requests with `Transfer-Encoding` are rejected | Medium | Regression prevention |
