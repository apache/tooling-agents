## Architecture Observations

### Framework Capabilities

**Quart Framework:** Natively supports WebSockets through `@app.websocket()` decorators, but architecture describes only HTTP endpoints. This creates a gap between framework capability and documented usage.

### Session Management Model

**Current State:** HTTP-centric session model using:
- Quart sessions
- Flash messages for state transfer
- Cookie-based session tokens

**WebSocket Implications:** If WebSocket support is added, the current session model would require:
1. Explicit token exchange mechanism
2. Session validation during WebSocket handshake
3. Dedicated WebSocket session tokens bound to authenticated users

### API Layer Gap

**Observation:** `api.py` is described as "currently minimal"

**Risk Assessment:** This is the most likely location where WebSocket endpoints could be introduced without being captured in architecture documentation.
