## Findings

### ASVS-811-CRT-001 — Missing Owner-Based Authorization on All Election Management Endpoints

**Severity:** CRITICAL (Type B: Control EXISTS in schema but is NEVER CALLED)

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `load_election` decorator
- **Lines:** ~158-172
- **Also affects:** `load_election_issue` decorator (lines ~175-209); every management endpoint

**Related Functions Checked:**
- `do_open_endpoint` (line ~309)
- `do_close_endpoint` (line ~323)
- `do_add_issue_endpoint` (line ~345)
- `do_edit_issue_endpoint` (line ~367)
- `do_delete_issue_endpoint` (line ~385)
- `do_set_open_at_endpoint` (line ~273)
- `do_set_close_at_endpoint` (line ~279)
- `manage_page` (line ~252)
- `manage_stv_page` (line ~266)

**Description:**

The application defines election ownership and authorization controls in the database schema (`owner_pid` and `authz` fields) with explicit documentation stating that only the owner or members of the specified LDAP group should be able to edit elections. However, these controls are never enforced in the web layer. The `load_election` decorator, which is applied to all 9 management endpoints, contains only a placeholder comment `### check authz` with no actual authorization logic.

**Vulnerable Code:**

```python
# pages.py, lines 158-172 (load_election decorator)
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            # ... 404 handling
            
        _LOGGER.debug(f'Loaded: {e}')

        ### check authz    # <-- PLACEHOLDER, NEVER IMPLEMENTED

        return await func(e)
    return loader
```

**Authorization Rule Defined (schema.md):**
> `authz` (TEXT): Specifies an LDAP group... allowed to edit the election. If NULL, only `owner_pid` can edit.

**Authorization Rule Defined (schema.sql):**
```sql
/* What authz group is allowed to edit this election? If NULL,
   then only the OWNER_PID can edit.  */
authz  TEXT,
```

**Data Flow:**
1. `owner_pid` and `authz` → Stored in `election` table
2. Retrieved by `get_metadata()`
3. **NEVER compared against session user**
4. Any committer can manage any election

**Proof of Concept:**

Any authenticated ASF committer (User B) who is NOT the owner of election `abc1234567` can:

```http
GET /do-open/abc1234567       → Opens someone else's election
GET /do-close/abc1234567      → Closes someone else's election  
POST /do-add-issue/abc1234567 → Adds issues to someone else's election
POST /do-delete-issue/abc1234567/def9876543 → Deletes issues
POST /do-set-open_at/abc1234567 → Changes dates
```

**Impact:**

Complete bypass of election ownership model. Any authenticated committer can manipulate any election — opening, closing, adding/editing/deleting issues, and changing dates — regardless of whether they are the owner or in the authorized group. This undermines the entire election integrity model and violates the documented authorization policy.

**Remediation:**

```python
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            result = await basic_info()
            result.title = 'Unknown Election'
            result.eid = eid
            raise_404(T_BAD_EID, result)

        # Enforce owner/authz authorization
        result = await basic_info()
        md = e.get_metadata()
        if not is_authorized_manager(result.uid, md):
            quart.abort(403, 'Not authorized to manage this election')

        return await func(e)
    return loader

def is_authorized_manager(uid, election_metadata):
    """Check if user is the owner or in the authz group."""
    if uid == election_metadata.owner_pid:
        return True
    if election_metadata.authz:
        # Check LDAP group membership
        return check_group_membership(uid, election_metadata.authz)
    return False
```

Additionally, document authorization rules in a formal policy:

```markdown
## Authorization Matrix — Election Management
| Function | Required Role | Required Relationship |
|----------|---------------|----------------------|
| Create election | pmc_member | None (becomes owner) |
| Open/Close election | committer | owner OR in authz group |
| Add/Edit/Delete issue | committer | owner OR in authz group |
| Set dates | committer | owner OR in authz group |
| View management page | committer | owner OR in authz group |
```

### ASVS-811-HGH-001 — Missing Voter Eligibility Check on Vote Submission Endpoint

**Severity:** HIGH (Type B: Eligibility control exists on GET page but NOT on POST action)

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint`
- **Lines:** ~285-307

**Related Functions Checked:**
- `vote_on_page` (lines ~222-228) — HAS the check
- `election.add_vote()` (election.py, lines ~201-207)
- `q_find_issues` query (queries.yaml)

**Description:**

The vote viewing page (`vote_on_page`) correctly checks voter eligibility using `q_find_issues` before rendering the ballot. However, the vote submission endpoint (`do_vote_endpoint`) does not perform this check before processing votes. Instead, it relies on an implicit exception when `add_vote()` attempts to access `.salt` on a `None` mayvote record, which is caught by a generic exception handler and returns a vague error message.

**Vulnerable Code (POST endpoint — missing check):**

```python
# pages.py, lines 285-307
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    ### check authz    # <-- PLACEHOLDER, NEVER IMPLEMENTED

    form = edict(await quart.request.form)
    # ... processes votes without verifying eligibility
    for iid, votestring in votes.items():
        # No eligibility check before add_vote
        election.add_vote(result.uid, iid, votestring)
```

**Control EXISTS on GET endpoint (vote_on_page, line ~222):**

```python
# pages.py, lines 222-228 (vote_on_page — HAS the check)
election.q_find_issues.perform(result.uid, election.eid)
if not election.q_find_issues.fetchall():
    result = await basic_info()
    result.title = 'Access Denied'
    result.eid = election.eid
    raise_404(T_BAD_EID, result)
```

**Data Flow:**
1. User loads `GET /vote-on/<eid>` → eligibility checked via `q_find_issues` ✅
2. User submits `POST /do-vote/<eid>` → **NO eligibility check**
3. `add_vote()` called → fails implicitly with `AttributeError` on `mayvote.salt` if not eligible ❌
4. Generic exception handler returns vague error message

**Proof of Concept:**

A committer NOT in the `mayvote` table for election `abc1234567` can POST directly:

```http
POST /do-vote/abc1234567
Content-Type: application/x-www-form-urlencoded

vote-def9876543=y
```

This will reach `add_vote()` and fail with an unhandled `AttributeError` (accessing `.salt` on `None`), which is caught by the generic `except Exception` and returned as a vague "Error submitting vote" flash message — not a proper 403 Forbidden.

**Impact:**

While the vote ultimately fails, the failure mode is an unhandled exception rather than a proper authorization denial. The generic error handler could mask real errors. An attacker can probe which issues exist in which elections by observing error vs. success responses, potentially leaking information about election structure.

**Remediation:**

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    # Verify voter eligibility before processing any votes
    election.q_find_issues.perform(result.uid, election.eid)
    eligible_issues = {row.iid for row in election.q_find_issues.fetchall()}
    if not eligible_issues:
        await flash_danger('You are not eligible to vote in this election.')
        return quart.redirect('/voter', code=303)

    form = edict(await quart.request.form)
    # ... process votes, also checking each iid is in eligible_issues
    for iid, votestring in votes.items():
        if iid not in eligible_issues:
            await flash_danger(f'You are not eligible to vote on issue {iid}.')
            return quart.redirect(f'/vote-on/{election.eid}', code=303)
        election.add_vote(result.uid, iid, votestring)
```

### ASVS-811-MED-001 — No Formal Authorization Policy Document Defining Access Rules

**Severity:** MEDIUM

**Location:**
- **File:** `v3/ARCHITECTURE.md` (incomplete security section)
- **File:** `v3/docs/schema.md` (authorization rules marked as "TBD")
- **File:** `v3/server/pages.py` (10 unresolved authorization placeholders)

**Related Functions Checked:**
- All functions with `### check authz` placeholders (10 instances)
- Documentation review across ARCHITECTURE.md, schema.md

**Description:**

The application lacks a formal authorization policy document that defines function-level and data-specific access rules. The existing documentation provides only minimal coverage, and critical authorization rules are explicitly marked as incomplete ("TBD"). This absence of comprehensive documentation has directly led to the implementation gaps identified in the other findings.

**Evidence of Incomplete Documentation:**

**In `ARCHITECTURE.md` (Security section):**
```markdown
### Security
- **Authorization**: Role-based (committer, PMC member) via decorators.
```
This is the entirety of authorization documentation — a single sentence. No function-level rules, no data-specific access rules, no consumer-resource permission mapping.

**In `schema.md`:**
```markdown
- `authz` (TEXT): Specifies an LDAP group... allowed to edit the election. 
  If NULL, only `owner_pid` can edit. The exact format is TBD...
```
Authorization rules are described as "TBD" — incomplete by the documentation's own admission.

**In `pages.py`, 10 instances of unresolved authorization placeholders:**
```python
### check authz                    # line ~101 (_set_election_date)
### check authz                    # line ~167 (load_election)
### check authz                    # line ~194 (load_election_issue)
### check authz                    # line ~290 (do_vote_endpoint)
### check authz                    # line ~335 (do_create_endpoint)
### check authz                    # line ~349 (do_open_endpoint)
### check authz                    # line ~363 (do_close_endpoint)
### check authz                    # line ~378 (do_add_issue_endpoint)
### check authz                    # line ~394 (do_edit_issue_endpoint)
### check authz                    # line ~413 (do_delete_issue_endpoint)
```

**Data Flow:**

The absence of formal documentation creates a gap between:
1. Schema-defined authorization fields (`owner_pid`, `authz`, `mayvote`)
2. Model-layer state enforcement (election states)
3. Web-layer access control (mostly missing)

Without a unifying authorization policy document, each layer operates independently, creating security gaps.

**Impact:**

Without documented authorization rules:
- Developers cannot implement consistent access controls
- Testers cannot verify authorization enforcement
- Administrators cannot audit compliance
- Security reviewers cannot assess completeness

The 10 unimplemented `### check authz` placeholders directly demonstrate that the absence of formal documentation has led to missing enforcement.

**What's Missing:**

1. No authorization matrix mapping roles to permitted functions
2. No documented rules for data-specific access (which user can manage which election)
3. No documented voter eligibility enforcement requirements per endpoint
4. No documented rules for the `authz` LDAP group field
5. No decision-making factors or environmental contexts documented (e.g., election state restrictions)

**Remediation:**

Create a formal authorization policy document:

```markdown
# Authorization Policy — Apache STeVe v3

## Roles
| Role | Source | Description |
|------|--------|-------------|
| Anonymous | No session | Can view public pages only |
| Authenticated | Valid session | Can view profile/settings |
| Committer | ASF OAuth + committer flag | Can vote on eligible elections |
| PMC Member | ASF OAuth + PMC membership | Can create elections |
| Election Owner | `election.owner_pid` matches uid | Full management of owned elections |
| Authz Group Member | `election.authz` LDAP group | Management per election authz field |

## Function-Level Access Rules
| Endpoint | Method | Min Role | Resource Check |
|----------|--------|----------|----------------|
| /voter | GET | Committer | None |
| /vote-on/<eid> | GET | Committer | Must have mayvote entries |
| /do-vote/<eid> | POST | Committer | Must have mayvote entries per issue |
| /admin | GET | Committer | Shows only owned elections |
| /manage/<eid> | GET | Committer | Must be owner or in authz group |
| /do-open/<eid> | POST | Committer | Must be owner or in authz group; election must be editable |
| /do-close/<eid> | POST | Committer | Must be owner or in authz group; election must be open |
| /do-create-election | POST | PMC Member | None |
| /docs/<iid>/<doc> | GET | Authenticated | Must have mayvote entry for issue |

## Data-Specific Rules
- Election management: restricted to owner_pid OR members of authz group
- Voting: restricted to persons with mayvote entries for specific issues
- Tallying: restricted to server-access holders; checks tamper detection first

## Decision-Making Factors
- **User Role**: Derived from ASF OAuth claims (committer, PMC membership)
- **Resource Ownership**: `election.owner_pid` compared to session uid
- **Group Membership**: `election.authz` LDAP group membership (if populated)
- **Voter Eligibility**: Presence of `mayvote` record for user/issue combination
- **Election State**: Current state (editable/open/closed) determines permitted operations
- **Tamper Status**: `opened_key` verification required before tallying

## Environmental Contexts
- All authorization checks require valid authenticated session
- Election state transitions enforce temporal constraints (open_at/close_at)
- Vote submission requires election to be in "open" state
- Management operations require election to be in "editable" state
```

## Positive Security Patterns

1. **Election State Machine**: The `election.py` model layer consistently enforces state transitions via assertions (`is_editable()`, `is_open()`, `is_closed()`), preventing operations in wrong states.

2. **Voter Eligibility on Vote Page**: `vote_on_page` properly checks `q_find_issues` before rendering the ballot, blocking ineligible users from even seeing the voting interface.

3. **Document Access Control**: `serve_doc()` correctly checks `mayvote` before serving per-issue documents, returning 404 for unauthorized users (security through obscurity is appropriate here).

4. **Tamper Detection**: `is_tampered()` recomputes the `opened_key` Argon2 hash from current election data, and the tally script checks this before proceeding.

5. **Metadata Sanitization**: `get_metadata()` explicitly excludes `salt` and `opened_key` from returned data. `get_issue()` and `get_person()` similarly exclude sensitive fields.

6. **Parameterized Queries**: All SQL in `queries.yaml` uses `?` placeholders, preventing SQL injection.

7. **Non-Predictable IDs**: Election and issue IDs are 10-character hex strings generated via `crypto.create_id()`, preventing URL enumeration.

8. **Separate Create Permission**: Election creation requires `R.pmc_member`, a higher privilege level than `R.committer`, establishing a two-tier role model.

---

## Architecture Observations

1. **Authorization Model is Split Across Layers**: The database schema defines ownership and group authorization fields, the model layer enforces state transitions, but the web layer performs NO resource-level authorization. This split creates dangerous gaps where each layer assumes the other handles authorization.

2. **`authz` Field is Dead Code**: The `authz` column on the `election` table is populated during creation, returned in metadata, and displayed in templates, but is never evaluated as an access control decision. It is purely decorative.

3. **Assertion-Based State Enforcement**: The model uses Python `assert` statements for state checks. In production with `-O` flag, these would be silently skipped. This should use explicit `if/raise` patterns.

4. **Decorator Order Matters**: The `@load_election` decorator runs AFTER `@asfquart.auth.require`, which is correct — authentication before authorization. But since `@load_election` has no authorization logic, this ordering currently has no security benefit beyond authentication.

---

## Recommendations Summary

### Priority 1 — Immediate (CRITICAL)

1. **Implement owner/authz checks in `load_election` and `load_election_issue`**: Add proper authorization logic checking `owner_pid` and `authz` against the session user. Every management endpoint flows through these decorators, so fixing them fixes 9 endpoints at once.

### Priority 2 — Short-term (HIGH)

2. **Add explicit voter eligibility check in `do_vote_endpoint`**: Before processing any votes, verify the user has `mayvote` entries for the election. Return 403 instead of relying on exception handling.

3. **Replace `assert` with `if/raise` in `election.py`**: State enforcement via `assert` can be disabled with `-O`. Use explicit exception raises.

### Priority 3 — Medium-term (MEDIUM)

4. **Create formal authorization policy document**: Define an authorization matrix mapping roles, resource relationships, and permitted operations. Include this in the project documentation alongside `ARCHITECTURE.md`.

5. **Implement the `authz` field evaluation**: Either implement LDAP group checking or remove the field to avoid false confidence that group-based authorization exists.

6. **Add authorization audit logging**: Log authorization decisions (both grants and denials) to enable security monitoring and incident investigation.
