### Summary

This audit evaluates the election system against ASVS 8.1.4, which requires documented authorization rules covering environmental/contextual factors, function-level, data-specific, and field-level authorization with defined attributes, risk thresholds, and actions. The codebase reveals a **critical systemic gap**: while the database schema defines ownership (`owner_pid`) and group authorization (`authz`) fields, **no ownership or role-based authorization checks are implemented at any management endpoint**. The code contains 10+ explicit `### check authz` placeholder comments confirming these controls were planned but never built, creating a false sense of security.

## Findings

### Critical Severity

#### ASVS-814-CRT-001: Missing Election Ownership Authorization on All Management Endpoints

**Severity:** Critical

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `load_election` decorator (lines 171-189), `load_election_issue` (lines 195-226)
- **Also affects:** Lines 97 (`_set_election_date`), 399 (`do_vote_endpoint`), 443 (`do_create_endpoint`), 465 (`do_open_endpoint`), 486 (`do_close_endpoint`), 503 (`do_add_issue_endpoint`), 526 (`do_edit_issue_endpoint`), 549 (`do_delete_issue_endpoint`)

**Related Functions Checked:**
- `load_election()` — decorator for election loading
- `load_election_issue()` — decorator for election + issue loading
- `do_open_endpoint()` — opens election
- `do_close_endpoint()` — closes election
- `do_add_issue_endpoint()` — adds issue to election
- `do_edit_issue_endpoint()` — edits election issue
- `do_delete_issue_endpoint()` — deletes election issue
- `_set_election_date()` — modifies election dates

**Description:**

The codebase reveals a critical systemic gap where database schema defines ownership (`owner_pid`) and group authorization (`authz`) fields, but **no ownership or role-based authorization checks are implemented at any management endpoint**. The code contains 10+ explicit `### check authz` placeholder comments confirming these controls were planned but never built. This is a Type B gap where the authorization need is explicitly recognized but the check is never implemented, creating dangerous false confidence.

**Vulnerable Code:**

```python
# pages.py, lines 171-189
def load_election(func):
    """Decorator to load/pass-argument an Election from EID."""
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            # ... 404 handling ...
        _LOGGER.debug(f'Loaded: {e}')
        ### check authz          # <-- PLACEHOLDER, NEVER IMPLEMENTED
        return await func(e)
    return loader
```

**Data Flow:**
1. **Source:** Authenticated committer session (`result.uid` from `basic_info()`)
2. **Sink:** Election management operations (`election.open()`, `election.close()`, `election.add_issue()`, etc.)
3. **Missing Control:** Verification that `result.uid == election.owner_pid` or `result.uid` is a member of `election.authz` group

**Proof of Concept:**

```bash
# Attacker "mallory" is an ASF committer but NOT the election owner.
# She discovers election ID "a1b2c3d4e5" (e.g., from voter page listing)

# Close someone else's active election (via simple GET!)
curl -b "session=mallory_session" \
  "https://vote.example.org/do-close/a1b2c3d4e5"

# Add a malicious issue to someone else's editable election
curl -b "session=mallory_session" \
  -d "title=Malicious+Issue&description=Injected" \
  "https://vote.example.org/do-add-issue/a1b2c3d4e5"

# Delete issues from someone else's election
curl -b "session=mallory_session" \
  -X POST "https://vote.example.org/do-delete-issue/a1b2c3d4e5/f6g7h8i9j0"
```

**Impact:**

Any authenticated ASF committer can manage, modify, open, close, or corrupt ANY election in the system, regardless of ownership. This enables:
- Premature closure of active elections to suppress votes
- Injection of fraudulent issues into elections
- Deletion of legitimate issues
- Modification of election dates
- Opening elections before they're ready

**Remediation:**

```python
async def check_election_authz(election, uid):
    """Verify the user is authorized to manage this election."""
    md = election.get_metadata()
    if md.owner_pid == uid:
        return  # Owner always authorized
    
    if md.authz:
        # Check if user is in the authz group (e.g., LDAP group check)
        if await is_member_of_group(uid, md.authz):
            return
    
    _LOGGER.warning(
        f'Authz denied: User[U:{uid}] attempted to manage election[E:{election.eid}]'
        f' owned by [U:{md.owner_pid}]'
    )
    quart.abort(403, 'Not authorized to manage this election')


def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        e = steve.election.Election(DB_FNAME, eid)
        # ... existing error handling ...
        
        # ENFORCE ownership/authz check
        result = await basic_info()
        await check_election_authz(e, result.uid)
        
        return await func(e)
    return loader
```

### High Severity

#### ASVS-814-HGH-001: Missing Voter Eligibility Check in Vote Submission Endpoint

**Severity:** High

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint`
- **Lines:** 394-435

**Related Functions Checked:**
- `do_vote_endpoint()` — POST handler for vote submission (lines 394-435)
- `vote_on_page()` — GET handler for vote form (lines 255-261, properly checks eligibility)
- `election.add_vote()` — adds vote to database (implicitly fails without explicit check)

**Description:**

The vote submission endpoint (`do_vote_endpoint`) lacks explicit voter eligibility verification before processing votes. While the GET endpoint (`vote_on_page`) properly checks the `mayvote` table, the POST endpoint relies on an implicit failure when `add_vote()` attempts to access `None.salt` for ineligible voters. This is a Type B gap where the eligibility check exists in one location but is not replicated in the critical submission path.

**Vulnerable Code:**

```python
# pages.py, lines 394-435
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    ### check authz                    # <-- PLACEHOLDER, NEVER IMPLEMENTED

    form = edict(await quart.request.form)
    # ... processes votes without eligibility check ...
    for iid, votestring in votes.items():
        # No check: is result.uid in mayvote for this iid?
        try:
            election.add_vote(result.uid, iid, votestring)
```

**Contrast with GET endpoint** (lines 255-261, `vote_on_page`):

```python
# The GET page DOES check eligibility:
election.q_find_issues.perform(result.uid, election.eid)
if not election.q_find_issues.fetchall():
    result = await basic_info()
    result.title = 'Access Denied'
    raise_404(T_BAD_EID, result)
```

**Data Flow:**
1. **Source:** Authenticated committer (`result.uid`) + form data with `vote-{iid}` keys
2. **Sink:** `election.add_vote(result.uid, iid, votestring)` → `crypto.gen_vote_token()` → database
3. **Missing Control:** Explicit eligibility verification before `add_vote()`

**Proof of Concept:**

```bash
# Attacker "mallory" is a committer but NOT in the mayvote table for this election.
# The vote-on page would deny access, but she submits directly to the POST endpoint:
curl -b "session=mallory_session" \
  -d "vote-a1b2c3d4e5=yes" \
  "https://vote.example.org/do-vote/f6g7h8i9j0"

# Result: 500 error (AttributeError) instead of 403 Forbidden
# The error is caught generically and produces "Error submitting vote for issue..."
# Information leakage: confirms the issue ID exists
```

**Impact:**
- Authorization failure produces a 500 error instead of a clear 403 denial
- Error message leaks information about valid issue IDs
- No audit log entry distinguishing unauthorized access from system errors
- The implicit failure depends on an `AttributeError` which is fragile and could change

**Remediation:**

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    # Verify voter eligibility BEFORE processing any votes
    election.q_find_issues.perform(result.uid, election.eid)
    eligible_issues = {row.iid for row in election.q_find_issues.fetchall()}
    if not eligible_issues:
        _LOGGER.warning(
            f'Authz denied: User[U:{result.uid}] attempted to vote in '
            f'election[E:{election.eid}] without eligibility'
        )
        quart.abort(403, 'Not authorized to vote in this election')

    form = edict(await quart.request.form)
    votes = {}
    for key, value in form.items():
        if key.startswith('vote-'):
            iid = key.split('-', 1)[1]
            if iid not in eligible_issues:
                _LOGGER.warning(
                    f'Authz denied: User[U:{result.uid}] attempted to vote on '
                    f'issue[I:{iid}] without eligibility'
                )
                quart.abort(403, f'Not authorized to vote on this issue')
            votes[iid] = value
    # ... continue processing ...
```

### Medium Severity

#### ASVS-814-MED-001: State-Changing Operations via GET Requests Enable Trivial CSRF

**Severity:** Medium

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `do_open_endpoint` (lines 460-479), `do_close_endpoint` (lines 481-498)

**Related Functions Checked:**
- `do_open_endpoint()` — opens election via GET request
- `do_close_endpoint()` — closes election via GET request
- Both lack CSRF token validation

**Description:**

Critical state-changing operations (opening and closing elections) are implemented as GET requests without CSRF protection. This violates HTTP semantics (GET should be idempotent and safe) and enables trivial cross-site request forgery attacks where any authenticated committer visiting a malicious page could have elections opened or closed without their knowledge.

**Vulnerable Code:**

```python
# Line 460 - Opens an election via GET
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    # ... no CSRF token check ...
    election.open(pdb)

# Line 481 - Closes an election via GET
@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    # ... no CSRF token check ...
    election.close()
```

**Data Flow:**
1. **Source:** GET request with election ID in URL path
2. **Sink:** `election.open()` / `election.close()` — irreversible state transitions
3. **Missing Control:** POST method requirement + CSRF token validation

**Proof of Concept:**

```html
<!-- Attacker embeds this in any page viewed by an authenticated committer -->
<!-- Closes an active election silently -->
<img src="https://vote.example.org/do-close/a1b2c3d4e5" style="display:none">

<!-- Or opens a premature election -->
<img src="https://vote.example.org/do-open/f6g7h8i9j0" style="display:none">
```

**Impact:**

Election state transitions (open/close) are irreversible operations that can be triggered by:
- Cross-site image tags, link prefetching, or browser extensions
- Browser history/bookmark sync services
- Web crawlers (if they somehow reach authenticated pages)

Combined with ASVS-814-CRT-001 (missing ownership checks), any authenticated committer visiting a malicious page could have elections opened or closed without their knowledge.

**Remediation:**

```python
@APP.post('/do-open/<eid>')  # Changed from GET to POST
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    # Validate CSRF token
    form = await quart.request.form
    validate_csrf_token(form.get('csrf_token'))
    
    result = await basic_info()
    await check_election_authz(election, result.uid)
    
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    # ...
```

#### ASVS-814-MED-002: Election State Machine Uses Assert Statements Bypassable in Production

**Severity:** Medium

**Location:**
- **File:** `v3/steve/election.py`
- **Functions:** Multiple locations throughout the file

**Related Functions Checked:**
- `delete()` — line 72, uses `assert self.is_editable()`
- `open()` — line 94, uses `assert self.is_editable()`
- `add_issue()` — line 149, uses `assert self.is_editable()` and `assert vtype in vtypes.TYPES`
- `edit_issue()` — line 168, uses `assert self.is_editable()` and `assert vtype in vtypes.TYPES`
- `delete_issue()` — line 175, uses `assert self.is_editable()`
- `add_vote()` — line 214, correctly uses exception-based check `_all_metadata(self.S_OPEN)`

**Description:**

The election state machine uses `assert` statements to enforce state transitions and validate input. When Python is run with optimization flags (`python -O` or `PYTHONOPTIMIZE=1`), all `assert` statements are removed from the bytecode. This is common in production deployments for performance, which would eliminate critical state machine enforcement and input validation.

**Vulnerable Code:**

```python
# election.py, line 72
def delete(self):
    assert self.is_editable()  # Stripped by python -O
    self.db.conn.execute('BEGIN TRANSACTION')
    self.c_delete_mayvote.perform(self.eid)
    # ...

# election.py, line 94
def open(self, pdb):
    assert self.is_editable()  # Stripped by python -O
    self.add_salts()
    # ...

# election.py, line 149
def add_issue(self, title, description, vtype, kv):
    assert self.is_editable()  # Stripped by python -O
    assert vtype in vtypes.TYPES  # Also stripped
    # ...

# election.py, line 168
def edit_issue(self, iid, title, description, vtype, kv):
    assert self.is_editable()  # Stripped by python -O
    assert vtype in vtypes.TYPES  # Also stripped
    # ...

# election.py, line 175
def delete_issue(self, iid):
    assert self.is_editable()  # Stripped by python -O
    # ...
```

**Contrast with `add_vote` (line 214) which uses proper exception:**
```python
def add_vote(self, pid, iid, votestring):
    md = self._all_metadata(self.S_OPEN)  # Raises ElectionBadState if wrong state
```

**Data Flow:**
1. **Source:** Any management endpoint call (already lacking ownership checks per ASVS-814-CRT-001)
2. **Sink:** State-modifying database operations (delete, add_issue, edit_issue, delete_issue)
3. **Missing Control:** Proper exception-based state validation that cannot be optimized away

**Impact:**

If the application is deployed with `python -O` or `PYTHONOPTIMIZE=1` (common in containerized production deployments for performance), all `assert` statements are removed. This eliminates:
- State machine enforcement: issues could be added to open/closed elections
- Vote type validation: arbitrary `vtype` values could be stored
- Election deletion protection: open elections could be deleted

Combined with ASVS-814-CRT-001, any committer could modify issues in active elections, corrupting ongoing votes.

**Remediation:**

```python
def _require_state(self, required_state):
    """Enforce election state requirement (not bypassable by -O)."""
    current = self.get_state()
    if current != required_state:
        raise ElectionBadState(self.eid, current, required_state)

def delete(self):
    self._require_state(self.S_EDITABLE)
    # ...

def add_issue(self, title, description, vtype, kv):
    self._require_state(self.S_EDITABLE)
    if vtype not in vtypes.TYPES:
        raise ValueError(f'Invalid vote type: {vtype}')
    # ...
```

## Positive Security Patterns

1. **Tamper Detection**: The `opened_key` Argon2 hash of election data provides integrity checking for election state, with `is_tampered()` available for verification.

2. **Cryptographic Vote Anonymity**: Per-voter salts in `mayvote`, vote tokens derived via Argon2, and Fernet encryption provide strong separation between voter identity and vote content.

3. **Non-Predictable IDs**: 10-character hex IDs generated via `crypto.create_id()` with collision retry logic prevent URL enumeration.

4. **Document Access Control**: `serve_doc()` properly checks the `mayvote` table before serving issue documents, implementing data-specific authorization.

5. **Database Constraints**: Schema enforces referential integrity with `STRICT` mode, `CHECK` constraints, and `FOREIGN KEY ... ON DELETE RESTRICT`.

6. **Audit Logging**: Security-relevant operations are logged with user identity and resource identifiers (e.g., `User[U:{uid}] opened election[E:{eid}]`).

7. **Differentiated Creation Authorization**: Election creation requires `R.pmc_member` (higher privilege) while other operations require `R.committer`.

## Architecture Observations

1. **Authorization Model is Designed but Unimplemented**: The database schema clearly defines an authorization model with `owner_pid` for ownership and `authz` for group-based access. The code has numerous `### check authz` comments confirming developers intended to implement these checks. This represents a dangerous intermediate state where the authorization infrastructure exists but enforcement does not.

2. **No Authorization Documentation Exists**: Per ASVS 8.1.4, there is no documentation defining:
   - Environmental/contextual factors (IP, time, device) in authorization decisions
   - Risk thresholds and escalation rules
   - Step-up authentication triggers
   - Complete mapping of which roles can perform which operations on which resources
   
   The `schema.md` describes database fields but not the decision-making process.

3. **Two-Tier Authentication Without Authorization**: The system distinguishes `R.committer` from `R.pmc_member` for authentication but has no resource-level authorization. All committers are equivalent for all operations except election creation.

4. **State Machine as Partial Control**: The election state machine (editable → open → closed) provides some protection against invalid operations but doesn't address WHO can perform operations, only WHEN they can be performed.

## Recommendations Summary

### Immediate Actions (Priority 1)

**Implement Ownership Authorization** *(Addresses ASVS-814-CRT-001)*

Create a reusable `check_election_authz()` function and integrate it into both `load_election` and `load_election_issue` decorators. This single change would close the most critical gap across all management endpoints.

```python
async def check_election_authz(election, uid):
    """Verify the user is authorized to manage this election."""
    md = election.get_metadata()
    if md.owner_pid == uid:
        return
    if md.authz and await is_member_of_group(uid, md.authz):
        return
    _LOGGER.warning(f'Authz denied: User[U:{uid}] attempted to manage election[E:{election.eid}]')
    quart.abort(403, 'Not authorized to manage this election')
```

### Short-Term Actions (Priority 2-3)

**Add Eligibility Check to Vote Submission** *(Addresses ASVS-814-HGH-001)*

Replicate the eligibility check from `vote_on_page` into `do_vote_endpoint` with proper 403 responses and audit logging for denied attempts.

**Convert State-Changing GET to POST** *(Addresses ASVS-814-MED-001)*

Change `/do-open/<eid>` and `/do-close/<eid>` from GET to POST, and implement CSRF token validation for all state-changing endpoints.

### Long-Term Actions (Priority 4-5)

**Replace Assert with Exception-Based Checks** *(Addresses ASVS-814-MED-002)*

Replace all `assert` statements in `election.py` with proper exception-raising checks that cannot be stripped by Python optimization flags.

**Create Authorization Documentation** *(Addresses ASVS 8.1.4 Directly)*

Document the complete authorization decision matrix including:
- Role → Resource → Operation mapping
- Ownership and group-based authorization rules
- Election state transitions and who can trigger them
- Voter eligibility verification points
- Environmental factors considered (if any)
- Risk thresholds and actions (allow/deny/step-up)
