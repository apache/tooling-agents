## Findings

**No findings were identified for ASVS 9.1.2.** The application does not use self-contained tokens with algorithm selection mechanisms that would be vulnerable to algorithm confusion or 'None' algorithm attacks.

### Rationale for Zero Findings

#### 1. No Self-Contained Tokens with Algorithm Selection

The ASVS chapter definition states: *"a token containing data or claims on which a receiving service will rely to make security decisions. This should be differentiated from a simple token containing only an identifier."*

The `vote_token` is explicitly an opaque identifier used for database lookups:

```python
# crypto.py, line ~50
def gen_vote_token(opened_key: bytes, pid: str, iid: str, salt: bytes) -> bytes:
    "Generate a person or issue token."
    return _hash(opened_key + pid.encode() + iid.encode(), salt)
```

```python
# election.py, line ~230 (add_vote)
vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
# ...
self.c_add_vote.perform(vote_token, ciphertext)
```

```python
# election.py, line ~280 (tally_issue)
row = self.q_recent_vote.first_row(vote_token)  # DB lookup by token
```

These tokens carry no embedded algorithm identifier, no claims, and no data that could be manipulated to switch algorithms.

#### 2. No 'None' Algorithm Possibility

All cryptographic operations use hardcoded algorithms with no selection mechanism:

- `_hash()` always uses `argon2.low_level.Type.D` (line ~95)
- `_b64_vote_key()` always uses `hkdf.HKDF(algorithm=hashes.SHA256(), ...)` (line ~61)
- `create_vote()` and `decrypt_votestring()` always use `cryptography.fernet.Fernet` (lines ~73, ~80)

There is no code path that accepts an algorithm parameter from external input, a token header, or configuration that could resolve to 'None'.

#### 3. No Symmetric/Asymmetric Key Confusion Risk

The entire cryptographic stack is purely symmetric:
- Argon2 (symmetric hash function)
- HKDF-SHA256 (symmetric key derivation)
- Fernet/AES-CBC (symmetric encryption)

No asymmetric algorithms (RSA, ECDSA, EdDSA) are present anywhere in the codebase. There is no possibility of key confusion between symmetric and asymmetric operations.

#### 4. Fernet's Fixed Algorithm Design

Fernet is specifically designed to prevent algorithm agility attacks. The format uses version `0x80` which mandates AES-128-CBC + HMAC-SHA256. The `cryptography` library rejects any other version byte during decryption:

```python
# crypto.py, lines ~71-74
def create_vote(vote_token: bytes, salt: bytes, votestring: str) -> bytes:
    b64key = _b64_vote_key(vote_token, salt)
    f = cryptography.fernet.Fernet(b64key)
    return f.encrypt(votestring.encode())
```

## Architecture Observations

### 1. No JWT/SAML Usage
The codebase uses opaque cryptographic tokens (Argon2 hashes) as database identifiers and Fernet for symmetric encryption. Neither pattern constitutes a "self-contained token" as defined by ASVS Chapter 9. Authentication token handling (if using JWTs or session tokens) is not present in the reviewed files and would be in the web layer.

### 2. Implicit Algorithm Allowlist
While there is no explicit allowlist configuration, the algorithms are hardcoded in source code, which is effectively a compile-time allowlist. This is arguably stronger than a runtime allowlist since it cannot be reconfigured without code changes.

**Advantages:**
- Cannot be misconfigured at runtime
- Requires code review to change algorithms
- Clear audit trail via version control

**Considerations:**
- Algorithm changes require code deployment
- No flexibility for algorithm negotiation (appropriate for this use case)

### 3. Scope Limitation
The provided files are cryptographic library (`crypto.py`) and election domain logic (`election.py`). If the application uses JWTs or other self-contained tokens for authentication at the HTTP layer, those files were not provided for review. ASVS 9.1.2 findings would most likely appear in web framework middleware, authentication handlers, or token validation code.

### 4. Token Design Philosophy
The application follows a secure-by-design approach:
- Tokens are opaque identifiers, not self-describing
- All security decisions rely on server-side data lookups
- No client-controlled cryptographic parameters

This architecture inherently avoids many JWT-related vulnerabilities.

## Recommendations Summary

Since no findings were identified for ASVS 9.1.2, recommendations are preventive and focused on maintaining current security posture.

### Immediate Actions
*No immediate remediation required.*

### Short-term Recommendations (1-3 months)

1. **Document the implicit algorithm allowlist**
   - **Priority:** Low
   - **Effort:** 1 hour
   - **Action:** Add a comment block or configuration constant that explicitly names the approved algorithms, making the implicit allowlist auditable:
   ```python
   # Approved algorithm allowlist for this application (ASVS 9.1.2):
   # - Argon2d (token hashing)
   # - HKDF-SHA256 (key derivation)  
   # - Fernet/AES-128-CBC+HMAC-SHA256 (vote encryption)
   # - BLAKE2b (pre-hashing)
   # No algorithm selection from external input is permitted.
   ```

2. **Clarify Argon2 variant usage**
   - **Priority:** Low
   - **Effort:** 30 minutes
   - **Action:** Document why `benchmark_argon2` (line ~118) uses `Type.ID` while production `_hash` uses `Type.D`. This is not a security issue for 9.1.2 but could cause confusion if benchmark results are used to tune production parameters.

### Long-term Recommendations (3-12 months)

3. **If JWTs are introduced in future**
   - **Priority:** High (if applicable)
   - **Effort:** 2-4 hours per implementation
   - **Action:** Implement an explicit algorithm allowlist (e.g., `["ES256"]` only), reject tokens with `alg: "none"`, and use asymmetric-only verification to prevent key confusion. Libraries like `PyJWT` should be configured with `algorithms=["ES256"]` in every `decode()` call.
   
   **Example secure configuration:**
   ```python
   import jwt
   
   # Secure JWT verification
   ALLOWED_ALGORITHMS = ["ES256"]  # Explicit allowlist, asymmetric only
   
   def verify_token(token: str, public_key: str) -> dict:
       try:
           payload = jwt.decode(
               token,
               public_key,
               algorithms=ALLOWED_ALGORITHMS,  # Explicit allowlist
               options={
                   "verify_signature": True,
                   "require": ["exp", "iat", "sub"]  # Require critical claims
               }
           )
           return payload
       except jwt.InvalidAlgorithmError:
           raise SecurityError("Algorithm not in allowlist")
       except jwt.DecodeError:
           raise SecurityError("Invalid token format")
   ```

4. **Consider explicit configuration for algorithm parameters**
   - **Priority:** Low
   - **Effort:** 4-8 hours
   - **Action:** While hardcoded algorithms are secure, consider moving algorithm selection to a validated configuration file for easier security audits and compliance documentation. Ensure configuration is:
     - Read-only at runtime
     - Validated against a schema
     - Logged on application startup
     - Subject to code review for changes
