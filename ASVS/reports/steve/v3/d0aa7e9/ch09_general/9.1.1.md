- **ASVS Requirement:** 9.1.1 — Verify that self-contained tokens are validated using their digital signature or MAC to protect against tampering before accepting the token's contents.
  - **Level:** 1
  - **Section:** Token source and integrity
  - **Section Description:** This section includes requirements to ensure that the token has been produced by a trusted party and has not been tampered with.
  - **Chapter:** Self-contained Tokens
  - **Control Objective:** The concept of a self-contained token is mentioned in the original RFC 6749 OAuth 2.0 from 2012. It refers to a token containing data or claims on which a receiving service will rely to make security decisions. This should be differentiated from a simple token containing only an identifier, which a receiving service uses to look up data locally. The most common examples of self-contained tokens are JSON Web Tokens (JWTs) and SAML assertions. The use of self-contained tokens has become very widespread, even outside of OAuth and OIDC. At the same time, the security of this mechanism relies on the ability to validate the integrity of the token and to ensure that the token is valid for a particular context. There are many pitfalls with this process, and this chapter provides specific details of the mechanisms that applications should have in place to prevent them.

### Summary

The audit against ASVS 9.1.1 identified one critical finding. While the codebase implements Fernet encryption with built-in HMAC validation for vote ciphertexts (a positive pattern), the election data integrity check (`is_tampered()`) exists as a defined control but is **never invoked** at any server entry point before sensitive operations, representing a classic Type B coverage gap.

## Findings

### Critical Severity

#### ASVS-911-SEV-001: Election Data Integrity Check (opened_key MAC) Exists But Is Never Called Before Sensitive Operations

**Severity**: CRITICAL (Type B Gap — Control EXISTS but NOT CALLED)

**Location**:
- **File**: `v3/steve/election.py`
- **Method**: `is_tampered()` (line ~255)
- **Uncovered Entry Points**: 
  - `v3/server/pages.py` — `do_vote_endpoint` (line ~410), `do_close_endpoint` (line ~482)
  - `v3/steve/election.py` — `add_vote()` (line ~200), `tally_issue()` (line ~215), `has_voted_upon()` (line ~258)

**Related Functions Checked**:
- `_all_metadata()` — called but does not invoke integrity check
- `gen_opened_key()` — creates the MAC but validation never occurs
- `add_vote()`, `tally_issue()`, `has_voted_upon()`, `close()` — all consume `opened_key` without verification

**Description**:  
The `opened_key` serves as a MAC/integrity hash of the complete election definition (election metadata, issues, eligible voters). It is generated via Argon2 when the election opens and stored in the database. The `is_tampered()` method recomputes this hash and compares it to the stored value to detect modifications. However, **no server endpoint or internal method ever calls `is_tampered()`** before accepting or acting on the election data.

**Vulnerable Code — The unused control** (`election.py` ~line 255):
```python
def is_tampered(self, pdb):
    md = self._all_metadata()  # no required state
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)
    return opened_key != md.opened_key
```

**Vulnerable Code — Vote submission trusts `opened_key` without integrity check** (`election.py` ~line 200):
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)  # Checks state, NOT integrity

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    #                                  ^^^^^^^^^^^^^^
    # opened_key read from DB and used directly — never validated
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

**Vulnerable Code — Server endpoint has no tamper check** (`pages.py` ~line 410):
```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    # ... no call to election.is_tampered() ...
    for iid, votestring in votes.items():
        election.add_vote(result.uid, iid, votestring)
        # Votes recorded using potentially tampered election data
```

**Data Flow**:
```
Database (opened_key, voter lists, issue definitions)
    ↓
_all_metadata() reads opened_key [NO integrity verification]
    ↓
add_vote() / tally_issue() / has_voted_upon() consume opened_key
    ↓
Cryptographic operations (vote tokens, encryption) proceed on unvalidated data
```

**Proof of Concept**:
1. Administrator or attacker with database access modifies the `mayvote` table (adds unauthorized voter) or `issue` table (modifies issue data) after the election is opened.
2. The `opened_key` in the database no longer matches the recomputed hash of the modified election data.
3. Despite this, `add_vote()` proceeds normally because `is_tampered()` is never called — votes are accepted under a tampered election configuration.
4. During tallying, `tally_issue()` similarly processes results without verifying election integrity, producing potentially fraudulent results.

**Impact**:
- Post-opening modifications to election data (voter rolls, issues, election metadata) go undetected during all voting and tallying operations.
- The entire purpose of the `opened_key` integrity mechanism — detecting tampering — is nullified because the check is never performed.
- False confidence: the existence of `is_tampered()` suggests integrity is being verified, but it is not.
- Attackers with database access can manipulate election outcomes without detection.

**Remediation**:

Add tamper checks before every sensitive operation that relies on election data. The most effective approach is to integrate it into `_all_metadata()` or create a wrapper:

```python
# Option A: Integrate into _all_metadata for open/closed elections
def _all_metadata(self, required_state=None, check_integrity=False):
    md = self.q_metadata.first_row(self.eid)
    if not md:
        raise self._disappeared()

    if required_state:
        state = self._compute_state(md)
        if state != required_state:
            raise ElectionBadState(self.eid, state, required_state)

    if check_integrity and md.opened_key is not None:
        # Requires pdb to be available; consider caching or passing it
        if self.is_tampered(self._pdb):
            raise ElectionTampered(self.eid)

    return md

# Option B: Add explicit checks at each entry point in pages.py
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    
    # Verify election integrity BEFORE accepting votes
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    if election.is_tampered(pdb):
        await flash_danger('Election integrity check failed. Voting halted.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
    
    # ... proceed with vote processing ...
```

Additionally, use constant-time comparison for the MAC check:

```python
import hmac

def is_tampered(self, pdb):
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)
    # Use constant-time comparison
    return not hmac.compare_digest(opened_key, md.opened_key)
```

## Positive Security Patterns

| Pattern | Location | Description |
|---------|----------|-------------|
| **Fernet HMAC validation on decrypt** | `crypto.py:74-79` | `Fernet.decrypt()` automatically validates HMAC-SHA256 before returning plaintext. All vote decryption paths use this correctly. |
| **Strong key derivation chain** | `crypto.py:37-47, 54-65` | Multi-stage derivation: BLAKE2b → Argon2 → HKDF-SHA256 provides defense-in-depth for key material. |
| **Per-voter cryptographic salts** | `election.py:~125-140` | Each voter/issue pair gets a unique 16-byte salt via `secrets.token_bytes()`, preventing cross-voter correlation. |
| **Secure random ID generation** | `crypto.py:109-112` | Uses `secrets.token_hex()` for election and issue IDs — cryptographically unpredictable. |
| **Cryptographic shuffle for tally privacy** | `crypto.py:93-106` | Fisher-Yates shuffle using `secrets.randbelow()` ensures votes are not in database insertion order during tallying. |
| **State validation before mutations** | `election.py:~200, ~215` | `_all_metadata(required_state)` enforces state machine constraints (e.g., only accept votes when election is OPEN). |

## Architecture Observations

1. **Integrity mechanism is well-designed but disconnected**: The `opened_key` approach (hash all election data, store hash, recompute to detect changes) is a sound integrity pattern. The implementation in `is_tampered()` correctly gathers election data and recomputes the hash. The critical flaw is purely in the call graph — no code path invokes this check.

2. **`_all_metadata()` is the natural integration point**: Every sensitive operation already calls `_all_metadata()` for state validation. Adding integrity verification here would provide comprehensive coverage with minimal code changes.

3. **Non-constant-time comparison**: The `is_tampered()` method uses Python's `!=` operator for hash comparison, which short-circuits. While Argon2's computation time makes timing attacks impractical, using `hmac.compare_digest()` would be defense-in-depth.

4. **Positive cryptographic foundation**: The underlying cryptographic primitives (Fernet, Argon2, HKDF) are industry-standard and correctly implemented. The issue is architectural — a missing link between the integrity check and the operations that depend on it.
