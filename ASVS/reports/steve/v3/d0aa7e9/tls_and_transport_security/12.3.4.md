## Findings

### High Severity

#### ASVS-1234-HIGH-001: No SSL Context Configuration — Missing TLS Protocol Version Minimum and Cipher Suite Controls

**Severity:** HIGH  
**CWE:** CWE-326 (Inadequate Encryption Strength)  
**File:** `v3/server/main.py`  
**Function:** `run_standalone()`  
**Lines:** 83-89

**Gap Type:** Type A — Entry point with NO control

**Vulnerable Code:**
```python
kwargs = {}
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
    extra_files.update((kwargs['certfile'], kwargs['keyfile']))

# Spool up the app!
app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Analysis:**  
The TLS configuration passes raw file paths to `app.runx()` without creating an `ssl.SSLContext`. This means:
- **No minimum TLS version** is enforced (TLS 1.0/1.1 may be accepted depending on underlying library defaults)
- **No cipher suite restrictions** are applied (weak ciphers like RC4, DES, or export-grade ciphers may be negotiated)
- **No client certificate verification** for the internal proxy-to-server connection (the config comments confirm "a proxy sits in front of this server")

The ASVS requirement explicitly states that TLS connections between internal services must use trusted certificates. Without an SSL context, there is no mechanism to enforce trust policy or protocol security on this internal connection.

**Data Flow:**  
`config.yaml certfile/keyfile` → `main.py kwargs dict` → `app.runx()` → **Missing: ssl.SSLContext with protocol/cipher/trust settings**

**Impact:**  
Internal connections between the reverse proxy and this server may negotiate deprecated TLS versions or weak cipher suites, enabling potential MITM attacks on the internal network segment. Per ASVS 12.3.4, internal service connections must enforce trusted certificates — without an SSL context, no trust policy exists.

**Remediation:**
```python
import ssl

kwargs = {}
if app.cfg.server.certfile:
    ssl_context = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
    ssl_context.minimum_version = ssl.TLSVersion.TLSv1_2
    ssl_context.set_ciphers(
        'ECDHE+AESGCM:ECDHE+CHACHA20:DHE+AESGCM:DHE+CHACHA20:!aNULL:!MD5:!DSS:!RC4'
    )
    ssl_context.load_cert_chain(
        certfile=CERTS_DIR / app.cfg.server.certfile,
        keyfile=CERTS_DIR / app.cfg.server.keyfile,
    )
    # For mutual TLS with the reverse proxy (ASVS 12.3.4):
    # ssl_context.verify_mode = ssl.CERT_REQUIRED
    # ssl_context.load_verify_locations(cafile=CERTS_DIR / 'internal-ca.pem')
    
    kwargs['ssl'] = ssl_context

app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

#### ASVS-1234-HIGH-002: No Certificate Trust Configuration for Outbound OAuth Connection

**Severity:** HIGH  
**CWE:** CWE-295 (Improper Certificate Validation)  
**File:** `v3/server/main.py`  
**Function:** `create_app()`  
**Lines:** 42-45

**Gap Type:** Type A — Entry point with NO control

**Vulnerable Code:**
```python
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Analysis:**  
The server makes outbound HTTPS connections to `oauth.apache.org` for authentication token exchange. ASVS 12.3.4 requires that TLS connections use trusted certificates and, where applicable, trust only specific CAs. The code configures OAuth URLs but provides:
- **No explicit SSL context** for the outbound HTTP client
- **No CA certificate pinning** for the OAuth endpoint
- **No custom trust store** limiting which CAs can sign the OAuth server's certificate

While Python 3's default `ssl` module validates certificates against the system trust store, ASVS 12.3.4's principle of trusting *specific* CAs is not met. The OAuth token exchange carries authorization codes — if an attacker compromises any CA in the system trust store, they could MITM this connection.

Since the `asfquart.generics` module handles the actual HTTP request (source not provided), we cannot confirm whether certificate validation is enforced or potentially disabled (e.g., `verify=False`).

**Data Flow:**  
`OAuth URL constants` → `asfquart.generics HTTP client` → `https://oauth.apache.org` → **Missing: explicit certificate trust/pinning configuration**

**Impact:**  
If the outbound HTTP client doesn't properly validate the OAuth server's certificate, or if any trusted CA is compromised, an attacker could intercept authorization codes and tokens during the OAuth callback flow, leading to authentication bypass.

**Remediation:**
```python
import ssl
import certifi

# Create a dedicated SSL context for outbound OAuth connections
oauth_ssl_context = ssl.create_default_context(cafile=certifi.where())
oauth_ssl_context.minimum_version = ssl.TLSVersion.TLSv1_2
# Optionally pin to Let's Encrypt / specific CA for oauth.apache.org
# oauth_ssl_context.load_verify_locations(cafile='certs/oauth-ca-bundle.pem')

# Pass to asfquart or configure the HTTP client used for OAuth
asfquart.generics.OAUTH_SSL_CONTEXT = oauth_ssl_context
```

### Medium Severity

#### ASVS-1234-MED-001: TLS Is Optional — Server Permits Plain HTTP Operation for Internal Service Communication

**Severity:** MEDIUM  
**CWE:** CWE-319 (Cleartext Transmission of Sensitive Information)  
**File:** `v3/server/config.yaml.example` (lines 30-32), `v3/server/main.py` (lines 83-86)  
**Function:** `run_standalone()`

**Gap Type:** Type B — Control EXISTS but NOT CALLED (when config is blank)

**Vulnerable Code (config.yaml.example):**
```yaml
# Specify the .pem files to serve using TLS, or leave these two
# fields blank for plain HTTP. These files are relative to the
# "certs" subdirectory of main.py (the app), or they can be absolute
# paths to their installed location.
certfile: localhost.apache.org+3.pem
keyfile: localhost.apache.org+3-key.pem
```

**Vulnerable Code (main.py):**
```python
kwargs = {}
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
```

**Analysis:**  
The TLS configuration is entirely optional. The example config explicitly documents that leaving `certfile`/`keyfile` blank results in plain HTTP. In `main.py`, the conditional `if app.cfg.server.certfile:` means the server silently degrades to unencrypted HTTP without any warning or startup failure.

ASVS 12.3.4 requires that TLS connections between internal services use trusted certificates. The config comments confirm "a proxy sits in front of this server" — meaning the proxy-to-application connection is an internal service link that should always be encrypted.

**Impact:**  
If deployed with blank TLS configuration (or if configuration is accidentally cleared), all internal communication between the reverse proxy and the application server occurs in plaintext. Authentication tokens, OAuth credentials, vote data, and session cookies would be transmitted unencrypted on the internal network.

**Remediation:**
```python
kwargs = {}
if app.cfg.server.certfile and app.cfg.server.keyfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
    extra_files.update((kwargs['certfile'], kwargs['keyfile']))
else:
    _LOGGER.critical(
        'TLS is not configured! Set server.certfile and server.keyfile '
        'in config.yaml. Refusing to start without TLS.'
    )
    sys.exit(1)
```

Additionally, update `config.yaml.example` to remove the "leave blank for plain HTTP" guidance:
```yaml
# REQUIRED: Specify the .pem files to serve using TLS.
# The server will not start without valid TLS configuration.
certfile: localhost.apache.org+3.pem
keyfile: localhost.apache.org+3-key.pem
```

## Architecture Observations

1. **Reverse Proxy Assumption:** Comments indicate a reverse proxy is expected ("a proxy sits in front of this server"). The proxy-to-app connection is the primary internal service-to-service link relevant to ASVS 12.3.4.

2. **ASGI Mode Gap:** When running under Hypercorn (`run_asgi()`), TLS configuration is entirely absent from this code. It depends on Hypercorn's external configuration, which is not visible or auditable here. This creates a risk that ASGI deployments may lack TLS.

3. **Development Certificate Names:** The example cert names (`localhost.apache.org+3.pem`) suggest mkcert-generated development certificates. There is no guidance for production certificate setup, CA trust configuration, or certificate rotation procedures.

4. **No Database Network TLS:** SQLite is file-based, so database TLS is not applicable (correctly identified in domain context).
