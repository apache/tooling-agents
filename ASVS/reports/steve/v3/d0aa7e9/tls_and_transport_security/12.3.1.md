## Findings

### Critical Severity

#### ASVS-1231-SEV-001: Application Explicitly Permits Plain HTTP Operation

**Severity:** CRITICAL  
**Gap Type:** Type B — TLS control EXISTS but is conditionally NOT APPLIED

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `run_standalone()`
- **Lines:** 77-80
- **Supporting File:** `v3/server/config.yaml.example`, lines 27-30

**Related Functions Checked:**
- `run_standalone()` (L77-80)
- `create_app()` (OAuth URL configuration)

**Vulnerable Code (main.py):**
```python
    kwargs = {}
    if app.cfg.server.certfile:
        kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
        kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
        extra_files.update((kwargs['certfile'], kwargs['keyfile']))

    # Spool up the app!
    app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Vulnerable Documentation (config.yaml.example):**
```yaml
    # Specify the .pem files to serve using TLS, or leave these two
    # fields blank for plain HTTP. These files are relative to the
    # "certs" subdirectory of main.py (the app), or they can be absolute
    # paths to their installed location.
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
```

**Data Flow:**
```
config.yaml (certfile: "")
    ↓
app.cfg.server.certfile evaluates falsy
    ↓
kwargs dict remains empty
    ↓
app.runx() starts without TLS
    ↓
plain HTTP listener on configured port
```

**Proof of Concept:**
```yaml
# config.yaml with TLS disabled
server:
    port: 58383
    certfile:
    keyfile:
db: steve.db
```
The server starts serving all election data, authentication tokens, and votes over unencrypted HTTP. Any network observer can intercept OAuth tokens, session cookies, and vote contents.

**Impact:**
- Complete loss of confidentiality for all data in transit including authentication credentials, session tokens, and encrypted vote payloads
- ASVS 12.3.1 explicitly requires: "The server must not fall back to insecure or unencrypted protocols"
- The application's vote encryption becomes moot if votes transit the network in cleartext before reaching the encryption layer
- An election system serving voters over plain HTTP enables trivial session hijacking and vote manipulation

**Remediation:**

The application must refuse to start without valid TLS configuration. Add a startup check:

```python
def run_standalone():
    # ... existing setup ...
    app = create_app()

    if not app.cfg.server.certfile or not app.cfg.server.keyfile:
        _LOGGER.critical("TLS certificate and key are REQUIRED. "
                         "Configure server.certfile and server.keyfile in config.yaml. "
                         "Plain HTTP is not permitted.")
        sys.exit(1)

    certfile_path = CERTS_DIR / app.cfg.server.certfile
    keyfile_path = CERTS_DIR / app.cfg.server.keyfile

    if not certfile_path.exists() or not keyfile_path.exists():
        _LOGGER.critical("TLS certificate or key file not found at %s / %s",
                         certfile_path, keyfile_path)
        sys.exit(1)

    kwargs = {
        'certfile': certfile_path,
        'keyfile': keyfile_path,
    }
    # ... rest of startup
```

Update `config.yaml.example` to remove the plain HTTP option documentation:
```yaml
    # TLS certificate and key files (REQUIRED). Relative to "certs"
    # subdirectory or absolute paths.
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
```

### High Severity

#### ASVS-1231-SEV-002: No TLS Protocol Version Enforcement (TLS 1.0/1.1 Potentially Accepted)

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `run_standalone()`
- **Lines:** 77-82

**Related Functions Checked:**
- `run_standalone()` (L77-82)
- `app.runx()` (L82)

**Vulnerable Code:**
```python
    kwargs = {}
    if app.cfg.server.certfile:
        kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
        kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
        extra_files.update((kwargs['certfile'], kwargs['keyfile']))

    app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Analysis:**

When TLS is enabled, only `certfile` and `keyfile` paths are passed to `app.runx()`. There is no `ssl.SSLContext` object created with:
- `ssl.TLSVersion.TLSv1_2` as `minimum_version`
- Disabled `SSLv3`, `TLSv1`, `TLSv1_1` protocols
- No `ssl_context` parameter passed to the server

Without explicit protocol version restrictions, the underlying framework and Python's `ssl` module defaults determine which TLS versions are accepted. While modern Python (3.10+) has reasonable defaults, the application takes no explicit action to enforce TLS 1.2+ as required by current security standards.

**Data Flow:**
```
config.yaml (certfile set)
    ↓
kwargs = {certfile, keyfile} only
    ↓
app.runx(**kwargs)
    ↓
framework creates ssl_context with default protocol settings
    ↓
potentially accepts TLS 1.0/1.1 connections
```

**Proof of Concept:**
```bash
# Test if TLS 1.1 is accepted (should be rejected)
openssl s_client -connect localhost:58383 -tls1_1

# Test if TLS 1.0 is accepted (should be rejected)
openssl s_client -connect localhost:58383 -tls1
```

**Impact:**
- TLS 1.0 and 1.1 have known vulnerabilities (BEAST, POODLE, Lucky13)
- Clients using deprecated TLS versions could be attacked via protocol downgrade
- Non-compliance with PCI DSS, NIST SP 800-52 Rev. 2, and ASVS requirements

**Remediation:**

Create an explicit `ssl.SSLContext` with hardened settings:

```python
import ssl

def create_ssl_context(certfile, keyfile):
    """Create a hardened SSL context enforcing TLS 1.2+."""
    ctx = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
    ctx.minimum_version = ssl.TLSVersion.TLSv1_2
    ctx.maximum_version = ssl.TLSVersion.TLSv1_3

    # Disable compression (CRIME attack)
    ctx.options |= ssl.OP_NO_COMPRESSION
    # Server chooses cipher order
    ctx.options |= ssl.OP_CIPHER_SERVER_PREFERENCE

    # Strong cipher suites only
    ctx.set_ciphers(
        'ECDHE+AESGCM:ECDHE+CHACHA20:DHE+AESGCM:DHE+CHACHA20'
        ':!aNULL:!MD5:!DSS:!RC4:!3DES'
    )

    ctx.load_cert_chain(certfile=certfile, keyfile=keyfile)
    return ctx
```

Then pass `ssl_context=ctx` to the server startup instead of raw cert paths.

#### ASVS-1231-SEV-003: ASGI Mode Lacks Any TLS Configuration

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `run_asgi()`
- **Lines:** 91-109

**Related Functions Checked:**
- `run_asgi()` (L91-109)
- `create_app()` (called from run_asgi)

**Vulnerable Code:**
```python
def run_asgi():
    "Run as an ASGI process; eg. Hypercorn"

    # NOTE: no-op if Hypercorn has set up the root logger.
    logging.basicConfig(
        level=logging.DEBUG,
        style='{',
        format='[{asctime}|{levelname}|{name}] {message}',
        datefmt=DATE_FORMAT,
    )

    logging.getLogger('watchfiles.main').setLevel(logging.INFO)
    _LOGGER.setLevel(logging.DEBUG)

    _LOGGER.info(' ** Run-mode: ASGI')

    global steve_app
    steve_app = create_app()
```

**Analysis:**

When the application runs in ASGI mode (Hypercorn), no TLS configuration is performed whatsoever. Unlike `run_standalone()` which at least optionally configures TLS, the ASGI path:
- Does not read or validate the `certfile`/`keyfile` configuration
- Does not create or pass any ssl_context
- Provides no Hypercorn TLS configuration file or binding configuration
- The documentation comment at the bottom suggests running as `uv run python -m hypercorn main:steve_app` with no `--certfile`/`--keyfile` flags

The config.yaml.example comment states "Typical usage is that a proxy sits in front of this server," but there is no enforcement that the proxy provides TLS, and no mechanism to reject non-TLS connections in ASGI mode.

**Data Flow:**
```
ASGI mode startup
    ↓
run_asgi() called
    ↓
create_app() with no TLS validation
    ↓
Hypercorn starts with default (non-TLS) configuration
    ↓
plain HTTP listener
```

**Impact:**
- ASGI deployment runs entirely over plain HTTP by default
- No documented Hypercorn TLS configuration exists in the codebase
- Even if a reverse proxy provides external TLS, the proxy-to-application link is unencrypted
- ASVS 12.3.1 requires encrypted protocols for ALL connections, including internal ones

**Remediation:**

Provide and enforce TLS configuration for Hypercorn deployments:

```python
def run_asgi():
    "Run as an ASGI process; eg. Hypercorn"
    # ... existing logging setup ...

    _LOGGER.info(' ** Run-mode: ASGI')

    global steve_app
    steve_app = create_app()

    # Validate TLS configuration exists even in ASGI mode
    if not steve_app.cfg.server.certfile or not steve_app.cfg.server.keyfile:
        _LOGGER.critical(
            "TLS configuration is required. When using Hypercorn, pass "
            "--certfile and --keyfile flags, or configure in hypercorn.toml"
        )
        sys.exit(1)
```

Additionally, provide a `hypercorn.toml` in the repository:
```toml
# hypercorn.toml
certfile = "certs/localhost.apache.org+3.pem"
keyfile = "certs/localhost.apache.org+3-key.pem"
bind = ["0.0.0.0:58383"]
```

## Positive Security Patterns

1. **HTTPS for OAuth endpoints**: Both `OAUTH_URL_INIT` and `OAUTH_URL_CALLBACK` use `https://` scheme, ensuring outbound OAuth communication is encrypted.

2. **Example config includes TLS by default**: The `config.yaml.example` ships with certificate file paths populated, meaning the default example configuration has TLS enabled.

3. **Certificate paths use pathlib**: Certificate paths are constructed using `pathlib.Path` operations (`CERTS_DIR / app.cfg.server.certfile`), which is safer than string concatenation.

4. **Dedicated certs directory**: Certificate files are organized in a dedicated `certs/` subdirectory, supporting clean file permission management.

---

## Architecture Observations

1. **Reverse Proxy Assumption**: The comment "Typical usage is that a proxy sits in front of this server" suggests TLS termination may be handled externally. However, ASVS 12.3.1 requires encryption for ALL connections including internal ones between proxy and application server. The application should still enforce TLS or explicitly validate that it's running behind a trusted proxy.

2. **Two Execution Modes with Different Security Postures**: The standalone and ASGI modes have completely different TLS handling. Standalone at least supports TLS; ASGI delegates it entirely to the deployment configuration with no validation.

3. **Framework Delegation**: TLS implementation is largely delegated to `asfquart` (likely wrapping Quart/Hypercorn). Without visibility into that framework's ssl_context creation, it's unclear whether passing only `certfile`/`keyfile` results in a hardened TLS configuration.
