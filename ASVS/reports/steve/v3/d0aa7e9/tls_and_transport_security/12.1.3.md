## Findings

### Medium Severity

#### ASVS-1213-MED-001: No SSL Context Configuration Prevents mTLS Client Certificate Validation

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with NO control  
**File:** `v3/server/main.py`  
**Function:** `run_standalone()`  
**Lines:** 83-90  
**Related Config:** `v3/server/config.yaml.example`, lines 28-30

**Related Functions Checked:**
- `create_app()` (L39-42) — OAuth setup only, no TLS context
- `run_asgi()` (L99-118) — No TLS configuration

**Description:**

The TLS configuration passes raw `certfile` and `keyfile` paths directly to `app.runx()` without constructing an `ssl.SSLContext`. This has several consequences directly relevant to ASVS 12.1.3:

1. **No client certificate verification**: There is no `ssl.SSLContext.verify_mode = ssl.CERT_REQUIRED` and no trusted CA (`ca_certs`) configured, so client certificates are never requested or validated.
2. **No configuration surface for mTLS**: The `config.yaml` schema has no fields for CA certificates, certificate verification mode, or CRL/OCSP configuration.
3. **No TLS version floor**: Without an explicit context, the server may accept TLS 1.0/1.1 depending on the underlying framework defaults.
4. **No cipher restrictions**: Default ciphers are used, which may include weak suites.

**Vulnerable Code:**

```python
# main.py lines 83-90
kwargs = {}
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
    extra_files.update((kwargs['certfile'], kwargs['keyfile']))

# Spool up the app!
app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

```yaml
# config.yaml.example lines 28-30
certfile: localhost.apache.org+3.pem
keyfile: localhost.apache.org+3-key.pem
# No ca_certs, no verify_mode, no tls_version_min, no ciphers
```

**Data Flow:**
```
Inbound TLS connections 
  → Quart/Hypercorn listener 
  → No SSL context 
  → No client cert request 
  → No cert validation 
  → Authentication relies solely on OAuth session cookies
```

**Proof of Concept:**

A client connecting to the server will never be prompted for a client certificate, regardless of whether they possess one:

```bash
# Client connection - no certificate requested
curl -v https://localhost:58383/
# Output shows no "Client certificate requested" message
```

**Impact:**
- Cannot implement mutual TLS authentication for service-to-service or client-to-server verification
- If the system were deployed in an environment expecting mTLS (e.g., service mesh, zero-trust network), client certificate identities would not be validated before use
- The missing SSL context also prevents enforcement of TLS 1.2+ and strong ciphers
- For a voting/election system handling authenticated ballot submission, the inability to layer mTLS as a defense-in-depth authentication mechanism is a notable gap

**Remediation:**

Create an explicit `ssl.SSLContext` with proper configuration and provide mTLS configuration options:

**Step 1:** Update configuration schema in `config.yaml`:

```yaml
# config.yaml - add mTLS and TLS hardening options
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
    # mTLS configuration (optional)
    ca_certs: ca-bundle.pem        # Trusted CA for client certificates
    verify_client: false            # Set true to require client certs
    # TLS hardening
    tls_min_version: "1.2"
    ciphers: "ECDHE+AESGCM:ECDHE+CHACHA20:DHE+AESGCM:DHE+CHACHA20"
```

**Step 2:** Implement SSL context creation in `main.py`:

```python
# main.py - create explicit SSL context
import ssl

def _create_ssl_context(cfg) -> ssl.SSLContext:
    """Create a hardened SSL context with optional mTLS support."""
    context = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
    
    # Enforce minimum TLS version
    context.minimum_version = ssl.TLSVersion.TLSv1_2
    context.maximum_version = ssl.TLSVersion.TLSv1_3
    
    # Load server certificate
    certfile = CERTS_DIR / cfg.server.certfile
    keyfile = CERTS_DIR / cfg.server.keyfile
    context.load_cert_chain(certfile=certfile, keyfile=keyfile)
    
    # Configure cipher suites
    context.set_ciphers('ECDHE+AESGCM:ECDHE+CHACHA20:DHE+AESGCM:DHE+CHACHA20')
    
    # mTLS: Require and validate client certificates if configured
    if getattr(cfg.server, 'verify_client', False):
        ca_certs = CERTS_DIR / cfg.server.ca_certs
        context.load_verify_locations(cafile=ca_certs)
        context.verify_mode = ssl.CERT_REQUIRED  # CRITICAL: validate before use
        _LOGGER.info('mTLS enabled: client certificates required and validated')
    else:
        context.verify_mode = ssl.CERT_NONE
    
    return context


def run_standalone():
    # ... existing setup ...
    app = create_app()
    
    kwargs = {}
    if app.cfg.server.certfile:
        ssl_context = _create_ssl_context(app.cfg)
        kwargs['ssl'] = ssl_context
    
    app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

#### ASVS-1213-MED-002: ASGI Deployment Mode Has No Application-Level TLS Controls

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with NO control  
**File:** `v3/server/main.py`  
**Function:** `run_asgi()`  
**Lines:** 99-118

**Related Functions Checked:**
- `create_app()` (L39-42) — No TLS validation
- `run_standalone()` (L83-90) — Has TLS config but only for standalone mode

**Description:**

When deployed via Hypercorn (the documented production path), TLS configuration is entirely delegated to the external ASGI server with no application-enforced constraints. The application:

1. Does not configure or pass any SSL context to the ASGI runner
2. Does not verify at startup that the ASGI server has TLS enabled
3. Provides no documentation or configuration validation for Hypercorn's mTLS settings (`--ca-certs`, `--verify-mode`)

This means the application relies entirely on operational configuration to enforce TLS and mTLS, with no programmatic safeguard ensuring client certificates are validated when used.

**Vulnerable Code:**

```python
# main.py lines 99-118
def run_asgi():
    # ... logging setup ...
    _LOGGER.info(' ** Run-mode: ASGI')
    global steve_app
    steve_app = create_app()
    # No TLS configuration whatsoever
```

**Data Flow:**
```
External ASGI server (Hypercorn)
  → TLS termination (if configured externally)
  → Application receives connection
  → No validation that TLS was used
  → No validation that client cert was checked
```

**Proof of Concept:**

Hypercorn can be started without TLS, and the application will not detect or prevent this:

```bash
# Insecure deployment - application does not prevent this
hypercorn main:steve_app --bind 0.0.0.0:58383
# Server runs over plain HTTP with no warning
```

**Impact:**
- Misconfigured Hypercorn deployment could serve over plain HTTP or accept unverified client certificates
- No defense-in-depth: if operational TLS config is wrong, the application has no fallback check
- For a voting system, this could allow unauthenticated or MITM'd connections
- Silent security degradation: no warning when deployed insecurely

**Remediation:**

Add startup validation that TLS is properly configured in ASGI mode:

```python
def run_asgi():
    _LOGGER.info(' ** Run-mode: ASGI')
    global steve_app
    steve_app = create_app()
    
    # Warn if no TLS configuration is detected
    # (Hypercorn sets these environment variables)
    import os
    if not os.environ.get('HYPERCORN_SSL_CERTFILE'):
        _LOGGER.warning(
            'SECURITY: No TLS certificate detected in ASGI mode. '
            'Ensure Hypercorn is configured with --certfile and --keyfile. '
            'For mTLS, also configure --ca-certs and --verify-mode.'
        )
    
    # Document required Hypercorn configuration
    _LOGGER.info(
        'For secure production deployment, run Hypercorn with: '
        '--certfile <cert.pem> --keyfile <key.pem> '
        '--ca-certs <ca-bundle.pem> --verify-mode CERT_REQUIRED'
    )
```

Additionally, create deployment documentation:

```markdown
# docs/deployment.md

## Production Deployment with mTLS

To enable mutual TLS authentication in production:

1. Configure Hypercorn with TLS certificates:
   ```bash
   hypercorn main:steve_app \
     --bind 0.0.0.0:58383 \
     --certfile /path/to/server-cert.pem \
     --keyfile /path/to/server-key.pem \
     --ca-certs /path/to/client-ca-bundle.pem \
     --verify-mode CERT_REQUIRED
   ```

2. Ensure client CA bundle contains all trusted certificate authorities
3. Monitor logs for certificate validation failures
```

## Architecture Observations

1. **Authentication Model**: The application uses Apache OAuth (`oauth.apache.org`) for all user authentication, not mTLS client certificates. This means ASVS 12.1.3 has limited direct applicability to the current authentication model — but the absence of any SSL context configuration still represents a gap in TLS hardening and prevents future mTLS implementation for defense-in-depth.

2. **Dual Deployment Modes**: Standalone mode (`run_standalone`) has code-level TLS configuration; ASGI mode (`run_asgi`) has none. This inconsistency means security posture depends heavily on which deployment mode is used and how the external ASGI server is configured. Production deployments using ASGI have no application-level TLS validation.

3. **Proxy Architecture**: The config comment "Typical usage is that a proxy sits in front of this server" (line 22) suggests TLS termination may occur at a reverse proxy. If so, mTLS validation would need to happen at the proxy layer and be forwarded via headers (e.g., `X-Client-Cert`) — but there is no code to validate such forwarded certificate identity headers, creating a potential spoofing vector.

4. **Framework Dependency**: The `asfquart` framework abstracts significant security behavior (OAuth flow, session management, TLS termination). A complete audit of mTLS capability requires reviewing `asfquart.construct()` and `app.runx()` implementations to understand what TLS controls may be available at the framework level.

5. **Election System Context**: As a voting/ballot submission system, the application handles highly sensitive operations. While OAuth provides user authentication, the lack of mTLS capability prevents implementing defense-in-depth through cryptographic client authentication, which would be valuable for service-to-service communication or high-security deployment environments.
