## Findings

### High Severity

#### ASVS-1233-HIGH-001: Server Permits Fallback to Unencrypted HTTP Communication

**Severity:** HIGH  
**Gap Type:** Type B – Control EXISTS but conditionally NOT CALLED  
**Location:** `v3/server/main.py` lines 98–104; `v3/server/config.yaml.example` lines 28–31

**Vulnerable Code (`main.py`):**
```python
kwargs = {}
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
    extra_files.update((kwargs['certfile'], kwargs['keyfile']))

# Spool up the app!
app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Vulnerable Configuration Comment (`config.yaml.example`):**
```yaml
# Specify the .pem files to serve using TLS, or leave these two
# fields blank for plain HTTP. These files are relative to the
```

**Data Flow:**
```
config.yaml (empty certfile)
  → app.cfg.server.certfile evaluates falsy
  → kwargs remains {}
  → app.runx() starts without TLS
  → Server listens on plain HTTP
```

**Related Functions Checked:**
- `run_standalone()` — TLS configuration logic
- `create_app()` — Application initialization
- Configuration loading from `config.yaml`

**Analysis:**

The TLS control exists but is entirely optional. The configuration file explicitly documents that leaving `certfile` and `keyfile` blank results in plain HTTP. There is no validation, warning, or error when the server starts without TLS. Any misconfiguration, accidental blank, or copy of the example config without modification results in an unencrypted service endpoint. This directly violates ASVS 12.3.3's requirement that communications "does not fall back to insecure or unencrypted communications."

For an election/voting system, this is particularly dangerous because authentication tokens, session cookies, and vote submissions would transit in cleartext.

**Proof of Concept:**
1. Deploy with `config.yaml` containing `certfile:` and `keyfile:` set to empty strings
2. Server starts on port 58383 over plain HTTP
3. All internal and external communication to this service is unencrypted
4. Network observer can capture OAuth tokens, session data, and votes

**Impact:**
- Authentication credentials (OAuth tokens, session cookies) exposed in transit
- Vote data transmitted in cleartext between client and server
- Man-in-the-middle attacks possible against the voting service
- Complete compromise of voter anonymity on the network path

**Remediation:**
```python
def run_standalone():
    # ... logging setup ...
    app = create_app()
    steve = sys.modules['steve']
    extra_files = {steve.election.QUERIES}

    # REQUIRE TLS - do not allow fallback to plain HTTP
    if not app.cfg.server.certfile or not app.cfg.server.keyfile:
        _LOGGER.critical(
            "TLS certificate and key MUST be configured. "
            "Set server.certfile and server.keyfile in config.yaml. "
            "Plain HTTP is not permitted."
        )
        sys.exit(1)

    kwargs = {
        'certfile': CERTS_DIR / app.cfg.server.certfile,
        'keyfile': CERTS_DIR / app.cfg.server.keyfile,
    }
    extra_files.update((kwargs['certfile'], kwargs['keyfile']))

    # Verify cert files exist before starting
    for key in ('certfile', 'keyfile'):
        if not kwargs[key].exists():
            _LOGGER.critical("TLS %s not found: %s", key, kwargs[key])
            sys.exit(1)

    app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

Update `config.yaml.example` to remove the "leave blank for plain HTTP" guidance:
```yaml
    # TLS certificate and key files are REQUIRED.
    # Paths are relative to the "certs" subdirectory, or absolute paths.
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
```

### Medium Severity

#### ASVS-1233-MED-001: No TLS Protocol Version or Cipher Suite Enforcement

**Severity:** MEDIUM  
**Gap Type:** Type A – Entry point with NO control  
**Location:** `v3/server/main.py` lines 98–104

**Vulnerable Code:**
```python
kwargs = {}
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
    extra_files.update((kwargs['certfile'], kwargs['keyfile']))

app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Related Functions Checked:**
- `run_standalone()` — TLS listener configuration
- `app.runx()` — Framework server startup

**Analysis:**

When TLS is enabled, the code passes only `certfile` and `keyfile` to `app.runx()` without creating or configuring an `ssl.SSLContext`. This means:

1. **No minimum TLS version enforcement** — The underlying framework's default SSL context is used. While modern Python (3.10+) defaults to TLS 1.2 minimum, this is not explicitly enforced and could vary by deployment environment or framework version.
2. **No cipher suite restriction** — No explicit cipher configuration means the full set of default ciphers is available, which may include weaker options.
3. **No explicit certificate validation parameters** — No configuration of certificate chain verification for mutual TLS or other hardening.

The application delegates all SSL/TLS security decisions to framework defaults, providing no defense-in-depth against protocol downgrade attacks or weak cipher negotiation.

**Data Flow:**
```
TLS configuration
  → certfile/keyfile passed to app.runx()
  → Framework default SSL context used
  → No explicit TLS version or cipher restrictions
  → Potentially weak configuration accepted
```

**Impact:**
- Potential acceptance of TLS 1.0/1.1 connections depending on Python/framework version
- Weaker cipher suites may be negotiated by clients
- No explicit security posture for the TLS listener

**Remediation:**
```python
import ssl

def create_ssl_context(certfile, keyfile):
    """Create a hardened SSL context for the server."""
    ctx = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
    ctx.minimum_version = ssl.TLSVersion.TLSv1_2
    ctx.maximum_version = ssl.TLSVersion.TLSv1_3
    
    # Strong cipher suites only
    ctx.set_ciphers(
        'ECDHE+AESGCM:ECDHE+CHACHA20:DHE+AESGCM:DHE+CHACHA20'
        ':!aNULL:!MD5:!DSS:!RC4:!3DES'
    )
    
    ctx.load_cert_chain(certfile=certfile, keyfile=keyfile)
    return ctx

# In run_standalone():
ssl_context = create_ssl_context(
    CERTS_DIR / app.cfg.server.certfile,
    CERTS_DIR / app.cfg.server.keyfile,
)
app.runx(port=app.cfg.server.port, extra_files=extra_files, ssl=ssl_context)
```

#### ASVS-1233-MED-002: ASGI Mode Lacks TLS Configuration or Validation

**Severity:** MEDIUM  
**Gap Type:** Type A – Entry point with NO control  
**Location:** `v3/server/main.py` lines 115–126

**Vulnerable Code:**
```python
def run_asgi():
    "Run as an ASGI process; eg. Hypercorn"

    # ... logging setup ...

    _LOGGER.info(' ** Run-mode: ASGI')

    global steve_app
    steve_app = create_app()
```

**Related Functions Checked:**
- `run_asgi()` — ASGI mode initialization
- `create_app()` — Application factory
- Hypercorn invocation (documented in comments)

**Analysis:**

The ASGI execution path creates the application without any TLS configuration or validation. When deployed behind Hypercorn, TLS must be configured entirely through Hypercorn's own configuration (command line flags or `hypercorn.toml`). The application performs no check that TLS is actually active in the ASGI server environment.

While it's architecturally common for ASGI apps to delegate TLS to the server or reverse proxy, for a security-sensitive voting application, the lack of any runtime validation means the application cannot detect or warn if it's inadvertently served over plain HTTP.

The Hypercorn invocation shown in the code comments also lacks any TLS flags:
```python
#   $ uv run python -m hypercorn main:steve_app
```

**Data Flow:**
```
ASGI mode startup
  → No TLS configuration in application
  → Delegates to Hypercorn/reverse proxy
  → No validation that TLS is active
  → Silent unencrypted deployment possible
```

**Impact:**
- ASGI deployment may run without TLS if Hypercorn is not separately configured
- No application-level detection of missing transport encryption
- Example invocation in comments suggests unencrypted deployment

**Remediation:**
```python
def run_asgi():
    "Run as an ASGI process; eg. Hypercorn"
    # ... logging setup ...

    _LOGGER.info(' ** Run-mode: ASGI')

    global steve_app
    steve_app = create_app()

    # Warn if TLS certificates are configured but ASGI mode won't use them directly
    if steve_app.cfg.server.certfile:
        _LOGGER.warning(
            "TLS cert/key configured in config.yaml but running in ASGI mode. "
            "Ensure Hypercorn or the reverse proxy is configured with TLS. "
            "Example: hypercorn --certfile %s --keyfile %s main:steve_app",
            CERTS_DIR / steve_app.cfg.server.certfile,
            CERTS_DIR / steve_app.cfg.server.keyfile,
        )
    else:
        _LOGGER.warning(
            "No TLS certificate configured. Ensure TLS is provided by "
            "the reverse proxy or ASGI server."
        )
```

Update comments to show secure invocation:
```python
    # Using Hypercorn with TLS:
    #   $ uv run python -m hypercorn --certfile certs/cert.pem --keyfile certs/key.pem main:steve_app
```

## Architecture Observations

1. **Dual execution model creates inconsistent TLS handling.** Standalone mode has optional TLS configuration; ASGI mode has none. Neither mode enforces encryption.

2. **Proxy assumption is undocumented.** The config comment mentions "a proxy sits in front of this server" but there's no enforcement that the proxy provides TLS when the application doesn't.

3. **SQLite is file-based (no network DB connection).** This means the database communication path is not a transport encryption concern — correctly excluded from findings.

4. **No LDAP integration visible in provided code.** Cannot assess LDAPS enforcement from the files provided.

## Recommendations Summary

### Immediate Actions (Priority 1)
- **Require TLS configuration** — Fail startup if certfile/keyfile are missing or empty (addresses ASVS-1233-HIGH-001)
- **Update config.yaml.example** — Remove "leave blank for HTTP" language, document TLS as mandatory (addresses ASVS-1233-HIGH-001)

### Short-term Actions (Priority 2-3)
- **Create explicit SSLContext** with TLS 1.2+ minimum and strong cipher suites (addresses ASVS-1233-MED-001)
- **Add ASGI mode TLS guidance** — Runtime warnings and secure example commands (addresses ASVS-1233-MED-002)

### Long-term Actions (Priority 4-5)
- **Add HSTS headers** to prevent protocol downgrade at the HTTP layer (defense-in-depth)
- Consider certificate rotation and monitoring mechanisms
