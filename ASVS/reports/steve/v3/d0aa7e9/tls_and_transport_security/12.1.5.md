## Findings

### High Severity

#### ASVS-1215-HIGH-001: Encrypted Client Hello (ECH) Not Implemented

| Attribute | Detail |
|-----------|--------|
| **Severity** | **HIGH** |
| **Finding ID** | ASVS-1215-HIGH-001 |
| **File** | `v3/server/main.py` lines 82-88; `v3/server/config.yaml.example` lines 28-31 |
| **Function** | `run_standalone()` |
| **Gap Type** | Type A — Entry point with NO control |

**Related Functions Checked:**
- `run_standalone()` (main.py L82-88)
- Configuration parsing (config.yaml.example L28-31)

**Description:**

The TLS setup passes raw file paths for certificate and key to `app.runx()`. There is:

1. **No ECH key pair** generated or referenced in `certs/` directory or configuration
2. **No `ech_config`** or equivalent parameter in TLS settings
3. **No `ssl.SSLContext`** created where ECH could be enabled
4. **No DNS HTTPS record** guidance or ECHConfig publication mechanism
5. **No ECH retry configuration** for client compatibility

Without ECH, the Server Name Indication (SNI) field is transmitted in plaintext during the TLS ClientHello, allowing network observers to identify which specific server/election the client is connecting to. For a voting system, this metadata leakage can reveal voter participation patterns.

**Vulnerable Configuration (config.yaml.example):**
```yaml
server:
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
```

**Vulnerable Code (main.py lines 82-88):**
```python
kwargs = {}
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
    extra_files.update((kwargs['certfile'], kwargs['keyfile']))

app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Data Flow:**
```
Client → TLS ClientHello (SNI: "voting.apache.org" in PLAINTEXT) → Network Observer can see target hostname → Server
```

**Proof of Concept:**
```bash
# An observer on the network path can capture SNI using:
tshark -i eth0 -f "tcp port 58383" -T fields -e tls.handshake.extensions_server_name
# Output: voting.apache.org (exposed without ECH)
```

**Impact:**
- Network observers (ISPs, corporate firewalls, nation-state actors) can determine which users are accessing the voting application
- In a voting context, this reveals voter participation metadata even if vote content is encrypted
- Violates privacy expectations of an election system

**Remediation:**

ECH requires server-side support in the TLS library and DNS publication. For Python applications using Hypercorn:

```yaml
# config.yaml - Add ECH configuration
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
    # ECH configuration
    ech_keyfile: ech_key.pem
    ech_config_list: ech_configs.bin
```

```python
# main.py - Create SSL context with ECH support
import ssl

def create_ssl_context(app):
    """Create hardened SSL context with ECH support."""
    ctx = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
    ctx.minimum_version = ssl.TLSVersion.TLSv1_3  # ECH requires TLS 1.3
    
    certfile = CERTS_DIR / app.cfg.server.certfile
    keyfile = CERTS_DIR / app.cfg.server.keyfile
    ctx.load_cert_chain(certfile=certfile, keyfile=keyfile)
    
    # ECH configuration (when supported by ssl module / OpenSSL 3.2+)
    # ctx.set_ech_keys(CERTS_DIR / app.cfg.server.ech_keyfile)
    
    # Until native Python ECH support is available, deploy behind a
    # reverse proxy (e.g., Cloudflare, nginx with OpenSSL 3.2+) that
    # supports ECH, and publish ECHConfig via DNS HTTPS records.
    
    return ctx

def run_standalone():
    # ... existing code ...
    app = create_app()
    
    if app.cfg.server.certfile:
        ssl_context = create_ssl_context(app)
        kwargs['ssl'] = ssl_context
    
    app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Recommended immediate approach:** Deploy behind a TLS-terminating reverse proxy (e.g., Cloudflare or nginx compiled with OpenSSL 3.2+) that supports ECH, and publish ECHConfig via DNS HTTPS resource records.

### Medium Severity

#### ASVS-1215-MED-001: No SSL Context Created — Prevents All TLS Hardening Including ECH

| Attribute | Detail |
|-----------|--------|
| **Severity** | **MEDIUM** |
| **Finding ID** | ASVS-1215-MED-001 |
| **File** | `v3/server/main.py` lines 82-88 |
| **Function** | `run_standalone()` |
| **Gap Type** | Type A — Foundational control missing |

**Related Functions Checked:**
- `run_standalone()` (main.py L82-88)
- `app.runx()` invocation (main.py L88)

**Description:**

The application passes raw certificate file paths rather than constructing an `ssl.SSLContext` object. This means:

1. **No TLS version minimum** is enforced (TLS 1.0/1.1 may be accepted depending on system defaults)
2. **No cipher suite restrictions** are applied
3. **No ECH can be configured** — ECH requires explicit SSL context manipulation
4. **Default system SSL settings** are inherited, which vary by OS and OpenSSL version

The lack of an SSL context is the foundational gap that prevents ECH implementation and all other TLS hardening.

**Vulnerable Code (main.py lines 82-88):**
```python
kwargs = {}
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
    extra_files.update((kwargs['certfile'], kwargs['keyfile']))

# Passes raw paths, no SSLContext
app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Impact:**
- Cannot enable ECH without an explicitly configured SSL context
- Potentially allows deprecated TLS versions and weak cipher suites
- System-dependent TLS security posture

**Remediation:**

```python
import ssl

def create_ssl_context(app):
    ctx = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
    ctx.minimum_version = ssl.TLSVersion.TLSv1_3
    ctx.set_ciphers(
        'ECDHE+AESGCM:ECDHE+CHACHA20:DHE+AESGCM:DHE+CHACHA20'
    )
    ctx.load_cert_chain(
        certfile=CERTS_DIR / app.cfg.server.certfile,
        keyfile=CERTS_DIR / app.cfg.server.keyfile,
    )
    return ctx
```

#### ASVS-1215-MED-002: ASGI Mode Has No TLS Configuration — ECH Cannot Be Applied

| Attribute | Detail |
|-----------|--------|
| **Severity** | **MEDIUM** |
| **Finding ID** | ASVS-1215-MED-002 |
| **File** | `v3/server/main.py` lines 95-115 |
| **Function** | `run_asgi()` |
| **Gap Type** | Type B — Control exists (standalone) but NOT applied (ASGI) |

**Related Functions Checked:**
- `run_asgi()` (main.py L107-115)
- `create_app()` (main.py L115)

**Description:**

When running in ASGI mode (Hypercorn), the `run_asgi()` function creates the app but performs **zero TLS configuration**. The standalone mode at least configures certfile/keyfile, but ASGI mode delegates all TLS settings entirely to the external Hypercorn configuration.

There is no:
- Documentation or example of required Hypercorn TLS/ECH configuration
- Validation that the ASGI server is configured with TLS
- Programmatic SSL context passed to the ASGI framework

**Vulnerable Code (main.py lines 107-115):**
```python
def run_asgi():
    "Run as an ASGI process; eg. Hypercorn"
    # ...logging setup...
    _LOGGER.info(' ** Run-mode: ASGI')

    global steve_app
    steve_app = create_app()
```

**Impact:**
- Operators deploying via Hypercorn receive no guidance on ECH or TLS configuration
- The application cannot verify or enforce ECH when run in production ASGI mode
- Creates a Type B gap: the standalone TLS control exists but is not applied in ASGI deployment

**Remediation:**

Provide Hypercorn configuration with ECH settings:

```toml
# hypercorn.toml
bind = ["0.0.0.0:58383"]
certfile = "certs/localhost.apache.org+3.pem"
keyfile = "certs/localhost.apache.org+3-key.pem"

# Enforce TLS 1.3 for ECH support
# ciphers = "ECDHE+AESGCM:ECDHE+CHACHA20"
```

Add runtime validation in `run_asgi()`:
```python
def run_asgi():
    _LOGGER.info(' ** Run-mode: ASGI')
    global steve_app
    steve_app = create_app()
    _LOGGER.warning(
        'ASGI mode: Ensure your ASGI server (Hypercorn) is configured with '
        'TLS 1.3 and ECH support. See deployment documentation.'
    )
```

## Architecture Observations

1. **Proxy Architecture Comment**: The config mentions "a proxy sits in front of this server" (line 25). If this proxy supports ECH (e.g., Cloudflare, nginx with OpenSSL 3.2+), ECH could be handled at the proxy layer. However, this is not documented, enforced, or verified.

2. **Dual Deployment Modes**: Standalone and ASGI modes have different TLS configuration paths, with ASGI mode having significantly less application-level control.

3. **Voting System Context**: For a voting application, SNI metadata exposure reveals which users are accessing the election system, creating a voter participation surveillance vector.
