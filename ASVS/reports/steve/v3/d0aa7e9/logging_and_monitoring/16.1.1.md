#### ASVS-1611-HIGH-001: Tally CLI Operations Lack Security Audit Trail

**Severity:** HIGH  
**Gap Type:** Type A (Entry point with NO control)

**Location:**
- **File:** `v3/server/bin/tally.py`
- **Functions:** `main()` (lines 136–160), `tally_election()` (lines 102–133)
- **Line Numbers:** 136-160, 102-133

**Related Functions Checked:**
- `main()` - Entry point for tally operations
- `tally_election()` - Core tallying logic
- `election.tally_issue()` - Vote decryption (called from `tally_election()`)
- `election.is_tampered()` - Integrity verification

**Description:**  
The tally operation—decrypting all votes and computing election results—is the most security-sensitive operation in the system. The CLI tool that performs it creates no meaningful security audit trail. There is no logging of who initiated the tally, no completion record, and no summary of results.

**Vulnerable Code:**

```python
# main() - lines 136-160
def main(spy_on_open, election_id, issue_id, db_fname, output_format):
    # ... election selection ...
    election = steve.election.Election(db_fname, election_id)

    pdb = steve.persondb.PersonDB.open(db_fname)
    if election.is_tampered(pdb):
        print(f'Error: Election {election_id} has been tampered with. Cannot proceed.')  # No security log!
        sys.exit(1)

    # Proceed with tally - NO log of who is running this
    tally_election(election, issue_id, output_format)
    # NO completion log
```

```python
# tally_election() - lines 102-133
def tally_election(election, issue_id, output_format):
    # ...
    for issue in issues:
        try:
            tally_result, issue_voters = election.tally_issue(issue.iid)
            # ... accumulate results, NO security logging ...
        except Exception as e:
            print(f'Error tallying issue {issue.iid}: {e}')  # print(), not logging!
            raise
```

**Data Flow:**  
`CLI execution` → `main()` → `tally_election()` → `election.tally_issue()` → vote decryption → results output — **no audit trail at any step**

**Impact:**  
- No forensic evidence of when tallying occurred or who performed it
- Tally operations cannot be audited for compliance
- If unauthorized tallying occurs, there is no detection mechanism
- Election results have no provenance chain

**Remediation:**
```python
import os
import getpass

def main(spy_on_open, election_id, issue_id, db_fname, output_format):
    operator = getpass.getuser()
    _LOGGER.info(
        f'Tally operation initiated by operator={operator} '
        f'pid={os.getpid()} for election[E:{election_id}] '
        f'issue_id={issue_id} spy_on_open={spy_on_open}'
    )

    # ... existing logic ...

    election = steve.election.Election(db_fname, election_id)

    pdb = steve.persondb.PersonDB.open(db_fname)
    if election.is_tampered(pdb):
        _LOGGER.critical(
            f'TAMPERING DETECTED: Election[E:{election_id}] failed integrity check. '
            f'Operator={operator}. Tally aborted.'
        )
        sys.exit(1)

    tally_election(election, issue_id, output_format)

    _LOGGER.info(
        f'Tally operation completed by operator={operator} '
        f'for election[E:{election_id}]'
    )
```

#### ASVS-1611-HIGH-002: Critical Security Events Bypass Logging Framework via print()

**Severity:** HIGH  
**Gap Type:** Type B (Control EXISTS but NOT CALLED — logging module is imported but `print()` is used for security events)

**Location:**
- **File:** `v3/server/bin/tally.py`, lines 124, 152
- **File:** `v3/server/pages.py`, lines 508, 537

**Related Functions Checked:**
- `main()` in `tally.py` - Tampering detection
- `tally_election()` in `tally.py` - Error handling
- `do_add_issue_endpoint()` in `pages.py` - Form processing
- `do_edit_issue_endpoint()` in `pages.py` - Form processing

**Description:**  
Security-critical events use `print()` instead of the `_LOGGER` logging framework. This means these events bypass log formatting, routing, storage, and any SIEM integration. The `_LOGGER` is defined and used elsewhere in the same files, creating false confidence that security events are captured.

**Vulnerable Code (tally.py):**

```python
# Line 152 - Tampering detection (CRITICAL security event)
if election.is_tampered(pdb):
    print(f'Error: Election {election_id} has been tampered with. Cannot proceed.')
    sys.exit(1)

# Line 124 - Tally error
except Exception as e:
    print(f'Error tallying issue {issue.iid}: {e}')
    raise  # Fail hard
```

**Vulnerable Code (pages.py):**

```python
# Line 508 (do_add_issue_endpoint) - Debug output of form data
form = edict(await quart.request.form)
print('FORM:', form)

# Line 537 (do_edit_issue_endpoint) - Debug output of form data
form = edict(await quart.request.form)
print('FORM:', form)
```

**Data Flow:**  
`Security event occurs` → `print()` → `stdout` (may not be captured by log collection) — bypasses `logging.basicConfig()` formatting, handlers, and routing

**Impact:**  
- Tampering detection—the most critical security signal—may not reach log aggregation systems
- Tally errors with exception details go to unformatted stdout
- Debug `print('FORM:', form)` outputs sensitive election data (titles, descriptions) outside the logging framework without timestamps, log levels, or access controls
- SIEM tools monitoring structured log output will miss these events entirely

**Remediation:**
```python
# tally.py - Replace print() with appropriate logging levels
if election.is_tampered(pdb):
    _LOGGER.critical(
        f'TAMPERING DETECTED: Election[E:{election_id}] integrity check failed. Aborting.'
    )
    sys.exit(1)

# tally.py - Replace print() with logging for errors
except Exception as e:
    _LOGGER.error(f'Failed to tally issue[I:{issue.iid}]: {type(e).__name__}')
    raise

# pages.py - Remove debug print statements entirely
# DELETE: print('FORM:', form)
_LOGGER.debug(f'Processing form submission for election[E:{election.eid}]')
```

#### ASVS-1611-HIGH-003: Authorization Failures Not Logged at Multiple Endpoints

**Severity:** HIGH  
**Gap Type:** Type A (Entry point with NO logging control for authorization denials)

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `vote_on_page()` (line 250), `serve_doc()` (line 547), `admin_page()` (line 308)

**Related Functions Checked:**
- `vote_on_page()` - Voter eligibility check
- `serve_doc()` - Document access authorization
- `admin_page()` - Admin access control
- `raise_404()` - 404 response helper
- `election.q_find_issues.perform()` - Database query for voter eligibility

**Description:**  
Multiple endpoints silently deny access by returning 404 responses without creating any security log entry. Authorization failures are high-signal security events that should always be logged for anomaly detection and incident investigation.

**Vulnerable Code:**

```python
# vote_on_page() - line 246-253: Voter not eligible, silent denial
election.q_find_issues.perform(result.uid, election.eid)
if not election.q_find_issues.fetchall():
    result = await basic_info()
    result.title = 'Access Denied'
    result.eid = election.eid
    raise_404(T_BAD_EID, result)  # NO logging of this denial
```

```python
# serve_doc() - lines 543-548: Document access denied, silent 404
row = db.q_get_mayvote.first_row(result.uid, iid)
if not row:
    # Nothing fancy. Just pretend the file does not exist
    quart.abort(404)  # NO logging of unauthorized document access attempt
```

```python
# admin_page() - lines 305-312: Unknown person denied admin, silent 404
except steve.persondb.PersonNotFound:
    result = await basic_info()
    result.title = 'Unknown Person'
    result.pid = result.uid
    raise_404(T_BAD_PID, result)  # NO logging of admin access attempt by unknown person
```

**Data Flow:**  
`Authenticated user` → `authorization check` → `denial` → `404 response` — **no security event logged**

**Impact:**  
- Cannot detect unauthorized access attempts or privilege escalation probing
- No visibility into users attempting to access elections they're not authorized for
- Cannot correlate authorization failures across endpoints for pattern detection
- Incident response cannot determine scope of unauthorized access attempts

**Remediation:**
```python
# vote_on_page() - Add authorization failure logging
election.q_find_issues.perform(result.uid, election.eid)
if not election.q_find_issues.fetchall():
    _LOGGER.warning(
        f'Authorization denied: User[U:{result.uid}] attempted to access '
        f'election[E:{election.eid}] without voter eligibility'
    )
    result = await basic_info()
    result.title = 'Access Denied'
    result.eid = election.eid
    raise_404(T_BAD_EID, result)

# serve_doc() - Add document access failure logging
if not row:
    _LOGGER.warning(
        f'Authorization denied: User[U:{result.uid}] attempted to access '
        f'document "{docname}" for issue[I:{iid}] without eligibility'
    )
    quart.abort(404)

# admin_page() - Add admin access failure logging
except steve.persondb.PersonNotFound:
    _LOGGER.warning(
        f'Authorization denied: User[U:{result.uid}] not found in PersonDB, '
        f'admin access denied'
    )
    # ... existing 404 handling ...
```

### Medium Severity

#### ASVS-1611-MEDIUM-004: Log Injection via Unsanitized User-Controlled Input

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `do_create_endpoint()` (line 455), `_set_election_date()` (line 101), `do_add_issue_endpoint()` (line 517), `do_edit_issue_endpoint()` (line 542)

**Related Functions Checked:**
- `do_create_endpoint()` - Election creation with title logging
- `_set_election_date()` - Date setting with user input logging
- `do_add_issue_endpoint()` - Issue creation with title logging
- `do_edit_issue_endpoint()` - Issue editing with title logging
- `flash_success()` - Flash message helper

**Description:**  
User-controlled input from form submissions is directly interpolated into log messages using f-strings without encoding newlines or other log control characters. An attacker can inject fake log entries by including newline characters in form fields, undermining log integrity for forensic analysis.

**Vulnerable Code:**

```python
# do_create_endpoint() - line 455: form.title is user-controlled
_LOGGER.info(
    f'User[U:{result.uid}] created election[E:{election.eid}];'
    f' title: "{form.title}"'
)

# _set_election_date() - line 101: date_str from JSON body is user-controlled
_LOGGER.info(
    f'User[U:{result.uid}] set {field} for election[E:{election.eid}] to {date_str}'
)

# do_add_issue_endpoint() - line 517: form.title is user-controlled
# do_edit_issue_endpoint() - line 542: form.title is user-controlled (via flash)
await flash_success(f'Issue "{form.title}" has been updated.')
```

**Data Flow:**  
`User form submission` → `form.title` (unsanitized) → `f-string interpolation` → `_LOGGER.info()` → **log file with injected content**

**Proof of Concept:**

```http
POST /do-create-election HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: [valid session]

title=Test%0A[01/15 09:00|INFO|pages] User[U:admin] opened election[E:e-sensitive-id]
```

This produces a log entry followed by a newline and a fake log entry:
```
[01/15 08:30|INFO|pages] User[U:attacker] created election[E:abc123]; title: "Test
[01/15 09:00|INFO|pages] User[U:admin] opened election[E:e-sensitive-id]"
```

**Impact:**  
- Attackers can forge log entries to cover tracks or frame other users
- Log analysis tools may misparse injected entries
- Incident investigation can be misled by fabricated audit trails
- Undermines trust in the entire logging infrastructure

**Remediation:**
```python
import re

def sanitize_for_log(value: str, max_length: int = 200) -> str:
    """Sanitize user input for safe inclusion in log messages."""
    if value is None:
        return '<none>'
    # Replace control characters (newlines, tabs, carriage returns)
    sanitized = re.sub(r'[\r\n\t\x00-\x1f]', ' ', str(value))
    # Truncate to prevent log flooding
    if len(sanitized) > max_length:
        sanitized = sanitized[:max_length] + '...[truncated]'
    return sanitized

# Usage:
_LOGGER.info(
    f'User[U:{result.uid}] created election[E:{election.eid}];'
    f' title: "{sanitize_for_log(form.title)}"'
)
```

#### ASVS-1611-MEDIUM-005: Exception Details in Error Logs May Expose Sensitive Data

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/pages.py`, line 419
- **File:** `v3/server/bin/tally.py`, line 124

**Related Functions Checked:**
- `do_vote_endpoint()` in `pages.py` - Vote submission error handling
- `tally_election()` in `tally.py` - Tally error handling
- `election.add_vote()` - Vote processing (called from `do_vote_endpoint()`)
- `crypto.create_vote()` - Cryptographic operations
- `db.c_add_vote.perform()` - Database operations

**Description:**  
Exception objects are directly interpolated into log messages. During vote processing, exceptions from cryptographic operations or database layer could expose sensitive internal state including cryptographic parameters, SQL queries, or partial vote data.

**Vulnerable Code:**

```python
# pages.py - do_vote_endpoint, line 417-420
except Exception as e:
    _LOGGER.error(
        f'Error adding vote for user[U:{result.uid}] on issue[I:{iid}]: {e}'
    )

# tally.py - tally_election, line 124
except Exception as e:
    print(f'Error tallying issue {issue.iid}: {e}')
    raise
```

**Data Flow:**  
`add_vote()` → `crypto.create_vote()` or `db.c_add_vote.perform()` → exception with internal details → `{e}` interpolated into log → **sensitive data exposed in logs**

**Impact:**  
- Cryptographic errors could expose key material, salts, or vote tokens in logs
- Database errors could expose SQL queries with parameter values
- Vote processing errors could leak partial vote content (violating ballot secrecy)
- Logs containing sensitive data become a high-value target for attackers

**Remediation:**
```python
# Log exception type and a safe message; use separate debug-level for details
except Exception as e:
    _LOGGER.error(
        f'Vote processing failed for user[U:{result.uid}] on issue[I:{iid}]: '
        f'{type(e).__name__}'
    )
    _LOGGER.debug(f'Vote error details (issue[I:{iid}]): {e}', exc_info=True)
    await flash_danger(f'Error submitting vote for issue {iid}.')
    return quart.redirect(f'/vote-on/{election.eid}', code=303)
```

## Architecture Observations

1. **No Centralized Security Logging Layer:** Security logging is ad-hoc per endpoint rather than implemented through middleware or decorators. This makes it easy to miss logging at new endpoints and hard to ensure consistency.

2. **No Log Destination Configuration:** `main.py` configures `logging.basicConfig()` which defaults to stderr. There is no configuration for file-based logging, log rotation, retention policies, or forwarding to a SIEM—requirements explicitly called out by ASVS 16.1.1.

3. **CLI vs. Web Logging Disparity:** Web endpoints in `pages.py` have reasonable logging coverage for successful operations, while the CLI tally tool in `tally.py` has almost none despite performing the most sensitive operations.

4. **No Logging Documentation:** No evidence of a logging inventory document describing what events are logged, their format, storage location, retention policy, or access controls—the core requirement of ASVS 16.1.1.

## Recommendations Summary

### Immediate Actions (Critical Priority)

1. **Create a formal logging inventory document** covering all layers, events, formats, storage, access controls, and retention periods as required by ASVS 16.1.1
   - **Addresses:** All findings, core ASVS requirement
   - **Effort:** 1-2 days

2. **Add comprehensive audit logging to tally.py** including operator identity, election ID, start/completion timestamps, and result summaries
   - **Addresses:** ASVS-1611-HIGH-001
   - **Effort:** 4-6 hours

3. **Replace all `print()` calls with `_LOGGER`** at appropriate levels (`CRITICAL` for tampering, `ERROR` for failures); remove debug `print('FORM:', ...)` statements
   - **Addresses:** ASVS-1611-HIGH-002
   - **Effort:** 2-3 hours

### Short-term Actions (High Priority)

4. **Add authorization failure logging** at all endpoints via a decorator or middleware pattern to ensure coverage on new endpoints
   - **Addresses:** ASVS-1611-HIGH-003
   - **Effort:** 1 day

5. **Implement a `sanitize_for_log()` utility** and apply to all user-controlled values before log interpolation
   - **Addresses:** ASVS-1611-MEDIUM-004
   - **Effort:** 4-6 hours

6. **Log only exception type names at INFO/ERROR level**; restrict full exception details to DEBUG level to prevent sensitive data leakage
   - **Addresses:** ASVS-1611-MEDIUM-005
   - **Effort:** 2-3 hours

### Long-term Actions (Strategic Priority)

7. **Configure structured logging output** (e.g., JSON format) with file handlers, rotation, and retention policies for production deployment
   - **Addresses:** Architecture observation #2
   - **Effort:** 2-3 days

8. **Implement a security logging decorator** that automatically captures user identity, action, resource, and outcome for all state-changing endpoints
   - **Addresses:** Architecture observation #1
   - **Effort:** 3-4 days
