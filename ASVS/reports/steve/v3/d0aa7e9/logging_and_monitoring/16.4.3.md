## Findings

### HIGH SEVERITY

#### ASVS-1643-HIGH-001: Complete Absence of Remote/Separate Log Transmission

**Location:**
- **File:** `v3/server/main.py`
- **Functions:** `run_standalone()`, `run_asgi()`
- **Lines:** 55-62, 97-103

**Related Functions Checked:**
- `logging.basicConfig()` configuration in both server modes
- All `_LOGGER` usage in `pages.py`, `election.py`, `tally.py`
- Log handler initialization

**Description:**

The application entirely lacks any mechanism for transmitting logs to a logically separate system. All security-relevant logs (authentication, voting, election state changes, tampering detection) are written exclusively to the local process's stderr via Python's default `logging.basicConfig()`. An attacker who compromises the application server would have full ability to modify or destroy the forensic audit trail — a critical deficiency for an election system.

**Vulnerable Code (Standalone mode):**
```python
# v3/server/main.py, lines 55-62
logging.basicConfig(
    level=logging.DEBUG,
    style='{',
    format='[{asctime}|{levelname}|{name}] {message}',
    datefmt=DATE_FORMAT,
)
```

**Vulnerable Code (ASGI mode):**
```python
# v3/server/main.py, lines 97-103
logging.basicConfig(
    level=logging.DEBUG,
    style='{',
    format='[{asctime}|{levelname}|{name}] {message}',
    datefmt=DATE_FORMAT,
)
```

**Data Flow:**
```
Security events (pages.py, election.py) 
  → _LOGGER.info/error() 
  → Python root logger 
  → logging.basicConfig() 
  → StreamHandler(stderr) 
  → LOCAL PROCESS OUTPUT ONLY
```

**Detailed Analysis:**

`logging.basicConfig()` with no `handlers` or `filename` argument creates a single `StreamHandler` writing to `stderr`. Every security-critical log statement across the entire application terminates at the local process:

| File | Security Events Logged Locally | Lines |
|------|-------------------------------|-------|
| `pages.py` | Election creation, opening, closing | 438, 459, 478 |
| `pages.py` | Vote submission | 412 |
| `pages.py` | Issue add/edit/delete | 502, 525, 546 |
| `pages.py` | Election date changes | 128 |
| `election.py` | Election/issue creation | 207, 222, 435 |
| `tally.py` | Tally operations | 117, 120 |

There is **no handler** configured for any of:
- `logging.handlers.SysLogHandler` (remote syslog)
- `logging.handlers.SocketHandler` (TCP/UDP log forwarding)
- `logging.handlers.HTTPHandler` (SIEM webhook)
- `logging.handlers.QueueHandler` (async forwarding)
- File handlers writing to a mounted remote filesystem
- Any third-party log shipper integration (fluentd, filebeat, etc.)

**Proof of Concept:**

An attacker with server access (post-breach) executes:
```bash
# All security logs are in the process stderr, typically captured to a local file
# Attacker modifies vote submission logs to hide unauthorized voting
sed -i '/voted on issue/d' /var/log/steve/app.log

# Or if running in a container, logs are in docker/journald on the same host
# Attacker can clear the journal
journalctl --vacuum-time=1s
```

No external system would detect the modification because no logs are transmitted externally.

**Impact:**

If the application server is compromised, an attacker can:
1. **Delete** all security logs (vote records, authentication events, election state transitions)
2. **Modify** logs to cover tracks of unauthorized election manipulation
3. **Prevent** incident detection, alerting, and escalation since no external system receives the logs
4. **Destroy** the forensic audit trail that is critical for election integrity and legal proceedings

For an election system, loss of log integrity directly undermines the ability to verify that the election was conducted fairly.

**Remediation:**

Configure a remote log handler in addition to local output. At minimum, add a `SysLogHandler` targeting a separate log aggregation server:

```python
import logging
from logging.handlers import SysLogHandler
import ssl
import socket

def configure_logging(app_config):
    """Configure logging with remote transmission to a separate system."""
    
    # Local handler (keep for operational debugging)
    logging.basicConfig(
        level=logging.DEBUG,
        style='{',
        format='[{asctime}|{levelname}|{name}] {message}',
        datefmt=DATE_FORMAT,
    )
    
    # Remote handler — transmit security logs to separate system
    remote_handler = SysLogHandler(
        address=(app_config.log_server_host, app_config.log_server_port),
        facility=SysLogHandler.LOG_AUTH,  # Security-relevant facility
        socktype=socket.SOCK_TCP,  # Reliable delivery
    )
    remote_handler.setLevel(logging.INFO)  # Security events are INFO+
    
    # Structured format for SIEM ingestion
    remote_formatter = logging.Formatter(
        fmt='steve-election: %(asctime)s %(levelname)s %(name)s %(message)s',
        datefmt='%Y-%m-%dT%H:%M:%S%z'
    )
    remote_handler.setFormatter(remote_formatter)
    
    # Add to root logger so ALL module loggers transmit remotely
    logging.getLogger().addHandler(remote_handler)
```

For production election systems, consider:
- **TLS-encrypted syslog** (RFC 5425) to prevent log interception in transit
- **SIEM integration** (Splunk HEC, Elasticsearch, etc.) via dedicated handlers
- **Write-once storage** (S3 with Object Lock, immutable log volumes)
- **Log signing** to detect tampering of archived logs

### MEDIUM SEVERITY

#### ASVS-1643-MEDIUM-002: Critical Security Events Bypass Logging Framework Entirely

**Location:**
- **File:** `v3/server/bin/tally.py`
- **Functions:** Tampering detection block, tally error handling
- **Lines:** 161-162, 132-133

**Related Functions Checked:**
- `_LOGGER` import and initialization (line 35)
- `election.is_tampered()` method
- Exception handling in tally computation loop

**Description:**

The `_LOGGER` object is imported and available, yet the most security-critical events in the tally process use `print()` instead of the logging framework. This causes critical security signals (especially tampering detection) to bypass the logging infrastructure entirely, making them invisible to any log aggregation, monitoring, or alerting systems.

**Vulnerable Code (Tampering detection — highest severity event):**
```python
# v3/server/bin/tally.py, lines 161-162
if election.is_tampered(pdb):
    print(f'Error: Election {election_id} has been tampered with. Cannot proceed.')
    sys.exit(1)
```

**Vulnerable Code (Tally error):**
```python
# v3/server/bin/tally.py, lines 132-133
except Exception as e:
    print(f'Error tallying issue {issue.iid}: {e}')
    raise  # Fail hard
```

**Data Flow:**
```
Tampering detected 
  → print() 
  → stdout 
  → NOT captured by logging framework
  → NOT transmitted to any log handler (local or remote)
  → NO audit trail
```

**Detailed Analysis:**

`print()` writes to `stdout`, which:
- Bypasses the logging framework entirely
- Cannot be routed to remote log handlers even if they were configured
- Is not captured by standard log rotation or aggregation tools
- Has no severity level, timestamp, or structured metadata

| Event | Current Method | Should Be | Severity |
|-------|---------------|-----------|----------|
| Election tampering detected | `print()` | `_LOGGER.critical()` | CRITICAL |
| Tally computation error | `print()` | `_LOGGER.error()` | ERROR |

The **tampering detection event** is arguably the most critical security signal the system can produce — evidence that election data has been modified after opening. This event being invisible to any logging infrastructure means it cannot trigger alerts, escalation, or forensic investigation.

**Impact:**

- Tampering of an election could occur without any auditable log record
- Even if ASVS-1643-HIGH-001 were remediated (remote logging added), these events would still not be transmitted
- Security monitoring/SIEM would have no signal for the highest-severity election integrity event
- Tally errors that might indicate data corruption go unlogged
- Forensic investigation after an incident would lack critical evidence

**Remediation:**

Replace `print()` calls with appropriate logging levels:

```python
# v3/server/bin/tally.py — tampering detection
if election.is_tampered(pdb):
    _LOGGER.critical(
        f'TAMPERING DETECTED: Election[E:{election_id}] '
        f'integrity check failed. Tally aborted.'
    )
    sys.exit(1)

# v3/server/bin/tally.py — tally errors
except Exception as e:
    _LOGGER.error(
        f'Tally computation failed for issue[I:{issue.iid}] '
        f'in election[E:{election.eid}]: {type(e).__name__}: {e}'
    )
    raise
```

Additionally, the successful completion of a tally should also be logged as an audit event:

```python
# After successful tally
_LOGGER.info(
    f'Tally completed for election[E:{election_id}], '
    f'{len(results)} issues tallied, {len(all_voters)} unique voters'
)
```

## Positive Security Patterns

1. **Structured log messages with identifiers**: `pages.py` consistently uses `User[U:{uid}]`, `election[E:{eid}]`, `issue[I:{iid}]` format for correlation — this is excellent for SIEM parsing if logs were transmitted remotely.

2. **Security-relevant events are identified**: Vote submissions, election state transitions, issue modifications, and date changes are all logged with actor identity — the signal is good, only the transport is missing.

3. **Separation of vote content from vote logging**: `pages.py` line 412 logs `User[U:{result.uid}] voted on issue[I:{iid}]` without logging the vote content, correctly preserving ballot secrecy.

4. **Tampering detection exists**: `tally.py` line 161 and `election.py` `is_tampered()` method provide integrity verification — the detection mechanism is sound even though its logging is deficient.

5. **Logging level hierarchy**: `main.py` sets different log levels for different subsystems (asyncio, hpack, etc.), showing awareness of log noise management.

---

## Architecture Observations

1. **Single-tier logging architecture**: The entire system uses Python's built-in logging with no external log infrastructure. For a standalone low-concurrency election system, this may have been a deliberate simplicity choice, but it fundamentally fails ASVS 16.4.3's requirement for logical separation.

2. **CLI tool logging gap**: `tally.py` operates as a CLI script with its own `logging.basicConfig(level=logging.INFO)`, completely independent of the server's logging configuration. Even if the server were configured with remote handlers, the tally script would not inherit them.

3. **No log aggregation configuration surface**: There is no configuration file or environment variable for specifying a remote log destination. Adding this capability requires architectural change, not just a code fix.

4. **Good event identification foundation**: The existing log messages use structured identifiers and include relevant context. This provides a solid foundation for SIEM integration once remote transmission is implemented.
