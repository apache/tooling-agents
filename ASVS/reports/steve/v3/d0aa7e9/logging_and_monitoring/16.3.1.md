### Key Findings

The application demonstrates solid patterns for logging business-significant actions (votes, election state changes) with user identifiers, but has **critical gaps in logging authentication operations themselves**. Despite using OAuth-based authentication across 15+ endpoints, the application does not log successful authentication, failed authentication attempts, or authorization rejections. This represents a fundamental gap in the ASVS 16.3.1 requirement.

## Findings

### High Severity

#### ASVS-1631-HIGH-001 — No Logging of Successful or Failed Authentication Attempts

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control  

**Location:**
- **File:** `v3/server/pages.py` (all authenticated endpoints) and `v3/server/main.py` (OAuth configuration)
- **Functions:** `create_app()` (main.py:36), all route handlers (pages.py)
- **Lines:** main.py:36-48, pages.py:63-92

**Related Functions Checked:**
- `basic_info()` (pages.py:63-92)
- `create_app()` (main.py:36-48)
- All decorated route handlers in pages.py

**Description:**

The application uses `@asfquart.auth.require` decorators for OAuth-based authentication across 15+ endpoints but never logs the outcome of authentication operations. There is no `@APP.before_request` handler, no `@APP.after_request` handler, no error handler for 401/403 responses, and no explicit authentication event logging anywhere in the application code. When the OAuth flow completes (success or failure), the application does not record this event.

**Vulnerable Code:**

*pages.py, lines 63-92:*
```python
async def basic_info():
    """Return base-level EZT template data."""
    basic = edict()
    # ...
    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
    else:
        # No session.
        basic.update(uid=None, name=None, email=None)
    # ^^^ Session read outcome is NEVER logged
    return basic
```

*main.py, lines 36-48:*
```python
def create_app():
    asfquart.generics.OAUTH_URL_INIT = (
        'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
    )
    asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    import pages
    import api
    # ^^^ No authentication event hooks registered
    return app
```

**Data Flow:**

```
User → OAuth callback → asfquart.auth framework → session created/rejected → NO LOG ENTRY
```

**Proof of Concept:**

1. A legitimate user successfully authenticates via OAuth → No log entry created
2. An unauthorized user attempts to access `/admin` and is rejected → No log entry created
3. An attacker performs brute-force or credential stuffing against the OAuth flow → No log entries to detect the pattern

**Impact:**

In an election system, the inability to audit authentication events means:
- Impossible to detect unauthorized access attempts
- No forensic trail for security incident investigation
- Cannot verify that only authorized individuals accessed the system during an election
- Compliance failure for election auditing requirements

**Remediation:**

```python
# In main.py, after app creation:
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    @app.before_request
    async def log_authentication():
        """Log authentication outcomes for all requests to protected endpoints."""
        s = await asfquart.session.read()
        if s:
            _LOGGER.info(
                'AUTH_SUCCESS: user=%s, method=oauth, '
                'ip=%s, user_agent=%s, path=%s',
                s['uid'],
                quart.request.remote_addr,
                quart.request.headers.get('User-Agent', 'unknown'),
                quart.request.path,
            )
        elif requires_auth(quart.request.path):
            _LOGGER.warning(
                'AUTH_FAILURE: method=oauth, ip=%s, user_agent=%s, path=%s',
                quart.request.remote_addr,
                quart.request.headers.get('User-Agent', 'unknown'),
                quart.request.path,
            )
    
    @app.errorhandler(401)
    async def handle_401(error):
        _LOGGER.warning(
            'AUTH_REJECTED: status=401, ip=%s, path=%s, user_agent=%s',
            quart.request.remote_addr,
            quart.request.path,
            quart.request.headers.get('User-Agent', 'unknown'),
        )
        return quart.Response('Unauthorized', status=401)
    
    @app.errorhandler(403)
    async def handle_403(error):
        s = await asfquart.session.read()
        _LOGGER.warning(
            'AUTH_FORBIDDEN: status=403, user=%s, ip=%s, path=%s',
            s['uid'] if s else 'anonymous',
            quart.request.remote_addr,
            quart.request.path,
        )
        return quart.Response('Forbidden', status=403)
    
    import pages
    import api
    return app
```

#### ASVS-1631-HIGH-002 — Authorization Rejection Events Not Logged

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control  

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `admin_page()` (line 330), `vote_on_page()` (line 239)
- **Lines:** 356-366, 274-279

**Related Functions Checked:**
- `admin_page()` (pages.py:330)
- `vote_on_page()` (pages.py:239)
- PersonDB lookup and mayvote eligibility checks

**Description:**

The application performs in-handler authorization checks (PersonDB lookup, mayvote eligibility) that silently reject users with a 404 response but never log the authorization failure. These are security-relevant authentication/authorization decisions that could indicate unauthorized access attempts.

**Vulnerable Code:**

*pages.py, lines 356-366 — admin_page:*
```python
try:
    me = pdb.get_person(result.uid)
except steve.persondb.PersonNotFound:
    ### the authn'd committer is not in "person"
    result = await basic_info()
    result.title = 'Unknown Person'
    result.pid = result.uid
    # Note: result.uid (and friends) are needed for the navbar.
    raise_404(T_BAD_PID, result)
    # NOTREACHED
    # ^^^ Authenticated user rejected — NOT LOGGED
```

*pages.py, lines 274-279 — vote_on_page:*
```python
# Check if user can vote in this election (has mayvote entries)
election.q_find_issues.perform(result.uid, election.eid)
if not election.q_find_issues.fetchall():
    result = await basic_info()
    result.title = 'Access Denied'
    result.eid = election.eid
    raise_404(T_BAD_EID, result)  # Reuse template for consistency
    # ^^^ Voter authorization rejected — NOT LOGGED
```

**Data Flow:**

```
Authenticated User → Authorization Check (PersonDB/mayvote) → Rejection (404) → NO LOG ENTRY
```

**Proof of Concept:**

An authenticated committer who is not in the PersonDB attempts to access `/admin`. They receive a 404, but no log entry records this authorization failure. An attacker who compromises an OAuth account could probe the system without detection.

**Impact:**

- Authorization bypass attempts are invisible to security monitoring
- No audit trail for access control decisions in an election system
- Impossible to detect enumeration or privilege escalation attempts

**Remediation:**

```python
# In admin_page():
except steve.persondb.PersonNotFound:
    _LOGGER.warning(
        'AUTHZ_FAILURE: user=%s attempted admin access but not in PersonDB, '
        'ip=%s, path=%s',
        result.uid,
        quart.request.remote_addr,
        quart.request.path,
    )
    raise_404(T_BAD_PID, result)

# In vote_on_page():
if not election.q_find_issues.fetchall():
    _LOGGER.warning(
        'AUTHZ_FAILURE: user=%s not eligible to vote in election=%s, '
        'ip=%s',
        result.uid,
        election.eid,
        quart.request.remote_addr,
    )
    raise_404(T_BAD_EID, result)
```

#### ASVS-1631-MEDIUM-003 — Authentication Metadata Not Collected in Security Logs

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry points with incomplete controls  

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** All logging statements (do_vote_endpoint, do_create_endpoint, do_open_endpoint, do_close_endpoint, etc.)
- **Lines:** ~475, ~498, ~513 (representative examples)

**Related Functions Checked:**
- `do_vote_endpoint()` (pages.py:~475)
- `do_create_endpoint()` (pages.py:~498)
- `do_open_endpoint()` (pages.py:~513)
- All other state-changing endpoints with logging

**Description:**

ASVS 16.3.1 requires that "additional metadata, such as the type of authentication or factors used, should also be collected." While the application logs user actions with UIDs, no authentication metadata is included: no IP address, user agent, session identifier, authentication method, or timestamp format suitable for correlation.

**Vulnerable Code Examples:**

*pages.py, line ~475 (do_vote_endpoint):*
```python
_LOGGER.info(
    f'User[U:{result.uid}] voted on issue[I:{iid}] in election[E:{election.eid}]'
)
```

*pages.py, line ~498 (do_create_endpoint):*
```python
_LOGGER.info(
    f'User[U:{result.uid}] created election[E:{election.eid}];'
    f' title: "{form.title}"'
)
```

*pages.py, line ~513 (do_open_endpoint):*
```python
_LOGGER.info(f'User[U:{result.uid}] opened election[E:{election.eid}]')
```

**Data Flow:**

```
User Action → Logging Statement → Log Entry (UID only) → Missing: IP, User-Agent, Auth Method
```

**Proof of Concept:**

Current log entry:
```
User[U:jdoe] voted on issue[I:123] in election[E:456]
```

Missing context prevents answering:
- What IP address did this request come from?
- What authentication method was used?
- What user agent/client was used?
- Can we correlate this with other security events?

**Impact:**

- Cannot correlate authentication events with source IPs for forensic investigation
- Cannot distinguish between authentication methods if multiple are supported
- Insufficient metadata for SIEM correlation and alerting
- Election audit logs lack the granularity needed for post-election security review

**Remediation:**

```python
# Create a helper that enriches log context:
async def security_log_context():
    """Build structured metadata for security logs."""
    s = await asfquart.session.read()
    return {
        'user': s['uid'] if s else 'anonymous',
        'ip': quart.request.remote_addr,
        'user_agent': quart.request.headers.get('User-Agent', 'unknown'),
        'method': quart.request.method,
        'path': quart.request.path,
        'auth_type': 'oauth',  # or derive from session
    }

# Usage:
ctx = await security_log_context()
_LOGGER.info(
    'VOTE_SUBMITTED: user=%(user)s, issue=%s, election=%s, '
    'ip=%(ip)s, auth_type=%(auth_type)s',
    iid, election.eid,
    ctx,
)
```

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with NO control  

**Location:**
- **File:** `v3/server/bin/tally.py`
- **Functions:** `main()` (line 138), `tally_election()` (line 98)

**Description:**

The tally CLI tool performs the most security-sensitive operation in the election system — decrypting and counting votes — but does not log the identity of the operator, their authorization basis, or create an audit trail of the tallying operation. Tampering detection results are printed to stdout rather than logged to the security log.

**Vulnerable Code:**

*tally.py, lines 155-160:*
```python

    print(f'Error: Election {election_id} has been tampered with. Cannot proceed.')
    sys.exit(1)

# ^^^ No logging of WHO ran the tally or WHEN
```

*tally.py, lines 98-135:*
```python
def tally_election(election, issue_id, output_format):
    """Tally all issues in the given election and output results."""
    issues = election.list_issues()
    # ... tallying occurs ...
    # ^^^ No logging of: operator identity, start/end time, 
    #     issues tallied, or tally completion
```

**Data Flow:**

```
Admin → CLI → tally.py → Database decryption → Results output → NO AUDIT LOG
```

**Proof of Concept:**

1. Administrator runs: `python tally.py election_123`
2. Votes are decrypted and tallied
3. Results are output to stdout
4. No log entry records who performed this operation or when

**Impact:**

- No forensic record of who performed the tally or when
- Tampering alerts go to stdout (easily lost/missed) rather than persistent security logs
- Cannot verify the chain of custody for election results
- Domain context specifically requires "tally operations create audit trails"

        'TALLY_INITIATED: operator=%s, election_id=%s, issue_id=%s, '
        'spy_on_open=%s, pid=%s',
        operator, election_id, issue_id, spy_on_open, os.getpid()
    )
    
    # ... existing logic ...
    
    # Check for tampering
    pdb = steve.persondb.PersonDB.open(db_fname)
    if election.is_tampered(pdb):
        _LOGGER.critical(
            'TAMPER_DETECTED: operator=%s, election_id=%s — tally aborted',

        )

        'TAMPER_CHECK_PASSED: operator=%s, election_id=%s',
        operator, election_id

        'TALLY_COMPLETED: operator=%s, election_id=%s',
        operator, election_id

## Positive Security Patterns

1. **Structured User-Action Logging**: All state-changing operations in `pages.py` consistently log with format `User[U:{uid}] <action> <resource>[<type>:{id}]` — this is a good pattern that just needs authentication metadata added.

2. **Tamper Detection Before Tally**: `tally.py` checks `election.is_tampered(pdb)` before proceeding — the integrity check exists, it just needs proper logging.

3. **Election State Machine Logging**: Election lifecycle changes (create, open, close) are all logged with admin identity — demonstrates awareness of audit trail needs.

4. **Vote Logging Without Content**: `_LOGGER.info(f'User[U:{result.uid}] voted on issue[I:{iid}]...')` correctly logs the vote submission without the vote content, preserving ballot secrecy.

5. **Error Logging on Vote Failure**: `_LOGGER.error(f'Error adding vote for user[U:{result.uid}]...')` captures vote submission failures with context.

---

## Architecture Observations

1. **Authentication Delegation Without Logging Bridge**: The application delegates authentication entirely to `asfquart.auth` but does not register any hooks to capture authentication events in the application's security log. Even if the framework logs internally, the application's security logs would be incomplete without bridging these events.

2. **Two Separate Execution Contexts**: The web server (`pages.py`) and CLI tool (`tally.py`) have completely separate logging configurations with no shared security logging infrastructure. The CLI tool's `logging.basicConfig(level=logging.INFO)` writes to stderr by default with no persistent destination configured.

3. **No Centralized Security Logger**: The application uses standard Python `logging.getLogger(__name__)` which mixes security events with debug/operational logs. A dedicated security logger (e.g., `logging.getLogger('security.auth')`) would enable better log routing and analysis.
