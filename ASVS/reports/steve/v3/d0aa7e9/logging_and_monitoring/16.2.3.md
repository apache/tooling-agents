## Findings

### HIGH SEVERITY

#### ASVS-1623-HIGH-001: Production Web Endpoints Output Form Data to Undocumented stdout Channel via print()

**Severity:** HIGH

**Gap Type:** Type B — Logging control EXISTS (`_LOGGER`) but NOT CALLED for these outputs, creating a false sense of logging completeness.

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `do_add_issue_endpoint()` (line ~493), `do_edit_issue_endpoint()` (line ~516)
- **Line Numbers:** 493, 516

**Related Functions Checked:**
- `do_create_endpoint()` — ✅ Uses only `_LOGGER`
- `do_open_endpoint()` — ✅ Uses only `_LOGGER`
- `do_close_endpoint()` — ✅ Uses only `_LOGGER`
- `do_delete_issue_endpoint()` — ✅ Uses only `_LOGGER`
- `do_vote_endpoint()` — ✅ Uses only `_LOGGER`

**Description:**

Two production web endpoints that handle sensitive election configuration data output full form contents to stdout using `print()` statements. This creates an undocumented logging channel that bypasses the application's configured logging framework.

**Vulnerable Code:**

In `do_add_issue_endpoint` (line ~493):
```python
async def do_add_issue_endpoint(election):
    result = await basic_info()
    ### check authz
    form = edict(await quart.request.form)
    print('FORM:', form)          # <-- Undocumented output to stdout

    ### do better with these
    vtype = 'yna'
    kv = None

    iid = election.add_issue(form.title, form.description, vtype, kv)

    _LOGGER.info(                 # <-- Proper logging used HERE but not for form data
        f'User[U:{result.uid}] added issue[I:{iid}] to election[E:{election.eid}]'
    )
```

In `do_edit_issue_endpoint` (line ~516):
```python
async def do_edit_issue_endpoint(election, issue):
    result = await basic_info()
    ### check authz
    form = edict(await quart.request.form)
    print('FORM:', form)          # <-- Undocumented output to stdout
    
    election.edit_issue(issue.iid, form.title, form.description, issue.vtype, issue.kv)

    _LOGGER.info(                 # <-- Proper logging used HERE but not for form data
        f'User[U:{result.uid}] edited issue[I:{issue.iid}]'
        f' in election[E:{election.eid}]'
    )
```

**Data Flow:**

1. User submits form → `quart.request.form` (user-controlled)
2. Form data wrapped as `edict` containing issue `title` and `description`
3. `print('FORM:', form)` writes full form contents to **process stdout**
4. stdout is NOT part of any documented logging infrastructure
5. Depending on deployment (Hypercorn, systemd, Docker), stdout may be captured by:
   - Container log drivers (sent to CloudWatch, Stackdriver, etc.)
   - systemd journal (accessible to any user with `journalctl`)
   - Process output files with no access controls
   - Log aggregators that weren't configured to receive election data

**Proof of Concept:**

```http
POST /do-add-issue/ELECTION123 HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: session=valid_session_cookie

title=Board+Election+2025&description=Sensitive+candidate+information+here
```

Process stdout will contain:
```
FORM: {'title': 'Board Election 2025', 'description': 'Sensitive candidate information here'}
```

This output bypasses the logging framework, meaning:
- It is not formatted with the `[{asctime}|{levelname}|{name}] {message}` structure
- It cannot be filtered by log level
- It cannot be routed by logging handlers
- It has no metadata (timestamp, user, log level)
- It goes to stdout while the logging framework outputs to stderr

**Impact:**

Election issue content (titles, descriptions) is broadcast to an undocumented destination (stdout) that is separate from the application's configured logging framework. This violates the requirement that the application only store or broadcast logs to documented destinations. In ASGI deployments, stdout may be captured and forwarded to services not included in any log inventory, potentially exposing election configuration data.

**Remediation:**

Remove the `print()` statements or convert them to proper logging at an appropriate level:

```python
async def do_add_issue_endpoint(election):
    result = await basic_info()
    form = edict(await quart.request.form)
    
    # Use the logging framework, at DEBUG level, without dumping full form
    _LOGGER.debug(
        f'User[U:{result.uid}] adding issue to election[E:{election.eid}]'
    )
    
    vtype = 'yna'
    kv = None
    iid = election.add_issue(form.title, form.description, vtype, kv)

    _LOGGER.info(
        f'User[U:{result.uid}] added issue[I:{iid}] to election[E:{election.eid}]'
    )
```

Apply the same pattern to `do_edit_issue_endpoint`.

### MEDIUM SEVERITY

#### ASVS-1623-MEDIUM-001: Security-Critical Events in Tally Tool Bypass Logging Framework

**Severity:** MEDIUM

**Gap Type:** Type B — Logger `_LOGGER` is configured and imported but not used for security-critical events (tampering detection, tally errors).

**Location:**
- **File:** `v3/server/bin/tally.py`
- **Functions:** `tally_election()` (line ~119), `main()` (line ~151)
- **Line Numbers:** 119, 151

**Related Functions Checked:**
- `tally_election()` line ~112 — ✅ Uses `_LOGGER.error` for no-issues error
- `tally_election()` line ~116 — ✅ Uses `_LOGGER.info` for single issue tally
- `tally_election()` line ~120 — ✅ Uses `_LOGGER.info` for multi-issue tally

**Description:**

The tally tool contains two security-critical events that output to stdout via `print()` instead of using the configured logging framework: tampering detection alerts and tally error exceptions.

**Vulnerable Code:**

Location 1 — `tally_election()`, approximately line 119:
```python
        except Exception as e:
            print(f'Error tallying issue {issue.iid}: {e}')
            raise  # Fail hard
```

Location 2 — `main()`, approximately line 151:
```python
    if election.is_tampered(pdb):
        print(f'Error: Election {election_id} has been tampered with. Cannot proceed.')
        sys.exit(1)
```

**Data Flow:**

1. **Tampering detection**: `election.is_tampered(pdb)` detects election data modification
2. Alert is output via `print()` to stdout instead of `_LOGGER.critical()` or `_LOGGER.error()`
3. The logger `_LOGGER = logging.getLogger(__name__)` is defined at line 35 and configured at line 157 but NOT USED for these critical events
4. Similarly, tally errors including exception messages go to stdout rather than the logging framework

**Proof of Concept:**

When an election is tampered with:
```
$ python tally.py --election-id ELEC123 --db-path steve.db
Error: Election ELEC123 has been tampered with. Cannot proceed.
```

This security-critical event:
- Goes to stdout (not stderr where the logging framework writes)
- Has no timestamp, no log level, no structured metadata
- Cannot be captured by log handlers or forwarded to a SIEM
- Would not appear in any log files configured for the Python logging framework

**Impact:**

Tampering detection is a critical security event in an election system. Broadcasting this event to stdout instead of the logging framework means it may not reach the documented log destinations (SIEM, audit log files, log aggregation services). Tally errors that might indicate cryptographic or integrity issues similarly bypass the logging infrastructure.

**Remediation:**

```python
def main(spy_on_open, election_id, issue_id, db_fname, output_format):
    # ... election loading ...
    
    # Check for tampering
    pdb = steve.persondb.PersonDB.open(db_fname)
    if election.is_tampered(pdb):
        _LOGGER.critical(
            f'TAMPERING DETECTED: Election[E:{election_id}] data integrity check failed. '
            f'Tally aborted.'
        )
        sys.exit(1)

    # Proceed with tally
    tally_election(election, issue_id, output_format)
```

And in `tally_election()`:
```python
        except Exception as e:
            _LOGGER.error(
                f'Tally error for issue[I:{issue.iid}]: {e}',
                exc_info=True
            )
            raise
```

#### ASVS-1623-MEDIUM-002: No Documented Log Inventory or Centralized Log Destination Configuration

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/main.py` (logging configuration)
- **Functions:** `run_standalone()` (lines 58-63), `run_asgi()` (lines 92-97)
- **Additional Location:** `v3/server/bin/tally.py` `__main__` (line 157)
- **Scope:** System-wide

**Related Functions Checked:**
- All three execution contexts (standalone, ASGI, CLI) use only `logging.basicConfig()` with default destinations

**Description:**

The application lacks a documented log inventory and uses only default logging destinations across all execution modes. No persistent log storage or centralized log destination is configured.

**Evidence:**

The application's logging infrastructure is configured solely through Python's `logging.basicConfig()` with default destinations:

```python
# main.py - standalone mode (lines 58-63)
def run_standalone():
    logging.basicConfig(
        level=logging.DEBUG,
        style='{',
        format='[{asctime}|{levelname}|{name}] {message}',
        datefmt=DATE_FORMAT,
    )
```

```python
# main.py - ASGI mode (lines 92-97)
def run_asgi():
    logging.basicConfig(
        level=logging.DEBUG,
        style='{',
        format='[{asctime}|{levelname}|{name}] {message}',
        datefmt=DATE_FORMAT,
    )
```

```python
# tally.py - CLI mode (line 157)
if __name__ == '__main__':
    logging.basicConfig(level=logging.INFO)
```

**Data Flow:**

1. Application starts in one of three modes (standalone/ASGI/CLI)
2. `logging.basicConfig()` is called with no `filename` or `handlers` parameter
3. Logs are sent to stderr by default
4. No persistent storage, file handlers, or service handlers are configured
5. Depending on deployment environment, stderr may be:
   - Discarded if not redirected
   - Captured by container runtime
   - Written to systemd journal
   - Sent to unknown destinations

**Analysis:**

1. **No log inventory document**: There is no configuration file, documentation, or code artifact that defines the approved log destinations for the application
2. **Default-only destinations**: `logging.basicConfig()` without a `filename` or `handlers` parameter sends logs to stderr only — no persistent log storage is configured
3. **Inconsistent output channels**: The codebase uses both `print()` (stdout) and `logging` (stderr), creating two separate undocumented output streams
4. **No file/service handlers**: No `FileHandler`, `SysLogHandler`, `SocketHandler`, or any other handler is configured to direct logs to specific documented destinations
5. **Three different logging configurations**: standalone, ASGI, and CLI each configure logging independently with no shared log destination policy

**Impact:**

Without a log inventory, it is impossible to verify that logs are only going to approved destinations. The three different logging configurations across execution modes mean logs may end up in different places depending on how the application is run, with no documentation of which destinations are approved. This makes compliance with ASVS 16.2.3 unverifiable.

**Remediation:**

1. Create a formal log inventory document specifying approved log destinations
2. Centralize logging configuration:

```python
# logging_config.py
import logging
import logging.config

LOG_CONFIG = {
    'version': 1,
    'disable_existing_loggers': False,
    'formatters': {
        'structured': {
            'format': '[{asctime}|{levelname}|{name}] {message}',
            'style': '{',
            'datefmt': '%m/%d %H:%M',
        },
    },
    'handlers': {
        'console': {
            'class': 'logging.StreamHandler',
            'formatter': 'structured',
            'stream': 'ext://sys.stderr',
        },
        'audit_file': {
            'class': 'logging.handlers.RotatingFileHandler',
            'filename': '/var/log/steve/audit.log',
            'formatter': 'structured',
            'maxBytes': 10485760,
            'backupCount': 10,
        },
    },
    'root': {
        'level': 'INFO',
        'handlers': ['console', 'audit_file'],
    },
}

def configure_logging():
    logging.config.dictConfig(LOG_CONFIG)
```

3. Add linting rules or code review checks to prevent `print()` in production modules

## Positive Security Patterns

1. **Consistent Structured Logging in Web Endpoints**: All state-changing operations in `pages.py` use `_LOGGER.info()` with a consistent format: `User[U:{uid}] {action} {object}[{type}:{id}]`. This provides excellent structured audit trail data.

2. **No Vote Content in Logs**: The `do_vote_endpoint` correctly logs `User[U:{result.uid}] voted on issue[I:{iid}] in election[E:{election.eid}]` — the vote content (`votestring`) is never logged, preserving ballot secrecy.

3. **Proper Logger Hierarchy**: Each module creates its own logger via `logging.getLogger(__name__)`, enabling per-module log level control.

4. **Error Logging in Vote Submission**: Vote submission errors are logged via `_LOGGER.error()` with user and issue context, creating an audit trail for failures.

5. **Election State Change Logging**: Opening and closing elections are logged with admin identity (`User[U:{result.uid}]`), meeting the domain requirement for election lifecycle auditing.

---

## Architecture Observations

1. **Dual Output Problem**: The application has two distinct output channels — `logging` (stderr) and `print()` (stdout). In a containerized/ASGI deployment, these may be captured by different log drivers or routed to different destinations. The `print()` calls constitute an undocumented secondary log channel.

2. **No Log Destination Persistence**: All three execution contexts (standalone, ASGI, CLI) configure `logging.basicConfig()` without persistent handlers. Security audit logs (authentication, voting, election state changes) exist only in the process's stderr stream with no configured file or service destination.

3. **Missing Log Inventory Artifact**: There is no log inventory document or configuration that enumerates approved log destinations. This makes it impossible to verify compliance with ASVS 16.2.3 from an audit perspective.
