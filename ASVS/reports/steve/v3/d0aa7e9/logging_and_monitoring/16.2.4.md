### Key Issues

The application implements logging across web server (pages.py/main.py), business logic (election.py), and CLI tooling (tally.py). While security events are generally logged via Python's `logging` module with a semi-structured message convention, several significant issues prevent reliable log reading and correlation by log processors—directly violating ASVS 16.2.4.

### HIGH Severity

#### ASVS-1624-HIGH-002: Critical Security Events in Tally CLI Bypass Logging Framework via print()

**Severity:** HIGH  
**File:** `v3/server/bin/tally.py`  
**Lines:** 129, 140-141  
**Functions:** `tally_election()`, `main()`

**Related Functions Checked:**
- `main()` - Entry point for tally CLI
- `tally_election()` - Core tallying logic
- Existing `_LOGGER` usage on lines 109, 112, 150

**Description:**

Election tampering detection and tally errors—the most critical security events in an election system—bypass the logging framework entirely and use `print()` statements instead. This is a Type B gap: the `_LOGGER` control EXISTS in tally.py (imported on line 23, used on lines 109/112/150) but is NOT CALLED for the most critical security events.

**Vulnerable Code:**

Tampering detection (lines 140-141):
```python
if election.is_tampered(pdb):
    print(f'Error: Election {election_id} has been tampered with. Cannot proceed.')
    sys.exit(1)
```

Tally error (line 129):
```python
except Exception as e:
    print(f'Error tallying issue {issue.iid}: {e}')
    raise  # Fail hard
```

**Data Flow:**
```
Tampering detection result 
  → print() 
  → stdout only 
  → NOT captured by log aggregator/SIEM 
  → Security event invisible to monitoring
```

**Impact:**

Election tampering is arguably the most critical security event in this system. When detected:
- The event goes to stdout only, with **no timestamp, no severity level, no logger name**
- Log processors/SIEMs configured to read the application's logging framework will **never see this event**
- There is **no audit trail** of tampering detection—if stdout is not captured, the event is lost entirely
- The tally process itself (who ran it, when, what election) has no structured log entry

Similarly, exceptions during tallying (line 129) may contain security-relevant information about vote integrity issues but bypass the logging framework.

**Proof of Concept:**

When an administrator runs `tally.py` and tampering is detected, the output is:
```
Error: Election abc123 has been tampered with. Cannot proceed.
```

This has no metadata. Compare to what a proper log entry would look like:
```
[2025-01-15T14:30:45+00:00|CRITICAL|tally] Election tampering detected for election[E:abc123]; aborting tally
```

**Remediation:**

```python
def main(spy_on_open, election_id, issue_id, db_fname, output_format):
    # ... existing code ...
    
    # Check for tampering
    pdb = steve.persondb.PersonDB.open(db_fname)
    if election.is_tampered(pdb):
        _LOGGER.critical(
            f'TAMPERING DETECTED: election[E:{election_id}] data integrity check failed;'
            f' tally aborted'
        )
        sys.exit(1)

    _LOGGER.info(f'Starting tally for election[E:{election_id}]')
    tally_election(election, issue_id, output_format)
    _LOGGER.info(f'Tally completed for election[E:{election_id}]')


def tally_election(election, issue_id, output_format):
    # ...
    for issue in issues:
        try:
            tally_result, issue_voters = election.tally_issue(issue.iid)
            # ...
        except Exception as e:
            _LOGGER.error(
                f'Tally failed for issue[I:{issue.iid}] in election[E:{election.eid}]: {e}'
            )
            raise
```

### MEDIUM Severity

#### ASVS-1624-MED-001: Timestamp Format Missing Year, Seconds, and Timezone Prevents Log Correlation

**Severity:** MEDIUM  
**File:** `v3/server/main.py`  
**Lines:** 20, 51-56, 85-90  
**Functions:** `run_standalone()`, `run_asgi()`

**Related Functions Checked:**
- `run_standalone()` - Standalone server configuration
- `run_asgi()` - ASGI server configuration
- All `_LOGGER.*()` calls across pages.py, election.py

**Description:**

The timestamp format `'%m/%d %H:%M'` configured in `DATE_FORMAT` (line 20) and applied in both server run modes omits critical temporal information: year, seconds, and timezone. This prevents reliable log correlation across time periods, rapid event sequences, and distributed systems.

**Vulnerable Code:**

```python
DATE_FORMAT = '%m/%d %H:%M'  # line 20

# Applied in both run modes:
logging.basicConfig(
    level=logging.DEBUG,
    style='{',
    format='[{asctime}|{levelname}|{name}] {message}',
    datefmt=DATE_FORMAT,  # '%m/%d %H:%M'
)
```

**Data Flow:**
```
All _LOGGER.*() calls across the application 
  → Python logging module 
  → DATE_FORMAT applied 
  → Log output lacks year, seconds, timezone
```

**Example Output:**
```
[01/15 14:30|INFO|pages] User[U:alice] voted on issue[I:xyz] in election[E:abc]
```

**Impact:**

- **No year:** Logs from 2024 and 2025 are indistinguishable, breaking historical correlation and audit trails. For an election system, this is particularly problematic since elections may need investigation months or years later.
- **No seconds:** Events within the same minute cannot be ordered. During rapid vote submission, the sequence of operations becomes ambiguous. A SIEM cannot reconstruct the precise timeline of security events.
- **No timezone:** When logs are collected from servers in different locations or compared against client-side timestamps, the time offset is unknown, making correlation impossible.

**Proof of Concept:**

Two events logged 30 seconds apart on January 15, 2025:
```
[01/15 14:30|INFO|pages] User[U:alice] voted on issue[I:001] in election[E:abc]
[01/15 14:30|INFO|pages] User[U:bob] voted on issue[I:001] in election[E:abc]
```

A log processor cannot determine which vote came first, whether these are from 2024 or 2025, or what timezone the server is in.

**Remediation:**

```python
# Use ISO 8601 format with timezone for SIEM compatibility
DATE_FORMAT = '%Y-%m-%dT%H:%M:%S%z'

# Or for even better precision:
DATE_FORMAT = '%Y-%m-%dT%H:%M:%S.%f%z'

# Ensure timezone is set (in logging config or globally):
import time
import os
os.environ['TZ'] = 'UTC'
time.tzset()
```

#### ASVS-1624-MED-003: Inconsistent Log Format Between Web Server and CLI Components Prevents Unified Log Processing

**Severity:** MEDIUM  
**File:** `v3/server/bin/tally.py`, `v3/server/main.py`  
**Lines:** tally.py:148; main.py:51-56  
**Functions:** `tally.py:__main__`, `main.py:run_standalone()`, `main.py:run_asgi()`

**Related Functions Checked:**
- `tally.py:main()` - CLI entry point
- `main.py:run_standalone()` - Web server configuration
- `main.py:run_asgi()` - ASGI server configuration

**Description:**

The web server and CLI tally tool use different log formats, preventing unified log processing. The web server uses a custom pipe-delimited format while the CLI uses Python's default colon-delimited format. This format divergence creates a correlation gap at the most critical phase of the election lifecycle.

**Vulnerable Code:**

Web server (main.py:51-56):
```python
logging.basicConfig(
    level=logging.DEBUG,
    style='{',
    format='[{asctime}|{levelname}|{name}] {message}',
    datefmt='%m/%d %H:%M',
)
```

CLI tally tool (tally.py:148):
```python
logging.basicConfig(level=logging.INFO)
# Uses Python default: "%(levelname)s:%(name)s:%(message)s"
# with default date format: "%Y-%m-%d %H:%M:%S,uuu"
```

**Data Flow:**
```
Security events generated during tallying 
  → Python default log format 
  → Different structure than web server logs 
  → Log processor cannot parse both formats with one configuration
```

**Impact:**

A log processor (SIEM, ELK, Splunk) configured to parse the web server's pipe-delimited format `[{asctime}|{levelname}|{name}] {message}` will fail to parse tally logs in Python's default colon-delimited format `levelname:name:message`. This means:

- **Tally operations cannot be correlated** with web server events (e.g., who closed the election, then who ran the tally)
- **Two parser configurations** are needed, and log processors may not correctly identify which format applies to which line
- The audit trail has a **gap at the most critical phase**—the transition from election close to vote tallying

**Proof of Concept:**

Example of the two formats for related events:
```
# Web server log (when admin closes election):
[01/15 14:30|INFO|pages] User[U:alice] closed election[E:abc123]

# CLI tally log (when same admin tallies):
INFO:__main__:Tallying 5 issues ...
```

**Remediation:**

```python
# tally.py - Use the same format as the web server
DATE_FORMAT = '%Y-%m-%dT%H:%M:%S%z'  # Fixed version from Finding 1

logging.basicConfig(
    level=logging.INFO,
    style='{',
    format='[{asctime}|{levelname}|{name}] {message}',
    datefmt=DATE_FORMAT,
)
```

Better yet, extract the log format to a shared configuration module:
```python
# steve/log_config.py
LOG_FORMAT = '[{asctime}|{levelname}|{name}] {message}'
LOG_DATE_FORMAT = '%Y-%m-%dT%H:%M:%S%z'
LOG_STYLE = '{'

def configure_logging(level=logging.INFO):
    logging.basicConfig(
        level=level,
        style=LOG_STYLE,
        format=LOG_FORMAT,
        datefmt=LOG_DATE_FORMAT,
    )
```

#### ASVS-1624-MED-004: Debug print() Statements in Production Handlers Bypass Structured Logging

**Severity:** MEDIUM  
**File:** `v3/server/pages.py`  
**Lines:** 510, 531  
**Functions:** `do_add_issue_endpoint()`, `do_edit_issue_endpoint()`

**Related Functions Checked:**
- `do_add_issue_endpoint()` - Issue creation handler
- `do_edit_issue_endpoint()` - Issue modification handler
- Other logging calls in pages.py (lines 115, 134, 413, 464, 480, 496, 511, 533, 554)

**Description:**

Debug `print()` statements in production request handlers bypass the structured logging framework. This is a Type B gap: `_LOGGER` is imported and used elsewhere in the same file for similar operations, but `print()` is used here instead.

**Vulnerable Code:**

```python
# do_add_issue_endpoint (line 510):
async def do_add_issue_endpoint(election):
    result = await basic_info()
    form = edict(await quart.request.form)
    print('FORM:', form)  # <-- bypasses logging framework
    # ...

# do_edit_issue_endpoint (line 531):
async def do_edit_issue_endpoint(election, issue):
    result = await basic_info()
    form = edict(await quart.request.form)
    print('FORM:', form)  # <-- bypasses logging framework
    # ...
```

**Data Flow:**
```
Admin form submission (issue title, description) 
  → print() 
  → stdout 
  → No timestamp, no severity, no logger metadata 
  → Cannot be correlated with other logged events
```

**Impact:**

- Form data from administrative operations (creating/editing election issues) is dumped to stdout without any log metadata
- A log processor cannot associate these outputs with the `_LOGGER.info()` entries that follow them in the same handler
- The raw form data may include user-supplied content that is unescaped, potentially causing log parsing issues if stdout is captured
- These entries have no timestamp, severity, or correlation identifiers

**Remediation:**

```python
# Replace print() with proper debug logging:
async def do_add_issue_endpoint(election):
    result = await basic_info()
    form = edict(await quart.request.form)
    _LOGGER.debug(f'Issue creation form received for election[E:{election.eid}]')
    # ... rest of handler

async def do_edit_issue_endpoint(election, issue):
    result = await basic_info()
    form = edict(await quart.request.form)
    _LOGGER.debug(f'Issue edit form received for election[E:{election.eid}], issue[I:{issue.iid}]')
    # ... rest of handler
```

## Architecture Observations

### 1. No Centralized Log Configuration

Each entry point (main.py, tally.py) configures logging independently, leading to format divergence. A shared logging configuration module would enforce consistency across all application components.

### 2. No Structured Format (JSON)

While the tagged identifier convention (`User[U:...]`) is commendable and enables basic grep-based correlation, it requires custom regex parsing. JSON-formatted logs would be natively parseable by all major log processors (ELK, Splunk, CloudWatch) without custom configuration.

### 3. CLI/Web Gap

The tally process is the most security-critical operation (decrypting and counting votes) but has the weakest logging. This creates an audit trail gap at the most sensitive phase of the election lifecycle—the transition from election close to vote tallying and results publication.

### 4. No Request Correlation ID

Web requests don't generate or log a correlation/trace ID. For multi-step operations (loading election → checking authorization → submitting vote → logging), individual log entries cannot be linked to a single HTTP request. This makes it difficult to reconstruct the full context of security events.

### 5. Logging Maturity by Component

| Component | Maturity Level | Rationale |
|-----------|---------------|-----------|
| `pages.py` | Good | Consistent use of structured logging with user context; only minor print() issues |
| `election.py` | Good | Library-level logging with proper separation of concerns |
| `main.py` | Partial | Logging configured but timestamp format critically deficient |
| `tally.py` | Poor | Mixed print()/logger usage; different format; most critical events unlogged |
