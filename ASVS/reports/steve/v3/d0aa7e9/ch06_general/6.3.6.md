## Positive Security Patterns

### 1. Clean Separation of Authentication and Notification
Email is used exclusively in `mail-voters.py` for voter notifications. The authentication layer (`asfquart.auth`) is completely separate and OAuth-based.

### 2. No Magic Links or Email OTPs
The codebase contains no code to generate single-use authentication tokens for email delivery. Specifically absent:
- Token generation for email-based login
- OTP code generation or verification
- Email verification as an authentication step

### 3. Consistent OAuth Enforcement
All 21 routes that require authentication use `@asfquart.auth.require` with appropriate role requirements (`R.committer` or `R.pmc_member`). No route accepts email-based credentials.

### 4. Vote Identity Derived from Authenticated Session
In `do_vote_endpoint()` (line 470), the voter identity comes from `result.uid` (the OAuth-authenticated session), not from any email-provided identifier:

```python
election.add_vote(result.uid, iid, votestring)
```

### 5. Voter Notification is Admin-Initiated CLI Tool
The `mail-voters.py` script is a privileged CLI tool (not a web endpoint), requiring server access to execute. This prevents unauthorized mass email distribution and is consistent with the known pattern for CLI tools with direct database access.

## Architecture Observations

### 1. External Authentication Dependency
The actual authentication implementation resides in `asfquart.auth`, which is not included in the provided source files. Based on usage patterns (OAuth roles like `R.committer`, `R.pmc_member`), this is ASF's LDAP/OAuth infrastructure — not email-based. Full compliance verification would benefit from auditing the `asfquart.auth` module source.

### 2. No Password Recovery via Email
The codebase contains no password reset, account recovery, or email verification flows, which are common places where email-as-authentication creeps in. Authentication is entirely delegated to the external ASF identity provider.

### 3. ARCHITECTURE.md Confirmation
From the Security section:
> Authentication: External (ASF OAuth); no local auth.

This architectural decision explicitly avoids local authentication mechanisms, including email-based ones.

## Recommendations Summary

Since no violations of ASVS 6.3.6 were found, the following are hardening recommendations to maintain compliance:

### Immediate Actions
None required — application is currently compliant.

### Short-term Improvements (1-3 months)

**1. Document the No-Email-Auth Policy**
- **Action:** Add an explicit statement in `ARCHITECTURE.md` or a security policy document that email must never be used as an authentication factor
- **Rationale:** Prevents future regression during feature development
- **Effort:** Low (documentation update)

**2. Template Content Review Process**
- **Action:** Establish a review process to ensure administrator-provided EZT templates never include authentication-equivalent information
- **Rationale:** Defense-in-depth, though current architecture makes this safe since all routes require OAuth
- **Effort:** Low (process documentation)

### Long-term Strategic Recommendations (3-6 months)

**3. Audit `asfquart.auth` Module**
- **Action:** Conduct a complete ASVS L3 audit of the external `asfquart.auth` module
- **Rationale:** Verify that the external authentication provider does not fall back to email-based authentication under any circumstances (e.g., account recovery flows)
- **Effort:** Medium (requires access to external dependency source)

## Conclusion

**ASVS 6.3.6 Compliance Status: ✅ PASS**

The Apache Steve voting platform demonstrates exemplary compliance with ASVS 6.3.6. The application uses external ASF OAuth for all authentication, with email used exclusively for voter notifications via a privileged CLI tool. No authentication tokens or credentials are transmitted via email, and no email-based single-factor or multi-factor authentication mechanisms exist in the audited codebase.

The architecture's clean separation between authentication (OAuth) and notification (email) provides a strong foundation for maintaining compliance as the application evolves.
