## Findings

### Summary

**No findings at MEDIUM severity or above were identified.**

All random value generation in the production codebase uses Python's `secrets` module, which provides cryptographically secure random numbers by wrapping the operating system's CSPRNG (`/dev/urandom` on Linux, `CryptGenRandom` on Windows, `arc4random()` on BSD/macOS). This fully satisfies ASVS requirement 6.5.3.

## Positive Security Patterns

### 1. Consistent CSPRNG Usage Throughout Codebase

All three random-generation functions in `crypto.py` exclusively use the `secrets` module:

```python
# Salt generation (line 35)
def gen_salt() -> bytes:
    return secrets.token_bytes(SALT_LEN)  # CSPRNG

# ID generation (line 111-112)
def create_id():
    return secrets.token_hex(5)  # CSPRNG

# Shuffle (line 104)
j = secrets.randbelow(i + 1)  # CSPRNG
```

**Security Benefit:** Eliminates predictability attacks against salts, IDs, and shuffle operations.

### 2. Centralized Random Generation Architecture

All random value generation is centralized in `crypto.py`. No other module generates random values independently.

**Security Benefit:** Prevents the common anti-pattern where developers accidentally use the non-cryptographic `random` module in new code. Single point of control for cryptographic randomness.

### 3. Proper Key Derivation Chain

The key derivation chain uses CSPRNG-generated salts at every stage:

1. **Election salt** → `secrets.token_bytes(16)` → Argon2 → `opened_key`
2. **Per-voter salt** → `secrets.token_bytes(16)` → Argon2 → `vote_token`
3. **Vote encryption** → HKDF-SHA256 derives Fernet key from `vote_token` + salt

**Security Benefit:** Ensures each derived key has unique, unpredictable entropy source.

### 4. Cryptographically Secure Fisher-Yates Shuffle

The custom `shuffle()` function correctly implements Fisher-Yates algorithm using `secrets.randbelow()` rather than Python's standard `random.shuffle()`:

```python
def shuffle(x):
    for i in range(len(x) - 1, 0, -1):
        j = secrets.randbelow(i + 1)  # Cryptographically uniform
        x[i], x[j] = x[j], x[i]
```

**Security Benefit:** Used in `tally_issue()` to randomize vote order before tallying, preventing timing/order-based analysis. Standard `random.shuffle()` uses Mersenne Twister (predictable after observing 624 outputs).

### 5. No Insecure Random Module in Production Code

The `random` standard library module is not imported anywhere in the production codebase.

**Security Benefit:** Eliminates entire class of vulnerabilities from accidental non-CSPRNG usage.

### 6. Schema-Level Enforcement of Cryptographic Parameters

SQL schema enforces salt and key sizes with CHECK constraints:

```sql
salt  BLOB  CHECK (salt IS NULL OR length(salt) = 16),
opened_key  BLOB  CHECK (opened_key IS NULL OR length(opened_key) = 32),
vote_token  BLOB NOT NULL  CHECK (length(vote_token) = 32),
```

**Security Benefit:** Defense-in-depth—prevents storage of improperly sized cryptographic material even if application logic fails.

## Architecture Observations

### Multi-Factor Authentication Scope

The application **does not implement its own MFA mechanisms**. Authentication is delegated to an external ASF OAuth system, as documented in `ARCHITECTURE.md`:

> **Authentication**: Relies on external ASF authentication via `asfquart.auth`. Requires committer-level access for most operations.

**Implication:** ASVS 6.5.3's specific requirements for lookup secrets, TOTP seeds, and out-of-band authentication codes are **not directly applicable** to this codebase's architectural scope. However, the underlying principle—using CSPRNG for all security-sensitive random values—is fully satisfied for the cryptographic operations the application does perform (salt generation, ID generation, shuffle operations).

### ID Generation Entropy Analysis (Informational)

`create_id()` generates 40-bit hexadecimal IDs via `secrets.token_hex(5)`:
- **Entropy:** 40 bits = 2^40 ≈ 1.1 trillion possible values
- **Purpose:** Obscured resource identifiers for elections and issues
- **Protection:** All endpoints require authentication; IDs are not authentication tokens

Per `schema.md`:
> 10-character hexadecimal IDs... to prevent URL deduction

**Assessment:** While 40 bits is lower than typical session token entropy (128+ bits), this is appropriate for the use case:
1. IDs are not authentication credentials
2. All access is behind authentication layer
3. Intentional design per documentation
4. Uses CSPRNG (compliant with ASVS 6.5.3)

**Status:** Informational observation; no security concern.

### Developer Tooling Exclusion

`crypto.py` contains a `benchmark_argon2()` function (line ~115) that uses a hardcoded salt:

```python
if __name__ == '__main__':
    benchmark_argon2(b'16_byte_salt_123')
```

**Assessment:** This is exclusively a developer benchmarking tool with no path to production use. Excluded from audit scope per standard practice for developer utilities.

## Recommendations Summary

**Note:** Since no findings at MEDIUM+ severity were identified, these are defense-in-depth recommendations to maintain current security posture.

### Immediate Actions
*None required* — codebase is compliant.

### Short-Term Enhancements (Optional)

1. **Add Pre-Commit Hook for CSPRNG Enforcement**
   - **Action:** Create a lint rule to flag `import random` in production code paths
   - **Benefit:** Prevents future regression if new developers introduce non-cryptographic randomness
   - **Implementation:** Add to `.pre-commit-config.yaml`:
     ```yaml
     - repo: local
       hooks:
         - id: no-random-module
           name: Prevent non-CSPRNG random usage
           entry: 'import random'
           language: pygrep
           files: '^v3/steve/.*\.py$'
           exclude: 'test_.*\.py$'
     ```

2. **Document CSPRNG Policy**
   - **Action:** Add developer documentation stating CSPRNG requirement
   - **Location:** `CONTRIBUTING.md` or `v3/steve/crypto.py` module docstring
   - **Content Example:**
     ```python
     """
     Cryptographic utilities for Apache Steve.
     
     SECURITY POLICY: All random value generation MUST use Python's
     `secrets` module to comply with ASVS 6.5.3 (CSPRNG requirement).
     Never use the `random` module for security-sensitive operations.
     """
     ```

### Long-Term Considerations

1. **Maintain Centralized Random Generation**
   - Continue routing all random value generation through `crypto.py`
   - Review new PRs to ensure no direct `secrets` usage outside this module
   - Benefit: Single point of control for future security audits

2. **Periodic Dependency Review**
   - Monitor Python `secrets` module for any security advisories
   - Review cryptographic library updates (Argon2, cryptography package)
   - Frequency: Quarterly or with major Python version updates
