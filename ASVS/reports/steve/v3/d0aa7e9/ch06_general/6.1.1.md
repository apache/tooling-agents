## Findings

### Medium Severity

#### Finding ASVS-611-MED-001

**Title:** Complete absence of documentation defining authentication defense controls

**Location:**
- **Files:** `v3/TODO.md`, `v3/docs/schema.md`, `v3/server/pages.py`, `v3/server/main.py`
- **Scope:** Project-wide documentation gap

**Gap Type:** Type A — No control exists

**Related Functions Checked:**
- `create_app()` in `v3/server/main.py` (line ~33)
- OAuth configuration in `v3/server/main.py` (lines ~39-43)
- All authentication decorators in `v3/server/pages.py`

**Description:**

ASVS 6.1.1 requires application documentation to explicitly define how rate limiting, anti-automation, and adaptive response controls defend against credential stuffing and password brute force, and how they prevent malicious account lockout. A thorough review of all provided documentation and code reveals **no documentation whatsoever** addressing these concerns.

**Evidence of Gap:**

1. **`v3/docs/schema.md`** — Documents database schema, cryptographic fields, and election lifecycle but contains zero references to rate limiting, brute force protection, login attempt tracking, account lockout, or anti-automation controls.

2. **`v3/TODO.md`** — Enumerates 9 known issues/improvements. While it identifies missing CSRF tokens (§6, §8), it makes **no mention** of rate limiting, brute force defense, or account lockout prevention despite these being higher-priority authentication concerns.

3. **`v3/server/main.py`** — Application bootstrap at `create_app()` (line ~33) attaches no rate limiting middleware:

**Vulnerable Code:**

```python
# v3/server/main.py, line ~33
def create_app():
    "Create the asfquart app and its endpoints."
    # ...
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    import pages  # noqa: F401
    import api  # noqa: F401
    return app
    # No rate limiting middleware attached
```

4. **`v3/server/pages.py`** — Authentication is delegated to Apache OAuth via `@asfquart.auth.require`, but no documentation explains:
   - What brute force protections `oauth.apache.org` provides
   - Whether there are retry limits on the OAuth callback flow
   - How the application would detect or respond to credential stuffing
   - How malicious account lockout is prevented at the identity provider level

5. **OAuth configuration** in `main.py` (lines ~39-43) provides URLs but no security documentation:

```python
# v3/server/main.py, lines ~39-43
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**

```
User → OAuth Provider (oauth.apache.org) → Application
                ↓
        [Undocumented security controls]
                ↓
        No documented rate limiting
        No documented anti-automation
        No documented lockout prevention
```

**Proof of Concept:**

N/A — This is a documentation gap rather than an exploitable vulnerability. However, the absence of documentation makes it impossible to verify that adequate controls exist at the OAuth provider level.

**Impact:**

Without documented authentication defense controls:
- Security auditors cannot verify that brute force protections are correctly configured
- Operations teams cannot tune rate limits or lockout thresholds
- It is unknown whether the external OAuth provider implements sufficient protections, creating an undocumented trust dependency
- If the OAuth provider does implement account lockout, there is no documented strategy for preventing an attacker from maliciously locking out legitimate voters during an active election — a particularly impactful scenario for a voting system

**Remediation:**

Create an authentication security document (e.g., `v3/docs/authentication-security.md`) that addresses:

```markdown
# Authentication Security Controls

## Authentication Flow
- Authentication is delegated to Apache OAuth (oauth.apache.org)
- [Document OAuth provider's brute force protections]

## Rate Limiting
- Login attempts: [X attempts per Y minutes per IP/account]
- Vote submission: [X votes per Y seconds per authenticated user]  
- API endpoints: [Requests per minute per session]
- Implementation: [middleware/library used, configuration location]

## Anti-Automation
- [CAPTCHA/challenge requirements after N failed attempts]
- [Bot detection mechanisms]

## Adaptive Response
- After N failed login attempts: [action taken]
- After N failed attempts from same IP: [action taken]
- Escalation policy: [describe progressive response]

## Account Lockout Prevention
- Lockout policy: [temporary vs permanent, duration]
- Anti-lockout measures: [how legitimate users recover]
- Election-specific: [protections against voter lockout during active elections]

## Configuration
- [Where settings are configured]
- [How to modify thresholds]
- [Monitoring and alerting for attack detection]
```

#### Finding ASVS-611-MED-002

**Title:** No rate limiting or throttling on vote submission and state-changing endpoints

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:**
  - `do_vote_endpoint()` (line ~367)
  - `do_create_endpoint()` (line ~408)
  - `do_open_endpoint()` (line ~429)
  - `do_close_endpoint()` (line ~448)

**Gap Type:** Type A — No control exists

**Related Functions Checked:**
- `do_vote_endpoint()` — vote submission handler
- `do_open_endpoint()` — election state change handler
- `do_close_endpoint()` — election state change handler
- `do_create_endpoint()` — election creation handler

**Description:**

Per the known false positive guidance, which directs auditors to "*focus on authentication brute-force protection and whether vote submission has appropriate throttling*," the vote submission and election state-change endpoints have no rate limiting or throttling controls, and no documentation exists describing how such controls should be configured.

**Vulnerable Code:**

```python
# v3/server/pages.py, line ~367
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    # No rate limiting check
    # No anti-automation check
    result = await basic_info()
    form = edict(await quart.request.form)
    # Processes vote immediately...
    for iid, votestring in votes.items():
        election.add_vote(result.uid, iid, votestring)
```

```python
# v3/server/pages.py, line ~429 — State change via GET (no CSRF, no throttle)
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    # No rate limiting, no throttle
    election.open(pdb)
```

**Data Flow:**

```
Authenticated User Request
        ↓
@asfquart.auth.require({R.committer})  [Auth check only]
        ↓
[NO RATE LIMITING]
        ↓
@load_election  [Load election data]
        ↓
Process vote/state change immediately
        ↓
Database write (SQLite single-writer)
```

**Proof of Concept:**

An authenticated attacker (any committer) could submit rapid automated requests:

```bash
# Rapid re-voting to potentially cause database contention or abuse
for i in $(seq 1 1000); do
  curl -X POST https://target/do-vote/$EID \
    -H "Cookie: session=$SESSION" \
    -d "vote-$IID=yes" &
done
```

**Impact:**

- Rapid vote submission could cause database contention in SQLite (single-writer model)
- Automated election state changes (`/do-open/`, `/do-close/` via GET) could be triggered by link prefetching or CSRF
- No documentation exists to describe what throttling *should* be applied, making it impossible to assess whether the current state is intentional or an oversight
- An insider threat or compromised account could abuse the absence of throttling to disrupt elections

**Remediation:**

1. Implement rate limiting on sensitive endpoints:

```python
from quart_rate_limiter import RateLimiter, rate_limit

rate_limiter = RateLimiter(app)

@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@rate_limit(1, timedelta(seconds=5))  # 1 vote per 5 seconds
@load_election
async def do_vote_endpoint(election):
    ...
```

2. Document the rate limiting configuration in the authentication security document referenced in Finding ASVS-611-MED-001.

3. Add similar rate limiting to election state-change endpoints:

```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@rate_limit(5, timedelta(minutes=1))  # 5 state changes per minute
@load_election
async def do_open_endpoint(election):
    ...
```

## Architecture Observations

1. **Authentication is fully delegated** to Apache OAuth (`oauth.apache.org`). The application itself never handles credentials, which significantly reduces the credential stuffing attack surface. However, this delegation is undocumented from a security controls perspective.

2. **Closed community model**: The known false positive guidance confirms this is a "closed-community voting system with authenticated users." This reduces but does not eliminate the need for rate limiting documentation — an insider threat or compromised account could still abuse the absence of throttling.

3. **State-changing GET requests** (`/do-open/<eid>`, `/do-close/<eid>`) are particularly concerning in the context of this requirement because they combine the absence of CSRF protection (acknowledged in TODO.md) with the absence of rate limiting, making automated abuse trivial.

4. **SQLite single-writer architecture** makes the application particularly vulnerable to database contention from rapid automated requests, as there is no concurrent write capability to absorb burst traffic.
