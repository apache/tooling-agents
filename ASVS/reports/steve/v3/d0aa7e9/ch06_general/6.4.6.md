**Key Findings**

This application **does not manage local authentication factors (passwords)**. Authentication is entirely delegated to an external identity provider (ASF OAuth) as documented in `ARCHITECTURE.md` ("Authentication: External (ASF OAuth); no local auth."). The database schema (`person` table) contains only `pid`, `name`, and `email` — no password or credential fields exist.

Because there is no local credential management, there is no admin-initiated password reset feature, nor any mechanism for administrators to set or choose user passwords. **ASVS 6.4.6 is therefore not directly applicable** to this codebase's authentication boundary.

## Findings

**No findings identified for ASVS 6.4.6.**

### Applicability Assessment

ASVS 6.4.6 is **not applicable** to this codebase because:

1. **No Local Authentication Factors**: The application does not manage passwords, security questions, or any other local authentication factors
2. **No Password Storage**: The database schema contains no password hash, salt, or credential fields for user authentication
3. **No Password Reset Functionality**: No endpoints or methods exist for password reset, password change, or credential recovery
4. **External Authentication Only**: All authentication is delegated to ASF OAuth provider via `asfquart.auth`
5. **Admin Scope Limited**: Administrative functions are scoped exclusively to election management, not user credential management

### Authentication Boundary Analysis

```
┌─────────────────────────────────────────┐
│   ASF OAuth Provider (External)         │
│   - Password storage                     │
│   - Password reset                       │
│   - Credential management                │
│   - ASVS 6.4.6 compliance point         │
└──────────────┬──────────────────────────┘
               │ OAuth tokens
               ↓
┌─────────────────────────────────────────┐
│   Apache Steve Application               │
│   - Receives authenticated identity      │
│   - No local credentials                 │
│   - No password operations               │
│   - Election/voting operations only      │
└─────────────────────────────────────────┘
```

The application's authentication boundary ends at `asfquart.auth.require()`. All credential lifecycle management, including password reset compliance with ASVS 6.4.6, is the responsibility of the external ASF OAuth provider.

## Positive Security Patterns

### 1. Complete External Authentication Delegation

**Location:** `ARCHITECTURE.md`, `pages.py` (all protected routes)

**Pattern:**
```
# ARCHITECTURE.md
Authentication: External (ASF OAuth); no local auth.
```

```python
# pages.py - All protected routes use external auth
@asfquart.auth.require({R.committer})
@asfquart.auth.require({R.pmc_member})
```

**Security Benefit:** By delegating authentication entirely to ASF's OAuth infrastructure, the application avoids all local password management risks, including:
- Password storage vulnerabilities
- Password reset implementation flaws
- Admin credential manipulation (ASVS 6.4.6 concern)
- Password complexity enforcement
- Credential stuffing attacks

This architectural decision eliminates an entire class of authentication vulnerabilities at the application level.

### 2. Zero Credential Storage

**Location:** `v3/schema.sql` (person table), `v3/steve/persondb.py`

**Pattern:**
```sql
-- schema.sql, person table
CREATE TABLE person (
    pid   TEXT PRIMARY KEY NOT NULL,
    name  TEXT,
    email TEXT NOT NULL
) STRICT;
```

```python
# persondb.py
def add_person(self, pid, name, email):
    """Add a person to the database."""
    self.q_add_person(pid, name, email)
```

**Security Benefit:** The `person` table stores only identity metadata — no password hashes, security questions, recovery tokens, or any authentication factors. This design:
- Eliminates attack surface for credential theft
- Prevents admin credential manipulation entirely
- Simplifies GDPR/privacy compliance
- Reduces breach impact to identity metadata only

### 3. Clear Separation of Cryptographic Material

**Location:** `v3/steve/persondb.py` (line 37), `v3/schema.sql`

**Pattern:**
```python
# persondb.py line 37
def get_person(self, pid):
    "Return NAME, EMAIL for Person identified by PID."
    # NEVER return person.salt
    person = self.q_get_person.first_row(pid)
```

**Security Benefit:** The developer explicitly documents that salt values should not be returned. These salts (in `mayvote` and `election` tables) serve vote anonymization purposes — they are not authentication factors. This clear separation:
- Prevents confusion between authentication credentials and cryptographic material
- Documents intent for future maintainers
- Reduces risk of accidentally exposing sensitive cryptographic values

### 4. Admin Operations Scoped to Election Management

**Location:** `v3/server/pages.py` (all admin endpoints)

**Pattern:**
```python
# pages.py — Admin endpoints
@APP.post('/do-create-election')    # Creates elections
@APP.get('/do-open/<eid>')          # Opens elections
@APP.get('/do-close/<eid>')         # Closes elections
@APP.post('/do-add-issue/<eid>')    # Manages issues
```

**Security Benefit:** Admin functionality is strictly limited to election lifecycle management:
- Create, open, close elections
- Manage election issues
- View election results

There are **zero** admin operations that touch user identity or credentials. This separation of concerns:
- Follows principle of least privilege
- Prevents privilege escalation to credential management
- Simplifies security auditing
- Reduces admin attack surface

### 5. Session-Based Identity from External Provider

**Location:** `v3/server/pages.py` (`basic_info()` function, line ~68)

**Pattern:**
```python
def basic_info():
    """Extract basic user information from session."""
    session = asfquart.session.read()
    # Identity populated by external OAuth flow
```

**Security Benefit:** The application:
- Never prompts for passwords
- Never handles password input
- Never validates credentials
- Only reads pre-authenticated identity from OAuth session

This "zero-knowledge" approach to credentials ensures the application cannot be exploited for credential theft, even if compromised.

## Architecture Observations

### 1. Authentication Boundary

The application's authentication boundary ends at `asfquart.auth.require()`. All credential management (including password reset, MFA enrollment, account recovery, etc.) is the responsibility of the ASF OAuth provider. 

**Implication for ASVS 6.4.6:** Compliance with this requirement must be verified at the external provider level, not within this application. The application inherits the security posture of the ASF OAuth implementation.

### 2. No Local Account Creation

There is no endpoint for creating user accounts through the web interface. The `PersonDB.add_person()` method exists for programmatic use (e.g., by `load-fakedata.py` for testing) and only sets identity metadata, never credentials.

**Security Implication:** Prevents attackers from:
- Creating unauthorized accounts
- Bypassing external authentication
- Injecting malicious user records with known credentials

### 3. Identity vs. Authentication Separation

The application maintains a clear separation:
- **Identity** (who you are): Stored in `person` table (`pid`, `name`, `email`)
- **Authentication** (proving who you are): Handled entirely by ASF OAuth
- **Authorization** (what you can do): Managed by `asfquart.auth.require()` with role checks

This separation follows security best practices and simplifies the codebase.

### 4. Cryptographic Material Purpose

Multiple cryptographic fields exist in the schema (`salt`, `vote_token`, `ciphertext`, `opened_key`) but **none** are authentication factors:
- `election.salt` / `mayvote.salt`: Vote anonymization
- `vote.vote_token`: Blind signature for vote privacy
- `vote.ciphertext`: Encrypted vote payload
- `election.opened_key`: Election decryption key

This demonstrates that the presence of cryptographic fields does not imply local credential management.

### 5. Admin Privilege Model

Admin privileges are election-scoped, not system-scoped:
- Admins can manage elections they own
- No "super admin" with user management capabilities
- No cross-election administrative access (without explicit ownership)

This limits the blast radius of compromised admin accounts.

## Recommendations Summary

### Immediate Actions

**None required.** The application correctly delegates all authentication to an external provider and has no local credential management that could violate ASVS 6.4.6.

### Short-term Recommendations

#### 1. Document Authentication Delegation (Informational)
**Priority:** Low  
**Effort:** 1-2 hours

Add explicit documentation stating that ASVS authentication factor lifecycle requirements (including 6.4.6) are satisfied by the external ASF OAuth provider.

**Suggested Documentation Location:** `SECURITY.md` or `ARCHITECTURE.md`

**Example Content:**
```markdown
## Authentication Security (ASVS Compliance)

This application delegates all authentication to ASF OAuth. The following 
ASVS Level 3 requirements are satisfied by the external provider:

- 6.4.6: Admin-initiated password reset (admins cannot set/choose passwords)
- [Other relevant ASVS requirements]

For verification of these controls, refer to ASF OAuth security documentation 
at [URL].
```

**Benefit:** 
- Clarifies security responsibility boundaries
- Assists future security audits
- Demonstrates due diligence

#### 2. Verify External Provider Compliance (Out-of-Scope for This Audit)
**Priority:** Medium  
**Effort:** External coordination required

Confirm that the ASF OAuth identity provider itself complies with ASVS 6.4.6 — specifically that ASF admins who manage user accounts cannot directly set/choose user passwords, only initiate resets.

**Verification Steps:**
1. Review ASF OAuth documentation
2. Confirm password reset workflow
3. Verify admin capabilities are limited to triggering resets
4. Document findings

**Benefit:**
- Ensures end-to-end ASVS compliance
- Identifies gaps in external provider
- Provides audit trail

### Long-term Recommendations

#### 3. Maintain No-Local-Auth Pattern
**Priority:** High (if local auth is ever considered)  
**Effort:** Design review for any future authentication changes

If future requirements introduce local authentication (e.g., API keys, service accounts, emergency access), ensure that admin-initiated credential reset follows the pattern required by ASVS 6.4.6.

**Design Principles to Maintain:**
- Admins can **initiate** resets (trigger email/SMS)
- Admins **cannot** directly set or view credentials
- Reset tokens are time-limited and single-use
- Reset process requires user interaction (email confirmation, etc.)

**Example Compliant Flow:**
```
Admin Action → System generates secure token → User receives token via 
out-of-band channel → User sets new password → Admin never sees password
```

**Benefit:**
- Prevents future introduction of ASVS 6.4.6 violations
- Maintains current security posture
- Reduces audit findings in future reviews

#### 4. Consider OAuth Provider Redundancy
**Priority:** Low  
**Effort:** Significant architectural change

Consider supporting multiple OAuth providers (e.g., ASF OAuth + GitHub OAuth) for redundancy, while maintaining the no-local-auth pattern.

**Benefit:**
- Reduces single point of failure
- Improves availability
- Maintains security posture

**Risk:** Increased complexity in identity mapping and authorization.
