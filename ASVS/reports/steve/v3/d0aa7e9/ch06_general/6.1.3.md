### Key Issues Identified

The application implements multiple distinct authentication levels (public, session-only, committer, PMC member) via `asfquart.auth` and a separate CLI access pathway. The audit reveals critical gaps in documentation and consistency:

- **Multiple authentication pathways lack unified documentation** - Five distinct authentication levels exist without comprehensive security control mapping
- **Resource-level authorization documented but never implemented** - 10+ `### check authz` comments indicate planned security controls that were never coded (Type B gap)
- **Inconsistent authentication strength across related operations** - Vote display checks eligibility at web layer while vote submission relies only on business layer

## Findings

### HIGH Severity Findings

#### Finding ASVS-613-HIGH-001

**Title:** Multiple Authentication Pathways With Inconsistent Strength Not Documented Together

**Location:** 
- File: `v3/ARCHITECTURE.md` (Security section)
- File: `v3/server/pages.py` (all route definitions)
- Lines: 443-444, 464-466, 486-488, 506-508

**Related Functions Checked:**
- All route handlers in `pages.py`
- Authentication decorators: `@asfquart.auth.require`
- Role definitions in `asfquart.auth`

**Description:**

The application has five distinct authentication pathways/levels (public, session-only, committer, PMC member, CLI direct access), but ARCHITECTURE.md only documents authentication as:

```markdown
### Security
- **Authentication**: External (ASF OAuth); no local auth.
- **Authorization**: Role-based (committer, PMC member) via decorators.
```

This documentation fails to:
1. Enumerate all authentication pathways and their strength levels
2. Map which endpoints require which authentication level and why
3. Document the CLI pathway's authentication model and how it relates to the web pathway
4. Specify what security controls must be consistently enforced across all pathways
5. Explain why related operations require different authentication strength (PMC for create vs. committer for manage)

**Vulnerable Code:**

```python
# pages.py line 443-444 — Creating elections requires PMC member
@APP.post('/do-create-election')
@asfquart.auth.require({R.pmc_member})  ### need general solution

# pages.py line 464-466 — Opening elections requires only committer
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution

# pages.py line 486-488 — Closing elections requires only committer
@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution

# pages.py line 506-508 — Adding issues requires only committer
@APP.post('/do-add-issue/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
```

The `### need general solution` comments on 13 separate endpoints confirm this authentication model is provisional and undocumented.

**Data Flow:** 

Incoming request → `asfquart.auth.require` decorator → Role check → Handler execution — but no documented standard for which role should protect which operation.

**Proof of Concept:**

N/A - This is a documentation and consistency gap rather than an exploitable vulnerability.

**Impact:** 

Without documented authentication pathways and strength requirements, there is no verifiable standard for security review, and developers cannot determine if new endpoints use the correct authentication level. The `### need general solution` comments confirm the development team recognizes the model is incomplete but has no documentation to guide resolution.

**Remediation:**

Create a formal Authentication and Authorization Matrix document:

```markdown
# Authentication Pathways and Security Controls

## Web Authentication Pathway
- **Mechanism:** ASF OAuth via asfquart.auth
- **Session Management:** Server-side sessions via asfquart.session
- **Levels:**
  | Level | Requirement | Use Case | Endpoints |
  |-------|------------|----------|-----------|
  | Public | None | Information pages | /, /about, /static/* |
  | Session | Valid ASF OAuth session | Profile, settings | /profile, /settings |
  | Committer | ASF committer role | Voting, viewing | /voter, /vote-on/* |
  | Election Owner | Committer + ownership check | Election management | /manage/*, /do-open/*, /do-close/* |
  | PMC Member | PMC membership | Election creation | /do-create-election |

## CLI Authentication Pathway
- **Mechanism:** OS-level server access control
- **Rationale:** Tallying requires privileged server access
- **Controls:** Command execution logging, read-only for tallying

## Consistent Controls Across All Pathways
- CSRF tokens required for all state-changing operations
- Audit logging for all administrative actions
- Rate limiting for authentication attempts
```

#### Finding ASVS-613-HIGH-002

**Title:** Resource-Level Authorization Documented as Required But Never Implemented (Type B Gap)

**Location:** 
- File: `v3/server/pages.py`
- Lines: 83, 190, 223, 405, 447, 468, 489, 510, 536, 560

**Related Functions Checked:**
- `do_vote_endpoint()` - line 405
- `do_open_endpoint()` - line 468
- `do_close_endpoint()` - line 489
- `do_add_issue_endpoint()` - line 510
- `do_delete_issue_endpoint()` - line 560
- `manage_page()` - line 190
- All management endpoints with `### check authz` comments

**Description:**

The codebase contains 10+ explicit `### check authz` comments indicating the developers intended to implement fine-grained authorization checks at each endpoint. However, none of these checks were ever implemented. This creates a Type B gap: the security control is DEFINED (via comments documenting the intent) but NOT CALLED, giving false confidence that the security model was considered.

As a result, authentication strength is not consistently enforced — any authenticated committer can perform administrative operations on ANY election, regardless of ownership.

**Vulnerable Code:**

```python
# pages.py line 405 — Vote endpoint, no ownership/eligibility check at web layer
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    ### check authz    # <-- NEVER IMPLEMENTED
    form = edict(await quart.request.form)
    ...
    election.add_vote(result.uid, iid, votestring)
```

```python
# pages.py line 468 — Open election endpoint, any committer can open any election
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz    # <-- NEVER IMPLEMENTED
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)  # Any committer can open ANY election
```

```python
# pages.py line 560 — Delete issue endpoint, any committer can delete issues
@APP.post('/do-delete-issue/<eid>/<iid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election_issue
async def do_delete_issue_endpoint(election, issue):
    result = await basic_info()
    ### check authz    # <-- NEVER IMPLEMENTED
    election.delete_issue(issue.iid)  # Any committer deletes any issue
```

**Data Flow:**

Request → Authentication check (role-based) → Election loaded → ~~Authorization check (ownership)~~ → Operation executed without ownership verification

**Proof of Concept:**

An authenticated ASF committer who is NOT the election owner can:
1. `GET /do-open/<eid>` — Open any election for voting
2. `GET /do-close/<eid>` — Close any election, ending voting
3. `POST /do-add-issue/<eid>` — Add unauthorized issues to any election
4. `POST /do-delete-issue/<eid>/<iid>` — Delete issues from any election
5. `POST /do-edit-issue/<eid>/<iid>` — Modify issues in any election

**Impact:** 

The authentication model implies per-resource authorization (election ownership is tracked via `owner_pid` in the database), but this is never enforced at the web layer. Any committer can manipulate any election's state and content, undermining the entire election integrity model. This is particularly dangerous because `election.py` stores `owner_pid` (line ~379) suggesting the data model supports ownership — it's just never checked.

**Remediation:**

Implement an authorization decorator that verifies election ownership:

```python
def require_election_owner(func):
    """Decorator to verify the authenticated user owns the election."""
    @functools.wraps(func)
    async def checker(election, *args, **kwargs):
        result = await basic_info()
        md = election.get_metadata()
        if md.owner_pid != result.uid:
            _LOGGER.warning(
                f'User[U:{result.uid}] denied access to election[E:{election.eid}]'
                f' owned by [U:{md.owner_pid}]'
            )
            quart.abort(403)
        return await func(election, *args, **kwargs)
    return checker

# Apply to management endpoints:
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
@require_election_owner  # NEW: enforce ownership
async def do_open_endpoint(election):
    ...
```

### MEDIUM Severity Findings

#### Finding ASVS-613-MEDIUM-001

**Title:** Inconsistent Authentication Level Between Vote Display Page and Vote Submission Endpoint

**Location:** 
- File: `v3/server/pages.py`
- Functions: `vote_on_page` (lines 231-290) vs `do_vote_endpoint` (lines 402-440)

**Related Functions Checked:**
- `vote_on_page()` - GET endpoint with eligibility check
- `do_vote_endpoint()` - POST endpoint without web-layer eligibility check
- `election.q_find_issues` - eligibility query
- `election.add_vote()` - business layer with `q_get_mayvote` check

**Description:**

The vote display page (`GET /vote-on/<eid>`) includes a voter eligibility check via `election.q_find_issues`, but the corresponding vote submission endpoint (`POST /do-vote/<eid>`) does not perform the same check at the web layer. While the business logic layer (`election.add_vote`) does check `q_get_mayvote`, the inconsistency in where and how security controls are applied across these related authentication pathways violates the principle of consistent enforcement.

**Vulnerable Code:**

```python
# pages.py lines 249-253 — GET endpoint checks eligibility at web layer
@APP.get('/vote-on/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def vote_on_page(election):
    ...
    # Check if user can vote in this election (has mayvote entries)
    election.q_find_issues.perform(result.uid, election.eid)
    if not election.q_find_issues.fetchall():
        ...
        raise_404(T_BAD_EID, result)  # Explicit eligibility check

# pages.py lines 402-440 — POST endpoint has NO eligibility check at web layer
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    ### check authz    # <-- NEVER IMPLEMENTED — no eligibility check here
    ...
    for iid, votestring in votes.items():
        election.add_vote(result.uid, iid, votestring)
        # Business layer checks mayvote, but error handling is generic
```

**Data Flow:** 

- **GET path:** Request → Auth check → Election load → **Eligibility check** → Render page
- **POST path:** Request → Auth check → Election load → ~~Eligibility check~~ → Process votes → Business layer may reject

**Proof of Concept:**

1. Authenticated committer accesses `GET /vote-on/<eid>` for an election where they are not eligible
2. Web layer correctly rejects with 404 error
3. Same committer directly POSTs to `/do-vote/<eid>` bypassing web-layer check
4. Vote is processed by business layer, which may reject with different error handling

**Impact:** 

The inconsistent placement of the eligibility check means that if the business-layer check in `add_vote` fails or is modified, the web layer provides no safety net. Additionally, error messages from the business layer are caught generically (`except Exception as e`), potentially revealing different information than the web-layer check would provide.

**Remediation:**

Apply consistent eligibility checking at the web layer for both endpoints:

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    
    # Consistent eligibility check (same as vote_on_page)
    election.q_find_issues.perform(result.uid, election.eid)
    if not election.q_find_issues.fetchall():
        await flash_danger('You are not eligible to vote in this election.')
        return quart.redirect('/voter', code=303)
    
    # ... rest of vote processing
```

## Positive Security Patterns

### 1. Consistent Authentication Decorator Usage

**Implementation:** Every protected endpoint uses `@asfquart.auth.require` with appropriate role requirements, even if the specific roles need refinement.

**Benefit:** Provides a clear, auditable pattern for authentication enforcement across all web endpoints.

### 2. Business-Layer State Enforcement

**Implementation:** `election.py` consistently uses state assertions (`is_editable()`, `is_open()`, `is_closed()`) before allowing operations.

**Benefit:** Provides defense-in-depth for election lifecycle integrity, preventing operations in invalid states regardless of web-layer controls.

### 3. Parameterized Database Queries

**Implementation:** All database operations use `cursor.perform()` with parameterized queries.

**Benefit:** Prevents SQL injection vulnerabilities across the entire application.

### 4. Cryptographic Vote Protection

**Implementation:** Vote encryption/decryption model with per-voter salts and vote tokens.

**Benefit:** Provides strong anonymity guarantees for the voting process.

### 5. Comprehensive Audit Logging

**Implementation:** Security-relevant operations (election creation, opening, closing, voting) are logged with user and resource identifiers.

**Benefit:** Enables security monitoring and forensic analysis of administrative actions.

### 6. Document Access Control

**Implementation:** The `serve_doc` endpoint correctly checks `q_get_mayvote` before serving per-issue documents.

**Benefit:** Ensures only eligible voters can access election-related documentation, despite using lighter session-only authentication.

## Architecture Observations

### 1. Single Authentication Provider Model

The application uses a single authentication provider (ASF OAuth) for all web access, which simplifies the authentication model. The different decorator levels (bare, committer, PMC member) represent authorization tiers within the same authentication pathway rather than separate authentication mechanisms.

**Implication:** This reduces complexity but requires clear documentation of authorization levels and their enforcement points.

### 2. Defense-in-Depth Gap

The business logic layer (`election.py`) enforces state-based controls but does not enforce user-level authorization. The web layer (`pages.py`) enforces role-based authentication but not resource-level authorization. Neither layer provides complete authorization coverage.

**Implication:** Modifications to either layer could inadvertently create security gaps without the other layer providing backup protection.

### 3. Known Technical Debt

The `### need general solution` and `### check authz` comments represent acknowledged technical debt that should be tracked and prioritized before production deployment.

**Implication:** The development team is aware of the gaps, but without formal tracking and remediation plans, these issues may persist indefinitely.

### 4. CLI Pathway Trust Model

CLI tools bypass all application-level authentication, relying entirely on server access controls. This is documented in the known false positive patterns but should be formally included in authentication pathway documentation.

**Implication:** The security boundary for CLI access is at the OS level rather than application level, which must be clearly communicated to operators and security reviewers.
