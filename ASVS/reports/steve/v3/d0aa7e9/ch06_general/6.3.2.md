## Positive Security Patterns

### 1. External Authentication by Design
**Location:** `v3/server/pages.py` via `asfquart.auth`

The application completely eliminates local credential management by delegating all authentication to ASF's OAuth infrastructure. This architectural decision removes the entire attack surface related to default accounts with known credentials.

```python
@auth.require({R.committer})
async def voter(result):
    # Authentication handled externally; result.uid contains verified identity
```

**Impact:** Prevents default account vulnerabilities by design.

### 2. No Local Credential Storage
**Location:** `v3/schema.sql`, `person` table

The person table functions as an identity registry, not an authentication database:

```sql
CREATE TABLE person (
    pid   TEXT PRIMARY KEY NOT NULL,
    name  TEXT,
    email TEXT NOT NULL
) STRICT;
```

**Impact:** Even if a person record named "admin" existed, it would have no associated credentials to exploit.

### 3. Role-Based (Not Identity-Based) Authorization
**Location:** `v3/server/pages.py`

All authorization checks use role membership rather than specific PIDs:

```python
@auth.require({R.pmc_member})  # Role-based
# NOT: if result.uid == 'admin':  # Identity-based (not used)
```

**Impact:** No account receives implicit elevated privileges by name.

### 4. Clean Schema Initialization
**Location:** `v3/schema.sql`

The SQL schema contains only DDL (Data Definition Language) statements with zero INSERT operations:

```sql
-- Pure structural definitions
CREATE TABLE person (...);
CREATE TABLE election (...);
-- No INSERT statements present
```

**Impact:** Fresh installations begin with zero user accounts.

### 5. Real Identity Sourcing
**Location:** `v3/server/bin/asf-load-ldap.py`

Person records originate from verified ASF LDAP directory entries:

```python
results = client.search_s(LDAP_DN, ldap.SCOPE_SUBTREE, 'uid=*', attrlist=None)
for r in results:
    uid = entry.uid[0].decode('utf-8')
    visname = entry.cn[0].decode('utf-8')
    email = entry['asf-committer-email'][0].decode('utf-8')
    pdb.add_person(uid, visname, email)
```

**Impact:** No synthetic "admin" or "root" accounts can be injected through identity synchronization.

## Architecture Observations

### Zero Local Accounts Design Pattern

The application implements a "zero local accounts" architecture that is inherently resistant to default account vulnerabilities:

1. **Authentication Layer:** Fully externalized to ASF OAuth
2. **Identity Storage:** Person table stores metadata only (no credentials)
3. **Authorization Model:** Role-based rather than identity-based
4. **Data Sources:** LDAP synchronization (real identities) or validated YAML imports

This design eliminates the entire class of "default account" vulnerabilities by never managing credentials locally.

### Person Table as Identity Registry

The `person` table functions as a registry of known identities for election management purposes, not as an authentication database. This separation of concerns prevents credential-related vulnerabilities.

### CLI Tools Assessment

Administrative CLI tools (`v3/server/bin/`) are server-access-only utilities that:
- Do not create accounts with credentials
- Operate on existing person records
- Require direct server access (excluded from web attack surface)

**Note:** The `_REMAP` dictionary in `create-election.py` is a deployment-time PID remapping utility, not a default account creation mechanism.
