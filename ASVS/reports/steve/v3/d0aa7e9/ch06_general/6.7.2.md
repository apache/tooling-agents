### Summary

After thorough analysis of the codebase, **this application does not implement a cryptographic device-based challenge-response authentication mechanism**. Authentication is handled externally via ASF OAuth (`asfquart.auth`). The cryptographic operations present (salts, vote tokens, encryption) are for vote integrity and confidentiality — not for device-based authentication challenge-response protocols. Therefore, ASVS 6.7.2 has limited direct applicability.

However, all nonce-like and salt values used in the application's cryptographic operations **meet or exceed** the 64-bit minimum specified by this requirement. No findings at MEDIUM severity or above were identified.

## Findings

**No findings at MEDIUM severity or above.**

ASVS 6.7.2 specifically targets challenge nonces in cryptographic authentication mechanisms (smart cards, FIDO keys). This application:

1. **Does not implement cryptographic device-based authentication** — authentication is via external ASF OAuth.
2. **Does not use a challenge-response protocol** — vote tokens are computed server-side deterministically from stored secrets, not as challenge-response exchanges with a client cryptographic device.
3. **All cryptographic salt/nonce values exceed 64 bits** — the minimum nonce-like value in any security-relevant context is 128 bits (16-byte salts).
4. **Statistical uniqueness is ensured** — all salts are generated via `secrets.token_bytes()` (CSPRNG), providing cryptographic-quality randomness. Per-voter-per-issue salts in `mayvote` ensure distinct derivation paths.

---

## Positive Security Patterns

### 1. CSPRNG for All Random Values
**Location**: `crypto.py:gen_salt()` (L36), `create_id()` (L105), `shuffle()` (L97)

All random value generation uses Python's `secrets` module, which draws from the OS CSPRNG. No use of `random` module for security-sensitive operations.

### 2. Generous Salt Sizes (128-bit)
**Location**: `crypto.py` L23 (`SALT_LEN = 16`)

All salts are 128 bits, double the 64-bit minimum required by ASVS 6.7.2. This provides substantial collision resistance (~2^64 expected attempts for birthday collision).

### 3. Per-Voter-Per-Issue Salt Uniqueness
**Location**: `election.py:add_salts()` (L127-140)

Each voter-issue pair in the `mayvote` table receives an independently generated 128-bit salt. This ensures:
- Vote tokens for different voter-issue pairs cannot be correlated
- Encryption keys are unique even if the same person votes on multiple issues

### 4. Schema-Level Enforcement
**Location**: `schema.sql` — CHECK constraints on `salt`, `opened_key`, `vote_token`

The database schema enforces minimum byte-length constraints on all cryptographic fields, providing defense-in-depth against programming errors that might store truncated values.

### 5. Strong Key Derivation Chain
**Location**: `crypto.py` (L39-86)

The key derivation chain uses industry-standard algorithms:
- BLAKE2b for pre-hashing (64-byte digest)
- Argon2d with 64MB memory cost for key derivation (resistant to GPU/ASIC attacks)
- HKDF-SHA256 for final key stretching

Each step uses independent, CSPRNG-generated salts.

### 6. Authenticated Encryption
**Location**: `crypto.py:create_vote()` (L64-67)

Fernet provides authenticated encryption (AES-128-CBC + HMAC-SHA256), with the library managing its own 128-bit IV per encryption operation. This prevents both tampering and nonce reuse at the encryption layer.

## Architecture Observations

### 1. Authentication Boundary
The application correctly delegates authentication to an external system (ASF OAuth). The cryptographic operations in `crypto.py` serve data protection (confidentiality, integrity) rather than authentication. If FIDO/smart card authentication were ever added, the challenge nonce generation would need to be implemented in the authentication layer, not in `crypto.py`.

### 2. Nonce Lifecycle
Salt values are generated once per election opening (`election.salt`) and once per voter-issue pair (`mayvote.salt`). They are stored persistently and reused for deterministic vote token computation. This is appropriate for the data protection use case but differs from the ephemeral challenge-nonce model in ASVS 6.7.2's target scenario.

### 3. Collision Handling for IDs
`create_id()` generates 40-bit random identifiers with database-level collision retry (`sqlite3.IntegrityError` catch in `Election.create()` L300 and `add_issue()` L170). While 40 bits is below the 64-bit threshold for nonces, these are not security nonces — they are identifiers for URL unpredictability, where collision retry and database uniqueness constraints are sufficient.

## Recommendations Summary

Since no findings at MEDIUM or above were identified for ASVS 6.7.2, the following are advisory recommendations for defense-in-depth:

### Long-term (Strategic)

**1. Document Non-Applicability of ASVS 6.7.2**

Document in the project's security documentation that challenge-response cryptographic authentication is not implemented. Note that if it were implemented in the future, the existing `gen_salt()` function already produces values exceeding the minimum requirement.

**2. Future Cryptographic Device Authentication Guidance**

If cryptographic device authentication is added in the future, ensure challenge nonces are:
- At least 128 bits (current salt size provides a good template)
- Generated via `secrets.token_bytes()`
- Single-use with server-side tracking to prevent replay
- Time-limited (bound to session or with expiration)

### Short-term (Hardening)

**3. Consider Constant-Time Comparison for Tamper Detection**

**Location**: `election.py:is_tampered()` L265

While not directly related to ASVS 6.7.2, this is relevant to the broader cryptographic authentication chapter. The current `!=` comparison could theoretically leak information about the `opened_key` value through timing side-channels.

**Current code:**
```python
return opened_key != md.opened_key
```

**Recommended:**
```python
import hmac
return not hmac.compare_digest(opened_key, md.opened_key)
```
