**Section:** General Multi-factor authentication requirements

**Section Description:** This section provides general guidance that will be relevant to various different multi-factor authentication methods.

The mechanisms include:

* Lookup Secrets
* Time based One-time Passwords (TOTPs)
* Out-of-Band mechanisms

Lookup secrets are pre-generated lists of secret codes, similar to Transaction Authorization Numbers (TAN), social media recovery codes, or a grid containing a set of random values. This type of authentication mechanism is considered "something you have" because the codes are deliberately not memorable so will need to be stored somewhere.

Time based One-time Passwords (TOTPs) are physical or soft tokens that display a continually changing pseudo-random one-time challenge. This type of authentication mechanism is considered "something you have". Multi-factor TOTPs are similar to single-factor TOTPs, but require a valid PIN code, biometric unlocking, USB insertion or NFC pairing, or some additional value (such as transaction signing calculators) to be entered to create the final One-time Password (OTP).

### Key Findings

The audited codebase delegates primary authentication to the `asfquart` framework but provides **no application-level mechanism** to revoke authentication factors, invalidate sessions, suspend user accounts, or remove voter access from active elections. The election's tamper-detection architecture cryptographically locks the voter roster upon election opening, making individual voter access revocation architecturally impossible without invalidating the entire election.

### HIGH Severity Findings

#### Finding ASVS-656-HIGH-001

**Title:** No mechanism exists to revoke user authentication or invalidate sessions upon credential compromise

**Severity:** HIGH  
**Finding ID:** ASVS-656-HIGH-001  
**Gap Type:** Type A — Entry point with NO control (the control doesn't exist at all)

**Location:**
- **File:** `v3/server/pages.py` (entire file)
- **Functions Reviewed:** `profile_page()`, `settings_page()`, all endpoints

**Related Functions Checked:**
- `profile_page()` (line 560)
- `settings_page()` (line 569)
- No `logout` endpoint exists
- No `revoke-session` endpoint exists
- No `suspend-user` endpoint exists

**Description:**

The `/settings` and `/profile` pages are empty stubs with zero authentication factor management. A comprehensive search confirms no revocation endpoint exists anywhere in the codebase.

**Vulnerable Code:**

```python
# v3/server/pages.py, lines 560-576
@APP.get('/profile')
@asfquart.auth.require  # Bare decorator means just require a valid session
@APP.use_template(TEMPLATES / 'profile.ezt')
async def profile_page():
    result = await basic_info()
    result.title = 'Profile'
    return result

@APP.get('/settings')
@asfquart.auth.require  # Bare decorator means just require a valid session
@APP.use_template(TEMPLATES / 'settings.ezt')
async def settings_page():
    result = await basic_info()
    result.title = 'Settings'
    return result
```

**Comprehensive search confirms:**
- No `POST /logout` or `GET /logout` endpoint
- No `POST /revoke-session` endpoint
- No `POST /revoke-factor` or similar endpoint
- No `POST /suspend-user` admin endpoint
- No session invalidation function anywhere in the codebase
- No user/account management functions beyond `PersonDB.get_person()` (read-only)

**Data Flow:**

```
User authentication (asfquart.auth) 
  → Session created
  → [NO MECHANISM TO DESTROY/REVOKE SESSION]
  → Compromised credentials allow indefinite access
```

**Proof of Concept:**

N/A — This is an absence of security control rather than an exploitable vulnerability. However, the attack scenario is:

1. Attacker steals user's authentication credentials (password, session cookie, SSO token)
2. Attacker authenticates as the victim
3. Victim discovers the compromise
4. Victim has NO way to revoke the compromised factor or invalidate active sessions
5. Attacker continues to have access indefinitely

**Impact:**

If a user's authentication credentials (password, session cookie, SSO token, or any MFA factor) are compromised through theft, phishing, or device loss, there is no way to:

1. Revoke the compromised authentication factor
2. Force-invalidate active sessions
3. Suspend the user's account
4. Prevent the attacker from casting votes on behalf of the compromised user

In a voting system, this means a stolen authentication factor could be used to cast fraudulent votes with no recourse during an active election.

**Remediation:**

```python
# Add to pages.py

@APP.post('/logout')
@asfquart.auth.require
async def logout_endpoint():
    """Destroy the current session."""
    session = await asfquart.session.read()
    if session:
        await asfquart.session.destroy()
        _LOGGER.info(f'User[U:{session["uid"]}] logged out')
    return quart.redirect('/', code=303)

@APP.post('/revoke-all-sessions')
@asfquart.auth.require
async def revoke_all_sessions_endpoint():
    """Revoke all active sessions for the current user."""
    result = await basic_info()
    # Invalidate all sessions for this user
    await asfquart.session.revoke_all(result.uid)
    _LOGGER.info(f'User[U:{result.uid}] revoked all sessions')
    await flash_success('All sessions have been revoked.')
    return quart.redirect('/login', code=303)

# Admin endpoint for suspending users
@APP.post('/admin/suspend-user')
@asfquart.auth.require({R.pmc_member})
async def suspend_user_endpoint():
    """Suspend a user's access due to compromised credentials."""
    data = await quart.request.get_json()
    target_pid = data.get('pid')
    # Add to suspended users table
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    pdb.suspend_person(target_pid)
    # Revoke all sessions for the target
    await asfquart.session.revoke_all(target_pid)
    _LOGGER.warning(f'Admin suspended user[U:{target_pid}]')
    return '', 204
```

#### Finding ASVS-656-HIGH-002

**Title:** Voter access cannot be revoked during open elections due to cryptographic locking of voter roster

**Severity:** HIGH  
**Finding ID:** ASVS-656-HIGH-002  
**Gap Type:** Type A — No control exists for voter revocation

**Location:**
- **File:** `v3/steve/election.py`
- **Lines:** 85-117 (primary), 210 (comparison), 292-303 (impact)
- **Functions:** `open()`, `gather_election_data()`, `add_salts()`, `is_tampered()`

**Related Functions Checked:**
- `open()` (line 85)
- `gather_election_data()` (line 108)
- `add_salts()` (line 117)
- `is_tampered()` (line 292)
- `add_voter()` (line ~210) — exists
- `remove_voter()` — **does not exist**
- `suspend_voter()` — **does not exist**
- `revoke_voter_access()` — **does not exist**

**Description:**

When an election is opened, the entire voter roster is cryptographically bound into the `opened_key` via tamper detection. Any modification to the voter list causes `is_tampered()` to return `True`, preventing the election from proceeding. The `Election` class has `add_voter()` but no `remove_voter()`, `suspend_voter()`, or `revoke_voter_access()` method.

**Vulnerable Code:**

```python
# v3/steve/election.py, lines 85-93
def open(self, pdb):
    # Double-check the Election is in the editing state.
    assert self.is_editable()
    self.add_salts()  # Locks salts for all voter/issue pairs
    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)
    self.c_open.perform(salt, opened_key, self.eid)
```

The `gather_election_data()` includes ALL voter PIDs and emails:

```python
# v3/steve/election.py, lines 108-113
# Include the PID and EMAIL for each Person who may vote
self.q_voting_persons.perform(self.eid)
pdata = ''.join(row.pid + row.email for row in self.q_voting_persons.fetchall())
return (mdata + idata + pdata).encode()
```

And tamper detection verifies this hash hasn't changed:

```python
# v3/steve/election.py, lines 292-303
def is_tampered(self, pdb):
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)
    return opened_key != md.opened_key  # Any voter change = tampered
```

**Data Flow:**

```
1. Voter roster assembled during gather_election_data()
   → Hashed into opened_key
2. Any modification to voter list
   → is_tampered() returns True
3. Tampered elections cannot proceed
   → Removing a single compromised voter invalidates entire election
4. Therefore: Individual voter revocation is architecturally impossible
```

**Proof of Concept:**

```python
# Scenario: Voter credentials are compromised during an active election

# Attempt 1: Remove the compromised voter
election.remove_voter('compromised_pid')  
# Result: Method does not exist

# Attempt 2: Manually delete from database
# DELETE FROM voters WHERE pid = 'compromised_pid' AND eid = 'election123'
# Result: election.is_tampered() now returns True
# Result: Election is invalidated, all votes lost

# Conclusion: No way to revoke individual voter access
```

**Impact:**

If a voter's authentication is compromised during an active election (e.g., stolen device, phished credentials), an administrator has only two options:

1. **Do nothing**: Allow the attacker to vote as the compromised user
2. **Close the entire election**: Invalidate all votes from all voters

Neither option satisfies ASVS 6.5.6's requirement for factor revocation. In a voting system, this creates a denial-of-service vector: compromising a single voter's credentials could force cancellation of an entire election.

**Remediation:**

Implement a voter suspension mechanism that works with the tamper detection model:

```python
# Add to election.py

def suspend_voter(self, pid: str, reason: str = ''):
    """Suspend a voter's access without modifying the election hash.
    
    Uses a separate suspension table that is checked during vote submission
    but is NOT part of the tamper-detection hash.
    """
    assert self.is_open()
    
    # Record suspension (not in edata hash)
    self.c_suspend_voter.perform(pid, self.eid, reason)
    _LOGGER.warning(
        f'Voter[P:{pid}] suspended from election[E:{self.eid}]: {reason}'
    )

def add_vote(self, pid: str, iid: str, votestring: str):
    """Add VOTESTRING as the (latest) vote by PID for IID."""
    md = self._all_metadata(self.S_OPEN)
    
    # CHECK: Is this voter suspended?
    if self._is_voter_suspended(pid):
        raise VoterSuspended(pid, self.eid)
    
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)

def _is_voter_suspended(self, pid: str) -> bool:
    """Check if a voter is suspended from this election."""
    row = self.q_check_suspension.first_row(pid, self.eid)
    return row is not None
```

Additionally, add the corresponding SQL table and admin endpoint:

```sql
-- Add to schema
CREATE TABLE suspended_voters (
    pid TEXT NOT NULL,
    eid TEXT NOT NULL,
    reason TEXT,
    suspended_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (pid, eid)
);
```

```python
# Add to pages.py
@APP.post('/admin/suspend-voter/<eid>')
@asfquart.auth.require({R.pmc_member})
@load_election
async def suspend_voter_endpoint(election):
    result = await basic_info()
    data = await quart.request.get_json()
    pid = data.get('pid')
    reason = data.get('reason', 'Authentication factor compromised')
    
    election.suspend_voter(pid, reason)
    _LOGGER.warning(
        f'Admin[U:{result.uid}] suspended voter[P:{pid}]'
        f' from election[E:{election.eid}]: {reason}'
    )
    return '', 204
```

### MEDIUM Severity Findings

#### Finding ASVS-656-MED-001

**Title:** State-changing operations (open/close election) use GET requests, enabling unintended execution via CSRF or link following

**Severity:** MEDIUM  
**Finding ID:** ASVS-656-MED-001  
**Gap Type:** Type A — No control (no CSRF protection, wrong HTTP method)

**Location:**
- **File:** `v3/server/pages.py`
- **Lines:** 453-487
- **Functions:** `do_open_endpoint()`, `do_close_endpoint()`

**Related Functions Checked:**
- `do_open_endpoint()` (line 453)
- `do_close_endpoint()` (line 470)
- `basic_info()` (line 88) — sets `csrf_token = 'placeholder'`

**Description:**

State-changing operations that open and close elections use GET requests without CSRF protection. Opening an election cryptographically locks the voter roster (Finding 2), making this a critical operation. The placeholder CSRF token (`basic.csrf_token = 'placeholder'` at line 88) provides no protection.

**Vulnerable Code:**

```python
# v3/server/pages.py, lines 453-469
@APP.get('/do-open/<eid>')        # ← GET for state-changing operation
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz                  # ← Authorization check missing
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)              # ← Irreversible state change
    # ...

@APP.get('/do-close/<eid>')       # ← GET for state-changing operation
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz                  # ← Authorization check missing
    election.close()                # ← Irreversible state change
    # ...
```

**Data Flow:**

```
1. Attacker crafts malicious page with image tag or link
2. Authenticated committer visits attacker's page
3. Browser automatically sends GET request to /do-open/<eid>
4. Election opens, voter roster locks cryptographically
5. No further voter modifications possible (including revocation)
```

**Proof of Concept:**

```html
<!-- Attacker places this on any page visited by an authenticated committer -->
<img src="https://voting-app.example.org/do-open/abc1234567" style="display:none">
<!-- Election is now open, voter list is locked, no revocation possible -->
```

**Relevance to ASVS 6.5.6:**

Opening an election cryptographically locks the voter roster (Finding 2). If this irreversible action can be triggered via CSRF (trivial with GET requests — an `<img>` tag suffices), it permanently prevents voter list modifications, including any revocation. Combined with the placeholder CSRF token (`basic.csrf_token = 'placeholder'` at line 88), any authenticated committer can be tricked into opening an election prematurely.

**Impact:**

An attacker could force-open an election before the administrator has finished configuring the voter list, permanently locking in an incomplete or incorrect voter roster. Combined with the absence of voter revocation (Finding 2), this could include unauthorized voters who cannot later be removed.

**Remediation:**

```python

@asfquart.auth.require({R.pmc_member})  # Restrict to election owners
@load_election

    result = await basic_info()
    
    # Verify ownership

    if md.owner_pid != result.uid:

    # Validate CSRF token
    form = await quart.request.form
    if form.get('csrf_token') != session_csrf_token:
        quart.abort(403)
    
    pdb = steve.persondb.PersonDB.open(DB_FNAME)

    # ...
```

## Architecture Observations

1. **Authentication is entirely delegated** to `asfquart.auth`, meaning the application has no direct control over authentication factor lifecycle. However, the application provides no compensating controls at the application layer (session revocation, account suspension, voter suspension).

2. **The tamper-detection model creates a tension** with ASVS 6.5.6: the cryptographic integrity of the election depends on an immutable voter roster, but security incident response requires the ability to revoke individual access. These goals need to be reconciled through a suspension mechanism that is separate from (and checked in addition to) the tamper-detection hash.

3. **The election state machine (editable → open → closed) is irreversible**, which is appropriate for election integrity but means access control decisions made before opening are permanent. This elevates the importance of having application-level suspension capabilities.
