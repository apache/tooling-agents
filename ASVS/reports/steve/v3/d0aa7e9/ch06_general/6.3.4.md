### Key Issues Identified

The application demonstrates a **single, documented authentication pathway** (ASF OAuth) but suffers from **critical inconsistencies in authorization enforcement** across election management operations. Twelve distinct route handlers contain unimplemented authorization checks, allowing any authenticated committer to perform administrative operations on any election regardless of ownership. Additionally, two state-changing operations use HTTP GET methods, creating CSRF vulnerabilities that bypass intended authentication protections.

### High Severity

#### ASVS-634-HIGH-001: Missing Owner Authorization on Election Management Creates Undocumented Privileged Pathways

**Severity:** HIGH  
**Gap Type:** Type B – Control documented (via `### check authz` comments) but NOT CALLED  
**CWE:** CWE-862 (Missing Authorization)

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** 12 route handlers (see evidence table below)
- **Lines:** 157-175 (`load_election` decorator), 336-489 (affected endpoints)

**Related Functions Checked:**
- `load_election()` — decorator at lines 157-175
- `load_election_issue()` — decorator at lines 177-200
- `do_open_endpoint()` — lines 336-350
- `do_close_endpoint()` — lines 352-366
- `do_vote_endpoint()` — lines 277-301
- `do_add_issue_endpoint()` — lines 375-394
- `do_edit_issue_endpoint()` — lines 396-412
- `do_delete_issue_endpoint()` — lines 414-432
- `do_set_open_at_endpoint()` — lines 262-266
- `do_set_close_at_endpoint()` — lines 268-275
- `manage_page()` — lines 249-258
- `manage_stv_page()` — lines 260-275

**Description:**

Twelve distinct route handlers contain `### check authz` placeholder comments indicating that per-election owner/authorization checks were designed but never implemented. As a result, **any authenticated committer** can perform administrative operations on **any election**, regardless of election ownership. This creates undocumented privileged pathways that violate consistent security enforcement required by ASVS 6.3.4.

The authorization gap exists at the decorator level (`load_election` and `load_election_issue`), which means all routes using these decorators inherit the vulnerability.

**Evidence – Affected Endpoints:**

|-------|----------|------|-----------|------------|-------------|

| `GET /do-close/<eid>` | `do_close_endpoint` | 355 | Close any election | `R.committer` | ❌ |
| `POST /do-vote/<eid>` | `do_vote_endpoint` | 277 | Vote in any election | `R.committer` | ❌ |
| `POST /do-add-issue/<eid>` | `do_add_issue_endpoint` | 375 | Add issues to any election | `R.committer` | ❌ |

| `POST /do-delete-issue/<eid>/<iid>` | `do_delete_issue_endpoint` | 414 | Delete issues from any election | `R.committer` | ❌ |
| `POST /do-set-open_at/<eid>` | `do_set_open_at_endpoint` | 262 | Set open date on any election | `R.committer` | ❌ |
| `POST /do-set-close_at/<eid>` | `do_set_close_at_endpoint` | 268 | Set close date on any election | `R.committer` | ❌ |

**Vulnerable Code:**

```python
# v3/server/pages.py, lines 157-175
def load_election(func):
    """Decorator to load/pass-argument an Election from EID."""
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            _LOGGER.warning(f'No such election: {eid}')
            quart.abort(404)
        
        _LOGGER.debug(f'Loaded: {e}')
        ### check authz          # <-- PLACEHOLDER: NEVER IMPLEMENTED
        return await func(e)
    
    return loader
```

```python
# v3/server/pages.py, lines 336-350
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz              # <-- PLACEHOLDER: NEVER IMPLEMENTED
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)           # <-- Opens ANY election without ownership verification
    
    return quart.redirect(f'/manage/{election.id}')
```

**Data Flow:**

1. **Authentication:** Attacker authenticates as any ASF committer → passes `@asfquart.auth.require({R.committer})`
2. **Election Loading:** Attacker sends `GET /do-open/{target_eid}` → `load_election` decorator loads election by EID
3. **Missing Authorization:** No ownership check performed (placeholder comment exists but no code)
4. **Unauthorized Operation:** `election.open(pdb)` executes → election state changed despite attacker not being the owner
5. **Result:** Attacker successfully manipulates election belonging to another user

**Proof of Concept:**

```http
# Authenticated committer (not election owner) closes another user's election
GET /do-close/abc123def456 HTTP/1.1
Host: steve.apache.org
Cookie: session=<valid_committer_session>

# Response: 302 redirect to /manage/abc123def456
# Election state changed from "open" to "closed" without authorization
```

**Impact:**

- **Election Integrity Compromise:** Any committer can manipulate election lifecycle (open/close) for all elections
- **Data Tampering:** Any committer can modify election issues (add/edit/delete) across all elections
- **Privilege Escalation:** Committers gain PMC-level election management capabilities without proper authorization
- **Inconsistent Security Model:** Election creation requires `R.pmc_member` but all subsequent management requires only `R.committer` with no ownership verification
- **Audit Trail Corruption:** Unauthorized changes may not be properly attributed to the actual perpetrator

**Remediation:**

```python
# v3/server/pages.py — Updated load_election decorator
def load_election(func):
    """Decorator to load/pass-argument an Election from EID."""
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            _LOGGER.warning(f'No such election: {eid}')
            quart.abort(404)
        
        _LOGGER.debug(f'Loaded: {e}')
        
        # VERIFY the current user is authorized for this election
        result = await basic_info()
        md = e.get_metadata()
        
        if md.owner_pid != result.uid:
            _LOGGER.warning(
                f'User[U:{result.uid}] unauthorized for election[E:{eid}] '
                f'owned by [U:{md.owner_pid}]'
            )
            quart.abort(403)
        
        return await func(e)
    
    return loader
```

**Alternative Remediation (Role-Based):**

If elections can be managed by PMC members collectively rather than individual owners:

#### ASVS-634-HIGH-002: State-Changing Operations via GET Requests Bypass Authentication Pathway Protections

**Severity:** HIGH  
**Gap Type:** Type A – Entry points with no control (no CSRF protection, no POST requirement)  
**CWE:** CWE-352 (Cross-Site Request Forgery)

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `do_open_endpoint()`, `do_close_endpoint()`
- **Lines:** 336-366

**Related Functions Checked:**
- `do_open_endpoint()` — lines 336-350
- `do_close_endpoint()` — lines 352-366
- Template confirmation handlers — `manage.ezt` (line ~236), `manage-stv.ezt`

**Description:**

Two state-changing endpoints that permanently alter election state use HTTP GET instead of POST. Since GET requests are linkable, prefetchable, and embeddable (e.g., via `<img>` tags), an attacker can trigger irreversible election state changes by tricking any committer into visiting a page containing the crafted URL. This vulnerability is compounded by the missing owner authorization (ASVS-634-HIGH-001) — the attacker need not target the election owner, as any committer's session suffices.

This violates ASVS 6.3.4 by creating an undocumented attack pathway that bypasses the intended authentication controls through CSRF exploitation.

**Vulnerable Code:**

```python
# v3/server/pages.py, lines 336-350
@APP.get('/do-open/<eid>')          # <-- GET for irreversible state change
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)              # <-- Irreversible state change
    
    return quart.redirect(f'/manage/{election.id}')
```

```python
# v3/server/pages.py, lines 352-366
@APP.get('/do-close/<eid>')         # <-- GET for irreversible state change
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz
    election.close()                # <-- Irreversible state change
    
    return quart.redirect(f'/manage/{election.id}')
```

**Template Confirmation (Client-Side JavaScript):**

**Data Flow:**

1. **Attack Setup:** Attacker crafts malicious page containing `<img>` tag with GET request URL
2. **Victim Visit:** Authenticated committer visits attacker-controlled page
3. **Automatic Request:** Browser automatically sends GET request with victim's session cookie
4. **Authentication Bypass:** Request passes `@asfquart.auth.require({R.committer})` using victim's session
5. **State Change:** Election state permanently altered (opened or closed)
6. **No User Interaction:** Victim never clicked, confirmed, or intended the action

**Proof of Concept:**

**Alternative Attack Vector (Email):**


**Impact:**

- **Premature Election Closure:** Attacker can close an open election before voting completes, disenfranchising voters
- **Unauthorized Election Opening:** Attacker can open an election before administrators are ready, allowing premature voting
- **Irreversible Operations:** Both operations are documented in the UI as **irreversible** and **permanent**
- **No User Consent:** Victim never sees confirmation dialog or approves the action
- **Mass Exploitation:** Single malicious page can target multiple elections simultaneously
- **Combined with ASVS-634-HIGH-001:** Any committer's session can be leveraged against any election (not just elections they own)

**Remediation:**

```python
# v3/server/pages.py — Change to POST with CSRF protection

@APP.post('/do-open/<eid>')   # Change from GET to POST
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    
    # Validate CSRF token
    form = await quart.request.form
    csrf_token = form.get('csrf_token')
    if not csrf_token or csrf_token != quart.session.get('csrf_token'):
        _LOGGER.warning(f'CSRF token validation failed for user {result.uid}')
        quart.abort(403)
    
    # Verify ownership (per ASVS-634-HIGH-001 fix)
    md = election.get_metadata()
    if md.owner_pid != result.uid:
        quart.abort(403)
    
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    
    return quart.redirect(f'/manage/{election.id}')
```

### Medium Severity

#### ASVS-634-MEDIUM-003: Inconsistent Authentication Strength for Election Document Access

**Severity:** MEDIUM  
**Gap Type:** Inconsistent authentication tier across similar resources  
**CWE:** CWE-287 (Improper Authentication)

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `serve_doc()`
- **Lines:** 469-489

**Related Functions Checked:**
- `serve_doc()` — lines 475-489
- `vote_on_page()` — lines 223-247 (comparison endpoint)
- `manage_page()` — lines 249-258 (comparison endpoint)

**Description:**

The `/docs/<iid>/<docname>` route serves election-related documents but requires only a bare session (`@asfquart.auth.require`) while all other election data access routes require committer-level authentication (`@asfquart.auth.require({R.committer})`). This creates an inconsistent authentication strength across the election data access pathway, violating ASVS 6.3.4's requirement for consistent security control enforcement.

While the `mayvote` check provides partial mitigation by verifying voter eligibility, the authentication tier is weaker than equivalent election routes. Additionally, the unimplemented `### verify the propriety of DOCNAME` comment suggests incomplete security hardening.

**Vulnerable Code:**

```python
# v3/server/pages.py, lines 469-489
@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require  # <-- Bare session: any authenticated user (weaker than R.committer)
async def serve_doc(iid, docname):
    """Serve a document associated with an issue."""
    result = await basic_info()
    
    # Verify the user may vote on this issue (and thus, access its docs).
    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)
    
    ### verify the propriety of DOCNAME.   # <-- Not implemented: path traversal risk
    
    # Serve the document from the issue's document directory.
    return await quart.send_from_directory(DOCSDIR / iid, docname)
```

**Comparison of Authentication Levels for Election Resources:**

| Route | Auth Level | Data Type | Line |
|-------|-----------|-----------|------|
| `GET /vote-on/<eid>` | `R.committer` | Election issues/ballot data | 223 |
| `GET /manage/<eid>` | `R.committer` | Election metadata/admin | 249 |
| `GET /manage-stv/<eid>/<iid>` | `R.committer` | STV issue management | 260 |
| `POST /do-vote/<eid>` | `R.committer` | Vote submission | 277 |
| `GET /docs/<iid>/<docname>` | **Bare session** | Election documents | 475 |

**Data Flow:**

1. **Weak Authentication:** Non-committer authenticated user (if any exist) accesses `/docs/<iid>/<docname>`
2. **Session Check Only:** Request passes `@asfquart.auth.require` (no role verification)
3. **Eligibility Check:** `mayvote` check provides partial mitigation (user must be eligible voter)
4. **Document Served:** File served from `DOCSDIR / iid / docname` without path validation
5. **Inconsistency:** Same election's ballot data requires `R.committer`, but documents require only session

**Proof of Concept:**

```http
# Scenario: Non-committer authenticated user (if such accounts exist in ASF auth)
GET /docs/issue_abc123/candidate_statement.pdf HTTP/1.1
Host: steve.apache.org
Cookie: session=<valid_non_committer_session>

# Request succeeds if user is in mayvote list, despite lacking R.committer role
```

**Path Traversal Risk (Secondary Issue):**

```http
# Unvalidated docname parameter
GET /docs/issue_abc123/../../../etc/passwd HTTP/1.1
Host: steve.apache.org
Cookie: session=<valid_session>

# Potential directory traversal if send_from_directory doesn't sanitize
```

**Impact:**

- **Inconsistent Security Posture:** Reduces confidence that authentication is systematically enforced across the application
- **Potential Unauthorized Access:** Non-committer authenticated users (if they exist) could access election documents
- **Path Traversal Risk:** Unvalidated `docname` parameter may allow directory traversal attacks
- **Incomplete Implementation:** Comment `### verify the propriety of DOCNAME` indicates the route is not fully hardened
- **Audit Confusion:** Inconsistent authentication tiers make security reviews more difficult

**Remediation:**

```python
# v3/server/pages.py — Strengthen authentication and add input validation

@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require({R.committer})  # Match other election routes
async def serve_doc(iid, docname):
    """Serve a document associated with an issue."""
    result = await basic_info()
    
    # Validate docname to prevent directory traversal
    if '..' in docname or '/' in docname or '\\' in docname:
        _LOGGER.warning(f'Invalid docname requested: {docname} by user {result.uid}')
        quart.abort(400)
    
    # Verify the user may vote on this issue (and thus, access its docs).
    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        _LOGGER.info(f'User {result.uid} not eligible for issue {iid}')
        quart.abort(404)
    
    # Serve the document from the issue's document directory.
    # send_from_directory provides additional path validation
    return await quart.send_from_directory(DOCSDIR / iid, docname)
```

## Architecture Observations

### 1. Single Authentication Provider (Positive)

The application correctly delegates authentication to ASF OAuth (`oauth.apache.org`), avoiding the complexity and risks of multiple authentication pathways. This design choice eliminates:
- Local password storage and management
- Password reset mechanisms
- Multiple credential types
- Authentication bypass vulnerabilities across providers

However, the **inconsistency exists at the authorization layer**, which ASVS 6.3.4 includes in "security controls... enforced consistently." While authentication is singular and well-implemented, authorization checks are incomplete or missing.

### 2. Decorator-Based Security Model (Positive with Gaps)

The authentication decorators (`@asfquart.auth.require`, `@asfquart.auth.require({R.committer})`, `@asfquart.auth.require({R.pmc_member})`) provide a clear, auditable pattern for authentication enforcement. However:

- Decorators only address **authentication tier** (who can access the route)
- They do not address **authorization** (what specific resources the user can access)
- The `load_election` decorator was designed to handle authorization but was never implemented
- This creates a gap between intended and actual security posture

### 3. Unaudited Module: `api.py` (Risk)

```python
# main.py, lines 40-41
import pages  # noqa: F401
import api  # noqa: F401    # <-- api.py not provided for audit
```

The `api.py` module is imported and registers routes with the application but was not provided for this audit. This represents a **potential undocumented authentication pathway** that cannot be verified for consistency. Without reviewing `api.py`, the audit cannot confirm:

- Whether API routes use the same authentication decorators
- Whether API routes require authentication at all
- Whether API routes perform authorization checks
- Whether API routes are subject to CSRF protections

**Recommendation:** Include `api.py` in security reviews to ensure comprehensive coverage.

### 4. State Machine Relies on Assertions (Risk)

```python
# election.py, lines 394-413
def is_editable(self):
    return not self.is_open() and not self.is_closed()

def is_open(self):
    return self.md.open_at and self.md.open_at <= time.time()

def is_closed(self):
    return self.md.close_at and self.md.close_at <= time.time()

# Usage in election.py:
assert self.is_editable()  # <-- Stripped when running with python -O
```

Election state enforcement uses Python `assert` statements, which are **stripped when running with `-O` optimization flag**. This is not a production-safe guard. If the application is deployed with optimization enabled, state checks are silently removed, allowing operations in invalid states.

**Recommendation:** Replace assertions with explicit exception raising:

```python
def require_editable(self):
    if not self.is_editable():
        raise ElectionStateError(
            f"Election {self.id} is not editable (open={self.is_open()}, closed={self.is_closed()})"
        )
```

### 5. Authorization Model Inconsistency

The application demonstrates an inconsistent authorization model:

| Operation | Required Role | Authorization Check |
|-----------|--------------|---------------------|
| Create election | `R.pmc_member` | ✅ Role-based |
| Manage election | `R.committer` | ❌ No ownership check |
| Vote in election | `R.committer` | ✅ Eligibility check |
| Access documents | Bare session | ✅ Eligibility check |

This creates confusion about the intended security model:
- Is election management intended for owners only? (Suggested by `### check authz` comments)
- Is election management intended for all PMC members? (Suggested by creation requiring `R.pmc_member`)
- Is election management intended for all committers? (Actual implementation)

**Recommendation:** Document and consistently implement the intended authorization model.
