## Findings

**No findings identified.** ASVS 6.5.4 requirements are not applicable to this codebase.

### Applicability Analysis

**ASVS 6.5.4 Scope:**
- Lookup secrets (recovery codes, TANs, pre-generated secret lists)
- Out-of-band authentication codes (SMS, email verification codes)

**Codebase Authentication Architecture:**

The application does not implement:
1. ❌ Recovery code generation
2. ❌ Transaction Authorization Numbers (TANs)
3. ❌ Pre-generated secret code lists
4. ❌ SMS verification codes
5. ❌ Email verification codes
6. ❌ Any OOB authentication mechanisms

Authentication is entirely delegated to external ASF infrastructure:

```python
# v3/server/pages.py, line ~80
s = await asfquart.session.read()
if s:
    basic.update(
        uid=s['uid'],
        name=s['fullname'],
        email=s['email'],
    )
```

All authorization checks use external decorators:
```python
@asfquart.auth.require({R.committer})
@asfquart.auth.require({R.pmc_member})
```

**Conclusion:** MFA implementation, including any lookup secrets or OOB codes, resides in the ASF identity provider, not this application. ASVS 6.5.4 compliance is the responsibility of the external authentication infrastructure.

## Positive Security Patterns

### ✅ Cryptographically Secure Random Number Generation

**Location:** `v3/steve/crypto.py` throughout

All random value generation correctly uses the `secrets` module (CSPRNG):

```python
# Line 36 — Salt generation (128 bits)
def gen_salt() -> bytes:
    return secrets.token_bytes(SALT_LEN)

# Line 108 — ID generation (40 bits)
def create_id():
    return secrets.token_hex(5)

# Line 100 — Secure shuffle
def shuffle(items: list):
    for i in range(len(items) - 1, 0, -1):
        j = secrets.randbelow(i + 1)
        items[i], items[j] = items[j], items[i]
```

**Assessment:** No use of insecure `random` module for security-sensitive operations.

### ✅ Generous Entropy for Identifiers

**Location:** `crypto.py:108` — `create_id()`

Even though election/issue IDs are not authentication codes, they use 40 bits of entropy, which exceeds the ASVS 6.5.4 minimum of 20 bits.

### ✅ Strong Cryptographic Primitives

**Location:** `crypto.py:62-78` — `_hash()`

```python
def _hash(password: str, salt: bytes) -> bytes:
    return argon2.low_level.hash_secret_raw(
        secret=password.encode('utf-8'),
        salt=salt,
        time_cost=TIME_COST,      # 2 iterations
        memory_cost=MEMORY_COST,  # 64 MB
        parallelism=PARALLELISM,  # 1 thread
        hash_len=HASH_LEN,        # 32 bytes
        type=argon2.low_level.Type.ID,
    )
```

**Assessment:** Argon2id with 64MB memory cost provides strong resistance against brute-force attacks.

### ✅ Clean Authentication Boundary

**Location:** Throughout `pages.py`

The application cleanly separates:
- **Authentication** (external ASF infrastructure)
- **Authorization** (internal session + database checks)

This architectural pattern appropriately delegates MFA responsibility to the identity provider.

### ✅ Cryptographic Shuffle for Sensitive Operations

**Location:** `election.py` — tally shuffling

Security-sensitive shuffling operations use `crypto.shuffle()` with CSPRNG, not the standard library's `random.shuffle()`.

**Implication:** Any ASVS requirements related to password storage, MFA, account recovery, or authentication code generation are the responsibility of the ASF identity provider.

The codebase contains no:
- Password reset mechanisms
- Account recovery flows
- Self-registration endpoints
- Email/SMS verification

These would be the typical locations requiring lookup secrets or OOB codes per ASVS 6.5.4.

## Recommendations Summary

### Immediate Actions (0-30 days)

**None required.** No security findings were identified related to ASVS 6.5.4.

### Short-term Actions (1-3 months)

1. **Document Authentication Architecture**
   - Create architectural documentation explicitly stating that authentication, including MFA, is delegated to ASF infrastructure
   - Document the security boundary between the application and identity provider
   - Include this in security review materials for future auditors

2. **Verify External Infrastructure Compliance**
   - Request ASVS 6.5.4 compliance documentation from ASF infrastructure team
   - Confirm that any lookup secrets or OOB codes generated by the identity provider meet the 20-bit entropy minimum
   - Document verification results

### Long-term Actions (3-12 months)

1. **Prepare for Standalone Deployment Scenarios**
   - If the application may ever be deployed outside ASF infrastructure, design recovery code generation using existing `crypto.py` patterns
   - Recommended implementation: `secrets.token_hex(3)` for 24-bit recovery codes (6 hex characters)
   - Document that this would be required for standalone deployments

2. **Consider Consistency Improvements**
   - Replace `random.shuffle()` at `pages.py:266` with `crypto.shuffle()` for consistency
   - This is a code quality improvement, not a security requirement
