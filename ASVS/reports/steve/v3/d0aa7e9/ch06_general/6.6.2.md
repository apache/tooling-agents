## Findings

### Medium Severity

#### ASVS-662-SEV-001: Email-based voter notification lacks request-bound authentication token generation

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with NO control

**Location:**
- **File:** `v3/server/bin/mail-voters.py`
- **Function:** `main()`
- **Lines:** 45–68

**Related Functions:**
- `v3/steve/election.py:get_voters_for_email()` (lines 455–460)
- `v3/steve/election.py:get_metadata()`

**Vulnerable Code:**
```python
# mail-voters.py lines 50-68
for voter in voters:
    # Prepare data dictionary for template rendering
    data = {
        'voter': voter,  # edict with pid, name, email
        'election': metadata,  # edict with eid, title, owner_pid, etc.
    }

    # Render the template to a StringIO stream
    output = io.StringIO()
    template.generate(output, data)
    rendered_body = output.getvalue()

    # Send the email
    asfpy.messaging.mail(
        sender='Apache Election Platform <voter@apache.org>',
        recipient=voter.email,
        subject=f'Vote in Election: {metadata.title}',
        message=rendered_body,
    )
```

**Data Flow:**
1. `get_voters_for_email()` returns **static** voter attributes: `pid`, `name`, `email`
2. `get_metadata()` returns **static** election attributes: `eid`, `title`, `owner_pid`, etc.
3. Template receives only these static identifiers — **no unique, time-limited, or request-bound token is generated**
4. Email is sent over an insecure OOB channel (email) with no cryptographic binding to any authentication session

**Description:**

The email notification system provides template data consisting entirely of static, reusable identifiers. There is no mechanism anywhere in the email sending flow to:

- Generate a unique, time-limited authentication token per voter per notification
- Bind any token to an originating authentication request/session
- Store or validate a nonce for later verification
- Create a cryptographically signed URL that expires

The `get_voters_for_email()` method returns only persistent attributes:
```python
# election.py lines 455-460
def get_voters_for_email(self):
    self.q_voting_persons.perform(self.eid)
    return [
        edict(pid=row.pid, name=row.name, email=row.email)
        for row in self.q_voting_persons.fetchall()
    ]
```

If the operator-provided EZT template constructs any voting link (e.g., `https://vote.example.org/election/[election.eid]?voter=[voter.pid]`), that link would be:
- Identical across multiple invocations of `mail-voters.py` for the same election
- Replayable indefinitely for the election's lifetime
- Not bound to any specific authentication request or session

This violates ASVS 6.6.2 which requires OOB tokens to be "bound to the original authentication request for which they were generated and are not usable for a previous or subsequent one." The ASVS section description also explicitly notes: "Unsafe out-of-band authentication mechanisms such as e-mail [...] are not permitted."

**Proof of Concept:**

An attacker who intercepts or obtains a voter notification email can extract the static voter identifiers. Because no request-bound token exists:

1. Attacker intercepts email to `voter_a@example.org` for election `abc1234567`
2. Email contains link: `https://vote.example.org/election/abc1234567` with voter PID
3. The same link/identifiers remain valid for any future session since nothing binds them to a specific authentication request
4. If re-sent (e.g., running `mail-voters.py` again), identical credentials are generated — no previous tokens are invalidated

**Impact:**
- OOB authentication identifiers sent via email are not bound to specific authentication requests
- Email-delivered credentials remain perpetually valid for the election duration
- No mechanism exists to detect or prevent replay of intercepted email content
- For an L2 voting system, this undermines the integrity of the voter authentication channel

**Remediation:**

Generate a cryptographically random, time-limited, single-use token per voter per email send, store it server-side bound to the authentication context, and validate it on use:

```python
import secrets
import time

def generate_voter_auth_token(election, voter_pid):
    """Generate a request-bound, time-limited OOB token."""
    token = secrets.token_urlsafe(32)
    expiry = int(time.time()) + 3600  # 1 hour validity
    
    # Store token bound to this specific authentication request
    election.c_store_oob_token.perform(
        token,
        voter_pid,
        election.eid,
        expiry,
        False  # not yet used
    )
    return token

def validate_voter_auth_token(election, token, voter_pid):
    """Validate and consume an OOB token."""
    row = election.q_get_oob_token.first_row(token)
    if row is None:
        return False
    if row.pid != voter_pid or row.eid != election.eid:
        return False
    if int(time.time()) > row.expiry:
        return False
    if row.used:
        return False  # Already consumed
    # Mark as used (single-use)
    election.c_mark_oob_token_used.perform(token)
    return True

# In mail-voters.py:
for voter in voters:
    token = generate_voter_auth_token(election, voter.pid)
    data = {
        'voter': voter,
        'election': metadata,
        'auth_token': token,  # Unique, time-limited, request-bound
    }
    # ... render and send
```

## Architecture Observations

1. **Authentication Architecture Gap:** The provided codebase contains a voter email notification system (`mail-voters.py`) but lacks any OOB authentication token infrastructure (generation, storage, validation, expiration). Authentication appears to be delegated to LDAP, but the email channel has no cryptographic binding to authentication sessions.

2. **Email as OOB Channel:** The ASVS section explicitly warns that "unsafe out-of-band authentication mechanisms such as e-mail [...] are not permitted." The system uses email to communicate with voters, and if this serves any authentication purpose, it operates on a channel ASVS designates as unsafe.

3. **Template-Controlled Security Boundary:** The email content is determined by an operator-provided EZT template (`--template` argument). Security-critical decisions about what identifiers appear in emails are delegated to template authors with no guardrails or token injection mechanism provided by the application.

4. **Strong Vote Integrity, Weak Authentication Channel:** The vote encryption pipeline (Argon2 → HKDF → Fernet) is well-constructed with per-voter salts and tamper detection. However, the authentication channel (email notification) lacks comparable cryptographic rigor.

## Recommendations Summary

### Immediate Actions (0-30 days)
1. **Implement request-bound, time-limited OOB tokens** for voter email notifications (ASVS 6.6.2) — **Medium Effort**
2. **Add database table for OOB token storage** with expiry and single-use enforcement (ASVS 6.6.2) — **Medium Effort**

### Short-term Actions (1-3 months)
3. **Provide application control over authentication token injection** into emails, removing this responsibility from templates (ASVS 6.6.2) — **Low Effort**
4. **Implement HMAC-signed URLs** with server-validated expiry timestamps if email must continue to be used (ASVS 6.6.2) — **Medium Effort**

### Long-term Actions (3-12 months)
5. **Migrate from email-based voter notification** to push notifications or TOTP for voter authentication, per ASVS section guidance (ASVS 6.6.2) — **High Effort**
