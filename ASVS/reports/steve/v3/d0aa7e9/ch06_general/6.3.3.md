**Key Findings:**
The application is an election/voting system that relies entirely on single-factor authentication through the ASF Quart authentication framework (`asfquart.auth`). ASVS 6.3.3 at Level 2 mandates that "either a multi-factor authentication mechanism or a combination of single-factor authentication mechanisms must be used in order to access the application."

**Vulnerable Code — Session Reading (no MFA status check):**

```python
# pages.py lines 78-88
async def basic_info():
    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
    # NOTE: No MFA status field read, no MFA verification performed
    basic.csrf_token = 'placeholder'
    return basic
```

**Vulnerable Code — Authentication decorator (single factor only):**

```python
# pages.py line 387-389
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})  # Single-factor only
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    ### check authz  # <-- No MFA check here either
```

**Data Flow:**
```
User credentials (single factor) → asfquart.auth.require → session created → 
vote submission accepted → NO second factor verification at any point
```

**Proof of Concept:**
An attacker who compromises a single authentication factor (e.g., stolen session cookie, compromised ASF password, OAuth token theft) can:
1. Access the voter dashboard and view all available elections
2. Cast votes on behalf of the compromised user via `POST /do-vote/<eid>`
3. If the user is an election owner, open/close elections and modify ballots

**Impact:**
- **Vote integrity compromise:** An attacker with a single stolen credential can cast fraudulent votes in an election system
- **Election manipulation:** Election owners' compromised accounts can be used to open/close elections or modify ballot items
- **No detection mechanism:** Without MFA, there is no additional barrier preventing session hijacking from resulting in complete account takeover within this application
- **Violation of NIST guidance:** NIST SP 800-63B §4.2.1 requires multi-factor authentication at AAL2, which this application should meet for its sensitivity level

**Remediation:**

**Option A — Verify MFA status from identity provider in session:**

```python
async def basic_info():
    basic = edict()
    s = await asfquart.session.read()
    if s:
        # Verify MFA was used for this session
        if not s.get('mfa_verified', False):
            quart.abort(403, 'Multi-factor authentication required')
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
            mfa_verified=s['mfa_verified'],
        )
    else:
        basic.update(uid=None, name=None, email=None)
    return basic
```

**Option B — Add MFA enforcement decorator for sensitive operations:**

```python
def require_mfa(func):
    """Decorator requiring MFA verification for sensitive operations."""
    @functools.wraps(func)
    async def wrapper(*args, **kwargs):
        s = await asfquart.session.read()
        if not s or not s.get('mfa_verified'):
            # Redirect to MFA challenge or reject
            quart.abort(403, 'This operation requires multi-factor authentication')
        return await func(*args, **kwargs)
    return wrapper

@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@require_mfa  # Enforce MFA for vote submission
@load_election
async def do_vote_endpoint(election):
    ...
```

### MEDIUM Severity

#### ASVS-633-MED-001: State-Changing Operations via GET Without MFA or Re-authentication

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry points with no additional authentication control  
**ASVS Requirement:** 6.3.3 (Level 2)

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `do_open_endpoint()`, `do_close_endpoint()`
- **Lines:** 452-469, 473-488

**Related Functions Checked:**
- `do_open_endpoint()` — Election opening (lines 452-469)
- `do_close_endpoint()` — Election closing (lines 473-488)
- `election.open()` — State change implementation (`v3/steve/election.py:85`)
- `election.close()` — State change implementation (`v3/steve/election.py:124`)

**Description:**  
Critical election lifecycle operations (opening and closing elections) are performed via GET requests with only single-factor authentication. These operations fundamentally alter the security posture of an election (opening enables vote acceptance; closing finalizes results). The combination of single-factor auth, GET method for state changes, and no re-authentication creates an elevated risk of unauthorized election manipulation.

**Vulnerable Code:**

```python
# pages.py lines 452-469
@APP.get('/do-open/<eid>')          # GET for state-changing operation
@asfquart.auth.require({R.committer})  # Single-factor only
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz                   # Authorization check NOT implemented
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)               # Election state changed without MFA
    ...

# pages.py lines 473-488
@APP.get('/do-close/<eid>')         # GET for state-changing operation
@asfquart.auth.require({R.committer})  # Single-factor only  
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz                   # Authorization check NOT implemented
    election.close()                 # Election state changed without MFA
    ...
```

**Data Flow:**
```
GET request → single-factor auth check → missing authz check → 
election.open()/close() → state changed without MFA verification
```

**Impact:**
- Any authenticated committer (not just the election owner) can open or close any election due to missing authorization checks (`### check authz` is a TODO)
- Combined with single-factor auth, a compromised committer session can manipulate election lifecycle
- GET requests are more susceptible to CSRF, link sharing, and prefetch attacks

**Remediation:**
1. Change to POST methods with CSRF protection
2. Implement MFA verification (see ASVS-633-HIGH-001)
3. Implement the missing authorization checks to verify the requesting user is the election owner

**Recommended Fix:**

```python
@APP.post('/do-open/<eid>')  # Changed to POST
@asfquart.auth.require({R.committer})
@require_mfa  # Add MFA requirement
@require_csrf  # Add CSRF protection
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    
    # Implement authorization check
    if election.owner != result.uid:
        quart.abort(403, 'Only election owner can open election')
    
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    ...
```

## Positive Security Patterns

The following security controls are implemented correctly and demonstrate security best practices:

1. **Cryptographic vote protection:** The `crypto.py` module implements a well-designed layered encryption scheme using Argon2 → HKDF → Fernet, with per-voter salts preventing vote correlation

2. **Anti-tamper mechanism:** `election.py:is_tampered()` provides integrity verification by recomputing the election key from current data

3. **Secure random generation:** Consistent use of `secrets` module for all security-sensitive random operations (salt generation, ID creation, vote shuffling)

4. **State machine enforcement:** Election state transitions are enforced with assertions (`is_editable()`, `is_open()`, `is_closed()`)

5. **Parameterized queries:** Database operations use the `asfpy.db` wrapper with parameterized queries, preventing SQL injection

6. **Document access control:** `serve_doc()` verifies per-user, per-issue authorization before serving files

## Architecture Observations

### Authentication Architecture

1. **Authentication is entirely delegated:** The application fully delegates authentication to the `asfquart.auth` framework with no application-level authentication controls beyond role checking

2. **No session attribute for MFA status:** The session object (`uid`, `fullname`, `email`) contains no indication of whether MFA was used during authentication

3. **Authorization is incomplete:** Multiple `### check authz` comments throughout `pages.py` indicate authorization checks are planned but not implemented, meaning any committer can manage any election

4. **Single trust boundary:** All authenticated users (committers) share the same trust level for most operations, with only election creation requiring `pmc_member` status

### Trust Model

The application's trust model assumes:
- ASF authentication framework provides sufficient authentication assurance
- All committers are equally trusted for most operations
- Cryptographic controls protect vote confidentiality and integrity
- Authorization enforcement happens at the identity provider level

This trust model creates risk when:
- A single committer credential is compromised
- Authorization checks are not implemented at the application level
- No MFA verification occurs within the application
