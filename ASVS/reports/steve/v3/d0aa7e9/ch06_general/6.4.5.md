## Findings

### Medium Severity

#### Finding ASVS-645-MED-001: No Automated Renewal Notification System for Expiring Authentication Factors

| Attribute | Value |
|-----------|-------|
| **Severity** | **MEDIUM** |
| **Gap Type** | Type A — Entry point with NO control |
| **File** | `v3/server/bin/mail-voters.py` |
| **Function** | `main()` |
| **Lines** | 34–73 |
| **ASVS Requirement** | 6.4.5 — Renewal instructions sent with enough time; automated reminders configured |
| **CWE** | CWE-287: Improper Authentication |

**Description:**

The email infrastructure lacks all critical components required by ASVS 6.4.5 for timely authentication factor renewal notifications. The script operates as a manual, context-agnostic broadcast tool with no awareness of authentication factor expiration timelines, no automated scheduling, and no configurable reminder logic.

**Vulnerable Code:**

```python
def main(eid, template_file):
    election = steve.election.Election(DB_FNAME, eid)
    metadata = election.get_metadata()
    voters = election.get_voters_for_email()  # No expiration-awareness

    # ...
    for voter in voters:
        # No check: is this voter's auth factor approaching expiration?
        # No check: how many days remain before expiration?
        # Template is fully manual — no standard renewal instructions
        asfpy.messaging.mail(
            sender='Apache Election Platform <voter@apache.org>',
            recipient=voter.email,
            subject=f'Vote in Election: {metadata.title}',
            message=rendered_body,
        )
```

**Related Functions Checked:**

- `steve.election.Election()` — No expiration-aware query methods visible
- `election.get_voters_for_email()` — Returns all eligible voters without expiration filtering
- `election.get_metadata()` — Provides election details but no authentication lifecycle data
- CLI argument parser (lines 81-88) — No parameters for expiration thresholds or scheduling

**Data Flow:**

```
Admin manually runs script → ALL voters retrieved indiscriminately →
Emails sent without expiration context → No scheduling → No follow-up →
Voters with approaching expiration receive no targeted renewal instructions
```

**Specific Gaps:**

1. **No expiration filtering**: `get_voters_for_email()` retrieves all eligible voters, not those with soon-to-expire credentials. There is no parameter like `--days-until-expiry` to target at-risk voters.

2. **No automated scheduling**: The script must be invoked manually by an administrator. If the administrator forgets or miscalculates timing, voters receive no renewal notice before their authentication mechanism expires.

3. **No configurable reminder thresholds**: ASVS 6.4.5 explicitly requires "configuring automated reminders if necessary." No reminder configuration, interval, or escalation logic exists.

4. **No standard renewal content**: The email body is entirely determined by a user-supplied `.ezt` template with no enforced inclusion of renewal instructions, expiration dates, or action links.

5. **No state tracking**: No mechanism to record that a reminder was sent, preventing deduplication and escalation logic (e.g., "send again if voter hasn't renewed after 7 days").

**Proof of Concept:**

```bash
# Scenario: Voter's authentication token expires in 3 days
# Administrator must manually:
# 1. Query database for voters expiring soon (no built-in tool)
# 2. Calculate appropriate timing
# 3. Remember to run script
# 4. Hope template includes renewal instructions

# Current invocation (no expiration awareness):
python mail-voters.py --eid 2026-board --template generic-email.ezt

# Result: ALL voters emailed, regardless of expiration status
# Voters with imminent expiration receive no targeted renewal guidance
# If admin forgets to run script, voters locked out without warning
```

**Impact:**

Voters whose authentication factors (tokens, credentials) are approaching expiration will not receive timely renewal instructions unless an administrator manually identifies them and runs the script at the correct time. This creates a window where voters may be locked out of the election system without warning, violating:

- **ASVS 6.4.5** requirement for timely renewal instructions with automated reminders
- **NIST SP 800-63B §6.1.2.3** replacement guidance for authentication factor lifecycle management
- **Operational continuity**: Voters unable to participate due to expired credentials without advance notice

**Risk Scenario:**

1. Voter's authentication token expires on Day 0
2. No automated reminder sent at Day -14 (two weeks before)
3. No automated reminder sent at Day -7 (one week before)
4. No automated reminder sent at Day -3 (urgent notice)
5. Day 0: Voter attempts to authenticate, discovers expired token
6. Voter must contact support for manual renewal (election may have ended)

**Remediation:**

Implement a comprehensive automated renewal notification system:

```python
import datetime

def main(eid, template_file, days_before_expiry=14, dry_run=False):
    """
    Send renewal reminders to voters whose authentication factors
    are approaching expiration.
    
    Args:
        eid: Election ID
        template_file: Path to email template
        days_before_expiry: Send reminders to voters expiring within this window
        dry_run: If True, log actions without sending emails
    """
    election = steve.election.Election(DB_FNAME, eid)
    metadata = election.get_metadata()

    # Retrieve only voters whose auth factors expire within the threshold
    expiry_cutoff = datetime.datetime.utcnow() + datetime.timedelta(days=days_before_expiry)
    voters = election.get_voters_expiring_before(expiry_cutoff)

    if not voters:
        _LOGGER.info(f'No voters with auth factors expiring before {expiry_cutoff} '
                     f'for election {eid}')
        return

    template = ezt.Template(template_file, compress_whitespace=False)

    for voter in voters:
        # Check if reminder already sent for this expiration window
        if election.reminder_already_sent(voter.pid, days_before_expiry):
            _LOGGER.debug(f'Skipping {voter.email}: reminder already sent for '
                          f'{days_before_expiry}-day window')
            continue

        data = {
            'voter': voter,
            'election': metadata,
            'expiry_date': voter.auth_expiry.strftime('%Y-%m-%d'),
            'days_remaining': (voter.auth_expiry - datetime.datetime.utcnow()).days,
            'renewal_url': f'{BASE_URL}/renew/{voter.renewal_token}',
        }
        output = io.StringIO()
        template.generate(output, data)

        if not dry_run:
            asfpy.messaging.mail(
                sender='Apache Election Platform <voter@apache.org>',
                recipient=voter.email,
                subject=f'Action Required: Renew your credentials for {metadata.title}',
                message=output.getvalue(),
            )
            # Record that reminder was sent to prevent duplicates
            election.record_reminder_sent(
                voter.pid,
                reminder_type='auth_renewal',
                days_before_expiry=days_before_expiry
            )

        _LOGGER.info(f'Renewal reminder sent to {voter.email} '
                     f'(expires {voter.auth_expiry}, {data["days_remaining"]} days remaining)')

# Update CLI argument parser:
parser.add_argument('--days-before-expiry', type=int, default=14,
                    help='Send reminders to voters whose auth expires within this many days')
parser.add_argument('--dry-run', action='store_true',
                    help='Log actions without sending emails')
```

**Required Database Schema Changes:**

```sql
-- Add expiration tracking to voters table
ALTER TABLE voters ADD COLUMN auth_expiry TIMESTAMP;
ALTER TABLE voters ADD COLUMN renewal_token VARCHAR(64);

-- Track sent reminders to prevent duplicates
CREATE TABLE renewal_reminders (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    voter_id INTEGER NOT NULL,
    election_id VARCHAR(255) NOT NULL,
    reminder_type VARCHAR(50) NOT NULL,
    days_before_expiry INTEGER NOT NULL,
    sent_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (voter_id) REFERENCES voters(pid),
    UNIQUE(voter_id, election_id, days_before_expiry)
);
```

Integrate with cron/systemd timer for automated daily execution:

## Architecture Observations

### Current State

1. **Single manual touchpoint**: The entire voter notification system depends on a human administrator knowing when to run `mail-voters.py` and with what parameters. This is architecturally incompatible with ASVS 6.4.5's requirement for timely, potentially automated reminders.

2. **No state tracking for reminders**: There is no mechanism to record that a reminder was sent, preventing deduplication and escalation logic (e.g., "send again if voter hasn't renewed after 7 days").

3. **Election model opacity**: The `steve.election.Election` class is imported but not provided. If authentication factor expiration dates are tracked in the database, the email script has no logic to query or utilize them.

4. **Template flexibility without guardrails**: While EZT templates provide flexibility, there are no enforced fields or validation to ensure renewal-critical information (expiration date, renewal URL, days remaining) is included in messages.

### Architectural Gaps for ASVS 6.4.5 Compliance

| Architectural Layer | Current State | Required for Compliance | Gap |
|---------------------|---------------|------------------------|-----|
| **Data Model** | Voter table (schema unknown) | Authentication factor expiration dates, renewal tokens | No expiration tracking visible |
| **Business Logic** | Manual script invocation | Automated expiration queries, reminder scheduling | No automation framework |
| **Notification Layer** | Generic email sending | Expiration-aware targeted notifications | No expiration context |
| **Scheduling Layer** | None | Cron/systemd timer integration | Completely absent |
| **State Management** | None | Reminder tracking, deduplication | No state persistence |
