## Positive Security Patterns

### 1. Consistent Use of `secrets` Module for All Token Generation

**Location:** `v3/steve/crypto.py`, lines ~32, ~117

```python
# crypto.py
def gen_salt() -> bytes:
    return secrets.token_bytes(SALT_LEN)  # 16 bytes of cryptographic randomness

def create_id():
    return secrets.token_hex(5)  # Cryptographically secure IDs
```

**Analysis:** The codebase exclusively uses `secrets` (CSPRNG) for generating random values, never `random` for security-critical operations. This directly satisfies ASVS 7.2.2's requirement for dynamically generated tokens.

### 2. Multi-Stage Key Derivation for Vote Tokens

**Location:** `v3/steve/crypto.py`, lines ~35-67

```python
# crypto.py — Vote token chain:
# 1. BLAKE2b(election_data) → Argon2(hash, salt) → opened_key
# 2. Argon2(opened_key + pid + iid, per-voter salt) → vote_token
# 3. HKDF-SHA256(vote_token, per-voter salt) → encryption key
```

**Analysis:** Vote tokens are derived through multiple cryptographic stages, each incorporating dynamically generated salts. This provides strong token uniqueness per voter, per issue, per election.

### 3. Per-Voter Salt Generation

**Location:** `v3/steve/election.py`, lines ~117-135

```python
# election.py — add_salts()
for mayvote in self.q_all_issues.fetchall():
    salt = crypto.gen_salt()  # Unique 16-byte salt per voter-issue pair
    self.c_salt_mayvote.perform(salt, mayvote.rowid)
```

**Analysis:** Each voter-issue combination receives a unique, cryptographically random salt, ensuring vote tokens are unique even across the same voter's different issue votes.

### 4. Collision Detection with Retry for Generated IDs

**Location:** `v3/steve/election.py`, lines ~408-426

```python
# election.py — create()
while True:
    eid = crypto.create_id()
    try:
        db.c_create.perform(eid, title, owner_pid, authz, open_at, close_at)
        break
    except sqlite3.IntegrityError:
        _LOGGER.debug('EID conflict(!!) ... trying again.')
```

**Analysis:** ID generation includes database-level uniqueness enforcement with automatic retry, ensuring tokens are never reused even in the astronomically unlikely case of collision.

### 5. Cryptographically Secure Shuffle for Vote Anonymization

**Location:** `v3/steve/crypto.py`, lines ~99-105

```python
# crypto.py — shuffle()
for i in range(len(x) - 1, 0, -1):
    j = secrets.randbelow(i + 1)  # CSPRNG for Fisher-Yates
    x[i], x[j] = x[j], x[i]
```

**Analysis:** The custom shuffle implementation uses `secrets.randbelow()` instead of `random.randint()`, ensuring vote ordering cannot leak information about submission order.

**Location:** `v3/server/pages.py`, lines ~56-84


**Analysis:** Session management is fully delegated to the `asfquart` framework rather than implementing custom session token logic, reducing the risk of static/predictable session tokens.

## Architecture Observations

### 1. Session Management Delegation (Unverifiable)

The application delegates all session token generation and validation to the `asfquart.session` and `asfquart.auth` framework. While this is a sound architectural decision (using a tested framework rather than custom implementation), the session token generation mechanism itself is outside the scope of the provided source code. Full ASVS 7.2.2 compliance verification requires auditing the `asfquart` framework's session token generation to confirm it uses cryptographically secure random generation.

### 2. Non-Cryptographic Shuffle for Candidate Display

**Location:** `v3/server/pages.py`, line ~247

```python
# pages.py, line ~247
random.shuffle(issue.candidates)  # Non-cryptographic PRNG
```

While the application provides a cryptographically secure `crypto.shuffle()` using `secrets.randbelow()`, the candidate display ordering in `vote_on_page` uses `random.shuffle()`. This is not a session management concern (ASVS 7.2.2), but represents an inconsistency — ballot ordering bias could theoretically be predicted if the PRNG state is known.

### 3. Election/Issue ID Entropy

Election and issue IDs are generated with 40 bits of entropy (`secrets.token_hex(5)`). While these are resource identifiers rather than session tokens, and authentication is enforced independently, they serve as the primary object reference in URLs. The 40-bit space (≈1 trillion values) is adequate for the intended low-concurrency use case but would be insufficient for high-volume systems.

### 4. Acknowledged Technical Debt — CSRF Token

The CSRF token in `basic_info()` is set to a static placeholder string. Per the known false positive list, this is acknowledged technical debt. When implemented, it should be a cryptographically random value bound to the user's session, aligning with both CSRF protection requirements and the spirit of ASVS 7.2.2's dynamic token generation mandate.

## Recommendations Summary

### Immediate Actions
- **Verify `asfquart` Session Token Generation** — Audit the `asfquart.session` module to confirm session tokens are generated using `secrets.token_bytes()` or equivalent CSPRNG with at least 128 bits of entropy per NIST SP 800-63B. Verify session cookies are configured with `Secure`, `HttpOnly`, and `SameSite` flags.

### Short-term Improvements
- **Replace `random.shuffle` with `crypto.shuffle`** — In `vote_on_page()` (line ~247), replace `random.shuffle(issue.candidates)` with `steve.crypto.shuffle(issue.candidates)` for consistency with the codebase's security posture.

### Long-term Enhancements
- **Implement Dynamic CSRF Tokens** — Replace `basic.csrf_token = 'placeholder'` with `basic.csrf_token = secrets.token_hex(32)`. Store in session and validate on all state-changing POST endpoints (per existing technical debt tracking).
