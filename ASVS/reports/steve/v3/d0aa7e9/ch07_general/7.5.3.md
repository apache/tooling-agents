- **Repository:** apache/steve

| Severity | Count | Description |
|----------|-------|-------------|
| **Critical** | 0 | - |
| **High** | 2 | Vote submission and election state changes lack re-authentication |
| **Medium** | 1 | Election administration operations lack re-authentication |
| **Low** | 0 | - |
| **TOTAL** | 3 | All findings are Type A gaps (entry point with NO control) |

### Key Findings

The voting system implements **zero re-authentication controls** for any sensitive operations. The most critical operations—**vote submission**, **election opening**, and **election closing**—rely solely on existing session authentication with no step-up or secondary verification mechanism. This creates significant risk for session hijacking and abuse, particularly when combined with the use of GET requests for state-changing operations and placeholder CSRF protection.

## Findings

### HIGH: ASVS-753-HIGH-001: No Re-authentication Before Vote Submission

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control  
**File:** `v3/server/pages.py`  
**Function:** `do_vote_endpoint`  
**Lines:** 372–413  
**Related Files:** `v3/steve/election.py`, method `add_vote`, lines 170–185

#### Description

The vote submission endpoint performs no re-authentication or secondary verification before recording votes. An attacker who has hijacked or abused an active committer session can submit or modify votes without any additional challenge.

#### Vulnerable Code

```python
# pages.py:372-413
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})  # Only session-level auth
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    ### check authz  # <-- Authorization check never implemented

    form = edict(await quart.request.form)
    # ... votes extracted and submitted directly ...
    for iid, votestring in votes.items():
        # No re-authentication, no password confirmation, no OTP
        election.add_vote(result.uid, iid, votestring)
```

```python
# election.py:170-185
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."
    md = self._all_metadata(self.S_OPEN)
    ### validate VOTESTRING for ISSUE.TYPE voting  # <-- Also not implemented
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)  # Vote recorded immediately
```

#### Data Flow

- **Source:** Hijacked or abused session (active `{R.committer}` session)
- **Path:** `do_vote_endpoint()` → `election.add_vote()` → encrypted vote stored in database
- **Missing Control:** No re-authentication, password confirmation, or session freshness check before vote recording

#### Proof of Concept

An attacker who has hijacked a committer's session (e.g., via XSS, session fixation, or network interception) can submit votes:

```http
POST /do-vote/<known_eid> HTTP/1.1
Cookie: <hijacked_session_cookie>
Content-Type: application/x-www-form-urlencoded

vote-<iid1>=y&vote-<iid2>=n&csrf_token=placeholder
```

**Note:** The CSRF token is a static placeholder (`'placeholder'` at `pages.py:79`), providing zero protection. The `add_vote` method records the *latest* vote, so an attacker can **overwrite** a legitimate voter's previously cast ballot.

#### Impact

- An attacker with an active session can cast or **modify** votes for any eligible election
- Vote integrity is compromised — the core security property of a voting system
- Since votes are encrypted using per-voter tokens, the modification appears legitimate and is indistinguishable from the real voter's action
- Combined with the lack of CSRF protection, this can be exploited through cross-site request forgery by malicious content on any website the victim visits

#### Remediation

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    
    # Require re-authentication before vote submission
    session = await asfquart.session.read()
    if not await verify_recent_authentication(session, max_age_seconds=300):
        # Redirect to re-authentication page with return URL
        return quart.redirect(
            f'/reauth?return_to=/vote-on/{election.eid}', code=303
        )
    
    # ... proceed with vote submission ...
```

Where `verify_recent_authentication` checks either:
1. A session timestamp indicating when the user last actively authenticated (within 5 minutes)
2. A step-up authentication challenge (password re-entry or TOTP)

### HIGH: ASVS-753-HIGH-002: No Re-authentication Before Irreversible Election State Changes

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control  
**File:** `v3/server/pages.py`  
**Functions:** `do_open_endpoint` (line 436), `do_close_endpoint` (line 455)  
**Related Files:** `v3/steve/election.py`, methods `open` (line 58), `close` (line 100)

#### Description

Election opening and closing operations are irreversible state transitions that require no re-authentication. Additionally, these critical operations use GET requests, making them exploitable through trivial content-loading vectors (images, iframes, link prefetching).

#### Vulnerable Code

```python
# pages.py:436-452 — Opening an election (irreversible)
@APP.get('/do-open/<eid>')  # GET request for state-changing operation
@asfquart.auth.require({R.committer})  # Only session-level auth
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz  # <-- Never implemented
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)  # Irreversible: generates salts, opened_key
    # ...

# pages.py:455-470 — Closing an election (irreversible)
@APP.get('/do-close/<eid>')  # GET request for state-changing operation
@asfquart.auth.require({R.committer})  # Only session-level auth
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz  # <-- Never implemented
    election.close()  # Irreversible: sets closed=1
    # ...
```

**Compounding factor — election.py state enforcement via assertions:**

```python
# election.py:58-63
def open(self, pdb):
    assert self.is_editable()  # Fails silently in production with -O flag
    self.add_salts()
    # ... generates opened_key, transitions to S_OPEN

# election.py:100-103
def close(self):
    assert self.is_open()  # Same concern
    self.c_close.perform(self.eid)
```

#### Data Flow

- **Source:** Hijacked session OR malicious content forcing victim's browser to load URL
- **Path:** GET request → `do_open_endpoint()`/`do_close_endpoint()` → `election.open()`/`election.close()` → permanent state transition in database
- **Missing Control:** No re-authentication, no confirmation, and these use GET requests

#### Proof of Concept

Since these are **GET requests**, an attacker can trigger them through trivial vectors:

```html
<!-- On any website, in an email, or via injected content -->
<img src="https://voting.example.org/do-open/KNOWN_EID" style="display:none">
<img src="https://voting.example.org/do-close/KNOWN_EID" style="display:none">
```

Any authenticated committer who views a page containing the above will have their session used to open or close the election. No CSRF token is required because these are GET endpoints.

#### Impact

- An election can be prematurely opened (before all issues/voters are configured), corrupting election integrity
- An election can be prematurely closed (cutting off voters), disenfranchising eligible voters
- Both transitions are irreversible by design (`election.open()` generates cryptographic material; `election.close()` sets the closed flag permanently)
- The GET method makes this exploitable through any content-loading vector (images, iframes, link prefetching)

#### Remediation

```python
@APP.post('/do-open/<eid>')  # Change to POST
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    
    # Verify ownership
    md = election.get_metadata()
    if md.owner_pid != result.uid:
        quart.abort(403)
    
    # Require re-authentication
    if not await verify_recent_authentication(await asfquart.session.read()):
        return quart.redirect(f'/reauth?return_to=/manage/{election.eid}', code=303)
    
    # Validate CSRF token
    form = await quart.request.form
    validate_csrf_token(form.get('csrf_token'))
    
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    # ...
```

### MEDIUM: ASVS-753-MED-001: No Re-authentication Before Election Administration Operations

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with NO control  
**File:** `v3/server/pages.py`  
**Functions:** Multiple administrative endpoints (lines 416, 472, 497, 518, 360, 366)

#### Description

Election administration operations (create, add/edit/delete issues, set dates) require no re-authentication beyond the initial session. While these operations are restricted to the editable state (before an election opens), they can corrupt election configuration when combined with the ability to open elections.

#### Affected Endpoints

| Endpoint | Function | Line | Operation |
|----------|----------|------|-----------|
| `POST /do-create-election` | `do_create_endpoint` | 416 | Create election |
| `POST /do-add-issue/<eid>` | `do_add_issue_endpoint` | 472 | Add issue to election |
| `POST /do-edit-issue/<eid>/<iid>` | `do_edit_issue_endpoint` | 497 | Modify issue |
| `POST /do-delete-issue/<eid>/<iid>` | `do_delete_issue_endpoint` | 518 | Delete issue |
| `POST /do-set-open_at/<eid>` | `do_set_open_at_endpoint` | 360 | Set election date |
| `POST /do-set-close_at/<eid>` | `do_set_close_at_endpoint` | 366 | Set election date |

#### Vulnerable Code (representative)

```python
# pages.py:416-433
@APP.post('/do-create-election')
@asfquart.auth.require({R.pmc_member})  # Only session-level auth
async def do_create_endpoint():
    result = await basic_info()
    ### check authz  # <-- Never implemented
    form = edict(await quart.request.form)
    election = steve.election.Election.create(DB_FNAME, form.title, result.uid)
    # No re-authentication required

# pages.py:518-534
@APP.post('/do-delete-issue/<eid>/<iid>')
@asfquart.auth.require({R.committer})  # Only session-level auth
@load_election_issue
async def do_delete_issue_endpoint(election, issue):
    result = await basic_info()
    ### check authz  # <-- Never implemented
    election.delete_issue(issue.iid)  # Permanent deletion, no re-auth
```

#### Impact

- A hijacked committer session can create spurious elections, add/modify/delete issues
- While these operations are restricted to the editable state (before an election opens), they can corrupt election configuration
- Combined with the ability to open elections (ASVS-753-HIGH-002), an attacker could configure AND open a manipulated election

#### Remediation

At minimum, administrative operations should require session freshness validation. Ideally, a step-up authentication mechanism should be implemented for all election management operations:

```python
@APP.post('/do-delete-issue/<eid>/<iid>')
@asfquart.auth.require({R.committer})
@load_election_issue
async def do_delete_issue_endpoint(election, issue):
    result = await basic_info()
    
    # Verify ownership
    md = election.get_metadata()
    if md.owner_pid != result.uid:
        quart.abort(403)
    
    # Require recent authentication (15 minutes for admin operations)
    session = await asfquart.session.read()
    if not await verify_recent_authentication(session, max_age_seconds=900):
        return quart.redirect(f'/reauth?return_to=/manage/{election.eid}', code=303)
    
    election.delete_issue(issue.iid)
    # ...
```

## Positive Security Patterns

1. **Vote encryption with per-voter salts** (`election.py:170-185`): Each vote is encrypted using a unique token derived from `opened_key + pid + iid + salt`, preventing vote correlation even if the database is compromised.

2. **Anti-tamper detection** (`election.py:256-270`): The `is_tampered()` method re-derives the `opened_key` from election data and compares it to the stored key, detecting unauthorized modifications.

3. **Voter eligibility verification** (`pages.py:240-245`): The `vote_on_page` checks `q_find_issues` to verify the user has mayvote entries before displaying the ballot.

4. **Candidate randomization** (`pages.py:271`): `random.shuffle(issue.candidates)` prevents positional bias in STV elections.

5. **Comprehensive audit logging**: All sensitive operations log user identity, action, and affected resources (e.g., `f'User[U:{result.uid}] voted on issue[I:{iid}]'`).

6. **Database state machine** (`election.py`): The `_all_metadata(required_state)` pattern provides reliable state enforcement for operations that use it (e.g., `add_vote` properly uses exception-based state checking rather than assertions).

---

## Architecture Observations

1. **No re-authentication framework exists**: There is no mechanism, decorator, utility function, or session attribute for tracking authentication freshness or performing step-up authentication anywhere in the codebase.

2. **Authorization is systematically absent**: Every state-changing endpoint contains `### check authz` as a placeholder comment but no implementation. This means any authenticated committer can manage any election, not just elections they own.

3. **Inconsistent state enforcement**: Some methods use `assert` (disabled with `-O` flag) while `add_vote` uses proper exception-based checking via `_all_metadata(self.S_OPEN)`. The critical path (`add_vote`) is correctly protected, but `open()`, `close()`, `delete()`, `add_issue()`, `edit_issue()`, `delete_issue()` rely on assertions.

4. **GET for state changes**: `/do-open/<eid>` and `/do-close/<eid>` use GET methods for irreversible operations, violating HTTP semantics and making session abuse trivially exploitable through any content-loading vector.

---

## Recommendations Summary

### Immediate (Critical Priority)

1. **Implement re-authentication for vote submission**: Add a session freshness check or step-up authentication challenge before `do_vote_endpoint` processes votes. This is the highest-value target in a voting system.

2. **Implement re-authentication for election lifecycle operations**: Require re-authentication before `do_open_endpoint` and `do_close_endpoint`, and change these to POST requests.

3. **Replace GET with POST for state changes**: Convert `/do-open/<eid>` and `/do-close/<eid>` to POST endpoints to prevent trivial CSRF exploitation.

### Short-term (High Priority)

4. **Implement authorization checks**: Replace all `### check authz` comments with actual ownership verification (e.g., `if md.owner_pid != result.uid: abort(403)`).

5. **Replace assertions with exception-based state checks**: Use `_all_metadata(required_state)` pattern consistently instead of `assert` for all state-dependent operations.

6. **Implement CSRF token validation**: Replace the placeholder CSRF token with a real per-session token validated on all state-changing endpoints.

### Long-term (Medium Priority)

7. **Add session freshness tracking**: Record `last_authenticated_at` in the session and check it before sensitive operations, requiring re-authentication if stale (e.g., >5 minutes for votes, >15 minutes for administration).

8. **Implement step-up authentication framework**: Create a reusable mechanism for password re-entry or TOTP challenges that can be applied to any sensitive endpoint via decorator.

9. **Add confirmation dialogs**: For irreversible operations (open, close, delete), require explicit user confirmation in addition to re-authentication.
