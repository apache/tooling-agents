## Findings

### Medium Severity

#### ASVS-723-MED-001: Insufficient Entropy in Reference Token Generation

| Field | Details |
|-------|---------|
| **Severity** | MEDIUM |
| **Finding ID** | ASVS-723-MED-001 |
| **File** | `v3/steve/crypto.py` |
| **Function** | `create_id()` (lines 115–117) |
| **Gap Type** | Type A — Insufficient entropy for reference token |
| **Related Functions** | `do_vote_endpoint` (pages.py:404), `do_open_endpoint` (pages.py:469), `do_close_endpoint` (pages.py:489), `do_add_issue_endpoint` (pages.py:506), `do_edit_issue_endpoint` (pages.py:529), `do_delete_issue_endpoint` (pages.py:549) |

**Description:**

`create_id()` generates reference tokens (election IDs `eid`, issue IDs `iid`) with only **40 bits of entropy** (`5 bytes × 8 = 40 bits`). ASVS 7.2.3 mandates a minimum of **128 bits** for reference tokens.

While these are resource identifiers rather than session tokens directly, they function as security-critical reference tokens within authenticated sessions.

**Vulnerable Code:**

```python
def create_id():
    "Create a standard ID value."
    # Use 10 hex characters for the ID
    return secrets.token_hex(5)  # 5 bytes
```

**Analysis:**

The insufficient entropy becomes a security issue due to three compounding factors:

1. **Authorization is systematically incomplete** — every state-changing endpoint in `pages.py` contains `### check authz` comments with no actual authorization enforcement:
   - `do_vote_endpoint` (line 404)
   - `do_open_endpoint` (line 469)
   - `do_close_endpoint` (line 489)
   - `do_add_issue_endpoint` (line 506)
   - `do_edit_issue_endpoint` (line 529)
   - `do_delete_issue_endpoint` (line 549)

2. **IDs are exposed in URLs** — patterns like `/manage/<eid>`, `/do-vote/<eid>`, `/do-open/<eid>` expose these identifiers.

3. **Brute-force feasibility** — 40 bits = ~1.1 trillion possible values. An authenticated attacker can enumerate valid election IDs systematically. Without authorization checks, discovering a valid `eid` grants full access.

**Data Flow:**

```
create_id() [40-bit token] → election.eid / issue.iid → URL path parameter → 
load_election decorator → NO authz check → full access to resource
```

**Proof of Concept:**

```python
# Authenticated attacker iterates hex IDs
import requests
session = requests.Session()
# ... authenticate as any committer ...
for i in range(0x0000000000, 0xFFFFFFFFFF):
    eid = f"{i:010x}"
    r = session.get(f"https://target/manage/{eid}")
    if r.status_code == 200:
        print(f"Found election: {eid}")
        # Can now open/close/modify this election
```

**Impact:** 

Any authenticated committer can enumerate and access/manage elections owned by other users, since the 40-bit ID space is the only practical access control boundary (authorization checks are unimplemented).

**Remediation:**

```python
def create_id():
    "Create a standard ID value with sufficient entropy."
    # Use at least 128 bits of entropy (16 bytes → 32 hex chars)
    return secrets.token_hex(16)  # 16 bytes = 128 bits
```

Additionally, implement the authorization checks marked as `### check authz` throughout `pages.py` — entropy alone should not be relied upon for access control (defense in depth).

## Positive Security Patterns

1. **CSPRNG for salt generation**: `gen_salt()` correctly uses `secrets.token_bytes(16)` providing 128 bits of entropy — meets ASVS 7.2.3 minimum.

2. **Cryptographic shuffle implementation**: `steve.crypto.shuffle()` implements Fisher-Yates with `secrets.randbelow()`, demonstrating awareness of CSPRNG requirements.

3. **Strong key derivation**: Argon2 with appropriate parameters (time_cost=2, memory_cost=65536, parallelism=4) and HKDF-SHA256 for key stretching.

4. **Authentication enforcement**: All sensitive endpoints use `@asfquart.auth.require` decorators — no unprotected state-changing operations.

5. **Proper use of `secrets` module**: All cryptographic random values (except `create_id()` entropy level and `random.shuffle` in pages.py) use the `secrets` module.

## Architecture Observations

1. **Session management is externalized**: Session token generation, storage, and validation are handled entirely by `asfquart.session`. The compliance of session tokens with ASVS 7.2.3 (128-bit entropy, CSPRNG, uniqueness) **cannot be verified** from the provided source code. This is the most significant audit gap for this requirement.

2. **Type B Gap — Cryptographic shuffle exists but not used**: `pages.py:279` uses `random.shuffle(issue.candidates)` (Mersenne Twister — predictable PRNG) while `steve.crypto.shuffle()` (CSPRNG-based) exists specifically for this purpose. While not a session management issue, this demonstrates a pattern where security controls are defined but not consistently applied.

   ```python
   # pages.py line 279 — INSECURE
   random.shuffle(issue.candidates)
   
   # Should use:
   steve.crypto.shuffle(issue.candidates)
   ```

3. **Authorization gap compounds token entropy issues**: The systematic absence of authorization checks (every `### check authz` comment is unimplemented) means that resource reference tokens (`eid`, `iid`) become the de facto access control mechanism — making their insufficient entropy more impactful.

## Recommendations Summary

### Immediate Actions (Priority 1)

1. **Audit `asfquart.session` token generation** — Session token entropy cannot be verified from provided code; this is the primary control for ASVS 7.2.3 compliance.

2. **Increase `create_id()` entropy to ≥128 bits** — Change `secrets.token_hex(5)` to `secrets.token_hex(16)` to meet minimum entropy requirements.

### Short-term Actions (Priority 2)

3. **Implement authorization checks** — All `### check authz` placeholders must be implemented so that resource security doesn't depend solely on ID entropy.

4. **Replace `random.shuffle` with `steve.crypto.shuffle`** — Use the CSPRNG-based shuffle that already exists in the codebase for election-sensitive randomization (pages.py:279).

### Long-term Actions

5. **Establish security control usage standards** — Create guidelines ensuring cryptographic functions from `steve.crypto` are consistently used across the codebase.

6. **Implement automated security testing** — Add tests to verify entropy requirements and authorization enforcement across all endpoints.
