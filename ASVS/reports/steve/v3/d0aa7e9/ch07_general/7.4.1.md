# ASVS 7.4.1 Security Audit Report

## Executive Summary

**Repository:** apache/steve  
**Audit Date:** Apr 22, 2026  
**Auditor:** Tooling Agents  
**ASVS Requirement:** 7.4.1 — Verify that when session termination is triggered (such as logout or expiration), the application disallows any further use of the session. For reference tokens or stateful sessions, this means invalidating the session data at the application backend. Applications using self-contained tokens will need a solution such as maintaining a list of terminated tokens, disallowing tokens produced before a per-user date and time or rotating a per-user signing key.

**Key Finding:** The application delegates authentication to the ASF OAuth SSO provider and session management to the `asfquart` framework. However, the application code contains **no logout endpoint** and **no session invalidation logic** anywhere in the codebase. This means users have no mechanism to terminate their sessions, and the application never ensures backend session data is destroyed upon any termination trigger.

## Findings

### High Severity

#### ASVS-741-HIGH-001: No Session Termination (Logout) Endpoint Exists

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control. The entire application lacks a logout mechanism.  
**CWE:** CWE-613: Insufficient Session Expiration

**Location:**
- **File:** `v3/server/pages.py`
- **Scope:** All routes (lines 1–508)
- **Related Functions:** `basic_info()` (lines 76-92), all route handlers

**ASVS 7.4.1 Violation:**

The requirement states: *"Verify that when session termination is triggered (such as logout or expiration), the application disallows any further use of the session. For reference tokens or stateful sessions, this means invalidating the session data at the application backend."*

**Violation Details:**
- No logout mechanism exists for users to trigger session termination
- No session invalidation code exists anywhere in the codebase
- Sessions can only end through framework-level timeout (configuration not visible) or browser closure
- Compromised sessions remain valid indefinitely with no user-initiated revocation mechanism

**Evidence — Session Read Without Destruction:**

`v3/server/pages.py`, lines 76–92, `basic_info()`:
```python
async def basic_info():
    # ...
    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
    else:
        basic.update(uid=None, name=None, email=None)
    # ...
    # NOTE: Session is only READ, never destroyed, invalidated, or cleared
```

**Data Flow Analysis:**

```
1. User Authentication
   └─> ASF OAuth Provider
       └─> asfquart creates server-side session
           └─> Session ID stored in cookie

2. Request Processing (all routes)
   └─> Cookie sent with request
       └─> asfquart.session.read() retrieves session data
           └─> User identified and authorized
               └─> Response generated

3. Session Termination (MISSING)
   └─> ❌ No logout route exists
       └─> ❌ User cannot trigger termination
           └─> ❌ No backend invalidation code
               └─> ⚠️ Session remains valid until timeout/expiry
```

**Proof of Concept:**

1. **Authenticate** as any committer via ASF OAuth at `/voter`
2. **Capture** the session cookie value (e.g., using browser DevTools)
3. **Search** for logout mechanism:
   - Check navigation UI — no logout link
   - Check `/logout` route — 404 Not Found
   - Check `/signout`, `/do-logout` — 404 Not Found
   - Review all routes in source — none perform session termination
4. **Close** the browser tab
5. **Reopen** browser and navigate to `/voter` with same cookie
6. **Result:** Session still active, user automatically authenticated

**Attack Scenario — Session Hijacking Without Revocation:**

```
Attacker's Perspective:
1. Obtain victim's session cookie via:
   - XSS attack (if cookie not properly secured)
   - Network interception (if not HTTPS-only)
   - Physical access to shared workstation
   - Browser history/cache on shared computer

2. Use stolen session cookie to:
   - Access victim's voting interface
   - Cast votes as the victim
   - Modify existing votes
   - Access administrative functions (if victim is PMC member)

3. Victim's options for defense:
   - ❌ Cannot logout to invalidate session
   - ❌ Cannot revoke session from profile/settings
   - ❌ Cannot force re-authentication
   - ⚠️ Must wait for timeout (duration unknown)
```

**Impact:**

| Impact Category | Severity | Description |
|----------------|----------|-------------|
| **Confidentiality** | High | Stolen sessions grant unauthorized access to voting records and user data |
| **Integrity** | High | Attackers with stolen sessions can cast/modify votes on behalf of legitimate users |
| **Availability** | Medium | Legitimate users cannot terminate compromised sessions |
| **Compliance** | High | Direct violation of ASVS 7.4.1 Level 1 requirement |
| **User Control** | High | Users have no control over their own session lifecycle |

**Specific Risks for Voting System:**
- **Shared Workstation Risk:** Users in offices/conferences cannot securely terminate sessions
- **Vote Manipulation:** Stolen sessions allow unauthorized vote casting with no revocation mechanism
- **Accountability:** If a session is compromised, determining legitimate vs. malicious votes becomes impossible
- **Trust Erosion:** Voters cannot be confident their sessions are secure if they cannot terminate them

**Remediation:**

**Immediate (Required):**

```python
# Add to v3/server/pages.py

@APP.get('/logout')
@asfquart.auth.require  # Require valid session to logout
async def logout():
    """
    Terminate the current user session.
    Implements ASVS 7.4.1 session termination requirement.
    """
    # Get user info for logging before destroying session
    session_data = await asfquart.session.read()
    user_id = session_data.get('uid', 'unknown') if session_data else 'unknown'
    
    # Destroy the server-side session
    await asfquart.session.destroy()
    
    # Create response that clears the session cookie
    response = quart.redirect('/')
    response.delete_cookie(
        'session',  # Adjust cookie name to match asfquart configuration
        path='/',
        domain=None,  # Adjust if using specific domain
        secure=True,  # Ensure secure flag
        httponly=True,  # Ensure httponly flag
        samesite='Lax'  # Adjust based on requirements
    )
    
    # Audit log
    _LOGGER.info(f'User {user_id} logged out successfully')
    
    return response

# Alternative: Logout with SSO provider redirect
@APP.get('/logout-sso')
@asfquart.auth.require
async def logout_sso():
    """
    Terminate session and redirect to SSO provider logout.
    Ensures SSO provider also terminates the session.
    """
    session_data = await asfquart.session.read()
    user_id = session_data.get('uid', 'unknown') if session_data else 'unknown'
    
    await asfquart.session.destroy()
    
    # Redirect to ASF OAuth logout with return URL
    sso_logout_url = (
        'https://oauth.apache.org/logout'
        '?redirect_uri=https://steve.apache.org/'
    )
    
    response = quart.redirect(sso_logout_url)
    response.delete_cookie('session', path='/', secure=True, httponly=True)
    
    _LOGGER.info(f'User {user_id} logged out with SSO redirect')
    
    return response
```

**If `asfquart.session.destroy()` is not available:**

```python
# Implement manual session invalidation
async def logout():
    session_data = await asfquart.session.read()
    
    if session_data:
        # Clear all session data
        for key in list(session_data.keys()):
            del session_data[key]
        
        # Force session write with empty data
        await asfquart.session.write(session_data)
    
    # Rotate session ID (if supported)
    await asfquart.session.regenerate()
    
    response = quart.redirect('/')
    response.set_cookie(
        'session',
        value='',
        expires=0,  # Expire immediately
        max_age=0,
        path='/',
        secure=True,
        httponly=True,
        samesite='Lax'
    )
    
    return response
```

## Architecture Observations

### 1. SSO Delegation Model

**Current Implementation:**
- Application uses ASF OAuth for authentication
- `asfquart` framework handles session management
- Session data stored server-side (stateful model)

**ASVS 7.4.1 Applicability:**

Per the requirement's commentary: *"Session termination may be handled either by the application itself or by the SSO provider if the SSO provider is handling session management instead of the application."*

**Analysis:**
- While ASF OAuth handles *authentication*, the application maintains its own *session*
- The SSO provider's session lifecycle is separate from the application's session
- Therefore, the application MUST implement its own session termination
- Logging out of the application should ideally also trigger SSO logout (or vice versa)

**Recommendation:** Implement both local logout (`/logout`) and SSO-integrated logout (`/logout-sso`) to ensure complete session termination across both layers.

### 2. Stateful Session Architecture

**Advantages for Session Termination:**
- Server-side session storage enables immediate invalidation
- No need for token revocation lists or key rotation (as required for stateless tokens)
- Backend can delete session data to instantly prevent reuse
- Framework likely provides built-in invalidation mechanisms

**Current Gap:**
- Infrastructure supports proper termination (stateful sessions)
- Code never invokes termination functionality
- Simple fix: call existing framework methods

**Contrast with Stateless Tokens:**

If the application used self-contained JWT tokens instead:
- Would need to maintain revocation list (blocklist)
- Would need per-user token generation timestamps
- Would need key rotation mechanisms
- Much more complex to implement

**Conclusion:** The stateful session architecture is correct for this use case; only the implementation is missing.

### 3. Session Configuration Visibility

**Observations:**
- No session timeout configuration visible in provided code
- No idle timeout settings found
- No absolute session lifetime defined
- Cookie security flags not explicitly set in code

**Implications:**
- Relying on `asfquart` defaults (not verified)
- Timeout behavior unknown
- Session lifecycle not under explicit application control

**Recommendation:** Make session configuration explicit in application code or configuration files to ensure timeouts are appropriate for a voting system.

### 4. Multi-Layer Session Management

**Layers Identified:**

```
┌─────────────────────────────────────┐
│   ASF OAuth Provider Session        │  ← SSO layer
│   (authentication session)          │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│   asfquart Framework Session        │  ← Framework layer
│   (application session)             │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│   Application Logic                 │  ← Application layer
│   (voting operations)               │
└─────────────────────────────────────┘
```

**Current State:**
- OAuth layer: Handles initial authentication
- Framework layer: Maintains session state
- Application layer: Reads session, never terminates

**Ideal State:**
- All layers should coordinate on session termination
- Logout should cascade: Application → Framework → SSO
- Or: SSO logout should trigger application session cleanup

### Configuration Recommendations

**Suggested Session Timeout Values for Voting System:**

```python
# Recommended configuration
SESSION_CONFIG = {
    # Absolute timeout: Maximum session lifetime regardless of activity
    'PERMANENT_SESSION_LIFETIME': 3600,  # 1 hour
    
    # Idle timeout: Session expires after this period of inactivity
    'SESSION_IDLE_TIMEOUT': 1800,  # 30 minutes
    
    # Cookie security
    'SESSION_COOKIE_SECURE': True,      # HTTPS only
    'SESSION_COOKIE_HTTPONLY': True,    # No JavaScript access
    'SESSION_COOKIE_SAMESITE': 'Lax',   # CSRF protection
    
    # Session refresh
    'SESSION_REFRESH_EACH_REQUEST': False,  # Don't extend on every request
    'SESSION_REFRESH_THRESHOLD': 300,   # Refresh if < 5 min remaining
}
```

**Rationale:**
- **1-hour absolute timeout:** Balances security with user experience for voting operations
- **30-minute idle timeout:** Protects against abandoned sessions on shared workstations
- **No auto-refresh:** Ensures sessions eventually expire even with background activity
- **Secure flags:** Standard security best practices
