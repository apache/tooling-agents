### Key Findings

The application **completely lacks session inactivity timeout mechanisms**. Once authenticated, user sessions persist indefinitely with no idle period enforcement or re-authentication requirements. For a voting system where session hijacking directly enables vote manipulation and election tampering, this represents a critical security gap. Additionally, no logout endpoint exists, preventing users from explicitly terminating their sessions.

## Findings

### High Severity

#### ASVS-731-HIGH-001: No Session Inactivity Timeout Implemented

**Severity:** HIGH

**Location:** Application-wide — `v3/server/pages.py` (all authenticated routes), `v3/server/main.py` (app creation)

**Gap Type:** Type A — Entry points with NO control

**Related Functions Checked:**
- `basic_info()` (pages.py:62-88)
- `create_app()` (main.py:33-46)
- All authenticated route handlers

**Description:**

The application has no session inactivity timeout mechanism. Once a user authenticates, their session persists indefinitely with no check for user activity or idle period. This violates ASVS 7.3.1 which requires an inactivity timeout enforced according to risk analysis.

**Vulnerable Code:**

Session reading without timeout check (`pages.py:62-88`):

```python
async def basic_info():
    """Return base-level EZT template data."""
    basic = edict()
    # ...
    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
    else:
        # No session.
        basic.update(uid=None, name=None, email=None)
    ### generate a real token and store in the session
    basic.csrf_token = 'placeholder'
    return basic
```

The session is read and its contents are used directly. There is no:
- Check for a `last_activity` timestamp in the session
- Comparison against a configured inactivity timeout duration
- Invalidation of stale sessions
- Update of a `last_activity` timestamp upon successful access

App creation without timeout configuration (`main.py:33-46`):

```python
def create_app():
    "Create the asfquart app and its endpoints."
    import asfquart.generics
    asfquart.generics.OAUTH_URL_INIT = (
        'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
    )
    asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    import pages  # noqa: F401
    import api  # noqa: F401
    return app
```

No session lifetime, idle timeout, or cookie expiration is configured during app construction. No `PERMANENT_SESSION_LIFETIME`, `SESSION_COOKIE_MAX_AGE`, or equivalent setting is visible.

Architecture documentation lacks timeout specification (`ARCHITECTURE.md`):

```markdown
- **Middleware**: Includes session management, CSRF protection (placeholder), and flash messages for user feedback.
```

Session management is mentioned but timeout requirements are not documented. ASVS 7.3.1 explicitly requires documented security decisions about timeout durations.

**Data Flow:**

```
User authenticates via ASF OAuth
→ Session created (asfquart.session)
→ No timeout/expiry configured
→ All subsequent requests: basic_info() reads session
→ Session check: exists? yes → grant access (no idle check)
→ Session persists indefinitely until browser close or manual cookie deletion
```

**Proof of Concept:**

```
1. User authenticates and receives session cookie
2. User walks away from computer without closing browser
3. 8 hours later, another person accesses the same browser
4. Navigates to /voter — session still valid, full access granted
5. Navigates to /do-vote/<eid> — can submit votes as the original user
6. No re-authentication is ever requested
```

**Impact:**

1. **Extended attack window for session hijacking:** If an attacker obtains a session token (via XSS, network interception, shared computer), the session remains valid indefinitely, providing unlimited access to vote, manage elections, or create elections on behalf of the victim.
2. **Shared/public computer risk:** A voter who forgets to close their browser on a shared machine leaves their authenticated session accessible to the next user, who could cast votes or modify elections.
3. **Stale session abuse:** Sessions from users who are no longer active (e.g., removed from a committee) may remain valid if no re-authentication is ever forced.
4. **Compliance gap:** The system handles election integrity — a high-trust operation. NIST SP 800-63 (referenced by ASVS) recommends idle timeouts of 30 minutes for AAL1 and 12 hours maximum for re-authentication at AAL2.

**Remediation:**

Implement a session inactivity timeout with last-activity tracking. For a voting system, a 30-minute inactivity timeout is recommended per NIST SP 800-63B.

```python
# In pages.py or as middleware

import time

INACTIVITY_TIMEOUT = 30 * 60  # 30 minutes in seconds

async def check_session_timeout():
    """Middleware/helper to enforce inactivity timeout."""
    s = await asfquart.session.read()
    if not s:
        return  # No session to check

    last_activity = s.get('last_activity', 0)
    now = time.time()

    if now - last_activity > INACTIVITY_TIMEOUT:
        # Session has been idle too long — invalidate and force re-auth
        await asfquart.session.destroy()
        await flash_warning('Your session has expired due to inactivity. Please log in again.')
        quart.abort(quart.redirect('/'))

    # Update last activity timestamp
    s['last_activity'] = now
    await asfquart.session.write(s)


async def basic_info():
    """Return base-level EZT template data."""
    # Enforce inactivity timeout before processing
    await check_session_timeout()

    basic = edict()
    # ... rest of existing logic ...


# Additionally, configure session cookie with max-age as an absolute backstop:
# In create_app() or config.yaml:
# app.config['PERMANENT_SESSION_LIFETIME'] = timedelta(hours=8)
# app.config['SESSION_COOKIE_SECURE'] = True
# app.config['SESSION_COOKIE_HTTPONLY'] = True
# app.config['SESSION_COOKIE_SAMESITE'] = 'Lax'
```

Document the timeout decision in `ARCHITECTURE.md`:

```markdown
### Session Timeout Policy
- **Inactivity timeout:** 30 minutes (based on NIST SP 800-63B AAL1 guidance)
- **Absolute timeout:** 8 hours (maximum session lifetime regardless of activity)
- **Rationale:** Voting operations are high-trust; shorter timeouts minimize
  session hijacking window while allowing reasonable voting session duration.
```

#### ASVS-731-HIGH-002: No Session Logout / Explicit Invalidation Endpoint

**Severity:** HIGH

**Location:** `v3/server/pages.py` — no logout route exists among all defined routes (lines 1–679)

**Gap Type:** Type A — Required entry point completely absent

**Related Functions Checked:**
- All route definitions in `pages.py`
- Session management functions

**Description:**

There is no logout endpoint (`/logout` or equivalent) anywhere in the application. Users have no mechanism to explicitly terminate their session. Combined with the absence of inactivity timeouts (ASVS-731-HIGH-001), this means sessions can only end when the browser deletes the session cookie. This directly impacts the ability to enforce re-authentication.

**Evidence:**

Complete route listing from `pages.py`:

```
GET  /
GET  /voter
GET  /vote-on/<eid>
GET  /admin
GET  /manage/<eid>
GET  /manage-stv/<eid>/<iid>
POST /do-set-open_at/<eid>
POST /do-set-close_at/<eid>
POST /do-vote/<eid>
POST /do-create-election
GET  /do-open/<eid>
GET  /do-close/<eid>
POST /do-add-issue/<eid>
POST /do-edit-issue/<eid>/<iid>
POST /do-delete-issue/<eid>/<iid>
GET  /profile
GET  /settings
GET  /about
GET  /docs/<iid>/<docname>
GET  /static/<path:filename>
GET  /favicon.ico
```

No `/logout`, `/signout`, or session destruction endpoint exists.

**Data Flow:**

```
User completes voting session
→ Wants to log out
→ No logout button/endpoint available
→ Session remains active indefinitely
→ Next user of same browser/computer inherits active session
```

**Proof of Concept:**

```
1. User authenticates and votes on shared/public computer
2. User looks for logout option — none exists
3. User closes browser tab (session cookie may persist if not session-only)
4. Next user opens browser, navigates to application
5. Session still active — full access to previous user's account
```

**Impact:**

Users cannot voluntarily end their sessions. On shared computers, after completing a vote, the voter cannot log out — their session remains active for any subsequent user of that machine. This undermines the entire session lifecycle and makes the missing inactivity timeout even more critical. This is particularly severe in voting contexts where users may access the system from public terminals or shared devices.

**Remediation:**

Add a logout endpoint that properly destroys the session:

```python
@APP.get('/logout')
@asfquart.auth.require  # Must have a session to destroy
async def logout():
    """Destroy the current session and redirect to home."""
    await asfquart.session.destroy()
    await flash_info('You have been logged out.')
    return quart.redirect('/', code=303)
```

Add logout links to all authenticated page templates:

```html
<!-- In base template header -->
{% if uid %}
<div class="user-menu">
    <span>Logged in as: {{ name }}</span>
    <a href="/logout">Logout</a>
</div>
{% endif %}
```

## Positive Security Patterns

1. **Authentication decorators are consistently applied:** All sensitive routes use `@asfquart.auth.require` with appropriate role requirements (`R.committer`, `R.pmc_member`). This provides a solid foundation for access control.

2. **Session-based identity propagation:** `basic_info()` properly reads session data and makes `uid` available to all handlers, ensuring consistent identity context across the application.

3. **Comprehensive audit logging:** All state-changing operations log the user identity and action (e.g., `f'User[U:{result.uid}] opened election[E:{election.eid}]'`), providing excellent forensic capabilities.

4. **Election state machine enforcement:** The `Election` class enforces state transitions via assertions (e.g., `assert self.is_editable()`, `assert self.is_open()`), preventing invalid operations and maintaining data integrity.

5. **Per-document authorization:** `serve_doc()` checks the `mayvote` table before serving issue documents, implementing fine-grained access control beyond simple authentication.

6. **Centralized session access point:** `basic_info()` is called by virtually every authenticated handler. This architectural pattern means adding a timeout check in this single function would provide near-complete coverage.

---

## Architecture Observations

1. **External session management:** Session implementation is delegated to `asfquart.session`, which is not provided for review. The external library *may* have built-in timeout capabilities, but the application does not configure them, and no documentation of such configuration exists.

2. **No middleware layer:** The application lacks a `@before_request` or equivalent middleware where a centralized session timeout check could be implemented. Each route independently calls `basic_info()`, which would be the natural place to add timeout enforcement.

3. **Centralized session access point:** `basic_info()` is called by virtually every authenticated handler. This is a positive architectural pattern because adding a timeout check in this single function would provide near-complete coverage.

4. **Missing security documentation:** `ARCHITECTURE.md` does not document session timeout policy, risk assessment for session lifetime, or re-authentication requirements — all explicitly required by ASVS 7.3.1's reference to "documented security decisions."

5. **High-value operations lack additional protection:** Critical state-changing operations (opening/closing elections) use GET requests with no re-authentication or confirmation step, compounding the session timeout issue.
