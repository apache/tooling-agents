**Key Findings:**
This audit evaluates the codebase against ASVS 7.6.1 — session lifetime and termination behavior between Relying Parties (RPs) and Identity Providers (IdPs), including re-authentication enforcement when maximum IdP authentication age is reached. The analysis reveals significant gaps in session lifetime enforcement and federated re-authentication behavior.

## Findings

### High Severity

#### Finding ASVS-761-HIGH-001

**Title:** No IdP Authentication Event Timestamp Tracking or Maximum Session Lifetime Enforcement

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `basic_info()`
- **Lines:** 44–74

**Related Functions Checked:**
- `asfquart.session.read()` (external dependency)
- All route handlers calling `basic_info()`

**Description:**

The application lacks any mechanism to track when the IdP authentication event occurred or enforce a maximum session lifetime. The `basic_info()` function reads session data but only validates session existence, not authentication recency. This violates NIST SP 800-63C requirements for federation session synchronization.

**Vulnerable Code:**
```python
async def basic_info():
    """Return base-level EZT template data."""
    basic = edict()
    basic.flashes = [
        edict(message=f[1], category=f[0])
        for f in quart.get_flashed_messages(with_categories=True)
    ]

    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
    else:
        basic.update(uid=None, name=None, email=None)

    ### generate a real token and store in the session
    basic.csrf_token = 'placeholder'

    return basic
```

**Gap Type:** Type A — Entry point with NO control

**Data Flow:**
1. User authenticates via IdP (ASF OAuth/SSO) at some point in time `T₀`
2. `asfquart.session.read()` returns session data with identity fields (`uid`, `fullname`, `email`)
3. The session data is checked **only for existence** (`if s:`) — no temporal validation occurs
4. No `auth_time`, `iat`, `max_age`, or session creation timestamp is evaluated
5. The session remains valid indefinitely until the server-side store is cleared or the cookie expires (not controlled in application code)

**Proof of Concept:**
1. User authenticates via IdP at time T₀
2. Hours/days pass; IdP session expires or is revoked
3. User's RP session cookie remains valid in the application
4. User continues accessing all protected endpoints (`/voter`, `/vote-on/<eid>`, `/do-vote/<eid>`) without re-authentication
5. Even after IdP revokes user's committer status, the existing RP session continues to authorize privileged operations

**Impact:**
- Stale sessions can persist beyond the IdP's intended authentication lifetime, violating NIST SP 800-63C requirements for federation session synchronization
- If a user's account is disabled at the IdP, the RP session remains active
- A compromised session token has no expiration boundary within the application, maximizing the attack window
- For a voting system, this means votes could be cast using sessions that the IdP would no longer consider valid

**Remediation:**
```python
import time

# Configuration
MAX_SESSION_AGE_SECONDS = 3600  # 1 hour - align with IdP policy
REAUTHENTICATION_AGE_SECONDS = 900  # 15 minutes for sensitive ops

async def basic_info():
    """Return base-level EZT template data."""
    basic = edict()
    basic.flashes = [
        edict(message=f[1], category=f[0])
        for f in quart.get_flashed_messages(with_categories=True)
    ]

    s = await asfquart.session.read()
    if s:
        # Validate session has not exceeded maximum lifetime
        auth_time = s.get('auth_time', 0)
        if time.time() - auth_time > MAX_SESSION_AGE_SECONDS:
            await asfquart.session.destroy()
            # Redirect to IdP for re-authentication
            return quart.redirect('/login?reason=session_expired')
        
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
            auth_time=auth_time,
        )
    else:
        basic.update(uid=None, name=None, email=None, auth_time=None)

    basic.csrf_token = 'placeholder'
    return basic
```

#### Finding ASVS-761-HIGH-002

**Title:** No Re-authentication Enforced Before High-Impact Voting and Election Management Operations

**Location:**
- **File:** `v3/server/pages.py`
- **Multiple Functions:** See table below

**Affected Endpoints (state-changing operations relying solely on `@asfquart.auth.require`):**

| Endpoint | Function | Line | Operation |
|----------|----------|------|-----------|
| `POST /do-vote/<eid>` | `do_vote_endpoint` | 393 | Submit votes |
| `GET /do-open/<eid>` | `do_open_endpoint` | 460 | Open election for voting |
| `GET /do-close/<eid>` | `do_close_endpoint` | 480 | Close election |
| `POST /do-create-election` | `do_create_endpoint` | 439 | Create new election |
| `POST /do-delete-issue/<eid>/<iid>` | `do_delete_issue_endpoint` | 545 | Delete election issue |

**Related Functions Checked:**
- `@asfquart.auth.require` decorator
- `basic_info()` (called by all affected endpoints)
- `load_election` decorator

**Description:**

Sensitive operations such as vote submission and election management do not enforce re-authentication requirements. The `@asfquart.auth.require` decorator only verifies session existence and role membership, not authentication freshness. Multiple `### check authz` comments indicate planned but unimplemented authorization logic.

**Vulnerable Code (representative — `do_vote_endpoint`):**
```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    ### check authz    # <-- No re-authentication check
    form = edict(await quart.request.form)
    # ... proceeds directly to casting votes
    election.add_vote(result.uid, iid, votestring)
```

**Gap Type:** Type B — Control concept acknowledged (`### check authz` comments) but NOT implemented. The `@asfquart.auth.require` decorator only gates on session existence and role, not on authentication freshness.

**Data Flow:**
1. `@asfquart.auth.require({R.committer})` verifies the session has a committer role
2. `basic_info()` reads session identity without checking authentication age
3. `### check authz` comment indicates planned but unimplemented authorization logic
4. Sensitive operation executes immediately — no check on when the IdP last authenticated this user

**Proof of Concept:**
1. User authenticates at T₀ via IdP
2. At T₀ + 24 hours (well past any reasonable re-authentication window), user submits a vote:
   ```
   POST /do-vote/abc123
   Content-Type: application/x-www-form-urlencoded
   Cookie: session=<stale-session-cookie>
   
   vote-issue1=yes&vote-issue2=no
   ```
3. Vote is accepted and recorded because only session existence is checked, not authentication recency
4. Per NIST SP 800-63C, the RP should have required re-authentication before this FAL-dependent operation

**Impact:**
- Votes can be cast using sessions whose IdP authentication occurred arbitrarily far in the past
- Election lifecycle operations (open/close) can be performed without fresh authentication
- Violates NIST SP 800-63C §5.3 which requires RPs to enforce re-authentication when the time since the last IdP authentication event exceeds the maximum defined interval
- In a voting system context, this significantly increases the window for session hijacking attacks to result in fraudulent votes

**Remediation:**
```python
async def require_fresh_auth(max_age_seconds=900):
    """Require that the IdP authentication event is within max_age_seconds."""
    s = await asfquart.session.read()
    if not s:
        quart.abort(401)
    
    auth_time = s.get('auth_time')
    if auth_time is None or (time.time() - auth_time) > max_age_seconds:
        # Store the original request URL for post-auth redirect
        session_data = await asfquart.session.read()
        session_data['return_to'] = quart.request.url
        await asfquart.session.write(session_data)
        # Redirect to IdP with prompt=login or max_age parameter
        return quart.redirect('/login?prompt=login&reason=reauth_required')
    
    return None  # Auth is fresh

@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    # Enforce fresh authentication for vote submission
    reauth_redirect = await require_fresh_auth(max_age_seconds=900)
    if reauth_redirect:
        return reauth_redirect
    
    result = await basic_info()
    # ... continue with vote processing
```

### Medium Severity

#### Finding ASVS-761-MEDIUM-003

**Title:** No Session Invalidation Mechanism or IdP Session Synchronization

**Location:**
- **File:** `v3/server/pages.py` (entire file — absence of control)

**Related Functions Checked:**
- Complete codebase search for logout endpoints
- Complete codebase search for session destruction calls
- Complete codebase search for back-channel logout handlers

**Description:**

The application has no mechanism to terminate sessions or synchronize session state with the IdP. There is no logout endpoint, no session destruction functionality, and no back-channel logout handler. This means sessions cannot be actively terminated by administrators or the IdP, and revoked users retain access until external mechanisms clear sessions.

**Evidence of Gap:**

A complete review of the codebase reveals:
- **No logout endpoint** exists in `pages.py` or any visible route handler
- **No session destruction function** is called anywhere in application code
- **No back-channel logout handler** (e.g., OIDC Back-Channel Logout) is implemented
- **No front-channel logout** redirect to IdP is present
- **No periodic session validation** against the IdP occurs

**Gap Type:** Type A — No control exists for session termination synchronization

**Comprehensive search results:**

| Pattern Searched | Found | Location |
|-----------------|-------|----------|
| `session.destroy` / `session.delete` / `session.clear` | No | — |
| `logout` route or handler | No | — |
| `backchannel_logout` | No | — |
| Session timeout configuration | No | — |
| IdP token introspection/validation | No | — |
| `max_age` / `auth_time` parameter handling | No | — |

**Proof of Concept:**
1. User A authenticates and receives a session
2. Administrator revokes User A's committer status at the IdP
3. User A's existing RP session remains fully functional
4. User A continues to access `/voter`, `/admin`, `/manage/<eid>`, and can cast votes
5. There is no mechanism to propagate the IdP status change to the RP

**Impact:**
- Sessions cannot be actively terminated by administrators or the IdP
- Revoked users retain access until some external mechanism clears sessions (server restart, store expiry)
- No documented session termination behavior between RP and IdP as required by ASVS 7.6.1
- In a voting context, a user whose eligibility has been revoked could continue participating

## Architecture Observations

1. **Federated Identity Architecture:** The application delegates authentication to ASF's identity infrastructure via `asfquart.auth`. The RP (this application) trusts the IdP session but has no mechanism to validate its ongoing validity or enforce re-authentication windows.

2. **Session-to-IdP Coupling Gap:** The session established by `asfquart.session` appears to be independent of the IdP session lifecycle. Once created, the RP session has no visible binding to the IdP session's validity or expiration.

3. **No Authentication Metadata Storage:** The session stores identity attributes (`uid`, `fullname`, `email`) but does not store authentication event metadata (`auth_time`, `acr`, `amr`) needed for NIST SP 800-63C compliance.

4. **Comment-Driven Security Debt:** Multiple `### check authz` comments throughout `pages.py` indicate that authorization checks beyond basic role requirements are planned but not implemented, creating a false confidence gap.
