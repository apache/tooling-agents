**Section:** Fundamental Session Management Security  
**Section Description:** This section satisfies the essential requirements of secure sessions by verifying that session tokens are securely generated and validated.  
**Chapter:** Session Management  
**Control Objective:** Session management mechanisms allow applications to correlate user and device interactions over time, even when using stateless communication protocols (such as HTTP). Modern applications may use multiple session tokens with distinct characteristics and purposes. A secure session management system is one that prevents attackers from obtaining, utilizing, or otherwise abusing a victim's session. Applications maintaining sessions must ensure that the following high-level session management requirements are met:

* Sessions are unique to each individual and cannot be guessed or shared.
* Sessions are invalidated when no longer required and are timed out during periods of inactivity.

Many of the requirements in this chapter relate to selected [NIST SP 800-63 Digital Identity Guidelines](https://pages.nist.gov/800-63-4/) controls, focusing on common threats and commonly exploited authentication weaknesses.

Note that requirements for specific implementation details of certain session management mechanisms can be found elsewhere:

* HTTP Cookies are a common mechanism for securing session tokens. Specific security requirements for cookies can be found in the "Web Frontend Security" chapter.
* Self-contained tokens are frequently used as a way of maintaining sessions. Specific security requirements can be found in the "Self-contained Tokens" chapter.

**Key Finding:** Session token verification is consistently performed on the backend through the `asfquart` framework across all protected endpoints. However, a critical gap exists where the session-verified user identity is obtained but not used for resource-level authorization checks on election management endpoints, representing a **Type C gap** (control called but result not fully utilized). This allows any authenticated committer to manage any election.

## Findings

### HIGH Severity

#### ASVS-721-HIGH-001: Session-Verified Identity Not Used for Election Ownership Authorization (Type C Gap)

**Severity:** HIGH

**File:** `v3/server/pages.py`  
**Affected Functions:** `do_open_endpoint` (L448), `do_close_endpoint` (L468), `do_add_issue_endpoint` (L486), `do_edit_issue_endpoint` (L513), `do_delete_issue_endpoint` (L537), `do_set_open_at_endpoint` (L369), `do_set_close_at_endpoint` (L375), `manage_page` (L319)

**Related Functions Checked:**
- `load_election` decorator (L164-185)
- `basic_info()` (L58-85)
- `asfquart.auth.require` decorator (framework)
- `asfquart.session.read()` (framework)

**Description:**  
The `@asfquart.auth.require({R.committer})` decorator correctly verifies the session token on the backend and confirms the user holds the committer role. The `basic_info()` function then extracts the verified user identity (`result.uid`) from the server-side session store. However, **this identity is never compared against the election's `owner_pid`** before allowing management operations. The `### check authz` TODO comments at each endpoint confirm the gap is known but unimplemented.

This is a **Type C gap**: the session verification control is called, the user identity result is obtained, but that result is not applied for resource-level authorization.

**Vulnerable Code (representative — same pattern across all 8 endpoints):**

```python
# v3/server/pages.py, line 448
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  # ← Session verified on backend ✅
@load_election
async def do_open_endpoint(election):
    result = await basic_info()  # ← result.uid extracted from session ✅

    ### check authz  # ← MISSING: no check that result.uid == election.owner_pid ❌

    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)  # ← State-changing operation proceeds without ownership check
```

```python
# v3/server/pages.py, line 164 (load_election decorator)
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            ...
        ### check authz  # ← MISSING: no ownership verification ❌
        return await func(e)
    return loader
```

**Data Flow:**
```
HTTP Request with session cookie
  → @asfquart.auth.require({R.committer}) [BACKEND: verifies token, extracts identity]
  → load_election(eid) [loads ANY election by EID from URL]
  → basic_info() → result.uid [reads identity from server-side session]
  → ### check authz [NOT IMPLEMENTED]
  → election.open() / .close() / .add_issue() / etc. [proceeds without ownership check]
```

**Proof of Concept:**
```http
# Attacker is authenticated as committer "mallory"
# Election with eid "abc1234567" is owned by "alice"

# Mallory can close Alice's election:
GET /do-close/abc1234567 HTTP/1.1
Cookie: session=<mallory's valid session token>

# Mallory can add a fraudulent issue:
POST /do-add-issue/abc1234567 HTTP/1.1
Cookie: session=<mallory's valid session token>
Content-Type: application/x-www-form-urlencoded

title=Fraudulent+Issue&description=Injected+by+attacker

# Mallory can delete an existing issue:
POST /do-delete-issue/abc1234567/def9876543 HTTP/1.1
Cookie: session=<mallory's valid session token>
```

**Impact:**
- Any authenticated committer can open, close, or manipulate any election regardless of ownership
- Election integrity is compromised: issues can be added, modified, or deleted by unauthorized users
- Premature election closure causes denial of service for legitimate voters
- The `opened_key` tamper-detection mechanism becomes ineffective if issues are modified before opening

**Remediation:**

```python
# In load_election decorator or each endpoint, add ownership verification:
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            result = await basic_info()
            result.title = 'Unknown Election'
            result.eid = eid
            raise_404(T_BAD_EID, result)

        return await func(e)
    return loader

# Create a separate decorator for management operations:
def require_election_owner(func):
    @functools.wraps(func)
    async def checker(election, *args, **kwargs):
        result = await basic_info()
        md = election.get_metadata()
        if md.owner_pid != result.uid:
            quart.abort(403, 'Not the election owner')
        return await func(election, *args, **kwargs)
    return checker

# Apply to management endpoints:
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
@require_election_owner  # ← New authorization check
async def do_open_endpoint(election):
    ...
```

### MEDIUM Severity

#### ASVS-721-MEDIUM-001: State-Changing Operations via GET Bypass Session CSRF Protections

**Severity:** MEDIUM

**File:** `v3/server/pages.py`  
**Affected Functions:** `do_open_endpoint` (L448), `do_close_endpoint` (L468)

**Related Functions Checked:**
- `basic_info()` (L84) — CSRF token generation
- `@asfquart.auth.require({R.committer})` — session verification

**Description:**  
Two critical state-changing operations (opening and closing elections) use GET methods. While session tokens are verified on the backend via `@asfquart.auth.require({R.committer})`, GET requests are inherently more vulnerable to cross-site request forgery because they can be triggered by image tags, link prefetching, or redirects without user interaction. Combined with the placeholder CSRF token (`basic.csrf_token = 'placeholder'` at line 84), a verified session can be abused through external trigger mechanisms.

**Vulnerable Code:**
```python
# v3/server/pages.py, line 448
@APP.get('/do-open/<eid>')       # ← GET for state-changing operation
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    ...
    election.open(pdb)  # Irreversible: adds salts, generates opened_key

# v3/server/pages.py, line 468
@APP.get('/do-close/<eid>')      # ← GET for state-changing operation
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    ...
    election.close()  # Irreversible: prevents further voting
```

```python
# v3/server/pages.py, line 84
basic.csrf_token = 'placeholder'  # ← No CSRF protection implemented
```

**Data Flow:**
```
Attacker's malicious page
  → <img src="https://voting.example.org/do-close/abc1234567">
  → Browser automatically sends GET with victim's session cookie
  → @asfquart.auth.require verifies valid session
  → No CSRF token check
  → election.close() executes
```

**Proof of Concept:**
```html
<!-- Attacker hosts this on any website. When a logged-in committer visits,
     their browser automatically sends the GET request with session cookie -->
<img src="https://voting.example.org/do-close/abc1234567" width="0" height="0">
```

**Impact:**
- An attacker can trick an authenticated user into opening or closing an election without their knowledge
- Election opening is irreversible (generates salts and opened_key)
- Election closure prevents all further voting, causing denial of service

**Remediation:**
```python
# Change to POST with CSRF token validation:
@APP.post('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    # Validate CSRF token from form data
    form = await quart.request.form
    validate_csrf_token(form.get('csrf_token'))
    
    result = await basic_info()
    
    ### check authz
    md = election.get_metadata()
    if md.owner_pid != result.uid:
        quart.abort(403, 'Not the election owner')
    
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    return quart.redirect(f'/manage/{election.eid}')

# Implement proper CSRF token generation:
async def basic_info():
    result = await asfquart.session.read()
    basic = BasicInfo()
    basic.uid = result.uid
    basic.csrf_token = secrets.token_urlsafe(32)  # Replace placeholder
    # Store token in session for validation
    await asfquart.session.write({'csrf_token': basic.csrf_token})
    return basic
```

## Positive Security Patterns

1. **Consistent Backend Session Verification:** All 17 protected endpoints use `@asfquart.auth.require` decorators for server-side session token verification. No client-side verification was found.

2. **Server-Side Session Data Extraction:** User identity (`uid`) is always read from the server-side session via `asfquart.session.read()` — never from request parameters, headers, or cookies directly.

3. **Vote Identity Binding:** The `add_vote()` method (election.py L236) correctly derives the vote token from the server-side session identity (`pid`), the issue ID, the election's `opened_key`, and a per-mayvote salt — making it cryptographically impossible to vote as another user.

4. **Election State Machine Enforcement:** State transitions (editable→open→closed) are enforced server-side via assertions in `election.py`, preventing operations like voting on un-opened elections.

5. **Document Access Authorization:** The `serve_doc` endpoint (pages.py L581) performs server-side authorization by checking the `mayvote` table before serving documents.

6. **Role-Based Access Differentiation:** Election creation requires `R.pmc_member` (higher privilege) while other operations require `R.committer`, showing intentional role differentiation.

---

## Architecture Observations

1. **Session Framework Delegation:** Session management is fully delegated to the `asfquart` framework. The application code never directly handles session tokens, cookies, or session storage — this is a positive pattern that centralizes session security in a trusted component.

2. **Two-Layer Security Model:** Authentication (session verification) is enforced at the decorator layer, while authorization (ownership, eligibility) is expected at the handler layer. The decorator layer is complete; the handler layer has significant gaps.

3. **Implicit vs. Explicit Authorization:** The `add_vote()` function provides implicit authorization (fails with AttributeError if user has no mayvote entry) rather than explicit authorization checks. While this prevents unauthorized votes, it creates unclear error handling and could mask issues.

4. **Dual Session Read Pattern:** The `@asfquart.auth.require` decorator verifies the session, and then `basic_info()` reads the session again to extract user data. This is functionally safe (same request lifecycle) but introduces a minor redundancy.
