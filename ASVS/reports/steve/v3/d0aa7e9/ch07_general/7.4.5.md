## Findings

### HIGH Severity

#### Finding ASVS-745-HIGH-001

| Attribute | Detail |
|-----------|--------|
| **Finding ID** | ASVS-745-HIGH-001 |
| **Title** | No administrator capability to terminate user sessions |
| **Severity** | **HIGH** |
| **Gap Type** | Type A — Entry point with NO control |
| **ASVS Requirement** | 7.4.5 (Level 2) |
| **CWE** | CWE-613: Insufficient Session Expiration |

**Location:**

| File | Function | Lines | Issue |
|------|----------|-------|-------|
| `v3/server/pages.py` | `admin_page()` | All routes | No session management endpoints exist |
| `v3/schema.sql` | Database schema | All tables | No session table defined |
| `v3/queries.yaml` | Query definitions | All queries | No session-related queries |
| `v3/steve/election.py` | All methods | All functions | No session management operations |

**Related Functions Checked:**
- ✅ `admin_page()` — Only manages elections, not sessions
- ✅ `voter_page()` — No logout or session management
- ✅ `basic_info()` — Only reads session, doesn't manage it
- ✅ All `/do-*` endpoints — No session termination actions
- ✅ `v3/server/bin/tally.py` — No CLI session tools

**Description:**

The application provides no mechanism for administrators to terminate active sessions, either for an individual user or for all users. This was confirmed by exhaustive review of all route handlers, backend classes, database schema, and CLI tools. Session management is entirely delegated to the external `asfquart` framework with no application-level override capability.

**Vulnerable Code:**

In `v3/server/pages.py`, the admin route only manages elections:

```python
@APP.get('/admin')
@asfquart.auth.require({R.committer})  ### need general solution
@APP.use_template(TEMPLATES / 'admin.ezt')
async def admin_page():
    result = await basic_info()
    result.title = 'Administration'
    # ... only fetches election data, no session management ...
    result.owned = [postprocess_election(e) for e in owned]
    return result
```

**Evidence — No session persistence:**

From `v3/schema.sql`, the complete table list:

```sql
CREATE TABLE election (...);
CREATE TABLE issue (...);
CREATE TABLE person (...);
CREATE TABLE mayvote (...);
CREATE TABLE vote (...);
```

**No session table exists** that would allow server-side session invalidation.

**Evidence — No session-related queries:**

Review of `v3/queries.yaml` shows queries for elections, issues, votes, and voters, but zero queries related to session management, listing, or termination.

**Data Flow Analysis:**

```
Admin detects compromised account
  ↓
❌ No admin endpoint to view active sessions
  ↓
❌ No admin endpoint to terminate specific user session
  ↓
❌ No admin endpoint to terminate all sessions
  ↓
❌ No database table to store/invalidate sessions
  ↓
❌ No CLI tool for session management
  ↓
⚠️ Compromised session remains active until natural expiry
```

**Proof of Concept — Attack Scenario:**

1. **Attacker compromises a committer's session** (via session fixation, XSS, or credential theft)
2. **Attacker casts fraudulent votes:**
   ```http
   POST /do-vote/election-2026-board HTTP/1.1
   Cookie: asfquart_session=<stolen_session_token>
   
   issue_123=candidate_malicious
   ```
3. **Administrator becomes aware of the compromise** and attempts to terminate the session
4. **❌ No mechanism exists** — Administrator cannot:
   - View active sessions for the compromised user
   - Terminate the specific session
   - Force a global session reset
5. **Attacker continues casting votes** until session naturally expires (potentially hours or days)
6. **Fraudulent votes are recorded** and may be irreversible due to vote anonymization

**Impact:**

- **Incident Response Failure:** Administrators cannot respond to account compromises by terminating sessions
- **No Emergency Controls:** No ability to force re-authentication after security policy changes or during active attacks
- **No "Kill Switch":** No emergency "terminate all sessions" capability during a security incident
- **Voting System Risk:** In a voting system, this means fraudulent votes can continue to be cast during an active compromise, potentially affecting election outcomes
- **Compliance Violation:** Direct failure of ASVS 7.4.5 Level 2 requirement

**Remediation:**

**Step 1: Add session storage table**

In `v3/schema.sql`:

```sql
CREATE TABLE user_session (
    session_id  TEXT PRIMARY KEY NOT NULL,
    pid         TEXT NOT NULL,
    created_at  INTEGER NOT NULL DEFAULT (unixepoch('now')),
    last_activity INTEGER NOT NULL DEFAULT (unixepoch('now')),
    expires_at  INTEGER NOT NULL,
    is_active   INTEGER NOT NULL DEFAULT 1,
    ip_address  TEXT,
    user_agent  TEXT,
    FOREIGN KEY (pid) REFERENCES person(pid)
) STRICT;

CREATE INDEX idx_session_pid ON user_session(pid);
CREATE INDEX idx_session_active ON user_session(is_active, expires_at);
```

**Step 2: Add session management queries**

In `v3/queries.yaml`:

```yaml
q_active_sessions:
  sql: |
    SELECT s.session_id, s.pid, p.name, s.created_at, s.last_activity, 
           s.expires_at, s.ip_address, s.user_agent
    FROM user_session s
    JOIN person p ON s.pid = p.pid
    WHERE s.is_active = 1 AND s.expires_at > unixepoch('now')
    ORDER BY s.last_activity DESC

q_user_sessions:
  sql: |
    SELECT session_id, created_at, last_activity, expires_at, ip_address
    FROM user_session
    WHERE pid = :pid AND is_active = 1 AND expires_at > unixepoch('now')
    ORDER BY last_activity DESC

c_terminate_user_sessions:
  sql: |
    UPDATE user_session SET is_active = 0 WHERE pid = :pid

c_terminate_session:
  sql: |
    UPDATE user_session SET is_active = 0 WHERE session_id = :session_id

c_terminate_all_sessions:
  sql: |
    UPDATE user_session SET is_active = 0 WHERE is_active = 1
```

**Step 3: Add admin session management endpoints**

In `v3/server/pages.py`:

```python
@APP.get('/admin/sessions')
@asfquart.auth.require({R.pmc_member})  # Or a dedicated admin role
@APP.use_template(TEMPLATES / 'admin-sessions.ezt')
async def admin_sessions_page():
    """List all active sessions."""
    result = await basic_info()
    result.title = 'Active Sessions'
    db = steve.election.Election.open_database(DB_FNAME)
    
    # Fetch active sessions
    db.q_active_sessions.perform()
    result.sessions = list(db.q_active_sessions.fetchall())
    
    return result


@APP.post('/admin/sessions/terminate/<pid>')
@asfquart.auth.require({R.pmc_member})
async def terminate_user_sessions(pid):
    """Terminate all sessions for a specific user."""
    result = await basic_info()
    db = steve.election.Election.open_database(DB_FNAME)
    
    # Terminate all sessions for user
    db.c_terminate_user_sessions.perform(pid)
    
    _LOGGER.warning(
        f'Admin[U:{result.uid}] terminated all sessions for user[U:{pid}]'
    )
    
    await flash_success(f'All sessions terminated for user: {pid}')
    return quart.redirect('/admin/sessions', code=303)


@APP.post('/admin/sessions/terminate-all')
@asfquart.auth.require({R.pmc_member})
async def terminate_all_sessions():
    """Terminate all active sessions (emergency)."""
    result = await basic_info()
    db = steve.election.Election.open_database(DB_FNAME)
    
    # Emergency: terminate ALL sessions
    db.c_terminate_all_sessions.perform()
    
    _LOGGER.critical(
        f'Admin[U:{result.uid}] terminated ALL active sessions (emergency)'
    )
    
    await flash_warning('All active sessions have been terminated.')
    return quart.redirect('/admin/sessions', code=303)


@APP.post('/admin/sessions/terminate-session/<session_id>')
@asfquart.auth.require({R.pmc_member})
async def terminate_specific_session(session_id):
    """Terminate a specific session by ID."""
    result = await basic_info()
    db = steve.election.Election.open_database(DB_FNAME)
    
    db.c_terminate_session.perform(session_id)
    
    _LOGGER.warning(
        f'Admin[U:{result.uid}] terminated session: {session_id}'
    )
    
    await flash_success(f'Session terminated: {session_id}')
    return quart.redirect('/admin/sessions', code=303)
```

**Step 4: Add session validation middleware**

```python
@APP.before_request
async def validate_session():
    """Validate session is still active in database."""
    session_id = quart.session.get('session_id')
    if session_id:
        db = steve.election.Election.open_database(DB_FNAME)
        db.q_validate_session.perform(session_id)
        session_data = db.q_validate_session.fetchone()
        
        if not session_data or not session_data['is_active']:
            # Session has been terminated by admin
            quart.session.clear()
            await flash_warning('Your session has been terminated by an administrator.')
            return quart.redirect('/', code=303)
        
        # Update last activity
        db.c_update_session_activity.perform(session_id)
```

## Positive Security Patterns

The following security controls are properly implemented:

1. **✅ Consistent Authentication Enforcement**
   - All sensitive routes use `@asfquart.auth.require` decorators with appropriate role requirements
   - Example: `@asfquart.auth.require({R.committer})` on admin routes
   - No authentication bypasses detected

2. **✅ Parameterized SQL Queries**
   - All SQL uses parameterized statements via `asfpy.db`
   - No string concatenation in SQL queries
   - Prevents SQL injection attacks

3. **✅ State Machine Enforcement**
   - Election lifecycle transitions use `assert` checks
   - Methods: `is_editable()`, `is_open()`, `is_closed()`
   - Prevents invalid state transitions

4. **✅ Comprehensive Audit Logging**
   - User actions logged with user IDs: create, open, close, vote, edit, delete
   - Uses `_LOGGER.info()` and `_LOGGER.warning()` appropriately
   - Provides accountability trail

5. **✅ Cryptographic Vote Anonymity**
   - Vote tokens and encryption prevent correlation between voters and votes
   - Protects voter privacy in election system

6. **✅ Role-Based Access Control**
   - Clear role separation: `R.committer`, `R.pmc_member`, `R.voter`
   - Roles enforced at route level

## Architecture Observations

### Session Management Architecture

1. **External Session Dependency**
   - Session management is delegated to the external `asfquart` framework
   - Application has no visibility into session internals
   - No application-level session store or management layer
   - **Risk:** Cannot implement session termination without framework support

2. **No Session Persistence Model**
   - Database schema (`schema.sql`) has no concept of user sessions
   - Tables: `election`, `issue`, `person`, `mayvote`, `vote` — no `user_session`
   - Without server-side session store, selective invalidation is impossible
   - **Gap:** Cannot track or terminate active sessions

3. **SSO Integration Considerations**
   - Authentication is via ASF OAuth (external SSO provider)
   - Even if SSO provider terminates its sessions, application sessions likely remain valid
   - Application needs its own session invalidation layer to complement SSO controls
   - **Risk:** SSO logout may not invalidate application session cookies/tokens

4. **Admin Role Underspecification**
   - `admin_page()` requires `R.committer` role
   - True administrative actions (session termination) should require higher privilege
   - `R.pmc_member` role exists but only used for election creation
   - **Recommendation:** Define dedicated `R.admin` role for session management

### Application Architecture

5. **Stateless Route Design**
   - Routes are designed to be stateless
   - Each request reads session via `basic_info()` but doesn't maintain session state
   - **Implication:** Need to add session validation middleware to check database on each request

6. **Database-Centric Design**
   - All persistent state stored in SQLite database
   - Session table would fit naturally into existing schema
   - Queries defined in `queries.yaml` for consistency
   - **Opportunity:** Session management can follow existing patterns
