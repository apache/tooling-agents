## Findings

### High Severity

#### ASVS-711-HIGH-001: No Documented or Configured Session Inactivity Timeout or Absolute Session Lifetime

| Field | Detail |
|-------|--------|
| **Severity** | HIGH |
| **Finding ID** | ASVS-711-HIGH-001 |
| **Title** | No Documented or Configured Session Inactivity Timeout or Absolute Session Lifetime |
| **File** | `v3/server/pages.py` (lines 44–71), `v3/docs/schema.md` (entire file) |
| **Function** | `basic_info()`, application-wide session management |

**Vulnerable Code:**

In `v3/server/pages.py`, the session is read without any timeout validation:

```python
async def basic_info():
    """Return base-level EZT template data."""
    basic = edict()
    # ...
    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
    else:
        # No session.
        basic.update(
            uid=None,
            name=None,
            email=None,
        )
    ### generate a real token and store in the session
    basic.csrf_token = 'placeholder'
    return basic
```

**Data Flow:**

`asfquart.session.read()` → session data retrieved → **no inactivity timestamp checked** → **no absolute lifetime checked** → session accepted unconditionally → user authorized for sensitive election operations (`do_vote_endpoint`, `do_open_endpoint`, `do_close_endpoint`, `do_create_endpoint`).

**Analysis:**

1. **No inactivity timeout**: The `basic_info()` function and all route handlers that call it read the session but never check when the user last performed an action. There is no `last_activity` timestamp stored in or checked against the session.

2. **No absolute lifetime**: There is no maximum session duration enforced. A session token obtained once could theoretically remain valid indefinitely.

3. **No documentation exists**: The `schema.md` documentation comprehensively covers the database schema but contains zero mention of session management policies, timeouts, or lifecycle. No separate session management documentation is referenced or exists in the provided codebase.

4. **No NIST SP 800-63B analysis**: NIST SP 800-63B Section 7.2 specifies re-authentication requirements including session timeouts (e.g., 30-minute inactivity for AAL2). There is no documentation justifying any deviation from these requirements, despite the application handling sensitive voting operations that arguably require AAL2-level controls.

5. **No risk analysis**: The ASVS section description explicitly states: "A risk analysis with documented security decisions related to session handling must be conducted as a prerequisite to implementation and testing." No such analysis is present.

**Evidence of gap in all authenticated endpoints:**

Every authenticated route delegates session handling to `asfquart.session.read()` without application-level timeout enforcement:

| Route | Action Sensitivity | Timeout Check |
|-------|-------------------|---------------|
| `POST /do-vote/<eid>` (line 385) | **Critical** – casts votes | None |
| `GET /do-open/<eid>` (line 457) | **Critical** – opens election | None |
| `GET /do-close/<eid>` (line 477) | **Critical** – closes election | None |
| `POST /do-create-election` (line 436) | **High** – creates election | None |
| `POST /do-add-issue/<eid>` (line 496) | **High** – modifies election | None |
| `POST /do-delete-issue/<eid>/<iid>` (line 546) | **High** – deletes content | None |

**Impact:**

For a **voting/election system**, indefinite sessions create concrete risks:
- An unattended workstation retains a valid voting session indefinitely, allowing unauthorized vote casting by a passerby.
- Session tokens obtained through any means (network sniffing, XSS, log exposure) remain valid without time bound.
- Without documented timeout policies, neither developers nor auditors can verify the session security posture.
- Non-compliance with NIST SP 800-63B, which the ASVS requirement specifically cross-references.

**Proof of Concept:**

1. User authenticates and receives a session token.
2. User walks away from their workstation (or closes browser but session cookie persists).
3. Hours, days, or weeks later, anyone accessing the same browser/token can:
   - Cast votes: `POST /do-vote/<eid>` 
   - Open/close elections: `GET /do-open/<eid>`, `GET /do-close/<eid>`
   - Create elections: `POST /do-create-election`

No server-side mechanism will reject the stale session.

**Remediation:**

1. **Implement session timeouts** at the application layer:

```python
import time

SESSION_INACTIVITY_TIMEOUT = 900   # 15 minutes (NIST AAL2: ≤30 min)
SESSION_ABSOLUTE_LIFETIME = 43200  # 12 hours

async def basic_info():
    """Return base-level EZT template data."""
    basic = edict()
    
    s = await asfquart.session.read()
    if s:
        now = time.time()
        
        # Check absolute session lifetime
        created_at = s.get('created_at', 0)
        if now - created_at > SESSION_ABSOLUTE_LIFETIME:
            await asfquart.session.destroy()
            quart.abort(401, 'Session expired. Please re-authenticate.')
        
        # Check inactivity timeout
        last_active = s.get('last_active', 0)
        if now - last_active > SESSION_INACTIVITY_TIMEOUT:
            await asfquart.session.destroy()
            quart.abort(401, 'Session timed out due to inactivity.')
        
        # Update last activity timestamp
        s['last_active'] = now
        await asfquart.session.write(s)
        
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
    else:
        basic.update(uid=None, name=None, email=None)
    
    return basic
```

2. **Consider re-authentication for critical operations** (vote casting, election open/close):

```python
async def require_recent_auth(max_age=300):  # 5 minutes
    """Require re-authentication for sensitive operations."""
    s = await asfquart.session.read()
    if not s or time.time() - s.get('auth_time', 0) > max_age:
        quart.abort(401, 'Re-authentication required for this operation.')
```

3. **Create session management documentation** (see Finding ASVS-711-MED-001).

### Medium Severity

#### ASVS-711-MED-001: Absence of Session Management Risk Analysis and Policy Documentation

| Field | Detail |
|-------|--------|
| **Severity** | MEDIUM |
| **Finding ID** | ASVS-711-MED-001 |
| **Title** | Absence of Session Management Risk Analysis and Policy Documentation |
| **File** | `v3/docs/schema.md` (entire file), project-wide |
| **Function** | N/A (documentation gap) |

**Analysis:**

ASVS 7.1.1 explicitly requires documentation that:
- States the session inactivity timeout value
- States the absolute maximum session lifetime
- Justifies these values in combination with other controls
- Includes justification for any deviations from NIST SP 800-63B

The project's only documentation file (`v3/docs/schema.md`) covers the database schema in detail but contains **no mention** of:
- Session management policies or configuration
- Session token storage mechanism (cookie? header?)
- Session timeout values
- SSO interaction considerations (noted as required by the ASVS section description)
- NIST SP 800-63B analysis or deviation justification
- Risk analysis for session handling decisions

The `schema.md` election lifecycle documentation (Editable → Open → Closed) discusses election state transitions but does not address how user sessions interact with these states or whether session policies change based on election phase.

**Impact:**

- The ASVS section description states: "A risk analysis with documented security decisions related to session handling must be conducted as a prerequisite to implementation and testing." Without this documentation, the session management implementation cannot be verified as intentional or appropriate.
- For an election system, the absence of documented session policies means there is no auditable basis for determining whether session handling is adequate for the sensitivity of the operations (voting, election administration).
- New developers or administrators have no reference for expected session behavior.

**Remediation:**

Create a `session-management.md` (or equivalent section in existing docs) containing:

```markdown
# Session Management Policy

## Overview
This document describes session management decisions for the Steve 
voting system, per ASVS 7.1.1 requirements.

## Session Timeout Values
- **Inactivity timeout**: 15 minutes
  - Justification: Voting sessions should be short-lived to prevent 
    unauthorized use of unattended workstations. NIST SP 800-63B 
    Section 7.2 permits up to 30 minutes for AAL2; we use a stricter 
    15-minute value given the sensitivity of vote casting.
- **Absolute session lifetime**: 12 hours
  - Justification: Elections may span workdays. 12 hours allows a 
    full work session while limiting token reuse risk.

## NIST SP 800-63B Compliance
- AAL level: [AAL1/AAL2] — justify based on authentication method
- Re-authentication: Required for vote submission if session is 
  older than 5 minutes since last authentication event
- Deviations: [Document any deviations and justification]

## SSO Interaction
- [Document how SSO session lifetime interacts with application 
  session lifetime]
- [Document session revocation on SSO logout]

## Risk Analysis
- Threat: Unattended workstation during voting period
  - Mitigation: 15-minute inactivity timeout
- Threat: Stolen session token
  - Mitigation: 12-hour absolute lifetime, HTTPS-only cookies
- [Additional threats and mitigations]
```

## Positive Security Patterns

1. **Authentication enforcement on routes**: All sensitive endpoints use `@asfquart.auth.require({R.committer})` or `@asfquart.auth.require({R.pmc_member})` decorators consistently.

2. **Session-based user identification**: `basic_info()` reads `uid` from the session consistently, providing centralized identity extraction.

3. **Per-document authorization**: `serve_doc()` performs mayvote-based authorization checking (line 598–602).

4. **Comprehensive audit logging**: Security-relevant actions (vote, open, close, create, add/edit/delete issue) are logged with user identifiers.

5. **Database schema integrity**: `STRICT` mode, foreign key constraints, CHECK constraints on cryptographic field sizes, and the `prevent_open_close_update` trigger demonstrate defense-in-depth at the data layer.

---

## Architecture Observations

1. **Session management is entirely delegated to `asfquart.session`**: The application code contains no session lifecycle management logic. Whether appropriate timeouts exist depends entirely on the framework's default configuration, which is not visible, configured, or documented in the application.

2. **No distinction between session sensitivity levels**: The same session mechanism is used for browsing (low sensitivity) and voting/election management (high sensitivity). NIST SP 800-63B would recommend re-authentication or step-up authentication for the latter.

3. **State-changing operations via GET**: `do_open_endpoint` and `do_close_endpoint` use `@APP.get()` for state-changing operations. While this is more of a CSRF concern, it interacts with session management because a valid session + predictable URL = unauthorized state change from any context that can trigger a GET request.
