## Findings

### High Severity

#### ASVS-1054-HIGH-001: No Visible ID Token Audience (`aud`) Claim Validation

**Severity:** HIGH

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `create_app()`
- **Lines:** 35–48

**Related Functions Checked:**
- `asfquart.session.read()` in `pages.py:79`
- `basic_info()` in `pages.py:79-86`
- All authentication decorators in `pages.py`

**Description:**

The application explicitly overrides OIDC behavior and implements custom OAuth URL configuration without any visible ID Token audience (`aud`) claim validation. The code contains a deliberate comment "Avoid OIDC" and manually configures OAuth endpoints, bypassing standard OIDC token validation mechanisms. No `client_id` is configured or referenced anywhere in the application code, and no `aud` validation logic exists.

This represents either a Type A gap (no control exists) or Type B gap (control may exist in the `asfquart` framework but is potentially circumvented by the explicit OIDC override).

**Vulnerable Code:**

```python
def create_app():
    "Create the asfquart app and its endpoints."

    ### is this really needed right now?
    # Avoid OIDC
    import asfquart.generics

    asfquart.generics.OAUTH_URL_INIT = (
        'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
    )
    asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'

    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
```

Session consumption without audience validation in `pages.py`:

```python
async def basic_info():
    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
```

**Data Flow:**

1. User authenticates via `oauth.apache.org` → Authorization code returned
2. `asfquart` framework exchanges code at `OAUTH_URL_CALLBACK` → Token(s) received
3. Session populated with identity (`uid`, `fullname`, `email`) from token data
4. **Missing step**: No visible validation that the token's `aud` claim equals this application's `client_id`
5. Session data consumed by `asfquart.session.read()` in `pages.py:79` for all protected endpoints

**Proof of Concept:**

If `oauth.apache.org` serves multiple relying parties (e.g., other ASF services), an attacker who obtains an ID Token or identity assertion issued for a *different* Apache service could potentially replay it against this application's callback endpoint. Without `aud` validation, the application cannot distinguish tokens meant for it from tokens meant for other clients — a classic token confusion/substitution attack.

Attack scenario:
1. Attacker authenticates to a different ASF service (Client B) via `oauth.apache.org`
2. Attacker captures the identity token/code intended for Client B
3. Attacker presents this token to STeVe's callback endpoint
4. Without `aud` == `client_id` validation, STeVe accepts the token and creates a valid session

**Impact:**

- **Authentication bypass** through token confusion if the OAuth/OIDC provider serves multiple clients
- An attacker with a token intended for another Apache service could gain authenticated access to the election system
- Undermines the integrity guarantees of the authentication layer that all election operations depend on
- Violates ASVS 10.5.4 Level 2 requirement for ID Token audience validation

**Remediation:**

If using OIDC (recommended for authentication use cases), explicitly validate the `aud` claim:

```python
def create_app():
    import asfquart.generics

    # Configure OIDC properly instead of avoiding it
    # Ensure the framework validates ID Token claims
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    # Set the client_id that must match the 'aud' claim
    app.config['OIDC_CLIENT_ID'] = 'steve-voting-app'
    
    # If asfquart doesn't validate aud, add middleware:
    @app.before_request
    async def validate_id_token_audience():
        # This should be handled by the framework, but verify:
        session = await asfquart.session.read()
        if session and 'id_token_aud' in session:
            if session['id_token_aud'] != app.config['OIDC_CLIENT_ID']:
                await asfquart.session.clear()
                quart.abort(401, 'Token audience mismatch')
    
    return app
```

Alternatively, if continuing with the `asfquart` framework:
1. Remove the "Avoid OIDC" override and use proper OIDC endpoints
2. Configure `client_id` in the framework configuration
3. Verify the framework's token handling code validates `aud` == `client_id`
4. Add integration tests that verify tokens with incorrect `aud` values are rejected

## Positive Security Patterns

1. **TLS for LDAP:** `LDAP_URL = 'ldaps://ldap-us.apache.org/'` — Uses LDAPS (encrypted) for LDAP queries
2. **HTTPS OAuth endpoints:** Both `OAUTH_URL_INIT` and `OAUTH_URL_CALLBACK` use HTTPS
3. **Authentication enforcement:** Consistent use of `@asfquart.auth.require` decorators on protected endpoints
4. **State parameter in OAuth:** `OAUTH_URL_INIT` includes `state=%s`, indicating CSRF protection in the OAuth flow
5. **Comprehensive audit logging:** All state-changing operations log the acting user's UID
6. **Role-based access:** Election creation requires `R.pmc_member` (higher privilege than `R.committer`)
7. **Voter eligibility checks:** `vote_on_page` validates mayvote entries before allowing vote access

## Architecture Observations

1. **Framework opacity:** The `asfquart` framework handles the entire OAuth flow (authorization, token exchange, session creation). Without its source code, critical OIDC security controls cannot be verified. This is the single most important gap for this ASVS requirement.

2. **"Avoid OIDC" design decision:** The explicit override from OIDC to custom OAuth suggests a deliberate architectural choice. However, using OAuth alone for authentication (without OIDC's identity layer and its `aud`/`nonce`/`iss` validation) reduces security guarantees against token confusion attacks.

3. **Single-provider assumption:** The application appears to trust `oauth.apache.org` as the sole identity source. If this provider serves multiple relying parties, the lack of audience validation becomes a concrete risk rather than a theoretical one.
