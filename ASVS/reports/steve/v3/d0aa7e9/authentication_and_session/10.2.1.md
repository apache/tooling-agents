The application uses the OAuth authorization code flow through the `asfquart` framework. The authorization URL template includes a `state` parameter placeholder, but PKCE is absent, and state parameter validation cannot be verified from the application code. ## Findings

### High Severity

#### ASVS-1021-HIGH-001

**Finding:** State-Changing Election Operations Exposed via GET Method — Trivially Exploitable CSRF

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `do_open_endpoint()` (line 510), `do_close_endpoint()` (line 532)
- **Lines:** 510-523, 532-545

**Related Functions Checked:**
- `basic_info()` (line 86)
- `load_election` decorator (line 108)
- `@asfquart.auth.require` decorator

**Description:**

Critical election state transitions (opening and closing elections) are exposed via GET endpoints without any CSRF protection. These operations are irreversible and can be triggered by simple cross-site requests such as image tags, iframes, or link prefetching.

**Vulnerable Code:**

```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)  # STATE-CHANGING OPERATION
    ...
    return quart.redirect(f'/manage/{election.eid}', code=303)


@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz
    election.close()  # STATE-CHANGING OPERATION
    ...
    return quart.redirect(f'/manage/{election.eid}', code=303)
```

**Data Flow:**

1. **Source:** Attacker-crafted URL (e.g., `<img>` tag, link, iframe)
2. **Sink:** `election.open(pdb)` or `election.close()` — irreversible election state transitions
3. **Missing control:** No CSRF token validation; GET method bypasses even SameSite cookie protections for cross-site navigations in some browsers

**Proof of Concept:**

An attacker sends an email or hosts a page targeting an authenticated election administrator:

```html
<!-- Opens an election without the admin's knowledge -->
<img src="https://steve.apache.org/do-open/KNOWN_EID" width="1" height="1">

<!-- Closes an active election, disrupting voting -->
<img src="https://steve.apache.org/do-close/KNOWN_EID" width="1" height="1">
```

Since these are GET requests, they:
- Are triggered by image tags, iframes, link prefetching, etc.
- Are included in browser history and referrer headers
- Cannot be protected by CSRF tokens (no form body)
- May bypass `SameSite=Lax` cookie protections (top-level navigation)

**Impact:**

An attacker can open or close elections without the administrator's consent. Opening an election prematurely exposes it to voters before it's ready. Closing an election prematurely terminates voting, potentially disenfranchising voters. Both are irreversible state transitions in the election lifecycle.

**Remediation:**

Change these to POST endpoints with CSRF token validation:

```python
@APP.post('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    # Validate CSRF token
    form = await quart.request.form
    if form.get('csrf_token') != (await asfquart.session.read()).get('csrf_token'):
        quart.abort(403, 'CSRF token validation failed')
    
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    ...

@APP.post('/do-close/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    form = await quart.request.form
    if form.get('csrf_token') != (await asfquart.session.read()).get('csrf_token'):
        quart.abort(403, 'CSRF token validation failed')
    
    election.close()
    ...
```

Update the management template to use forms with POST and CSRF tokens instead of simple links.

#### ASVS-1021-HIGH-002

**Finding:** CSRF Token Is a Hardcoded Placeholder — All POST State-Changing Operations Lack CSRF Validation

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `basic_info()` (line 96)
- **Affected endpoints:** All POST handlers (lines 428, 435, 442, 488, 551, 576, 601)

**Related Functions Checked:**
- `do_vote_endpoint()` (line 442)
- `do_create_endpoint()` (line 488)
- `do_add_issue_endpoint()` (line 551)
- `do_edit_issue_endpoint()` (line 576)
- `do_delete_issue_endpoint()` (line 601)
- `do_set_open_at_endpoint()` (line 428)
- `do_set_close_at_endpoint()` (line 435)

**Description:**

The CSRF token is hardcoded as the string `'placeholder'` and is never validated in any POST handler. This creates a false sense of security while leaving all state-changing operations vulnerable to CSRF attacks.

**Vulnerable Code — Token "Generation":**

```python
async def basic_info():
    ...
    ### generate a real token and store in the session
    basic.csrf_token = 'placeholder'
    return basic
```

**Vulnerable Code — All POST Handlers (no validation):**

| Endpoint | Function | Line | Operation |
|----------|----------|------|-----------|
| `POST /do-vote/<eid>` | `do_vote_endpoint` | 442 | Submit votes |
| `POST /do-create-election` | `do_create_endpoint` | 488 | Create election |
| `POST /do-add-issue/<eid>` | `do_add_issue_endpoint` | 551 | Add election issue |
| `POST /do-edit-issue/<eid>/<iid>` | `do_edit_issue_endpoint` | 576 | Edit issue |
| `POST /do-delete-issue/<eid>/<iid>` | `do_delete_issue_endpoint` | 601 | Delete issue |
| `POST /do-set-open_at/<eid>` | `do_set_open_at_endpoint` | 428 | Set open date |
| `POST /do-set-close_at/<eid>` | `do_set_close_at_endpoint` | 435 | Set close date |

**Data Flow:**

1. **Source:** `basic_info()` generates `csrf_token = 'placeholder'` → passed to templates
2. **Templates:** May render the token into forms (creating false confidence)
3. **Sink:** POST handlers process `await quart.request.form` without any CSRF validation
4. **Missing control:** No token generation (not random), no session storage, no validation in any handler

**Proof of Concept — Forging a vote submission:**

```html
<form id="csrf" method="POST" action="https://steve.apache.org/do-vote/TARGET_EID">
    <input type="hidden" name="vote-issue1" value="y">
    <input type="hidden" name="vote-issue2" value="n">
    <!-- csrf_token not needed since it's never validated -->
</form>
<script>document.getElementById('csrf').submit();</script>
```

An attacker can submit votes on behalf of an authenticated voter, or create/modify elections as an authenticated administrator, by hosting this page and tricking the victim into visiting it.

**Impact:**

All state-changing operations on the OAuth client are unprotected against CSRF:
- **Vote manipulation:** Attacker can submit or change votes for authenticated voters
- **Election manipulation:** Attacker can create elections, add/edit/delete issues, set dates
- The placeholder token creates false confidence that protection exists

**Remediation:**

Implement proper CSRF token generation and validation:

```python
import secrets

async def basic_info():
    ...
    s = await asfquart.session.read()
    if s:
        # Generate and store CSRF token in session if not present
        if 'csrf_token' not in s:
            s['csrf_token'] = secrets.token_urlsafe(32)
            await asfquart.session.write(s)
        basic.csrf_token = s['csrf_token']
    return basic

async def validate_csrf():
    """Validate CSRF token from form/header against session."""
    s = await asfquart.session.read()
    if not s or 'csrf_token' not in s:
        quart.abort(403, 'Missing session CSRF token')
    
    form = await quart.request.form
    token = form.get('csrf_token') or quart.request.headers.get('X-CSRF-Token')
    if not secrets.compare_digest(token or '', s['csrf_token']):
        quart.abort(403, 'CSRF token validation failed')

# Apply to all POST handlers:
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    await validate_csrf()  # Add this line
    ...
```

### Medium Severity

#### ASVS-1021-MEDIUM-001

**Finding:** OAuth Authorization Code Flow Lacks PKCE; State Parameter Validation Delegated to Unauditable Framework

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `create_app()`
- **Lines:** 38-42

**Related Functions Checked:**
- OAuth URL template configuration (lines 38-42)
- `asfquart.generics` module (external dependency)

**Description:**

The OAuth authorization code flow is configured without PKCE (Proof Key for Code Exchange), and the state parameter validation is entirely delegated to the `asfquart` framework, which cannot be verified from the application code. Additionally, the application deliberately downgrades from OIDC to plain OAuth, removing the `nonce` parameter protection.

**Vulnerable Code:**

```python
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**

1. **Source:** User initiates login → framework constructs authorization URL from `OAUTH_URL_INIT` template
2. **Authorization server:** Returns authorization code via redirect to the application callback
3. **Sink:** Framework exchanges code via `OAUTH_URL_CALLBACK` → **No PKCE `code_verifier` sent with token request**
4. **Missing control:** No `code_challenge`/`code_challenge_method` parameters in authorization request; no `code_verifier` in token exchange

**Analysis:**

The OAuth authorization URL template includes `state=%s`, indicating the `asfquart` framework is expected to generate and insert a state value. However:

1. **No PKCE parameters:** The authorization URL contains no `code_challenge` or `code_challenge_method` parameter, and the token exchange URL contains no `code_verifier`. PKCE is completely absent from the configuration.

2. **State validation unverifiable:** The state parameter generation, session storage, and callback validation are entirely within the `asfquart` framework, which is not part of the audited codebase. The application provides no application-level verification.

3. **Deliberate OIDC downgrade:** The comment `# Avoid OIDC` and the override of `OAUTH_URL_INIT`/`OAUTH_URL_CALLBACK` indicate a deliberate downgrade from OIDC (which includes `nonce` as an additional CSRF protection) to plain OAuth, removing an extra layer of protection.

**Proof of Concept:**

Without PKCE, if an attacker obtains an authorization code (e.g., via referrer header leakage, open redirect, or log exposure), they can exchange it without proving they initiated the flow. A CSRF attack on the callback endpoint could associate the attacker's identity with the victim's session:

```html
<!-- Attacker forces victim to complete OAuth flow with attacker's code -->
<img src="https://steve.apache.org/oauth/callback?code=ATTACKER_CODE&state=GUESSED_OR_EMPTY">
```

If the framework's state validation is weak (e.g., missing, accepts empty state, or uses predictable values), this attack succeeds.

**Impact:**

An attacker could potentially trigger token exchange with a stolen or forged authorization code, leading to session fixation or account takeover. The severity is mitigated IF the `asfquart` framework properly validates the state parameter, but this cannot be verified from the application code.

**Remediation:**

Add PKCE parameters to the OAuth configuration:

```python
import secrets
import hashlib
import base64

# In the OAuth flow initialization
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
    '&code_challenge=%s&code_challenge_method=S256'
)

# Generate PKCE values per-session
def generate_pkce():
    code_verifier = secrets.token_urlsafe(64)
    code_challenge = base64.urlsafe_b64encode(
        hashlib.sha256(code_verifier.encode()).digest()
    ).rstrip(b'=').decode()
    return code_verifier, code_challenge
```

If `asfquart` does not support PKCE natively, file a feature request or switch to an OAuth library that does (e.g., `authlib`).

## Positive Security Patterns

1. **State parameter present in OAuth URL template:** The `state=%s` placeholder in `OAUTH_URL_INIT` indicates the `asfquart` framework is designed to generate and use a state parameter in the authorization code flow.

2. **Authentication required on all state-changing routes:** Every endpoint that modifies data uses `@asfquart.auth.require({R.committer})` or `@asfquart.auth.require({R.pmc_member})`, ensuring only authenticated users can reach these endpoints.

3. **HTTPS enforced for OAuth:** Both `OAUTH_URL_INIT` and `OAUTH_URL_CALLBACK` use `https://` URLs, preventing code interception over the network.

4. **LDAPS used for directory access:** `LDAP_URL = 'ldaps://ldap-us.apache.org/'` uses LDAP over TLS.

5. **POST method used for most state-changing operations:** 7 of 9 state-changing operations correctly use POST method, limiting the attack surface for cross-site request forgery.

6. **Voter eligibility verification:** The `vote_on_page` function checks `mayvote` entries before rendering the voting interface, providing authorization beyond authentication.

---

## Architecture Observations

1. **Framework-delegated OAuth security:** The application delegates all OAuth flow handling to `asfquart`, making it impossible to audit the critical state parameter generation and validation from the application code alone. A complete audit requires examining the `asfquart` framework source.

2. **OIDC-to-OAuth downgrade:** The explicit override from OIDC to plain OAuth (`# Avoid OIDC`) removes the OIDC `nonce` parameter, which provides an additional layer of CSRF protection beyond the `state` parameter.

3. **No defense-in-depth for CSRF:** The application has a single layer of CSRF protection (the framework's state parameter for OAuth flow), with no application-level CSRF tokens. If the framework's protection fails, there is no fallback.
