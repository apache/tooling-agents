## Findings

### Medium Severity

#### ASVS-1062-MED-001: Complete Absence of Logout Mechanism Prevents OIDC Logout Flow Participation

- **Severity:** MEDIUM
- **Gap Type:** Type A (Entry point with NO control)
- **CWE:** CWE-613 (Insufficient Session Expiration)
- **File:** `v3/server/pages.py` (entire file — no logout route exists)
- **Related Files:** `v3/server/main.py:37-41` (OAuth configuration)
- **Functions Checked:**
  - All 21 route handlers in `pages.py`
  - OAuth configuration in `main.py`
  - Session management calls throughout codebase

**Description:**

The application configures itself as an OIDC/OAuth Relying Party against `oauth.apache.org` but implements **zero** logout functionality. There is no `/logout` endpoint, no session destruction mechanism, no front-channel logout URI, and no back-channel logout handler. This creates the following security gaps:

1. **Users cannot terminate their sessions** — once authenticated, sessions persist until natural expiry
2. **The RP cannot initiate logout with the OP** — no RP-Initiated Logout per [OIDC RP-Initiated Logout 1.0](https://openid.net/specs/openid-connect-rpinitiated-1_0.html)
3. **The RP cannot process OP-initiated logout** — if `oauth.apache.org` implements [OIDC Front-Channel](https://openid.net/specs/openid-connect-frontchannel-1_0.html) or [Back-Channel Logout](https://openid.net/specs/openid-connect-backchannel-1_0.html), this RP would not honor those notifications, leaving orphaned sessions

**Vulnerable Code (Absence Evidence):**

OAuth configuration in `main.py` (lines 37-41) lacks logout URL:
```python
# main.py lines 37-41
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
# No OAUTH_URL_LOGOUT or equivalent
```

Session read without destruction capability in `pages.py` (lines 92-101):
```python
# pages.py lines 92-101
s = await asfquart.session.read()
if s:
    basic.update(uid=s['uid'], name=s['fullname'], email=s['email'])
# No session.destroy(), session.clear(), or equivalent anywhere in codebase
```

**Data Flow:**

```
┌─────────────────┐
│ User Session    │
│ Created by      │
│ asfquart OAuth  │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Session Persists│
│ Indefinitely    │◄─── ❌ No termination path
└─────────────────┘
         │
         ▼
┌─────────────────┐
│ Orphaned Session│
│ Remains Valid   │
└─────────────────┘
```

**Attack Scenario:**

1. User A authenticates and votes on `https://steve.apache.org/voter`
2. User A navigates away (no logout option exists)
3. User B accesses the same browser — User A's session cookie is still valid
4. User B can now cast votes as User A

**Impact:**

- **Session Hijacking Risk:** Users on shared workstations cannot end their sessions, leaving subsequent users with access to the voting system under the previous user's identity
- **Orphaned Sessions:** If `oauth.apache.org` revokes a user's access or the user logs out of the OP, the RP session remains valid — orphaned sessions continue to authorize actions
- **Unauthorized Vote Casting:** For a voting system, this means unauthorized vote casting is possible via session reuse
- **Compliance Gap:** Fails to implement standard OIDC logout flows, preventing proper session lifecycle management

**Remediation:**

**1. Add RP-Initiated Logout Endpoint:**

```python
@APP.get('/logout')
@asfquart.auth.require
async def logout():
    # Read current session to get id_token_hint if available
    s = await asfquart.session.read()
    id_token_hint = s.get('id_token') if s else None
    
    # Destroy the RP-side session
    await asfquart.session.clear()
    
    # RP-Initiated Logout (OIDC RP-Initiated Logout 1.0)
    if id_token_hint:
        logout_url = (
            f"https://oauth.apache.org/logout?"
            f"id_token_hint={id_token_hint}&"
            f"post_logout_redirect_uri={quart.url_for('home_page', _external=True)}"
        )
        return quart.redirect(logout_url)
    
    await flash_info('You have been logged out.')
    return quart.redirect('/')
```

**2. Implement Back-Channel Logout Handler:**

```python
@APP.post('/backchannel-logout')
async def backchannel_logout():
    # Validate the logout token (JWT signed by the OP)
    logout_token = (await quart.request.form).get('logout_token')
    if not logout_token:
        quart.abort(400)
    
    # Verify signature, iss, aud, and events claim
    claims = verify_logout_token(logout_token)  # implementation needed
    
    # Invalidate session(s) for the subject
    await invalidate_sessions_for_sub(claims['sub'])
    
    return '', 200
```

**3. Configure Logout URL in `main.py`:**

```python
asfquart.generics.OAUTH_URL_LOGOUT = (
    'https://oauth.apache.org/logout?id_token_hint=%s&post_logout_redirect_uri=%s'
)
```

**4. Register Logout URIs with the OP:**
- Register `https://steve.apache.org/backchannel-logout` as the back-channel logout URI
- Register `https://steve.apache.org/` as an allowed post-logout redirect URI

## Architecture Observations

### 1. Relying Party Architecture

This application is strictly a Relying Party. The OpenID Provider is `oauth.apache.org`, which is external. The core ASVS 10.6.2 requirement (OP-side logout DoS mitigation via `id_token_hint` validation or user confirmation) must be verified against the OP's implementation, not this RP.

### 2. Framework Abstraction

The `asfquart` framework (not provided in the audit scope) may include built-in logout handling, session management, or OIDC flows. The audit is limited to what's visible in the application code. If `asfquart` provides a default `/logout` route, this finding may require adjustment — this should be verified against the framework source.

### 3. OAuth vs. OIDC Implementation

The comment `# Avoid OIDC` in `main.py:35` and the use of plain OAuth URLs (without `.well-known/openid-configuration` discovery) suggest the application may be using OAuth without full OIDC. If no ID Tokens are issued, OIDC logout specifications may not apply, but session termination capability is still required for security.

### 4. Session Management Model

Sessions are read-only from the application perspective. All session creation and management appears to be delegated to the `asfquart` framework, which limits visibility into session lifecycle controls.
