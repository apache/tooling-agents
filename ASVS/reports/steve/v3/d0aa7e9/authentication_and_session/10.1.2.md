**Summary**

Audit of ASVS 10.1.2 focuses on whether the OAuth client properly binds authorization flow responses to the originating user agent session and transaction using cryptographic secrets (state, PKCE, nonce). Analysis reveals that the application explicitly overrides OAuth URL templates without PKCE support, and the token exchange endpoint similarly lacks PKCE `code_verifier` binding. The `state` parameter is present in the authorization URL template, but its generation, session-binding, and validation are delegated entirely to the opaque `asfquart` framework and cannot be verified from the available source code.

## Findings

### Medium Severity

#### ASVS-1012-MED-001: OAuth Authorization Flow Lacks PKCE (Proof Key for Code Exchange)

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/main.py`
- **Lines:** 35–38
- **Function:** `create_app()`

**Related Functions Checked:**
- `create_app()` — OAuth URL configuration
- OAuth callback handler (not present in provided code)

**Description:**

The application explicitly overrides the framework's OAuth URL templates. The authorization URL (`OAUTH_URL_INIT`) includes only `state` and `redirect_uri` — there are no `code_challenge` or `code_challenge_method` parameters. The token exchange URL (`OAUTH_URL_CALLBACK`) includes only `code` — there is no `code_verifier` parameter.

ASVS 10.1.2 specifically requires that _"client-generated secrets, such as the proof key for code exchange (PKCE) 'code_verifier'"_ are used to cryptographically bind the authorization code to the specific transaction. Without PKCE:

- An attacker who intercepts an authorization code (via referrer headers, browser history, open redirector, or log exposure) can exchange it at the token endpoint since no proof of the original requestor is required.
- The `state` parameter alone prevents cross-site request forgery on the OAuth flow but does **not** prevent authorization code injection by an attacker who obtains a valid code through other means.

**Vulnerable Code:**
```python
# v3/server/main.py, lines 35-38
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**
```
User Agent → /auth?state=X&redirect_uri=Y (no code_challenge)
                ↓
Authorization Server issues authorization code
                ↓
Callback → /token?code=Z (no code_verifier to bind code to original client)
```

**Proof of Concept:**
1. Attacker initiates their own OAuth flow to obtain a valid authorization code.
2. Attacker tricks victim into submitting the attacker's authorization code to the application's callback endpoint.
3. Without PKCE, the application cannot distinguish the legitimate code from the injected one — only the `state` check prevents this. If state validation has any weakness (see ASVS-1012-MED-002), the attack succeeds.

Alternatively, if an authorization code leaks (e.g., via server logs, referrer header), the attacker can exchange it directly since no `code_verifier` is required.

**Impact:** 

An attacker could potentially complete an OAuth flow with a stolen or injected authorization code, gaining access as the victim user. This is partially mitigated if the `state` parameter is correctly implemented, but defense-in-depth requires PKCE.

**Remediation:**

```python
# main.py - Add PKCE parameters to OAuth URL templates
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
    '&code_challenge=%s&code_challenge_method=S256'
)
asfquart.generics.OAUTH_URL_CALLBACK = (
    'https://oauth.apache.org/token?code=%s&code_verifier=%s'
)

# The framework must also be updated to:
# 1. Generate a cryptographically random code_verifier (>= 43 chars, unreserved chars)
# 2. Derive code_challenge = BASE64URL(SHA256(code_verifier))
# 3. Store code_verifier in server-side session before redirect
# 4. Send code_verifier with the token exchange request
```

**Note:** This also requires that `oauth.apache.org` supports PKCE (RFC 7636). If the authorization server does not support PKCE, this should be raised as an architectural gap requiring coordination with the OAuth provider.

#### ASVS-1012-MED-002: OAuth State Parameter Security Properties Unverifiable — Framework Delegation Without Audit Visibility

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/main.py`
- **Lines:** 35–38
- **Function:** `create_app()`
- **Additional Reference:** `v3/server/pages.py`, line 89, `basic_info()`

**Related Functions Checked:**
- `create_app()` — OAuth URL configuration
- `basic_info()` — CSRF token handling (analogous security control)
- OAuth callback handler (not present in provided code)

**Description:**

ASVS 10.1.2 requires that the `state` parameter is:
1. **Not guessable** — generated with a cryptographically secure random number generator
2. **Specific to the transaction** — unique per authorization request
3. **Securely bound to the client and user agent session** — stored server-side and validated on callback

The `state=%s` placeholder in the URL template confirms the framework is expected to populate this value. However:

- The **OAuth callback handler** is not present in any of the provided source files. It is entirely within the `asfquart` framework, which is not available for audit.
- The **state generation logic** is not visible — we cannot verify entropy source, length, or uniqueness.
- The **state validation logic** is not visible — we cannot verify that the callback checks the state against the session-stored value before accepting the authorization code.
- The **session binding mechanism** is opaque — `asfquart.session.read()` is used but its security properties (server-side vs. client-side, tamper-resistance) cannot be assessed.

The `basic.csrf_token = 'placeholder'` on line 89 of `pages.py` demonstrates that session-bound cryptographic tokens are not yet implemented for form CSRF protection. This pattern of deferred security controls raises concern about whether the analogous OAuth state parameter handling in the framework is robust.

**Vulnerable Code (URL template):**
```python
# v3/server/main.py, lines 35-38
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
```

**Vulnerable Code (CSRF token showing security posture):**
```python
# v3/server/pages.py, line 89
### generate a real token and store in the session
basic.csrf_token = 'placeholder'
```

**Data Flow:**
```
Authorization Request:
  Application → Generate state (unverifiable) → Store in session (unverifiable)
                ↓
  User Agent → Authorization Server with state parameter
                ↓
Authorization Callback:
  Authorization Server → User Agent with code + state
                ↓
  User Agent → Application callback (not in provided code)
                ↓
  Application → Validate state against session (unverifiable)
```

**Impact:** 

If the `state` parameter is predictable, not session-bound, or not validated on callback, an attacker could perform OAuth CSRF — forcing a victim to log in as the attacker's account, or injecting an authorization code from a different session.

**Remediation:**

1. **Obtain and audit the `asfquart` framework source code** — specifically the OAuth callback handler, state generation, and state validation logic.
2. **Verify** that `state` is generated using `secrets.token_urlsafe(32)` or equivalent:
   ```python
   import secrets
   state = secrets.token_urlsafe(32)
   ```
3. **Verify** that `state` is stored in a server-side session before the redirect:
   ```python
   session['oauth_state'] = state
   ```
4. **Verify** that the callback handler rejects requests where the returned `state` does not match the session-stored value:
   ```python
   if request.args.get('state') != session.get('oauth_state'):
       abort(403, "Invalid state parameter")
   session.pop('oauth_state')  # Consume the state
   ```
5. Document the framework's OAuth security properties as part of the application's security architecture.

## Architecture Observations

1. **OAuth flow is entirely framework-managed.** The application configures URL templates but does not implement any OAuth logic (authorization redirect, callback handling, token exchange, state management). This means the security of the OAuth flow is entirely dependent on the `asfquart` framework, which is not available for this audit.

2. **OIDC explicitly bypassed.** The comment `# Avoid OIDC` and the use of Apache's custom OAuth endpoint (`oauth.apache.org` instead of an OIDC provider) means OIDC-specific protections like `nonce` binding and ID Token validation are not applicable. This is a deliberate architectural choice, but it removes a layer of transaction binding.

3. **Token exchange URL suggests server-to-server flow.** The `OAUTH_URL_CALLBACK` points to a token endpoint (`/token?code=%s`), which the server-side application calls directly. This makes the application a confidential client, which partially mitigates the PKCE absence (the authorization server authenticates the client via credentials). However, PKCE is still recommended for defense-in-depth per current best practices (RFC 9126, OAuth 2.1 draft).

4. **Security control implementation deferred.** The placeholder CSRF token (`basic.csrf_token = 'placeholder'`) indicates that critical security controls may be deferred to future implementation, creating risk that OAuth security controls may similarly be incomplete.
