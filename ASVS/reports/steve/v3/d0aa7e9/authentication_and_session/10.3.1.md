## Findings

### Medium Severity

#### ASVS-1031-MED-001: No Access Token Audience Validation in OAuth Configuration

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `create_app()`
- **Lines:** 36–41

**Related Functions Checked:**
- `asfquart.session.read()` in `pages.py:83`
- OAuth configuration in `main.py:38-41`
- Session creation flow (framework-internal)

**Description:**

The application configures OAuth authentication without any visible mechanism to validate that access tokens are intended specifically for the STeVe voting service. The OAuth flow is explicitly configured to avoid OIDC (which would provide standardized `aud` claim validation), and no audience parameter is included in either the authorization request or token exchange.

**Vulnerable Code:**

```python
# Avoid OIDC
import asfquart.generics

asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**

```
User authenticates → oauth.apache.org issues token → 
asfquart exchanges code via OAUTH_URL_CALLBACK → 
Token accepted without audience validation → 
Session created with uid/fullname/email from token
```

**Analysis:**

1. **Authorization Request** (`OAUTH_URL_INIT`): Only includes `state` and `redirect_uri` parameters. Missing:
   - `audience` parameter (RFC 8707 Resource Indicators for OAuth 2.0)
   - `resource` parameter to scope the token to this service
   - Any mechanism to indicate the intended token consumer

2. **Token Exchange** (`OAUTH_URL_CALLBACK`): Only includes the `code` parameter. No audience constraint is communicated to the authorization server.

3. **Explicit OIDC Avoidance**: The comment `# Avoid OIDC` and the override of generic OAuth URLs bypasses OIDC's standardized `aud` claim in ID tokens. OIDC would provide:
   - An ID token with an `aud` claim set to the client's `client_id`
   - Standard validation libraries that check `aud` automatically

4. **No Visible Audience Validation in Token Processing**: Session creation (via `asfquart.session.read()` in `pages.py:83`) extracts `uid`, `fullname`, and `email` from the session but there is no evidence of audience verification either:
   - In the OAuth configuration
   - In the session creation flow
   - In any middleware or decorator

**Proof of Concept:**

If `oauth.apache.org` issues tokens for multiple Apache services (e.g., JIRA, Confluence, STeVe), a token obtained by authenticating to another Apache OAuth-integrated service could potentially be replayed against STeVe. Without audience validation:

```
1. Attacker authenticates to ServiceA (another Apache OAuth service)
2. Attacker intercepts/obtains the access token issued for ServiceA
3. Attacker presents this token to STeVe's callback handler
4. STeVe accepts the token (no aud check) and creates a valid session
5. Attacker gains authenticated access to the voting system
```

**Impact:**

- Tokens intended for other Apache services within the same OAuth provider could be accepted by STeVe
- In a voting system, unauthorized access could enable illegitimate voting or election manipulation
- Violates the principle that each resource server should only accept tokens scoped to it
- Potential for cross-service authentication bypass within the Apache OAuth ecosystem

**Remediation:**

**Option 1 — Use OIDC with audience validation:**

```python
def create_app():
    import asfquart.generics
    
    # Use OIDC for standardized audience validation
    # The framework should validate that id_token.aud == client_id
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    # Configure expected audience (this service's client_id)
    app.config['OAUTH_EXPECTED_AUDIENCE'] = 'steve-voting-app'
    
    import pages
    import api
    return app
```

**Option 2 — Add explicit audience validation in token processing:**

```python
def create_app():
    import asfquart.generics

    asfquart.generics.OAUTH_URL_INIT = (
        'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
        '&resource=https://steve.apache.org'  # RFC 8707
    )
    asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
    
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    # Register token validation middleware
    @app.before_serving
    async def configure_token_validation():
        app.config['EXPECTED_AUDIENCE'] = 'steve-voting-app'
    
    import pages
    import api
    return app
```

**Option 3 — If using JWT access tokens, add validation middleware:**

```python
import jwt

EXPECTED_AUDIENCE = 'steve-voting-app'

def validate_token_audience(token_payload):
    """Validate the aud claim matches this service."""
    aud = token_payload.get('aud')
    if isinstance(aud, list):
        if EXPECTED_AUDIENCE not in aud:
            raise ValueError(f'Token audience {aud} does not include {EXPECTED_AUDIENCE}')
    elif aud != EXPECTED_AUDIENCE:
        raise ValueError(f'Token audience {aud} does not match {EXPECTED_AUDIENCE}')
```

## Positive Security Patterns

1. **Consistent Authentication Decorators**: All state-changing and data-access endpoints use `@asfquart.auth.require` with appropriate role requirements (`R.committer`, `R.pmc_member`). No protected endpoint is missing an auth decorator.

2. **LDAPS for Directory Access**: `asf-load-ldap.py` uses `ldaps://` (line 30), ensuring encrypted LDAP communication.

3. **Tiered Authorization Model**: The application distinguishes between `R.committer` (voters) and `R.pmc_member` (election creators), providing role-based access control.

4. **Per-Election Voter Eligibility**: `vote_on_page()` (lines 314-318) explicitly checks `q_find_issues` to verify the authenticated user is eligible to vote in the specific election before displaying the ballot.

5. **Secure OAuth Provider**: The hardcoded use of `https://oauth.apache.org/` ensures token exchange occurs over TLS with Apache's own infrastructure.

---

## Architecture Observations

1. **Framework Opacity**: The `asfquart` framework handles the OAuth token exchange and session creation internally. Audience validation may exist within the framework but is not configurable or visible from the application code. This creates an **auditability gap** — the security property cannot be verified from the application layer alone.

2. **Session-Based Architecture**: After initial OAuth authentication, the application uses session cookies rather than validating access tokens per-request. This means audience validation is only critical at the initial token exchange, but that single point of validation is not evidenced.

3. **Dual Role**: The application acts as both an OAuth client (redirecting to the OAuth provider) and a resource server (protecting API endpoints). ASVS 10.3.1 applies to both roles.

4. **Single OAuth Provider**: The application is tightly coupled to Apache's OAuth infrastructure (`oauth.apache.org`), which may provide implicit audience scoping through organizational policies, but this cannot be verified from the code.

## Recommendations Summary

### Immediate Actions (0-30 days)

1. **Implement Audience Validation** [MEDIUM]: Configure the `asfquart` framework or add middleware to validate the `aud` claim in tokens received from `oauth.apache.org`. If using JWT access tokens, verify `aud` matches the application's `client_id`. If using opaque tokens, use the token introspection endpoint with audience verification.

### Short-term Actions (1-3 months)

2. **Consider Enabling OIDC** [MEDIUM]: Remove the "Avoid OIDC" workaround and use OIDC's standardized ID token with `aud` claim validation. This provides defense-in-depth audience verification through well-tested library code.

3. **Document Framework Security Properties** [INFORMATIONAL]: Since `asfquart` handles the core OAuth flow, document what token validation it performs (signature verification, expiration, issuer, **audience**) to enable future security audits without requiring framework source access.

### Long-term Actions (3-6 months)

4. **Framework Security Audit**: Conduct a security audit of the `asfquart` framework to verify its OAuth implementation includes proper audience validation and other security controls required by ASVS.

5. **Implement Resource Indicators**: Consider adopting RFC 8707 Resource Indicators for OAuth 2.0 to explicitly scope tokens to the STeVe service during the authorization request.
