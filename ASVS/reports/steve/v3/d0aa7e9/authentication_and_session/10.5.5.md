## Positive Security Patterns

The following security-positive patterns were observed during the audit:

### 1. Authentication Delegation to Framework
- **Location:** `main.py` lines 37-38
- **Pattern:** The application delegates authentication to `asfquart`, which handles the OAuth flow with Apache's identity infrastructure
- **Benefit:** Reduces attack surface for custom authentication bugs by leveraging vetted framework code

### 2. HTTPS OAuth Endpoints
- **Location:** `main.py` lines 40-43
- **Pattern:** Both `OAUTH_URL_INIT` and `OAUTH_URL_CALLBACK` use `https://` scheme
- **Benefit:** Ensures encrypted communication with OAuth provider, preventing token interception

### 3. Session-Based Access Control
- **Location:** Throughout `pages.py`
- **Pattern:** All sensitive routes use `@asfquart.auth.require({R.committer})` or `@asfquart.auth.require({R.pmc_member})`
- **Benefit:** Ensures authentication is enforced before route handlers execute

### 4. Secure LDAP Communication
- **Location:** `asf-load-ldap.py` line 33
- **Pattern:** LDAP connection uses `ldaps://` protocol
- **Benefit:** Ensures encrypted communication for sensitive directory operations

## Architecture Observations

### Why ASVS 10.5.5 Is Not Applicable

#### 1. OAuth, Not OIDC
- The application explicitly avoids OIDC (`main.py` line 37: `# Avoid OIDC`)
- Configured plain OAuth URLs point to `oauth.apache.org`
- No OIDC discovery/token endpoints that would issue `id_token`s or support back-channel logout

#### 2. No OIDC Relying Party Behavior
The codebase exhibits none of the characteristic OIDC RP behaviors:
- No `id_token` parsing or validation
- No OIDC discovery metadata consumption
- No `userinfo` endpoint calls
- No OIDC session management artifacts (e.g., `session_state`, `sid` claim tracking)

#### 3. Conditional Requirement
ASVS 10.5.5 explicitly states "**when using** OIDC back-channel logout" — this is a conditional requirement that only applies when the feature is implemented. The absence of the feature means the requirement does not apply.

### Related Architectural Observation

**Note:** This observation is out of scope for ASVS 10.5.5 but architecturally relevant.

The complete absence of any logout mechanism (neither front-channel nor back-channel, neither OIDC nor plain session destruction) means user sessions persist until cookie expiration. If the upstream OAuth provider (`oauth.apache.org`) revokes access, existing application sessions at STeVe would remain valid.

This is **not a violation of ASVS 10.5.5 specifically**, but may be relevant under other ASVS session management requirements (e.g., 3.3.x series on session termination).

## Recommendations Summary

Since no MEDIUM or higher findings were identified for ASVS 10.5.5, the following are **architectural recommendations** rather than vulnerability remediations:

### Future Considerations (If OIDC Is Adopted)

**Context:** The comment "Avoid OIDC" and "is this really needed right now?" in `main.py` suggests OIDC adoption may be considered in the future.

**If OIDC is adopted**, implement a back-channel logout endpoint that:

1. **Validates `typ` header** is `logout+jwt`
2. **Verifies the `event` claim** contains `http://schemas.openid.net/event/backchannel-logout`
3. **Rejects tokens** containing a `nonce` claim
4. **Enforces short `exp` window** (≤2 minutes as recommended)
5. **Invalidates the corresponding server-side session** for the subject identified in the logout token

### General Session Management Enhancement

**Recommendation:** Implement a logout route

Even with plain OAuth, the application should provide a mechanism for users to explicitly destroy their session. This is separate from ASVS 10.5.5 but is a foundational session management control covered by other ASVS requirements.

**Suggested implementation:**
- Add a `/logout` route
- Clear server-side session data
- Invalidate session cookie
- Optionally redirect to OAuth provider's logout endpoint if supported

1. Explicitly uses plain OAuth (not OIDC) as documented in `main.py` line 37

3. Does not process logout tokens of any kind

The conditional nature of the requirement ("**when using** OIDC back-channel logout") means there is no compliance violation. **No CRITICAL, HIGH, MEDIUM, or LOW findings are reported for this requirement.**

Should the application architecture change to adopt OIDC and implement back-channel logout in the future, the recommendations in this report should be implemented to ensure compliance with ASVS 10.5.5.
