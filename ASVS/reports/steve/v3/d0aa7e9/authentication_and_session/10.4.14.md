### Key Findings

This application functions as an **OAuth client** (not an authorization server) consuming tokens from `oauth.apache.org`. While ASVS 10.4.14 technically targets authorization servers, the audit reveals the client-side OAuth integration has **zero support** for sender-constrained token mechanisms. The token exchange uses plain bearer tokens with no proof-of-possession (DPoP) or mutual TLS (mTLS) certificate binding, meaning stolen tokens can be replayed from any context without detection.

## Findings

### High Severity

#### ASVS-10414-HIGH-001: OAuth Token Exchange Uses Plain Bearer Tokens Without Sender-Constraining Mechanism

**Severity:** HIGH  
**File:** `v3/server/main.py`  
**Function:** `create_app()`  
**Lines:** 37-41

**Related Functions Checked:**
- OAuth URL configuration in `main.py:37-41`
- TLS certificate configuration in `main.py:82-84`
- Session creation handlers in `pages.py` (using `asfquart.session.read()`)

**Description:**

The OAuth token exchange implementation uses plain bearer tokens without any sender-constraining mechanism (DPoP or mTLS). The application performs a simple authorization code exchange with no cryptographic binding between the token and the client's identity.

**Vulnerable Code:**

```python
# v3/server/main.py, lines 37-41
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**

```
1. User authenticates at oauth.apache.org
   ↓
2. Authorization code returned to application
   ↓
3. Application calls OAUTH_URL_CALLBACK with code only
   • No DPoP proof header
   • No mTLS client certificate
   ↓
4. Plain bearer access token received
   • No token_type validation
   • No cnf claim verification
   ↓
5. Token used without proof-of-possession binding
   ↓
6. Session created from token claims (pages.py: asfquart.session.read())
```

**Technical Analysis:**

The implementation lacks all required components for sender-constrained tokens:

1. **DPoP (RFC 9449) - Not Implemented:**
   - No DPoP keypair generation
   - No `DPoP` HTTP header construction
   - No `DPoP-Nonce` handling
   - No `token_type: "DPoP"` validation
   - No access token hash (`ath`) claim verification

2. **mTLS Certificate Binding (RFC 8705) - Not Implemented:**
   - Server certificates configured (`main.py:82-84`) are for HTTPS server operation only
   - No client certificate presented to authorization server
   - No `cnf` (confirmation) claim binding verification
   - No `x5t#S256` thumbprint validation

3. **Token Type Enforcement - Not Implemented:**
   - No validation that token response `token_type` is `"DPoP"` vs `"Bearer"`

**Proof of Concept:**

An attacker who intercepts the bearer access token can replay it from any context:

```http
# Legitimate token exchange (current implementation)
GET https://oauth.apache.org/token?code=AUTH_CODE_HERE
Response: {"access_token": "eyJ...", "token_type": "Bearer"}

# Attacker replays stolen token from different machine/context
# No DPoP proof or certificate binding prevents this
GET /userinfo
Authorization: Bearer eyJ...
Response: {"uid": "victim_user", "email": "victim@apache.org"}
```

**Attack Scenarios:**

1. **Network Interception:** Token captured via compromised network
2. **Log Exposure:** Token leaked in application logs or monitoring systems
3. **Memory Dump:** Token extracted from process memory
4. **Token Exfiltration:** Token stolen via XSS or other client-side vulnerabilities

**Impact:**

- **Token Replay:** Stolen access tokens can be replayed from any client without detection
- **Session Impersonation:** Token theft enables full session impersonation of legitimate users
- **Voting Manipulation:** In a voting system context, attackers could cast unauthorized votes
- **Compliance Violation:** Fails ASVS 10.4.14 Level 3 requirement for sender-constrained tokens
- **Privilege Escalation:** Stolen admin tokens grant unauthorized administrative access

**Remediation:**

**Option 1: Implement DPoP (RFC 9449) - Recommended**

```python
import json
import time
import uuid
import hashlib
import base64
from cryptography.hazmat.primitives.asymmetric import ec
from cryptography.hazmat.primitives import serialization
import jwt  # PyJWT

# Generate an ephemeral DPoP keypair per session
_dpop_private_key = ec.generate_private_key(ec.SECP256R1())
_dpop_public_key = _dpop_private_key.public_key()

def create_dpop_proof(http_method: str, http_uri: str, access_token: str = None) -> str:
    """Create a DPoP proof JWT per RFC 9449."""
    public_numbers = _dpop_public_key.public_numbers()
    jwk = {
        "kty": "EC",
        "crv": "P-256",
        "x": base64.urlsafe_b64encode(
            public_numbers.x.to_bytes(32, 'big')
        ).decode().rstrip('='),
        "y": base64.urlsafe_b64encode(
            public_numbers.y.to_bytes(32, 'big')
        ).decode().rstrip('='),
    }
    
    headers = {"typ": "dpop+jwt", "alg": "ES256", "jwk": jwk}
    payload = {
        "jti": str(uuid.uuid4()),
        "htm": http_method,
        "htu": http_uri,
        "iat": int(time.time()),
    }
    
    # When using with an access token, include ath (access token hash)
    if access_token:
        token_hash = hashlib.sha256(access_token.encode('ascii')).digest()
        payload["ath"] = base64.urlsafe_b64encode(token_hash).decode().rstrip('=')
    
    return jwt.encode(payload, _dpop_private_key, algorithm="ES256", headers=headers)

# Updated token exchange to include DPoP header
async def exchange_code_for_token(code: str) -> dict:
    token_endpoint = "https://oauth.apache.org/token"
    dpop_proof = create_dpop_proof("POST", token_endpoint)
    
    async with httpx.AsyncClient() as client:
        response = await client.post(
            token_endpoint,
            data={"code": code, "grant_type": "authorization_code"},
            headers={"DPoP": dpop_proof}
        )
        token_data = response.json()
        
        # Validate token_type is "DPoP"
        if token_data.get("token_type") != "DPoP":
            raise ValueError("Authorization server did not issue DPoP token")
        
        return token_data

# When using the access token
async def call_protected_resource(access_token: str, resource_url: str):
    dpop_proof = create_dpop_proof("GET", resource_url, access_token)
    
    async with httpx.AsyncClient() as client:
        response = await client.get(
            resource_url,
            headers={
                "Authorization": f"DPoP {access_token}",
                "DPoP": dpop_proof
            }
        )
        return response.json()
```

**Option 2: Implement mTLS Certificate Binding (RFC 8705)**

```python
import httpx

async def exchange_code_for_token(code: str) -> dict:
    """Exchange authorization code using mTLS client certificate."""
    async with httpx.AsyncClient(
        cert=("/path/to/client.crt", "/path/to/client.key"),
    ) as client:
        response = await client.post(
            "https://oauth.apache.org/token",
            data={"code": code, "grant_type": "authorization_code"},
        )
        token_data = response.json()
        
        # Verify the token contains cnf.x5t#S256 claim matching our cert
        access_token_decoded = jwt.decode(
            token_data["access_token"],
            options={"verify_signature": False}
        )
        
        if "cnf" not in access_token_decoded:
            raise ValueError("Token is not certificate-bound")
        
        # Verify certificate thumbprint matches
        with open("/path/to/client.crt", "rb") as f:
            cert_bytes = f.read()
            cert_hash = hashlib.sha256(cert_bytes).digest()
            expected_thumbprint = base64.urlsafe_b64encode(cert_hash).decode().rstrip('=')
            
            if access_token_decoded["cnf"].get("x5t#S256") != expected_thumbprint:
                raise ValueError("Certificate thumbprint mismatch")
        
        return token_data
```

## Architecture Observations

### OAuth Role Classification

This application functions as an **OAuth client** consuming tokens from `oauth.apache.org` (the external authorization server). While ASVS 10.4.14 technically targets authorization servers, the client must also support sender-constraining mechanisms for end-to-end security.

### Token Lifecycle

OAuth tokens are used briefly during authentication to establish a cookie-based session (`asfquart.session`). This reduces the token exposure window but does not eliminate replay risks during the authentication phase.

### Framework Dependencies

The actual token exchange logic is encapsulated in `asfquart.generics`, which is not provided in the source code. The URL templates suggest the framework performs a simple GET-based token exchange, which is incompatible with DPoP (requires POST with headers).

### External Dependencies

Implementation of sender-constrained tokens requires coordination with `oauth.apache.org`:
- Authorization server must support DPoP or mTLS token binding
- Both client and server must be updated in coordination
- Configuration changes required on both sides
