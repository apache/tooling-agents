## Findings

### High Severity

#### ASVS-104016-HIGH-001

**Title:** OAuth Client Authentication Lacks Public-Key-Based Methods (mTLS / private_key_jwt)

**Location:**
- **File:** `v3/server/main.py`
- **Lines:** 38–43
- **Function:** `create_app()`

**Related Functions Checked:**
- `asfquart.generics.OAUTH_URL_INIT` configuration
- `asfquart.generics.OAUTH_URL_CALLBACK` configuration
- TLS configuration in `main.py:83-86`

**Description:**

ASVS 10.4.16 requires that the OAuth client uses strong, public-key-based client authentication methods (mutual TLS or `private_key_jwt`) that are resistant to replay attacks. The application shows no evidence of configuring or using any public-key-based client authentication method for the token endpoint exchange.

**Vulnerable Code:**

```python
# Avoid OIDC
import asfquart.generics

asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**

```
User authenticates → OAuth redirect to oauth.apache.org →
Authorization code returned → Token exchange at OAUTH_URL_CALLBACK →
    ↳ Only `code` parameter formatted into URL
    ↳ NO client certificate (mTLS) configuration
    ↳ NO client_assertion / client_assertion_type (private_key_jwt)
    ↳ Missing: Strong client authentication to authorization server
```

**Evidence of Gap:**

1. **No mTLS Configuration for OAuth Flow:** The TLS certificate configuration in `main.py:83-86` is exclusively for the application's *own* HTTPS server binding, not for mutual TLS client authentication against the authorization server's token endpoint:
   ```python
   # This is SERVER TLS, not client auth to the AS
   kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
   kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
   ```

2. **No `private_key_jwt` Configuration:** The token endpoint URL template `'https://oauth.apache.org/token?code=%s'` formats only the authorization code. There is no:
   - `client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer` parameter
   - `client_assertion` parameter containing a signed JWT
   - JWT signing key configuration anywhere in the codebase

3. **Explicit OIDC Avoidance:** The comment `# Avoid OIDC` on line 38 confirms that a more complete authentication protocol stack is deliberately not used, which typically would enable stronger client authentication registration.

4. **No Client Authentication Metadata:** There is no configuration property for `token_endpoint_auth_method` or equivalent anywhere in the provided code, meaning the `asfquart` framework is not being instructed to use `tls_client_auth`, `self_signed_tls_client_auth`, or `private_key_jwt`.

**Proof of Concept:**

An attacker who obtains the client credentials could:
1. Intercept or steal an authorization code
2. Exchange the code for an access token without needing to prove possession of private key material
3. Replay the client credentials indefinitely (symmetric secrets are not time-bound or cryptographically bound to the request)

**Impact:**

- Without public-key-based client authentication, the client likely authenticates using `client_secret_basic` or `client_secret_post` (symmetric shared secrets), or possibly no client authentication at all (public client behavior).
- Symmetric secrets are vulnerable to credential theft if the secret is leaked (logs, config exposure, memory dumps).
- Symmetric methods are not resistant to replay attacks — a captured `client_secret` can be reused indefinitely.
- An attacker who obtains the client credentials could impersonate the application to the authorization server, exchanging stolen authorization codes for access tokens.
- This fails ASVS L3 requirement 10.4.16 which mandates authentication based on public-key cryptography and resistant to replay attacks.

**Remediation:**

Configure the OAuth client to use one of the following methods:

**Option A: Mutual TLS (`tls_client_auth`):**

```python
import httpx

# Configure HTTP client with mTLS for token endpoint
oauth_http_client = httpx.AsyncClient(
    cert=("/path/to/client-cert.pem", "/path/to/client-key.pem"),
    verify="/path/to/ca-bundle.pem",
)

# Register with AS using token_endpoint_auth_method = "tls_client_auth"
# per RFC 8705 Section 2
```

**Option B: Private Key JWT (`private_key_jwt`):**

```python
import time
import jwt
import secrets

def build_client_assertion(client_id, token_endpoint, private_key):
    now = int(time.time())
    claims = {
        "iss": client_id,
        "sub": client_id,
        "aud": token_endpoint,
        "iat": now,
        "exp": now + 60,  # Short-lived to prevent replay
        "jti": secrets.token_hex(16),  # Unique ID prevents replay
    }
    return jwt.encode(claims, private_key, algorithm="RS256")

# Token request includes:
# client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer
# client_assertion=<signed_jwt>
```

**Note:** This finding is limited by the `asfquart` framework source not being available for review. If `asfquart` internally implements strong client authentication that is not visible in the URL template configuration, this finding should be re-evaluated.

### Medium Severity

#### ASVS-104016-MEDIUM-001

**Title:** OAuth Client Confidentiality Classification Cannot Be Verified — No Client Type Enforcement

**Location:**
- **File:** `v3/server/main.py`
- **Lines:** 35–51
- **Function:** `create_app()`

**Related Functions Checked:**
- `asfquart.construct()` initialization
- OAuth URL configuration
- Application configuration loading

**Description:**

ASVS 10.4.16 first requires: *"Verify that the client is confidential."* A confidential client must demonstrate the ability to maintain credential confidentiality by authenticating with the authorization server using credentials that are not exposed to end users.

The application is a server-side Quart application, which architecturally *should* be a confidential client. However, no explicit client credential configuration or client type enforcement is visible in the codebase.

**Vulnerable Code:**

```python
def create_app():
    "Create the asfquart app and its endpoints."

    ### is this really needed right now?
    # Avoid OIDC
    import asfquart.generics

    asfquart.generics.OAUTH_URL_INIT = (
        'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
    )
    asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'

    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    # ...
```

**Analysis:**

1. **No explicit client credential configuration** is visible — no `client_id`/`client_secret` in config, no key material for client authentication.
2. **No client registration metadata** is configured showing `token_endpoint_auth_method` is set to a confidential method.
3. The token endpoint URL passes only the authorization code (`code=%s`), which mirrors a **public client** pattern where the client cannot authenticate itself.

**Impact:**

- If the client is treated as a public client by the AS, it loses the security benefits of confidential client flows (client authentication at token endpoint, reduced authorization code injection risk).
- Several ASVS requirements that offer relaxed controls for confidential clients would not apply.
- Increased risk of authorization code interception and injection attacks.

**Remediation:**

- Explicitly register the client as a confidential client with the authorization server (`oauth.apache.org`).
- Configure the application with the appropriate client credentials and authentication method.
- Document the client type classification in application security documentation.
- Add configuration validation to ensure confidential client credentials are present and properly secured.

## Positive Security Patterns

1. **Server-side OAuth Flow:** The application is a server-side Quart application, which is architecturally appropriate for a confidential client — the token exchange happens server-to-server, not in the browser.

2. **OAuth State Parameter:** The `OAUTH_URL_INIT` template includes `state=%s`, indicating CSRF protection in the OAuth authorization flow.

3. **HTTPS Enforcement:** Both the OAuth authorization (`https://oauth.apache.org/`) and the application's own server support TLS configuration.

4. **Consistent Auth Decorators:** All sensitive routes use `@asfquart.auth.require()` with appropriate role requirements, ensuring the OAuth-derived session is validated before access.

5. **LDAP over TLS:** The LDAP connection uses `ldaps://` (line 34 of `asf-load-ldap.py`), protecting credential transit.

---

## Architecture Observations

1. **External AS Dependency:** The authorization server (`oauth.apache.org`) is external. Full verification of ASVS 10.4.16 requires inspecting the AS's client registration for this application — specifically the `token_endpoint_auth_method` registered for the `steve` client.

2. **Framework Opacity:** The `asfquart` framework handles the actual OAuth token exchange. The URL templates suggest simple code-for-token exchange, but the framework *could* add client authentication headers/parameters internally. The absence of any key material or certificate configuration for the OAuth flow strongly suggests it does not.

3. **Comment-Driven Technical Debt:** The `### is this really needed right now?` and `# Avoid OIDC` comments suggest the OAuth integration was simplified or deferred, which typically means stronger authentication methods were not prioritized.
