## Findings

### Medium Severity

#### ASVS-1023-MED-001: OAuth authorization request does not specify required scopes, relying on server defaults

**Location:** `v3/server/main.py`, function `create_app()`, lines 38-41

**Related Functions Checked:**
- `pages.py:basic_info()` (lines 85-91) — session data consumption
- All endpoint handlers in `pages.py` — user identity requirements

**Description:**

The OAuth authorization URL template does not include a `scope` parameter, violating the principle of least privilege for OAuth scope requests. The application relies entirely on the authorization server's default scope configuration rather than explicitly requesting only the minimal scopes it requires.

**Vulnerable Code:**

```python
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**

1. **Source**: Hardcoded OAuth URL template in `create_app()` (line 38)
2. **Sink**: Authorization request sent to `oauth.apache.org` when user authenticates
3. **Missing Control**: No `scope` parameter in the authorization URL template

**Evidence from session usage** (`pages.py:basic_info()`, lines 85-91):

```python
s = await asfquart.session.read()
if s:
    basic.update(
        uid=s['uid'],
        name=s['fullname'],
        email=s['email'],
    )
```

The application consumes only three fields from the OAuth session: `uid`, `fullname`, and `email`. This defines the minimum data the application requires. However, the authorization URL contains only `state` and `redirect_uri` parameters — no `scope` parameter restricts what the authorization server grants.

The URL template has exactly two `%s` format placeholders (for `state` and `redirect_uri`), confirming that the `asfquart` framework cannot inject a scope value through this template.

**Proof of Concept:**

When a user initiates login, the authorization request sent to `oauth.apache.org` would be:

```
GET https://oauth.apache.org/auth?state=<random>&redirect_uri=https://steve.apache.org/callback
```

Missing from this request:

```
&scope=openid+uid+profile+email
```

Without an explicit scope, the AS may grant broader access (e.g., group memberships, commit access details, or other ASF-specific attributes) than the three fields (`uid`, `fullname`, `email`) actually consumed by the application.

**Impact:**

- **ASVS 10.2.3 Non-Compliance**: Direct violation of the requirement to request only required scopes
- **Principle of Least Privilege Violation**: The authorization server's default scope may be broader than necessary
- **Increased Attack Surface**: If session data or tokens are compromised, they may contain more permissions or expose more user data than this election application requires
- **Data Minimization Concern**: Potential privacy impact if excessive user data is collected without justification

**Remediation:**

**Option 1 — Direct URL Template Modification:**

```python
# Determine the minimal scopes needed by this application
REQUIRED_SCOPES = 'openid uid email'  # Adjust to match oauth.apache.org's scope vocabulary

asfquart.generics.OAUTH_URL_INIT = (
    f'https://oauth.apache.org/auth?state=%s&redirect_uri=%s&scope={REQUIRED_SCOPES}'
)
```

**Option 2 — Framework Configuration (if supported):**

```python
app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
app.config['OAUTH_SCOPES'] = 'openid uid email'  # Framework-specific configuration
```

**Additional Steps:**

1. Document the rationale for each requested scope
2. Map scopes to specific session fields consumed by the application
3. Coordinate with `oauth.apache.org` administrators to confirm available scopes
4. Verify that the minimal scope set still provides all required functionality

## Positive Security Patterns

The following security controls are properly implemented:

1. **State Parameter Protection**: The `OAUTH_URL_INIT` template includes a `state` parameter (`state=%s`), which helps protect against CSRF attacks in the OAuth flow.

2. **HTTPS-Only OAuth Endpoints**: Both `OAUTH_URL_INIT` and `OAUTH_URL_CALLBACK` use `https://` URLs, ensuring encrypted communication with the authorization server.

3. **Server-Side Token Exchange**: The `OAUTH_URL_CALLBACK` pattern suggests an authorization code flow (exchanging `code` for token server-side), which is the recommended approach for confidential clients.

4. **Authentication Enforcement**: All sensitive endpoints use `@asfquart.auth.require()` decorators with appropriate role requirements (`R.committer`, `R.pmc_member`), ensuring proper authorization checks.

5. **Redirect URI Specification**: The authorization URL template includes an explicit `redirect_uri` parameter, preventing open redirect vulnerabilities.

## Architecture Observations

### Framework Abstraction Layer

The `asfquart` framework handles the actual OAuth flow mechanics (redirect, callback, session creation). Without its source code, we can only audit the configuration surface it exposes. The URL templates are the only visible control point for OAuth parameters.

### First-Party OAuth Provider

The authorization server (`oauth.apache.org`) is controlled by the same organization. This mitigates the risk somewhat — the AS operator can configure conservative default scopes — but does not eliminate the ASVS requirement for the client to explicitly request minimal scopes.

### OIDC Avoidance

The comment `# Avoid OIDC` (lines 36-37) suggests a deliberate choice to use a simpler OAuth flow rather than OpenID Connect. This means standardized scope negotiation (`openid`, `profile`, `email`) may not apply, and the scope vocabulary is specific to Apache's OAuth implementation.

### Minimal Data Consumption

Analysis of `pages.py` confirms the application only consumes three user attributes (`uid`, `fullname`, `email`) from the OAuth session, demonstrating good data minimization practices at the application layer — though this should be reinforced at the OAuth authorization layer.

## Recommendations Summary

### Immediate Actions (Critical Priority)

None identified.

### Short-Term Actions (High Priority)

1. **Add Explicit Scope Parameter** (Effort: Low)
   - Modify `OAUTH_URL_INIT` to include a `scope` parameter limiting authorization to only the identity fields the application consumes
   - Test the modified authorization flow to ensure compatibility with `oauth.apache.org`

2. **Document Scope Requirements** (Effort: Low)
   - Create documentation mapping required scopes to specific session fields (`uid`, `fullname`, `email`) used by the application
   - Include rationale for each scope in the codebase comments

### Long-Term Actions (Medium Priority)

3. **Coordinate with Authorization Server** (Effort: Medium)
   - Work with `oauth.apache.org` administrators to confirm available scopes and their exact semantics
   - Verify that the default scope set is not overly broad for this application's needs
   - Establish a process for reviewing scope requirements when new features are added

4. **Framework Enhancement Evaluation** (Effort: Medium)
   - Investigate whether `asfquart` supports scope configuration through application config rather than URL template modification
   - If not available, consider contributing this feature to the framework
