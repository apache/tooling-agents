## Findings

### Medium Severity

#### ASVS-10410-MED-001: No Visible Client Authentication for OAuth Token Exchange
**Severity:** MEDIUM  
**Gap Type:** Type A — No visible client authentication control for OAuth backchannel requests  
**CWE:** CWE-306 (Missing Authentication for Critical Function)

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `create_app()`
- **Lines:** 38-41

**Related Functions Checked:**
- `asfquart.generics.OAUTH_URL_INIT` configuration (line 38-40)
- `asfquart.generics.OAUTH_URL_CALLBACK` configuration (line 41)
- `asfquart.construct()` initialization (line 43)
- Framework internals (not accessible for audit)

**Description:**

This server-side web application (Quart/Python) is inherently capable of maintaining credential confidentiality and should operate as a confidential OAuth client per RFC 6749 §2.1. ASVS 10.4.10 requires that confidential clients authenticate themselves to the authorization server during backchannel requests such as token exchange.

The audit reveals no visible client authentication mechanism in the codebase. The only OAuth configuration present consists of two URL templates pointing to `oauth.apache.org`. Critical authentication elements are missing:

1. **No `client_secret` configuration** — No client credentials visible in the code or configuration
2. **No client certificate for mTLS** — While server TLS certificates are configured, no mutual TLS client authentication is set up
3. **No `private_key_jwt` configuration** — No JWT assertion signing keys configured
4. **Framework opacity** — The `asfquart` package handles the actual HTTP request to the token endpoint, but its internals cannot be verified without source code access

Additionally, the token URL format uses query parameters (`token?code=%s`) rather than the RFC 6749 §4.1.3 recommended POST body approach, potentially exposing authorization codes in server logs.

**Vulnerable Code:**
```python
# v3/server/main.py, lines 38-41
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**
```
1. User completes authentication at oauth.apache.org
   ↓
2. Browser redirected back with authorization code
   ↓
3. asfquart framework calls OAUTH_URL_CALLBACK with code
   ↓
4. Token request sent to https://oauth.apache.org/token?code=<code>
   ↓
5. ⚠️ MISSING: No client_id, client_secret, client assertion, or mTLS configured
   ↓
6. Authorization server receives potentially unauthenticated token request
   ↓
7. Access token returned and used for session establishment
```

**Proof of Concept:**

If no client authentication is enforced, an attacker who obtains a valid authorization code (e.g., from server logs, referrer headers, or network interception) can exchange it for tokens without proving client identity:

```bash
# Attacker with stolen authorization code
curl 'https://oauth.apache.org/token?code=STOLEN_AUTH_CODE'

# If no client authentication is required, this returns a valid access token
# The authorization server cannot distinguish this request from a legitimate one
```

**Impact:**

Operating as an unauthenticated public client introduces several security risks:

- **Authorization code injection attacks** — Attackers who obtain valid authorization codes can exchange them for access tokens without proving client identity
- **No client identity verification** — The authorization server cannot distinguish legitimate token requests from this application versus requests from attackers replaying stolen codes
- **Token revocation vulnerability** — If token revocation is implemented, those requests would also lack authentication
- **Refresh token exposure** — Any refresh token operations would be similarly unauthenticated

**ASVS 10.4.10 Compliance Status:** ❌ Cannot verify compliance

**Remediation:**

**Immediate Actions:**

1. **Verify current configuration** — Obtain and review the `asfquart` framework source code and any external configuration files (e.g., `APP.cfg`) to determine if client authentication is already configured but not visible in the audited files.

2. **Confirm client registration** — Verify with Apache Infrastructure that the STeVe application is registered as a confidential client at `oauth.apache.org` and determine the configured authentication method.

**Implementation Options (choose one based on AS capabilities):**

**Option 1: Client Secret (Minimum Acceptable)**
```python
# main.py — Add explicit client authentication configuration
asfquart.generics.OAUTH_CLIENT_ID = 'steve-voting-app'
asfquart.generics.OAUTH_CLIENT_SECRET = os.environ['OAUTH_CLIENT_SECRET']
asfquart.generics.OAUTH_CLIENT_AUTH_METHOD = 'client_secret_post'

# Ensure secret is stored securely, not in source code
# Use environment variables or secure configuration management
```

**Option 2: Private Key JWT (Recommended per ASVS)**
```python
# main.py — JWT-based client authentication
asfquart.generics.OAUTH_CLIENT_ID = 'steve-voting-app'
asfquart.generics.OAUTH_CLIENT_AUTH_METHOD = 'private_key_jwt'
asfquart.generics.OAUTH_SIGNING_KEY_PATH = '/etc/steve/oauth-signing-key.pem'
asfquart.generics.OAUTH_SIGNING_ALG = 'RS256'

# Generate and securely store private key
# Register corresponding public key with oauth.apache.org
```

**Option 3: Mutual TLS (RFC 8705)**
```python
# main.py — mTLS client authentication
asfquart.generics.OAUTH_CLIENT_ID = 'steve-voting-app'
asfquart.generics.OAUTH_CLIENT_AUTH_METHOD = 'tls_client_auth'
asfquart.generics.OAUTH_CLIENT_CERT_PATH = '/etc/steve/client-cert.pem'
asfquart.generics.OAUTH_CLIENT_KEY_PATH = '/etc/steve/client-key.pem'

# Ensure client certificate is registered with oauth.apache.org
# Certificate must meet RFC 8705 section 2.1 requirements
```

**Token Exchange Protocol Fix:**

Ensure the authorization code is transmitted via POST body parameters rather than query parameters:

```python
# Instead of:
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'

# Use POST with form-encoded body (ensure framework supports this):
asfquart.generics.OAUTH_TOKEN_ENDPOINT = 'https://oauth.apache.org/token'
asfquart.generics.OAUTH_TOKEN_METHOD = 'POST'
# Framework should handle: grant_type=authorization_code&code=<code>&redirect_uri=<uri>&client_id=<id>
```

**Verification Steps:**

1. Enable debug logging for OAuth token exchange requests
2. Confirm client authentication parameters are included in token requests
3. Test that token exchange fails when client credentials are invalid
4. Document the authentication method in security configuration documentation

## Positive Security Patterns

### 1. HTTPS for All OAuth Communications
**Location:** `main.py:38-41`

Both `OAUTH_URL_INIT` and `OAUTH_URL_CALLBACK` use the `https://` scheme, ensuring that OAuth authorization codes, tokens, and client credentials (if present) are protected in transit via TLS encryption.

```python
asfquart.generics.OAUTH_URL_INIT = 'https://oauth.apache.org/auth?...'
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?...'
```

### 2. OAuth State Parameter Implementation
**Location:** `main.py:38-40`

The authorization URL template includes a `state=%s` parameter, which is critical for CSRF protection in the OAuth authorization flow. This prevents attackers from injecting authorization codes obtained from other sessions.

```python
'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
```

### 3. Consistent Authentication Enforcement
**Location:** `pages.py:148, 156, 165, 183`

All state-changing endpoints use `@asfquart.auth.require` decorators with appropriate role requirements, ensuring that OAuth-derived sessions are validated before processing sensitive operations:

```python
@asfquart.auth.require({R.committer})
async def create_issue():
    # Only authenticated committers can create issues
```

### 4. LDAP Over TLS
**Location:** `bin/asf-load-ldap.py:30`

The LDAP integration uses `ldaps://` protocol, encrypting LDAP authentication traffic:

```python
ldap.initialize('ldaps://ldap-us.apache.org/')
```

### 5. Server TLS Configuration
**Location:** `main.py:50-52`

The application supports TLS for its own server endpoints with configurable certificate and key files, protecting the application's web interface and API:

```python
certfile=app.cfg.server.certfile,
keyfile=app.cfg.server.keyfile,
```

## Architecture Observations

### 1. Application Role Clarification

This application operates as an **OAuth Client/Relying Party**, NOT as an OAuth Authorization Server. The authorization server is `oauth.apache.org`, which is operated by Apache Infrastructure. 

**Implication for ASVS 10.4.10:** This requirement is categorized under "OAuth Authorization Server" responsibilities. Since STeVe is a client, full compliance verification requires:
- Auditing the client's authentication behavior (covered in this report)
- Verifying that `oauth.apache.org` enforces client authentication (outside audit scope)

### 2. Framework Abstraction Layer

The `asfquart` framework provides a high-level abstraction over OAuth protocol details. While this simplifies application development, it creates an audit evidence gap:

```python
# All OAuth logic is delegated to the framework
asfquart.construct('steve', app_dir=THIS_DIR, ...)
s = await asfquart.session.read()
```

**Audit Impact:** Without access to `asfquart` source code, the audit cannot definitively confirm or deny the presence of client authentication mechanisms. The framework may:
- Implement client authentication internally using configuration from `APP.cfg`
- Rely on external configuration not visible in the audited codebase
- Lack client authentication entirely

### 3. Configuration Separation

Application configuration is loaded via `asfquart.construct('steve', app_dir=THIS_DIR)` and accessed through `APP.cfg` object. Client credentials may exist in this configuration but are not visible in the source files provided for audit:

```python
app.cfg.server.certfile  # Accessible configuration
app.cfg.oauth.client_secret  # Potentially exists but not confirmed
```

### 4. Plain OAuth 2.0 (Not OIDC)

The code includes an explicit comment avoiding OIDC:

```python
# Line 36: "Avoid OIDC"
```

**Security Implications:**
- No ID tokens with standardized user identity verification
- No `nonce` parameter for replay protection
- Increased reliance on the security of the token exchange process
- User identity derived from access token introspection or userinfo endpoint

This architectural choice makes proper client authentication for token exchange even more critical, as there are fewer protocol-level protections against token substitution attacks.

### 5. Session Security Dependency Chain

The application's authorization enforcement depends on OAuth session integrity:

```
OAuth Token Exchange (potentially unauthenticated)
  ↓
Session Establishment (asfquart.session)
  ↓
Authorization Checks (@asfquart.auth.require)
  ↓
Access to Sensitive Operations
```

If the token exchange lacks client authentication, the entire security chain is weakened at its foundation.

### 6. External Authorization Server Trust

The application places complete trust in `oauth.apache.org` for:
- User authentication
- Authorization decisions
- Token issuance and validation
- Session lifecycle management

This is appropriate for an internal Apache project, but it means ASVS 10.4.10 compliance depends on both:
- The client properly authenticating itself (this audit's focus)
- The AS properly enforcing client authentication (assumed but not verified)
