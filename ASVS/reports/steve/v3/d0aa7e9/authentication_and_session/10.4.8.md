## Findings

### Medium Severity

#### ASVS-1048-SEV-001: No Visible Session/Token Absolute Expiration Enforcement in OAuth Client

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/main.py`
- **Lines:** 36-48
- **Function:** `create_app()`

**Additional Locations:**
- **File:** `v3/server/pages.py`
- **Lines:** 60-90
- **Function:** `basic_info()`

**Related Functions Checked:**
- `asfquart.construct()` (main.py:45)
- `asfquart.session.read()` (pages.py:77)
- All authenticated route handlers in `pages.py`

**Description:**

The application lacks visible enforcement of absolute session or token expiration at the client level. While the application delegates authentication to an external OAuth Authorization Server (`oauth.apache.org`), there is no application-level mechanism to ensure sessions derived from OAuth tokens respect absolute expiration boundaries.

**Vulnerable Code:**

`main.py:36-48`:
```python
def create_app():
    "Create the asfquart app and its endpoints."

    import asfquart.generics

    asfquart.generics.OAUTH_URL_INIT = (
        'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
    )
    asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'

    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    # No session lifetime, token expiration, or cookie configuration visible
```

`pages.py:60-90`:
```python
async def basic_info():
    """Return base-level EZT template data."""
    basic = edict()
    # ...
    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
    else:
        basic.update(uid=None, name=None, email=None)
    # No check for session age, token expiration, or creation timestamp
    # ...
    return basic
```

**Data Flow:**

1. User authenticates via `oauth.apache.org` (external AS)
2. Authorization code exchanged for token via `OAUTH_URL_CALLBACK`
3. `asfquart` creates a session (mechanism not visible in provided code)
4. All subsequent requests read session via `asfquart.session.read()` — **no expiration or freshness check at the application level**
5. Session persists until explicitly invalidated or framework default (unknown)

**Gap Classification:** 

Type A — No visible control for session/token absolute expiration at the application level. The `asfquart.construct()` call includes no session lifetime configuration, and `basic_info()` performs no timestamp-based session validation.

**Proof of Concept:**

```
1. User authenticates via OAuth → session established
2. Time passes beyond the AS's intended refresh token expiration
3. User continues accessing authenticated routes (e.g., GET /voter, POST /do-vote/<eid>)
4. Application reads session via basic_info() — no timestamp check
5. If asfquart doesn't enforce absolute timeout, access continues indefinitely
```

**Impact:**

- If `asfquart` does not internally enforce absolute session expiration, sessions derived from OAuth tokens could persist indefinitely
- Even if the AS properly expires refresh tokens (per ASVS 10.4.8), the client session may outlive the intended token lifetime
- Long-lived sessions increase the window for session hijacking, particularly for a voting application where temporal bounds on voting windows are security-critical
- For an election system, a session that persists beyond an election's close could theoretically allow late vote submission (dependent on other controls)

**Remediation:**

**Step 1:** Configure explicit session absolute expiration in the application:

```python
def create_app():
    import datetime
    
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    # Enforce absolute session expiration
    app.config['PERMANENT_SESSION_LIFETIME'] = datetime.timedelta(hours=8)
    app.config['SESSION_COOKIE_SECURE'] = True
    app.config['SESSION_COOKIE_HTTPONLY'] = True
    app.config['SESSION_COOKIE_SAMESITE'] = 'Lax'
    
    return app
```

**Step 2:** Store the authentication timestamp in the session and validate it:

```python
import time

async def basic_info():
    basic = edict()
    s = await asfquart.session.read()
    if s:
        # Check absolute session age
        created_at = s.get('created_at')
        if created_at:
            age = time.time() - created_at
            MAX_SESSION_AGE = 8 * 3600  # 8 hours absolute maximum
            if age > MAX_SESSION_AGE:
                await asfquart.session.invalidate()
                quart.abort(401, 'Session expired')
        basic.update(uid=s['uid'], name=s['fullname'], email=s['email'])
    else:
        basic.update(uid=None, name=None, email=None)
    return basic
```

**Step 3:** Ensure the OAuth callback handler stores the creation timestamp:

```python
# In the OAuth callback handler (not visible in provided code)
async def oauth_callback():
    # ... token exchange logic ...
    session_data = {
        'uid': user_id,
        'fullname': full_name,
        'email': email,
        'created_at': time.time()  # Add creation timestamp
    }
    await asfquart.session.write(session_data)
```

## Conclusion

The application does not implement an OAuth Authorization Server; it is an OAuth client delegating to `oauth.apache.org`. Therefore, the core ASVS 10.4.8 requirement (refresh token absolute expiration at the AS) **cannot be verified from this codebase**. The external AS (`oauth.apache.org`) must be audited separately against this requirement.

On the client side, one MEDIUM finding was identified: the absence of visible session/token absolute expiration enforcement, which is the client-side complement needed to fully realize the security intent of ASVS 10.4.8. The `asfquart` framework may enforce this internally, but this cannot be confirmed from the provided source code.

**Risk Assessment:** The identified gap creates a potential window for session persistence beyond intended token lifetimes. For a voting application where temporal boundaries are security-critical (e.g., preventing votes after election close), this represents a moderate risk that should be addressed through the recommended session lifetime controls.
