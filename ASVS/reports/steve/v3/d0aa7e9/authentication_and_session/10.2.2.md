## Findings

### Summary

**No findings at MEDIUM severity or above.**

The ASVS 10.2.2 requirement is conditional on the OAuth client interacting with more than one authorization server. This application is configured to interact with exactly one authorization server (`oauth.apache.org`), making the requirement not applicable to the current codebase. No mix-up attack defense is needed because the attack vector does not exist in a single-AS configuration.

### Requirement Decomposition

The requirement has a conditional structure:

> **IF** the OAuth client can interact with more than one authorization server,  
> **THEN** it has a defense against mix-up attacks (e.g., `iss` parameter validation).

### Conditional Evaluation

| Factor | Evidence | Result |
|--------|----------|--------|
| Multiple AS configured? | Only `oauth.apache.org` in `main.py:38-42` | **No** |
| Dynamic AS selection? | No IdP discovery, no AS selection logic | **No** |
| Federation support? | No OIDC discovery, comment says "Avoid OIDC" (`main.py:37`) | **No** |
| Configurable AS list? | Hardcoded URLs, no config-driven AS array | **No** |

**Conclusion**: The conditional precondition is **not met**. The application interacts with exactly one authorization server.

## Positive Security Patterns

### 1. Single Authorization Server Architecture
**Location:** `main.py:38-42`

The hardcoded configuration to a single AS (`oauth.apache.org`) eliminates the mix-up attack vector entirely for the current deployment. By design, the application cannot be tricked into sending authorization codes to an unintended authorization server.

### 2. State Parameter Implementation
**Location:** `main.py:39`

The OAuth initialization URL includes a `state` parameter (`state=%s`), which when properly validated by the `asfquart` framework, provides CSRF protection during the OAuth flow, preventing attackers from injecting malicious authorization responses.

### 3. HTTPS-Only OAuth Endpoints
**Location:** `main.py:38-42`

Both the authorization and token URLs use `https://`, preventing downgrade attacks on the OAuth flow and ensuring confidentiality and integrity of OAuth parameters and tokens in transit.

### 4. Framework-Delegated OAuth Implementation
**Location:** Throughout codebase

The OAuth callback handling is delegated to the `asfquart` framework, centralizing the security-critical OAuth flow logic rather than implementing it in application code. This reduces the likelihood of implementation errors in the security-sensitive token exchange process.

### 5. Consistent Authentication Decorator Pattern
**Location:** `pages.py` (multiple functions)

All protected routes consistently use `@asfquart.auth.require` decorators, ensuring session validation is not accidentally omitted and providing a clear security boundary between public and authenticated endpoints.

## Architecture Observations

### 1. OAuth Flow Not Fully Auditable
The actual OAuth callback handler (where authorization codes are exchanged for tokens and sessions are created) lives within the `asfquart` framework, which is not provided. A complete ASVS audit would require reviewing `asfquart.generics` and `asfquart.auth` to verify:
- State parameter generation and validation
- Redirect URI validation
- Token response validation
- Session binding and management
- Token storage and lifecycle

### 2. OIDC Explicitly Avoided
The comment at `main.py:37` (`# Avoid OIDC`) and the use of plain OAuth URLs (rather than OIDC `.well-known/openid-configuration` discovery) indicate the application intentionally avoids OIDC. This means OIDC-specific protections like ID Token `iss` claim validation are not applicable. The application uses OAuth 2.0 for authorization/authentication rather than the OpenID Connect identity layer.

### 3. No Future-Proofing for Multi-AS Scenarios
If the application were to add a second authorization server in the future (e.g., for federation, testing environments, or multi-tenant scenarios), there is no `iss` validation infrastructure in place. This is a design consideration, not a current vulnerability, but would require architectural changes to maintain security compliance with ASVS 10.2.2.

### 4. Framework Dependency for Security Controls
The application's security posture for OAuth flows is heavily dependent on the `asfquart` framework's implementation. Any vulnerabilities in the framework's OAuth handling would directly impact this application. Regular framework updates and security monitoring are critical.

## Recommendations Summary

While no findings warrant MEDIUM+ severity for ASVS 10.2.2, the following recommendations improve security posture and future readiness:

### Immediate Actions
*No immediate actions required for ASVS 10.2.2 compliance.*

### Short-term Recommendations

1. **Audit `asfquart` Framework OAuth Implementation** (Priority: High, but out of scope for this codebase)
   - **Action:** Request source code for `asfquart.generics` OAuth callback handling
   - **Verify:** 
     - `state` parameter validation against session-stored values
     - Redirect URI validation against registered URIs
     - Token response validation (signature, expiry, audience)
     - Secure session binding to prevent session fixation
   - **Rationale:** The application delegates critical OAuth security controls to the framework, which must be verified independently

2. **Document Single-AS Assumption** (Priority: Low)
   - **Action:** Add an explicit architectural decision record (ADR) noting:
     - Application assumes a single authorization server (`oauth.apache.org`)
     - ASVS 10.2.2 is not applicable in current architecture
     - Adding a second AS would require implementing `iss` parameter validation per RFC 9207 (OAuth 2.0 Authorization Server Issuer Identification)
   - **Rationale:** Prevents future developers from inadvertently introducing multi-AS support without proper security controls

### Long-term Recommendations

3. **Prepare for Multi-AS Support** (Priority: Low, if federation is planned)
   - **Action:** If multi-tenant, federated, or test/production environment separation is planned:
     - Design `iss` parameter validation mechanism
     - Implement authorization server metadata validation
     - Consider OIDC adoption for standardized issuer identification
     - Add configuration management for multiple AS endpoints
   - **Rationale:** Proactive security architecture for anticipated features

4. **Framework Security Monitoring** (Priority: Medium)
   - **Action:** Establish monitoring for `asfquart` framework security advisories
   - **Implement:** Automated dependency scanning and update procedures
   - **Rationale:** Maintain security posture as framework evolves
