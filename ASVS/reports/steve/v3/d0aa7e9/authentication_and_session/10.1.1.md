**Audit Scope**

This audit examined the OAuth integration architecture, session management, and data flow to frontend templates in the STeVe voting application. The application delegates all OAuth token handling to the `asfquart` framework. The application code itself does not directly manipulate OAuth tokens in its routes. However, the explicit avoidance of OIDC removes standardized audience restriction mechanisms, and the opaque delegation to the framework without any visible token confinement configuration creates unverifiable trust boundaries.

## Findings

### Medium Severity

#### ASVS-1011-MED-001: Missing OIDC Audience Restriction Control

| Attribute | Detail |
|-----------|--------|
| **Finding ID** | ASVS-1011-MED-001 |
| **Severity** | MEDIUM |
| **File** | `v3/server/main.py` |
| **Function** | `create_app()` |
| **Lines** | 36–43 |
| **Gap Type** | Type A — No standardized audience restriction control |
| **CWE** | CWE-346: Origin Validation Error |

**Related Functions Checked:**
- `asfquart.construct()` — Framework initialization
- `asfquart.generics.OAUTH_URL_INIT` — OAuth authorization URL override
- `asfquart.generics.OAUTH_URL_CALLBACK` — OAuth token exchange URL override

**Vulnerable Code:**

```python
### is this really needed right now?
# Avoid OIDC
import asfquart.generics

asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Description:**

The application explicitly overrides the framework's default OIDC configuration to use a plain OAuth flow against `oauth.apache.org`. By disabling OIDC, the application loses the standardized ID Token `aud` (audience) claim verification that ensures tokens issued by the authorization server are intended exclusively for this specific client. Without audience-restricted tokens, there is no verifiable mechanism at the application layer to confirm that:

1. The access token obtained was issued specifically for the STeVe application (and not a different OAuth client registered with the same AS).
2. Token confusion attacks (where a token issued for one relying party is replayed against another) are prevented.

The developer comment `### is this really needed right now?` indicates uncertainty about whether this OIDC override is still necessary, suggesting this may be a transitional configuration that was never revisited.

**Data Flow:**

```
Authorization Server (oauth.apache.org)
    ↓ [OAuth 2.0 flow — NO OIDC]
    ↓ [Token issued without aud claim validation]
asfquart.generics.OAUTH_URL_CALLBACK
    ↓ [Token exchange — no audience verification]
Application Session
    ↓ [Token stored — cannot verify intended recipient]
Protected Resources
```

- **Source:** `main.py` `create_app()` overrides `asfquart.generics.OAUTH_URL_INIT` and `OAUTH_URL_CALLBACK`
- **Missing Control:** No `aud` claim validation on tokens; no OIDC ID Token verification
- **Result:** Tokens issued by the authorization server have no application-level audience binding

**Proof of Concept:**

In a multi-client environment where multiple applications use `oauth.apache.org`:

1. An attacker with a legitimate OAuth client registered at `oauth.apache.org` obtains an access token for their own client.
2. The attacker presents this token to STeVe's session layer.
3. Without audience restriction checking, STeVe cannot distinguish tokens issued for itself from tokens issued for other clients.

This risk is bounded by the ASF infrastructure's OAuth server configuration, but the application provides no defense-in-depth against token confusion.

**Impact:**

Without audience-restricted tokens, the application cannot independently verify that OAuth tokens were issued for this specific client. In a shared authorization server environment (which `oauth.apache.org` is), this weakens the guarantee that tokens are consumed only by the intended component. This creates potential for token confusion attacks where tokens intended for one client could be misused by another client sharing the same authorization server.

**Remediation:**

Re-enable OIDC and validate the ID Token's `aud` claim:

```python
def create_app():
    # Use OIDC for audience-restricted token issuance
    # Do NOT override OAUTH_URL_INIT/CALLBACK to avoid OIDC
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    # Configure OIDC client_id for audience validation
    # The framework should verify id_token.aud == expected_client_id
    app.config['OIDC_CLIENT_ID'] = 'steve-voting-app'
    app.config['OIDC_VALIDATE_AUDIENCE'] = True
    
    # Remove the OAUTH_URL overrides to use OIDC defaults
    # import asfquart.generics
    # asfquart.generics.OAUTH_URL_INIT = ...  # DELETE THESE LINES
    # asfquart.generics.OAUTH_URL_CALLBACK = ...
    
    import pages
    import api
    return app
```

#### ASVS-1011-MED-002: Unverified Session Transport May Expose Tokens to Browser

| Attribute | Detail |
|-----------|--------|
| **Finding ID** | ASVS-1011-MED-002 |
| **Severity** | MEDIUM |
| **File** | `v3/server/pages.py` |
| **Function** | `basic_info()` |
| **Lines** | 65–95 |
| **Gap Type** | Type A — No verification that session transport does not expose tokens |
| **CWE** | CWE-522: Insufficiently Protected Credentials |

**Related Functions Checked:**
- `asfquart.session.read()` — Session data retrieval
- All template rendering functions using `basic_info()` — Session data consumption
- `quart.redirect()` — Redirect responses (verified clean)

**Vulnerable Code:**

```python
async def basic_info():
    """Return base-level EZT template data."""
    basic = edict()
    # ...
    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
    # ...
    basic.csrf_token = 'placeholder'
    return basic
```

**Description:**

The application reads session data via `asfquart.session.read()` in every authenticated handler. Quart's default session implementation stores **all session data** in a client-side signed cookie (itsdangerous-signed, base64-encoded). If the `asfquart.session` follows Quart's default and the framework stores the OAuth access token or refresh token in the session (a common pattern for maintaining authenticated state), these tokens would be:

1. Serialized into the session cookie sent to the browser with every HTTP response.
2. Readable by any JavaScript on the page (if the cookie lacks `HttpOnly`).
3. Sent by the browser with every subsequent request to the domain, including to static file endpoints and public pages that do not require authentication.

The application code extracts only `uid`, `fullname`, and `email` from the session — but the entire session dictionary (including any tokens the framework may have stored) travels through the browser via the cookie. There is no visible configuration in the application ensuring:
- Server-side session storage (keeping tokens on the server, sending only a session ID to the browser)
- Session cookie attributes (`HttpOnly`, `Secure`, `SameSite=Lax`)
- Token exclusion from the session cookie payload

**Data Flow:**

```
OAuth Authorization Server
    ↓ [Token exchange]
asfquart framework
    ↓ [Store tokens + user data in session]
Session Cookie (potentially client-side)
    ↓ [Full session payload sent to browser]
Browser
    ↓ [Cookie included in all requests]
    ↓ [Potentially accessible to JavaScript if not HttpOnly]
Application (receives session cookie on every request)
```

- **Source:** `asfquart` framework stores OAuth session data after authentication
- **Transport:** Quart session cookie (potentially client-side by default)
- **Sink:** Browser receives full session cookie with every response
- **Missing Control:** No explicit server-side session configuration; no session cookie hardening visible

**Proof of Concept:**

If using Quart's default client-side sessions:

```bash
# Decode a Quart session cookie (base64 + itsdangerous signature)
# The session payload is readable (signed, not encrypted by default)
echo "<session_cookie_value>" | base64 -d
# May reveal: {"uid": "user123", "fullname": "...", "email": "...", 
#              "access_token": "eyJ...", "refresh_token": "..."}
```

If `HttpOnly` is not set:

```javascript
// Any XSS payload can read the session cookie
document.cookie  // includes full session with potential tokens
```

**Impact:**

If OAuth tokens are stored in the session and the session uses client-side storage, access and refresh tokens are sent to the browser (a component that should never receive them in a server-rendered application). This directly violates ASVS 10.1.1's requirement that tokens are only accessible to components that need them. The browser becomes an unnecessary holder of sensitive tokens, expanding the attack surface for token theft via XSS, browser extensions, or client-side malware.

**Remediation:**

Configure server-side session storage and secure cookie attributes:

```python
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    # Use server-side sessions (only session ID in cookie)
    app.config['SESSION_TYPE'] = 'filesystem'  # or 'redis', 'database'
    app.config['SESSION_FILE_DIR'] = '/secure/session/storage'
    
    # Harden session cookie
    app.config['SESSION_COOKIE_HTTPONLY'] = True
    app.config['SESSION_COOKIE_SECURE'] = True
    app.config['SESSION_COOKIE_SAMESITE'] = 'Lax'
    app.config['SESSION_COOKIE_NAME'] = '__Host-steve_session'  # __Host- prefix for additional security
    
    # Ensure framework does not store tokens in session
    # If tokens must be in session, ensure they're server-side only
    # Never store access/refresh tokens in client-side cookies
    
    import pages
    import api
    return app
```

Additionally, audit the `asfquart` framework to confirm:
- Tokens are stored server-side only
- Session cookies contain only a session identifier
- Session identifiers are cryptographically random and sufficiently long

## Positive Security Patterns

The following security-positive patterns were identified during the audit:

1. **Server-side rendering architecture**: The application uses EZT server-side templates rather than a JavaScript SPA. This inherently confines token handling to the backend — the browser receives HTML, not tokens. This architectural choice significantly reduces the attack surface for token exposure.

2. **Minimal session data extraction**: `basic_info()` extracts only `uid`, `fullname`, and `email` from the session object, rather than passing the entire session to templates. This limits exposure even if the session contains additional data and demonstrates defense-in-depth thinking.

3. **No tokens in URLs or redirects**: All `quart.redirect()` calls use simple paths without query parameters containing tokens. Authorization codes are not visible in application-level URL construction. This prevents token leakage via browser history, referrer headers, and server logs.

4. **No token-returning API endpoints visible**: The provided endpoints return HTML pages, 204 status codes, or redirects — none return tokens in response bodies. This confirms tokens remain backend-only in the reviewed code paths.

5. **Authentication decorator consistency**: All state-changing endpoints use `@asfquart.auth.require`, ensuring the authentication token (session) is validated before processing. This provides centralized authentication enforcement.

6. **Template data sanitization**: The `basic_info()` function creates a clean dictionary for template rendering, avoiding accidental exposure of the entire session object to the template layer.

## Architecture Observations

1. **Opaque framework delegation**: ALL OAuth token lifecycle management (acquisition, storage, refresh, revocation) is handled by the `asfquart` framework. While this centralizes security-critical logic, it means the application has no defense-in-depth if the framework mishandles tokens. The application cannot be fully audited for ASVS 10.1.1 without also auditing `asfquart`. This creates a single point of trust that should be verified.

2. **`api.py` module not provided**: The `main.py` file imports `import api` which registers additional endpoints. These API endpoints could potentially return token-bearing responses. Full verification of 10.1.1 requires auditing this module. The audit scope is incomplete without reviewing these endpoints.

3. **OIDC avoidance intent unclear**: The comment `### is this really needed right now?` suggests the OIDC override may have been a temporary workaround that persisted. The original intent for avoiding OIDC should be reviewed, as OIDC provides token audience restriction directly relevant to 10.1.1. This appears to be technical debt that should be addressed.

4. **Session storage mechanism unknown**: Whether `asfquart.session` uses server-side or client-side storage is not visible in the application configuration. This is the single most important factor for ASVS 10.1.1 compliance in this architecture. Without this verification, the application's compliance status cannot be definitively determined.

5. **Backend-for-Frontend pattern implicit**: The architecture follows BFF principles (server-rendered pages, no SPA), which is the ideal pattern for ASVS 10.1.1 compliance. However, the lack of explicit configuration makes it difficult to verify that tokens never reach the browser.
