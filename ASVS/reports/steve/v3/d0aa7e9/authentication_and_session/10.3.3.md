## Findings

### Medium Severity

#### ASVS-1033-MED-001: User Identity Derived from Opaque `uid` Session Field Without Verifiable `iss`+`sub` Claim Origin

**Severity:** MEDIUM

**Location:** 
- **File:** `v3/server/pages.py`
- **Function:** `basic_info()`
- **Lines:** 89-98

**Related Functions Checked:**
- `voter_page()` - pages.py:157
- `vote_on_page()` - pages.py:274
- `admin_page()` - pages.py:329
- `do_vote_endpoint()` - pages.py:438
- `do_create_endpoint()` - pages.py:475
- `do_open_endpoint()` - pages.py:496
- `do_close_endpoint()` - pages.py:514
- `serve_doc()` - pages.py:626

**Description:**

The application derives user identity from a session field `uid` without verifiable proof that this identifier originates from non-reassignable OAuth token claims (`iss` + `sub`). All authorization decisions throughout the application depend on this single `uid` field, which is populated by the opaque `asfquart` framework during OAuth token exchange.

**Vulnerable Code:**

```python
# pages.py lines 89-98
s = await asfquart.session.read()
if s:
    basic.update(
        uid=s['uid'],       # <-- sole identifier for all authz decisions
        name=s['fullname'],
        email=s['email'],
    )
```

**Data Flow:**

```
OAuth token from oauth.apache.org
    → asfquart framework (opaque token processing)
    → Session populated with 'uid', 'fullname', 'email'
    → basic_info() reads s['uid']
    → result.uid used for ALL authorization decisions:
        - Voter eligibility: election.q_find_issues.perform(result.uid, ...)
        - Vote recording: election.add_vote(result.uid, ...)
        - Ownership queries: Election.owned_elections(DB_FNAME, result.uid)
        - Admin operations
    → MISSING: No verification that uid maps to iss+sub
```

**Authorization Decision Points Using `uid`:**

| Endpoint | File:Line | Usage |
|----------|-----------|-------|
| `voter_page` | pages.py:157 | `Election.open_to_pid(DB_FNAME, result.uid)` |
| `vote_on_page` | pages.py:274 | `election.q_find_issues.perform(result.uid, election.eid)` |
| `admin_page` | pages.py:329 | `Election.owned_elections(DB_FNAME, result.uid)` |
| `do_vote_endpoint` | pages.py:438 | `election.add_vote(result.uid, iid, votestring)` |
| `do_create_endpoint` | pages.py:475 | `Election.create(DB_FNAME, form.title, result.uid)` |
| `do_open_endpoint` | pages.py:496 | Logged as acting user |
| `do_close_endpoint` | pages.py:514 | Logged as acting user |
| `serve_doc` | pages.py:626 | `db.q_get_mayvote.first_row(result.uid, iid)` |

**Corresponding LDAP Identity Source:**

```python
# asf-load-ldap.py lines 53-56 — uid is LDAP username, not a stable sub claim
uid = entry.uid[0].decode('utf-8')
pdb.add_person(uid, visname, email)
```

**Proof of Concept:**

If the `asfquart` framework populates `uid` from a reassignable claim (such as `preferred_username`, `email`, or a custom attribute) rather than the immutable `sub` claim combined with `iss` validation, the following scenario becomes possible:

1. User A with identifier "committer123" leaves the organization
2. The identifier "committer123" is later reassigned to User B
3. User B authenticates via OAuth and receives a token
4. If the framework uses a reassignable claim, User B's session receives `uid='committer123'`
5. User B now has access to all of User A's election permissions, votes, and administrative privileges

**Impact:**

The `uid` field from the session is used as the sole user identifier for every authorization decision in the system, including vote eligibility and vote submission. The application code has no mechanism to verify that this `uid` was derived from the non-reassignable combination of `iss` (issuer) and `sub` (subject) claims from the OAuth token response.

If the `asfquart` framework populates `uid` from a reassignable claim, a user who inherits a recycled identifier could gain access to another user's election permissions, votes, and administrative privileges. While Apache's infrastructure does not currently recycle committer UIDs, the application architecture itself provides no defense-in-depth against this class of identity confusion.

The entire authorization chain depends on the external framework making the correct claim selection—a trust assumption that is neither documented nor verified.

**Remediation:**

The application should explicitly verify that user identity is derived from `iss` + `sub` claims, or at minimum document and validate this at the application layer:

```python
async def basic_info():
    basic = edict()
    # ... flash handling ...

    s = await asfquart.session.read()
    if s:
        # ASVS 10.3.3: Verify identity from non-reassignable claims
        # The session should contain the issuer and subject from the
        # OAuth/OIDC token to prevent identity confusion.
        issuer = s.get('iss')
        subject = s.get('sub')
        
        if not issuer or not subject:
            _LOGGER.error('Session missing iss/sub claims for unique identification')
            quart.abort(401)
        
        # Validate expected issuer
        EXPECTED_ISSUER = 'https://oauth.apache.org'
        if issuer != EXPECTED_ISSUER:
            _LOGGER.error(f'Unexpected token issuer: {issuer}')
            quart.abort(401)
        
        # Use iss+sub combination as the canonical identity,
        # mapping to uid via a verified lookup
        canonical_id = f'{issuer}#{subject}'
        
        basic.update(
            uid=s['uid'],
            canonical_id=canonical_id,
            name=s['fullname'],
            email=s['email'],
        )
    else:
        basic.update(uid=None, canonical_id=None, name=None, email=None)
    
    basic.csrf_token = 'placeholder'
    return basic
```

If the `asfquart` framework cannot be modified to expose `iss` and `sub` in the session, the framework's token-to-session mapping should be audited to confirm that `uid` is derived from the `sub` claim (or equivalent non-reassignable identifier) and the issuer is validated during token exchange.

## Positive Security Patterns

1. **Single OAuth Provider**: The application is configured with a single trusted issuer (`oauth.apache.org`), which reduces but does not eliminate the risk of issuer confusion.

2. **Framework-Level Authentication**: The `asfquart.auth.require` decorator pattern ensures authentication is checked before route handlers execute, preventing accidental unauthenticated access.

3. **Role-Based Authorization**: The application uses role-based checks (`R.committer`, `R.pmc_member`) at the decorator level, providing a layered authorization model.

4. **Voter Eligibility Verification**: The `vote_on_page` function explicitly checks voter eligibility against the `mayvote` table using the user's identifier before allowing access.

5. **Comprehensive Audit Logging**: All state-changing operations log the acting user's `uid`, supporting forensic analysis.

---

## Architecture Observations

1. **Opaque Framework Boundary**: The `asfquart` framework handles the entire OAuth token exchange and session creation pipeline. The application code has zero visibility into how claims are extracted from OAuth tokens and mapped to session fields. This means ASVS 10.3.3 compliance depends entirely on the framework's implementation, which cannot be verified from the provided source code.

2. **Identity Coupling**: The LDAP `uid` attribute (populated by `asf-load-ldap.py`) is used as both the person database primary key and the session-based user identifier. If the OAuth token's `uid` field ever diverges from the LDAP `uid`, authorization decisions would break silently or grant incorrect access.

3. **No Token-Level Resource Server**: This application is not a traditional OAuth Resource Server that validates access tokens per-request. Instead, it's a web application using OAuth for authentication, then relying on server-side sessions. The ASVS 10.3.3 requirement still applies to the identity derivation step, but the attack surface is narrower since tokens aren't transmitted per-request.

## Recommendations Summary

### Immediate Actions (Priority 1)

| Recommendation | Effort | Impact |
|----------------|--------|--------|
| **Audit the `asfquart` framework** to verify that the `uid` session field is derived from non-reassignable token claims (`sub` claim from the OAuth/OIDC token issued by `oauth.apache.org`). Document this mapping explicitly. | Low | High |

### Short-term Actions (Priority 2)

| Recommendation | Effort | Impact |
|----------------|--------|--------|
| **Expose `iss` and `sub` in the session** so that the application can validate the identity derivation chain, even if only as an assertion/logging check. | Medium | High |
| **Add an issuer validation check** in `basic_info()` to confirm the session was established from the expected OAuth provider, preventing any future multi-provider confusion. | Low | Medium |

### Long-term Actions (Priority 3)

| Recommendation | Effort | Impact |
|----------------|--------|--------|
| **Document the identity model**: Explicitly document that `uid` in sessions corresponds to Apache LDAP `uid`, which maps to the OAuth `sub` claim, and that Apache does not recycle committer IDs. This converts an implicit assumption into an auditable policy. | Low | Medium |
