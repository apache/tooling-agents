**Key Issue Identified**

The application uses OAuth Authorization Code grant flow via the `asfquart` framework, delegating to `oauth.apache.org` as the authorization server. Analysis reveals that **PKCE (Proof Key for Code Exchange) parameters are completely absent** from the OAuth flow configuration, leaving the authorization code exchange vulnerable to interception attacks. This represents a critical gap in authorization security controls for a voting system handling ASF member authentication.

#### Finding ASVS-1046-HIGH-001: Missing PKCE Implementation in OAuth Authorization Code Flow

**Classification**

| Attribute | Value |
|-----------|-------|
| **Finding ID** | ASVS-1046-HIGH-001 |
| **Severity** | HIGH |
| **Gap Type** | Type A — Entry point with NO control |

| **CWE** | CWE-862: Missing Authorization |

**Location**

| File | Function | Lines | Component |
|------|----------|-------|-----------|
| `v3/server/main.py` | `create_app()` | 37-43 | OAuth URL template configuration |

**Related Functions Checked**

- `create_app()` in `main.py` — OAuth initialization
- All authenticated route handlers in `pages.py` — Dependent on OAuth session security
- `asfquart.generics.OAUTH_URL_INIT` — Authorization request template
- `asfquart.generics.OAUTH_URL_CALLBACK` — Token request template

**Vulnerable Code**

```python
# File: v3/server/main.py, Lines 37-43
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Technical Description**

The application explicitly overrides the `asfquart` framework's OAuth URL templates, which define the complete parameter set for the OAuth Authorization Code flow. Analysis reveals critical security gaps:

1. **Authorization Request (`OAUTH_URL_INIT`)**: Contains only `state` and `redirect_uri` parameters. The required PKCE parameters are absent:
   - ❌ Missing `code_challenge` parameter
   - ❌ Missing `code_challenge_method` parameter

2. **Token Request (`OAUTH_URL_CALLBACK`)**: Contains only the `code` parameter. The required PKCE validation parameter is absent:
   - ❌ Missing `code_verifier` parameter

The URL templates use fixed format strings with `%s` placeholders only for the existing parameters. This structure prevents PKCE parameters from being injected unless the framework independently modifies URLs post-formatting — which the explicit override contradicts.

**Data Flow Analysis**

```
┌─────────────────────────────────────────────────────────────────┐

│    → Application calls create_app()                             │
│    → OAUTH_URL_INIT template applied (NO code_challenge)        │
```

**Attack Scenario**

An attacker can exploit the missing PKCE controls to perform authorization code interception:

**Prerequisites:**
- Attacker has network visibility (e.g., shared WiFi, compromised router, malicious proxy)
- OR attacker can exploit browser history/logs on shared devices
- OR attacker can exploit referrer header leakage
- OR attacker controls a malicious application with access to redirect endpoints

**Attack Steps:**

1. **Victim initiates login**: User clicks "Login with ASF OAuth" on the voting application
2. **Authorization code issued**: `oauth.apache.org` generates authorization code and redirects:
   ```
   https://steve.apache.org/callback?code=INTERCEPTABLE_CODE&state=abc123
   ```
3. **Attacker intercepts code**: Via one of the following vectors:
   - Network packet capture on shared WiFi
   - Browser history access on shared/compromised device
   - Server access logs if redirect endpoint logging is verbose
   - Referrer header if callback page makes external requests
   - Malicious browser extension monitoring navigation
4. **Attacker replays code**: Submits stolen code to token endpoint:
   ```
   POST https://oauth.apache.org/token?code=INTERCEPTABLE_CODE
   ```
5. **Token issued to attacker**: Without `code_verifier` validation, the AS cannot distinguish legitimate client from attacker
6. **Attacker authenticates as victim**: Gains full session access with victim's identity and permissions

**Proof of Concept**

```bash
# Step 1: Legitimate authorization URL (no PKCE)
https://oauth.apache.org/auth?state=abc123&redirect_uri=https://steve.apache.org/callback

# Step 2: Callback with interceptable authorization code
https://steve.apache.org/callback?code=STOLEN_AUTH_CODE&state=abc123

# Step 3: Attacker replays code (simplified for demonstration)
curl -X POST 'https://oauth.apache.org/token' \
  -d 'code=STOLEN_AUTH_CODE' \
  -d 'redirect_uri=https://steve.apache.org/callback' \
  -d 'client_id=steve_voting_app'

# Result: Token issued with NO code_verifier validation
# → Attacker receives valid access token
# → Attacker can now authenticate as victim
```

**Remediation**

**Immediate Actions Required:**

1. **Implement PKCE Parameter Generation:**

```python
# File: v3/server/main.py
import hashlib
import base64
import secrets

def generate_pkce_pair():
    """
    Generate PKCE code_verifier and S256 code_challenge per RFC 7636.
    
    Returns:
        tuple: (code_verifier, code_challenge)
    """
    # Generate cryptographically random verifier (43-128 characters)
    code_verifier = secrets.token_urlsafe(64)
    
    # Create S256 challenge
    digest = hashlib.sha256(code_verifier.encode('ascii')).digest()
    code_challenge = base64.urlsafe_b64encode(digest).rstrip(b'=').decode('ascii')
    
    return code_verifier, code_challenge
```

2. **Update OAuth URL Templates:**

```python
# File: v3/server/main.py, Lines 37-43 (MODIFIED)

    import asfquart.generics

    # Updated URL templates with PKCE parameters
    asfquart.generics.OAUTH_URL_INIT = (
        'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
        '&code_challenge=%s&code_challenge_method=S256'
    )
    asfquart.generics.OAUTH_URL_CALLBACK = (
        'https://oauth.apache.org/token?code=%s&code_verifier=%s'
    )
    
    # ... rest of configuration
```

3. **Integrate PKCE into OAuth Flow:**

```python

def handle_oauth_callback(request):
```

## Architecture Observations

### OAuth Role Clarification

**Client vs. Server Responsibilities:**

This codebase implements the **OAuth Client (Relying Party)** role, not the authorization server. However, the client's configuration directly determines PKCE feasibility:

```
┌─────────────────────────────────────────────────────────────┐
│                     OAuth Architecture                       │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────────────┐         ┌──────────────────┐         │
│  │  Steve Voting    │         │ oauth.apache.org │         │
│  │  Application     │────────▶│  Authorization   │         │
│  │  (OAuth Client)  │         │      Server      │         │
│  └──────────────────┘         └──────────────────┘         │
│         │                              │                    │
│         │ 1. Auth request              │                    │
│         │    (NO code_challenge) ❌    │                    │
│         │                              │                    │
│         │ 2. Auth code returned        │                    │
│         │    (interceptable) ⚠️         │                    │
│         │                              │                    │
│         │ 3. Token request             │                    │
│         │    (NO code_verifier) ❌     │                    │
│         │                              │                    │
│         ▼                              ▼                    │
│    ⚠️ PKCE gap in client config prevents AS enforcement     │
└─────────────────────────────────────────────────────────────┘
```

**Key Insight:** Even if `oauth.apache.org` supports PKCE, it cannot enforce it unless the client sends PKCE parameters. The client's URL template configuration is the critical control point.

   - OAuth configuration was simplified/rushed
   - OIDC features (which often include PKCE by default) were intentionally avoided
   - Security implications may not have been fully evaluated

3. **Framework Source Unavailability**: Without access to `asfquart` source code, we cannot confirm:
   - Whether PKCE is handled internally despite URL template overrides
   - Whether framework defaults include PKCE
   - Whether OIDC mode provides automatic PKCE

**Recommendation:** Audit `asfquart` framework source to determine if removing URL template overrides would restore PKCE functionality.

### Security vs. Simplicity Trade-off

**Observed Pattern:**

**Assessment:** For a voting system handling authentication for Apache Software Foundation members, the security benefits of PKCE far outweigh the implementation complexity. The current simplicity comes at an unacceptable security cost.
