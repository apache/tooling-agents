## Findings

### Medium Severity

#### ASVS-1053-MED-001: No Authorization Server Issuer Validation Mechanism Configured

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with NO control  
**ASVS Requirement:** 10.5.3  

**Location:**
- **File:** `main.py`
- **Lines:** 37-42
- **Function:** `create_app()`

**Related Functions Checked:**
- `pages.py:83-89` — Session data consumption without provenance validation
- `main.py:44` — Framework initialization (opaque)

**Description:**

The application configures OAuth endpoints via hardcoded URL strings but defines no expected issuer URL and implements no mechanism to validate that authorization server metadata or token responses originate from the expected issuer. The comment "Avoid OIDC" indicates a deliberate bypass of OIDC discovery, which also bypasses the metadata issuer validation this requirement mandates.

**Vulnerable Code:**

```python
# main.py, lines 37-42
# Avoid OIDC
import asfquart.generics

asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**

1. **Source:** OAuth endpoints hardcoded as format strings in `main.py`
2. **Processing:** `asfquart` framework handles the complete OAuth flow (callback, token exchange, session creation)
3. **Sink:** Session data consumed in `pages.py:83-89` without issuer validation:
   ```python
   s = await asfquart.session.read()
   if s:
       basic.update(
           uid=s['uid'],
           name=s['fullname'],
           email=s['email'],
       )
   ```
4. **Missing Control:** No `expected_issuer` configuration; no validation that `iss` claim or metadata issuer matches `https://oauth.apache.org`; no metadata fetching or validation at all

**Evidence Chain:**
- `main.py:37` — Comment `# Avoid OIDC` explicitly skips OIDC discovery
- `main.py:39-42` — OAuth URLs set as raw strings, no issuer configuration
- `pages.py:83-89` — Session data consumed without any provenance validation of which authorization server provided it
- No `.well-known/openid-configuration` or `.well-known/oauth-authorization-server` metadata endpoint is fetched or validated anywhere in the provided code
- No `iss` claim comparison exists in any provided file

**Proof of Concept:**

If an attacker can perform a DNS hijack or man-in-the-middle on the connection to `oauth.apache.org`, or if the `asfquart` framework were to support metadata discovery in the future:

1. Attacker stands up a rogue authorization server at a controlled domain
2. The rogue AS metadata contains `"issuer": "https://oauth.apache.org"` (impersonating the legitimate AS)
3. Without issuer validation, the client would accept the metadata and redirect users to the attacker's authorization endpoint
4. Users authenticate against the attacker's AS, exposing credentials and receiving attacker-controlled tokens
5. The application session is populated with attacker-chosen identity claims

**Impact:**

- **Authorization server impersonation** could allow an attacker to inject arbitrary identity claims into user sessions
- **Identity forgery:** An attacker could authenticate as any ASF member (including PMC members with election creation privileges)
- **Vote manipulation** through impersonated voter identities
- **Election management operations** performed under forged identities

**Remediation:**

Configure an expected issuer URL and validate it against authorization server metadata and token responses:

```python
def create_app():
    import asfquart.generics

    # Configure expected issuer for validation
    EXPECTED_ISSUER = 'https://oauth.apache.org'
    
    asfquart.generics.OAUTH_URL_INIT = (
        f'{EXPECTED_ISSUER}/auth?state=%s&redirect_uri=%s'
    )
    asfquart.generics.OAUTH_URL_CALLBACK = f'{EXPECTED_ISSUER}/token?code=%s'
    
    # If asfquart supports issuer validation, configure it:
    # asfquart.generics.EXPECTED_ISSUER = EXPECTED_ISSUER
    # asfquart.generics.VALIDATE_ISSUER = True
    
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    # If framework doesn't support issuer validation natively,
    # add middleware to validate iss claim in session/tokens:
    @app.before_request
    async def validate_session_issuer():
        s = await asfquart.session.read()
        if s and s.get('iss') != EXPECTED_ISSUER:
            # Reject sessions from unexpected issuers
            await asfquart.session.clear()
            quart.abort(403, 'Invalid authorization server')
    
    import pages
    import api
    return app
```

If migrating to OIDC discovery in the future, ensure the fetched metadata's `issuer` field is compared:

```python
import httpx

async def fetch_and_validate_metadata(expected_issuer: str) -> dict:
    """Fetch AS metadata and validate issuer matches exactly."""
    metadata_url = f"{expected_issuer}/.well-known/openid-configuration"
    async with httpx.AsyncClient() as client:
        resp = await client.get(metadata_url)
        resp.raise_for_status()
        metadata = resp.json()
    
    # ASVS 10.5.3: Exact match required
    if metadata.get('issuer') != expected_issuer:
        raise ValueError(
            f"Issuer mismatch: expected '{expected_issuer}', "
            f"got '{metadata.get('issuer')}'"
        )
    return metadata
```

## Positive Security Patterns

1. **Hardcoded OAuth URLs:** Using hardcoded `https://` URLs for the authorization and token endpoints (`main.py:39-42`) prevents some classes of metadata-based attacks, since the endpoints aren't dynamically derived from potentially-tampered metadata.

2. **TLS enforcement:** OAuth URLs use `https://` scheme, and LDAP uses `ldaps://` (`asf-load-ldap.py:35`), ensuring transport-layer security for credential exchanges.

3. **Authentication decorators consistently applied:** All state-changing endpoints use `@asfquart.auth.require` decorators, ensuring authentication is checked before OAuth session data is consumed.

4. **Framework delegation:** By delegating OAuth to `asfquart`, the application avoids implementing custom OAuth flows, reducing the attack surface for implementation errors.

---

## Architecture Observations

1. **Opaque security dependency:** The `asfquart` framework handles the entire OAuth flow (authorization redirect, callback, token exchange, session creation), but its source code is not provided. This makes it impossible to fully verify ASVS 10.5.3 compliance — the framework may or may not validate the issuer claim internally.

2. **No OIDC discovery:** The explicit "Avoid OIDC" comment and hardcoded URLs mean the application does not fetch authorization server metadata at all. While this incidentally prevents some metadata impersonation attacks, it also means there is no programmatic enforcement of issuer identity — the security relies entirely on DNS and TLS certificate validation.

3. **Single authorization server:** The application is designed for a single trusted authorization server (`oauth.apache.org`). There is no multi-tenant or federated identity scenario, which reduces (but does not eliminate) the risk of AS impersonation.
