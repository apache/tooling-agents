## Findings

### Medium Severity

#### ASVS-1034-MED-001: Missing Authentication Recentness Verification

| Attribute | Value |
|-----------|-------|
| **Severity** | MEDIUM |
| **Gap Type** | Type A — Entry points with NO control |
| **File** | `v3/server/main.py` (lines 37-43), `v3/server/pages.py` (lines 85-95) |
| **ASVS Requirement** | 10.3.4 — Verify authentication recentness via `auth_time` or equivalent |

**Description:** 

The application explicitly disables OIDC and uses plain OAuth, thereby removing the standard mechanism (`auth_time` claim) for verifying authentication recentness. The session object contains only `uid`, `fullname`, and `email` — no authentication timestamp is stored or checked. Sensitive operations (voting, opening/closing elections) proceed without verifying when the user last authenticated.

**Vulnerable Code (main.py:37-43):**

```python
### is this really needed right now?
# Avoid OIDC
import asfquart.generics

asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Session read with no auth_time (pages.py:85-95):**

```python
s = await asfquart.session.read()
if s:
    basic.update(
        uid=s['uid'],
        name=s['fullname'],
        email=s['email'],
    )
# ...
### generate a real token and store in the session
basic.csrf_token = 'placeholder'
```

**Data Flow:**

1. User authenticates via Apache OAuth → session established with `uid`/`fullname`/`email` only
2. No `auth_time` or authentication timestamp stored in session
3. Hours/days later, user accesses `/do-vote/<eid>` → session is still valid
4. Vote is cast without any check on authentication freshness

**Affected Entry Points (sensitive operations with no recentness check):**

| Route | Method | Operation | Risk |
|-------|--------|-----------|------|
| `/do-vote/<eid>` | POST | Cast vote | HIGH — vote integrity |
| `/do-open/<eid>` | GET | Open election | HIGH — state change |
| `/do-close/<eid>` | GET | Close election | HIGH — state change |
| `/do-create-election` | POST | Create election | MEDIUM — resource creation |
| `/do-delete-issue/<eid>/<iid>` | POST | Delete issue | MEDIUM — destructive |

**Related Functions Checked:**
- `asfquart.session.read()` (pages.py:85)
- `do_vote_endpoint()` (pages.py:443-482)
- `do_open_endpoint()` (pages.py:507-525)
- `do_close_endpoint()` (pages.py:528-544)
- `do_create_endpoint()` (pages.py:485-504)

**Proof of Concept:**

1. Attacker obtains a valid session cookie (e.g., via XSS, physical access, or session fixation)
2. Original user authenticated 24+ hours ago
3. Attacker submits `POST /do-vote/<eid>` with the stale session
4. Vote is accepted because no `auth_time` check exists
5. No mechanism detects that the authentication is stale

**Impact:** 

In a voting system, stale sessions can be exploited to cast votes on behalf of another user without requiring recent authentication. This undermines vote integrity — the core security property of the system.

**Remediation:**

```python
# In session creation (after OAuth callback), record auth_time:
import time

async def on_oauth_callback():
    # After successful OAuth validation...
    session_data = {
        'uid': user_id,
        'fullname': full_name,
        'email': email,
        'auth_time': int(time.time()),  # Record authentication timestamp
    }
    await asfquart.session.write(session_data)

# In pages.py, add a recentness check helper:
MAX_AUTH_AGE_VOTING = 3600  # 1 hour for voting operations

async def require_recent_auth(max_age_seconds=MAX_AUTH_AGE_VOTING):
    s = await asfquart.session.read()
    auth_time = s.get('auth_time', 0)
    if (time.time() - auth_time) > max_age_seconds:
        # Redirect to re-authenticate
        return quart.redirect('/login?prompt=login&reason=reauth_required')
    return None

# Apply before sensitive operations:
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    reauth = await require_recent_auth(MAX_AUTH_AGE_VOTING)
    if reauth:
        return reauth
    # ... proceed with voting
```

#### ASVS-1034-MED-002: Missing Authentication Method and Strength Verification

| Attribute | Value |
|-----------|-------|
| **Severity** | MEDIUM |
| **Gap Type** | Type A — Entry points with NO control |
| **File** | `v3/server/pages.py` (multiple endpoints) |
| **ASVS Requirement** | 10.3.4 — Verify authentication method/strength via `acr`/`amr` or equivalent |

**Description:** 

The application has operations of varying sensitivity (viewing elections, voting, managing elections, creating elections) but performs no verification of authentication method or strength. The framework distinguishes `R.committer` from `R.pmc_member` roles but these are authorization checks on group membership — not authentication quality. There is no verification that the user authenticated with an appropriate method (e.g., MFA for administrative operations) regardless of the authentication provider's capabilities.

**Evidence — No authentication method differentiation (pages.py):**

All sensitive operations use the same auth gate without strength differentiation:

```python
# Voting (high integrity requirement) - same auth level as viewing
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})  # Only checks role, not auth method
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    ### check authz          # <-- Even authorization is incomplete
    ...

# Election management (administrative) - same auth level  
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  # Only checks role, not auth method
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz          # <-- Authorization also incomplete
    ...

# Election creation (highest privilege) - checks role but not auth strength
@APP.post('/do-create-election')
@asfquart.auth.require({R.pmc_member})  # Role check only, no acr/amr
async def do_create_endpoint():
    result = await basic_info()
    ### check authz
    ...
```

**Data Flow:**

1. User authenticates via any method Apache OAuth supports (password, SSO, etc.)
2. Session is established with no record of authentication method (`amr`) or context (`acr`)
3. User accesses administrative operations (open/close elections, create elections)
4. No verification that authentication strength matches the sensitivity of the operation
5. A weak authentication method grants the same access as a strong one

**Related Functions Checked:**
- `asfquart.auth.require()` decorator (pages.py, multiple locations)
- `do_vote_endpoint()` (pages.py:443-482)
- `do_open_endpoint()` (pages.py:507-525)
- `do_close_endpoint()` (pages.py:528-544)
- `do_create_endpoint()` (pages.py:485-504)

**Impact:** 

Administrative operations on elections (open, close, create, delete issues) can be performed with any authentication method, including potentially weak ones. For a voting system, this means election integrity relies entirely on the initial authentication quality, which is neither verified nor enforced by the resource server.

**Remediation:**

```python
# If using OIDC (recommended), verify acr/amr claims:
async def require_auth_strength(required_acr=None, required_amr=None):
    """Verify authentication method meets requirements for sensitive operations."""
    s = await asfquart.session.read()
    
    if required_acr:
        actual_acr = s.get('acr', '')
        if actual_acr not in required_acr:
            quart.abort(403, f'Insufficient authentication level. Required: {required_acr}')
    
    if required_amr:
        actual_amr = set(s.get('amr', []))
        if not actual_amr.intersection(required_amr):
            quart.abort(403, f'Required authentication method not used.')

# Usage for election management requiring MFA:
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    await require_auth_strength(required_amr={'mfa', 'otp', 'hwk'})
    # ... proceed
```

## Positive Security Patterns

1. **Role-based authentication decorators**: The consistent use of `@asfquart.auth.require({R.committer})` and `@asfquart.auth.require({R.pmc_member})` shows a structured approach to role separation, providing a foundation for adding strength-based checks.

2. **Tiered authorization concept**: The distinction between `R.committer` (general access) and `R.pmc_member` (election creation) demonstrates awareness that different operations require different privilege levels.

3. **Mayvote access control**: The `vote_on_page` function checks `q_find_issues` to verify voter eligibility before displaying the ballot — a proper resource-level authorization check.

4. **Comprehensive audit logging**: All sensitive operations log the user, action, and target resource, which provides accountability even in the absence of authentication quality checks.

5. **LDAPS usage**: The LDAP script uses `ldaps://` (lines in asf-load-ldap.py), ensuring encrypted communication for identity data.

---

## Architecture Observations

1. **Session-centric rather than token-centric**: The application converts OAuth tokens into server-side sessions immediately after authentication. This means token-based claims (acr, amr, auth_time) are lost at the session boundary. Any remediation must capture these claims during session creation.

2. **OIDC explicitly avoided**: The comment `# Avoid OIDC` in `main.py` and the use of Apache's custom OAuth URLs (`oauth.apache.org`) suggest a deliberate architectural choice. This removes the standard mechanism for conveying authentication quality claims. If Apache's OAuth provider doesn't support OIDC claims, an alternative mechanism (e.g., storing auth timestamp in the session) is needed.

3. **Incomplete authorization model**: The pervasive `### check authz` comments indicate that resource-level authorization (e.g., "does this user own this election?") is not yet implemented. While this falls under different ASVS requirements, it compounds the risk from missing authentication quality checks.

4. **State-changing operations via GET**: `do_open_endpoint` and `do_close_endpoint` use GET requests for state-changing operations, which is an anti-pattern that exacerbates risk when combined with stale sessions.

---

## Recommendations Summary

### Immediate Actions

1. **Store `auth_time` in session** during OAuth callback. Even without OIDC, record `time.time()` when the session is established. (Effort: Low, Impact: Enables recentness checks)

2. **Implement recentness verification** for voting operations (`/do-vote/<eid>`). Require authentication within the last 1 hour before accepting votes. (Effort: Medium, Impact: Protects vote integrity)

### Short-term Actions

3. **Implement the missing `### check authz` checks** to ensure resource-level authorization complements authentication quality checks. (Effort: Medium, Impact: Defense in depth)

4. **Implement step-up authentication** for administrative operations (open/close/create elections) requiring re-authentication or MFA. (Effort: High, Impact: Protects election administration)

### Long-term Actions

5. **Evaluate OIDC adoption** to gain standard `acr`/`amr`/`auth_time` claims from the identity provider. The current `# Avoid OIDC` decision removes the standard mechanism for authentication quality verification. (Effort: High, Impact: Full 10.3.4 compliance)
