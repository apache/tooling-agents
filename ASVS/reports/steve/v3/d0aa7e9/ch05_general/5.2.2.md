## Findings

### Medium Severity

#### ASVS-522-MED-001: Document Serving Lacks Filename Sanitization and Content-Type Enforcement

**Severity:** MEDIUM

**Location:** `v3/server/pages.py`, function `serve_doc`, lines 594–609

**Related Functions Checked:**
- `rewrite_description` (line 47) — generates links to this endpoint
- `send_from_directory` (line 609) — serves files without content validation
- Database authorization check (lines 600-604) — verifies voter eligibility

**Description:**

The `serve_doc` endpoint serves per-issue documents to authorized voters without validating filename extensions or enforcing safe content types. The code contains an explicit TODO comment acknowledging this gap (`### verify the propriety of DOCNAME.`), but the validation was never implemented. This creates a control gap where documents used in business decision workflows (voters reading documents before voting) are served without proper content validation.

While this endpoint *serves* files rather than *accepting uploads*, ASVS 5.2.2 requires applications to validate that "contents correspond to the type represented by the extension." At Level 1, files used for business decisions are in scope, and these documents directly influence voting decisions.

**Vulnerable Code:**

```python
@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require  # fine-grained per-doc authz within the handler
async def serve_doc(iid, docname):
    result = await basic_info()  # get PID  ### grr: called uid

    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        # Nothing fancy. Just pretend the file does not exist
        quart.abort(404)
        # NOTREACHED

    ### verify the propriety of DOCNAME.    # <-- Acknowledged TODO, never implemented

    # Return per-issue document.
    return await quart.send_from_directory(DOCSDIR / iid, docname)
```

**Data Flow:**

```
URL parameter `docname` (user-controlled)
  → No extension allowlist check
  → No content-type enforcement
  → quart.send_from_directory() serves file with MIME type guessed from extension
  → Browser renders content inline with application's origin
```

**Proof of Concept:**

If an administrator (or compromised CLI process) places a file `malicious.html` in `DOCSDIR/<valid_iid>/`, and an issue description references `doc:malicious.html`:

1. An authorized voter views the issue
2. `rewrite_description` generates a link: `<a href="/docs/<iid>/malicious.html">malicious.html</a>`
3. Clicking the link serves `malicious.html` as `text/html` from the application's origin
4. JavaScript in that file executes in the application's security context (same-origin XSS)

**Attack Request:**
```http
GET /docs/<valid_iid>/malicious.html HTTP/1.1
Host: voting.example.org
Cookie: session=<valid_session>
```

**Impact:**

- **Stored XSS Risk:** Files served with HTML/SVG content types could execute JavaScript in the application's security context
- **No Extension Validation:** Any file type can be served without restrictions
- **Content-Type Sniffing:** Browsers may interpret files differently than intended
- **Business Logic Bypass:** Malicious documents could mislead voters in the decision-making process
- **Known Gap:** The acknowledged-but-unimplemented validation represents technical debt that could be exploited if document management is compromised

**Remediation:**

Implement comprehensive filename validation and content-type enforcement:

```python
import mimetypes
import pathlib

# Allowlist of safe file extensions for election documents
ALLOWED_DOC_EXTENSIONS = {'.pdf', '.txt', '.md', '.png', '.jpg', '.jpeg', '.gif'}

# Safe MIME types (force download for anything else)
SAFE_MIME_TYPES = {
    '.pdf': 'application/pdf',
    '.txt': 'text/plain',
    '.md': 'text/plain',
    '.png': 'image/png',
    '.jpg': 'image/jpeg',
    '.jpeg': 'image/jpeg',
    '.gif': 'image/gif',
}

@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require
async def serve_doc(iid, docname):
    result = await basic_info()

    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)

    # Validate filename: no path separators, no special components
    if '..' in docname or '/' in docname or '\\' in docname:
        quart.abort(400, 'Invalid document name')

    # Check file extension against allowlist
    ext = pathlib.Path(docname).suffix.lower()
    if ext not in ALLOWED_DOC_EXTENSIONS:
        quart.abort(403, 'File type not allowed')

    # Validate IID format (should be a hex ID, no path components)
    if '..' in iid or '/' in iid or '\\' in iid:
        quart.abort(400, 'Invalid issue ID')

    # Serve with explicit safe content type and security headers
    response = await quart.send_from_directory(DOCSDIR / iid, docname)
    response.headers['Content-Type'] = SAFE_MIME_TYPES.get(ext, 'application/octet-stream')
    response.headers['Content-Disposition'] = f'inline; filename="{docname}"'
    response.headers['X-Content-Type-Options'] = 'nosniff'
    return response
```

For full ASVS 5.2.2 compliance, add content validation where files are *accepted* into the docs directory (in the CLI/admin tooling):

```python
import magic  # python-magic library
import pathlib

def validate_document(filepath):
    """Validate file content matches its extension before accepting."""
    ext = pathlib.Path(filepath).suffix.lower()
    if ext not in ALLOWED_DOC_EXTENSIONS:
        raise ValueError(f'Extension {ext} not in allowlist')
    
    # Verify magic bytes match claimed type
    detected = magic.from_file(str(filepath), mime=True)
    expected_mimes = {
        '.pdf': 'application/pdf',
        '.png': 'image/png',
        '.jpg': 'image/jpeg',
        '.jpeg': 'image/jpeg',
        '.gif': 'image/gif',
        '.txt': 'text/plain',
        '.md': 'text/plain',
    }
    if detected != expected_mimes.get(ext, 'application/octet-stream'):
        raise ValueError(f'Content mismatch: extension={ext}, detected={detected}')
```

## Positive Security Patterns

### 1. Standard Framework File Serving Functions

**Location:** `v3/server/pages.py:609, 614, 619`

The application consistently uses Quart's `send_from_directory()` function for serving files, which provides built-in path traversal protection for the `docname` parameter within the specified directory. This is a secure baseline that prevents directory traversal attacks like `../../etc/passwd`.

### 2. Authorization Before Document Access

**Location:** `v3/server/pages.py:600-604`

The `serve_doc` endpoint implements fine-grained authorization by checking the `mayvote` table to verify that the requesting voter is eligible for the specific issue before serving documents. This prevents unauthorized access to election materials.

```python
row = db.q_get_mayvote.first_row(result.uid, iid)
if not row:
    quart.abort(404)
```

### 3. No File Upload Attack Surface

**Location:** All POST routes in `v3/server/pages.py`

All POST handlers use `request.form` (text) or `request.get_json()` for data input. No endpoints use `request.files`, which eliminates the primary attack surface for malicious file uploads. This significantly reduces the application's exposure to file-based attacks.

### 4. Safe Path Construction

**Location:** `v3/server/pages.py:37-41`

The application uses `pathlib.Path` for path construction rather than string concatenation, which provides safer path joining and manipulation:

```python
DOCSDIR = pathlib.Path('/path/to/docs')
# Later: DOCSDIR / iid / docname
```

### 5. Authentication on Sensitive Routes

**Location:** All file-serving routes with `@asfquart.auth.require` decorator

File-related endpoints require authentication, ensuring that only authenticated users can access documents. This is combined with the authorization checks for defense-in-depth.

## Architecture Observations

### 1. Document Management is External to Web Layer

The `serve_doc` endpoint and `rewrite_description` function reference documents in `DOCSDIR/<iid>/`, but no web endpoint uploads files to this location. Documents are likely managed through CLI tools or administrative interfaces outside the web application scope.

**Implication:** ASVS 5.2.2 compliance must be verified in the CLI/admin tooling where files are actually accepted into the system. The web layer only serves pre-existing documents, so content validation should occur at the point of document ingestion.

### 2. No Content-Security-Policy for Document Responses

The `serve_doc` endpoint doesn't set `Content-Security-Policy` headers on responses. If documents are served inline (as HTML or SVG), scripts could execute in the application's origin context.

**Recommendation:** Implement CSP headers like `Content-Security-Policy: sandbox` to prevent script execution even if malicious content is served.

### 3. IID Used in Path Construction Without Format Validation

The `iid` URL parameter is used as a directory component (`DOCSDIR / iid`). While the `mayvote` database check provides indirect validation (the IID must exist in the database), explicit format validation of `iid` would be a stronger defense-in-depth measure.

**Note:** The Quart default string converter prevents `/` in `iid`, but `..` is still possible as a value, though the database check would likely fail for invalid IIDs.

### 4. Acknowledged Technical Debt

The explicit TODO comment (`### verify the propriety of DOCNAME.`) indicates that filename validation was planned but never implemented. This represents known technical debt that should be prioritized for remediation.
