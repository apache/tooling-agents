## Findings

### HIGH Severity

#### ASVS-532-HIGH-001: User-Controlled `iid` Used in Directory Path Construction Without Path Validation

**Severity:** HIGH  
**Gap Type:** Type B — Control acknowledged as needed but NOT implemented  
**File:** `v3/server/pages.py`  
**Function:** `serve_doc` (lines 585–600)  
**Related Functions Checked:** `rewrite_description` (L53-60), `serve_static` (L604-606), `serve_favicon` (L609-610)

##### Vulnerable Code

```python
# Line 585-600
@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require  # fine-grained per-doc authz within the handler
async def serve_doc(iid, docname):
    result = await basic_info()

    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)

    ### verify the propriety of DOCNAME.       # <-- TODO: acknowledged missing control

    # Return per-issue document.
    return await quart.send_from_directory(DOCSDIR / iid, docname)  # <-- SINK
```

##### Data Flow

```
Source: URL parameter `iid` → Route /docs/<iid>/<docname>
            ↓
Intermediate: Database check (incidental protection, not path-safety control)
            ↓
Sink: pathlib.Path concatenation → DOCSDIR / iid (directory argument)
            ↓
quart.send_from_directory(DOCSDIR / iid, docname)
            ↓
Missing Control: No explicit path traversal validation on `iid`
                 Note: send_from_directory ONLY validates `docname` (2nd arg),
                       NOT the directory (1st arg)
```

##### Description

The `iid` URL parameter is directly concatenated into a filesystem directory path without explicit validation:

```python
DOCSDIR / iid  # e.g., /app/server/docs/ + user_input
```

Quart's `send_from_directory(directory, filename)` uses Werkzeug's `safe_join` to protect the **`filename`** parameter against traversal, but the **`directory`** parameter is **trusted and not validated**. This means if `iid` contains `..`, the base directory escapes `DOCSDIR`:

- `iid = ".."` → `DOCSDIR / ".."` = `THIS_DIR` (the `server/` directory)
- `docname = "config.yaml"` → serves `server/config.yaml` (server configuration)

**Current indirect protection:** The database query `q_get_mayvote.first_row(result.uid, iid)` returns no row for `iid=".."` because IIDs in the database are cryptographically generated hex strings. This is **incidental protection** — the database check is an authorization check, not a path validation control.

**Why this is still HIGH severity:**

1. **The developer explicitly acknowledged the gap** with `### verify the propriety of DOCNAME.` — this is a Type B gap (control acknowledged but not implemented)
2. **No defense-in-depth** — if the authorization logic changes (e.g., adding a public documents feature, loosening the mayvote check, or adding an admin bypass), the path traversal becomes directly exploitable
3. **The framework does not protect this parameter** — unlike `docname`, the `iid`-constructed directory path has zero framework-level protection
4. **ASVS 5.3.2 requires explicit validation** — "strict validation and sanitization must be applied" to user-submitted data used in file paths

##### Proof of Concept

```http
GET /docs/../config.yaml HTTP/1.1
Cookie: [valid_session_cookie]
```

- `iid` = `..`
- `docname` = `config.yaml`
- Path becomes: `send_from_directory(DOCSDIR / '..', 'config.yaml')` = `send_from_directory(server_dir, 'config.yaml')`
- **Currently blocked** by database check (no mayvote row for `iid=".."`)
- **Would serve `server/config.yaml`** if database check were removed or bypassed

More targeted attack if database check were weakened:
```http
GET /docs/..%00/templates/manage.ezt HTTP/1.1
```
```http
GET /docs/%2e%2e/config.yaml HTTP/1.1
```

##### Impact

- **Configuration disclosure:** Server configuration file (`config.yaml`) containing database paths and potentially sensitive settings
- **Template source disclosure:** EZT templates revealing application logic and variable names
- **Source code disclosure:** If `pages.py` or other Python files are reachable via traversal
- **Database file access:** If the SQLite database path is within reach of the traversed directory

##### Remediation

Add explicit path validation for both `iid` and `docname` before any filesystem operations:

```python
import re

# Strict allowlist pattern for IIDs (crypto-generated hex strings)
SAFE_ID_PATTERN = re.compile(r'^[a-zA-Z0-9_-]+$')
# Strict allowlist for document names
SAFE_DOCNAME_PATTERN = re.compile(r'^[a-zA-Z0-9][a-zA-Z0-9._-]*$')

@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require
async def serve_doc(iid, docname):
    result = await basic_info()

    # ASVS 5.3.2: Validate user-submitted path components
    if not SAFE_ID_PATTERN.match(iid):
        quart.abort(404)
    if not SAFE_DOCNAME_PATTERN.match(docname):
        quart.abort(404)

    # Additional defense-in-depth: resolve and verify containment
    doc_path = (DOCSDIR / iid / docname).resolve()
    if not str(doc_path).startswith(str(DOCSDIR.resolve())):
        quart.abort(404)

    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)

    return await quart.send_from_directory(DOCSDIR / iid, docname)
```

## Positive Security Patterns

1. **Hardcoded base paths:** All base directory constants (`TEMPLATES`, `STATICDIR`, `DOCSDIR`, `DB_FNAME`) are computed from `__file__` using `pathlib.Path.resolve()`, preventing relative path confusion.

2. **`send_from_directory` for static files:** The `serve_static` endpoint correctly uses `send_from_directory(STATICDIR, filename)` with a trusted directory and framework-protected filename — this is the standard secure pattern.

3. **Cryptographically generated IDs:** Both `eid` and `iid` values are generated via `crypto.create_id()`, producing values that inherently cannot contain path traversal characters. This provides a strong secondary defense.

4. **Parameterized database queries:** All SQL operations in `election.py` use prepared statements via `asfpy.db`, preventing SQL injection that could be used to manipulate IID values in the database.

5. **Authorization check before document serving:** The mayvote database check in `serve_doc` ensures only authorized voters can access issue documents, providing both authorization and incidental path safety.

6. **Favicon uses hardcoded filename:** `serve_favicon` passes a literal string `'favicon.ico'` to `send_from_directory`, eliminating any user input from the file path.

---

## Architecture Observations

1. **Document storage model:** Per-issue documents are stored in `server/docs/<iid>/` directories. The `iid` is crypto-generated but used directly in filesystem paths from URL parameters, creating a mismatch between the ID-based data model and the filesystem-based storage.

2. **Single point of file access:** All document serving goes through `serve_doc`, making it the single point where path validation must be enforced. This is architecturally sound — there is one place to fix.

3. **Framework reliance without explicit controls:** The codebase relies on Quart/Werkzeug's `send_from_directory` for path safety but doesn't add application-level validation. The `safe_join` protection only covers the filename parameter, not the directory parameter which includes user input.

4. **TODO comment as risk indicator:** The `### verify the propriety of DOCNAME.` comment at line 597 is a clear indicator that the developer identified the need for validation during development but deferred implementation. This is a tracked technical debt item that directly impacts ASVS compliance.

## Recommendations Summary

### Immediate (Priority 1)

**Add explicit path validation to `serve_doc`:**
- Validate `iid` against an allowlist pattern (alphanumeric + limited special chars)
- Validate `docname` against an allowlist pattern (no `..`, no path separators)
- Add `resolve()`-based containment check as defense-in-depth
- Perform validation **before** any filesystem operations or database queries

### Short-term (Priority 2)

**Normalize the document serving architecture:**
- Consider mapping document access through database records rather than filesystem paths (e.g., store document metadata in the database and serve by internal ID)
- If filesystem storage is retained, validate that the resolved path is contained within `DOCSDIR` using `pathlib.Path.resolve()` and `str.startswith()`

### Long-term (Priority 3)

**Resolve all `### verify` / `### check` TODO comments:**
- The codebase contains multiple `### check authz` comments indicating unimplemented authorization checks
- Create tracking issues for each TODO and prioritize implementation
- Establish a policy that security-related TODOs must be resolved before deployment
