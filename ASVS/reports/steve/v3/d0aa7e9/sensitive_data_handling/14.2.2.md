## Findings

### HIGH Severity

#### ASVS-1422-HIGH-001: Complete Absence of Cache-Control Headers on All Sensitive Endpoints

**Severity:** HIGH  
**Gap Type:** Type A — Entry points with NO control  
**File:** `v3/server/pages.py` (affects all route handlers)

**Description:**

Every authenticated endpoint serving sensitive data returns responses without `Cache-Control`, `Pragma`, or `Expires` headers. Per HTTP/1.1 (RFC 7234), responses without explicit cache directives are cacheable by intermediaries. Any load balancer, CDN, or reverse proxy fronting this application may cache and serve authenticated pages to unauthorized users.

**Affected Endpoints (all sensitive):**

| Route | Line | Sensitive Data Exposed |
|-------|------|----------------------|
| `GET /voter` | ~137 | Voter eligibility, election list per user |
| `GET /vote-on/<eid>` | ~220 | **Ballot content**, candidate lists, voter-election mapping |
| `GET /admin` | ~283 | Owned elections, person data |
| `GET /manage/<eid>` | ~320 | Election metadata, issue details |
| `GET /manage-stv/<eid>/<iid>` | ~343 | STV issue data, candidates |
| `GET /profile` | ~530 | User PII (name, email, uid) |
| `GET /settings` | ~540 | User settings |
| `GET /docs/<iid>/<docname>` | ~555 | Authorization-protected election documents |

**Related Functions Checked:**
- `basic_info()` (line ~60) — Populates base template data, no cache headers set
- `voter_page()` (line ~137) — Returns voter eligibility data
- `vote_on_page()` (line ~220) — Returns ballot content
- `admin_page()` (line ~283) — Returns election ownership data
- `manage_page()` (line ~320) — Returns election metadata
- `manage_stv_page()` (line ~343) — Returns STV issue data
- `profile_page()` (line ~530) — Returns user PII
- `settings_page()` (line ~540) — Returns user settings

**Vulnerable Code:**

```python
# v3/server/pages.py, line ~60
async def basic_info():
    """Return base-level EZT template data."""
    basic = edict()
    # ... populates uid, name, email from session ...
    basic.csrf_token = 'placeholder'
    return basic
    # NOTE: No Cache-Control headers set on the response object
```

```python
# v3/server/pages.py, line ~220
@APP.get('/vote-on/<eid>')
@asfquart.auth.require({R.committer})
@load_election
@APP.use_template(TEMPLATES / 'vote-on.ezt')
async def vote_on_page(election):
    result = await basic_info()
    # ... builds ballot with issues, candidates, voter eligibility ...
    result.issues = issues_stv + issues_yna
    return result
    # Response rendered from template, no Cache-Control headers
```

**Data Flow:**
1. User requests `/vote-on/<eid>` → authentication verified
2. Server builds response containing ballot issues, candidate lists, voter-election mapping
3. Response sent **without `Cache-Control: no-store`**
4. Intermediate load balancer / CDN may cache the full HTML response
5. Subsequent request (same URL) may be served from cache, potentially to a different user or after session expiration

**Proof of Concept:**

```http
# Request from authenticated user
GET /vote-on/a1b2c3d4e5 HTTP/1.1
Host: voting.example.org
Cookie: session=<valid_token>

# Response (observed)
HTTP/1.1 200 OK
Content-Type: text/html
# ← No Cache-Control header
# ← No Pragma header
# ← Body contains: ballot issues, candidates, voter eligibility

# A caching reverse proxy (e.g., Varnish, nginx proxy_cache, AWS ALB)
# stores this response. A subsequent request to the same URL 
# could be served from cache without authentication verification.
```

**Impact:**
- **Ballot exposure:** Cached voting pages reveal which issues a voter is eligible for, leaking voter-to-election correlations that undermine ballot secrecy
- **Proxy cache poisoning:** A shared cache (CDN, corporate proxy) could serve one voter's personalized ballot page to another user
- **Persistent data exposure:** After logout, cached pages remain in intermediate caches containing user identity and ballot information
- **CSRF token leakage:** The (placeholder) CSRF token embedded in cached pages could be reused

**Remediation:**

Add an after-request handler to set `Cache-Control: no-store` on all authenticated responses:

```python
# v3/server/pages.py — Add near the top, after APP initialization

@APP.after_request
async def set_cache_control(response):
    """Prevent caching of sensitive responses by server components."""
    # Check if this is an authenticated/sensitive response
    # Static assets (CSS/JS) should still be cacheable
    if response.content_type and 'text/html' in response.content_type:
        response.headers['Cache-Control'] = 'no-store, no-cache, must-revalidate, max-age=0'
        response.headers['Pragma'] = 'no-cache'
        response.headers['Expires'] = '0'
    return response
```

Or more granularly, protect all authenticated routes:

```python
# Alternative: decorator for sensitive routes
def no_cache(func):
    @functools.wraps(func)
    async def wrapper(*args, **kwargs):
        response = await func(*args, **kwargs)
        if isinstance(response, quart.Response):
            response.headers['Cache-Control'] = 'no-store'
            response.headers['Pragma'] = 'no-cache'
        return response
    return wrapper
```

### MEDIUM Severity

#### ASVS-1422-MED-001: Authorization-Protected Documents Served via `send_from_directory` Without Cache Prevention

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with NO control  
**File:** `v3/server/pages.py`  
**Function:** `serve_doc`  
**Lines:** ~555-565

**Description:**

The `/docs/<iid>/<docname>` endpoint serves election documents after verifying voter eligibility via the `mayvote` table. However, `quart.send_from_directory()` uses framework defaults which typically set `Cache-Control: public` or include `max-age` based on `SEND_FILE_MAX_AGE_DEFAULT` config. This actively **encourages** intermediate caches to store authorization-protected documents.

**Related Functions Checked:**
- `serve_doc()` (lines ~555-565) — Serves authorization-protected documents
- Database query `q_get_mayvote.first_row()` — Verifies voter eligibility

**Vulnerable Code:**

```python
# v3/server/pages.py, lines ~555-565
@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require
async def serve_doc(iid, docname):
    result = await basic_info()

    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)

    ### verify the propriety of DOCNAME.

    # Return per-issue document — no cache headers set
    return await quart.send_from_directory(DOCSDIR / iid, docname)
```

**Data Flow:**
1. Eligible voter requests `/docs/<iid>/ballot_details.pdf`
2. Authorization check passes (mayvote row exists)
3. `send_from_directory` returns the file with **default caching headers**
4. Load balancer / CDN caches the document with the URL as key
5. Non-eligible user requests same URL → receives cached document (bypassing authorization)

**Impact:**
- Election documents containing ballot details, candidate information, or voting instructions could be served from cache to unauthorized users
- Authorization check is bypassed when response is served from an intermediate cache

**Remediation:**

```python
@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require
async def serve_doc(iid, docname):
    result = await basic_info()

    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)

    response = await quart.send_from_directory(DOCSDIR / iid, docname)
    response.headers['Cache-Control'] = 'no-store, no-cache, must-revalidate, private'
    response.headers['Pragma'] = 'no-cache'
    response.headers['Expires'] = '0'
    return response
```

## Architecture Observations

1. **No response middleware layer:** The application lacks any `@APP.after_request` handler, meaning there is no centralized point where security headers (Cache-Control, HSTS, X-Content-Type-Options, etc.) can be applied. This is a structural gap that goes beyond caching.

2. **Template rendering via decorator:** The `@APP.use_template()` decorator handles response rendering, but provides no hook for setting response headers. This architectural choice makes it harder to add per-route cache control.

3. **`send_from_directory` for auth-protected content:** Using a static file serving function for authorization-gated content is inherently risky because the function is designed for public static assets and its default caching behavior works against security.

4. **State-changing GET endpoints:** `/do-open/<eid>` and `/do-close/<eid>` use GET method, making them susceptible to caching by proxies that cache GET responses. While the primary concern is CSRF (acknowledged as known debt), from a caching perspective these redirects could also be cached.
