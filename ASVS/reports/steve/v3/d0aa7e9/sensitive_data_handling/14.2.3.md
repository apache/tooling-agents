## Findings

### Medium Severity

#### ASVS-1423-MED-001 — External Image Loaded on All Pages Leaks Voter Activity Metadata to Third-Party Server

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/templates/header.ezt`
- **Line:** ~22 (in `<nav>` element)
- **Function Context:** Included in all page templates via `[include "header.ezt"]`

**Related Functions Checked:**
- All template rendering endpoints in `pages.py`
- Template inclusion mechanism in EZT templating system
- HTTP header configuration in application middleware

**Description:**

The application's navigation header includes an external image resource loaded from `https://www.apache.org/foundation/press/kit/feather.svg`. This image is automatically fetched by the browser on every page load, including sensitive voting pages. The HTTP request to apache.org transmits voter metadata outside the application's control, creating an externally-observable record of voting activity.

**Vulnerable Code:**

```html
<!-- v3/server/templates/header.ezt, line ~22 -->
<a class="navbar-brand" href="/">
    <img src="https://www.apache.org/foundation/press/kit/feather.svg" 
         alt="Logo" 
         width="30" 
         height="30" 
         class="d-inline-block align-text-top">
    Apache STeVe
</a>
```

**Data Flow:**

1. Voter authenticates successfully and navigates to any page (e.g., `/vote-on/<eid>`, `/voter`, `/manage/<eid>`)
2. Server renders page with `[include "header.ezt"]` directive
3. Browser receives HTML containing external image URL
4. Browser automatically issues: `GET https://www.apache.org/foundation/press/kit/feather.svg`
5. HTTP request transmits to apache.org web server:
   - **Voter IP address** (network layer)
   - **User-Agent** header (browser fingerprinting data)
   - **Referer** header (origin URL, potentially including election ID)
   - **Precise timestamp** of page access (server logs)
6. No `referrerpolicy` attribute is set on the `<img>` tag
7. No `Referrer-Policy` HTTP header or `<meta>` tag is configured in the application
8. Apache.org's web server logs record: `<voter_ip> - - [timestamp] "GET /foundation/press/kit/feather.svg" ... "Referer: https://voting.example.org/vote-on/<eid>"`

**Proof of Concept:**

```http
# Step 1: Voter visits ballot page
GET /vote-on/a1b2c3d4e5f6 HTTP/1.1
Host: voting.example.org
Cookie: session=<session_token>

# Step 2: Browser automatically issues external request
GET /foundation/press/kit/feather.svg HTTP/1.1
Host: www.apache.org
Referer: https://voting.example.org/vote-on/a1b2c3d4e5f6
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36
X-Forwarded-For: <voter_ip_address>
```

**Apache.org server logs now contain:**
```
203.0.113.42 - - [22/Apr/2026:14:32:15 +0000] "GET /foundation/press/kit/feather.svg HTTP/1.1" 200 1523 "https://voting.example.org/vote-on/a1b2c3d4e5f6" "Mozilla/5.0..."
```

This creates an external record correlating:
- Voter network identity (IP: 203.0.113.42)
- Voting system origin (voting.example.org)
- Specific election being accessed (eid: a1b2c3d4e5f6)
- Precise timing (14:32:15 UTC)

**Impact:**

1. **Voter Privacy Violation:** Voter IP addresses are transmitted to a server outside the application's control on every page load, including sensitive ballot pages. This creates an externally-observable record of voting activity.

2. **Timing Correlation Risk:** In low-electorate elections, timing data revealing when specific voters are active on the voting system could contribute to voter-vote correlation when combined with other signals (e.g., vote tallies published immediately after submission).

3. **Uncontrolled Data Retention:** The application cannot control apache.org's logging policies, retention periods, or data sharing practices. This data may be retained indefinitely or shared with third parties.

4. **Ballot Secrecy Compromise:** The system's core security goal is ballot secrecy. Creating an external record that correlates network identities with voting activity timing undermines this goal, even if vote content itself is not exposed.

5. **Regulatory Compliance Risk:** Depending on jurisdiction, transmitting voter activity data to third parties may violate election privacy regulations or data protection laws (e.g., GDPR's data minimization principle).

6. **ASVS 14.2.3 Non-Compliance:** The requirement explicitly states "defined sensitive data is not sent to untrusted parties." Voter IP addresses and activity timing constitute sensitive data in a voting context, and apache.org (while trustworthy) is an "untrusted party" from the application's data protection perspective.

**Remediation:**

**Immediate Fix (Priority 1):**

Download and host the Apache feather logo locally:

```bash
# Download the image to static assets directory
curl -o v3/server/static/img/feather.svg \
  https://www.apache.org/foundation/press/kit/feather.svg
```

Update `header.ezt`:

```html
<!-- v3/server/templates/header.ezt — replace external URL with local copy -->
<a class="navbar-brand" href="/">
    <img src="/static/img/feather.svg" 
         alt="Logo" 
         width="30" 
         height="30" 
         class="d-inline-block align-text-top">
    Apache STeVe
</a>
```

**Defense-in-Depth (Priority 2):**

Add a `Referrer-Policy` header at the application level to protect against any future external resources:

```python
# In application startup / middleware (e.g., pages.py or main application file)
@APP.after_request
async def set_security_headers(response):
    """Set security headers on all responses."""
    response.headers['Referrer-Policy'] = 'same-origin'
    # Optionally add CSP to prevent future external resource inclusion
    response.headers['Content-Security-Policy'] = "default-src 'self'; img-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'"
    return response
```

**Verification:**

After remediation, verify:
1. All pages load without external HTTP requests (check browser DevTools Network tab)
2. `Referrer-Policy: same-origin` header is present in all responses
3. Logo displays correctly on all pages
4. No console errors related to image loading

## Positive Security Patterns

The audit identified several exemplary security practices that effectively prevent sensitive data transmission to untrusted parties:

### 1. Local Asset Serving

**Implementation:** All JavaScript and CSS resources are served from `/static/` directory.

**Evidence:**
- Bootstrap CSS/JS served locally
- Bootstrap Icons served locally
- SortableJS (`Sortable.min.js`) served locally from `/static/js/`
- Custom application scripts served locally

**Security Benefit:** Eliminates third-party CDN tracking. External CDNs can log IP addresses, User-Agents, and timing data for every asset request. Local serving ensures no voter activity metadata is transmitted to CDN providers.

**Notable Detail:** `vote-on.ezt` includes SRI (Subresource Integrity) attribute on `Sortable.min.js` despite being locally served:
```html
<script src="/static/js/Sortable.min.js" 
        integrity="sha384-..." 
        crossorigin="anonymous"></script>
```
This demonstrates security awareness and defense-in-depth, protecting against compromised static file servers.

### 2. Vote Content Exclusion from Logs

**Implementation:** The `do_vote_endpoint` function (pages.py:286-290) logs vote submission events without including vote content.

**Evidence:**
```python
# pages.py, line ~286
_LOGGER.info(f'User[U:{result.uid}] voted on issue[I:{iid}] in election[E:{election.eid}]')
```

**Security Benefit:** Prevents vote content from appearing in application logs, which may be:
- Stored in external log aggregation systems
- Backed up to third-party cloud storage
- Accessible to system administrators
- Subject to legal discovery requests

Only metadata (user ID, issue ID, election ID) is logged, never the votestring itself.

### 3. Cryptographic Material Exclusion from API Responses

**Implementation:** Data retrieval functions explicitly exclude sensitive cryptographic material.

**Evidence:**

`election.py:153-167` — `get_metadata()`:
```python
def get_metadata(self, eid):
    """Get election metadata, excluding cryptographic secrets."""
    row = self.q_get_metadata.first_row(eid)
    if not row:
        return None
    # Explicitly construct dict without salt/opened_key
    return EasyDict({
        'eid': row[0],
        'title': row[1],
        'owner': row[2],
        'monitors': row[3],
        # salt and opened_key deliberately omitted
    })
```

`election.py:169-175` — `get_issue()`:
```python
def get_issue(self, iid):
    """Get issue data, excluding salt."""
    row = self.q_get_issue.first_row(iid)
    if not row:
        return None
    return EasyDict({
        'iid': row[0],
        'title': row[1],
        # salt deliberately omitted
    })
```

`persondb.py:36-43` — `get_person()`:
```python
def get_person(self, uid):
    """Get person data, excluding password and salt."""
    row = self.q_get_person.first_row(uid)
    if not row:
        return None
    return EasyDict({
        'uid': row[0],
        'name': row[1],
        # password and salt deliberately omitted
    })
```

**Security Benefit:** Prevents accidental leakage of cryptographic secrets through:
- Template rendering (if developer accidentally includes `{{election.salt}}`)
- API responses (if endpoint returns full object)
- Error messages (if exception handler serializes object)
- Debug output (if object is printed/logged)

This is a **secure-by-design** pattern that makes cryptographic material leakage structurally impossible at the application boundary.

## Architecture Observations

### 1. Missing Referrer-Policy Configuration

**Observation:** No `Referrer-Policy` HTTP header or `<meta name="referrer">` tag is configured.

**Current Behavior:** Modern browsers default to `strict-origin-when-cross-origin`, which limits cross-origin referrer to origin-only (not full URL). However, this is browser-dependent and not guaranteed.

**Risk:** Without explicit policy:
- Older browsers may send full URL in `Referer` header to external resources
- Election IDs in URLs (`/vote-on/<eid>`) could be transmitted to external parties
- Future external resources could receive more data than intended

**Recommendation:** Implement explicit `Referrer-Policy: same-origin` to ensure referrer is only sent to same-origin requests. This provides defense-in-depth against future external resource inclusion.

### 2. Debug Print Statements in Production Code

**Observation:** Two endpoints contain `print('FORM:', form)` statements:
- `do_add_issue_endpoint` (pages.py, line ~395)
- `do_edit_issue_endpoint` (pages.py, line ~411)

**Current Behavior:** These statements output election management form data to stdout.

**Risk:**
- Stdout may be captured by external logging infrastructure (e.g., Docker logs, systemd journal, cloud logging services)
- Form data includes election titles, descriptions, candidate names, and CSRF tokens
- While not vote content, this creates an uncontrolled data channel outside the application's logging framework
- External logging systems may have different retention policies or access controls

**Data Exposed:**
```python
# Example output to stdout
FORM: {'title': 'Board Election 2026', 'description': '...', 'candidates': '...', 'csrf_token': '...'}
```

**Recommendation:** Replace `print()` statements with structured logging at DEBUG level:
```python
_LOGGER.debug('Election form data: %s', {
    'title': form.get('title'),
    'issue_count': len(form.getlist('candidates')),
    # Exclude sensitive details from logs
})
```

### 3. Cache-Control Headers Not Observed

**Observation:** The provided code does not show explicit `Cache-Control` headers on voting-related responses.

**Current Behavior:** Without explicit cache control, intermediate proxies may cache voting pages.

**Risk:**
- Proxy caches could store ballot pages containing:
  - Candidate names
  - Issue descriptions
  - CSRF tokens (though these expire)
- Cached pages could be served to wrong users
- Sensitive election data could persist in proxy caches beyond election period

**Recommendation:** Add `Cache-Control: no-store, no-cache, must-revalidate` to all voting-related responses:
```python
@APP.after_request
async def set_cache_headers(response):
    """Prevent caching of sensitive pages."""
    if request.path.startswith(('/vote-on/', '/voter', '/manage/')):
        response.headers['Cache-Control'] = 'no-store, no-cache, must-revalidate, max-age=0'
        response.headers['Pragma'] = 'no-cache'
        response.headers['Expires'] = '0'
    return response
```

### 4. No Content Security Policy

**Observation:** No `Content-Security-Policy` (CSP) header is configured.

**Current Behavior:** Without CSP, the application cannot prevent future developers from accidentally introducing external resources.

**Risk:**
- Future code changes could introduce external scripts, images, or stylesheets
- XSS vulnerabilities could be exploited to load external tracking code
- No technical control prevents external resource inclusion

**Recommendation:** Implement restrictive CSP:
```python
response.headers['Content-Security-Policy'] = (
    "default-src 'self'; "
    "img-src 'self'; "
    "script-src 'self'; "
    "style-src 'self' 'unsafe-inline'; "  # Bootstrap requires inline styles
    "font-src 'self'; "
    "connect-src 'self'; "
    "frame-ancestors 'none'; "
    "base-uri 'self'; "
    "form-action 'self'"
)
```

This provides defense-in-depth by making external resource inclusion structurally impossible at the browser level.

### 5. Election ID Exposure in URLs

**Observation:** Election IDs appear in URLs (e.g., `/vote-on/<eid>`, `/manage/<eid>`).

**Current Behavior:** Election IDs are random hex strings (e.g., `a1b2c3d4e5f6`), not sequential integers.

**Security Assessment:** ✅ **Acceptable** — Random election IDs prevent enumeration attacks and don't reveal sensitive information. However, they may appear in:
- Browser history
- Proxy logs
- `Referer` headers (if external resources present)

**Current Mitigation:** The use of random hex IDs (vs. sequential integers) prevents:
- Election enumeration
- Inference of election count
- Timing attacks based on ID sequence

**No Action Required:** This is an acceptable trade-off. Alternative approaches (e.g., session-based election access without IDs in URLs) would significantly complicate the architecture without meaningful security benefit, given that:
1. Election IDs are not secret (elections are visible to eligible voters)
2. Access control is enforced server-side via `mayvote` checks
3. Random IDs prevent enumeration

        content = template_file.read_text()
        matches = external_pattern.findall(content)
        assert not matches, f'{template_file.name} contains external URLs: {matches}'
