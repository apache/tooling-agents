## Findings

### MEDIUM Severity

#### Finding ASVS-1524-MED-001

**Title:** Dependency confusion risk for ASF-namespaced internal package `asfquart`

**Location:**
- **File:** `v3/server/main.py`
- **Lines:** 32-38
- **Function:** Module-level imports and application initialization

**Related Functions Checked:**
- `asfquart.construct()` (line 38)
- `asfquart.generics.OAUTH_URL_INIT` configuration (lines 34-35)
- `asfquart.generics.OAUTH_URL_CALLBACK` configuration (line 36)

**Description:**

The `asfquart` package is an ASF-internal web framework wrapper that provides critical security infrastructure including OAuth integration, authentication, and application construction. This package presents an elevated dependency confusion risk.

If `asfquart` is distributed via an internal ASF package repository and the name is **not** defensively registered on PyPI:

1. An attacker could register `asfquart` on PyPI with a higher version number
2. If `pip` or `uv` is configured with `--extra-index-url` (adding internal repo alongside PyPI), the public malicious package could be preferred due to version precedence
3. The malicious package would execute during import, with full access to the OAuth configuration, authentication flow, and application construction

No configuration restricting the package index source (e.g., `--index-url` pointing exclusively to an internal repo, or `uv` workspace config with explicit source restrictions) was provided for audit.

**Vulnerable Code:**

```python
import asfquart.generics

asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'

app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
```

**Data Flow:**

```
pip/uv install 
  → resolves `asfquart` 
  → no source restriction 
  → could fetch from public PyPI instead of ASF internal 
  → malicious code executes at `import asfquart.generics`
```

**Proof of Concept:**

An attacker could:
1. Register `asfquart` version 99.0.0 on PyPI
2. Include malicious code in `__init__.py` that exfiltrates environment variables
3. When the application installs dependencies without explicit source restrictions, the malicious package is installed
4. OAuth credentials and application secrets are compromised at import time

**Impact:**

Complete application compromise. A malicious `asfquart` package could:
- Intercept OAuth tokens
- Modify authentication flows
- Exfiltrate voter data
- Manipulate election results

This is the foundational framework of the application, making it the highest-value target for a supply chain attack.

**Remediation:**

```toml
# pyproject.toml - Example using uv workspace sources
[tool.uv.sources]
asfquart = { index = "asf-internal" }
asfpy = { index = "asf-internal" }

[[tool.uv.index]]
name = "asf-internal"
url = "https://packages.apache.org/simple/"
explicit = true  # Only use this index for packages explicitly mapped to it
```

Additionally:
- Defensively register the `asfquart` package name on PyPI (even as an empty placeholder) to prevent name squatting
- Configure `uv` or `pip` to use `--index-url` exclusively for ASF packages, preventing fallback to public PyPI
- Document the expected repository source for all internal packages

#### Finding ASVS-1524-MED-002

**Title:** No dependency version pinning or integrity verification visible in codebase

**Location:**
- **File:** `v3/server/main.py`
- **Line:** 1
- **Function:** Script execution configuration

**Related Functions Checked:**
- All import statements across `main.py`, `crypto.py`, and `election.py`
- Package manager configuration (expected but not found)

**Description:**

The shebang indicates the `uv` package manager is used with inline script mode (PEP 723):

```python
#!/usr/bin/env -S uv run --script
```

However, **no inline metadata block** specifying dependencies with version constraints is present in the file. Additionally, no `requirements.txt`, `pyproject.toml` `[project.dependencies]` section, `uv.lock`, `poetry.lock`, or `Pipfile.lock` was provided for audit review.

**Vulnerable Code:**

```python
#!/usr/bin/env -S uv run --script
# Expected but absent — PEP 723 inline metadata:
# /// script
# requires-python = ">=3.11"
# dependencies = [
#     "asfquart==1.2.3",
#     "cryptography==43.0.0",
#     "argon2-cffi==23.1.0",
#     "easydict==1.13",
# ]
# ///
```

**Data Flow:**

```
Application startup 
  → uv resolves dependencies 
  → no version constraints 
  → latest versions pulled from index 
  → potentially malicious or vulnerable versions installed
```

**Proof of Concept:**

Without version pins:
1. A vulnerability is discovered in `cryptography` 42.0.0
2. Version 43.0.0 is released with a fix
3. An attacker compromises PyPI and pushes malicious version 43.0.1
4. Next deployment automatically pulls the malicious version
5. Cryptographic operations are compromised

**Impact:**

Without version pins:
- Any dependency update (including malicious ones pushed to PyPI) would be automatically adopted
- Builds are not reproducible — different environments may resolve different versions
- No integrity verification means a compromised package mirror could serve tampered packages

Without a lock file with integrity hashes:
- The `cryptography` package (providing Fernet, HKDF) could be replaced with a backdoored version
- The `argon2-cffi` package (providing Argon2 hashing for vote tokens and opened keys) could be weakened

**Remediation:**

**Option 1 — `uv` lock file with hashes:**
```bash
# Generate locked dependencies
uv lock
# Verify: uv.lock should contain sha256 hashes for all packages
```

**Option 2 — `requirements.txt` with hashes:**
```text
cryptography==43.0.0 \
    --hash=sha256:0a4f28...
argon2-cffi==23.1.0 \
    --hash=sha256:879c3e...
asfpy==0.48 \
    --hash=sha256:abc123...
easydict==1.13 \
    --hash=sha256:def456...
```

**Option 3 — PEP 723 inline metadata in `main.py`:**
```python
# /// script
# requires-python = ">=3.11"
# dependencies = [
#     "asfquart>=1.0,<2.0",
#     "cryptography>=43.0,<44.0",
#     "argon2-cffi>=23.1,<24.0",
#     "easydict>=1.13,<2.0",
# ]
# ///
```

#### Finding ASVS-1524-MED-003

**Title:** No SBOM documenting transitive dependency tree

**Location:**
- **File:** Project root (expected location)
- **Lines:** N/A
- **Function:** Build and deployment pipeline

**Related Functions Checked:**
- All third-party package imports
- Transitive dependency chains for `cryptography`, `argon2-cffi`, `asfquart`, `asfpy`, and `easydict`

**Description:**

The application's direct dependencies pull in significant transitive dependency chains. None of these transitive dependencies are documented in the provided audit materials.

| Direct Dependency | Known Transitive Dependencies |
|-------------------|-------------------------------|
| `cryptography` | `cffi`, `pycparser`, OS-level OpenSSL bindings |
| `argon2-cffi` | `argon2-cffi-bindings`, `cffi`, `pycparser` |
| `asfquart` | `quart`, `hypercorn`, `h11`, `h2`, `wsproto`, `priority`, `hpack`, and more |
| `asfpy` | `PyYAML`, `requests`, `ldap3`, and others |
| `easydict` | (none expected, but unverified) |

Without an SBOM:
- Vulnerabilities in transitive dependencies (e.g., a CVE in `h2` or `cffi`) cannot be tracked
- The full attack surface of the application is unknown
- Compliance with ASVS 15.2.4's requirement to verify "all of their transitive dependencies" cannot be satisfied

**Vulnerable Code:**

N/A — This is an absence of documentation rather than vulnerable code.

**Data Flow:**

```
Direct dependency installation 
  → transitive dependencies resolved automatically 
  → no visibility into transitive tree 
  → vulnerable or malicious transitive dependencies undetected
```

**Proof of Concept:**

If `pycparser` (a transitive dependency of both `cryptography` and `argon2-cffi`) were compromised:
1. Attacker pushes malicious version to PyPI
2. During installation, `cffi` pulls the malicious `pycparser`
3. Malicious code injects backdoors during C extension compilation
4. Both cryptographic libraries are compromised at build time
5. No detection occurs because `pycparser` is not tracked

**Impact:**

A compromised or vulnerable transitive dependency would go undetected. Transitive dependencies can:
- Introduce vulnerabilities that affect the application
- Be compromised to inject malicious code during installation
- Create hidden attack vectors not visible in direct dependency audits

**Remediation:**

**Generate and maintain an SBOM:**

```bash
# Using CycloneDX (recommended for security auditing)
pip install cyclonedx-bom
cyclonedx-py environment -o sbom.json --format json

# Or using syft
syft dir:./v3 -o cyclonedx-json > sbom.json
```

**Integrate SBOM generation into CI/CD:**

```yaml
# .github/workflows/sbom.yml
- name: Generate SBOM
  run: cyclonedx-py environment -o sbom.json
  
- name: Audit SBOM
  run: grype sbom:sbom.json --fail-on medium
  
- name: Archive SBOM
  uses: actions/upload-artifact@v3
  with:
    name: sbom
    path: sbom.json
```

**Additional steps:**
- Store SBOM artifacts with each release
- Implement automated vulnerability scanning against the SBOM
- Review transitive dependency changes during dependency updates

## Architecture Observations

1. **Minimal dependency footprint for cryptographic core**: The `crypto.py` module uses only `cryptography`, `argon2-cffi`, and standard library modules. This is a sound architectural decision that minimizes the supply chain attack surface for the most sensitive operations.

2. **`asfquart` as a trust anchor**: The entire application's authentication and authorization flows depend on `asfquart`. This makes it the highest-value target for a dependency confusion attack. Its provenance must be verifiable.

3. **`easydict` convenience vs. risk tradeoff**: The `easydict` package (last PyPI release: 2021) is used only for attribute-style dict access in `election.py`. This is a low-complexity dependency but introduces an additional supply chain node. Consider whether `types.SimpleNamespace` or Python 3.10+ structural pattern matching could replace it.

4. **`uv` package manager**: The use of `uv` (shebang line) is positive for dependency resolution speed and lock file support, but the `uv` configuration enforcing repository restrictions must exist outside the audited files.
