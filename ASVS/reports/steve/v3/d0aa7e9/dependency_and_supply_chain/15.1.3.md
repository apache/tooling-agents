### Key Issues Identified

ASVS 15.1.3 requires documentation identifying time-consuming or resource-demanding functionality, defenses against availability loss, and strategies to avoid response times exceeding consumer timeouts. The audited codebase contains **multiple high-cost cryptographic operations (Argon2 with 64MB memory allocation)** executed in web request paths, with **no documentation identifying these as resource-intensive, no documented defenses, and no timeout avoidance strategies**. Furthermore, synchronous CPU-bound operations are called within an async framework without appropriate offloading, creating a concrete availability risk.

## Findings

### High Severity

#### ASVS-1513-HIGH-001

**Title:** Missing Documentation of Resource-Intensive Argon2 Cryptographic Operations and Availability Defenses

**Location:**
- **File:** `v3/steve/crypto.py`
- **Lines:** 91–102
- **Related Files:** `v3/steve/election.py` (lines 270–285, 353–378)

**Related Functions Checked:**
- `crypto._hash()` (L91)
- `crypto.gen_vote_token()` (called by multiple election methods)
- `election.add_vote()` (L270)
- `election.has_voted_upon()` (L353)
- `election.tally_issue()` (L306)
- `election.open()` (invokes `gen_opened_key()`)
- `election.is_tampered()` (invokes `gen_opened_key()`)

**Description:**

The application uses Argon2 key derivation with significant resource requirements (64MB memory, ~200-500ms CPU time per invocation) in multiple web request paths. There is no documentation identifying these operations as resource-intensive, no documented defenses against availability loss, and no documented strategies to avoid response times exceeding consumer timeouts, creating a direct violation of ASVS 15.1.3.

**Vulnerable Code:**

```python
# crypto.py L91-102 — Called in every vote submission and ballot status check
def _hash(data: bytes, salt: bytes) -> bytes:
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,  # 64 MB per invocation
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,
    )
```

**Data Flow:**

1. User submits vote → `election.py:add_vote()` L270
2. `add_vote()` → `crypto.gen_vote_token()` → `crypto._hash()` — allocates **64MB RAM**, blocks CPU for ~200–500ms
3. User views ballot → `election.py:has_voted_upon()` L353
4. `has_voted_upon()` → `crypto.gen_vote_token()` × N issues — **N × 64MB peak, N × 200–500ms blocking**
5. **Missing control:** No documentation identifying these as resource-intensive; no documented defense against concurrent invocations exhausting memory or CPU; no documented timeout strategy

**Specific Resource Analysis:**

| Operation | Argon2 Calls | Memory per Call | CPU Time per Call | Scaling |
|-----------|-------------|-----------------|-------------------|---------|
| `add_vote()` | 1 | 64 MB | ~200–500ms | Per vote submission |
| `has_voted_upon()` | N (issues) | 64 MB each | ~200–500ms each | Per ballot page load |
| `tally_issue()` | M (voters) | 64 MB each | ~200–500ms each | Per issue tallied |
| `open()` → `gen_opened_key()` | 1 | 64 MB | ~200–500ms | Per election open |
| `is_tampered()` | 1 | 64 MB | ~200–500ms | Per tamper check |

**Proof of Concept:**

With 10 concurrent vote submissions: 10 × 64MB = **640MB peak memory** + 10 × 500ms of CPU blocking. With `has_voted_upon()` for a voter eligible on 20 issues: **20 sequential Argon2 calls = ~10 seconds response time**, likely exceeding client timeout.

**Impact:**

Without documentation identifying these costs, operators cannot:
- Size infrastructure appropriately
- Configure reverse proxy timeouts
- Set connection pool limits
- Implement request queuing for heavy operations
- Understand why the application becomes unresponsive under moderate load

**Remediation:**

Create an operations/architecture document that:

1. Identifies each resource-intensive operation with its CPU/memory profile
2. Documents maximum concurrent requests the server can handle based on Argon2 memory
3. Specifies recommended reverse proxy timeout settings
4. Describes the recommended deployment configuration (worker count, memory limits)

Example documentation section:

```markdown
## Resource-Intensive Operations

### Vote Submission (`add_vote`)
- **Resource cost:** 1× Argon2 derivation (64MB RAM, ~500ms CPU)
- **Concurrency limit:** Max concurrent vote submissions = available_memory / 64MB
- **Recommended defense:** Configure reverse proxy to limit concurrent connections;
  set worker count = (available_memory - base_usage) / 64MB
- **Timeout guidance:** Client timeout should be ≥ 2 seconds for vote submission

### Ballot Status (`has_voted_upon`)
- **Resource cost:** N × Argon2 where N = number of issues (64MB × N RAM, ~500ms × N CPU)
- **Recommended defense:** Limit elections to ≤ 20 issues, or implement caching
  of voted-upon status; consider async processing with progress indication
- **Timeout guidance:** For N issues, expect N × 0.5s response time
```

#### ASVS-1513-HIGH-002

**Title:** Synchronous CPU/Memory-Bound Operations Block Async Event Loop Without Documented Mitigation

**Location:**
- **File:** `v3/steve/election.py`
- **Lines:** 270–285, 353–378
- **Related Files:** `v3/server/main.py` (line 39)

**Related Functions Checked:**
- `election.add_vote()` (L270-285)
- `election.has_voted_upon()` (L353-378)
- `crypto.gen_vote_token()` (synchronous, CPU-bound)
- `main.py:create_app()` (L39, creates Quart async app)

**Description:**

The application uses Quart, an async web framework, but calls synchronous CPU-bound Argon2 operations directly within the async event loop without offloading to a thread pool. This blocks the entire event loop during cryptographic operations (~500ms per call), preventing the server from handling any other requests. There is no documentation of this architectural constraint or mitigation strategy.

**Vulnerable Code:**

```python
# election.py L270-285 — Called from async web handler
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    # Synchronous Argon2 — blocks entire event loop for ~500ms
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

```python
# main.py L39 — Creates Quart (async) application
app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
```

**Data Flow:**

1. Quart async framework handles incoming HTTP request
2. Async handler calls `Election.add_vote()` (synchronous)
3. `add_vote()` calls `crypto.gen_vote_token()` → `argon2.low_level.hash_secret_raw()` (synchronous, CPU-bound, ~500ms)
4. **During this 500ms, the entire async event loop is blocked** — no other requests can be served
5. **Missing control:** No `asyncio.run_in_executor()` wrapping, no thread pool offloading, and **no documentation** of this architectural constraint

**Proof of Concept:**

```
# 5 voters submit votes within 1 second
# Without event loop offloading:
# Voter 1: waits 0–500ms
# Voter 2: waits 500–1000ms  
# Voter 3: waits 1000–1500ms
# Voter 4: waits 1500–2000ms (likely timeout)
# Voter 5: waits 2000–2500ms (definitely timeout)
```

**Impact:**

Even modest concurrent usage causes cascading response delays. A single `has_voted_upon()` call for 10 issues blocks the event loop for ~5 seconds, during which ALL other requests (including health checks) are unservable. This is a direct availability concern that ASVS 15.1.3 requires to be documented with defenses.

**Remediation:**

1. **Document** that Argon2 operations must be offloaded from the event loop

2. **Implement** `asyncio.run_in_executor()` for all Argon2-calling paths:

```python
import asyncio
from concurrent.futures import ThreadPoolExecutor

# Limit concurrent Argon2 operations based on available memory
# e.g., 4 concurrent × 64MB = 256MB Argon2 budget
_CRYPTO_POOL = ThreadPoolExecutor(max_workers=4, thread_name_prefix='argon2')

async def add_vote_async(self, pid: str, iid: str, votestring: str):
    """Add vote with non-blocking cryptographic operations."""
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    
    loop = asyncio.get_event_loop()
    # Offload CPU-bound Argon2 to thread pool
    vote_token = await loop.run_in_executor(
        _CRYPTO_POOL,
        crypto.gen_vote_token, md.opened_key, pid, iid, mayvote.salt
    )
    ciphertext = await loop.run_in_executor(
        _CRYPTO_POOL,
        crypto.create_vote, vote_token, mayvote.salt, votestring
    )
    self.c_add_vote.perform(vote_token, ciphertext)
```

3. **Document** the thread pool size as the concurrency control mechanism:

```markdown
### Concurrency Control
Argon2 operations are offloaded to a bounded thread pool (max_workers=4).
This limits peak memory to 4 × 64MB = 256MB for cryptographic operations
and prevents event loop blocking. Excess requests queue at the executor.
```

### Medium Severity

#### ASVS-1513-MEDIUM-003

**Title:** Tally Operation Has Unbounded O(N) Argon2 Complexity Without Documented Timeout or Processing Strategy

**Location:**
- **File:** `v3/steve/election.py`
- **Lines:** 287–351
- **Function:** `tally_issue()`

**Related Functions Checked:**
- `election.tally_issue()` (L306-340)
- `crypto.gen_vote_token()` (called N times where N = eligible voters)
- `crypto.decrypt_votestring()` (called N times)

**Description:**

The `tally_issue()` method performs O(N) Argon2 key derivations where N equals the number of eligible voters for an issue. There is no documentation of the expected execution time, no guidance on maximum supported election sizes, and no documented timeout or processing strategy. This creates an undocumented availability risk for large elections.

**Vulnerable Code:**

```python
# election.py L306-340 — O(N) Argon2 operations where N = eligible voters
def tally_issue(self, iid):
    md = self._all_metadata(self.S_CLOSED)
    issue = self.q_get_issue.first_row(iid)
    voters = set()
    votes = []

    self.q_tally.perform(iid)
    for mayvote in self.q_tally.fetchall():
        # Each iteration: 1× Argon2 (64MB, ~500ms) + 1× HKDF + 1× Fernet decrypt
        vote_token = crypto.gen_vote_token(
            md.opened_key, mayvote.pid, iid, mayvote.salt,
        )
        row = self.q_recent_vote.first_row(vote_token)
        if row is None:
            continue
        voters.add(mayvote.pid)
        votestring = crypto.decrypt_votestring(
            vote_token, mayvote.salt, row.ciphertext,
        )
        votes.append(votestring)
```

**Resource Scaling Analysis:**

| Eligible Voters | Argon2 Calls | Peak Memory | Estimated Wall Time |
|----------------|-------------|-------------|---------------------|
| 100 | 100 | 64 MB (sequential) | ~50 seconds |
| 500 | 500 | 64 MB (sequential) | ~250 seconds |
| 1,000 | 1,000 | 64 MB (sequential) | ~500 seconds |
| 5,000 | 5,000 | 64 MB (sequential) | ~2,500 seconds |

**Data Flow:**

1. Tally operation initiated (CLI or web endpoint)
2. For each eligible voter on the issue:
   - Call `crypto.gen_vote_token()` → Argon2 derivation (64MB, ~500ms)
   - Decrypt vote using derived key
3. No progress indication, no timeout handling, no documented execution strategy

**Impact:**

If `tally_issue()` is invoked via a web endpoint (not just CLI), the response will exceed any reasonable HTTP timeout. Even as a CLI operation, there is no documented guidance on expected duration, no progress reporting, and no documented maximum election size. Operators have no basis for setting infrastructure timeouts or planning tally windows.

**Remediation:**

Document the tally operation's resource profile and recommended execution strategy:

```markdown
### Tally Operation
- **Complexity:** O(N) Argon2 derivations where N = eligible voters per issue
- **Expected time:** ~0.5 seconds × N voters per issue
- **Execution model:** CLI only; NOT exposed via web API
- **Recommendation:** For elections > 200 voters, schedule tallying 
  during low-usage windows. Consider batched processing with 
  progress output for elections > 1000 voters.
- **Maximum supported:** Tested up to N voters at Y time
```

## Positive Security Patterns

1. **`benchmark_argon2()` function** (`crypto.py` L129–162): Demonstrates developer awareness of Argon2 performance characteristics. This benchmarking capability is a good foundation for creating resource documentation.

2. **Cryptographic shuffle** (`crypto.py` L105–119): Uses `secrets.randbelow()` — correct use of CSPRNG with O(n) time complexity, not resource-intensive.

3. **Transaction-based salt generation** (`election.py` L129–148): `add_salts()` wraps all salt generation in a single transaction, reducing database contention.

4. **State machine enforcement** (`election.py`): `assert self.is_editable()` / `self._all_metadata(self.S_OPEN)` prevents operations in wrong states, reducing unnecessary resource consumption from invalid requests.

5. **`gen_salt()` uses `secrets.token_bytes()`** (`crypto.py` L36–37): Fast, non-blocking CSPRNG operation — not a resource concern.

## Architecture Observations

1. **Framework mismatch:** Quart is an async framework designed for I/O-bound workloads. Argon2 is CPU-bound and memory-intensive. Without explicit offloading via `run_in_executor()`, every Argon2 call serializes request processing on the event loop. This is a fundamental architectural concern that ASVS 15.1.3 specifically asks to be documented.

2. **Memory budgeting not addressed:** Each Argon2 call allocates 64MB. With no documented limit on concurrent operations, an operator cannot determine safe worker/connection counts. For a server with 2GB available memory, only ~30 concurrent Argon2 operations can execute before OOM.

3. **Inherent tension:** The cryptographic design prioritizes vote anonymity (Argon2-based key derivation per voter per issue), which creates linear scaling of expensive operations. This is a legitimate design trade-off that **must be documented** per ASVS 15.1.3, including the maximum supported election size.

4. **`has_voted_upon()` multiplier effect:** This is likely called on every ballot page load. If a voter is eligible for 10 issues, that's 10 Argon2 calls (~5 seconds) per page load, blocking the event loop. This needs both documentation and a caching strategy.
