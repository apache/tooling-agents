## Findings

### CRITICAL

#### ASVS-1521-CRT-001: Complete Absence of SBOM and Dependency Version Tracking

**Severity:** CRITICAL  
**Gap Type:** Type A — No control exists for a foundational requirement  
**CWE:** CWE-1395 (Dependency on Vulnerable Third-Party Component)

**Location:**
- **File:** Project-wide (no `pyproject.toml`, `requirements.txt`, `uv.lock`, `Pipfile.lock`, or SBOM document found)
- **Affected Files:** `main.py` (line 1), `crypto.py` (imports), `election.py` (imports)

**Related Functions Checked:**
- All import statements across `main.py`, `crypto.py`, `election.py`
- `uv run --script` invocation in `main.py:1`

**Description:**

The application has **no Software Bill of Materials (SBOM)**, no dependency version pinning, and no documented update/remediation timeframes. This renders ASVS 15.2.1 completely unauditable — the requirement mandates verification that components "have not breached the documented update and remediation time frames," but with no documented timeframes and no recorded component versions, this verification is impossible.

**Vulnerable Code:**

```python
# main.py:1
#!/usr/bin/env -S uv run --script
# ❌ No inline script metadata (PEP 723) declaring dependencies

# crypto.py
import argon2.low_level                          # ❌ No version constraint
import cryptography.fernet                        # ❌ No version constraint
from cryptography.hazmat.primitives.kdf import hkdf  # ❌ No version constraint

# election.py
import asfpy.db                                   # ❌ No version constraint
from easydict import EasyDict as edict            # ❌ No version constraint

# main.py
import asfquart                                   # ❌ No version constraint
```

**Data Flow:**
1. **Source:** Unversioned dependency declarations across all modules
2. **Resolution:** `uv run --script` resolves dependencies at install time without constraints
3. **Sink:** Production runtime environment with potentially vulnerable versions
4. **Missing Control:** No SBOM, no version pinning, no lock file, no documented remediation timeframes

**Proof of Concept:**

The following demonstrates the risk:

```bash
# Scenario 1: Initial deployment resolves to secure versions
$ uv run main.py
# Installs: cryptography==44.0.0 (secure)

# Scenario 2: Six months later, re-deployment on new server
$ uv run main.py  
# Installs: cryptography==38.0.0 (contains CVE-2023-49083)
# ❌ No lock file to prevent version drift
# ❌ No SBOM to verify what's running
# ❌ No policy to mandate updates within X days
```

**Impact:**

1. **Cryptographic Vulnerability Exposure:** The application's entire security model depends on `cryptography` (Fernet encryption of votes) and `argon2-cffi` (key derivation). Historical CVEs in the `cryptography` package (e.g., CVE-2023-49083, CVE-2024-26130) demonstrate that running outdated versions can introduce exploitable vulnerabilities. Without pinned versions, there is no way to verify the deployed version is patched.

2. **Non-Reproducible Builds:** Each deployment may resolve to different dependency versions, including ones with known vulnerabilities. The `uv run --script` invocation without a lock file will resolve dependencies at install time, creating inconsistent environments.

3. **Supply Chain Attack Surface:** Without version pinning, dependency confusion attacks and malicious package updates can silently enter production.

4. **Compliance Failure:** This is an ASVS Level 1 requirement (minimum baseline). No audit trail exists for when dependencies were last reviewed or updated, creating regulatory/compliance burden.

5. **Vote Integrity Risk:** Vulnerable versions of cryptographic libraries could compromise vote encryption/decryption integrity, undermining the entire election system.

**Remediation:**

**Step 1: Create dependency manifest with pinned versions**

```toml
# pyproject.toml
[project]
name = "steve"
version = "3.0.0"
requires-python = ">=3.11"
dependencies = [
    "asfquart>=0.x.x",
    "asfpy>=0.x.x",
    "cryptography>=44.0.0",  # Pinned to patched version
    "argon2-cffi>=23.1.0",   # Pinned to patched version
    "easydict>=1.13",
]

[tool.uv]
# Generate and commit uv.lock for reproducible builds
```

**Step 2: Generate and commit lock file**

```bash
uv lock
git add uv.lock
git commit -m "Add dependency lock file for reproducible builds"
```

**Step 3: Generate formal SBOM**

```bash
pip install cyclonedx-bom
cyclonedx-py environment -o sbom.json --format json
git add sbom.json
git commit -m "Add Software Bill of Materials"
```

**Step 4: Document update/remediation timeframes**

```markdown
# SECURITY.md

## Dependency Update Policy

- **Critical CVEs:** Patch within 48 hours of disclosure
- **High CVEs:** Patch within 7 days of disclosure
- **Medium CVEs:** Patch within 30 days of disclosure
- **Routine updates:** Monthly dependency review cycle
- **SBOM refresh:** On every release and quarterly audit
- **Cryptographic dependencies:** Review within 24 hours of any security advisory

## Vulnerability Monitoring

- Automated scanning via `pip-audit` in CI/CD
- GitHub Dependabot enabled for automated security PRs
- Security mailing lists monitored for `cryptography` and `argon2-cffi`
```

**Step 5: Integrate automated vulnerability scanning**

```yaml
# .github/workflows/security.yml
name: Dependency Security Scan
on: [push, pull_request, schedule]

jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v1
      - run: uv pip install pip-audit
      - run: pip-audit --requirement requirements.txt --format json
```

**Step 6: Enable automated dependency updates**

```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: "pip"
    directory: "/"
    schedule:
      interval: "weekly"
    open-pull-requests-limit: 10
    labels:
      - "dependencies"
      - "security"
```

## Architecture Observations

### 1. Dangerous Functionality Identification

`crypto.py` correctly centralizes all cryptographic operations (encryption, decryption, key derivation, hashing). Per the domain context, this module contains "dangerous functionality" and its dependencies (`cryptography`, `argon2-cffi`) are the highest-priority components for version tracking — making the absence of version management especially impactful.

### 2. `uv` Tooling Partial Adoption

The `#!/usr/bin/env -S uv run --script` shebang in `main.py` indicates the project uses `uv` as its package manager. `uv` supports `uv.lock` files for reproducible resolution. The tooling infrastructure exists but the lock file and version constraints are not present in the provided codebase.

### 3. No Evidence of Dependency Auditing Pipeline

No CI/CD configuration, pre-commit hooks, or scanning tool configuration was found that would automatically flag outdated or vulnerable dependencies before they reach production.

## Recommendations Summary

### Immediate Actions (P0 — Within 48 Hours)

1. **Create `pyproject.toml` with pinned dependency versions and generate `uv.lock`**
   - **Rationale:** Enables reproducible builds and version verification — prerequisite for all other controls
   - **Effort:** 2-4 hours
   - **Impact:** Prevents version drift and establishes audit baseline

2. **Generate and maintain a formal SBOM (CycloneDX JSON)**
   - **Rationale:** Directly addresses ASVS 15.2.1 auditability requirement
   - **Effort:** 1-2 hours (automated)
   - **Impact:** Enables compliance verification and incident response

3. **Document update/remediation timeframes in `SECURITY.md`**
   - **Rationale:** Establishes the "documented time frames" that ASVS 15.2.1 requires verification against
   - **Effort:** 2-3 hours
   - **Impact:** Creates policy framework for ongoing compliance

### Short-term Actions (P1 — Within 1 Week)

4. **Integrate `pip-audit` or `osv-scanner` into CI/CD**
   - **Rationale:** Automated detection of known vulnerabilities in dependencies
   - **Effort:** 4-6 hours
   - **Impact:** Continuous monitoring prevents vulnerable deployments

5. **Enable GitHub Dependabot or Renovate for automated dependency PRs**
   - **Rationale:** Reduces mean-time-to-remediation for vulnerable components
   - **Effort:** 2-3 hours
   - **Impact:** Proactive security updates with minimal manual effort

### Long-term Actions (P2 — Within 1 Month)

6. **Implement pre-commit hooks for dependency validation**
   - **Rationale:** Shift-left security by catching issues before commit
   - **Effort:** 3-4 hours
   - **Impact:** Developer awareness and early detection

7. **Establish quarterly dependency review process**
   - **Rationale:** Proactive identification of outdated components beyond CVEs
   - **Effort:** 4 hours per quarter
   - **Impact:** Reduces technical debt and maintains security posture
