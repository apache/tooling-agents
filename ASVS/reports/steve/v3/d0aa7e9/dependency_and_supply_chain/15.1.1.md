## Findings

### Critical Severity

#### ASVS-1511-CRI-001: No Documented Risk-Based Remediation Timeframes for Security-Critical Cryptographic Dependencies

**Severity:** CRITICAL

**Gap Type:** Type A — Complete absence of control for security-critical components explicitly identified as "dangerous functionality"

**Location:**
- **Files Affected:** All application files (policy documentation missing)
- **Primary Impact:** `v3/steve/crypto.py` (lines 58-94)
- **Secondary Impact:** `v3/steve/election.py` (lines 283-287, 320-333)

**Related Functions Checked:**
- `crypto.py::_hash()` (lines 85-94)
- `crypto.py::create_vote()` (lines 71-76)
- `crypto.py::decrypt_votestring()` (lines 78-82)
- `crypto.py::_b64_vote_key()` (lines 60-68)
- `election.py::decrypt_votestring()` (lines 283-287)
- `election.py::is_tampered()` (lines 320-333)

**Description:**

The application's entire vote secrecy guarantee depends on two third-party cryptographic libraries (`argon2-cffi` and `cryptography`), both used extensively in `crypto.py` — a module explicitly identified as containing "dangerous functionality" (encryption/decryption). No documentation exists anywhere in the provided codebase that defines:

1. Risk-based remediation timeframes (e.g., Critical CVE → 24h, High → 7d, Medium → 30d)
2. A general schedule for updating libraries
3. An SBOM enumerating component versions
4. A classification of which components are "dangerous" or "risky"

**Vulnerable Code:**

```python
# crypto.py:85-94 — Argon2 key derivation (vote key material)
import argon2.low_level
# ...
return argon2.low_level.hash_secret_raw(
    secret=data,
    salt=salt,
    time_cost=2,
    memory_cost=65536,
    parallelism=4,
    hash_len=32,
    type=argon2.low_level.Type.D,
)
```

```python
# crypto.py:71-76 — Fernet vote encryption
import cryptography.fernet
# ...
def create_vote(vote_token: bytes, salt: bytes, votestring: str) -> bytes:
    b64key = _b64_vote_key(vote_token, salt)
    f = cryptography.fernet.Fernet(b64key)
    return f.encrypt(votestring.encode())
```

```python
# crypto.py:78-82 — Fernet vote decryption (used during tally)
def decrypt_votestring(vote_token: bytes, salt: bytes, ciphertext: bytes) -> str:
    b64key = _b64_vote_key(vote_token, salt)
    f = cryptography.fernet.Fernet(b64key)
    return f.decrypt(ciphertext).decode()
```

**Data Flow:**

```
argon2-cffi vulnerability → _hash() compromised → gen_opened_key() / gen_vote_token() weakened
    → Vote encryption keys derivable → ALL stored votes decryptable
    → Election integrity (is_tampered) bypassed

cryptography vulnerability → Fernet broken → create_vote() / decrypt_votestring() compromised
    → Vote contents exposed → Complete loss of ballot secrecy
```

**Proof of Concept:**

Without documented remediation timeframes:
1. CVE-XXXX-YYYY is published affecting `cryptography.fernet`
2. No organizational policy exists to determine response urgency
3. Vulnerability remains unpatched for extended period
4. Attacker exploits vulnerability to decrypt stored votes
5. Election integrity and ballot secrecy completely compromised

**Impact:**

A vulnerability in either `argon2-cffi` or the `cryptography` library could directly compromise:
- **Vote secrecy**: All encrypted votes in the database could be decrypted (`decrypt_votestring` in `election.py:283-287`)
- **Election integrity**: The tamper detection mechanism (`is_tampered` in `election.py:320-333`) relies on `crypto.gen_opened_key()` which depends on both libraries
- **Key derivation security**: `_hash()` using `argon2.low_level` is the foundation of all vote tokens and election keys

Without documented remediation timeframes, a published CVE in these libraries could remain unpatched indefinitely, with no organizational accountability to apply fixes.

**Remediation:**

Create a **Dependency Security Policy** document (e.g., `DEPENDENCY-POLICY.md`) that includes:

```markdown
# Dependency Security Policy

## Software Bill of Materials
See `sbom.json` (CycloneDX format) generated by CI pipeline.

## Component Risk Classification

### Dangerous Functionality Components
Components performing cryptographic operations, deserialization, or 
direct memory manipulation:
- `cryptography` (Fernet encryption, HKDF key derivation) — crypto.py
- `argon2-cffi` (Argon2 key derivation) — crypto.py

### Risky Components
Third-party components requiring enhanced monitoring:
- [List components with poor maintenance, EOL status, or vulnerability history]

## Vulnerability Remediation Timeframes

| Severity (CVSS) | Timeframe | Dangerous Functionality | Standard Components |
|------------------|-----------|------------------------|---------------------|
| Critical (9.0+)  | Immediate | 24 hours               | 48 hours            |
| High (7.0-8.9)   | Urgent    | 72 hours               | 7 days              |

## General Update Cadence
- Security-critical libraries: Monthly review, update within 7 days of patch
- All other dependencies: Quarterly review and update cycle
- Automated CVE monitoring via [tool, e.g., GitHub Dependabot/Snyk/OSV-Scanner]
```

## Positive Security Patterns

1. **Strong cryptographic primitives**: Use of Argon2 for key derivation with reasonable parameters (`time_cost=2`, `memory_cost=65536`, `parallelism=4`) in `crypto.py:85-94`
2. **Cryptographically secure randomness**: Consistent use of `secrets` module throughout (`gen_salt`, `create_id`, `shuffle`) — no use of `random` module
3. **HKDF key stretching**: Proper use of HKDF with salt and info context for deriving encryption keys (`crypto.py:60-68`)
4. **Anti-tamper verification**: `election.py:320-333` recomputes election data hash to detect modification
5. **Strict state machine**: Election lifecycle enforced via `assert` statements preventing operations in wrong states
6. **TLS support**: Server configuration includes certificate/key paths for encrypted transport (`config.yaml.example`, `main.py:67-69`)

---

## Architecture Observations

1. **Crypto module is a single point of failure**: `crypto.py` concentrates all cryptographic operations using two third-party libraries. A vulnerability in either library compromises the entire voting system's security guarantees.
2. **No dependency pinning visible**: No `requirements.txt`, `pyproject.toml`, or lock file was provided for review — version pinning status is unknown.
3. **Argon2 Type.D usage**: `crypto.py:93` uses `Type.D` (data-dependent) rather than `Type.ID` (hybrid), though the benchmark function at line 137 tests `Type.ID`. This inconsistency should be documented with rationale.
