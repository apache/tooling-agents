## Findings

**No findings at MEDIUM severity or above.**

The application does not implement or consume the `postMessage` API anywhere in its frontend code. There are:

- Zero `window.addEventListener('message', ...)` or `window.onmessage` handlers
- Zero `postMessage()` calls to other windows or frames
- Zero `<iframe>` elements that could be targets or sources of cross-origin messages
- Zero `window.open()` calls that could create popup windows for cross-origin communication

---

## Positive Security Patterns

1. **No unnecessary cross-origin messaging surface**: The application avoids `postMessage` entirely, using same-origin form submissions and `fetch()` calls instead. This eliminates the entire class of postMessage-related vulnerabilities.

2. **SRI integrity on third-party scripts**: SortableJS is loaded with a Subresource Integrity hash (line 177 of `vote-on.ezt`), preventing supply-chain tampering that could inject malicious `message` event listeners.

3. **Server-side data embedding over XHR**: STV candidate data is rendered directly into the page as a JavaScript object (`STV_CANDIDATES`), avoiding any need for cross-origin data fetching or messaging.

4. **Client-side HTML escaping**: The `escapeHtml()` function in `vote-on.ezt` sanitizes candidate names before DOM insertion, which would also protect against injection if postMessage were introduced in the future.

5. **Same-origin fetch with JSON**: The `saveDate()` function in `manage.ezt` communicates with the server via `fetch()` using `Content-Type: application/json`, which stays within the same origin and triggers CORS preflight if misconfigured.

---

## Architecture Observations

- **Communication model**: The application follows a traditional server-rendered architecture with minimal client-side interactivity. All data flows through standard HTTP requests (form POSTs, fetch calls) rather than inter-window messaging.

- **No embedded third-party widgets**: There are no embedded iframes (payment processors, analytics widgets, social media embeds, etc.) that would necessitate `postMessage` communication.

- **Future risk**: If the application ever introduces iframes, OAuth popup flows, or cross-origin widget integrations, postMessage origin validation (per ASVS 3.5.5) should be implemented at that time.

## Recommendations Summary

No remediation actions are required for ASVS 3.5.5. As a defensive measure for future development:

### Immediate Actions
None required — ASVS 3.5.5 is not applicable to current codebase.

### Short-term Actions
**Document the no-postMessage policy** — Add a comment in `steve.js` or a development guideline noting that if `postMessage` is introduced, origin validation must be implemented per ASVS 3.5.5.

### Long-term Actions
**Example pattern for future use** — If postMessage is ever needed, implement the following validation pattern:

```javascript
window.addEventListener('message', function(event) {
    // ASVS 3.5.5: Validate origin
    if (event.origin !== 'https://trusted.example.org') {
        return; // Discard untrusted origin
    }
    
    // ASVS 3.5.5: Validate message syntax
    let data;
    try {
        data = JSON.parse(event.data);
        if (!data.type || !data.payload) {
            return; // Discard invalid syntax
        }
    } catch (e) {
        return; // Discard unparseable messages
    }
    
    // Process validated message
    handleMessage(data);
});
```
