## Findings

### Medium Severity

#### ASVS-361-MED-001 — Externally Hosted SVG Image Without SRI or Documented Security Decision

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with NO control  
**Location:**
- **File:** `v3/server/templates/header.ezt`
- **Line:** 18
- **Function/Context:** Header template - Logo rendering

**Related Functions Checked:**
- Template rendering in header.ezt
- Static asset loading in fetch-thirdparty.sh

**Vulnerable Code:**
```html
<img src="https://www.apache.org/foundation/press/kit/feather.svg" alt="Logo" width="30" height="30" class="d-inline-block align-text-top">
```

**Data Flow:**  
Browser → HTTPS request to `www.apache.org` → SVG content loaded and rendered → No integrity verification

**Description:**  
The Apache feather logo is loaded at runtime from an external domain (`www.apache.org`). This resource:
- Is **not versioned** (the URL has no version identifier, meaning content can change)
- Has **no SRI** integrity attribute (the `integrity` attribute is not supported on `<img>` elements)
- Has **no documented security decision** justifying this external dependency

ASVS 3.6.1 requires that when SRI is not possible, "there should be a documented security decision to justify this for each resource." While SVG loaded via `<img>` is sandboxed (no script execution), a compromised resource could still be used for phishing (visual replacement) or tracking.

**Impact:**  
If the external host is compromised or the resource is modified, the application would display attacker-controlled visual content to all users. In a voting application, this could undermine trust or be used for social engineering.

**Proof of Concept:**
N/A - Configuration issue

**Remediation:**  
Self-host the SVG image alongside other static assets:

```bash
# In fetch-thirdparty.sh, add:
FEATHER_URL="https://www.apache.org/foundation/press/kit/feather.svg"
echo "Fetching: ${FEATHER_URL}"
curl -q --fail "${FEATHER_URL}" --output "${STATIC_DIR}/img/feather.svg"
```

```html
<!-- In header.ezt -->
<img src="/static/img/feather.svg" alt="Logo" width="30" height="30" class="d-inline-block align-text-top">
```

#### ASVS-361-MED-002 — Missing SRI for Self-Hosted Third-Party Library (bootstrap-icons.css)

**Severity:** MEDIUM  
**Gap Type:** Type B — Control EXISTS (SRI for Bootstrap CSS/JS) but NOT APPLIED to bootstrap-icons.css  
**Location:**
- **File:** `v3/server/templates/header.ezt`
- **Line:** 10
- **File:** `v3/server/bin/fetch-thirdparty.sh`
- **Lines:** 70–74
- **Function/Context:** Template CSS loading and build script asset extraction

**Related Functions Checked:**
- SRI hash generation in fetch-thirdparty.sh (lines 60, 62, 92)
- Bootstrap CSS loading with SRI in header.ezt (lines 8-9)
- Bootstrap Icons extraction process (lines 70-74)

**Vulnerable Code (header.ezt):**
```html
<link href="/static/css/bootstrap-icons.css" rel="stylesheet">
```

Compare with the protected resource on the preceding line:
```html
<link href="/static/css/bootstrap.min.css" rel="stylesheet"
      integrity="sha384-4bw+/aepP/YC94hEpVNVgiZdgIC5+VKNBQNGCHeKRQN+PtmoHDEXuppvnDJzQIu9">
```

**Vulnerable Code (fetch-thirdparty.sh):**
```bash
# Hash generated for bootstrap.min.css (line 60):
echo -n "sha384-" ; openssl dgst -sha384 -binary "${STATIC_DIR}/css/bootstrap.min.css" | openssl base64 -A

# NO hash generated for bootstrap-icons.css after extraction at line 71
echo "Extracting: bootstrap-icons.css"
unzip -joq "${I_ZIPFILE}" "${I_DIST}/bootstrap-icons.css" -d "${STATIC_DIR}/css"
# <-- missing hash generation
```

**Data Flow:**  
`fetch-thirdparty.sh` downloads and extracts `bootstrap-icons.css` → No SRI hash generated → Template includes file without `integrity` attribute → Browser loads CSS without integrity check → Tampered CSS would load undetected

**Description:**  
The SRI defense-in-depth pattern is applied to `bootstrap.min.css` and `bootstrap.bundle.min.js` but explicitly skipped for `bootstrap-icons.css`. This third-party CSS file controls `@font-face` declarations for web fonts. If tampered with after deployment, it could:

1. Redirect font loading to an attacker-controlled origin
2. Inject CSS-based data exfiltration (e.g., attribute selectors with background URLs)
3. Modify visual rendering to mislead voters

The inconsistency creates a **false confidence** that third-party resources are integrity-protected when a significant gap exists.

**Impact:**  
An attacker who can modify server-side files or intercept during deployment could alter `bootstrap-icons.css` without detection, while other Bootstrap files would trigger integrity failures. This creates a targeted attack vector through the weakest link.

**Proof of Concept:**
N/A - Configuration issue

**Remediation:**  
Add SRI hash generation and template integration:

```bash
# In fetch-thirdparty.sh, after extracting bootstrap-icons.css:
echo "bootstrap-icons.css:"
echo -n "sha384-" ; openssl dgst -sha384 -binary "${STATIC_DIR}/css/bootstrap-icons.css" | openssl base64 -A ; echo ""
```

```html
<!-- In header.ezt -->
<link href="/static/css/bootstrap-icons.css" rel="stylesheet"
      integrity="sha384-GENERATED_HASH_HERE">
```

#### ASVS-361-MED-003 — Build Script Downloads Third-Party Assets Without Pre-Download Integrity Verification

**Severity:** MEDIUM  
**Gap Type:** Type C — Control CALLED (hash generation) but RESULT IGNORED (hash computed after download, not verified against known-good values)  
**Location:**
- **File:** `v3/server/bin/fetch-thirdparty.sh`
- **Lines:** 47, 60–62, 67, 82, 92
- **Function/Context:** Third-party asset download and hash generation process

**Related Functions Checked:**
- Bootstrap download (line 47)
- Bootstrap extraction (lines 50-57)
- Bootstrap Icons download (line 67)
- Bootstrap Icons extraction (lines 70-74)
- Sortable download (line 82)
- SRI hash generation (lines 60, 62, 92)

**Vulnerable Code:**
```bash
# Download with no integrity check:
curl -q --location "${B_URL}" --output "${ZIPFILE}"

# Extract immediately:
unzip -joq "${ZIPFILE}" "${DIST}/css/bootstrap.min.css" -d "${STATIC_DIR}/css"

# Generate hash AFTER extraction (from potentially compromised content):
echo -n "sha384-" ; openssl dgst -sha384 -binary "${STATIC_DIR}/css/bootstrap.min.css" | openssl base64 -A
```

**Data Flow:**  
GitHub release URL → `curl` download (no checksum verification) → `unzip` extraction → Hash generated from extracted file → Hash embedded in templates → SRI "validates" against the (potentially compromised) file

**Description:**  
The build script generates SRI hashes **from the downloaded content** rather than verifying downloads **against known-good hashes**. This means:

1. `curl` does not use `--fail` flag (HTTP errors silently produce non-library content)
2. No pre-defined SHA-256/SHA-384 checksums are checked before extraction
3. No GPG signature verification of release packages
4. The generated SRI hash will match whatever was downloaded, including compromised content

If a supply chain attack targets the download (e.g., compromised GitHub release, DNS hijacking), the SRI mechanism would be rendered ineffective because the integrity hash would be computed from the malicious payload.

**Impact:**  
A supply chain compromise during the build process would result in malicious JavaScript/CSS being served to all voters, with SRI hashes that appear valid. The existing SRI provides zero protection against this attack vector.

**Proof of Concept:**
N/A - Build process vulnerability

**Remediation:**  
Add known-good hash verification before extraction:

```bash
# Define expected hashes (from official Bootstrap release notes)
EXPECTED_BS_SHA256="a4a04c..."  # From https://github.com/twbs/bootstrap/releases
EXPECTED_ICONS_SHA256="b5b5b5..."
EXPECTED_SORTABLE_SHA256="c6c6c6..."

# Download
curl -q --fail --location "${B_URL}" --output "${ZIPFILE}"

# Verify before extraction
ACTUAL_HASH=$(sha256sum "${ZIPFILE}" | cut -d' ' -f1)
if [ "${ACTUAL_HASH}" != "${EXPECTED_BS_SHA256}" ]; then
    echo "ERROR: Bootstrap download integrity check failed!"
    echo "Expected: ${EXPECTED_BS_SHA256}"
    echo "Got:      ${ACTUAL_HASH}"
    rm -f "${ZIPFILE}"
    exit 1
fi

# Only then extract
unzip -joq "${ZIPFILE}" "${DIST}/css/bootstrap.min.css" -d "${STATIC_DIR}/css"
```

## Positive Security Patterns

1. **Self-hosting third-party assets:** All JavaScript libraries, CSS, and fonts are downloaded at build time and served from the application's own static directory, avoiding runtime CDN dependencies. This is the preferred approach per ASVS 3.6.1.

2. **SRI on critical Bootstrap resources:** Both `bootstrap.min.css` and `bootstrap.bundle.min.js` include SHA-384 integrity attributes, providing defense-in-depth against post-deployment file tampering.

3. **Versioned third-party dependencies:** The build script pins specific versions (`VERSION="5.3.1"`, `I_VERSION="1.13.1"`, `S_VERSION="1.15.7"`), preventing silent updates.

4. **SRI hash generation tooling:** The build script includes `openssl dgst` commands to generate SRI hashes, demonstrating awareness and intent to maintain integrity verification.

5. **HTTPS for all external downloads:** All `curl` commands use HTTPS URLs, providing transport-layer integrity during build.

## Architecture Observations

1. **Manual SRI workflow:** SRI hashes are generated by the build script and must be manually copied into `.ezt` templates. This manual process is error-prone and explains the inconsistency between resources.

2. **Template engine (EZT):** The EZT templating system is used for HTML generation. No dynamic SRI hash injection is visible, suggesting hashes are hard-coded in templates.

3. **No Content-Security-Policy:** Neither template includes a CSP meta tag or suggests CSP headers. A CSP with `require-sri-for` (deprecated but illustrative) or strict source directives would complement SRI.
