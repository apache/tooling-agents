## Findings

### Medium Severity

#### ASVS-333-SEV-001: Session Cookie Missing `__Host-` Prefix

**Severity:** MEDIUM  
**Gap Type:** Type A — No control exists  
**Location:** `v3/server/main.py`, lines 44-46  
**Related Functions:** `v3/server/pages.py`, line 70 (`asfquart.session.read()`)

**Description:**

The application does not configure the session cookie with the `__Host-` prefix as required by ASVS 3.3.3. The session cookie carries authentication state (user ID, name, email) and is read on every authenticated page load via `basic_info()`. Without the `__Host-` prefix, the cookie is vulnerable to subdomain injection attacks.

**Vulnerable Code:**

App Construction (`main.py:44-46`):
```python
app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
```

No session cookie name configuration is provided. Quart's default session cookie name is `session`, which lacks the `__Host-` prefix.

Session Usage (`pages.py:70-76`):
```python
s = await asfquart.session.read()
if s:
    basic.update(
        uid=s['uid'],
        name=s['fullname'],
        email=s['email'],
    )
```

**Data Flow:**

```
Login (asfquart OAuth) → Session cookie set (name: "session") → Cookie sent on every request → basic_info() reads session → User identity established
```

**Impact:**

On the `apache.org` domain (which hosts this application per the OAuth configuration), the absence of `__Host-` prefix means:

1. **Subdomain cookie injection**: A compromised or attacker-controlled subdomain (e.g., `evil.apache.org`) could set a cookie named `session` for the parent domain, overriding the legitimate session cookie and performing session fixation.
2. **No browser-enforced binding**: Without `__Host-`, the browser does not enforce that the cookie was set with `Secure`, without a `Domain` attribute, and with `Path=/`. These constraints must be manually configured and maintained.
3. **Cookie tossing**: An attacker on any `*.apache.org` subdomain could "toss" a session cookie to the voting application, potentially hijacking sessions for an election system handling sensitive voting data.

**Proof of Concept:**

An attacker controlling any subdomain of `apache.org` could inject:
```
Set-Cookie: session=attacker-controlled-value; Domain=.apache.org; Path=/
```

This would override the legitimate session cookie in the victim's browser when visiting the STeVe application.

**Remediation:**

Configure the session cookie name to use the `__Host-` prefix in the Quart/asfquart application configuration:

```python
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    # Configure session cookie with __Host- prefix per ASVS 3.3.3
    app.config['SESSION_COOKIE_NAME'] = '__Host-session'
    app.config['SESSION_COOKIE_SECURE'] = True      # Required for __Host- prefix
    app.config['SESSION_COOKIE_HTTPONLY'] = True      # Defense in depth
    app.config['SESSION_COOKIE_SAMESITE'] = 'Lax'    # CSRF mitigation
    app.config['SESSION_COOKIE_PATH'] = '/'           # Required for __Host- prefix
    # Do NOT set SESSION_COOKIE_DOMAIN — __Host- prefix requires no Domain attribute

    import pages
    import api

    return app
```

If `asfquart` wraps cookie configuration differently, the equivalent configuration should be applied through its API. The key requirements for `__Host-` prefixed cookies are:
- `Secure` flag must be set
- `Path` must be `/`
- `Domain` attribute must **not** be set

## Architecture Observations

1. **Cookie management is fully delegated to framework**: The application relies entirely on `asfquart`/Quart for cookie handling. No explicit `set_cookie()` calls exist in application code, meaning the fix must be applied at the framework configuration level.

2. **Shared domain risk**: The application runs on `apache.org` infrastructure (per OAuth URLs), which hosts hundreds of services on subdomains. This amplifies the risk of subdomain cookie injection attacks.

3. **Single cookie dependency**: All authenticated state flows through a single session cookie (read in `basic_info()` which is called by every authenticated handler). Compromising this cookie compromises the entire user session.

4. **CSRF token gap compounds risk**: The placeholder CSRF token (`pages.py:75`) means that if session fixation succeeds via cookie injection, there is no secondary defense against cross-site request forgery on state-changing operations.
