**Key Findings:**
The application has **no CORS preflight mechanism** and **no alternative cross-origin request protection** (no Origin/Referer validation, no Content-Type enforcement, no custom header requirements, and CSRF tokens are placeholder-only). Multiple state-changing endpoints are exploitable from any cross-origin page. Two critical architectural patterns make CORS preflight protection impossible: state-changing GET endpoints and form-accepting POST endpoints using CORS-safelisted content types.

## Findings

### HIGH Severity

#### ASVS-352-SEV-001: State-Changing Operations via GET Bypass All CORS Preflight Protections

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `do_open_endpoint` (line 448), `do_close_endpoint` (line 469)
- **Lines:** 448-466, 469-484

**Affected Endpoints:**

| Route | Handler | Line | Operation |
|-------|---------|------|-----------|
| `GET /do-open/<eid>` | `do_open_endpoint` | `pages.py:448` | Opens an election |
| `GET /do-close/<eid>` | `do_close_endpoint` | `pages.py:469` | Closes an election |

**Related Functions Checked:**
- `@asfquart.auth.require` decorator (authentication only, no cross-origin protection)
- `@load_election` decorator (election loading, no security validation)
- `election.open()` method
- `election.close()` method

**Description:**

Two critical state-changing operations use GET requests, which architecturally **cannot** trigger CORS preflight checks. GET requests are always considered "simple requests" by the browser and will never initiate a preflight OPTIONS request, regardless of origin. This makes it impossible to use CORS preflight as a cross-origin protection mechanism for these endpoints.

**Vulnerable Code:**

```python
# pages.py:448-466
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    # ...
    return quart.redirect(f'/manage/{election.eid}', code=303)

# pages.py:469-484
@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz
    election.close()
    # ...
    return quart.redirect(f'/manage/{election.eid}', code=303)
```

**Data Flow:**

1. Attacker hosts malicious page on `attacker.com`
2. Authenticated victim visits attacker's page
3. Attacker's page contains: `<img src="https://steve.apache.org/do-close/KNOWN_EID">`
4. Browser sends GET request with victim's session cookie (no CORS preflight)
5. Server validates authentication (passes)
6. Server closes election
7. No cross-origin checks performed at any stage

**Proof of Concept:**

```html
<!-- Hosted on attacker.com, visited by authenticated admin -->
<html>
<body>
  <!-- Close an election silently -->
  <img src="https://steve.apache.org/do-close/abc123-election-id" style="display:none">
  <!-- Open an election prematurely -->
  <img src="https://steve.apache.org/do-open/def456-election-id" style="display:none">
</body>
</html>
```

**Impact:**

An attacker who knows or can guess an election ID (EIDs may be predictable UUIDs) can trick an authenticated committer into prematurely opening or closing any election they have access to. This directly compromises election integrity. GET requests **never** trigger CORS preflight — this is architecturally unfixable via CORS alone.

**Remediation:**

Change these to POST endpoints with anti-forgery protection:

```python
@APP.post('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    # Validate Origin or require custom header
    origin = quart.request.headers.get('Origin')
    if origin and origin not in ALLOWED_ORIGINS:
        quart.abort(403, 'Invalid origin')
    
    # Require non-safelisted Content-Type to force preflight
    content_type = quart.request.content_type or ''
    if 'application/json' not in content_type:
        quart.abort(415, 'Content-Type must be application/json')
    
    # ... existing logic ...
```

#### ASVS-352-SEV-002: POST Endpoints Accept CORS-Safelisted Content Types Without Cross-Origin Verification

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `do_vote_endpoint` (line 380), `do_create_endpoint` (line 425), `do_add_issue_endpoint` (line 490), `do_edit_issue_endpoint` (line 516), `do_delete_issue_endpoint` (line 540)
- **Lines:** 380-423, 425-446, 490-514, 516-538, 540-558

**Affected Endpoints (all use `quart.request.form` → `application/x-www-form-urlencoded`):**

| Route | Handler | Line | Operation |
|-------|---------|------|-----------|
| `POST /do-vote/<eid>` | `do_vote_endpoint` | `pages.py:380` | Submits votes |
| `POST /do-create-election` | `do_create_endpoint` | `pages.py:425` | Creates election |
| `POST /do-add-issue/<eid>` | `do_add_issue_endpoint` | `pages.py:490` | Adds issue |
| `POST /do-edit-issue/<eid>/<iid>` | `do_edit_issue_endpoint` | `pages.py:516` | Edits issue |
| `POST /do-delete-issue/<eid>/<iid>` | `do_delete_issue_endpoint` | `pages.py:540` | Deletes issue |

**Related Functions Checked:**
- `quart.request.form` (accepts form-urlencoded without validation)
- `@asfquart.auth.require` decorator (authentication only)
- `@load_election` decorator
- `election.add_vote()` method
- `steve.election.Election.create()` method
- `election.add_issue()` method
- `election.edit_issue()` method
- `election.delete_issue()` method

**Description:**

All state-changing POST endpoints accept `application/x-www-form-urlencoded` content type, which is a [CORS-safelisted content type](https://developer.mozilla.org/en-US/docs/Glossary/CORS-safelisted_request_header). Requests using this content type do **not** trigger CORS preflight checks, allowing cross-origin form submissions with credentials. No Origin header validation, Content-Type enforcement, custom header requirements, or CSRF token validation exists on any of these endpoints.

**Vulnerable Code (representative — `do_vote_endpoint`):**

```python
# pages.py:380-423
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    ### check authz              # <-- No authorization beyond "is committer"
    form = edict(await quart.request.form)  # Accepts form-encoded (CORS-safelisted)
    # ... no Origin check, no Content-Type enforcement, no CSRF validation ...
    for iid, votestring in votes.items():
        election.add_vote(result.uid, iid, votestring)
```

**Data Flow:**

1. Attacker hosts malicious page with HTML form on `attacker.com`
2. Authenticated victim visits attacker's page
3. JavaScript auto-submits form with `Content-Type: application/x-www-form-urlencoded`
4. Browser sends POST request with session cookie (no CORS preflight triggered)
5. Server validates authentication (passes)
6. Server processes form data and performs state-changing operation
7. No cross-origin checks performed at any stage

**Proof of Concept (vote manipulation):**

```html
<!-- Hosted on attacker.com -->
<html>
<body>
  <form id="voteForm" method="POST" 
        action="https://steve.apache.org/do-vote/target-election-id">
    <input type="hidden" name="vote-issue1" value="y">
    <input type="hidden" name="vote-issue2" value="n">
  </form>
  <script>document.getElementById('voteForm').submit();</script>
</body>
</html>
```

**Proof of Concept (election creation):**

```html
<form id="f" method="POST" action="https://steve.apache.org/do-create-election">
  <input type="hidden" name="title" value="Attacker's Fake Election">
</form>
<script>document.getElementById('f').submit();</script>
```

**Impact:**

- **Vote manipulation:** An attacker can cast or change votes on behalf of any authenticated committer who visits a malicious page during an open election
- **Election creation:** Unauthorized election creation under a PMC member's identity
- **Issue tampering:** Adding, editing, or deleting issues in elections

All of these requests use `Content-Type: application/x-www-form-urlencoded`, which is a CORS-safelisted content type and will **not** trigger a CORS preflight request.

**Remediation — Option A (Content-Type enforcement to force preflight):**

```python
def require_json_content_type(func):
    """Require application/json to force CORS preflight for cross-origin requests."""
    @functools.wraps(func)
    async def wrapper(*args, **kwargs):
        ct = quart.request.content_type or ''
        if 'application/json' not in ct:
            quart.abort(415, 'Content-Type must be application/json')
        return await func(*args, **kwargs)
    return wrapper
```

**Remediation — Option B (Custom header requirement):**

```python
def require_custom_header(func):
    """Require a custom header that forces CORS preflight."""
    @functools.wraps(func)
    async def wrapper(*args, **kwargs):
        if not quart.request.headers.get('X-Requested-With'):
            quart.abort(403, 'Missing required header')
        return await func(*args, **kwargs)
    return wrapper
```

**Remediation — Option C (Origin validation):**

```python
def validate_origin(func):
    """Check Origin header on state-changing requests."""
    @functools.wraps(func)
    async def wrapper(*args, **kwargs):
        origin = quart.request.headers.get('Origin')
        if origin and origin not in APP.cfg.allowed_origins:
            quart.abort(403, 'Disallowed origin')
        return await func(*args, **kwargs)
    return wrapper
```

### MEDIUM Severity

#### ASVS-352-SEV-003: JSON Endpoints Lack Explicit Content-Type Validation (Incidental Protection Only)

**Severity:** MEDIUM  
**Gap Type:** Type B — Control EXISTS incidentally but NOT explicitly CALLED

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `_set_election_date` (line 88), `do_set_open_at_endpoint` (line 368), `do_set_close_at_endpoint` (line 374)
- **Lines:** 88-108, 368-372, 374-378

**Affected Endpoints:**

| Route | Handler | Line |
|-------|---------|------|
| `POST /do-set-open_at/<eid>` | `do_set_open_at_endpoint` | `pages.py:368` |
| `POST /do-set-close_at/<eid>` | `do_set_close_at_endpoint` | `pages.py:374` |

**Related Functions Checked:**
- `quart.request.get_json()` (incidentally enforces Content-Type)
- `@asfquart.auth.require` decorator
- `@load_election` decorator
- `election.set_open_at()` method
- `election.set_close_at()` method

**Description:**

These endpoints use `quart.request.get_json()` without the `force=True` parameter, which incidentally requires `Content-Type: application/json`. This Content-Type is not CORS-safelisted, so it forces a preflight check. However, this protection is **incidental**, not intentional — the code does not explicitly validate the Content-Type header as a security control.

**Vulnerable Code:**

```python
# pages.py:88-108
async def _set_election_date(election, field):
    result = await basic_info()
    ### check authz
    data = await quart.request.get_json()   # <-- Incidentally requires application/json
    date_str = data.get('date')             # <-- Would throw AttributeError if data is None
    # ... no explicit Content-Type check, no Origin check ...
```

**Analysis:**

Quart's `get_json()` (without `force=True`) returns `None` when `Content-Type` is not `application/json`. This means `data.get('date')` would raise an `AttributeError`, effectively rejecting non-JSON requests. However:

1. This protection is **incidental**, not intentional — if the code is later changed to `get_json(force=True)` or adds a `None` check, the protection disappears
2. The error is an unhandled exception (500) rather than a proper 403/415, potentially leaking stack traces
3. No Origin header validation exists as a defense-in-depth measure

**Data Flow:**

1. Cross-origin request sent with `Content-Type: application/x-www-form-urlencoded`
2. `get_json()` returns `None` (no JSON body with correct Content-Type)
3. `data.get('date')` raises `AttributeError: 'NoneType' object has no attribute 'get'`
4. Unhandled exception returns 500 error (protection works but is fragile)

**Impact:**

Currently low exploitability due to Quart's default `get_json()` behavior, but fragile. The `application/json` Content-Type requirement incidentally forces CORS preflight, but this is not an explicit security control and could be accidentally removed during refactoring.

**Remediation:**

Make the Content-Type requirement explicit:

```python
async def _set_election_date(election, field):
    result = await basic_info()
    
    # Explicitly require application/json (forces CORS preflight for cross-origin)
    content_type = quart.request.content_type or ''
    if 'application/json' not in content_type:
        quart.abort(415, 'Content-Type must be application/json')
    
    data = await quart.request.get_json()
    if not data:
        quart.abort(400, 'Invalid JSON body')
    
    date_str = data.get('date')
    # ... rest of function ...
```

## Positive Security Patterns

1. **Authentication decorators consistently applied**: All state-changing endpoints require `@asfquart.auth.require({R.committer})` or higher, preventing unauthenticated access.

2. **POST for most write operations**: Most data-modifying operations use POST (except open/close — see findings).

3. **POST-Redirect-GET pattern**: POST handlers redirect with 303, reducing accidental resubmission.

4. **JSON endpoints incidentally force preflight**: The `_set_election_date` helper uses `get_json()` which effectively requires `application/json`, a non-safelisted Content-Type.

---

## Architecture Observations

1. **No CORS middleware or configuration exists anywhere in the codebase**. The application neither sets `Access-Control-Allow-Origin` headers nor explicitly restricts cross-origin access. This means it relies entirely on the browser's Same-Origin Policy and cookie SameSite attributes (which are not visible in the provided code).

2. **Cookie SameSite attribute is unknown**: The `asfquart.session` module's cookie configuration is not visible. If cookies default to `SameSite=None` or `SameSite=Lax`, the GET-based state-changing endpoints remain vulnerable even with cookies.

3. **The `api.py` module is a stub**: This suggests API endpoints may be added in the future. CORS and cross-origin controls should be established before API expansion.

4. **CSRF token infrastructure exists but is not implemented**: The code includes CSRF token generation (`pages.py:84`) but uses a placeholder value (`'placeholder'`) and never validates tokens on any endpoint.

## Recommendations Summary

### Immediate Actions (HIGH Priority)

**Address HIGH severity findings ASVS-352-SEV-001 and ASVS-352-SEV-002:**

1. **Convert GET state-changing endpoints to POST**: Change `/do-open/<eid>` and `/do-close/<eid>` from `@APP.get` to `@APP.post`. Update corresponding frontend links to forms or JavaScript fetch calls. GET requests for state changes can never be protected by CORS preflight.

2. **Implement cross-origin protection for all POST endpoints**: Choose at least one of:
   - **Recommended**: Require a custom header (e.g., `X-Steve-Request: 1`) on all state-changing POST endpoints. This forces CORS preflight and is framework-agnostic.
   - **Alternative**: Enforce `Content-Type: application/json` on all endpoints and switch from form parsing to JSON parsing.
   - **Alternative**: Validate the `Origin` header against a configured allowlist on all state-changing endpoints.

### Short-term Actions (MEDIUM Priority)

**Address MEDIUM severity finding ASVS-352-SEV-003:**

3. **Make Content-Type enforcement explicit on JSON endpoints**: Add explicit `Content-Type` validation in `_set_election_date` rather than relying on `get_json()` behavior.

4. **Verify SameSite cookie attribute**: Ensure session cookies are set with `SameSite=Strict` or at minimum `SameSite=Lax` as a defense-in-depth measure (note: `Lax` still allows top-level GET navigations).

### Long-term Actions

5. **Implement CSRF tokens**: Replace the `'placeholder'` CSRF token with real per-session tokens, validated on every state-changing request. This provides defense-in-depth alongside CORS preflight enforcement.

6. **Add CORS middleware**: If any endpoints need to be called cross-origin in the future, configure explicit CORS with restricted allowed origins, methods, and headers.

7. **Establish security review process**: Implement code review guidelines that require explicit cross-origin protection checks for all new state-changing endpoints.
