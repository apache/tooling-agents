## Findings

### Medium Severity

#### ASVS-375-MED-001: Complete Absence of Browser Security Feature Detection

| Field | Value |
|-------|-------|
| **Severity** | Medium |
| **Finding ID** | ASVS-375-MED-001 |
| **Gap Type** | Type A — Entry point with NO control |
| **File** | `v3/server/static/js/steve.js` |
| **Lines** | 1–76 (entire file) |
| **Functions Affected** | All application entry points |

**Related Functions Checked:**
- `showModal()` (lines 28-34)
- `validateRequiredField()` (lines 38-51)
- `submitFormWithLoading()` (lines 55-76)

**Description:**

The application's common JavaScript utility file — described as "Common JavaScript utilities for the Apache STeVe voting system" — contains zero browser security feature detection. The application implicitly depends on modern browser features (Bootstrap 5 Modal API, ES6 template literals, `classList` API, `const` declarations) but never checks whether the browser supports the security features the application relies upon.

For a voting system, the browser must support:
- **Content Security Policy (CSP)** — to prevent XSS
- **Strict-Transport-Security** — to enforce HTTPS
- **SameSite cookie attribute** — to mitigate CSRF
- **Secure cookie flag enforcement** — to prevent cookie leakage
- **SubtleCrypto / Web Crypto API** — if any client-side cryptographic operations are used

No feature detection, no user warning, and no access-blocking logic exists anywhere in the provided client-side code.

**Vulnerable Code (absence of control):**

```javascript
// steve.js — The ENTIRE common utility file contains only:
// 1. showModal() — UI helper
// 2. validateRequiredField() — Basic form validation
// 3. submitFormWithLoading() — Form submission helper
//
// MISSING: Any form of browser capability/security feature detection
// MISSING: Any warning or blocking for unsupported browsers
// MISSING: Any documented graceful degradation behavior
```

**Data Flow:**

```
User opens application in any browser (including outdated/insecure)
  → Application loads steve.js
  → No security feature checks performed
  → Application renders normally regardless of browser capabilities
  → Server-sent security headers (CSP, HSTS, etc.) may be silently ignored by old browsers
  → User believes they are protected but may not be
```

**Proof of Concept:**

A user accessing the voting application with an outdated browser (e.g., IE11, older Android WebView) that does not support CSP Level 2, SameSite cookies, or HSTS preloading would:

1. Receive the page normally with no warning
2. Have server-sent security headers silently ignored
3. Be vulnerable to attacks (XSS, session hijacking) that the security headers were designed to prevent
4. Have no indication their session is less secure than expected

**Impact:**

In a voting system, users on browsers lacking security feature support could be targeted with XSS or session hijacking attacks that would succeed due to missing CSP/HSTS enforcement. Voters may unknowingly cast votes on compromised sessions. The application provides false confidence that security is enforced uniformly.

**Remediation:**

Add a browser security feature detection module to `steve.js` that runs on page load:

```javascript
/**
 * Check browser support for required security features.
 * Warns or blocks users if critical features are missing.
 */
function checkBrowserSecurityFeatures() {
  const issues = [];

  // Check for Content Security Policy support (CSP Level 2)
  if (!window.SecurityPolicyViolationEvent) {
    issues.push('Content Security Policy (CSP)');
  }

  // Check for crypto API (if needed for vote integrity)
  if (!window.crypto || !window.crypto.subtle) {
    issues.push('Web Cryptography API');
  }

  // Check for Fetch API with credentials support
  if (!window.fetch) {
    issues.push('Fetch API');
  }

  // Check HTTPS enforcement
  if (location.protocol !== 'https:' && location.hostname !== 'localhost') {
    issues.push('Secure connection (HTTPS)');
  }

  // Check for SameSite cookie support (heuristic)
  if (!window.cookieStore && !document.cookie.includes('SameSite')) {
    // Indirect check; server should also enforce
  }

  if (issues.length > 0) {
    const warningDiv = document.createElement('div');
    warningDiv.className = 'alert alert-danger m-3';
    warningDiv.setAttribute('role', 'alert');
    warningDiv.textContent =
      'Your browser does not support the following security features required ' +
      'for secure voting: ' + issues.join(', ') + '. ' +
      'Please update your browser before proceeding.';
    document.body.prepend(warningDiv);

    // Optionally block access for critical missing features
    const forms = document.querySelectorAll('form');
    forms.forEach(form => {
      form.querySelectorAll('button[type="submit"], input[type="submit"]')
        .forEach(btn => { btn.disabled = true; });
    });

    console.warn('[STeVe Security] Unsupported browser features:', issues);
  }
}

// Run on DOM ready
if (document.readyState === 'loading') {
  document.addEventListener('DOMContentLoaded', checkBrowserSecurityFeatures);
} else {
  checkBrowserSecurityFeatures();
}
```

## Architecture Observations

1. **Single client-side JS file provided:** The common utility file (`steve.js`) is the natural location for cross-cutting concerns like browser feature detection. Its absence here strongly suggests it doesn't exist elsewhere in the client-side stack, representing a systemic gap rather than an isolated omission.

2. **Implicit modern browser dependency:** The code uses ES6+ features (template literals, `const`, arrow-like patterns in Bootstrap) which means very old browsers would fail with JavaScript errors — an uncontrolled failure rather than a documented graceful degradation. This creates unpredictable behavior rather than the controlled, documented behavior ASVS 3.7.5 requires.

3. **Voting system context amplifies risk:** Unlike a general web application, a voting system has integrity requirements where security feature gaps could directly undermine election validity. The absence of browser security checks in this context represents a higher risk than in typical web applications.

4. **No defense-in-depth for client security:** The application appears to rely entirely on server-side security headers without verifying client-side enforcement, creating a silent failure mode where users believe they are protected but may not be.
