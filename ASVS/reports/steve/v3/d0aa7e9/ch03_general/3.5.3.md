## Findings

### Critical Severity

#### ASVS-353-CRT-001: Election Open Operation Uses GET Method for State-Changing Functionality

**Severity:** CRITICAL  
**Gap Type:** Type B — Control EXISTS (POST used on 7 of 9 state-changing endpoints) but NOT CALLED on this endpoint  
**File:** `v3/server/pages.py`  
**Function:** `do_open_endpoint`  
**Lines:** ~536–553

**Related Functions Checked:**
- `election.open(pdb)` in `v3/steve/election.py` (~line 73)
- `election.add_salts()` in `v3/steve/election.py`
- `election.gather_election_data(pdb)` in `v3/steve/election.py`
- `crypto.gen_opened_key()` in crypto module

##### Vulnerable Code

```python
@APP.get('/do-open/<eid>')          # ← GET method for state-changing operation
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()

    ### check authz

    pdb = steve.persondb.PersonDB.open(DB_FNAME)

    # Open the Election.
    election.open(pdb)                # ← Irreversible state change via GET
    ...
    return quart.redirect(f'/manage/{election.eid}', code=303)
```

##### State Change Implementation

From `v3/steve/election.py` (`Election.open()`, ~line 73):

```python
def open(self, pdb):
    assert self.is_editable()
    self.add_salts()                          # Generates crypto salts for all voters
    edata = self.gather_election_data(pdb)    # Assembles election data
    salt = crypto.gen_salt()                  # Generates election-level salt
    opened_key = crypto.gen_opened_key(edata, salt)  # Derives tamper-detection key
    self.c_open.perform(salt, opened_key, self.eid)  # Writes to database
```

This operation:
1. Generates cryptographic salts for every voter/issue pair
2. Derives an anti-tamper key from the entire election definition
3. Transitions the election from `editable` → `open` (irreversible)
4. After this point, issues and voters **cannot** be added/removed

##### Data Flow

```
Attacker-controlled URL → GET /do-open/<eid> → election.open(pdb) → Database state change
                          ↑ No method restriction    ↑ No Sec-Fetch-* check
```

##### Proof of Concept

An attacker who knows an election ID can forge a request that a logged-in committer will trigger:

**Via image tag injection** (e.g., in any HTML context the victim renders):
```html
<img src="https://voting.example.org/do-open/abc123def456" style="display:none">
```

**Via link prefetch:**
```html
<link rel="prefetch" href="https://voting.example.org/do-open/abc123def456">
```

**Via simple link in email/chat:**
```
Click here for the election details: https://voting.example.org/do-open/abc123def456
```

When any authenticated committer's browser processes these, the GET request fires with their session cookie (assuming `SameSite=Lax` or `None`), opening the election prematurely.

##### Impact

- **Election integrity compromised**: An election can be opened before all issues/voters are configured, locking the election definition permanently
- **Cryptographic binding to incomplete data**: The `opened_key` anti-tamper hash binds to whatever data exists at the time of opening — missing voters or issues become permanent
- **No rollback**: The `open()` operation has no undo; once opened, the election cannot return to editable state
- **Trivially exploitable**: Requires only knowing an election ID (which appears in URLs and logs) and getting a committer to load an attacker-controlled page
- **Violates election integrity guarantees**: The cryptographic tamper-detection mechanism becomes meaningless if an attacker can force premature opening

##### Remediation

```python
# Change from GET to POST
@APP.post('/do-open/<eid>')          # ← Changed from @APP.get
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz — verify result.uid is the election owner
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    _LOGGER.info(f'User[U:{result.uid}] opened election[E:{election.eid}]')
    title = election.get_metadata().title
    await flash_success(f'Opened election: {title}')
    return quart.redirect(f'/manage/{election.eid}', code=303)
```

**Additional Required Changes:**
- Update corresponding template/JavaScript to use a form POST or fetch with `method: 'POST'` instead of a link/anchor tag
- Add CSRF token validation once CSRF protection is implemented (see TODO.md items #6/#8)

#### ASVS-353-CRT-002: Election Close Operation Uses GET Method for State-Changing Functionality

**Severity:** CRITICAL  
**Gap Type:** Type B — Control EXISTS (POST used on 7 of 9 state-changing endpoints) but NOT CALLED on this endpoint  
**File:** `v3/server/pages.py`  
**Function:** `do_close_endpoint`  
**Lines:** ~555–571

**Related Functions Checked:**
- `election.close()` in `v3/steve/election.py` (~line 94)
- `election.is_open()` in `v3/steve/election.py`

##### Vulnerable Code

```python
@APP.get('/do-close/<eid>')          # ← GET method for state-changing operation
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    result = await basic_info()

    ### check authz

    # Close the Election.
    election.close()                  # ← Irreversible state change via GET
    ...
    return quart.redirect(f'/manage/{election.eid}', code=303)
```

##### State Change Implementation

From `v3/steve/election.py` (`Election.close()`, ~line 94):

```python
def close(self):
    assert self.is_open()
    self.c_close.perform(self.eid)    # Sets closed=1 in database
```

This transitions the election from `open` → `closed`, permanently ending the voting period.

##### Data Flow

```
Attacker-controlled URL → GET /do-close/<eid> → election.close() → Database state change
                          ↑ No method restriction    ↑ No Sec-Fetch-* check
```

##### Proof of Concept

```html
<!-- Injected anywhere a logged-in committer will render -->
<img src="https://voting.example.org/do-close/abc123def456" width="0" height="0">
```

Or via browser navigation:
```
https://voting.example.org/do-close/abc123def456
```

##### Impact

- **Election disruption**: An active election can be closed prematurely, disenfranchising voters who haven't yet cast ballots
- **No rollback**: Once closed, the election cannot be reopened — the state machine only moves forward
- **Denial of service against democratic process**: A single forged GET request ends an entire election
- **Any committer can trigger**: The authorization check is `{R.committer}` (not owner-only), meaning any authenticated committer's session can be weaponized
- **Democratic legitimacy compromised**: Premature closure can invalidate election results and undermine trust in the voting system

##### Remediation

```python
# Change from GET to POST
@APP.post('/do-close/<eid>')         # ← Changed from @APP.get
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz — verify result.uid is the election owner
    election.close()
    _LOGGER.info(f'User[U:{result.uid}] closed election[E:{election.eid}]')
    title = election.get_metadata().title
    await flash_success(f'Closed election: {title}')
    return quart.redirect(f'/manage/{election.eid}', code=303)
```

**Additional Required Changes:**
- Update corresponding template/JavaScript to use a form POST or fetch with `method: 'POST'` instead of a link/anchor tag
- Add CSRF token validation once CSRF protection is implemented (see TODO.md items #6/#8)

## Positive Security Patterns

1. **Consistent POST usage for most mutations**: 7 of 9 state-changing endpoints correctly use `@APP.post()`, showing the development team understands the principle and has established the correct pattern throughout most of the codebase.

2. **Authentication on all sensitive routes**: All state-changing endpoints have `@asfquart.auth.require()` decorators, ensuring only authenticated users with appropriate roles can access sensitive functionality.

3. **POST-Redirect-GET pattern**: POST endpoints correctly redirect with `code=303`, preventing form resubmission issues and following HTTP best practices.

4. **Election state machine assertions**: `election.py` uses state assertions (`is_editable()`, `is_open()`) before performing operations, providing defense-in-depth against operations in wrong state (though see Architecture Observations regarding `assert` usage).

5. **Structured route naming**: The `do-` prefix convention for action endpoints makes it easy to audit which routes perform mutations and provides clear semantic separation between read and write operations.

6. **Authorization decorators**: Consistent use of `@asfquart.auth.require()` with role-based access control (e.g., `{R.committer}`) demonstrates security-conscious design.

---

## Architecture Observations

1. **No Sec-Fetch-* validation layer**: The application has no middleware or per-route validation of `Sec-Fetch-Site`, `Sec-Fetch-Mode`, or `Sec-Fetch-Dest` headers. Adding this as a middleware would provide defense-in-depth for all endpoints and serve as the alternative control specified in ASVS 3.5.3.

2. **TODO.md acknowledges CSRF debt**: The placeholder CSRF token (`basic.csrf_token = 'placeholder'`) and the TODO.md item #6/#8 show awareness of CSRF protection needs, but the GET method issue on open/close is a separate, more fundamental problem that CSRF tokens alone wouldn't fully address (GET requests can be triggered without any form interaction).

3. **`assert` used for security checks**: `election.py` uses Python `assert` for state validation (e.g., `assert self.is_editable()`, `assert self.is_open()`). These are disabled when Python runs with `-O` (optimized mode). State checks should use explicit `if`/`raise` patterns for security-critical validation to ensure they cannot be bypassed in production environments.

4. **State machine design is sound**: The election lifecycle (editable → open → closed) is well-designed with clear transitions. The security issue is purely in the HTTP method choice for triggering these transitions, not in the state machine logic itself.

5. **Inconsistent application of security pattern**: The fact that 7 of 9 endpoints use the correct method suggests this is a development oversight rather than a systemic misunderstanding, making remediation straightforward.
