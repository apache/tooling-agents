## Findings

### Medium Severity

#### ASVS-373-MED-001: Complete Absence of External URL Navigation Warning

| Attribute | Detail |
|-----------|--------|
| **Finding ID** | ASVS-373-MED-001 |
| **Severity** | **MEDIUM** |
| **Gap Type** | Type A — Entry point with NO control |
| **ASVS Requirement** | 3.7.3 — Missing external URL navigation warning |
| **File** | `v3/server/pages.py` |
| **Function** | `rewrite_description()` (lines 52-59), application-wide |
| **Lines** | 52-59, 349-350 |

**Related Functions Checked:**
- `rewrite_description()` (L52-59)
- `vote_on_page()` (L268-354)
- `do_add_issue_endpoint()` (issue creation)
- `do_edit_issue_endpoint()` (issue modification)
- All `quart.redirect()` calls (server-side redirects)

**Description:**

The application has no mechanism whatsoever to warn users before navigating to URLs outside the application's control. There is no interstitial warning page, no client-side JavaScript intercept for external links, and no server-side redirect proxy. This is a complete absence of the ASVS 3.7.3 control.

The `rewrite_description()` function injects unescaped HTML into the page, allowing arbitrary HTML including external links to be rendered directly to voters without any warning or cancellation option.

**Vulnerable Code:**

```python
# v3/server/pages.py, lines 52-59
def rewrite_description(issue):
    """Rewrite issue description: wrap in <pre> and convert doc:filename to links."""
    import re
    desc = issue.description
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'  # No HTML escaping
```

Rendered in the `vote_on_page()` handler:

```python
# v3/server/pages.py, lines 349-350
for issue in result.issues:
    rewrite_description(issue)
```

**Data Flow:**

```
Source: Issue description (set by admin via do_add_issue_endpoint / do_edit_issue_endpoint)
    ↓
Stored in database unescaped
    ↓
Retrieved by vote_on_page() via election.list_issues()
    ↓
Passed through rewrite_description() — no HTML escaping applied
    ↓
Rendered as raw HTML in template
    ↓
External <a href="..."> tags rendered as clickable links
    ↓
User navigates to external URL with NO warning or cancellation option
```

**Proof of Concept:**

1. An election administrator creates an issue via `POST /do-add-issue/<eid>` with:
   ```
   title=Board+Vote&description=See+details+at+<a+href="https://attacker.example.com/phishing">official+documentation</a>
   ```

2. A voter visits `GET /vote-on/<eid>` and sees the rendered description containing a clickable external link.

3. Clicking the link navigates directly to `https://attacker.example.com/phishing` with no interstitial warning and no option to cancel.

**Impact:**

- Voters can be directed to phishing sites that mimic the voting application, potentially capturing credentials or manipulating vote decisions
- In an election context, this could be used for social engineering to influence voter behavior
- No audit trail of outbound navigation exists
- Violates ASVS 3.7.3 Level 3 requirement completely

**Remediation:**

Implement a three-part solution:

**Part 1 — Server-side redirect proxy route:**

```python
from urllib.parse import urlparse, quote

ALLOWED_DOMAINS = {'your-app-domain.example.org'}

@APP.get('/redirect-warning')
@asfquart.auth.require
@APP.use_template(TEMPLATES / 'redirect_warning.ezt')
async def redirect_warning():
    result = await basic_info()
    result.title = 'External Link Warning'
    target_url = quart.request.args.get('url', '')
    
    parsed = urlparse(target_url)
    if not parsed.scheme or not parsed.netloc:
        quart.abort(400, 'Invalid URL')
    
    if parsed.netloc in ALLOWED_DOMAINS:
        return quart.redirect(target_url, code=303)
    
    result.target_url = target_url
    result.target_domain = parsed.netloc
    return result
```

**Part 2 — Interstitial template (`redirect_warning.ezt`):**

```html
<h2>You are leaving this application</h2>
<p>You are about to navigate to an external website:</p>
<p><strong>[target_domain]</strong></p>
<p>This site is not controlled by this application. Proceed at your own risk.</p>
<a href="[target_url]" rel="noopener noreferrer">Continue to external site</a>
<a href="javascript:history.back()">Cancel and go back</a>
```

**Part 3 — HTML escaping in `rewrite_description()` and client-side interception:**

```python
import html

def rewrite_description(issue):
    desc = html.escape(issue.description)  # Escape HTML first
    import re
    def repl(match):
        filename = html.escape(match.group(1))
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

Add client-side JavaScript to intercept any remaining external links:

```javascript
document.addEventListener('click', function(e) {
    const link = e.target.closest('a[href]');
    if (link && link.hostname !== window.location.hostname) {
        e.preventDefault();
        window.location = '/redirect-warning?url=' + encodeURIComponent(link.href);
    }
});
```

## Positive Security Patterns

1. **All server-side redirects use hardcoded internal paths**: Every `quart.redirect()` in the codebase uses a hardcoded path prefix (e.g., `/manage/`, `/vote-on/`, `/voter`), preventing open redirect vulnerabilities through server-side redirects.

2. **Authentication required on all sensitive pages**: All voter-facing and admin-facing routes use `@asfquart.auth.require`, ensuring unauthenticated users cannot reach pages that may contain external links.

3. **Document serving uses `send_from_directory`**: The `/docs/` route uses Quart's `send_from_directory` which provides basic path containment, preventing redirect-via-file-serving attacks.

---

## Architecture Observations

1. **No centralized URL policy**: The application has no concept of "internal vs external" URLs. There is no allowlist of trusted domains, no URL rewriting middleware, and no external link detection anywhere in the codebase.

2. **Template rendering not auditable**: The EZT templates are not provided for review. External links could also be hardcoded in templates (e.g., links to documentation sites, Apache Foundation URLs, etc.) — all of which would bypass any server-side control.

3. **Issue descriptions are admin-controlled but unescaped**: While descriptions originate from authenticated administrators (not arbitrary users), the lack of HTML escaping means any HTML including external links is rendered verbatim to voters. This creates a trust boundary violation even within the authenticated user population.

4. **Single point of failure for content sanitization**: All issue content flows through `rewrite_description()`, making it a critical control point that currently provides no security protection.
