## Findings

### Medium Severity

#### ASVS-335-MED-001: No Cookie Size Validation Control

**Severity:** MEDIUM  
**Gap Type:** Type A — No cookie size validation control exists anywhere in the application layer  
**File:** `v3/server/pages.py` (application-wide concern)  
**ASVS Requirement:** 3.3.5 — Cookie name + value ≤ 4096 bytes

**Location:**
- **File:** `v3/server/pages.py`
- **Functions affected:** All functions using session/flash (application-wide)
- **Lines:** 63-94 (session read), 73-78 (flash consumption), 356, 519 (flash writes)

**Related Functions Checked:**
- `basic_info()` — lines 63-94
- Flash message handlers — lines 73-78, 121-128
- Vote processing — line 519
- Election creation — line 356

**Description:**

The application has no mechanism to validate or enforce the 4096-byte cookie size limit. All session cookie management is delegated to the Quart/`asfquart` framework with no application-level guard. While the current session payload (uid, fullname, email, flash messages) is likely small enough, there is no defensive control preventing oversized cookies if session data grows (e.g., additional session attributes, accumulated data from framework internals, or future code changes).

**Vulnerable Code:**

Session data written to cookie (indirect via framework):
```python
# lines 80-87 — session data read, implying it was written at login
s = await asfquart.session.read()
if s:
    basic.update(
        uid=s['uid'],
        name=s['fullname'],
        email=s['email'],
    )
```

Flash messages contribute to session/cookie size:
```python
# lines 73-78 — flash messages stored in session until consumed
basic.flashes = [
    edict(message=f[1], category=f[0])
    for f in quart.get_flashed_messages(with_categories=True)
]
```

Multiple flash paths write user-influenced content into the session:
```python
# line 519 — vote processing stores user-controlled iid in flash
await flash_danger(f'Invalid issue ID: {iid}')

# line 356 — election title from form stored in flash
await flash_success(f'Created election: {form.title}')
```

**Data Flow:**

User action → `quart.flash()` with dynamic content → session store → session cookie written by framework → no size check before browser receives it.

**Proof of Concept:**

Under the current codebase, practical exploitation is limited because:
1. Flash messages are consumed on the next page render
2. Session fields are small strings
3. No endpoint allows unbounded session data injection

However, if `asfquart.session` uses signed/encrypted cookie-based sessions (Quart's default), the encoding overhead (base64 + HMAC signature ≈ 30-40% inflation) reduces the effective payload capacity to approximately 2,900 bytes — leaving less headroom than expected.

**Impact:**

If the session cookie exceeds 4096 bytes (through future code changes, framework overhead growth, or unforeseen session data accumulation), the browser will silently discard it. The user's session would effectively be invalidated, preventing authentication and use of all protected functionality. This is a denial-of-service condition against individual users.

**Remediation:**

Implement middleware that validates cookie size before the response is sent:

```python
@APP.after_request
async def validate_cookie_size(response):
    MAX_COOKIE_SIZE = 4096
    for header_name, header_value in response.headers:
        if header_name.lower() == 'set-cookie':
            # Extract cookie name=value portion (before attributes)
            cookie_nv = header_value.split(';')[0]
            if len(cookie_nv.encode('utf-8')) > MAX_COOKIE_SIZE:
                _LOGGER.error(
                    f'Cookie exceeds {MAX_COOKIE_SIZE} bytes: '
                    f'{len(cookie_nv.encode("utf-8"))} bytes'
                )
                # Take corrective action: clear session, log, alert
                # Do NOT silently send the oversized cookie
    return response
```

## Architecture Observations

1. **Cookie management is entirely framework-delegated**: `pages.py` never calls `set_cookie()` directly. All cookies are managed through `asfquart.session` and Quart's session layer. This is architecturally clean but means cookie size validation must happen in the framework layer, which was not provided for review.

2. **Session storage mechanism unknown**: Whether `asfquart.session` uses server-side session storage (small cookie with session ID) or client-side cookie storage (all data in cookie) is critical to this ASVS requirement. Server-side sessions would make oversized cookies nearly impossible; client-side sessions make them possible.

3. **Flash messages contain dynamic content**: Several flash calls embed user-controlled data (election titles, issue IDs). While these are small strings, this pattern could compound if error handling becomes more verbose.
