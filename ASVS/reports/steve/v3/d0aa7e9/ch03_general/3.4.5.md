## Findings

### Medium Severity

#### ASVS-345-MED-001: Missing Referrer-Policy Header on All Application Responses

**Severity:** MEDIUM

**Gap Type:** Type A — Entry points with NO control

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `create_app()`
- **Lines:** 31–47
- **Additional File:** `v3/server/pages.py` — all route handlers (no `after_request` hook defined)

**Related Functions Checked:**
- `create_app()` (main.py:31–47)
- `run_standalone()` (main.py:50–84)
- `run_asgi()` (main.py:90–109)
- All route handlers in `pages.py` (lines 125–602)

**Description:**

The application does not set a `Referrer-Policy` HTTP response header on any responses, nor is there evidence of HTML meta tag configuration in the provided code. This violates ASVS requirement 3.4.5 and exposes sensitive election identifiers, issue IDs, and document names in URL paths to third-party services via the browser's `Referer` header.

**Vulnerable Code (main.py:31–47):**

```python
def create_app():
    "Create the asfquart app and its endpoints."

    ### is this really needed right now?
    # Avoid OIDC
    import asfquart.generics

    asfquart.generics.OAUTH_URL_INIT = (
        'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
    )
    asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'

    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    # Now that we have an APP, import modules that will add page
    # and API endpoints into the APP.
    import pages  # noqa: F401
    import api  # noqa: F401

    return app  # also available as asfquart.APP now
```

No `Referrer-Policy` header is configured at the application level, and no `after_request` handler sets it on any response.

**Data Flow:**

1. User navigates to a sensitive page (e.g., `/vote-on/abc123` or `/manage-stv/abc123/issue456`)
2. The HTML response is rendered and served **without** a `Referrer-Policy` header
3. If any page contains links to third-party resources (e.g., CDN-hosted CSS/JS, external documentation links) or the user clicks an external link, the browser sends the **full URL** including the path (election ID, issue ID, document name) in the `Referer` header to the third party
4. Third-party services learn internal election identifiers and navigation patterns

**Sensitive URL Paths Exposed:**

- `/vote-on/<eid>` — reveals which election a user is voting on
- `/manage/<eid>` — reveals election management activity
- `/manage-stv/<eid>/<iid>` — reveals both election and issue identifiers
- `/docs/<iid>/<docname>` — reveals issue IDs and supporting document names
- `/do-open/<eid>`, `/do-close/<eid>` — reveals election state-change operations (these are GET requests that leak via Referer on the subsequent redirect target's page load)

**Proof of Concept:**

```
1. User visits: https://steve.example.org/vote-on/election-2024-board
2. That page loads a third-party resource or user clicks an external link
3. Browser sends request to third party with header:
   Referer: https://steve.example.org/vote-on/election-2024-board
4. Third party now knows the election identifier and that voting is occurring
```

**Impact:**

- Election identifiers, issue identifiers, and document names leak to any third-party service referenced in the HTML pages
- For this election/voting system, leaking which elections exist and are active has organizational sensitivity
- Combined with timing information from the Referer, an observer could correlate voting activity patterns

**Remediation:**

Add a global `after_request` handler that sets `Referrer-Policy` on all responses. For an election system, `strict-origin-when-cross-origin` (minimum) or `no-referrer` (strictest) is recommended:

```python
# In main.py create_app(), after app construction:
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    @app.after_request
    async def set_security_headers(response):
        # Prevent leakage of election/issue IDs in URL paths
        response.headers['Referrer-Policy'] = 'strict-origin-when-cross-origin'
        # Consider also adding other security headers here:
        # response.headers['X-Content-Type-Options'] = 'nosniff'
        # response.headers['X-Frame-Options'] = 'DENY'
        return response

    import pages
    import api

    return app
```

For maximum protection (recommended for a voting system):

```python
response.headers['Referrer-Policy'] = 'no-referrer'
```

Alternatively, if templates are controlled, a fallback HTML meta tag can be added in the base template:

```html
<meta name="referrer" content="strict-origin-when-cross-origin">
```

## Architecture Observations

1. **No security header middleware exists anywhere.** The application factory (`create_app`) configures OAuth URLs and imports route modules but does not set any HTTP security headers. This is a systemic gap — not just for Referrer-Policy, but likely for other security headers (CSP, X-Frame-Options, etc.) as well.

2. **State-changing GET endpoints are particularly vulnerable.** The routes `GET /do-open/<eid>` and `GET /do-close/<eid>` perform state changes via GET requests. After the redirect (303), the next page load will send the previous URL (containing the election ID and operation type) as the Referer to any third-party resources loaded on the redirect target page.

3. **The `asfquart` framework** is used as the application foundation, but there is no evidence from the provided code that it sets `Referrer-Policy` by default. The application must explicitly configure this.
