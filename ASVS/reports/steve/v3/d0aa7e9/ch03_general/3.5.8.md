## Findings

### High Severity Findings

#### Finding ASVS-358-HIGH-001

**Cross-Origin Resource Loading of Authenticated Documents**

| Attribute | Value |
|-----------|-------|
| **Severity** | HIGH |
| **Finding ID** | ASVS-358-HIGH-001 |
| **File** | `v3/server/pages.py` |
| **Function** | `serve_doc()` (lines 587–603) |
| **Gap Type** | Type A — No control exists |

**Description:**

The `/docs/<iid>/<docname>` endpoint serves authenticated documents (images, scripts, PDFs, and other files associated with election issues) without setting a `Cross-Origin-Resource-Policy` response header and without validating `Sec-Fetch-*` request headers. This allows a malicious cross-origin page to embed or load these authenticated resources on behalf of a logged-in user.

**Vulnerable Code:**

```python
# Serve supporting documents for an issue
@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require  # fine-grained per-doc authz within the handler
async def serve_doc(iid, docname):
    result = await basic_info()  # get PID

    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)

    ### verify the propriety of DOCNAME.  # <-- validation TODO noted but not implemented

    # Return per-issue document — NO CORP header, NO Sec-Fetch check
    return await quart.send_from_directory(DOCSDIR / iid, docname)
```

**Data Flow:**

1. **Source:** Cross-origin page controlled by attacker includes `<img src="https://voting-app.example.com/docs/ISSUE_ID/secret-ballot.pdf">`
2. **Sink:** `quart.send_from_directory()` returns the authenticated document
3. **Missing Control:** No `Sec-Fetch-Site` validation to reject cross-origin requests; No `Cross-Origin-Resource-Policy: same-origin` header on the response

**Proof of Concept:**

```html
<!-- Attacker-controlled page at evil.example.com -->
<html>
<body>
  <!-- Load authenticated election document cross-origin -->
  <img src="https://voting-app.example.com/docs/abc123/candidate-photo.jpg" 
       onload="exfiltrate(this)" onerror="tryNext()">
  
  <!-- Embed as script to probe for existence/size -->
  <script src="https://voting-app.example.com/docs/abc123/notes.js"></script>
  
  <!-- Or embed in iframe -->
  <iframe src="https://voting-app.example.com/docs/abc123/ballot-details.pdf"></iframe>
</body>
</html>
```

If the user is authenticated and visits the attacker page, authenticated documents are loaded cross-origin. Even if content cannot be directly read (due to Same-Origin Policy for some resource types), image dimensions, load/error timing, and existence probing are possible.

**Impact:**

- Authenticated election documents (images, PDFs, scripts) can be loaded by cross-origin pages when the user has an active session
- Attackers can confirm existence of specific documents and issues
- Image content is directly rendered; document metadata leaks via timing/size
- Election-sensitive material (candidate information, ballot details referenced via `doc:filename` in issue descriptions) exposed

**Remediation:**

```python
@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require
async def serve_doc(iid, docname):
    # Validate Sec-Fetch-* headers to ensure same-origin navigation
    sec_fetch_site = quart.request.headers.get('Sec-Fetch-Site', '')
    sec_fetch_dest = quart.request.headers.get('Sec-Fetch-Dest', '')
    
    # Only allow same-origin or same-site requests
    if sec_fetch_site not in ('same-origin', 'same-site', 'none'):
        quart.abort(403)
    
    # Only allow document/image/empty destinations (navigation, link clicks)
    if sec_fetch_dest not in ('document', 'image', 'empty', ''):
        quart.abort(403)
    
    result = await basic_info()
    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)

    response = await quart.send_from_directory(DOCSDIR / iid, docname)
    response.headers['Cross-Origin-Resource-Policy'] = 'same-origin'
    return response
```

#### Finding ASVS-358-HIGH-002

**State-Changing GET Endpoints Vulnerable to Cross-Origin Resource Embedding**

| Attribute | Value |
|-----------|-------|
| **Severity** | HIGH |
| **Finding ID** | ASVS-358-HIGH-002 |
| **File** | `v3/server/pages.py` |
| **Functions** | `do_open_endpoint()` (line 462), `do_close_endpoint()` (line 481) |
| **Gap Type** | Type A — No control exists |

**Description:**

The `/do-open/<eid>` and `/do-close/<eid>` endpoints perform irreversible state-changing operations (opening and closing elections) via GET requests. Combined with the complete absence of `Sec-Fetch-*` header validation, these endpoints can be triggered by a cross-origin page embedding the URL as a resource (e.g., `<img>`, `<link>`, `<script>`). This is distinct from general CSRF because the attack vector is specifically through cross-origin resource loading — the mechanism ASVS 3.5.8 is designed to prevent.

**Vulnerable Code:**

```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz  # <-- authz TODO not implemented
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)  # Irreversible state change
    # ...
    return quart.redirect(f'/manage/{election.eid}', code=303)


@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz  # <-- authz TODO not implemented
    election.close()  # Irreversible state change
    # ...
    return quart.redirect(f'/manage/{election.eid}', code=303)
```

**Data Flow:**

1. **Source:** Cross-origin page includes `<img src="https://voting-app.example.com/do-open/KNOWN_EID">`
2. **Sink:** `election.open(pdb)` or `election.close()` executes the irreversible state transition
3. **Missing Control:** No `Sec-Fetch-Dest` check to reject `image`/`script`/`style` destinations; No `Sec-Fetch-Site` check to reject `cross-site` origin; No `Sec-Fetch-Mode` check to reject `no-cors` mode

**Proof of Concept:**

```html
<!-- Attacker page: forces election open/close via resource embedding -->
<html>
<body>
  <!-- Trigger election opening via image embed -->
  <img src="https://voting-app.example.com/do-open/e_abc123def" width="1" height="1">
  
  <!-- Or close an election -->
  <img src="https://voting-app.example.com/do-close/e_xyz789" width="1" height="1">
  
  <!-- Multiple elections can be targeted simultaneously -->
  <link rel="prefetch" href="https://voting-app.example.com/do-open/e_target2">
</body>
</html>
```

When a committer visits this page while authenticated, the browser sends GET requests with session cookies. The server has no way to distinguish these from legitimate navigations because `Sec-Fetch-*` headers are not checked.

**Impact:**

- **Election integrity compromise:** An attacker can force-open elections prematurely (before proper voter rolls, issues, or dates are finalized)
- **Denial of service:** An attacker can force-close open elections, permanently ending voting
- Both operations are explicitly irreversible (`assert self.is_editable()` / `assert self.is_open()` — once changed, cannot revert)
- The `election.py` code confirms: `open()` triggers salt generation and key derivation (line 88-95); `close()` permanently marks the election closed (line 120)

**Remediation:**

```python
# 1. Change to POST methods (defense-in-depth alongside Sec-Fetch)
@APP.post('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    validate_sec_fetch(quart.request)  # See middleware below
    # ... existing logic ...

@APP.post('/do-close/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    validate_sec_fetch(quart.request)
    # ... existing logic ...

# 2. Update the JavaScript in manage.ezt template:
# Change: window.location.href = '/do-open/[eid]';
# To:     fetch('/do-open/[eid]', { method: 'POST', headers: {...} })
#         .then(r => window.location.href = '/manage/[eid]');
```

### Medium Severity Findings

#### Finding ASVS-358-MEDIUM-003

**Systemic Absence of Cross-Origin Resource Protection**

| Attribute | Value |
|-----------|-------|
| **Severity** | MEDIUM |
| **Finding ID** | ASVS-358-MEDIUM-003 |
| **File** | `v3/server/pages.py` |
| **Function** | All authenticated endpoints (global) |
| **Gap Type** | Type A — No control exists at any layer |

**Description:**

The application has no global mechanism — neither middleware, `after_request` hook, nor per-endpoint logic — to set `Cross-Origin-Resource-Policy` response headers or validate `Sec-Fetch-*` request headers on any response. This is a systemic architectural gap affecting all 15+ authenticated endpoints. ASVS 3.5.8 requires one of these mechanisms; neither is present.

**Vulnerable Code (representative — `basic_info()` and all endpoints):**

```python
async def basic_info():
    """Return base-level EZT template data."""
    basic = edict()
    # ... session data ...
    ### generate a real token and store in the session
    basic.csrf_token = 'placeholder'  # No CORP/Sec-Fetch logic anywhere
    return basic

# No @APP.after_request or middleware adding CORP headers
# No Sec-Fetch validation decorator or utility function exists
```

**Impact:**

- No browser-enforced cross-origin resource blocking on any authenticated response
- Authenticated HTML pages can be iframed by malicious sites (clickjacking vector; no `X-Frame-Options` visible either)
- Cross-origin scripts can probe authenticated endpoints for timing/error-based information disclosure
- The application relies solely on Same-Origin Policy, which does not prevent resource loading (only reading in some contexts)

**Remediation — Global Middleware:**

```python
@APP.after_request
async def add_security_headers(response):
    """Set Cross-Origin-Resource-Policy and validate Sec-Fetch-* on all responses."""
    # Set CORP on all responses to block cross-origin resource loads
    response.headers['Cross-Origin-Resource-Policy'] = 'same-origin'
    
    # Additional defense-in-depth headers
    response.headers['X-Frame-Options'] = 'DENY'
    response.headers['X-Content-Type-Options'] = 'nosniff'
    
    return response

def validate_sec_fetch(request):
    """Validate Sec-Fetch-* headers for state-changing and sensitive endpoints."""
    sec_fetch_site = request.headers.get('Sec-Fetch-Site', '')
    
    # For browsers that send Sec-Fetch-*, reject cross-origin
    if sec_fetch_site and sec_fetch_site not in ('same-origin', 'same-site', 'none'):
        quart.abort(403, 'Cross-origin request blocked')
    
    # For state-changing endpoints, also check mode
    sec_fetch_mode = request.headers.get('Sec-Fetch-Mode', '')
    if sec_fetch_mode and sec_fetch_mode == 'no-cors':
        quart.abort(403, 'Unexpected request mode')

# Apply to sensitive endpoints as a decorator:
def require_same_origin(func):
    @functools.wraps(func)
    async def wrapper(*args, **kwargs):
        validate_sec_fetch(quart.request)
        return await func(*args, **kwargs)
    return wrapper
```

## Positive Security Patterns

The following security patterns are correctly implemented and should be maintained:

1. **Authentication decorators consistently applied**: All sensitive endpoints use `@asfquart.auth.require` with appropriate role requirements (`R.committer`, `R.pmc_member`).

2. **Template escaping for JavaScript context**: The `manage.ezt` template uses `[format "js,html"]` when passing issue titles and descriptions to JavaScript functions, preventing XSS in that context.

3. **Authorization check on document serving**: The `serve_doc()` function performs per-document authorization via `q_get_mayvote.first_row(result.uid, iid)` before serving files.

4. **State machine enforcement in election.py**: Assertions like `assert self.is_editable()` and `assert self.is_open()` enforce correct state transitions server-side, preventing logical misuse even if browser controls are bypassed.

5. **Most state-changing operations use POST**: Endpoints like `/do-vote`, `/do-create-election`, `/do-add-issue`, `/do-edit-issue`, `/do-delete-issue`, `/do-set-open_at`, `/do-set-close_at` correctly use POST method.

6. **Sensitive metadata exclusion**: `get_metadata()` in `election.py` explicitly excludes `salt` and `opened_key` from returned data.

---

## Architecture Observations

1. **No middleware layer for security headers**: The application has no `@APP.after_request` or equivalent hook to add security headers globally. This is the root cause of the systemic CORP gap.

2. **Mixed GET/POST for state changes**: While most mutation endpoints correctly use POST, the critical `do-open` and `do-close` endpoints use GET, making them uniquely vulnerable to cross-origin resource embedding attacks.

3. **Template-driven JavaScript navigation**: The `manage.ezt` template uses `window.location.href` for state-changing actions, which generates simple GET navigations. This pattern drives the GET method choice for those endpoints.

4. **Placeholder security controls**: The `csrf_token = 'placeholder'` pattern (line 78) and multiple `### check authz` TODO comments indicate security controls that are designed-for but not yet implemented, creating false confidence if the template infrastructure suggests they're active.
