**Key Findings:**

The application exhibits several type safety deficiencies that violate ASVS 15.3.5. The most critical issues are:

1. **Complete absence of vote value validation** against expected vote types, creating a Type B gap where the need is explicitly acknowledged but unimplemented
2. **Use of Python `assert` statements** for security-critical type and state validation, which are silently stripped in optimized runtime mode
3. **Missing type checks on JSON-deserialized data** used throughout the election lifecycle

### HIGH Severity

#### ASVS-1535-HIGH-001: Missing Vote Value Type/Format Validation (Type B Gap)

**Severity:** HIGH  
**Gap Type:** Type B — Control is explicitly identified as needed but NOT implemented

**Location:**
- **Primary File:** `v3/steve/election.py`
- **Function:** `add_vote`
- **Lines:** 244-260
- **Also Affects:** `v3/server/pages.py`, `do_vote_endpoint`, lines 415-455

**Related Functions Checked:**
- `election.add_vote()` (election.py:244-260)
- `pages.do_vote_endpoint()` (pages.py:415-455)
- `crypto.create_vote()` (called from add_vote)
- `vtypes.vtype_module()` (available but not used for validation)

**Description:**

The application accepts vote values from users without validating them against the expected vote type for the issue. The code contains an explicit TODO comment acknowledging this validation requirement, but the validation is not implemented. This allows arbitrary strings to be encrypted and stored as votes, regardless of whether they match the issue's voting format (YNA, STV, etc.).

**Vulnerable Code:**

```python
# election.py, add_vote():
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting    # <-- EXPLICITLY ACKNOWLEDGED, NOT DONE

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

```python
# pages.py, do_vote_endpoint():
for iid, votestring in votes.items():
    if iid not in issue_dict:
        await flash_danger(f'Invalid issue ID: {iid}')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
    # No validation that votestring matches issue_dict[iid].vtype
    try:
        election.add_vote(result.uid, iid, votestring)
```

**Data Flow:**

```
User form POST → quart.request.form → votes dict → election.add_vote(votestring) 
→ crypto.create_vote() → database
                              ↑
                     NO TYPE/FORMAT CHECK HERE
```

**Proof of Concept:**

```http
POST /do-vote/abc123 HTTP/1.1
Content-Type: application/x-www-form-urlencoded

vote-issue1=INVALID_GARBAGE_VALUE&vote-issue2={"__proto__":{"admin":true}}
```

For a YNA issue (should only accept "yes"/"no"/"abstain"), any arbitrary string is accepted and encrypted. For an STV issue (should accept a valid ranking), any string bypasses format validation.

**Impact:**

Election integrity is compromised. Invalid votes are encrypted and stored, then when tallied by `vtypes` modules, the behavior is unpredictable — tallying may crash, produce incorrect results, or silently discard/miscount votes. An attacker who is an eligible voter can submit malformed votes to corrupt STV tallying results.

**Remediation:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."
    md = self._all_metadata(self.S_OPEN)

    # Validate votestring type and format
    if not isinstance(votestring, str):
        raise ValueError(f"votestring must be a string, got {type(votestring).__name__}")

    issue = self.q_get_issue.first_row(iid)
    if not issue:
        raise IssueNotFound(iid)

    # Validate against the issue's vote type
    m = vtypes.vtype_module(issue.type)
    if not m.validate_vote(votestring, self.json2kv(issue.kv)):
        raise ValueError(f"Invalid vote format for {issue.type}: {votestring!r}")

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

#### ASVS-1535-HIGH-002: Security-Critical Validation Uses `assert` Statements

**Severity:** HIGH  
**Gap Type:** Type C — Control is called but result is ignorable (Python `-O` removes all `assert`)

**Location:**
- **File:** `v3/steve/election.py`
- **Functions:** Multiple methods throughout the file
- **Lines:** 57, 76, 127, 183, 201, 214, 236, 356

**Related Functions Checked:**
- `delete()` (line ~57)
- `open()` (line ~76)
- `add_salts()` (line ~127)
- `add_issue()` (line ~183)
- `edit_issue()` (line ~201)
- `delete_issue()` (line ~214)
- `add_voter()` (line ~236)
- `_compute_state()` (line ~356)

**Description:**

The application uses Python `assert` statements for security-critical type and state validation throughout the election lifecycle. Python's `assert` statement is a debugging aid that is **completely removed** when the interpreter runs with the `-O` (optimize) flag (`python -O` or `PYTHONOPTIMIZE=1`). This means every state validation and type check using `assert` can be silently bypassed.

**Vulnerable Code Locations:**

```python
# Line ~57: delete()
assert self.is_editable()

# Line ~76: open()
assert self.is_editable()

# Line ~127: add_salts()
assert self.is_editable()

# Line ~183: add_issue()
assert self.is_editable()
assert vtype in vtypes.TYPES

# Line ~201: edit_issue()
assert self.is_editable()
assert vtype in vtypes.TYPES

# Line ~214: delete_issue()
assert self.is_editable()

# Line ~236: add_voter()
assert self.is_editable()

# Line ~356: _compute_state()
assert md.closed in (None, 0)
assert md.salt is not None and md.opened_key is not None
```

**Data Flow:**

```
Method call → assert check → [REMOVED IN -O MODE] → security-sensitive operation
```

**Proof of Concept:**

Running the application with `python -O server.py` or setting `PYTHONOPTIMIZE=1` in the environment completely removes all `assert`-based validation. Then:

```http
POST /do-add-issue/OPEN_ELECTION_EID HTTP/1.1
Content-Type: application/x-www-form-urlencoded

title=Injected+Issue&description=Added+after+election+opened
```

This would succeed because `assert self.is_editable()` is no longer checked.

**Impact:**

Complete bypass of the election state machine. An attacker with committer access could:
- Add/edit/delete issues in open elections
- Corrupt vote tallying
- Delete elections that have already collected votes
- Use arbitrary strings as vote types, not just the allowed types
- The entire election lifecycle state machine becomes unenforceable

**Remediation:**

```python
def add_issue(self, title, description, vtype, kv):
    "Add a new issue with a generated unique IID."
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
    if not isinstance(vtype, str) or vtype not in vtypes.TYPES:
        raise ValueError(f"Invalid vote type: {vtype!r}. Must be one of {vtypes.TYPES}")

    # ... rest of method
```

Apply the same pattern to all methods using `assert` for security checks: replace with explicit `if not ...: raise`.

#### ASVS-1535-MEDIUM-003: Missing Type Validation on JSON Request Body

**Severity:** MEDIUM  
**Gap Type:** Type A — No type control at entry point

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `_set_election_date`
- **Lines:** 93-113

**Related Functions Checked:**
- `_set_election_date()` (pages.py:93-113)
- `quart.request.get_json()` (framework function)
- `datetime.datetime.fromisoformat()` (standard library)

**Description:**

The `_set_election_date` function accepts JSON request bodies without validating that the parsed data is the expected type (dict) or that nested fields have the expected types (string for date). The code makes type assumptions that can lead to unhandled exceptions.

**Vulnerable Code:**

```python
async def _set_election_date(election, field):
    result = await basic_info()
    data = await quart.request.get_json()       # Can return None if body isn't valid JSON
    date_str = data.get('date')                  # AttributeError if data is None
                                                 # date_str could be int, bool, list, dict, null
    if not date_str:
        quart.abort(400, 'Missing date')

    try:
        dt = datetime.datetime.fromisoformat(date_str).date()  # TypeError if not string
    except ValueError:
        quart.abort(400, 'Invalid date format')
    # TypeError from fromisoformat(non_string) is NOT caught
```

**Data Flow:**

```
POST JSON body → get_json() → data (any type or None) → data.get('date') 
→ date_str (any JSON type) → fromisoformat() 
                                          ↑
                              Only catches ValueError, not TypeError
```

**Proof of Concept:**

**Test 1: Integer date value**
```http
POST /do-set-open_at/abc123 HTTP/1.1
Content-Type: application/json

{"date": 12345}
```

This sends an integer for the `date` field. `fromisoformat(12345)` raises `TypeError`, which is NOT caught by the `except ValueError` block, resulting in a 500 error with potential stack trace exposure.

**Test 2: Invalid JSON body**
```http
POST /do-set-open_at/abc123 HTTP/1.1
Content-Type: text/plain

not json at all
```

This causes `get_json()` to return `None`, then `None.get('date')` raises `AttributeError` → 500 error.

**Impact:**

Unhandled exceptions cause 500 errors that may leak implementation details. More importantly, the code makes a type assumption (that the JSON body is a dict and `date` is a string) without verification, violating ASVS 15.3.5.

**Remediation:**

```python
async def _set_election_date(election, field):
    result = await basic_info()
    data = await quart.request.get_json()
    
    if not isinstance(data, dict):
        quart.abort(400, 'Invalid request body')
    
    date_str = data.get('date')
    if not isinstance(date_str, str) or not date_str:
        quart.abort(400, 'Missing or invalid date field')

    try:
        dt = datetime.datetime.fromisoformat(date_str).date()
    except (ValueError, TypeError):
        quart.abort(400, 'Invalid date format')
    
    # ... rest of function
```

#### ASVS-1535-MEDIUM-004: Deserialized KV Data Used Without Type Validation

**Severity:** MEDIUM  
**Gap Type:** Type A — No type control between deserialization and consumption

**Location:**
- **Primary File:** `v3/steve/election.py`
- **Function:** `json2kv`
- **Line:** ~365
- **Consumers:** `v3/server/pages.py` (multiple locations)

**Related Functions Checked:**
- `election.json2kv()` (election.py:~365)
- `pages.vote_on_page()` (pages.py:~278-281)
- `election.tally_issue()` (election.py:~299)
- `vtypes.vtype_module().tally()` (called from tally_issue)

**Description:**

The `json2kv` method deserializes JSON strings from the database without validating that the result is the expected type (dict). Consumers of this data throughout the application assume it's a dict and call `.get()` methods on it, which will fail if the deserialized value is a different JSON type (array, string, number, etc.).

**Vulnerable Code (Definition):**

```python
# election.py
@staticmethod
def json2kv(j):
    "Convert the KV JSON string back into its structured value."
    return j and json.loads(j)  # Returns ANY JSON type: dict, list, str, int, bool, None
```

**Vulnerable Code (Consumption in pages.py, vote_on_page):**

```python
# pages.py, ~line 278-281
issue.seats = issue.kv.get('seats', 0)          # Assumes kv is a dict
issue.labelmap = edict(issue.kv.get('labelmap', {}))  # Assumes kv is a dict
issue.candidates = [{'label': k, 'name': v} for k, v in issue.labelmap.items()]
```

**Vulnerable Code (Consumption in election.py, tally_issue):**

```python
# election.py, ~line 299
m = vtypes.vtype_module(issue.type)
return m.tally(votes, self.json2kv(issue.kv)), voters  # Passes unvalidated KV to tally
```

**Data Flow:**

```
Database KV column (TEXT) → json.loads() → ANY Python type 
→ .get('seats', 0)  [AttributeError if not dict]
→ .get('labelmap', {})  [AttributeError if not dict]  
→ edict(labelmap)  [TypeError if not mapping]
→ tally module  [undefined behavior]
```

**Proof of Concept:**

If an election is created and an issue's KV is set to a JSON array instead of an object (via a race condition, direct DB manipulation, or future code path that doesn't validate):

```sql
UPDATE issue SET kv = '["not", "a", "dict"]' WHERE iid = 'target_iid';
```

Subsequent access to this issue would crash on `issue.kv.get('seats', 0)` since lists don't have `.get()`.

More subtly, if `seats` is stored as the string `"3"` instead of integer `3`, STV tallying logic would receive a string where it expects an integer, potentially causing incorrect seat allocation.

**Impact:**

Runtime errors during election display or tallying. If KV data contains unexpected types for nested values (e.g., `seats` as a string), tallying could silently produce incorrect results without raising an error, as many Python operations implicitly handle strings and integers differently.

**Remediation:**

```python
@staticmethod
def json2kv(j):
    "Convert the KV JSON string back into its structured value."
    if not j:
        return None
    parsed = json.loads(j)
    if parsed is not None and not isinstance(parsed, dict):
        raise ValueError(f"KV data must be a JSON object, got {type(parsed).__name__}")
    return parsed

# Additionally, validate specific fields:
def _validate_kv(kv, vtype):
    """Validate KV fields match expected types for the vote type."""
    if kv is None:
        return
    if vtype == 'stv':
        seats = kv.get('seats')
        if seats is not None and not isinstance(seats, int):
            raise ValueError(f"'seats' must be an integer, got {type(seats).__name__}")
        labelmap = kv.get('labelmap')
        if labelmap is not None and not isinstance(labelmap, dict):
            raise ValueError(f"'labelmap' must be a dict, got {type(labelmap).__name__}")
```

## Positive Security Patterns

The audit identified several commendable security practices:

1. **Parameterized queries**: The `asfpy.db` module uses YAML-defined parameterized queries, preventing SQL injection via type confusion of parameters.

2. **`fromisoformat()` for date parsing**: The date validation in `_set_election_date` correctly uses a strict parsing function rather than loose string matching (though it needs broader exception handling).

3. **State machine pattern**: The election lifecycle (`EDITABLE → OPEN → CLOSED`) is well-structured with explicit state computation in `_compute_state`, preventing invalid transitions (when assertions are active).

4. **Explicit field selection in `get_metadata()`**: Returns only specified fields, excluding sensitive `salt` and `opened_key` — this is correct data typing and field control.

5. **Type annotations on `add_voter`**: Uses `pid: str, iid: str | None = None` type hints, though these are not enforced at runtime.

6. **EasyDict usage**: Provides attribute access with clear field names, making code readable and reducing dict key typos.

---

## Architecture Observations

1. **Python's Type System**: Python uses strong dynamic typing (no implicit type coercion between strings/integers for `==`), which inherently mitigates many type juggling attacks common in PHP/JavaScript. However, ASVS 15.3.5 requires *explicit* type verification, not reliance on language behavior.

2. **JSON Deserialization Boundary**: The primary type confusion risk in this application is at the JSON deserialization boundary (`get_json()`, `json.loads()`) where any JSON type can appear. The application consistently assumes dict types without verification.

3. **SQLite Type Affinity**: SQLite uses type affinity rather than strict typing. While values stored as integers typically come back as integers in Python, the database schema doesn't enforce types. The application relies on this implicit behavior.

4. **No Runtime Type Enforcement**: Python type hints (`pid: str`) are used in a few places but are not enforced at runtime. No runtime type checking library (e.g., `pydantic`, `typeguard`, `attrs`) is used.

## Recommendations Summary

### Immediate Actions (Priority 1)

1. **Replace ALL `assert` statements used for security validation** with explicit `if/raise` patterns. This is a single-pass refactor that immediately hardens the state machine against optimized Python runtimes.
   - **Affected:** All methods in `election.py` using `assert` for state/type checks
   - **Effort:** 2-4 hours
   - **Risk if not fixed:** Complete bypass of election state machine in production

2. **Implement votestring validation** in `add_vote()` that checks the value against the issue's vote type before encryption. This is the most impactful type validation gap.
   - **Affected:** `election.py:add_vote()`, `pages.py:do_vote_endpoint()`
   - **Effort:** 4-6 hours (includes implementing validation in vtypes modules)
   - **Risk if not fixed:** Election integrity compromise, invalid vote tallying

### Short-term Actions (Priority 2)

3. **Add `isinstance` checks on JSON-parsed request bodies** before accessing properties. Wrap `get_json()` results with type verification.
   - **Affected:** `pages.py:_set_election_date()` and similar JSON endpoints
   - **Effort:** 2-3 hours
   - **Risk if not fixed:** 500 errors, potential information disclosure

4. **Add type validation to `json2kv()`** to ensure the deserialized value is a dict, and add field-level type checks for known KV fields (`seats` → int, `labelmap` → dict).
   - **Affected:** `election.py:json2kv()` and all consumers
   - **Effort:** 3-4 hours
   - **Risk if not fixed:** Runtime errors during tallying, incorrect election results

### Medium-term Actions (Priority 3)

5. **Adopt a validation library** (e.g., Pydantic, marshmallow, or attrs) for form data and JSON request parsing to enforce schemas with type constraints at the API boundary.
   - **Effort:** 1-2 weeks (architectural change)
   - **Benefit:** Comprehensive type safety across all API endpoints

6. **Add explicit type casting** at database boundaries where values have expected types (e.g., `int(md.closed)` or use a typed ORM layer).
   - **Effort:** 1 week
   - **Benefit:** Eliminates SQLite type affinity assumptions
