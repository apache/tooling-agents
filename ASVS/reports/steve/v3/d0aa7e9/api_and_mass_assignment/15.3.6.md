## Findings

### Medium Severity

#### ASVS-1536-MED-001: Unsanitized JSON Object Keys in Data Pipeline to Client-Side Templates

**Severity:** MEDIUM

**Location:**
- **File:** `v3/steve/election.py`
- **Lines:** 444-452 (data origin)
- **File:** `v3/server/pages.py`
- **Lines:** 258-265 (data flow to template)

**Related Functions Checked:**
- `json2kv()` (election.py:448)
- `list_issues()` (election.py:256)
- `vote_on_page()` (pages.py:258)

**Description:**

The `labelmap` field in STV issue KV data is an arbitrary key-value dictionary where keys represent candidate labels. These keys are:
1. Deserialized from JSON without any key filtering (`json.loads()`)
2. Converted to an `EasyDict` without filtering
3. Iterated to produce `candidates` list where each key becomes a `label` value
4. Passed to the template for client-side rendering

If these labels are used to construct JavaScript objects on the client side (e.g., `{[label]: value}` or `Object.assign({}, labelData)`), keys like `__proto__`, `constructor`, or `prototype` could pollute JavaScript prototypes.

**Vulnerable Code (Data Deserialization):**
```python
# election.py:448-449
@staticmethod
def json2kv(j):
    "Convert the KV JSON string back into its structured value."
    return j and json.loads(j)
```

**Vulnerable Code (Data Flow to Template):**
```python
# pages.py:258-265
for issue in issues_stv:
    issue.seats = issue.kv.get('seats', 0)
    issue.labelmap = edict(issue.kv.get('labelmap', {}))
    issue.candidates = [{'label': k, 'name': v} for k, v in issue.labelmap.items()]
    random.shuffle(issue.candidates)
```

**Data Flow:**
```
Database KV column (JSON text)
  → json2kv() [election.py:448] — raw json.loads(), no key filtering
  → list_issues() [election.py:256] — returns edict with unfiltered KV
  → vote_on_page() [pages.py:258] — extracts labelmap as dictionary
  → issue.candidates list with arbitrary 'label' keys
  → EZT template rendering → client-side JavaScript
```

**Current Attack Surface Assessment:**
- The web interface currently hardcodes `kv = None` in `do_add_issue_endpoint` (pages.py:468-469)
- `do_edit_issue_endpoint` passes existing KV through unchanged (pages.py:489)
- KV values would need to be set via CLI tools, direct database manipulation, or future web endpoints
- This limits immediate exploitability but represents a defense-in-depth gap

**Proof of Concept (Database-level):**
```sql
-- Injecting prototype-polluting key into issue KV
UPDATE issue SET kv = '{"seats": 1, "labelmap": {"__proto__": {"polluted": true}, "A": "Candidate A"}}' WHERE iid = 'target-issue-id';
```

If the client-side JavaScript processes this as:
```javascript
// Hypothetical client-side code (not available for review)
let candidateMap = {};
candidates.forEach(c => { candidateMap[c.label] = c.name; });
// If c.label === "__proto__", this pollutes Object.prototype
```

**Impact:** 

If client-side JavaScript constructs objects using these labels as keys, an attacker with write access to KV data could pollute JavaScript prototypes, potentially leading to XSS, authentication bypass, or denial of service on the client side.

**Remediation:**

Server-side key filtering in the data pipeline:

```python
# election.py — Add a safe key filter
DANGEROUS_KEYS = frozenset({'__proto__', 'constructor', 'prototype'})

@staticmethod
def json2kv(j):
    "Convert the KV JSON string back into its structured value."
    if not j:
        return None
    parsed = json.loads(j)
    return Election._sanitize_keys(parsed)

@staticmethod
def _sanitize_keys(obj):
    "Recursively remove prototype-polluting keys from parsed JSON."
    if isinstance(obj, dict):
        return {
            k: Election._sanitize_keys(v)
            for k, v in obj.items()
            if k not in Election.DANGEROUS_KEYS
        }
    if isinstance(obj, list):
        return [Election._sanitize_keys(item) for item in obj]
    return obj
```

Additionally, validate `labelmap` keys when they are set:
```python
def add_issue(self, title, description, vtype, kv):
    assert self.is_editable()
    assert vtype in vtypes.TYPES
    
    # Validate KV keys if present
    if kv and 'labelmap' in kv:
        for key in kv['labelmap']:
            if key in self.DANGEROUS_KEYS:
                raise ValueError(f"Invalid label key: {key}")
    # ... rest of method
```

## Positive Security Patterns

1. **Parameterized Database Queries**: All database operations use parameterized queries via `asfpy.db`, preventing SQL injection that could be used to inject malicious KV data.

2. **State Machine Enforcement**: Election state checks (`is_editable()`, `is_open()`, `is_closed()`) restrict when data modifications can occur, limiting windows for KV data manipulation.

3. **Selective Metadata Return**: `get_metadata()` (election.py:182-195) explicitly selects which fields to return rather than dumping the full database row, preventing exposure of sensitive fields like `salt` and `opened_key`.

4. **EasyDict Usage**: While `EasyDict` doesn't filter prototype-polluting keys, Python itself is not vulnerable to prototype pollution — this is only a concern when data flows to the JavaScript client.

---

## Architecture Observations

1. **Data Serialization Boundary**: The KV field serves as a flexible schema-less data store (JSON in SQLite). This design pattern is common but creates a trust boundary issue — data stored by privileged operations (CLI/admin) is rendered to potentially different users in a different execution context (browser JavaScript).

2. **Template Engine (EZT)**: The EZT template engine is used for server-side rendering. Without seeing the templates, it's unknown whether data is rendered as inline JavaScript objects, JSON in `<script>` tags, or as HTML `data-*` attributes — each pattern has different prototype pollution implications.

3. **No Content-Type Enforcement for JSON APIs**: The `_set_election_date` function calls `await quart.request.get_json()` but doesn't verify the `Content-Type` header, which is a separate concern but relates to how data enters the system.

## Recommendations Summary

### Immediate Actions (Priority 1)

**Provide JavaScript Source Code for Full ASVS 15.3.6 Audit**
- **Impact:** Audit Gap Closure
- **Action:** The JavaScript files in `v3/server/static/` and templates in `v3/server/templates/` must be reviewed to:
  - Verify that `Map()` or `Set()` are used instead of object literals for user-influenced data
  - Check for `Object.assign()`, spread operators, or recursive merge functions on untrusted data
  - Verify `Object.create(null)` or `Object.freeze(Object.prototype)` patterns where appropriate
  - Confirm that `JSON.parse()` results are not merged into existing objects without key filtering

### Short-term Actions (Priority 2)

**Implement Server-Side Key Sanitization**
- **Impact:** Defense in Depth
- **Action:** Add prototype-polluting key filtering in `json2kv()` as shown in ASVS-1536-MED-001 remediation. This provides server-side protection regardless of client-side implementation.

**Add KV Schema Validation for STV Issues**
- **Impact:** Input Validation
- **Action:** Define and enforce an explicit schema for STV issue KV data:
```python
STV_KV_SCHEMA = {
    'seats': int,
    'labelmap': dict,  # keys must be alphanumeric labels
}

def validate_stv_kv(kv):
    if not isinstance(kv.get('labelmap', {}), dict):
        raise ValueError("labelmap must be a dictionary")
    for key in kv.get('labelmap', {}):
        if not re.match(r'^[A-Za-z0-9_-]+$', key):
            raise ValueError(f"Invalid labelmap key: {key}")
```

### Long-term Actions (Priority 3)

**Implement Content Security Policy**
- **Impact:** Mitigation Layer
- **Action:** Even if prototype pollution occurs, a strict CSP can limit its impact by preventing inline script execution that might result from successful prototype pollution attacks.
