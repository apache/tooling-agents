## Findings

### Medium Severity

#### ASVS-1534-MED-001: Complete Absence of Client IP Address in Security Audit Logs

**Severity:** MEDIUM  
**Gap Type:** Type A – Entry points with NO control  
**Location:** `v3/server/pages.py` (lines 115, 405, 412, 434, 446, 459, 475, 492, 510, 524)

**Description:**

The application logs security-critical actions (voting, election creation, opening, closing, issue management) but never includes the client's IP address. All 10+ logging statements reference only `result.uid` (authenticated user ID). This means forensic investigation of suspicious activity (e.g., compromised accounts, unauthorized access from unusual locations) cannot correlate actions with source IPs.

**Vulnerable Code:**

`v3/server/pages.py`, line ~405 (vote submission):
```python
_LOGGER.info(
    f'User[U:{result.uid}] voted on issue[I:{iid}] in election[E:{election.eid}]'
)
```

`v3/server/pages.py`, line ~434 (election creation):
```python
_LOGGER.info(
    f'User[U:{result.uid}] created election[E:{election.eid}];'
    f' title: "{form.title}"'
)
```

`v3/server/pages.py`, line ~446 (election opening):
```python
_LOGGER.info(f'User[U:{result.uid}] opened election[E:{election.eid}]')
```

`v3/server/pages.py`, line ~459 (election closing):
```python
_LOGGER.info(f'User[U:{result.uid}] closed election[E:{election.eid}]')
```

**All Affected Logging Sites:**

**Related Functions Checked:**
- `_set_election_date` (line ~115)
- `do_vote_endpoint` (lines ~405, ~412)
- `do_create_endpoint` (line ~434)

**Data Flow:**
```
Client IP → [reverse proxy] → quart.request → [NEVER READ] → _LOGGER.info() → logs (IP absent)
```

**Impact:**

Incomplete audit trail for a voting system. Without IP addresses in logs, administrators cannot:
- Detect account compromise via geolocation anomalies
- Perform forensic analysis of disputed elections
- Identify coordinated manipulation from common network origins
- Satisfy audit requirements for election integrity

**Remediation:**

Create a helper to extract the client IP from the trusted proxy header and include it in all security log entries:

```python
def get_client_ip():
    """Extract client IP from trusted proxy header or fall back to remote_addr."""
    # Only trust X-Forwarded-For from configured trusted proxies
    if quart.request.access_route:
        # access_route[0] is the leftmost (client) IP; only trust if
        # the immediate upstream (remote_addr) is a known proxy
        trusted_proxies = APP.cfg.get('trusted_proxies', set())
        if quart.request.remote_addr in trusted_proxies:
            return quart.request.access_route[0]
    return quart.request.remote_addr

# Usage in handlers:
client_ip = get_client_ip()
_LOGGER.info(
    f'User[U:{result.uid}] IP[{client_ip}] voted on'
    f' issue[I:{iid}] in election[E:{election.eid}]'
)
```

#### ASVS-1534-MED-002: No Trusted Proxy Configuration for IP Forwarding

**Severity:** MEDIUM  
**Gap Type:** Type A – Entry point with NO control  
**Location:** `v3/server/api.py` (entire file), `v3/server/pages.py` (module-level configuration)

**Description:**

The application has no configuration for trusted proxy headers. When deployed behind a reverse proxy (standard for production web applications), `request.remote_addr` returns the proxy's IP rather than the client's. The Quart framework supports `ProxyFix`-style middleware or explicit `X-Forwarded-For` parsing, but neither is configured. This means even if the application were to start reading IP addresses, it would obtain the wrong value.

**Vulnerable Code:**

`v3/server/api.py` (complete file):
```python
import asfquart

APP = asfquart.APP
```

No proxy middleware is registered. No trusted proxy IPs are defined in visible configuration.

`v3/server/pages.py` (module-level, no proxy configuration):
```python
APP = asfquart.APP
# ... no ProxyFix or forwarded-header configuration ...
```

**Related Functions Checked:**
- Application initialization in `api.py`
- Module-level configuration in `pages.py`

**Data Flow:**
```
Client (IP: 203.0.113.50) → Reverse Proxy (IP: 10.0.0.1) 
    → [X-Forwarded-For: 203.0.113.50] → Quart App
    → request.remote_addr = 10.0.0.1 (proxy IP, not client)
    → X-Forwarded-For header: ATTACKER-SPOOFABLE without trusted proxy config
```

**Proof of Concept:**

Without trusted proxy configuration, an attacker can spoof their IP by injecting headers directly:

```http
POST /do-vote/abc123 HTTP/1.1
Host: voting.example.org
X-Forwarded-For: 127.0.0.1
X-Real-IP: 127.0.0.1
```

If the application naively reads `X-Forwarded-For` without validating the sender is a trusted proxy, any client can claim any IP address.

**Impact:**

- Any future IP-based security controls (rate limiting, geo-blocking) would operate on incorrect data
- Audit logs would record proxy IPs instead of real client IPs
- Spoofable headers could be used to bypass IP-based restrictions

**Remediation:**

Configure trusted proxy middleware at application startup:

```python
# In api.py or app initialization
from quart import Quart

APP = asfquart.APP

# Configure trusted proxies (use actual proxy IPs)
APP.config['TRUSTED_PROXIES'] = {'10.0.0.1', '10.0.0.2'}

# Or use Quart's built-in proxy handling
@APP.before_serving
async def configure_proxy():
    # Quart supports proxy_fix via its config
    APP.config['FORWARDED_ALLOW_IPS'] = '10.0.0.1,10.0.0.2'
```

For Quart specifically, use the `ProxyFixMiddleware` or configure `forwarded_allow_ips`:

```python
from quart_proxy_fix import ProxyFix  # or equivalent

app = ProxyFix(APP, x_for=1, x_proto=1, x_host=1)
```

## Positive Security Patterns

1. **Authentication on all sensitive routes**: Every state-changing endpoint uses `@asfquart.auth.require({R.committer})` or `@asfquart.auth.require({R.pmc_member})`, ensuring authenticated access.

2. **User ID in all audit logs**: While IP is missing, the authenticated `uid` is consistently included in all security log messages, providing user-level accountability.

3. **Structured logging pattern**: The logging format `User[U:{uid}]` / `election[E:{eid}]` / `issue[I:{iid}]` is consistent across all handlers, making log parsing reliable.

4. **State machine enforcement**: Election state transitions are protected by assertions (`assert self.is_editable()`, `assert self.is_open()`), preventing operations in invalid states.

5. **Transactional database operations**: Critical multi-step operations (delete, add_salts) use explicit `BEGIN TRANSACTION` / `COMMIT`.

## Architecture Observations

1. **Framework dependency**: The application relies on `asfquart` (an ASF-specific Quart wrapper) for session management and authentication. Proxy configuration may need to be applied at the `asfquart` framework level rather than in this application code.

2. **Deployment model unknown**: Without seeing deployment configuration (e.g., Nginx/Apache reverse proxy configs, Docker compose, etc.), the actual proxy chain cannot be verified. However, the application code must be prepared regardless.

3. **ASVS acknowledgment in requirement**: The requirement explicitly notes "even the original IP address may not be reliable due to dynamic IPs, VPNs, or corporate firewalls." For a closed-community ASF voting system where users authenticate via OAuth, IP is supplementary to identity — but still valuable for forensics.
