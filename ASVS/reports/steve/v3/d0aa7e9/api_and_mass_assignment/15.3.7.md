**Overview:**

The application uses Quart (async Flask-like framework), which inherently distinguishes between parameter sources (`request.form`, `request.args`, `request.get_json()`, `request.cookies`). The codebase consistently uses source-specific accessors, which is a strong positive pattern against cross-source HPP. However, there is a systemic pattern across all form-processing endpoints where `MultiDict` objects are silently coerced into single-value dictionaries via `EasyDict`, providing no defense against intra-source parameter pollution and no detection/logging of duplicate parameters.

## Findings

### Medium Severity

#### Finding: ASVS-1537-MED-001

**Title:** No Explicit Defense Against Intra-Source HTTP Parameter Pollution in Form-Processing Endpoints

**File/Location:** `v3/server/pages.py` — Multiple endpoints

**Vulnerable Code Locations:**

1. `do_vote_endpoint` (line 405):
```python
form = edict(await quart.request.form)
```

2. `do_create_endpoint` (line 448):
```python
form = edict(await quart.request.form)
```

3. `do_add_issue_endpoint` (line 508):
```python
form = edict(await quart.request.form)
```

4. `do_edit_issue_endpoint` (line 532):
```python
form = edict(await quart.request.form)
```

**Data Flow:**

```
HTTP POST body (user-controllable, may contain duplicate parameter names)
  → quart.request.form (MultiDict preserving all values)
    → edict() constructor (silently discards all but first value per key)
      → form.title / form.description / form.items() (single value used, no duplicate detection)
```

**Description:**

Quart's `request.form` returns a Werkzeug `MultiDict` that preserves multiple values for the same parameter name. When this `MultiDict` is passed to `EasyDict()` (which inherits from `dict`), the constructor calls `dict.__init__()`, which invokes `MultiDict.__getitem__()` for each unique key — returning only the **first** submitted value and silently discarding all subsequent duplicates.

This means:
- Duplicate parameters are silently dropped with no validation, logging, or error
- The application has no mechanism to detect or reject HTTP parameter pollution attempts
- The behavior (first-value-wins) is an implicit framework artifact, not an explicit security decision

**Proof of Concept:**

For the vote submission endpoint:
```http
POST /do-vote/abc123def HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: session=...

vote-issue1=yes&vote-issue1=no&vote-issue1=abstain
```

After `edict(await quart.request.form)`, the form contains `{'vote-issue1': 'yes'}` — the `no` and `abstain` values are silently discarded. No log entry, no error, no indication that the request was anomalous.

For the election creation endpoint:
```http
POST /do-create-election HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: session=...

title=Legitimate+Election&title=Injected+Title
```

`form.title` resolves to `"Legitimate Election"` while `"Injected Title"` is silently discarded.

**Impact:**

- Fails to meet ASVS 15.3.7 requirement for explicit HPP defenses
- Silent parameter deduplication could mask attack attempts (no logging of anomalous duplicate parameters)
- The implicit first-value-wins behavior is fragile — if any future code path accesses `request.form.getlist()` or iterates the raw MultiDict differently, inconsistent parameter resolution would occur
- In the voting context (`do_vote_endpoint`), while the current impact is limited to a voter's own ballot, the lack of detection means HPP probes would go unnoticed

**Remediation:**

Create a helper function that validates form inputs are single-valued and rejects requests with duplicate parameters:

```python
from quart import request, abort
from easydict import EasyDict as edict

async def get_single_value_form():
    """Parse request.form, rejecting duplicate parameters (HPP defense)."""
    form = await request.form
    
    # Detect duplicate parameters
    seen = set()
    for key in form:
        values = form.getlist(key)
        if len(values) > 1:
            _LOGGER.warning(
                f'HPP attempt detected: parameter "{key}" submitted '
                f'{len(values)} times from {request.remote_addr}'
            )
            abort(400, f'Duplicate parameter: {key}')
        seen.add(key)
    
    return edict(form)
```

Then replace all instances of `edict(await quart.request.form)` with:

```python
# Before (vulnerable):
form = edict(await quart.request.form)

# After (defended):
form = await get_single_value_form()
```

Additionally, for the vote endpoint, validate that only expected parameter names are present:

```python
async def do_vote_endpoint(election):
    form = await get_single_value_form()
    
    # Whitelist: only accept keys matching 'vote-<hex_id>' pattern
    import re
    for key in form:
        if not re.match(r'^vote-[a-f0-9]+$', key):
            abort(400, f'Unexpected parameter: {key}')
    ...
```

## Positive Security Patterns

1. **Source-Specific Parameter Access:** All handlers use either `request.form` or `request.get_json()` — never `request.values` (which merges query string and form body). This provides strong cross-source HPP protection by design.

2. **URL Path Parameters for Resource IDs:** Election IDs (`eid`) and Issue IDs (`iid`) are extracted from URL path segments via Quart's routing, completely isolated from query string and form body parameters.

3. **State Machine Enforcement:** The `Election` class enforces state preconditions (`is_editable()`, `is_open()`, `is_closed()`) via assertions before any state-changing operation, preventing parameter manipulation from bypassing workflow controls.

4. **Metadata Field Filtering:** `get_metadata()` in `election.py` (lines 165-180) explicitly constructs a return dict with whitelisted fields, excluding sensitive `salt` and `opened_key` values — protecting against mass assignment on read.

5. **Parameterized Queries:** All database operations use `asfpy.db` with prepared statements via YAML-defined queries, preventing SQL injection regardless of parameter pollution.

---

## Architecture Observations

1. **Framework HPP Posture:** Quart (Werkzeug-based) natively separates `request.args`, `request.form`, `request.cookies`, and `request.headers` into distinct `MultiDict` objects. The application benefits from this architecture but does not add explicit duplicate-detection logic on top.

2. **EasyDict Anti-Pattern:** Using `edict(await quart.request.form)` as a convenience pattern systematically destroys the `MultiDict`'s ability to represent duplicate parameters. This creates a "normalization gap" between what the HTTP request contains and what the application logic processes.

3. **Incomplete Authorization Placeholders:** Multiple endpoints contain `### check authz` comments (lines 86, 245, 294, 402, 445, 465, 484, 504, 527, 552) indicating authorization logic is not yet implemented. While not directly HPP-related, this means any user with the `committer` role can manage any election — parameter pollution attacks could have broader impact if combined with missing authorization.
