### Key Observations

ASVS 11.6.2 requires that approved cryptographic algorithms are used for key exchange with secure parameters. This codebase does **not** employ public key cryptography or key exchange protocols (Diffie-Hellman, ECDH); all key material is derived server-side using symmetric key derivation functions (Argon2, HKDF). While the absence of key exchange is an intentional design decision, the audit identified issues with the cryptographic parameters used in the key derivation chain that constitutes the key establishment process for vote encryption. Three findings of MEDIUM severity were identified affecting the security parameters of the cryptographic key establishment process.

## Findings

### Medium Severity

#### Finding ASVS-1162-MED-001: Argon2d Variant Used Instead of Recommended Argon2id for Key Derivation

**Severity:** MEDIUM  
**Gap Type:** Type A — Incorrect algorithm parameter selection  
**File:** `v3/steve/crypto.py`  
**Function:** `_hash()` (line 80)  
**ASVS Requirement:** 11.6.2 — Secure parameters for cryptographic mechanisms

**Description:**

The key derivation function uses Argon2d instead of the recommended Argon2id variant. Argon2d is specifically vulnerable to side-channel attacks (cache-timing attacks). RFC 9106 §4 states: *"If side-channel attacks are a viable threat, Argon2id SHOULD be used."* On shared server infrastructure (cloud VMs, containers), cache-timing side-channels are a realistic attack vector.

**Vulnerable Code:**

```python
# v3/steve/crypto.py, line 80-89
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2 to hash the data, with default tuning parameters."

    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,  # <-- Argon2d
    )
```

**Evidence of Awareness — Benchmark uses Argon2id:**

```python
# v3/steve/crypto.py, line 117 (benchmark_argon2)
                argon2.low_level.hash_secret_raw(
                    ...
                    type=argon2.low_level.Type.ID,  # <-- Argon2id in benchmark
                )
```

**Data Flow:**

1. `gen_opened_key()` → `_hash(digest, salt)` with Type.D → `opened_key` stored in DB
2. `gen_vote_token()` → `_hash(opened_key + pid + iid, salt)` with Type.D → `vote_token` used for encryption key derivation

**Related Functions Checked:**
- `gen_opened_key()` — election tamper detection key (called from `election.py:open()`, line 82)
- `gen_vote_token()` — per-voter encryption token (called from `election.py:add_vote()`, line 193 and `tally_issue()`, line 223)

**Impact:**

An attacker co-located on the same physical hardware could potentially extract key material from the Argon2d computation through cache-timing analysis. This affects ALL key derivation in the system, compromising both the election tamper detection mechanism and per-voter encryption tokens.

**Remediation:**

```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2id to hash the data, with tuning parameters per RFC 9106."

    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.ID,  # Argon2id per RFC 9106 recommendation
    )
```

> **Note:** Changing the Argon2 variant will change all derived keys. This requires a migration strategy: existing elections with stored `opened_key` values derived using Argon2d would fail tamper detection. Consider versioning the key derivation or migrating all open/closed elections before switching.

#### Finding ASVS-1162-MED-002: HKDF Domain Separation Label Mismatches Actual Algorithm (Migration Risk)

**Severity:** MEDIUM  
**Gap Type:** Type A — Incorrect cryptographic parameter  
**File:** `v3/steve/crypto.py`  
**Function:** `_b64_vote_key()` (line 53)  
**ASVS Requirement:** 11.6.2 — Secure parameters prevent cryptographic breaks

**Description:**

The HKDF `info` parameter serves as a domain separation label per NIST SP 800-56C Rev. 2 §4. The current label `b'xchacha20_key'` does not match the actual algorithm (Fernet/AES-128-CBC+HMAC-SHA256), creating both algorithm migration vulnerability and audit confusion.

**Vulnerable Code:**

```python
# v3/steve/crypto.py, lines 53-62
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    "Key-stretch the vote_token. (ref: PBKDF)"

    ### still using Fernet now, but will switch soon. Leaving comments.
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,  # 32-byte key for XChaCha20-Poly1305   <-- WRONG: used for Fernet
        salt=salt,
        info=b'xchacha20_key',  # <-- WRONG: key is used for Fernet (AES-128-CBC)
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

**Data Flow:**

`add_vote()` → `crypto.create_vote(vote_token, salt, votestring)` → `_b64_vote_key(vote_token, salt)` with info=`b'xchacha20_key'` → key used for **Fernet**, not XChaCha20

**Related Functions Checked:**
- `create_vote()` — uses derived key for Fernet encryption (line 68)
- `decrypt_votestring()` — uses derived key for Fernet decryption (line 72)

**Impact:**

This creates two risks:
1. **Algorithm migration vulnerability**: The comment indicates a planned migration to XChaCha20-Poly1305. If the migration occurs without changing the info label, the **same key material** would be derived for both the old (Fernet) and new (XChaCha20) algorithm contexts. This violates the principle that keys derived for one purpose must not be reused for another (NIST SP 800-56C §4.1).
2. **Audit confusion**: The mislabeled parameter creates false confidence that domain separation is correctly implemented, when it is not.

**Remediation:**

```python
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    "Key-stretch the vote_token for vote encryption. (ref: PBKDF)"

    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,  # 32 bytes: 16 for HMAC signing key + 16 for AES-128 encryption key
        salt=salt,
        info=b'fernet_vote_key_v1',  # Correct domain separation for current algorithm
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

When migrating to XChaCha20-Poly1305, use a distinct info label (e.g., `b'xchacha20_vote_key_v2'`) to ensure complete key separation between algorithm generations.

> **Note:** Changing the info label will change all derived keys. Existing encrypted votes would become undecryptable. This change should be coordinated with the algorithm migration, or applied only to new elections.

#### Finding ASVS-1162-MED-003: Tamper Detection Control Not Applied Before Vote Submission (Key Establishment Integrity Gap)

**Severity:** MEDIUM  
**Gap Type:** Type B — Control EXISTS but NOT CALLED at all entry points  
**File:** `v3/steve/election.py`  
**Function:** `add_vote()` (line 186)  
**ASVS Requirement:** 11.6.2 — Prevent attacks on key establishment process

**Description:**

The `opened_key` is the root of trust for the entire key derivation chain. The system includes an `is_tampered()` function to verify election integrity, but this control is only applied at tally time, not during vote submission. This allows the key establishment process to proceed without verifying that the key material was established over untampered election data.

**Control Definition:**

```python
# v3/steve/election.py, lines 247-260
def is_tampered(self, pdb):
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)
    return opened_key != md.opened_key
```

**Control Applied (tally path only):**

```python
# v3/server/bin/tally.py, lines 131-134
if election.is_tampered(pdb):
    print(f'Error: Election {election_id} has been tampered with. Cannot proceed.')
    sys.exit(1)
```

**Control NOT Applied (vote submission):**

```python
# v3/steve/election.py, lines 186-199
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    # ^^^ Uses md.opened_key WITHOUT verifying it hasn't been tampered with

    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

**Coverage Analysis:**

| Entry Point | `is_tampered()` Called? | Uses `opened_key`? |
|-------------|------------------------|-------------------|
| `tally.py:main()` | ✅ Yes (line 131) | Yes (via `tally_issue()`) |
| `election.py:add_vote()` | ❌ No | Yes (line 193) |
| `election.py:has_voted_upon()` | ❌ No | Yes (line 242) |
| `election.py:tally_issue()` | ❌ No (relies on caller) | Yes (line 223) |

**Related Functions Checked:**
- `has_voted_upon()` — uses `opened_key` without tamper check (line 238-245)
- `tally_issue()` — uses `opened_key` without tamper check (line 200-242)

**Impact:**

If election data is modified after opening (e.g., unauthorized voter additions, issue modifications) while the stored `opened_key` remains unchanged, `add_vote()` will continue accepting and encrypting votes using the original key material without detecting the integrity violation. The tampering would only be discovered at tally time, potentially after an election has completed with compromised integrity.

**Remediation:**

Add a tamper check at the beginning of operations that depend on the key establishment:

```python
def add_vote(self, pid: str, iid: str, votestring: str, pdb):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # Verify election integrity before using key material
    if self.is_tampered(pdb):
        raise ElectionTampered(self.eid)

    md = self._all_metadata(self.S_OPEN)
    # ... rest of method
```

> **Note:** The `is_tampered()` check requires a `PersonDB` instance, which may not be readily available in all calling contexts. Consider caching the tamper state or passing the `pdb` reference during election initialization. Also note that calling `is_tampered()` on every vote submission involves Argon2 computation; consider caching the result with a short TTL or checking on election load.

## Positive Security Patterns

1. **CSPRNG Usage**: All random values (`gen_salt()`, `create_id()`, `shuffle()`) use Python's `secrets` module, which provides cryptographically secure randomness. No use of `random` module for security-sensitive operations.

2. **Memory-Hard Key Derivation**: Argon2 with 64MB memory cost provides strong resistance against GPU-based brute-force attacks on key material. Parameters (time_cost=2, memory_cost=65536, parallelism=4) align with Passlib defaults and are reasonable for server-side use.

3. **Authenticated Encryption**: Fernet provides both confidentiality (AES-128-CBC) and integrity (HMAC-SHA256). Votes cannot be silently modified without detection.

4. **Per-Voter Salts**: Each (person, issue) pair gets a unique 16-byte salt, preventing correlation between voters even if the database is fully compromised. Salt uniqueness is enforced by the primary key constraint on mayvote(pid, iid).

5. **Cryptographic Shuffling**: Vote results are shuffled using Fisher-Yates with `secrets.randbelow()` before processing, preventing database-order information leakage during tallying.

6. **Schema-Level Constraints**: The database enforces salt and key lengths via CHECK constraints, providing defense-in-depth against programming errors.

7. **Tamper Detection Architecture**: The `opened_key` mechanism captures a cryptographic snapshot of the election state at opening time, enabling detection of post-opening modifications.

8. **Two-Phase Key Derivation**: The separation of vote_token generation (Argon2) from vote_key derivation (HKDF) follows good cryptographic engineering practices with distinct key derivation phases.

## Architecture Observations

1. **No Public Key Cryptography**: The system relies entirely on symmetric key derivation. While this is an intentional design choice, it means ASVS 11.6.2's core requirement about key exchange algorithms is not directly applicable. The server is both the encryptor and decryptor.

2. **Single Point of Key Material**: The `opened_key` stored in the election table is the root of all vote encryption keys. Compromise of this single value, combined with the mayvote salts and voter/issue identifiers, enables derivation of all vote encryption keys.

3. **Key Derivation is Computationally Expensive by Design**: Each vote submission requires one Argon2 computation (for vote_token), which provides brute-force resistance but could be a performance concern under load.

4. **Algorithm Migration in Progress**: Comments in `_b64_vote_key()` indicate a planned migration from Fernet to XChaCha20-Poly1305. This migration will require careful key versioning to maintain backward compatibility with existing encrypted votes.
