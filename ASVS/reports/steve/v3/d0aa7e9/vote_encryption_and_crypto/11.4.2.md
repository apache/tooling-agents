## Findings

### High Severity

#### ASVS-1142-HIGH-001: Argon2d Used Instead of Recommended Argon2id

| Attribute | Detail |
|-----------|--------|
| **Severity** | HIGH |
| **Finding ID** | ASVS-1142-HIGH-001 |
| **File** | `v3/steve/crypto.py` |
| **Function** | `_hash()` |
| **Lines** | 82-92 |
| **Gap Type** | Type B — Correct variant known (used in benchmark) but wrong variant deployed in production |

**Vulnerable Code:**
```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2 to hash the data, with default tuning parameters."
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,  # ← Argon2d
    )
```

**Evidence of Type B Gap — Benchmark uses the correct variant:**
```python
def benchmark_argon2():
    # ...
    argon2.low_level.hash_secret_raw(
        # ...
        type=argon2.low_level.Type.ID,  # ← Argon2id (correct)
    )
```

**Data Flow:**
- `_hash()` is called by `gen_opened_key()` (election integrity key) and `gen_vote_token()` (voter anonymity token)
- `gen_opened_key()` ← `election.py:open()` (election opening) and `election.py:is_tampered()` (tamper verification)
- `gen_vote_token()` ← `election.py:add_vote()` (vote submission), `election.py:tally_issue()` (vote counting), `election.py:has_voted_upon()` (voting status)
- **Every cryptographic key derivation operation in the system flows through this function using the wrong Argon2 variant**

**Analysis:**

OWASP Password Storage Cheat Sheet explicitly recommends **Argon2id** as the first choice. RFC 9106 Section 4 recommends Argon2id for password hashing and key derivation where side-channel resistance matters.

**Argon2d** uses data-dependent memory access patterns, making it vulnerable to cache-timing side-channel attacks. An attacker with the ability to observe memory access patterns on the server (e.g., co-located VMs, compromised hypervisor, or SGX-like side channels) could potentially recover:
- The `opened_key` (compromising election integrity verification)
- `vote_token` values (compromising voter anonymity by linking voters to encrypted votes)

**Argon2id** provides the same brute-force resistance as Argon2d while using data-independent access patterns in the first half of execution, mitigating side-channel attacks.

**Impact:**
- Election integrity: Recovery of `opened_key` could allow crafting tamper-undetectable modifications to election data
- Voter anonymity: Recovery of `vote_token` derivation could link specific voters to their votes, destroying ballot secrecy
- The benchmark function proves the developer considered Argon2id, making this a clear oversight in production code

**Remediation:**
```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2id to hash the data with tuned parameters."
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.ID,  # Argon2id per OWASP/RFC 9106
    )
```

> **Note:** This change will alter all derived keys. Existing elections' `opened_key` values will no longer match, causing `is_tampered()` to return `True`. A migration strategy is required: either re-derive keys for elections in editable state, or maintain a version flag to use the correct algorithm for legacy elections.

### Medium Severity

#### ASVS-1142-MED-001: Non-Constant-Time Comparison of Cryptographic Hash Material

| Attribute | Detail |
|-----------|--------|
| **Severity** | MEDIUM |
| **Finding ID** | ASVS-1142-MED-001 |
| **File** | `v3/steve/election.py` |
| **Function** | `is_tampered()` |
| **Lines** | 260-263 |
| **Gap Type** | Type A — No timing-safe comparison control applied |

**Vulnerable Code:**
```python
def is_tampered(self, pdb):
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)
    
    # Standard Python != operator — NOT constant-time
    return opened_key != md.opened_key
```

**Data Flow:**
- `is_tampered()` ← `tally.py:main()` (pre-tally integrity check)
- `opened_key` is a 32-byte Argon2 hash output
- `md.opened_key` is the stored 32-byte reference value from the database
- Python's `!=` performs byte-by-byte comparison with early termination

**Analysis:**

Python's built-in `!=` operator on `bytes` objects performs a short-circuit comparison — it returns as soon as a mismatched byte is found. This timing difference can leak information about how many leading bytes of the computed hash match the stored hash.

In this system's context, the `opened_key` is derived from election data (metadata + issues + voters) and the election salt. If an attacker can:
1. Modify election data in the database (e.g., SQL injection elsewhere, or database access)
2. Repeatedly trigger `is_tampered()` checks (e.g., through the web interface or by manipulating the tally process)
3. Measure response timing with sufficient precision

They could incrementally determine the stored `opened_key` value, then craft election modifications that produce the same hash — effectively bypassing tamper detection.

**Impact:**
- Potential bypass of tamper detection mechanism through iterative timing analysis
- Could allow undetected modification of election configuration (adding/removing voters, changing issues)

**Remediation:**
```python
import hmac

def is_tampered(self, pdb):
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)
    
    # Use constant-time comparison to prevent timing attacks
    return not hmac.compare_digest(opened_key, md.opened_key)
```

## Positive Security Patterns

1. **Layered Key Derivation**: The system uses BLAKE2b → Argon2 → HKDF, providing defense-in-depth. The BLAKE2b pre-hash reduces arbitrary-length election data to a fixed 64 bytes before Argon2, which is a sound approach.

2. **Per-Voter Unique Salts**: Each `(pid, iid)` pair gets a unique 16-byte cryptographically random salt via `secrets.token_bytes()`, stored in the `mayvote` table. This prevents vote correlation even by database administrators.

3. **Argon2 Parameter Strength**: With `time_cost=2`, `memory_cost=65536` (64 MiB), and `parallelism=4`, the parameters exceed OWASP's minimum recommendation of 19 MiB / t=2 / p=1. The system also includes a `benchmark_argon2()` tool for parameter tuning.

4. **CSPRNG Throughout**: All random generation uses the `secrets` module (`token_bytes`, `token_hex`, `randbelow`), ensuring cryptographically strong randomness for IDs, salts, and vote shuffling.

5. **Schema-Level Crypto Material Validation**: The SQLite schema enforces exact lengths for cryptographic material (`salt` at 16 bytes, `opened_key` at 32 bytes, `vote_token` at 32 bytes) via CHECK constraints with STRICT mode.

6. **Vote Shuffling Before Tallying**: `crypto.shuffle()` uses Fisher-Yates with `secrets.randbelow()` to destroy database insertion order before tally processing, protecting voter privacy.

7. **Tamper Detection Architecture**: The `is_tampered()` check before tallying recomputes the election's integrity hash from current data, catching any post-opening modifications to election metadata, issues, or voter rolls.

## Architecture Observations

1. **Key Derivation Chain**: The cryptographic chain is well-structured: election salt → opened_key (Argon2) → vote_token (Argon2 with per-voter salt) → vote_key (HKDF) → Fernet encryption. Each layer adds domain-specific binding.

2. **Benchmark/Production Mismatch**: The `benchmark_argon2()` function tests Argon2id (`Type.ID`) while production uses Argon2d (`Type.D`). This suggests the benchmark was developed with correct intent but the production code wasn't updated to match, possibly due to a copy-paste error or intentional choice that should be documented.

3. **HKDF Info String Mismatch**: The `_b64_vote_key()` function uses `info=b'xchacha20_key'` with a comment about XChaCha20-Poly1305, but the actual encryption is Fernet (AES-128-CBC + HMAC-SHA256). While not a vulnerability (HKDF info is for domain separation and the 32-byte output length works for Fernet), this creates technical debt and confusion for future maintainers.

4. **State Machine Integrity**: The election state machine (EDITABLE → OPEN → CLOSED) is enforced via assertions in `election.py` and schema constraints in `schema.sql`. Crypto operations are properly gated by state (salts generated at open, votes only during open, tally only when closed).
