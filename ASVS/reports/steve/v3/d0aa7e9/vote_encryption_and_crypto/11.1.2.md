#### ASVS-1112-HIGH-001: Absence of Formal Cryptographic Inventory Document

**Severity:** HIGH  
**Gap Type:** Type A — No control exists  
**ASVS Requirement:** 11.1.2 (Level 2)

**Location:**
- **Files Affected:** `v3/steve/crypto.py` (entire file), `v3/steve/election.py` (entire file), `v3/schema.sql` (entire file)
- **Functions:** All cryptographic functions across the codebase
- **Lines:** N/A (systemic documentation gap)

**Related Functions Checked:**
- `crypto.gen_salt()` (line 34)
- `crypto.gen_opened_key()` (lines 38-49)
- `crypto.gen_vote_token()` (lines 52-56)
- `crypto._b64_vote_key()` (lines 59-69)
- `crypto.create_vote()` (lines 72-77)
- `crypto.decrypt_votestring()` (lines 80-85)
- `election.open()` (line 73)
- `election.add_salts()` (line 130)
- `election.is_tampered()` (lines 362-375)

**Description:**

ASVS 11.1.2 requires a cryptographic inventory that is "performed, maintained, regularly updated, and includes all cryptographic keys, algorithms, and certificates used by the application." No such document exists in the codebase. Cryptographic usage is spread across `crypto.py`, `election.py`, and `schema.sql` with inline comments that serve as the only documentation—and these are demonstrably inconsistent (see subsequent findings).

**Evidence — Missing Documentation Areas:**

1. **No algorithm registry:** Algorithms are discoverable only by reading source code. No centralized list of approved algorithms, their parameters, or their intended use.

2. **No key boundary documentation (where keys can/cannot be used):**
   - `opened_key` is used in `gen_vote_token()` and `is_tampered()` — not documented
   - Per-voter `salt` is used for both vote_token derivation AND vote encryption key derivation — not documented
   - No documentation that election salts must NOT be used for vote encryption

3. **No data protection mapping (what data can/cannot be protected):**
   - Vote strings are encrypted — not formally specified
   - Election metadata (titles, descriptions) is NOT encrypted — no justification documented
   - Voter eligibility data is NOT encrypted — acknowledged in domain context but not in code documentation

4. **No key lifecycle documentation:**
   - Salt generation occurs at election open (`election.py:119-138`) — lifecycle not documented
   - `opened_key` is never rotated — policy not documented
   - Key destruction after tallying — no policy exists

**Vulnerable Code:**

N/A — This is a documentation gap, not a code vulnerability. However, the absence of inventory has allowed inconsistencies to persist (see findings 002 and 003).

**Data Flow:**

```
Election Creation → Salt Generation → Key Derivation → Vote Encryption
     ↓                    ↓                  ↓                ↓
  No docs          No lifecycle docs   No boundary docs   No algorithm docs
```

**Impact:**

Without a formal inventory, cryptographic drift goes undetected (as evidenced by findings 002 and 003 below). Algorithm transitions cannot be planned safely. Security audits require complete code review rather than document verification. ### Medium Severity

#### ASVS-1112-MED-002: Undocumented Algorithm Variant Mismatch — Argon2d in Production vs Argon2id in Benchmark

**Severity:** MEDIUM  
**Gap Type:** Type B — Control exists (benchmark uses Argon2id) but NOT applied (production uses Argon2d)  
**ASVS Requirement:** 11.1.2 (Level 2)

**Location:**
- **File:** `v3/steve/crypto.py`
- **Functions:** `_hash()` (lines 88-98) vs `benchmark_argon2()` (lines 124-156)
- **Line:** 97 (production), 150 (benchmark)

**Related Functions Checked:**
- `gen_opened_key()` (lines 38-49) — calls `_hash()` with Argon2d
- `gen_vote_token()` (lines 52-56) — calls `_hash()` with Argon2d
- `benchmark_argon2()` (lines 124-156) — tests Argon2id parameters

**Description:**

The production code uses Argon2d (`Type.D`) while the benchmark function tests Argon2id (`Type.ID`). This creates a discrepancy where:
1. Performance and security testing is conducted on a different algorithm variant than production uses
2. The recommended variant (Argon2id per RFC 9106 and OWASP guidance) is tested but not deployed
3. A cryptographic inventory would have caught this inconsistency by requiring explicit documentation of algorithm variants

**Vulnerable Code:**

Production implementation in `_hash()` (line 97):
```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2id to hash the data, with default tuning parameters."
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,  # ← Argon2d - side-channel vulnerable
    )
```

Benchmark implementation (line 150):
```python
def benchmark_argon2():
    # ... setup code ...
    hash_value = argon2.low_level.hash_secret_raw(
        secret=secret,
        salt=salt,
        time_cost=time_cost,
        memory_cost=memory_cost,
        parallelism=parallelism,
        hash_len=32,
        type=argon2.low_level.Type.ID,  # ← Argon2id - recommended variant
    )
```

**Data Flow:**

```
Election Data → gen_opened_key() → _hash(Type.D) → opened_key (32 bytes)
                                         ↓
                                  Side-channel vulnerable

Voter + Issue → gen_vote_token() → _hash(Type.D) → vote_token (32 bytes)
                                         ↓
                                  Side-channel vulnerable

Benchmark Test → benchmark_argon2() → Type.ID → Performance metrics
                                         ↓
                                  NOT testing production code path
```

**Proof of Concept:**

Demonstrating the algorithm mismatch:

```python
import argon2.low_level

# Production configuration
prod_hash = argon2.low_level.hash_secret_raw(
    secret=b"test_data",
    salt=b"0123456789abcdef",
    time_cost=2,
    memory_cost=65536,
    parallelism=4,
    hash_len=32,
    type=argon2.low_level.Type.D,  # Production
)

# Benchmark configuration
bench_hash = argon2.low_level.hash_secret_raw(
    secret=b"test_data",
    salt=b"0123456789abcdef",
    time_cost=2,
    memory_cost=65536,
    parallelism=4,
    hash_len=32,
    type=argon2.low_level.Type.ID,  # Benchmark
)

print(f"Production: {prod_hash.hex()}")
print(f"Benchmark:  {bench_hash.hex()}")
print(f"Match: {prod_hash == bench_hash}")  # False - different algorithms
```

**Impact:**

1. **Side-Channel Vulnerability:** Argon2d is vulnerable to cache-timing side-channel attacks, which could allow a co-located attacker (e.g., in cloud environments) to extract key material during key derivation. While exploitation requires precise timing measurement and repeated attempts, this is a known weakness of the Argon2d variant.

2. **False Security Confidence:** The benchmark function testing Argon2id parameters gives false confidence that the recommended variant is in use, when production actually uses the less secure variant.

3. **Inventory Gap:** A cryptographic inventory would have caught this discrepancy by requiring documentation of the specific algorithm variant in use, along with justification for choosing Argon2d over Argon2id.

**Remediation:**

Change production code to use Argon2id:

```python
# In crypto.py, _hash() function, line 97:
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2id to hash the data, with default tuning parameters."
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.ID,  # Changed from Type.D to Type.ID
    )
```

**Migration Plan:**

> **CRITICAL:** This change will invalidate all existing `opened_key` values and `vote_token` values. A migration plan is required for any elections that have been opened:

1. **For new elections:** Will automatically use Argon2id after code deployment
2. **For open elections:** Must complete tallying before upgrade, or implement dual-algorithm support
3. **For closed elections:** Historical data remains valid; tamper detection will need recalculation if verification is required

This is precisely the kind of change that a cryptographic inventory with versioning would manage systematically.

#### ASVS-1112-MED-003: HKDF Domain Separation Label Mismatched with Actual Encryption Algorithm

**Severity:** MEDIUM  
**Gap Type:** Type B — Domain separation exists but labels wrong algorithm, creating false inventory record  
**ASVS Requirement:** 11.1.2 (Level 2)

**Location:**
- **File:** `v3/steve/crypto.py`
- **Function:** `_b64_vote_key()`
- **Lines:** 59-69

**Related Functions Checked:**
- `create_vote()` (lines 72-77) — uses Fernet with keys from `_b64_vote_key()`
- `decrypt_votestring()` (lines 80-85) — uses Fernet with keys from `_b64_vote_key()`
- `gen_vote_token()` (lines 52-56) — provides input to `_b64_vote_key()`

**Description:**

The HKDF `info` parameter provides cryptographic domain separation per NIST SP 800-56C. The code uses `info=b'xchacha20_key'` while the actual encryption algorithm is Fernet (AES-128-CBC + HMAC-SHA256). This creates two problems:

1. **Inventory Falsification:** Any automated or manual inventory reading the `info` field would incorrectly record XChaCha20-Poly1305 as the encryption algorithm.

2. **Unsafe Algorithm Migration:** When the planned migration to XChaCha20-Poly1305 occurs (per the comment "will switch soon"), if the same `info=b'xchacha20_key'` is retained, the derived keys will be identical to the current Fernet keys. This eliminates cryptographic domain separation between the old and new algorithms.

**Vulnerable Code:**

```python
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    "Key-stretch the vote_token. (ref: PBKDF)"

    ### still using Fernet now, but will switch soon. Leaving comments.
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,  # 32-byte key for XChaCha20-Poly1305  ← COMMENT: Wrong algorithm
        salt=salt,
        info=b'xchacha20_key',  # ← CRITICAL: Actual cipher is AES-128-CBC (Fernet)
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

Actual usage in `create_vote()` (line 75):
```python
def create_vote(vote_token: bytes, salt: bytes, votestring: str) -> bytes:
    vote_key = _b64_vote_key(vote_token, salt)
    f = Fernet(vote_key)  # ← Using Fernet (AES-128-CBC), not XChaCha20
    return f.encrypt(votestring.encode("utf-8"))
```

**Data Flow:**

```
vote_token (32 bytes)
    ↓
HKDF-SHA256(info=b'xchacha20_key')  ← MISLEADING LABEL
    ↓
vote_key (32 bytes, base64-encoded)
    ↓
Fernet(vote_key)  ← ACTUAL ALGORITHM: AES-128-CBC + HMAC-SHA256
    ↓
ciphertext
```

**Proof of Concept:**

Demonstrating the mismatch between label and actual usage:

```python
from cryptography.hazmat.primitives import hashes
from cryptography.hazmat.primitives.kdf.hkdf import HKDF
from cryptography.fernet import Fernet
import base64

# Current implementation
vote_token = b"0" * 32
salt = b"1" * 16

keymaker = hkdf.HKDF(
    algorithm=hashes.SHA256(),
    length=32,
    salt=salt,
    info=b'xchacha20_key',  # Label says XChaCha20
)
vote_key = base64.urlsafe_b64encode(keymaker.derive(vote_token))

# But actual usage is Fernet (AES-128-CBC)
f = Fernet(vote_key)
ciphertext = f.encrypt(b"test vote")

# Inventory would record: "XChaCha20-Poly1305"
# Actual algorithm: "AES-128-CBC + HMAC-SHA256"
print(f"Label claims: XChaCha20-Poly1305")
print(f"Actual algorithm: Fernet (AES-128-CBC + HMAC-SHA256)")
```

**Impact:**

1. **Incorrect Inventory Records:** Any cryptographic inventory process that examines the code will record XChaCha20-Poly1305 as the vote encryption algorithm, when it is actually Fernet (AES-128-CBC + HMAC-SHA256).

2. **Failed Domain Separation During Migration:** When migrating to XChaCha20-Poly1305, if the same `info` value is retained:
   ```python
   # Old system: HKDF(info=b'xchacha20_key') → Fernet
   # New system: HKDF(info=b'xchacha20_key') → XChaCha20-Poly1305
   # Result: Same keys for different algorithms - defeats domain separation
   ```

3. **Misleading Code Comments:** The comment "32-byte key for XChaCha20-Poly1305" is incorrect—Fernet uses 16 bytes for AES-128 and 16 bytes for HMAC signing key.

**Remediation:**

Correct the `info` parameter to match the actual algorithm:

```python
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    "Key-stretch the vote_token into a Fernet-compatible key."
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,  # 32 bytes: 16-byte signing key + 16-byte AES-128 key (Fernet spec)
        salt=salt,
        info=b'fernet_vote_key_v1',  # Matches actual algorithm; version for migration
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

When migrating to XChaCha20-Poly1305, use a distinct `info` value:

```python
def _b64_vote_key_v2(vote_token: bytes, salt: bytes) -> bytes:
    "Key-stretch the vote_token into an XChaCha20-Poly1305-compatible key."
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,  # 32 bytes for XChaCha20-Poly1305
        salt=salt,
        info=b'xchacha20_vote_key_v2',  # New domain for new algorithm
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

#### ASVS-1112-MED-004: Non-Constant-Time Cryptographic Comparison in Tamper Detection

**Severity:** MEDIUM  
**Gap Type:** Type C — Control exists (tamper check) but comparison result may leak information through timing  
**ASVS Requirement:** 11.1.2 (Level 2)

**Location:**
- **File:** `v3/steve/election.py`
- **Function:** `is_tampered()`
- **Line:** 375

**Related Functions Checked:**
- `gen_opened_key()` in `crypto.py` (lines 38-49) — generates the comparison value
- `gather_election_data()` in `election.py` (line 368) — gathers data for integrity check
- `tally()` in `tally.py` (lines 152-154) — calls `is_tampered()` before processing votes

**Description:**

The `is_tampered()` function uses Python's `!=` operator to compare cryptographic keys. This operator performs byte-by-byte comparison that short-circuits on the first differing byte, creating a timing side-channel. While both values being compared are Argon2 hashes (making practical exploitation difficult), cryptographic best practice mandates constant-time comparison for all secret material.

A proper cryptographic inventory would document that all cryptographic comparisons must use constant-time functions to prevent timing side-channels.

**Vulnerable Code:**

```python
def is_tampered(self, pdb):
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)

    # The computed key should be unchanged.
    return opened_key != md.opened_key  # ← Python != short-circuits on first differing byte
```

**Data Flow:**

```
Election Metadata → gather_election_data() → edata
                                               ↓
                                    gen_opened_key(edata, salt)
                                               ↓
                                    computed_opened_key (32 bytes)
                                               ↓
                      Python != comparison ← stored_opened_key (32 bytes)
                                ↓
                    Returns on first byte mismatch
                                ↓
                    Timing reveals position of first difference
```

**Proof of Concept:**

Demonstrating timing difference in non-constant-time comparison:

```python
import time
import secrets

# Simulate stored and computed keys
stored_key = secrets.token_bytes(32)

# Case 1: Keys differ in first byte
computed_key_early = bytes([stored_key[0] ^ 0xFF]) + stored_key[1:]

start = time.perf_counter_ns()
for _ in range(100000):
    result = stored_key != computed_key_early
end = time.perf_counter_ns()
time_early = end - start

# Case 2: Keys differ in last byte
computed_key_late = stored_key[:-1] + bytes([stored_key[-1] ^ 0xFF])

start = time.perf_counter_ns()
for _ in range(100000):
    result = stored_key != computed_key_late
end = time.perf_counter_ns()
time_late = end - start

print(f"Time for early mismatch: {time_early} ns")
print(f"Time for late mismatch:  {time_late} ns")
print(f"Difference: {abs(time_late - time_early)} ns")
# Typically shows measurable difference
```

**Impact:**

1. **Timing Side-Channel:** An attacker with the ability to modify election data and observe response timing could theoretically deduce bytes of the stored `opened_key` through repeated attempts and precise timing measurement.

2. **Limited Practical Exploitability:** Both values being compared are Argon2 hashes, which makes practical exploitation difficult. The attacker would need:
   - Ability to modify election data
   - Precise timing measurement capability
   - Many repeated attempts
   - Statistical analysis of timing variations

3. **Cryptographic Best Practice Violation:** Regardless of practical exploitability, constant-time comparison is a fundamental cryptographic hygiene requirement. A proper inventory would have documented this requirement.

**Remediation:**

Use `hmac.compare_digest()` for constant-time comparison:

```python
import hmac

def is_tampered(self, pdb):
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)

    # Use constant-time comparison to prevent timing side-channels
    return not hmac.compare_digest(opened_key, md.opened_key)
```

### 6. Cryptographic Shuffling

**Location:** `crypto.py:101-114`  
**Pattern:** Fisher-Yates shuffle with cryptographic randomness

```python
def shuffle(lst):
    "Shuffle a list in-place, using Fisher-Yates."
    for i in range(len(lst) - 1, 0, -1):
        j = secrets.randbelow(i + 1)  # ← CSPRNG for shuffle
        lst[i], lst[j] = lst[j], lst[i]
```

**Security Benefit:** Vote order is randomized before tallying using cryptographically secure randomness, preventing database-order correlation attacks. Uses Fisher-Yates algorithm which provides uniform distribution.

### 3. No PQC Readiness Documentation

**ASVS Context:** The ASVS 11.1.2 section description specifically mentions:
> "The need to future-proof cryptographic systems against the eventual rise of quantum computing is also critical. Post-Quantum Cryptography (PQC) refers to cryptographic algorithms designed to remain secure against attacks by quantum computers."

**Current State:** The application's reliance on symmetric primitives (AES via Fernet, Argon2, BLAKE2b, HMAC-SHA256) provides natural resistance to quantum attacks for stored data, as Grover's algorithm only provides quadratic speedup against symmetric cryptography (effectively halving key strength).

**Gap:** This quantum resistance is not documented or formally assessed. The inventory should include:

```markdown
## Post-Quantum Cryptography (PQC) Assessment

| Component | Algorithm | Quantum Threat | Current Strength | PQC-Ready | Notes |
|-----------|-----------|----------------|------------------|-----------|-------|
| Vote Encryption | Fernet (AES-128-CBC) | Grover's algorithm | ~64-bit effective | ⚠️ Marginal | Upgrade to AES-256 recommended |
| Key Derivation | Argon2id | Grover's algorithm | ~128-bit effective | ✅ Yes | Sufficient for PQC era |
| Integrity (HMAC) | HMAC-SHA256 | Grover's algorithm | ~128-bit effective | ✅ Yes | Sufficient for PQC era |
| Pre-hashing | BLAKE2b | Grover's algorithm | ~256-bit effective | ✅ Yes | Sufficient for PQC era |

### Recommendations:
1. **Immediate:** Upgrade Fernet (AES-128) to XChaCha20-Poly1305 (256-bit) for better PQC margins
2. **Monitor:** Track NIST PQC standardization for any future asymmetric crypto needs
3. **Policy:** Document that symmetric primitives with ≥256-bit keys are considered PQC-ready
```

### 5. Cryptographic Agility Partially Implemented

**Positive:** The codebase demonstrates some cryptographic agility:

```python
# Add version prefix to ciphertext
def create_vote_v2(vote_token: bytes, salt: bytes, votestring: str) -> bytes:
    version = b'\x02'  # Version 2: XChaCha20-Poly1305
    vote_key = _b64_vote_key_v2(vote_token, salt)
    cipher = XChaCha20Poly1305(vote_key)
    nonce = secrets.token_bytes(24)
    ciphertext = cipher.encrypt(nonce, votestring.encode("utf-8"), None)
    return version + nonce + ciphertext

def decrypt_votestring_v2(vote_token: bytes, salt: bytes, ciphertext: bytes) -> str:
    version = ciphertext[0:1]
    if version == b'\x01':
        # Fernet (legacy)
        return decrypt_votestring_v1(vote_token, salt, ciphertext[1:])
    elif version == b'\x02':
        # XChaCha20-Poly1305
        nonce = ciphertext[1:25]
        ct = ciphertext[25:]
        vote_key = _b64_vote_key_v2(vote_token, salt)
        cipher = XChaCha20Poly1305(vote_key)
        plaintext = cipher.decrypt(nonce, ct, None)
        return plaintext.decode("utf-8")
    else:
        raise ValueError(f"Unknown ciphertext version: {version.hex()}")
```
