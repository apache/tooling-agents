### Overview

Audit of the STeVe voting system's cryptographic implementations against ASVS 11.2.4 (constant-time cryptographic operations) reveals two distinct timing side-channel vulnerabilities: a non-constant-time comparison of security-critical cryptographic keys, and the use of the data-dependent Argon2d variant which leaks information through memory access patterns.

## Findings

### HIGH Severity

#### Finding ASVS-1124-HIGH-001: Non-Constant-Time Comparison of Tamper Detection Key

| Attribute | Value |
|-----------|-------|
| **Finding ID** | ASVS-1124-HIGH-001 |
| **Severity** | HIGH |
| **Gap Type** | Type B — Control exists (tamper detection) but comparison is not performed securely |
| **File** | `v3/steve/election.py` |
| **Function** | `is_tampered()` |
| **Line** | 383 |

**Description:**

The `is_tampered()` function uses Python's `!=` operator to compare cryptographic keys, which performs byte-by-byte comparison and short-circuits at the first difference. This creates a timing side-channel that leaks information about the position of the first differing byte.

**Vulnerable Code:**

```python
def is_tampered(self, pdb):
    md = self._all_metadata()  # no required state

    # Compute an opened_key based on the current data.
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)

    # The computed key should be unchanged.
    return opened_key != md.opened_key  # <-- Non-constant-time comparison
```

**Data Flow:**

```
md.opened_key (stored 32-byte Argon2 hash from database)
    → compared with freshly computed opened_key (32-byte Argon2 hash)
    → Python's `!=` operator on bytes objects
    → Short-circuits at first differing byte
    → Timing difference leaks byte position of first mismatch
```

**Related Functions Checked:**

| Call Site | File | Line | Context |
|-----------|------|------|---------|
| `tally.py:main()` | `tally.py` | 152 | `election.is_tampered(pdb)` — called before every tally operation |
| Any web route checking tampering | (not shown) | — | Any endpoint validating election integrity |

**Proof of Concept:**

An attacker with the ability to modify election data in the database and observe response timing (e.g., through the tally process exit timing, or a web endpoint that calls `is_tampered()`) could:

1. Modify a single aspect of election data (e.g., voter eligibility in `mayvote`)
2. Trigger the tamper check via the tally CLI or any web endpoint checking election validity
3. Measure response time — a faster response means the recomputed `opened_key` diverges earlier from the stored key
4. Through iterative modification and observation, reconstruct the stored `opened_key` byte-by-byte
5. Once the `opened_key` is known, craft election modifications that produce matching keys, bypassing tamper detection entirely

**Impact:**

The `opened_key` is the sole integrity mechanism protecting election definitions from post-opening tampering. A timing side-channel that leaks information about this key directly undermines the election integrity guarantee — the most critical security property of the system.

**Remediation:**

```python
import hmac

def is_tampered(self, pdb):
    md = self._all_metadata()

    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)

    # Use constant-time comparison to prevent timing attacks
    return not hmac.compare_digest(opened_key, md.opened_key)
```

Alternatively, add a constant-time comparison utility to `crypto.py`:

```python
import hmac

def constant_time_compare(a: bytes, b: bytes) -> bool:
    """Constant-time comparison of two byte strings."""
    return hmac.compare_digest(a, b)
```

### MEDIUM Severity

#### Finding ASVS-1124-MED-001: Argon2d (Data-Dependent) Used Instead of Argon2id

| Attribute | Value |
|-----------|-------|
| **Finding ID** | ASVS-1124-MED-001 |
| **Severity** | MEDIUM |
| **Gap Type** | Type A — Inappropriate algorithm variant selection |
| **File** | `v3/steve/crypto.py` |
| **Function** | `_hash()` |
| **Line** | 97 |

**Description:**

The cryptographic hash function uses Argon2d, which has data-dependent memory access patterns that are vulnerable to side-channel attacks. The recommended variant for most applications is Argon2id, which provides resistance to both side-channel and GPU attacks.

**Vulnerable Code:**

```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2 to hash the data, with default tuning parameters."

    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,  # <-- Argon2d: data-dependent memory access
    )
```

**Internal Inconsistency Evidence:**

The benchmark function at line 148 uses `Type.ID` (Argon2id), while production code uses `Type.D` (Argon2d):

```python
# In benchmark_argon2() (line 148):
type=argon2.low_level.Type.ID,   # Argon2id — the recommended variant

# In _hash() (line 97):
type=argon2.low_level.Type.D,    # Argon2d — data-dependent, side-channel vulnerable
```

**Technical Explanation:**

Argon2 has three variants:
- **Argon2d**: Memory access pattern depends on the input data. Optimized for resistance to GPU attacks but **vulnerable to side-channel attacks** (cache timing, memory access pattern observation).
- **Argon2i**: Memory access pattern is input-independent. Resistant to side-channel attacks but less resistant to GPU attacks.
- **Argon2id**: Hybrid — first pass uses Argon2i (side-channel resistant), subsequent passes use Argon2d (GPU resistant). **Recommended by RFC 9106 and OWASP** for most applications.

**Related Functions Checked:**

All call sites using `_hash()` are affected:

| Caller | Purpose | Risk |
|--------|---------|------|
| `gen_opened_key()` (line 48) | Election tamper-detection key | Memory access pattern leaks election data hash |
| `gen_vote_token()` (line 55) | Per-voter vote identification token | Memory access pattern could correlate voters to tokens |

**Data Flow:**

```
User input (voter ID, issue ID, election data)
    → _hash() with Argon2d
    → Data-dependent memory access patterns
    → Observable through cache timing on shared hardware
    → Potential correlation of voters to vote tokens
```

**Impact:**

On shared hosting or VM environments, an attacker co-located on the same physical hardware could observe cache access patterns during Argon2d execution. Since `gen_vote_token()` is called during the voting process with voter-specific inputs (`opened_key + pid + iid`), an attacker could potentially:
- Distinguish which voter is casting a vote based on memory access patterns
- Correlate vote tokens back to specific voters, breaking ballot secrecy

Per RFC 9106 §4: "Argon2id is recommended for most applications, as it provides a reasonable level of resistance against both side-channel and GPU attacks."

**Remediation:**

```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2 to hash the data, with default tuning parameters."

    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.ID,  # Argon2id: hybrid, side-channel resistant
    )
```

> **Migration Note:** Changing the Argon2 type will produce different hash outputs. This change must be coordinated with any elections currently in the "open" state, as their `opened_key` and `vote_token` values would no longer match. Apply during a maintenance window when no elections are open.

## Positive Security Patterns

1. **Fernet HMAC Verification**: The `cryptography` library's Fernet implementation uses `hmac.compare_digest()` internally for all token authentication, correctly preventing timing attacks during vote decryption.

2. **CSPRNG Usage Throughout**: All random value generation (`gen_salt`, `create_id`, `shuffle`) uses the `secrets` module, which provides cryptographically secure randomness.

3. **HKDF for Key Derivation**: The `_b64_vote_key()` function correctly uses HKDF-SHA256 from the `cryptography` library, which implements constant-time key derivation.

4. **Fisher-Yates Shuffle**: The custom `shuffle()` function correctly implements the Fisher-Yates algorithm with `secrets.randbelow()`, providing uniform distribution with cryptographic randomness. This prevents vote ordering leakage during tallying.

5. **BLAKE2b Pre-hashing**: Using BLAKE2b to reduce arbitrary-length election data to a fixed-size digest before Argon2 is a sound pattern that avoids Argon2's input size limitations.

6. **Per-Voter Salt Isolation**: Each (person, issue) pair in `mayvote` gets a unique 16-byte salt, providing strong cryptographic separation between voters.

## Architecture Observations

1. **Single Comparison Point**: The `is_tampered()` function is the *only* location in the codebase where application code directly compares cryptographic output values. All other cryptographic comparisons (e.g., Fernet HMAC verification) are handled within well-audited library code. This means the remediation surface is minimal — fixing one line resolves the constant-time comparison gap.

2. **Argon2d Selection May Be Intentional but Undocumented**: The use of `Type.D` in production while `Type.ID` appears in benchmarks suggests this may have been a development oversight rather than a deliberate choice. If Argon2d was chosen for maximum GPU resistance (its strength), this should be documented with a risk acceptance note for the side-channel tradeoff.

3. **Tamper Detection Criticality**: The `opened_key` comparison in `is_tampered()` is the sole mechanism preventing post-opening manipulation of election definitions (voter eligibility, issue lists, etc.). Any weakness in this comparison has outsized impact on the system's integrity guarantees.
