## Findings

### Medium Severity

#### ASVS-1161-MED-001: Argon2d Variant Used Instead of Recommended Argon2id for Key Derivation

**Severity:** MEDIUM  
**Finding ID:** ASVS-1161-MED-001  
**Gap Type:** Type B — Correct variant is known (used in benchmark) but NOT applied in production code

**Location:**
- **File:** `v3/steve/crypto.py`
- **Function:** `_hash()` (lines 88–98)
- **Related Functions Checked:** `gen_opened_key()`, `gen_vote_token()`

**Description:**

The system uses Argon2d (data-dependent variant) for all key derivation operations instead of the recommended Argon2id (hybrid variant). Argon2d uses data-dependent memory access patterns, making it vulnerable to side-channel attacks such as cache-timing and memory-access pattern analysis. RFC 9106 Section 4 explicitly recommends Argon2id for general-purpose use: *"If you do not know the difference between them or you consider side-channel attacks as a viable threat, choose Argon2id."*

**Vulnerable Code:**
```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2 to hash the data, with default tuning parameters."

    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,  # <-- Argon2d: side-channel vulnerable
    )
```

**Benchmark code uses the correct variant (line ~148):**
```python
type=argon2.low_level.Type.ID,  # <-- Argon2id: recommended variant
```

**Data Flow:**
- `election.py:open()` → `crypto.gen_opened_key()` → `_hash(digest, salt)` → Argon2**d** → `opened_key` stored in DB
- `election.py:add_vote()` → `crypto.gen_vote_token()` → `_hash(opened_key + pid + iid, salt)` → Argon2**d** → `vote_token` used to derive encryption key

**Proof of Concept:**

In a shared hosting or VM environment, an attacker on the same physical host could use cache-timing analysis tools (e.g., Flush+Reload) to observe Argon2d's data-dependent memory access patterns during `gen_vote_token()` execution, potentially recovering the vote token and thus the vote encryption key.

**Impact:**

In multi-tenant server environments, a co-located attacker could potentially extract key material through memory access timing. This directly impacts both the election integrity key (`opened_key`) and per-voter encryption key derivation (`vote_token`), which protect ballot secrecy.

**Remediation:**
```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2id to hash the data, with default tuning parameters."

    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.ID,  # Argon2id: hybrid, side-channel resistant
    )
```

> **Note:** Changing the variant will produce different output for the same input. Any existing elections that have been opened will have their `opened_key` invalidated, triggering tamper detection. This change requires a migration strategy for any active elections.

#### ASVS-1161-MED-002: HKDF Domain Separation Label Misrepresents Actual Encryption Algorithm

**Severity:** MEDIUM  
**Finding ID:** ASVS-1161-MED-002  
**Gap Type:** Type A — No correct domain separation for the actual algorithm in use

**Location:**
- **File:** `v3/steve/crypto.py`
- **Function:** `_b64_vote_key()` (lines 59–69)
- **Related Functions Checked:** `create_vote()`, `decrypt_votestring()`

**Description:**

The HKDF key derivation function uses `info=b'xchacha20_key'` as the domain separation label, but the derived key is actually used with Fernet (AES-128-CBC + HMAC-SHA256), not XChaCha20-Poly1305. The HKDF `info` parameter provides cryptographic domain separation per NIST SP 800-56C. This mismatch violates proper domain separation and creates risk during algorithm transitions.

**Vulnerable Code:**
```python
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    "Key-stretch the vote_token. (ref: PBKDF)"

    ### still using Fernet now, but will switch soon. Leaving comments.
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,  # 32-byte key for XChaCha20-Poly1305
        salt=salt,
        info=b'xchacha20_key',  # <-- Label for algorithm NOT in use
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

**Actual usage (lines 72–77, 80–85):**
```python
f = cryptography.fernet.Fernet(b64key)  # AES-128-CBC + HMAC-SHA256
```

**Data Flow:**
- `vote_token` + `salt` → HKDF(SHA-256, info=`xchacha20_key`) → 32-byte key → base64 → **Fernet** (AES-128-CBC + HMAC-SHA256)

**Proof of Concept:**

During algorithm transition, if a developer adds XChaCha20-Poly1305 encryption using the existing `_b64_vote_key()` function (since its label already says `xchacha20_key`), the same key material would encrypt votes under both AES-128-CBC (old votes) and XChaCha20-Poly1305 (new votes). An attacker with access to ciphertexts under both algorithms using the same key could exploit algorithmic differences to recover plaintext.

**Impact:**

When XChaCha20-Poly1305 is added (as indicated by the comment "will switch soon"), if the same HKDF parameters are used, identical key material would be derived for both algorithms. This creates a **key reuse vulnerability across different ciphers** — a well-documented cryptographic failure mode that can leak plaintext. The incorrect label also makes cryptographic inventory auditing unreliable, as the documented algorithm doesn't match the actual implementation.

**Remediation:**
```python
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    "Key-stretch the vote_token into a Fernet-compatible key."

    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,  # Fernet: 16-byte signing key + 16-byte AES-128 key
        salt=salt,
        info=b'fernet_vote_key_v1',  # Correct domain label for actual algorithm
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

> **Note:** As with Finding 001, changing the `info` parameter changes the derived key. This requires coordinated migration of any existing encrypted votes.

#### ASVS-1161-MED-003: Non-Constant-Time Comparison of Cryptographic Key Material in Tamper Detection

**Severity:** MEDIUM  
**Finding ID:** ASVS-1161-MED-003  
**Gap Type:** Type B — `hmac.compare_digest()` is available in stdlib but NOT used for key comparison

**Location:**
- **File:** `v3/steve/election.py`
- **Function:** `is_tampered()` (lines 359–377)
- **Related Functions Checked:** Entry points in `tally.py:main()` (line ~158)

**Description:**

The tamper detection function uses Python's `!=` operator to compare cryptographic key material (`opened_key`). This operator performs byte-by-byte comparison and returns early on the first mismatch, creating a timing side-channel vulnerability. For a 32-byte key, this creates a measurable timing differential depending on how many leading bytes match.

**Vulnerable Code:**
```python
def is_tampered(self, pdb):
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)

    # The computed key should be unchanged.
    return opened_key != md.opened_key  # <-- Non-constant-time comparison
```

**Entry Points Where Called:**
- `tally.py:main()` (line ~158): Called before every tally operation
- Potentially called from web endpoints during election state transitions

**Data Flow:**
- `gather_election_data(pdb)` → `crypto.gen_opened_key()` → 32-byte `opened_key` → Python `!=` operator (short-circuits on first byte difference) → boolean result

**Proof of Concept:**

An authenticated administrator making repeated modifications to election data (e.g., voter list) while measuring the time `is_tampered()` takes to respond could observe timing variations correlating with how many bytes of the recomputed key match the stored key. While direct key recovery is impractical due to Argon2's avalanche properties, the principle of defense in depth requires constant-time comparison for all cryptographic material.

**Impact:**

While the Argon2 output is pseudorandom (making targeted byte-by-byte recovery impractical through input manipulation alone), this violates cryptographic implementation best practices. In scenarios where an attacker can repeatedly trigger tamper checks with controlled election data modifications and measure server response times (e.g., through a web-facing endpoint), information about the stored `opened_key` could leak.

**Remediation:**
```python
import hmac

def is_tampered(self, pdb):
    md = self._all_metadata()
    edata = self.gather_election_data(pdb)
    opened_key = crypto.gen_opened_key(edata, md.salt)

    # Use constant-time comparison to prevent timing side channels
    return not hmac.compare_digest(opened_key, md.opened_key)
```

## Architecture Observations

1. **Layered Key Derivation:** The system uses a well-designed multi-layer key hierarchy: election salt → opened_key (Argon2) → vote_token (Argon2 with per-voter salt) → vote_key (HKDF) → Fernet encryption. Each layer provides different security properties (election isolation, voter anonymity, key stretching).

2. **Crypto Centralization:** All cryptographic operations are properly centralized in `crypto.py`, making audit tractable. No raw cryptographic primitives are used in `election.py` — only the `crypto` module's API.

3. **Algorithm Transition Risk:** The codebase shows clear signs of a planned transition from Fernet to XChaCha20-Poly1305 (comments, HKDF info label). This transition needs careful management to avoid key reuse and ensure backward compatibility for existing encrypted votes.

4. **No Digital Signatures:** The system uses Argon2 hashing rather than digital signatures for tamper detection. This is acknowledged as intentional in the false positive patterns, and provides adequate integrity checking for the state machine model.
