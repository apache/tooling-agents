**Key Findings:**
The audit examines whether nonces, initialization vectors, and other single-use numbers are properly managed across the cryptographic operations in the STeVe election system. The core encryption (Fernet) correctly relies on the `cryptography` library's internal IV generation using `os.urandom(16)`, which prevents IV reuse. However, one finding was identified where the HKDF key derivation method uses an incorrect algorithm context binding, violating the "method of generation must be appropriate for the algorithm being used" clause.

```
Vote Decryption:
  Same chain → Fernet.decrypt(ciphertext) → votestring
```

## Findings

### Medium Severity

#### ASVS-1134-MED-001: HKDF Key Derivation Uses Incorrect Algorithm Context Binding

| Attribute | Value |
|-----------|-------|
| **Severity** | MEDIUM |
| **Gap Type** | Type B — Control EXISTS (HKDF with info parameter) but MISCONFIGURED |
| **File** | `v3/steve/crypto.py` |
| **Function** | `_b64_vote_key` |
| **Lines** | 59–69 |
| **Related Functions** | `create_vote()` (L73-76), `decrypt_votestring()` |

**Description:**

The HKDF `info` parameter serves as a cryptographic domain separator that binds derived key material to its intended algorithm and purpose (NIST SP 800-56C Rev. 2, Section 4). The current implementation uses `info=b'xchacha20_key'` while the derived key is actually consumed by Fernet, which uses AES-128-CBC + HMAC-SHA256. The comment on line 61 explicitly states: *"still using Fernet now, but will switch soon."*

This violates ASVS 11.3.4's requirement that "the method of generation must be appropriate for the algorithm being used" because:

1. **Incorrect context binding**: The key derivation is bound to XChaCha20-Poly1305 via the info parameter, but the key is used with AES-128-CBC (Fernet). The domain separator does not match the actual algorithm.
2. **Migration key-reuse risk**: The code explicitly signals a planned migration to XChaCha20-Poly1305. If the `info` parameter is not changed during migration, the identical key material would be used with both AES-128-CBC (old ciphertexts) and XChaCha20-Poly1305 (new ciphertexts), creating cross-algorithm key reuse — a direct nonce/key-pair violation.
3. **Both algorithms use 32-byte keys**: Fernet takes 32 bytes (split into 16B HMAC + 16B AES) and XChaCha20-Poly1305 takes 32 bytes for the cipher key. The HKDF output would be byte-identical for both, meaning no cryptographic separation exists between algorithm versions.

**Vulnerable Code:**

```python
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    "Key-stretch the vote_token. (ref: PBKDF)"

    ### still using Fernet now, but will switch soon. Leaving comments.
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,  # 32-byte key for XChaCha20-Poly1305
        salt=salt,
        info=b'xchacha20_key',  # ← Does not match actual algorithm
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

**Data Flow:**

```
vote_token (32B) 
  → HKDF(SHA256, salt=mayvote.salt, info=b'xchacha20_key', length=32)
  → vote_key (32B) 
  → base64url encode 
  → Fernet(AES-128-CBC + HMAC-SHA256)  ← Algorithm mismatch with info
```

**Proof of Concept:**

The misconfiguration can be demonstrated by examining the key derivation path:

1. Current state: `info=b'xchacha20_key'` → key used with Fernet (AES-128-CBC)
2. Post-migration: `info=b'xchacha20_key'` (unchanged) → key used with XChaCha20-Poly1305
3. Result: Same key material protects both old and new ciphertexts under different algorithms

**Impact:**

- **Current**: The incorrect `info` string does not weaken the derived key for Fernet usage, but it misrepresents the cryptographic binding, creating false confidence in algorithm separation.
- **During migration**: If XChaCha20-Poly1305 is adopted without updating `info`, the same derived key would encrypt votes under two different algorithms. An attacker with access to both old (Fernet) and new (XChaCha20) ciphertexts encrypted under the same key gains additional material for cross-protocol analysis.

**Remediation:**

Update the `info` parameter to reflect the actual algorithm, and plan for key separation during migration:

```python
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    "Key-stretch the vote_token for Fernet encryption."
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,  # 32-byte key for Fernet (16B HMAC + 16B AES-128)
        salt=salt,
        info=b'fernet_vote_key_v1',  # Correct algorithm binding
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

When migrating to XChaCha20-Poly1305, use a distinct info value (e.g., `b'xchacha20_vote_key_v2'`) to ensure cryptographic separation from legacy keys. Consider a versioned migration strategy that re-encrypts existing votes during the transition.

## Positive Security Patterns

1. **Fernet IV Generation**: The `cryptography` library's Fernet implementation generates a fresh 16-byte random IV via `os.urandom()` for every `encrypt()` call. This ensures IV uniqueness even when the same key encrypts multiple plaintexts (re-voting scenario).

2. **CSPRNG Throughout**: All random values (`secrets.token_bytes`, `secrets.token_hex`, `secrets.randbelow`) use Python's `secrets` module backed by `os.urandom()`, which is the appropriate source for cryptographic randomness.

3. **Per-Voter Salt Uniqueness**: Each (voter, issue) pair receives a unique 16-byte salt generated at election open time (`add_salts()`), ensuring distinct key derivation paths. The `PRIMARY KEY (pid, iid)` constraint in the `mayvote` table enforces uniqueness at the database level.

4. **No Static/Hardcoded Nonces**: No hardcoded IVs, nonces, or salts were found. All cryptographic randomness is generated at runtime.

5. **Collision-Resistant ID Generation**: Election and issue IDs use `secrets.token_hex(5)` with retry loops on `IntegrityError`, preventing ID reuse.

6. **Vote Shuffling with CSPRNG**: The Fisher-Yates shuffle in `crypto.shuffle()` uses `secrets.randbelow()` to prevent vote ordering leakage during tallying.

## Architecture Observations

1. **Argon2d vs. Argon2id**: Production code (`_hash()`, line 89) uses `argon2.low_level.Type.D` (Argon2d) while the benchmark function (`benchmark_argon2()`) uses `Type.ID` (Argon2id). RFC 9106 and OWASP recommend Argon2id for most use cases as it combines Argon2i's side-channel resistance with Argon2d's GPU-resistance. While not directly an ASVS 11.3.4 concern (Argon2d generates key material, not nonces/IVs), this discrepancy warrants review under algorithm selection requirements.

2. **Non-Constant-Time Comparison in Tamper Detection**: The `is_tampered()` function (election.py, line ~303) uses `opened_key != md.opened_key` which is not constant-time. While not an ASVS 11.3.4 concern, the domain context explicitly calls for constant-time operations. Consider `hmac.compare_digest()` for timing-attack resistance.

3. **Re-voting Key Reuse Pattern**: When a voter re-votes on the same issue, the same Fernet key is used (deterministic from vote_token + salt). Multiple ciphertexts exist in the `vote` table under the same key but with distinct random IVs. This is cryptographically safe for AES-128-CBC with random IVs within practical usage bounds, but reveals that a re-vote occurred to anyone with database access (the same `vote_token` appears multiple times).

4. **HKDF Salt Dual-Use**: The `mayvote.salt` serves as both the Argon2d salt (for vote_token generation) and the HKDF salt (for encryption key derivation). While using different algorithms with different internal structures mitigates direct risk, dedicated salts per cryptographic operation would improve defense-in-depth.
