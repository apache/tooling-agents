## Findings

### Medium Severity

#### ASVS-1151-MED-001: Election and Issue IDs Generated with Insufficient Entropy (40 bits vs. 128-bit minimum)

**Severity:** MEDIUM

**Location:**
- **File:** `v3/steve/crypto.py`
- **Function:** `create_id()`
- **Line:** ~118
- **Also enforced by:** `v3/schema.sql`, `election.eid` CHECK constraint (line ~61), `issue.iid` CHECK constraint (line ~104)

**Related Functions Checked:**
- `election.py:Election.create()` L370 (caller for EID)
- `election.py:Election.add_issue()` L195 (caller for IID)

**Description:**

Election and issue identifiers are generated with only 40 bits of entropy (5 bytes = 10 hex characters), significantly below the ASVS 11.5.1 requirement of at least 128 bits for non-guessable values. The schema explicitly documents that these IDs are intended to be non-guessable to prevent URL enumeration.

**Vulnerable Code:**

```python
# crypto.py
def create_id():
    "Create a standard ID value."
    # Use 10 hex characters for the ID
    return secrets.token_hex(5)  # 5 bytes  ← 40 bits of entropy
```

```sql
-- schema.sql (election table)
eid  TEXT
       PRIMARY KEY NOT NULL
       CHECK (length(eid) = 10                          -- ← enforces 40-bit ceiling
              AND eid GLOB '[0-9a-f]...[0-9a-f]'),

-- schema.sql (issue table)
iid  TEXT
       PRIMARY KEY NOT NULL
       CHECK (length(iid) = 10                          -- ← enforces 40-bit ceiling
              AND iid GLOB '[0-9a-f]...[0-9a-f]'),
```

**Data Flow:**

| Step | Location | Detail |
|------|----------|--------|
| **Source** | `crypto.create_id()` | `secrets.token_hex(5)` → 5 bytes → 10 hex chars |
| **Callers** | `election.py:Election.create()` L370 | EID for new elections |
| | `election.py:Election.add_issue()` L195 | IID for new issues |
| **Schema Constraint** | `schema.sql` | `CHECK (length(eid) = 10)` and `CHECK (length(iid) = 10)` prevent longer values |
| **Missing Control** | — | No path to achieve 128-bit entropy without changing both code and schema |

**Explicit Design Intent (Non-Guessability):**

The schema comments explicitly state these IDs are intended to be non-guessable:
> *"The Election ID; 10 hex characters. We do not use AUTOINCREMENT, so that URLs for Elections cannot be deduced."* — `schema.sql` line 59

> *"The Issue ID; 10 hex characters. We do not use AUTOINCREMENT, so that URLs for Issues cannot be deduced."* — `schema.sql` line 100

**Impact:**
- **Keyspace:** 2^40 ≈ 1.1 trillion possible IDs
- **Birthday bound:** ~2^20 ≈ 1 million — collisions become probable at this threshold
- **Enumeration:** An authenticated attacker without rate limiting (noted as absent per domain context) could probe for valid EIDs/IIDs. At 1,000 req/sec, the full 40-bit space could be scanned in ~12.7 days; targeted probing against a small number of active elections would be much faster.
- **Information disclosure:** Discovery of active election IDs could reveal election existence, titles, and participation status to unauthorized authenticated users.
- **Note:** Vote secrecy is NOT directly impacted because per-voter salts (128-bit) protect vote tokens independently.

**Proof of Concept:**

```python
import itertools
import requests

# With 40-bit keyspace and known hex charset, enumeration is feasible
# Try first ~1M values to find active elections (birthday-probability approach)
session = requests.Session()
session.cookies.set('session', VALID_AUTH_COOKIE)

for candidate in itertools.product('0123456789abcdef', repeat=10):
    eid = ''.join(candidate)
    resp = session.get(f'https://vote.example.org/election/{eid}')
    if resp.status_code == 200:
        print(f'Found election: {eid}')
        # In practice, use smarter sampling — random probes
```

**Remediation:**

Increase ID entropy to at least 128 bits (16 bytes → 32 hex characters). This requires coordinated changes to both code and schema:

```python
# crypto.py - fixed
def create_id():
    "Create a standard ID value with 128 bits of entropy."
    return secrets.token_hex(16)  # 16 bytes = 128 bits → 32 hex characters
```

```sql
-- schema.sql - fixed
eid  TEXT
       PRIMARY KEY NOT NULL
       CHECK (length(eid) = 32
              AND eid GLOB '[0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f][0-9a-f]'),

-- Same change for iid in the issue table
```

**Gap Type Classification:** Type A — The control (CSPRNG) exists and is applied, but the entropy parameter is insufficient. This is not a missing control but an incorrectly configured one.

## Positive Security Patterns

1. **Consistent CSPRNG usage**: All random generation uses Python's `secrets` module — no use of `random`, `os.urandom()` directly, or other weaker sources. This is exemplary.

2. **Proper salt generation**: `gen_salt()` generates 16 bytes (128 bits) using `secrets.token_bytes()`, meeting the ASVS requirement exactly.

3. **Cryptographically secure shuffle**: The `shuffle()` function implements Fisher-Yates with `secrets.randbelow()` rather than using `random.shuffle()`. This prevents vote ordering leakage during tallying.

4. **ID collision handling**: Both `Election.create()` and `Election.add_issue()` handle `IntegrityError` with retry loops, preventing ID collision issues despite the smaller keyspace.

5. **Strong key derivation chain**: The vote encryption key derivation (`opened_key` → `vote_token` → `vote_key`) uses Argon2 (memory-hard) followed by HKDF-SHA256, providing robust key stretching even though the underlying IDs have lower entropy. The per-voter salt (128-bit) ensures vote token uniqueness.

6. **Library-managed Fernet IV**: Vote encryption uses `cryptography.fernet.Fernet`, which internally generates a 128-bit IV using `os.urandom()` for each encryption operation.

---

## Architecture Observations

1. **Entropy isolation**: The 40-bit EID/IID entropy weakness does NOT propagate to vote security because per-voter salts (128-bit) are independently generated and used as primary entropy sources in vote token derivation. The `gen_vote_token()` function combines `opened_key + pid + iid + salt`, where the salt dominates entropy.

2. **Schema as enforcement layer**: The `CHECK` constraints in `schema.sql` act as a secondary enforcement of ID format. Any fix to ID entropy must update both the Python code and the SQL schema, making this a coordinated multi-file change.

3. **Single `create_id()` function**: All non-guessable identifiers flow through one function, making remediation straightforward — a single code change plus schema migration addresses all instances.

4. **No UUID usage**: The codebase correctly avoids UUIDs (which ASVS 11.5.1 explicitly calls out as non-compliant) in favor of random hex strings.
