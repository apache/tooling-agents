## Findings

### Medium Severity

#### ASVS-1135-MED-001: HKDF Key Derivation Context Bound to Wrong Algorithm Creates Key Reuse Risk

**Severity:** MEDIUM

**Location:**
- **File:** `v3/steve/crypto.py`
- **Function:** `_b64_vote_key()`
- **Lines:** 60-71

**Related Functions Checked:**
- `create_vote()` (lines 74-79)
- `decrypt_votestring()` (lines 82-87)
- `gen_vote_token()` (lines 52-57)

**Vulnerable Code:**
```python
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    "Key-stretch the vote_token. (ref: PBKDF)"

    ### still using Fernet now, but will switch soon. Leaving comments.
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,  # 32-byte key for XChaCha20-Poly1305
        salt=salt,
        info=b'xchacha20_key',  # <-- Incorrect context for Fernet
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

**Description:**

The HKDF `info` parameter provides cryptographic domain separation — it binds the derived key material to a specific purpose, ensuring keys derived for one algorithm are not reused for another. The current implementation uses `info=b'xchacha20_key'`, but the derived key is consumed by Fernet (AES-128-CBC + HMAC-SHA256), not XChaCha20-Poly1305.

This creates two concrete problems:

1. **Incorrect cryptographic context binding:** The key is semantically bound to XChaCha20 usage but is actually used for a different encrypt-then-MAC scheme (Fernet). HKDF's `info` parameter exists specifically to prevent this type of cross-algorithm key usage.

2. **Direct path to key reuse during planned migration:** The code comments (`### still using Fernet now, but will switch soon`) confirm an active plan to migrate to XChaCha20-Poly1305. When this migration occurs, if the `info` label remains `b'xchacha20_key'`, the same derived key material will be used for both the old Fernet EtM scheme and the new XChaCha20-Poly1305 AEAD scheme. Reusing key material across different encryption/MAC constructions can undermine the security guarantees of both.

**Data Flow:**
```
vote_token (Argon2 output) 
  → HKDF(info=b'xchacha20_key') 
  → 32-byte key 
  → base64 encode 
  → Fernet(AES-128-CBC + HMAC-SHA256)
```

**Proof of Concept:**

Current flow:
```python
# vote_token derived from Argon2
vote_token = crypto._hash(b'voter_uuid', b'salt')

# HKDF with xchacha20_key context
vote_key = crypto._b64_vote_key(vote_token, salt)

# But key is used with Fernet (AES-CBC + HMAC)
encrypted = crypto.create_vote(vote_key, plaintext)
```

During migration scenario:
```python
# Same HKDF info parameter
vote_key_old = HKDF(info=b'xchacha20_key').derive(vote_token)  # Used for Fernet
vote_key_new = HKDF(info=b'xchacha20_key').derive(vote_token)  # Used for XChaCha20

# vote_key_old == vote_key_new → Key reuse across algorithms
```

**Impact:**
- **Currently:** Fernet's encrypt-then-MAC mode operates correctly regardless of the `info` value, so there is no active compromise of EtM guarantees.
- **During migration:** Key material reuse between Fernet (CBC+HMAC) and XChaCha20-Poly1305 violates NIST SP 800-108 key separation requirements. An attacker with access to ciphertexts from both schemes could potentially exploit the shared key relationship.

**Remediation:**

Update the HKDF `info` parameter to correctly reflect the current algorithm, and plan for a distinct `info` value when migrating:

```python
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    "Key-stretch the vote_token. (ref: PBKDF)"
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,
        salt=salt,
        info=b'fernet_vote_key_v1',  # Correct context for current algorithm
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

When migrating to XChaCha20-Poly1305, use a different `info` value:
```python
info=b'xchacha20poly1305_vote_key_v2'
```

**Migration Note:** Changing the `info` value will change all derived keys, making existing ciphertexts undecryptable. This must be coordinated with a data migration that re-encrypts all stored votes, or the system must support dual decryption during a transition period.

## Positive Security Patterns

### P1: Fernet Correctly Implements Encrypt-then-MAC ✅
**Location:** `crypto.py:74-87` (`create_vote()`, `decrypt_votestring()`)

The `cryptography.fernet.Fernet` library implements the Fernet specification:
- **Encryption path:** AES-128-CBC encrypts plaintext → HMAC-SHA256 authenticates (Version ‖ Timestamp ‖ IV ‖ Ciphertext) → Token = Version ‖ Timestamp ‖ IV ‖ Ciphertext ‖ HMAC
- **Decryption path:** HMAC is verified **before** any decryption occurs → Only if MAC is valid does AES-128-CBC decryption proceed

This satisfies ASVS 11.3.5's encrypt-then-MAC requirement.

### P2: No Custom Encryption+MAC Constructions ✅
**Location:** All crypto operations in `crypto.py`

The codebase uses a well-vetted library implementation (Fernet) for all encryption+MAC operations rather than constructing custom combinations. There are no instances of bare AES operations paired with manual HMAC computation, which eliminates the risk of incorrect ordering.

### P3: Strong Key Derivation Chain ✅
**Location:** `crypto.py:39-71`

The key derivation chain (BLAKE2b → Argon2 → HKDF → Fernet) ensures that:
- Election data is efficiently reduced for KDF input (BLAKE2b)
- Key material is expensive to brute-force (Argon2, 64MB memory, 2 iterations)
- Final encryption keys are properly derived with domain separation intent (HKDF)

### P4: Cryptographically Secure Randomness Throughout ✅
**Location:** `crypto.py:34-36, 104-117, 120-124`

All random operations use the `secrets` module:
- `gen_salt()`: `secrets.token_bytes(16)`
- `shuffle()`: `secrets.randbelow()` for Fisher-Yates
- `create_id()`: `secrets.token_hex(5)`

### P5: Vote Order Anonymization ✅
**Location:** `election.py:287-291`

Votes are shuffled using a cryptographically secure shuffle before tallying, preventing database-order correlation even if decryption occurs.

## Architecture Observations

### Cryptographic Layer Separation
The architecture properly separates concerns:
- `crypto.py`: All cryptographic primitives (encryption, hashing, key derivation)
- `election.py`: Business logic that calls crypto operations at appropriate state transitions
- `tally.py`: Offline decryption with tamper checking

### State Machine Enforcement
Encryption operations are gated by election state:
- `add_vote()` requires `S_OPEN` state (line 226)
- `tally_issue()` requires `S_CLOSED` state (line 244)
- `add_salts()` requires `S_EDITABLE` state (line 137)

This ensures cryptographic operations occur only at appropriate lifecycle stages.

### Single Encryption Scheme
The entire system uses exactly one encryption+MAC scheme (Fernet). This simplifies EtM verification — there is only one combination to audit.
