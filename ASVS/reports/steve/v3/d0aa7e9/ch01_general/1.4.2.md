## Findings

### Medium Severity

#### ASVS-142-MED-001: Missing Upper-Bound Range Validation on STV `seats` Integer Parameter

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/bin/create-election.py`
  - **Function:** `validate_issue()`
  - **Lines:** 60-61 — validates `> 0` only
- **File:** `v3/steve/election.py`
  - **Function:** `add_issue()`
  - **Line:** 174 — no KV content validation
- **File:** `v3/steve/vtypes/stv.py`
  - **Function:** `tally()`
  - **Line:** 65 — uses `seats` without range check

**Related Functions Checked:**
- `create-election.py:validate_issue()` — partial validation present
- `election.py:add_issue()` — no validation
- `stv.py:tally()` — no validation
- `pages.py:vote_on_page()` — reads value for display only

**Gap Type:** Type B — Validation EXISTS (partially) in CLI but NOT APPLIED at the API layer (`add_issue()`), and missing upper bound everywhere.

**Description:**

The STV (Single Transferable Vote) election type accepts a `seats` parameter that determines how many candidates should be elected. While the CLI import tool validates that `seats` is a positive integer, there is no upper-bound validation anywhere in the codebase. Additionally, the core API function `election.add_issue()` performs no validation on the `kv` dictionary contents at all, creating a defense-in-depth gap.

**Vulnerable Code:**

In `create-election.py:validate_issue()`:
```python
# Line 60-61: Only validates positive, no upper bound
if not isinstance(kv['seats'], int) or kv['seats'] <= 0:
    raise ValueError('STV seats must be a positive integer')
```

In `election.py:add_issue()`:
```python
# Line 174-179: No validation of kv contents at all
def add_issue(self, title, description, vtype, kv):
    assert self.is_editable()
    assert vtype in vtypes.TYPES
    # kv passed directly without any integer range validation
    ...
    self.c_add_issue.perform(
        iid, self.eid, title, description, vtype, self.kv2json(kv)
    )
```

In `stv.py:tally()`:
```python
# Line 65-68: Uses seats directly without range checking
seats = kv['seats']
# ...
results = stv_tool.run_stv(names, votes, seats)
```

**Data Flow:**

```
YAML file (seats: <integer>) 
  → create-election.py:validate_issue() [checks > 0, no upper bound]
  → election.add_issue(kv={"seats": N}) [no validation]
  → Database (KV JSON column)
  → stv.py:tally() → stv_tool.run_stv(names, votes, N) [no validation]
```

**Proof of Concept:**

YAML file with extreme `seats` value:
```yaml
election:
  title: "Test Election"
  owner_pid: "attacker"
issues:
  - title: "Board Election"
    vtype: stv
    kv:
      version: 1
      seats: 2147483647  # INT32_MAX - passes validation
      labelmap:
        a: "Alice"
        b: "Bob"
record:
  - voter1
```

This value passes `validate_issue()` (it is a positive integer), gets stored in the database, and when `tally()` is called, `seats: 2147483647` is passed to `stv_tool.run_stv()`. Depending on the STV algorithm's implementation:
- If it allocates arrays proportional to `seats`, it could exhaust memory
- If `seats > len(candidates)`, election results are logically incorrect
- If the underlying STV tool uses C extensions with fixed-width integers, actual integer overflow could occur

**Impact:**
- Resource exhaustion during tally (denial of service)
- Logically incorrect election results if `seats` exceeds number of candidates
- Potential integer overflow if the STV tool module uses C-based numeric processing
- Bypass of validation when new entry points are added (e.g., web-based STV creation per TODO in `do_add_issue_endpoint`)

**Remediation:**

Add range validation at multiple layers for defense-in-depth:

**1. In `election.py:add_issue()` — API layer validation:**
```python
def add_issue(self, title, description, vtype, kv):
    assert self.is_editable()
    assert vtype in vtypes.TYPES
    
    # Validate STV-specific integer parameters
    if vtype == 'stv' and kv:
        seats = kv.get('seats')
        if not isinstance(seats, int) or seats <= 0:
            raise ValueError('STV seats must be a positive integer')
        if seats > 100:  # Reasonable upper bound for any election
            raise ValueError('STV seats exceeds maximum allowed (100)')
        labelmap = kv.get('labelmap', {})
        if seats > len(labelmap):
            raise ValueError('STV seats cannot exceed number of candidates')
    ...
```

**2. In `stv.py:tally()` — Validate before algorithm execution:**
```python
def tally(votestrings, kv):
    seats = kv['seats']
    labelmap = kv['labelmap']
    
    # Range validation at point of use
    if not isinstance(seats, int) or seats <= 0:
        raise ValueError('Invalid seats value')
    if seats > len(labelmap):
        raise ValueError(f'seats ({seats}) exceeds candidate count ({len(labelmap)})')
    ...
```

**3. In `create-election.py:validate_issue()` — Add upper bound:**
```python
if not isinstance(kv['seats'], int) or kv['seats'] <= 0:
    raise ValueError('STV seats must be a positive integer')
if kv['seats'] > 100:
    raise ValueError('STV seats exceeds maximum allowed value')
labelmap = kv.get('labelmap', {})
if kv['seats'] > len(labelmap):
    raise ValueError('STV seats cannot exceed number of candidates')
```

## Positive Security Patterns

1. **Hardcoded Cryptographic Parameters**: All Argon2 parameters (`time_cost`, `memory_cost`, `parallelism`, `hash_len`) and HKDF parameters (`length=32`) are hardcoded constants in `crypto.py`, preventing any user-controlled integer manipulation at the cryptographic boundary with C libraries.

2. **Secure Random Generation**: All randomness uses the `secrets` module (`token_bytes`, `token_hex`, `randbelow`), which properly handles integer ranges internally and is cryptographically secure.

3. **Python Arbitrary Precision**: The codebase benefits from Python's native big-integer support, eliminating traditional integer overflow/underflow vulnerabilities in arithmetic operations throughout `crypto.py`, `election.py`, and `pages.py`.

4. **Constant Salt Length**: `SALT_LEN = 16` is defined once as a module constant and used consistently, preventing length-related integer errors.

5. **Safe Fisher-Yates Implementation**: The `shuffle()` function in `crypto.py` correctly uses `range(len(x) - 1, 0, -1)` and `secrets.randbelow(i + 1)`, which safely handles edge cases (empty list produces empty range).

6. **Date Validation in Web Path**: `_set_election_date()` uses `datetime.datetime.fromisoformat()` which inherently bounds the parsed date to valid datetime ranges (year 1–9999).

7. **Vote Type Validation**: Both `create-election.py` and `election.add_issue()` validate `vtype` against allowed sets (`VALID_VTYPES` and `vtypes.TYPES` respectively), preventing unexpected vote type processing.

---

## Architecture Observations

1. **Language Choice Mitigates Core Risk**: Python's arbitrary-precision integers mean that ASVS 1.4.2's primary concern (memory corruption from integer overflow) is structurally prevented for pure Python code. The remaining risk surface is at C library boundaries and in algorithmic complexity.

2. **C Library Boundaries Are Well-Controlled**: The two C library interfaces (`argon2.low_level` and `cryptography`) both receive only hardcoded integer parameters. No user-controlled integers reach these boundaries.

3. **Validation Layer Gap**: Integer validation for STV configuration (`seats`) exists only in the CLI import tool (`create-election.py`), not in the core API (`election.add_issue()`). This creates a defense-in-depth gap where new entry points (e.g., future web-based STV creation, noted in the TODO comment in `do_add_issue_endpoint`) would bypass validation.

4. **Dynamic Module Loading**: `stv.py` loads `stv_tool` from a relative file path. If `stv_tool.run_stv()` is implemented with C extensions or numpy, the unbounded `seats` parameter becomes a more concrete integer overflow risk at that boundary.
