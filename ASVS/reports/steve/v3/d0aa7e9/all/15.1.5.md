## ASVS-15.1.5: Secure Coding and Architecture Documentation - Dangerous Functionality

### Findings for 15.1.5

### [HIGH]

**Finding ID**: ASVS-1515-HIGH-001
**Location**: `v3/steve/vtypes/stv.py` (lines 21-30), `v3/server/pages.py` (line 574)
**Issue**: No documentation highlights parts of the application using "dangerous functionality"

**Evidence**:

The following dangerous operations exist without documentation:

**1. Dynamic code execution via `importlib`** in `stv.py`:
```python
def load_stv():
    pathname = os.path.join(os.path.dirname(__file__), STV_RELPATH)
    spec = importlib.util.spec_from_file_location('stv_tool', pathname)
    module = importlib.util.module_from_spec(spec)
    spec.loader.exec_module(module)
    return module
```
This loads and executes arbitrary Python code from a relative path (`../../../monitoring/stv_tool.py`). If the relative path is manipulated or the target file is compromised, arbitrary code execution occurs.

**2. File serving from user-influenced path** in `pages.py` (line 574):
```python
@APP.get('/docs/<iid>/<docname>')
async def serve_doc(iid, docname):
    ### verify the propriety of DOCNAME.
    return await quart.send_from_directory(DOCSDIR / iid, docname)
```
The comment `### verify the propriety of DOCNAME.` indicates the developer is aware this is dangerous but has not implemented validation. This is raw file path handling based on user input.

**3. JSON deserialization of database-stored data** in `election.py` (line 368):
```python
@staticmethod
def json2kv(j):
    return j and json.loads(j)
```
While the data originates from the database, it was originally user-supplied and stored via `kv2json`. The KV data is then used to construct file paths, candidate lists, and render in templates.

**4. YAML deserialization** in `create-election.py` (line 61):
```python
with open(yaml_file, 'r') as f:
    data = yaml.safe_load(f)
```
Uses `safe_load` (good), but the operation itself is parsing untrusted file data that drives election creation.

**Missing documentation**: `ARCHITECTURE.md` does not contain a "Dangerous Functionality" or "High-Risk Operations" section. `TODO.md` notes some issues informally but is not structured documentation.

**Impact**: Without formal documentation of dangerous operations, security reviewers and developers may not apply appropriate scrutiny to these areas. The dynamic module loading in particular is a high-risk pattern that should be explicitly documented with its security justification and constraints.

**Remediation**: Add to ARCHITECTURE.md:
```markdown
## Dangerous Functionality

### Dynamic Module Loading
- **Location**: `steve/vtypes/stv.py`
- **Operation**: `importlib.util.spec_from_file_location()` loads external STV algorithm
- **Risk**: Arbitrary code execution if target file is compromised
- **Mitigation**: Fixed relative path, file integrity monitoring recommended

### File Serving from User Input
- **Location**: `server/pages.py` `serve_doc()`
- **Operation**: Serves files from DOCSDIR based on URL parameters
- **Risk**: Path traversal
- **Mitigation**: Use `send_from_directory` (built-in sandboxing), add explicit path validation

### Cryptographic Key Material
- **Location**: `steve/election.py`, `steve/crypto.py`
- **Operation**: Salt/key generation, Argon2 KDF, vote encryption/decryption
- **Risk**: Key exposure leads to vote de-anonymization
- **Mitigation**: Keys never returned in public APIs, salts are per-voter-issue
```

## Cross-cutting Architecture Observations

1. **Documentation Maturity Gap**: The project has good inline code documentation (`ARCHITECTURE.md`, `schema.md`, `tally.md`) but lacks formal security architecture documentation. Security decisions are documented in comments and TODOs rather than dedicated security documents.

2. **Development vs. Production Boundary**: Multiple indicators suggest the codebase is in an active development state being deployed (or ready to deploy) with development artifacts still present. The `TODO.md` essentially serves as both a bug tracker and security audit log.

3. **Authentication vs. Authorization**: The project correctly implements authentication (via ASF OAuth/session), but authorization (ownership checks, per-election access control) is consistently marked with placeholder comments (`### check authz`). This creates a Type B gap — the decorator exists but fine-grained authorization is not implemented.

4. **Cryptographic Design is Strong**: The vote encryption/anonymization design (Argon2 KDF, per-voter salts, encrypted ballots, anti-tamper hashing) is well-implemented. However, the deliberate slowness of these operations creates availability concerns that are only partially addressed.

5. **Dependency Management Awareness**: Version constraints show intentional dependency management, but no formal vulnerability tracking or update policy exists.
