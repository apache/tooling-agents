## ASVS-5.2.2: File Extension and Content Validation

### Findings for 5.2.2

#### HIGH
**Finding ID:** ASVS-522-HIGH-001
**Location:** `v3/server/pages.py`, function `serve_doc` (lines 560-574)

**Description:** The `serve_doc` endpoint serves files from the filesystem without any validation of file extension or content type. The `docname` parameter is passed directly to `send_from_directory` with no check that:
1. The file extension matches an allowlist of permitted types
2. The file content (magic bytes) matches the extension
3. The file is safe for the end-user to download

The comment `### verify the propriety of DOCNAME.` explicitly acknowledges this gap.

**Vulnerable code quote:**
```python
@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require  # fine-grained per-doc authz within the handler
async def serve_doc(iid, docname):
    result = await basic_info()

    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)

    ### verify the propriety of DOCNAME.

    # Return per-issue document.
    return await quart.send_from_directory(DOCSDIR / iid, docname)
```

**Data flow:** User request with arbitrary `docname` → authorization check (mayvote) → file served from filesystem with no content validation → potential serving of malicious content

**Proof of concept:** If a malicious file (e.g., `malware.html` containing JavaScript, or `exploit.svg` with embedded scripts) is placed in `DOCSDIR/<iid>/`, any authorized user could be served this file. The browser may execute embedded scripts based on the file's MIME type.

**Impact:** Without extension allowlisting and content validation, the application could serve polyglot files, HTML/SVG with embedded scripts (leading to stored XSS), or files masquerading as safe types (e.g., `.pdf` containing HTML).

**Remediation:**
```python
import magic  # python-magic library

ALLOWED_EXTENSIONS = {'.pdf', '.txt', '.png', '.jpg', '.jpeg'}
EXTENSION_MIME_MAP = {
    '.pdf': 'application/pdf',
    '.txt': 'text/plain',
    '.png': 'image/png',
    '.jpg': 'image/jpeg',
    '.jpeg': 'image/jpeg',
}

@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require
async def serve_doc(iid, docname):
    result = await basic_info()
    
    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)

    # Validate extension
    ext = pathlib.Path(docname).suffix.lower()
    if ext not in ALLOWED_EXTENSIONS:
        quart.abort(403, 'File type not permitted')

    # Validate content matches extension
    filepath = DOCSDIR / iid / docname
    if not filepath.is_file():
        quart.abort(404)
    
    detected_mime = magic.from_file(str(filepath), mime=True)
    expected_mime = EXTENSION_MIME_MAP.get(ext)
    if detected_mime != expected_mime:
        _LOGGER.warning(f'MIME mismatch for {docname}: expected {expected_mime}, got {detected_mime}')
        quart.abort(403, 'File content does not match extension')

    response = await quart.send_from_directory(DOCSDIR / iid, docname)
    response.headers['X-Content-Type-Options'] = 'nosniff'
    response.headers['Content-Disposition'] = f'attachment; filename="{docname}"'
    return response
```

#### MEDIUM
**Finding ID:** ASVS-522-MED-001
**Location:** `v3/server/pages.py`, function `rewrite_description` (lines 49-57)

**Description:** The `rewrite_description` function creates clickable links from `doc:filename` patterns in issue descriptions. There is no validation that the referenced filename has a safe extension. This allows issue descriptions to reference any file type, potentially leading users to download malicious files.

**Vulnerable code quote:**
```python
def rewrite_description(issue):
    """Rewrite issue description: wrap in <pre> and convert doc:filename to links."""
    import re
    desc = issue.description
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

**Data flow:** Issue description containing `doc:malicious.exe` → regex match → creates link without validating extension → user may click and download dangerous file

**Impact:** Users could be directed to download executable or otherwise dangerous file types through issue descriptions.

**Remediation:** Validate that the filename in `doc:` references matches an allowed extension list before generating the link.

## Cross-cutting Architecture Observations

1. **Missing Upload Mechanism:** The application has a clear document serving architecture (`DOCSDIR/<iid>/<docname>`, `serve_doc` endpoint, `rewrite_description` link generation) but no visible upload handler in the provided code. This creates a Type B gap — the authorization control for serving documents exists, but without a corresponding upload handler with security controls, the safety of served files cannot be guaranteed.

2. **Acknowledged Technical Debt:** The comment `### verify the propriety of DOCNAME.` in `serve_doc` (line 571) indicates the development team is aware of the gap but has not yet addressed it. This is a conscious risk acceptance without documented mitigation.

3. **Framework-Level Protections:** The application relies on `quart.send_from_directory` for path safety, which is a reasonable baseline but insufficient for ASVS L2/L3 compliance. The framework prevents directory traversal via path manipulation but does not address file content safety, symlinks, or archive bombs.

4. **Schema Has No File Metadata:** The `v3/schema.sql` database schema has no table for tracking uploaded files, their sizes, types, or ownership. This makes it architecturally impossible to enforce per-user quotas, file type policies, or audit file uploads without schema changes.

5. **Defense-in-Depth Absent for Files:** The application has multiple layers for election security (cryptographic salts, anti-tamper keys, encrypted votes) but zero layers for file security. This asymmetry suggests file handling was not part of the initial security design.

## Cross-cutting Recommendations

### Immediate (Critical)
1. **Add file extension allowlist to `serve_doc`** — Implement a whitelist of permitted extensions (`.pdf`, `.txt`, `.png`) and reject requests for unlisted types. This is the minimum viable control for the existing serving endpoint.
2. **Add `X-Content-Type-Options: nosniff` and `Content-Disposition: attachment`** headers to all file serve responses to prevent content sniffing and force download behavior.
3. **Add symlink check** — Before serving any file, verify `not filepath.is_symlink()`.

### Short-term (High)
4. **Implement a documented file upload handler** with:
   - `MAX_CONTENT_LENGTH` configured in Quart
   - Extension validation against an allowlist
   - Magic bytes verification using `python-magic`
   - Maximum file size per upload
5. **Create file handling documentation** that satisfies ASVS 5.1.1 — define permitted types, extensions, sizes, and malicious file handling behavior.
6. **Add a `user_files` tracking table** to the database schema to enable quota enforcement.

### Long-term (Medium)
7. **Implement compressed file validation** — detect archives, check uncompressed size and file count, reject symlinks.
8. **Implement per-user storage quotas** with configurable limits and enforcement at upload time.
9. **Add virus/malware scanning** integration (e.g., ClamAV) for uploaded documents.
10. **Implement image rewriting** for any accepted image files to strip metadata and neutralize polyglot attacks.
