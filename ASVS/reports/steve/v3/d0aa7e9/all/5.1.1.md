## ASVS-5.1.1: File Handling Documentation

### Findings for 5.1.1

#### MEDIUM
**Finding ID:** ASVS-511-MED-001
**Location:** `v3/server/pages.py`, function `serve_doc` (lines 560-574); `v3/docs/schema.md` (full document)

**Description:** The application serves documents from `DOCSDIR/<iid>/` via the `/docs/<iid>/<docname>` endpoint, and issue descriptions can reference documents via `doc:filename` syntax (processed by `rewrite_description`). However, there is no documentation anywhere in the provided codebase that defines:
- Permitted file types for documents associated with issues
- Expected file extensions
- Maximum file size (or maximum unpacked size)
- How the application handles malicious files detected during download/processing

**Vulnerable code quote:**
```python
# Serve supporting documents for an issue
@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require  # fine-grained per-doc authz within the handler
async def serve_doc(iid, docname):
    result = await basic_info()  # get PID  ### grr: called uid

    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        # Nothing fancy. Just pretend the file does not exist
        quart.abort(404)
        # NOTREACHED

    ### verify the propriety of DOCNAME.

    # Return per-issue document.
    return await quart.send_from_directory(DOCSDIR / iid, docname)
```

**Data flow:** Documents placed in `DOCSDIR/<iid>/` → served directly to users via `send_from_directory` → no content-type enforcement, no safe-handling documentation

**Impact:** Without documented policies on accepted file types, extensions, and sizes, developers cannot implement consistent file validation. End users downloading served documents have no assurance that files have been vetted for malware or unsafe content (e.g., polyglot files, malicious macros).

**Remediation:**
Create explicit documentation specifying:
```markdown
## File Upload Policy

### Permitted File Types for Issue Documents
- PDF (.pdf) - application/pdf
- Plain text (.txt) - text/plain
- PNG images (.png) - image/png

### Maximum File Size
- Individual file: 10 MB
- Per-issue total: 50 MB

### Malicious File Handling
- Files are scanned with ClamAV on upload
- Files failing validation are rejected with HTTP 415 and logged
- Served files include `Content-Disposition: attachment` header
- X-Content-Type-Options: nosniff is set on all responses
```

## Cross-cutting Architecture Observations

1. **Missing Upload Mechanism:** The application has a clear document serving architecture (`DOCSDIR/<iid>/<docname>`, `serve_doc` endpoint, `rewrite_description` link generation) but no visible upload handler in the provided code. This creates a Type B gap — the authorization control for serving documents exists, but without a corresponding upload handler with security controls, the safety of served files cannot be guaranteed.

2. **Acknowledged Technical Debt:** The comment `### verify the propriety of DOCNAME.` in `serve_doc` (line 571) indicates the development team is aware of the gap but has not yet addressed it. This is a conscious risk acceptance without documented mitigation.

3. **Framework-Level Protections:** The application relies on `quart.send_from_directory` for path safety, which is a reasonable baseline but insufficient for ASVS L2/L3 compliance. The framework prevents directory traversal via path manipulation but does not address file content safety, symlinks, or archive bombs.

4. **Schema Has No File Metadata:** The `v3/schema.sql` database schema has no table for tracking uploaded files, their sizes, types, or ownership. This makes it architecturally impossible to enforce per-user quotas, file type policies, or audit file uploads without schema changes.

5. **Defense-in-Depth Absent for Files:** The application has multiple layers for election security (cryptographic salts, anti-tamper keys, encrypted votes) but zero layers for file security. This asymmetry suggests file handling was not part of the initial security design.

## Cross-cutting Recommendations

### Immediate (Critical)
1. **Add file extension allowlist to `serve_doc`** — Implement a whitelist of permitted extensions (`.pdf`, `.txt`, `.png`) and reject requests for unlisted types. This is the minimum viable control for the existing serving endpoint.
2. **Add `X-Content-Type-Options: nosniff` and `Content-Disposition: attachment`** headers to all file serve responses to prevent content sniffing and force download behavior.
3. **Add symlink check** — Before serving any file, verify `not filepath.is_symlink()`.

### Short-term (High)
4. **Implement a documented file upload handler** with:
   - `MAX_CONTENT_LENGTH` configured in Quart
   - Extension validation against an allowlist
   - Magic bytes verification using `python-magic`
   - Maximum file size per upload
5. **Create file handling documentation** that satisfies ASVS 5.1.1 — define permitted types, extensions, sizes, and malicious file handling behavior.
6. **Add a `user_files` tracking table** to the database schema to enable quota enforcement.

### Long-term (Medium)
7. **Implement compressed file validation** — detect archives, check uncompressed size and file count, reject symlinks.
8. **Implement per-user storage quotas** with configurable limits and enforcement at upload time.
9. **Add virus/malware scanning** integration (e.g., ClamAV) for uploaded documents.
10. **Implement image rewriting** for any accepted image files to strip metadata and neutralize polyglot attacks.
