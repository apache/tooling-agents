## ASVS-4.1.2: HTTP to HTTPS Redirect Policy

### Findings for 4.1.2

### [HIGH] ASVS-412-HIGH-001
**Finding: No HTTP-to-HTTPS Redirect Policy Defined; Application Accepts Plain HTTP**

- **File:** `v3/server/main.py` and `v3/server/config.yaml.example`
- **Location:** `run_standalone()` function, lines 56-87

**Vulnerable code quote:**
```python
# main.py
kwargs = {}
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
    extra_files.update((kwargs['certfile'], kwargs['keyfile']))

# Spool up the app!
app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

```yaml
# config.yaml.example
server:
    # Specify the .pem files to serve using TLS, or leave these two
    # fields blank for plain HTTP.
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
```

- **Data flow:** 
  1. If `certfile` is blank → server runs on plain HTTP → ALL endpoints (including API like `/do-vote/<eid>`) accessible over unencrypted HTTP
  2. If TLS is configured → server only listens on one port (TLS) → No HTTP listener exists → No redirect needed
  3. No differentiation between user-facing and API endpoints for redirect behavior
- **Proof of concept:** With `certfile` left blank as the config comment suggests is possible:
```
POST /do-vote/abc123 HTTP/1.1
Host: example.com
Content-Type: application/x-www-form-urlencoded

vote-issue1=y&csrf_token=placeholder
```
This vote submission containing sensitive ballot data would be transmitted in cleartext.

- **Impact:** Sensitive voting data, session tokens, and authentication credentials could be intercepted if the application is deployed without TLS or if a proxy misconfiguration allows HTTP access. The comment "leave these two fields blank for plain HTTP" explicitly supports insecure deployment.
- **Gap Type:** Type A — No redirect policy or enforcement mechanism exists.

**Remediation:**
```python
# Add to create_app() or as middleware:
@app.before_request
async def enforce_https():
    # For API/service endpoints, reject HTTP outright (don't redirect)
    if not quart.request.is_secure:
        if quart.request.path.startswith('/do-') or quart.request.path.startswith('/api/'):
            quart.abort(403, 'HTTPS required for API endpoints')
        else:
            # User-facing endpoints: redirect to HTTPS
            url = quart.request.url.replace('http://', 'https://', 1)
            return quart.redirect(url, code=301)
```

### [MEDIUM] ASVS-412-MED-001
**Finding: State-Changing Operations on GET Endpoints Allow Silent Data Leakage**

- **File:** `v3/server/pages.py`
- **Location:** `do_open_endpoint()` line 431, `do_close_endpoint()` line 451

**Vulnerable code quote:**
```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    # ...
    election.open(pdb)
    return quart.redirect(f'/manage/{election.eid}', code=303)

@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    # ...
    election.close()
    return quart.redirect(f'/manage/{election.eid}', code=303)
```

- **Data flow:** If HTTP is accidentally used, a GET request to `/do-open/<eid>` would leak the EID (election identifier) in the URL, in Referer headers, and in proxy/server logs — all without any error since no HTTPS enforcement exists.
- **Impact:** Election identifiers and the action being performed are exposed in URLs. Combined with no HTTPS enforcement, this creates a data leakage risk through intermediary logs and browser history.

**Remediation:** These should be POST endpoints, and HTTPS should be enforced for all state-changing operations:
```python
@APP.post('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    # ...
```

## Cross-cutting Architecture Observations

1. **CSRF Protection is Completely Non-Functional:** The `basic_info()` function sets `basic.csrf_token = 'placeholder'` (line 83). This value is rendered in templates and sent in `X-CSRFToken` headers, but **no server-side validation exists**. This is a Type B gap (control exists in UI but is never validated), creating false confidence. All state-changing endpoints are vulnerable to CSRF, compounded by the GET-method issue on open/close.

2. **Authorization Gaps (Pervasive):** Multiple `### check authz` placeholders indicate that fine-grained authorization (e.g., "is this user the owner of this election?") is NOT implemented. Any authenticated committer can manage/open/close any election. This is explicitly acknowledged in code comments but represents a critical security gap for a voting system.

3. **Defense-in-Depth Layers Missing:** The application lacks:
   - Security response headers (CSP, X-Content-Type-Options, X-Frame-Options)
   - Rate limiting
   - Request size limits
   - Audit trail integrity

4. **Deployment Architecture Dependency:** The config comment "a proxy sits in front of this server" means some controls (HSTS, HTTP→HTTPS redirect, proxy header trust) might be implemented at the infrastructure layer. However, the application provides no documentation, verification, or enforcement of these assumptions.

5. **Template vs API Endpoint Confusion:** The application mixes template-rendered pages with JSON API endpoints (`/do-set-open_at`, `/do-set-close_at`) without clear architectural separation. This makes it difficult to apply appropriate security policies (Content-Type, redirect behavior, CORS) per endpoint type.

## Cross-cutting Recommendations

### Immediate (Critical — must fix before production)
1. **Implement CSRF protection:** Replace `'placeholder'` token with a cryptographically random token tied to the session. Validate on all POST endpoints. Consider using `quart-csrf` or similar library.
2. **Change `/do-open` and `/do-close` to POST methods** to prevent CSRF via GET and accidental triggering by prefetchers/crawlers.
3. **Add HSTS header with preload directive** and submit domain to hstspreload.org.
4. **Add charset to all responses:** Fix `raise_404()` and add an `after_request` handler ensuring all `text/*` responses include `; charset=utf-8`.

### Short-term (High — fix within current sprint)
5. **Configure trusted proxy headers:** Add Hypercorn `forwarded_allow_ips` or application-level middleware to prevent header spoofing.
6. **Implement HTTP rejection for API endpoints:** Non-browser endpoints should return 403 on HTTP rather than redirecting.
7. **Add browser feature detection:** Implement client-side checks that warn users if their browser cannot support the voting interface.
8. **Add security headers via `after_request` middleware:**
   ```python
   @app.after_request
   async def security_headers(response):
       response.headers['X-Content-Type-Options'] = 'nosniff'
       response.headers['X-Frame-Options'] = 'DENY'
       response.headers['Strict-Transport-Security'] = 'max-age=63072000; includeSubDomains; preload'
       response.headers['Content-Security-Policy'] = "default-src 'self'"
       return response
   ```

### Long-term (Medium — plan for next release)
9. **Implement fine-grained authorization** (replace all `### check authz` placeholders) — verify election ownership before allowing management operations.
10. **Architectural separation:** Create distinct middleware chains for browser-facing pages vs. JSON API endpoints with different security policies.
11. **Document deployment requirements:** Create a security deployment guide specifying required proxy configuration, TLS requirements, and header trust settings.
12. **Add integration tests** that verify security headers, method restrictions, and CSRF protection are functioning correctly in the deployed configuration.
