## ASVS-6.2.2: Users Can Change Their Password

### Findings for 6.2.2

No password change functionality exists in the application. The `/profile` and `/settings` endpoints are present but only render templates with basic user information:

```python
@APP.get('/profile')
@asfquart.auth.require  # Bare decorator means just require a valid session
@APP.use_template(TEMPLATES / 'profile.ezt')
async def profile_page():
    result = await basic_info()
    result.title = 'Profile'
    return result

@APP.get('/settings')
@asfquart.auth.require  # Bare decorator means just require a valid session
@APP.use_template(TEMPLATES / 'settings.ezt')
async def settings_page():
    result = await basic_info()
    result.title = 'Settings'
    return result
```

No POST handler for password change exists anywhere in the codebase.

**Determination**: Not applicable. The application does not manage passwords locally. Users would change their password through ASF's identity management system. If the application were to be deployed with local authentication in the future, this would be a critical gap.

## Cross-cutting Architecture Observations

1. **Authentication Architecture**: This application exclusively uses external authentication via `asfquart.auth` (ASF's authentication infrastructure). All ASVS 6.2.x password requirements are implicitly delegated to this external provider. The application has **zero** local password management functionality.

2. **Future Risk**: The codebase contains explicit comments (lines 19-22 of `pages.py`) indicating that the current ASF-only authentication model is not intended to be the final solution. When local authentication is added, ALL password security controls (6.2.2 through 6.2.12) will need to be implemented from scratch.

3. **Shared Responsibility Model**: In the current architecture, password security is a shared responsibility:
   - **External auth provider (ASF)**: Responsible for password creation, storage, validation, change, and all ASVS 6.2.x controls
   - **This application**: Responsible only for properly integrating with the external auth system (session management, auth requirement enforcement)

4. **No Credential Storage**: The database schema (`person` table) intentionally stores no credentials, only identity information (`pid`, `name`, `email`). This is a clean separation of concerns.

5. **Session Management**: While passwords aren't managed locally, the application does handle sessions via `asfquart.session`. The session handling appears to properly read authenticated state from the external provider.

## Cross-cutting Recommendations

### Immediate (if deploying with local auth)
- **Implement all ASVS 6.2.x controls**: If the application ever moves away from external-only auth, a complete password management module must be developed including:
  - Password change endpoint requiring current password (6.2.2, 6.2.3)
  - Top 3000+ common password blocklist (6.2.4)
  - No character composition rules (6.2.5)
  - Context-specific word list (election terms, application name, etc.) (6.2.11)
  - Integration with HIBP or similar breached password database (6.2.12)

### Short-term
- **Document authentication delegation**: Create explicit documentation stating which ASVS controls are satisfied by the external authentication provider (ASF's system), and obtain assurance that the provider meets Level 2 requirements.
- **Verify external provider compliance**: Confirm that ASF's authentication system implements breached password checking, common password rejection, and context-specific word blocking.

### Long-term
- **Design local auth module**: If multi-deployment scenarios are pursued (as comments suggest), design the password management module with all ASVS 6.2.x requirements from the start, using libraries like `zxcvbn` for strength estimation and the HIBP k-Anonymity API for breached password checking.
- **Configuration-driven auth**: Implement a pluggable authentication architecture that can switch between external providers and local credential management, with password security controls automatically enabled when local auth is active.
