### [CRITICAL]
**Finding ID: ASVS-221-CRIT-001**
**Title: Vote string (votestring) has no validation against vote type rules**

**File:** `v3/steve/election.py`, line 231-244
**Function:** `Election.add_vote()`

The `add_vote` method accepts a `votestring` parameter and encrypts it without any validation against the expected format for the issue's vote type. The code contains an explicit TODO comment acknowledging this gap:

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting  # <-- NOT IMPLEMENTED

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)

    # Pass MAYVOTE.SALT for PBKDF.
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

**Caller** (`pages.py` line 401-404):
```python
for iid, votestring in votes.items():
    if iid not in issue_dict:
        await flash_danger(f'Invalid issue ID: {iid}')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
    try:
        election.add_vote(result.uid, iid, votestring)
```

**Gap Type: Type A** — Entry point with NO control on `votestring` content.

**Data flow:** HTML form `vote-<iid>` field → `do_vote_endpoint()` → `election.add_vote()` → encrypted and stored without validation

**Proof of concept:**
```http
POST /do-vote/1a2b3c4d5e HTTP/1.1
Content-Type: application/x-www-form-urlencoded

vote-a1b2c3d4e5=INVALID_GARBAGE_STRING_NOT_YES_NO_ABSTAIN
```

For a YNA issue, arbitrary strings like "maybe", "1234", or empty strings would be accepted and encrypted. For STV, invalid rankings or non-existent candidate labels would be stored.

**Impact:** 
- Invalid votes corrupt election results during tallying
- The tally function (`vtypes` module) may crash or produce incorrect results when processing invalid vote strings
- Undermines election integrity — the core purpose of the application

**Remediation:**
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    # Validate the votestring against the issue's vote type
    issue = self.q_get_issue.first_row(iid)
    if not issue:
        raise IssueNotFound(iid)
    
    m = vtypes.vtype_module(issue.type)
    if not m.validate_vote(votestring, self.json2kv(issue.kv)):
        raise InvalidVoteString(votestring, issue.type)

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

### [HIGH]
**Finding ID: ASVS-221-HIGH-001**
**Title: Election title and issue title/description have no server-side length or content validation**

**File:** `v3/server/pages.py`, lines 410-425 (do_create_endpoint), 469-490 (do_add_issue_endpoint), 493-515 (do_edit_issue_endpoint)
**Functions:** `do_create_endpoint()`, `do_add_issue_endpoint()`, `do_edit_issue_endpoint()`

User-supplied text fields are passed directly to the database without any validation:

```python
# do_create_endpoint (line 420)
form = edict(await quart.request.form)
election = steve.election.Election.create(DB_FNAME, form.title, result.uid)

# do_add_issue_endpoint (line 483)
form = edict(await quart.request.form)
iid = election.add_issue(form.title, form.description, vtype, kv)

# do_edit_issue_endpoint (line 505)
form = edict(await quart.request.form)
election.edit_issue(issue.iid, form.title, form.description, issue.vtype, issue.kv)
```

**Gap Type: Type A** — Entry points with NO validation on text content.

**Data flow:** Form field → `await quart.request.form` → EasyDict → `Election.create()` / `add_issue()` / `edit_issue()` → database INSERT/UPDATE

**Proof of concept:**
```http
POST /do-create-election HTTP/1.1
Content-Type: application/x-www-form-urlencoded

title=AAAA....(100000 characters)....AAAA
```

Or:
```http
POST /do-add-issue/1a2b3c4d5e HTTP/1.1
Content-Type: application/x-www-form-urlencoded

title=&description=
```
(Empty title — the database column is NOT NULL but allows empty strings)

**Impact:**
- Extremely long titles could cause display issues, memory exhaustion, or denial of service
- Empty titles violate logical expectations (NOT NULL ≠ non-empty)
- No character set validation allows control characters or other problematic content
- Affects the `gather_election_data()` anti-tamper hash — maliciously crafted titles could cause issues

**Remediation:**
```python
MAX_TITLE_LENGTH = 200
MAX_DESCRIPTION_LENGTH = 10000

async def do_create_endpoint():
    form = edict(await quart.request.form)
    
    title = form.get('title', '').strip()
    if not title:
        await flash_danger('Election title is required.')
        return quart.redirect('/admin', code=303)
    if len(title) > MAX_TITLE_LENGTH:
        await flash_danger(f'Title must be {MAX_TITLE_LENGTH} characters or less.')
        return quart.redirect('/admin', code=303)
    
    election = steve.election.Election.create(DB_FNAME, title, result.uid)
```

### [HIGH]
**Finding ID: ASVS-221-HIGH-002**
**Title: Document filename (docname) relies solely on framework protection without explicit validation**

**File:** `v3/server/pages.py`, lines 560-574
**Function:** `serve_doc()`

The endpoint serves files based on user-supplied `iid` and `docname` parameters. While `send_from_directory` provides framework-level path traversal protection, the code explicitly acknowledges missing validation:

```python
@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require
async def serve_doc(iid, docname):
    result = await basic_info()
    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)

    ### verify the propriety of DOCNAME.  # <-- NOT IMPLEMENTED

    return await quart.send_from_directory(DOCSDIR / iid, docname)
```

**Gap Type: Type A** — Entry point where explicit validation is planned but not implemented.

**Data flow:** URL path parameter `docname` → `serve_doc()` → `send_from_directory(DOCSDIR / iid, docname)`

**Proof of concept:**
```http
GET /docs/a1b2c3d4e5/../../etc/passwd HTTP/1.1
```
Note: Quart's `send_from_directory` uses Werkzeug's `safe_join` which should block `../`, but the `iid` parameter constructs a directory path (`DOCSDIR / iid`) without validating that `iid` is actually a valid 10-char hex string (it's only checked against the mayvote table which may return empty results for crafted values).

Additionally, `docname` could contain special characters, encoded sequences, or be a symlink target if the filesystem allows.

**Impact:** Depending on framework version and edge cases, unauthorized file access outside the intended document directory. The explicit TODO comment indicates the developers recognized this risk.

**Remediation:**
```python
import re

VALID_DOCNAME = re.compile(r'^[a-zA-Z0-9_\-][a-zA-Z0-9_\-\.]{0,254}$')

@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require
async def serve_doc(iid, docname):
    result = await basic_info()
    
    # Validate docname format
    if not VALID_DOCNAME.match(docname):
        quart.abort(400)
    
    # Reject hidden files and directory traversal attempts
    if docname.startswith('.') or '..' in docname:
        quart.abort(400)
    
    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)

    return await quart.send_from_directory(DOCSDIR / iid, docname)
```

### [MEDIUM]
**Finding ID: ASVS-221-MED-001**
**Title: Date validation checks format but not logical business constraints**

**File:** `v3/server/pages.py`, lines 88-110
**Function:** `_set_election_date()`

The date validation only checks ISO format parsing, not business logic validity:

```python
async def _set_election_date(election, field):
    data = await quart.request.get_json()
    date_str = data.get('date')
    if not date_str:
        quart.abort(400, 'Missing date')

    try:
        dt = datetime.datetime.fromisoformat(date_str).date()
    except ValueError:
        quart.abort(400, 'Invalid date format')

    # No validation:
    # - Is date in the future? 
    # - Is close_at after open_at?
    # - Is the date within a reasonable range?
    if field == 'open_at':
        election.set_open_at(dt)
    elif field == 'close_at':
        election.set_close_at(dt)
```

**Data flow:** JSON body `{ "date": "..." }` → `fromisoformat()` → `.date()` → `set_open_at()` / `set_close_at()` → SQL UPDATE

**Proof of concept:**
```http
POST /do-set-close_at/1a2b3c4d5e HTTP/1.1
Content-Type: application/json

{"date": "1900-01-01"}
```

**Impact:** Elections can be set with nonsensical dates (past dates, close before open, far future), confusing administrators and potentially breaking UI display logic.

**Remediation:**
```python
dt = datetime.datetime.fromisoformat(date_str).date()
today = datetime.date.today()

if dt < today:
    quart.abort(400, 'Date cannot be in the past')

if field == 'close_at':
    md = election.get_metadata()
    if md.open_at:
        open_date = datetime.date.fromtimestamp(md.open_at)
        if dt <= open_date:
            quart.abort(400, 'Close date must be after open date')
```

### Positive Patterns for 2.2.1

- **Parameterized queries throughout:** All SQL queries in `queries.yaml` use `?` placeholders, preventing SQL injection regardless of input validation.
- **Election existence verified on instantiation:** `Election.__init__()` immediately checks `q_check_election` and raises `ElectionNotFound` if the EID doesn't exist.
- **Vote type enumeration enforced:** `assert vtype in vtypes.TYPES` prevents creation of issues with invalid vote types.
- **IID validated against election context:** `do_vote_endpoint()` verifies each voted IID exists in the election's issue set.
- **Database CHECK constraints:** Schema-level enforcement of format for IDs, salts, and keys provides defense-in-depth.
- **STV issue validation in CLI:** `create-election.py` validates required kv fields (version, labelmap, seats) and seat count for STV issues.

## Cross-cutting Architecture Observations

1. **Documentation vs. Implementation Gap:** The application has solid database-level documentation (`schema.md`) describing data types and constraints, but lacks a separate input validation specification document. The database schema enforces internal ID formats but not user-input content rules.

2. **Defense-in-Depth Pattern:** The architecture employs multiple layers — database CHECK constraints, foreign keys, application-level state assertions, and authentication decorators. However, the input validation layer (between user input and business logic) is largely absent.

3. **Type A Gaps Dominate:** Most vulnerabilities are "entry point with no control" rather than controls that exist but aren't applied. This suggests input validation was deferred during development (evidenced by multiple TODO/`###` comments).

4. **Asymmetric Validation:** The CLI tool (`create-election.py`) has more validation (STV kv fields, seat count, PID existence) than the web endpoints for equivalent operations. This creates inconsistent security posture between entry points.

5. **State Machine is Well-Implemented:** The election lifecycle (editable → open → closed) is consistently enforced via assertions in `election.py`. This is a strong positive architectural pattern for business logic.

6. **Cryptographic Layer is Solid:** Vote encryption, token generation, and anti-tamper detection are well-structured and correctly separate concerns from the input validation layer.

7. **GET for State Changes:** Using GET requests for irreversible operations (`do-open`, `do-close`) violates REST conventions and makes CSRF prevention more difficult.

## Cross-cutting Recommendations

### Immediate (Critical/High Priority)

1. **Implement votestring validation** in `Election.add_vote()` — this is the core function of the application and currently accepts arbitrary input. Create `validate_vote()` methods in each vtype module.

2. **Convert state-changing GET endpoints to POST** (`/do-open/<eid>`, `/do-close/<eid>`) and implement CSRF token generation and validation. Replace the placeholder token.

3. **Add server-side required field validation** for all form-submitted text fields (title, description) in `do_create_endpoint`, `do_add_issue_endpoint`, and `do_edit_issue_endpoint`.

### Short-term (Medium Priority)

4. **Add cross-field date validation** ensuring `close_at > open_at` when both are set, and consider restricting past dates.

5. **Implement explicit mayvote NULL check** in `add_vote()` to provide clear error handling rather than relying on AttributeError.

6. **Add length limits** on text input fields (title: 200 chars, description: 10,000 chars) both in validation code and as database CHECK constraints.

7. **Create an input validation specification document** covering all user-input fields, their formats, constraints, and cross-field rules.

8. **Validate `docname`** explicitly with an allowlist pattern, even though `send_from_directory` provides framework protection.

### Long-term (Architectural)

9. **Implement authorization (authz) checks** — currently all `### check authz` comments indicate no ownership/permission validation beyond basic authentication.

10. **Create a centralized input validation module** that can be reused across endpoints, rather than inline validation in each handler.

11. **Add per-user and global rate limits** for election creation, issue management, and vote submission to prevent abuse.

12. **Document all business logic rules** in a dedicated specification, including state transitions, temporal constraints, limits, and cross-entity consistency rules.
