## ASVS-15.4.2: TOCTOU Race Conditions

### Findings for 15.4.2

### [HIGH]
**Finding ID:** ASVS-1542-HIGH-001

**File:** `v3/steve/election.py`, method `open`, lines 70-79

**Vulnerable code:**
```python
def open(self, pdb):
    # Double-check the Election is in the editing state.
    assert self.is_editable()

    # Add salts first.
    self.add_salts()

    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)
    self.c_open.perform(salt, opened_key, self.eid)
```

**Data flow:** `is_editable()` check (reads DB) → time gap → `add_salts()` (writes DB) → `c_open.perform()` (writes DB)

**Proof of concept:** Two administrators simultaneously hit `GET /do-open/<eid>`:
1. Request A: `is_editable()` → True (state is editable)
2. Request B: `is_editable()` → True (state still editable, Request A hasn't committed yet)
3. Request A: `add_salts()` → generates salts, commits transaction
4. Request A: `c_open.perform()` → sets opened_key, state becomes OPEN
5. Request B: `add_salts()` → OVERWRITES salts with new values (election is now "open" but add_salts only asserts editable)
6. Request B: `c_open.perform()` → OVERWRITES opened_key

**Impact:** The election's cryptographic integrity is compromised. Salts are overwritten, making the original `opened_key` invalid relative to the new salts. The `is_tampered()` check would subsequently fail, and votes cast between steps 4 and 5 would become undecryptable.

**Remediation:**
```python
def open(self, pdb):
    # Use a single transaction with state check inside
    self.db.conn.execute('BEGIN IMMEDIATE')  # Acquire write lock immediately
    try:
        # Re-check state within the transaction
        md = self._all_metadata()
        if self._compute_state(md) != self.S_EDITABLE:
            self.db.conn.execute('ROLLBACK')
            raise ElectionBadState(self.eid, self._compute_state(md), self.S_EDITABLE)
        
        # All operations within the same transaction
        self._add_salts_internal()  # No separate transaction
        edata = self.gather_election_data(pdb)
        salt = crypto.gen_salt()
        opened_key = crypto.gen_opened_key(edata, salt)
        self.c_open.perform(salt, opened_key, self.eid)
        
        self.db.conn.execute('COMMIT')
    except:
        self.db.conn.execute('ROLLBACK')
        raise
```

### [HIGH]
**Finding ID:** ASVS-1542-HIGH-002

**File:** `v3/steve/election.py`, method `close`, lines 109-114

**Vulnerable code:**
```python
def close(self):
    "Close an election."
    assert self.is_open()
    self.c_close.perform(self.eid)
```

**Data flow:** `is_open()` reads state from DB → time gap → `c_close.perform()` writes to DB

**Proof of concept:** 
1. Admin hits `GET /do-close/<eid>` 
2. `is_open()` returns True
3. Concurrently, the election is already being closed by another admin
4. `c_close.perform()` executes — if the query is `UPDATE metadata SET closed=1 WHERE eid=?`, it succeeds silently even if already closed

**Impact:** While double-closing is relatively benign, the pattern also applies to the reverse: if someone manages to reopen a closed election (via direct DB access or a future API), votes submitted during the race window could be lost or the tamper check could be invalidated.

**Remediation:**
```python
def close(self):
    "Close an election atomically."
    # Use conditional UPDATE that only succeeds if state matches
    self.c_close_if_open.perform(self.eid)  # UPDATE ... WHERE eid=? AND closed IS NULL AND salt IS NOT NULL
    if self.c_close_if_open.rowcount == 0:
        raise ElectionBadState(self.eid, self.get_state(), self.S_OPEN)
```

### [MEDIUM]
**Finding ID:** ASVS-1542-MED-001

**File:** `v3/steve/election.py`, method `add_vote`, lines 231-244

**Vulnerable code:**
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

**Data flow:** `_all_metadata(S_OPEN)` checks state → time gap → `c_add_vote.perform()` inserts vote

**Proof of concept:** An admin closes the election between the state check and the vote insertion. The vote would be recorded for a closed election. While the `_all_metadata(self.S_OPEN)` check prevents obviously-closed elections from accepting votes, a race between closing and a vote submission could allow a vote to be recorded after the close timestamp.

**Impact:** A vote could be recorded after an election is closed, potentially affecting tally results if the timing is tight enough.

**Remediation:** Use a conditional INSERT or wrap in a transaction that re-verifies state:
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    self.db.conn.execute('BEGIN IMMEDIATE')
    try:
        md = self._all_metadata(self.S_OPEN)
        mayvote = self.q_get_mayvote.first_row(pid, iid)
        vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
        ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
        self.c_add_vote.perform(vote_token, ciphertext)
        self.db.conn.execute('COMMIT')
    except:
        self.db.conn.execute('ROLLBACK')
        raise
```

### Positive Patterns for 15.4.2

1. **Atomic ID generation**: `create()` and `add_issue()` use a retry loop with `sqlite3.IntegrityError`, letting the database enforce uniqueness atomically rather than checking existence then inserting.
2. **State validation in _all_metadata**: The `required_state` parameter in `_all_metadata()` provides a consistent check mechanism, even though it's not atomic with subsequent operations.
3. **Anti-tamper detection**: `is_tampered()` provides a post-hoc integrity check that can detect if data was modified by a race condition after the fact.

## Cross-cutting Architecture Observations

1. **Python + Async + SQLite Architecture**: The combination of Quart (async framework) with SQLite (file-based database) creates an inherent tension. SQLite's file-level locking doesn't align well with concurrent async request handling. While the single-threaded nature of asyncio mitigates threading issues, the lack of connection pooling and proper busy-timeout handling means production deployments could experience contention.

2. **Assertion-based validation**: The codebase extensively uses `assert` statements for state validation (`assert self.is_editable()`). In production Python (`-O` flag), assertions are stripped entirely, removing all state guards. This is a systemic architectural risk that spans multiple ASVS requirements.

3. **EasyDict usage throughout**: The pervasive use of `EasyDict` means type information is lost at runtime. While convenient, it makes it impossible to enforce schemas on data structures, creating a gap for type safety (15.3.5) and mass assignment concerns.

4. **Separation of concerns**: The `Election` class handles both data access and business logic. The web layer (`pages.py`) directly instantiates `Election` objects rather than going through a service layer, which makes it harder to apply cross-cutting concerns like atomic transactions spanning business operations.

5. **Authorization gap pattern**: The `### check authz` placeholders throughout `pages.py` indicate a systematic Type B gap — the authentication framework exists and is applied, but fine-grained authorization (ownership verification) is consistently missing.

## Cross-cutting Recommendations

### Immediate (Critical/High Priority)

1. **Replace assertions with proper validation**: Convert all `assert self.is_editable()` patterns to explicit checks that raise proper exceptions and cannot be disabled with `-O`:
   ```python
   if not self.is_editable():
       raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
   ```

2. **Add transaction error handling**: Wrap all `BEGIN TRANSACTION`/`COMMIT` blocks in try/except with explicit `ROLLBACK` on failure to prevent database lock leaks.

3. **Make election state transitions atomic**: Combine state checks and state changes into single transactions using `BEGIN IMMEDIATE` to prevent TOCTOU races on open/close operations.

### Short-term (Medium Priority)

4. **Configure SQLite busy timeout**: Add `PRAGMA busy_timeout = 5000` on every new connection to handle concurrent access gracefully.

5. **Add type validation for JSON inputs**: Verify that `get_json()` returns a dict and that expected fields have correct types before processing.

6. **Implement duplicate parameter detection**: Check for multi-valued form parameters in vote submission and other form-processing endpoints.

7. **Unify transaction patterns**: Adopt a consistent approach (e.g., context manager) for all database write operations.

### Long-term (Architectural)

8. **Consider database migration**: For production use with concurrent access, consider migrating from SQLite to PostgreSQL, which has proper row-level locking and better concurrency support.

9. **Add application-level write serialization**: Implement an asyncio Lock for state-changing operations on the same election to prevent concurrent modification.

10. **Implement a service layer**: Separate business logic from data access to centralize transaction management and make it easier to apply atomic operation patterns consistently.
