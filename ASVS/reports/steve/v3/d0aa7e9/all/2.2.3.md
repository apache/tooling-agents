## ASVS-2.2.3: Combinations of Related Data Items

### Findings for 2.2.3

### [HIGH]
**Finding ID: ASVS-223-HIGH-001**
**Title: No validation that close_at is after open_at when setting dates**

**File:** `v3/server/pages.py`, lines 88-110
**Function:** `_set_election_date()`

When setting `open_at` or `close_at` dates, the application does not validate the logical relationship between them:

```python
async def _set_election_date(election, field):
    data = await quart.request.get_json()
    date_str = data.get('date')
    # ... format validation ...
    dt = datetime.datetime.fromisoformat(date_str).date()
    
    if field == 'open_at':
        election.set_open_at(dt)  # No check against existing close_at
    elif field == 'close_at':
        election.set_close_at(dt)  # No check against existing open_at
```

In `election.py`:
```python
def set_open_at(self, timestamp):
    "Set the open_at timestamp for this Election."
    self.c_set_open_at.perform(timestamp, self.eid)

def set_close_at(self, timestamp):
    "Set the close_at timestamp for this Election."
    self.c_set_close_at.perform(timestamp, self.eid)
```

**Gap Type: Type A** — No cross-field consistency validation exists.

**Data flow:** 
1. User sets `close_at` to Jan 1, 2024
2. User sets `open_at` to Feb 1, 2024
3. Now `close_at < open_at` — logically inconsistent

**Proof of concept:**
```http
POST /do-set-open_at/1a2b3c4d5e HTTP/1.1
Content-Type: application/json

{"date": "2025-12-31"}
```
Then:
```http
POST /do-set-close_at/1a2b3c4d5e HTTP/1.1
Content-Type: application/json

{"date": "2025-01-01"}
```

**Impact:** Illogical date combinations can confuse voters, break UI display logic (showing negative time remaining), and undermine the election schedule's credibility.

**Remediation:**
```python
async def _set_election_date(election, field):
    data = await quart.request.get_json()
    date_str = data.get('date')
    if not date_str:
        quart.abort(400, 'Missing date')

    try:
        dt = datetime.datetime.fromisoformat(date_str).date()
    except ValueError:
        quart.abort(400, 'Invalid date format')

    # Validate cross-field consistency
    md = election.get_metadata()
    if field == 'open_at' and md.close_at:
        close_date = datetime.date.fromtimestamp(md.close_at)
        if dt >= close_date:
            quart.abort(400, 'Open date must be before close date')
    elif field == 'close_at' and md.open_at:
        open_date = datetime.date.fromtimestamp(md.open_at)
        if dt <= open_date:
            quart.abort(400, 'Close date must be after open date')

    if field == 'open_at':
        election.set_open_at(int(datetime.datetime.combine(dt, datetime.time()).timestamp()))
    elif field == 'close_at':
        election.set_close_at(int(datetime.datetime.combine(dt, datetime.time()).timestamp()))
```

### [MEDIUM]
**Finding ID: ASVS-223-MED-001**
**Title: do_vote_endpoint does not verify voter eligibility for each specific issue**

**File:** `v3/server/pages.py`, lines 372-407
**Function:** `do_vote_endpoint()`

The endpoint validates that the IID exists in the election's issue list, but does not verify that the authenticated user (PID) is eligible to vote on that *specific* issue (has a mayvote entry for it) before calling `add_vote`:

```python
async def do_vote_endpoint(election):
    result = await basic_info()
    form = edict(await quart.request.form)
    
    votes = {}
    for key, value in form.items():
        if key.startswith('vote-'):
            iid = key.split('-', 1)[1]
            votes[iid] = value

    issues = election.list_issues()
    issue_dict = {i.iid: i for i in issues}

    for iid, votestring in votes.items():
        if iid not in issue_dict:  # Only checks issue exists in election
            await flash_danger(f'Invalid issue ID: {iid}')
            return quart.redirect(f'/vote-on/{election.eid}', code=303)

        try:
            election.add_vote(result.uid, iid, votestring)  # Defers to add_vote
```

Note: The `add_vote` method in `election.py` *does* check mayvote:
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    # If mayvote is None, accessing mayvote.salt will raise AttributeError
```

This is partially mitigated — if the user has no mayvote entry, `mayvote` will be `None` and `mayvote.salt` will raise an `AttributeError`, caught by the generic `except Exception` handler. However, this is accidental error handling rather than explicit validation.

**Gap Type: Type C (partial)** — The control indirectly exists (mayvote lookup) but the result isn't explicitly checked — failure manifests as an unhandled attribute error rather than a clear authorization denial.

**Impact:** While the operation won't succeed (it will error), the error handling is generic and doesn't distinguish between "not authorized" and "system error," potentially masking authorization failures in logs.

**Remediation:**
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    if mayvote is None:
        raise VoterNotEligible(pid, iid)
    
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    # ...
```

### [MEDIUM]
**Finding ID: ASVS-223-MED-002**
**Title: No validation that STV vote rankings reference valid candidates from issue's labelmap**

**File:** `v3/steve/election.py`, line 231-244
**Function:** `Election.add_vote()` (same root cause as ASVS-221-CRIT-001, different facet)

For STV issues, the `votestring` should contain a ranking of candidates whose labels exist in the issue's `kv.labelmap`. This combined data consistency (vote references valid candidates from the issue's metadata) is not validated:

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    # No check that votestring candidates ∈ issue.kv.labelmap
    # No check that votestring ranking count ≤ len(issue.kv.labelmap)
    # No check for duplicate candidates in ranking
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

**Data flow:** Form data with STV ranking → `add_vote()` → stored without validating candidate labels match issue's labelmap

**Proof of concept:** For an STV issue with labelmap `{"a": "Alice", "b": "Bob", "c": "Carlos"}`:
```http
POST /do-vote/1a2b3c4d5e HTTP/1.1
Content-Type: application/x-www-form-urlencoded

vote-a1b2c3d4e5=z,x,y
```
Submits rankings for candidates "z", "x", "y" which don't exist in the labelmap.

**Impact:** Invalid candidate references in votes would corrupt STV tallying results or cause tally errors.

**Remediation:** Validate that each ranked candidate label exists in the issue's `kv.labelmap` and there are no duplicates.

### Positive Patterns for 2.2.3

- **Anti-tamper mechanism:** `gather_election_data()` hashes the combination of election metadata, all issues, and all voter PIDs/emails together. Any modification to these combined values after opening is detectable via `is_tampered()`.
- **Mayvote eligibility tied to specific issue:** The `add_vote` method queries the specific (pid, iid) pair from the mayvote table, ensuring the voter is eligible for that particular issue.
- **Foreign key cascade prevention:** `ON DELETE RESTRICT` on foreign keys prevents deletion of persons who are referenced in elections/mayvote, maintaining referential consistency.
- **Election lifecycle state machine:** The code enforces that operations (add issue, vote, tally) can only occur in the appropriate state, preventing logically inconsistent combinations of operations.
- **Trigger-based consistency:** The `prevent_open_close_update` trigger ensures that once an election is closed, its dates cannot be modified — enforcing temporal consistency at the database level.

## Cross-cutting Architecture Observations

1. **Documentation vs. Implementation Gap:** The application has solid database-level documentation (`schema.md`) describing data types and constraints, but lacks a separate input validation specification document. The database schema enforces internal ID formats but not user-input content rules.

2. **Defense-in-Depth Pattern:** The architecture employs multiple layers — database CHECK constraints, foreign keys, application-level state assertions, and authentication decorators. However, the input validation layer (between user input and business logic) is largely absent.

3. **Type A Gaps Dominate:** Most vulnerabilities are "entry point with no control" rather than controls that exist but aren't applied. This suggests input validation was deferred during development (evidenced by multiple TODO/`###` comments).

4. **Asymmetric Validation:** The CLI tool (`create-election.py`) has more validation (STV kv fields, seat count, PID existence) than the web endpoints for equivalent operations. This creates inconsistent security posture between entry points.

5. **State Machine is Well-Implemented:** The election lifecycle (editable → open → closed) is consistently enforced via assertions in `election.py`. This is a strong positive architectural pattern for business logic.

6. **Cryptographic Layer is Solid:** Vote encryption, token generation, and anti-tamper detection are well-structured and correctly separate concerns from the input validation layer.

7. **GET for State Changes:** Using GET requests for irreversible operations (`do-open`, `do-close`) violates REST conventions and makes CSRF prevention more difficult.

## Cross-cutting Recommendations

### Immediate (Critical/High Priority)

1. **Implement votestring validation** in `Election.add_vote()` — this is the core function of the application and currently accepts arbitrary input. Create `validate_vote()` methods in each vtype module.

2. **Convert state-changing GET endpoints to POST** (`/do-open/<eid>`, `/do-close/<eid>`) and implement CSRF token generation and validation. Replace the placeholder token.

3. **Add server-side required field validation** for all form-submitted text fields (title, description) in `do_create_endpoint`, `do_add_issue_endpoint`, and `do_edit_issue_endpoint`.

### Short-term (Medium Priority)

4. **Add cross-field date validation** ensuring `close_at > open_at` when both are set, and consider restricting past dates.

5. **Implement explicit mayvote NULL check** in `add_vote()` to provide clear error handling rather than relying on AttributeError.

6. **Add length limits** on text input fields (title: 200 chars, description: 10,000 chars) both in validation code and as database CHECK constraints.

7. **Create an input validation specification document** covering all user-input fields, their formats, constraints, and cross-field rules.

8. **Validate `docname`** explicitly with an allowlist pattern, even though `send_from_directory` provides framework protection.

### Long-term (Architectural)

9. **Implement authorization (authz) checks** — currently all `### check authz` comments indicate no ownership/permission validation beyond basic authentication.

10. **Create a centralized input validation module** that can be reused across endpoints, rather than inline validation in each handler.

11. **Add per-user and global rate limits** for election creation, issue management, and vote submission to prevent abuse.

12. **Document all business logic rules** in a dedicated specification, including state transitions, temporal constraints, limits, and cross-entity consistency rules.
