## ASVS-16.4.2: Log Protection

### Findings for 16.4.2

### [MEDIUM]
**Finding ID**: ASVS-1642-MED-001

**File**: `v3/server/pages.py` (lines 28, throughout)  
**Function**: Application-wide logging configuration

**Vulnerable code quote**:
```python
_LOGGER = logging.getLogger(__name__)
```

**Description**: The application uses Python's standard `logging` module but does not configure any log protection mechanisms. There is no evidence of:
- Log file permissions being set (e.g., append-only, read-restricted)
- Log integrity verification (e.g., hashing, chaining, signing)
- Separation of log storage from application write access
- Protection against log truncation or deletion by the application process

The application process that writes logs has implicit read/write access to any local log files, meaning if the application is compromised, the attacker can modify or delete logs.

**Data flow**: Application events → `_LOGGER.info/debug/error` → Python logging → (no configured protection)

**Proof of concept**: If an attacker gains code execution via the application (e.g., through a template injection or path traversal vulnerability), they would have the same filesystem permissions as the application process and could modify/delete any log files written to the local filesystem.

**Impact**: Forensic evidence could be altered or destroyed post-breach, making incident investigation unreliable.

**Remediation**:
```python
import logging.handlers

# Configure immutable logging - write to a separate process/system
# Option 1: Syslog (remote or local socket)
syslog_handler = logging.handlers.SysLogHandler(
    address='/dev/log',  # or ('remote-syslog-host', 514)
    facility=logging.handlers.SysLogHandler.LOG_LOCAL0
)
_LOGGER.addHandler(syslog_handler)

# Option 2: If local files are used, set restrictive permissions at creation
import os
log_path = '/var/log/steve/app.log'
os.makedirs(os.path.dirname(log_path), exist_ok=True)
handler = logging.FileHandler(log_path)
os.chmod(log_path, 0o640)  # Root or logging group can read; app can write
_LOGGER.addHandler(handler)
```

### [LOW]
**Finding ID**: ASVS-1642-LOW-001

**File**: `v3/server/bin/tally.md` (Security and Concerns section)

**Vulnerable code quote**:
```markdown
- **Logging**: No internal logging is performed, as it could be tampered with. Results are only output to stdout/stderr.
```

**Description**: The tally script intentionally omits logging because of tampering concerns. While the reasoning acknowledges the risk, the solution (no logging at all) means there is no audit trail of who ran tallies, when, or what parameters were used. This is a gap in forensic capability rather than a direct vulnerability.

**Impact**: No audit trail for tally operations, which are the most security-sensitive operations in a voting system.

**Remediation**: Send tally audit events to the same protected remote logging system used by the main application.

## Cross-cutting Architecture Observations

1. **Security control existence vs. application gap (Type B)**: The most significant systemic issue is that security controls are *identified* (via `### check authz` comments and `assert` statements) but either not implemented or implemented using an unreliable mechanism (`assert`). This creates false confidence that security is handled.

2. **Logging infrastructure immaturity**: The application generates good audit events but lacks the infrastructure to protect, transmit, and store those events securely. This is a foundational gap that affects multiple ASVS requirements.

3. **Error handling philosophy inconsistency**: Some endpoints (like `do_vote_endpoint`) have try/except blocks while others (like `do_open_endpoint`, `do_close_endpoint`) have no error handling at all. There's no consistent strategy.

4. **Single-tier architecture**: The application, database, and logs all reside on the same system/process. This means a compromise of any one component grants access to all others, including logs. A multi-tier or defense-in-depth approach is absent.

5. **Admin tooling (`tally.py`) shares the same deficiencies**: The tally script uses `raise` to "fail hard" on errors, but has no structured error handling, no audit logging to a secure system, and outputs results directly to stdout where they could be captured or lost.

6. **Assertion-based validation in a cryptographic voting system**: Using `assert` for state checks in a system designed to guarantee election integrity is a fundamental design flaw. The security model depends on state transitions being enforced, but the enforcement mechanism can be silently disabled.

## Cross-cutting Recommendations

### Immediate (Critical/High - address before deployment)

1. **Replace all `assert` statements with proper exception raising** in `election.py`. Every `assert self.is_editable()`, `assert self.is_open()`, and `assert self.is_closed()` must be converted to `if not ...: raise ElectionBadState(...)`. This is the single highest-impact change.

2. **Implement a global exception handler** (`@APP.errorhandler(Exception)`) that:
   - Logs the full exception to the structured logger
   - Returns a generic error page to the user
   - Prevents stack trace disclosure

3. **Implement authorization checks** at all `### check authz` locations. At minimum, verify the authenticated user owns the election for management operations.

### Short-term (Medium - address within sprint)

4. **Configure log transmission** to a logically separate system (remote syslog, centralized logging service, or at minimum a separate filesystem with restricted permissions).

5. **Add database error handling** with graceful degradation. Wrap database operations in try/except for `sqlite3.OperationalError` and related exceptions.

6. **Remove debug `print()` statements** from production endpoints.

7. **Add log file protection**: If logs are stored locally, configure append-only permissions and ensure the application process cannot delete or modify existing log entries.

### Long-term (Architecture improvements)

8. **Implement a circuit breaker pattern** for database access, with health checks and graceful degradation.

9. **Add log integrity verification**: Implement hash chaining or digital signatures on log entries to detect tampering.

10. **Separate the logging infrastructure**: Deploy a dedicated log aggregation system (e.g., ELK stack, Loki, or cloud-native logging) that is network-isolated from the application.

11. **Add comprehensive integration tests** that verify error handling paths, including simulated database failures and invalid state transitions.
