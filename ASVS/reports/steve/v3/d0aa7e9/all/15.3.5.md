## ASVS-15.3.5: Type Safety and Strict Equality

### Findings for 15.3.5

### [MEDIUM]
**Finding ID:** ASVS-1535-MED-001

**File:** `v3/server/pages.py`, function `_set_election_date`, lines 91-93

**Vulnerable code:**
```python
data = await quart.request.get_json()
date_str = data.get('date')
if not date_str:
    quart.abort(400, 'Missing date')
```

**Data flow:** HTTP request body → `get_json()` → `data` (could be `None`) → `.get('date')` → `AttributeError`

**Proof of concept:** Send a POST request to `/do-set-open_at/<eid>` with `Content-Type: text/plain` or malformed JSON body. `get_json()` returns `None`, and calling `.get('date')` on `None` raises `AttributeError`.

**Impact:** Unhandled exception could expose stack traces or cause inconsistent application state. The code assumes `data` is always a dict without verifying its type.

**Remediation:**
```python
data = await quart.request.get_json()
if not isinstance(data, dict):
    quart.abort(400, 'Invalid request body')
date_str = data.get('date')
if not isinstance(date_str, str) or not date_str:
    quart.abort(400, 'Missing or invalid date')
```

### [MEDIUM]
**Finding ID:** ASVS-1535-MED-002

**File:** `v3/steve/election.py`, method `_compute_state`, lines 351-362

**Vulnerable code:**
```python
@classmethod
def _compute_state(cls, md):
    if md.closed == 1:
        assert md.salt is not None and md.opened_key is not None
        return cls.S_CLOSED
    assert md.closed in (None, 0)
```

**Data flow:** SQLite row data → `md.closed` → comparison with integer `1`

**Proof of concept:** If the database column `closed` is stored or returned as the string `"1"` (e.g., due to a migration issue or manual DB edit), the `== 1` comparison would be `False` in Python, causing the state computation to incorrectly report the election as not closed. The assertion `md.closed in (None, 0)` would then fire for `"1"`.

**Impact:** Election state could be computed incorrectly if the database value type doesn't match expectations. An election thought to be open could actually be closed, or vice-versa, affecting vote integrity.

**Remediation:**
```python
@classmethod
def _compute_state(cls, md):
    closed_val = int(md.closed) if md.closed is not None else None
    if closed_val == 1:
        assert md.salt is not None and md.opened_key is not None
        return cls.S_CLOSED
    assert closed_val in (None, 0)
```

### [LOW]
**Finding ID:** ASVS-1535-LOW-001

**File:** `v3/server/pages.py`, function `do_vote_endpoint`, lines 384-394

**Vulnerable code:**
```python
form = edict(await quart.request.form)
if not form:
    await flash_danger('No vote data provided.')
    return quart.redirect(f'/vote-on/{election.eid}', code=303)

votes = {}
for key, value in form.items():
    if key.startswith('vote-'):
        iid = key.split('-', 1)[1]
        votes[iid] = value
```

**Data flow:** Form data → `form.items()` → `value` (always string) → `votes[iid]` → `election.add_vote(result.uid, iid, votestring)`

**Impact:** No type verification is performed on `votestring` before passing to `add_vote()`. While HTML form values are always strings, a programmatic attacker could submit unexpected types via JSON content-type manipulation. The `add_vote` method then encrypts whatever string is passed without validating it against the issue's expected vote type (as noted by the `### validate VOTESTRING` comment in `add_vote`).

**Remediation:**
```python
if not isinstance(votestring, str):
    await flash_danger(f'Invalid vote format for issue {iid}.')
    return quart.redirect(f'/vote-on/{election.eid}', code=303)
# Also validate against expected vtype format
```

### Positive Patterns for 15.3.5

1. **Vote type assertion**: `assert vtype in vtypes.TYPES` in `add_issue` and `edit_issue` explicitly validates that the vote type is from an allowed set.

3. **State comparison constants**: Using class-level string constants (`S_EDITABLE`, `S_OPEN`, `S_CLOSED`) compared with `==` reduces typo-related bugs.
4. **Explicit None checks**: `if row is None` pattern used consistently (e.g., `not_found()`, `tally_issue`).

## Cross-cutting Architecture Observations

1. **Python + Async + SQLite Architecture**: The combination of Quart (async framework) with SQLite (file-based database) creates an inherent tension. SQLite's file-level locking doesn't align well with concurrent async request handling. While the single-threaded nature of asyncio mitigates threading issues, the lack of connection pooling and proper busy-timeout handling means production deployments could experience contention.

2. **Assertion-based validation**: The codebase extensively uses `assert` statements for state validation (`assert self.is_editable()`). In production Python (`-O` flag), assertions are stripped entirely, removing all state guards. This is a systemic architectural risk that spans multiple ASVS requirements.

3. **EasyDict usage throughout**: The pervasive use of `EasyDict` means type information is lost at runtime. While convenient, it makes it impossible to enforce schemas on data structures, creating a gap for type safety (15.3.5) and mass assignment concerns.

4. **Separation of concerns**: The `Election` class handles both data access and business logic. The web layer (`pages.py`) directly instantiates `Election` objects rather than going through a service layer, which makes it harder to apply cross-cutting concerns like atomic transactions spanning business operations.

5. **Authorization gap pattern**: The `### check authz` placeholders throughout `pages.py` indicate a systematic Type B gap — the authentication framework exists and is applied, but fine-grained authorization (ownership verification) is consistently missing.

## Cross-cutting Recommendations

### Immediate (Critical/High Priority)

1. **Replace assertions with proper validation**: Convert all `assert self.is_editable()` patterns to explicit checks that raise proper exceptions and cannot be disabled with `-O`:
   ```python
   if not self.is_editable():
       raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
   ```

2. **Add transaction error handling**: Wrap all `BEGIN TRANSACTION`/`COMMIT` blocks in try/except with explicit `ROLLBACK` on failure to prevent database lock leaks.

3. **Make election state transitions atomic**: Combine state checks and state changes into single transactions using `BEGIN IMMEDIATE` to prevent TOCTOU races on open/close operations.

### Short-term (Medium Priority)

4. **Configure SQLite busy timeout**: Add `PRAGMA busy_timeout = 5000` on every new connection to handle concurrent access gracefully.

5. **Add type validation for JSON inputs**: Verify that `get_json()` returns a dict and that expected fields have correct types before processing.

6. **Implement duplicate parameter detection**: Check for multi-valued form parameters in vote submission and other form-processing endpoints.

7. **Unify transaction patterns**: Adopt a consistent approach (e.g., context manager) for all database write operations.

### Long-term (Architectural)

8. **Consider database migration**: For production use with concurrent access, consider migrating from SQLite to PostgreSQL, which has proper row-level locking and better concurrency support.

9. **Add application-level write serialization**: Implement an asyncio Lock for state-changing operations on the same election to prevent concurrent modification.

10. **Implement a service layer**: Separate business logic from data access to centralize transaction management and make it easier to apply atomic operation patterns consistently.
