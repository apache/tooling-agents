## ASVS-15.1.4: Secure Coding and Architecture Documentation - Risky Components

### Findings for 15.1.4

### [MEDIUM]

**Finding ID**: ASVS-1514-MED-001
**Location**: `v3/ARCHITECTURE.md`, `v3/pyproject.toml`
**Issue**: No documentation identifies third-party libraries considered "risky components"

**Evidence**:

The `pyproject.toml` declares these dependencies:
```toml
dependencies = [
    "asfpy>=0.56",
    "asfquart>=0.1.12",
    "ezt>=1.1",
    "easydict>=1.13",
    "cryptography>=46.0.5,<47",
    "argon2-cffi>=25.1.0,<26",
]
```

`ARCHITECTURE.md` lists these in the "Dependencies" section:
```markdown
- **Dependencies**: Managed via `uv` (Python package manager); includes Quart, asfpy, etc.
```

**No documentation anywhere** performs a risk assessment or highlights any library as potentially risky. Candidates that should be evaluated include:

1. **`asfquart>=0.1.12`** — Version 0.1.x indicates pre-stable/immature software with potentially incomplete security controls
2. **`easydict>=1.13`** — Simple utility library; maintenance status and security posture undocumented
3. **`ezt>=1.1`** — Legacy templating engine; maintenance status undocumented
4. **`python-ldap~=3.0.0`** (dev dependency) — Has had historical vulnerabilities related to LDAP injection

**Impact**: Without documenting risky components, the development team lacks awareness of which dependencies require closer scrutiny, more frequent updates, or architectural isolation.

**Remediation**: Add a "Dependency Risk Assessment" section to ARCHITECTURE.md:
```markdown
## Dependency Risk Assessment

### Potentially Risky Components
| Library | Risk Factor | Mitigation |
|---------|------------|------------|
| asfquart (0.1.x) | Pre-stable, ASF-internal, limited review | Monitor closely, pin versions |
| ezt | Legacy, minimal maintenance | Consider migration to Jinja2 |
| easydict | Limited maintenance | Evaluate replacement if abandoned |

### Assessment Criteria
Components are flagged if: unmaintained (>12mo no release), pre-stable (0.x),
history of critical CVEs, or limited security review.
```

## Cross-cutting Architecture Observations

1. **Documentation Maturity Gap**: The project has good inline code documentation (`ARCHITECTURE.md`, `schema.md`, `tally.md`) but lacks formal security architecture documentation. Security decisions are documented in comments and TODOs rather than dedicated security documents.

2. **Development vs. Production Boundary**: Multiple indicators suggest the codebase is in an active development state being deployed (or ready to deploy) with development artifacts still present. The `TODO.md` essentially serves as both a bug tracker and security audit log.

3. **Authentication vs. Authorization**: The project correctly implements authentication (via ASF OAuth/session), but authorization (ownership checks, per-election access control) is consistently marked with placeholder comments (`### check authz`). This creates a Type B gap — the decorator exists but fine-grained authorization is not implemented.

4. **Cryptographic Design is Strong**: The vote encryption/anonymization design (Argon2 KDF, per-voter salts, encrypted ballots, anti-tamper hashing) is well-implemented. However, the deliberate slowness of these operations creates availability concerns that are only partially addressed.

5. **Dependency Management Awareness**: Version constraints show intentional dependency management, but no formal vulnerability tracking or update policy exists.
