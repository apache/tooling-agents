## ASVS-3.5.3: Browser Origin Separation - HTTP Methods

### Findings for 3.5.3

### [CRITICAL]

**Finding ID:** ASVS-353-CRIT-001

**Title:** Irreversible State-Changing Operations Use GET Method

**File:** `v3/server/pages.py`, lines 431-465

**Vulnerable Code:**
```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    _LOGGER.info(f'User[U:{result.uid}] opened election[E:{election.eid}]')
    title = election.get_metadata().title
    await flash_success(f'Opened election: {title}')
    return quart.redirect(f'/manage/{election.eid}', code=303)


@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz
    election.close()
    _LOGGER.info(f'User[U:{result.uid}] closed election[E:{election.eid}]')
    title = election.get_metadata().title
    await flash_success(f'Closed election: {title}')
    return quart.redirect(f'/manage/{election.eid}', code=303)
```

**Data Flow:** GET request to `/do-open/<eid>` or `/do-close/<eid>` → `election.open()` / `election.close()` called → Election state permanently changed

**Client-side invocation confirming GET usage (`manage.ezt`):**
```javascript
confirmOpen.addEventListener('click', () => {
    openModal.hide();
    window.location.href = '/do-open/[eid]';
});

confirmClose.addEventListener('click', () => {
    closeModal.hide();
    window.location.href = '/do-close/[eid]';
});
```

**Proof of Concept:**

1. **Image tag attack:**
```html
<img src="https://target.example.com/do-open/ELECTION_EID">
```

2. **Prefetch/link attack:**
```html
<link rel="prefetch" href="https://target.example.com/do-close/ELECTION_EID">
```

3. **Browser history/back button:** If user navigates away and back, the operation could be replayed.

4. **Search engine crawlers or link previews** could trigger these endpoints.

**Impact:** 
- Elections can be opened prematurely, allowing voting before intended
- Elections can be closed prematurely, disenfranchising voters
- Both operations are documented as **irreversible** in the UI text
- Triggers from img tags, prefetch, link previews, or any context that loads URLs
- No Sec-Fetch-* validation as alternative protection

**Remediation:**
```python
@APP.post('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    # Validate CSRF token
    form = await quart.request.form
    validate_csrf(form.get('csrf_token'))
    
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    _LOGGER.info(f'User[U:{result.uid}] opened election[E:{election.eid}]')
    title = election.get_metadata().title
    await flash_success(f'Opened election: {title}')
    return quart.redirect(f'/manage/{election.eid}', code=303)
```

And update the template:
```javascript
confirmOpen.addEventListener('click', () => {
    openModal.hide();
    // Use a form POST instead of navigation
    const form = document.createElement('form');
    form.method = 'POST';
    form.action = '/do-open/[eid]';
    const csrf = document.createElement('input');
    csrf.type = 'hidden';
    csrf.name = 'csrf_token';
    csrf.value = csrfToken;
    form.appendChild(csrf);
    document.body.appendChild(form);
    form.submit();
});
```

## Cross-cutting Architecture Observations

1. **CSRF Protection is Entirely Absent Despite Apparent Intent:** The codebase shows clear architectural intent to implement CSRF protection (token fields in templates, header passing in fetch calls, TODO items acknowledging it), but the actual enforcement is completely missing. This creates a dangerous false sense of security — developers and reviewers may believe protection exists because they see the tokens in templates.

2. **GET-Based State Changes Compound the CSRF Problem:** The `/do-open/<eid>` and `/do-close/<eid>` endpoints using GET method means that even implementing CSRF tokens for POST requests would leave these critical operations unprotected. These must be converted to POST before any CSRF solution is effective.

3. **Authorization Model is Incomplete:** Multiple `### check authz` placeholders indicate that ownership verification (e.g., "does this user own this election?") is not enforced. Any authenticated committer can manage any election. While this is beyond the scope of ASVS 3.5.x, it amplifies the impact of CSRF — an attacker doesn't need to target the election owner specifically.

4. **Template Security is Well-Implemented:** The use of `[format "js,html"]` in EZT templates for user-supplied data injected into JavaScript contexts shows awareness of XSS prevention, which is relevant because XSS can bypass CSRF protections.

5. **Client-Server Contract Mismatch:** The client (templates) sends CSRF tokens and custom headers that the server completely ignores. This indicates the frontend was developed with security in mind but the backend implementation was deferred.

## Cross-cutting Recommendations

### Immediate (Critical — deploy blockers)

1. **Implement CSRF token validation on all POST endpoints.** Generate a cryptographically random token per session, store it server-side, and validate it on every state-changing request. Use a decorator pattern for consistency.

2. **Convert `/do-open/<eid>` and `/do-close/<eid>` from GET to POST.** These perform irreversible operations and must not be triggerable via simple navigation, image loads, or link prefetch.

3. **Add Origin header validation** as defense-in-depth for all POST endpoints, rejecting requests with an Origin header that doesn't match the application's own origin.

### Short-term (High priority — within sprint)

4. **Enforce Content-Type validation** on form endpoints — reject requests that don't match `application/x-www-form-urlencoded` or `multipart/form-data` as appropriate. For JSON endpoints, reject non-`application/json` requests.

5. **Configure SameSite=Lax (minimum) or SameSite=Strict on session cookies.** This provides browser-level CSRF protection for most scenarios while the token-based approach covers older browsers.

6. **Add Sec-Fetch-Site header checking** as an additional defense layer — reject requests where `Sec-Fetch-Site: cross-site` on sensitive endpoints.

### Long-term (Architecture improvements)

7. **Consider hostname-based separation** for admin and voter functionality to leverage Same-Origin Policy restrictions between functional domains.

8. **Implement a comprehensive security middleware stack** that handles CSRF, CORS, and Content-Type validation in one place rather than requiring per-endpoint implementation.

9. **Add integration tests** that verify CSRF protection by attempting state-changing requests without valid tokens and asserting they fail with 403.
