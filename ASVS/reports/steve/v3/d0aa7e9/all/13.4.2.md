## ASVS-13.4.2: Debug Modes Disabled in Production

### Findings for 13.4.2

### [MEDIUM]

**Finding ID**: ASVS-1342-MED-001

**File**: `v3/server/pages.py` (lines ~476, ~497)

**Vulnerable Code**:
```python
@APP.post('/do-add-issue/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_add_issue_endpoint(election):
    ...
    form = edict(await quart.request.form)
    print('FORM:', form)
    ...
```

```python
@APP.post('/do-edit-issue/<eid>/<iid>')
@asfquart.auth.require({R.committer})
@load_election_issue
async def do_edit_issue_endpoint(election, issue):
    ...
    form = edict(await quart.request.form)
    print('FORM:', form)
    ...
```

**Data Flow**: User-submitted form data → `print()` → stdout/process output → potentially captured in server logs, process monitors, or container output.

**Impact**: Debug `print()` statements output user-submitted form data (issue titles, descriptions) to stdout in production. This could expose sensitive election data to log aggregation systems, container runtime logs, or process monitoring tools that capture stdout. This violates the principle of disabling debug features in production.

**Proof of Concept**: Submit a POST to `/do-add-issue/<eid>` with form data. The complete form contents are printed to stdout regardless of environment.

**Remediation**:
```python
# Remove print statements, use proper logging at DEBUG level
@APP.post('/do-add-issue/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_add_issue_endpoint(election):
    ...
    form = edict(await quart.request.form)
    _LOGGER.debug('Form data received for add-issue')  # Don't log values
    ...
```

### [MEDIUM]

**Finding ID**: ASVS-1342-MED-002

**File**: `v3/server/pages.py` (line 83)

**Vulnerable Code**:
```python
async def basic_info():
    ...
    ### generate a real token and store in the session
    basic.csrf_token = 'placeholder'
    return basic
```

**Impact**: The hardcoded placeholder CSRF token is a debugging/development artifact left in production code. While this is primarily a CSRF vulnerability, the presence of a placeholder token with a comment admitting it's not real indicates the application has not been hardened for production. This exposes that security controls are incomplete, which is information an attacker could leverage.

**Remediation**: Generate a proper cryptographic CSRF token:
```python
import secrets
basic.csrf_token = secrets.token_hex(32)
# Store in session for validation
```

## Cross-cutting Architecture Observations

1. **Key Material Exposure Pattern**: The application handles all cryptographic operations in-process with no isolation boundary. The `opened_key` (master election key) flows through multiple code paths: `election.open()` → SQLite storage → `add_vote()` → `tally_issue()` → `has_voted_upon()`. Any single compromise point in this chain exposes all election vote data.

2. **Lifecycle Gap**: The election lifecycle (editable → open → closed) provides a natural framework for key management, but key material persists in the `closed` state indefinitely. There's no "archived" or "destroyed" state where key material would be removed.

3. **Defense-in-Depth Gaps**: Multiple requirements share a common theme of missing explicit hardening. The framework provides reasonable defaults (no TRACE, no directory listing, method restrictions) but there's no explicit defense-in-depth configuration. This creates risk if framework behavior changes or if deployment differs from assumptions.

4. **Development Artifacts in Production Code**: Multiple indicators suggest the code is not fully production-hardened: placeholder CSRF token, active `print()` statements, `### check authz` comments indicating incomplete authorization, and commented-out debug statements for sensitive values.

5. **Single Database as Secret Store**: SQLite serves as both the application database and the secrets store. There's no separation between operational data and key material, making it impossible to apply different access controls or audit requirements to secrets vs. general data.

## Cross-cutting Recommendations

### Immediate (Critical Path)
1. **Remove debug `print()` statements** from `do_add_issue_endpoint` and `do_edit_issue_endpoint` - these actively leak form data to stdout in production
2. **Add explicit source control metadata blocking** to the `/static/` route to prevent serving `.git` contents if accidentally deployed
3. **Document deployment requirements** explicitly stating that TRACE must be disabled at reverse proxy level and directory listing must be disabled

### Short-term (1-3 months)
4. **Implement key lifecycle management** - add `key_created_at` tracking to the schema, implement secret cleanup after election finalization (e.g., zeroing `opened_key` after results are exported/verified)
5. **Add explicit TRACE blocking middleware** and source control path filtering as defense-in-depth measures
6. **Remove all commented-out debug code** that could expose sensitive key material if uncommented
7. **Implement environment-based configuration** to ensure debug-level features are disabled in production builds

### Long-term (3-12 months)
8. **Integrate with a key management service** (HashiCorp Vault, AWS KMS, or similar) to perform cryptographic operations in an isolated security boundary - this is the most significant gap for a Level 3 ASVS target
9. **Implement secret rotation** - design a mechanism to rotate election keys for long-running elections, or implement a maximum election duration policy
10. **Separate secret storage** from operational database - consider storing key material in a dedicated secrets backend with its own access controls and audit trail
