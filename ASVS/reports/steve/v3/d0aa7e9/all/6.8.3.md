## ASVS-6.8.3: Authentication with Identity Provider - SAML Replay Prevention

### Findings for 6.8.3

#### [MEDIUM]

**Finding ID**: ASVS-683-MED-001

**Title**: No visible SAML assertion replay prevention mechanism

**File**: `v3/server/pages.py` (entire authentication flow)

**Description**: The application code contains no visible mechanism for:
1. Tracking processed SAML assertion IDs
2. Checking assertion validity periods
3. Preventing replay of authentication assertions

The authentication is entirely delegated to `asfquart.auth` and `asfquart.session`. Without visibility into these libraries, it's impossible to confirm SAML replay prevention is implemented.

Additionally, the CSRF token is a placeholder (`basic.csrf_token = 'placeholder'` at line 83 of pages.py), which means even session-level replay protections are absent:

```python
### generate a real token and store in the session
basic.csrf_token = 'placeholder'
```

**Impact**: If SAML is used as the authentication mechanism, an attacker who captures a valid SAML assertion could replay it to establish an authenticated session, particularly if:
- The IdP assertion has a long validity window
- No assertion ID tracking is implemented
- Network interception is possible

**Remediation**: If SAML is the authentication protocol, ensure the framework:
```python
# Example assertion tracking
PROCESSED_ASSERTIONS = set()  # or Redis/database backed

def validate_saml_assertion(assertion):
    assertion_id = assertion.get_id()
    
    # Check for replay
    if assertion_id in PROCESSED_ASSERTIONS:
        raise SecurityError("SAML assertion replay detected")
    
    # Check validity period
    if assertion.not_on_or_after < datetime.utcnow():
        raise SecurityError("SAML assertion expired")
    
    # Record as processed
    PROCESSED_ASSERTIONS.add(assertion_id)
```

## Cross-cutting Architecture Observations

1. **Single Trust Boundary**: The application places complete trust in the `asfquart` authentication framework. There is no defense-in-depth for authentication - if the framework is bypassed, all controls fail simultaneously.

2. **Cryptographic Architecture vs. Authentication Architecture Mismatch**: The election voting system has sophisticated cryptographic protections (Argon2 hashing, per-voter salts, Fernet encryption, anti-tamper detection), but the authentication and authorization layer is incomplete with numerous `### check authz` placeholder comments.

3. **No Separation of Duties**: The same `R.committer` role can both manage elections and vote in them. There is no enforcement that the election owner cannot also modify votes or bypass the anti-tamper mechanism.

4. **CSRF Protection Gap**: The placeholder CSRF token (`'placeholder'`) means that even with proper IdP integration, session-based attacks remain possible across all POST endpoints and the GET-based state-changing endpoints (`/do-open/`, `/do-close/`).

5. **Opaque Authentication Dependencies**: The security posture of the application is heavily dependent on the `asfquart` library, which is not included in the audit scope. The application cannot be fully assessed for ASVS 6.8.x requirements without visibility into this dependency.

6. **Authorization is Role-Based Only**: The system checks if a user is a "committer" or "pmc_member" but does not check if they are the *owner* of a specific election or have been specifically authorized for it. Any committer can manage any election.

## Cross-cutting Recommendations

### Immediate (Critical Priority)
1. **Implement CSRF protection**: Replace the placeholder token with a real CSRF mechanism. This affects all POST endpoints and the GET-based state-changing endpoints.
2. **Convert state-changing GET endpoints to POST**: `/do-open/<eid>` and `/do-close/<eid>` must not use GET for irreversible operations.
3. **Implement authorization checks**: Replace all `### check authz` placeholders with actual ownership/permission verification before allowing election management operations.

### Short-term (High Priority)
4. **Add authentication strength verification**: Capture and validate `acr`, `amr`, and `auth_time` from the IdP for sensitive operations (election open/close, vote submission).
5. **Namespace user identities by IdP**: Add an IdP identifier to the `person.pid` to prevent future identity collision if multiple IdPs are supported.
6. **Document authentication framework requirements**: Create explicit documentation of what `asfquart.auth` must verify (signature validation, replay prevention, token expiry) as a contractual interface.
7. **Add database integrity protection**: Implement an HMAC or signature over the `opened_key` to detect unauthorized modification at the database level.

### Long-term (Architectural)
8. **Implement step-up authentication**: For critical operations (opening/closing elections), require re-authentication or MFA verification.
9. **Add audit logging for authentication events**: Log authentication method, strength, and time for security monitoring.
10. **Consider database encryption**: Protect the SQLite database at rest to prevent direct modification of cryptographic material.
11. **Implement proper secret management**: The cryptographic keys and salts should be managed according to the "Secret Management" principles referenced in the ASVS section description, potentially using a dedicated key management service.
