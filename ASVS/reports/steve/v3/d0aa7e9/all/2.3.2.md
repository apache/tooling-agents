## ASVS-2.3.2: Business Logic Limits

### Findings for 2.3.2

### [CRITICAL]

**Finding ID**: ASVS-232-CRIT-001
**Title**: Missing Authorization — Any Committer Can Manage Any Election (Type B Gap)

**Location**: `v3/server/pages.py`, all management endpoints: `do_open_endpoint` (line 431), `do_close_endpoint` (line 451), `do_add_issue_endpoint` (line 469), `do_edit_issue_endpoint` (line 493), `do_delete_issue_endpoint` (line 516), `manage_page` (line 318), `do_set_open_at_endpoint` (line 360), `do_set_close_at_endpoint` (line 366)

**Vulnerable code**:
```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    result = await basic_info()

    ### check authz   # <--- NOT IMPLEMENTED

    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)
    ...
```

**Control EXISTS**: The `election` table has `owner_pid` and `authz` columns (schema.sql lines 15-20). The Election model stores these values. Comments `### check authz` exist in every endpoint.

**Control NOT CALLED**: No endpoint ever compares `result.uid` against `election.owner_pid` or `election.authz`. This is a Type B gap (control exists but not applied).

**Data flow**: Authenticated committer (any) → `/do-open/<any_eid>` → `election.open()` → unauthorized state change

**Proof of Concept**:
```
# As any ASF committer (not the election owner):
GET /do-open/abc1234567 HTTP/1.1
Cookie: [valid_session_cookie]
```

**Impact**: Any authenticated committer can open, close, add/edit/delete issues in ANY election. This completely undermines the ownership model and allows unauthorized manipulation of other users' elections.

**Remediation**:
```python
def check_election_owner(election, uid):
    """Verify the current user owns or is authorized for this election."""
    md = election.get_metadata()
    if md.owner_pid != uid:
        # Check authz group membership if defined
        if md.authz:
            # Check if uid is in the authz group
            if not is_member_of_group(uid, md.authz):
                quart.abort(403, 'Not authorized to manage this election')
        else:
            quart.abort(403, 'Not authorized to manage this election')
```

### [HIGH]

**Finding ID**: ASVS-232-HIGH-001
**Title**: No Vote String Validation Against Issue Vote Type

**Location**: `v3/steve/election.py`, `add_vote()` method (line 231)

**Vulnerable code**:
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting   # <--- NOT IMPLEMENTED

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

**Data flow**: User form input → `do_vote_endpoint()` → `election.add_vote(result.uid, iid, votestring)` → stored without validation

**Proof of Concept**:
```
POST /do-vote/abc1234567 HTTP/1.1
Content-Type: application/x-www-form-urlencoded

vote-issue123456=GARBAGE_DATA_NOT_YNA
```

For a YNA issue, a votestring of "garbage" would be encrypted and stored. At tally time, this could cause tally logic errors or produce incorrect results.

**Impact**: Arbitrary vote strings can be stored for any issue type. For YNA issues, values other than y/n/a would corrupt tallies. For STV issues, invalid labels could crash or corrupt the STV algorithm during tally.

**Remediation**:
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    
    # Validate votestring against issue type
    issue = self.q_get_issue.first_row(iid)
    if not issue:
        raise IssueNotFound(iid)
    
    m = vtypes.vtype_module(issue.type)
    if not m.validate_vote(votestring, self.json2kv(issue.kv)):
        raise ValueError(f'Invalid vote for issue type {issue.type}')
    
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    ...
```

### [MEDIUM]

**Finding ID**: ASVS-232-MED-001
**Title**: No Logical Constraint Validation on Election Dates

**Location**: `v3/server/pages.py`, `_set_election_date()` (line 88)

**Vulnerable code**:
```python
async def _set_election_date(election, field):
    ...
    data = await quart.request.get_json()
    date_str = data.get('date')
    if not date_str:
        quart.abort(400, 'Missing date')

    try:
        dt = datetime.datetime.fromisoformat(date_str).date()
    except ValueError:
        quart.abort(400, 'Invalid date format')

    # Set the date on the election (field is 'open_at' or 'close_at')
    if field == 'open_at':
        election.set_open_at(dt)
    elif field == 'close_at':
        election.set_close_at(dt)
    ...
```

**Data flow**: JSON body `{"date": "2020-01-01"}` → `_set_election_date()` → `election.set_close_at(past_date)` — no logical validation

**Proof of Concept**:
```json
POST /do-set-close_at/abc1234567
{"date": "2020-01-01"}
```
Sets a close date in the past, or sets close_at before open_at.

**Impact**: Confusing/misleading date information displayed to voters. Could be used to manipulate voter behavior (e.g., showing a close date has passed to discourage voting while election is still open).

**Remediation**: Add logical date validation (close_at > open_at, dates not in unreasonable past).

## Cross-cutting Architecture Observations

1. **Type B Gaps Dominate**: The most critical pattern is controls that EXIST in the codebase (owner_pid, authz, CSRF tokens, state validation) but are NOT APPLIED at the enforcement point. This creates false confidence — the schema and comments suggest security is in place, but enforcement is absent.

2. **Assert vs. Exception Pattern Inconsistency**: The codebase uses `assert` for state checks in some methods (`open()`, `close()`, `add_issue()`) but proper exceptions in others (`add_vote()` via `_all_metadata(required_state)`). This inconsistency means security enforcement depends on which code path is taken.

3. **GET for State Changes**: The use of GET requests for irreversible operations (`/do-open/<eid>`, `/do-close/<eid>`) combines with the CSRF gap to create a particularly dangerous attack surface.

4. **Single-Layer Defense**: Security controls exist primarily at the model layer (`election.py`) with minimal web-layer enforcement. The web layer (pages.py) trusts that the model will enforce constraints, but the model uses bypassable assertions.

5. **SQLite Concurrency Model**: The application relies on SQLite's implicit serialization for concurrency safety, but the check-then-act patterns in the application layer create TOCTOU windows that could be exploited in multi-process deployments.

6. **Audit Trail Strength**: Logging of user actions is consistently implemented, providing after-the-fact accountability even where preventive controls are missing.

## Cross-cutting Recommendations

### Immediate (P0 — Address Before Production Use)

1. **Replace all `assert` statements** with proper exception raising (`ElectionBadState`) in `election.py`. This is the most critical fix as it affects all state enforcement.

2. **Implement authorization checks** in all management endpoints — compare `result.uid` against `election.owner_pid`/`election.authz`. This closes the most impactful Type B gap.

3. **Implement CSRF protection** — generate real tokens per session and validate them in all POST endpoints. Change `/do-open/` and `/do-close/` to POST methods.

4. **Add vote string validation** in `add_vote()` — validate against issue type before encryption.

### Short-term (P1 — Within Next Sprint)

5. **Wrap vote submission in a transaction** — ensure atomic ballot submission in `do_vote_endpoint()`.

6. **Make election open operation atomic** — combine `add_salts()` and the state change in a single transaction.

7. **Add rate limiting** on all mutating endpoints, prioritizing `/do-vote/<eid>` and `/do-create-election`.

8. **Add prerequisite validation** before election opening (verify issues exist, voters are registered).

### Long-term (P2 — Architecture Improvements)

9. **Implement multi-user approval** for irreversible state changes (open/close elections), particularly for elections with many eligible voters.

10. **Add explicit optimistic locking** (e.g., version column) on the election table to prevent concurrent state modifications.

11. **Re-enable and complete transactions** in the `create-election.py` script.

12. **Consider vote garbage collection** — periodically remove superseded vote rows to prevent unbounded table growth.
