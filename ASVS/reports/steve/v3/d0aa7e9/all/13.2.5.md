## ASVS-13.2.5: Server Allowlist Configuration

### Findings for 13.2.5

### [LOW]

**Finding ID**: ASVS-1325-LOW-001

**Title**: No evidence of server-level allowlist configuration for outbound requests or file access

**File**: Application-wide (configuration gap)

**Description**: The codebase does not contain any server-level configuration defining which resources or systems the server can send requests to or load data from. While the application primarily uses local SQLite databases and local file system access, there is no evidence of:
- Web server configuration restricting outbound connections
- Application-level configuration restricting file system access paths
- Network-level allowlist documentation

**Impact**: If the application is extended to make outbound requests (e.g., for email notifications as suggested by the `get_voters_for_email()` method in `election.py` line 445), there would be no allowlist to prevent SSRF or unauthorized outbound communication.

**Remediation**:
```python
# In application configuration
ALLOWED_FILE_PATHS = {
    'database': THIS_DIR / APP.cfg.db,
    'templates': THIS_DIR / 'templates',
    'static': THIS_DIR / 'static',
    'docs': THIS_DIR / 'docs',
}

# Server-level: configure reverse proxy (e.g., nginx) to restrict outbound
# Add iptables/firewall rules to whitelist only necessary outbound destinations
```

## Cross-cutting Architecture Observations

1. **Authentication without Authorization**: The application has robust authentication (via `asfquart.auth.require`) but authorization is largely unimplemented. Multiple `### check authz` placeholders throughout `pages.py` mean any authenticated committer can manage any election. This impacts the backend communication security posture because it means the application's internal access control doesn't properly protect backend resources.

2. **Local-First Architecture**: The application uses SQLite (local file database) rather than remote services. This significantly reduces the attack surface for ASVS 13.2.x requirements about backend communication, as there are no remote database credentials, API keys, or network connections to external services to protect.

3. **Cryptographic Material Lifecycle**: The application has a well-designed cryptographic lifecycle (salt generation → key derivation → vote encryption) but all material resides in the same SQLite file. There's no separation of privilege between the application accessing election metadata and accessing cryptographic secrets.

4. **Single Process Model**: The application appears to run as a single Quart process, connecting to local SQLite. This means connection configuration (13.2.6) is less critical than in distributed architectures, but still relevant for concurrency handling.

5. **Defense in Depth Gap**: While queries are properly parameterized (SQL injection protection) and sensitive data is filtered at the API layer, the single-layer design means a compromise at the application level exposes all secrets simultaneously.

---

## Cross-cutting Recommendations

### Immediate (Critical/High Priority)

1. **Implement secrets management**: Integrate a key vault (e.g., HashiCorp Vault) to store `opened_key` values separately from the election database. At minimum, enable SQLite encryption (SQLCipher) for the database file.

2. **Restrict `__getattr__` proxy**: Limit the Election class's attribute proxy to only expose a defined set of public operations, preventing unintended access to raw database cursors that return sensitive columns.

### Short-term (Medium Priority)

3. **Document and configure connection behavior**: Add SQLite `PRAGMA busy_timeout`, WAL mode, and explicit connection lifecycle management. Document expected concurrency limits.

4. **Implement file serving allowlist**: Add file extension and name validation to `serve_doc()` to define permitted document types.

5. **Add database encryption at rest**: Use SQLCipher or similar to encrypt the SQLite database, ensuring that file system access alone cannot expose secrets.

### Long-term (Architecture)

6. **Separate secrets storage**: Move cryptographic material (election salts, opened_keys) to a dedicated secrets store with access control, audit logging, and key rotation capabilities.

7. **Implement connection pooling**: Replace ad-hoc `open_database()` calls with an application-level connection pool with documented configuration for timeouts, max connections, and retry behavior.

8. **Add outbound request allowlist**: If the application evolves to include email notifications or external integrations (suggested by `get_voters_for_email()`), implement an allowlist of permitted outbound destinations at the application and network layers.
