## ASVS-14.2.6: Minimum Required Sensitive Data Returned

### Findings for 14.2.6

### [MEDIUM]

**Finding ID:** ASVS-1426-MED-001

**Title:** Full Email Addresses Exposed in Session Context Without Masking

**File:** `v3/server/pages.py`, lines 68-77  
**Function:** `basic_info`

**Vulnerable Code:**
```python
async def basic_info():
    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],  # Full email passed to all templates
        )
```

**Data Flow:** Session store → `basic_info()` → template context → **all HTML page responses** → client browser.

**Impact:** The full email address from the user's session is passed to every template render. While this may be the user's own email (acceptable for profile page), it's passed to ALL templates including public-facing ones. If any template exposes this in HTML or JavaScript context, the full email is visible. The email should only be returned where functionally required (e.g., profile page).

**Remediation:**
```python
async def basic_info():
    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            # Only include email in contexts that need it
            # email=s['email'],  # Don't include by default
        )
    return basic

# In profile_page specifically:
async def profile_page():
    result = await basic_info()
    s = await asfquart.session.read()
    result.email = s['email']  # Only where needed
```

### [LOW]

**Finding ID:** ASVS-1426-LOW-001

**Title:** Voter Email Addresses Returned in Bulk by get_voters_for_email Without Pagination or Need-Based Filtering

**File:** `v3/steve/election.py`, lines 445-450  
**Function:** `get_voters_for_email`

**Vulnerable Code:**
```python
def get_voters_for_email(self):
    "Return a list of distinct voters (pid, name, email) eligible for this election."
    self.q_voting_persons.perform(self.eid)
    return [
        edict(pid=row.pid, name=row.name, email=row.email)
        for row in self.q_voting_persons.fetchall()
    ]
```

**Data Flow:** Database → `q_voting_persons` → full list of voter PII (pid, name, email) → calling script.

**Impact:** This function returns all voter emails in bulk. Currently used only by `mail-voters.py` (admin script) and `gather_election_data()`. The function itself returns more data than strictly needed for the email-sending operation (pid is not needed for sending mail). The risk is low since this is an admin function, but follows the principle of returning minimum data.

**Remediation:** This is acceptable for the admin email script use case. Ensure this function is not exposed through web endpoints to unauthorized users.

## Cross-cutting Architecture Observations

1. **Security Control Consistency Gap**: The application has strong cryptographic controls (Argon2, Fernet encryption, BLAKE2b, CSPRNG for IDs and shuffling) but lacks fundamental HTTP-layer security headers (Cache-Control, X-Content-Type-Options). This suggests the security design focused on cryptographic vote integrity but did not address transport/caching layer concerns.

2. **Internal vs. External API Surface**: The `Election._all_metadata()` method is documented as "INTERNAL ONLY" and returns sensitive fields (salt, opened_key), while `get_metadata()` is the safe public API. This separation is correct but relies on convention rather than enforcement (no access modifier, just naming convention with underscore prefix).

3. **Authorization Gap (Cross-cutting)**: Multiple endpoints have `### check authz` placeholder comments. While authentication is enforced via decorators, fine-grained authorization (e.g., only election owner can open/close/manage) is not implemented. This affects multiple ASVS requirements indirectly — if an unauthorized user can access management endpoints, they could view sensitive election data that should be minimized.

4. **Data Flow Segregation**: The system properly segregates sensitive cryptographic material (salt, opened_key) from public-facing queries at the SQL level (separate queries for internal vs. external use). This is a strong architectural pattern.

5. **Template Security**: The EZT templating engine used for HTML rendering requires careful attention to output encoding. The `vote-on.ezt` template includes a client-side `escapeHtml()` function for JavaScript-rendered content, but server-side rendering relies on EZT's default behavior.

6. **Operational Tooling**: The `mail-voters.py` script runs as a standalone process with direct database access. It accesses voter PII (email addresses) without the web application's session/auth layer, relying instead on OS-level access controls to the script and database file.

## Cross-cutting Recommendations

### Immediate (Within Sprint)

1. **Remove debug print statements** (`pages.py` lines 481, 497) — Direct PII/data exposure risk.
2. **Add `Cache-Control: no-store, private` header** to all authenticated responses via an `@APP.after_request` handler.
3. **Add `X-Content-Type-Options: nosniff` header** globally.

### Short-term (Within Quarter)

4. **Implement authorization checks** at all `### check authz` placeholder locations — verify the authenticated user is the election owner or in the authorized group before allowing management operations.
5. **Document data retention policy** — classify each data type (elections, votes, persons) with explicit retention periods and deletion triggers.
6. **Implement CSRF token validation** — replace the placeholder token with a real per-session token and validate on all POST endpoints.
7. **Validate `docname` parameter** in `serve_doc()` — restrict to safe filename characters.

### Long-term (Within Year)

8. **Implement automated data purge** — scheduled job to delete closed elections past retention period and purge superseded votes after tally certification.
9. **Complete migration to XChaCha20-Poly1305** with AAD binding vote ciphertext to vote_token.
10. **Implement log classification** — ensure logs containing user-action mappings have appropriate retention limits and access controls distinct from application data.
11. **Add per-response caching policy** — static assets should have appropriate cache headers (`max-age`), while dynamic authenticated content should have `no-store`.
