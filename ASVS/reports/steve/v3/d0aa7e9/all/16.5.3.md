## ASVS-16.5.3: Fail Securely / Prevent Fail-Open

### Findings for 16.5.3

### [CRITICAL]
**Finding ID**: ASVS-1653-CRIT-001

**File**: `v3/steve/election.py`, lines 53, 112, 119, 169, 186, 198, 225  
**Function**: `delete`, `close`, `add_salts`, `add_issue`, `edit_issue`, `delete_issue`, `add_voter`

**Vulnerable code quote**:
```python
def delete(self):
    "Delete this Election and its Issues and Person/Issue pairs."
    # Can't delete if it has been opened (even if later closed).
    assert self.is_editable()
    # ...

def close(self):
    "Close an election."
    # The Election should be open.
    assert self.is_open()

def add_salts(self):
    "Set the SALT column in the MAYVOTE table."
    # The Election should be editable.
    assert self.is_editable()

def add_issue(self, title, description, vtype, kv):
    "Add a new issue with a generated unique IID."
    assert self.is_editable()
    assert vtype in vtypes.TYPES

def add_voter(self, pid: str, iid: str | None = None) -> None:
    "Add PID (Person) to Issue IID, or to all Issues (None)."
    # This is only allowed while the Election is editable.
    assert self.is_editable()
```

**Description**: Critical security state checks use Python `assert` statements. **In Python, all `assert` statements are completely removed when the interpreter runs with the `-O` (optimize) flag.** This means:

1. If deployed with `python -O` (or `PYTHONOPTIMIZE=1`), ALL election state checks are silently bypassed
2. Issues could be added/edited/deleted in open or closed elections
3. Elections could be deleted after being opened
4. Voters could be added after an election is open
5. Salts could be regenerated on an open election

This is a textbook **fail-open condition** — the security control exists but can be completely disabled by a deployment configuration, with no runtime indication that protection is absent.

**Data flow**: HTTP request → endpoint → `election.add_issue()` → `assert self.is_editable()` (REMOVED if -O) → operation proceeds without state validation

**Proof of concept**: 
```bash
# Deploy the application with optimization:
python -O -m quart run
# or set environment variable:
PYTHONOPTIMIZE=1 python -m quart run

# Now POST to add an issue to an OPEN election (should be blocked):
POST /do-add-issue/OPEN_ELECTION_EID
Content-Type: application/x-www-form-urlencoded

title=Injected+Issue&description=Tampered
# This succeeds because assert is removed!
```

**Impact**: Complete bypass of election state integrity controls. An authenticated attacker could modify open or closed elections, add/remove issues during voting, or corrupt election integrity. This undermines the entire security model of the voting system.

**Remediation**:
```python
def add_issue(self, title, description, vtype, kv):
    "Add a new issue with a generated unique IID."
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
    if vtype not in vtypes.TYPES:
        raise ValueError(f'Invalid vote type: {vtype}')
    # ... rest of method

def close(self):
    "Close an election."
    if not self.is_open():
        raise ElectionBadState(self.eid, self.get_state(), self.S_OPEN)
    self.c_close.perform(self.eid)

# Apply this pattern to ALL assert statements in the class
```

### [HIGH]
**Finding ID**: ASVS-1653-HIGH-001

**File**: `v3/server/pages.py`, lines 379, 417, 438, 458, 479, 501, 524  
**Function**: `do_vote_endpoint`, `do_create_endpoint`, `do_open_endpoint`, `do_close_endpoint`, `do_add_issue_endpoint`, `do_edit_issue_endpoint`, `do_delete_issue_endpoint`

**Vulnerable code quote**:
```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    ### check authz  # <-- NOT IMPLEMENTED
    # ...
    election.add_vote(result.uid, iid, votestring)

@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz  # <-- NOT IMPLEMENTED
    election.open(pdb)

@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz  # <-- NOT IMPLEMENTED
    election.close()
```

**Description**: Multiple state-changing endpoints have `### check authz` placeholder comments but no actual authorization logic. This is a **Type B gap** — the developers clearly identified the need for authorization checks (the comment exists), but the control is not implemented. Any authenticated committer can:

1. Open/close ANY election (not just ones they own)
2. Add/edit/delete issues in ANY election
3. Vote in ANY open election (partially mitigated by `add_vote`'s mayvote check)

The `do_vote_endpoint` specifically does not verify that the user is authorized to vote in the election before attempting `election.add_vote()`. While `add_vote()` checks `q_get_mayvote`, a failure there would result in an unhandled exception rather than a graceful denial.

**Data flow**: Authenticated committer request → endpoint → `### check authz` (SKIPPED) → state-changing operation proceeds

**Proof of concept**:
```
# Any committer can open someone else's election:
GET /do-open/SOMEONE_ELSES_ELECTION_EID

# Any committer can close someone else's election:
GET /do-close/SOMEONE_ELSES_ELECTION_EID

# Any committer can add issues to someone else's election:
POST /do-add-issue/SOMEONE_ELSES_ELECTION_EID
title=Malicious+Issue&description=Injected
```

**Impact**: Unauthorized users (any authenticated committer) can manipulate elections they don't own, violating election integrity. This is a fail-open condition where the absence of authorization checks allows operations to proceed.

**Remediation**:
```python
async def check_election_owner(election, uid):
    """Verify the current user owns this election."""
    md = election.get_metadata()
    if md.owner_pid != uid:
        quart.abort(403, 'You do not have permission to manage this election.')

@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    await check_election_owner(election, result.uid)
    # ... proceed
```

**Description**: The `_set_election_date` helper does not verify the election state before setting dates. It's possible to set `open_at` on an already-open election or `close_at` on a closed election. The `set_open_at` and `set_close_at` methods in `election.py` directly execute SQL updates without state checks:

```python
def set_open_at(self, timestamp):
    "Set the open_at timestamp for this Election."
    self.c_set_open_at.perform(timestamp, self.eid)
```

**Impact**: Election metadata could be manipulated after the election transitions to a locked state.

**Remediation**: Add state validation in `set_open_at`/`set_close_at` methods.

## Cross-cutting Architecture Observations

1. **Security control existence vs. application gap (Type B)**: The most significant systemic issue is that security controls are *identified* (via `### check authz` comments and `assert` statements) but either not implemented or implemented using an unreliable mechanism (`assert`). This creates false confidence that security is handled.

2. **Logging infrastructure immaturity**: The application generates good audit events but lacks the infrastructure to protect, transmit, and store those events securely. This is a foundational gap that affects multiple ASVS requirements.

3. **Error handling philosophy inconsistency**: Some endpoints (like `do_vote_endpoint`) have try/except blocks while others (like `do_open_endpoint`, `do_close_endpoint`) have no error handling at all. There's no consistent strategy.

4. **Single-tier architecture**: The application, database, and logs all reside on the same system/process. This means a compromise of any one component grants access to all others, including logs. A multi-tier or defense-in-depth approach is absent.

5. **Admin tooling (`tally.py`) shares the same deficiencies**: The tally script uses `raise` to "fail hard" on errors, but has no structured error handling, no audit logging to a secure system, and outputs results directly to stdout where they could be captured or lost.

6. **Assertion-based validation in a cryptographic voting system**: Using `assert` for state checks in a system designed to guarantee election integrity is a fundamental design flaw. The security model depends on state transitions being enforced, but the enforcement mechanism can be silently disabled.

## Cross-cutting Recommendations

### Immediate (Critical/High - address before deployment)

1. **Replace all `assert` statements with proper exception raising** in `election.py`. Every `assert self.is_editable()`, `assert self.is_open()`, and `assert self.is_closed()` must be converted to `if not ...: raise ElectionBadState(...)`. This is the single highest-impact change.

2. **Implement a global exception handler** (`@APP.errorhandler(Exception)`) that:
   - Logs the full exception to the structured logger
   - Returns a generic error page to the user
   - Prevents stack trace disclosure

3. **Implement authorization checks** at all `### check authz` locations. At minimum, verify the authenticated user owns the election for management operations.

### Short-term (Medium - address within sprint)

4. **Configure log transmission** to a logically separate system (remote syslog, centralized logging service, or at minimum a separate filesystem with restricted permissions).

5. **Add database error handling** with graceful degradation. Wrap database operations in try/except for `sqlite3.OperationalError` and related exceptions.

6. **Remove debug `print()` statements** from production endpoints.

7. **Add log file protection**: If logs are stored locally, configure append-only permissions and ensure the application process cannot delete or modify existing log entries.

### Long-term (Architecture improvements)

8. **Implement a circuit breaker pattern** for database access, with health checks and graceful degradation.

9. **Add log integrity verification**: Implement hash chaining or digital signatures on log entries to detect tampering.

10. **Separate the logging infrastructure**: Deploy a dedicated log aggregation system (e.g., ELK stack, Loki, or cloud-native logging) that is network-isolated from the application.

11. **Add comprehensive integration tests** that verify error handling paths, including simulated database failures and invalid state transitions.
