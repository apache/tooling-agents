## ASVS-7.6.1: Federated Re-authentication

### Findings for 7.6.1

### [MEDIUM]

**Finding ID:** ASVS-761-MED-001

**File:** `v3/server/pages.py`, lines 68-84 (`basic_info`)

**Description:** The application reads session data from the SSO provider but does not validate session freshness or enforce maximum time between IdP authentication events. There is no check of when the user last authenticated at the IdP level.

```python
async def basic_info():
    """Return base-level EZT template data."""
    basic = edict()
    # ...
    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
    else:
        # No session.
        basic.update(uid=None, name=None, email=None)
    
    ### generate a real token and store in the session
    basic.csrf_token = 'placeholder'
    return basic
```

**Data flow:** User authenticates via IdP (`/auth?login=/`) → session created → application reads session → **no check of IdP authentication timestamp or session age** → access granted indefinitely until session naturally expires

**Impact:** If the IdP session expires or is revoked, the application's local session may continue to be valid, creating a disconnect between the IdP's authentication state and the RP's session state. This is particularly relevant for this voting system since:
- A user removed from a PMC/committer group at the IdP level may retain voting access
- The application cannot enforce re-authentication when IdP policies change

**Remediation:**

```python
MAX_IDP_AUTH_AGE = 3600  # Maximum 1 hour since last IdP authentication

async def basic_info():
    basic = edict()
    s = await asfquart.session.read()
    if s:
        # Check if IdP authentication is still fresh
        idp_auth_time = s.get('auth_time')  # IdP-provided auth timestamp
        if idp_auth_time and (time.time() - idp_auth_time) > MAX_IDP_AUTH_AGE:
            # Force re-authentication at IdP
            return quart.redirect('/auth?login=' + quart.request.path)
        
        basic.update(uid=s['uid'], name=s['fullname'], email=s['email'])
    else:
        basic.update(uid=None, name=None, email=None)
    return basic
```

### [LOW]

**Finding ID:** ASVS-761-LOW-001

**File:** `v3/server/pages.py` — entire file and `v3/server/templates/header.ezt`

**Description:** There is no documentation of the expected session lifetime behavior between this application (RP) and the ASF SSO provider (IdP). The login (`/auth?login=/`) and logout (`/auth?logout=/`) endpoints delegate to `asfquart` but session lifetime policies are not explicitly configured or documented in the application code.

**Impact:** Without documented session behavior, it's impossible to verify correct implementation. Operations teams cannot validate that session termination at the IdP properly propagates to this application, and vice versa.

**Remediation:** Document and configure session behavior explicitly:

```python
# In application configuration
APP.config.update(
    SESSION_LIFETIME=3600,          # Max session duration: 1 hour
    IDP_REAUTH_INTERVAL=1800,       # Re-verify with IdP every 30 minutes  
    IDLE_TIMEOUT=900,               # Idle timeout: 15 minutes
    # Document: logout at /auth?logout=/ terminates both RP session
    # and IdP session (single logout)
)
```

## Cross-cutting Architecture Observations

1. **Session Management Delegation**: The application delegates authentication/session management to `asfquart` (ASF Quart framework). This means some session controls (lifetime, storage, IdP synchronization) may be implemented at the framework level but are not visible or configurable in application code. This creates an audit gap — the framework would need separate review.

2. **Placeholder Security Controls (Type B Gaps)**: The most critical architectural issue is the prevalence of placeholder/non-functional security controls:
   - CSRF token: `basic.csrf_token = 'placeholder'` (line 83) — **exists in templates but provides no protection**
   - Authorization checks: `### check authz` comments in 8+ locations — **control points identified but not implemented**
   - These represent **Type B gaps** (control EXISTS but NOT CALLED) — providing false confidence to developers and auditors who see the control infrastructure without realizing it's non-functional.

3. **State-Changing GET Requests**: The `/do-open/<eid>` and `/do-close/<eid>` endpoints use GET methods for irreversible state changes. This violates HTTP semantics and makes these operations trivially exploitable via CSRF (no form submission needed — simple link/image inclusion suffices).

4. **Authorization Model Immaturity**: The code repeatedly notes `### need general solution` next to auth decorators. The current model only distinguishes between `R.committer` and `R.pmc_member`, but there's no ownership-based authorization (any committer can manage any election, open/close any election, etc.). The `### check authz` placeholders indicate awareness but no implementation.

5. **Voting System Sensitivity**: This is an election/voting system where integrity is paramount. The combination of:
   - No step-up authentication for voting
   - Non-functional CSRF protection
   - No session viewing/termination for users
   - GET-based state-changing operations
   
   Creates a scenario where a single compromised session could manipulate election outcomes.

## Cross-cutting Recommendations

### Immediate (Critical — before any production deployment)

1. **Implement CSRF protection**: Replace `'placeholder'` with actual token generation and validation. All state-changing endpoints MUST validate CSRF tokens. Priority: all POST endpoints and especially convert GET state-change endpoints to POST.

2. **Convert state-changing GETs to POSTs**: `/do-open/<eid>` and `/do-close/<eid>` MUST use POST methods with CSRF protection.

3. **Implement authorization checks**: Replace all `### check authz` placeholders with actual ownership/permission verification. At minimum, verify that the authenticated user is the election owner before allowing management operations.

### Short-term (Within next development sprint)

4. **Implement user session viewing**: Add `/sessions` page allowing users to see active sessions with device/time metadata.

5. **Implement user session termination**: Allow users to terminate other sessions (requiring re-authentication with at least one factor per ASVS 7.5.2).

6. **Add step-up authentication for voting**: Require recent authentication (within 5 minutes) before casting votes and performing irreversible election operations.

7. **Document session lifetime behavior**: Explicitly configure and document the session lifetime contract between this application and the ASF SSO provider.

### Long-term (Architecture improvements)

8. **Implement admin session management**: Provide administrators the ability to terminate sessions for specific users or all users (ASVS 7.4.5).

9. **Add IdP authentication freshness checking**: Validate `auth_time` claims from the IdP and force re-authentication when the maximum authentication age is exceeded (ASVS 7.6.1).

10. **Implement session binding**: Consider binding sessions to client characteristics (IP range, user agent) to detect session hijacking, with automatic step-up challenges on anomalous changes.

11. **Role and permission system overhaul**: Replace the `### need general solution` comments with a proper RBAC system that distinguishes election owners, election administrators, voters, and system administrators with appropriate permission boundaries.
