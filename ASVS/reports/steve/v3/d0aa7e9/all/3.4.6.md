## ASVS-3.4.6: frame-ancestors Directive in CSP

### Findings for 3.4.6

### [HIGH]
**Finding ID:** ASVS-346-HIGH-001  
**Location:** `v3/server/pages.py` (all HTML-rendering routes)  
**Title:** No frame-ancestors directive or X-Frame-Options header to prevent clickjacking

**Vulnerable Code:**
```python
# No CSP frame-ancestors anywhere in the application
# No X-Frame-Options header set
# All HTML pages can be freely embedded in iframes

@APP.get('/')
@APP.use_template(TEMPLATES / 'home.ezt')
async def home_page():
    result = await basic_info()
    result.title = 'Home'
    return result

# Similarly for all other HTML-rendering endpoints
```

**Data Flow:** Attacker creates malicious page → embeds voting application in iframe → overlays deceptive UI → user interacts with hidden voting form → votes cast without user awareness (clickjacking)

**Proof of Concept:**
```html
<!-- Attacker's page -->
<html>
<body>
<h1>Click here to win a prize!</h1>
<iframe src="https://steve.apache.org/vote-on/target-election-id" 
        style="opacity: 0.01; position: absolute; top: 0; left: 0; width: 100%; height: 100%;">
</iframe>
</body>
</html>
```

**Impact:**
- Voting pages can be embedded in attacker-controlled pages enabling clickjacking attacks
- Administrative actions (open/close elections, add/delete issues) can be triggered through UI redressing
- The `do-open/<eid>` and `do-close/<eid>` endpoints use GET requests, making them particularly susceptible to iframe-based CSRF/clickjacking

**Remediation:**
```python
@APP.after_request
async def add_frame_protection(response):
    # Set via CSP (preferred, per requirement)
    if 'Content-Security-Policy' not in response.headers:
        response.headers['Content-Security-Policy'] = "frame-ancestors 'self'"
    elif 'frame-ancestors' not in response.headers.get('Content-Security-Policy', ''):
        response.headers['Content-Security-Policy'] += "; frame-ancestors 'self'"
    return response
```

## Cross-cutting Architecture Observations

1. **Complete Absence of Security Header Infrastructure**: The application has NO `after_request` handler, middleware, or any other mechanism to inject security headers into HTTP responses. This is a systemic gap — not a per-header oversight, but a fundamental architectural omission.

2. **Gap Type B (Control EXISTS but NOT CALLED)**: The Quart framework fully supports `after_request` hooks and response header manipulation, but neither is utilized. This creates false confidence if anyone assumes the framework provides these headers by default (it does not).

3. **Template Architecture Lacks Security Meta Tags**: The `header.ezt` template, which is included by all HTML pages, contains no security-related `<meta>` tags that could serve as a fallback for missing HTTP headers.

4. **Custom Response Objects Bypass Future Middleware**: The `raise_404` function creates raw `quart.Response` objects. If an `after_request` handler is added in the future, these responses will be processed by it (good), but the pattern of manually constructing responses should be noted for consistency.

5. **Sensitive Application Context**: This is a **voting application** handling election data. The consequences of XSS, clickjacking, or tabnabbing attacks are severe — vote manipulation, election fraud, or voter impersonation. The absence of all browser security headers in this context is particularly critical.

6. **External Resource Loading**: The application loads resources from external origins (Apache logo from `www.apache.org`) in `header.ezt`, which would need to be accounted for in any CSP policy.

## Cross-cutting Recommendations

### Immediate (Critical Priority)
1. **Create a centralized `after_request` handler** that sets ALL required security headers on every response:
   ```python
   @APP.after_request
   async def add_security_headers(response):
       response.headers['Content-Security-Policy'] = (
           "default-src 'self'; "
           "script-src 'self'; "
           "style-src 'self'; "
           "img-src 'self' https://www.apache.org; "
           "object-src 'none'; "
           "base-uri 'none'; "
           "frame-ancestors 'self'; "
           "report-uri /csp-report"
       )
       response.headers['X-Content-Type-Options'] = 'nosniff'
       response.headers['Referrer-Policy'] = 'strict-origin-when-cross-origin'
       response.headers['Cross-Origin-Opener-Policy'] = 'same-origin'
       return response
   ```

2. **Add `rel="noopener noreferrer"` to ALL external links** in templates (`footer.ezt`, `header.ezt`).

### Short-term (High Priority)
3. **Implement per-response CSP nonces** for L3 compliance — generate nonces in `basic_info()`, pass to templates, and include in CSP header.
4. **Create a CSP violation reporting endpoint** (`/csp-report`) and integrate with existing logging.
5. **Deploy CSP in report-only mode first** (`Content-Security-Policy-Report-Only`) to identify legitimate resource loads that need allowlisting.
6. **Add SRI attributes** to `bootstrap-icons.css` and `steve.css` resources that currently lack them.

### Long-term (Architectural)
7. **Consider a security header library** like `secure` (Python) or `Talisman` (adapted for Quart) that enforces headers by default and reduces risk of regression.
8. **Add integration tests** that verify security headers are present on all response types (HTML, JSON, static files, error pages).
9. **Implement Content-Security-Policy with nonces** across all templates, requiring template engine integration to inject nonces into script/style tags dynamically.
10. **Self-host the Apache feather logo** to eliminate the need for `img-src https://www.apache.org` in CSP, tightening the policy further.
