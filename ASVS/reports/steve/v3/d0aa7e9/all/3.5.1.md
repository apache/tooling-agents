## ASVS-3.5.1: Browser Origin Separation - Anti-Forgery Tokens

### Findings for 3.5.1

### [CRITICAL]

**Finding ID:** ASVS-351-CRIT-001

**Title:** CSRF Token is a Hardcoded Placeholder — No Server-Side Validation Exists

**File:** `v3/server/pages.py`, function `basic_info()`, line 83

**Vulnerable Code:**
```python
### generate a real token and store in the session
basic.csrf_token = 'placeholder'
```

**Data Flow:** Template renders placeholder token → Forms/fetch include static "placeholder" value → Server never validates any token on any endpoint

**Gap Type:** Type B — Control EXISTS (token fields in templates, header in fetch) but NOT VALIDATED (no server-side checking occurs)

**Evidence of non-validation in all POST endpoints:**

In `do_vote_endpoint` (line 372):
```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    ### check authz
    form = edict(await quart.request.form)
    # No csrf_token validation anywhere
```

In `do_create_endpoint` (line 410):
```python
@APP.post('/do-create-election')
@asfquart.auth.require({R.pmc_member})
async def do_create_endpoint():
    result = await basic_info()
    ### check authz
    form = edict(await quart.request.form)
    # No csrf_token validation anywhere
```

In `_set_election_date` (line 88):
```python
async def _set_election_date(election, field):
    result = await basic_info()
    ### check authz
    data = await quart.request.get_json()
    # No X-CSRFToken header validation despite client sending it
```

**Proof of Concept:**
An attacker hosts the following on `https://evil.example.com`:
```html
<form id="csrf" action="https://target.example.com/do-create-election" method="POST">
  <input type="hidden" name="title" value="Attacker-Controlled Election">
  <input type="hidden" name="csrf_token" value="anything-works">
</form>
<script>document.getElementById('csrf').submit();</script>
```

If a PMC member visits this page while authenticated, a new election is created under their account.

**Impact:** All state-changing operations (vote submission, election creation, election opening/closing, issue add/edit/delete) are vulnerable to CSRF. An attacker can forge requests for any authenticated user who visits a malicious page.

**Affected Endpoints (all POST handlers):**
- `/do-vote/<eid>` — forge votes
- `/do-create-election` — create rogue elections
- `/do-add-issue/<eid>` — inject issues into elections
- `/do-edit-issue/<eid>/<iid>` — modify issue content
- `/do-delete-issue/<eid>/<iid>` — delete issues
- `/do-set-open_at/<eid>` — modify election dates
- `/do-set-close_at/<eid>` — modify election dates

**Remediation:**
```python
import secrets

async def basic_info():
    # ...
    s = await asfquart.session.read()
    if s:
        # Generate real CSRF token and store in session
        if 'csrf_token' not in s:
            s['csrf_token'] = secrets.token_hex(32)
            await asfquart.session.write(s)
        basic.csrf_token = s['csrf_token']
    return basic

def require_csrf(func):
    """Decorator to validate CSRF token on POST requests."""
    @functools.wraps(func)
    async def wrapper(*args, **kwargs):
        s = await asfquart.session.read()
        if not s or 'csrf_token' not in s:
            quart.abort(403, 'Missing session CSRF token')
        
        # Check form field or X-CSRFToken header
        token = None
        if quart.request.content_type and 'json' in quart.request.content_type:
            token = quart.request.headers.get('X-CSRFToken')
        else:
            form = await quart.request.form
            token = form.get('csrf_token')
        
        if not token or not secrets.compare_digest(token, s['csrf_token']):
            quart.abort(403, 'CSRF token validation failed')
        
        return await func(*args, **kwargs)
    return wrapper
```

### [CRITICAL]

**Finding ID:** ASVS-351-CRIT-002

**Title:** State-Changing GET Endpoints Have Zero CSRF Protection

**File:** `v3/server/pages.py`, lines 431 and 451

**Vulnerable Code:**
```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    # ...
    election.open(pdb)
    # ...

@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    # ...
    election.close()
    # ...
```

**Data Flow:** Attacker crafts URL → User's browser loads it (img tag, redirect) → GET request sent with cookies → Election state permanently changed

**Gap Type:** Type A — No anti-CSRF control exists for these endpoints (GET requests cannot carry CSRF tokens via standard mechanisms)

**Proof of Concept:**
```html
<!-- On any page the victim visits -->
<img src="https://target.example.com/do-open/KNOWN_EID" style="display:none">
<!-- Or to close: -->
<img src="https://target.example.com/do-close/KNOWN_EID" style="display:none">
```

**Impact:** Irreversible operations (opening/closing elections) can be triggered by any web page the victim visits. Opening an election prematurely or closing it early directly impacts election integrity.

**Remediation:** Convert to POST (see ASVS-3.5.3 finding) and add CSRF validation.

## Cross-cutting Architecture Observations

1. **CSRF Protection is Entirely Absent Despite Apparent Intent:** The codebase shows clear architectural intent to implement CSRF protection (token fields in templates, header passing in fetch calls, TODO items acknowledging it), but the actual enforcement is completely missing. This creates a dangerous false sense of security — developers and reviewers may believe protection exists because they see the tokens in templates.

2. **GET-Based State Changes Compound the CSRF Problem:** The `/do-open/<eid>` and `/do-close/<eid>` endpoints using GET method means that even implementing CSRF tokens for POST requests would leave these critical operations unprotected. These must be converted to POST before any CSRF solution is effective.

3. **Authorization Model is Incomplete:** Multiple `### check authz` placeholders indicate that ownership verification (e.g., "does this user own this election?") is not enforced. Any authenticated committer can manage any election. While this is beyond the scope of ASVS 3.5.x, it amplifies the impact of CSRF — an attacker doesn't need to target the election owner specifically.

4. **Template Security is Well-Implemented:** The use of `[format "js,html"]` in EZT templates for user-supplied data injected into JavaScript contexts shows awareness of XSS prevention, which is relevant because XSS can bypass CSRF protections.

5. **Client-Server Contract Mismatch:** The client (templates) sends CSRF tokens and custom headers that the server completely ignores. This indicates the frontend was developed with security in mind but the backend implementation was deferred.

## Cross-cutting Recommendations

### Immediate (Critical — deploy blockers)

1. **Implement CSRF token validation on all POST endpoints.** Generate a cryptographically random token per session, store it server-side, and validate it on every state-changing request. Use a decorator pattern for consistency.

2. **Convert `/do-open/<eid>` and `/do-close/<eid>` from GET to POST.** These perform irreversible operations and must not be triggerable via simple navigation, image loads, or link prefetch.

3. **Add Origin header validation** as defense-in-depth for all POST endpoints, rejecting requests with an Origin header that doesn't match the application's own origin.

### Short-term (High priority — within sprint)

4. **Enforce Content-Type validation** on form endpoints — reject requests that don't match `application/x-www-form-urlencoded` or `multipart/form-data` as appropriate. For JSON endpoints, reject non-`application/json` requests.

5. **Configure SameSite=Lax (minimum) or SameSite=Strict on session cookies.** This provides browser-level CSRF protection for most scenarios while the token-based approach covers older browsers.

6. **Add Sec-Fetch-Site header checking** as an additional defense layer — reject requests where `Sec-Fetch-Site: cross-site` on sensitive endpoints.

### Long-term (Architecture improvements)

7. **Consider hostname-based separation** for admin and voter functionality to leverage Same-Origin Policy restrictions between functional domains.

8. **Implement a comprehensive security middleware stack** that handles CSRF, CORS, and Content-Type validation in one place rather than requiring per-endpoint implementation.

9. **Add integration tests** that verify CSRF protection by attempting state-changing requests without valid tokens and asserting they fail with 403.
