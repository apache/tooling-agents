## ASVS-13.3.2: Least Privilege Access to Secrets

### Findings for 13.3.2

### [MEDIUM]

**Finding ID**: ASVS-1332-MED-001

**Title**: Internal `_all_metadata()` method exposes SALT and OPENED_KEY to any code with Election object access

**File**: `v3/steve/election.py`, function `_all_metadata()`, line 130

**Vulnerable Code**:
```python
def _all_metadata(self, required_state=None):
    "INTERNAL ONLY: return all metadata about this Election."

    # NOTE: this returns the SALT and OPENED_KEY columns. This
    # API is not for public use.
    md = self.q_metadata.first_row(self.eid)
    if not md:
        raise self._disappeared()

    if required_state:
        state = self._compute_state(md)
        if state != required_state:
            raise ElectionBadState(self.eid, state, required_state)

    return md
```

**Data flow**: `q_metadata` (SELECT * FROM election) → returns all columns including `salt` and `opened_key` → accessible to any caller with an `Election` instance.

**Description**: The `_all_metadata()` method is marked as "INTERNAL ONLY" by convention (leading underscore and comment), but Python does not enforce access control. Any code with access to an `Election` instance can call `election._all_metadata()` to retrieve the salt and opened_key. The `__getattr__` proxy method (line 48) further means that query cursors like `q_metadata` are directly accessible on the Election object.

**Impact**: If authorization gaps exist (and they do — multiple `### check authz` placeholders), any authenticated user who obtains an Election object could access sensitive cryptographic material through the internal API.

**Remediation**:
```python
def _all_metadata(self, required_state=None):
    "INTERNAL ONLY: return all metadata about this Election."
    md = self.q_metadata.first_row(self.eid)
    if not md:
        raise self._disappeared()
    if required_state:
        state = self._compute_state(md)
        if state != required_state:
            raise ElectionBadState(self.eid, state, required_state)
    # Return as a named tuple or frozen object to prevent accidental exposure
    return _InternalMetadata(md)

class _InternalMetadata:
    """Wrapper that prevents serialization of sensitive fields."""
    __slots__ = ('_md',)
    def __init__(self, md):
        object.__setattr__(self, '_md', md)
    def __getattr__(self, name):
        return getattr(self._md, name)
    def __repr__(self):
        return f'<Metadata eid={self._md.eid} [REDACTED]>'
```

### [LOW]

**Finding ID**: ASVS-1332-LOW-001

**Title**: `__getattr__` proxy exposes all database cursors without access restriction

**File**: `v3/steve/election.py`, line 48

**Vulnerable Code**:
```python
def __getattr__(self, name):
    "Proxy the cursors."
    return self.__dict__.get(name, getattr(self.db, name))
```

**Description**: The `__getattr__` proxy on the `Election` class makes all database cursors (including `q_metadata` which returns sensitive columns) accessible as attributes of any Election instance. This violates least privilege by exposing internal database operations to any code that holds a reference to an Election object.

**Impact**: Code in `pages.py` directly accesses query cursors (e.g., `election.q_find_issues.perform(...)` at line 236), demonstrating that internal DB operations are available at the same privilege level as public API methods.

**Remediation**: Restrict proxy to only expose documented public cursors:
```python
_PUBLIC_CURSORS = {'q_check_election', 'q_issues', 'q_get_issue', 'q_find_issues'}

def __getattr__(self, name):
    "Proxy only public cursors."
    if name in self._PUBLIC_CURSORS:
        return getattr(self.db, name)
    raise AttributeError(f"'{type(self).__name__}' has no attribute '{name}'")
```

### Positive Patterns for 13.3.2

1. **Deliberate data filtering**: `get_metadata()` constructs a new `edict` with only non-sensitive fields, deliberately excluding `salt` and `opened_key`.
2. **Query-level protection**: Public-facing queries (`q_open_to_me`, `q_owned`, `q_upcoming_to_me`) are designed to never return `salt` or `opened_key` columns.
3. **Comment discipline**: Multiple explicit comments document that sensitive fields must never be returned (e.g., "NOTE: do not return the SALT or OPENED_KEY", "NEVER return issue.salt").
4. **Separation of internal/public**: The naming convention (`_all_metadata` vs `get_metadata`) clearly delineates internal and public APIs.

## Cross-cutting Architecture Observations

1. **Authentication without Authorization**: The application has robust authentication (via `asfquart.auth.require`) but authorization is largely unimplemented. Multiple `### check authz` placeholders throughout `pages.py` mean any authenticated committer can manage any election. This impacts the backend communication security posture because it means the application's internal access control doesn't properly protect backend resources.

2. **Local-First Architecture**: The application uses SQLite (local file database) rather than remote services. This significantly reduces the attack surface for ASVS 13.2.x requirements about backend communication, as there are no remote database credentials, API keys, or network connections to external services to protect.

3. **Cryptographic Material Lifecycle**: The application has a well-designed cryptographic lifecycle (salt generation → key derivation → vote encryption) but all material resides in the same SQLite file. There's no separation of privilege between the application accessing election metadata and accessing cryptographic secrets.

4. **Single Process Model**: The application appears to run as a single Quart process, connecting to local SQLite. This means connection configuration (13.2.6) is less critical than in distributed architectures, but still relevant for concurrency handling.

5. **Defense in Depth Gap**: While queries are properly parameterized (SQL injection protection) and sensitive data is filtered at the API layer, the single-layer design means a compromise at the application level exposes all secrets simultaneously.

## Cross-cutting Recommendations

### Immediate (Critical/High Priority)

1. **Implement secrets management**: Integrate a key vault (e.g., HashiCorp Vault) to store `opened_key` values separately from the election database. At minimum, enable SQLite encryption (SQLCipher) for the database file.

2. **Restrict `__getattr__` proxy**: Limit the Election class's attribute proxy to only expose a defined set of public operations, preventing unintended access to raw database cursors that return sensitive columns.

### Short-term (Medium Priority)

3. **Document and configure connection behavior**: Add SQLite `PRAGMA busy_timeout`, WAL mode, and explicit connection lifecycle management. Document expected concurrency limits.

4. **Implement file serving allowlist**: Add file extension and name validation to `serve_doc()` to define permitted document types.

5. **Add database encryption at rest**: Use SQLCipher or similar to encrypt the SQLite database, ensuring that file system access alone cannot expose secrets.

### Long-term (Architecture)

6. **Separate secrets storage**: Move cryptographic material (election salts, opened_keys) to a dedicated secrets store with access control, audit logging, and key rotation capabilities.

7. **Implement connection pooling**: Replace ad-hoc `open_database()` calls with an application-level connection pool with documented configuration for timeouts, max connections, and retry behavior.

8. **Add outbound request allowlist**: If the application evolves to include email notifications or external integrations (suggested by `get_voters_for_email()`), implement an allowlist of permitted outbound destinations at the application and network layers.
