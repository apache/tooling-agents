## ASVS-12.1.2: General TLS Security Guidance - Cipher Suites

### Findings for 12.1.2

### [HIGH]

**Finding ID**: ASVS-1212-HIGH-001

**File**: `v3/server/main.py`, function `run_standalone()`, lines 75-82

**Vulnerable code:**
```python
kwargs = {}
if app.cfg.server.certfile:
    kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
    kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
    extra_files.update((kwargs['certfile'], kwargs['keyfile']))

# Spool up the app!
app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

**Data flow**: TLS setup → no cipher suite specification → framework default cipher suites used

**Proof of concept**: Without explicit cipher suite configuration, weak ciphers may be negotiated:
```bash
# Test for weak ciphers
nmap --script ssl-enum-ciphers -p 58383 localhost
# Or:
openssl s_client -connect localhost:58383 -cipher RC4-SHA
```

**Impact**: The application has no control over which cipher suites are enabled. Depending on the Python/OpenSSL version, this could allow:
- Non-forward-secrecy ciphers (e.g., RSA key exchange)
- Weak encryption algorithms (e.g., 3DES, RC4)
- Weak hash algorithms in cipher suites (e.g., SHA-1 based MACs)

For L3 compliance, the application MUST only support cipher suites providing forward secrecy (ECDHE/DHE key exchange).

**Remediation**:
```python
import ssl

def create_ssl_context(certfile, keyfile):
    ctx = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
    ctx.minimum_version = ssl.TLSVersion.TLSv1_2
    
    # Only allow strong cipher suites with forward secrecy
    ctx.set_ciphers(
        'ECDHE+AESGCM:ECDHE+CHACHA20:DHE+AESGCM:DHE+CHACHA20'
        ':!aNULL:!MD5:!DSS:!RC4:!3DES'
    )
    ctx.load_cert_chain(certfile, keyfile)
    return ctx
```

## Cross-cutting Architecture Observations

1. **TLS Configuration is Entirely Application-Agnostic**: The application treats TLS as an optional transport layer concern, passing cert/key paths to the framework without any security controls around protocol versions, cipher suites, or enforcement. This is a systemic design gap.

2. **Dual Deployment Modes Create Inconsistency**: The standalone mode (`run_standalone()`) accepts TLS configuration via `config.yaml`, while the ASGI mode (`run_asgi()`) delegates all TLS to Hypercorn's external configuration. Neither mode enforces TLS or validates its presence.

3. **No Security Headers Infrastructure**: There is no middleware or after-request hook for setting security-critical HTTP headers (HSTS, X-Content-Type-Options, etc.). This suggests TLS enforcement at the application layer has not been considered.

4. **Development vs. Production Gap**: The development setup uses self-signed certificates (mkcert-generated `localhost.apache.org+3.pem`), which is appropriate for development but provides no path toward production TLS hardening (OCSP, ECH, cipher suite restrictions).

5. **Proxy Assumption Undocumented**: The config comment mentions "a proxy sits in front of this server," but there is no documentation of required proxy TLS configuration, no `X-Forwarded-Proto` validation, and no deployment hardening guide.

## Cross-cutting Recommendations

### Immediate (Critical/High Priority)

1. **Enforce TLS as mandatory**: Remove the option to run without TLS. If plain HTTP is needed for development behind a proxy, require an explicit `--insecure` flag or environment variable.

2. **Add TLS protocol version enforcement**: Create an SSL context with `minimum_version = TLSv1_2` for standalone mode. Document required Hypercorn TLS configuration for ASGI mode.

3. **Implement HSTS**: Add `Strict-Transport-Security` header via after-request middleware to prevent protocol downgrade attacks.

4. **Configure cipher suites**: Restrict to AEAD cipher suites with forward secrecy (ECDHE+AESGCM, ECDHE+CHACHA20).

### Short-term (Medium Priority)

5. **Document production TLS requirements**: Create a deployment guide specifying required TLS configuration for both standalone and proxy-fronted deployments.

6. **Add HTTP→HTTPS redirect**: If port 80 is accessible, redirect all traffic to HTTPS.

7. **Configure OCSP stapling**: For production deployments, configure either at the application level (if possible) or document proxy-level OCSP stapling requirements.

8. **Add secure cookie attributes**: Ensure session cookies have `Secure`, `HttpOnly`, and `SameSite=Strict` flags.

### Long-term (Low Priority)

9. **Implement ECH**: Once Python/Hypercorn gains native ECH support, or document proxy-level ECH configuration (e.g., via Cloudflare or nginx with ECH patches).

10. **Automated TLS testing**: Integrate TLS configuration testing (e.g., testssl.sh, sslyze) into CI/CD pipeline to detect configuration regressions.

11. **Certificate management**: Implement automated certificate rotation, separate certificates per environment (as noted in the ASVS section description about wildcard certificates), and certificate expiry monitoring.
