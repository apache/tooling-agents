## ASVS-6.5.2: Hashing of Lookup Secrets with Proper Storage

### Findings for 6.5.2

No lookup secrets (recovery codes, TANs, grid values) are generated or stored in this application. The cryptographic tokens present in the system serve a different purpose (vote encryption/integrity) and are not authentication lookup secrets.

**No findings detected** — the requirement is not applicable to this code scope.

**Relevant observation on stored sensitive material:**

The `vote_token` (32 bytes, stored in `vote` table) is the output of Argon2 hashing:

```python
# v3/steve/crypto.py, line 51
def gen_vote_token(opened_key: bytes, pid: str, iid: str, salt: bytes) -> bytes:
    "Generate a person or issue token."
    return _hash(opened_key + pid.encode() + iid.encode(), salt)
```

```python
# v3/steve/crypto.py, line 83
def _hash(data: bytes, salt: bytes) -> bytes:
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,
    )
```

While this is not a "lookup secret" per ASVS definition, it demonstrates proper use of a memory-hard hashing function (Argon2) with per-record salts for sensitive cryptographic material derivation.

| Control | Location | Status |
|---------|----------|--------|
| Argon2 hashing for vote tokens | `crypto.py:83` (`_hash()`) | Applied — 32-byte output with 16-byte salt |

### Positive Patterns for 6.5.2

- **Argon2 with proper parameters**: `time_cost=2`, `memory_cost=65536` (64MB), `parallelism=4`, `hash_len=32` — these are reasonable defaults for key derivation.
- **Unique salts per mayvote pair**: Each person-issue pair gets its own 16-byte cryptographic salt, preventing rainbow table attacks.
- **Salt is 16 bytes (128 bits)**: Exceeds the 32-bit minimum specified in the requirement for password storage hashing.

## Cross-cutting Architecture Observations

1. **No MFA in Scope**: This codebase is an election/voting application that delegates all user authentication (including any MFA) to the external `asfquart.auth` library. The ASVS 6.5.x requirements for multi-factor authentication mechanisms are **not directly applicable** to this code.

2. **Strong Cryptographic Foundations**: Despite MFA not being in scope, the cryptographic practices are robust:
   - CSPRNG (`secrets` module) for all random value generation
   - Argon2 for key derivation with appropriate parameters
   - Per-record 128-bit salts
   - Fernet (AES-CBC + HMAC-SHA256) for vote encryption
   - HKDF for key stretching

3. **Separation of Concerns**: Authentication, authorization, and election logic are separated:
   - `asfquart.auth` handles authentication (including any MFA)
   - `pages.py` handles authorization decorators and route logic
   - `election.py` handles business logic with state assertions
   - `crypto.py` handles all cryptographic operations

4. **Authorization Gaps (Related but Distinct)**: Multiple `### check authz` placeholder comments indicate that fine-grained authorization (e.g., "is this user the election owner?") is not yet implemented. This is not an MFA issue but is relevant to overall security.

5. **CSRF Placeholder**: The `csrf_token = 'placeholder'` pattern indicates CSRF protection is not implemented, which could allow unauthorized actions on behalf of authenticated users, potentially undermining the value of any MFA enforcement at the authentication layer.

## Cross-cutting Recommendations

### Immediate (if MFA is planned for this application)
- No immediate MFA-related changes needed in this codebase since authentication is delegated externally.
- Replace `random.shuffle()` with `crypto.shuffle()` in `pages.py:269` for consistency in using CSPRNG.

### Short-term
- **Implement CSRF protection**: Replace the placeholder token with a proper implementation. Without CSRF protection, an attacker could force authenticated users to perform actions (voting, election management) via cross-site request forgery, bypassing the security benefit of any MFA at the authentication layer.
- **Implement authorization checks**: The `### check authz` placeholders should be filled in to ensure that election owners are the only ones who can manage their elections.

### Long-term
- **Document MFA requirements at the infrastructure level**: Since authentication is delegated to `asfquart.auth`, ensure that the external authentication system properly implements ASVS 6.5.x requirements (one-time-use codes, CSPRNG generation, proper lifetimes, factor revocation).
- **Consider voter-level revocation**: Add the ability to revoke individual voter eligibility after an election is opened, in case of compromised credentials.
- **Audit the external auth system**: The `asfquart.auth` library should be separately audited against ASVS 6.5.x requirements to ensure MFA is properly implemented at that layer.
