## ASVS-1.2.1: Injection Prevention - Context-Appropriate Output Encoding

### Findings for 1.2.1

#### [HIGH]
**Finding ID:** ASVS-121-HIGH-001

**File:** `v3/server/templates/vote-on.ezt`, multiple locations (HTML context)

**Vulnerable Code (line ~80):**
```html
<h1 class="h4 mb-0 fw-semibold">[election.title]</h1>
```

**Additional instances:**
- Line ~82: `<span>Authority: <strong>[election.owner_name]</strong></span>`
- Line ~83: `<span>Electorate: <strong>[election.authz]</strong></span>`
- Line ~111: `<strong>[issues.title]</strong>` (in issue list items)

**Data Flow:** `election.title` / `issues.title` (from database, set by admin) → EZT template (no `[format "html"]`) → HTML document → browser HTML parser

**Issue:** User-controlled values are output in HTML element content without HTML encoding. EZT's default behavior is to output raw values (evidenced by the explicit use of `[format "js,html"]` in manage.ezt which would be unnecessary if encoding were automatic).

**Proof of Concept:** Election title: `<script>fetch('/do-vote/EID',{method:'POST',body:new FormData(document.getElementById('voteForm'))})</script>`

**Impact:** Stored XSS affecting all voters who view the ballot page.

**Remediation:** Use `[format "html"]` for all user-derived values in HTML contexts:
```html
<h1 class="h4 mb-0 fw-semibold">[format "html"][election.title][end]</h1>
```

#### [HIGH]
**Finding ID:** ASVS-121-HIGH-002

**File:** `v3/server/templates/manage.ezt`, line ~248 (HTML context for descriptions)

**Vulnerable Code:**
```html
<div id="description-[issues.iid]" class="description mt-2">[issues.description]</div>
```

**Data Flow:** `issues.description` (from database, user-set via form) → raw output in HTML div → browser HTML parser

**Issue:** Issue descriptions are output without HTML encoding in the management template. Unlike vote-on.ezt (where `rewrite_description` is called), manage.ezt outputs descriptions directly. A description containing `<script>` tags would execute.

**Proof of Concept:** Issue description: `<img src=x onerror="fetch('/do-delete-issue/EID/IID')">`

**Impact:** Stored XSS on the management page. Could trigger unauthorized actions if a different admin views the page.

**Remediation:**
```html
<div id="description-[issues.iid]" class="description mt-2">[format "html"][issues.description][end]</div>
```

## Cross-cutting Architecture Observations

### 1. Inconsistent Encoding Strategy (Type B Pattern)
The most critical cross-cutting observation is the **inconsistent application of encoding controls**:
- `manage.ezt` and `manage-stv.ezt` correctly use `[format "js,html"]` for JavaScript-in-HTML contexts
- `vote-on.ezt` (the voter-facing page) does NOT use any format directives for the same type of data

This is a Type B gap (control exists but not applied) and represents false confidence — developers may believe the voter page is safe because they applied controls elsewhere.

### 2. EZT Template Engine Default Behavior
The application relies on an EZT template engine that does NOT HTML-encode output by default. This is evidenced by:
- The `rewrite_description` function constructing raw HTML that must pass through without escaping
- The selective use of `[format]` directives only in specific contexts
- No visible `base_format` configuration

This architectural choice means every template variable is a potential XSS vector unless explicitly wrapped in a `[format]` directive.

### 3. Data Storage in Original Form (Positive)
The application correctly stores data in its original form in the database — no pre-encoding. This prevents double-encoding issues but places the full burden of output encoding on the template layer, which currently has gaps.

### 4. Mixed HTML Construction Patterns
HTML is constructed in multiple locations:
- Server-side in Python (`rewrite_description`)
- Server-side in EZT templates (all templates)
- Client-side in JavaScript (`makeItem()`)

Each location needs its own encoding strategy, but only the client-side JavaScript consistently applies it.

### 5. State-Changing GET Requests
The `/do-open/<eid>` and `/do-close/<eid>` endpoints use GET methods for state-changing operations. Combined with the placeholder CSRF tokens, these are trivially exploitable via CSRF (e.g., `<img src="/do-open/KNOWN_EID">`).

## Cross-cutting Recommendations

### Immediate (Critical — address before next deployment)

1. **Add `[format "js"]` encoding to ALL JavaScript-embedded values in `vote-on.ezt`** (ASVS-123-CRIT-001). This is the highest-priority fix as it affects the voter-facing ballot page and could allow manipulation of election results.

2. **Fix `rewrite_description` to HTML-encode description content** before wrapping in `<pre>` tags (ASVS-112-HIGH-001). Apply `html.escape()` to the description text and `urllib.parse.quote()` to filenames in constructed URLs.

### Short-term (within current sprint)

3. **Establish a template encoding standard:** Either:
   - Configure EZT with `base_format = FORMAT_HTML` and use `[format "raw"]` only where intentional raw HTML is needed, OR
   - Audit ALL templates and add explicit `[format "html"]` to every user-derived variable in HTML contexts

4. **Add `[format "html"]` to all unencoded template outputs** including:
   - `election.title`, `election.owner_name`, `election.authz` in all templates
   - `issues.title`, `issues.description` in manage templates
   - Flash message content in `flashes.ezt`

5. **Fix flash messages** to either encode at creation or ensure template-level encoding (ASVS-112-MED-001).

6. **Add explicit validation to `serve_doc`** for the `docname` parameter — restrict to alphanumeric plus safe punctuation (ASVS-111-MED-001).

### Long-term (architectural)

7. **Consider migrating to JSON embedding pattern** for complex JavaScript data: serialize server-side with `json.dumps()` and parse client-side with `JSON.parse()` from a `<script type="application/json">` tag. This is inherently safer than template-based JS construction.

8. **Implement a Content Security Policy (CSP)** header to mitigate XSS impact. At minimum, restrict `script-src` to prevent inline script execution, which would require refactoring inline scripts to external files.

9. **Implement real CSRF protection** — the placeholder tokens provide zero security. Use a proper CSRF library that generates per-session tokens and validates them on all state-changing requests.

10. **Convert state-changing GET endpoints** (`/do-open/`, `/do-close/`) to POST methods with CSRF validation.
