## ASVS-14.2.7: Sensitive Information Data Retention Classification

### Findings for 14.2.7

### [HIGH]

**Finding ID:** ASVS-1427-HIGH-001

**Title:** No Data Retention Policy or Automatic Purge Mechanism for Election Records and Vote Data

**File:** `v3/schema.sql` (entire schema), `v3/steve/election.py`  
**Functions:** All CRUD operations

**Vulnerable Code (schema.sql):**
```sql
CREATE TABLE vote (
    vid  INTEGER PRIMARY KEY AUTOINCREMENT,
    vote_token  BLOB NOT NULL  CHECK (length(vote_token) = 32),
    ciphertext  BLOB NOT NULL
) STRICT;
```

**Evidence from documentation (v3/docs/schema.md):**
```
Re-voting: The vote table allows multiple votes per person-issue pair. 
The latest vote is identified using MAX(vid). Older votes are retained for auditing.
```

**Data Flow:** Votes are inserted into the `vote` table → **retained indefinitely** → no scheduled deletion → no TTL → no retention classification.

**Impact:** 
- **Vote records**: All vote records (including superseded re-votes) are retained indefinitely without any documented retention schedule. Even after an election is closed and tallied, encrypted vote data persists.
- **Person records**: The `person` table retains PID, name, and email indefinitely with no cleanup mechanism.
- **Election records**: Closed elections with their associated crypto material (salt, opened_key) persist forever.
- **Mayvote records**: Voter eligibility data (linking persons to issues) has no expiry.

This violates data minimization principles (especially for PII like email addresses) and creates an expanding attack surface for data breach scenarios.

**Remediation:**
```python
# Add to election.py:
import time

RETENTION_DAYS_CLOSED_ELECTION = 365  # 1 year after closing
RETENTION_DAYS_PERSON_INACTIVE = 730  # 2 years of inactivity

@classmethod
def purge_expired_elections(cls, db_fname):
    """Delete closed elections older than retention period."""
    db = cls.open_database(db_fname)
    cutoff = int(time.time()) - (RETENTION_DAYS_CLOSED_ELECTION * 86400)
    
    db.conn.execute('BEGIN TRANSACTION')
    # Get expired elections
    expired = db.conn.execute(
        'SELECT eid FROM election WHERE closed = 1 AND close_at < ?',
        (cutoff,)
    ).fetchall()
    
    for (eid,) in expired:
        # Delete in referential order
        db.conn.execute('''
            DELETE FROM vote WHERE vote_token IN (
                SELECT vote_token FROM mayvote m
                JOIN issue i ON m.iid = i.iid
                WHERE i.eid = ?
            )''', (eid,))
        db.conn.execute('''
            DELETE FROM mayvote WHERE iid IN (
                SELECT iid FROM issue WHERE eid = ?
            )''', (eid,))
        db.conn.execute('DELETE FROM issue WHERE eid = ?', (eid,))
        db.conn.execute('DELETE FROM election WHERE eid = ?', (eid,))
    
    db.conn.execute('COMMIT')
    return len(expired)
```

Additionally, add a retention schedule document:
```markdown
# Data Retention Policy
| Data Type | Retention Period | Purge Trigger |
|-----------|-----------------|---------------|
| Closed election records | 1 year after close_at | Automated daily job |
| Superseded vote records | Until election tally is certified | Post-certification cleanup |
| Person records | 2 years after last activity | Automated monthly job |
| Election documents | Same as election record | Cascade with election deletion |
```

### [MEDIUM]

**Finding ID:** ASVS-1427-MED-001

**Title:** Superseded Vote Records Retained Indefinitely Without Purge Mechanism

**File:** `v3/schema.sql`, `v3/steve/election.py`  
**Function:** `add_vote` (line 231)

**Vulnerable Code:**
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."
    # ...
    self.c_add_vote.perform(vote_token, ciphertext)
    # Previous votes with same vote_token remain in the table
```

**Schema comment:**
```sql
/* duplicates of (person, issue) may occur (the vote_token will be the
   same), as re-voting is allowed. Only the latest is used. */
```

**Data Flow:** Re-vote submitted → new row inserted → old vote row remains → `q_recent_vote` only uses latest → **old votes accumulate forever**.

**Impact:** When voters re-vote, old encrypted ballots remain in the database permanently. While only the latest vote is used for tallying (`ORDER BY _ROWID_ DESC LIMIT 1`), the retained historical votes:
1. Increase the attack surface for key compromise scenarios
2. Consume storage indefinitely
3. Could potentially reveal voting pattern changes if the encryption key is compromised
4. Are not needed once the election is closed and tallied

**Remediation:**
```python

        DELETE FROM vote 
        WHERE vid NOT IN (
            SELECT MAX(vid) FROM vote GROUP BY vote_token
        )
        AND vote_token IN (
            SELECT DISTINCT v.vote_token FROM vote v
            JOIN mayvote m ON v.vote_token = (/* derived token */)
            -- Or: use a separate tracking mechanism
        )
    ''')
```

## Cross-cutting Architecture Observations

1. **Security Control Consistency Gap**: The application has strong cryptographic controls (Argon2, Fernet encryption, BLAKE2b, CSPRNG for IDs and shuffling) but lacks fundamental HTTP-layer security headers (Cache-Control, X-Content-Type-Options). This suggests the security design focused on cryptographic vote integrity but did not address transport/caching layer concerns.

2. **Internal vs. External API Surface**: The `Election._all_metadata()` method is documented as "INTERNAL ONLY" and returns sensitive fields (salt, opened_key), while `get_metadata()` is the safe public API. This separation is correct but relies on convention rather than enforcement (no access modifier, just naming convention with underscore prefix).

3. **Authorization Gap (Cross-cutting)**: Multiple endpoints have `### check authz` placeholder comments. While authentication is enforced via decorators, fine-grained authorization (e.g., only election owner can open/close/manage) is not implemented. This affects multiple ASVS requirements indirectly — if an unauthorized user can access management endpoints, they could view sensitive election data that should be minimized.

4. **Data Flow Segregation**: The system properly segregates sensitive cryptographic material (salt, opened_key) from public-facing queries at the SQL level (separate queries for internal vs. external use). This is a strong architectural pattern.

5. **Template Security**: The EZT templating engine used for HTML rendering requires careful attention to output encoding. The `vote-on.ezt` template includes a client-side `escapeHtml()` function for JavaScript-rendered content, but server-side rendering relies on EZT's default behavior.

6. **Operational Tooling**: The `mail-voters.py` script runs as a standalone process with direct database access. It accesses voter PII (email addresses) without the web application's session/auth layer, relying instead on OS-level access controls to the script and database file.

## Cross-cutting Recommendations

### Immediate (Within Sprint)

1. **Remove debug print statements** (`pages.py` lines 481, 497) — Direct PII/data exposure risk.
2. **Add `Cache-Control: no-store, private` header** to all authenticated responses via an `@APP.after_request` handler.
3. **Add `X-Content-Type-Options: nosniff` header** globally.

### Short-term (Within Quarter)

4. **Implement authorization checks** at all `### check authz` placeholder locations — verify the authenticated user is the election owner or in the authorized group before allowing management operations.
5. **Document data retention policy** — classify each data type (elections, votes, persons) with explicit retention periods and deletion triggers.
6. **Implement CSRF token validation** — replace the placeholder token with a real per-session token and validate on all POST endpoints.
7. **Validate `docname` parameter** in `serve_doc()` — restrict to safe filename characters.

### Long-term (Within Year)

8. **Implement automated data purge** — scheduled job to delete closed elections past retention period and purge superseded votes after tally certification.
9. **Complete migration to XChaCha20-Poly1305** with AAD binding vote ciphertext to vote_token.
10. **Implement log classification** — ensure logs containing user-action mappings have appropriate retention limits and access controls distinct from application data.
11. **Add per-response caching policy** — static assets should have appropriate cache headers (`max-age`), while dynamic authenticated content should have `no-store`.
