## ASVS-13.2.2: Least Privilege for Backend Communications

### Findings for 13.2.2

#### [MEDIUM]
**Finding ID:** ASVS-1322-MED-001

**Title:** All application components use identical full-privilege database access

**Location:** `v3/steve/election.py`, `v3/steve/persondb.py`, `v3/server/bin/create-election.py`, `v3/server/bin/mail-voters.py`

**Description:** All application modules — web handlers, business logic, command-line tools — access the SQLite database with the same full read/write permissions. There is no privilege separation:

**Web request handlers (full access):**
```python
# v3/server/pages.py, line 40
DB_FNAME = THIS_DIR / APP.cfg.db
```

**Election model (can modify all tables):**
```python
# v3/steve/election.py, lines 35-37
@staticmethod
def open_database(db_fname):
    return asfpy.db.DB(db_fname, yaml_fname=QUERIES, yaml_section='election')
```

**PersonDB (can modify person table):**
```python
# v3/steve/persondb.py, lines 25-26
@classmethod
def open(cls, db_fname):
    return cls(asfpy.db.DB(db_fname, yaml_fname=QUERIES, yaml_section='person'))
```

**CLI tools (same DB access as web server):**
```python
# v3/server/bin/create-election.py, line 31
DB_FNAME = THIS_DIR.parent / 'steve.db'
```

```python
# v3/server/bin/mail-voters.py, line 28
DB_FNAME = THIS_DIR.parent / 'steve.db'
```

**Specific concerns:**
1. The `mail-voters.py` script only needs READ access to election metadata and voter emails, but has full write access to the database
2. The voter-facing endpoints (e.g., `/vote-on/<eid>`) can technically invoke any database operation (create elections, delete data) since the same `Election` class is used
3. No OS-level service account separation between the web server and CLI tools

**Impact:** If any component is compromised (e.g., through a web vulnerability), the attacker gains full database access including ability to modify election results, delete elections, or alter voter records. The `mail-voters.py` script could be leveraged to modify data if compromised.

**Remediation:**
1. For SQLite: Implement application-level privilege separation by using different database wrapper classes with restricted query sets:

```python
# Proposed: read-only voter context
class VoterDB:
    """Read-only access for voter-facing operations."""
    ALLOWED_QUERIES = {'q_open_to_me', 'q_get_issue', 'q_find_issues', 'q_has_voted'}
    
    def __init__(self, db_fname):
        # Open in read-only mode where possible
        self.conn = sqlite3.connect(f'file:{db_fname}?mode=ro', uri=True)
```

2. For CLI tools, use read-only database connections where write access is not needed:
```python
# mail-voters.py - should use read-only connection
import sqlite3
conn = sqlite3.connect(f'file:{DB_FNAME}?mode=ro', uri=True)
```

3. Document the principle of least privilege for each component and its required access level.

#### [LOW]
**Finding ID:** ASVS-1322-LOW-001

**Title:** CLI scripts run without documented privilege requirements

**Location:** `v3/server/bin/create-election.py`, `v3/server/bin/mail-voters.py`

**Description:** Command-line utilities that directly modify the database or send emails have no documented OS-level privilege requirements or access controls:

```python
# v3/server/bin/create-election.py - can create elections, add voters
if __name__ == '__main__':
    logging.basicConfig(level=logging.INFO)
    parser = argparse.ArgumentParser(...)
    args = parser.parse_args()
    main(args.yaml_file)
```

```python
# v3/server/bin/mail-voters.py - can read voter emails and send messages
if __name__ == '__main__':
    logging.basicConfig(level=logging.INFO)
    parser = argparse.ArgumentParser(...)
    args = parser.parse_args()
    main(args.eid, args.template)
```

These scripts have the same database privileges as the web server but no authentication or authorization layer.

**Impact:** Anyone with shell access to the server can create elections, modify voter rolls, or send emails to voters without audit trail beyond system-level logging.

**Remediation:** Document required OS permissions and consider adding authentication checks or restricted execution contexts for CLI tools.

### Positive Patterns for 13.2.2

- Web endpoints use role-based access control decorators (`R.committer`, `R.pmc_member`) to restrict who can perform actions
- The `Election` class enforces state-based restrictions (e.g., cannot delete an opened election, cannot vote on a closed election) via assertions
- `PersonDB` and `Election` use separate query sections (`yaml_section='person'` vs `yaml_section='election'`) suggesting a design toward separation of concerns
- Foreign key constraints with `ON DELETE RESTRICT` prevent unauthorized cascade deletions at the database level
- The `q_open_to_me` and `q_owned` queries explicitly exclude sensitive fields (`salt`, `opened_key`) from results

## Cross-cutting Architecture Observations

1. **Documentation vs. Implementation Gap**: The application has good architectural documentation (`ARCHITECTURE.md`, `schema.md`) covering design intent, but lacks operational documentation required by ASVS Level 2-3 (communication inventories, resource management strategies, secrets rotation schedules).

2. **Single Database Pattern**: The use of SQLite as an embedded database simplifies many backend communication concerns (no network authentication needed, no connection pool management) but makes it impossible to apply traditional service-account or role-based database access controls.

3. **Trust Boundary Ambiguity**: The `TODO.md` file acknowledges multiple unimplemented security controls (`### check authz` placeholders, CSRF placeholder). The system currently relies primarily on the ASF OAuth committer-level authentication as its sole access control boundary, with no fine-grained authorization distinguishing election owners from other committers.

4. **Operational Tooling Without Controls**: CLI scripts (`create-election.py`, `mail-voters.py`) operate with the same privileges as the web server but without any authentication or authorization layer, creating a potential privilege escalation path for anyone with shell access.

5. **Consistent Cryptographic Approach**: The cryptographic design (Argon2 key derivation, separate salts per voter-issue pair, encrypted votes, anti-tamper keys) is well-documented and consistently applied, representing a strength in the security architecture.

## Cross-cutting Recommendations

### Immediate (Address within current sprint)
1. **Create a Communications Inventory document** listing all external services (OAuth, SMTP, LDAP), protocols, authentication methods, and network requirements — satisfies ASVS 13.1.1 at minimal implementation cost
2. **Document existing secrets** and their criticality in a secrets inventory, even without rotation automation — satisfies ASVS 13.1.4 documentation requirement

### Short-term (1-3 months)
3. **Add resource management documentation** defining timeouts, connection limits, and failure handling for SQLite, OAuth, and SMTP — satisfies ASVS 13.1.2 and 13.1.3
4. **Implement read-only database connections** for components that only need read access (mail script, voter-facing queries) — improves ASVS 13.2.2 compliance
5. **Implement TLS certificate rotation procedures** and document the rotation schedule

### Long-term (3-6 months)
6. **Evaluate migration from SQLite** to a database that supports role-based access control if the application scales beyond single-server deployment — would fully satisfy ASVS 13.2.1 and 13.2.2
7. **Implement connection lifecycle management** with explicit open/close patterns (context managers) for all database access paths to prevent resource leaks
8. **Add circuit breaker patterns** for OAuth and SMTP communication with documented fallback behavior
9. **Implement proper CSRF tokens and session-based secrets** with defined rotation (currently placeholder) — prerequisite for ASVS 13.1.4 compliance on session secrets
