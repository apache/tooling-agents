## ASVS-1.3.3: Sanitization Before Dangerous Contexts

### Findings for 1.3.3

### [HIGH] ASVS-133-HIGH-001: User-Supplied Filename Passed to HTML Context Without Encoding

**File:** `v3/server/pages.py`  
**Function:** `rewrite_description()` (lines 48-55)

**Vulnerable Code:**
```python
def repl(match):
    filename = match.group(1)
    return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
```

**Data Flow:** The `filename` is extracted from user-supplied description text via regex, then injected into both an HTML attribute context (href) and HTML text content context without URL-encoding or HTML-encoding respectively.

**Proof of Concept:**
```
description=See doc:" onmouseover="alert(1)" data-x=" for details
```

The regex captures `"` (not whitespace), producing:
```html
<a href="/docs/iid/" onmouseover="alert(1)" data-x="">" onmouseover="alert(1)" data-x="</a>
```

This breaks out of the href attribute and injects an event handler.

**Impact:** Attribute injection enabling XSS in all voters' browsers when viewing ballots.

**Remediation:**
```python
import html
from urllib.parse import quote

def repl(match):
    filename = match.group(1)
    safe_href = quote(filename, safe='')
    safe_text = html.escape(filename)
    return f'<a href="/docs/{issue.iid}/{safe_href}">{safe_text}</a>'
```

### [MEDIUM] ASVS-133-MED-001: Document Filename Not Validated Before Filesystem Access

**File:** `v3/server/pages.py`  
**Function:** `serve_doc()` (lines 557-574)

**Vulnerable Code:**
```python
@APP.get('/docs/<iid>/<docname>')
async def serve_doc(iid, docname):
    ...
    ### verify the propriety of DOCNAME.
    return await quart.send_from_directory(DOCSDIR / iid, docname)
```

**Data Flow:** `iid` and `docname` from URL → filesystem path construction. The developer explicitly flagged this with a TODO comment acknowledging the missing validation.

**Mitigating Factors:**
- `send_from_directory` uses Werkzeug's `safe_join` internally, which rejects `..` in `docname`
- The database check `db.q_get_mayvote.first_row(result.uid, iid)` ensures `iid` must exist in the database
- These mitigations make exploitation unlikely but the code lacks explicit defense-in-depth

**Impact:** If `safe_join` has an edge-case bypass or if the database is manipulated, arbitrary files could be served. The acknowledged TODO indicates awareness but no remediation.

**Remediation:**
```python
import os
import re

@APP.get('/docs/<iid>/<docname>')
async def serve_doc(iid, docname):
    result = await basic_info()
    
    # Validate IID format (should be hex/alphanumeric ID)
    if not re.match(r'^[a-zA-Z0-9_-]+$', iid):
        quart.abort(404)
    
    # Validate docname - only allow safe filenames
    if not re.match(r'^[a-zA-Z0-9_.-]+$', docname):
        quart.abort(404)
    
    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)
    
    return await quart.send_from_directory(DOCSDIR / iid, docname)
```

### [MEDIUM] ASVS-133-MED-002: Election Title and Issue Data Stored Without Length Limits

**File:** `v3/server/pages.py`  
**Functions:** `do_create_endpoint()` (line 424), `do_add_issue_endpoint()` (line 480), `do_edit_issue_endpoint()` (line 503)

**Vulnerable Code:**
```python
# do_create_endpoint
election = steve.election.Election.create(DB_FNAME, form.title, result.uid)

# do_add_issue_endpoint
iid = election.add_issue(form.title, form.description, vtype, kv)

# do_edit_issue_endpoint
election.edit_issue(issue.iid, form.title, form.description, issue.vtype, issue.kv)
```

**Data Flow:** `form.title` and `form.description` are passed directly to database storage without length validation or character filtering. The ASVS requirement specifically mentions "trimming input which is too long."

**Impact:** Excessively long inputs could cause denial of service (database bloat, template rendering issues, email subject overflow). Unrestricted characters could cause downstream issues in multiple contexts (HTML, email, logging).

**Remediation:**
```python
MAX_TITLE_LENGTH = 200
MAX_DESCRIPTION_LENGTH = 5000

form = edict(await quart.request.form)
title = form.title.strip()[:MAX_TITLE_LENGTH]
description = form.description.strip()[:MAX_DESCRIPTION_LENGTH]
```

### [LOW] ASVS-133-LOW-001: User Input in Log Messages Without Sanitization

**File:** `v3/server/pages.py`  
**Functions:** Multiple logging statements

**Vulnerable Code:**
```python
_LOGGER.info(
    f'User[U:{result.uid}] created election[E:{election.eid}];'
    f' title: "{form.title}"'
)
```

**Data Flow:** `form.title` (user-controlled) → log output. If logs are viewed in a context that interprets formatting (web-based log viewers, terminal escape sequences), this could enable log injection.

**Impact:** Log forgery, potential terminal escape injection, misleading audit trails.

**Remediation:**
```python
safe_title = form.title.replace('\n', '\\n').replace('\r', '\\r')
_LOGGER.info(f'User[U:{result.uid}] created election[E:{election.eid}]; title: "{safe_title}"')
```

## Cross-cutting Architecture Observations

1. **Defense-in-Depth Gap:** The application relies primarily on authentication (`@asfquart.auth.require`) as its sole security layer. There is no input sanitization layer, no output encoding layer, and no content security policy mentioned.

2. **Input → Storage → Rendering Pipeline:** User input (election titles, issue descriptions) flows from forms → database → templates/emails without sanitization at any point in the chain. This creates a systemic vulnerability where any context (HTML, email, logs) that renders stored data is potentially exploitable.

3. **TODO-Driven Security:** Multiple security concerns are acknowledged via comments (`### check authz`, `### verify the propriety of DOCNAME`, CSRF placeholder) but remain unimplemented. This creates a "Type B" gap — the team is aware of the need but the controls don't exist.

4. **Separation of Concerns:** The `mail-voters.py` script operates independently from the web layer, meaning data sanitized for web contexts may not be sanitized for email contexts (and vice versa). Sanitization should occur at storage time for multi-context safety.

5. **Template Engine Trust Boundary:** The code produces raw HTML strings (in `rewrite_description()`) and passes them to EZT templates, suggesting the templates render them unescaped. This inverts the normal trust model where templates provide auto-escaping.

## Cross-cutting Recommendations

### Immediate (Critical/High severity)

1. **Implement HTML sanitization in `rewrite_description()`** — Use `html.escape()` on the description content before wrapping in `<pre>` tags. URL-encode filenames in href attributes and HTML-escape them in text content.

2. **Sanitize election titles for SMTP safety** — Add CRLF stripping (`\r`, `\n`, `\x00`) to election title input at the point of creation in `do_create_endpoint()`.

3. **Add input length limits** — Enforce maximum lengths on `form.title` (200 chars) and `form.description` (5000 chars) in all form-processing endpoints.

### Short-term (Medium severity)

4. **Implement a centralized input sanitization utility** — Create a `sanitize` module with context-specific functions: `sanitize_for_html()`, `sanitize_for_email_header()`, `sanitize_for_filename()`, `sanitize_for_log()`.

5. **Validate `docname` parameter** — Add explicit allowlist validation (e.g., `^[a-zA-Z0-9._-]+$`) in `serve_doc()` before the filesystem operation.

6. **Add CSRF protection** — Replace the placeholder token with real generation/validation (noted in TODO.md but critical for POST endpoints that modify state).

### Long-term (Architecture)

7. **Adopt a sanitization-at-storage pattern** — Sanitize all user input when it enters the system (at form submission), ensuring stored data is safe for all downstream contexts.

8. **Migrate to auto-escaping templates** — Consider Jinja2 with auto-escaping enabled instead of EZT, which would provide default XSS protection for all template output.

9. **Implement Content Security Policy headers** — Add CSP headers to mitigate XSS impact even if injection occurs.

10. **Complete authorization model** — The `### check authz` placeholders represent a fundamental security gap. Implement ownership verification (e.g., only election owners can modify their elections).
