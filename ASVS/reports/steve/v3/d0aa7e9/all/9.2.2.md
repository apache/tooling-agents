## ASVS-9.2.2: Token Type and Purpose Validation

### Findings for 9.2.2

#### [MEDIUM]
**Finding ID**: ASVS-922-MED-001

**Title**: CSRF Token Placeholder Never Validated for Intended Purpose

**File**: `v3/server/pages.py`, function `basic_info()`, line 83

```python
### generate a real token and store in the session
basic.csrf_token = 'placeholder'
```

**Data flow**: CSRF token generated as static string → injected into template → sent in `X-CSRFToken` header by client → **never validated server-side**

**Description**: The CSRF "token" is defined as a static placeholder string. All POST endpoints (`/do-vote/<eid>`, `/do-create-election`, `/do-add-issue/<eid>`, `/do-edit-issue/<eid>/<iid>`, `/do-delete-issue/<eid>/<iid>`) accept requests without any CSRF token validation. The token exists in the templates (confirmed in `manage.ezt`: `<input type="hidden" id="csrf-token" value="[csrf_token]">`) and is sent in fetch headers (`'X-CSRFToken': csrfToken`), but no server-side handler validates it.

**Gap Type**: Type B (Control EXISTS in templates/client-side but NOT VALIDATED server-side — false confidence)

**Proof of Concept**: Any cross-origin request to `/do-vote/<eid>` will succeed because no CSRF token validation occurs:
```html
<form action="https://target/do-vote/abcdef0123" method="POST">
  <input name="vote-iid1234" value="yes">
  <input type="submit">
</form>
```

**Impact**: Attackers can forge votes, create/open/close elections, and modify issues on behalf of authenticated users via cross-site request forgery.

**Remediation**:
```python
async def validate_csrf():
    """Validate CSRF token from request against session."""
    s = await asfquart.session.read()
    token = quart.request.headers.get('X-CSRFToken') or (await quart.request.form).get('csrf_token')
    if not token or not s or token != s.get('csrf_token'):
        quart.abort(403, 'CSRF validation failed')

async def basic_info():
    # ...
    s = await asfquart.session.read()
    if s:
        # Generate real CSRF token
        if 'csrf_token' not in s:
            s['csrf_token'] = secrets.token_hex(32)
            await asfquart.session.write(s)
        basic.csrf_token = s['csrf_token']
```

#### [LOW]
**Finding ID**: ASVS-922-LOW-001

**Title**: Authentication Token Type Validation Not Verifiable in Provided Code

**File**: `v3/server/pages.py`, function `basic_info()`, lines 68-80

```python
s = await asfquart.session.read()
if s:
    basic.update(
        uid=s['uid'],
        name=s['fullname'],
        email=s['email'],
    )
```

**Description**: The application delegates all authentication token handling to `asfquart.session` and `asfquart.auth`. The provided code does not show any validation that distinguishes token types (e.g., access token vs. ID token vs. refresh token). The `basic_info()` function reads session data and uses it directly for authorization decisions (e.g., `result.uid` is used in database queries to determine voter eligibility). Without visibility into the `asfquart` module, it cannot be confirmed whether the underlying session framework validates that the token presented is of the correct type for the operation being performed.

**Impact**: If the framework uses self-contained tokens (JWTs) without type validation, a token intended for one purpose (e.g., refresh) could potentially be accepted for authentication decisions.

**Remediation**: Verify that `asfquart.auth` validates the `typ` claim (or equivalent) when processing tokens. Document the token validation chain for audit purposes.

### Positive Patterns for 9.2.2

1. **Vote token used correctly for single purpose**: The `vote_token` in `election.py` is generated via `crypto.gen_vote_token()` and exclusively used for vote encryption/lookup. It's never repurposed for authentication or other security decisions.
2. **State-based enforcement**: The Election class validates state before allowing operations (e.g., `_all_metadata(self.S_OPEN)` in `add_vote()` ensures votes are only accepted when the election is open).
3. **Sensitive data not returned**: `get_metadata()` explicitly excludes `salt` and `opened_key` from returned data, preventing internal tokens from leaking to presentation layer.

## Cross-cutting Architecture Observations

1. **Single-service architecture**: This is a monolithic web application, not a microservices system. The ASVS 9.2.x requirements about token type/audience validation are primarily designed for distributed systems where tokens flow between services. The relevance is reduced (but not eliminated) for this architecture.

2. **Authentication delegation**: All authentication and session management is delegated to `asfquart.session` and `asfquart.auth`. The security posture of the application heavily depends on the correctness of these external modules, which are not in scope for this audit.

3. **Internal token design is sound**: The `vote_token` system demonstrates good cryptographic design — tokens are derived from multiple inputs making them naturally scoped and resistant to cross-purpose reuse.

4. **Authorization gaps more critical than token issues**: The repeated `### check authz` placeholders throughout the code represent a more immediate threat than token validation issues. Any authenticated committer can manage/open/close/delete any election.

5. **CSRF as a token validation failure**: The placeholder CSRF token represents a concrete failure to validate a security token for its intended purpose (9.2.2), but the root cause is incomplete implementation rather than a design flaw.

## Cross-cutting Recommendations

### Immediate (Critical)
1. **Implement CSRF token validation**: Replace the placeholder with a cryptographically random token stored in the session, validated on all state-changing endpoints.
2. **Implement authorization checks**: Replace all `### check authz` placeholders with actual ownership/permission validation to ensure only election owners can manage their elections.

### Short-term (High)
3. **Document and verify framework token validation**: Audit `asfquart.auth` to confirm it validates token type and audience claims (if using JWTs/OIDC). Document the validation chain.
4. **Add audience configuration**: If the ASF OAuth system issues tokens for multiple services, configure this application with an expected audience identifier and validate it.

### Long-term (Medium)
5. **Security architecture documentation**: Document the complete authentication and authorization flow, including which components validate which token properties, to enable future audits.
6. **Consider token binding**: If transitioning to token-based authentication, implement sender-constrained tokens (DPoP or mTLS) to prevent token theft and replay across services.
