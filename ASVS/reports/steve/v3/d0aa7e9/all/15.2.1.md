## ASVS-15.2.1: Security Architecture and Dependencies - Update Timeframes

### Findings for 15.2.1

### [HIGH]

**Finding ID**: ASVS-1521-HIGH-001
**Location**: Project-wide (`v3/pyproject.toml`, `v3/ARCHITECTURE.md`)
**Issue**: No documented update or remediation timeframes for third-party components

**Evidence**:

The `ARCHITECTURE.md` states:
```markdown
- **Dependencies**: Managed via `uv` (Python package manager); includes Quart, asfpy, etc.
```

And the ASVS section description specifies:
> This section also emphasizes the importance of defining appropriate timeframes for addressing vulnerabilities in third-party components.

**No document in the codebase** defines:
- Maximum acceptable age for dependencies
- Timeframe for applying security patches (e.g., critical: 24h, high: 7d, medium: 30d)
- Process for monitoring dependency vulnerabilities
- Criteria for when a component must be updated or replaced

The `pyproject.toml` uses version constraints:
```toml
"cryptography>=46.0.5,<47",
"argon2-cffi>=25.1.0,<26",
```

Without a documented policy, there is no way to verify whether current component versions are within acceptable timeframes. The version constraints show some management but no documented cadence.

**Impact**: Without defined timeframes, vulnerability remediation becomes ad-hoc. Critical vulnerabilities in dependencies may persist beyond acceptable risk windows because there's no SLA for patching.

**Remediation**: Create a `SECURITY-POLICY.md` or add to `ARCHITECTURE.md`:
```markdown
## Dependency Update Policy

### Remediation Timeframes
| Severity | Timeframe | Action |
|----------|-----------|--------|
| Critical (CVSS 9.0+) | 48 hours | Immediate patch/upgrade |
| High (CVSS 7.0-8.9) | 7 days | Scheduled patch |
| Medium (CVSS 4.0-6.9) | 30 days | Next release cycle |
| Low (CVSS < 4.0) | 90 days | Planned maintenance |

### Monitoring
- Automated vulnerability scanning via GitHub Dependabot/Safety
- Monthly manual review of dependency health

### Current Component Status
All dependencies verified within timeframes as of: [DATE]
```

## Cross-cutting Architecture Observations

1. **Documentation Maturity Gap**: The project has good inline code documentation (`ARCHITECTURE.md`, `schema.md`, `tally.md`) but lacks formal security architecture documentation. Security decisions are documented in comments and TODOs rather than dedicated security documents.

2. **Development vs. Production Boundary**: Multiple indicators suggest the codebase is in an active development state being deployed (or ready to deploy) with development artifacts still present. The `TODO.md` essentially serves as both a bug tracker and security audit log.

3. **Authentication vs. Authorization**: The project correctly implements authentication (via ASF OAuth/session), but authorization (ownership checks, per-election access control) is consistently marked with placeholder comments (`### check authz`). This creates a Type B gap — the decorator exists but fine-grained authorization is not implemented.

4. **Cryptographic Design is Strong**: The vote encryption/anonymization design (Argon2 KDF, per-voter salts, encrypted ballots, anti-tamper hashing) is well-implemented. However, the deliberate slowness of these operations creates availability concerns that are only partially addressed.

5. **Dependency Management Awareness**: Version constraints show intentional dependency management, but no formal vulnerability tracking or update policy exists.
