## ASVS-8.3.1: Operation Level Authorization — Trusted Service Layer

### Findings for 8.3.1

### [CRITICAL]
**Finding ID**: ASVS-831-CRIT-001  
**Title**: State-Changing Actions Triggered via GET Requests Without Server-Side CSRF or Authorization Beyond Role  
**File**: `v3/server/pages.py`, lines 431 and 451

**Vulnerable Code (server-side):**
```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz           # <--- NOT IMPLEMENTED
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)        # Permanently opens election
    ...

@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz           # <--- NOT IMPLEMENTED
    election.close()          # Permanently closes election
    ...
```

**Client-side trigger (from template):**
```javascript
confirmOpen.addEventListener('click', () => {
    openModal.hide();
    window.location.href = '/do-open/[eid]';
});
```

**Analysis**: While the `@asfquart.auth.require({R.committer})` decorator provides server-side authentication, this is insufficient:
1. The "authorization" is merely role-based (any committer), not ownership-based
2. Using GET for state-changing operations violates HTTP semantics and makes CSRF trivial (no need to forge a POST)
3. The actual ownership check (`### check authz`) is not implemented

The client-side confirmation modal provides a UX safeguard but no security - it can be completely bypassed with a direct URL visit.

**Proof of Concept**: An attacker can embed this in any page a committer might visit:
```html
<img src="https://steve.apache.org/do-close/a1b2c3d4e5" style="display:none">
```

**Impact**: Any committer's browser can be tricked into closing/opening elections without consent. Combined with the IDOR issue, any election can be targeted.

**Remediation**:
1. Change to POST with CSRF token validation
2. Add ownership authorization check

```python
@APP.post('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
@require_csrf  # new decorator
async def do_open_endpoint(election):
    result = await basic_info()
    md = election.get_metadata()
    if md.owner_pid != result.uid:
        quart.abort(403)
    election.open(pdb)
```

### [HIGH]
**Finding ID**: ASVS-831-HIGH-001  
**Title**: CSRF Token is a Hardcoded Placeholder — No Server-Side Validation  
**File**: `v3/server/pages.py`, line 83

**Vulnerable Code:**
```python
async def basic_info():
    ...
    ### generate a real token and store in the session
    basic.csrf_token = 'placeholder'
    return basic
```

**Template usage (manage.ezt):**
```html
<input type="hidden" id="csrf-token" value="[csrf_token]">
```

```javascript
headers: { 'X-CSRFToken': csrfToken },
```

**Analysis**: 
- CSRF tokens ARE included in forms and fetch headers (positive intent)
- But the token is literally the string `'placeholder'` for all users/sessions
- No server-side validation of the token exists in any POST handler
- This is **Type C** — the control is CALLED (tokens sent) but the RESULT IS IGNORED (never validated)

**Impact**: All POST endpoints are vulnerable to CSRF attacks. An attacker can craft a form that submits votes, creates elections, or modifies issues on behalf of any logged-in user.

**Proof of Concept**:
```html
<form action="https://steve.apache.org/do-create-election" method="POST" id="csrf">
  <input name="csrf_token" value="placeholder">
  <input name="title" value="Attacker's Election">
</form>
<script>document.getElementById('csrf').submit();</script>
```

1. **Server-side authentication enforcement**: All sensitive endpoints use `@asfquart.auth.require` which executes on the server before any handler logic.
2. **Election state assertions**: Business logic in `election.py` uses `assert` statements to enforce state transitions server-side (e.g., `assert self.is_editable()` before modifications).
3. **Database-level constraints**: SQL schema enforces integrity (foreign keys, CHECK constraints, triggers like `prevent_open_close_update`).
4. **Parameterized queries**: All SQL uses `?` placeholders preventing SQL injection regardless of client input.

## Cross-cutting Architecture Observations

1. **Authorization Architecture Gap (Type B Pattern)**: The application has a clear architectural intent for object-level authorization — the database schema includes `owner_pid` and `authz` columns, `q_owned` queries filter by PID, and `### check authz` comments exist at 8+ locations. However, NONE of these placeholder checks are implemented. This creates false confidence: developers and auditors see the comments and may assume authorization is handled.

2. **Inconsistent Authorization Granularity**: 
   - `/vote-on/<eid>` (GET) verifies voter eligibility via `q_find_issues`
   - `/do-vote/<eid>` (POST) does NOT verify voter eligibility before attempting the vote
   - `/serve_doc/<iid>/<docname>` (GET) properly checks `q_get_mayvote`
   - Management endpoints check NOTHING beyond role

3. **GET for State Changes**: Using HTTP GET for `do-open` and `do-close` violates REST semantics and eliminates CSRF protection even if tokens were properly implemented (CSRF tokens protect POST/PUT/DELETE, not GET).

4. **Placeholder Security Controls**: The CSRF token system illustrates a dangerous anti-pattern — templates include tokens, JavaScript sends them in headers, but the server never validates them. This provides zero actual security while appearing secure during code review.

5. **Assertion-Based State Enforcement**: The `election.py` model uses Python `assert` statements for state checks. While these work in development, `assert` statements are removed when Python runs with optimization (`-O` flag). Production deployments MUST NOT use `-O` or these security checks disappear silently.

## Cross-cutting Recommendations

### Immediate (Critical — Address Before Production)

1. **Implement Object-Level Authorization**: Add ownership/authz checks in `load_election` and `load_election_issue` decorators. Compare `election.owner_pid` against `result.uid` (or check `election.authz` group membership).

2. **Implement CSRF Protection**: Replace the placeholder token with real per-session tokens. Add server-side validation in a `@before_request` handler or decorator for all POST endpoints.

3. **Change `do-open`/`do-close` to POST**: These perform irreversible state changes and must not be accessible via GET requests.

4. **Add Eligibility Check to `do_vote_endpoint`**: Replicate the `q_find_issues` check from `vote_on_page` before processing votes.

### Short-term (High Priority)

5. **Replace `assert` with Explicit Checks**: Replace `assert self.is_editable()` patterns with explicit `if not self.is_editable(): raise ElectionBadState(...)` to ensure enforcement regardless of Python optimization flags.

6. **Add Comprehensive Access Logging**: Log authorization denials (not just successful actions) to support security monitoring.

7. **Validate `docname` in `serve_doc`**: While `send_from_directory` likely prevents path traversal, add explicit validation per the code comment (`### verify the propriety of DOCNAME`).

### Long-term (Architecture Enhancement)

8. **Implement Adaptive Controls**: For Level 3 compliance, add IP/device tracking to sessions with re-authentication on context changes.

9. **Centralize Authorization Logic**: Create a dedicated authorization module rather than scattering checks across decorators and handlers. Consider a policy-based approach (e.g., checking owner, authz group, and election state in one call).

10. **Add Real-Time State Notifications**: Implement WebSocket or polling to notify active voting sessions when an election closes, preventing wasted user effort.
