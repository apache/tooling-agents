## ASVS-6.3.4: Multiple Authentication Pathways Consistency

### Findings for 6.3.4

And in decorators:
```python

    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
```

**Data flow:** Any authenticated committer → `/do-open/<eid>`, `/do-close/<eid>`, `/manage/<eid>`, etc. → No ownership check → Full access to any election

**Description:** The codebase contains 10+ instances of `### check authz` comments indicating the developers intended to implement ownership/authorization checks. These checks are NEVER implemented. The result is that ANY authenticated committer can:
- Open or close ANY election (not just ones they own)
- Add/edit/delete issues in ANY election
- Set dates on ANY election
- Access management pages for ANY election

Only `do_create_endpoint` requires `R.pmc_member`; all other management operations only require `R.committer`, creating an inconsistent and insufficient authorization model.

**Impact:** A malicious committer could tamper with elections they do not own — opening/closing elections prematurely, adding fake issues, or deleting legitimate issues. This breaks the trust model entirely.

**Proof of Concept:**
```http
GET /do-close/a1b2c3d4e5 HTTP/1.1
Cookie: session=<any_valid_committer_session>
```
This closes an election owned by a different user, with no authorization check.

**Remediation:**
```python
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            ...
        
        # Enforce ownership or authorized group membership
        result = await basic_info()
        md = e.get_metadata()
        if md.owner_pid != result.uid:
            # Check if user is in the authz group
            if md.authz and not user_in_group(result.uid, md.authz):
                quart.abort(403, 'Not authorized to manage this election')
        
        return await func(e)
    return loader
```

#### [HIGH] ASVS-634-HIGH-001
**Finding:** Inconsistent Authentication Levels Across Equivalent Operations

**File:** `v3/server/pages.py`, lines 431-449

**Vulnerable Code:**
```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    ...

@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_close_endpoint(election):
    ...
```

**Description:** State-changing operations that permanently alter elections (open, close) use GET requests with the same authentication level (`R.committer`) as read-only operations. Creating an election requires `R.pmc_member`, but opening/closing/managing one only requires `R.committer`. This is an undocumented inconsistency in authentication strength across pathways:

| Operation | Required Auth | Method |
|-----------|--------------|--------|
| Create election | `R.pmc_member` | POST |
| Open election (irreversible) | `R.committer` | GET |
| Close election (irreversible) | `R.committer` | GET |
| Delete issue | `R.committer` | POST | **Impact:** A lower-privileged user (committer vs PMC member) can perform irreversible operations on elections they didn't create.

**Remediation:** Ensure all election management operations require at minimum the same authorization level as creation, and add ownership verification.

## Cross-cutting Architecture Observations

1. **Authentication is externalized:** The actual login flow, credential verification, and session establishment are handled by `asfquart.auth` (not included in this codebase). Many ASVS 6.3 requirements would need to be assessed against that external system. The audited code only handles post-authentication authorization.

2. **Authorization is the critical gap:** The most severe finding is the systematic failure to implement ownership-based authorization (Type B gap — control is intended but never implemented). This spans all management endpoints and creates a flat permission model where any committer can modify any election.

3. **Separation of concerns:** The application cleanly separates:
   - Authentication (external, via `asfquart.auth`)
   - Session management (via `asfquart.session`)
   - Business logic (via `steve.election.Election`)
   - Cryptographic vote privacy (via `steve.crypto`)
   
   However, the authorization layer between authentication and business logic is entirely missing.

4. **Security-conscious cryptographic design:** The election system has sophisticated anti-tamper mechanisms (`is_tampered()`), vote encryption, and privacy-preserving tallying. This contrasts sharply with the weak authentication/authorization controls.

5. **TODO.md as documentation:** The application's own TODO file acknowledges CSRF and authorization gaps, confirming these are known issues awaiting resolution rather than oversights.

## Cross-cutting Recommendations

### Immediate (Critical Risk)
1. **Implement authorization checks** at all `### check authz` locations. Verify election ownership (`md.owner_pid == current_uid`) or group membership (`user in md.authz group`) before allowing management operations.
2. **Enforce MFA** via the external authentication provider or add an application-level MFA step after initial authentication.

### Short-term (High Risk)
3. **Implement CSRF protection** — replace the placeholder token with real per-session cryptographic tokens and validate on all state-changing endpoints.
4. **Change GET to POST** for state-changing operations (`/do-open/<eid>`, `/do-close/<eid>`) to prevent CSRF via simple link injection.
5. **Add audit logging** for person record changes and authentication events.

### Long-term (Medium Risk)
6. **Implement suspicious login detection** — track login metadata (IP, user agent, timestamp) and notify users of anomalies.
7. **Add credential change notifications** — send email alerts when person records are modified.
8. **Document all authentication pathways** formally, including the external auth provider's capabilities, MFA support, and integration points.
9. **Implement rate limiting** on authentication-related endpoints to prevent brute-force attacks (per NIST SP 800-63B § 5.2.2).
