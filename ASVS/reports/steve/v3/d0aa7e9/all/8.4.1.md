## ASVS-8.4.1: Multi-tenant Cross-Tenant Controls

### Findings for 8.4.1

### [CRITICAL]
**Finding ID:** ASVS-841-CRIT-001
**Location:** `v3/server/pages.py` - `load_election` decorator (line 165), `do_open_endpoint` (line 431), `do_close_endpoint` (line 451), `do_add_issue_endpoint` (line 469), `do_edit_issue_endpoint` (line 493), `do_delete_issue_endpoint` (line 516)

**Vulnerable Code:**
```python
def load_election(func):
    """Decorator to load/pass-argument an Election from EID."""

    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            # ...
            raise_404(T_BAD_EID, result)

        _LOGGER.debug(f'Loaded: {e}')

        ### check authz   # <-- NOT IMPLEMENTED

        return await func(e)

    return loader
```

**Data Flow:** Authenticated committer → provides any EID → `load_election` loads election without ownership check → management operation executes on another user's election

**Gap Type:** Type B — Control placeholder EXISTS but NOT CALLED. The `### check authz` comment indicates the developer intends to verify that the current user is the owner or is in the `authz` group, but this is never implemented.

**Proof of Concept:**
Any authenticated committer can:
1. `GET /do-open/a1b2c3d4e5` → Opens another owner's election
2. `GET /do-close/a1b2c3d4e5` → Closes another owner's election
3. `POST /do-add-issue/a1b2c3d4e5` → Adds issues to another owner's election
4. `POST /do-delete-issue/a1b2c3d4e5/f6g7h8i9j0` → Deletes issues from another owner's election

**Impact:** Complete cross-tenant boundary violation. Any authenticated committer can manage, open, close, or delete any election created by any other user. This breaks election integrity and allows sabotage of other users' elections.

**Remediation:**
```python
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            e = steve.election.Election(DB_FNAME, eid)
        except steve.election.ElectionNotFound:
            result = await basic_info()
            result.title = 'Unknown Election'
            result.eid = eid
            raise_404(T_BAD_EID, result)

        result = await basic_info()
        md = e.get_metadata()
        
        # Enforce ownership or authz group membership
        if md.owner_pid != result.uid:
            if not md.authz or not user_in_authz_group(result.uid, md.authz):
                quart.abort(403)
        
        return await func(e)
    return loader
```

### [HIGH]
**Finding ID:** ASVS-841-HIGH-001
**Location:** `v3/server/pages.py` - `do_vote_endpoint` (line 372)

**Vulnerable Code:**
```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()

    ### check authz   # <-- NOT IMPLEMENTED

    form = edict(await quart.request.form)
    # ...
    for iid, votestring in votes.items():
        if iid not in issue_dict:
            await flash_danger(f'Invalid issue ID: {iid}')
            return quart.redirect(f'/vote-on/{election.eid}', code=303)

        try:
            election.add_vote(result.uid, iid, votestring)
```

**Data Flow:** Authenticated committer → submits vote for any election EID → `add_vote` calls `q_get_mayvote` which DOES check eligibility at the data layer (line 237 of election.py)

**Mitigating Factor:** The `add_vote` method in `election.py` does query `q_get_mayvote.first_row(pid, iid)` which would fail if the user isn't eligible to vote. However, this failure path is not gracefully handled — if `mayvote` is `None`, `mayvote.salt` would raise an `AttributeError`, not a clean denial.

**Impact:** Unhandled exception rather than clean access denial when ineligible user attempts to vote. This could cause server errors but not data corruption.

**Remediation:**
```python
async def do_vote_endpoint(election):
    result = await basic_info()
    
    # Verify user is eligible to vote in this election
    election.q_find_issues.perform(result.uid, election.eid)
    if not election.q_find_issues.fetchall():
        quart.abort(403)
    # ...
```

## Cross-cutting Architecture Observations

1. **Authorization Model Gap**: The system has a well-designed data model for authorization (`owner_pid`, `authz` fields, `mayvote` table) but the web layer does not enforce ownership boundaries. This creates a Type B vulnerability pattern — the security infrastructure exists but is not applied.

2. **Defense-in-Depth Absence**: The application relies on a single authentication boundary (`R.committer`). Once past this boundary, all operations within the system are available regardless of role or ownership. The only exception is `do_create_election` which requires `R.pmc_member`.

3. **Cryptographic Architecture Strength**: The cryptographic design is sound — Argon2 for key derivation, Fernet for authenticated encryption, HKDF for key stretching, cryptographic shuffling for vote privacy. The security issues are entirely in the authorization/access-control layer, not the cryptographic layer.

4. **No Self-Contained Auth Tokens**: The system does not use JWTs or similar self-contained tokens for authentication/authorization. This means ASVS 9.x requirements for token validation are largely not applicable, with security relying instead on server-side session management (external `asfquart.session`).

5. **State Machine as Security Control**: The election state machine (`editable` → `open` → `closed`) enforced via assertions provides a secondary authorization layer for operations — certain operations are only possible in certain states, regardless of who requests them.

## Cross-cutting Recommendations

### Immediate (Critical)

1. **Implement ownership verification in `load_election`**: Compare `election.owner_pid` against `result.uid` and check `authz` group membership. This single change fixes ASVS-841-CRIT-001 across all endpoints using the decorator.

2. **Replace `assert` with proper access control**: Python assertions can be disabled with `-O` flag. Replace `assert self.is_editable()` with proper exception raising:
   ```python
   if not self.is_editable():
       raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
   ```

3. **Convert state-changing GETs to POSTs**: `/do-open/<eid>` and `/do-close/<eid>` must be POST endpoints to prevent CSRF via simple link/image attacks.

### Short-term (High)

4. **Implement CSRF protection**: Replace placeholder token with cryptographic CSRF tokens validated on all state-changing endpoints.

5. **Create separate admin decorator**: Build a `load_election_admin` decorator that verifies ownership/authz, distinct from voter-facing `load_election`:
   ```python
   def load_election_admin(func):
       """Load election and verify administrative access."""
       @functools.wraps(func)
       async def loader(eid):
           e = steve.election.Election(DB_FNAME, eid)
           result = await basic_info()
           md = e.get_metadata()
           if md.owner_pid != result.uid and not check_authz(result.uid, md.authz):
               quart.abort(403)
           return await func(e)
       return loader
   ```

6. **Verify session token expiration**: Audit `asfquart.session` to confirm tokens have bounded lifetimes.

### Long-term (Architecture)

7. **Implement multi-layer admin security**: For administrative operations, add step-up authentication (re-verification for destructive operations), contextual risk scoring (new IP/device detection), and rate limiting on management operations.

8. **Implement proper RBAC system**: Resolve the `authz` field usage — integrate with LDAP group membership verification to support the multi-tenant model described in the schema documentation.

9. **Add audit logging with context**: Enhance logging to include IP addresses, device fingerprints, and request context for forensic analysis of administrative actions.
