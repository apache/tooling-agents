## ASVS-14.2.4: Controls Around Sensitive Data

### Findings for 14.2.4

### [HIGH]

**Finding ID:** ASVS-1424-HIGH-001

**Title:** Debug Print Statements Expose Sensitive Form Data to Server Logs/Output

**File:** `v3/server/pages.py`  
**Functions:** `do_add_issue_endpoint` (line 481), `do_edit_issue_endpoint` (line 497)

**Vulnerable Code:**
```python
@APP.post('/do-add-issue/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_add_issue_endpoint(election):
    result = await basic_info()
    form = edict(await quart.request.form)
    print('FORM:', form)  # Dumps entire form data to stdout/logs
    # ...
```

**Data Flow:** User form submission → `quart.request.form` → `print()` → stdout/container logs/journald → potential log aggregation systems → **uncontrolled retention**.

**Impact:** Form data (including issue titles, descriptions, and any future sensitive fields) is dumped to server stdout without any log protection controls. In containerized environments, stdout is typically captured by logging infrastructure (e.g., CloudWatch, Stackdriver, ELK) with potentially different retention and access policies than the application database. This violates the requirement that sensitive data logging follows defined protection level controls.

**Remediation:**
```python
# Remove debug prints entirely:
# print('FORM:', form)  # DELETE THIS

# If debug logging is needed, use structured logging with appropriate level:
_LOGGER.debug(f'Issue form received for election[E:{election.eid}]')
# Never log the form contents
```

### [MEDIUM]

**Finding ID:** ASVS-1424-MED-001

**Title:** Vote Logging Records User-to-Issue Mapping Without Documented Retention or Access Controls

**File:** `v3/server/pages.py`, lines 401-407  
**Function:** `do_vote_endpoint`

**Vulnerable Code:**
```python
try:
    election.add_vote(result.uid, iid, votestring)
    _LOGGER.info(
        f'User[U:{result.uid}] voted on issue[I:{iid}] in election[E:{election.eid}]'
    )
```

**Data Flow:** Vote submission → `_LOGGER.info()` → application log files → potential log aggregation → **no documented retention policy**.

**Impact:** The logging creates a record that links a specific user (UID) to a specific issue (IID) and election (EID), which when correlated with the vote table could potentially deanonymize votes. While the log does not contain the actual vote value, it records that a user voted on a specific issue at a specific time. The vote system is designed to be anonymous (encrypted votes, shuffled tallying), but this log creates a side-channel. No documented retention policy or access control mechanism for these logs is present.

**Note:** This is noted as medium because the vote value itself is not logged, and the system design already links voters to issues via the `mayvote` table. However, the timing information in logs adds additional correlation data.

**Remediation:** Document log retention policy. Consider reducing vote logging to election-level granularity:
```python
_LOGGER.info(f'User[U:{result.uid}] submitted votes for election[E:{election.eid}]')
# Don't log individual issue-level voting activity
```

### [MEDIUM]

**Finding ID:** ASVS-1424-MED-002

**Title:** Fernet Encryption for Vote Ciphertext Does Not Provide AEAD with Associated Data Binding

**File:** `v3/steve/crypto.py`, lines 65-70  
**Function:** `create_vote`

**Vulnerable Code:**
```python
def create_vote(vote_token: bytes, salt: bytes, votestring: str) -> bytes:
    "Encrypt VOTESTRING using the VOTE_TOKEN and SALT."
    b64key = _b64_vote_key(vote_token, salt)
    f = cryptography.fernet.Fernet(b64key)
    return f.encrypt(votestring.encode())
```

**Impact:** Fernet provides authenticated encryption but does not support associated data (AAD). The ciphertext is not cryptographically bound to the vote_token or IID, meaning that while the key derivation provides implicit binding, there is no explicit integrity check that a ciphertext belongs to a specific (person, issue) pair at the crypto level. The documentation notes a planned transition to XChaCha20-Poly1305, which also lacks native AAD support without additional framing. The HKDF `info` parameter (`b'xchacha20_key'`) provides some domain separation but doesn't bind to the specific vote context.

**Remediation:** When transitioning to XChaCha20-Poly1305, use the AEAD associated data field to bind ciphertext to the vote_token:
```python
# Future: Use XChaCha20-Poly1305 with AAD
from cryptography.hazmat.primitives.ciphers.aead import ChaCha20Poly1305
# Include vote_token as associated data for binding
aead = ChaCha20Poly1305(key)
ciphertext = aead.encrypt(nonce, votestring.encode(), associated_data=vote_token)
```

### Positive Patterns for 14.2.4

- **Votes are never logged in plaintext** — `add_vote()` encrypts before storage, and the log only records that a vote occurred.
- **Salt/opened_key exclusion** — `get_metadata()` explicitly omits cryptographic material from public-facing responses.
- **Cryptographic shuffle** — `crypto.shuffle()` uses `secrets.randbelow()` (CSPRNG) to randomize vote order before tallying, preventing database-order information leakage.
- **Argon2 key derivation** — `_hash()` uses Argon2 with appropriate parameters (64MB memory, 2 iterations, parallelism 4) providing resistance against brute-force attacks on vote tokens.
- **BLAKE2b pre-hashing** — Election data is reduced to 64 bytes via BLAKE2b before Argon2 processing, efficiently handling arbitrary-length election definitions.
- **Anti-tamper detection** — `is_tampered()` recomputes the opened_key from current election data and compares to the stored value, detecting unauthorized modifications to issues, voters, or metadata.

## Cross-cutting Architecture Observations

1. **Security Control Consistency Gap**: The application has strong cryptographic controls (Argon2, Fernet encryption, BLAKE2b, CSPRNG for IDs and shuffling) but lacks fundamental HTTP-layer security headers (Cache-Control, X-Content-Type-Options). This suggests the security design focused on cryptographic vote integrity but did not address transport/caching layer concerns.

2. **Internal vs. External API Surface**: The `Election._all_metadata()` method is documented as "INTERNAL ONLY" and returns sensitive fields (salt, opened_key), while `get_metadata()` is the safe public API. This separation is correct but relies on convention rather than enforcement (no access modifier, just naming convention with underscore prefix).

3. **Authorization Gap (Cross-cutting)**: Multiple endpoints have `### check authz` placeholder comments. While authentication is enforced via decorators, fine-grained authorization (e.g., only election owner can open/close/manage) is not implemented. This affects multiple ASVS requirements indirectly — if an unauthorized user can access management endpoints, they could view sensitive election data that should be minimized.

4. **Data Flow Segregation**: The system properly segregates sensitive cryptographic material (salt, opened_key) from public-facing queries at the SQL level (separate queries for internal vs. external use). This is a strong architectural pattern.

5. **Template Security**: The EZT templating engine used for HTML rendering requires careful attention to output encoding. The `vote-on.ezt` template includes a client-side `escapeHtml()` function for JavaScript-rendered content, but server-side rendering relies on EZT's default behavior.

6. **Operational Tooling**: The `mail-voters.py` script runs as a standalone process with direct database access. It accesses voter PII (email addresses) without the web application's session/auth layer, relying instead on OS-level access controls to the script and database file.

## Cross-cutting Recommendations

### Immediate (Within Sprint)

1. **Remove debug print statements** (`pages.py` lines 481, 497) — Direct PII/data exposure risk.
2. **Add `Cache-Control: no-store, private` header** to all authenticated responses via an `@APP.after_request` handler.
3. **Add `X-Content-Type-Options: nosniff` header** globally.

### Short-term (Within Quarter)

4. **Implement authorization checks** at all `### check authz` placeholder locations — verify the authenticated user is the election owner or in the authorized group before allowing management operations.
5. **Document data retention policy** — classify each data type (elections, votes, persons) with explicit retention periods and deletion triggers.
6. **Implement CSRF token validation** — replace the placeholder token with a real per-session token and validate on all POST endpoints.
7. **Validate `docname` parameter** in `serve_doc()` — restrict to safe filename characters.

### Long-term (Within Year)

8. **Implement automated data purge** — scheduled job to delete closed elections past retention period and purge superseded votes after tally certification.
9. **Complete migration to XChaCha20-Poly1305** with AAD binding vote ciphertext to vote_token.
10. **Implement log classification** — ensure logs containing user-action mappings have appropriate retention limits and access controls distinct from application data.
11. **Add per-response caching policy** — static assets should have appropriate cache headers (`max-age`), while dynamic authenticated content should have `no-store`.
