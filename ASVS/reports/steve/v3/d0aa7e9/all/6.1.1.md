## ASVS-6.1.1: Authentication Documentation - Rate Limiting and Anti-Automation

### Findings for 6.1.1

### [HIGH]

**Finding ID**: ASVS-611-HIGH-001

**File**: `v3/server/pages.py` (entire file, and system architecture)

**Description**: No rate limiting, anti-automation, or adaptive response controls are implemented or documented anywhere in the application.

**Evidence of absence:**
```python
# All authentication endpoints use only the decorator-based auth:
@APP.get('/voter')
@asfquart.auth.require({R.committer})  ### need general solution

@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})

@APP.post('/do-create-election')
@asfquart.auth.require({R.pmc_member})  ### need general solution
```

No middleware, decorator, or configuration for:
- Request rate limiting
- Login attempt throttling
- Account lockout protection
- Anti-automation (CAPTCHA, behavioral analysis)
- Adaptive response to suspicious activity

**Data flow**: Unauthenticated request → ASF auth system → application endpoints → no throttling at application layer

**Proof of concept**: An attacker can make unlimited rapid requests to any endpoint. While authentication is delegated to ASF infrastructure, the application layer has no defense against:
- Credential stuffing attacks against the ASF auth gateway
- Automated vote manipulation attempts (rapid `/do-vote/<eid>` submissions)
- Election state manipulation (rapid `/do-open/<eid>`, `/do-close/<eid>` requests)

**Impact**: Without rate limiting documentation or implementation, there is no defense-in-depth against brute force or automated attacks. Additionally, the absence of documentation means security teams cannot verify the adequacy of controls or configure them correctly.

**Remediation**: Document and implement rate limiting controls:

```python
# Example documentation (to be placed in docs/security-controls.md):
"""
## Rate Limiting and Anti-Automation Controls

### Configuration
- Login attempts: 5 per minute per IP, 10 per hour per account
- Vote submissions: 10 per minute per authenticated user
- Election state changes: 1 per 5 seconds per user
- API endpoints: 60 requests per minute per authenticated user

### Lockout Prevention
- Account lockout is NOT performed at the application layer
  (delegated to ASF LDAP/OAuth infrastructure)
- IP-based rate limits use progressive delays (1s, 5s, 30s, 300s)
- Rate limit headers (X-RateLimit-*) are returned to clients

### Adaptive Response
- 10+ failed auth attempts from single IP: temporary IP block (15 min)
- Unusual voting patterns: alert to election administrator
"""

# Implementation example using quart-rate-limiter:
from quart_rate_limiter import RateLimiter, rate_limit

rate_limiter = RateLimiter(APP)

@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@rate_limit(10, timedelta(minutes=1))
@load_election
async def do_vote_endpoint(election):
    ...
```

## Cross-cutting Architecture Observations

1. **Authentication Delegation Pattern**: The application consistently delegates authentication to ASF infrastructure (`asfquart.auth`). This means ASVS requirements 6.2.1 and 6.2.10 (password security) are the responsibility of the external identity provider, not this application. However, documentation requirements (6.1.1, 6.1.2, 6.1.3) still apply to the application.

2. **Authorization Gap (Type B — Control EXISTS but NOT CALLED)**: The code has 9+ instances of `### check authz` placeholder comments. While authentication (identity verification) is enforced, authorization (permission verification — e.g., "is this user the election owner?") is NOT implemented. Any committer can manage any election, open/close elections they don't own, and delete others' issues. This is a critical Type B gap.

3. **CSRF Protection is a Placeholder**: `basic.csrf_token = 'placeholder'` (line 83) combined with no server-side validation means all state-changing endpoints are vulnerable to CSRF. The template includes the token in forms/headers, creating false confidence (Type B gap).

4. **State-Changing GET Requests**: `/do-open/<eid>` and `/do-close/<eid>` use GET methods for state-changing operations, making CSRF exploitation trivial (no form submission needed — a simple `<img>` tag suffices).

5. **File Serving Pipeline**: Documents flow from upload (not shown in provided code) → storage in `DOCSDIR` → serving via `serve_doc()` with no content inspection or antivirus scanning at any stage.

## Cross-cutting Recommendations

### Immediate (Critical/High Priority)

1. **Implement authorization checks**: Replace all `### check authz` comments with actual ownership/permission verification:
   ```python
   if election.get_metadata().owner_pid != result.uid:
       quart.abort(403)
   ```

2. **Implement CSRF protection**: Replace placeholder token with a real implementation (e.g., `quart-csrf` or similar), and validate on all POST endpoints.

3. **Convert state-changing GETs to POSTs**: `/do-open/<eid>` and `/do-close/<eid>` must use POST method with CSRF validation.

4. **Document authentication pathways**: Create a security document covering all auth levels, their required controls, and configuration guidance.

### Short-term (Medium Priority)

5. **Implement rate limiting**: Add request rate limiting middleware (e.g., `quart-rate-limiter`) with documented thresholds for authentication, voting, and administrative operations.

6. **Add antivirus scanning**: Integrate ClamAV or similar scanner into the document upload pipeline and/or download endpoint.

7. **Validate `docname` in `serve_doc()`**: While `send_from_directory` provides some protection, add explicit path traversal validation:
   ```python
   safe_path = (DOCSDIR / iid / docname).resolve()
   if not safe_path.is_relative_to(DOCSDIR / iid):
       quart.abort(404)
   ```

8. **Document context-specific password blocklist**: Even though authentication is delegated, document the blocklist and communicate it to the ASF identity management team.

### Long-term (Low Priority)

9. **Remove debug statements**: Remove `print('FORM:', form)` from production endpoints.

10. **Add Content-Security-Policy headers**: For served documents, add appropriate headers to prevent execution of embedded scripts.

11. **Implement comprehensive security documentation**: Cover all ASVS documentation requirements in a single security architecture document.
