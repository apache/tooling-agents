## Cross-cutting Architecture Observations

1. **OAuth AS Not In Scope**: The entire set of ASVS 10.4.x requirements targets OAuth Authorization Server functionality. The provided codebase is an **election/voting application** that *consumes* authentication via the `asfquart` framework but does not *implement* an OAuth Authorization Server. The actual OAuth AS (likely an Apache infrastructure component) is not included in the provided source files.

2. **Authentication Delegation**: The application delegates authentication to `asfquart.auth` with role-based requirements (`R.committer`, `R.pmc_member`). The implementation of these checks, including any underlying OAuth/OIDC flows, is external to this codebase.

3. **Session Management**: The application uses `asfquart.session.read()` to obtain user identity (`uid`, `fullname`, `email`). This session is presumably established after successful OAuth authentication upstream, but the OAuth token exchange, code validation, and refresh mechanisms are handled elsewhere.

4. **Audit Scope Limitation**: To properly audit ASVS 10.4.x requirements, the OAuth Authorization Server implementation used by this application (likely part of the `asfquart` or Apache ID infrastructure) would need to be included in the review scope.

## Cross-cutting Recommendations

**Immediate:**
- No immediate action required for ASVS 10.4.x within this codebase, as it does not implement OAuth AS functionality.

**Short-term:**
- **Identify the OAuth AS component**: Determine which system provides OAuth authorization server functionality for this application (e.g., Apache's OAuth infrastructure or a third-party identity provider) and audit that component against ASVS 10.4.x requirements.
- **Document the trust boundary**: Explicitly document the dependency on the external authentication system and what security guarantees it provides (authorization code one-time use, PKCE enforcement, refresh token rotation, etc.).

**Long-term:**
- **Ensure upstream compliance**: Verify that the upstream OAuth AS (consumed by `asfquart.auth`) meets all ASVS 10.4.x requirements, particularly:
  - Authorization code one-time use with revocation on reuse (10.4.2)
  - Authorization code lifetime ≤ 10 minutes (10.4.3)
  - No implicit or ROPC grant support (10.4.4)
  - Refresh token replay protection via rotation or sender-constraining (10.4.5)
  - PKCE enforcement with S256 (10.4.6)
- **Supply chain audit**: Include the `asfquart` authentication framework source code in future security audits to verify OAuth security controls are properly implemented at the authorization server level.
