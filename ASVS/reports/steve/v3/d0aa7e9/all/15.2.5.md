## ASVS-15.2.5: Additional Protections Around Dangerous Functionality

### Findings for 15.2.5

### [MEDIUM]

**Finding ID:** ASVS-1525-MED-001

**Title:** No Sandboxing or Isolation for Cryptographic and Sensitive Election Operations

**File:** `v3/steve/election.py` (entire module), `v3/server/pages.py` (entire module)

**Issue:** The application contains highly sensitive functionality (cryptographic vote encryption/decryption, election state management, tamper detection) that runs within the same process and trust boundary as the web server and administrative functions. There is no evidence of:

1. **Process isolation**: Cryptographic operations (`steve.crypto`) run in the same process as the web handler
2. **Network isolation**: Admin endpoints (`/do-open/<eid>`, `/do-close/<eid>`, `/do-delete-issue/`) are served by the same application instance as voter endpoints (`/do-vote/<eid>`)
3. **Containerization**: No Docker/container configuration visible
4. **Privilege separation**: The `open()`, `close()`, and `tally_issue()` methods (which handle cryptographic material) execute with the same privileges as `serve_static()`
5. **File system isolation**: `serve_doc()` accesses the filesystem from the same context that handles election encryption keys

**Relevant dangerous operations:**
- `election.open(pdb)` — generates cryptographic salts and opened_key (`election.py` line 70)
- `election.add_vote()` — encrypts votes (`election.py` line 231)
- `election.tally_issue()` — decrypts all votes (`election.py` line 248)
- `serve_doc()` — serves files from filesystem (`pages.py` line 560)

**Impact:** If an attacker compromises one part of the application (e.g., through the document serving endpoint or a template injection), they could pivot to access cryptographic keys, decrypt votes, or tamper with elections.

**Remediation:**
1. Separate admin/management functions from voter-facing endpoints into distinct services
2. Run cryptographic operations (tally, vote encryption) in an isolated subprocess or container
3. Use network policies to restrict which components can access the database directly
4. Implement least-privilege access: the web-facing service should not have direct access to `opened_key` values
5. Example architecture:
```
[Voter Service] → [Vote API] → [Crypto Service (isolated)]
                                        ↓
[Admin Service] → [Admin API] → [Election DB (restricted)]
```

### [MEDIUM]

**Finding ID:** ASVS-1525-MED-002

**Title:** State-Changing Operations Exposed via GET Requests Without CSRF Protection

**File:** `v3/server/pages.py`, lines 431-466

**Vulnerable code:**
```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    # ...
    election.open(pdb)
    # ...

@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_close_endpoint(election):
    # ...
    election.close()
    # ...
```

**Data flow:** Browser GET request (or attacker-controlled link/img src) → no CSRF validation → permanent election state change

**Proof of Concept:**
```html
<img src="https://steve.apache.org/do-open/a1b2c3d4e5" />
<img src="https://steve.apache.org/do-close/a1b2c3d4e5" />
```

**Issue:** Opening and closing elections are irreversible operations performed via GET requests with no CSRF protection. Combined with the placeholder CSRF token (`basic.csrf_token = 'placeholder'`), these "dangerous functions" have no additional protection layer.

**Impact:** An attacker can permanently open or close an election by tricking an authenticated admin into visiting a page with a crafted image tag or link.

**Remediation:** Convert to POST with CSRF validation:
```python
@APP.post('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    validate_csrf_token(await quart.request.form)
    # ... proceed with opening
```

## Cross-cutting Architecture Observations

1. **Authentication vs. Authorization Gap**: The application has authentication (`@asfquart.auth.require`) but lacks fine-grained authorization. All committers can manage any election — there are multiple `### check authz` placeholder comments indicating this is known but unimplemented. This creates a horizontal privilege escalation risk.

2. **CSRF Protection Entirely Missing**: The `csrf_token = 'placeholder'` on line 83 of `pages.py` means every state-changing endpoint is vulnerable to CSRF. This affects multiple requirements:
   - 15.2.5: No protection for dangerous operations (open/close election)
   - 15.3.3: Mass assignment protection is weakened when attackers can submit forms cross-origin

3. **Defense in Depth Architecture**: The application relies on a single-layer architecture where the web server, business logic, and cryptographic operations all run in the same process. This maximizes blast radius from any single vulnerability.

4. **State Machine as Security Control**: The `assert`-based state machine in `election.py` (editable → open → closed) provides an important security boundary. However, `assert` statements can be disabled with Python's `-O` flag, making this a fragile control for production.

5. **Query Design Philosophy**: The codebase shows two competing patterns — some queries (external-facing) explicitly select safe columns, while others (internal) use `SELECT *`. This split is well-documented but creates maintenance risk.

6. **Session-Based Identity**: The consistent use of `result.uid` from authenticated sessions for identity (rather than client-supplied values) is a strong positive pattern across all endpoints.

## Cross-cutting Recommendations

### Immediate (Critical/High Priority)
1. **Implement CSRF protection**: Replace `'placeholder'` with actual token generation and validation on all state-changing endpoints
2. **Convert GET state-changers to POST**: `/do-open/<eid>` and `/do-close/<eid>` must be POST endpoints
3. **Implement authorization checks**: Fill in all `### check authz` placeholders — at minimum, verify that the authenticated user is the election owner before allowing management operations
4. **Replace `assert` with explicit checks**: State validation should use `if not ... : raise` rather than `assert` which can be optimized away

### Short-term (Medium Priority)
5. **Configure trusted proxy middleware**: Add ProxyFix or equivalent with appropriate trust depth
6. **Add IP addresses to audit logs**: Include client IP (from trusted proxy chain) in security-relevant log entries
7. **Replace `SELECT *` queries**: Use explicit column lists in all queries, especially those touching tables with cryptographic material
8. **Implement formal input schemas**: Add dataclass or Pydantic-based input validation with explicit field allowlists
9. **Remove debug print statements**: Replace with structured logging at appropriate levels
10. **Pin dependency sources**: Add repository configuration and generate/commit lock file with hashes

### Long-term (Architecture)
11. **Service separation**: Split admin, voter, and cryptographic operations into separate services with network isolation
12. **Containerization**: Deploy with container-level isolation between components
13. **Least privilege**: The crypto/tally service should be the only component with access to opened_key values
14. **Rate limiting**: Implement rate limiting using correctly-resolved client IPs
15. **Dependency scanning**: Integrate automated dependency vulnerability scanning (e.g., `pip-audit`, `safety`) into CI/CD
