## ASVS-11.2.2: Crypto Agility

### Findings for 11.2.2

#### [HIGH]
**Finding ID:** ASVS-1122-HIGH-001  
**Title:** No Crypto Agility - Algorithms Hardcoded Without Abstraction or Versioning

**Location:** `v3/steve/crypto.py` (entire file), `v3/schema.sql`

**Vulnerable code:**
```python
# crypto.py - All algorithms hardcoded with no abstraction layer

def gen_opened_key(edata: bytes, salt: bytes) -> bytes:
    digest = hashlib.blake2b(edata).digest()  # Hardcoded BLAKE2b
    return _hash(digest, salt)  # Hardcoded Argon2d

def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),  # Hardcoded SHA-256
        length=32,
        salt=salt,
        info=b'xchacha20_key',  # Hardcoded info string
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)

def create_vote(vote_token: bytes, salt: bytes, votestring: str) -> bytes:
    b64key = _b64_vote_key(vote_token, salt)
    f = cryptography.fernet.Fernet(b64key)  # Hardcoded Fernet
    return f.encrypt(votestring.encode())
```

```sql
-- schema.sql - No algorithm version field stored with ciphertext
CREATE TABLE vote (
    vid  INTEGER PRIMARY KEY AUTOINCREMENT,
    vote_token  BLOB NOT NULL CHECK (length(vote_token) = 32),
    ciphertext  BLOB NOT NULL  -- No algorithm identifier
) STRICT;
```

**Description:**  
The application has no crypto-agility design. Specific issues:

1. **No algorithm abstraction layer:** All crypto operations directly call specific implementations with hardcoded parameters. There's no configuration, registry, or strategy pattern that would allow swapping algorithms.

2. **No algorithm versioning in stored data:** The `vote` table stores `ciphertext` without any algorithm identifier. When transitioning from Fernet to XChaCha20-Poly1305, there will be no way to determine which algorithm was used for existing ciphertext without external tracking.

3. **No key versioning:** If keys need to be rotated or algorithms changed, there's no version field to indicate which key/algorithm generated a given ciphertext.

4. **No re-encryption mechanism:** No tooling exists to decrypt existing votes with the old algorithm and re-encrypt with a new one.

5. **Fixed key lengths in schema:** Database CHECK constraints enforce exact lengths (`length(vote_token) = 32`, `length(opened_key) = 32`, `length(salt) = 16`), preventing algorithm changes that require different sizes without schema migration.

**Impact:** Algorithm upgrades require code changes, database schema migrations, and potential data loss. The planned Fernet → XChaCha20-Poly1305 transition will require careful coordination. Future PQC migrations will be significantly more complex without agility designed in.

**Remediation:**

1. Add algorithm version to stored ciphertext:
```sql
CREATE TABLE vote (
    vid  INTEGER PRIMARY KEY AUTOINCREMENT,
    vote_token  BLOB NOT NULL,
    crypto_version  INTEGER NOT NULL DEFAULT 1,  -- Algorithm version
    ciphertext  BLOB NOT NULL
) STRICT;
```

2. Implement a crypto abstraction layer:

#### [LOW]
**Finding ID:** ASVS-1122-LOW-001  
**Title:** Misleading HKDF Info Parameter Suggests Incomplete Migration

**Location:** `v3/steve/crypto.py`, lines 63-67

**Code:**
```python
### still using Fernet now, but will switch soon. Leaving comments.
keymaker = hkdf.HKDF(
    algorithm=hashes.SHA256(),
    length=32,  # 32-byte key for XChaCha20-Poly1305
    salt=salt,
    info=b'xchacha20_key',  # <-- References algorithm not yet in use
)
```

**Description:**  
The HKDF `info` parameter is set to `b'xchacha20_key'` but the actual encryption uses Fernet (AES-128-CBC). This is a domain separation label that typically identifies the intended key usage. When the migration to XChaCha20-Poly1305 occurs, this `info` value must NOT change (or old ciphertext becomes undecryptable), creating a confusing situation where the info label is correct for new data but was technically wrong for historical data.

**Impact:** Minor confusion during migration. The info parameter domain separation is inconsistent with actual use, which could lead to errors during the transition.

**Remediation:** Either use a generic `info` value like `b'vote_encryption_key_v1'` or document that the current `info` value was chosen proactively for the planned migration.

## Cross-cutting Architecture Observations

1. **Centralized Crypto Module:** All cryptographic operations are centralized in `v3/steve/crypto.py`. This is excellent for auditability but currently lacks the abstraction needed for agility. The centralization makes future refactoring feasible.

2. **Key Derivation Chain:** The system uses a well-designed multi-level key derivation: `election_data → BLAKE2b → Argon2 → opened_key → (+ pid + iid + salt) → Argon2 → vote_token → HKDF → vote_key → Fernet(vote)`. This provides good key separation and limits blast radius, but the entire chain is hardcoded.

3. **Documentation Gap:** The project has partial documentation (`schema.md`) but lacks formal security/crypto documentation. The existing documentation focuses on database schema rather than cryptographic design decisions.

4. **Anti-Tamper Design:** The `opened_key` serves dual purposes: (1) tamper detection via `is_tampered()` and (2) vote token derivation. This coupling means the anti-tamper mechanism is tied to the same cryptographic algorithm as the voting mechanism — upgrading one requires upgrading both.

5. **No Asymmetric Cryptography:** The system uses only symmetric primitives and key derivation. This reduces PQC urgency compared to systems using RSA/ECC, but the AES-128 effective key length under Grover's algorithm (64-bit) is a concern for long-term data confidentiality.

6. **Inconsistency Between Production and Benchmark:** Production uses Argon2d while the benchmark uses Argon2id, suggesting either incomplete migration or a deliberate but undocumented choice.

## Cross-cutting Recommendations

### Immediate (Sprint-level)
1. **Change Argon2d to Argon2id** in `_hash()` function (ASVS-1121-MED-001). This is a one-line change with significant security improvement. Note: this will change all derived values, so must be coordinated with any existing deployed databases.

2. **Add `crypto_version` column** to the `vote` table to enable future algorithm transitions without data loss.

### Short-term (1-3 months)
3. **Create formal cryptographic inventory document** mapping all algorithms, keys, their purposes, access scopes, and protection requirements (satisfies 11.1.1, 11.1.2).

4. **Implement crypto abstraction layer** in `crypto.py` with version-based algorithm selection and a configuration mechanism for algorithm parameters (satisfies 11.2.2).

5. **Add cryptographic linting** to CI/CD pipeline using `semgrep` or equivalent with rules for:
   - Flagging new crypto imports
   - Detecting weak algorithms
   - Ensuring CSPRNG usage (satisfies 11.1.3)

### Long-term (3-12 months)
6. **Complete Fernet → XChaCha20-Poly1305 migration** with proper versioning, re-encryption tooling, and rollback capability.

7. **Document PQC migration plan** assessing quantum risk for each primitive and establishing upgrade timelines, particularly upgrading to AES-256 for long-term vote confidentiality (satisfies 11.1.4).

8. **Implement key lifecycle management** including maximum key age, scheduled rotation for long-lived keys, and secure key destruction after election archival (satisfies 11.1.1).
