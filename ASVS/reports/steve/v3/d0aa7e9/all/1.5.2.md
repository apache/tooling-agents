## ASVS-1.5.2: Safe Deserialization

### Findings for 1.5.2

### [LOW]

**Finding ID**: ASVS-152-LOW-001

**Title**: Deserialized JSON `kv` data lacks schema validation before use in vote tallying

**File**: `v3/steve/election.py`, function `tally_issue`, line ~292 and `json2kv` at line 368

**Vulnerable code**:
```python
# v3/steve/election.py:292-293
m = vtypes.vtype_module(issue.type)
return m.tally(votes, self.json2kv(issue.kv)), voters

# v3/steve/election.py:368
@staticmethod
def json2kv(j):
    "Convert the KV JSON string back into its structured value."
    return j and json.loads(j)
```

**Data flow**: Database `issue.kv` column (TEXT) → `json.loads()` → passed directly to `vtypes` module `tally()` function without structural validation.

**Analysis**: While `json.loads()` is inherently safe from code execution (only produces basic Python types: dict, list, str, int, float, bool, None), the deserialized structure is passed to vote-type-specific modules without validation against an expected schema. If the `kv` data were corrupted or maliciously set (e.g., via an authorization bypass — the code inventory notes multiple `### check authz` placeholders are unimplemented), unexpected data structures could cause logic errors in tallying.

**Mitigating factors**:
- Data originates from authorized election administrators (write path is `add_issue`/`edit_issue` which assert `is_editable()`)
- `json.loads()` cannot instantiate arbitrary objects
- `issue.type` is validated against `vtypes.TYPES` at creation time
- Risk is LOW because exploitation requires bypassing multiple controls

**Impact**: Potential logic errors in vote tallying if `kv` structure is unexpected.

**Remediation**:
```python
@staticmethod
def json2kv(j):
    "Convert the KV JSON string back into its structured value."
    if not j:
        return None
    parsed = json.loads(j)
    # Validate that parsed result is a dict (expected schema)
    if not isinstance(parsed, dict):
        raise ValueError(f"Expected dict for kv, got {type(parsed).__name__}")
    return parsed
```

Additionally, each `vtypes` module should validate the `kv` structure it receives in its `tally()` function.

### Positive Patterns for 1.5.2

1. **Safe deserialization choice**: The application exclusively uses `json.loads()` which cannot instantiate arbitrary object types — only basic Python primitives are produced.
2. **No insecure deserialization**: No use of `pickle.loads()`, `yaml.load()` (without SafeLoader), `marshal.loads()`, or `eval()` on user data.
3. **Write-path validation**: The `vtype` field is validated against `vtypes.TYPES` before being stored, preventing arbitrary module loading during tallying.
4. **Database as trust boundary**: The `kv` JSON is stored and retrieved from the application's own SQLite database, not directly from user input during deserialization.

## Cross-cutting Architecture Observations

1. **Language-Level Safety**: The choice of Python as the implementation language inherently satisfies ASVS 1.4.x requirements (memory safety, integer overflow prevention, automatic resource management). These requirements are primarily relevant for applications written in C, C++, or other systems languages with manual memory management.

2. **Cryptographic Library Trust**: The application delegates all cryptographic operations to well-maintained libraries (`cryptography`, `argon2-cffi`, `hashlib`). These libraries handle their own internal memory safety for their C/Rust backends.

3. **Data Serialization Strategy**: The exclusive use of JSON for structured data interchange is a sound architectural decision that avoids both XXE (no XML) and unsafe deserialization (no pickle/YAML) attack surfaces.

4. **Consistent Single-Parser Approach**: The codebase maintains consistency by using a single parser for each data format, preventing parser differential attacks.

5. **State Machine Enforcement**: The election lifecycle (Editable → Open → Closed) is enforced through assertions and metadata checks, providing defense-in-depth against operations performed in inappropriate states.

## Cross-cutting Recommendations

### Immediate (No issues requiring immediate action for these requirements)

None. The audited requirements (1.4.1–1.4.3, 1.5.1–1.5.3) are well-satisfied by the current architecture.

### Short-term

1. **Add schema validation for `kv` JSON** (ASVS 1.5.2): Implement per-vtype JSON schema validation in `json2kv()` or at the `vtypes` module level to ensure deserialized data conforms to expected structure before use in tallying logic.

2. **Document language safety rationale**: Add a security design document noting that Python was chosen in part for its memory safety properties, satisfying ASVS 1.4.x requirements. This aids future auditors and prevents introduction of unsafe native code without review.

### Long-term

3. **Dependency security monitoring**: While the current cryptographic libraries are safe, establish automated monitoring (e.g., `pip-audit`, Dependabot) for vulnerabilities in `cryptography`, `argon2-cffi`, and other dependencies that contain native code — these are the only paths through which memory safety issues could be introduced.

4. **Consider typed data classes**: Replace `EasyDict` usage with Python `dataclass` or `TypedDict` types for deserialized data, providing compile-time type safety and documentation of expected data structures.
