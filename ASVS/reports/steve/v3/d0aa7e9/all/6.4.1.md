## ASVS-6.4.1: System Generated Initial Passwords/Activation Codes

### Findings for 6.4.1

No findings detected.

This application delegates all user authentication to an external Apache Software Foundation (ASF) identity provider via `asfquart.auth.require()`. There is no local user registration, no initial password generation, and no activation code generation within this codebase.

The `crypto.create_id()` function generates 10-hex-character election IDs and issue IDs, but these are resource identifiers (used in URLs), not authentication credentials:

```python
# v3/steve/election.py, line 379
while True:
    eid = crypto.create_id()
    try:
        db.c_create.perform(eid, title, owner_pid, authz, open_at, close_at)
        break
    except sqlite3.IntegrityError:
        _LOGGER.debug('EID conflict(!!) ... trying again.')
```

These identifiers do not serve as authentication factors — users still must authenticate via their ASF committer credentials before accessing any election by EID.

### Security Controls Inventory for 6.4.1

| Control | Location | Coverage Status |
|---------|----------|----------------|
| External authentication delegation | `v3/server/pages.py` (all `@asfquart.auth.require` decorators) | APPLIED — all sensitive endpoints require external auth |
| No local credential storage | `v3/schema.sql` — `person` table | VERIFIED — stores only `pid`, `name`, `email`; no passwords |
| Cryptographic ID generation | `v3/steve/election.py`, `crypto.create_id()` | APPLIED — for resource identifiers only, not auth factors |

### Positive Patterns for 6.4.1

1. **No local password storage**: The `person` table schema (`v3/schema.sql`) contains only `pid TEXT`, `name TEXT`, and `email TEXT` — no password hashes, no temporary credentials.
2. **External authentication**: All authentication is delegated to the ASF infrastructure, which is responsible for its own password/credential lifecycle.
3. **Secure ID generation**: Election and issue IDs use `crypto.create_id()` (presumed cryptographically random), preventing guessability even though they are not authentication factors.

## Cross-cutting Architecture Observations

1. **Complete Authentication Delegation**: This application entirely delegates authentication and credential management to the Apache Software Foundation's external identity infrastructure (accessed via `asfquart.auth` and `asfquart.session`). This is a sound architectural decision that eliminates an entire class of credential lifecycle vulnerabilities from this application's attack surface.

2. **No Local Credential Storage**: The database schema (`v3/schema.sql`) deliberately contains no password hashes, secret questions, recovery codes, MFA seeds, or any other authentication factor data. The `person` table stores only identity references (`pid`, `name`, `email`).

3. **ASVS 6.4.x Applicability**: All six requirements (6.4.1–6.4.6) relate to authentication factor lifecycle management. Since this application has no such lifecycle (delegating entirely to an external IdP), these requirements are architecturally addressed through delegation rather than implementation. The security posture for these requirements depends entirely on the external ASF identity provider's compliance.

4. **Potential Concern — Session Lifecycle**: While not directly covered by 6.4.x, the session management (read via `asfquart.session.read()`) is the closest analog to an authentication mechanism in this application. Session token lifecycle, expiration, and rotation would be relevant concerns but fall under the responsibility of the `asfquart` framework rather than this application code.

5. **Authorization Gap (Adjacent Concern)**: While not an authentication factor lifecycle issue, the numerous `### check authz` placeholder comments throughout `pages.py` indicate that authorization controls are incomplete. Any authenticated committer can potentially manage any election, which is an access control issue adjacent to authentication concerns.

## Cross-cutting Recommendations

### Immediate (No Action Required for 6.4.x)
- **No immediate remediation needed** for ASVS 6.4.1–6.4.6 within this application's scope, as authentication factor management is correctly delegated to the external identity provider.

### Short-term
- **Document the security dependency**: Explicitly document that ASVS 6.4.x compliance relies on the ASF identity provider (IdP) implementing these controls. This should be noted in the application's security architecture documentation.
- **Verify external IdP compliance**: Confirm that the ASF authentication infrastructure meets ASVS 6.4.1–6.4.6 requirements (secure initial password generation, no KBA, MFA-safe password reset, etc.).

### Long-term
- **If the application ever adds local accounts**: Implement proper password lifecycle management including secure random initial passwords, forced change on first use, short expiration for initial credentials, and MFA-aware password reset flows.
- **Session lifecycle hardening**: Review `asfquart.session` for proper session token rotation, expiration, and invalidation — the application's closest analog to authentication factor lifecycle management.
- **Complete authorization implementation**: Address the `### check authz` placeholders to ensure election owners are properly validated, as this is the application's primary security boundary after authentication.
