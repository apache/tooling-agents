## ASVS-7.2.3: Reference Tokens - CSPRNG with 128+ Bits Entropy

### Findings for 7.2.3

#### [LOW]

**Finding ID:** ASVS-723-LOW-001

**Title:** Unable to verify session token entropy from available code; election identifiers use only 40 bits

**File:** `v3/steve/crypto.py` (line 101)

**Code Reference:**
```python
def create_id():
    "Create a standard ID value."

    # Use 10 hex characters for the ID
    return secrets.token_hex(5)  # 5 bytes
```

**Analysis:**
- The `create_id()` function generates only 5 bytes (40 bits) of entropy for election/issue IDs. However, these are **not session tokens** — they are resource identifiers. The schema documentation explicitly states: "We do not use AUTOINCREMENT, so that URLs for Elections cannot be deduced." The 40-bit space (~1 trillion possible values) provides unpredictability for URL-based resource identifiers, though it wouldn't meet session token requirements.
- The actual **session tokens** are managed by `asfquart.session`, which is an external dependency not included in the provided source. We cannot verify whether session reference tokens meet the 128-bit entropy requirement.
- The `crypto.py` module demonstrates use of `secrets.token_bytes(16)` (128 bits) for salts, indicating awareness of appropriate entropy levels for cryptographic operations.

**Impact:** Low — the identifiers with 40 bits are not session tokens and have additional security layers (authentication required to access). However, the inability to audit the actual session token generation in `asfquart.session` represents an audit gap.

**Remediation:**
1. Verify that `asfquart.session` generates session tokens with at least 128 bits of entropy using a CSPRNG.
2. Document the session token generation mechanism:
```python
# Session Token Specification:
# - Generated by asfquart.session using [specific mechanism]
# - Token length: [N] bytes ([M] bits of entropy)
# - Source: [e.g., os.urandom / secrets module]
# - Verified to meet ASVS 7.2.3 (128+ bits, CSPRNG)
```

## Cross-cutting Architecture Observations

1. **Session Management Delegation:** The application delegates all session management to `asfquart.session` and `asfquart.auth`. While this is architecturally sound (avoiding custom session code), it means critical session security properties (timeouts, entropy, concurrent limits) cannot be audited from the application code alone. The framework becomes a trust boundary that must be separately verified.

2. **Authentication vs. Authorization Gap:** Authentication is consistently enforced (`@asfquart.auth.require`), but fine-grained authorization (election ownership checks) is marked with `### check authz` placeholder comments throughout `pages.py`. This means any authenticated committer can currently manage any election. This is not directly a session management issue but creates a significant authorization gap that session controls alone cannot mitigate.

3. **CSRF Token Placeholder:** The `basic.csrf_token = 'placeholder'` (line 83) represents a critical gap. While not a session *token* issue per ASVS 7.2.x, it means that session-based CSRF protection is non-functional. An attacker could forge requests that ride on a victim's valid session.

4. **State-Changing Operations via GET:** The `/do-open/<eid>` and `/do-close/<eid>` endpoints use GET requests for state-changing operations. Combined with the absent CSRF protection, this creates a trivially exploitable session riding vulnerability.

5. **No Session Invalidation on Privilege Change:** When election state changes (opened, closed), there is no session re-validation or re-authentication requirement. A session established before an election opened remains valid for voting without any freshness check.

## Cross-cutting Recommendations

### Immediate (Critical/High Priority)

1. **Document Session Management Policy:** Create a dedicated session management policy document covering:
   - Inactivity timeout (recommend 30 minutes per NIST SP 800-63B)
   - Absolute session lifetime (recommend 8-12 hours)
   - Concurrent session limit (recommend 3-5)
   - Federated session coordination with ASF SSO
   - Re-authentication triggers for sensitive operations

2. **Verify Framework Session Security:** Audit `asfquart.session` to confirm:
   - Session tokens use CSPRNG with ≥128 bits of entropy
   - Tokens are unique (collision-resistant)
   - Session store supports timeout/expiration

3. **Implement CSRF Protection:** Replace the placeholder token with a real CSRF mechanism tied to the session, as this directly undermines session security by enabling session riding attacks.

### Short-term (1-2 Sprints)

4. **Implement Authorization Checks:** Replace all `### check authz` placeholders with actual ownership/permission verification to ensure sessions only provide access to authorized resources.

5. **Convert State-Changing GETs to POSTs:** Change `/do-open/<eid>` and `/do-close/<eid>` to POST endpoints with CSRF protection.

6. **Add Concurrent Session Controls:** Implement session counting and enforcement, with user notification of other active sessions.

### Long-term (Architecture)

7. **Session Lifecycle Documentation:** Maintain living documentation of session management decisions as the application evolves from ASF-specific to a general-purpose solution (as noted in code comments).

8. **Re-authentication for Critical Operations:** Implement step-up authentication for election lifecycle operations (open, close, tally).

9. **Session Monitoring:** Add logging for session creation, destruction, and anomalous patterns (multiple geographic locations, rapid session cycling).
