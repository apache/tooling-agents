## ASVS-11.2.3: Minimum 128-bits of Security

### Findings for 11.2.3

### [MEDIUM]

**Finding ID:** ASVS-1123-MED-001

**File:** `v3/steve/crypto.py`, function `create_id()`, line 100

**Vulnerable Code:**
```python
def create_id():
    "Create a standard ID value."
    # Use 10 hex characters for the ID
    return secrets.token_hex(5)  # 5 bytes
```

**Data Flow:** `secrets.token_hex(5)` → 5 bytes (40 bits of entropy) → used as EID/IID identifiers

**Context:** While `create_id()` is not a traditional cryptographic primitive (it's an unpredictability token), it provides the security property of preventing URL/identifier guessing. With only 40 bits of entropy (~10^12 possibilities), this falls significantly below the 128-bit threshold recommended for cryptographic security parameters. The schema documentation explicitly states these are designed to "prevent URL guessing."

**Impact:** An attacker with sustained access could enumerate valid election IDs or issue IDs. With ~1 trillion possible values and only a small subset being valid, this is mitigated by the online attack constraint (rate limiting, network latency). However, for a voting system where ballot secrecy is paramount, the margin is thin.

**Proof of Concept:**
```python
# With 5 bytes = 40 bits, brute force requires ~2^40 attempts
# At 1000 requests/sec, exhaustive search takes ~12.7 days
# Finding ONE valid EID among, say, 100 elections is much faster
import itertools
for candidate in itertools.product('0123456789abcdef', repeat=10):
    test_eid = ''.join(candidate)
    # attempt access to /vote-on/{test_eid}
```

**Remediation:**
```python
def create_id():
    "Create a standard ID value with sufficient entropy."
    # Use 16 bytes (128 bits) encoded as 32 hex characters
    return secrets.token_hex(16)
```
Note: This requires schema changes to accommodate 32-character IDs and updating CHECK constraints in `schema.sql`.

### Positive Patterns for 11.2.3

1. **Strong salt generation**: `secrets.token_bytes(SALT_LEN)` with `SALT_LEN = 16` provides 128 bits of randomness for all salt values.
2. **Adequate key lengths**: HKDF derives 32-byte (256-bit) keys, Argon2 produces 32-byte outputs, BLAKE2b uses full 64-byte digest.
3. **Appropriate Argon2 tuning**: `memory_cost=65536` (64 MiB), `time_cost=2`, `parallelism=4` provides strong resistance against brute-force attacks on the derived keys.
4. **Cryptographically secure RNG**: All random values use `secrets` module throughout.

## Cross-cutting Architecture Observations

1. **Cryptographic key hierarchy is well-designed**: The system uses a layered key derivation approach: Election data → BLAKE2b → Argon2 → `opened_key` → per-voter tokens → per-vote encryption keys. This provides good key separation and ensures that compromise of one vote doesn't compromise others.

2. **Incomplete migration from Fernet to AEAD**: The codebase is in a transitional state, as evidenced by the `### still using Fernet now, but will switch soon` comment and the `info=b'xchacha20_key'` parameter. The HKDF derives 32 bytes intended for XChaCha20-Poly1305 but actually feeds it to Fernet which only uses 16 bytes for AES-128. This means the full key material isn't utilized.

3. **Security margin inconsistency**: The system uses strong 256-bit outputs from Argon2 and HKDF but then feeds them into Fernet which only provides 128-bit encryption security (AES-128). The 40-bit IDs (`create_id()`) are the weakest link in the entropy chain.

4. **Side-channel considerations partially addressed**: The use of `secrets` module throughout is excellent. However, Argon2d and non-constant-time comparison in `is_tampered()` show that side-channel resistance wasn't uniformly applied.

5. **Fernet's timestamp exposure**: Fernet tokens include a plaintext timestamp of when encryption occurred. In a voting system, this could leak information about when a specific vote was cast, potentially de-anonymizing voters if combined with server access logs.

## Cross-cutting Recommendations

### Immediate (Before Next Release)
1. **Fix `is_tampered()` to use `hmac.compare_digest()`** — Simple one-line fix that eliminates the timing oracle (ASVS-1124-HIGH-001).
2. **Switch from Argon2d to Argon2id** — Change `Type.D` to `Type.ID` in `_hash()` to gain side-channel resistance (ASVS-1124-MED-001).
3. **Add error handling around `decrypt_votestring()`** — Wrap in try/except to prevent DoS via corrupted votes (ASVS-1125-MED-001).

### Short-term (Next Development Cycle)
4. **Complete migration to XChaCha20-Poly1305** — Replace Fernet with a proper AEAD cipher as already planned. This addresses ASVS 11.3.2 and eliminates the key material waste (ASVS-1132-MED-001).
5. **Increase `create_id()` entropy to at least 128 bits** — Use `secrets.token_hex(16)` for 32-character hex IDs, updating schema constraints accordingly (ASVS-1123-MED-001).

### Long-term (Architecture)
6. **Implement a unified cryptographic abstraction layer** — Create a single module interface for all crypto operations with versioning, allowing cipher agility without code changes throughout the application.
7. **Add integration tests for timing resistance** — Automated tests that verify constant-time behavior of security-critical comparisons.
8. **Consider removing Fernet timestamp from vote tokens** — Or migrate to a scheme that doesn't embed plaintext metadata in ciphertext.
