## ASVS-7.3.2: Session Timeout

### Findings for 7.3.2

#### [HIGH]

**Finding ID:** ASVS-732-HIGH-001

**File:** `v3/server/pages.py` — entire file; no absolute session lifetime configuration

**Vulnerable Code:**
```python
# No absolute session lifetime enforcement found.
# Session is read without checking creation time:

s = await asfquart.session.read()
if s:
    basic.update(
        uid=s['uid'],
        name=s['fullname'],
        email=s['email'],
    )
```

**Data Flow:** User authenticates → Session created with no creation timestamp → Session used over days/weeks/months → No absolute expiration enforced → Session remains valid permanently as long as it exists.

**Analysis:** The application code contains:
1. No absolute session lifetime limit
2. No session creation timestamp stored in session data
3. No validation of session age before granting access
4. No documented security decision regarding maximum session lifetime

For an election system where elections have defined open/close periods, a session that outlives an election's lifecycle could allow unintended access patterns. An absolute maximum lifetime ensures that compromised sessions have bounded impact.

**Impact:** A compromised session token could be used indefinitely by an attacker. Even if a user changes their password or credentials are rotated, the existing session may remain valid. This is particularly concerning for election management operations where election states are time-sensitive.

**Remediation:**
```python
# Configuration:
SESSION_MAX_LIFETIME = 8 * 60 * 60  # 8 hours absolute maximum

# In session creation (post-authentication):
session['created_at'] = time.time()

# In basic_info() or middleware:
async def basic_info():
    s = await asfquart.session.read()
    if s:
        created_at = s.get('created_at', 0)
        if time.time() - created_at > SESSION_MAX_LIFETIME:
            await asfquart.session.destroy()
            quart.abort(401, 'Session expired. Please log in again.')
        # ...
```

## Cross-cutting Architecture Observations

1. **Session Management Delegation**: The application delegates all session management to the `asfquart` framework. While this is architecturally sound for separation of concerns, it means the application code has ZERO visibility into session lifecycle events. There is no evidence the framework provides the required controls (timeouts, token rotation, termination).

2. **Missing Security Layer**: The application has authentication (via decorators) but lacks a complete session management layer. There are no:
   - Session creation/destruction endpoints
   - Session metadata tracking (creation time, last activity)
   - Session inventory or listing
   - Per-user session management

3. **Election Lifecycle vs Session Lifecycle Mismatch**: Elections have well-defined states (editable → open → closed) with proper state transitions and assertions. Sessions have NO lifecycle management at all — a stark contrast suggesting sessions were not part of the security design.

4. **TODO Acknowledgment**: The `TODO.md` acknowledges CSRF issues but does NOT mention any session management concerns, indicating these requirements were not considered during design.

5. **Person Deletion Without Session Awareness**: The database schema and queries support person deletion (`c_delete_person`), but the application has no mechanism to link person records to active sessions. The `person` table has no "active/disabled" status field.

6. **No State-Change-Triggered Session Actions**: Sensitive state changes (opening/closing elections, casting votes) do not trigger any session-related security actions. There's no step-up authentication for high-privilege operations.

## Cross-cutting Recommendations

### Immediate (Critical)

1. **Implement a logout endpoint** — Users currently have no way to terminate their sessions. This is the most basic session management requirement.
   
2. **Add inactivity timeout** — Configure the `asfquart` session framework or implement middleware to enforce a 30-minute inactivity timeout for this election management context.

3. **Add absolute session lifetime** — Implement an 8-hour maximum session lifetime to bound the impact of session compromise.

### Short-term (High)

4. **Implement session-to-user mapping** — Add a session store or index that allows finding all sessions for a given PID. This enables requirements 7.4.2 and 7.4.3.

5. **Add session termination on person deletion** — Before or after calling `c_delete_person`, invalidate all sessions for that PID.

6. **Verify session token regeneration** — Audit the `asfquart.auth` framework to confirm it regenerates session tokens on authentication. If not, add this capability.

### Long-term (Medium)

7. **Add "terminate all other sessions" feature** — Implement in the `/settings` page, allowing users to invalidate all sessions except their current one.

8. **Document session security decisions** — Create a security design document that specifies chosen timeout values, session lifetime limits, and the risk analysis supporting these decisions (as required by ASVS Level 2).

9. **Implement session activity dashboard** — Allow users to view their active sessions (device, last activity, IP) and terminate individual ones.

10. **Add step-up authentication** — For critical operations (opening/closing elections, person management), require re-authentication even within a valid session.
