## ASVS-16.4.1: Log Injection Prevention

### Findings for 16.4.1

### [HIGH]
**Finding ID:** ASVS-1641-HIGH-001
**Location:** `v3/server/pages.py`, `do_create_endpoint()` (line 423)
**Vulnerable Code:**
```python
_LOGGER.info(
    f'User[U:{result.uid}] created election[E:{election.eid}];'
    f' title: "{form.title}"'
)
```
**Data Flow:** User form input (`form.title`) → f-string interpolation → `_LOGGER.info()` → log output
**Proof of Concept:**
```
POST /do-create-election HTTP/1.1
Content-Type: application/x-www-form-urlencoded

title=Legit%20Election%22%0A2024-01-01%2012:00:00%20INFO%20User[U:admin]%20opened%20election[E:secret123]
```
This injects a newline followed by a fake log entry, making it appear that `admin` opened a different election.
**Impact:** An attacker can inject newlines (`\n`) and carriage returns (`\r`) into log entries via the election title field, allowing log forging. This can be used to: (1) create fake audit entries, (2) obscure real malicious activity, (3) inject misleading data into SIEM systems.
**Remediation:** Sanitize all user-controlled data before logging:
```python
import re

def sanitize_for_log(value: str) -> str:
    """Remove control characters that could forge log entries."""
    return re.sub(r'[\r\n\x00-\x1f\x7f]', '', str(value))

_LOGGER.info(
    f'User[U:{sanitize_for_log(result.uid)}] created election[E:{election.eid}];'
    f' title: "{sanitize_for_log(form.title)}"'
)
```

### [HIGH]
**Finding ID:** ASVS-1641-HIGH-002
**Location:** `v3/server/pages.py`, multiple logging statements with user input
**Vulnerable Code (representative examples):**
```python
# Line 108 - date_str from JSON input
_LOGGER.info(
    f'User[U:{result.uid}] set {field} for election[E:{election.eid}] to {date_str}'
)

# Line 485 - form.title from user form
_LOGGER.info(
    f'User[U:{result.uid}] added issue[I:{iid}] to election[E:{election.eid}]'
)
```
**Affected logging locations with user-controlled data:**
- Line 108: `{date_str}` — from JSON request body
- Line 423-425: `{form.title}` — from form submission
- Line 485: IID is generated (safe), but `result.uid` comes from session

**Data Flow:** User-controlled values → f-string → logger → log storage/SIEM
**Impact:** All user-controllable strings logged via f-strings without encoding allow potential log injection via control characters. While `result.uid` comes from authenticated session data (lower risk), `form.title` and `date_str` come directly from user input.
**Remediation:** Create a centralized logging utility:
```python
class SafeLogger:
    """Logger wrapper that sanitizes interpolated values."""
    
    _CONTROL_CHARS = re.compile(r'[\x00-\x1f\x7f]')
    
    def __init__(self, logger):
        self._logger = logger
    
    @staticmethod
    def _clean(msg):
        return SafeLogger._CONTROL_CHARS.sub('', msg)
    
    def info(self, msg, *args, **kwargs):
        self._logger.info(self._clean(msg), *args, **kwargs)
    
    def warning(self, msg, *args, **kwargs):
        self._logger.warning(self._clean(msg), *args, **kwargs)
    
    def error(self, msg, *args, **kwargs):
        self._logger.error(self._clean(msg), *args, **kwargs)

_LOGGER = SafeLogger(logging.getLogger(__name__))
```

### [MEDIUM]
**Finding ID:** ASVS-1641-MED-001
**Location:** `v3/server/pages.py`, `print('FORM:', form)` (lines 481, 497)
**Vulnerable Code:**
```python
form = edict(await quart.request.form)
print('FORM:', form)
```
**Data Flow:** Raw form data (all keys/values) → `print()` → stdout → potential log aggregation
**Proof of Concept:**
```
POST /do-add-issue/abc1234567
Content-Type: application/x-www-form-urlencoded

title=Test%0ACRITICAL%20SQL_INJECTION_DETECTED&description=normal
```
**Impact:** `print()` provides no encoding or sanitization. Multi-line form values will create multiple lines in stdout, and if stdout is captured by a log aggregator, forged log entries can be injected.
**Remediation:** Remove debug print statements entirely.

### Positive Patterns for 16.4.1

1. **Fixed-format identifiers** — EIDs and IIDs are 10 hex character strings generated by `crypto.create_id()`, which constrains them to `[0-9a-f]{10}` — safe for log injection purposes when coming from the database.
2. **Session-sourced UIDs** — `result.uid` comes from the authenticated session rather than raw user input, reducing (but not eliminating) injection risk.
3. **Structured log prefix pattern** — The logging uses consistent patterns like `User[U:{uid}]`, `election[E:{eid}]`, `issue[I:{iid}]` which could facilitate parsing, though they don't prevent injection.

## Cross-cutting Architecture Observations

1. **Logging maturity is low** — The application has action logging for administrative operations but lacks authentication event logging, authorization failure logging, input validation failure logging, and error logging across most endpoints. Only `do_vote_endpoint()` demonstrates the full pattern of try/except with error logging.

2. **Type B gaps are pervasive** — The codebase contains numerous `### check authz` comments indicating the developer intended to add authorization checks. These are never implemented, creating a false sense of security and making authorization failure logging impossible.

3. **No centralized logging infrastructure** — There is no logging middleware, no structured log format (e.g., JSON), no log sanitization layer, and no consistent error handling pattern. Each endpoint handles (or doesn't handle) logging independently.

4. **Debug code in production paths** — `print('FORM:', form)` statements exist in two production endpoints, bypassing all logging controls and dumping raw user data.

5. **Security assertions vs. proper error handling** — The `election.py` module uses Python `assert` statements for security-critical state checks. These are: (a) disabled with `-O` flag, (b) generate no log entries when they fire, and (c) provide no graceful error handling.

6. **Single responsibility gap** — The authentication framework (`asfquart.auth`) handles enforcement but there's no evidence it handles logging. The application layer logs actions but not auth events. This creates a gap where neither layer owns auth logging.

## Cross-cutting Recommendations

### Immediate (Critical/High priority)
1. **Remove debug `print()` statements** — Lines 481 and 497 in `pages.py` dump raw form data to stdout. Remove immediately.
2. **Implement log sanitization** — Create a wrapper or formatter that strips control characters (`\r`, `\n`, `\x00`-`\x1f`) from all logged values before output.
3. **Add error handling with logging to all state-changing endpoints** — Wrap `election.open()`, `election.close()`, `Election.create()` in try/except with `_LOGGER.error()`.
4. **Log authorization denials** — Add `_LOGGER.warning()` calls in `vote_on_page()` and `serve_doc()` where access is denied.

### Short-term (1-2 sprints)
5. **Implement authentication event logging** — Add a middleware or decorator that logs all authentication successes and failures with metadata (user, endpoint, method, IP, auth type).
6. **Replace `assert` with proper exceptions** — Convert all security-relevant `assert` statements in `election.py` to explicit checks with `ElectionBadState` exceptions and logging.
7. **Implement authorization checks** — Address all `### check authz` placeholders with ownership verification and log failures.
8. **Add input validation failure logging** — Log when date parsing fails, when invalid IIDs are submitted, and when other validation rejects input.

### Long-term (Architecture)
9. **Adopt structured logging (JSON)** — Migrate to structured log format with consistent fields (`timestamp`, `level`, `event_type`, `user_id`, `resource_id`, `action`, `outcome`) for machine-readable analysis.
10. **Implement centralized security event framework** — Create a `SecurityEvent` class or module that provides typed events (`AUTH_SUCCESS`, `AUTH_FAILURE`, `AUTHZ_DENIED`, `INPUT_VALIDATION_FAILED`, `BUSINESS_LOGIC_VIOLATION`) with automatic metadata collection.
11. **Add anti-automation detection** — Implement rate limiting with logging of threshold violations to detect brute-force or automated abuse.
12. **PII handling policy in logs** — Establish a logging policy that defines which fields may be logged in full, which must be masked, and which must never be logged, then enforce via the structured logging framework.
