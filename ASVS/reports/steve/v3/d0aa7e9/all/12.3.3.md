## ASVS-12.3.3: TLS for Internal HTTP-based Services

### Findings for 12.3.3

#### [LOW]
**Finding ID**: ASVS-1233-LOW-001

**Title**: No documented enforcement of TLS between application and reverse proxy

**File**: `v3/ARCHITECTURE.md`, `v3/server/config.yaml.example` (line 22)

**Relevant Documentation**:
```yaml
# config.yaml.example
    # Default port for the server. Typical usage is that a proxy sits
    # in front of this server, so we don't use ports 80 or 443.
```

**Analysis**: The architecture states a proxy sits in front of the application server. The communication between the proxy and the application backend is not explicitly configured to use TLS. If the TLS configuration is left blank (as the config allows), the proxy-to-application link would be unencrypted.

In a typical deployment:
- External client → Proxy (TLS terminated) → Application server (plain HTTP on port 58383)

This internal hop may be unencrypted if the application's `certfile`/`keyfile` are not set.

**Impact**: If the proxy and application are on different hosts or cross a network boundary, internal traffic between them could be intercepted. This is a lower severity concern if both are on the same host/container.

**Remediation**:
- Document that TLS must be configured on the application even when behind a proxy (unless on the same localhost)
- Or enforce TLS between proxy and backend:
```yaml
# config.yaml production
server:
    port: 58383
    certfile: internal-service.pem  # Required even behind proxy
    keyfile: internal-service-key.pem
```

## Cross-cutting Architecture Observations

1. **Monolithic Architecture Reduces TLS Surface**: The application uses SQLite (local file) and runs as a single process, meaning most "internal communication" happens in-process. The primary TLS concern is the external-facing server endpoint and the proxy-to-app connection.

2. **Proxy-Dependent Security Model**: The architecture explicitly states "a proxy sits in front of this server." This means TLS termination and certificate management may be delegated to the proxy layer (e.g., Apache httpd, nginx, HAProxy), which is not represented in this codebase. This is a common and valid pattern but creates an implicit dependency that isn't enforced.

3. **Optional TLS is the Primary Risk**: The most significant cross-cutting issue is that TLS is conditional on configuration. The server can run in plain HTTP mode, which violates multiple requirements simultaneously (12.2.2, 12.3.1, 12.3.3).

4. **External Dependencies Not Auditable**: The `asfquart` framework handles OAuth HTTP clients and potentially TLS configuration. Without access to this code, certificate validation behavior cannot be fully verified.

5. **Development vs. Production Gap**: The documentation and configuration examples target development (self-signed certs, localhost). There is no production deployment guide addressing TLS requirements.

---

## Cross-cutting Recommendations

### Immediate (Critical/High Priority)

1. **Enforce TLS at Startup**: Modify `main.py` to require TLS configuration and fail with a clear error if certificates are missing or invalid. Allow an explicit `--insecure` flag for development only:
   ```python
   if not app.cfg.server.certfile and not os.environ.get('STEVE_DEV_MODE'):
       _LOGGER.critical('TLS is required. Set certfile/keyfile or STEVE_DEV_MODE=1')
       sys.exit(1)
   ```

2. **Add HSTS Headers**: Configure Strict-Transport-Security headers to prevent protocol downgrade:
   ```python
   @APP.after_request
   async def add_security_headers(response):
       response.headers['Strict-Transport-Security'] = 'max-age=31536000; includeSubDomains'
       return response
   ```

### Short-term (Within 1-2 Sprints)

3. **Production Deployment Guide**: Create documentation for production TLS deployment covering:
   - Publicly trusted certificate acquisition (Let's Encrypt/ACME)
   - Reverse proxy TLS configuration
   - Internal communication encryption requirements

4. **Verify OAuth Client TLS**: Audit the `asfquart` library to confirm certificate validation is enabled and cannot be accidentally disabled.

5. **Certificate Validation for LDAP**: Ensure `asf-load-ldap.py` uses LDAPS or StartTLS with certificate validation.

### Long-term (Architecture Level)

6. **Internal CA Infrastructure**: If the deployment grows beyond a single-host model, establish an internal CA for service-to-service TLS with proper certificate lifecycle management.

7. **Consider mTLS for Proxy-App Communication**: Implement mutual TLS between the reverse proxy and the application server for defense-in-depth.

8. **Automated Certificate Rotation**: Integrate certificate rotation mechanisms (e.g., cert-manager, ACME) to prevent certificate expiry-related outages.
