## ASVS-3.3.5: Cookie Size Limit (≤ 4096 bytes)

### Findings for 3.3.5

#### [LOW]
**Finding ID**: ASVS-335-LOW-001

**Title**: No validation of cookie size before writing

**File**: `v3/server/pages.py`

**Analysis**: The application uses framework-managed sessions (`asfquart.session`). Session data includes:
- `uid` (LDAP username, typically short)
- `fullname` (person name)
- `email` (email address)
- Flash messages stored in session (via `quart.flash()`)

Flash messages are accumulated in the session via multiple `flash_*` helper functions (lines 120-127) and are consumed on next page load. If numerous flash messages accumulate (e.g., repeated redirect loops or programmatic abuse), the session cookie could theoretically exceed 4096 bytes.

**Impact**: If the session cookie exceeds 4096 bytes, the browser will silently drop it, logging the user out and preventing them from using authenticated functionality. This is a denial-of-service concern rather than a confidentiality issue.

**Remediation**:
```python
# Add middleware to check cookie size before setting
@APP.after_request
async def check_cookie_size(response):
    for header_name, header_value in response.headers.getlist('Set-Cookie'):
        # Parse cookie name and value
        if len(header_value) > 4096:
            _LOGGER.warning(f'Cookie exceeds 4096 bytes: {len(header_value)}')
            # Truncate flash messages or use server-side session storage
    return response
```

## Cross-cutting Architecture Observations

1. **Session Management Framework Opacity**: The `asfquart` framework handles session cookies, but its configuration is not visible in the audited code. Critical security attributes (SameSite, HttpOnly, Secure, cookie name) are delegated to the framework with no explicit override. This creates an audit gap — the security posture depends on undocumented framework defaults.

2. **No Security Header Middleware**: The application lacks any `after_request` handler or middleware for setting security headers. This means HSTS, CSP, X-Content-Type-Options, X-Frame-Options, and other security headers are all absent from responses.

3. **CSRF Defense Completely Absent**: The combination of:
   - Placeholder CSRF token (`'placeholder'`)
   - No SameSite cookie attribute (verified)
   - State-changing GET endpoints (`/do-open`, `/do-close`)
   
   Creates a complete absence of CSRF protection for the voting system. This is the most critical cross-cutting concern.

4. **Defense-in-Depth Gap**: The application relies solely on authentication (`@asfquart.auth.require`) for security. There are no layered defenses (CSRF, origin checking, SameSite cookies, authorization checks). If any single layer is bypassed, there is no fallback.

5. **Acknowledged Technical Debt**: The `TODO.md` and inline `### check authz` comments indicate the development team is aware of security gaps but has not yet addressed them. The code appears to be in active development with security controls planned but not implemented.

## Cross-cutting Recommendations

### Immediate (Before Any Production Deployment)

1. **Add Security Headers Middleware** (addresses ASVS 3.4.1):
```python
@APP.after_request
async def security_headers(response):
    response.headers['Strict-Transport-Security'] = 'max-age=31536000; includeSubDomains'
    response.headers['X-Content-Type-Options'] = 'nosniff'
    response.headers['X-Frame-Options'] = 'DENY'
    return response
```

2. **Configure Session Cookie Securely** (addresses ASVS 3.3.2, 3.3.3, 3.3.4):
```python
app.config.update(
    SESSION_COOKIE_NAME='__Host-steve_session',
    SESSION_COOKIE_HTTPONLY=True,
    SESSION_COOKIE_SECURE=True,
    SESSION_COOKIE_SAMESITE='Strict',
    SESSION_COOKIE_PATH='/',
)
```

3. **Convert State-Changing GET Endpoints to POST** (addresses ASVS 3.3.2):
   - `/do-open/<eid>` → POST with CSRF token
   - `/do-close/<eid>` → POST with CSRF token

### Short-term (Within Next Sprint)

4. **Implement Real CSRF Protection** (supports ASVS 3.3.2):
   - Generate cryptographically random tokens per session
   - Validate tokens on all state-changing endpoints
   - Combine with `SameSite=Strict` for defense-in-depth

5. **Add Cookie Size Monitoring** (addresses ASVS 3.3.5):
   - Log warnings if session cookie approaches 4096 bytes
   - Consider server-side session storage if flash messages grow

### Long-term (Architecture)

6. **Audit asfquart Framework**: Review the `asfquart` framework source code to verify default cookie attributes, as the application delegates all cookie security to this dependency.

7. **Implement Authorization Layer**: Address the `### check authz` placeholders throughout the codebase to ensure election owners/authorized groups are properly validated, complementing the cookie security controls.

8. **Security Testing**: Add integration tests that verify response headers include HSTS, cookie attributes include SameSite/HttpOnly/Secure, and no CORS headers leak on sensitive endpoints.
