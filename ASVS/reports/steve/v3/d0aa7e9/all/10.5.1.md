## ASVS-10.5.1: ID Token Replay Attack Mitigation (Nonce Validation)

### Findings for 10.5.1

### [MEDIUM]

**Finding ID:** ASVS-1051-MED-001

**Title:** No visible nonce generation or validation in OIDC authentication flow

**File:** `v3/server/pages.py` (entire authentication flow)

**Vulnerable Pattern:**

The application delegates all OIDC authentication to `asfquart.auth` and `asfquart.session`, but there is no evidence in the provided code of:
1. Nonce generation before redirecting to the OpenID Provider
2. Nonce validation when processing the ID Token response
3. Any state parameter handling for the authorization request

```python
# pages.py:68-80 - Session is read but no nonce validation visible
async def basic_info():
    """Return base-level EZT template data."""
    basic = edict()
    # ...
    s = await asfquart.session.read()
    if s:
        basic.update(
            uid=s['uid'],
            name=s['fullname'],
            email=s['email'],
        )
```

**Data Flow:** User authentication → `asfquart.auth.require` decorator → session established → `asfquart.session.read()` returns user data. The intermediate OIDC token validation (including nonce) is not visible.

**Impact:** If the `asfquart` library does not properly validate the `nonce` claim in the ID Token against the `nonce` sent in the authentication request, an attacker could replay a stolen ID Token to impersonate a user.

**Note:** This finding is of MEDIUM severity because the control may exist in the `asfquart` library but cannot be verified from the provided code. The lack of visibility into the authentication flow creates an audit gap.

**Remediation:** Verify that the `asfquart` library:
1. Generates a cryptographically random `nonce` and includes it in the authentication request
2. Stores the `nonce` in the server-side session before redirect
3. Validates that the `nonce` claim in the received ID Token matches the stored value
4. Rejects tokens where the nonce does not match

## Cross-cutting Architecture Observations

1. **Authentication Delegation Pattern**: The entire OIDC client functionality is delegated to the `asfquart` library. While this is architecturally sound (centralized security controls), it means compliance with ASVS 10.5.x requirements cannot be verified from the application code alone. The security posture depends entirely on the correctness of `asfquart`.

2. **Not an OAuth Authorization Server**: This application is purely an OIDC relying party / OAuth client. Requirements 10.4.8 and 10.4.9 are not applicable to this codebase as it does not issue, manage, or revoke tokens.

3. **Single Identity Provider Architecture**: The application appears to use only the ASF identity provider, which simplifies OIDC security (single issuer, single trust relationship) but the lack of visible configuration makes it impossible to verify that issuer pinning is enforced.

4. **Session-Based Security Model**: After initial OIDC authentication, the application operates on server-side sessions rather than tokens. This means token replay/audience attacks are limited to the initial authentication flow (handled by `asfquart`), not to subsequent requests.

5. **Evolving Authentication**: Comments like `### need general solution` on auth decorators suggest the authentication/authorization system is still under development, which increases risk of security gaps.

## Cross-cutting Recommendations

### Immediate
1. **Audit `asfquart` Library**: The provided code cannot satisfy ASVS 10.5.x verification without examining the `asfquart` library. Conduct a focused audit of `asfquart.auth` and `asfquart.session` for:
   - Nonce generation and validation (10.5.1)
   - `sub` claim usage for user identification (10.5.2)
   - Issuer URL validation (10.5.3)
   - Audience claim validation (10.5.4)

2. **Document Security Assumptions**: Create explicit documentation stating which OIDC security controls are expected from `asfquart` vs. implemented in the application.

### Short-term
3. **Add Integration Tests**: Write tests that verify the OIDC flow end-to-end, including:
   - Rejection of tokens with incorrect nonce
   - Rejection of tokens with mismatched audience
   - Rejection of tokens from unauthorized issuers
   - Proper mapping of `sub` claim to `uid`/`pid`

4. **Configuration Visibility**: Surface the OIDC client configuration (issuer URL, client_id) in the application configuration rather than burying it in the library, making security audits more straightforward.

### Long-term
5. **Resolve Auth TODOs**: Address all `### need general solution` and `### check authz` placeholders to ensure the authorization model is complete and enforceable.

6. **Consider Multi-IdP Support**: If the application may support multiple identity providers in the future, implement issuer-scoped user identification (`issuer` + `sub` as the unique key) now to prevent account confusion attacks later.
