## ASVS-15.2.3: Security Architecture and Dependencies - Extraneous Functionality

### Findings for 15.2.3

### [HIGH]

**Finding ID**: ASVS-1523-HIGH-001
**Location**: `v3/server/pages.py` — `do_add_issue_endpoint()` (line 476), `do_edit_issue_endpoint()` (line 497)
**Issue**: Debug `print()` statements present in production endpoint code

```python
@APP.post('/do-add-issue/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_add_issue_endpoint(election):
    # ...
    form = edict(await quart.request.form)
    print('FORM:', form)  # DEBUG: prints all form data to stdout/logs
```

```python
@APP.post('/do-edit-issue/<eid>/<iid>')
@asfquart.auth.require({R.committer})
@load_election_issue
async def do_edit_issue_endpoint(election, issue):
    # ...
    form = edict(await quart.request.form)
    print('FORM:', form)  # DEBUG: prints all form data to stdout/logs
```

**Impact**: In production, these print statements output form data (including election titles, descriptions) to process stdout. This is debug code that:
1. Clutters production logs with sensitive data
2. May expose form data in shared logging systems
3. Indicates incomplete development practices in production code

**Remediation**: Remove debug prints or replace with proper logging:
```python
_LOGGER.debug(f'Form data received for issue in election[E:{election.eid}]')
```

### [MEDIUM]

**Finding ID**: ASVS-1523-MED-001
**Location**: `v3/server/pages.py` — `postprocess_election()` (line 619)
**Issue**: Hardcoded placeholder value in production code

```python
def postprocess_election(e):
    # ...
    ### temporary. need to adjust input query.
    if 'issue_count' not in e:
        e.issue_count = 5  ### arbitrary. just provide a value
```

This is explicitly marked as temporary test/placeholder code with comments acknowledging it provides an arbitrary value. This masks real data issues and provides incorrect information to users.

**Impact**: Users see incorrect issue counts (always "5" when data is missing), creating confusion and indicating incomplete/test functionality in production.

**Remediation**: Fix the underlying query to always return `issue_count`, then remove the placeholder:
```python
def postprocess_election(e):
    if 'issue_count' not in e:
        e.issue_count = 0  # Default for elections without issues
```

### [MEDIUM]

**Finding ID**: ASVS-1523-MED-002
**Location**: `v3/server/pages.py` — `basic_info()` (line 83)
**Issue**: Placeholder CSRF token exposed in production

```python
async def basic_info():
    # ...
    ### generate a real token and store in the session
    basic.csrf_token = 'placeholder'
    return basic
```

A non-functional placeholder token is rendered into every page template (see `manage.ezt` line 11):
```html
<input type="hidden" id="csrf-token" value="[csrf_token]">
```

This is development stub code that:
1. Provides false confidence that CSRF protection exists
2. Exposes the word "placeholder" as a static value an attacker can trivially use
3. Is explicitly noted as needing replacement

**Impact**: This development-stage stub in production means all POST forms send a token that is never validated, making the application appear protected while being fully vulnerable to CSRF.

## Cross-cutting Architecture Observations

1. **Documentation Maturity Gap**: The project has good inline code documentation (`ARCHITECTURE.md`, `schema.md`, `tally.md`) but lacks formal security architecture documentation. Security decisions are documented in comments and TODOs rather than dedicated security documents.

2. **Development vs. Production Boundary**: Multiple indicators suggest the codebase is in an active development state being deployed (or ready to deploy) with development artifacts still present. The `TODO.md` essentially serves as both a bug tracker and security audit log.

3. **Authentication vs. Authorization**: The project correctly implements authentication (via ASF OAuth/session), but authorization (ownership checks, per-election access control) is consistently marked with placeholder comments (`### check authz`). This creates a Type B gap — the decorator exists but fine-grained authorization is not implemented.

4. **Cryptographic Design is Strong**: The vote encryption/anonymization design (Argon2 KDF, per-voter salts, encrypted ballots, anti-tamper hashing) is well-implemented. However, the deliberate slowness of these operations creates availability concerns that are only partially addressed.

5. **Dependency Management Awareness**: Version constraints show intentional dependency management, but no formal vulnerability tracking or update policy exists.

## Cross-cutting Recommendations

### Immediate (0-7 days)
1. **Remove debug prints** from `pages.py` (`do_add_issue_endpoint`, `do_edit_issue_endpoint`) — trivial fix
2. **Remove/fix placeholder `issue_count = 5`** — incorrect data shown to users
3. **Remove `_REMAP` hack** from `create-election.py` — test code in production tool

### Short-term (1-4 weeks)
4. **Create Security Architecture Document** addressing:
   - Dangerous functionality registry (ASVS 15.1.5)
   - Risky component assessment (ASVS 15.1.4)
   - Resource-demanding functionality inventory with mitigations (ASVS 15.1.3)
   - Dependency update policy with timeframes (ASVS 15.2.1)
5. **Implement rate limiting** on Argon2-heavy endpoints (`/do-vote/<eid>`, `/do-open/<eid>`)
6. **Implement real CSRF tokens** — replace placeholder, add validation middleware

### Long-term (1-3 months)
7. **Implement authorization layer** — replace all `### check authz` placeholders with ownership/permission verification
8. **Establish CI/CD pipeline** with:
   - Automated dependency vulnerability scanning (e.g., `pip-audit`, `safety`)
   - Linting for debug prints and placeholder code
   - Separation of dev/test scripts from production distribution
9. **Add async/queue processing** for election `open()` operations with many voters
10. **Document and enforce consumer timeout strategy** for all web endpoints interacting with Argon2
