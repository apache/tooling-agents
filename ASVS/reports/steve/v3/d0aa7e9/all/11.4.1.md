## ASVS-11.4.1: Approved Hash Functions

### Findings for 11.4.1

### [LOW]

**Finding ID**: ASVS-1141-LOW-001  
**File**: `v3/steve/crypto.py`, function `gen_opened_key`, line 45  

```python
def gen_opened_key(edata: bytes, salt: bytes) -> bytes:
    "Generate the OpenedKey for this election."

    # The data is of arbitrary length. Use BLAKE2b to quickly hash all
    # this down into a manageable size for Argon2.
    # Note: BLAKE2b is the internal primitive of Argon2, so a good fit.
    digest = hashlib.blake2b(edata).digest()
```

**Impact**: BLAKE2b is a modern, secure hash function (RFC 7693) with a 512-bit output, providing strong collision resistance. However, it is not listed in NIST FIPS 180-4 (SHA-2) or FIPS 202 (SHA-3) as an "approved" hash function. Depending on organizational compliance requirements, this could be a concern. In practice, BLAKE2b is used within Argon2 itself (which is NIST-recognized via SP 800-63B), making this a very low-risk finding.  
**Remediation**: If strict NIST compliance is required, replace with SHA-512:

```python
import hashlib
digest = hashlib.sha512(edata).digest()  # 64 bytes, NIST-approved
```

## Cross-cutting Architecture Observations

1. **Cryptographic Architecture is Well-Designed**: The multi-layered approach (election data → BLAKE2b → Argon2 → opened_key; then opened_key + identifiers → Argon2 → vote_token → HKDF → encryption key) provides defense in depth with appropriate separation of concerns.

2. **Argon2 Variant Inconsistency**: The benchmark function uses `Type.ID` (Argon2id) while production uses `Type.D` (Argon2d). This appears to be an oversight rather than a deliberate design choice, as there's no documented rationale for choosing Argon2d.

3. **Fernet as Transitional Algorithm**: The code explicitly marks Fernet as temporary (comments mention switching to XChaCha20-Poly1305). The HKDF `info` parameter is already set for XChaCha20, which could cause key equivalence issues during migration if not properly managed.

4. **No Password Storage**: The system delegates authentication to external infrastructure (ASF/LDAP), meaning traditional password hashing requirements (11.4.2) have limited direct applicability. The Argon2 usage is for key derivation and anti-tamper rather than password verification.

5. **Anti-Tamper Integrity**: The `opened_key` mechanism provides cryptographic assurance that election configuration hasn't been modified after opening — a strong integrity control binding election metadata, issues, and voter lists.

## Cross-cutting Recommendations

### Immediate (Critical/High Priority)
1. **Switch Argon2 variant to Argon2id** (`Type.ID`): Single-line change in `_hash()` function, aligning production with benchmark code and OWASP recommendations. This affects all derived keys.

### Short-term (Medium Priority)
2. **Plan migration path for Fernet → XChaCha20-Poly1305**: Ensure the HKDF `info` parameter changes during migration to prevent key reuse across algorithms. Document the migration strategy for existing encrypted votes.
3. **Review Argon2 parameters against current hardware**: Run `benchmark_argon2()` on production hardware to determine if `time_cost` can be increased (current value of 2 is the minimum recommended).

### Long-term (Low Priority)
4. **Consider BLAKE2b → SHA-512 substitution**: If strict NIST compliance is required, replace BLAKE2b with SHA-512 for the data reduction step. Functionally equivalent with no performance concern for the single-call use case.
5. **Add cryptographic agility documentation**: Document the algorithm choices, migration strategy, and key derivation chain in a security design document to support future audits.
