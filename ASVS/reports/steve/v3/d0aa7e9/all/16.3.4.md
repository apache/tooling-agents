## ASVS-16.3.4: Unexpected Errors and Security Control Failures

### Findings for 16.3.4

### [HIGH]
**Finding ID:** ASVS-1634-HIGH-001
**Location:** `v3/server/pages.py`, `do_open_endpoint()` (line 431)
**Vulnerable Code:**
```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()

    ### check authz

    pdb = steve.persondb.PersonDB.open(DB_FNAME)

    # Open the Election.
    election.open(pdb)  # No try/except - assertion errors, crypto failures, DB errors all unhandled

    _LOGGER.info(f'User[U:{result.uid}] opened election[E:{election.eid}]')
```
**Data Flow:** Request → `election.open(pdb)` → potential failure in crypto/DB → unhandled exception → framework error → no application-level log
**Proof of Concept:** If `crypto.gen_salt()`, `crypto.gen_opened_key()`, or the database transaction in `add_salts()` fails, the error propagates as an unlogged exception.
**Impact:** Critical security operations (salt generation, key derivation, database transactions) can fail without application-level error logging. The same pattern affects `do_close_endpoint()`, `do_create_endpoint()`, and `do_add_issue_endpoint()`.
**Similar locations:**
- `do_close_endpoint()` — line 451
- `do_create_endpoint()` — line 410
- `do_add_issue_endpoint()` — line 469
- `do_edit_issue_endpoint()` — line 493

**Remediation:**
```python
async def do_open_endpoint(election):
    result = await basic_info()
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    try:
        election.open(pdb)
    except Exception as e:
        _LOGGER.error(
            f'SECURITY_CONTROL_FAILURE: Failed to open election[E:{election.eid}] '
            f'by user[U:{result.uid}]: {type(e).__name__}',
            exc_info=True
        )
        await flash_danger('Failed to open election. Please contact an administrator.')
        return quart.redirect(f'/manage/{election.eid}', code=303)
    _LOGGER.info(f'User[U:{result.uid}] opened election[E:{election.eid}]')
```

### [HIGH]
**Finding ID:** ASVS-1634-HIGH-002
**Location:** `v3/steve/election.py`, `add_salts()` (line 116) and `open()` (line 70)
**Vulnerable Code:**
```python
def add_salts(self):
    "Set the SALT column in the MAYVOTE table."
    assert self.is_editable()

    self.db.conn.execute('BEGIN TRANSACTION')

    self.q_all_issues.perform(self.eid)
    for mayvote in self.q_all_issues.fetchall():
        salt = crypto.gen_salt()
        self.c_salt_mayvote.perform(salt, mayvote.rowid)

    self.db.conn.execute('COMMIT')
    # No error handling - if COMMIT fails or gen_salt() fails mid-transaction,
    # partial state is left and no error is logged
```
**Data Flow:** Transaction start → salt operations → potential failure → no rollback logic → no error log
**Impact:** If a failure occurs during the transaction (e.g., disk full, crypto failure), the transaction may be left in an inconsistent state. There's no `try/except` with `ROLLBACK` and no error logging for these security-critical operations.
**Remediation:**
```python
def add_salts(self):
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
    
    try:
        self.db.conn.execute('BEGIN TRANSACTION')
        self.q_all_issues.perform(self.eid)
        for mayvote in self.q_all_issues.fetchall():
            salt = crypto.gen_salt()
            self.c_salt_mayvote.perform(salt, mayvote.rowid)
        self.db.conn.execute('COMMIT')
    except Exception as e:
        self.db.conn.execute('ROLLBACK')
        _LOGGER.error(f'SECURITY_FAILURE: Failed to add salts for election[E:{self.eid}]: {type(e).__name__}')
        raise
```

### [MEDIUM]
**Finding ID:** ASVS-1634-MED-001
**Location:** `v3/server/pages.py`, `serve_doc()` (line 560) and other endpoints
**Vulnerable Code:**
```python
@APP.get('/docs/<iid>/<docname>')
@asfquart.auth.require
async def serve_doc(iid, docname):
    result = await basic_info()

    db = steve.election.Election.open_database(DB_FNAME)
    row = db.q_get_mayvote.first_row(result.uid, iid)
    if not row:
        quart.abort(404)
        # NOTREACHED

    ### verify the propriety of DOCNAME.

    # Return per-issue document.
    return await quart.send_from_directory(DOCSDIR / iid, docname)
    # No error handling if send_from_directory fails (file not found, permission error, etc.)
```
**Data Flow:** File request → `send_from_directory()` → potential IO error → unlogged
**Impact:** File system errors (permission denied, file not found after auth check passes, symlink resolution failures) would not be logged as unexpected errors. The comment `### verify the propriety of DOCNAME.` indicates a known gap in security controls.
**Remediation:**
```python
try:
    return await quart.send_from_directory(DOCSDIR / iid, docname)
except Exception as e:
    _LOGGER.error(f'FILE_SERVE_ERROR: Failed to serve doc={docname} for issue={iid}: {type(e).__name__}')
    quart.abort(404)
```

## Cross-cutting Architecture Observations

1. **Logging maturity is low** — The application has action logging for administrative operations but lacks authentication event logging, authorization failure logging, input validation failure logging, and error logging across most endpoints. Only `do_vote_endpoint()` demonstrates the full pattern of try/except with error logging.

2. **Type B gaps are pervasive** — The codebase contains numerous `### check authz` comments indicating the developer intended to add authorization checks. These are never implemented, creating a false sense of security and making authorization failure logging impossible.

3. **No centralized logging infrastructure** — There is no logging middleware, no structured log format (e.g., JSON), no log sanitization layer, and no consistent error handling pattern. Each endpoint handles (or doesn't handle) logging independently.

4. **Debug code in production paths** — `print('FORM:', form)` statements exist in two production endpoints, bypassing all logging controls and dumping raw user data.

5. **Security assertions vs. proper error handling** — The `election.py` module uses Python `assert` statements for security-critical state checks. These are: (a) disabled with `-O` flag, (b) generate no log entries when they fire, and (c) provide no graceful error handling.

6. **Single responsibility gap** — The authentication framework (`asfquart.auth`) handles enforcement but there's no evidence it handles logging. The application layer logs actions but not auth events. This creates a gap where neither layer owns auth logging.

## Cross-cutting Recommendations

### Immediate (Critical/High priority)
1. **Remove debug `print()` statements** — Lines 481 and 497 in `pages.py` dump raw form data to stdout. Remove immediately.
2. **Implement log sanitization** — Create a wrapper or formatter that strips control characters (`\r`, `\n`, `\x00`-`\x1f`) from all logged values before output.
3. **Add error handling with logging to all state-changing endpoints** — Wrap `election.open()`, `election.close()`, `Election.create()` in try/except with `_LOGGER.error()`.
4. **Log authorization denials** — Add `_LOGGER.warning()` calls in `vote_on_page()` and `serve_doc()` where access is denied.

### Short-term (1-2 sprints)
5. **Implement authentication event logging** — Add a middleware or decorator that logs all authentication successes and failures with metadata (user, endpoint, method, IP, auth type).
6. **Replace `assert` with proper exceptions** — Convert all security-relevant `assert` statements in `election.py` to explicit checks with `ElectionBadState` exceptions and logging.
7. **Implement authorization checks** — Address all `### check authz` placeholders with ownership verification and log failures.
8. **Add input validation failure logging** — Log when date parsing fails, when invalid IIDs are submitted, and when other validation rejects input.

### Long-term (Architecture)
9. **Adopt structured logging (JSON)** — Migrate to structured log format with consistent fields (`timestamp`, `level`, `event_type`, `user_id`, `resource_id`, `action`, `outcome`) for machine-readable analysis.
10. **Implement centralized security event framework** — Create a `SecurityEvent` class or module that provides typed events (`AUTH_SUCCESS`, `AUTH_FAILURE`, `AUTHZ_DENIED`, `INPUT_VALIDATION_FAILED`, `BUSINESS_LOGIC_VIOLATION`) with automatic metadata collection.
11. **Add anti-automation detection** — Implement rate limiting with logging of threshold violations to detect brute-force or automated abuse.
12. **PII handling policy in logs** — Establish a logging policy that defines which fields may be logged in full, which must be masked, and which must never be logged, then enforce via the structured logging framework.
