## ASVS-4.3.1: GraphQL DoS Prevention

### Findings for 4.3.1

No findings detected.

The codebase does not implement any GraphQL API. The application is built on the Quart framework using traditional HTTP REST-style routes (GET/POST) with form submissions and JSON payloads. There are no GraphQL schemas, resolvers, or GraphQL-related dependencies (e.g., `graphene`, `ariadne`, `strawberry`, `graphql-core`) present in any of the provided source files.

**Evidence of non-applicability:**
- `pages.py` exclusively uses `@APP.get(...)` and `@APP.post(...)` decorators for standard HTTP routes
- Data is exchanged via HTML forms (`quart.request.form`) and JSON (`quart.request.get_json()`)
- No GraphQL endpoint (e.g., `/graphql`) is defined
- No GraphQL library imports exist

### Security Controls Inventory for 4.3.1

No applicable controls in this scope. GraphQL is not implemented in this application.

### Positive Patterns for 4.3.1

- The application uses traditional REST-style endpoints with well-defined route parameters, limiting the attack surface for complex query-based DoS attacks that GraphQL would introduce.
- Each endpoint has a fixed, predictable data retrieval pattern (no user-controlled query depth or nesting).

## Cross-cutting Architecture Observations

1. **Technology Stack Alignment**: The application is a traditional server-rendered web application using Quart (async Flask-like framework) with EZT templates. It does not employ GraphQL or WebSocket technologies. All six audited ASVS requirements (4.3.1, 4.3.2, 4.4.1, 4.4.2, 4.4.3, 4.4.4) are **not applicable** to this codebase in its current form.

2. **Communication Pattern**: The application uses:
   - Standard HTTP GET/POST for page loads and form submissions
   - `fetch()` with JSON payloads for AJAX-style date saving (in `manage.ezt`)
   - No real-time bidirectional communication channels

3. **Future Risk**: If the application were to add real-time features (e.g., live vote counting, live election status updates), WebSocket might be introduced. Similarly, if a mobile app or third-party integration layer were planned, GraphQL might be considered. At that point, these ASVS requirements would become applicable.

4. **Out-of-Scope but Notable Observations** (not part of the audited requirements but observed during analysis):
   - CSRF token is a placeholder (`'placeholder'` on line 83) with no server-side validation
   - Multiple `### check authz` placeholders indicate incomplete authorization enforcement
   - The `serve_doc()` function has a commented note about verifying `docname` propriety (potential path traversal)
   - State-changing operations (`/do-open/<eid>`, `/do-close/<eid>`) use GET methods, which is an anti-pattern

## Cross-cutting Recommendations

**Immediate** (if adding GraphQL or WebSocket in future):
- N/A for current architecture — no GraphQL or WebSocket present

**Short-term** (architecture hygiene related to audited categories):
1. If real-time features are planned, ensure WebSocket implementation uses WSS exclusively, validates Origin headers, and ties into the existing `asfquart.session` mechanism
2. If a GraphQL layer is planned for API consumers, implement query complexity analysis (e.g., using `graphql-cost-analysis`) and disable introspection in production from day one

**Long-term** (general posture relevant to these requirement families):
1. Document the application's communication architecture to clarify that GraphQL and WebSocket are intentionally excluded, preventing accidental introduction without security review
2. If the `fetch()`-based AJAX pattern in `manage.ezt` expands significantly, consider whether a more structured API layer (REST or GraphQL) would benefit from centralized security controls
3. Maintain the current standard HTTP architecture unless there is a clear requirement for real-time bidirectional communication — it provides a simpler security model
