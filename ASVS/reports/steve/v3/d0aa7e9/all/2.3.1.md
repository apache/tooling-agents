## ASVS-2.3.1: Business Logic Sequential Step Order

### Findings for 2.3.1

### [CRITICAL]

**Finding ID**: ASVS-231-CRIT-001
**Title**: Election State Enforcement Uses Python `assert` Statements That Can Be Disabled

**Location**: `v3/steve/election.py`, methods `open()` (line 70), `close()` (line 109), `add_issue()` (line 169), `edit_issue()` (line 184), `delete_issue()` (line 196), `add_voter()` (line 225), `add_salts()` (line 119), `delete()` (line 53)

**Vulnerable code**:
```python
def open(self, pdb):
    # Double-check the Election is in the editing state.
    assert self.is_editable()
    ...

def close(self):
    "Close an election."
    # The Election should be open.
    assert self.is_open()
    ...

def add_issue(self, title, description, vtype, kv):
    "Add a new issue with a generated unique IID."
    assert self.is_editable()
    assert vtype in vtypes.TYPES
    ...

def delete(self):
    # Can't delete if it has been opened (even if later closed).
    assert self.is_editable()
    ...
```

**Data flow**: User request → endpoint handler → `election.open()`/`election.close()`/`election.add_issue()` → `assert self.is_editable()` (SKIPPED if Python run with `-O` flag)

**Proof of Concept**: If the application is run with `python -O` or `PYTHONOPTIMIZE=1`, all `assert` statements are compiled away. An attacker could then:
1. Call `/do-open/<eid>` on an already-open election
2. Call `/do-close/<eid>` on an editable (never opened) election  
3. Add issues to an open or closed election
4. Delete issues from an open election

**Impact**: Complete bypass of the election lifecycle state machine. An attacker (any committer) could corrupt election integrity by performing operations out of order, adding/deleting issues during voting, or closing elections that were never properly opened.

**Remediation**:
```python
def open(self, pdb):
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
    ...

def close(self):
    if not self.is_open():
        raise ElectionBadState(self.eid, self.get_state(), self.S_OPEN)
    ...

def add_issue(self, title, description, vtype, kv):
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
    if vtype not in vtypes.TYPES:
        raise ValueError(f'Invalid vote type: {vtype}')
    ...
```

### [HIGH]

**Finding ID**: ASVS-231-HIGH-001
**Title**: State-Changing Operations Use GET Requests, Bypassing Sequential User Confirmation

**Location**: `v3/server/pages.py`, `do_open_endpoint()` (line 431), `do_close_endpoint()` (line 451)

**Vulnerable code**:
```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_open_endpoint(election):
    ...
    election.open(pdb)
    ...

@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})  ### need general solution
@load_election
async def do_close_endpoint(election):
    ...
    election.close()
    ...
```

**Data flow**: Browser prefetch / link click / CSRF → GET `/do-open/<eid>` → `election.open()` → irreversible state change

**Proof of Concept**: 
1. Embed `<img src="/do-open/abc1234567">` in any page a committer visits
2. Browser prefetch extensions may preload the URL
3. The confirmation modal in `manage.ezt` is purely client-side JavaScript and provides no server-side sequential enforcement: `window.location.href = '/do-open/[eid]'`

**Impact**: Irreversible election state changes can be triggered without user intentionally following the expected step sequence (view management page → confirm modal → execute). The intended sequential flow (confirmation) exists only client-side.

**Remediation**: Change to POST endpoints with server-side confirmation token:
```python
@APP.post('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    form = await quart.request.form
    # Verify confirmation token from session
    if form.get('confirm_token') != session_confirm_token:
        quart.abort(403)
    election.open(pdb)
    ...
```

### [MEDIUM]

**Finding ID**: ASVS-231-MED-001
**Title**: No Prerequisite Validation Before Election Opening

**Location**: `v3/steve/election.py`, `open()` method (line 70); `v3/server/pages.py`, `do_open_endpoint()` (line 431)

**Vulnerable code**:
```python
def open(self, pdb):
    # Double-check the Election is in the editing state.
    assert self.is_editable()

    # Add salts first. If this is gonna fail, then make sure it
    # happens before we move to the "opened" state.
    self.add_salts()

    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)
    self.c_open.perform(salt, opened_key, self.eid)
```

**Data flow**: Any committer → `/do-open/<eid>` → `election.open()` → election opened with zero issues/voters

**Proof of Concept**: Create an election, then immediately call `/do-open/<eid>` without adding any issues or voters. The election opens successfully but is useless and irreversible.

**Impact**: An election can be opened without any issues or eligible voters, which is logically invalid and irreversible. This represents a skipped step in the business flow (configure → validate → open).

**Remediation**:
```python
def open(self, pdb):
    if not self.is_editable():
        raise ElectionBadState(...)
    
    # Verify prerequisites
    issues = self.list_issues()
    if not issues:
        raise ValueError("Cannot open election with no issues")
    
    self.q_voting_persons.perform(self.eid)
    voters = self.q_voting_persons.fetchall()
    if not voters:
        raise ValueError("Cannot open election with no eligible voters")
    ...
```

- The `add_vote()` method uses `self._all_metadata(self.S_OPEN)` which properly raises `ElectionBadState` instead of using `assert` — this is the correct pattern.

## Cross-cutting Architecture Observations

1. **Type B Gaps Dominate**: The most critical pattern is controls that EXIST in the codebase (owner_pid, authz, CSRF tokens, state validation) but are NOT APPLIED at the enforcement point. This creates false confidence — the schema and comments suggest security is in place, but enforcement is absent.

2. **Assert vs. Exception Pattern Inconsistency**: The codebase uses `assert` for state checks in some methods (`open()`, `close()`, `add_issue()`) but proper exceptions in others (`add_vote()` via `_all_metadata(required_state)`). This inconsistency means security enforcement depends on which code path is taken.

3. **GET for State Changes**: The use of GET requests for irreversible operations (`/do-open/<eid>`, `/do-close/<eid>`) combines with the CSRF gap to create a particularly dangerous attack surface.

4. **Single-Layer Defense**: Security controls exist primarily at the model layer (`election.py`) with minimal web-layer enforcement. The web layer (pages.py) trusts that the model will enforce constraints, but the model uses bypassable assertions.

5. **SQLite Concurrency Model**: The application relies on SQLite's implicit serialization for concurrency safety, but the check-then-act patterns in the application layer create TOCTOU windows that could be exploited in multi-process deployments.

6. **Audit Trail Strength**: Logging of user actions is consistently implemented, providing after-the-fact accountability even where preventive controls are missing.

## Cross-cutting Recommendations

### Immediate (P0 — Address Before Production Use)

1. **Replace all `assert` statements** with proper exception raising (`ElectionBadState`) in `election.py`. This is the most critical fix as it affects all state enforcement.

2. **Implement authorization checks** in all management endpoints — compare `result.uid` against `election.owner_pid`/`election.authz`. This closes the most impactful Type B gap.

3. **Implement CSRF protection** — generate real tokens per session and validate them in all POST endpoints. Change `/do-open/` and `/do-close/` to POST methods.

4. **Add vote string validation** in `add_vote()` — validate against issue type before encryption.

### Short-term (P1 — Within Next Sprint)

5. **Wrap vote submission in a transaction** — ensure atomic ballot submission in `do_vote_endpoint()`.

6. **Make election open operation atomic** — combine `add_salts()` and the state change in a single transaction.

7. **Add rate limiting** on all mutating endpoints, prioritizing `/do-vote/<eid>` and `/do-create-election`.

8. **Add prerequisite validation** before election opening (verify issues exist, voters are registered).

### Long-term (P2 — Architecture Improvements)

9. **Implement multi-user approval** for irreversible state changes (open/close elections), particularly for elections with many eligible voters.

10. **Add explicit optimistic locking** (e.g., version column) on the election table to prevent concurrent state modifications.

11. **Re-enable and complete transactions** in the `create-election.py` script.

12. **Consider vote garbage collection** — periodically remove superseded vote rows to prevent unbounded table growth.
