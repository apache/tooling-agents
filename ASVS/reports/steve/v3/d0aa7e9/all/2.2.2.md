## ASVS-2.2.2: Input Validation at Trusted Service Layer

### Findings for 2.2.2

### [HIGH]
**Finding ID: ASVS-222-HIGH-001**
**Title: State-modifying operations use GET requests without server-side CSRF validation**

**File:** `v3/server/pages.py`, lines 431-447 (do_open_endpoint), 451-465 (do_close_endpoint)
**Functions:** `do_open_endpoint()`, `do_close_endpoint()`

Critical state-changing operations (opening and closing elections) are exposed as GET endpoints with no CSRF token validation on the server side:

```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    ### check authz
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)  # Irreversible state change!
    # ...

@APP.get('/do-close/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_close_endpoint(election):
    result = await basic_info()
    ### check authz
    election.close()  # Irreversible state change!
    # ...
```

**Client-side trigger** (manage.ezt):
```javascript
confirmOpen.addEventListener('click', () => {
    openModal.hide();
    window.location.href = '/do-open/[eid]';  // GET request
});
```

**Gap Type: Type B** — CSRF token control EXISTS (placeholder in `basic_info()`) but is NOT VALIDATED server-side, AND the endpoint uses GET instead of POST.

**Data flow:** Browser GET request (or `<img>` tag/link from attacker site) → `/do-open/<eid>` → `election.open()` → irreversible state change

**Proof of concept:**
An attacker could embed this in any page visited by the election admin:
```html
<img src="https://target-app/do-open/1a2b3c4d5e" style="display:none">
```

**Impact:** An attacker who knows an election EID can trick an authenticated admin into opening or closing an election via CSRF. These are irreversible operations that permanently change election state.

**Remediation:**
1. Change to POST methods
2. Implement real CSRF token generation and validation:
```python
@APP.post('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@csrf_required  # decorator that validates token
@load_election
async def do_open_endpoint(election):
    # ...
```

### [MEDIUM]
**Finding ID: ASVS-222-MED-001**
**Title: Client-side required field validation not replicated server-side**

**File:** `v3/server/pages.py` (do_add_issue_endpoint, do_create_endpoint)
**Templates:** `v3/server/templates/manage.ezt` (line 92)

The HTML form uses client-side `required` attribute and JavaScript validation:
```html
<input type="text" class="form-control" id="issueTitle" name="title" required ...>
```

```javascript
function saveIssue() {
    if (!validateRequiredField('issueTitle')) {
        return;
    }
    submitFormWithLoading('issueForm', 'saveIssueBtn', 'Saving...');
}
```

But the server-side handler does NOT verify the field is non-empty:
```python
async def do_add_issue_endpoint(election):
    form = edict(await quart.request.form)
    # No check: if not form.title or not form.title.strip()
    iid = election.add_issue(form.title, form.description, vtype, kv)
```

**Gap Type: Type A** — Server-side required-field validation is absent.

**Proof of concept:**
```http
POST /do-add-issue/1a2b3c4d5e HTTP/1.1
Content-Type: application/x-www-form-urlencoded

title=&description=
```
Bypasses client-side `required` attribute, creates an issue with empty title.

**Impact:** Empty or whitespace-only titles can be stored, violating data quality expectations. The database `NOT NULL` constraint doesn't prevent empty strings.

**Remediation:**
```python
async def do_add_issue_endpoint(election):
    form = edict(await quart.request.form)
    title = form.get('title', '').strip()
    if not title:
        await flash_danger('Issue title is required.')
        return quart.redirect(f'/manage/{election.eid}', code=303)
    # ...
```

### Positive Patterns for 2.2.2

- **Authentication enforcement at server layer:** All protected endpoints use `@asfquart.auth.require` decorators, enforced at the framework level before reaching handler code.
- **Database-level constraints as defense-in-depth:** CHECK constraints and FOREIGN KEY relationships enforce data integrity regardless of application-layer validation.
- **Server-side date validation:** `_set_election_date()` validates date format server-side, not relying on client-side input type.
- **Server-side state assertions:** All state-changing operations in `election.py` use `assert` to verify the election is in the correct state before proceeding.

## Cross-cutting Architecture Observations

1. **Documentation vs. Implementation Gap:** The application has solid database-level documentation (`schema.md`) describing data types and constraints, but lacks a separate input validation specification document. The database schema enforces internal ID formats but not user-input content rules.

2. **Defense-in-Depth Pattern:** The architecture employs multiple layers — database CHECK constraints, foreign keys, application-level state assertions, and authentication decorators. However, the input validation layer (between user input and business logic) is largely absent.

3. **Type A Gaps Dominate:** Most vulnerabilities are "entry point with no control" rather than controls that exist but aren't applied. This suggests input validation was deferred during development (evidenced by multiple TODO/`###` comments).

4. **Asymmetric Validation:** The CLI tool (`create-election.py`) has more validation (STV kv fields, seat count, PID existence) than the web endpoints for equivalent operations. This creates inconsistent security posture between entry points.

5. **State Machine is Well-Implemented:** The election lifecycle (editable → open → closed) is consistently enforced via assertions in `election.py`. This is a strong positive architectural pattern for business logic.

6. **Cryptographic Layer is Solid:** Vote encryption, token generation, and anti-tamper detection are well-structured and correctly separate concerns from the input validation layer.

7. **GET for State Changes:** Using GET requests for irreversible operations (`do-open`, `do-close`) violates REST conventions and makes CSRF prevention more difficult.

## Cross-cutting Recommendations

### Immediate (Critical/High Priority)

1. **Implement votestring validation** in `Election.add_vote()` — this is the core function of the application and currently accepts arbitrary input. Create `validate_vote()` methods in each vtype module.

2. **Convert state-changing GET endpoints to POST** (`/do-open/<eid>`, `/do-close/<eid>`) and implement CSRF token generation and validation. Replace the placeholder token.

3. **Add server-side required field validation** for all form-submitted text fields (title, description) in `do_create_endpoint`, `do_add_issue_endpoint`, and `do_edit_issue_endpoint`.

### Short-term (Medium Priority)

4. **Add cross-field date validation** ensuring `close_at > open_at` when both are set, and consider restricting past dates.

5. **Implement explicit mayvote NULL check** in `add_vote()` to provide clear error handling rather than relying on AttributeError.

6. **Add length limits** on text input fields (title: 200 chars, description: 10,000 chars) both in validation code and as database CHECK constraints.

7. **Create an input validation specification document** covering all user-input fields, their formats, constraints, and cross-field rules.

8. **Validate `docname`** explicitly with an allowlist pattern, even though `send_from_directory` provides framework protection.

### Long-term (Architectural)

9. **Implement authorization (authz) checks** — currently all `### check authz` comments indicate no ownership/permission validation beyond basic authentication.

10. **Create a centralized input validation module** that can be reused across endpoints, rather than inline validation in each handler.

11. **Add per-user and global rate limits** for election creation, issue management, and vote submission to prevent abuse.

12. **Document all business logic rules** in a dedicated specification, including state transitions, temporal constraints, limits, and cross-entity consistency rules.
