## ASVS-11.6.1: Approved Cryptographic Algorithms and Modes

### Findings for 11.6.1

### [MEDIUM]

**Finding ID**: ASVS-1161-MED-001

**File**: `v3/steve/crypto.py`, function `_hash()`, line 84

**Vulnerable code:**
```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2 to hash the data, with default tuning parameters."

    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,  # Passlib default
        memory_cost=65536,  # Passlib default
        parallelism=4,  # Passlib default
        hash_len=32,  # Standard Argon2 digest length
        type=argon2.low_level.Type.D,
    )
```

**Data flow**: `_hash()` → called by `gen_opened_key()` and `gen_vote_token()` → derives security-critical keys using Argon2**d** variant

**Issue**: RFC 9106 (Argon2) recommends **Argon2id** as the primary variant for most use cases. Argon2d is vulnerable to side-channel attacks (timing attacks, cache-timing attacks) because its memory access pattern depends on the password/secret. In a shared hosting environment or when an attacker has local access, this could leak information about the derived keys.

**Evidence of awareness**: The `benchmark_argon2()` function on line 108 uses `argon2.low_level.Type.ID` (Argon2id), showing the developers are aware of this variant but chose Argon2d for production.

**Impact**: In environments with shared resources (cloud, containerized deployments), side-channel attacks against Argon2d could potentially recover vote tokens or the opened_key, compromising vote secrecy or enabling tamper of the anti-tamper mechanism.

**Remediation**:
```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2id to hash the data, with default tuning parameters."

    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.ID,  # Argon2id: recommended variant
    )
```

### Positive Patterns for 11.6.1

1. **HKDF with appropriate parameters**: The `_b64_vote_key()` function uses HKDF-SHA256 with a proper salt and context info (`b'xchacha20_key'`), deriving a 32-byte key — approved per NIST SP 800-56C.
2. **Fernet provides authenticated encryption**: Using `cryptography.fernet.Fernet` ensures both confidentiality (AES-128-CBC) and integrity (HMAC-SHA256), preventing undetected ciphertext manipulation.
3. **BLAKE2b as pre-hash**: Using BLAKE2b to reduce arbitrary-length input before Argon2 is a sound approach that avoids Argon2's input size limitations.
4. **Appropriate key lengths**: 32-byte hash outputs from Argon2, 32-byte key from HKDF, 16-byte salts — all meet standard security levels.

## Cross-cutting Architecture Observations

1. **Cryptographic Design is Sound**: The overall cryptographic architecture (salt generation → Argon2 key derivation → HKDF key stretching → Fernet encryption) follows a well-structured layered approach with proper separation of concerns.

2. **ID Entropy vs. Authorization**: The low entropy in `create_id()` (40 bits) is partially mitigated by the incomplete but intended authorization layer. However, since authz is currently placeholder (`### check authz`), the low-entropy IDs are the *only* barrier to unauthorized access, making the entropy issue more critical in practice.

3. **Consistent Use of CSPRNG**: The codebase consistently uses `secrets` for security-sensitive randomness, with only the candidate display shuffle using `random` (acceptable for non-security display ordering).

4. **Anti-Tamper Mechanism**: The `opened_key` / `is_tampered()` design provides integrity verification for elections, but its effectiveness depends on the security of the underlying Argon2 derivation.

5. **Encryption Transition Planning**: The comments indicate a planned migration from Fernet (AES-128-CBC + HMAC-SHA256) to XChaCha20-Poly1305, which would be an improvement in terms of modern algorithm selection and nonce management.

## Cross-cutting Recommendations

### Immediate (Critical)
1. **Increase ID entropy to 128 bits**: Change `create_id()` to use `secrets.token_hex(16)` and update the database schema constraints. This is essential given the absence of proper authorization checks.

### Short-term (High)
2. **Switch from Argon2d to Argon2id**: Change `type=argon2.low_level.Type.D` to `type=argon2.low_level.Type.ID` in the `_hash()` function. This is a one-line change that improves resistance to side-channel attacks per RFC 9106 recommendations.

3. **Implement authorization checks**: The multiple `### check authz` placeholders create false confidence — the decorator-based auth ensures authentication but not authorization (ownership/permission checks).

### Long-term (Medium)
4. **Memory protection for sensitive operations**: Consider using `mlock()` for memory pages containing decrypted votes during tally operations, and implement explicit memory clearing patterns.

5. **Incremental hashing in `gather_election_data()`**: Refactor to use streaming hash computation instead of accumulating all voter data in memory.

6. **Complete the Fernet → XChaCha20-Poly1305 migration**: As noted in the code comments, this transition would bring the system to a more modern AEAD construction with larger nonce space.
