## ASVS-1.3.10: Format String Sanitization

### Findings for 1.3.10

No findings detected.

**Analysis:** The codebase uses Python f-strings exclusively for string formatting. F-strings are evaluated at the call site with a fixed format pattern — the format specification itself is never user-controlled. No use of `str.format()` with user-provided format strings, no `%`-operator formatting with user-controlled patterns, and no analogous mechanisms (like Java's `MessageFormat` or C's `printf`) that would allow format string attacks.

### Security Controls Inventory for 1.3.10

| Control | Location | Applied At | Coverage Status |
|---------|----------|------------|-----------------|
| F-string usage (compile-time patterns) | Throughout all files | All string formatting | **ADEQUATE** — Format patterns are hardcoded |

### Positive Patterns for 1.3.10

- Consistent use of Python f-strings (e.g., `f'User[U:{result.uid}] created election...'`) ensures format patterns are never dynamically constructed from user input.
- EZT template rendering separates data from format logic.

## Cross-cutting Architecture Observations

1. **Defense-in-Depth Gap:** The application relies primarily on authentication (`@asfquart.auth.require`) as its sole security layer. There is no input sanitization layer, no output encoding layer, and no content security policy mentioned.

2. **Input → Storage → Rendering Pipeline:** User input (election titles, issue descriptions) flows from forms → database → templates/emails without sanitization at any point in the chain. This creates a systemic vulnerability where any context (HTML, email, logs) that renders stored data is potentially exploitable.

3. **TODO-Driven Security:** Multiple security concerns are acknowledged via comments (`### check authz`, `### verify the propriety of DOCNAME`, CSRF placeholder) but remain unimplemented. This creates a "Type B" gap — the team is aware of the need but the controls don't exist.

4. **Separation of Concerns:** The `mail-voters.py` script operates independently from the web layer, meaning data sanitized for web contexts may not be sanitized for email contexts (and vice versa). Sanitization should occur at storage time for multi-context safety.

5. **Template Engine Trust Boundary:** The code produces raw HTML strings (in `rewrite_description()`) and passes them to EZT templates, suggesting the templates render them unescaped. This inverts the normal trust model where templates provide auto-escaping.

---

## Cross-cutting Recommendations

### Immediate (Critical/High severity)

1. **Implement HTML sanitization in `rewrite_description()`** — Use `html.escape()` on the description content before wrapping in `<pre>` tags. URL-encode filenames in href attributes and HTML-escape them in text content.

2. **Sanitize election titles for SMTP safety** — Add CRLF stripping (`\r`, `\n`, `\x00`) to election title input at the point of creation in `do_create_endpoint()`.

3. **Add input length limits** — Enforce maximum lengths on `form.title` (200 chars) and `form.description` (5000 chars) in all form-processing endpoints.

### Short-term (Medium severity)

4. **Implement a centralized input sanitization utility** — Create a `sanitize` module with context-specific functions: `sanitize_for_html()`, `sanitize_for_email_header()`, `sanitize_for_filename()`, `sanitize_for_log()`.

5. **Validate `docname` parameter** — Add explicit allowlist validation (e.g., `^[a-zA-Z0-9._-]+$`) in `serve_doc()` before the filesystem operation.

6. **Add CSRF protection** — Replace the placeholder token with real generation/validation (noted in TODO.md but critical for POST endpoints that modify state).

### Long-term (Architecture)

7. **Adopt a sanitization-at-storage pattern** — Sanitize all user input when it enters the system (at form submission), ensuring stored data is safe for all downstream contexts.

8. **Migrate to auto-escaping templates** — Consider Jinja2 with auto-escaping enabled instead of EZT, which would provide default XSS protection for all template output.

9. **Implement Content Security Policy headers** — Add CSP headers to mitigate XSS impact even if injection occurs.

10. **Complete authorization model** — The `### check authz` placeholders represent a fundamental security gap. Implement ownership verification (e.g., only election owners can modify their elections).
