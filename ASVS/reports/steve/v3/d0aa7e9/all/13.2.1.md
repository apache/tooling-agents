## ASVS-13.2.1: Backend Communication Authentication

### Findings for 13.2.1

#### [LOW]
**Finding ID:** ASVS-1321-LOW-001

**Title:** SQLite database access lacks authentication mechanism

**Location:** `v3/steve/election.py:35-37`, `v3/steve/persondb.py:25-26`

**Description:** Communications between application components and the SQLite database do not use any authentication mechanism. The database is accessed via file path with no credentials:

```python
# v3/steve/election.py, lines 35-37
@staticmethod
def open_database(db_fname):
    return asfpy.db.DB(db_fname, yaml_fname=QUERIES, yaml_section='election')
```

```python
# v3/steve/persondb.py, lines 25-26
@classmethod
def open(cls, db_fname):
    return cls(asfpy.db.DB(db_fname, yaml_fname=QUERIES, yaml_section='person'))
```

SQLite is a file-based database and does not natively support user authentication. Access control depends entirely on file system permissions.

**Note:** This is classified as LOW because SQLite's embedded nature means it runs in-process and is not a network service. However, the requirement applies to "data layers" which includes databases. The application relies solely on OS file permissions rather than cryptographic authentication.

**Data Flow:** Application process → file system → `steve.db` → no authentication layer

**Impact:** If the file system permissions are misconfigured, any process on the same host could read or modify election data, cryptographic salts, and encrypted votes.

**Remediation:** For SQLite specifically:
- Document that file system permissions serve as the authentication mechanism
- Ensure the database file has restrictive permissions (e.g., `0600`, owned by the application service account)
- Consider SQLite encryption extensions (e.g., SQLCipher) for data-at-rest protection
- If migrating to a networked database, implement service account authentication with short-term credentials

#### [LOW]
**Finding ID:** ASVS-1321-LOW-002

**Title:** Email service communication lacks documented authentication configuration

**Location:** `v3/server/bin/mail-voters.py:67-73`

**Description:** The email sending function does not show explicit SMTP authentication:

```python
# v3/server/bin/mail-voters.py, lines 67-73
asfpy.messaging.mail(
    sender='Apache Election Platform <voter@apache.org>',
    recipient=voter.email,
    subject=f'Vote in Election: {metadata.title}',
    message=rendered_body,
    # Add other parameters as needed (e.g., auth, headers)
)
```

The comment `# Add other parameters as needed (e.g., auth, headers)` suggests authentication is not yet configured. The `asfpy.messaging` library may handle this internally, but it is not documented or visible in this codebase.

**Impact:** If SMTP authentication is not configured, the application may be relying on network-level trust (e.g., sending from within the same network as the mail relay) which could be spoofed or abused.

**Remediation:** Document SMTP authentication method and ensure it uses short-term tokens or certificate-based authentication rather than static passwords.

## Cross-cutting Architecture Observations

1. **Documentation vs. Implementation Gap**: The application has good architectural documentation (`ARCHITECTURE.md`, `schema.md`) covering design intent, but lacks operational documentation required by ASVS Level 2-3 (communication inventories, resource management strategies, secrets rotation schedules).

2. **Single Database Pattern**: The use of SQLite as an embedded database simplifies many backend communication concerns (no network authentication needed, no connection pool management) but makes it impossible to apply traditional service-account or role-based database access controls.

3. **Trust Boundary Ambiguity**: The `TODO.md` file acknowledges multiple unimplemented security controls (`### check authz` placeholders, CSRF placeholder). The system currently relies primarily on the ASF OAuth committer-level authentication as its sole access control boundary, with no fine-grained authorization distinguishing election owners from other committers.

4. **Operational Tooling Without Controls**: CLI scripts (`create-election.py`, `mail-voters.py`) operate with the same privileges as the web server but without any authentication or authorization layer, creating a potential privilege escalation path for anyone with shell access.

5. **Consistent Cryptographic Approach**: The cryptographic design (Argon2 key derivation, separate salts per voter-issue pair, encrypted votes, anti-tamper keys) is well-documented and consistently applied, representing a strength in the security architecture.

## Cross-cutting Recommendations

### Immediate (Address within current sprint)
1. **Create a Communications Inventory document** listing all external services (OAuth, SMTP, LDAP), protocols, authentication methods, and network requirements — satisfies ASVS 13.1.1 at minimal implementation cost
2. **Document existing secrets** and their criticality in a secrets inventory, even without rotation automation — satisfies ASVS 13.1.4 documentation requirement

### Short-term (1-3 months)
3. **Add resource management documentation** defining timeouts, connection limits, and failure handling for SQLite, OAuth, and SMTP — satisfies ASVS 13.1.2 and 13.1.3
4. **Implement read-only database connections** for components that only need read access (mail script, voter-facing queries) — improves ASVS 13.2.2 compliance
5. **Implement TLS certificate rotation procedures** and document the rotation schedule

### Long-term (3-6 months)
6. **Evaluate migration from SQLite** to a database that supports role-based access control if the application scales beyond single-server deployment — would fully satisfy ASVS 13.2.1 and 13.2.2
7. **Implement connection lifecycle management** with explicit open/close patterns (context managers) for all database access paths to prevent resource leaks
8. **Add circuit breaker patterns** for OAuth and SMTP communication with documented fallback behavior
9. **Implement proper CSRF tokens and session-based secrets** with defined rotation (currently placeholder) — prerequisite for ASVS 13.1.4 compliance on session secrets
