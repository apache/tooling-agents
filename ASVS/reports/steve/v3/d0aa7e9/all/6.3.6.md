### Positive Patterns for 6.3.6

- Email is stored in the session (`s['email']`) and person table but is only used for display and notification purposes.
- The authentication mechanism uses `asfquart.auth.require` with role-based requirements (`R.committer`, `R.pmc_member`), not email-based login.
- No magic link, email OTP, or email-based authentication flow exists in the codebase.
- The `mail-voters.py` script sends notification emails about elections but does not create authentication tokens.
- No endpoint accepts email as a login credential or authentication factor.

## Cross-cutting Architecture Observations

1. **Authentication is externalized:** The actual login flow, credential verification, and session establishment are handled by `asfquart.auth` (not included in this codebase). Many ASVS 6.3 requirements would need to be assessed against that external system. The audited code only handles post-authentication authorization.

2. **Authorization is the critical gap:** The most severe finding is the systematic failure to implement ownership-based authorization (Type B gap — control is intended but never implemented). This spans all management endpoints and creates a flat permission model where any committer can modify any election.

3. **Separation of concerns:** The application cleanly separates:
   - Authentication (external, via `asfquart.auth`)
   - Session management (via `asfquart.session`)
   - Business logic (via `steve.election.Election`)
   - Cryptographic vote privacy (via `steve.crypto`)
   
   However, the authorization layer between authentication and business logic is entirely missing.

4. **Security-conscious cryptographic design:** The election system has sophisticated anti-tamper mechanisms (`is_tampered()`), vote encryption, and privacy-preserving tallying. This contrasts sharply with the weak authentication/authorization controls.

5. **TODO.md as documentation:** The application's own TODO file acknowledges CSRF and authorization gaps, confirming these are known issues awaiting resolution rather than oversights.

## Cross-cutting Recommendations

### Immediate (Critical Risk)
1. **Implement authorization checks** at all `### check authz` locations. Verify election ownership (`md.owner_pid == current_uid`) or group membership (`user in md.authz group`) before allowing management operations.
2. **Enforce MFA** via the external authentication provider or add an application-level MFA step after initial authentication.

### Short-term (High Risk)
3. **Implement CSRF protection** — replace the placeholder token with real per-session cryptographic tokens and validate on all state-changing endpoints.
4. **Change GET to POST** for state-changing operations (`/do-open/<eid>`, `/do-close/<eid>`) to prevent CSRF via simple link injection.
5. **Add audit logging** for person record changes and authentication events.

### Long-term (Medium Risk)
6. **Implement suspicious login detection** — track login metadata (IP, user agent, timestamp) and notify users of anomalies.
7. **Add credential change notifications** — send email alerts when person records are modified.
8. **Document all authentication pathways** formally, including the external auth provider's capabilities, MFA support, and integration points.
9. **Implement rate limiting** on authentication-related endpoints to prevent brute-force attacks (per NIST SP 800-63B § 5.2.2).
