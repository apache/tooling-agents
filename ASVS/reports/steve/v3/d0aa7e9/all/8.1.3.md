## ASVS-8.1.3: Environmental/Contextual Attributes Documentation

### Findings for 8.1.3

#### [LOW]

**Finding ID:** ASVS-813-LOW-001

**Files:** All files in scope

**Issue:** There is no documentation defining environmental or contextual attributes used in security decisions. The application does not appear to use:
- Time of day restrictions
- User location/geolocation
- IP address filtering or logging for security decisions
- Device fingerprinting
- Network-based restrictions

The only temporal element is the advisory `open_at`/`close_at` dates in the election table, but `schema.md` explicitly states:
```
Note: the "prevent_open_close_update" trigger will prevent these two values from changing once
an election is closed. They never need to be set, but once an election is closed: they are fixed.
```

And from the schema:
```sql
/* The approximate times this Election will be opened and closed,
   if known. NULL if unknown. These are purely advisory, for humans,
   and have no effect upon the actual Election operation. */
```

These dates have NO effect on security decisions — elections are opened/closed only by explicit user action.

**Impact:** For a Level 3 requirement, the absence of documentation about environmental/contextual attributes means:
1. There's no defined policy about whether such attributes SHOULD be used
2. There's no record of a conscious decision to NOT use them
3. If contextual security is later needed (e.g., restrict voting to certain time windows), there's no framework for it

**Remediation:** Document the decision explicitly:
```markdown
# Environmental and Contextual Security Attributes

## Current Implementation
This application does NOT currently use environmental or contextual attributes
for authentication or authorization decisions.

## Rationale
- Elections are opened/closed by explicit administrator action
- open_at/close_at dates are advisory only (displayed to users)
- IP-based restrictions are not appropriate for a distributed voting system
- Time-based access is controlled at the election state level, not clock-based

## Future Considerations
- [ ] Consider IP logging for audit trail
- [ ] Consider rate limiting by IP for vote submission
- [ ] Consider device binding for session security
```

## Cross-cutting Architecture Observations

1. **Systematic Authorization Gap (Type B):** The most critical finding across all requirements is the pattern of authorization placeholders (`### check authz`) that are never implemented. This creates a dangerous false sense of security — developers clearly intended to add authorization but it was never done. Eight separate endpoints share this identical gap.

2. **Authentication ≠ Authorization Confusion:** The application conflates authentication (proving identity) with authorization (proving permission). Being an ASF committer is sufficient to perform ANY operation on ANY election. The `authz` and `owner_pid` fields exist in the schema but are completely unused in the application layer.

3. **Defense-in-Depth at Data Layer:** Despite the application-level authorization failures, the `election.py` module provides some defense via state assertions (`assert self.is_editable()`) and the cryptographic design (can't decrypt votes without the proper keys). However, these protect election integrity, not confidentiality or availability.

4. **Missing Documentation Pattern:** Across all 8.1.x requirements, there is no formal authorization documentation. The code comments and `TODO.md` acknowledge this debt explicitly. The `schema.md` provides data-level documentation but not security-decision documentation.

5. **HTTP Method Misuse:** State-mutating operations (`do_open`, `do_close`) use GET, making them vulnerable to CSRF even before considering the placeholder CSRF token. This compounds the authorization gap.

6. **Incomplete Enforcement Pattern:** Where authorization IS implemented (e.g., `vote_on_page` checks mayvote), the corresponding action endpoint (`do_vote_endpoint`) does NOT repeat the check. This creates a bypass where viewing is protected but acting is not.

## Cross-cutting Recommendations

### Immediate (Critical - must fix before deployment)

1. **Implement ownership checks in `load_election` decorator** — Compare `election.owner_pid` against `result.uid`. This single fix addresses ASVS-821-CRIT-001, CRIT-003, HIGH-002, and MED-001 across 8+ endpoints.

2. **Add voter eligibility check in `do_vote_endpoint`** — Mirror the check from `vote_on_page()` to prevent ineligible users from submitting votes (ASVS-821-CRIT-002).

3. **Change `do_open` and `do_close` from GET to POST** — Prevents trivial CSRF via link/image tags (ASVS-821-HIGH-001).

### Short-term (High priority - within next sprint)

4. **Implement the `authz` field enforcement** — Define and implement group-based authorization for elections that specify an authz group.

5. **Create authorization documentation** (ASVS-811-HIGH-001) — Document the role hierarchy, function-level access matrix, and field-level restrictions that are already partially implemented in code.

6. **Implement CSRF token generation and validation** — Replace the placeholder with real tokens, validated on all POST endpoints.

7. **Document field-level access rules** — Formalize the implicit rules about salt/key exclusion from public APIs.

### Long-term (Strategic improvements)

8. **Consider contextual security factors** — For a voting system, logging IP addresses and implementing rate limiting would improve auditability and prevent abuse.

9. **Implement session consent mechanism** — For federated authentication, add a consent step or session confirmation to meet ASVS 7.6.2.

10. **Centralize authorization logic** — Instead of scattered checks, implement a policy-based authorization system (e.g., ABAC) that the decorators call consistently.
