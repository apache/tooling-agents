## ASVS-6.2.8: Password Verified Exactly as Received

### Findings for 6.2.8

No password verification logic exists in the audited code. The application does not:
- Accept passwords from users
- Perform password comparison/hashing for authentication
- Apply any transformation (truncation, case normalization, trimming) to authentication credentials

The `crypto.py` module handles vote encryption (not passwords), using Argon2 for key derivation and Fernet for encryption of vote strings. The `add_vote` method in `election.py` (line 231) processes `votestring` — not passwords:

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

This processes the votestring without modification, which is a positive pattern for data integrity, but it is not password verification.

**Conclusion:** This requirement is **not directly applicable** to the audited code scope as authentication is delegated externally.

## Cross-cutting Architecture Observations

1. **Delegated Authentication:** The application entirely delegates authentication to the `asfquart.auth` module (ASF OAuth/CAS). This means password-specific ASVS requirements (6.2.6–6.2.9) are not directly applicable to this codebase but would need to be verified against the external identity provider.

2. **Authorization Gap Pattern:** Multiple endpoints have `### check authz` placeholder comments without implementation. While authentication (authn) is enforced via decorators, authorization (authz — "is this user the owner of this election?") is not verified. This is a cross-cutting concern that affects the trust model around brute-force protections — any authenticated committer can access any election's management functions.

3. **CSRF Placeholder:** The `csrf_token = 'placeholder'` pattern creates a false sense of security. Templates reference it, JavaScript sends it in headers, but no server-side validation occurs. This is a **Type B gap** (control exists but not enforced).

4. **External Auth Boundary:** The security boundary between this application and `asfquart.auth` is critical. Rate limiting, brute force protection, password policies, and account management all depend on the external system's implementation, which is outside the audit scope.

5. **Cryptographic Design (Positive):** The vote encryption scheme using Argon2 key derivation, per-voter salts, and Fernet encryption demonstrates strong cryptographic practices for protecting ballot secrecy. The anti-tamper mechanism (`is_tampered()`) provides integrity verification.

## Cross-cutting Recommendations

### Immediate (Critical/High Priority)

1. **Implement real CSRF protection** — Replace the placeholder token with a cryptographically random per-session token and validate it on all state-changing endpoints. This directly impacts ASVS 6.3.1 by preventing automated cross-site attacks.

2. **Convert state-changing GET endpoints to POST** — `/do-open/<eid>` and `/do-close/<eid>` must use POST to prevent trivial automation and link-based attacks.

3. **Implement authorization checks** — Replace all `### check authz` placeholders with actual ownership verification (e.g., verify `result.uid == election.owner_pid`).

### Short-term (Medium Priority)

4. **Add rate limiting** — Implement per-user, per-endpoint rate limiting for sensitive operations (voting, election creation/modification). This addresses ASVS 6.3.1 at the application layer.

5. **Document external auth requirements** — Create security documentation specifying what brute force protections must be provided by the `asfquart.auth` system to satisfy ASVS 6.3.1.

6. **Remove debug prints** — Remove `print('FORM:', form)` statements in `do_add_issue_endpoint` and `do_edit_issue_endpoint`.

### Long-term (Architecture)

7. **Verify external IdP compliance** — Audit the ASF identity provider against ASVS 6.2.6–6.2.9 to ensure password handling requirements are met at the authentication boundary.

8. **Implement request anomaly detection** — Add logging and alerting for unusual patterns (rapid successive requests, bulk operations) to complement rate limiting.

9. **Security documentation** — Create an application-level security document describing the authentication architecture, trust boundaries, and how each ASVS requirement is satisfied (either locally or by the external IdP).
