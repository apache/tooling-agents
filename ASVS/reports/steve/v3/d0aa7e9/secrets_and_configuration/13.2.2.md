## Findings

### HIGH Severity

#### ASVS-1322-HIGH-001: State-Based Operation Restrictions Use Strippable Python Assert Statements

**Severity:** HIGH  
**Gap Type:** Type B — Security control EXISTS but is NOT RELIABLY CALLED (creates false confidence)

**Location:**
- **File:** `v3/steve/election.py`
- **Affected Methods and Lines:**

| Method | Line | Assert Statement |
|--------|------|-----------------|
| `delete()` | ~66 | `assert self.is_editable()` |
| `open()` | ~86 | `assert self.is_editable()` |
| `close()` | ~130 | `assert self.is_open()` |
| `add_salts()` | ~138 | `assert self.is_editable()` |
| `add_issue()` | ~190–191 | `assert self.is_editable()` / `assert vtype in vtypes.TYPES` |
| `edit_issue()` | ~211–212 | `assert self.is_editable()` / `assert vtype in vtypes.TYPES` |
| `delete_issue()` | ~226 | `assert self.is_editable()` |
| `add_voter()` | ~253 | `assert self.is_editable()` |

**Related Functions Checked:**
- `is_editable()` — state check method
- `is_open()` — state check method
- `_all_metadata(required_state)` — proper exception-based enforcement (vote operations)

**Description:**

Eight critical election management methods rely on Python `assert` statements to enforce state-based access control. These assertions are completely removed when Python runs with optimization flags (`-O` or `PYTHONOPTIMIZE=1`), which is common in production deployments for performance reasons.

This creates a false sense of security: the code *appears* to enforce least privilege through state restrictions, but these restrictions silently disappear in optimized execution modes.

**Vulnerable Code:**

```python
def delete(self):
    "Delete this Election and its Issues and Person/Issue pairs."
    # Can't delete if it has been opened (even if later closed).
    assert self.is_editable()  # <-- removed by python -O

    self.db.conn.execute('BEGIN TRANSACTION')
    self.c_delete_mayvote.perform(self.eid)
    self.c_delete_issues.perform(self.eid)
    self.c_delete_election.perform(self.eid)
    self.db.conn.execute('COMMIT')
```

```python
def add_salts(self):
    "Set the SALT column in the MAYVOTE table."
    assert self.is_editable()  # <-- removed by python -O

    self.db.conn.execute('BEGIN TRANSACTION')
    self.q_all_issues.perform(self.eid)
    for mayvote in self.q_all_issues.fetchall():
        salt = crypto.gen_salt()
        self.c_salt_mayvote.perform(salt, mayvote.rowid)
    self.db.conn.execute('COMMIT')
```

**Inconsistency with Vote Operations:**

The vote-related methods use proper exception-based validation that survives optimization:

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)  # raises ElectionBadState ← CORRECT

def tally_issue(self, iid):
    md = self._all_metadata(self.S_CLOSED)  # raises ElectionBadState ← CORRECT
```

**Data Flow:**

API layer → `Election.delete()` / `add_salts()` / etc. → `assert` (removed by `-O`) → unrestricted database write operations

**Proof of Concept:**

```bash
# Running the application with Python optimization strips all assertions:
PYTHONOPTIMIZE=1 uv run python -m hypercorn main:steve_app

# Or directly:
python -O -m hypercorn main:steve_app

# Now any API call to delete/modify elections bypasses state checks,
# allowing deletion of opened elections, salt regeneration on live elections,
# or issue modification after election opening.
```

**Impact:**

- An opened or closed election can be deleted, destroying all voting records
- `add_salts()` on an open election regenerates all salts, permanently orphaning all cast votes (making them undecryptable)
- Issues can be added/modified on open elections, corrupting the anti-tamper hash and invalidating the election
- Voters can be added after election opening without proper cryptographic setup (no salt assigned)
- This directly violates least privilege: backend database write operations proceed without their intended state-based restrictions

**Remediation:**

Replace all security-critical `assert` statements with proper conditional checks that raise exceptions:

```python
def delete(self):
    "Delete this Election and its Issues and Person/Issue pairs."
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
    
    self.db.conn.execute('BEGIN TRANSACTION')
    # ... rest of method

def add_salts(self):
    "Set the SALT column in the MAYVOTE table."
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
    # ... rest of method

def close(self):
    "Close an election."
    if not self.is_open():
        raise ElectionBadState(self.eid, self.get_state(), self.S_OPEN)
    self.c_close.perform(self.eid)

def add_issue(self, title, description, vtype, kv):
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
    if vtype not in vtypes.TYPES:
        raise ValueError(f'Invalid vote type: {vtype}')
    # ... rest of method
```

This aligns with the pattern already used by `_all_metadata(required_state)` and ensures state enforcement survives all Python execution modes.

### MEDIUM Severity

#### ASVS-1322-MED-001: All Database Connections Use Uniform Read-Write Privileges Without Least-Privilege Separation

**Severity:** MEDIUM  
**Gap Type:** Type A — No read-only database connection mechanism exists

**Location:**
- **File:** `v3/steve/election.py`
- **Function:** `open_database()` at line ~45

**Related Functions Checked:**
- `open_to_pid()` — class method, read-only operation
- `owned_elections()` — class method, read-only operation
- `upcoming_to_pid()` — class method, read-only operation
- `list_closed_election_ids()` — class method, read-only operation
- `tally_issue()` — read-only operation
- `has_voted_upon()` — read-only operation
- `get_metadata()` — read-only operation
- `__getattr__()` — cursor proxy mechanism

**Description:**

All database operations, regardless of whether they require read or write access, use the same connection type with full read-write privileges. There is no mechanism to open read-only database connections for operations that only query data.

**Vulnerable Code:**

```python
@staticmethod
def open_database(db_fname):
    return asfpy.db.DB(db_fname, yaml_fname=QUERIES, yaml_section='election')
```

**All callers — including read-only class methods — receive full read-write connections:**

```python
@classmethod
def open_to_pid(cls, db_fname, pid):
    "List of elections are OPEN for PID to vote upon."
    db = cls.open_database(db_fname)  # Full r/w connection for a SELECT query
    db.q_open_to_me.perform(pid,)
    return [row for row in db.q_open_to_me.fetchall()]

@classmethod
def owned_elections(cls, db_fname, pid):
    "List of elections are that PID has created."
    db = cls.open_database(db_fname)  # Full r/w connection for a SELECT query
    db.q_owned.perform(pid,)
    return [row for row in db.q_owned.fetchall()]

@classmethod
def upcoming_to_pid(cls, db_fname, pid):
    db = cls.open_database(db_fname)  # Full r/w connection for a SELECT query
    db.q_upcoming_to_me.perform(pid,)
    return [row for row in db.q_upcoming_to_me.fetchall()]

@classmethod
def list_closed_election_ids(cls, db_fname, include_open=False):
    db = cls.open_database(db_fname)  # Full r/w connection for SELECT queries
    # ... only performs SELECT queries
```

**Additionally, the `__getattr__` proxy exposes all database cursors (including write cursors) to any code with an Election instance:**

```python
def __getattr__(self, name):
    "Proxy the cursors."
    return self.__dict__.get(name, getattr(self.db, name))
```

**Data Flow:**

Read-only API endpoint → Class method (e.g., `open_to_pid()`) → `open_database()` → Full read-write connection → SELECT query execution

**Impact:**

- Read-only operations (listing elections, checking vote status, retrieving metadata, tallying) hold connections capable of writing to the database
- If any code path is compromised (e.g., through a future bug in the web layer), the existing connection has full write access even when the intended operation is read-only
- The `__getattr__` proxy means any code with an `Election` reference can invoke any database cursor, including `c_delete_election`, `c_delete_issues`, etc.
- Class method connections are not explicitly closed, extending the window where overprivileged connections exist

**Remediation:**

Implement separate read-only and read-write database connection methods:

```python
@staticmethod
def open_database(db_fname, readonly=False):
    """Open database with appropriate privilege level."""
    if readonly:
        # SQLite supports read-only connections via URI mode
        import urllib.parse
        db_uri = f'file:{urllib.parse.quote(str(db_fname))}?mode=ro'
        return asfpy.db.DB(db_uri, yaml_fname=QUERIES, 
                          yaml_section='election', uri=True)
    return asfpy.db.DB(db_fname, yaml_fname=QUERIES, 
                      yaml_section='election')

@classmethod
def open_to_pid(cls, db_fname, pid):
    "List of elections are OPEN for PID to vote upon."
    db = cls.open_database(db_fname, readonly=True)
    try:
        db.q_open_to_me.perform(pid,)
        return [row for row in db.q_open_to_me.fetchall()]
    finally:
        db.conn.close()

# Apply readonly=True to: owned_elections, upcoming_to_pid,
# list_closed_election_ids, tally_issue, has_voted_upon, get_metadata
```

If `asfpy.db.DB` does not support URI mode, an alternative is to use `PRAGMA query_only = ON` after connection:

```python
@staticmethod
def open_database(db_fname, readonly=False):
    db = asfpy.db.DB(db_fname, yaml_fname=QUERIES, yaml_section='election')
    if readonly:
        db.conn.execute('PRAGMA query_only = ON')
    return db
```

## Positive Security Patterns

1. **Proper state enforcement for vote operations:** `add_vote()` and `tally_issue()` use `_all_metadata(required_state)` which raises `ElectionBadState` exceptions — this survives Python optimization and is the correct pattern.

2. **Sensitive field stripping in `get_metadata()`:** The public metadata accessor explicitly excludes `salt` and `opened_key` from returned data, with a clear comment documenting why.

3. **Transactional database operations:** Multi-step write operations (`delete()`, `add_salts()`) use explicit `BEGIN TRANSACTION` / `COMMIT` for atomicity.

4. **Non-privileged server port:** The default port 58383 does not require root/elevated privileges to bind, following the principle of least privilege for the application process.

5. **Strong cryptographic separation:** Per-voter salts with Argon2 key derivation, HKDF key stretching, and Fernet authenticated encryption provide proper cryptographic isolation between vote records.

6. **Cryptographically secure randomness:** All random values (`gen_salt()`, `create_id()`, `shuffle()`) use the `secrets` module, not `random`.

7. **TLS configured by default in example:** The example configuration includes certificate and key paths, encouraging TLS deployment.

## Architecture Observations

1. **Two enforcement patterns coexist:** The codebase uses `assert` for election-management state checks and `_all_metadata(required_state)` for vote-related state checks. This inconsistency suggests the vote-related pattern was developed later or with more security awareness. All state checks should use the exception-based pattern.

2. **Class methods create and abandon database connections:** `open_to_pid()`, `owned_elections()`, `upcoming_to_pid()`, and `list_closed_election_ids()` create database connections that are never explicitly closed, relying on garbage collection. These connections hold read-write access unnecessarily.

3. **`__getattr__` proxy is overly permissive:** The attribute proxy on `Election` delegates any unknown attribute lookup to the database wrapper, exposing all defined query and command cursors. This means any code path with an `Election` reference can invoke any database operation, not just the ones appropriate to its context.

4. **Debug logging in ASGI production mode:** `run_asgi()` sets `logging.DEBUG` as the default level. While the election module doesn't log sensitive cryptographic material, debug-level logging in production increases the risk of information leakage through framework or library loggers.

## Recommendations Summary

### Immediate Actions (Priority 1)

| Finding | Effort | Impact |
|---------|--------|--------|
| Replace all security `assert` statements with `if/raise` pattern (ASVS-1322-HIGH-001) | Low — straightforward refactor; pattern already exists in `_all_metadata()` | Eliminates false confidence in state enforcement; prevents catastrophic operations under `-O` |

**Rationale:** This is a critical security control bypass that can be exploited simply by running Python with standard optimization flags. The remediation is straightforward since the correct pattern already exists in the codebase.

### Short-term Actions (Priority 2)

| Finding | Effort | Impact |
|---------|--------|--------|
| Implement read-only database connections for read-only operations (ASVS-1322-MED-001) | Medium — requires `open_database()` modification and caller updates | Enforces least privilege at the database connection level; limits blast radius of code-path compromise |
| Explicitly close database connections in class methods | Low — add `try/finally` blocks | Reduces window of overprivileged connection availability |

**Rationale:** Implementing connection-level privilege separation provides defense-in-depth. Even if application-level controls fail, database-level restrictions limit the damage.

### Long-term Actions (Priority 3-4)

| Action | Effort | Impact |
|--------|--------|--------|
| Configure production logging level to INFO/WARNING | Low — configuration change in `run_asgi()` | Reduces risk of sensitive information leakage through debug logs |
| Review and restrict `__getattr__` proxy scope | Medium — requires careful analysis of cursor usage | Prevents unintended database operations through attribute access |

**Rationale:** These improvements further harden the application against information disclosure and unintended database access patterns.
