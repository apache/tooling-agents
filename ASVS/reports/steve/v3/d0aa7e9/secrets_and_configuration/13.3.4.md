## Findings

### High Severity

#### ASVS-1334-SEV-001

| Attribute | Value |
|-----------|-------|
| **Severity** | **HIGH** |
| **Gap Type** | Type A — No control exists |
| **File** | `v3/steve/election.py` |
| **Functions** | `open()` (L79-92), `close()` (L126-132), `tally_issue()` (L285-351) |
| **Lines** | 79-92, 126-132, 285-351 |

**Description**

The election root key (`opened_key`) and its associated `salt` are generated when an election opens and persist in the database indefinitely after the election is closed and tallied. There is no expiration, rotation, or secure destruction mechanism. These are the **root secrets** from which all per-voter vote encryption keys are derived.

**Vulnerable Code**

In `election.py`, key generation during `open()`:
```python
def open(self, pdb):
    assert self.is_editable()
    self.add_salts()
    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)
    self.c_open.perform(salt, opened_key, self.eid)  # Stored permanently
```

In `election.py`, `close()` does not destroy or expire key material:
```python
def close(self):
    assert self.is_open()
    self.c_close.perform(self.eid)  # Only sets closed flag; keys persist
```

After `tally_issue()` completes, all cryptographic material remains:
```python
def tally_issue(self, iid):
    md = self._all_metadata(self.S_CLOSED)
    # ... decrypts all votes using md.opened_key ...
    # No cleanup of opened_key or salts after tally
```

**Related Functions Checked**
- `open()` — Generates and stores keys
- `close()` — Only updates state, no key destruction
- `tally_issue()` — Uses keys but doesn't clean up
- `get_metadata()` — Correctly excludes secrets from API
- `is_tampered()` — Validates key integrity but doesn't manage lifecycle

**Data Flow**

```
crypto.gen_salt() + crypto.gen_opened_key()
    ↓
Stored in metadata table
    ↓
Never expired, rotated, or destroyed
    ↓
Remains available for indefinite future database compromise
```

**Proof of Concept**

If the database file (`steve.db`) is compromised at any point after elections have been tallied—even years later—an attacker can:

1. Read `opened_key` and `salt` from the `metadata` table
2. Read voter `pid`, `iid`, and per-voter `salt` from the `mayvote` table
3. Re-derive every `vote_token` via `crypto.gen_vote_token()`
4. Derive every vote encryption key via `crypto._b64_vote_key()`
5. Decrypt every stored vote, breaking voter anonymity retroactively

**Impact**

- **Confidentiality:** Complete compromise of voter anonymity for all historical elections
- **Compliance:** Violation of ASVS 13.3.4 requirement for secret expiration
- **Long-term Risk:** Database backups become permanent vulnerability vectors
- **Forensic Exposure:** No ability to prove keys were destroyed after election completion

**Remediation**

Add a `purge_crypto()` method and integrate it into the post-tally lifecycle:

```python
def purge_crypto(self):
    """Securely destroy cryptographic material after tallying is verified."""
    assert self.is_closed()
    
    self.db.conn.execute('BEGIN TRANSACTION')
    
    # Nullify all per-voter salts
    self.c_purge_mayvote_salts.perform(self.eid)
    
    # Nullify the election opened_key and salt
    self.c_purge_election_keys.perform(self.eid)
    
    # Optionally delete encrypted vote ciphertext
    self.c_purge_votes.perform(self.eid)
    
    self.db.conn.execute('COMMIT')
    
    # Force SQLite to overwrite freed pages
    self.db.conn.execute('VACUUM')
    
    _LOGGER.info(f'Purged cryptographic material for election[E:{self.eid}]')
```

### Medium Severity

#### ASVS-1334-SEV-002

| Attribute | Value |
|-----------|-------|
| **Severity** | **MEDIUM** |
| **Gap Type** | Type A — No control exists |
| **File** | `v3/steve/election.py`, `v3/steve/crypto.py` |
| **Functions** | `add_salts()` (L134-154) |
| **Lines** | 134-154 |

**Description**

Per-voter cryptographic salts in the `mayvote` table are generated once when an election is opened and never expire or rotate. While per-voter salts are critical for vote anonymity (preventing cross-voter correlation), there is no mechanism for the salts to have a defined maximum lifetime or be securely zeroed after use.

**Vulnerable Code**

```python
def add_salts(self):
    "Set the SALT column in the MAYVOTE table."
    assert self.is_editable()
    self.db.conn.execute('BEGIN TRANSACTION')
    self.q_all_issues.perform(self.eid)
    for mayvote in self.q_all_issues.fetchall():
        salt = crypto.gen_salt()
        self.c_salt_mayvote.perform(salt, mayvote.rowid)  # Stored permanently
    self.db.conn.execute('COMMIT')
```

**Related Functions Checked**
- `add_salts()` — Generates salts with no expiration metadata
- `crypto.gen_salt()` — Secure generation but no lifecycle tracking
- `get_issue()` — Correctly excludes salts from API response

**Data Flow**

```
crypto.gen_salt() for each voter
    ↓
Stored in mayvote table
    ↓
No expiration timestamp
    ↓
Persists indefinitely alongside opened_key
```

**Impact**

Combined with Finding SEV-001, the indefinite persistence of per-voter salts means the complete key derivation chain (`opened_key` → `vote_token` → `vote_key`) remains reconstructable from database contents alone, without any time-bound protection.

**Remediation**

Integrate salt destruction into the `purge_crypto()` method from SEV-001. Additionally, add a `created_at` timestamp to the `mayvote` salt records to enable age-based expiration policies:

```python
# In add_salts():
salt = crypto.gen_salt()
created_at = int(time.time())
self.c_salt_mayvote.perform(salt, created_at, mayvote.rowid)
```

#### ASVS-1334-SEV-003

| Attribute | Value |
|-----------|-------|
| **Severity** | **MEDIUM** |
| **Gap Type** | Type A — No control exists |
| **File** | `v3/server/config.yaml.example` |
| **Location** | Lines 30-32 (TLS configuration) |
| **Lines** | 30-32 |

**Description**

TLS certificate and private key files are configured as static filesystem references with no expiration monitoring, rotation procedure, or lifecycle management. The configuration provides no guidance on certificate renewal timelines or procedures.

**Vulnerable Code**

```yaml
server:
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
```

**Related Functions Checked**
- `main.py` certificate loading — No expiration checking
- Configuration parsing — No validation of cert validity periods

**Impact**

Production TLS certificates will silently expire, causing either:
- Service outage if clients enforce certificate validity
- Downgrade to unencrypted communication if error handling falls through
- Extended use of potentially compromised key material with no rotation forcing function

**Remediation**

1. Add certificate expiration monitoring at startup:

```python
import ssl
import datetime

def check_cert_expiry(certfile, warn_days=30):
    cert = ssl.PEM_cert_to_DER_cert(open(certfile).read())
    x509 = crypto_x509.load_der_x509_certificate(cert)
    days_remaining = (x509.not_valid_after_utc - datetime.datetime.now(datetime.UTC)).days
    if days_remaining < warn_days:
        _LOGGER.warning(f'TLS certificate expires in {days_remaining} days!')
    if days_remaining <= 0:
        _LOGGER.critical('TLS certificate has EXPIRED!')
```

2. Document rotation procedure in `config.yaml.example`:

```yaml
server:
    certfile: localhost.apache.org+3.pem  # Rotate annually; see docs/tls-rotation.md
    keyfile: localhost.apache.org+3-key.pem
    # cert_warn_days: 30  # Warn this many days before expiry
```

## Architecture Observations

### Key Architectural Patterns

1. **Missing lifecycle phase:** The election state machine (`editable` → `open` → `closed`) lacks a terminal `archived`/`purged` state where cryptographic material is securely destroyed. This is the root cause of SEV-001 and SEV-002.

2. **Key hierarchy is sound but permanent:** The derivation chain (`edata` → `opened_key` → `vote_token` → `vote_key`) is cryptographically well-constructed, but its permanence means compromise at any point in time retroactively exposes all elections.

3. **SQLite VACUUM requirement:** Even if key material is `NULL`-ed in the database, SQLite may retain the original values in free pages. A `VACUUM` operation is necessary after purging to physically overwrite deleted data.

4. **No configuration for external secret stores:** The `config.yaml.example` shows only file-based secrets. There is no integration point for HashiCorp Vault, AWS Secrets Manager, or similar services that provide built-in rotation and expiration.

### Security Design Strengths

- Strong cryptographic primitives (Argon2, HKDF, XChaCha20-Poly1305)
- Proper key derivation hierarchy with domain separation
- API endpoints correctly exclude sensitive data
- State machine prevents improper transitions

### Security Design Weaknesses

- No secret lifecycle management beyond creation
- Database persistence without cleanup mechanisms
- Static configuration with no dynamic secret support
- No monitoring or alerting for secret expiration
