## Findings

### Medium Severity

#### Finding ASVS-1324-MED-001

| Attribute | Detail |
|-----------|--------|
| **Severity** | **MEDIUM** |
| **Gap Type** | Type A — No control exists |
| **File** | `v3/server/config.yaml.example` (entire file), `v3/server/main.py:43-46` |
| **Function** | Configuration and `create_app()` |
| **ASVS Requirement** | 13.2.4 — Allowlist for external communications |

**Description:** 

The application configuration (`config.yaml.example`) defines no allowlist of permitted external resources. The application communicates with at least one external service (`oauth.apache.org`), and the domain context indicates LDAP and potentially email services are also used. No centralized, configurable allowlist exists to define and restrict these communications.

**Vulnerable Configuration (config.yaml.example):**
```yaml
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem

db: steve.db
# No external_resources_allowlist or equivalent section exists
```

**Hardcoded OAuth URLs (main.py:43-46):**
```python
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow:**
- **Outbound OAuth**: Application → `oauth.apache.org` (hardcoded, no configurable allowlist)
- **Outbound LDAP**: Application → LDAP server (mentioned in domain context, no restriction visible)
- **Outbound Email**: Implied by `get_voters_for_email()` in `election.py:636-643` (no destination restriction visible)

**Impact:**
1. No single auditable location documents all permitted external communications
2. If any code path (including imported `pages`/`api` modules or `asfquart` framework) makes outbound requests based on user-controlled data, there is no enforcement mechanism to prevent SSRF or unauthorized data exfiltration
3. New external integrations can be added without updating a central policy
4. Deployment hardening cannot reference an application-defined list for firewall rules

**Remediation:**

Add an external communications allowlist to the application configuration:

```yaml
# config.yaml
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem

db: steve.db

# ASVS 13.2.4: Permitted external communications
allowed_external_resources:
  oauth:
    host: oauth.apache.org
    port: 443
    protocol: https
    purpose: "OAuth authentication"
  ldap:
    host: ldap.apache.org
    port: 636
    protocol: ldaps
    purpose: "Voter directory lookup"
```

Implement an enforcement wrapper for outbound connections:

```python
# outbound.py
import urllib.parse

class OutboundAllowlist:
    def __init__(self, allowed_resources: dict):
        self._allowed_hosts = set()
        for resource in allowed_resources.values():
            self._allowed_hosts.add(
                (resource['host'], resource.get('port', 443))
            )

    def is_permitted(self, url: str) -> bool:
        parsed = urllib.parse.urlparse(url)
        port = parsed.port or (443 if parsed.scheme == 'https' else 80)
        return (parsed.hostname, port) in self._allowed_hosts

    def validate_or_raise(self, url: str):
        if not self.is_permitted(url):
            raise PermissionError(
                f"Outbound request to {url} blocked: "
                f"not in allowed_external_resources"
            )
```

#### Finding ASVS-1324-MED-002

| Attribute | Detail |
|-----------|--------|
| **Severity** | **MEDIUM** |
| **Gap Type** | Type A — No control exists |
| **File** | `v3/server/main.py:43-46` |
| **Function** | `create_app()` |
| **Lines** | 43-46 |

**Description:** 

The OAuth `redirect_uri` parameter in `OAUTH_URL_INIT` is constructed using string formatting (`%s`), and the value that populates this parameter comes from the `asfquart` framework (not visible in provided code). If the `redirect_uri` is derived from request-controlled data (e.g., the `Host` header) without validation against an allowlist of permitted callback URLs, this could allow an attacker to redirect OAuth callbacks to an attacker-controlled server, capturing authorization codes.

**Vulnerable Code (main.py:43-44):**
```python
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
```

**Data Flow:**
- Source: Incoming request (potentially `Host` header or request URL) → `asfquart` framework → `redirect_uri` parameter → sent to `oauth.apache.org`

**Proof of Concept:**
```
GET /login HTTP/1.1
Host: attacker-controlled.example.com
```
If `redirect_uri` is derived from the `Host` header, the OAuth flow would redirect the authorization code to `attacker-controlled.example.com`.

**Impact:** 

OAuth authorization code theft enabling account takeover, particularly impactful for election administrators.

**Remediation:**

Define an explicit allowlist for permitted OAuth redirect URIs:

```yaml
# config.yaml
oauth:
  provider: https://oauth.apache.org
  allowed_redirect_uris:
    - https://steve.apache.org/oauth/callback
    - https://steve-staging.apache.org/oauth/callback
```

Validate the redirect URI against the allowlist:

```python
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    
    allowed_redirects = app.cfg.oauth.allowed_redirect_uris
    # Ensure redirect_uri is from allowlist, not derived from request
    asfquart.generics.OAUTH_URL_INIT = (
        f'https://oauth.apache.org/auth?state=%s'
        f'&redirect_uri={urllib.parse.quote(allowed_redirects[0])}'
    )
```

## Positive Security Patterns

1. **Hardcoded OAuth Destination** (`main.py:43-46`): OAuth provider URLs are hardcoded to `oauth.apache.org` over HTTPS, preventing arbitrary OAuth provider injection. This is a basic form of implicit allowlisting for this specific communication channel.

2. **Local-Only Database Access** (`election.py`): All database operations use a local SQLite file specified in configuration. No remote database connections exist, eliminating one class of backend communication risk.

3. **No Dynamic URL Fetching in Core Logic** (`crypto.py`, `election.py`): The cryptographic and election logic modules make zero outbound network requests. All operations are local computation and local database I/O.

4. **TLS Support** (`config.yaml.example:29-31`, `main.py:82-85`): The configuration supports TLS certificates for encrypted communications, with cert/key path validation via `pathlib`.

5. **Secure Cryptographic Practices** (`crypto.py`): Uses `secrets` module for all random generation, Argon2 for key derivation, and HKDF for key stretching — all are strong cryptographic choices.

## Architecture Observations

1. **Minimal External Surface**: The application has a small external communication footprint (primarily OAuth), which limits the attack surface for SSRF and data exfiltration. However, the domain context mentions LDAP and email services that are not visible in the provided code.

2. **Framework Dependency for Network Policy**: The `asfquart` framework handles actual OAuth HTTP requests, meaning the application delegates outbound connection policy to the framework without visible enforcement.

3. **No Egress Monitoring**: There is no application-level logging of outbound requests, making it difficult to detect unauthorized external communications.

4. **Configuration Simplicity vs. Security**: The `config.yaml` is minimal, which reduces complexity but omits security-relevant configuration such as external resource allowlists, permitted origins, and network policies.
