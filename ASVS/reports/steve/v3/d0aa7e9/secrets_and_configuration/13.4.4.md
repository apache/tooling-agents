## Findings

### Medium Severity

#### ASVS-1344-MED-001: No Explicit HTTP TRACE Method Blocking at Application or Server Level

**Severity:** MEDIUM

**Location:**
- **File:** `v3/server/main.py`
- **Function:** `create_app()`
- **Lines:** 33–45
- **Additional File:** `v3/server/config.yaml.example`

**Related Functions Checked:**
- `create_app()` in `v3/server/main.py`
- Configuration parsing in `v3/server/config.yaml.example`
- Route registration in imported `pages` and `api` modules (not provided)

**Vulnerable Code:**

```python
# main.py - create_app()
def create_app():
    "Create the asfquart app and its endpoints."
    # ...
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    import pages  # noqa: F401
    import api    # noqa: F401

    return app  # No middleware to reject TRACE requests
```

```yaml
# config.yaml.example - No HTTP method restrictions
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
db: steve.db
```

**Data Flow:**

```
HTTP TRACE request → Hypercorn (ASGI server) → Quart/asfquart app → [no explicit TRACE rejection middleware] → relies solely on framework default 405 for unregistered routes
```

**Description:**

The application relies entirely on Quart's implicit behavior (returning 405 for methods not registered on any route) to prevent TRACE handling. There is no explicit, defense-in-depth control:

1. **No application middleware** exists to reject TRACE requests before route dispatch.
2. **No server configuration** (`config.yaml.example`) includes HTTP method restrictions.
3. **No reverse proxy configuration** is included in the codebase, despite the comment on line 4 of `config.yaml.example` stating *"Typical usage is that a proxy sits in front of this server."* The proxy configuration, which would be the primary defense point, is not provided or templated.
4. **No ASGI middleware** is registered that would block TRACE before it reaches routing logic.

While Quart's default behavior provides implicit protection (no route explicitly accepts TRACE), this is fragile:
- A catch-all error handler or routing change could inadvertently respond to TRACE.
- The `pages` and `api` modules (imported but not provided for review) could register routes accepting all methods.
- Without the reverse proxy config, there's no verifiable TRACE blocking at the infrastructure tier.

**Proof of Concept:**

```bash
# Against the standalone server or Hypercorn:
curl -X TRACE https://localhost:58383/ -k -v

# Expected (implicit framework behavior): 405 Method Not Allowed
# Risk: No explicit guarantee; dependent on route registration behavior
# If any route uses methods=None or a catch-all, TRACE would be served
```

**Impact:**

If TRACE is inadvertently enabled (through route misconfiguration, middleware changes, or proxy misconfiguration), an attacker could use Cross-Site Tracing (XST) to:
- Reflect HTTP request headers including authentication cookies and tokens
- Bypass `HttpOnly` cookie protections when chained with XSS
- Leak OAuth session tokens used for voter authentication

**Remediation:**

Add explicit TRACE blocking middleware to the application and provide production proxy configuration:

```python
# In main.py, after app creation:
from quart import request, abort

def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    @app.before_request
    async def block_trace():
        if request.method == 'TRACE':
            abort(405)

    import pages
    import api
    return app
```

Additionally, provide a production reverse proxy configuration template:

```nginx
# nginx.conf.example (production proxy template)
server {
    # Block TRACE and TRACK methods at the proxy level
    if ($request_method ~ ^(TRACE|TRACK)$) {
        return 405;
    }
    # ... proxy_pass to Hypercorn backend
}
```

Or for Apache httpd:
```apache
# Disable TRACE at the server level
TraceEnable Off
```

## Architecture Observations

1. **Proxy dependency without configuration:** The `config.yaml.example` explicitly states a proxy is expected in front of the application (line 4 comment: *"a proxy sits in front of this server"*), yet no proxy configuration template is provided. This makes it impossible to verify TRACE blocking at the infrastructure tier — the most effective place to block it.

2. **Two runtime modes:** The application supports standalone and ASGI modes. Both paths lack explicit TRACE blocking, meaning the gap exists regardless of deployment method.

3. **Framework reliance:** The current protection is entirely implicit through Quart's routing behavior. This is an example of **security by absence** rather than **security by design**.
