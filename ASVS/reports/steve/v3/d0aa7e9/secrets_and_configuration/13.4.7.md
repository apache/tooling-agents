### Assessment Summary

The application demonstrates a strong positive pattern by setting `static_folder=None`, disabling the framework's default static file serving. However, sensitive data files (database, configuration, certificates, query definitions) are co-located within the application directory tree, and no defense-in-depth file-extension filtering middleware exists at the application layer. Combined with the documented but unverified reverse proxy deployment model, this creates a risk of unintentional information leakage.

## Findings

### MEDIUM Severity

#### Finding ASVS-1347-MED-001

**Title:** Sensitive data files co-located in application directory without file-extension serving restrictions

**Location:**
- **Files:** `v3/server/config.yaml.example:34`, `v3/server/main.py:28`
- **Functions Reviewed:** Application initialization, path configuration
- **Gap Type:** Type A — No defense-in-depth control for file-type restriction

**Description:**

The SQLite database (`steve.db`), configuration file (`config.yaml`), query definitions (`queries.yaml`), TLS private key (`*.pem`), and Python source files (`.py`) all reside within or directly adjacent to the application directory tree. While `static_folder=None` prevents the Quart framework from serving these files, the documented deployment model uses a reverse proxy ("a proxy sits in front of this server"), and no proxy configuration is provided or enforced to restrict served file types.

**Vulnerable Configuration:**

```yaml
# config.yaml.example
db: steve.db   # Database in/near application directory
```

```python
# main.py
THIS_DIR = pathlib.Path(__file__).resolve().parent
CERTS_DIR = THIS_DIR / 'certs'   # TLS keys under app directory
```

```python
# election.py
QUERIES = THIS_DIR.parent / 'queries.yaml'  # Query definitions in app tree
```

**Data Flow:**

```
External request → Reverse proxy (unconfigured/misconfigured) → Application directory filesystem → Sensitive file served
```

**Proof of Concept:**

If a reverse proxy (e.g., Nginx) is configured to serve files from the application directory for static assets:

```http
GET /steve.db HTTP/1.1
Host: voting.example.org

# Returns the entire SQLite database containing encrypted votes,
# cryptographic salts, and opened_keys
```

```http
GET /config.yaml HTTP/1.1
Host: voting.example.org

# Returns server configuration including potential secrets
```

```http
GET /certs/localhost.apache.org+3-key.pem HTTP/1.1
Host: voting.example.org

# Returns TLS private key
```

**Impact:**

If the reverse proxy is misconfigured or a new route handler is added that inadvertently serves file contents, an attacker could obtain:
- The SQLite database containing all election data, encrypted votes, cryptographic salts, and opened_keys — enabling offline decryption of all votes
- TLS private keys enabling man-in-the-middle attacks
- Application source code enabling targeted vulnerability discovery
- Git history potentially containing committed secrets

**Remediation:**

1. **Move sensitive data files outside the application directory tree:**

```yaml
# config.yaml - use absolute path outside web root
db: /var/lib/steve/steve.db
```

```python
# main.py - certificates outside app tree
CERTS_DIR = pathlib.Path('/etc/steve/certs')
```

2. **Add application-level middleware to restrict response content types:**

```python
from quart import request, abort

ALLOWED_CONTENT_TYPES = {'text/html', 'application/json', 'text/css', 'application/javascript'}

@app.after_request
async def restrict_content_types(response):
    content_type = response.content_type.split(';')[0].strip()
    if content_type not in ALLOWED_CONTENT_TYPES:
        # Log unexpected content type serving
        _LOGGER.warning(f'Blocked response with content-type: {content_type} for {request.path}')
        abort(403)
    return response
```

3. **Provide and document required reverse proxy configuration:**

```nginx
# nginx.conf - Example for production deployment
server {
    # Block access to sensitive file extensions
    location ~* \.(db|sqlite|yaml|yml|py|pyc|pem|key|git|env|cfg|ini|log)$ {
        deny all;
        return 404;
    }

    # Block hidden files and directories
    location ~ /\. {
        deny all;
        return 404;
    }

    # Only proxy defined API/page routes
    location / {
        proxy_pass https://127.0.0.1:58383;
    }
}
```

#### Finding ASVS-1347-MED-002

**Title:** No documented or enforced production web tier hardening for file-type restrictions

**Location:**
- **Files:** `v3/server/main.py` (deployment configuration), `v3/server/config.yaml.example`
- **Functions Reviewed:** Application startup, configuration loading
- **Gap Type:** Type A — Required production configuration absent from codebase

**Description:**

The `config.yaml.example` references a reverse proxy deployment model ("a proxy sits in front of this server"), but the codebase contains no reverse proxy configuration templates, deployment hardening documentation, or automated configuration validation to ensure file extension restrictions are applied in production. ASVS 13.4.7 Level 3 requires verification that the **web tier** (not just the application framework) restricts served file types. Without enforceable proxy configuration, this requirement cannot be verified as satisfied.

**Evidence:**

```yaml
# config.yaml.example, line 21-23
# Default port for the server. Typical usage is that a proxy sits
# in front of this server, so we don't use ports 80 or 443.
```

The deployment involves two run modes (standalone and ASGI/Hypercorn), both of which expose the application directly without file-extension filtering:

```python
# main.py - standalone mode
app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)

# main.py - ASGI mode (Hypercorn)
steve_app = create_app()
```

Neither mode configures file-extension restrictions at the ASGI server level.

**Impact:**

Without documented and enforced web tier configuration:
- Deployments may omit file-extension restrictions entirely
- New team members or automated deployments may expose the application directly without a properly configured proxy
- The ASVS 13.4.7 requirement cannot be verified as consistently met across deployments

**Remediation:**

1. **Include a production reverse proxy configuration template in the repository:**

```
v3/deploy/
├── nginx.conf.example      # File extension restrictions
├── deployment-checklist.md  # Security verification steps
└── hardening.md             # Web tier hardening documentation
```

2. **Add a startup check that verifies the application is not directly exposed:**

```python
@app.before_serving
async def verify_deployment():
    """Warn if application appears directly exposed without a proxy."""
    port = app.cfg.server.port
    if port in (80, 443):
        _LOGGER.critical(
            'Application should not be directly exposed on ports 80/443. '
            'Deploy behind a reverse proxy with file-extension restrictions.'
        )
```

3. **Add ASGI middleware to reject requests for common sensitive extensions:**

```python
from quart import request, abort

BLOCKED_EXTENSIONS = {'.db', '.sqlite', '.yaml', '.yml', '.py', '.pyc',
                       '.pem', '.key', '.env', '.cfg', '.ini', '.log',
                       '.git', '.bak', '.swp', '.old'}

@app.before_request
async def block_sensitive_extensions():
    path = request.path.lower()
    for ext in BLOCKED_EXTENSIONS:
        if path.endswith(ext):
            _LOGGER.warning(f'Blocked request for sensitive file type: {request.path}')
            abort(404)
```

## Positive Security Patterns

### 1. Explicit Disabling of Static File Serving

**Location:** `v3/server/main.py`, line 37

```python
app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
```

This explicitly disables the Quart/Flask default static file serving, which is a strong baseline control. The application will only respond to explicitly defined route handlers from the `pages` and `api` modules, not serve arbitrary files from the filesystem.

### 2. Non-Standard Port

**Location:** `v3/server/config.yaml.example`, line 24

```yaml
port: 58383
```

The application runs on a non-standard port with the expectation that a reverse proxy sits in front, which is a correct architectural pattern for isolating the application from direct file requests.

### 3. Sensitive Metadata Not Returned in Public APIs

**Location:** `v3/steve/election.py`, lines 148–158

```python
def get_metadata(self):
    md = self._all_metadata()
    # NOTE: do not return the SALT or OPENED_KEY
    return edict(
        eid=md.eid,
        title=md.title,
        ...
    )
```

Salt and opened_key values are explicitly excluded from public metadata responses.

## Architecture Observations

1. **Single-layer defense**: The only protection against file leakage is `static_folder=None` at the Quart framework level. There is no defense-in-depth through reverse proxy configuration, filesystem isolation, or application middleware.

2. **Data/code co-location**: The SQLite database, TLS certificates, configuration files, and application source code all reside in the same directory tree, which is the application's working directory. This violates the principle of separating data from web-accessible resources.

3. **Deployment model gap**: The codebase acknowledges proxy deployment but provides no proxy configuration, creating a reliance on operations teams to independently implement file-type restrictions.
