### Summary

Analysis of the apache/steve repository reveals a comprehensive cryptographic architecture with multiple categories of secrets, but **no accompanying documentation** inventorying these secrets, classifying their criticality, or defining rotation procedures. Furthermore, the cryptographic design in `crypto.py` contains no versioning or rotation mechanism, making key rotation structurally impossible for active election data.

**Description**:


In `election.py`, secrets are generated and stored with no external documentation:
```python

```

```yaml

    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem

```

**Data Flow**:
```
[Election Creation] → Generate salt (crypto.gen_salt)
                   → Derive opened_key (crypto.gen_opened_key)
                   → Store in database (metadata table)
                   ↓
[Voter Registration] → Generate per-voter salt

[Vote Casting] → Derive vote token

               → Encrypt vote

❌ NO DOCUMENTATION exists at any stage defining:

   - Where they are stored

   - What their criticality level is
```

#### Finding ASVS-1314-HIGH-002: No Secret Rotation Schedule or Rotation Capability

**Severity**: HIGH  
**Gap Type**: Type A — No control exists  
**ASVS Requirement**: 13.1.4 (Rotation schedule)  
**Files**: `v3/steve/crypto.py` (entire module), `v3/steve/election.py` (key lifecycle)  
**CWE**: CWE-320 (Key Management Errors)

**Location**:
- **Primary File**: `v3/steve/crypto.py`
- **Related File**: `v3/steve/election.py`
- **Functions**: `_b64_vote_key()` (crypto.py:68-77), `open()` (election.py:82-94), `add_salts()` (election.py:143-151)

**Functions Checked**:
- `crypto._b64_vote_key()` — No version parameter
- `crypto.gen_opened_key()` — No version support
- `election.open()` — One-time key generation
- `election.add_salts()` — One-time salt generation
- `election.cast_vote()` — Uses fixed key derivation

**Description**:  
No rotation schedule is defined for any secret in the application. More critically, the cryptographic architecture structurally prevents rotation for election-bound secrets: once an election is opened, its `salt`, `opened_key`, per-voter salts, and derived vote tokens are permanently fixed. There is no key versioning, no re-encryption mechanism, and no documented procedure for rotating even infrastructure secrets (TLS certificates, OAuth credentials).

**Vulnerable Code**:

In `crypto.py`, the HKDF `info` parameter is a fixed constant with no version indicator:
```python
# Lines 68-77
def _b64_vote_key(vote_token: bytes, salt: bytes) -> bytes:
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,
        salt=salt,
        info=b'xchacha20_key',  # ❌ No version field - rotation impossible
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)
```

In `election.py`, salts are set exactly once and never updated:
```python
# Lines 143-151
def add_salts(self):
    "Set the SALT column in the MAYVOTE table."
    assert self.is_editable()
    # ...
    for mayvote in self.q_all_issues.fetchall():
        salt = crypto.gen_salt()  # ❌ Generated once, no update mechanism
        self.c_salt_mayvote.perform(salt, mayvote.rowid)
```

The election open procedure creates the `opened_key` once with no rotation path:
```python
# Lines 82-94
def open(self, pdb):
    assert self.is_editable()
    self.add_salts()
    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()  # ❌ One-time generation
    opened_key = crypto.gen_opened_key(edata, salt)  # ❌ Fixed for election lifetime
    self.c_open.perform(salt, opened_key, self.eid)
    # No mechanism to update these values after this point
```

**Data Flow**:
```
[Election Lifecycle]

EDITABLE → OPEN (keys generated) → CLOSED → TALLYING
           ↓                       ↓
       ❌ No rotation          ❌ No rotation
       ❌ No versioning        ❌ Keys remain fixed
       ❌ No re-keying         ❌ Compromise = permanent

[Infrastructure Secrets]

TLS Certificate → No documented renewal procedure
OAuth Secret → No documented rotation procedure
Database File → No documented key rotation for election keys within
```

**Proof of Concept**:

**Scenario 1: Compromised Election Key**
```python

    return opened_key != self.opened_key  # ❌ Would always return True if key changed

# 3. No re-encryption mechanism for existing votes
# 4. Only option: Close election and create new one (data loss)
```

**Scenario 2: TLS Certificate Expiry**
```bash

# No documented procedure exists for:
# 1. Monitoring certificate expiry
# 2. Generating new certificate
# 3. Installing new certificate
# 4. Restarting service
# Result: Unplanned outage
```

**Impact**:
- **Permanent Compromise Risk**: If any election-scoped secret is compromised, there is no recovery path without closing the election
- **Long-Lived Elections**: Elections running for months/years accumulate risk with no mitigation
- **Infrastructure Outages**: TLS certificate expiry could cause unplanned service disruption
- **Compliance Violation**: Many security frameworks require defined rotation schedules (PCI-DSS, SOC 2, etc.)
- **Defense-in-Depth Failure**: A single compromise has permanent effect for the election's lifetime

**Short-term Action** — Add key versioning support to `crypto.py` for future rotation capability:

```python
# Add to crypto.py

CURRENT_KEY_VERSION = 1

def _b64_vote_key(vote_token: bytes, salt: bytes, version: int = CURRENT_KEY_VERSION) -> bytes:
    """
    Derive a Fernet key from vote token and salt.
    
    Args:
        vote_token: Unique token for this vote
        salt: Election-specific salt
        version: Key derivation version (for future rotation support)
    
    Returns:
        Base64-encoded Fernet key
    
    Note:
        Version parameter enables future key rotation by allowing multiple
        key derivation algorithms to coexist. When rotating:
        1. Increment CURRENT_KEY_VERSION
        2. Update info parameter for new version
        3. Maintain decryption support for old versions
    """
    keymaker = hkdf.HKDF(
        algorithm=hashes.SHA256(),
        length=32,
        salt=salt,
        info=f'vote_key_v{version}'.encode(),  # ✅ Version-aware
    )
    vote_key = keymaker.derive(vote_token)
    return base64.urlsafe_b64encode(vote_key)

def decrypt(vote_token: bytes, salt: bytes, ciphertext: bytes, version: int = CURRENT_KEY_VERSION) -> bytes:
    """
    Decrypt vote data, supporting multiple key versions.
    
    This allows reading votes encrypted with older key versions during
    a rotation period.
    """
    key = _b64_vote_key(vote_token, salt, version)
    fernet = Fernet(key)
    return fernet.decrypt(ciphertext)
```

```python
#!/usr/bin/env python3
"""
Validate STeVe configuration security before deployment.
Usage: python3 validate_config.py
"""

import os
import sys
import stat

def check_file_permissions(filepath, expected_mode=0o600):
    """Check if file has secure permissions."""
    if not os.path.exists(filepath):
        return None, f"File not found: {filepath}"
    
    st = os.stat(filepath)
    actual_mode = stat.S_IMODE(st.st_mode)
    
    if actual_mode != expected_mode:
        return False, f"Insecure permissions {oct(actual_mode)}, expected {oct(expected_mode)}"
    return True, "OK"

def check_gitignore():
    """Verify .gitignore contains required entries."""
    required = ['config.yaml', '*.db', 'certs/*.pem']
    if not os.path.exists('.gitignore'):
        return False, ".gitignore not found"
    
    with open('.gitignore') as f:
        content = f.read()
    
    missing = [entry for entry in required if entry not in content]
    if missing:
        return False, f"Missing entries: {missing}"
    return True, "OK"

def check_environment_variables():
    """Verify required environment variables are set."""
    required = ['STEVE_OAUTH_SECRET']
    missing = [var for var in required if var not in os.environ]
    
    if missing:
        return False, f"Missing environment variables: {missing}"
    return True, "OK"

def main():
    """Run all security checks."""
    print("STeVe Configuration Security Validation")
    print("=" * 60)
    
    checks = [
        ("config.yaml permissions", lambda: check_file_permissions("config.yaml")),
        ("steve.db permissions", lambda: check_file_permissions("steve.db")),
        ("TLS key permissions", lambda: check_file_permissions("certs/localhost.apache.org+3-key.pem")),
        (".gitignore entries", check_gitignore),
        ("Environment variables", check_environment_variables),
    ]
    
    all_passed = True
    for name, check_func in checks:
        status, message = check_func()
        symbol = "✅" if status else "❌" if status is False else "⚠️ "
        print(f"{symbol} {name}: {message}")
        if status is False:
            all_passed = False
    
    print("=" * 60)
    if all_passed:
        print("✅ All security checks passed")
        return 0
    else:
        print("❌ Security checks failed - review errors above")
        return 1

if __name__ == "__main__":
    sys.exit(main())
```

The codebase demonstrates several strong security practices that should be maintained and used as a reference for future development:

| Pattern | Location | Assessment | Recommendation |

| **Cryptographically Secure Random Generation** | `crypto.py:gen_salt()` using `secrets.token_bytes(16)` | ✅ Excellent — Uses Python's cryptographically secure random number generator with appropriate 128-bit length | Maintain this pattern for all random value generation |
| **Strong Key Derivation Chain** | `crypto.py` — BLAKE2b → Argon2d → HKDF → Fernet | ✅ Excellent — Multi-layer defense with memory-hard and CPU-hard functions | Document the rationale for each layer in security documentation |

### Security Architecture Strengths

1. **Layered Cryptographic Defense**: The system employs multiple cryptographic primitives in sequence (BLAKE2b → Argon2d → HKDF), ensuring that compromise of any single algorithm doesn't immediately compromise the entire system.

2. **Explicit Security Boundaries**: Code comments and API design clearly distinguish between public data and sensitive secrets, reducing the risk of accidental exposure.

3. **State-Based Access Control**: The election state machine (editable → open → closed) enforces temporal access control, preventing operations outside their valid lifecycle phases.

4. **Constant-Time Operations**: Use of cryptographic libraries (Fernet, Argon2) that implement constant-time comparisons prevents timing attacks.

## Architecture Observations

### 1. Election-Scoped Key Lifecycle

**Observation**: The cryptographic design binds all secrets to the election lifecycle (editable → open → closed). Keys are generated during the "open" transition and remain fixed until the election closes.

**Implications**:
- ✅ **Positive**: Appropriate for the use case — elections have defined start and end dates
- ✅ **Positive**: Simplifies key management — no mid-election rotation complexity
- ⚠️ **Constraint**: Key rotation is only possible at election boundaries
- ⚠️ **Risk**: Long-running elections accumulate risk over time

**Recommendation**: This design constraint should be explicitly documented in `SECURITY.md` as a fundamental architectural decision with the following guidance:

```markdown
## Cryptographic Architecture: Election-Scoped Keys

### Design Principle
All election-specific cryptographic secrets are generated during the election "open" 
transition and remain immutable for the election's lifetime. This design ensures:
- Cryptographic consistency across all votes
- Simplified integrity verification
- Clear audit boundaries

### Constraints
- **No mid-election key rotation**: Keys cannot be rotated while an election is active
- **Long-lived elections**: Elections open for >90 days should be reviewed for risk
- **Compromise response**: Key compromise requires closing and recreating the election

### Operational Guidance
- **Maximum election duration**: Recommend 90 days maximum between open and close
- **Key compromise procedure**: Close election immediately, notify voters, create new election
- **Risk assessment**: Longer elections require additional monitoring and access controls
```

### 2. Argon2d vs Argon2id Selection

**Observation**: Production code in `_hash()` uses `Type.D` (Argon2d) while the benchmark function uses `Type.ID` (Argon2id).

**Current Implementation**:
```python
# crypto.py line 25
hash = argon2.low_level.hash_secret_raw(
    secret=secret,
    salt=salt,
    time_cost=3,
    memory_cost=256 * 1024,
    parallelism=4,
    hash_len=32,
    type=argon2.Type.D,  # Data-dependent
)
```

**Analysis**:
- **Argon2d**: Data-dependent memory access, resistant to time-memory tradeoff attacks, but vulnerable to side-channel attacks
- **Argon2id**: Hybrid approach, recommended for most use cases, resistant to both GPU attacks and side-channel attacks
- **Context**: Server-side key derivation in controlled environment

**Recommendation**: Document the algorithm choice with rationale:

```python
# crypto.py
def _hash(secret: bytes, salt: bytes) -> bytes:
    """
    Derive a key using Argon2d.
    
    Algorithm Selection Rationale:
    - Argon2d (data-dependent) chosen over Argon2id for maximum resistance to
      GPU-based attacks in server-side key derivation context
    - Side-channel resistance (Argon2id's advantage) less critical in server
      environment with controlled access
    - 256MB memory cost provides strong protection against parallel attacks
    
    If deployment environment has side-channel concerns (shared hosting, VMs),
    consider migrating to Argon2id (Type.ID).
    """
    hash = argon2.low_level.hash_secret_raw(
        secret=secret,
        salt=salt,
        time_cost=3,
        memory_cost=256 * 1024,
        parallelism=4,
        hash_len=32,
        type=argon2.Type.D,
    )
    return hash
```

### 5. Database as Key Storage

**Observation**: The SQLite database (`steve.db`) serves dual purposes:
1. Application data storage (votes, voter registration, election metadata)

**Security Implications**:

| Aspect | Implication | Risk Level |
|--------|-------------|------------|
| **Single Point of Failure** | Database compromise exposes both data and keys | HIGH |
| **Backup Security** | Backups must be treated as key material | HIGH |
| **Access Control** | Database access = key access | HIGH |
| **Key Extraction** | Keys can be extracted with SQL: `SELECT salt, opened_key FROM metadata` | HIGH |

**Current Mitigations**:
- ✅ File permissions (0600) recommended in updated documentation
- ✅ Sensitive fields excluded from public APIs
- ✅ Integrity verification via `is_tampered()`

```markdown
- **Simplicity**: Single file for backup and deployment
- **Atomicity**: Keys and data remain synchronized
- **Portability**: No external key management system required
```

```python
# Add to election.py or create new monitoring module

import hashlib
import json
from datetime import datetime

class DatabaseIntegrityMonitor:
    """Monitor database integrity and detect unauthorized access."""
    
    def __init__(self, db_path: str):
        self.db_path = db_path
        self.integrity_log = f"{db_path}.integrity.log"
    
    def compute_hash(self) -> str:
        """Compute SHA-256 hash of database file."""
        sha256 = hashlib.sha256()
        with open(self.db_path, 'rb') as f:
            for chunk in iter(lambda: f.read(4096), b''):
                sha256.update(chunk)
        return sha256.hexdigest()
    
    def record_checkpoint(self, operation: str):
        """Record integrity checkpoint after sensitive operations."""
        checkpoint = {
            'timestamp': datetime.utcnow().isoformat(),
            'operation': operation,
            'hash': self.compute_hash()
        }
        with open(self.integrity_log, 'a') as f:
            f.write(json.dumps(checkpoint) + '\n')
    
    def verify_integrity(self) -> bool:
        """Verify database hasn't been modified unexpectedly."""
        if not os.path.exists(self.integrity_log):
            return True  # No baseline yet
        
        with open(self.integrity_log, 'r') as f:
            last_checkpoint = json.loads(f.readlines()[-1])
        
        current_hash = self.compute_hash()
        return current_hash == last_checkpoint['hash']

# Usage in election.py
def open(self, pdb):
    # ... existing code ...
    monitor = DatabaseIntegrityMonitor(self.db_path)
    monitor.record_checkpoint('election_opened')
```

```python
# crypto.py

# Argon2d Parameters
# Selected based on OWASP recommendations and server capacity analysis
# Benchmark: ~500ms on 4-core server (acceptable for server-side KDF)
ARGON2_TIME_COST = 3           # Iterations (OWASP minimum: 2)
ARGON2_MEMORY_COST = 256*1024  # 256 MB (OWASP minimum: 19 MB)
ARGON2_PARALLELISM = 4         # Threads (match server core count)
ARGON2_HASH_LENGTH = 32        # 256-bit output

def _hash(secret: bytes, salt: bytes) -> bytes:
    """
    Derive a key using Argon2d with hardened parameters.
    
    Parameter Selection:
    - time_cost=3: Provides ~500ms computation time, acceptable for server-side
    - memory_cost=256MB: Exceeds OWASP minimum (19MB) by 13x for GPU resistance
    - parallelism=4: Matches typical server core count
    - hash_len=32: 256-bit output for AES-256 compatibility
    
    Performance: ~500ms per derivation on 4-core server
    Security: Resistant to GPU attacks (256MB per attempt), ~$1M cost for 1B attempts
    
    To adjust for different server specs, run: python3 crypto.py benchmark
    """
    hash = argon2.low_level.hash_secret_raw(
        secret=secret,
        salt=salt,
        time_cost=ARGON2_TIME_COST,
        memory_cost=ARGON2_MEMORY_COST,
        parallelism=ARGON2_PARALLELISM,
        hash_len=ARGON2_HASH_LENGTH,
        type=argon2.Type.D,
    )
    return hash
```
