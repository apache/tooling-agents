## Findings

### High Severity

#### ASVS-1331-HIGH-001: No Secrets Management Solution for Backend Cryptographic Material

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control  
**Location:** `v3/steve/election.py`, `v3/server/main.py`, `v3/server/config.yaml.example`

**Description:**  
ASVS 13.3.1 (L2) requires a secrets management solution (e.g., key vault) to securely create, store, control access to, and destroy backend secrets. The application has **no integration** with any secrets management system. All cryptographic key material is stored directly in SQLite or referenced by plain file paths.

**Affected Secrets:**

| Secret | Current Storage | Risk |
|--------|----------------|------|
| `opened_key` (election master key) | Raw bytes in SQLite `metadata` table | DB compromise → all votes decryptable |
| Per-voter salts | Raw bytes in SQLite `mayvote` table | Combined with `opened_key` enables vote decryption |
| TLS private key | File path in `config.yaml` | No vault-backed access control or audit |
| OAuth integration secrets | Presumably `config.yaml` or env vars (via asfquart) | No vault integration visible |

**Vulnerable Code — Election master key stored directly in DB:**

```python
# v3/steve/election.py, lines 75-88 (open method)
def open(self, pdb):
    assert self.is_editable()
    self.add_salts()
    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)
    # Master key written directly to SQLite - no vault, no access control
    self.c_open.perform(salt, opened_key, self.eid)
```

```python
# v3/steve/election.py, lines 258-274 (add_vote method)
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    # opened_key read directly from DB row - no vault retrieval, no access audit
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

**Vulnerable Code — Config references secrets by plain path:**

```yaml
# v3/server/config.yaml.example, lines 28-29
certfile: localhost.apache.org+3.pem
keyfile: localhost.apache.org+3-key.pem
```

```python
# v3/server/main.py, lines 77-78
kwargs['certfile'] = CERTS_DIR / app.cfg.server.certfile
kwargs['keyfile'] = CERTS_DIR / app.cfg.server.keyfile
```

**Data Flow:**
```
crypto.gen_opened_key() → raw bytes → SQLite database (no vault)
config.yaml → TLS key path → file system (no vault)
Every vote operation → reads opened_key from SQLite → derives vote tokens
```

**Impact:**  
Any compromise of the SQLite database file (file access, backup exposure, unintended copy) exposes all cryptographic material needed to decrypt every vote in every election. No access controls, audit trail, or monitoring exist around secret retrieval. The application cannot meet ASVS 13.3.1 L2 without a secrets management solution.

**Remediation:**

Integrate a secrets management solution to protect at minimum the election master key. For example, using a vault to wrap/unwrap the `opened_key`:

```python
# Example: vault-backed key protection
import hvac  # HashiCorp Vault client

class SecretManager:
    def __init__(self, vault_url, vault_token):
        self.client = hvac.Client(url=vault_url, token=vault_token)
    
    def wrap_key(self, election_id: str, opened_key: bytes) -> str:
        """Encrypt opened_key with vault transit engine, return ciphertext."""
        import base64
        b64_key = base64.b64encode(opened_key).decode()
        result = self.client.secrets.transit.encrypt_data(
            name='steve-election-keys',
            plaintext=b64_key,
        )
        return result['data']['ciphertext']
    
    def unwrap_key(self, election_id: str, wrapped_key: str) -> bytes:
        """Decrypt opened_key from vault transit engine."""
        import base64
        result = self.client.secrets.transit.decrypt_data(
            name='steve-election-keys',
            ciphertext=wrapped_key,
        )
        return base64.b64decode(result['data']['plaintext'])

# In election.py open():
wrapped = secret_mgr.wrap_key(self.eid, opened_key)
self.c_open.perform(salt, wrapped, self.eid)  # Store wrapped, not raw

# In election.py add_vote():
opened_key = secret_mgr.unwrap_key(self.eid, md.wrapped_opened_key)
```

For TLS keys and OAuth secrets, source them from vault rather than filesystem/config:

```yaml
# config.yaml - reference vault paths, not file paths
secrets:
    vault_url: https://vault.internal:8200
    tls_cert_path: secret/steve/tls/cert
    tls_key_path: secret/steve/tls/key
```

#### ASVS-1331-HIGH-002: No Secret Destruction or Lifecycle Management for Election Key Material

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with NO control  
**Location:** `v3/steve/election.py`, lines 119-125 (`close` method) and lines 52-73 (`delete` method)

**Description:**  
Once an election's `opened_key` is generated and stored in the database, it persists indefinitely. There is no mechanism to destroy, expire, or rotate the cryptographic key material after tallying is complete. ASVS 13.3.1 explicitly requires the ability to "destroy backend secrets."

**Vulnerable Code — Close operation retains all key material:**

```python
# v3/steve/election.py, lines 119-125
def close(self):
    "Close an election."
    assert self.is_open()
    # Only flips a flag - opened_key and all salts remain in database
    self.c_close.perform(self.eid)
```

**Vulnerable Code — Delete only works for editable (never-opened) elections:**

```python
# v3/steve/election.py, lines 52-73
def delete(self):
    "Delete this Election and its Issues and Person/Issue pairs."
    # Can't delete if it has been opened (even if later closed).
    assert self.is_editable()  # <-- Only editable elections can be deleted
    # ... delete operations ...
```

**Data Flow:**
```
Election opened → opened_key + salts stored in DB
Election closed → flag set, key material RETAINED
Tallying complete → no mechanism to destroy key material
Database persists → opened_key available forever for decryption
```

**Impact:**  
After tallying is complete and results are finalized, the full cryptographic chain (opened_key → vote_token → Fernet key → plaintext vote) remains reconstructable from the database. An attacker who gains database access months or years later can still decrypt all votes. This violates the principle of data minimization and the ASVS requirement for secret destruction capability.

**Remediation:**

Add a post-tally key destruction step:

```python
def destroy_key_material(self):
    """Securely destroy election key material after tallying is complete."""
    assert self.is_closed()
    
    self.db.conn.execute('BEGIN TRANSACTION')
    
    # Destroy the election master key
    self.c_destroy_opened_key.perform(self.eid)
    
    # Destroy per-voter salts
    self.c_destroy_salts.perform(self.eid)
    
    # Destroy vote tokens and ciphertexts
    self.c_destroy_votes.perform(self.eid)
    
    self.db.conn.execute('COMMIT')
    
    # Force SQLite to reclaim space (overwrite deleted pages)
    self.db.conn.execute('VACUUM')
    
    _LOGGER.info(f'Key material destroyed for election[E:{self.eid}]')
```

### Medium Severity

#### ASVS-1331-MED-003: Debug Logging Enabled in Production Deployment Paths

**Severity:** MEDIUM  
**Gap Type:** Type A — No production logging hardening  
**Location:** `v3/server/main.py`, lines 56-62 (`run_standalone`) and lines 95-100 (`run_asgi`)

**Description:**  
Both production deployment code paths (`run_standalone` and `run_asgi`) set root logging to `DEBUG` level. While the application code itself avoids logging secrets directly (commented-out `print` statements confirm this was considered), the underlying `asfpy.db` database wrapper and `asfquart` framework may log SQL query parameters at DEBUG level, potentially exposing `opened_key`, salt values, or vote tokens.

**Vulnerable Code:**

```python
# v3/server/main.py, lines 56-62 (run_standalone)
def run_standalone():
    logging.basicConfig(
        level=logging.DEBUG,  # <-- DEBUG in production
        style='{',
        format='[{asctime}|{levelname}|{name}] {message}',
        datefmt=DATE_FORMAT,
    )
```

```python
# v3/server/main.py, lines 95-100 (run_asgi)
def run_asgi():
    logging.basicConfig(
        level=logging.DEBUG,  # <-- DEBUG in production
        style='{',
        format='[{asctime}|{levelname}|{name}] {message}',
        datefmt=DATE_FORMAT,
    )
```

**Evidence of developer awareness (secrets were nearly logged):**

```python
# v3/steve/election.py, lines 85-86 (commented out)
#print('SALT:', salt)
#print('KEY:', opened_key)
```

**Data Flow:**
```
Vote submission → election.add_vote() → SQL query with opened_key bytes
asfpy.db at DEBUG → may log query params → log file/stdout
Log file → exposed opened_key, salts, vote_tokens
```

**Impact:**  
If `asfpy.db` or any middleware logs SQL parameters at DEBUG level, the election master key (`opened_key`), voter salts, and vote tokens will appear in application logs. Log aggregation systems, log files, or stdout in containerized deployments could expose this material to operators without database access.

**Remediation:**

Set production logging to INFO or WARNING, and explicitly suppress sensitive loggers:

```python
def run_standalone():
    logging.basicConfig(
        level=logging.INFO,  # Production level
        style='{',
        format='[{asctime}|{levelname}|{name}] {message}',
        datefmt=DATE_FORMAT,
    )

def run_asgi():
    logging.basicConfig(
        level=logging.INFO,  # Production level
        style='{',
        format='[{asctime}|{levelname}|{name}] {message}',
        datefmt=DATE_FORMAT,
    )
    # Explicitly suppress database query logging
    logging.getLogger('asfpy.db').setLevel(logging.WARNING)
```

Alternatively, make the log level configurable via config.yaml:

```yaml
server:
    log_level: INFO  # Override to DEBUG only for development
```

## Positive Security Patterns

1. **Sensitive field filtering:** `get_metadata()` explicitly strips `salt` and `opened_key` from public responses, with clear code comments documenting this decision.

2. **Cryptographic randomness throughout:** All random values use `secrets` module (CSPRNG) — salts, IDs, and shuffle operations.

3. **Example config without secrets:** `config.yaml.example` contains only structural configuration with no real credentials, suggesting awareness of secret separation.

4. **Developer discipline on logging:** Commented-out `print` statements for `SALT` and `KEY` demonstrate deliberate avoidance of secret logging, though the DEBUG log level undermines this.

5. **Per-voter salt uniqueness:** Each voter-issue pair gets an independent cryptographic salt, providing strong cryptographic separation between votes.

6. **Multi-layer key derivation:** The chain `election_data → BLAKE2b → Argon2 → HKDF → Fernet` provides defense-in-depth for key material.

---

## Architecture Observations

1. **Single-file database as key store:** The SQLite file simultaneously serves as the application database AND the key store for all election cryptographic material. This means database backups, file copies, or filesystem snapshots expose all secrets. A proper secrets management solution would separate key material from application data.

2. **No access audit trail for secret retrieval:** Every `_all_metadata()` call retrieves the `opened_key`, but no logging or auditing records when or by whom this sensitive material was accessed.

3. **Argon2d vs Argon2id:** The production `_hash()` function uses `argon2.low_level.Type.D` (Argon2d), while the benchmark uses `Type.ID` (Argon2id). Argon2id is the recommended variant for server-side use as it provides resistance to side-channel attacks while maintaining memory-hardness. This is a cryptographic configuration concern adjacent to secret management.
