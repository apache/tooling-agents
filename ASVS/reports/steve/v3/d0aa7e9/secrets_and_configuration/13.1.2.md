### High Severity

#### ASVS-1312-HIGH-001: Unbounded Database Connection Creation Without Pooling, Limits, or Documented Recovery

**Severity**: HIGH

**Location**: 
- **File**: `v3/steve/election.py`
- **Lines**: 42-48 (constructor), multiple class methods
- **File**: `v3/server/config.yaml.example`

**Related Functions Checked**:

| Method | New Connection | Pool Limited | Documented |
|--------|---------------|-------------|------------|
| `Election.__init__` | ✅ Yes | ❌ No | ❌ No |
| `Election.open_to_pid` | ✅ Yes | ❌ No | ❌ No |
| `Election.owned_elections` | ✅ Yes | ❌ No | ❌ No |
| `Election.upcoming_to_pid` | ✅ Yes | ❌ No | ❌ No |
| `Election.list_closed_election_ids` | ✅ Yes | ❌ No | ❌ No |
| `Election.create` | ✅ Yes (×2) | ❌ No | ❌ No |
| `Election.delete_by_eid` | ✅ Yes | ❌ No | ❌ No |

**Description**: 
The `Election` class opens a new, independent SQLite database connection for every operation via `open_database()`. There is no connection pool, no maximum connection limit, no timeout configuration, and no documented behavior for when the database becomes unavailable or connections are exhausted. Class-level methods each independently open new connections, meaning concurrent API requests create unbounded parallel connections.

**Vulnerable Code**:

`v3/steve/election.py` lines 42-48 — Each `Election` instance creates a new connection:
```python
@staticmethod
def open_database(db_fname):
    return asfpy.db.DB(db_fname, yaml_fname=QUERIES, yaml_section='election')

def __init__(self, db_fname, eid, op='Opening'):
    _LOGGER.debug(f'{op} election ID "{eid}"')
    self.db = self.open_database(db_fname)
```

Class methods that independently open additional connections with no limits:
```python
@classmethod
def open_to_pid(cls, db_fname, pid):
    db = cls.open_database(db_fname)  # New connection every call
    db.q_open_to_me.perform(pid,)
    return [row for row in db.q_open_to_me.fetchall()]

@classmethod
def owned_elections(cls, db_fname, pid):
    db = cls.open_database(db_fname)  # Another new connection
    ...

@classmethod
def upcoming_to_pid(cls, db_fname, pid):
    db = cls.open_database(db_fname)  # Yet another new connection
    ...

@classmethod
def list_closed_election_ids(cls, db_fname, include_open=False):
    db = cls.open_database(db_fname)  # And another
    ...
```

`v3/server/config.yaml.example` — No database connection limits:
```yaml
# Path to the database, absolute or relative to APP.
db: steve.db
```

**Data Flow**: 
`HTTP Request → API endpoint → Election() or Election.classmethod() → open_database() → asfpy.db.DB() → new SQLite connection (unbounded)`

**Proof of Concept**: 
N concurrent authenticated users simultaneously loading their dashboard triggers N calls to `open_to_pid` + N calls to `upcoming_to_pid` + N calls to `owned_elections` = 3N independent database connections opened simultaneously with no cap.

**Impact**: 
Under concurrent load, each inbound request opens at least one new SQLite connection. SQLite uses file-level locking; under write contention, connections queue on the lock with no configured timeout. Concurrent read-heavy operations (listing elections) exhaust file descriptors. No fallback or circuit-breaker exists—the application will produce unhandled exceptions (e.g., `sqlite3.OperationalError: unable to open database file` or `database is locked`), leading to cascading failures.

**Remediation**:

1. Add connection pool configuration to `config.yaml.example`:
```yaml
db:
  path: steve.db
  pool_size: 10           # Maximum concurrent connections
  pool_timeout: 5         # Seconds to wait for a connection
  max_overflow: 5         # Burst connections beyond pool_size
  # Behavior when pool exhausted: return HTTP 503 with Retry-After header
```

2. Implement a connection pool or singleton pattern in `election.py`:
```python
import threading

class ConnectionPool:
    _lock = threading.Lock()
    _pool = {}
    MAX_CONNECTIONS = 10
    POOL_TIMEOUT = 5.0

    @classmethod
    def get_connection(cls, db_fname):
        """Get a connection from the pool, blocking if exhausted."""
        # Implementation using queue.Queue with maxsize=MAX_CONNECTIONS
        # Raises ServiceUnavailable after POOL_TIMEOUT
        ...
```

3. Document fallback behavior when limits are reached.

#### ASVS-1312-HIGH-002: No Concurrency Limits on Memory-Intensive Argon2 Operations Enabling Resource Exhaustion

**Severity**: HIGH

**Location**:
- **File**: `v3/steve/crypto.py`
- **Lines**: 88-98
- **File**: `v3/steve/election.py`
- **Lines**: 230-243

**Related Functions Checked**:
- `crypto._hash()` (lines 88-98)
- `crypto.gen_vote_token()` (calls `_hash`)
- `Election.add_vote()` (lines 230-243)
- `Election.tally_issue()` (line 266, loops over voters)

**Description**: 
Each vote submission (`add_vote`) triggers Argon2 hashing with `memory_cost=65536` (64 MiB) per invocation. There are no documented or implemented limits on concurrent cryptographic operations. An authenticated attacker can submit concurrent vote requests, each consuming 64 MiB of memory with no cap, leading to memory exhaustion and denial of service. This constitutes an undocumented resource consumption pattern for an internal service (the cryptographic subsystem).

**Vulnerable Code**:

`v3/steve/crypto.py` lines 88-98:
```python
def _hash(data: bytes, salt: bytes) -> bytes:
    "Use Argon2 to hash the data, with default tuning parameters."
    return argon2.low_level.hash_secret_raw(
        secret=data,
        salt=salt,
        time_cost=2,
        memory_cost=65536,  # 64 MiB per invocation
        parallelism=4,
        hash_len=32,
        type=argon2.low_level.Type.D,
    )
```

`v3/steve/election.py` lines 230-243 — Two Argon2 calls per vote (via `gen_vote_token` and internally during key derivation):
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)  # Argon2: 64 MiB
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

**Data Flow**: 
`Authenticated HTTP Request → add_vote() → crypto.gen_vote_token() → crypto._hash() → argon2 (64 MiB allocation, no concurrency limit)`

**Proof of Concept**:
With 20 concurrent vote submissions, the application allocates 20 × 64 MiB = 1.28 GiB of memory simultaneously for Argon2 alone.

**Impact**: 
With 20 concurrent vote submissions, the application allocates 20 × 64 MiB = 1.28 GiB of memory simultaneously for Argon2 alone. The `tally_issue` method is worse—it calls `gen_vote_token` once per eligible voter in a loop (line 266), meaning tallying an election with 1,000 voters allocates 64 GiB sequentially (or concurrently if multiple tallies run). No documentation defines these resource requirements or maximum concurrent operation limits.

**Remediation**:

1. Document the resource profile of cryptographic operations:
```yaml
# In config.yaml or operations documentation
crypto:
  argon2_memory_mb: 64            # Memory per Argon2 invocation
  max_concurrent_hash_ops: 4      # Limit concurrent Argon2 operations
  # When limit reached: queue requests, return 429 after 10s timeout
```

2. Implement a semaphore to bound concurrent Argon2 operations:
```python
import asyncio

_ARGON2_SEMAPHORE = asyncio.Semaphore(4)  # Max 4 concurrent = 256 MiB max

async def bounded_hash(data: bytes, salt: bytes) -> bytes:
    async with _ARGON2_SEMAPHORE:
        return await asyncio.to_thread(_hash, data, salt)
```

#### ASVS-1312-MED-003: No Web Server Concurrency Limits Configured or Documented

**Severity**: MEDIUM

**Location**:
- **File**: `v3/server/config.yaml.example`
- **File**: `v3/server/main.py`
- **Lines**: 50-88, 91-108

**Related Functions Checked**:
- `app.runx()` (line 78-80, standalone mode)
- `create_app()` (lines 103-108, ASGI mode)

**Description**: 
The server configuration and startup code define no maximum concurrent connections, worker limits, request queue sizes, or keepalive timeouts. The `config.yaml.example` only specifies port and TLS settings. Neither standalone nor ASGI mode documents or configures concurrency boundaries.

**Vulnerable Code**:

`v3/server/config.yaml.example` — Configuration is limited to port and TLS:
```yaml
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
db: steve.db
```

`v3/server/main.py` lines 78-80 — Standalone server started without concurrency parameters:
```python
app.runx(port=app.cfg.server.port, extra_files=extra_files, **kwargs)
```

`v3/server/main.py` lines 103-108 — ASGI mode with no Hypercorn configuration documented:
```python
global steve_app
steve_app = create_app()
```

**Data Flow**:
`HTTP Connection → asfquart/Hypercorn (no limits) → Application handlers → Unbounded resource consumption`

**Proof of Concept**:
Deploy the application with default configuration and establish thousands of concurrent connections. No documented limit exists to prevent resource exhaustion.

**Impact**: 
Without documented and configured connection limits, the application relies entirely on the default behavior of asfquart/Hypercorn, which may accept thousands of concurrent connections. Combined with the database and Argon2 resource issues above, this creates a multiplier effect for resource exhaustion. Operations teams have no documented guidance on capacity planning or expected failure modes.

**Remediation**:

1. Add server concurrency configuration:
```yaml
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
    max_connections: 100        # Maximum concurrent connections
    workers: 2                  # Worker processes
    keepalive_timeout: 30       # Seconds
    request_timeout: 60         # Seconds per request
    # When max_connections reached: new connections receive 503
```

2. For Hypercorn ASGI deployment, document and provide a configuration file:
```toml
# hypercorn.toml
bind = "0.0.0.0:58383"
workers = 2
backlog = 100
graceful_timeout = 10
```

#### ASVS-1312-MED-004: No OAuth Service Connection Limits or Failure Handling Documented

**Severity**: MEDIUM

**Location**:
- **File**: `v3/server/main.py`
- **Lines**: 35-38

**Related Functions Checked**:
- `asfquart.generics.OAUTH_URL_INIT` configuration
- `asfquart.generics.OAUTH_URL_CALLBACK` configuration

**Description**: 
The application integrates with an external OAuth service (`oauth.apache.org`) for authentication. There is no documented or configured connection limit, timeout, retry policy, or fallback behavior for when the OAuth service is unreachable or slow. The URLs are hardcoded with no resilience configuration.

**Vulnerable Code**:

```python
asfquart.generics.OAUTH_URL_INIT = (
    'https://oauth.apache.org/auth?state=%s&redirect_uri=%s'
)
asfquart.generics.OAUTH_URL_CALLBACK = 'https://oauth.apache.org/token?code=%s'
```

**Data Flow**:
`User Authentication Request → OAuth service call (no timeout) → Indefinite hang on service unavailability`

**Proof of Concept**:
Simulate OAuth service unavailability (DNS failure, network partition, or slow response). Authentication requests will hang indefinitely without timeout, consuming server resources.

**Impact**: 
If `oauth.apache.org` becomes slow or unresponsive, authentication requests will hang indefinitely (no timeout configured), consuming server resources (connections, worker threads). A slowloris-style attack against the OAuth provider or DNS manipulation could cause cascading failure in the voting application. No documentation exists for operators on how to detect or respond to OAuth service degradation.

**Remediation**:

1. Document OAuth service dependencies and limits:
```yaml
oauth:
    base_url: https://oauth.apache.org
    connect_timeout: 5          # Seconds
    read_timeout: 10            # Seconds
    max_retries: 2
    circuit_breaker_threshold: 5  # Failures before opening circuit
    # Fallback: Display "Authentication service unavailable" page
    # Recovery: Auto-retry after 30 seconds
```

## Positive Security Patterns

1. **Transaction Management**: `election.py` properly uses `BEGIN TRANSACTION`/`COMMIT` for multi-step database operations (lines 61-70, 127-136), which provides atomicity and reduces lock contention.

2. **Cryptographically Secure Randomness**: `crypto.py` consistently uses `secrets` module for all random operations (salt generation, ID creation, shuffling).

3. **Strong Key Derivation**: Argon2 with reasonable parameters (64 MiB memory, time_cost=2, parallelism=4) provides strong key derivation, even if the resource profile needs documentation.

4. **Connection Cleanup on Deletion**: `Election.delete()` and `_disappeared()` close database connections after use, preventing leaked connections in those specific paths.

5. **TLS Support**: Configuration supports TLS certificate/key configuration for encrypted transport.

---

## Architecture Observations

1. **Connection-Per-Operation Pattern**: The `Election` class follows a connection-per-operation model where every class method and instance creates its own database connection. This is a fundamental architectural choice that makes connection pooling difficult to retrofit. The class methods (`open_to_pid`, `owned_elections`, etc.) don't close connections, relying on garbage collection.

2. **Synchronous Argon2 in Async Context**: The application appears to use an async framework (asfquart/Quart) but Argon2 operations in `crypto.py` are synchronous and CPU-bound. Without `asyncio.to_thread()` wrapping, these block the event loop, effectively serializing all concurrent requests through a single thread during vote operations.

3. **No Service Health Checks**: There is no health check endpoint or internal monitoring for database connectivity, OAuth reachability, or resource utilization that operators could use to detect degradation.
