## Findings

### Medium Severity

#### ASVS-1346-MED-001: Hypercorn Server Header Exposes Backend Component Identity and Version

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with no control  
**File:** `v3/server/main.py` (lines 32–42, 82–103)  
**Config:** `v3/server/config.yaml.example` (entire file)

**Description:**

The application uses Hypercorn as its production ASGI server (confirmed by the explicit comments and code path in `run_asgi()`). Hypercorn, by default, sends a `Server` response header on every HTTP response (e.g., `server: hypercorn-h11` or `server: hypercorn-h2`), which directly discloses the server software name and transport protocol version to any client.

Neither the application startup code nor the configuration template includes any mechanism to suppress or override this header.

**Vulnerable Code:**

```python
# v3/server/main.py — run_asgi() (production path)
def run_asgi():
    # ...
    _LOGGER.info(' ** Run-mode: ASGI')
    global steve_app
    steve_app = create_app()
    # No Hypercorn configuration to suppress Server header
```

```python
# v3/server/main.py — create_app()
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)
    import pages
    import api
    return app
    # No after_request hook to strip or override response headers
```

```yaml
# v3/server/config.yaml.example — No header suppression settings
server:
    port: 58383
    certfile: localhost.apache.org+3.pem
    keyfile: localhost.apache.org+3-key.pem
db: steve.db
# Missing: no server_header, no include_server_header: false
```

**Data Flow:**

```
Any HTTP request → Hypercorn ASGI Server → HTTP response with `Server: hypercorn-h11` header → Client
                                            ↑
                                  No suppression configured
```

**Proof of Concept:**

```bash
$ curl -sI https://steve.example.org:58383/
HTTP/1.1 200 OK
server: hypercorn-h11
date: ...
content-type: text/html; charset=utf-8
```

The `server: hypercorn-h11` header reveals the exact ASGI server software.

**Impact:**

An attacker can fingerprint the backend technology stack without any application interaction. This enables targeted attacks against known Hypercorn vulnerabilities and reduces attacker reconnaissance effort. While not directly exploitable alone, this information is commonly chained with other vulnerabilities (as noted in the ASVS section description).

**Remediation:**

**Option A — Hypercorn configuration file (recommended):**

Create a `hypercorn.toml` configuration:
```toml
include_server_header = false
```

Launch with:
```bash
$ hypercorn --config hypercorn.toml main:steve_app
```

**Option B — After-request middleware in the application:**

```python
# In create_app() or a dedicated middleware module
@app.after_request
async def strip_server_header(response):
    response.headers.pop('Server', None)
    response.headers.pop('X-Powered-By', None)
    return response
```

**Option C — Add to config.yaml and apply during app creation:**

```yaml
server:
    port: 58383
    suppress_server_header: true
```

#### ASVS-1346-MED-002: Missing Custom Error Handlers Allow Default Framework Error Pages

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with no control  
**File:** `v3/server/main.py`, function `create_app()` (lines 32–42)

**Description:**

The `create_app()` function constructs the application and imports route modules (`pages`, `api`) but does not register any custom error handlers for HTTP error codes (400, 404, 405, 500, etc.). Quart-based frameworks (which `asfquart` wraps) generate default error pages that can reveal the framework name, version, and in DEBUG-adjacent configurations, full Python stack traces including library paths and versions.

Compounding this, the `run_asgi()` production path configures the root logger at `DEBUG` level:

```python
def run_asgi():
    logging.basicConfig(
        level=logging.DEBUG,  # <-- DEBUG in production
        ...
    )
```

While log level alone doesn't guarantee version exposure in responses, it increases the verbosity of information available if any unhandled exception propagates to the default error handler.

**Vulnerable Code:**

```python
# v3/server/main.py — create_app()
def create_app():
    "Create the asfquart app and its endpoints."

    import asfquart.generics
    # ... OAuth config ...

    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    import pages   # noqa: F401
    import api     # noqa: F401

    return app
    # No @app.errorhandler(404), @app.errorhandler(500), etc.
    # No generic exception handler registered
```

**Data Flow:**

```
HTTP request for invalid route → Quart routing → No matching route → Default 404 handler
                                                                       ↓
                                                        HTML page with "Quart" branding,
                                                        potentially including version string
                                                                       ↓
                                                                    Client

HTTP request triggering unhandled exception → Default 500 handler → Traceback including
                                                                     library paths like:
                                                                     /lib/python3.x/site-packages/quart/...
                                                                     /lib/python3.x/site-packages/cryptography/...
```

**Proof of Concept:**

```bash
# Trigger a 404 - default error page reveals framework
$ curl -s https://steve.example.org:58383/nonexistent-path

# Response may contain:
# <!DOCTYPE HTML PUBLIC...>
# <title>404 Not Found</title>
# <p>The requested URL was not found on the server.</p>
# <!-- Powered by Quart x.y.z -->

# Trigger a 405 - wrong method on known endpoint
$ curl -s -X DELETE https://steve.example.org:58383/
```

**Impact:**

Default error pages disclose the web framework name and version, and unhandled exceptions can expose Python version, library paths, and dependency versions. This assists attackers in identifying specific vulnerabilities in the exact versions of Quart, Hypercorn, cryptography, argon2, and other dependencies.

**Remediation:**

Register custom error handlers that return generic responses:

```python
def create_app():
    app = asfquart.construct('steve', app_dir=THIS_DIR, static_folder=None)

    # Register custom error handlers to prevent version disclosure
    @app.errorhandler(404)
    async def not_found(error):
        return {"error": "Not found"}, 404

    @app.errorhandler(405)
    async def method_not_allowed(error):
        return {"error": "Method not allowed"}, 405

    @app.errorhandler(500)
    async def internal_error(error):
        # Log the real error server-side, return generic message
        _LOGGER.exception("Internal server error")
        return {"error": "Internal server error"}, 500

    @app.errorhandler(Exception)
    async def generic_error(error):
        _LOGGER.exception("Unhandled exception")
        return {"error": "An error occurred"}, 500

    import pages
    import api
    return app
```

Additionally, set production logging to `INFO` rather than `DEBUG`:

```python
def run_asgi():
    logging.basicConfig(
        level=logging.INFO,  # Not DEBUG in production
        ...
    )
```

## Architecture Observations

1. **No production vs. development configuration distinction**: The `config.yaml.example` has no mechanism to toggle between development and production modes. Both `run_standalone()` and `run_asgi()` use DEBUG-level logging.

2. **asfquart abstraction layer**: The application delegates to `asfquart.construct()` for app creation. Without visibility into this library's defaults, it's unknown whether it provides any built-in header suppression or error handling. The audit assumes standard Quart/Hypercorn defaults.

3. **Reverse proxy assumption**: The config comments suggest "a proxy sits in front of this server." If a reverse proxy (e.g., Apache httpd, nginx) is used, it may override or strip the `Server` header from the upstream Hypercorn response. However, defense-in-depth requires the application itself to not leak this information, as proxy configuration is outside the application's control.
