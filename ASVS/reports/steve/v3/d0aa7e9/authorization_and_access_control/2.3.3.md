## Findings

### High Severity

#### Finding ASVS-233-HIGH-001: Election Opening Operation Lacks Atomic Transaction Control

**Severity:** HIGH  
**Finding ID:** ASVS-233-HIGH-001  
**Gap Type:** Type B (Control EXISTS in `add_salts` but NOT APPLIED to the enclosing `open()` operation)

**Location:**
- **File:** `v3/steve/election.py`
- **Function:** `Election.open()` (lines 74-89) and `Election.add_salts()` (lines 126-140)

**Related Functions Checked:**
- `Election.add_salts()` (L126-140)
- `Election.gather_election_data()` (referenced)
- `crypto.gen_salt()` (referenced)
- `crypto.gen_opened_key()` (referenced)

**Description:**

The election opening operation is a critical state transition that involves multiple database modifications across two separate committed transactions. The `open()` method first calls `add_salts()`, which commits its own transaction containing per-voter salt generation, then separately executes cryptographic operations and commits the election state change. This split-transaction approach violates ASVS 2.3.3's requirement for atomic business operations.

**Vulnerable Code:**

```python
def open(self, pdb):
    # Double-check the Election is in the editing state.
    assert self.is_editable()

    # Add salts first. If this is gonna fail, then make sure it
    # happens before we move to the "opened" state.
    self.add_salts()                                    # <-- Step 1: COMMITS its own transaction

    edata = self.gather_election_data(pdb)              # <-- Step 2: reads data (can fail)
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)     # <-- Step 3: computation (can fail)

    self.c_open.perform(salt, opened_key, self.eid)     # <-- Step 4: SEPARATE auto-commit
```

```python
def add_salts(self):
    assert self.is_editable()
    self.db.conn.execute('BEGIN TRANSACTION')           # <-- Transaction A begins
    self.q_all_issues.perform(self.eid)
    for mayvote in self.q_all_issues.fetchall():
        salt = crypto.gen_salt()
        self.c_salt_mayvote.perform(salt, mayvote.rowid)
    self.db.conn.execute('COMMIT')                      # <-- Transaction A commits
```

**Data Flow:**
1. `open()` calls `add_salts()` which commits per-voter salts in its own transaction
2. Control returns to `open()` which then performs `gather_election_data()`
3. Cryptographic key generation via `gen_opened_key()` is executed
4. Finally, `c_open.perform()` auto-commits the election state change
5. If steps 2-4 fail after `add_salts()` has committed, the database retains committed salts while the election remains in "editable" state

**Proof of Concept:**

If `crypto.gen_opened_key(edata, salt)` raises an exception (e.g., Argon2 memory allocation failure, corrupt election data), or if `c_open.perform()` fails (e.g., disk full, database locked):

1. All mayvote entries have non-NULL salts (committed by `add_salts()`)
2. The election table still has `salt IS NULL AND opened_key IS NULL` (editable state)
3. The election is in a state not cleanly represented by the state machine: editable with populated per-voter salts
4. A retry of `open()` will regenerate all salts (losing the originals), but if the failure is persistent, the election cannot transition states while salts are irrecoverably set

**Impact:**

The election opening operation—a critical, irreversible state transition—is not atomic. A partial failure leaves the database in an inconsistent state where per-voter cryptographic salts are committed but the election is not marked as open. While the system can recover by retrying, this violates the fundamental requirement that business logic operations either succeed entirely or roll back to the previous correct state. In a voting system, state transition integrity is foundational to trust.

**Remediation:**

```python
def open(self, pdb):
    assert self.is_editable()

    self.db.conn.execute('BEGIN TRANSACTION')
    try:
        # Inline salt generation (instead of calling add_salts with its own txn)
        self.q_all_issues.perform(self.eid)
        for mayvote in self.q_all_issues.fetchall():
            salt = crypto.gen_salt()
            self.c_salt_mayvote.perform(salt, mayvote.rowid)

        edata = self.gather_election_data(pdb)
        salt = crypto.gen_salt()
        opened_key = crypto.gen_opened_key(edata, salt)
        self.c_open.perform(salt, opened_key, self.eid)

        self.db.conn.execute('COMMIT')
    except Exception:
        self.db.conn.execute('ROLLBACK')
        raise
```

#### Finding ASVS-233-HIGH-002: Batch Vote Submission Lacks Transactional Atomicity

**Severity:** HIGH  
**Finding ID:** ASVS-233-HIGH-002  
**Gap Type:** Type A (Entry point with NO transaction control for multi-step operation)

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint()` (lines 376–417)
- **Related File:** `v3/steve/election.py`
- **Related Function:** `Election.add_vote()` (lines 201–212)

**Related Functions Checked:**
- `Election.add_vote()` (L201-212)
- `Election._all_metadata()` (referenced)
- `crypto.gen_vote_token()` (referenced)
- `crypto.create_vote()` (referenced)

**Description:**

The vote submission endpoint processes multiple votes from a single user ballot submission by iterating through each vote and calling `add_vote()` individually. Each `add_vote()` call performs a single INSERT that auto-commits immediately. If any vote in the sequence fails, all previously committed votes remain in the database while subsequent votes are lost, resulting in a partial ballot submission that violates voter intent and election integrity.

**Vulnerable Code:**

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    # ...
    for iid, votestring in votes.items():
        if iid not in issue_dict:
            await flash_danger(f'Invalid issue ID: {iid}')
            return quart.redirect(f'/vote-on/{election.eid}', code=303)

        try:
            election.add_vote(result.uid, iid, votestring)      # <-- Auto-commits per iteration
            # ...
        except Exception as e:
            # ...
            await flash_danger(f'Error submitting vote for issue {iid}.')
            return quart.redirect(f'/vote-on/{election.eid}', code=303)  # <-- Partial votes committed

    await flash_success('Votes submitted successfully!')
```

And in `election.py`, `add_vote()` (lines 201–212):

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)              # <-- Single auto-commit INSERT
```

**Data Flow:**

1. User submits form with votes for N issues
2. `do_vote_endpoint` iterates through each vote
3. For each iteration, calls `election.add_vote()` which auto-commits each INSERT individually
4. If vote K of N fails, votes 1 through K-1 are permanently committed
5. Votes K through N are lost
6. User receives error but partial votes exist in database

**Proof of Concept:**

A user votes on 5 issues in a single form submission. If the election is closed by an administrator between the 3rd and 4th `add_vote()` call:

1. `add_vote()` for issue 4 calls `_all_metadata(self.S_OPEN)` which raises `ElectionBadState`
2. The exception is caught, user sees "Error submitting vote for issue [4]"
3. Votes for issues 1, 2, 3 are committed; votes for issues 4, 5 are not
4. User sees the error but has no way to know which votes were recorded
5. They cannot retry because the election is now closed

Alternatively, if `crypto.create_vote()` fails for one issue (e.g., encoding error on the votestring), the same partial-commit scenario occurs.

**Impact:**

In a voting system, the user's ballot submission is the most critical business operation. A partial ballot violates voter intent—the user believed they were submitting all votes together. The tallied results may include incomplete ballots, undermining election integrity. This directly violates ASVS 2.3.3's requirement that business operations succeed entirely or roll back.

**Remediation:**

```python
# In election.py, add a batch vote method:
def add_votes(self, pid: str, votes: dict[str, str]):
    """Add multiple votes atomically. VOTES is {iid: votestring}."""
    md = self._all_metadata(self.S_OPEN)

    self.db.conn.execute('BEGIN TRANSACTION')
    try:
        for iid, votestring in votes.items():
            mayvote = self.q_get_mayvote.first_row(pid, iid)
            if not mayvote:
                raise ValueError(f'Not eligible to vote on issue {iid}')
            vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
            ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
            self.c_add_vote.perform(vote_token, ciphertext)
        self.db.conn.execute('COMMIT')
    except Exception:
        self.db.conn.execute('ROLLBACK')
        raise
```

```python
# In pages.py, use the batch method:
async def do_vote_endpoint(election):
    result = await basic_info()
    form = edict(await quart.request.form)
    votes = {key.split('-', 1)[1]: value for key, value in form.items() if key.startswith('vote-')}
```

### Medium Severity

#### Finding ASVS-233-MED-001: Missing ROLLBACK Handling in Transactional Methods

**Severity:** MEDIUM  
**Finding ID:** ASVS-233-MED-001  
**Gap Type:** Type C (Transaction control CALLED but ROLLBACK handling absent — partial success can be committed on retry or leave DB locked)

**Location:**
- **File:** `v3/steve/election.py`
- **Functions:** `Election.delete()` (lines 55–70), `Election.add_salts()` (lines 126–140)

**Related Functions Checked:**
- `Election.delete()` (L55-70)
- `Election.add_salts()` (L126-140)
- `c_delete_mayvote.perform()` (referenced)
- `c_delete_issues.perform()` (referenced)
- `c_delete_election.perform()` (referenced)
- `c_salt_mayvote.perform()` (referenced)

**Description:**

The `delete()` and `add_salts()` methods both use explicit `BEGIN TRANSACTION` / `COMMIT` blocks to wrap multi-step database operations. However, neither method includes exception handling with explicit `ROLLBACK` statements. When an exception occurs during the transaction, the `COMMIT` is never reached, leaving the database connection with an open, uncommitted transaction that holds SQLite's exclusive write lock until garbage collection occurs.

**Vulnerable Code — `delete()`:**

```python
def delete(self):
    assert self.is_editable()
    self.db.conn.execute('BEGIN TRANSACTION')        # <-- Transaction begins
    self.c_delete_mayvote.perform(self.eid)
    self.c_delete_issues.perform(self.eid)           # <-- If this fails...
    self.c_delete_election.perform(self.eid)
    self.db.conn.execute('COMMIT')                   # <-- ...COMMIT never reached
    self.db.conn.close()                             # <-- ...close() never reached
    self.db = None                                   # <-- Connection left open with uncommitted txn
```

**Vulnerable Code — `add_salts()`:**

```python
def add_salts(self):
    assert self.is_editable()
    self.db.conn.execute('BEGIN TRANSACTION')        # <-- Transaction begins
    self.q_all_issues.perform(self.eid)
    for mayvote in self.q_all_issues.fetchall():
        salt = crypto.gen_salt()
        self.c_salt_mayvote.perform(salt, mayvote.rowid)  # <-- If this fails mid-loop...
    self.db.conn.execute('COMMIT')                   # <-- ...COMMIT never reached, no ROLLBACK
```

**Data Flow:**

1. `BEGIN TRANSACTION` is executed
2. Multiple SQL operations proceed
3. If any operation raises an exception, the exception propagates up the call stack
4. `COMMIT` is never reached
5. No `ROLLBACK` is executed
6. The database connection retains an open write transaction
7. SQLite write lock held indefinitely until garbage collection

**Proof of Concept:**

If `c_delete_issues.perform()` fails (e.g., a foreign key violation because a mayvote was added concurrently, or a disk I/O error), the exception propagates out of `delete()`. The connection remains open with an uncommitted transaction, holding SQLite's exclusive write lock. Any subsequent write attempt on the database will block or fail with "database is locked" until the Python garbage collector closes the connection or the process restarts.

**Impact:**

Without explicit `ROLLBACK` on exception, failed transactions leave the database connection in a dirty state. For SQLite, this means an exclusive write lock is held, potentially blocking all other write operations to the election database. While SQLite will implicitly roll back when the connection is eventually closed, relying on garbage collection for transactional cleanup is unreliable and can cause prolonged service disruption in the election system.

**Remediation:**

```python
def delete(self):
    assert self.is_editable()
    self.db.conn.execute('BEGIN TRANSACTION')
    try:
        self.c_delete_mayvote.perform(self.eid)
        self.c_delete_issues.perform(self.eid)
        self.c_delete_election.perform(self.eid)
        self.db.conn.execute('COMMIT')
    except Exception:
        self.db.conn.execute('ROLLBACK')
        raise
    finally:
        self.db.conn.close()
        self.db = None
```

```python
def add_salts(self):
    assert self.is_editable()
    self.db.conn.execute('BEGIN TRANSACTION')
    try:
        self.q_all_issues.perform(self.eid)
        for mayvote in self.q_all_issues.fetchall():
            salt = crypto.gen_salt()
            self.c_salt_mayvote.perform(salt, mayvote.rowid)
        self.db.conn.execute('COMMIT')
    except Exception:
        self.db.conn.execute('ROLLBACK')
        raise
```

## Positive Security Patterns

1. **State machine enforcement via assertions**: Methods like `add_issue()`, `edit_issue()`, `delete_issue()`, `add_vote()`, `close()`, and `add_salts()` all verify the election is in the correct state before proceeding (`assert self.is_editable()`, `assert self.is_open()`). State is computed from database metadata, not cached.

2. **Explicit transaction usage in `delete()` and `add_salts()`**: The codebase demonstrates awareness of the need for transactions in multi-step operations. Both methods use `BEGIN TRANSACTION` / `COMMIT` blocks.

3. **Referential integrity via foreign keys**: The schema defines `FOREIGN KEY` constraints with `ON DELETE RESTRICT`, preventing orphaned records at the database level.

4. **Unique constraint retry logic**: Both `create()` and `add_issue()` handle `sqlite3.IntegrityError` by retrying with a new ID, correctly handling the case where a randomly generated ID collides.

5. **Election state immutability**: The `prevent_open_close_update` trigger prevents modification of advisory timestamps after election closure, reinforcing state finality at the database level.

6. **Tamper detection**: The `is_tampered()` method recomputes the `opened_key` from current election data and compares it with the stored value, detecting unauthorized modifications to election configuration after opening.

---

## Architecture Observations

1. **Auto-commit default mode**: The codebase operates in SQLite auto-commit mode by default, with explicit `BEGIN TRANSACTION` only in `delete()` and `add_salts()`. This is fine for single-statement operations but creates gaps when multiple statements must be atomic.

2. **Database connection lifecycle**: Each `Election` instance holds its own `asfpy.db.DB` connection. The connection is not managed via context manager patterns, increasing risk of leaked connections on exception paths.

3. **Business operation boundaries**: The boundary between "business operation" (e.g., "open an election") and "database operations" (add salts, set opened_key) is not explicitly marked. Transaction boundaries should align with business operation boundaries.

4. **Async handler / sync database mismatch**: The web handlers are `async` (Quart), but `Election` methods perform synchronous SQLite operations. While SQLite serializes writes anyway, the async context means multiple coroutines could interleave their non-transactional operations within the same event loop tick.
