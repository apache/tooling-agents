### Key Findings

The audit reveals **no business logic locking mechanisms** protecting critical election operations. The vote submission pathway has a Time-of-Check-to-Time-of-Use (TOCTOU) race condition that allows votes to be recorded after election closure. Election state transitions (`open`/`close`) lack transactional atomicity, enabling concurrent corruption. State enforcement relies on Python `assert` statements that can be globally disabled.

## Findings

### High Severity

#### ASVS-234-HIGH-001: TOCTOU Race Condition Allows Vote Insertion After Election Closure

**Severity:** HIGH  
**Gap Type:** Type D — Control called but sensitive operation occurs after a window where state can change  
**File:** `v3/steve/election.py`, method `add_vote`, lines 258–269  
**Related Functions:** `close` method, lines 113–119; `do_vote_endpoint` in `v3/server/pages.py`, lines 403–446

**Description:**

The vote submission pathway has a Time-of-Check-to-Time-of-Use (TOCTOU) race condition. The `add_vote` method checks that the election is open at line 261, but the actual vote insertion at line 269 occurs after CPU-intensive cryptographic operations. During this window, the election can be closed by another request, yet the vote will still be recorded.

**Vulnerable Code:**

```python
# election.py, add_vote (line 258)
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)  # CHECK: election is open

    ### validate VOTESTRING for ISSUE.TYPE voting

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)  # CPU-intensive

    # Pass MAYVOTE.SALT for PBKDF.
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)  # CPU-intensive

    self.c_add_vote.perform(vote_token, ciphertext)  # WRITE: no lock held, state may have changed
```

```python
# election.py, close (line 113)
def close(self):
    "Close an election."
    assert self.is_open()
    self.c_close.perform(self.eid)  # Commits immediately in autocommit mode
```

The `vote` table schema confirms no database-level protection:

```sql
-- schema.sql, line ~179
CREATE TABLE vote (
    vid  INTEGER PRIMARY KEY AUTOINCREMENT,
    vote_token  BLOB NOT NULL  CHECK (length(vote_token) = 32),
    ciphertext  BLOB NOT NULL
) STRICT;
-- No foreign key to election, no trigger checking election state
```

**Data Flow:**

`do_vote_endpoint` → `await quart.request.form` (yield point) → `add_vote()` → `_all_metadata(S_OPEN)` (check passes) → `crypto.gen_vote_token()` + `crypto.create_vote()` (CPU gap) → `c_add_vote.perform()` (vote inserted into closed election)

**Proof of Concept:**

```
# Attacker (legitimate voter) times vote submission against election closure

# Step 1: Admin sends close request
GET /do-close/abc1234567 HTTP/1.1

# Step 2: Attacker sends vote concurrently (same moment or slightly before)
POST /do-vote/abc1234567 HTTP/1.1
Content-Type: application/x-www-form-urlencoded

vote-def9876543=yes

# Race window: add_vote reads state as OPEN, then close() commits,
# then add_vote inserts the vote. The vote_token is computed using
# opened_key from the OPEN state, so it's cryptographically valid
# and will be included in subsequent tally.
```

With multi-worker deployments (e.g., Hypercorn with multiple processes), the window between `_all_metadata(S_OPEN)` and `c_add_vote.perform()` is widened by the CPU-intensive `gen_vote_token()` and `create_vote()` operations (key derivation with PBKDF/Argon2).

**Impact:**

Votes can be recorded and tallied for elections that have already been officially closed. In a contested election, this undermines the integrity of the result. The tampered vote is cryptographically valid (uses the correct opened_key and salt), so it cannot be distinguished from a legitimate vote during tallying.

**Remediation:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # Wrap the entire check-and-write in a transaction with
    # IMMEDIATE mode to acquire a write lock before reading state
    self.db.conn.execute('BEGIN IMMEDIATE')
    try:
        md = self._all_metadata(self.S_OPEN)
        
        mayvote = self.q_get_mayvote.first_row(pid, iid)
        if not mayvote:
            raise ValueError(f'PID {pid} not eligible for IID {iid}')
            
        vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
        ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
        self.c_add_vote.perform(vote_token, ciphertext)
        
        self.db.conn.execute('COMMIT')
    except Exception:
        self.db.conn.execute('ROLLBACK')
        raise
```

Additionally, add a database trigger as defense-in-depth:

```sql
-- Requires a way to check election state from the vote table.
-- Since vote doesn't reference election directly, add IID to vote
-- or create a trigger that checks via mayvote → issue → election.
```

#### ASVS-234-HIGH-002: Election `open()` Operation Lacks Transactional Atomicity Enabling Concurrent State Corruption

**Severity:** HIGH  
**Gap Type:** Type B — Transaction control EXISTS (used in `delete()` and `add_salts()`) but NOT APPLIED to `open()` as a whole  
**File:** `v3/steve/election.py`, method `open`, lines 73–84; method `add_salts`, lines 121–139

**Description:**

The `open()` method performs multiple state-changing operations that are not wrapped in a single transaction. First, `add_salts()` executes in its own transaction and commits. Then, after a gap where the election remains in editable state, the main opening operation commits separately. Concurrent `open()` calls can interleave, causing cryptographic material to be overwritten.

**Vulnerable Code:**

```python
# election.py, open (line 73)
def open(self, pdb):
    # Double-check the Election is in the editing state.
    assert self.is_editable()           # CHECK (assert, can be disabled)

    # Add salts first.
    self.add_salts()                    # Has its OWN transaction, commits independently

    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()             # Generates random salt
    opened_key = crypto.gen_opened_key(edata, salt)

    self.c_open.perform(salt, opened_key, self.eid)  # Separate autocommit write
```

```python
# election.py, add_salts (line 121)
def add_salts(self):
    "Set the SALT column in the MAYVOTE table."
    assert self.is_editable()
    
    self.db.conn.execute('BEGIN TRANSACTION')      # Transaction A
    self.q_all_issues.perform(self.eid)
    for mayvote in self.q_all_issues.fetchall():
        salt = crypto.gen_salt()
        self.c_salt_mayvote.perform(salt, mayvote.rowid)
    self.db.conn.execute('COMMIT')                 # Transaction A ends
    # Gap: election is still "editable" here — salt/opened_key not yet set
```

**Data Flow:**

`do_open_endpoint` → `election.open(pdb)` → `assert is_editable()` (no lock) → `add_salts()` (own transaction, commits) → *gap: election still editable* → `c_open.perform()` (separate autocommit)

**Proof of Concept:**

```
# Two administrators (or same admin double-clicking) open election simultaneously

# Request A:                           # Request B:
assert is_editable() → True            assert is_editable() → True
add_salts() → sets salts_A, commits    
                                        add_salts() → overwrites with salts_B, commits
gen_salt() → election_salt_A
gen_opened_key() → key_A               
c_open(salt_A, key_A) → commits        gen_salt() → election_salt_B
                                        gen_opened_key() → key_B
                                        c_open(salt_B, key_B) → overwrites

# Final state: election has salt_B/key_B, mayvote has salts_B
# But any votes cast between Request A completing and Request B 
# overwriting used salt_A/key_A — those votes are now undecryptable
```

**Impact:**

If the race window is exploited, the election's cryptographic material can be overwritten after voters have already begun casting votes. Votes encrypted with the first set of keys become permanently undecryptable, effectively destroying cast ballots. The `is_tampered()` check would also produce unpredictable results.

**Remediation:**

```python
def open(self, pdb):
    """Open the election with full transactional atomicity."""
    
    # Use IMMEDIATE to acquire write lock before checking state
    self.db.conn.execute('BEGIN IMMEDIATE')
    try:
        md = self._all_metadata(self.S_EDITABLE)  # Proper exception, not assert
        
        # Set mayvote salts within same transaction
        self.q_all_issues.perform(self.eid)
        for mayvote in self.q_all_issues.fetchall():
            salt = crypto.gen_salt()
            self.c_salt_mayvote.perform(salt, mayvote.rowid)
        
        edata = self.gather_election_data(pdb)
        salt = crypto.gen_salt()
        opened_key = crypto.gen_opened_key(edata, salt)
        self.c_open.perform(salt, opened_key, self.eid)
        
        self.db.conn.execute('COMMIT')
    except Exception:
        self.db.conn.execute('ROLLBACK')
        raise
```

### Medium Severity

#### ASVS-234-MEDIUM-003: State Machine Enforcement Uses `assert` Statements Disableable at Runtime

**Severity:** MEDIUM  
**Gap Type:** Type C — Control called but result can be globally suppressed  
**File:** `v3/steve/election.py`, multiple methods

**Description:**

All state-changing election operations (except `add_vote`) rely on `assert` statements for state enforcement. Python's `assert` statements are removed when the interpreter runs with optimization flags (`-O` or `-OO`), which is common in production deployments for performance. This completely bypasses the election state machine.

**Vulnerable Code:**

All state-changing election operations rely on `assert` for state enforcement:

```python
# election.py line 52 — delete()
assert self.is_editable()

# election.py line 73 — open()
assert self.is_editable()

# election.py line 116 — close()
assert self.is_open()

# election.py line 127 — add_salts()
assert self.is_editable()

# election.py line 190 — add_issue()
assert self.is_editable()

# election.py line 208 — edit_issue()
assert self.is_editable()

# election.py line 220 — delete_issue()
assert self.is_editable()

# election.py line 248 — add_voter()
assert self.is_editable()
```

**Contrast with `add_vote` (line 261)** which correctly uses exception-based enforcement:

```python
md = self._all_metadata(self.S_OPEN)  # raises ElectionBadState if not open
```

**Data Flow:**

If Python runs with `-O` or `-OO` flags → all `assert` statements are removed at bytecode compilation → `open()`, `close()`, `add_issue()`, `edit_issue()`, `delete_issue()`, `add_voter()`, `delete()` execute without any state validation.

**Impact:**

If the application is deployed with Python optimization enabled (a common production practice for performance), the entire election state machine is bypassed. Issues could be added to or deleted from open elections, elections could be re-opened after closure, and election deletion could occur while voting is active. This would corrupt elections and destroy voting records.

**Remediation:**

Replace all `assert`-based state checks with the existing `_all_metadata(required_state)` pattern that raises `ElectionBadState`:

```python
def close(self):
    "Close an election."
    # Replace: assert self.is_open()
    self._all_metadata(self.S_OPEN)  # raises ElectionBadState if not open
    self.c_close.perform(self.eid)

def add_issue(self, title, description, vtype, kv):
    # Replace: assert self.is_editable()
    self._all_metadata(self.S_EDITABLE)  # raises ElectionBadState
    if vtype not in vtypes.TYPES:
        raise ValueError(f'Invalid vote type: {vtype}')
    # ... rest of method
```

#### ASVS-234-MEDIUM-004: Batch Vote Submission Lacks Transactional Atomicity

**Severity:** MEDIUM  
**Gap Type:** Type A — No transaction control at entry point  
**File:** `v3/server/pages.py`, function `do_vote_endpoint`, lines 403–446; `v3/steve/election.py`, method `add_vote`, lines 258–269

**Description:**

When a voter submits votes for multiple issues in a single request, each vote is processed as an independent transaction in autocommit mode. If the election closes or an error occurs mid-batch, some votes may be recorded while others are lost, with no clear feedback to the voter about which votes succeeded.

**Vulnerable Code:**

```python
# pages.py, do_vote_endpoint (line 430)
for iid, votestring in votes.items():
    if iid not in issue_dict:
        await flash_danger(f'Invalid issue ID: {iid}')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    try:
        election.add_vote(result.uid, iid, votestring)  # Each vote is a separate autocommit
```

Each `add_vote` call executes `c_add_vote.perform()` in SQLite autocommit mode — each vote is an independent transaction:

```python
# election.py, add_vote (line 269)
self.c_add_vote.perform(vote_token, ciphertext)  # Individual autocommit
```

**Data Flow:**

`do_vote_endpoint` → loop over `votes.items()` → `add_vote()` × N (each independent autocommit) → partial failure possible if election closes mid-loop or error occurs

**Proof of Concept:**

```
POST /do-vote/abc1234567 HTTP/1.1
Content-Type: application/x-www-form-urlencoded

vote-issue1=yes&vote-issue2=no&vote-issue3=abstain

# If the election closes after issue1's vote is committed but before issue2:
# - issue1: vote recorded
# - issue2: ElectionBadState raised, vote lost
# - issue3: never processed
# User sees error flash but has no way to know which votes were recorded
```

**Impact:**

In elections with multiple issues, voters may have a partial set of votes recorded without clear feedback about which votes succeeded. This undermines voter intent and creates an inconsistent voting record. The user receives a generic error message and is redirected, with no indication of partial success.

**Remediation:**

```python
async def do_vote_endpoint(election):
    result = await basic_info()
    form = edict(await quart.request.form)
    # ... extract votes ...
    
    # Wrap all votes in a single transaction
    election.db.conn.execute('BEGIN IMMEDIATE')
    try:
        for iid, votestring in votes.items():
            if iid not in issue_dict:
                raise ValueError(f'Invalid issue ID: {iid}')
            election.add_vote_within_transaction(result.uid, iid, votestring)
        election.db.conn.execute('COMMIT')
    except Exception as e:
        election.db.conn.execute('ROLLBACK')
        await flash_danger(f'Error submitting votes: {e}')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
    
    await flash_success('Votes submitted successfully!')
    return quart.redirect('/voter', code=303)
```

## Positive Security Patterns

1. **Per-voter cryptographic salts** (`mayvote.salt`): Each voter-issue pair gets a unique 16-byte salt, preventing vote correlation — `add_salts()` at line 121.

2. **State derivation from data** (`_compute_state`): Election state is computed from actual field values (`salt`, `opened_key`, `closed`) rather than a simple status flag, making state spoofing harder — line 389.

3. **Tamper detection** (`is_tampered`): Computes opened_key from current election data and compares against stored value — line 351.

4. **Non-deducible identifiers**: EIDs and IIDs use 10-character hex random values with CHECK constraints, preventing enumeration — `schema.sql` lines 10, 98.

5. **Re-voting by design**: The vote table intentionally allows duplicate `vote_token` entries with only the latest used (`q_recent_vote`), preventing vote-locking attacks.

6. **Trigger-based protection**: `prevent_open_close_update` trigger prevents modification of advisory dates after election closure — `schema.sql` line 94.

7. **Referential integrity**: `ON DELETE RESTRICT` foreign keys prevent orphaned records — throughout `schema.sql`.

8. **Exception-based state check in `add_vote`**: Unlike other methods, `add_vote` uses `_all_metadata(self.S_OPEN)` which raises `ElectionBadState` — a proper exception that cannot be disabled with `-O`.

## Architecture Observations

1. **No application-level locking primitives**: The codebase has zero instances of mutex, semaphore, `asyncio.Lock`, or database-level advisory locks. The `BEGIN TRANSACTION` used in `delete()` and `add_salts()` provides atomicity for those specific operations but is not applied to the critical `open()`/`close()`/`add_vote()` paths.

2. **Synchronous SQLite in async framework**: SQLite operations are synchronous within Quart's async framework. While this limits intra-process interleaving, multi-worker deployments (standard for production) expose all TOCTOU vulnerabilities through inter-process races.

3. **Authorization placeholder pattern**: Multiple endpoints have `### check authz` comments indicating intended but unimplemented owner authorization checks. This amplifies the state transition vulnerabilities since any authenticated committer can trigger `open()`/`close()` on any election.
