**Key Findings:**
The application **has no SVG sanitization controls whatsoever**. User-supplied content (election titles, issue titles, and issue descriptions) is stored in the database and rendered as raw HTML in EZT templates. This allows authenticated users to inject SVG elements containing scripts, `<foreignObject>`, and event handlers that execute in other users' browsers. This represents a **Type A gap** (entry points with NO control).

## Findings

### HIGH Severity Findings

#### Finding ASVS-134-HIGH-001

**Title:** Stored XSS via SVG Scriptable Content in Issue Descriptions

**Severity:** HIGH  
**Gap Type:** Type A (No control exists)  
**ASVS Requirement:** 1.3.4  
**CWE:** CWE-79 (Cross-site Scripting), CWE-116 (Improper Encoding or Escaping of Output)

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `rewrite_description()` (lines 42–49), `do_add_issue_endpoint()` (lines 399–417), `do_edit_issue_endpoint()` (lines 420–439)
- **Templates:** `v3/server/templates/vote-on.ezt`, `v3/server/templates/manage.ezt`

**Related Functions Checked:**
- `election.add_issue()` — No sanitization before database storage
- `election.edit_issue()` — No sanitization before database update
- `vote_on_page()` — Passes unsanitized data to template
- `manage_page()` — Passes unsanitized data to template

**Description:**

The application accepts user-supplied HTML content in issue descriptions without any sanitization for SVG scriptable elements. The `rewrite_description()` function processes descriptions by wrapping them in `<pre>` tags and converting `doc:` references to HTML links, but performs **no sanitization** of SVG elements, scripts, or event handlers. This content is then rendered as raw HTML in multiple templates, allowing authenticated attackers to inject malicious SVG content that executes in other users' browsers.

**Vulnerable Code:**

`v3/server/pages.py` — description storage (no sanitization before storing):
```python
@APP.post('/do-add-issue/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_add_issue_endpoint(election):
    form = edict(await quart.request.form)
    # ...
    iid = election.add_issue(form.title, form.description, vtype, kv)  # raw user input
```

`v3/server/pages.py` — description processing (adds HTML but no sanitization):
```python
def rewrite_description(issue):
    desc = issue.description
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'  # wraps in <pre>, does NOT sanitize SVG/HTML
```

`v3/server/templates/vote-on.ezt` — raw rendering:
```html
<div id="description-[issues.iid]" class="description mt-2 text-muted small">
  [issues.description]
</div>
```

`v3/server/templates/manage.ezt` — raw rendering:
```html
<div id="description-[issues.iid]" class="description mt-2">[issues.description]</div>
```

**Data Flow:**
```
User form (POST /do-add-issue/<eid>)
  → form.description (no sanitization)
    → election.add_issue() → c_add_issue SQL INSERT
      → Stored in DB issue.description column
        → vote_on_page() → rewrite_description() wraps in <pre> (no SVG strip)
          → vote-on.ezt [issues.description] rendered as raw HTML
            → Browser parses SVG with embedded scripts ⚠️
```

**Proof of Concept:**

An authenticated committer creates an issue with this description:
```http
POST /do-add-issue/abc123 HTTP/1.1
Content-Type: application/x-www-form-urlencoded

title=Budget+Vote&description=<svg+onload="fetch('/do-close/'+document.querySelector('[data-eid]').dataset.eid)"><foreignObject><body+xmlns="http://www.w3.org/1999/xhtml"><script>document.cookie</script></body></foreignObject></svg>&csrf_token=placeholder
```

When any voter views this issue at `/vote-on/abc123`, the SVG executes JavaScript in their browser session.

**Impact:**

- **Confidentiality:** HIGH — Session tokens and authentication cookies can be stolen
- **Integrity:** HIGH — Attackers can submit votes, close elections, or modify data on behalf of victims
- **Availability:** MEDIUM — Malicious scripts could disrupt the voting interface
- **Scope:** All voters who view the compromised election
- **Attack Vector:** Requires authenticated committer access, but affects all users

Stored cross-site scripting affecting all voters who view the election. An attacker with committer access can execute arbitrary JavaScript in other voters' browsers, potentially manipulating election outcomes or hijacking sessions.

**Remediation:**

**Immediate (Required):**

Sanitize all user-supplied HTML content, stripping SVG scriptable elements:

```python
import bleach

# Define safe tags (no SVG scriptable elements)
SAFE_TAGS = ['b', 'i', 'em', 'strong', 'a', 'p', 'br', 'ul', 'ol', 'li', 'pre', 'code']
SAFE_ATTRS = {'a': ['href', 'title']}

def sanitize_html(content):
    """Remove SVG scriptable content and other dangerous HTML."""
    if content is None:
        return None
    return bleach.clean(
        content,
        tags=SAFE_TAGS,
        attributes=SAFE_ATTRS,
        strip=True,
    )

def rewrite_description(issue):
    import re
    desc = sanitize_html(issue.description)  # sanitize FIRST
    def repl(match):
        filename = bleach.clean(match.group(1))  # escape filename too
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

**Alternative (Defense-in-Depth):**

Apply HTML entity encoding in templates:
```html
<div id="description-[issues.iid]" class="description mt-2">
  [format "html"][issues.description][end]
</div>
```

**Testing:**

1. Test with SVG payloads containing `<script>`, `<foreignObject>`, and event handlers
2. Verify legitimate HTML formatting (bold, links) is preserved
3. Test `doc:` reference conversion still functions correctly
4. Verify no double-encoding occurs

#### Finding ASVS-134-HIGH-002

**Title:** Stored XSS via SVG Scriptable Content in Election and Issue Titles

**Severity:** HIGH  
**Gap Type:** Type A (No control exists)  
**ASVS Requirement:** 1.3.4  
**CWE:** CWE-79 (Cross-site Scripting), CWE-116 (Improper Encoding or Escaping of Output)

**Location:**
- **File:** `v3/server/pages.py`
- **Functions:** `do_create_endpoint()` (line 353), `do_add_issue_endpoint()` (lines 399–417)
- **Templates:** `v3/server/templates/voter.ezt`, `v3/server/templates/admin.ezt`, `v3/server/templates/manage.ezt`, `v3/server/templates/vote-on.ezt`

- `admin_page()` — Passes unsanitized titles to template
- `manage_page()` — Passes unsanitized titles to template
- `vote_on_page()` — Passes unsanitized titles to template

**Description:**

Election titles and issue titles are stored without any HTML/SVG sanitization and rendered as raw HTML across multiple high-traffic pages. Since titles appear on dashboard pages viewed by all users, this vulnerability has maximum exposure. Unlike descriptions (which may contain legitimate HTML formatting), titles should contain only plain text, making this a clear sanitization failure.

```python

@asfquart.auth.require({R.pmc_member})
async def do_create_endpoint():
    form = edict(await quart.request.form)
    election = steve.election.Election.create(DB_FNAME, form.title, result.uid)  # raw title
```

`v3/server/templates/voter.ezt` — raw rendering:
```html
<h5 class="card-title mb-3">[open_elections.title]</h5>
```

`v3/server/templates/admin.ezt` — raw rendering:
```html
<h5 class="card-title">[owned.title]...</h5>
```

`v3/server/templates/manage.ezt` — raw rendering:
```html
<h2>[e_title]</h2>
```

`v3/server/templates/vote-on.ezt` — raw rendering:
```html
<h1 class="h4 mb-0 fw-semibold">[election.title]</h1>
```

Issue titles in `v3/server/templates/manage.ezt`:
```html

```

**Data Flow:**
```
User form (POST /do-create-election)
  → form.title (no sanitization)
    → Election.create() → c_create SQL INSERT
      → Stored in DB election.title column
        → voter_page() → postprocess_election()
          → voter.ezt [open_elections.title] rendered as raw HTML
            → Browser parses SVG with scripts ⚠️
```

**Proof of Concept:**

```http
POST /do-create-election HTTP/1.1
Content-Type: application/x-www-form-urlencoded

title=<svg/onload=alert(document.domain)>Board+Election&csrf_token=placeholder
```

Every user who visits `/voter`, `/admin`, or `/manage/<eid>` will have the SVG execute.

**Impact:**

- **Confidentiality:** HIGH — Session tokens exposed on dashboard pages
- **Integrity:** HIGH — Can manipulate election data for all users
- **Availability:** MEDIUM — Can disrupt user interface
- **Scope:** ALL authenticated users (dashboards are high-traffic pages)
- **Attack Vector:** Requires PMC member access for election titles, committer access for issue titles

Stored XSS visible to all authenticated users browsing elections. Election titles appear on high-traffic pages (voter dashboard, admin dashboard), maximizing exposure.

**Remediation:**

**Option A: Sanitize on input (Recommended)**

```python
import html

@APP.post('/do-create-election')
@asfquart.auth.require({R.pmc_member})
async def do_create_endpoint():
    form = edict(await quart.request.form)
    safe_title = html.escape(form.title)  # strip all HTML/SVG
    election = steve.election.Election.create(DB_FNAME, safe_title, result.uid)

@APP.post('/do-add-issue/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_add_issue_endpoint(election):
    form = edict(await quart.request.form)
    safe_title = html.escape(form.title)
    safe_description = sanitize_html(form.description)  # Use bleach for descriptions
    iid = election.add_issue(safe_title, safe_description, vtype, kv)
```

**Option B: Use EZT format directive in ALL templates**

In templates, replace:
```html
[open_elections.title]
```

With:
```html
[format "html"][open_elections.title][end]
```

Apply to all instances in:
- `voter.ezt`
- `admin.ezt`
- `manage.ezt`
- `manage-stv.ezt`
- `vote-on.ezt`

**Testing:**

1. Test with SVG payloads in election titles
2. Test with SVG payloads in issue titles
3. Verify titles display correctly after encoding
4. Test that special characters (quotes, ampersands) are properly handled

### MEDIUM Severity Findings

#### Finding ASVS-134-MEDIUM-003

**Title:** Inconsistent Output Encoding — `[format]` Used in JS Contexts but Missing in HTML Body

**Severity:** MEDIUM  
**Gap Type:** Type B (Control exists for JS contexts but NOT applied to HTML body contexts)  
**ASVS Requirement:** 1.3.4  
**CWE:** CWE-116 (Improper Encoding or Escaping of Output)

**Location:**
- **Files:** Multiple EZT templates
- **Primary:** `v3/server/templates/manage.ezt`, `v3/server/templates/manage-stv.ezt`
- **Secondary:** All other templates rendering user-derived content

**Related Functions Checked:**
- All template rendering functions in `pages.py`
- EZT template engine's `[format]` directive implementation

**Description:**

The application demonstrates awareness of output encoding by using the EZT `[format "js,html"]` directive for JavaScript string contexts in `manage.ezt` and `manage-stv.ezt`. However, the **same values** are rendered without any escaping in HTML body contexts just lines away. This inconsistency creates false confidence that output encoding is properly handled, while leaving the primary XSS vector (HTML body rendering) completely unprotected.

**Vulnerable Code:**

In `v3/server/templates/manage.ezt` and `v3/server/templates/manage-stv.ezt`, EZT `[format]` escaping IS used for JavaScript string contexts:
```html
onclick="openEditIssueModal('[issues.iid]',
                            '[format "js,html"][issues.title][end]',
                            '[format "js,html"][issues.description][end]')"
```

But the SAME values are rendered **without** `[format]` in HTML body contexts, just lines away:
```html
<strong>[issues.title]</strong>
<!-- ... -->
<div id="description-[issues.iid]" class="description mt-2">[issues.description]</div>
```

**Templates Affected:**

| Template | JS-escaped values | Unescaped HTML body values |
|----------|------------------|--------------------------|
| `manage.ezt` | `issues.title`, `issues.description` (in onclick) | `issues.title`, `issues.description`, `e_title` |
| `manage-stv.ezt` | `issues.title`, `issues.description` (in onclick) | `issues.title`, `issues.description`, `e_title` |
| `vote-on.ezt` | N/A | `election.title`, `issues.title`, `issues.description` |
| `voter.ezt` | N/A | All `*.title`, `*.owner_name`, `*.authz` |
| `admin.ezt` | N/A | `owned.title`, `owned.owner_name`, `owned.authz` |
| `e_bad_eid.ezt` | N/A | `[eid]` |
| `e_bad_iid.ezt` | N/A | `[iid]` |
| `e_bad_pid.ezt` | N/A | `[pid]` |
| `flashes.ezt` | N/A | `[flashes.message]` |

**Data Flow:**
```
User-supplied content
  → Stored in database (no sanitization)
    → Passed to template
      → [format "js,html"] applied for onclick attribute (✅ protected)
      → Raw output in HTML body (❌ vulnerable)
        → Browser parses SVG and executes scripts
```

**Analysis:**

The `[format "js,html"]` directive demonstrates that:
1. Developers are aware of context-specific encoding requirements
2. The template engine supports proper escaping
3. Encoding is intentionally applied in some contexts

However, the encoding is **only** applied to prevent JavaScript string injection in `onclick` attributes, while the more critical HTML body context — where SVG scriptable content would actually render and execute — is left completely unescaped. This selective application suggests a misunderstanding of XSS attack vectors.

**Impact:**

- **Confidentiality:** MEDIUM — SVG content will execute in HTML body contexts
- **Integrity:** MEDIUM — Same as HIGH findings, but demonstrates partial security awareness
- **Availability:** LOW — Limited impact beyond XSS execution
- **Scope:** All users viewing affected templates
- **Risk Factor:** Creates false confidence that output encoding is handled properly

The partial application of output encoding gives a false sense of security. SVG content with `<script>`, `<foreignObject>`, or event handler attributes will be rendered and executed in all unescaped contexts.

**Remediation:**

**Immediate (Required):**

Apply `[format "html"]` to ALL user-derived values in HTML body contexts:

```html
<!-- Before: -->
<strong>[issues.title]</strong>
<div class="description">[issues.description]</div>
<h2>[e_title]</h2>

<!-- After: -->
<strong>[format "html"][issues.title][end]</strong>
<div class="description">[format "html"][issues.description][end]</div>
<h2>[format "html"][e_title][end]</h2>
```

**Comprehensive Template Updates:**

Create a checklist for all templates:

**`manage.ezt` and `manage-stv.ezt`:**

**`vote-on.ezt`:**

Consider setting EZT's default format to HTML-escaped if the engine supports it, to prevent future omissions. Alternatively, migrate to a template engine with auto-escaping by default (e.g., Jinja2 with `autoescape=True`).

## Architecture Observations

### 1. EZT Template Engine — No Default Auto-Escaping

**Observation:**
The EZT template engine defaults to raw output, requiring explicit `[format]` directives for escaping. This is contrary to modern template engines like Jinja2, Django Templates, or React/Vue, which escape by default and require explicit opt-out for raw HTML.

**Security Implication:**
- Makes escape omissions highly likely as the application grows
- Places security burden on developers to remember encoding in every context
- No compiler/linter warnings for missing escapes

**Comparison:**

| Template Engine | Default Behavior | Raw Output Syntax |
|----------------|------------------|-------------------|
| EZT (current) | Raw output | `[variable]` |
| Jinja2 | Auto-escape | `{{ variable }}` (escaped), `{{ variable\|safe }}` (raw) |
| Django | Auto-escape | `{{ variable }}` (escaped), `{{ variable\|safe }}` (raw) |
| React JSX | Auto-escape | `{variable}` (escaped), `dangerouslySetInnerHTML` (raw) |

**Recommendation:** Consider migrating to Jinja2 with `autoescape=True` or implement a custom EZT wrapper that escapes by default.

### 2. No Centralized Input Sanitization Layer

**Observation:**
Content flows directly from `quart.request.form` to database storage without passing through any sanitization layer:

```python
form = edict(await quart.request.form)
election.add_issue(form.title, form.description, vtype, kv)  # Direct storage
```

**Security Implication:**
- No single point of enforcement for HTML/SVG sanitization
- Each endpoint must remember to sanitize (prone to omissions)
- Difficult to audit or update sanitization rules globally

**Best Practice Comparison:**

```python
# Current (vulnerable):

# Recommended (centralized sanitization):
from app.security import sanitize_html, sanitize_text

safe_description = sanitize_html(form.description)  # Allow safe HTML subset
election.add_issue(safe_title, safe_description, vtype, kv)
```

**Recommendation:** Implement a `security.py` module with centralized sanitization functions used by all input endpoints.

### 3. `rewrite_description()` Generates HTML from User Input

**Observation:**
The `rewrite_description()` function intentionally generates HTML markup from user input:

```python
def rewrite_description(issue):
    desc = issue.description
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

**Security Implication:**
- Output encoding alone is insufficient (would escape the intended `<a>` and `<pre>` tags)
- **Must** use input sanitization to allow safe HTML subset while blocking SVG/scripts
- Creates a "rich content" context that requires careful handling

**Analysis:**
This is a legitimate use case for allowing some HTML in user content (links, formatting). However, it requires a whitelist-based approach:

1. Sanitize input to allow only safe tags (`<a>`, `<pre>`, `<code>`, etc.)
2. Strip all SVG-related tags and event handlers
3. Validate attributes (e.g., `href` should not contain `javascript:` URLs)

**Recommendation:** Use `bleach` or `nh3` library with strict tag/attribute whitelists.

### 4. No Content Security Policy (CSP) Observed

**Observation:**
No CSP headers were observed in the codebase review. CSP is a critical defense-in-depth measure against XSS.

**Security Implication:**
Even if sanitization is bypassed, a properly configured CSP would prevent inline script execution and restrict script sources.

**Recommended CSP Header:**
```
Content-Security-Policy: 
    default-src 'self'; 
    script-src 'self'; 
    style-src 'self' 'unsafe-inline'; 
    img-src 'self' data:; 
    object-src 'none'; 
    base-uri 'self'; 
    form-action 'self';
```

**Implementation:**
```python
@APP.after_request
async def add_security_headers(response):
    response.headers['Content-Security-Policy'] = (
        "default-src 'self'; "
        "script-src 'self'; "
        "style-src 'self' 'unsafe-inline'; "
        "object-src 'none';"
    )
    return response
```

**Recommendation:** Implement CSP as defense-in-depth, even after fixing sanitization issues.

### 5. Authentication Reduces but Doesn't Eliminate Risk

**Observation:**
Only authenticated users with specific roles can create content:
- PMC members can create elections
- Committers can create/edit issues

**Security Implication:**
- **Positive:** Prevents anonymous attackers from injecting malicious content
- **Limitation:** Does not prevent:
  - Malicious insiders
  - Compromised accounts
  - Social engineering attacks
  - Privilege escalation exploits

**Risk Assessment:**
While authentication significantly reduces the attack surface, it should not be considered a primary XSS defense. Trusted users can still:
- Be compromised (phishing, credential theft)
- Act maliciously (disgruntled employees, attackers with stolen credentials)
- Make mistakes (accidentally pasting malicious content)

**Recommendation:** Implement proper input sanitization regardless of authentication requirements. Defense-in-depth is essential.

### 6. Database Design — No Separation of Raw and Rendered Content

**Observation:**
The database appears to store user content in a single field that is both:
1. Modified by `rewrite_description()` (adds HTML)
2. Rendered directly in templates

**Security Implication:**
- No clear separation between "user-supplied raw content" and "application-rendered HTML"
- Difficult to re-sanitize content if security policies change
- Cannot easily audit what is user-supplied vs. application-generated

**Best Practice:**
```sql
-- Recommended schema:
CREATE TABLE issue (
    iid TEXT PRIMARY KEY,
    title TEXT,
    description_raw TEXT,  -- User input, never rendered directly
    description_html TEXT,  -- Sanitized and processed, safe to render
    -- ...
);
```

**Recommendation:** Consider schema changes to separate raw user input from rendered HTML, enabling easier security updates.

## Implementation Guidance

### Sanitization Library Selection

**Recommended: `bleach` (Python)**

```python
# Install:
pip install bleach

# Usage:
import bleach

ALLOWED_TAGS = ['b', 'i', 'em', 'strong', 'a', 'p', 'br', 'ul', 'ol', 'li', 'pre', 'code']
ALLOWED_ATTRS = {
    'a': ['href', 'title'],
}

def sanitize_html(content):
    """Sanitize HTML content, removing SVG and scripts."""
    if not content:
        return content
    return bleach.clean(
        content,
        tags=ALLOWED_TAGS,
        attributes=ALLOWED_ATTRS,
        strip=True,
    )

def sanitize_text(content):
    """Remove all HTML tags."""
    if not content:
        return content
    return bleach.clean(content, tags=[], strip=True)
```

**Alternative: `nh3` (Rust-based, faster)**

```python
# Install:
pip install nh3

# Usage:
import nh3

def sanitize_html(content):
    """Sanitize HTML content using nh3."""
    if not content:
        return content
    return nh3.clean(
        content,
        tags={'b', 'i', 'em', 'strong', 'a', 'p', 'br', 'ul', 'ol', 'li', 'pre', 'code'},
        attributes={'a': {'href', 'title'}},
    )
```

## Audit Conclusion

**Compliance Status:** ❌ **NON-COMPLIANT** with ASVS 1.3.4

**Summary:**
The apache/steve application fails ASVS requirement 1.3.4 due to complete absence of SVG scriptable content sanitization. User-supplied content containing SVG elements with scripts, `<foreignObject>`, and event handlers is stored without sanitization and rendered as raw HTML, enabling stored cross-site scripting attacks.

**Risk Level:** **HIGH**

**Key Gaps:**
1. No input sanitization for HTML/SVG content
2. Inconsistent output encoding (partial in JS contexts, absent in HTML body)
3. No Content Security Policy for defense-in-depth
4. Template engine defaults to raw output without auto-escaping

**Required Actions:**
Implement comprehensive input sanitization using `bleach` or `nh3`, apply output encoding to all user-derived template variables, and deploy Content Security Policy headers. See Recommendations Summary for detailed implementation plan.
