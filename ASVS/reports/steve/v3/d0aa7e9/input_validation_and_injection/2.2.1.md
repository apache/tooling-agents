## Findings

### Critical Severity

#### ASVS-221-CRI-001: Vote Content (votestring) Not Validated Against Issue Vote Type

**Severity:** CRITICAL  
**Gap Type:** Type B — Control EXISTS (documented intent) but NOT CALLED  
**File:** `v3/steve/election.py`  
**Function:** `add_vote()`  
**Line:** ~229  
**Also affects:** `v3/server/pages.py`, function `do_vote_endpoint()`, lines ~410-447

**Related Functions Checked:**
- `election.py::add_vote()` — Missing validation implementation
- `pages.py::do_vote_endpoint()` — No validation before calling `add_vote()`
- `vtypes/*` modules — No `validate()` interface defined

##### Description

The voting system accepts and stores arbitrary vote content without validating it against the issue's vote type. While a comment in the code explicitly marks this validation as needed (`### validate VOTESTRING for ISSUE.TYPE voting`), the validation is never implemented. This allows attackers to submit invalid vote values that are encrypted and permanently stored, corrupting the election's vote record.

##### Vulnerable Code

**election.py — `add_vote()`:**
```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting    # <── CONTROL DOCUMENTED BUT NOT IMPLEMENTED

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)

    # Pass MAYVOTE.SALT for PBKDF.
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)

    self.c_add_vote.perform(vote_token, ciphertext)
```

**pages.py — `do_vote_endpoint()`:**
```python
votes = {}
for key, value in form.items():
    if key.startswith('vote-'):
        iid = key.split('-', 1)[1]
        votes[iid] = value            # <── arbitrary string accepted

for iid, votestring in votes.items():
    if iid not in issue_dict:          # <── IID validated
        ...
    try:
        election.add_vote(result.uid, iid, votestring)  # <── votestring NOT validated
```

##### Data Flow

```
Source: HTTP POST form field value (e.g., vote-abc123=<attacker_value>)
  ↓
pages.py do_vote_endpoint(): extracted from form, IID validated, VALUE NOT validated
  ↓
election.py add_vote(): comment marks validation as needed, NOT implemented
  ↓
Sink: crypto.create_vote() encrypts and stores arbitrary string as official vote
  ↓
tally_issue() decrypts and passes to vtypes tally module with corrupted data
```

##### Proof of Concept

For a **YNA issue** (valid values: `y`, `n`, `a`):
```http
POST /do-vote/a1b2c3d4e5 HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: [valid_session]

vote-f6g7h8i9j0=INVALID_STRING&vote-k1l2m3n4o5=ynaynayna
```

For an **STV issue** (valid values: comma-separated candidate labels from the issue's `labelmap`):
```http
POST /do-vote/a1b2c3d4e5 HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: [valid_session]

vote-p6q7r8s9t0=nonexistent_candidate1,nonexistent_candidate2,duplicate,duplicate
```

Both requests will be encrypted and stored as valid votes. At tally time, the `vtypes` module will receive these corrupted votestrings, potentially causing:
- Incorrect tally results (invalid values counted or causing exceptions)
- Denial of service during tallying (unexpected input to counting algorithms)
- Duplicate candidate rankings in STV breaking the single-transferable-vote invariant

##### Impact

This is the **core business-critical input** of a voting system. Without server-side validation:

1. **Election integrity**: Arbitrary vote values are encrypted and stored, corrupting the permanent vote record
2. **Tally corruption**: The `vtypes` tally module receives invalid data — STV algorithms specifically depend on valid candidate labels and unique rankings to function correctly
3. **Silent corruption**: Because votes are encrypted, invalid values cannot be detected without decrypting all votes during tallying — there is no early warning
4. **No recovery**: Once encrypted with per-voter tokens, invalid votes are indistinguishable from valid ones until tally time

The client-side form constrains YNA to radio buttons (`y`/`n`/`a`) and STV to drag-and-drop from known candidates, but these controls are trivially bypassed via direct HTTP requests.

##### Remediation

**In `election.py`, implement validation before encryption:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    # Validate votestring against the issue's vote type
    issue = self.q_get_issue.first_row(iid)
    if not issue:
        raise IssueNotFound(iid)

    vtype_mod = vtypes.vtype_module(issue.type)
    kv = self.json2kv(issue.kv)
    if not vtype_mod.validate(votestring, kv):
        raise InvalidVoteString(iid, issue.type, votestring)

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    # ... rest of method unchanged
```

**Add validation functions to each vtype module (e.g., `vtypes/yna.py`):**

```python
VALID_YNA = {'y', 'n', 'a'}

def validate(votestring: str, kv: dict) -> bool:
    """Validate that votestring is a valid YNA vote."""
    return votestring in VALID_YNA
```

**For STV (`vtypes/stv.py`):**

```python
def validate(votestring: str, kv: dict) -> bool:
    """Validate that votestring contains valid, unique candidate labels."""
    if not votestring:
        return False
    labels = votestring.split(',')
    valid_labels = set(kv.get('labelmap', {}).keys())
    # Each label must be valid and appear at most once
    return (all(label in valid_labels for label in labels)
            and len(labels) == len(set(labels)))
```

**Additionally, add validation in the web handler (`pages.py`) for defense-in-depth:**

```python
for iid, votestring in votes.items():
    if iid not in issue_dict:
        await flash_danger(f'Invalid issue ID: {iid}')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    # Validate vote content against issue type
    issue = issue_dict[iid]
    if issue.vtype == 'yna' and votestring not in ('y', 'n', 'a'):
        await flash_danger(f'Invalid vote value for issue {issue.title}.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
    elif issue.vtype == 'stv':
        labels = set(votestring.split(',')) if votestring else set()
        valid_labels = set(issue.kv.get('labelmap', {}).keys())
        if not labels or not labels.issubset(valid_labels):
            await flash_danger(f'Invalid candidate ranking for {issue.title}.')
            return quart.redirect(f'/vote-on/{election.eid}', code=303)
```

## Positive Security Patterns

1. **Date validation in `_set_election_date()`**: Properly validates date format using `fromisoformat()` with `try/except` and null check — well-implemented input validation pattern.

2. **Election/Issue ID existence validation**: The `load_election` and `load_election_issue` decorators consistently validate that database objects exist before proceeding, returning proper 404 responses.

3. **Vote type assertion on issue creation**: `assert vtype in vtypes.TYPES` in `add_issue()` enforces the allowlist of valid vote types.

4. **State-machine assertions**: Election state transitions are guarded by assertions (`is_editable()`, `is_open()`, `is_closed()`) preventing operations in invalid states.

5. **EZT template escaping**: The templates use `[format "js,html"]` for values embedded in JavaScript contexts (e.g., `onclick` handlers in `manage.ezt`), showing awareness of context-specific encoding.

6. **`send_from_directory` usage**: File serving uses Quart's `send_from_directory` which internally employs `safe_join` for path traversal protection.

7. **Issue ID validation in vote handler**: The `do_vote_endpoint` validates submitted issue IDs against the election's actual issue list before processing.

---

## Architecture Observations

1. **Validation responsibility is split inconsistently**: Date validation happens at the web handler level (`pages.py`), vote type validation happens at the model level (`election.py` via assertion), but vote content validation happens at **neither** level.

2. **The `vtypes` module infrastructure exists** (referenced via `vtypes.TYPES` and `vtypes.vtype_module()`), providing the natural location for vote content validation functions, but no `validate()` interface is defined or called.

3. **Assert statements for state validation**: Using Python `assert` for business logic validation is fragile — running with `-O` flag disables all assertions. These should be converted to explicit conditional checks with proper exceptions.

---

## Recommendations Summary

### Immediate Actions (Critical Priority)

1. **Implement votestring validation** in `election.add_vote()` and the `do_vote_endpoint` handler — this is the highest priority as it affects the core business function of the voting system. Add a `validate()` method to each vtypes module (YNA: allowlist of `{y,n,a}`; STV: valid labels from labelmap, uniqueness check) and call it before encryption.

### Short-term Actions (Recommended)

2. **Convert assert-based state checks to explicit validation** — Replace `assert self.is_editable()` patterns with `if not self.is_editable(): raise ElectionBadState(...)` to ensure protection cannot be disabled by Python optimization flags.

3. **Add length and content validation for text fields** — Election titles, issue titles, and descriptions should have maximum length limits and basic content validation to prevent storage abuse and ensure data quality.

### Long-term Actions

4. **Establish consistent validation architecture** — Define clear guidelines for where validation should occur (web layer vs. business logic layer) and implement defense-in-depth with validation at both layers for critical inputs.

5. **Add comprehensive input validation testing** — Create automated tests that verify all input validation controls function correctly and cannot be bypassed.
