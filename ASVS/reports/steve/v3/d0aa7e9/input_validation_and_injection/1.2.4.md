## Positive Security Patterns

The following security-positive patterns were observed during the audit:

### 1. **Consistent Parameterization Architecture**

The `asfpy.db` wrapper enforces parameterized queries by design. Query SQL is defined in YAML, parameters are bound at runtime via `.perform()` — this separates code from data structurally, making SQL injection architecturally impossible.

**Location:** `election.py:36`, `persondb.py:24`  
**Pattern:** All 39 database interactions use the same safe interface

### 2. **Proxy Pattern Prevents Ad-Hoc Queries**

`Election.__getattr__` and `PersonDB.__getattr__` proxy only to pre-defined cursors from the YAML, making it impossible to accidentally construct queries from user input.

**Location:** `election.py:47`, `persondb.py:29`  
**Security benefit:** No code path exists to bypass parameterization

### 3. **Vote Data Encrypted Before Storage**

In `add_vote()`, `votestring` is encrypted via `crypto.create_vote()` before being passed to `c_add_vote.perform()`, adding defense-in-depth even though parameterization already prevents injection.

**Location:** `election.py:270-279`  
**Security benefit:** Multiple layers of protection for sensitive vote data

### 4. **Type Validation Before DB Writes**

`add_issue()` and `edit_issue()` validate `vtype in vtypes.TYPES` before database operations.

**Location:** `election.py:201`, `election.py:219`  
**Security benefit:** Application-level validation prevents invalid data from reaching the database

### 5. **Transaction Boundaries for Multi-Step Operations**

`delete()` and `add_salts()` properly use `BEGIN TRANSACTION` / `COMMIT` for referential integrity.

**Location:** `election.py:58-70`, `election.py:140-150`  
**Security benefit:** Prevents partial state corruption

### 6. **ID Generation from Cryptographic Source**

`crypto.create_id()` generates IDs that are then used in parameterized queries, preventing any injection through ID values.

**Location:** Throughout `election.py`  
**Security benefit:** High-entropy, unpredictable identifiers

## Architecture Observations

### 1. **Clean Separation of Concerns**

Web handlers (`pages.py`) never construct SQL — all database logic lives in `election.py` and `persondb.py`, which consistently use the parameterized wrapper.

**Security impact:** Reduces attack surface by centralizing database access patterns

### 2. **Query Definition Externalization**

SQL lives in `queries.yaml`, not in Python code. This makes query review and parameterization verification straightforward.

**Security impact:** Easier security audits, clear separation between code and queries

### 3. **Single Database Access Pattern**

The entire codebase uses exactly one mechanism for database access (`asfpy.db.DB` with `.perform()` / `.first_row()`). There are no alternative paths that bypass parameterization.

**Security impact:** No "escape hatches" that could introduce vulnerabilities

### 4. **SQLite Strict Mode**

Per domain context, the database uses `strict mode` which provides additional type safety at the database level.

**Security impact:** Additional defense-in-depth through database-level type enforcement

## Recommendations Summary

While no CRITICAL injection vulnerabilities were found, the following defense-in-depth improvements are noted for completeness:

### Immediate Actions (0-30 days)

*None required* — No security vulnerabilities identified

### Short-term Improvements (1-3 months)

1. **Add input validation for `votestring`**  
   **Location:** `election.py:270` (comment: `### validate VOTESTRING for ISSUE.TYPE voting`)  
   **Rationale:** While parameterization prevents SQL injection, validating vote content against expected patterns (e.g., `y`/`n`/`a` for YNA, comma-separated labels for STV) would catch application logic errors early  
   **Priority:** Low (defense-in-depth)

2. **Validate `docname` in `serve_doc()`**  
   **Location:** `pages.py:545` (comment: `### verify the propriety of DOCNAME`)  
   **Rationale:** While `send_from_directory` provides path traversal protection, explicit validation of document names against expected patterns would add defense-in-depth  
   **Priority:** Low (defense-in-depth)

### Long-term Enhancements (3-12 months)

3. **Document the parameterization guarantee**  
   **Location:** Developer documentation or code comments in `election.py`, `persondb.py`  
   **Rationale:** Consider adding documentation confirming that `asfpy.db.DB.perform()` uses SQLite parameter binding (`?` placeholders), making the security property explicit for future maintainers  
   **Priority:** Low (documentation)
