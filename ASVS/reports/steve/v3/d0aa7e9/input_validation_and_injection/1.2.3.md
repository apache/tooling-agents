- **ASVS Requirement:** 1.2.3 — Verify that output encoding or escaping is used when dynamically building JavaScript content (including JSON), to avoid changing the message or document structure (to avoid JavaScript and JSON injection).
  - **Level:** 1
  - **Section:** Injection Prevention
  - **Section Description:** Output encoding or escaping, performed close to or adjacent to a potentially dangerous context, is critical to the security of any application. Attempting to perform this too early may result in malformed content or render the encoding or escaping ineffective.

## Findings

### Critical Severity

#### ASVS-123-CRI-001: Missing JavaScript Output Encoding in `vote-on.ezt` STV_CANDIDATES Object

**Severity:** CRITICAL  
**Gap Type:** Type B — Control EXISTS (`[format "js,html"]`) but NOT CALLED at this entry point  
**File:** `v3/server/templates/vote-on.ezt`, lines within the `<script>` block  
**Function:** Template rendering triggered by `vote_on_page()` in `v3/server/pages.py` (~line 158)

##### Vulnerable Code

```javascript
// vote-on.ezt — inside <script> block
const STV_CANDIDATES = {
[for issues][is issues.vtype "stv"]    "[issues.iid]": {
      seats: [issues.seats],
      title: "[issues.title]",                              // ← NO ENCODING
      candidates: [
[for issues.candidates]        { label: "[issues.candidates.label]", name: "[issues.candidates.name]" },  // ← NO ENCODING
[end]      ]
    },
[end][end]  };
```

##### Evidence the Control Exists But Is Not Applied

In `manage.ezt` and `manage-stv.ezt`, the **same data fields** are correctly encoded:

```html
<!-- manage.ezt — CORRECT usage -->
onclick="openEditIssueModal('[issues.iid]',
                            '[format "js,html"][issues.title][end]',
                            '[format "js,html"][issues.description][end]')"
```

The `vote-on.ezt` template renders the identical `[issues.title]` into a JavaScript string context but **omits** the `[format "js,html"]` directive.

##### Data Flow

```
Source:  POST /do-add-issue/<eid>  →  form.title (user input from admin)
         ↓
Store:   election.add_issue(form.title, ...)  →  database INSERT
         ↓
Load:    vote_on_page()  →  election.list_issues()  →  result.issues
         ↓
Sink:    EZT renders [issues.title] directly inside JavaScript string literal
         ↓
Missing: No [format "js,html"] encoding applied
```

##### Proof of Concept

**Step 1:** Authenticated committer creates an STV election with an issue titled:
```
"+alert(document.cookie)+"
```

This is done via `POST /do-add-issue/<eid>` with form parameter `title="+alert(document.cookie)+"`.

**Step 2:** Any voter visits `/vote-on/<eid>`. The rendered JavaScript becomes:

```javascript
const STV_CANDIDATES = {
    "a1b2c3d4e5": {
      seats: 1,
      title: ""+alert(document.cookie)+"",
      candidates: [
        { label: "A", name: "Candidate One" },
      ]
    },
};
```

The JavaScript engine evaluates `"" + alert(document.cookie) + ""` as a valid expression. `alert(document.cookie)` executes in the voter's browser session.

**Step 3:** A real attacker would replace `alert()` with cookie exfiltration, session hijacking, or vote manipulation (e.g., programmatically setting hidden form values before submission).

##### Impact

- **Stored XSS** affecting all voters who view the ballot page for the targeted election
- Attacker (malicious election admin) can execute arbitrary JavaScript in voters' authenticated sessions
- Can silently manipulate vote form data, steal session tokens, or redirect voters
- Undermines the integrity of the voting process — the core security objective of the application

##### Remediation

Apply the existing `[format "js"]` encoding to all server-side values injected into JavaScript contexts:

```javascript
// vote-on.ezt — FIXED
const STV_CANDIDATES = {
[for issues][is issues.vtype "stv"]    "[issues.iid]": {
      seats: parseInt("[format "js"][issues.seats][end]", 10),
      title: "[format "js"][issues.title][end]",
      candidates: [
[for issues.candidates]        { label: "[format "js"][issues.candidates.label][end]", name: "[format "js"][issues.candidates.name][end]" },
[end]      ]
    },
[end][end]  };
```

**Alternatively**, use a safer architecture pattern — serialize the data as JSON from the Python side and embed it as a data attribute on a DOM element, avoiding inline JavaScript entirely:

```python
# pages.py — in vote_on_page()
import json
result.stv_candidates_json = json.dumps({
    issue.iid: {
        'seats': issue.seats,
        'title': issue.title,
        'candidates': issue.candidates,
    }
    for issue in issues_stv
})
```

```html
<!-- vote-on.ezt -->
<div id="stv-data" data-candidates="[stv_candidates_json]"></div>
<script>
  const STV_CANDIDATES = JSON.parse(
    document.getElementById('stv-data').dataset.candidates
  );
</script>
```

This approach uses `json.dumps()` (which escapes special characters correctly) and `JSON.parse()` (which only produces data, not executable code), eliminating the injection class entirely.

## Architecture Observations

1. **EZT templating does not auto-escape** — Unlike Jinja2 with `autoescape=True`, EZT requires explicit `[format ...]` directives for each variable. This creates a high risk of omission, especially as new templates are added.

2. **Inline JavaScript with server-side interpolation** — The pattern of embedding server data directly in `<script>` blocks is inherently fragile. Every new interpolation point must be manually identified and encoded. A data-attribute + `JSON.parse()` pattern (as shown in remediation) would be structurally safer.

3. **The control exists and developers know to use it** — The presence of `[format "js,html"]` in `manage.ezt` shows awareness of the requirement. The omission in `vote-on.ezt` appears to be an oversight during implementation of the voting UI, which was likely developed separately from the admin UI.

---

## Recommendations Summary

### Immediate Actions (Critical Priority)

1. **Apply `[format "js"]` encoding to all server-side values in the `STV_CANDIDATES` JavaScript object** in `vote-on.ezt`, specifically `[issues.title]`, `[issues.seats]`, `[issues.candidates.label]`, and `[issues.candidates.name]`. This addresses ASVS-123-CRI-001.

### Short-term Actions (1-3 months)

2. **Migrate all inline JavaScript data embedding to the `data-attribute + JSON.parse()` pattern**. This eliminates the entire class of JavaScript injection through template interpolation and provides a more maintainable architecture.

### Long-term Actions (3-6 months)

3. **Establish development guidelines** that **every** EZT variable appearing inside a `<script>` block or HTML event attribute **must** use the appropriate `[format]` directive. Add template review checklists specifically for JavaScript context encoding to prevent future regressions.

4. **Consider migrating to a templating engine with auto-escaping capabilities** (e.g., Jinja2) to reduce the burden of manual encoding and minimize the risk of developer oversight.
