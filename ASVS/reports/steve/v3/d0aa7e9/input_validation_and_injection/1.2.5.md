## Positive Security Patterns

### 1. ✅ Zero OS Command Execution Surface

**Location:** Entire codebase  
**Pattern:** No imports or usage of OS command execution primitives

The application does not import or use any of the following:
- `os.system()`
- `os.popen()`
- `subprocess` module
- `commands` module
- `exec()` / `eval()` for system commands
- `os.execv*()` family

**Security Impact:** Eliminates entire class of OS command injection vulnerabilities at the architectural level.

### 2. ✅ Framework-Level File Serving

**Location:** `v3/server/pages.py` lines 572, 575  
**Pattern:** `quart.send_from_directory()`

```python
# Line 572
return await quart.send_from_directory(DOCSDIR / iid, docname)

# Line 575
return await quart.send_from_directory(STATICDIR, filename)
```

**Security Impact:** Uses Quart's built-in file serving with directory confinement. This is an in-process file-read operation with no shell involvement, preventing command injection via path manipulation.

### 3. ✅ Exclusive Use of Safe Data Sinks

**Location:** All request handlers  
**Pattern:** User input flows only to safe operations

All user-controllable data flows exclusively to:
- **Parameterized SQL queries** via `asfpy.db.perform()` calls
- **Cryptographic functions** (`crypto.create_vote()`, etc.)
- **Template rendering** (EZT, in-process)
- **Type parsing** (`datetime.fromisoformat()`)

**Security Impact:** No user data reaches shell or command execution contexts.

### 4. ✅ Safe Path Construction

**Location:** Throughout codebase  
**Pattern:** `pathlib.Path` operators

```python
# Example from pages.py
DOCSDIR / iid / docname
STATICDIR / filename
```

**Security Impact:** Uses `pathlib.Path` operators instead of string concatenation, providing cleaner path handling that avoids patterns that could be misused if later adapted to shell commands.

### 5. ✅ In-Process Architecture

**Location:** Entire application architecture  
**Pattern:** Pure Python implementation

All operations are handled by in-process Python libraries:
- Database access via `asfpy.db`
- Cryptography via `crypto` module
- Template rendering via EZT
- File serving via Quart framework
- Tally operations via direct Python functions

**Security Impact:** No external process invocations required, eliminating command injection vectors at architectural level.

## Architecture Observations

### Pure Python Web Application Design

The application follows a pure Python architecture with no dependencies on OS-level command execution:

1. **Database Layer:** All database operations use `asfpy.db` with parameterized queries
2. **Cryptography:** Vote encryption handled by in-process Python crypto libraries
3. **Template Rendering:** EZT template engine operates entirely in-process
4. **File Operations:** Framework-level file serving via Quart's built-in functions
5. **Business Logic:** Tally tool (`tally_issue`) uses direct database access

### No External System Integrations via Shell

The application has no integrations that commonly introduce command injection risks:
- ❌ No email sending via `sendmail` command
- ❌ No PDF generation via external tools
- ❌ No image processing via ImageMagick/GraphicsMagick
- ❌ No file compression via `tar`/`zip` commands
- ❌ No version control operations via `git` commands

### Security-by-Design Assessment

The architectural choices demonstrate security-by-design principles:
- **Minimal attack surface:** Only necessary functionality exposed
- **Library-first approach:** Leverages well-tested Python libraries
- **Framework security features:** Uses Quart's built-in security mechanisms
- **Type safety:** Structured data handling reduces injection risks
