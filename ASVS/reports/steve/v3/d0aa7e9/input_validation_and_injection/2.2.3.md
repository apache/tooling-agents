### Key Findings Overview

The audit identified **3 significant gaps** in data combination validation:

1. **Vote format validation is explicitly acknowledged but not implemented** — arbitrary vote strings are accepted and encrypted, potentially corrupting election tallies
2. **Election date relationships are not validated** — close dates can precede open dates, creating logically impossible configurations
3. **Elections can be opened without issues or voters** — allowing irreversible state transitions that render elections permanently unusable

### HIGH SEVERITY

#### ASVS-223-HIGH-001: Missing Vote Format Validation Against Issue Type

**Severity:** HIGH  
**Gap Type:** Type B — Control explicitly acknowledged as needed but not implemented  
**File:** `v3/steve/election.py`, function `add_vote()`, line ~295  
**Also affects:** `v3/server/pages.py`, function `do_vote_endpoint()`, lines ~479–498

**Vulnerable Code:**

```python
# v3/steve/election.py — add_vote()
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)

    ### validate VOTESTRING for ISSUE.TYPE voting    # <── TODO: NOT IMPLEMENTED

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)

    # Pass MAYVOTE.SALT for PBKDF.
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)

    self.c_add_vote.perform(vote_token, ciphertext)
```

```python
# v3/server/pages.py — do_vote_endpoint()
for iid, votestring in votes.items():
    if iid not in issue_dict:
        await flash_danger(f'Invalid issue ID: {iid}')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)

    # No validation that votestring matches issue_dict[iid].vtype
    try:
        election.add_vote(result.uid, iid, votestring)
```

**Data Flow:**

```
Form POST (vote-<iid>=<arbitrary_string>)
  → pages.py do_vote_endpoint(): extracts iid + votestring from form
  → Validates iid exists in election (✓)
  → Does NOT validate votestring format against issue_dict[iid].vtype (✗)
  → election.add_vote(): encrypts and stores arbitrary votestring
  → tally_issue(): decrypts and passes invalid data to vtype module tally()
```

**Proof of Concept:**

```http
POST /do-vote/abc123 HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Cookie: <valid_session>

vote-yna_issue_id=GARBAGE_VALUE&vote-stv_issue_id=fake_label1,fake_label1,fake_label1
```

For a **YNA** issue: any string is accepted (should only be `y`, `n`, or `a`).  
For an **STV** issue: non-existent candidate labels, duplicate labels, and arbitrarily long rankings are all accepted.

**Impact:**
- Invalid votes are encrypted and stored, then decrypted during tallying
- YNA tally may count invalid votes or crash when encountering unexpected values
- STV tally receives non-existent candidate labels and duplicate entries, potentially corrupting ranked-choice results or causing the tally module to fail
- For a voting system, data integrity of recorded votes is the core function — accepting garbage vote data undermines the entire system's purpose

**Remediation:**

```python
# v3/steve/election.py

# Add constants or import from vtypes
YNA_VALID = {'y', 'n', 'a'}

def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    md = self._all_metadata(self.S_OPEN)

    # Fetch issue to determine vote type
    issue = self.q_get_issue.first_row(iid)
    if not issue:
        raise IssueNotFound(iid)

    # Validate votestring against issue type
    self._validate_votestring(issue.type, votestring, self.json2kv(issue.kv))

    mayvote = self.q_get_mayvote.first_row(pid, iid)
    if not mayvote:
        raise ValueError(f'Voter {pid} not eligible for issue {iid}')
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
    self.c_add_vote.perform(vote_token, ciphertext)

@staticmethod
def _validate_votestring(vtype, votestring, kv):
    """Validate that votestring is reasonable for the given vote type."""
    if vtype == 'yna':
        if votestring not in ('y', 'n', 'a'):
            raise ValueError(f'Invalid YNA vote: {votestring!r}')
    elif vtype == 'stv':
        if not kv or 'labelmap' not in kv:
            raise ValueError('STV issue missing labelmap')
        valid_labels = set(kv['labelmap'].keys())
        ranked = votestring.split(',') if votestring else []
        # Check for invalid labels
        for label in ranked:
            if label not in valid_labels:
                raise ValueError(f'Invalid candidate label: {label!r}')
        # Check for duplicate labels
        if len(ranked) != len(set(ranked)):
            raise ValueError('Duplicate candidate labels in ranking')
    else:
        raise ValueError(f'Unknown vote type: {vtype}')
```

### MEDIUM SEVERITY

#### ASVS-223-MED-002: Missing Cross-Validation of Election Open/Close Dates

**Severity:** MEDIUM  
**Gap Type:** Type A — No control exists  
**File:** `v3/server/pages.py`, function `_set_election_date()`, lines ~77–101  
**Also affects:** `v3/steve/election.py` `Election.create()`, `v3/server/bin/create-election.py` `main()`

**Vulnerable Code:**

```python
# v3/server/pages.py — _set_election_date()
async def _set_election_date(election, field):
    result = await basic_info()
    data = await quart.request.get_json()
    date_str = data.get('date')
    if not date_str:
        quart.abort(400, 'Missing date')

    # Validate date (basic check)
    try:
        dt = datetime.datetime.fromisoformat(date_str).date()
    except ValueError:
        quart.abort(400, 'Invalid date format')

    # No cross-validation: close_at > open_at? dates in the future?
    if field == 'open_at':
        election.set_open_at(dt)
    elif field == 'close_at':
        election.set_close_at(dt)
```

```python
# v3/steve/election.py — set_open_at / set_close_at
def set_open_at(self, timestamp):
    "Set the open_at timestamp for this Election."
    self.c_set_open_at.perform(timestamp, self.eid)

def set_close_at(self, timestamp):
    "Set the close_at timestamp for this Election."
    self.c_set_close_at.perform(timestamp, self.eid)
```

**Data Flow:**

```
POST /do-set-close_at/<eid>  {"date": "2020-01-01"}
  → _set_election_date(): parses date, checks format (✓)
  → Does NOT compare close_at against current open_at value (✗)
  → election.set_close_at(): blindly stores date
  
POST /do-set-open_at/<eid>   {"date": "2026-12-31"}
  → open_at set to date AFTER close_at, no validation (✗)
```

**Proof of Concept:**

```http
POST /do-set-close_at/abc123 HTTP/1.1
Content-Type: application/json
Cookie: <valid_session>

{"date": "2024-01-01"}
```

Then:

```http
POST /do-set-open_at/abc123 HTTP/1.1
Content-Type: application/json
Cookie: <valid_session>

{"date": "2025-12-31"}
```

Result: `open_at` is 2025-12-31, `close_at` is 2024-01-01 — logically impossible date configuration.

**Impact:**
- Elections can have illogical date configurations (close before open, dates in the past)
- Time-based queries and displays will show contradictory information to voters
- The `q_open_to_me` query relies on `salt IS NOT NULL` rather than date comparisons, so a past close_at doesn't auto-close the election, but voters see confusing closing dates
- Undermines trust in the election process

**Remediation:**

```python
async def _set_election_date(election, field):
    result = await basic_info()
    data = await quart.request.get_json()
    date_str = data.get('date')
    if not date_str:
        quart.abort(400, 'Missing date')

    try:
        dt = datetime.datetime.fromisoformat(date_str).date()
    except ValueError:
        quart.abort(400, 'Invalid date format')

    # Cross-validate dates
    md = election.get_metadata()
    
    if field == 'open_at':
        # If close_at exists, open_at must be before it
        if md.close_at:
            close_date = datetime.datetime.fromtimestamp(md.close_at).date()
            if dt >= close_date:
                quart.abort(400, 'Open date must be before close date')
        election.set_open_at(int(datetime.datetime.combine(
            dt, datetime.time()).timestamp()))
    elif field == 'close_at':
        # close_at must be after open_at if set
        if md.open_at:
            open_date = datetime.datetime.fromtimestamp(md.open_at).date()
            if dt <= open_date:
                quart.abort(400, 'Close date must be after open date')
        election.set_close_at(int(datetime.datetime.combine(
            dt, datetime.time()).timestamp()))
```

#### ASVS-223-MED-003: Election Can Be Opened Without Issues or Eligible Voters

**Severity:** MEDIUM  
**Gap Type:** Type A — No control exists  
**File:** `v3/steve/election.py`, function `open()`, lines ~72–87  
**Also affects:** `v3/server/pages.py`, function `do_open_endpoint()`, lines ~530–547

**Vulnerable Code:**

```python
# v3/steve/election.py — open()
def open(self, pdb):
    # Double-check the Election is in the editing state.
    assert self.is_editable()

    # No check that issues exist or voters are assigned
    self.add_salts()

    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)
    self.c_open.perform(salt, opened_key, self.eid)
```

```python
# v3/server/pages.py — do_open_endpoint()
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    pdb = steve.persondb.PersonDB.open(DB_FNAME)
    election.open(pdb)  # No pre-condition check on election completeness
```

**Data Flow:**

```
GET /do-open/<eid>
  → load_election: verifies EID exists (✓)
  → do_open_endpoint: calls election.open(pdb)
  → election.open(): checks is_editable (✓), does NOT check issues/voters exist (✗)
  → add_salts(): iterates q_all_issues — if empty, commits empty transaction
  → gather_election_data(): generates opened_key from possibly empty data
  → Election is now permanently in OPEN state with no voteable content
```

**Proof of Concept:**

1. Create an election via `POST /do-create-election` with just a title
2. Without adding any issues or voters, navigate to `GET /do-open/<eid>`
3. The election transitions to OPEN state irreversibly

**Impact:**
- An election in the OPEN state cannot be returned to EDITABLE (`open()` is irreversible)
- No voters can participate (no mayvote entries), making the election permanently useless
- The `opened_key` is derived from an empty dataset, providing weak anti-tamper protection
- Election administrators must create a new election, losing the EID and any configured metadata

**Remediation:**

```python
# v3/steve/election.py

def open(self, pdb):
    assert self.is_editable()

    # Verify the election has at least one issue
    self.q_issues.perform(self.eid)
    issues = self.q_issues.fetchall()
    if not issues:
        raise ValueError(
            f'Election[E:{self.eid}] cannot be opened: no issues defined'
        )

    # Verify at least one voter is assigned
    self.q_all_issues.perform(self.eid)
    mayvote_rows = self.q_all_issues.fetchall()
    if not mayvote_rows:
        raise ValueError(
            f'Election[E:{self.eid}] cannot be opened: no voters assigned'
        )

    self.add_salts()
    edata = self.gather_election_data(pdb)
    salt = crypto.gen_salt()
    opened_key = crypto.gen_opened_key(edata, salt)
    self.c_open.perform(salt, opened_key, self.eid)
```

## Architecture Observations

1. **Validation is concentrated in election.py but incomplete**: The `election.py` module acts as the business logic layer and enforces state transitions, but lacks data-combination validation for the most critical operation (vote submission).

2. **Client-side validation present, server-side absent**: The `vote-on.ezt` template includes JavaScript that validates YNA choices (checks if radio is selected) and STV rankings (checks if at least one candidate is ranked). However, these are trivially bypassable, and the server performs no equivalent validation.

3. **State enforcement uses mixed mechanisms**: Some state checks use `assert` (which can be disabled with `python -O`) while others use `_all_metadata(required_state)` which raises proper exceptions. The `add_vote()` method uses the latter (proper), while `add_issue()`, `delete_issue()`, `open()`, and `close()` use the former (weaker).

## Recommendations Summary

### Immediate Actions (Critical/High Priority)

1. **Implement votestring validation in `add_vote()`** (Finding ASVS-223-HIGH-001)
   - **Priority:** CRITICAL
   - **Effort:** Medium (2-3 days)
   - **Impact:** Prevents vote data corruption and tally failures
   - Implement `_validate_votestring()` method covering YNA and STV vote formats
   - Add validation before vote encryption in both `election.py` and `pages.py`

### Short-term Actions (Medium Priority)

2. **Add election date cross-validation** (Finding ASVS-223-MED-002)
   - **Priority:** HIGH
   - **Effort:** Low (1 day)
   - **Impact:** Prevents logically impossible election configurations
   - Implement relationship checks in `_set_election_date()` and `Election.create()`
   - Ensure close_at is always after open_at

3. **Add pre-open completeness check** (Finding ASVS-223-MED-003)
   - **Priority:** HIGH
   - **Effort:** Low (1 day)
   - **Impact:** Prevents unusable elections from being opened
   - Verify at least one issue and one eligible voter exist before allowing state transition to OPEN

### Long-term Actions (Defense-in-Depth)

4. **Replace `assert` statements with explicit exceptions**
   - **Priority:** MEDIUM
   - **Effort:** Medium (2-3 days)
   - **Impact:** Ensures state checks cannot be bypassed with Python optimization flags
   - Replace all state-checking `assert` statements in `add_issue()`, `edit_issue()`, `delete_issue()`, `open()`, `close()`, and `add_voter()`
   - Use the `_all_metadata(required_state)` pattern consistently

5. **Add comprehensive integration tests for data combination validation**
   - **Priority:** MEDIUM
   - **Effort:** High (1 week)
   - **Impact:** Prevents regression of validation logic
   - Test all invalid vote format combinations
   - Test date relationship edge cases
   - Test election state transition preconditions
