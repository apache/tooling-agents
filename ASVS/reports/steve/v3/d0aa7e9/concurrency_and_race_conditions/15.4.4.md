### Overview

The application uses Quart's async/await model for concurrent request handling but executes **all database and cryptographic operations synchronously within async route handlers** without offloading to a thread pool. This directly violates ASVS 15.4.4 by allowing any single request's blocking I/O to starve all other concurrent requests. Three distinct findings were identified at MEDIUM and HIGH severity.

#### ASVS-1544-HIGH-001: Synchronous Blocking Database I/O in Async Event Loop Without Thread Pool

**Severity:** HIGH  
**Gap Type:** Type A — No thread pool control exists anywhere in the codebase  
**ASVS Requirement:** 15.4.4 — "…by leveraging thread pools, allowing lower-priority threads to proceed within a reasonable timeframe."

**Location:**
- **File:** `v3/steve/election.py`
- **Lines:** 38–43 (`__init__`, `open_database`)
- **Function:** Synchronous DB connection
- **File:** `v3/server/pages.py`
- **Line:** ~181 (`load_election` decorator)
- **Function:** Sync `Election()` in async handler
- **File:** `v3/server/pages.py`
- **Lines:** 399–432 (`do_vote_endpoint`)
- **Function:** Sync `add_vote()` in async handler
- **File:** `v3/server/pages.py`
- **Lines:** 144–172 (`voter_page`)
- **Function:** Sync `open_to_pid()` in async handler

**Related Functions Checked:**
- `Election.__init__()` — Creates synchronous database connection
- `Election.open_database()` — Returns synchronous `asfpy.db.DB` wrapper
- `load_election()` decorator — Instantiates `Election` in async context
- `do_vote_endpoint()` — Calls `add_vote()` synchronously
- `voter_page()` — Calls multiple synchronous DB methods
- All 12 database-accessing endpoints listed in affected endpoints table

**Description:**

The application uses Quart's async/await model for concurrent request handling but executes all database and cryptographic operations synchronously within async route handlers without offloading to a thread pool. This creates a systemic thread starvation vulnerability where every database-accessing endpoint blocks the event loop, preventing all other concurrent requests from proceeding.

**Vulnerable Code (`election.py`):**
```python
@staticmethod
def open_database(db_fname):
    return asfpy.db.DB(db_fname, yaml_fname=QUERIES, yaml_section='election')

def __init__(self, db_fname, eid, op='Opening'):
    self.db = self.open_database(db_fname)  # Synchronous sqlite3 connection
    self.eid = eid
    if not_found(self.q_check_election, eid):  # Synchronous query
        raise ElectionNotFound(eid)
```

**Vulnerable Code (`pages.py` — `load_election` decorator):**
```python
async def loader(eid):
    try:
        e = steve.election.Election(DB_FNAME, eid)  # BLOCKS event loop
    except steve.election.ElectionNotFound:
        ...
    return await func(e)
```

**Data Flow:**
1. Quart receives HTTP request → dispatches to async handler
2. Async handler calls `Election(DB_FNAME, eid)` → creates synchronous `sqlite3` connection → executes synchronous SQL query
3. **Event loop is blocked** — all other pending async tasks (other HTTP requests) cannot proceed
4. This pattern repeats for every database-touching endpoint

**All Affected Endpoints:**

| Route | Handler | Sync Blocking Call |
|-------|---------|-------------------|
| `GET /voter` | `voter_page()` | `Election.open_to_pid()`, `Election.owned_elections()`, `Election.upcoming_to_pid()` |
| `GET /vote-on/<eid>` | `vote_on_page()` | `Election()`, `q_find_issues`, `list_issues()` |
| `POST /do-vote/<eid>` | `do_vote_endpoint()` | `Election()`, `add_vote()` × N issues |
| `GET /do-open/<eid>` | `do_open_endpoint()` | `Election()`, `open()` (includes Argon2) |
| `GET /do-close/<eid>` | `do_close_endpoint()` | `Election()`, `close()` |
| `POST /do-create-election` | `do_create_endpoint()` | `Election.create()` |
| `POST /do-add-issue/<eid>` | `do_add_issue_endpoint()` | `Election()`, `add_issue()` |
| `POST /do-edit-issue/<eid>/<iid>` | `do_edit_issue_endpoint()` | `Election()`, `edit_issue()` |
| `POST /do-delete-issue/<eid>/<iid>` | `do_delete_issue_endpoint()` | `Election()`, `delete_issue()` |
| `GET /manage/<eid>` | `manage_page()` | `Election()`, `get_metadata()`, `list_issues()` |
| `GET /admin` | `admin_page()` | `open_to_pid()`, `owned_elections()`, `PersonDB.open()` |
| `GET /docs/<iid>/<docname>` | `serve_doc()` | `Election.open_database()`, `q_get_mayvote` |

**Proof of Concept:**
```bash
# Two concurrent requests demonstrate starvation:
# Terminal 1 - Trigger a database-heavy operation (voter page with many elections):
curl -b session_cookie https://vote.example.org/voter &

# Terminal 2 - Simple request that should be fast but is starved:
time curl -b session_cookie https://vote.example.org/about
# Expected: ~1ms (no DB). Actual: blocked until Terminal 1's sync DB completes.
```

**Impact:**
- Any single request performing synchronous database I/O blocks **all** other concurrent request processing in the Quart event loop.
- Under even modest concurrency (2+ simultaneous users), requests queue behind blocking operations, causing cascading latency.
- During heavy operations (election opening, bulk vote submission), the application becomes effectively single-threaded and unresponsive.

**Remediation:**

Wrap all synchronous `Election` method calls in `asyncio.to_thread()` to offload them to a thread pool:

```python
import asyncio

# In pages.py - Updated load_election decorator
def load_election(func):
    @functools.wraps(func)
    async def loader(eid):
        try:
            # Offload synchronous DB operation to thread pool
            e = await asyncio.to_thread(
                steve.election.Election, DB_FNAME, eid
            )
        except steve.election.ElectionNotFound:
            result = await basic_info()
            result.title = 'Unknown Election'
            result.eid = eid
            raise_404(T_BAD_EID, result)
        return await func(e)
    return loader

# In do_vote_endpoint - Offload blocking vote submission
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    form = edict(await quart.request.form)
    ...
    for iid, votestring in votes.items():
        await asyncio.to_thread(
            election.add_vote, result.uid, iid, votestring
        )
    ...
```

Alternatively, adopt an async SQLite driver such as `aiosqlite`:
```python
import aiosqlite

class AsyncElection:
    @staticmethod
    async def open_database(db_fname):
        return await aiosqlite.connect(str(db_fname))
```

#### ASVS-1544-MEDIUM-002: CPU-Intensive Argon2 Key Derivation Blocks Event Loop During Election Opening

**Severity:** MEDIUM  
**Gap Type:** Type A — No offloading mechanism for CPU-bound operations  
**ASVS Requirement:** 15.4.4 — "…ensuring fair access to resources…allowing lower-priority threads to proceed within a reasonable timeframe."

**Location:**
- **File:** `v3/steve/election.py`
- **Lines:** 71–85
- **Function:** `open()` method — Orchestrates all blocking operations
- **File:** `v3/steve/election.py`
- **Lines:** 116–131
- **Function:** `add_salts()` — Transaction loop over all voters
- **File:** `v3/server/pages.py`
- **Lines:** 452–469
- **Function:** `do_open_endpoint()` — Async handler calling sync `open()`

**Related Functions Checked:**
- `Election.open()` — Orchestrates salt generation, data gathering, and Argon2 computation
- `Election.add_salts()` — Loops over all voters/issues generating salts
- `Election.gather_election_data()` — Multiple sequential DB reads
- `crypto.gen_opened_key()` — Argon2 key derivation (CPU-intensive)
- `do_open_endpoint()` — Async handler that calls all of the above synchronously

**Description:**

Election opening performs CPU-intensive Argon2 key derivation and holds a database write lock during an unbounded iteration over all voter-issue combinations. The entire operation executes synchronously in the async event loop, blocking all concurrent requests for potentially 1-5+ seconds depending on election size and Argon2 parameters.

**Vulnerable Code (`election.py` — `open()`):**
```python
def open(self, pdb):
    assert self.is_editable()
    self.add_salts()                                    # Step 1: Transaction loop (N iterations)
    edata = self.gather_election_data(pdb)              # Step 2: Multiple DB queries
    salt = crypto.gen_salt()                            # Step 3: RNG
    opened_key = crypto.gen_opened_key(edata, salt)     # Step 4: Argon2 (CPU-intensive!)
    self.c_open.perform(salt, opened_key, self.eid)     # Step 5: DB write
```

**Vulnerable Code (`election.py` — `add_salts()`):**
```python
def add_salts(self):
    assert self.is_editable()
    self.db.conn.execute('BEGIN TRANSACTION')            # Acquire write lock
    self.q_all_issues.perform(self.eid)
    for mayvote in self.q_all_issues.fetchall():         # N voters × M issues
        salt = crypto.gen_salt()
        self.c_salt_mayvote.perform(salt, mayvote.rowid) # N×M writes while holding lock
    self.db.conn.execute('COMMIT')                       # Release write lock
```

**Vulnerable Code (`pages.py` — `do_open_endpoint()`):**
```python
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    pdb = steve.persondb.PersonDB.open(DB_FNAME)  # Sync connection
    election.open(pdb)                             # ALL of the above blocks event loop
```

**Data Flow:**
1. Admin requests `GET /do-open/<eid>` → async handler enters
2. `election.open(pdb)` begins:
   - `add_salts()`: Acquires SQLite write lock → loops over `voters × issues` entries (potentially hundreds) generating salts and writing to DB → releases lock
   - `gather_election_data()`: Multiple sequential DB reads
   - `crypto.gen_opened_key()`: **Argon2 hash computation** — deliberately CPU/memory-intensive (100ms–2s depending on parameters)
   - `c_open.perform()`: Final DB write
3. **Total event loop blocking: potentially 1–5+ seconds** depending on election size and Argon2 parameters
4. All concurrent requests (vote submissions, page loads) are starved for this entire duration

**Proof of Concept:**
```bash
# Open an election with 50 voters × 10 issues = 500 mayvote rows:
# Terminal 1:
curl -b admin_session https://vote.example.org/do-open/a1b2c3d4e5

# Terminal 2 (immediately after):
time curl -b voter_session https://vote.example.org/voter
# Result: Voter's page load blocked for 1-5 seconds while Argon2 computes
# and 500 salt writes complete in a held transaction
```

**Impact:**
- Election opening is an infrequent but high-impact event. During the operation, **all** concurrent user requests are starved.
- The `add_salts()` transaction holds SQLite's file-level write lock for the entire iteration, blocking even separate database connections from writing.
- Argon2 key derivation is deliberately CPU-intensive; running it synchronously in the event loop blocks all async task scheduling for its full duration.
- Combined, these create a multi-second window where the application is completely unresponsive.

**Remediation:**

Offload the entire `election.open()` call to a thread pool, and consider breaking the `add_salts()` transaction into batches:

```python
# In pages.py - Offload election opening to thread pool
@APP.get('/do-open/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_open_endpoint(election):
    result = await basic_info()
    pdb = await asyncio.to_thread(
        steve.persondb.PersonDB.open, DB_FNAME
    )
    # Offload the entire blocking open() to thread pool
    await asyncio.to_thread(election.open, pdb)
    ...
```

```python
# In election.py - Batch the salt generation to limit lock duration
def add_salts(self, batch_size=100):
    assert self.is_editable()
    self.q_all_issues.perform(self.eid)
    all_mayvotes = list(self.q_all_issues.fetchall())
    
    for i in range(0, len(all_mayvotes), batch_size):
        batch = all_mayvotes[i:i + batch_size]
        self.db.conn.execute('BEGIN TRANSACTION')
        for mayvote in batch:
            salt = crypto.gen_salt()
            self.c_salt_mayvote.perform(salt, mayvote.rowid)
        self.db.conn.execute('COMMIT')
```

#### ASVS-1544-MEDIUM-003: Unbounded Synchronous Vote Processing Loop Amplifies Event Loop Starvation

**Severity:** MEDIUM  
**Gap Type:** Type A — No yielding mechanism in processing loop  
**ASVS Requirement:** 15.4.4 — "…prevent thread starvation by ensuring fair access to resources…"

**Location:**
- **File:** `v3/server/pages.py`
- **Lines:** 399–432
- **Function:** `do_vote_endpoint()` — Loop over votes
- **File:** `v3/steve/election.py`
- **Lines:** 231–244
- **Function:** `add_vote()` — Per-vote DB + crypto operations

**Related Functions Checked:**
- `do_vote_endpoint()` — Iterates over submitted votes
- `Election.add_vote()` — Performs DB reads, PBKDF key derivation, encryption, and DB write per vote
- `Election._all_metadata()` — Redundantly re-queried on each iteration
- `Election.q_get_mayvote` — Per-vote eligibility check

**Description:**

Vote submission loops over all issues synchronously, performing database reads, PBKDF key derivation, encryption, and database writes for each issue without yielding to the event loop. For elections with many issues, this creates extended blocking proportional to the number of issues, with redundant state checks amplifying the problem.

**Vulnerable Code (`pages.py`):**
```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    ...
    for iid, votestring in votes.items():
        if iid not in issue_dict:
            ...
        try:
            election.add_vote(result.uid, iid, votestring)  # Blocks per iteration
```

**Vulnerable Code (`election.py` — `add_vote()`):**
```python
def add_vote(self, pid, iid, votestring):
    md = self._all_metadata(self.S_OPEN)              # DB query (state check)
    mayvote = self.q_get_mayvote.first_row(pid, iid)  # DB query (eligibility)
    vote_token = crypto.gen_vote_token(                # HMAC/hash computation
        md.opened_key, pid, iid, mayvote.salt
    )
    ciphertext = crypto.create_vote(                   # PBKDF key derivation + encryption
        vote_token, mayvote.salt, votestring
    )
    self.c_add_vote.perform(vote_token, ciphertext)    # DB write
```

**Data Flow:**
1. Voter submits form with votes for N issues → `do_vote_endpoint` receives all `vote-<iid>` fields
2. For each issue: `add_vote()` performs 2 DB reads + crypto key derivation + encryption + 1 DB write — **all synchronous**
3. With N issues per election: total blocking time = N × (2 queries + PBKDF + Fernet encrypt + 1 insert)
4. For an election with 20 issues: ~20 × 5 operations = ~100 synchronous blocking operations in a single request
5. Event loop is blocked for the entire duration; no other requests can be processed

**Proof of Concept:**
```bash
# Submit votes for 20 issues simultaneously:
curl -X POST -b session_cookie \
  -d "vote-a1b2c3d4e5=Yes&vote-b2c3d4e5f6=No&vote-c3d4e5f6g7=Abstain&..." \
  https://vote.example.org/do-vote/x1y2z3a4b5

# While this processes (~20 sequential DB+crypto operations):
# All other users' requests are queued and starved
```

**Impact:**
- Vote submission for elections with many issues creates extended event loop blocking proportional to the number of issues.
- Each `add_vote()` call includes key derivation (PBKDF), which is deliberately slow. This multiplied across N issues creates significant starvation.
- Multiple voters submitting simultaneously will serialize completely, with each voter's full submission blocking all others.
- Additionally, `_all_metadata(self.S_OPEN)` is re-queried on every iteration, performing redundant state checks that add unnecessary blocking time.

**Remediation:**

```python
# Option 1: Offload entire vote loop to thread pool
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    result = await basic_info()
    form = edict(await quart.request.form)
    ...
    
    async def process_votes():
        for iid, votestring in votes.items():
            if iid not in issue_dict:
                raise ValueError(f'Invalid issue ID: {iid}')
            # Offload each blocking vote to thread pool
            await asyncio.to_thread(
                election.add_vote, result.uid, iid, votestring
            )
    
    try:
        await process_votes()
    except Exception as e:
        await flash_danger(f'Error submitting votes: {e}')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
    ...
```

```python
# Option 2 (election.py): Add bulk vote method that caches metadata
def add_votes_bulk(self, pid: str, votes: dict[str, str]):
    """Submit multiple votes efficiently, caching repeated queries."""
    md = self._all_metadata(self.S_OPEN)  # Single state check
    
    for iid, votestring in votes.items():
        mayvote = self.q_get_mayvote.first_row(pid, iid)
        if not mayvote:
            raise ValueError(f'Not eligible to vote on {iid}')
        vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
        ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
        self.c_add_vote.perform(vote_token, ciphertext)
```

## Positive Security Patterns

1. **State Machine Enforcement**: The `Election` class uses `assert self.is_editable()` / `is_open()` / `is_closed()` guards before state-changing operations, preventing operations in wrong states.

2. **Transaction Usage for Multi-Step Operations**: `delete()` and `add_salts()` properly wrap multi-table operations in `BEGIN TRANSACTION` / `COMMIT` blocks to maintain referential integrity.

3. **Separate Database Connections Per Request**: Each request creates its own `Election` instance with its own DB connection, avoiding shared mutable database state between concurrent async tasks.

4. **Anti-Tamper Detection**: The `opened_key` hash over election data provides integrity verification, ensuring election parameters haven't been modified after opening.

5. **Non-Sequential IDs**: Election and issue IDs use random 10-hex-character values, preventing enumeration.

6. **Comprehensive Audit Logging**: All state-changing operations log the acting user ID, target resource, and action performed.

---

## Architecture Observations

1. **Fundamental Async/Sync Mismatch**: The application is built on Quart (async) but uses `asfpy.db.DB` (synchronous `sqlite3` wrapper) throughout. This creates a systemic thread starvation vulnerability where **every** database-accessing endpoint blocks the event loop. This is not a single-point failure — it's a pervasive architectural gap.

2. **No Connection Pooling**: Each `Election` instantiation creates a new `sqlite3` connection (`open_database()`). While this avoids connection-sharing hazards, it also means there is no fair resource scheduling, no connection limits, and no reuse optimization.

3. **SQLite File-Level Locking**: Even if synchronous operations were offloaded to a thread pool, SQLite's file-level write lock would serialize all write operations. The `add_salts()` transaction exacerbates this by holding the write lock for the entire voter iteration.

4. **Crypto Operations Inline**: CPU-intensive cryptographic operations (Argon2 key derivation, PBKDF for vote encryption) are performed inline in request handlers without offloading to a dedicated thread or process pool.

## Recommendations Summary

### Immediate Actions (Priority 1)

| Recommendation | Effort | Impact | Timeline |
|----------------|--------|--------|----------|
| **Wrap all `Election` method calls in `asyncio.to_thread()`** in `pages.py` handlers. This is the minimum viable fix and addresses all three findings. | Low | HIGH — Prevents event loop blocking for all endpoints | Immediate |

### Short-term Actions (Priority 2-3)

| Recommendation | Effort | Impact | Timeline |
|----------------|--------|--------|----------|
| **Offload Argon2 computation** in `crypto.gen_opened_key()` specifically to a process pool (`ProcessPoolExecutor`) since it's CPU-bound, not I/O-bound. | Medium | HIGH — Prevents seconds-long starvation during election opening | 1-2 weeks |
| **Batch `add_salts()` transactions** to limit write-lock duration. Use smaller batches (e.g., 50–100 rows) to allow other writers between batches. | Low | MEDIUM — Reduces write-lock contention window | 1 week |
| **Add bulk `add_votes()` method** that performs a single state check and wraps all inserts in one transaction, reducing per-vote overhead. | Low | MEDIUM — Reduces blocking time for multi-issue votes | 1 week |

### Long-term Actions (Priority 4-6)

| Recommendation | Effort | Impact | Timeline |
|----------------|--------|--------|----------|
| **Consider migrating to `aiosqlite`** for native async database operations, eliminating the need for `to_thread()` wrappers. | High | HIGH — Architecturally correct solution for async+SQLite | 1-3 months |
| **Configure thread pool size** via `asyncio.get_event_loop().set_default_executor(ThreadPoolExecutor(max_workers=N))` to match expected concurrency and prevent thread pool exhaustion. | Low | MEDIUM — Ensures fair scheduling of offloaded operations | 2 weeks |
