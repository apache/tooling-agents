### Overview

The application processes concurrent requests via Quart's async model, with all critical election data in a shared SQLite database. While each request creates its own database connection (avoiding shared object issues at the Python level), critical TOCTOU (Time-of-Check to Time-of-Use) race conditions exist in election state transitions and vote submission due to non-atomic check-then-act patterns across separate SQL statements.

## Findings

### Critical Severity

#### ASVS-1541-CRI-001: Race Condition in Election Opening Can Corrupt Cryptographic State

**Severity:** CRITICAL  
**Gap Type:** Type A — Entry point with no concurrency control

**Location:**
- **File:** `v3/steve/election.py`
- **Method:** `open()`
- **Lines:** 76-90
- **Related Endpoint:** `v3/server/pages.py`, `do_open_endpoint()`, line 461

**Related Functions Checked:**
- `is_editable()` — State check performed separately
- `add_salts()` — Commits its own transaction
- `gather_election_data()` — Reads data after salt generation
- `c_open.perform()` — Final state write (autocommit)

**Description:**

The election opening process performs multiple separate database operations without transactional protection, creating a critical race condition window. The state check (`is_editable()`), salt generation (`add_salts()`), and final state transition (`c_open.perform()`) are not atomic, allowing concurrent open requests to corrupt the cryptographic state.

**Vulnerable Code:**

```python
def open(self, pdb):
    # Double-check the Election is in the editing state.
    assert self.is_editable()                          # Step 1: Read state (separate query)

    # Add salts first. If this is gonna fail, then make sure it
    # happens before we move to the "opened" state.
    self.add_salts()                                   # Step 2: Write salts (own transaction, commits)

    edata = self.gather_election_data(pdb)             # Step 3: Read election data
    salt = crypto.gen_salt()                           # Step 4: Generate new salt
    opened_key = crypto.gen_opened_key(edata, salt)    # Step 5: Derive opened_key

    self.c_open.perform(salt, opened_key, self.eid)    # Step 6: Write election state (autocommit)
```

**Data Flow:**

1. Request A reads election state → editable ✓
2. Request B reads election state → editable ✓ (concurrent)
3. Request A calls `add_salts()` → writes per-voter salts, commits transaction
4. Request B calls `add_salts()` → **overwrites** all per-voter salts with new values, commits
5. Request A calls `gather_election_data()` → reads salts written by **Request B**
6. Request A computes `opened_key` based on Request B's salts
7. Request A writes `salt_A` + `opened_key_A` to election
8. Request B computes `opened_key` based on its own salts → writes `salt_B` + `opened_key_B`
9. **Final state**: Election has `salt_B`/`opened_key_B` but mayvote table may contain salts from either request depending on interleaving

**Proof of Concept:**

Two administrators simultaneously hit `GET /do-open/<eid>` (note: this is a GET for a state-changing operation). Since there is no lock, mutex, or atomic compare-and-swap:

```
Admin A: GET /do-open/a1b2c3d4e5 → passes is_editable() → add_salts() commits
Admin B: GET /do-open/a1b2c3d4e5 → passes is_editable() → add_salts() overwrites salts
Admin A: c_open.perform(salt_A, key_A, eid) → writes election record
Admin B: c_open.perform(salt_B, key_B, eid) → overwrites election record
```

**Impact:**

- Election `opened_key` will not match recomputed key during tamper detection (`is_tampered()` always returns `True`)
- Per-voter salts may be inconsistent with the election-level key, making vote decryption fail during tally
- The election becomes permanently corrupted—votes cast cannot be decrypted, and the election appears tampered with
- Complete loss of election integrity for all voters in the affected election

**Remediation:**

Wrap the entire open operation in a single transaction with an atomic state check:

```python
def open(self, pdb):
    self.db.conn.execute('BEGIN IMMEDIATE TRANSACTION')
    try:
        md = self.q_metadata.first_row(self.eid)
        if not md or self._compute_state(md) != self.S_EDITABLE:
            self.db.conn.execute('ROLLBACK')
            raise ElectionBadState(self.eid, self._compute_state(md), self.S_EDITABLE)

        # Add salts within this transaction
        self.q_all_issues.perform(self.eid)
        for mayvote in self.q_all_issues.fetchall():
            salt = crypto.gen_salt()
            self.c_salt_mayvote.perform(salt, mayvote.rowid)

        edata = self.gather_election_data(pdb)
        salt = crypto.gen_salt()
        opened_key = crypto.gen_opened_key(edata, salt)
        self.c_open.perform(salt, opened_key, self.eid)

        self.db.conn.execute('COMMIT')
    except Exception:
        self.db.conn.execute('ROLLBACK')
        raise
```

Additionally, verify that `c_open` uses an atomic WHERE clause:

```sql
UPDATE election SET salt=?, opened_key=? 
WHERE eid=? AND salt IS NULL AND opened_key IS NULL
```

And check `rowcount == 1` after execution.

### High Severity

#### ASVS-1541-HIG-002: TOCTOU in Vote Submission — Eligibility Check and Vote Insert Not Transactional

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with no concurrency control

**Location:**
- **File:** `v3/steve/election.py`
- **Method:** `add_vote()`
- **Lines:** 268-285
- **Related Endpoint:** `v3/server/pages.py`, `do_vote_endpoint()`, line 397

**Related Functions Checked:**
- `_all_metadata()` — Election state check (separate query)
- `q_get_mayvote.first_row()` — Voter eligibility check (separate query)
- `crypto.gen_vote_token()` — Token generation
- `crypto.create_vote()` — Vote encryption
- `c_add_vote.perform()` — Vote insertion (autocommit)

**Description:**

The vote submission process performs eligibility verification and vote insertion as separate database operations without transactional protection. This creates a TOCTOU vulnerability where an election can be closed between the eligibility check and the vote insertion, allowing votes to be recorded after the official closure.

**Vulnerable Code:**

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    "Add VOTESTRING as the (latest) vote by PID for IID."

    # The Election should be open.
    md = self._all_metadata(self.S_OPEN)                          # CHECK 1: Election is open

    ### validate VOTESTRING for ISSUE.TYPE voting

    mayvote = self.q_get_mayvote.first_row(pid, iid)              # CHECK 2: Voter is eligible
    vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)

    # Pass MAYVOTE.SALT for PBKDF.
    ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)

    self.c_add_vote.perform(vote_token, ciphertext)               # ACT: Insert vote (no transaction)
```

**Data Flow:**

```
Source: Concurrent requests to POST /do-vote/<eid>
  → _all_metadata(S_OPEN)      [READ: election.closed, election.salt]
  → q_get_mayvote.first_row()  [READ: mayvote eligibility]
  → <gap: no lock held, election could be closed here>
  → crypto.gen_vote_token()    [compute]
  → crypto.create_vote()       [compute]
  → c_add_vote.perform()       [WRITE: insert into vote table]
Missing control: No transaction wrapping CHECK and ACT operations
```

**Proof of Concept:**

```
Time T1: Voter sends POST /do-vote/a1b2c3d4e5 → _all_metadata returns S_OPEN ✓
Time T2: Admin sends GET /do-close/a1b2c3d4e5 → election.close() succeeds, election is now CLOSED
Time T3: Voter's request continues → c_add_vote.perform() inserts vote into closed election
Time T4: Tally reads this vote (highest vid) as the voter's most recent vote
```

**Impact:**

- Votes can be recorded after an election has been officially closed
- During tally, the post-closure vote (having the highest `vid` via `AUTOINCREMENT`) will be treated as the voter's most recent vote, overriding any legitimate vote cast before closure
- An attacker could deliberately time vote submissions to coincide with election closure to change their vote after the window was supposed to be closed
- Undermines the integrity guarantee that only votes submitted while an election is open are counted

**Remediation:**

Wrap the eligibility check and vote insertion in a single transaction with a re-check of election state:

```python
def add_vote(self, pid: str, iid: str, votestring: str):
    self.db.conn.execute('BEGIN IMMEDIATE TRANSACTION')
    try:
        # Atomically verify election is open AND voter is eligible
        md = self._all_metadata(self.S_OPEN)
        mayvote = self.q_get_mayvote.first_row(pid, iid)
        if not mayvote:
            self.db.conn.execute('ROLLBACK')
            raise ValueError(f'Voter {pid} not eligible for issue {iid}')

        vote_token = crypto.gen_vote_token(md.opened_key, pid, iid, mayvote.salt)
        ciphertext = crypto.create_vote(vote_token, mayvote.salt, votestring)
        self.c_add_vote.perform(vote_token, ciphertext)

        self.db.conn.execute('COMMIT')
    except Exception:
        self.db.conn.execute('ROLLBACK')
        raise
```

With `BEGIN IMMEDIATE`, SQLite acquires a reserved lock immediately, preventing concurrent writers from modifying the election state between the check and the insert.

#### ASVS-1541-HIG-003: Election Close Operation Not Atomic — No State Guard in SQL

**Severity:** HIGH  
**Gap Type:** Type A — Entry point with no concurrency control

**Location:**
- **File:** `v3/steve/election.py`
- **Method:** `close()`
- **Lines:** 121-127
- **Related Endpoint:** `v3/server/pages.py`, `do_close_endpoint()`, line 482

**Related Functions Checked:**
- `is_open()` — State check (separate query)
- `_all_metadata()` — Called by `is_open()`
- `c_close.perform()` — State update (separate query, no WHERE clause verification)

**Description:**

The election close operation performs a state check and state update as separate database operations without transactional protection or atomic state verification in the UPDATE statement. This creates a race condition where multiple close requests can execute concurrently, and more critically, allows votes to be submitted during the close operation.

**Vulnerable Code:**

```python
def close(self):
    "Close an election."

    # The Election should be open.
    assert self.is_open()                    # CHECK: separate read query

    # Simple tweak of the metadata to close the Election.
    self.c_close.perform(self.eid)           # ACT: separate write query
```

**Data Flow:**

```
Source: Concurrent admin requests to GET /do-close/<eid>
  → is_open() → _all_metadata() → q_metadata query [READ]
  → <gap: another request could change state>
  → c_close.perform(eid)  [WRITE: presumably UPDATE election SET closed=1 WHERE eid=?]
Missing control: UPDATE query does not include WHERE clause checking current state
Missing control: No rowcount verification after update
```

**Proof of Concept:**

Scenario 1 — Double close:

```
Admin A: GET /do-close/a1b2c3d4e5 → is_open() returns True
Admin B: GET /do-close/a1b2c3d4e5 → is_open() returns True
Admin A: c_close.perform() → sets closed=1 (idempotent, but...)
Admin B: c_close.perform() → sets closed=1 again
```

While double-close is idempotent, the pattern demonstrates the race window.

Scenario 2 — Close racing with vote submission (interaction with SEV-002):

```
Time T1: Admin: is_open() → True, begins close
Time T2: Voter: _all_metadata(S_OPEN) → True, begins voting
Time T3: Admin: c_close.perform() → election closed
Time T4: Voter: c_add_vote.perform() → vote recorded in closed election
```

**Impact:**

- Combined with Finding SEV-002, the non-atomic close creates a window where votes can be submitted after the election administrator believes it is closed
- The `c_close` SQL likely does not include `WHERE closed IS NULL OR closed = 0`, meaning it doesn't atomically verify the election was actually open before closing
- No `rowcount` check means a failed state transition is silently ignored

**Remediation:**

Use an atomic UPDATE with a state-checking WHERE clause and verify the result:

```python
def close(self):
    "Close an election."
    # Atomic close: only succeeds if election is currently open
    self.c_close_atomic.perform(self.eid)  
    # SQL: UPDATE election SET closed=1 WHERE eid=? 
    #      AND salt IS NOT NULL AND opened_key IS NOT NULL 
    #      AND (closed IS NULL OR closed = 0)
    if self.c_close_atomic.rowcount != 1:
        raise ElectionBadState(self.eid, self.get_state(), self.S_OPEN)
```

### Medium Severity

#### ASVS-1541-MED-004: Security-Critical State Enforcement Uses `assert` — Disableable in Production

**Severity:** MEDIUM  
**Gap Type:** Type B — Control exists but would not be called in optimized Python

**Location:**
- **File:** `v3/steve/election.py`
- **Multiple Methods:** `delete()`, `open()`, `close()`, `add_salts()`, `add_issue()`, `edit_issue()`, `delete_issue()`, `add_voter()`

**Related Functions Checked:**
- `is_editable()` — State check called by assertions
- `is_open()` — State check called by assertions
- All state-changing database operations following assertions

**Description:**

Security-critical state enforcement throughout the election lifecycle relies on Python `assert` statements, which are removed when Python is run with optimization flags (`-O` or `-OO`). This is a common production optimization that would completely bypass all election state validation.

**Vulnerable Code Locations:**

```python
# election.py line 56 (delete)
assert self.is_editable()

# election.py line 78 (open)
assert self.is_editable()

# election.py line 124 (close)
assert self.is_open()

# election.py line 131 (add_salts)
assert self.is_editable()

# election.py line 193-194 (add_issue)
assert self.is_editable()
assert vtype in vtypes.TYPES

# election.py line 214-215 (edit_issue)
assert self.is_editable()
assert vtype in vtypes.TYPES

# election.py line 228 (delete_issue)
assert self.is_editable()

# election.py line 261 (add_voter)
assert self.is_editable()
```

**Data Flow:**

```
Source: Any state-changing operation
  → assert self.is_editable() / is_open()  [CONTROL: can be removed by -O flag]
  → state-changing database operation        [SENSITIVE OPERATION: proceeds unchecked]
Missing control: Proper conditional checks that cannot be disabled
```

**Impact:**

- Running Python with `-O` or `-OO` flags (common in production for performance) removes all `assert` statements
- With assertions disabled, ALL election state enforcement is bypassed:
  - Issues can be added/edited/deleted on open or closed elections
  - Voters can be added to open elections
  - Elections can be opened multiple times or closed when editable
  - Vote types are not validated
- This is a **Type B gap**: the control EXISTS (state checks) but would NOT BE CALLED in optimized Python

**Remediation:**

Replace all security-critical `assert` statements with proper conditional checks:

```python
def close(self):
    "Close an election."
    if not self.is_open():
        raise ElectionBadState(self.eid, self.get_state(), self.S_OPEN)
    self.c_close.perform(self.eid)

def add_issue(self, title, description, vtype, kv):
    if not self.is_editable():
        raise ElectionBadState(self.eid, self.get_state(), self.S_EDITABLE)
    if vtype not in vtypes.TYPES:
        raise ValueError(f'Invalid vote type: {vtype}')
    # ... rest of method
```

#### ASVS-1541-MED-005: Batch Vote Submission Without Transactional Integrity

**Severity:** MEDIUM  
**Gap Type:** Type A — Entry point with no concurrency control

**Location:**
- **File:** `v3/server/pages.py`
- **Function:** `do_vote_endpoint()`
- **Lines:** 397-437
- **Related:** `v3/steve/election.py`, `add_vote()`, lines 268-285

**Related Functions Checked:**
- `add_vote()` — Called in loop, each call autocommits
- Form validation logic
- Error handling and flash message generation

**Description:**

When a voter submits votes for multiple issues in a single request, each vote is inserted individually without wrapping the entire batch in a transaction. If an error occurs mid-submission or if the election is closed during processing, some votes are committed while others are not, creating an inconsistent state.

**Vulnerable Code:**

```python
@APP.post('/do-vote/<eid>')
@asfquart.auth.require({R.committer})
@load_election
async def do_vote_endpoint(election):
    # ...
    for iid, votestring in votes.items():
        if iid not in issue_dict:
            await flash_danger(f'Invalid issue ID: {iid}')
            return quart.redirect(f'/vote-on/{election.eid}', code=303)

        try:
            election.add_vote(result.uid, iid, votestring)    # Individual, non-transactional
            # ...
        except Exception as e:
            # ... error handling returns, leaving partial votes committed
            await flash_danger(f'Error submitting vote for issue {iid}.')
            return quart.redirect(f'/vote-on/{election.eid}', code=303)

    await flash_success('Votes submitted successfully!')
```

**Data Flow:**

```
Source: POST /do-vote/<eid> with form data containing votes for multiple issues
  → Loop iteration 1: add_vote(uid, iid1, vote1)  [autocommit - COMMITTED]
  → Loop iteration 2: add_vote(uid, iid2, vote2)  [autocommit - COMMITTED]
  → Loop iteration 3: add_vote(uid, iid3, vote3)  [FAILS - Exception]
  → Return with error; votes 1 and 2 are already committed
Missing control: No wrapping transaction for the entire batch
```

**Impact:**

- If the election is closed mid-submission (race with close), some votes are recorded and others are not
- If an error occurs on any individual vote (e.g., database error, crypto failure), previously submitted votes in the batch are already committed
- Voter sees an error message but has no way to know which votes were actually recorded
- This creates inconsistent voter records—some issues have votes, others don't—which the voter may not realize when re-submitting

**Remediation:**

Wrap the entire vote batch in a single transaction:

```python
async def do_vote_endpoint(election):
    result = await basic_info()
    form = edict(await quart.request.form)
    # ...
    
    election.db.conn.execute('BEGIN IMMEDIATE TRANSACTION')
    try:
        for iid, votestring in votes.items():
            if iid not in issue_dict:
                election.db.conn.execute('ROLLBACK')
                await flash_danger(f'Invalid issue ID: {iid}')
                return quart.redirect(f'/vote-on/{election.eid}', code=303)
            election.add_vote_no_state_check(result.uid, iid, votestring)
        election.db.conn.execute('COMMIT')
    except Exception as e:
        election.db.conn.execute('ROLLBACK')
        await flash_danger('Error submitting votes. No votes were recorded.')
        return quart.redirect(f'/vote-on/{election.eid}', code=303)
```

## Positive Security Patterns

1. **Per-Request Database Connections**: Each `Election()` instance creates its own `asfpy.db.DB` connection, avoiding shared connection state between concurrent requests.

2. **Transaction Usage in Multi-Step Deletes**: The `delete()` method (line 59) and `add_salts()` method (line 135) correctly use `BEGIN TRANSACTION`/`COMMIT` for multi-step operations.

3. **ID Collision Retry**: Both `create()` (line 446) and `add_issue()` (line 195) handle `sqlite3.IntegrityError` with retry loops for ID generation conflicts.

4. **Schema Trigger for Closed Elections**: The `prevent_open_close_update` trigger prevents modification of `open_at`/`close_at` after an election is closed, providing database-level enforcement.

5. **Re-voting by Design**: The vote table intentionally allows duplicate `vote_token` values, with `AUTOINCREMENT` on `vid` providing a deterministic ordering. This means a duplicate vote doesn't corrupt data—it simply records a newer vote that supersedes the older one during tally.

6. **Tamper Detection**: The `is_tampered()` method recomputes the `opened_key` from current data and compares it to the stored key, detecting unauthorized modifications.

---

## Architecture Observations

1. **Synchronous SQLite in Async Handlers**: All database operations are synchronous within Quart's async framework. While this prevents intra-request interleaving at the Python level, it creates event-loop blocking. More importantly, it does NOT prevent inter-request TOCTOU because separate SQL statements release locks between execution.

2. **Missing WAL Mode Configuration**: No `PRAGMA journal_mode=WAL` is visible. Default rollback journal mode severely limits concurrent reads during writes. WAL mode would improve concurrent access.

3. **Missing Foreign Key Pragma**: The schema comments explicitly require `PRAGMA foreign_keys = ON`, but no code enables it, potentially allowing referential integrity violations.

4. **State Machine Without Database Enforcement**: The three-state election model (editable → open → closed) is enforced only at the application layer via `assert` statements. There are no database-level constraints preventing invalid state transitions (e.g., a CHECK constraint or trigger that prevents setting `closed=1` when `salt IS NULL`).

## Recommendations Summary

### Immediate Actions (Critical Priority)

1. **Fix Election Opening Race Condition (ASVS-1541-CRI-001)**
   - Wrap entire `open()` operation in `BEGIN IMMEDIATE TRANSACTION`
   - Move state check inside transaction
   - Add atomic WHERE clause to UPDATE statement
   - Verify rowcount after state transition

### Short-term Actions (High Priority)

2. **Fix Vote Submission TOCTOU (ASVS-1541-HIG-002)**
   - Wrap eligibility check and vote insert in single transaction
   - Re-verify election state inside transaction
   - Use `BEGIN IMMEDIATE` to acquire write lock

3. **Fix Election Close Race Condition (ASVS-1541-HIG-003)**
   - Add atomic WHERE clause checking current state
   - Verify rowcount after UPDATE
   - Combine with vote submission fix to eliminate race window

4. **Replace Assert Statements (ASVS-1541-MED-004)**
   - Replace all security-critical `assert` with `if/raise`
   - Add unit tests verifying behavior with `-O` flag
   - Document that assertions should never be used for security checks

### Long-term Actions (Medium Priority)

5. **Add Batch Vote Transaction (ASVS-1541-MED-005)**
   - Wrap multi-issue vote loop in single transaction
   - Implement all-or-nothing semantics
   - Improve error messages to clarify transaction rollback

6. **Enable WAL Mode**
   - Add `PRAGMA journal_mode=WAL` at connection setup
   - Improves concurrent read performance during writes

7. **Enable Foreign Keys**
   - Add `PRAGMA foreign_keys = ON` at connection setup
   - Enforces referential integrity as schema requires

8. **Add Database-Level State Constraints**
   - Add CHECK constraints enforcing valid state transitions
   - Add triggers preventing invalid state changes
   - Example: `closed=1` should require `salt IS NOT NULL`
