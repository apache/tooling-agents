## ASVS-9.1.3: Key Material from Trusted Pre-configured Sources

### Findings for 9.1.3

### [MEDIUM]

**Finding ID:** ASVS-913-MED-001

**Title:** Abstract provider interface stores configuration without key material validation constraints

**File:** `webapp/packages/api/user-service/auth/base.py`, lines 100-107

```python
class AuthProvider(ABC):
    type: str = ""

    def __init__(self, config: dict):
        # Provider-specific config; subclasses read keys they care about.
        # Storing the whole dict makes config debugging easier.
        self.config = config
```

**Analysis:**

The `AuthProvider.__init__()` accepts a raw `config: dict` without any schema validation for key material sources. For OAuth providers that validate JWTs (ID tokens), the configuration should specify:
- Trusted JWKS URIs (not user-controllable)
- Expected issuers (pinned, not from token headers)
- Whether `jku`, `x5u`, `jwk` header claims are ever followed

The abstract interface does not:
1. Define required config keys for key material
2. Validate that key sources are from trusted, pre-configured URIs
3. Prohibit following untrusted key references in token headers

**Impact:** A misconfigured provider could accept key material references from within tokens themselves (e.g., following a `jku` header to an attacker-controlled JWKS endpoint), allowing token forgery.

**Remediation:**

Document and optionally enforce key material requirements:

```python
class AuthProvider(ABC):
    # Required config keys that subclasses must validate in __init__
    REQUIRED_KEY_CONFIG = []  # Subclasses override, e.g., ["jwks_uri", "issuer"]
    
    def __init__(self, config: dict):
        self.config = config
        self._validate_key_config()
    
    def _validate_key_config(self):
        """Ensure key material sources are pre-configured, not dynamic."""
        for key in self.REQUIRED_KEY_CONFIG:
            if key not in self.config:
                raise ValueError(
                    f"Provider {self.type} requires '{key}' in config "
                    f"for trusted key material resolution"
                )
```

**Mitigating factors:** 
- The docstring states "A provider is instantiated once at startup from its slice of `AUTH_CONFIG['providers']`" — configuration comes from a trusted source (config file/environment), not from user input.
- The application's session tokens are opaque and don't contain key references.

## Cross-cutting Architecture Observations

1. **Opaque token design is the primary security control**: The decision to use server-side sessions with opaque random identifiers (`secrets.token_urlsafe(32)`) effectively eliminates most self-contained token vulnerabilities (algorithm confusion, signature bypass, key confusion, header injection). This is an excellent architectural choice that makes ASVS 9.1.x requirements largely not applicable to the session layer itself.

2. **OAuth token validation is delegated to concrete providers**: The abstract interface (`AuthProvider`) defines the contract but doesn't enforce cryptographic validation requirements. Security depends entirely on correct implementation in subclasses. Without seeing concrete providers (Google, ASF, GitHub implementations), we cannot verify that OAuth ID tokens are properly validated.

3. **No cryptographic operations in audited code**: Neither file uses encryption, hashing, or digital signatures directly. The only cryptographic primitive is `secrets.token_urlsafe(32)` for session ID generation, which is the correct choice. ASVS 11.x requirements are largely not applicable to this specific code.

4. **Session integrity model**: The application achieves session integrity through database-backed server-side state rather than cryptographic signatures on tokens. This is valid but requires:
   - Database access controls (only the application can write to `user_sessions`)
   - TLS for cookie transmission (not visible in this code)
   - Proper cookie attributes (HttpOnly, Secure, SameSite — not visible in this code)

5. **Trust boundary at provider implementations**: The `exchange_code()` abstract method is where external tokens (potentially JWTs from OAuth providers) enter the system. This is the critical point where ASVS 9.1.x controls must be applied, but enforcement is not visible in the abstract base class.

## Cross-cutting Recommendations

### Immediate (before next release)

1. **Document cryptographic requirements for provider implementers**: Add explicit documentation to `AuthProvider.exchange_code()` requiring signature validation, algorithm allowlisting, and trusted key sources for any JWT/ID tokens received during OAuth code exchange.

2. **Create a shared token validation utility**: Provide a helper function in the `auth` package that enforces algorithm allowlists and trusted JWKS resolution, so concrete providers don't each independently implement (and potentially misconfigure) JWT validation.

### Short-term (next sprint)

3. **Audit concrete provider implementations**: The security of the system depends critically on how concrete providers (not shown in this audit) implement `exchange_code()`. Verify that each:
   - Validates JWT signatures before extracting claims
   - Uses an explicit algorithm allowlist (no "none")
   - Resolves signing keys only from pre-configured JWKS URIs
   - Validates issuer, audience, and expiry claims

4. **Verify cookie security attributes**: Confirm that the `gofannon_sid` cookie is set with `HttpOnly`, `Secure`, and appropriate `SameSite` attributes at the route handler level (not visible in the audited service code).

### Long-term (architectural)

5. **Consider session binding**: For high-security environments, consider binding sessions to client fingerprints (TLS channel binding, user-agent hashing) to mitigate session token theft.

6. **Add interface-level validation hooks**: Consider making `AuthProvider` enforce validation through template method patterns rather than relying on documentation alone — e.g., a `validate_provider_token()` method that the base class calls with assertions on algorithm and issuer.
