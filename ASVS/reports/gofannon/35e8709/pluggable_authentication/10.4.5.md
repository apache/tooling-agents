### Medium Severity

#### ASVS-1045-MED-001: Open Redirect via `return_to` Parameter

**Severity:** MEDIUM  
**CWE:** CWE-601 (URL Redirection to Untrusted Site)

**Location:**
- **File:** `webapp/packages/api/user-service/routes_auth.py`
- **Function:** `login_callback`
- **Lines:** ~170-180

**Related Functions Checked:**
- `login_redirect` (sets `return_to` cookie)
- `_validate_return_to` (not present - missing validation)

**Description:**

The `return_to` parameter allows arbitrary absolute URLs to be used as post-authentication redirect targets without validation against an allowlist. This creates an open redirect vulnerability that can be exploited for phishing attacks.

**Vulnerable Code:**

```python
raw_target = return_to or "/"
if raw_target.startswith(("http://", "https://")):
    redirect_url = raw_target  # ← No validation against allowed origins
else:
    frontend_base = os.getenv("FRONTEND_URL", "http://localhost:3000").rstrip("/")
    if not raw_target.startswith("/"):
        raw_target = "/" + raw_target
    redirect_url = frontend_base + raw_target
```

**Data Flow:**

1. **Source:** `return_to` query parameter → `GET /auth/login/{provider_type}?return_to=https://evil.com`
2. **Storage:** Set as `gofannon_return_to` cookie (httponly) in `login_redirect`
3. **Processing:** Retrieved from cookie in `login_callback` after OAuth flow completes
4. **Sink:** Used directly as redirect target with `RedirectResponse(url=redirect_url)`
5. **Missing Control:** No allowlist validation of absolute URLs

**Proof of Concept:**

```http
GET /auth/login/github?return_to=https://attacker.com/phish HTTP/1.1
Host: app.example.com
```

After successful OAuth authentication, the user is redirected to:
```
https://attacker.com/phish
```

**Attack Scenario:**

1. Attacker crafts a phishing link: `https://app.example.com/auth/login/github?return_to=https://evil-app.example.com/fake-2fa`
2. Victim clicks the link and completes legitimate OAuth authentication
3. After successful authentication, victim is redirected to attacker's site
4. Attacker's page displays "Please re-enter your password for additional verification"
5. Victim trusts the request because they just completed a legitimate sign-in flow

**Impact:**

- **Post-authentication phishing:** Users trust redirects after successful authentication
- **Credential theft:** Attackers can serve fake credential collection pages
- **Session hijacking:** If attacker domain is similar, users may not notice the switch
- **Reputation damage:** Legitimate application facilitates phishing attacks

**Remediation:**

```python
import urllib.parse

ALLOWED_REDIRECT_ORIGINS = {
    os.getenv("FRONTEND_URL", "http://localhost:3000").rstrip("/")
}

def _validate_return_to(raw_target: str, request: Request) -> str:
    """Validate and resolve return_to, preventing open redirect."""
    if not raw_target or raw_target == "/":
        frontend_base = os.getenv("FRONTEND_URL", "http://localhost:3000").rstrip("/")
        return frontend_base + "/"
    
    if raw_target.startswith(("http://", "https://")):
        parsed = urllib.parse.urlparse(raw_target)
        origin = f"{parsed.scheme}://{parsed.netloc}"
        if origin not in ALLOWED_REDIRECT_ORIGINS:
            # Reject absolute URLs to non-allowed origins
            logger.warning(f"Rejected redirect to untrusted origin: {origin}")
            frontend_base = os.getenv("FRONTEND_URL", "http://localhost:3000").rstrip("/")
            return frontend_base + "/"
        return raw_target
    
    # Relative path — safe to prefix with frontend base
    frontend_base = os.getenv("FRONTEND_URL", "http://localhost:3000").rstrip("/")
    if not raw_target.startswith("/"):
        raw_target = "/" + raw_target
    return frontend_base + raw_target

# In login_callback:
redirect_url = _validate_return_to(return_to or "/", request)
```

#### ASVS-1045-MED-002: Reflected XSS in Dev Stub Picker

**Severity:** MEDIUM  
**CWE:** CWE-79 (Improper Neutralization of Input During Web Page Generation)

**Location:**
- **File:** `webapp/packages/api/user-service/routes_auth.py`
- **Function:** `dev_stub_picker`
- **Lines:** ~253-266

**Related Functions Checked:**
- `_render_deny_page` (properly uses `html.escape()`)
- Provider `exchange_code` methods (handle `state` parameter)

**Description:**

The dev stub picker endpoint generates HTML by directly interpolating unsanitized query parameters (`state` and `users`) into the response, creating a reflected XSS vulnerability.

**Vulnerable Code:**

```python
@router.get("/auth/dev-stub-picker", response_class=HTMLResponse)
async def dev_stub_picker(
    state: str = Query(...),
    redirect_uri: str = Query(...),
    users: str = Query(default=""),
) -> HTMLResponse:
    user_list = [u for u in users.split(",") if u]
    if not user_list:
        return HTMLResponse(content="<h1>No dev_stub users configured</h1>", status_code=500)

    links = "".join(
        f'<li><a href="/auth/callback/dev_stub?code={u}&state={state}">{u}</a></li>'
        for u in user_list
    )
    html = f"""<!doctype html>
<html>
  <head><title>Dev Stub Login</title></head>
  <body>
    <h1>Select User</h1>
    <ul>{links}</ul>
  </body>
</html>"""
    return HTMLResponse(content=html)
```

**Data Flow:**

1. **Source:** `state` and `users` query parameters from URL
2. **Processing:** Split `users` by comma, iterate to build HTML
3. **Sink:** Direct string interpolation into HTML without encoding
4. **Missing Control:** No HTML entity encoding applied to `state` or user values

**Proof of Concept:**

```http
GET /auth/dev-stub-picker?state=%22%3E%3Cimg+src=x+onerror=alert(document.cookie)%3E&redirect_uri=/&users=alice HTTP/1.1
Host: app.example.com
```

Decoded: `state="><img src=x onerror=alert(document.cookie)>`

**Generated HTML:**

```html
<a href="/auth/callback/dev_stub?code=alice&state="><img src=x onerror=alert(document.cookie)>">alice</a>
```

**Attack Scenario:**

1. Attacker sends victim link to dev stub picker with malicious `state` parameter
2. Victim's browser renders the page with injected JavaScript
3. JavaScript executes with victim's session context
4. Attacker can steal session cookies, perform actions as the victim, or redirect to phishing page

**Impact:**

- **Session hijacking:** If httponly flag is not set on all cookies, attacker can steal session
- **Account takeover:** Attacker can perform authenticated actions as the victim
- **Phishing:** Inject fake login forms or redirect to malicious sites
- **Limited scope:** Only affects deployments with dev_stub enabled, but code exists in production codebase

**Remediation:**

```python
import html as _html
from urllib.parse import quote

@router.get("/auth/dev-stub-picker", response_class=HTMLResponse)
async def dev_stub_picker(
    state: str = Query(...),
    redirect_uri: str = Query(...),
    users: str = Query(default=""),
) -> HTMLResponse:
    user_list = [u for u in users.split(",") if u]
    if not user_list:
        return HTMLResponse(
            content="<h1>No dev_stub users configured</h1>", 
            status_code=500
        )

    # URL-encode parameters for href attribute
    safe_state = quote(state, safe='')
    
    # HTML-escape user values for display
    links = "".join(
        f'<li><a href="/auth/callback/dev_stub?code={quote(u, safe="")}&state={safe_state}">'
        f'{_html.escape(u)}</a></li>'
        for u in user_list
    )
    
    html = f"""<!doctype html>
<html>
  <head>
    <title>Dev Stub Login</title>
    <meta charset="UTF-8">
  </head>
  <body>
    <h1>Select User</h1>
    <ul>{links}</ul>
  </body>
</html>"""
    return HTMLResponse(content=html)
```

Add environment-based guard to prevent dev_stub from loading in production:

### Low Severity

#### ASVS-1045-LOW-003: OAuth Access Tokens Stored on Mutable Object Without Clear Lifecycle

**Severity:** LOW  
**CWE:** CWE-922 (Insecure Storage of Sensitive Information)

**Location:**
- **Files:** `auth/providers/github.py`, `auth/providers/google.py`, `auth/providers/microsoft.py`
- **Functions:** `exchange_code` methods
- **Lines:** github.py:~153, google.py:~172, microsoft.py:~156

**Related Functions Checked:**
- All provider `exchange_code` implementations
- `routes_auth.py:login_callback` (token usage scope)
- Session management functions (tokens not persisted to session)

**Description:**

OAuth access tokens from upstream providers are attached to `UserInfo` dataclass objects as ad-hoc private attributes using dynamic attribute assignment. While these tokens appear to be short-lived and scoped to a single request, the lifecycle is implicit and the pattern lacks explicit cleanup.

**Vulnerable Code Pattern:**

```python
# github.py line ~153
user_info._access_token = access_token  # type: ignore[attr-defined]
user_info._login = login  # type: ignore[attr-defined]

# google.py line ~172
user_info._access_token = access_token  # type: ignore[attr-defined]

# microsoft.py line ~156
user_info._access_token = access_token  # type: ignore[attr-defined]
```

**Data Flow:**

1. **Source:** OAuth provider token endpoint response
2. **Storage:** Attached as private attribute to `UserInfo` dataclass
3. **Usage:** Used for subsequent membership/group queries within same request
4. **Cleanup:** Implicit via garbage collection when request completes
5. **Risk:** If `UserInfo` object is ever serialized (logged, cached, stored), token leaks

**Concern:**

The `# type: ignore[attr-defined]` annotations indicate the developers recognize this as a design smell. While current code review shows tokens are not persisted beyond the request scope, the pattern creates risk if:

- Future code adds logging that serializes `UserInfo` objects
- Caching is introduced for user information
- Session storage accidentally includes the `UserInfo` object
- Debug endpoints expose object state

**Current Mitigations:**

✅ Tokens are not stored in sessions (verified in `login_callback`)  
✅ Tokens appear to be request-scoped only  
✅ No evidence of `UserInfo` serialization in current codebase  
✅ Access tokens are short-lived (typical OAuth token lifetime)  

**Impact:**

- **Low likelihood:** No current code path persists these tokens
- **Medium severity if exploited:** Access tokens could be used to query provider APIs
- **Limited scope:** Tokens only grant access to user's own data at provider
- **Time-limited:** Tokens expire according to provider policy

**Remediation:**

Use an explicit context object to separate concerns:

```python
from dataclasses import dataclass
from typing import Optional

@dataclass
class ExchangeResult:
    """Result of OAuth code exchange, including ephemeral access token."""
    user_info: UserInfo
    access_token: str  # Not persisted; used only for membership queries
    
    def __del__(self):
        # Explicit cleanup (defense in depth)
        self.access_token = ""

# In provider exchange_code methods:
async def exchange_code(self, code: str, redirect_uri: str) -> UserInfo:
    # ... existing token exchange logic ...
    
    result = ExchangeResult(
        user_info=user_info,
        access_token=access_token
    )
    
    # Use token for membership queries
    memberships = await self._fetch_memberships(result.access_token)
    result.user_info.memberships = memberships
    
    # Return only UserInfo, token goes out of scope
    return result.user_info
```

**Alternative (Minimal Change):**

Add explicit token cleanup:

```python
# At end of exchange_code, after all token usage:
try:
    delattr(user_info, '_access_token')
except AttributeError:
    pass
```

## Positive Security Patterns

The codebase demonstrates several strong security practices:

### 1. **Constant-Time State Comparison**
- **Location:** `routes_auth.py:login_callback`
- **Implementation:** `secrets.compare_digest(state, expected_state)`
- **Benefit:** Prevents timing attacks on OAuth state parameter validation

### 2. **CSRF Protection via State + Cookie Binding**
- **Location:** `routes_auth.py:login_redirect` and `login_callback`
- **Implementation:** State generated server-side with `secrets.token_urlsafe(24)`, stored in httponly cookie, verified on callback
- **Benefit:** Robust CSRF protection for OAuth flow

### 3. **Secure Cookie Defaults**
- **Location:** `routes_auth.py:_is_secure_cookie` and session cookie setting
- **Implementation:** Auto-detects HTTPS for Secure flag; always sets httponly + samesite=lax
- **Benefit:** Protects session cookies from XSS and CSRF attacks

### 4. **Server-Side Session Architecture**
- **Location:** Overall architecture
- **Implementation:** Session ID is opaque; all authorization state is server-side; no JWT/bearer tokens
- **Benefit:** Inherently prevents token replay attacks; immediate revocation capability

### 5. **Hosted Domain Enforcement**
- **Location:** `google.py:exchange_code`
- **Implementation:** Server-side validation of `hd` claim in ID token
- **Benefit:** Prevents authorization bypass by not trusting client-side authorize parameters alone

### 6. **Banned User Hard Deny**
- **Location:** `asf.py:evaluate_login`
- **Implementation:** Ban check overrides even site-admin privileges
- **Benefit:** Defense in depth; prevents privilege escalation via admin status

### 7. **Provider Configuration Fail-Fast**
- **Location:** All provider `__init__` methods
- **Implementation:** Missing secrets raise exceptions at startup
- **Benefit:** Prevents runtime failures; ensures proper configuration before accepting traffic

### 8. **HTML Escaping in Error Pages**
- **Location:** `routes_auth.py:_render_deny_page`
- **Implementation:** Uses `html.escape()` for user-facing deny messages
- **Benefit:** Prevents XSS in error handling paths

### 9. **HTTP Client Timeouts**
- **Location:** All provider modules
- **Implementation:** `httpx.AsyncClient(timeout=10.0)`
- **Benefit:** Prevents hanging requests and resource exhaustion

### 10. **Soft-Fail on LDAP Outage**
- **Location:** `asf.py`
- **Implementation:** LDAP unavailability doesn't crash login; preserves existing memberships
- **Benefit:** Graceful degradation; maintains availability during dependency failures

## Architecture Observations

### 1. Application Role: OAuth Client, Not Authorization Server

**Evidence of OAuth Client Role:**
- ✅ Redirects users to external IdPs via `get_authorize_url`
- ✅ Receives authorization codes at callback endpoints
- ✅ Exchanges codes for tokens at provider token endpoints
- ✅ Uses tokens for one-time user info/membership queries during login
- ✅ Creates its own server-side sessions (not OAuth tokens)

**Evidence Authorization Server Functions Are Absent:**
- ❌ No token endpoint (`/token`) for issuing tokens
- ❌ No authorization endpoint for issuing authorization codes
- ❌ No refresh token issuance or management
- ❌ No client registration functionality
- ❌ No token introspection or revocation endpoints

**Conclusion:** ASVS 10.4.5 is **NOT APPLICABLE** to this codebase. The requirement specifically addresses authorization servers that issue refresh tokens to public clients. This application exclusively acts as an OAuth client/relying party.

### 2. Session Management Model

The application uses **server-side sessions with opaque cookie identifiers** (`gofannon_sid`), not bearer tokens:

```python
# routes_auth.py - Session creation (not token issuance)
resp.set_cookie(
    key=SessionService.cookie_name(),
    value=session.id,  # Opaque identifier
    max_age=int((session.expires_at - session.created_at).total_seconds()),
    httponly=True,
    samesite="lax",
    secure=_is_secure_cookie(request),
    path="/",
)
```

**Security Properties:**
1. Session ID is an opaque reference to server-side state (not a self-contained token)
2. No refresh token exists to replay — session validity checked server-side on each request
3. Session revocation is immediate (server-side delete)
4. No token signature validation or expiry checking needed client-side

This architecture **inherently avoids refresh token replay attacks** because there are no refresh tokens in the system.

### 3. Token Scope and Lifecycle

**Upstream OAuth Tokens (from providers):**
- Requested without `offline_access` scope
- Used once during login for membership queries
- Discarded after login flow completes
- Never stored in sessions or persisted to database
- Not exposed to frontend or downstream services

**Session Tokens (application-managed):**
- Opaque identifiers with no embedded claims
- Stored server-side only
- Revoked via database delete operation
- Cannot be replayed because validity is checked server-side

### 4. `/auth/refresh-workspaces` Endpoint Clarification

This endpoint refreshes **workspace membership data**, not OAuth tokens:
- Relies on existing valid session
- Re-queries provider APIs for current group memberships
- Updates server-side session state
- Does not involve token refresh grant flow

### 5. Future Considerations

If the system ever needs to act as an Authorization Server (e.g., issuing tokens to a mobile app or SPA as a public client), ASVS 10.4.5 would become applicable and would require:

- **Preferred:** DPoP (RFC 9449) or mTLS (RFC 8705) for sender-constrained refresh tokens
- **Minimum (L1/L2):** Refresh token rotation with:
  - Invalidation on use
  - Revocation of all refresh tokens in family if invalidated token is presented
  - Detection of replay attempts

#### 1. Open Redirect Fix (ASVS-1045-MED-001)

```python
# Add to routes_auth.py
import urllib.parse

def _get_allowed_redirect_origins() -> set[str]:
    """Get set of allowed redirect origins from configuration."""
    frontend_url = os.getenv("FRONTEND_URL", "http://localhost:3000").rstrip("/")
    allowed = {frontend_url}
    
    # Optional: Support multiple allowed origins
    extra_origins = os.getenv("ALLOWED_REDIRECT_ORIGINS", "")
    if extra_origins:
        allowed.update(o.strip().rstrip("/") for o in extra_origins.split(",") if o.strip())
    
    return allowed

def _validate_return_to(raw_target: str, request: Request) -> str:
    """Validate and resolve return_to parameter, preventing open redirect."""
    frontend_base = os.getenv("FRONTEND_URL", "http://localhost:3000").rstrip("/")
    
    if not raw_target or raw_target == "/":
        return frontend_base + "/"
    
    if raw_target.startswith(("http://", "https://")):
        parsed = urllib.parse.urlparse(raw_target)
        origin = f"{parsed.scheme}://{parsed.netloc}"
        
        if origin not in _get_allowed_redirect_origins():
            logger.warning(
                f"Rejected redirect to untrusted origin: {origin}",
                extra={"remote_addr": request.client.host if request.client else "unknown"}
            )
            return frontend_base + "/"
        
        return raw_target
    
    # Relative path — safe to prefix
    if not raw_target.startswith("/"):
        raw_target = "/" + raw_target
    
    return frontend_base + raw_target

# Update login_callback:
redirect_url = _validate_return_to(return_to or "/", request)
```

# ADR: ASVS 10.4.5 Not Applicable to OAuth Client Architecture

## Status
Accepted

## Context
ASVS 10.4.5 requires OAuth Authorization Servers to implement refresh token 
replay attack mitigation through sender-constrained tokens (DPoP/mTLS) or 
refresh token rotation.

## Decision
This application is an OAuth **Client/Relying Party**, not an Authorization 
Server. It does not issue tokens to downstream clients and therefore ASVS 
10.4.5 is not applicable.

## Architecture
- Uses server-side sessions with opaque cookie identifiers
- Delegates authentication to external OAuth providers (Google, GitHub, etc.)
- Does not request offline_access scope (no refresh tokens obtained)
- Provider access tokens used once during login, then discarded
- Session revocation is immediate (server-side delete)

## Security Posture
The session-based architecture inherently avoids refresh token replay attacks 
because no refresh tokens exist in the system.

## Future Considerations
If the application evolves to act as an Authorization Server (e.g., issuing 
tokens to mobile apps), ASVS 10.4.5 will become applicable and require:
- DPoP (RFC 9449) or mTLS (RFC 8705) for sender-constrained tokens, or
- Refresh token rotation with invalidation-on-reuse

## References
- ASVS 10.4.5
- Audit Report: May 05, 2026

## Conclusion

**ASVS 10.4.5 is NOT APPLICABLE to this application.** The codebase implements an OAuth Client/Relying Party, not an OAuth Authorization Server. No refresh tokens are issued, managed, or rotated by this system.

The session management approach (server-side sessions with opaque cookie identifiers) is architecturally sound and avoids the refresh token replay threat model entirely. The security posture of the OAuth client implementation is generally strong, with proper CSRF protection, cookie security flags, and provider validation.

**Two actionable security findings were identified:**
1. **Open redirect vulnerability** (ASVS-1045-MED-001) - Requires immediate remediation
2. **Reflected XSS in dev stub** (ASVS-1045-MED-002) - Requires immediate remediation

These findings are orthogonal to ASVS 10.4.5 but represent real security issues that should be addressed promptly to maintain the application's security posture.
