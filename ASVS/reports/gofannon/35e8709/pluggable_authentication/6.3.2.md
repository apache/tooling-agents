## ASVS-6.3.2: Default User Accounts

### Findings for 6.3.2

### [MEDIUM]

**Finding ID:** ASVS-632-MED-001

**Title:** Default Site Admin Account Without Hard Production Block

**File:** `.dev-auth.yaml`, lines 37-41 and 43-44

**Vulnerable code:**

```yaml
          - uid: site_admin_1
            display_name: Site Admin
            email: admin@dev.local
            workspaces: []
  site_admins:
    - dev_stub:site_admin_1
```

**Context from** `webapp/packages/api/user-service/auth/providers/dev_stub.py`:
```python
# The ``dev_stub`` provider must never be enabled in production. The
# session service logs a warning if it sees ``dev_stub`` enabled while
# ``APP_ENV`` is set to anything other than ``local``/``dev``/``test``.
```

**Data flow:** `.dev-auth.yaml` loaded as AUTH_CONFIG → `dev_stub` provider initialized with `site_admin_1` → session service only WARNS (does not block) → site admin access granted

**Proof of concept:**
If a deployment accidentally uses the dev config (e.g., `AUTH_CONFIG_PATH` not overridden):
```bash
# Attacker navigates to the dev_stub picker (no credentials needed)
curl "https://target/auth/login/dev_stub"
# Follows redirect to picker, clicks site_admin_1
curl -b "gofannon_auth_state=<state>" \
     "https://target/auth/callback/dev_stub?code=site_admin_1&state=<state>"
# Now has site_admin session - can bypass workspace boundaries
```

**Impact:** If `dev_stub` is accidentally enabled in production (only a logged warning prevents this), an unauthenticated attacker gains full site administrator access with a well-known default account identifier. Site admins can bypass workspace boundaries per the application's design.

**Remediation:**
```python
# In dev_stub provider __init__, hard-fail if APP_ENV is production:
import os

class DevStubProvider(AuthProvider):
    def __init__(self, config: dict):
        app_env = os.getenv("APP_ENV", "local").lower()
        if app_env not in ("local", "dev", "test"):
            raise RuntimeError(
                "FATAL: dev_stub provider cannot be enabled in "
                f"APP_ENV={app_env}. This is a misconfiguration."
            )
        super().__init__(config)
        ...
```

## Cross-cutting Architecture Observations

1. **OAuth Client vs. Authorization Server Scope:** This application functions as an OAuth client, not an authorization server. ASVS 10.4.x requirements primarily target authorization servers. The application correctly delegates code issuance, single-use enforcement, and lifetime management to external providers (Google, GitHub, Microsoft, ASF). The dev_stub provider is the only component that exhibits authorization-server-like behavior, and it explicitly violates multiple authorization server requirements by design.

2. **Defense-in-Depth Gap:** The application relies heavily on external OAuth providers for security properties (code single-use, lifetime, brute-force protection on credentials) but adds no application-layer defense for scenarios where those external controls might be insufficient or misconfigured. The CSRF state validation is the only application-layer control protecting the OAuth callback.

3. **Dev/Production Boundary Weakness:** The separation between development and production configurations relies on documentation and environment variable overrides rather than code-level enforcement. The dev_stub provider with its default accounts can be enabled in production with only a logged warning as consequence. This represents a systemic risk where a single configuration mistake (forgetting to set `AUTH_CONFIG_PATH`) grants unauthenticated access with site-admin privileges.

4. **Token Stashing Pattern:** Multiple providers attach `_access_token` as an ad-hoc attribute to the `UserInfo` dataclass. This pattern works but means access tokens exist in memory attached to objects that may be serialized, logged, or passed through middleware. The tokens' lifecycle is tied to the request/response cycle with no explicit cleanup.

5. **Soft-Fail Posture:** The system consistently soft-fails (returns empty memberships) when external services are unavailable (LDAP, GitHub API, Google Admin SDK). While this prevents outages, it means a sustained external service disruption silently degrades security posture (users lose workspace access rather than being explicitly denied).

## Cross-cutting Recommendations

### Immediate (Critical/High severity, implement before next release)

1. **Add rate limiting to authentication endpoints** — Deploy a rate-limiting middleware (e.g., SlowAPI, Redis-backed limiter) on `/auth/login/*`, `/auth/callback/*`, and `/auth/dev-stub-picker`. Suggested limits: 10 login initiations/minute/IP, 5 callback attempts/minute/IP.

2. **Fix open redirect in `return_to`** — Validate `return_to` against a configured allowlist of origins (at minimum, the `FRONTEND_URL` origin). Reject absolute URLs pointing to external domains.

3. **Fix XSS in dev_stub_picker** — HTML-escape `state` and `users` values before embedding in HTML output. Apply URL-encoding in `href` attributes.

### Short-term (Medium severity, implement within 30 days)

4. **Hard-block dev_stub in production** — Change the session service warning to a hard failure: if `APP_ENV` is not in `{local, dev, test}`, refuse to initialize the dev_stub provider. This prevents accidental deployment of default accounts.

5. **Use configured base URL for redirect_uri** — Replace `request.base_url` in `_default_redirect_uri` with a configured environment variable (`APP_BASE_URL`) to prevent Host header manipulation from influencing the OAuth redirect_uri.

6. **Add authentication attempt auditing** — Log all login attempts (successful and failed) with IP, timestamp, provider type, and outcome. This enables detection of credential stuffing and supports incident response.

### Long-term (Architectural improvements)

7. **Implement PKCE (Proof Key for Code Exchange)** — Add PKCE to the OAuth flows for Google, GitHub, and Microsoft providers. This provides additional protection against authorization code interception, particularly relevant for the ASF provider's non-standard flow.

8. **Formalize token lifecycle** — Replace the ad-hoc `_access_token` attribute stashing with an explicit token context that is cleared after the workspace membership query completes, reducing the window where access tokens exist in memory.

9. **Consider WAF/CDN-level protections** — Document the requirement for upstream rate limiting and bot detection (e.g., Cloudflare, AWS WAF) as part of the deployment security requirements, given that application-layer rate limiting alone may not suffice against distributed attacks.
