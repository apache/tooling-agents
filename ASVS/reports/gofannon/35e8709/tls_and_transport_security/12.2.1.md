## ASVS-12.2.1: HTTPS Communication with External Facing Services

### Findings for 12.2.1

#### HIGH
**Finding ID:** ASVS-1221-HIGH-001

**Title:** No HSTS Header Configuration — Allows Downgrade to HTTP

**File:** `webapp/packages/api/user-service/app_factory.py`, function `create_app()`, lines ~56-83

**Description:** The application does not set the `Strict-Transport-Security` (HSTS) header. Without HSTS, browsers will allow HTTP connections on first visit or after header expiry, enabling protocol downgrade attacks (SSL stripping). No middleware for HSTS is added to the middleware stack.

**Vulnerable Code (absence):**
```python
def create_app() -> FastAPI:
    """Create and configure a FastAPI application instance."""
    app = FastAPI(lifespan=lifespan)
    app.add_middleware(ObservabilityMiddleware)
    _configure_cors(app)
    # No HSTS middleware or response header injection
    # Middleware stack: ObservabilityMiddleware, CORSMiddleware — nothing else
```

**Data Flow:** Client HTTP request → (No HSTS enforcement) → Response without `Strict-Transport-Security` header → Client may retry over HTTP

**Impact:** An attacker performing a man-in-the-middle attack can use SSL stripping (e.g., via `sslstrip`) to intercept the initial plaintext HTTP request before a user's browser upgrades to HTTPS. Session cookies, authentication tokens, and sensitive data could be intercepted.

**Proof of Concept:**
```bash
# Response headers will not include HSTS:
curl -I https://api.example.com/health
# Missing: Strict-Transport-Security: max-age=31536000; includeSubDomains
```

**Remediation:**
```python
from starlette.middleware.base import BaseHTTPMiddleware
from starlette.requests import Request
from starlette.responses import Response

class HSTSMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        response = await call_next(request)
        response.headers["Strict-Transport-Security"] = (
            "max-age=31536000; includeSubDomains; preload"
        )
        return response

def create_app() -> FastAPI:
    app = FastAPI(lifespan=lifespan)
    app.add_middleware(HSTSMiddleware)
    app.add_middleware(ObservabilityMiddleware)
    _configure_cors(app)
    # ...
```

#### MEDIUM
**Finding ID:** ASVS-1221-MED-001

**Title:** Default CORS Origin Uses HTTP Scheme — No HTTPS Enforcement for Frontend Communication

**File:** `webapp/packages/api/user-service/app_factory.py`, function `_configure_cors()`, line ~28

**Description:** The default `FRONTEND_URL` environment variable is `http://localhost:3000` (plaintext HTTP). While this is a development default, the CORS configuration directly uses this value as the allowed origin without validating that it uses HTTPS in production deployments.

**Vulnerable Code:**
```python
def _configure_cors(app: FastAPI) -> None:
    frontend_url = os.getenv("FRONTEND_URL", "http://localhost:3000")
    allowed_origins = [frontend_url]
    print(f"Configured allowed CORS origins: {allowed_origins}")
```

**Data Flow:** Environment variable (or default `http://`) → CORS allowed_origins → Browsers allow cross-origin requests from HTTP origins

**Impact:** If `FRONTEND_URL` is accidentally left as default or configured with `http://` in production, the CORS policy explicitly permits cross-origin requests from an insecure origin. This means:
1. Browsers will allow the insecure origin to make credentialed requests
2. There's no programmatic check ensuring the frontend communicates over HTTPS

**Proof of Concept:**
```bash
# If FRONTEND_URL env var is unset in production:
# The API will accept requests from http://localhost:3000
# If set to http://app.example.com instead of https://app.example.com:
curl -H "Origin: http://app.example.com" https://api.example.com/endpoint
# Response: Access-Control-Allow-Origin: http://app.example.com
```

**Remediation:**
```python
def _configure_cors(app: FastAPI) -> None:
    frontend_url = os.getenv("FRONTEND_URL", "http://localhost:3000")
    
    # Warn if non-HTTPS origin in non-development environment
    env = os.getenv("ENVIRONMENT", "development")
    if env != "development" and not frontend_url.startswith("https://"):
        raise ValueError(
            f"FRONTEND_URL must use HTTPS in {env} environment. "
            f"Got: {frontend_url}"
        )
    
    allowed_origins = [frontend_url]
```

#### MEDIUM
**Finding ID:** ASVS-1221-MED-002

**Title:** No HTTP-to-HTTPS Redirect Middleware

**File:** `webapp/packages/api/user-service/app_factory.py`, function `create_app()`, lines ~56-83

**Description:** The application does not include Starlette's `HTTPSRedirectMiddleware` or any equivalent mechanism to redirect plaintext HTTP requests to HTTPS. If the application receives HTTP traffic (e.g., due to infrastructure misconfiguration or direct access), it will serve responses over plaintext.

**Vulnerable Code (absence):**
```python
def create_app() -> FastAPI:
    """Create and configure a FastAPI application instance."""
    app = FastAPI(lifespan=lifespan)
    app.add_middleware(ObservabilityMiddleware)
    _configure_cors(app)
    # No HTTPSRedirectMiddleware
```

**Data Flow:** HTTP request → Application processes normally → Plaintext response (no redirect to HTTPS)

**Impact:** If a client connects over HTTP (directly or due to infrastructure misconfiguration), the application will process the request and return sensitive data in plaintext, violating the requirement that communications "do not fall back to insecure or unencrypted communications."

**Remediation:**
```python
from starlette.middleware.httpsredirect import HTTPSRedirectMiddleware

def create_app() -> FastAPI:
    app = FastAPI(lifespan=lifespan)
    
    # Only enforce redirect in production (behind TLS proxy, X-Forwarded-Proto is set)
    if os.getenv("ENVIRONMENT", "development") != "development":
        app.add_middleware(HTTPSRedirectMiddleware)
    
    app.add_middleware(ObservabilityMiddleware)
    _configure_cors(app)
```

## Cross-cutting Architecture Observations

1. **Complete Infrastructure Delegation for TLS**: The application code contains zero TLS-related configuration. All three ASVS requirements (12.1.1, 12.2.1, 12.2.2) are entirely delegated to infrastructure (cloud load balancers, reverse proxies). This is architecturally valid but creates a **Type B gap** — security controls exist (at infra level) but the application cannot verify they're actually applied.

2. **No Defense-in-Depth**: The application has no fallback security mechanisms if infrastructure-level TLS is misconfigured. Missing HSTS, missing HTTPS redirect, and missing origin scheme validation mean a single infrastructure misconfiguration could expose all traffic.

3. **Development Defaults in Production Path**: The `FRONTEND_URL` defaults to `http://localhost:3000` — an HTTP URL. This value flows directly into security-sensitive CORS configuration without any scheme validation, creating risk of accidental HTTP exposure in production.

4. **Observability Gap**: While `ObservabilityMiddleware` is present, there is no logging of TLS-related events (protocol version, certificate status, connection security level). This makes it impossible to detect TLS degradation in production through application-level monitoring.

5. **Single File Limitation**: This audit covers only the application factory. TLS configuration for outbound connections (to CouchDB, Firestore, DynamoDB, LDAP, LLM providers) is not visible in this file and would require inspection of service client configurations.

## Cross-cutting Recommendations

### Immediate (Sprint 0 / Pre-deployment)

1. **Add HSTS middleware** with `max-age=31536000; includeSubDomains` to all responses
2. **Add HTTPS scheme validation** for `FRONTEND_URL` in non-development environments — fail startup if HTTP origin is configured in production
3. **Add `HTTPSRedirectMiddleware`** for production deployments (respecting `X-Forwarded-Proto` from load balancers)

### Short-term (Next 2-4 weeks)

4. **Create a deployment security checklist** that documents TLS requirements and provides verification commands
5. **Add startup TLS verification** that confirms the service's public endpoint is using valid, publicly-trusted certificates
6. **Configure response headers** middleware that sets security headers including HSTS, `X-Content-Type-Options`, and `X-Frame-Options`
7. **Audit outbound connections** (database clients, LDAP, LLM API clients) to verify they enforce TLS 1.2+ and certificate validation

### Long-term (Quarterly)

8. **Implement TLS monitoring** — log TLS protocol version and cipher suite for each connection (available via proxy access logs or application-level inspection)
9. **Add certificate rotation automation** with alerting for expiry within 30 days
10. **Consider OCSP stapling** configuration at the infrastructure level and document verification procedures
11. **Evaluate Encrypted Client Hello (ECH)** support as browser adoption increases
