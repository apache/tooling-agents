## ASVS-3.5.3: HTTP Methods for Sensitive Functionality

### Findings for 3.5.3

### [MEDIUM]

**Finding ID:** ASVS-353-MED-001

**File:** `webapp/packages/api/user-service/app_factory.py` — `_configure_cors()` (line 31)

**Vulnerable Code:**

```python
cors_options = {
    "allow_origins": allowed_origins,
    "allow_credentials": True,
    "allow_methods": ["*"],          # All HTTP methods allowed via CORS
    "allow_headers": ["*"],
}
```

**Data Flow:** CORS preflight response → `Access-Control-Allow-Methods: *` → Browser permits any method cross-origin → No server-side method restriction visible at middleware level.

**Gap Type:** Type A — No visible `Sec-Fetch-*` header validation exists, and no global middleware enforces that state-changing operations use only `POST`/`PUT`/`PATCH`/`DELETE`. The wildcard methods in CORS configuration signals no method restrictions are enforced.

**Proof of Concept:**

Without seeing route definitions, the risk is that:
1. A GET request to a sensitive endpoint could trigger state changes (e.g., `/api/v1/agents/{id}/execute?action=delete`)
2. There is no `Sec-Fetch-*` validation that could distinguish navigation/resource-load GET requests from legitimate API GETs

```html
<!-- Image tag triggers GET request with cookies (if SameSite=None) -->
<img src="https://api.target.com/api/v1/agents/agent123/delete" />

<!-- Or via navigation (works with SameSite=Lax) -->
<a href="https://api.target.com/api/v1/export/all-data">Click for prize</a>
```

**Impact:** If any state-changing or resource-intensive endpoints respond to GET requests, they can be exploited via simple resource loads (`<img>`, `<script>`, `<link>`) or navigation, bypassing CORS entirely since these are not cross-origin script requests.

**Remediation:**

1. Ensure all sensitive endpoints use appropriate HTTP methods at the route level (this likely happens in the route files not shown):

```python
@router.post("/agents/{agent_id}/execute")  # POST, not GET
async def execute_agent(agent_id: str, ...):
    ...

@router.delete("/agents/{agent_id}")  # DELETE, not GET
async def delete_agent(agent_id: str, ...):
    ...
```

2. Add Sec-Fetch-* validation middleware for defense-in-depth:

```python
class SecFetchValidationMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        sec_fetch_site = request.headers.get("sec-fetch-site")
        sec_fetch_mode = request.headers.get("sec-fetch-mode")
        
        # Block cross-site navigation requests to API endpoints
        if (sec_fetch_site == "cross-site" and 
            sec_fetch_mode == "navigate" and
            request.method in {"GET", "HEAD"}):
            return JSONResponse(
                {"detail": "Cross-site navigation to API not allowed"},
                status_code=403,
            )
        return await call_next(request)
```

3. Restrict CORS methods to only those needed:

```python
cors_options = {
    "allow_origins": allowed_origins,
    "allow_credentials": True,
    "allow_methods": ["GET", "POST", "PUT", "PATCH", "DELETE"],  # Explicit list
    "allow_headers": ["Content-Type", "Authorization", "X-Requested-With"],
}
```

## Cross-cutting Architecture Observations

1. **CORS provides false sense of security:** The `CORSMiddleware` in Starlette/FastAPI operates on the response side. It does NOT reject or block requests from non-allowed origins for simple (non-preflight) requests. The application appears to rely solely on CORS for cross-origin protection without additional server-side origin validation or CSRF tokens. This is a systemic architectural gap.

2. **Cookie-based authentication without visible CSRF countermeasures:** The combination of:
   - Server: `allow_credentials: True`
   - Client: `credentials: 'include'` (automatic via fetch interceptor)
   - No CSRF token exchange
   - No custom header requirement
   
   Creates a classic CSRF-vulnerable architecture. The only mitigations would be browser defaults (`SameSite=Lax` on cookies) which are not explicitly configured and cannot be relied upon across all browsers/configurations.

3. **Security header strategy appears delegated:** The absence of HSTS, CSP, X-Frame-Options, and other security headers in the application code suggests these are expected to be configured at the infrastructure level (reverse proxy, CDN). This is a valid architecture but creates a documentation/verification gap.

4. **Fetch interceptor centralizes credential handling:** The `fetchInterceptor.js` provides a single point of control for credential inclusion, which is architecturally sound but does not compensate for missing server-side CSRF protections.

5. **Wildcard `allow_methods` and `allow_headers`** in CORS configuration is more permissive than necessary. While this doesn't directly create vulnerabilities (CORS is a browser feature for protecting users), it signals a permissive security posture and reduces the effectiveness of CORS preflight as a defense mechanism.

## Cross-cutting Recommendations

### Immediate (Critical/High — Deploy within sprint)

1. **Implement CSRF protection middleware** (addresses ASVS 3.5.1 and 3.5.2):
   - Require `Content-Type: application/json` on all state-changing API endpoints (this triggers CORS preflight)
   - OR require a custom header like `X-Requested-With: XMLHttpRequest` on all authenticated requests
   - Update `fetchInterceptor.js` to always include the custom header for API requests

2. **Add server-side Origin validation** for state-changing requests:
   - Validate the `Origin` header against the allowlist before processing POST/PUT/PATCH/DELETE
   - Return 403 if Origin is present but not in the allowlist

### Short-term (Medium — Deploy within 2 weeks)

3. **Add HSTS header** (addresses ASVS 3.4.1):
   - Either via application middleware or reverse proxy configuration
   - Minimum `max-age=31536000; includeSubDomains`
   - Document where HSTS is enforced in the deployment architecture

4. **Tighten CORS configuration** (addresses ASVS 3.4.2 and 3.5.3):
   - Replace `allow_methods: ["*"]` with explicit method list
   - Replace `allow_headers: ["*"]` with explicit header list
   - Require `FRONTEND_URL` to be explicitly set (fail if unset) in production

5. **Explicitly configure SameSite cookie attribute**:
   - Set `SameSite=Lax` or `SameSite=Strict` on session cookies
   - Ensure `Secure` flag is set in production

### Long-term (Low — Deploy within quarter)

6. **Implement Sec-Fetch-* header validation** as defense-in-depth against cross-site attacks that bypass CORS
7. **Create security headers documentation** specifying which headers are set at the application vs. infrastructure level, with deployment validation tests
8. **Add integration tests** that verify CSRF protection by attempting cross-origin simple requests without custom headers and asserting rejection
