## ASVS-7.4.2: Session Termination — Account Disable/Delete

### Findings for 7.4.2

### [HIGH]

**Finding ID:** ASVS-742-HIGH-001

**Title:** No mechanism to terminate all sessions when user account is disabled or deleted

**File:** `webapp/packages/api/user-service/services/session_service.py` — `SessionService` class

**Vulnerable code (absence):**

```python
class SessionService:
    """CRUD over session docs plus helpers for creating + refreshing."""

    def __init__(self, db: DatabaseService):
        self.db = db
        # ...

    async def delete(self, sid: Optional[str]) -> None:
        """Log out / delete a session. Safe on missing sessions."""
        if not sid:
            return
        try:
            self.db.delete(_SESSIONS_COLLECTION, sid)
        except HTTPException as e:
            if e.status_code == 404:
                return
            raise

    # NO METHOD EXISTS for:
    # - delete_all_for_user(user_uid: str)
    # - find_sessions_by_user(user_uid: str)
    # - invalidate_user_sessions(user_uid: str)
```

**Gap Type:** Type A — Entry point with NO control. There is no function, endpoint, or mechanism anywhere in the provided code to find and terminate all sessions belonging to a specific user.

**Data flow:** Admin disables/deletes user account → (no code path exists) → user's existing sessions remain valid in CouchDB until individual TTL expiry (default: 24 hours).

**Proof of concept:**
1. Employee has active sessions on laptop and phone (session IDs `s1`, `s2`)
2. Employee is terminated; admin disables their account
3. Both `s1` and `s2` remain valid in the `user_sessions` collection
4. Former employee continues to access the application for up to 24 hours (default TTL)
5. Any API calls validated via `get_by_id()` succeed because the session documents exist and haven't expired

**Impact:** A disabled or deleted user retains access for up to the session TTL (default 24 hours). In security-critical scenarios (employee termination, compromised account), this window is unacceptable. The application cannot enforce immediate access revocation.

**Remediation:**

```python
# Add to SessionService:

async def terminate_all_for_user(self, user_uid: str) -> int:
    """Terminate all active sessions for a user.
    
    Called when a user account is disabled, deleted, or when
    an admin forces logout of all sessions.
    
    Returns count of sessions terminated.
    """
    # Option 1: CouchDB view-based query
    sessions = self.db.query(
        _SESSIONS_COLLECTION,
        {"user_uid": user_uid}
    )
    count = 0
    for session_doc in sessions:
        try:
            self.db.delete(_SESSIONS_COLLECTION, session_doc["id"])
            count += 1
        except Exception:
            pass
    return count

# Add admin endpoint in routes_auth.py or an admin router:

@router.post("/admin/terminate-user-sessions/{user_uid}")
async def terminate_user_sessions(
    user_uid: str,
    # ... admin auth check ...
    session_svc: SessionService = Depends(_get_session_svc),
):
    """Terminate all sessions for a user (admin action)."""
    count = await session_svc.terminate_all_for_user(user_uid)
    return {"terminated": count}
```

Additionally, consider adding a CouchDB secondary index on `user_uid` in the `user_sessions` collection to support efficient queries.

## Cross-cutting Architecture Observations

1. **Session-per-request validation model is sound**: The reference token architecture with server-side CouchDB storage provides a strong foundation. Invalidation is effective once triggered.

2. **Concurrent sessions are an explicit design choice**: The docstring states sessions are never revoked at login time of another browser. This is documented but creates the 7.2.4 gap where old sessions persist after re-authentication.

3. **No session index by user**: The session service only supports lookup by session ID (the CouchDB `_id`). There's no secondary index or view for finding all sessions belonging to a user, which is the root cause of the 7.4.2 gap.

4. **Open redirect potential in `return_to`**: The callback handler (line 161) passes absolute URLs through without origin validation: `if raw_target.startswith(("http://", "https://")): redirect_url = raw_target`. While the `return_to` originates from an httponly cookie set during login initiation, the original query parameter is user-controlled. This is outside session management scope but worth noting.

5. **Dev stub picker XSS**: The `state` parameter in `dev_stub_picker` is embedded into HTML without escaping (line 247). While `state` is normally generated by `secrets.token_urlsafe()`, the endpoint accepts arbitrary query string input. This is a minor concern for a development-only endpoint.

6. **Legacy Firebase coexistence**: The main `routes.router` is always mounted but auth routes are conditional. The session service only handles Phase B auth. Legacy Firebase sessions presumably use a different mechanism — the security properties of that path should be verified separately.

## Cross-cutting Recommendations

### Immediate (Critical/High priority)

1. **Implement `terminate_all_for_user(user_uid)`** in `SessionService`:
   - Add a CouchDB view/index on `user_uid` in the `user_sessions` collection
   - Expose an admin API endpoint for bulk session termination
   - Wire account disable/delete actions to this method
   - *Addresses:* ASVS-742-HIGH-001

### Short-term (Medium priority)

2. **Terminate existing session on re-authentication**:
   - Read the existing `gofannon_sid` cookie in the callback handler
   - Call `session_svc.delete(existing_sid)` before creating the new session
   - *Addresses:* ASVS-724-MED-001

3. **Add session management admin capabilities**:
   - List active sessions for a user (for support/compliance)
   - Force-terminate specific or all sessions
   - Audit log session termination events

### Long-term (Architecture improvements)

4. **Consider session event bus for real-time revocation**:
   - For scenarios where the 15-minute refresh interval is too slow
   - Publish account-disable events that session validation can subscribe to
   - Useful when workspace membership removal needs immediate effect

5. **Add session metadata tracking**:
   - IP address, user agent, last-active timestamp
   - Enables anomaly detection and informed session management decisions
   - Supports "terminate sessions from unknown locations" use case
