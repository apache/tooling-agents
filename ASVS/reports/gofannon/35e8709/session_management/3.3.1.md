## Findings

### Medium Severity

#### Finding ASVS-331-MED-001: Cookie Names Lack Required `__Secure-` or `__Host-` Prefix

**Severity**: MEDIUM  
**Gap Type**: Type A (Entry point with NO control)

**Location:**
- **File:** `session_service.py`
- **Line:** 35
- **Function:** Cookie name constant definition
- **Related Files:** `routes_auth.py` (lines 114, 122, 188-194)

**Related Functions Checked:**
- `SessionService.cookie_name()` - returns unprefixed name
- `login_redirect()` - sets `gofannon_auth_state` cookie
- `login_callback()` - sets `gofannon_return_to` and `gofannon_sid` cookies

**Description:** 

ASVS 3.3.1 requires that if the `__Host-` prefix is not used for a cookie name, the `__Secure-` prefix **must** be used. All three authentication-related cookies use unprefixed names, removing a browser-enforced security constraint that prevents cookie injection from insecure contexts or subdomains.

**Affected Cookies:**

| Cookie Name | File | Line | Purpose |
|-------------|------|------|---------|
| `gofannon_sid` | `session_service.py` | 35 | Primary session identifier |
| `gofannon_auth_state` | `routes_auth.py` | 114 | OAuth CSRF token |
| `gofannon_return_to` | `routes_auth.py` | 122 | Post-login redirect target |

**Vulnerable Code (Definition):**
```python
# session_service.py, line 35
_COOKIE_NAME = "gofannon_sid"
```

**Vulnerable Code (Usage):**
```python
# routes_auth.py, line 188-194
resp.set_cookie(
    key=SessionService.cookie_name(),  # Returns "gofannon_sid"
    value=session.id,
    max_age=int((session.expires_at - session.created_at).total_seconds()),
    httponly=True,
    samesite="lax",
    secure=_is_secure_cookie(request),
    path="/",
)
```

```python
# routes_auth.py, lines 113-118
resp.set_cookie(
    key="gofannon_auth_state",
    value=state,
    max_age=600,
    httponly=True,
    samesite="lax",
    secure=_is_secure_cookie(request),
)
```

**Data Flow:** 

Session ID generated → stored in cookie named `gofannon_sid` → browser accepts cookie without prefix enforcement → cookie can be set by HTTP responses from same domain (no prefix-based browser protection)

**Attack Scenario:** 

A network attacker (MITM on HTTP traffic) or an attacker controlling a subdomain can inject a `gofannon_sid` cookie with a known value. While the server generates new session IDs post-login (mitigating classic fixation), the lack of prefix means the browser won't reject cookies set over non-HTTPS channels for this name. The `__Secure-` prefix instructs browsers to only accept cookies set with the `Secure` attribute, providing an additional browser-level enforcement layer.

**Impact:** 

Reduced defense-in-depth against cookie injection attacks. The `__Secure-` prefix provides browser-enforced guarantee that the cookie was only set over a secure channel, complementing the server-side `Secure` attribute.

**Remediation:**

```python
# session_service.py - update the cookie name constant
_COOKIE_NAME = "__Secure-gofannon_sid"

# routes_auth.py - update auxiliary cookie names
# In login_redirect():
resp.set_cookie(
    key="__Secure-gofannon_auth_state",
    value=state,
    max_age=600,
    httponly=True,
    samesite="lax",
    secure=True,  # Must be True when using __Secure- prefix
)

# In login_callback() - update cookie alias in parameter:
# expected_state: Optional[str] = Cookie(default=None, alias="__Secure-gofannon_auth_state")

# Update delete_cookie calls:
resp.delete_cookie("__Secure-gofannon_auth_state", path="/")
resp.delete_cookie("__Secure-gofannon_return_to", path="/")
```

#### Finding ASVS-331-MED-002: `Secure` Attribute Conditionally Set Based on Scheme Auto-Detection

**Severity**: MEDIUM  
**Gap Type**: Type B (Control EXISTS but conditionally applied — creates false confidence)

**Location:**
- **File:** `routes_auth.py`
- **Function:** `_is_secure_cookie()`
- **Lines:** 62-68

**Related Functions Checked:**
- `login_redirect()` - line 117: applies conditional Secure to `gofannon_auth_state`
- `login_callback()` - line 125: applies conditional Secure to `gofannon_return_to`
- `login_callback()` - line 192: applies conditional Secure to `gofannon_sid`

**Description:** 

The `Secure` attribute is determined by `_is_secure_cookie()` which checks `request.url.scheme == "https"`. This means in deployments behind TLS-terminating reverse proxies (common in production) where forwarded headers are not properly configured, the session cookie will be sent without the `Secure` flag, allowing it to be transmitted over unencrypted HTTP connections.

**Vulnerable Code:**
```python
# routes_auth.py, lines 62-68
def _is_secure_cookie(request: Request) -> bool:
    """Secure flag is True only when the request is over HTTPS.

    Setting Secure=True on a plain-http dev server makes browsers drop
    the cookie silently. Autodetect rather than hardcode.
    """
    return request.url.scheme == "https"
```

**Applied at:**
- Line 117: `gofannon_auth_state` cookie
- Line 125: `gofannon_return_to` cookie  
- Line 192: `gofannon_sid` session cookie

**Data Flow:** 

HTTP request arrives (possibly via reverse proxy) → `request.url.scheme` checked → if proxy doesn't set `X-Forwarded-Proto` or uvicorn doesn't trust it → `scheme == "http"` → `Secure=False` → cookie sent without Secure flag → cookie transmitted over any HTTP request

**Attack Scenario:** 

1. Production deployment behind nginx/ALB terminates TLS at the load balancer
2. Internal traffic to FastAPI app is HTTP (common pattern)
3. If `--proxy-headers` is not configured in uvicorn, `request.url.scheme` returns `"http"`
4. Session cookie is set without `Secure` flag
5. If user ever visits an HTTP URL on the same domain, the session cookie is exposed in cleartext

**Proof of Concept:**
```
# Request as seen by the application behind a reverse proxy without proper forwarding:
GET /auth/callback/google?code=xxx&state=yyy HTTP/1.1
Host: app.example.com
# Note: no X-Forwarded-Proto header or not trusted

# Response sets cookie WITHOUT Secure:
Set-Cookie: gofannon_sid=abc123; Path=/; HttpOnly; SameSite=Lax
# Missing: Secure
```

**Impact:** 

Session cookies could be transmitted over unencrypted connections in misconfigured production deployments, exposing session IDs to network-level attackers.

**Remediation:**

```python
# Option 1: Environment-driven override (recommended)
def _is_secure_cookie(request: Request) -> bool:
    """Determine if cookies should have Secure flag.
    
    In production, FORCE_SECURE_COOKIES=true ensures the flag is always set
    regardless of detected scheme. Only disable for local development.
    """
    force_secure = os.getenv("FORCE_SECURE_COOKIES", "").lower() in ("true", "1", "yes")
    if force_secure:
        return True
    return request.url.scheme == "https"

# Option 2: If using __Host- prefix (strongest, but requires path="/")
_COOKIE_NAME = "__Host-gofannon_sid"
# __Host- prefix requires: Secure=True, Path=/, no Domain attribute
# Browser enforces these constraints, so Secure MUST be True

# Option 3: Use __Secure- prefix which requires Secure=True at browser level
# This makes it a hard failure if Secure is accidentally omitted
```

## Positive Security Patterns

1. **Strong session ID generation**: `secrets.token_urlsafe(32)` provides 256 bits of entropy — exceeds recommendations.

2. **HttpOnly consistently applied**: All authentication cookies set `httponly=True`, preventing JavaScript access.

3. **SameSite=lax consistently applied**: Provides CSRF protection for top-level navigations while allowing the OAuth callback flow to work.

4. **OAuth state CSRF protection**: Uses `secrets.token_urlsafe(24)` with constant-time comparison (`secrets.compare_digest`).

5. **Server-side session with hard expiry**: Sessions checked for expiry on every access; expired sessions auto-evicted.

6. **Session fixation prevention**: New session ID generated on each login via `_new_session_id()`.

7. **HTML escaping in error pages**: `_render_deny_page` uses `html.escape()` to prevent reflected XSS.

8. **Short-lived state cookies**: OAuth state cookies expire in 10 minutes (`max_age=600`).

---

## Architecture Observations

1. **Development vs. Production concern**: The `_is_secure_cookie` function is designed to support both development (HTTP) and production (HTTPS) without code changes. While practical, it violates the ASVS requirement for unconditional `Secure` attribute enforcement.

2. **Cookie prefix adoption**: Adopting `__Host-gofannon_sid` would be ideal since the session cookie already uses `path="/"` and doesn't set a Domain attribute. The `__Host-` prefix provides the strongest browser-level guarantee (Secure required, Path=/, no Domain).

3. **Reverse proxy dependency**: The security of the `Secure` flag depends entirely on the ASGI server configuration (e.g., uvicorn `--proxy-headers`) and the reverse proxy correctly setting `X-Forwarded-Proto`. This infrastructure dependency should be documented as a deployment requirement.

4. **No cookie name used with `__Secure-` prefix alias in Cookie() dependency**: If cookie names are changed to use `__Secure-` prefix, all `Cookie(alias=...)` parameters in route functions must be updated accordingly.
