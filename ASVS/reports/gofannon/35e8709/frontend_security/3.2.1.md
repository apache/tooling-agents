## ASVS-3.2.1: Unintended Content Interpretation - Context Controls

### Findings for 3.2.1

### [LOW]
**Finding ID:** ASVS-321-LOW-001

**Title:** Fetch interceptor does not enforce `Accept: application/json` header on API requests

**File:** `webapp/packages/webui/src/services/fetchInterceptor.js`, function `installFetchInterceptor`, lines 43-47

**Vulnerable code:**
```javascript
window.fetch = async (input, init = {}) => {
    if (apiMatches(input) && init.credentials === undefined) {
      init = { ...init, credentials: 'include' };
    }
    return originalFetch(input, init);
};
```

**Data flow:** All API requests → fetch interceptor → only `credentials` added → no `Accept` header enforcement → server may respond with unexpected content type

**Description:** The fetch interceptor wraps all API calls to add `credentials: 'include'` but does not enforce an `Accept: application/json` header. While server-side content-type enforcement is the primary defense, sending an explicit `Accept` header provides defense-in-depth by signaling to the server that only JSON responses are expected. If an intermediate proxy or CDN serves cached responses with incorrect content types, the lack of `Accept` header means the client hasn't communicated its expected context.

**Impact:** Minimal direct impact because response bodies are consumed programmatically via `resp.json()` (which would throw on non-JSON content). However, this reduces the signal available for server-side Sec-Fetch validation and content negotiation.

**Remediation:**
```javascript
window.fetch = async (input, init = {}) => {
    if (apiMatches(input)) {
      const headers = new Headers(init.headers || {});
      if (!headers.has('Accept')) {
        headers.set('Accept', 'application/json');
      }
      init = { 
        ...init, 
        headers,
        credentials: init.credentials || 'include' 
      };
    }
    return originalFetch(input, init);
};
```

### Positive Patterns for 3.2.1

1. **Programmatic API consumption only**: All API responses are processed through `resp.json()` (authService.js lines 73, 157, 178), preventing the browser from ever rendering API response bodies as HTML content.

2. **URL-scoped credential injection**: The fetch interceptor carefully scopes credential inclusion to API URLs only (`fetchInterceptor.js:30-42`), preventing session cookies from being sent to third-party origins. The exclusion of protocol-relative URLs (`!url.startsWith('//')`) prevents credential leakage to attacker-controlled domains.

3. **Explicit Accept headers in auth calls**: The sessionAuth implementation explicitly sets `headers: { Accept: 'application/json' }` on critical auth endpoints (`authService.js:68, 152, 177`), signaling expected response context.

4. **No direct content rendering**: None of the provided code opens API responses in new browser contexts (no `window.open()` with API URLs, no `<iframe>` with API endpoints, no blob URL creation from API responses).

## Cross-cutting Architecture Observations

1. **React's inherent XSS protection serves as the primary defense for ASVS 3.2.2**: The codebase consistently uses React's JSX rendering model without any bypasses. This provides systematic protection against content injection across all components in scope.

2. **Server-side responsibility for ASVS 3.2.1**: The majority of ASVS 3.2.1 controls (CSP headers, Sec-Fetch-* validation, Content-Disposition, X-Content-Type-Options) are server-side concerns. The frontend's architecture of exclusively consuming JSON APIs programmatically means the browser never directly renders API response bodies, which is itself a strong contextual control.

3. **Fetch interceptor is a centralized control point**: The `fetchInterceptor.js` provides a single location where all API requests can be augmented with security-relevant headers. Currently it only handles credentials, but it could be extended to enforce Accept headers and other request-context signals.

4. **Defense-in-depth layering**: The combination of HttpOnly cookies (preventing JS access to session tokens), programmatic JSON consumption (preventing HTML rendering of API responses), and React's auto-escaping (preventing DOM injection) creates multiple layers of protection against content interpretation attacks.

## Cross-cutting Recommendations

**Immediate (no findings requiring immediate action):**
- No critical or high-severity findings identified in the provided frontend code for these requirements.

**Short-term:**
1. **Extend fetch interceptor to set `Accept: application/json`** on all API requests (Finding ASVS-321-LOW-001). This provides defense-in-depth and enables server-side content negotiation validation.
2. **Document the webui serving configuration**: Verify that the static file server (nginx, CDN, or build tool) serving the React SPA sets appropriate security headers including `X-Content-Type-Options: nosniff`, `Content-Security-Policy`, and proper `Content-Type` for all served assets.

**Long-term:**
1. **Add a project-wide lint rule** prohibiting `dangerouslySetInnerHTML` usage without security review (e.g., eslint-plugin-react's `react/no-danger` rule) to maintain the positive pattern observed in ASVS 3.2.2.
2. **Consider response content-type validation** in the fetch interceptor — verifying that API responses actually have `Content-Type: application/json` before parsing, to detect and log potential MITM or misconfiguration scenarios.
