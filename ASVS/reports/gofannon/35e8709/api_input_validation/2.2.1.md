## ASVS-2.2.1: Input Validation

### Findings for 2.2.1

### [HIGH]
**Finding ID**: ASVS-221-HIGH-001  
**Title**: No schema validation on deployed agent input  
**Location**: `routes.py`, `run_deployed_agent_route`, line ~395

```python
@router.post("/rest/{friendly_name}")
async def run_deployed_agent_route(
    friendly_name: str, 
    request: Request, 
    db: DatabaseService = Depends(get_db),
    user: dict = Depends(get_current_user)
):
    """Run a deployed agent by its friendly_name. Requires authentication."""
    input_dict = await request.json()
    user_basic_info = {
        "email": user.get("email"),
        "name": user.get("name") or user.get("displayName"),
    }
    return await run_deployed_agent_logic(
        friendly_name, 
        input_dict, 
        db, 
        user_id=user.get("uid"),
        user_basic_info=user_basic_info,
    )
```

**Data flow**: HTTP POST body → `request.json()` (raw parse, no model) → `run_deployed_agent_logic` → agent code execution  
**Proof of concept**: 
```http
POST /rest/my-agent HTTP/1.1
Content-Type: application/json

{"__proto__": {"admin": true}, "constructor": {"prototype": {"isAdmin": true}}, "deeply": {"nested": {"payload": "A".repeat(10000000)}}}
```
**Issue**: The deployed agent endpoint reads the request body as raw JSON without any Pydantic model validation. Each deployed agent has a defined `input_schema` stored in its configuration, but this schema is never validated against the incoming request. This is a Type B gap — the control (input_schema) EXISTS but is NOT CALLED at this entry point.  
**Impact**: Arbitrarily structured or sized payloads reach agent execution. Could enable denial of service via oversized inputs, unexpected behavior in agent code, or injection if agent code uses input values in unsafe contexts (e.g., string formatting into shell commands or queries).  
**Remediation**: Validate `input_dict` against the agent's declared `input_schema` before execution:

```python
from jsonschema import validate, ValidationError

@router.post("/rest/{friendly_name}")
async def run_deployed_agent_route(...):
    input_dict = await request.json()
    # Fetch agent and validate input against declared schema
    deployment = await get_agent_deployment(friendly_name, db)
    agent_doc = db.get("agents", deployment["agent_id"])
    if agent_doc.get("inputSchema"):
        try:
            validate(instance=input_dict, schema=agent_doc["inputSchema"])
        except ValidationError as e:
            raise HTTPException(status_code=422, detail=f"Input validation failed: {e.message}")
    ...
```

### [MEDIUM]
**Finding ID**: ASVS-221-MED-001  
**Title**: Provider and model fields not validated against allowed values  
**Location**: `models/chat.py`, `ChatRequest`, line ~44

```python
class ChatRequest(BaseModel):
    messages: List[ChatMessage]
    provider: str = "openai"
    model: str = "gpt-3.5-turbo"
    parameters: Dict[str, Any] = {}
```

**Data flow**: Client request → `ChatRequest.provider`/`model` → `process_chat` → LLM provider selection  
**Proof of concept**:
```json
{"messages": [{"role": "user", "content": "hello"}], "provider": "../../../../etc", "model": "../secret"}
```
**Issue**: The `provider` and `model` fields accept arbitrary strings without validation against the known set of providers from `PROVIDER_CONFIG`. The `_ensure_mutually_exclusive` validator accesses `PROVIDER_CONFIG.get(provider, {})` which silently returns empty for unknown providers, bypassing all parameter validation. This is a business logic validation gap — the application has a known set of allowed providers but doesn't enforce it at input time.  
**Impact**: Unknown provider/model combinations bypass parameter validation entirely; could cause confusing errors deep in processing rather than clear 422 rejections; provider selection logic may be exploitable depending on downstream implementation.  
**Remediation**: Add a model validator to check against known providers:

```python
@model_validator(mode="after")
def validate_provider_model(self):
    if self.provider not in PROVIDER_CONFIG:
        raise ValueError(f"Unknown provider: {self.provider}")
    models = PROVIDER_CONFIG[self.provider].get("models", {})
    if self.model not in models:
        raise ValueError(f"Unknown model '{self.model}' for provider '{self.provider}'")
    return self
```

### [MEDIUM]
**Finding ID**: ASVS-221-MED-002  
**Title**: No range validation on business-critical numeric inputs  
**Location**: `routes.py` (UpdateMonthlyAllowanceRequest line ~138, UpdateSpendRemainingRequest line ~148, AddUsageRequest line ~153), `models/agent.py` (LlmSettings line ~107)

```python
class UpdateMonthlyAllowanceRequest(BaseModel):
    monthly_allowance: float = Field(..., alias="monthlyAllowance")

class AddUsageRequest(BaseModel):
    response_cost: float = Field(..., alias="responseCost")
```

**Data flow**: Client → Pydantic model (float accepted, no range) → user_service → database  
**Proof of concept**:
```json
{"monthlyAllowance": -999999999.0}
```
or
```json
{"responseCost": -100.0}
```
**Issue**: Financial/billing fields accept any float value including negative numbers and extreme values. A negative `response_cost` would increase the user's remaining spend. A negative `monthly_allowance` has undefined business semantics.  
**Impact**: Users could manipulate their billing by submitting negative costs; extremely large allowance values could overflow downstream calculations.  
**Remediation**:

```python
class UpdateMonthlyAllowanceRequest(BaseModel):
    monthly_allowance: float = Field(..., alias="monthlyAllowance", gt=0, le=100000)

class AddUsageRequest(BaseModel):
    response_cost: float = Field(..., alias="responseCost", ge=0)
```

### [MEDIUM]
**Finding ID**: ASVS-221-MED-003  
**Title**: No length constraints on string inputs that control resource allocation  
**Location**: `models/agent.py` (CreateAgentRequest.name line ~57, .description line ~58), `models/chat.py` (ChatMessage.content line ~43)

```python
class ChatMessage(BaseModel):
    role: str = Field(..., pattern="^(user|assistant|system)$")
    content: str  # No max_length
```

```python
class CreateAgentRequest(BaseModel):
    name: str  # No max_length
    description: str  # No max_length
```

**Data flow**: Client → Pydantic (accepts any length string) → database storage / LLM prompt  
**Proof of concept**:
```json
{"messages": [{"role": "user", "content": "A" * 10000000}]}
```
**Issue**: String fields have no `max_length` constraints. An extremely long `content` field flows into LLM prompts (potential token cost abuse) and an extremely long `name`/`description` stores unbounded data in the database.  
**Impact**: Denial of service through memory exhaustion; excessive LLM token consumption driving up costs; database storage abuse.  
**Remediation**: Add `max_length` constraints appropriate to the business context:

```python
class ChatMessage(BaseModel):
    role: str = Field(..., pattern="^(user|assistant|system)$")
    content: str = Field(..., max_length=100000)
```

### [LOW]
**Finding ID**: ASVS-221-LOW-001  
**Title**: UpdateApiKeyRequest.provider not validated against allowed provider list  
**Location**: `routes.py`, `UpdateApiKeyRequest`, line ~218

```python
class UpdateApiKeyRequest(BaseModel):
    provider: str
    api_key: str
```

**Data flow**: Client → `provider` field → `user_service.update_api_key` → database field name derivation  
**Proof of concept**:
```json
{"provider": "nonexistent_provider", "apiKey": "sk-test"}
```
**Issue**: The `provider` field is not validated against the known set of API key providers defined in `ApiKeys` model (openai, anthropic, gemini, perplexity, openrouter). Invalid provider names may cause silent failures or unexpected database writes depending on the `update_api_key` implementation.  
**Remediation**:
```python
class UpdateApiKeyRequest(BaseModel):
    provider: str = Field(..., pattern="^(openai|anthropic|gemini|perplexity|openrouter)$")
    api_key: str = Field(..., min_length=1)
```

## Cross-cutting Architecture Observations

1. **Dual HTML-serving and API-serving role**: While the domain context states "FastAPI serves API endpoints, not HTML," the auth router serves multiple HTML pages (`dev_stub_picker`, `_render_deny_page`). These HTML-generating endpoints have inconsistent security controls — one uses `html.escape()` while the other does not.

2. **Validation exists but coverage is incomplete**: The codebase demonstrates awareness of input validation (regex patterns, Literal types, mutual exclusivity validators) but these are applied inconsistently. Business-critical inputs like billing amounts, LLM parameters, and deployed agent inputs lack the same rigor.

3. **Type B gaps are the most significant risk**: The `input_schema` field exists on every agent but is never validated at the `/rest/{friendly_name}` entry point. The `html.escape` function is imported and used in one HTML generator but not another. These "control exists but not applied" gaps are more dangerous than missing controls because they create false confidence.

4. **Output encoding is generally correct for JSON contexts**: The application primarily operates as a JSON API where FastAPI/Pydantic handle serialization correctly. The XSS risks are concentrated in the few HTML-generating endpoints in the auth router.

5. **Open redirect risk in OAuth flow**: The post-authentication redirect accepts arbitrary absolute URLs, which is a common OAuth implementation mistake. The CSRF protection (state parameter) is well-implemented but doesn't protect against attacker-initiated flows with malicious `return_to` values.

## Cross-cutting Recommendations

### Immediate (Fix within current sprint)

1. **Fix XSS in `dev_stub_picker`** (ASVS-121-HIGH-001): Apply `html.escape()` and `urllib.parse.quote()` to all user-controlled values interpolated into HTML/URLs. This is a 5-line fix in a single function.

2. **Fix open redirect in `login_callback`** (ASVS-122-HIGH-001): Validate that absolute `return_to` URLs match the configured `FRONTEND_URL` domain. Reject or ignore external domains.

3. **Add input validation to `/rest/{friendly_name}`** (ASVS-221-HIGH-001): Validate incoming JSON against the agent's declared `input_schema` before execution.

### Short-term (Next 2-4 weeks)

4. **Add range constraints to numeric fields**: Apply `ge`, `le`, `gt`, `lt` constraints to billing amounts, LLM parameters, and other business-critical numbers.

5. **Validate provider/model against known values**: Add model validators to `ChatRequest` and `ProviderConfig` that check against `PROVIDER_CONFIG`.

6. **Add `max_length` to string fields**: Especially `ChatMessage.content`, agent `name`/`description`, and API key values.

7. **Validate URL protocols on SSRF-sensitive inputs**: Use `AnyHttpUrl` or custom validators for `FetchSpecRequest.url` and `ListMcpToolsRequest.mcp_url`.

### Long-term (Next quarter)

8. **Create a validation specification document**: Document all input validation rules in a central specification that maps each endpoint's expected formats, ranges, and constraints. This satisfies ASVS 2.1.1 fully.

9. **Implement request body size limits**: Add middleware or FastAPI configuration to limit maximum request body size, protecting against memory exhaustion attacks on unbounded string fields.

10. **Add JSON Schema validation middleware for deployed agents**: Create a reusable middleware that validates request bodies against stored schemas, making the `input_schema` field enforceable rather than advisory.
