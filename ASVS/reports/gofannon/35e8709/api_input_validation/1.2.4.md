## ASVS-1.2.4: Injection Prevention â€” Database Injection

### Findings for 1.2.4

### [MEDIUM]

**Finding ID:** ASVS-124-MED-001

**Title:** Database abstraction passes user-controllable path parameters to `db.find()` without explicit NoSQL injection sanitization

**File:** `webapp/packages/api/user-service/routes.py` â€” functions `get_namespace_stats` (line ~405), `list_records` (line ~420)

**Vulnerable Code:**
```python
@router.get("/data-store/namespaces/{namespace}", response_model=NamespaceStats)
async def get_namespace_stats(
    namespace: str,
    db: DatabaseService = Depends(get_db),
    user: dict = Depends(get_current_user),
):
    user_id = user.get("uid", "anonymous")
    docs = db.find("agent_data_store", {"userId": user_id, "namespace": namespace})
```

```python
@router.get("/data-store/namespaces/{namespace}/records", response_model=List[DataStoreRecord])
async def list_records(
    namespace: str,
    db: DatabaseService = Depends(get_db),
    user: dict = Depends(get_current_user),
):
    user_id = user.get("uid", "anonymous")
    docs = db.find("agent_data_store", {"userId": user_id, "namespace": namespace})
```

**Data Flow:** HTTP path parameter `{namespace}` â†’ FastAPI string coercion â†’ `db.find()` filter dictionary value â†’ NoSQL query execution (CouchDB/Firestore/DynamoDB)

**Analysis:**
The `namespace` path parameter flows directly into the `db.find()` filter dictionary. While FastAPI ensures path parameters are strings (mitigating MongoDB-style operator injection like `{"$gt": ""}`), the `DatabaseService` abstraction does not provide any documented safety guarantees. For backends like CouchDB that may use Mango selectors, or custom query builders, there is no explicit validation that filter values cannot be interpreted as operators or contain special characters that alter query semantics.

The reliance is entirely on:
1. FastAPI path parameter typing (always string) â€” implicit protection
2. The undocumented behavior of the `DatabaseService.find()` implementation

This is a **Type B** gap: the application uses a database abstraction layer (analogous to an ORM), but the abstraction doesn't explicitly enforce or document injection prevention for its `find()` method.

**Proof of Concept:**
For a MongoDB-backed implementation that doesn't validate filter value types:
```
GET /data-store/namespaces/{"$regex":".*"}/records
```
Note: This specific vector is mitigated by FastAPI's path parameter handling (slashes aren't matched), but other backends could have different special character handling for values like `*`, `%`, or regex meta-characters in CouchDB Mango `$regex` operators.

**Impact:** If the database backend implementation doesn't properly isolate filter values from operators, an attacker could potentially bypass user-scoping or read records from other namespaces. Severity is reduced because FastAPI's type system provides a first layer of protection.

**Remediation:**
Add explicit input validation for all values passed to `db.find()` filters:
```python
import re

SAFE_NAMESPACE_PATTERN = re.compile(r"^[a-zA-Z0-9_\-\.]{1,128}$")

@router.get("/data-store/namespaces/{namespace}/records")
async def list_records(namespace: str, ...):
    if not SAFE_NAMESPACE_PATTERN.match(namespace):
        raise HTTPException(status_code=400, detail="Invalid namespace format")
    user_id = user.get("uid", "anonymous")
    docs = db.find("agent_data_store", {"userId": user_id, "namespace": namespace})
```

Additionally, the `DatabaseService.find()` method should document and enforce that filter values are treated as literal equality matches, never as operators.

### [LOW]

**Finding ID:** ASVS-124-LOW-001

**Title:** Session and ticket IDs from path parameters used directly in `db.get()` without format validation

**File:** `webapp/packages/api/user-service/routes.py` â€” functions `get_chat_status`, `get_session_config`, `delete_session`, `get_agent`, `delete_agent`, `update_agent`, `get_demo_app`

**Vulnerable Code:**
```python
@router.get("/chat/{ticket_id}")
async def get_chat_status(ticket_id: str, db: DatabaseService = Depends(get_db), ...):
    ticket_data = db.get("tickets", ticket_id)
```

```python
@router.post("/sessions/{session_id}/config")
async def update_session_config(session_id: str, config: ProviderConfig, db: DatabaseService = Depends(get_db), ...):
    session_doc = db.get("sessions", session_id)
```

**Data Flow:** HTTP path parameter â†’ string â†’ `db.get(collection, id)` â†’ direct document lookup

**Analysis:**
Multiple endpoints pass user-controlled path parameters directly to `db.get()` as document IDs. While ID-based lookups are inherently safe against injection in most NoSQL databases (they map to direct key lookups), there's no validation that these IDs conform to expected formats (UUID, etc.). If the `DatabaseService` implementation for CouchDB constructs REST URLs like `/{db}/{id}`, malformed IDs could potentially lead to path traversal within the CouchDB HTTP API.

**Impact:** Low. Direct ID lookups in NoSQL databases are generally safe. The risk exists only if the underlying implementation concatenates the ID into a URL or query string without encoding.

**Remediation:**
Validate that document IDs conform to the expected format:
```python
UUID_PATTERN = re.compile(r"^[a-f0-9\-]{36}$")

@router.get("/agents/{agent_id}", response_model=Agent)
async def get_agent(agent_id: str, ...):
    if not UUID_PATTERN.match(agent_id):
        raise HTTPException(status_code=400, detail="Invalid agent ID format")
    agent_doc = db.get("agents", agent_id)
```

### Positive Patterns for 1.2.4

1. **Structural query API:** The `DatabaseService` uses method-based access (`get()`, `find()`, `save()`, `delete()`, `list_all()`) rather than raw query strings. Filter dictionaries separate structure from values, analogous to parameterized queries.

2. **Pydantic `extra="ignore"`:** Both `CreateAgentRequest` and `UpdateAgentRequest` use `extra="ignore"`, preventing unexpected fields from being saved to the database and later used in queries.

3. **User-scoped queries:** Data store queries always include `{"userId": user_id}` from the authenticated session, not from request parameters, preventing unauthorized cross-user data access at the query level.

4. **`_rev` stripping on update:** The `update_agent` function explicitly removes `_rev` and `_id` from merged data before constructing the update, preventing CouchDB revision manipulation.

## Cross-cutting Architecture Observations

1. **Inconsistent HTML encoding practices within the same file:** `routes_auth.py` demonstrates knowledge of HTML escaping (`_render_deny_page` uses `html.escape()`) but fails to apply it in `dev_stub_picker`, just 30 lines away. This is a classic Type B gap â€” the control exists but isn't applied uniformly.

2. **Defense-in-depth reliance on framework typing:** The application's primary defense against injection (both database and OS) relies on FastAPI's type system and Pydantic models. While effective for current code patterns, this provides no protection if future changes introduce routes that pass complex types (dicts, lists) to the database layer.

3. **Database abstraction as implicit parameterization:** The `DatabaseService` API design (method calls with separate collection/ID/filter parameters) is structurally equivalent to parameterized queries. However, this safety property is not documented, tested, or enforced by the interface â€” it's an emergent property that could be undermined by implementation changes.

4. **Dev-mode endpoints reachable in production configurations:** The `dev_stub_picker` endpoint lacks a guard checking whether the dev_stub auth provider is actually enabled. Since `routes_auth.py` is mounted whenever any auth provider is enabled, development-mode attack surfaces persist in production.

5. **Open redirect in login callback:** The `login_callback` function allows arbitrary absolute URLs in the `return_to` cookie value (`if raw_target.startswith(("http://", "https://")): redirect_url = raw_target`). While not directly related to the three audited ASVS requirements, this represents a phishing vector that compounds the XSS finding.

## Cross-cutting Recommendations

### Immediate (0-2 weeks)
1. **Fix dev_stub_picker XSS** (ASVS-131-MED-001): Apply `html.escape()` for HTML text content and `urllib.parse.quote()` for URL attributes. Add a guard that returns 404 if the dev_stub provider is not enabled in the registry.

2. **Add format validation to namespace and key path parameters** (ASVS-124-MED-001): Apply regex validation (`^[a-zA-Z0-9_\-\.]+$`) on `namespace` and `key` parameters before they reach the database layer.

### Short-term (2-6 weeks)
3. **Document and test database abstraction safety contract:** Create explicit interface documentation for `DatabaseService.find()` stating that filter values MUST be treated as literal matches, not operators. Add integration tests that verify this behavior for each backend (CouchDB, Firestore, Memory).

4. **Add an HTML escaping utility and mandate its use for all HTML-rendering routes:** Create a shared template helper that auto-escapes all interpolated values. Consider using a minimal template engine (Jinja2 with autoescape) instead of f-string HTML construction.

5. **Restrict open redirect scope:** Validate that `return_to` URLs either are relative paths or match a configured allowlist of trusted domains.

### Long-term (1-3 months)
6. **Implement a query builder pattern** for the database layer that enforces safe filter construction and logs/rejects any non-primitive filter values.

7. **Add server-side content sanitization** for fields that may be rendered as HTML by frontends (agent descriptions, docstrings). Use a library like `nh3` (Rust-based, fast) to strip dangerous HTML while preserving formatting.

8. **Security linting in CI:** Add rules that flag f-string HTML construction (`HTMLResponse(content=f"..."`) without sanitization, and raw dict construction in `db.find()` calls without validation.
