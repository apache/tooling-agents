## ASVS-1.3.1: Sanitization — HTML Input Sanitization

### Findings for 1.3.1

### [MEDIUM]

**Finding ID:** ASVS-131-MED-001

**Title:** Reflected XSS in dev-stub picker endpoint via unsanitized query parameters rendered into HTML

**File:** `webapp/packages/api/user-service/routes_auth.py` — function `dev_stub_picker` (line ~195)

**Vulnerable Code:**
```python
@router.get("/auth/dev-stub-picker", response_class=HTMLResponse)
async def dev_stub_picker(
    request: Request,
    state: str = Query(...),
    redirect_uri: str = Query(...),
    users: str = Query(...),
) -> HTMLResponse:
    user_list = [u for u in users.split(",") if u]
    links = "".join(
        f'<li><a href="/auth/callback/dev_stub?code={u}&state={state}">{u}</a></li>'
        for u in user_list
    )
    html = f"""<!doctype html>
<html>
  <head><title>Dev stub login</title></head>
  <body style="font-family: sans-serif; max-width: 600px; margin: 60px auto;">
    <h1>Dev stub login</h1>
    <p>Not for production. Pick a configured user to sign in as:</p>
    <ul>{links}</ul>
  </body>
</html>"""
    return HTMLResponse(content=html)
```

**Data Flow:** Query parameter `users` / `state` → f-string interpolation → HTML response body (no encoding or sanitization)

**Gap Type:** Type A — Entry point with NO control (HTML encoding exists elsewhere in this file via `html.escape()` but is NOT applied here)

**Proof of Concept:**
```
GET /auth/dev-stub-picker?state=x&redirect_uri=x&users=<img%20src=x%20onerror=alert(document.cookie)>
```

Response body will contain:
```html
<li><a href="/auth/callback/dev_stub?code=<img src=x onerror=alert(document.cookie)>&state=x"><img src=x onerror=alert(document.cookie)></a></li>
```

The injected `onerror` handler executes JavaScript in the context of the API domain. Since the `gofannon_sid` cookie is `httponly`, it cannot be exfiltrated directly, but the attacker can:
- Perform authenticated API actions via `fetch()` (session cookie is sent automatically)
- Redirect users to phishing pages
- Read sensitive API responses (agent code, API key metadata)

**Attack Vector:** An attacker crafts a malicious link and sends it to a victim (phishing). The victim's browser renders the XSS payload within the API origin, granting the attacker access to same-origin API calls.

**Scope:** This endpoint is included in `routes_auth.py` which is mounted whenever ANY auth provider is enabled (`ProviderRegistry.has_any()`). It does NOT check whether the `dev_stub` provider is actually configured, so the endpoint may be reachable in production deployments.

**Impact:** Reflected XSS enabling session hijacking (via API actions), data exfiltration, and phishing. The attack surface is the API domain origin with session cookie access.

**Remediation:**
```python
import html as _html
from urllib.parse import quote as url_quote

@router.get("/auth/dev-stub-picker", response_class=HTMLResponse)
async def dev_stub_picker(
    request: Request,
    state: str = Query(...),
    redirect_uri: str = Query(...),
    users: str = Query(...),
) -> HTMLResponse:
    # Guard: only available when dev_stub provider is enabled
    registry = get_registry()
    if not registry.get("dev_stub"):
        raise HTTPException(status_code=404, detail="Not found")
    
    user_list = [u for u in users.split(",") if u]
    links = "".join(
        f'<li><a href="/auth/callback/dev_stub?code={url_quote(u)}&state={url_quote(state)}">'
        f'{_html.escape(u)}</a></li>'
        for u in user_list
    )
    html = f"""<!doctype html>...<ul>{links}</ul>..."""
    return HTMLResponse(content=html)
```

### [LOW]

**Finding ID:** ASVS-131-LOW-001

**Title:** No HTML sanitization on user-generated content fields stored for potential frontend rendering

**File:** `webapp/packages/api/user-service/models/agent.py` — classes `CreateAgentRequest`, `SwaggerSpec`

**Relevant Code:**
```python
class SwaggerSpec(BaseModel):
    name: str
    content: str  # Could contain HTML/script if spec is user-provided

class CreateAgentRequest(BaseModel):
    name: str
    description: str  # Rendered in UI components
    code: str
    docstring: Optional[str] = None
```

**Analysis:**
Fields like `description`, `name`, `docstring`, and `SwaggerSpec.content` accept arbitrary string content without sanitization. While these are returned as JSON (not rendered as HTML by the server), they flow to the frontend where they may be rendered in the DOM. The ASVS requirement specifically addresses "WYSIWYG editors or similar" content.

Per the domain context, "The application does not show explicit output encoding for HTML contexts... suggesting potential XSS or injection risks if user data is rendered unsafely." However, since the server returns JSON and the frontend framework (likely React) escapes by default, this is a LOW risk that depends on frontend implementation.

**Impact:** If the frontend renders any of these fields using `dangerouslySetInnerHTML` or similar unsafe patterns, stored XSS could affect other users viewing the same agent.

**Remediation:** For fields expected to contain rich text or markdown, apply server-side sanitization using a library like `bleach` or `nh3`:
```python
import nh3

class CreateAgentRequest(BaseModel):
    description: str
    
    @field_validator("description")
    @classmethod
    def sanitize_description(cls, v):
        return nh3.clean(v)  # Strips dangerous HTML while preserving safe content
```

### Positive Patterns for 1.3.1

1. **`_render_deny_page` uses `html.escape()`:** The login-denied page properly sanitizes the provider-supplied `reason` string before embedding it in HTML, demonstrating awareness of XSS risks.

2. **JSON API responses for data endpoints:** All agent, demo, and data store endpoints return JSON via Pydantic serialization, which does not render HTML and is safe from server-side XSS.

3. **Regex validation on ChatMessage.role:** The `ChatMessage` model uses `pattern="^(user|assistant|system)$"` to constrain the role field, preventing injection of arbitrary content in that field.

4. **`httponly=True` on session cookies:** Even if XSS is achieved, direct session token theft via `document.cookie` is prevented by the httponly flag on `gofannon_sid`.

## Cross-cutting Architecture Observations

1. **Inconsistent HTML encoding practices within the same file:** `routes_auth.py` demonstrates knowledge of HTML escaping (`_render_deny_page` uses `html.escape()`) but fails to apply it in `dev_stub_picker`, just 30 lines away. This is a classic Type B gap — the control exists but isn't applied uniformly.

2. **Defense-in-depth reliance on framework typing:** The application's primary defense against injection (both database and OS) relies on FastAPI's type system and Pydantic models. While effective for current code patterns, this provides no protection if future changes introduce routes that pass complex types (dicts, lists) to the database layer.

3. **Database abstraction as implicit parameterization:** The `DatabaseService` API design (method calls with separate collection/ID/filter parameters) is structurally equivalent to parameterized queries. However, this safety property is not documented, tested, or enforced by the interface — it's an emergent property that could be undermined by implementation changes.

4. **Dev-mode endpoints reachable in production configurations:** The `dev_stub_picker` endpoint lacks a guard checking whether the dev_stub auth provider is actually enabled. Since `routes_auth.py` is mounted whenever any auth provider is enabled, development-mode attack surfaces persist in production.

5. **Open redirect in login callback:** The `login_callback` function allows arbitrary absolute URLs in the `return_to` cookie value (`if raw_target.startswith(("http://", "https://")): redirect_url = raw_target`). While not directly related to the three audited ASVS requirements, this represents a phishing vector that compounds the XSS finding.

## Cross-cutting Recommendations

### Immediate (0-2 weeks)
1. **Fix dev_stub_picker XSS** (ASVS-131-MED-001): Apply `html.escape()` for HTML text content and `urllib.parse.quote()` for URL attributes. Add a guard that returns 404 if the dev_stub provider is not enabled in the registry.

2. **Add format validation to namespace and key path parameters** (ASVS-124-MED-001): Apply regex validation (`^[a-zA-Z0-9_\-\.]+$`) on `namespace` and `key` parameters before they reach the database layer.

### Short-term (2-6 weeks)
3. **Document and test database abstraction safety contract:** Create explicit interface documentation for `DatabaseService.find()` stating that filter values MUST be treated as literal matches, not operators. Add integration tests that verify this behavior for each backend (CouchDB, Firestore, Memory).

4. **Add an HTML escaping utility and mandate its use for all HTML-rendering routes:** Create a shared template helper that auto-escapes all interpolated values. Consider using a minimal template engine (Jinja2 with autoescape) instead of f-string HTML construction.

5. **Restrict open redirect scope:** Validate that `return_to` URLs either are relative paths or match a configured allowlist of trusted domains.

### Long-term (1-3 months)
6. **Implement a query builder pattern** for the database layer that enforces safe filter construction and logs/rejects any non-primitive filter values.

7. **Add server-side content sanitization** for fields that may be rendered as HTML by frontends (agent descriptions, docstrings). Use a library like `nh3` (Rust-based, fast) to strip dangerous HTML while preserving formatting.

8. **Security linting in CI:** Add rules that flag f-string HTML construction (`HTMLResponse(content=f"..."`) without sanitization, and raw dict construction in `db.find()` calls without validation.
