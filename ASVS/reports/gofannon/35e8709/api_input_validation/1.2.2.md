## ASVS-1.2.2: URL Encoding and Safe Protocols

### Findings for 1.2.2

### [HIGH]
**Finding ID**: ASVS-122-HIGH-001  
**Title**: Open redirect via unvalidated `return_to` parameter accepting absolute URLs  
**Location**: `routes_auth.py`, `login_callback`, lines ~149-157

```python
raw_target = return_to or "/"
if raw_target.startswith(("http://", "https://")):
    redirect_url = raw_target
else:
    frontend_base = os.getenv("FRONTEND_URL", "http://localhost:3000").rstrip("/")
    if not raw_target.startswith("/"):
        raw_target = "/" + raw_target
    redirect_url = frontend_base + raw_target
resp = RedirectResponse(url=redirect_url, status_code=302)
```

**Data flow**: `/auth/login/{type}?return_to=https://evil.com` → cookie `gofannon_return_to` → `/auth/callback/{type}` reads cookie → if starts with `http://`/`https://` → used directly as `RedirectResponse(url=...)` 
**Proof of concept**:
```
1. Attacker crafts: https://app.example.com/auth/login/google?return_to=https://evil.com/phish
2. User clicks the link, authenticates with Google
3. After successful OAuth, user is redirected to https://evil.com/phish
4. evil.com shows a fake "session expired" page to harvest credentials
```
**Issue**: The `return_to` value originates from a user-controlled query parameter (`/auth/login/{type}?return_to=...`), is stored in a cookie, and then used as the redirect target after authentication. When it's an absolute URL, no validation is performed to ensure it points to a trusted domain. Only safe URL protocols are checked (`http://`, `https://`), but the domain is unrestricted.  
**Impact**: Post-authentication open redirect enabling phishing attacks. Users trust the initial login page (legitimate domain) and may not notice the redirect to a malicious site after successful authentication.  
**Remediation**: Validate that absolute `return_to` URLs point to the configured frontend domain:

```python
from urllib.parse import urlparse

frontend_base = os.getenv("FRONTEND_URL", "http://localhost:3000").rstrip("/")
frontend_host = urlparse(frontend_base).netloc

raw_target = return_to or "/"
if raw_target.startswith(("http://", "https://")):
    parsed = urlparse(raw_target)
    if parsed.netloc != frontend_host:
        # Reject or fall back to root
        redirect_url = frontend_base + "/"
    else:
        redirect_url = raw_target
else:
    if not raw_target.startswith("/"):
        raw_target = "/" + raw_target
    redirect_url = frontend_base + raw_target
```

### [MEDIUM]
**Finding ID**: ASVS-122-MED-001  
**Title**: URL parameters not encoded in dynamically built HTML href attributes  
**Location**: `routes_auth.py`, `dev_stub_picker`, line ~210

```python
links = "".join(
    f'<li><a href="/auth/callback/dev_stub?code={u}&state={state}">{u}</a></li>'
    for u in user_list
)
```

**Data flow**: Query parameters `users` and `state` → split/used directly in URL construction within `href` attribute → no URL encoding applied  
**Proof of concept**:
```
GET /auth/dev-stub-picker?state=foo%26extra=bar&redirect_uri=x&users=user1%26admin=true
```
The `&` in `state` would inject additional query parameters into the constructed href URL.  
**Issue**: Query parameter values interpolated into href URLs are not URL-encoded. Characters like `&`, `=`, `#`, and space break URL structure, allowing parameter injection in the constructed callback URL.  
**Remediation**:

```python
from urllib.parse import quote

links = "".join(
    f'<li><a href="/auth/callback/dev_stub?code={quote(u, safe="")}&state={quote(state, safe="")}">{_html.escape(u)}</a></li>'
    for u in user_list
)
```

### [LOW]
**Finding ID**: ASVS-122-LOW-001  
**Title**: No URL protocol validation on FetchSpecRequest and ListMcpToolsRequest  
**Location**: `routes.py`, line ~131 and ~124

```python
class FetchSpecRequest(BaseModel):
    url: str

class ListMcpToolsRequest(BaseModel):
    mcp_url: str
```

**Issue**: URL fields accept any string without validating that only safe protocols (http/https) are used. A `javascript:`, `file:`, or `data:` protocol URL could potentially be processed if the downstream fetcher doesn't validate protocols.  
**Remediation**: Use Pydantic's `AnyHttpUrl` type or add a validator:

```python
from pydantic import field_validator

class FetchSpecRequest(BaseModel):
    url: str
    
    @field_validator('url')
    @classmethod
    def validate_url_protocol(cls, v):
        if not v.startswith(('http://', 'https://')):
            raise ValueError('Only http and https protocols are allowed')
        return v
```

## Cross-cutting Architecture Observations

1. **Dual HTML-serving and API-serving role**: While the domain context states "FastAPI serves API endpoints, not HTML," the auth router serves multiple HTML pages (`dev_stub_picker`, `_render_deny_page`). These HTML-generating endpoints have inconsistent security controls — one uses `html.escape()` while the other does not.

2. **Validation exists but coverage is incomplete**: The codebase demonstrates awareness of input validation (regex patterns, Literal types, mutual exclusivity validators) but these are applied inconsistently. Business-critical inputs like billing amounts, LLM parameters, and deployed agent inputs lack the same rigor.

3. **Type B gaps are the most significant risk**: The `input_schema` field exists on every agent but is never validated at the `/rest/{friendly_name}` entry point. The `html.escape` function is imported and used in one HTML generator but not another. These "control exists but not applied" gaps are more dangerous than missing controls because they create false confidence.

4. **Output encoding is generally correct for JSON contexts**: The application primarily operates as a JSON API where FastAPI/Pydantic handle serialization correctly. The XSS risks are concentrated in the few HTML-generating endpoints in the auth router.

5. **Open redirect risk in OAuth flow**: The post-authentication redirect accepts arbitrary absolute URLs, which is a common OAuth implementation mistake. The CSRF protection (state parameter) is well-implemented but doesn't protect against attacker-initiated flows with malicious `return_to` values.

## Cross-cutting Recommendations

### Immediate (Fix within current sprint)

1. **Fix XSS in `dev_stub_picker`** (ASVS-121-HIGH-001): Apply `html.escape()` and `urllib.parse.quote()` to all user-controlled values interpolated into HTML/URLs. This is a 5-line fix in a single function.

2. **Fix open redirect in `login_callback`** (ASVS-122-HIGH-001): Validate that absolute `return_to` URLs match the configured `FRONTEND_URL` domain. Reject or ignore external domains.

3. **Add input validation to `/rest/{friendly_name}`** (ASVS-221-HIGH-001): Validate incoming JSON against the agent's declared `input_schema` before execution.

### Short-term (Next 2-4 weeks)

4. **Add range constraints to numeric fields**: Apply `ge`, `le`, `gt`, `lt` constraints to billing amounts, LLM parameters, and other business-critical numbers.

5. **Validate provider/model against known values**: Add model validators to `ChatRequest` and `ProviderConfig` that check against `PROVIDER_CONFIG`.

6. **Add `max_length` to string fields**: Especially `ChatMessage.content`, agent `name`/`description`, and API key values.

7. **Validate URL protocols on SSRF-sensitive inputs**: Use `AnyHttpUrl` or custom validators for `FetchSpecRequest.url` and `ListMcpToolsRequest.mcp_url`.

### Long-term (Next quarter)

8. **Create a validation specification document**: Document all input validation rules in a central specification that maps each endpoint's expected formats, ranges, and constraints. This satisfies ASVS 2.1.1 fully.

9. **Implement request body size limits**: Add middleware or FastAPI configuration to limit maximum request body size, protecting against memory exhaustion attacks on unbounded string fields.

10. **Add JSON Schema validation middleware for deployed agents**: Create a reusable middleware that validates request bodies against stored schemas, making the `input_schema` field enforceable rather than advisory.
