## ASVS-1.2.5: Injection Prevention — OS Command Injection

### Findings for 1.2.5

No findings detected.

The provided source code does not contain any direct OS command execution calls (`os.system()`, `subprocess.run()`, `subprocess.Popen()`, `os.popen()`, etc.). User input does not flow into operating system command construction in any of the analyzed files.

**Note:** The `exec()` usage for agent code execution is an intentional design decision (known false positive per audit instructions) and is not OS command injection — it executes Python code within a sandboxed interpreter context, not shell commands.

**Note:** The `fetch_spec_content(request.url)` call in `/specs/fetch` and `mcp_service.list_tools_for_server(request.mcp_url)` accept user-controlled URLs, which represents an SSRF risk rather than OS command injection. These are not in scope for this requirement.

### Security Controls Inventory for 1.2.5

| Control | Location | Coverage Status |
|---------|----------|-----------------|
| No shell/subprocess calls | All analyzed files | No OS command execution present ✓ |
| Agent code sandbox (restricted globals) | `_execute_agent_code` (imported dependency) | Execution uses Python interpreter, not OS shell ✓ |
| Pydantic validation on all request bodies | Route handlers | Type coercion prevents injection payload shaping ✓ |

### Positive Patterns for 1.2.5

1. **No OS command execution in request handlers:** All route handlers delegate to service functions or database abstractions without invoking shell commands.

2. **URL-based remote operations:** Operations that interact with external systems (spec fetching, MCP tool listing) use HTTP client libraries rather than shell utilities like `curl` or `wget`.

3. **Environment variable access is read-only:** The only `os` module usage (`os.getenv("FRONTEND_URL")`) reads configuration — it does not construct commands from environment values combined with user input.

## Cross-cutting Architecture Observations

1. **Inconsistent HTML encoding practices within the same file:** `routes_auth.py` demonstrates knowledge of HTML escaping (`_render_deny_page` uses `html.escape()`) but fails to apply it in `dev_stub_picker`, just 30 lines away. This is a classic Type B gap — the control exists but isn't applied uniformly.

2. **Defense-in-depth reliance on framework typing:** The application's primary defense against injection (both database and OS) relies on FastAPI's type system and Pydantic models. While effective for current code patterns, this provides no protection if future changes introduce routes that pass complex types (dicts, lists) to the database layer.

3. **Database abstraction as implicit parameterization:** The `DatabaseService` API design (method calls with separate collection/ID/filter parameters) is structurally equivalent to parameterized queries. However, this safety property is not documented, tested, or enforced by the interface — it's an emergent property that could be undermined by implementation changes.

4. **Dev-mode endpoints reachable in production configurations:** The `dev_stub_picker` endpoint lacks a guard checking whether the dev_stub auth provider is actually enabled. Since `routes_auth.py` is mounted whenever any auth provider is enabled, development-mode attack surfaces persist in production.

5. **Open redirect in login callback:** The `login_callback` function allows arbitrary absolute URLs in the `return_to` cookie value (`if raw_target.startswith(("http://", "https://")): redirect_url = raw_target`). While not directly related to the three audited ASVS requirements, this represents a phishing vector that compounds the XSS finding.

## Cross-cutting Recommendations

### Immediate (0-2 weeks)
1. **Fix dev_stub_picker XSS** (ASVS-131-MED-001): Apply `html.escape()` for HTML text content and `urllib.parse.quote()` for URL attributes. Add a guard that returns 404 if the dev_stub provider is not enabled in the registry.

2. **Add format validation to namespace and key path parameters** (ASVS-124-MED-001): Apply regex validation (`^[a-zA-Z0-9_\-\.]+$`) on `namespace` and `key` parameters before they reach the database layer.

### Short-term (2-6 weeks)
3. **Document and test database abstraction safety contract:** Create explicit interface documentation for `DatabaseService.find()` stating that filter values MUST be treated as literal matches, not operators. Add integration tests that verify this behavior for each backend (CouchDB, Firestore, Memory).

4. **Add an HTML escaping utility and mandate its use for all HTML-rendering routes:** Create a shared template helper that auto-escapes all interpolated values. Consider using a minimal template engine (Jinja2 with autoescape) instead of f-string HTML construction.

5. **Restrict open redirect scope:** Validate that `return_to` URLs either are relative paths or match a configured allowlist of trusted domains.

### Long-term (1-3 months)
6. **Implement a query builder pattern** for the database layer that enforces safe filter construction and logs/rejects any non-primitive filter values.

7. **Add server-side content sanitization** for fields that may be rendered as HTML by frontends (agent descriptions, docstrings). Use a library like `nh3` (Rust-based, fast) to strip dangerous HTML while preserving formatting.

8. **Security linting in CI:** Add rules that flag f-string HTML construction (`HTMLResponse(content=f"..."`) without sanitization, and raw dict construction in `db.find()` calls without validation.
