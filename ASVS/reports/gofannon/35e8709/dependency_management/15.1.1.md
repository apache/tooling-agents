## ASVS-15.1.1: Secure Coding and Architecture Documentation — Risk-Based Remediation Timeframes

### Findings for 15.1.1

### [MEDIUM]

**Finding ID:** ASVS-1511-MED-001

**Title:** No Evidence of Documented Risk-Based Remediation Timeframes for Third-Party Components

**File Location:** `webapp/packages/api/user-service/services/llm_service.py` (Lines 1-20), `webapp/packages/api/user-service/app_factory.py` (Lines 1-10)

**Vulnerable Code Quote:**
```python
# llm_service.py - Critical dependency with no version or remediation policy reference
import litellm
from services.litellm_logger import ensure_litellm_logging

# Configure litellm defaults
litellm.drop_params = True
litellm.set_verbose = False
```

```python
# app_factory.py - Core framework dependency with no policy reference
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
```

**Data Flow:** Third-party libraries (litellm, FastAPI, Pydantic, OAuth) → Application runtime → No referenced policy governing update cadence or vulnerability remediation deadlines

**Evidence of Gap:**
- No code comments, configuration files, or documentation references defining SLA timelines for vulnerability remediation (e.g., "Critical: 48hrs, High: 7 days, Medium: 30 days, Low: 90 days")
- No SBOM file reference or generation mechanism visible in the application factory
- LiteLLM is identified in the domain context as a "critical trust point" but the codebase shows no corresponding classification or tiered remediation policy
- No configuration referencing a dependency management policy document
- The `PROVIDER_CONFIG` import suggests model configuration exists but lacks security classification of the provider library itself

**Impact:** Without documented remediation timeframes:
- Teams lack clear deadlines for patching known vulnerabilities in dependencies
- Prioritization of security patches becomes ad-hoc rather than risk-based
- Compliance audit evidence cannot demonstrate timely response to disclosed CVEs
- Critical components like LiteLLM (which handles API keys, makes external HTTP calls, processes untrusted LLM output) may remain unpatched during active exploitation

**Remediation:**

Create a `DEPENDENCY_POLICY.md` or equivalent governance document and reference it in the codebase:

```markdown
# Dependency Management Policy

## Risk Classification
| Classification | Criteria | Examples |
|---------------|----------|----------|
| Critical | Handles auth, secrets, or direct user input | litellm, OAuth libs, FastAPI |
| High | Core data processing | Pydantic, CouchDB client |
| Medium | Supporting functionality | logging, utilities |
| Low | Dev/build tooling | pytest, linters |

## Remediation SLAs (from CVE publication date)
| Vulnerability Severity | Critical Component | Standard Component |
|------------------------|-------------------|-------------------|
| Critical (CVSS 9.0+) | 24 hours | 72 hours |
| High (CVSS 7.0-8.9) | 72 hours | 7 days |
| Medium (CVSS 4.0-6.9) | 14 days | 30 days |
| Low (CVSS < 4.0) | 30 days | 90 days |

## Regular Update Cadence
- Critical components: Monthly minimum review
- All components: Quarterly update cycle
- Automated scanning: Daily (via dependabot/pip-audit/npm audit)
```

Reference this in code:
```python
# app_factory.py
# Dependency governance: see docs/DEPENDENCY_POLICY.md
# SBOM generated by: scripts/generate_sbom.sh
```

### [LOW]

**Finding ID:** ASVS-1511-LOW-001

**Title:** Critical Dependency (LiteLLM) Not Classified as "Dangerous Functionality" Component

**File Location:** `webapp/packages/api/user-service/services/llm_service.py` (Lines 14-20)

**Vulnerable Code Quote:**
```python
ensure_litellm_logging()

# Configure litellm defaults
litellm.drop_params = True
litellm.set_verbose = False

# Default timeout (seconds) for LLM calls. Override with LLM_TIMEOUT_SECONDS env var.
DEFAULT_LLM_TIMEOUT = int(os.getenv("LLM_TIMEOUT_SECONDS", "600"))
```

**Evidence of Gap:**
- LiteLLM performs HTTP calls to external services, processes API keys, handles untrusted binary/JSON data from external APIs, and dynamically routes to different backends
- Per ASVS 15.1.1 section description, components performing "raw file or binary data parsing" and handling sensitive operations should be documented as containing "dangerous functionality"
- No documentation classifying LiteLLM's risk profile or listing it as requiring enhanced scrutiny during updates
- The domain context explicitly identifies "LiteLLM's model loading" as resource-intensive and "LiteLLM and OAuth libraries" as critical trust points, but this is not reflected in the codebase

**Impact:** Without classifying LiteLLM as a dangerous/critical component, it may not receive expedited remediation when vulnerabilities are disclosed, despite being the primary interface to external LLM providers with access to user API keys.

**Remediation:** Add component classification documentation:
```yaml
# dependency_classifications.yaml
critical_trust_components:
  - name: litellm
    reason: "Handles user API keys, makes external HTTP calls, processes untrusted API responses"
    dangerous_operations: ["external HTTP with secrets", "dynamic API routing", "response parsing"]
    remediation_tier: critical
  - name: authlib  # or whichever OAuth library
    reason: "Handles OAuth flows, token validation, session establishment"
    dangerous_operations: ["cryptographic token validation", "external IdP communication"]
    remediation_tier: critical
```

## Cross-cutting Architecture Observations

1. **Documentation-as-Code Gap**: The codebase shows good operational awareness (timeouts, retries, allowance enforcement) but lacks the governance documentation layer that ASVS 15.x requires. The security controls are implemented in code but not connected to a formal dependency management policy.

2. **LiteLLM as Critical Trust Point**: LiteLLM is the most significant third-party dependency visible in these files. It:
   - Handles user API keys (`api_key` parameter)
   - Makes HTTP calls to external LLM providers
   - Parses untrusted response data from external APIs
   - Has a rapid release cadence with frequent security-relevant changes
   
   This component requires the most aggressive remediation timeline and monitoring.

3. **Implicit Version Requirements**: The use of `litellm.aresponses` and `litellm.aget_responses` implies a minimum version requirement that is not explicitly enforced at runtime. If a deployment uses an older version, the code falls through to `acompletion` silently rather than failing with a clear version mismatch error.

4. **Auth Library Resilience Pattern**: The `create_app()` function's try/except around auth registry initialization (Lines 58-68 in `app_factory.py`) shows a design pattern where third-party auth library failures are absorbed rather than blocking startup. While operationally pragmatic, this could mask version incompatibility issues with OAuth libraries.

5. **Observable but Not Governed**: The `ObservabilityMiddleware` and `ensure_litellm_logging()` provide visibility into third-party component behavior, which is necessary for detecting exploitation of dependency vulnerabilities, but there's no proactive governance mechanism (scheduled audits, automated scanning, SLA tracking).

## Cross-cutting Recommendations

### Immediate (0-7 days)

1. **Create `DEPENDENCY_POLICY.md`** defining risk-based remediation timeframes for all severity levels, with specific tiers for critical components (LiteLLM, OAuth libraries, FastAPI) vs. standard components.

2. **Generate and commit an SBOM** (using `cyclonedx-bom` for Python, `@cyclonedx/cyclonedx-npm` for frontend) covering all direct and transitive dependencies.

3. **Add `pip-audit` to CI/CD pipeline** to block deployments when dependencies have known vulnerabilities that breach defined SLAs.

### Short-term (1-4 weeks)

4. **Implement runtime version logging** at application startup for all critical dependencies (LiteLLM, FastAPI, OAuth libraries) to support audit trail requirements.

5. **Add minimum version assertions** for LiteLLM to fail fast if an incompatible/vulnerable version is deployed:
   ```python
   import litellm
   assert hasattr(litellm, 'aresponses'), "LiteLLM >= X.Y.Z required"
   ```

6. **Configure Dependabot/Renovate** with update schedules aligned to the documented remediation policy, with auto-merge for patch-level updates of non-critical components.

7. **Implement streaming cost estimation** to close the resource tracking gap in `stream_llm`.

### Long-term (1-3 months)

8. **Establish quarterly dependency review process** documenting decisions to retain, update, or replace each component, with special attention to "risky components" (poorly maintained or end-of-life).

9. **Implement supply chain verification** (e.g., `pip install --require-hashes`, npm `package-lock.json` integrity checks) to ensure dependencies come from exact repositories, not mirrors.

10. **Create a "dangerous functionality" register** documenting all components that perform operations like raw data parsing, dynamic code execution, or secret handling, with enhanced review requirements for version updates to these components.
