**Evidence of Gap:**
- No CI/CD gate or runtime check visible that validates dependencies against known CVE databases
- No version assertion or compatibility check for LiteLLM at startup (the code uses `litellm.aresponses` which is a relatively new API; older vulnerable versions might not have it but could silently fail)
- No `pip-audit`, `safety`, or equivalent integration referenced in the codebase
- No deployment-time SBOM validation against a vulnerability database
- Without a documented remediation timeframe (ASVS 15.1.1), compliance with update deadlines cannot be verified or enforced

**Impact:** 
- Applications may run with dependencies containing known CVEs beyond acceptable remediation windows
- The use of `litellm.aresponses` (OpenAI Responses API) suggests a specific minimum version requirement, but without enforcement, a deployment with an older, potentially vulnerable version could function partially (falling through to `acompletion`) without alerting operators
- Given LiteLLM's rapid development cycle and history of security-relevant fixes, stale versions pose elevated risk

**Proof of Concept:** Consider a scenario where:
1. A CVE is published for LiteLLM affecting API key handling
2. No automated alert or blocking mechanism exists in CI/CD
3. The vulnerable version remains deployed beyond any remediation deadline
4. User API keys are potentially exposed through the vulnerability

**Remediation:**

Implement multi-layer dependency compliance:

```python
# scripts/check_dependency_compliance.py
"""
Run in CI/CD and optionally at application startup.
Validates all dependencies against documented SLA timeframes.
"""
import subprocess
import json
import sys
from datetime import datetime, timedelta

REMEDIATION_SLAS = {
    "critical": timedelta(hours=24),
    "high": timedelta(days=7),
    "medium": timedelta(days=30),
    "low": timedelta(days=90),
}

def check_compliance():
    result = subprocess.run(
        ["pip-audit", "--format=json", "--progress-spinner=off"],
        capture_output=True, text=True
    )
    vulnerabilities = json.loads(result.stdout)
    
    breaches = []
    for vuln in vulnerabilities:
        severity = vuln.get("severity", "medium").lower()
        published = datetime.fromisoformat(vuln["published_date"])
        sla = REMEDIATION_SLAS.get(severity, timedelta(days=90))
        deadline = published + sla
        
        if datetime.now() > deadline:
            breaches.append({
                "package": vuln["name"],
                "version": vuln["version"],
                "cve": vuln["id"],
                "severity": severity,
                "days_overdue": (datetime.now() - deadline).days,
            })
    
    if breaches:
        print(f"❌ {len(breaches)} dependencies have breached remediation SLAs:")
        for b in breaches:
            print(f"  - {b['package']}=={b['version']}: {b['cve']} "
                  f"({b['severity']}, {b['days_overdue']} days overdue)")
        sys.exit(1)
    
    print("✅ All dependencies within remediation SLA timeframes")

if __name__ == "__main__":
    check_compliance()
```

**Finding ID:** ASVS-1521-LOW-001

**Title:** Streaming LLM Path Lacks Cost Tracking, Creating Potential for Uncontrolled Resource Consumption

**File Location:** `webapp/packages/api/user-service/services/llm_service.py` (Lines 337-340)

**Vulnerable Code Quote:**
```python

    ...

    """
    ...
    Note: Cost tracking is not available for streaming responses.
    """
```

**Evidence of Gap:**
- The `stream_llm` function explicitly notes cost tracking is unavailable
- While `require_allowance` is called before streaming (positive), actual cost is never deducted from user allowance after streaming completes
- This relates to ASVS 15.2.1's section on "prevent loss of availability due to overusing resource-demanding functionality"
- LiteLLM's streaming mode is identified in the domain context as potentially resource-intensive

**Impact:** Users could accumulate LLM costs through streaming without proper accounting, potentially exhausting shared resources or infrastructure budgets without triggering allowance enforcement post-call.

**Remediation:** Implement post-stream cost estimation:
```python
async def stream_llm(...) -> AsyncGenerator[Any, None]:
    # ... existing code ...
    token_count = 0
    try:
        response = await litellm.acompletion(**kwargs)
        async for chunk in response:
            # Estimate tokens from chunks
            if hasattr(chunk, 'usage') and chunk.usage:
                token_count = getattr(chunk.usage, 'total_tokens', token_count)
            yield chunk
    finally:
        # Best-effort cost estimation after stream completes
        if user_service and user_id and token_count > 0:
            estimated_cost = _estimate_streaming_cost(model_string, token_count)
            if estimated_cost:
                user_service.add_usage(user_id, estimated_cost, basic_info=user_basic_info)
```

### Positive Patterns for 15.2.1

1. **User allowance enforcement before LLM calls** (Lines 107-109, 373-375): Both `call_llm` and `stream_llm` enforce `require_allowance()` before making expensive third-party API calls, preventing unbounded resource consumption by individual users.
2. **Configurable timeout for LLM operations** (Lines 22-26, 127-129): `LLM_TIMEOUT_SECONDS` and `LLM_TIMEOUT_RETRIES` provide defense against resource exhaustion from hung connections to third-party LLM providers.
3. **Bounded retry logic** (Lines 248-267): Timeout retries are capped by `MAX_TIMEOUT_RETRIES` (default: 0) with exponential-ish backoff, preventing runaway retry storms against third-party services.
4. **Polling limits for Responses API** (Line 155): The `for _ in range(15)` loop prevents infinite polling against OpenAI's Responses API, limiting resource consumption to a bounded 30-second window.
5. **`litellm.drop_params = True`** (Line 18): Prevents errors when parameters aren't supported by a specific provider version, improving compatibility across library updates.

## Cross-cutting Architecture Observations

1. **Documentation-as-Code Gap**: The codebase shows good operational awareness (timeouts, retries, allowance enforcement) but lacks the governance documentation layer that ASVS 15.x requires. The security controls are implemented in code but not connected to a formal dependency management policy.

2. **LiteLLM as Critical Trust Point**: LiteLLM is the most significant third-party dependency visible in these files. It:
   - Handles user API keys (`api_key` parameter)
   - Makes HTTP calls to external LLM providers
   - Parses untrusted response data from external APIs
   - Has a rapid release cadence with frequent security-relevant changes
   
   This component requires the most aggressive remediation timeline and monitoring.

3. **Implicit Version Requirements**: The use of `litellm.aresponses` and `litellm.aget_responses` implies a minimum version requirement that is not explicitly enforced at runtime. If a deployment uses an older version, the code falls through to `acompletion` silently rather than failing with a clear version mismatch error.

4. **Auth Library Resilience Pattern**: The `create_app()` function's try/except around auth registry initialization (Lines 58-68 in `app_factory.py`) shows a design pattern where third-party auth library failures are absorbed rather than blocking startup. While operationally pragmatic, this could mask version incompatibility issues with OAuth libraries.

5. **Observable but Not Governed**: The `ObservabilityMiddleware` and `ensure_litellm_logging()` provide visibility into third-party component behavior, which is necessary for detecting exploitation of dependency vulnerabilities, but there's no proactive governance mechanism (scheduled audits, automated scanning, SLA tracking).

## Cross-cutting Recommendations

### Immediate (0-7 days)

1. **Create `DEPENDENCY_POLICY.md`** defining risk-based remediation timeframes for all severity levels, with specific tiers for critical components (LiteLLM, OAuth libraries, FastAPI) vs. standard components.

2. **Generate and commit an SBOM** (using `cyclonedx-bom` for Python, `@cyclonedx/cyclonedx-npm` for frontend) covering all direct and transitive dependencies.

3. **Add `pip-audit` to CI/CD pipeline** to block deployments when dependencies have known vulnerabilities that breach defined SLAs.

### Short-term (1-4 weeks)

4. **Implement runtime version logging** at application startup for all critical dependencies (LiteLLM, FastAPI, OAuth libraries) to support audit trail requirements.

5. **Add minimum version assertions** for LiteLLM to fail fast if an incompatible/vulnerable version is deployed:
   ```python
   import litellm
   assert hasattr(litellm, 'aresponses'), "LiteLLM >= X.Y.Z required"
   ```

6. **Configure Dependabot/Renovate** with update schedules aligned to the documented remediation policy, with auto-merge for patch-level updates of non-critical components.

7. **Implement streaming cost estimation** to close the resource tracking gap in `stream_llm`.

### Long-term (1-3 months)

8. **Establish quarterly dependency review process** documenting decisions to retain, update, or replace each component, with special attention to "risky components" (poorly maintained or end-of-life).

9. **Implement supply chain verification** (e.g., `pip install --require-hashes`, npm `package-lock.json` integrity checks) to ensure dependencies come from exact repositories, not mirrors.

10. **Create a "dangerous functionality" register** documenting all components that perform operations like raw data parsing, dynamic code execution, or secret handling, with enhanced review requirements for version updates to these components.
