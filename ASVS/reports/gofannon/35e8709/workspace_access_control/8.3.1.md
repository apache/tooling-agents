## ASVS-8.3.1: Operation Level Authorization

### Findings for 8.3.1

#### HIGH
**Finding ID:** ASVS-831-HIGH-001  
**Title:** Background Task Loses Authorization Context — No Re-verification of Permissions During Execution  
**File:** `webapp/packages/api/user-service/dependencies.py` (`process_chat`, lines ~383-504)

```python
async def process_chat(ticket_id: str, request: ChatRequest, user: dict, req: Request):
    # Background tasks don't have access to dependency injection, so we get service instances directly
    db_service = get_database_service(settings)
    ...
    # 'user' dict passed from the original request — permissions not re-checked
    # If user's permissions change (revoked) between request submission and execution,
    # the stale authorization context persists
    
    deployment_doc = db_service.get("deployments", agent_friendly_name)
    agent_id = deployment_doc["agentId"]
    agent_data = db_service.get("agents", agent_id)  # No workspace check
    agent = Agent(**agent_data)
    
    result, _ops = await _execute_agent_code(
        code=agent.code,
        ...
        user_id=user.get("uid"),  # Stale user context
    )
```

**Data flow:** HTTP request → `get_current_user` validates at request time → `user` dict captured → background task runs later with stale credentials → no re-verification before agent access

**Impact:** If a user's workspace membership or role is revoked between submitting a chat request and the background task executing, the operation proceeds with stale permissions. Given LLM calls can take 15-30 minutes (per timeout configuration), this window is significant.

**Note:** This partially overlaps with the known false positive about session caching, but differs because this is about background tasks operating outside the session lifecycle entirely — there's no session to expire or invalidate.

#### MEDIUM
**Finding ID:** ASVS-831-MED-001  
**Title:** `run_deployed_agent` Performs No Authorization Check on Agent Workspace Membership  
**File:** `webapp/packages/api/user-service/dependencies.py` (lines ~877-900), `routes.py` (line ~779)

The deployed agent execution path only validates that a deployment exists by its friendly name, without checking whether the authenticated user has permission to invoke that agent:

```python
# dependencies.py
async def run_deployed_agent(friendly_name: str, input_dict: dict, db: DatabaseService, user_id=None, ...):
    deployment_doc = db.get("deployments", friendly_name)  # Public lookup
    agent_id = deployment_doc["agentId"]
    agent_data = db.get("agents", agent_id)  # No permission check
    agent = Agent(**agent_data)
    result, _ops = await _execute_agent_code(code=agent.code, ...)  # Executes without authz
    return result

# routes.py
@router.post("/rest/{friendly_name}")
async def run_deployed_agent_route(friendly_name: str, request: Request, db=..., user=...):
    input_dict = await request.json()
    return await run_deployed_agent_logic(friendly_name, input_dict, db, user_id=user.get("uid"), ...)
```

**Data flow:** User-controlled `friendly_name` → deployment lookup → agent code execution (no workspace membership verification)

**Impact:** Any authenticated user can execute any deployed agent by knowing or guessing the friendly name. Since friendly names are discoverable via `GET /deployments` and `GET /providers` (which lists gofannon models), this is easily exploitable.

#### LOW
**Finding ID:** ASVS-831-LOW-001  
**Title:** Authorization Enforcement Occurs Correctly at Server Layer (Positive Finding with Gap Note)  
**File:** `webapp/packages/api/user-service/routes.py` (all routes)

All authorization checks are performed server-side via FastAPI dependency injection. No authorization logic is delegated to or relies on client-side enforcement. The `get_current_user` dependency validates sessions/tokens at the trusted service layer.

However, while the enforcement *location* is correct (server-side), the enforcement *completeness* is deficient (authentication without authorization for most resource operations, as documented in 8.2.1 and 8.2.2 findings).

### Security Controls Inventory for 8.3.1

|---------|----------|-------------------|--------|
| Session validation | `routes.py:_verify_session_cookie` | Server (FastAPI dependency) | ✅ Correct layer |
| Firebase token validation | `routes.py:_verify_firebase_token` | Server (FastAPI dependency) | ✅ Correct layer |
| Admin password check | `dependencies.py:require_admin_access` | Server (FastAPI dependency) | ✅ Correct layer |
| Workspace membership check | N/A | N/A | ❌ Not implemented |
| Background task re-auth | N/A | N/A | ❌ Not implemented |
| Data store user scoping | `routes.py:data store routes` | Server (query filter) | ✅ Correct |

### Positive Patterns for 8.3.1

- All authentication is performed server-side through FastAPI's dependency injection system
- No client-side JavaScript authorization or `X-Forwarded-*` header trust for access decisions
- The OAuth2 scheme (`oauth2_scheme`) extracts tokens server-side
- Session cookie validation (`_verify_session_cookie`) calls the session service directly on the backend
- Data store access filtering is applied at the database query level, not post-retrieval

## Cross-cutting Architecture Observations

1. **Phase B-1 vs B-3 Gap:** The workspace model (`models/workspace.py`) is well-designed with clear semantics, but workspace-based authorization enforcement is deferred to B-3. Meanwhile, the system operates as essentially a single-tenant application where all authenticated users share all resources (except data store entries). This creates a critical window where the multi-tenant claim in the domain context is aspirational, not actual.

2. **Authentication ≠ Authorization Pattern:** The codebase consistently authenticates users (positive) but conflates authentication with authorization. Every route has `user: dict = Depends(get_current_user)` but the `user` dict is used primarily for logging/context, not for access control decisions.

3. **Data Store as the Exception:** The data store routes demonstrate the correct pattern — filtering by `user_id` at the query level. This pattern should be replicated to agents, demos, deployments, and chat tickets.

4. **Audit Service Infrastructure Without Instrumentation:** `audit_service.py` is well-designed and ready for use, but no route currently calls it. This is acknowledged as B-3 scope but represents a Type B gap (control exists but is not called).

5. **Dual Admin Systems:** The `require_admin_access` (shared password) and `is_site_admin` (session-based) systems serve overlapping purposes without clear delineation, creating potential for privilege confusion.

## Cross-cutting Recommendations

### Immediate (Critical — before production multi-tenant deployment)

1. **Implement resource-level ownership validation** for all agent, demo, and deployment routes. At minimum, associate a `workspace_id` or `owner_uid` with each resource and validate it against the authenticated user's workspace memberships before any read/write operation.

2. **Add user ownership validation to chat ticket retrieval** (`GET /chat/{ticket_id}`). Store the user ID with the ticket during creation and verify it on retrieval.

3. **Add ownership validation to session config routes** (`/sessions/{session_id}/config`). Validate that the requesting user owns the session being modified.

### Short-term (High — within next sprint)

4. **Create a `require_workspace_access(resource_type, resource_id)` dependency** that can be applied uniformly to all resource routes, enforcing workspace-scoped access.

5. **Bind admin panel routes to user identity** by adding `get_current_user` alongside `require_admin_access`, enabling audit logging of admin actions.

6. **Scope list endpoints** (`GET /agents`, `GET /demos`, `GET /deployments`) to the user's workspace memberships rather than returning all records.

7. **Instrument cross-workspace access with the audit service** — connect `AuditService.record()` calls to the site-admin bypass path.

### Long-term (Medium — architecture alignment)

8. **Unify the admin authorization model** — replace or supplement the shared-password admin panel with the session-based `is_site_admin` flag, providing per-actor attribution.

9. **Add background task authorization refresh** — before executing long-running operations, re-validate the user's current permissions against the session service rather than relying on the snapshot from request time.

10. **Create and maintain an authorization matrix** document that maps routes → required roles/permissions → enforcement mechanism, enabling automated coverage verification in CI.
