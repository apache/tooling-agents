## ASVS-4.1.1: Generic Web Service Security - Content-Type Header

### Findings for 4.1.1

### [MEDIUM]

**Finding ID:** ASVS-411-MED-001

**File:** `webapp/packages/api/user-service/routes.py`, function `run_agent_code_stream`, lines ~600-744

**Description:** The Server-Sent Events (SSE) streaming endpoint sets `media_type="text/event-stream"` without including the `charset=utf-8` parameter. Since `text/event-stream` is a `text/*` content type, the ASVS requirement mandates a charset specification. Without it, intermediary proxies or older clients could misinterpret the character encoding, potentially leading to content misinterpretation or injection vectors in multi-byte character contexts.

**Vulnerable Code:**
```python
return StreamingResponse(
    event_generator(),
    media_type="text/event-stream",
    headers={
        "Cache-Control": "no-cache",
        "X-Accel-Buffering": "no",
        "Connection": "keep-alive",
    },
)
```

**Data Flow:** Agent trace events → JSON serialization → SSE frames → StreamingResponse → Client browser. The content is UTF-8 encoded by Python's default string encoding and the `json.dumps()` call, but this encoding assumption is never communicated to the client in the Content-Type header.

**Proof of Concept:** A response is delivered with header `Content-Type: text/event-stream` instead of `Content-Type: text/event-stream; charset=utf-8`. If a reverse proxy or client has a different default charset assumption (e.g., ISO-8859-1 per HTTP/1.1 spec for text/* without charset), multi-byte UTF-8 characters in agent trace output could be misinterpreted.

**Impact:** Potential content misinterpretation in edge cases with non-ASCII agent output. Low exploitability but violates defense-in-depth encoding hygiene.

**Remediation:**
```python
return StreamingResponse(
    event_generator(),
    media_type="text/event-stream; charset=utf-8",
    headers={
        "Cache-Control": "no-cache",
        "X-Accel-Buffering": "no",
        "Connection": "keep-alive",
    },
)
```

### Security Controls Inventory for 4.1.1

| Control | Location | Status |
|---------|----------|--------|
| FastAPI default JSONResponse Content-Type (`application/json`) | All JSON endpoints | ✅ Applied — `application/json` has no charset parameter per RFC 8259 (JSON is always UTF-8) |
| HTMLResponse Content-Type | `routes_auth.py` — `dev_stub_picker`, `login_callback` deny page | ✅ Applied — Starlette's HTMLResponse defaults to `text/html; charset=utf-8` |
| StreamingResponse Content-Type | `routes.py:run_agent_code_stream` | ❌ Missing charset for `text/event-stream` |
| Response(status_code=204) | `routes_auth.py:logout`, various DELETE endpoints | ✅ N/A — No message body |
| RedirectResponse | `routes_auth.py:login_redirect`, `login_callback` | ✅ N/A — 302 with Location header, no meaningful body |

### Positive Patterns for 4.1.1

1. **FastAPI's built-in JSON serialization** automatically sets `Content-Type: application/json` for all route handlers returning dicts or Pydantic models. Per RFC 8259, JSON text exchanged between systems is always UTF-8, so the absence of a charset parameter is correct for `application/json`.

2. **HTMLResponse usage in `routes_auth.py`** correctly inherits Starlette's default of `text/html; charset=utf-8` for all HTML content (dev-stub-picker, deny page).

3. **Explicit `media_type` on StreamingResponse** demonstrates awareness of content-type matching — the type matches the actual SSE content format.

4. **204 No Content responses** for DELETE operations correctly avoid body content, sidestepping Content-Type requirements entirely.

## Cross-cutting Architecture Observations

1. **Inconsistent response filtering pattern (Type B gap):** FastAPI's `response_model` is a powerful control that serializes responses through Pydantic schemas, automatically stripping undeclared fields. This control EXISTS in the codebase (used on ~60% of data endpoints) but is NOT APPLIED to provider configuration, session config, agent chain, agent deployment, MCP tools, and deployed agent execution endpoints. This creates a false sense of security — developers may assume all endpoints are filtered because many are.

2. **Content-Type handling is largely delegated to framework defaults:** FastAPI/Starlette handles Content-Type automatically for most response types. The only gap is the SSE streaming endpoint where manual `media_type` specification omits the charset. This is a narrow, low-risk gap.

3. **Two architectural layers of data flow control:** The application has both Pydantic request models (controlling input/mass assignment) and response models (controlling output/field exposure). Input models are consistently applied across all POST/PUT endpoints. Output models have gaps as documented above.

4. **Service layer abstraction creates opacity:** Functions like `get_available_providers()`, `build_agent_chain()`, and `run_deployed_agent_logic()` are imported from `dependencies` and their internal data shapes are not visible in the route layer. Without response_model enforcement, the route layer cannot guarantee what fields these services include in their return values.

---

## Cross-cutting Recommendations

### Immediate (Sprint 1)
1. **Add `response_model` to all data-returning endpoints** — Prioritize `GET /sessions/{session_id}/config` (returns raw DB document) and provider configuration endpoints. Define explicit Pydantic models that declare only the fields the client needs.

2. **Add `charset=utf-8`** to the SSE streaming endpoint's `media_type` parameter.

### Short-term (Sprint 2-3)
3. **Audit service layer return values** — Review `get_available_providers()`, `build_agent_chain()`, `get_agent_deployment()`, and `run_deployed_agent_logic()` to understand what fields they return and whether any contain sensitive internal data.

4. **Establish a code review checklist** requiring `response_model` on all new endpoints that return data objects, enforced via linting or PR template.

5. **Add integration tests** that verify response payloads contain ONLY expected fields — catch regressions when service functions add new internal fields.

### Long-term (Quarter)
6. **Consider a response serialization middleware** that logs warnings when an endpoint returns a dict or object without a declared response_model, making gaps visible during development.

7. **Implement OpenAPI response schema validation** in CI that compares actual response shapes against declared schemas, catching drift between implementation and contract.
