## ASVS-15.3.1: Defensive Coding - Return Only Required Fields

### Findings for 15.3.1

### [MEDIUM]

**Finding ID:** ASVS-1531-MED-001

**File:** `webapp/packages/api/user-service/routes.py`, function `get_session_config`, line ~410-414

**Description:** The endpoint returns raw database document content without a `response_model` or explicit field selection. The `provider_config` sub-document is returned directly from the database service with no schema-based filtering, meaning any field stored in that sub-document (including potentially sensitive internal metadata, revision fields, or inadvertently stored secrets) will be exposed to the client.

**Vulnerable Code:**
```python
@router.get("/sessions/{session_id}/config")
async def get_session_config(session_id: str, db: DatabaseService = Depends(get_db), user: dict = Depends(get_current_user)):
    """Get session configuration"""
    session_doc = db.get("sessions", session_id)
    return session_doc.get("provider_config")
```

**Data Flow:** Database (`sessions` collection) → `db.get()` → raw document → `.get("provider_config")` → client response. No Pydantic response_model filters the output.

**Proof of Concept:**
```
GET /sessions/some-session-id/config
Authorization: Bearer <valid-token>
```
If the `provider_config` object stored in the database contains internal fields (e.g., a cached API key, internal model pricing data, or debug flags added by background processing), all fields are returned verbatim.

**Impact:** Information disclosure of potentially sensitive session configuration data. The severity depends on what gets stored in `provider_config` — at minimum it violates the principle of only returning explicitly declared fields.

**Remediation:**
```python
from models.chat import ProviderConfig

@router.get("/sessions/{session_id}/config", response_model=ProviderConfig)
async def get_session_config(session_id: str, db: DatabaseService = Depends(get_db), user: dict = Depends(get_current_user)):
    """Get session configuration"""
    session_doc = db.get("sessions", session_id)
    config_data = session_doc.get("provider_config")
    if not config_data:
        raise HTTPException(status_code=404, detail="No configuration found")
    return ProviderConfig(**config_data)
```

### [MEDIUM]

**Finding ID:** ASVS-1531-MED-002

**File:** `webapp/packages/api/user-service/routes.py`, functions `get_providers`, `get_provider_config_route`, `get_model_config`, lines ~206-237

**Description:** Provider configuration endpoints return the full output of `get_available_providers()` or sub-objects thereof without any `response_model`. This function returns provider configuration dictionaries that may contain internal implementation details (endpoint URLs, pricing metadata, capability flags, internal identifiers) not intended for client consumption.

**Vulnerable Code:**
```python
@router.get("/providers")
def get_providers(user: dict = Depends(get_current_user)):
    """Get all available providers and their configurations"""
    return get_available_providers(user.get("uid", "anonymous"), user)

@router.get("/providers/{provider}")
def get_provider_config_route(provider: str, user: dict = Depends(get_current_user)):
    """Get configuration for a specific provider"""
    available_providers = get_available_providers(user.get("uid", "anonymous"), user)
    if provider not in available_providers:
        raise HTTPException(status_code=404, detail="Provider not found or not configured")
    return available_providers[provider]

@router.get("/providers/{provider}/models/{model}")
def get_model_config(provider: str, model: str, user: dict = Depends(get_current_user)):
    """Get configuration for a specific model"""
    available_providers = get_available_providers(user.get("uid", "anonymous"), user)
    # ...
    return available_providers[provider]["models"][model]
```

**Data Flow:** `get_available_providers()` service function → full provider configuration dict → directly serialized to client. No response_model declaration strips internal fields.

**Proof of Concept:**
```
GET /providers
Authorization: Bearer <valid-token>
```
Response may contain internal fields like base URLs, token limits, internal pricing data, feature flags, or other configuration that should remain server-side.

**Impact:** Information disclosure of internal service configuration. Depending on what `get_available_providers()` includes, this could reveal infrastructure details useful for further attacks.

**Remediation:** Define explicit Pydantic response models:
```python
class ProviderResponse(BaseModel):
    name: str
    display_name: str
    models: List[str]  # Only model names, not full config

class ProviderDetailResponse(BaseModel):
    name: str
    display_name: str
    models: Dict[str, ModelSummary]

@router.get("/providers", response_model=Dict[str, ProviderResponse])
def get_providers(user: dict = Depends(get_current_user)):
    return get_available_providers(user.get("uid", "anonymous"), user)
```

### [MEDIUM]

**Finding ID:** ASVS-1531-MED-003

**File:** `webapp/packages/api/user-service/routes.py`, function `get_agent_chain`, lines ~469-481; function `get_agent_deployment_route`, lines ~586-589

**Description:** Agent chain and deployment endpoints return service function results without `response_model` filtering. The `build_agent_chain()` function walks transitive dependencies and may include internal IDs, raw database document fields, or infrastructure details (MCP server URLs, internal routing information) that should be filtered before client delivery.

**Vulnerable Code:**
```python
@router.get("/agents/{agent_id}/chain")
async def get_agent_chain(
    agent_id: str,
    db: DatabaseService = Depends(get_db),
    user: dict = Depends(get_current_user),
):
    """Return the transitive call graph for an agent."""
    return await build_agent_chain(agent_id, db)

@router.get("/agents/{agent_id}/deployment")
async def get_agent_deployment_route(agent_id: str, db: DatabaseService = Depends(get_db), user: dict = Depends(get_current_user)):
    """Checks if an agent is deployed and returns its public-facing name."""
    return await get_agent_deployment(agent_id, db)
```

**Data Flow:** Database documents (agents, deployments collections) → dependency resolution logic → full result dict → client without field filtering.

**Proof of Concept:**
```
GET /agents/<agent-id>/chain
Authorization: Bearer <valid-token>
```
The response may include internal database IDs (`_id`, `_rev`), MCP server endpoint URLs (potentially internal network addresses), or other agent configuration not needed by the Chain View UI.

**Impact:** Potential exposure of internal infrastructure details (MCP server addresses, internal identifiers) that could aid in lateral movement or targeted attacks.

**Remediation:** Define response models that explicitly declare the fields the Chain View UI needs:
```python
class ChainNode(BaseModel):
    agent_id: str
    name: str
    type: str  # "agent" or "mcp_server"
    dependencies: List[str] = []

@router.get("/agents/{agent_id}/chain", response_model=List[ChainNode])
async def get_agent_chain(...):
    return await build_agent_chain(agent_id, db)
```

### [LOW]

**Finding ID:** ASVS-1531-LOW-001

**File:** `webapp/packages/api/user-service/routes.py`, function `run_deployed_agent_route`, lines ~810-827

**Description:** The deployed agent execution endpoint returns the full result from `run_deployed_agent_logic()` without a `response_model`. While agent output is inherently dynamic, the lack of any response envelope or filtering means internal execution metadata could leak alongside the intended result.

**Vulnerable Code:**
```python
@router.post("/rest/{friendly_name}")
async def run_deployed_agent_route(
    friendly_name: str, 
    request: Request, 
    db: DatabaseService = Depends(get_db),
    user: dict = Depends(get_current_user)
):
    """Run a deployed agent by its friendly_name. Requires authentication."""
    input_dict = await request.json()
    user_basic_info = {
        "email": user.get("email"),
        "name": user.get("name") or user.get("displayName"),
    }
    return await run_deployed_agent_logic(
        friendly_name, 
        input_dict, 
        db, 
        user_id=user.get("uid"),
        user_basic_info=user_basic_info,
    )
```

**Impact:** If `run_deployed_agent_logic` returns execution metadata alongside the agent result (e.g., execution time, internal error details, token counts, cost data), these could be exposed to consumers of the deployed API.

**Remediation:** Wrap the response in a defined envelope model:
```python
class DeployedAgentResponse(BaseModel):
    result: Any
    status: str = "success"

@router.post("/rest/{friendly_name}", response_model=DeployedAgentResponse)
async def run_deployed_agent_route(...):
    result = await run_deployed_agent_logic(...)
    return DeployedAgentResponse(result=result)
```

### [LOW]

**Finding ID:** ASVS-1531-LOW-002

**File:** `webapp/packages/api/user-service/routes.py`, function `list_mcp_tools`, lines ~544-553

**Description:** The MCP tools endpoint returns the raw tool list from `mcp_service.list_tools_for_server()` without a response model. MCP tool definitions returned by remote servers could contain internal metadata fields not appropriate for client exposure.

**Vulnerable Code:**
```python
@router.post("/mcp/tools")
async def list_mcp_tools(
    request: ListMcpToolsRequest,
    mcp_service: McpClientService = Depends(get_mcp_client_service),
    user: dict = Depends(get_current_user)
):
    """Connects to a remote MCP server and lists its available tools."""
    tools = await mcp_service.list_tools_for_server(request.mcp_url, request.auth_token)
    return {"mcp_url": request.mcp_url, "tools": tools}
```

**Impact:** External MCP server responses are passed through without field filtering. If an MCP server includes internal annotations, debug info, or metadata in tool definitions, these are forwarded to the client.

**Remediation:** Define a response model that explicitly declares which tool fields the UI needs:
```python
class McpTool(BaseModel):
    name: str
    description: Optional[str] = None
    input_schema: Optional[Dict[str, Any]] = None

class ListMcpToolsResponse(BaseModel):
    mcp_url: str
    tools: List[McpTool]

@router.post("/mcp/tools", response_model=ListMcpToolsResponse)
```

### Positive Patterns for 15.3.1

1. **Consistent use of Pydantic response models for CRUD operations** — Agent, DemoApp, DataStoreRecord, User, and ChatResponse models ensure that database documents are serialized through a defined schema, stripping internal fields like `_id` and `_rev`.

2. **API key masking in `get_user_api_keys`** — The `GET /users/me/api-keys` endpoint returns an `ApiKeys` response model that explicitly masks key values, demonstrating awareness of field-level sensitivity.

3. **Effective API key endpoint** (`GET /users/me/api-keys/{provider}/effective`) intentionally returns only `has_key` and `source` without exposing the actual key value — a textbook example of returning only the required subset.

4. **`SessionUser` model in `GET /auth/me`** explicitly selects only the fields the frontend needs (uid, display_name, email, provider_type, workspaces, is_site_admin) from the full session object.

5. **`RunCodeResponse` model** wraps sandbox execution results in a structured envelope with defined fields (result, error, schema_warnings, ops_log, trace), preventing internal execution details from leaking.

6. **`exclude_unset=True` in update operations** — The `update_agent` endpoint uses `request.model_dump(by_alias=True, exclude_unset=True)` preventing mass assignment of unintended fields through the update request model.

## Cross-cutting Architecture Observations

1. **Inconsistent response filtering pattern (Type B gap):** FastAPI's `response_model` is a powerful control that serializes responses through Pydantic schemas, automatically stripping undeclared fields. This control EXISTS in the codebase (used on ~60% of data endpoints) but is NOT APPLIED to provider configuration, session config, agent chain, agent deployment, MCP tools, and deployed agent execution endpoints. This creates a false sense of security — developers may assume all endpoints are filtered because many are.

2. **Content-Type handling is largely delegated to framework defaults:** FastAPI/Starlette handles Content-Type automatically for most response types. The only gap is the SSE streaming endpoint where manual `media_type` specification omits the charset. This is a narrow, low-risk gap.

3. **Two architectural layers of data flow control:** The application has both Pydantic request models (controlling input/mass assignment) and response models (controlling output/field exposure). Input models are consistently applied across all POST/PUT endpoints. Output models have gaps as documented above.

4. **Service layer abstraction creates opacity:** Functions like `get_available_providers()`, `build_agent_chain()`, and `run_deployed_agent_logic()` are imported from `dependencies` and their internal data shapes are not visible in the route layer. Without response_model enforcement, the route layer cannot guarantee what fields these services include in their return values.

## Cross-cutting Recommendations

### Immediate (Sprint 1)
1. **Add `response_model` to all data-returning endpoints** — Prioritize `GET /sessions/{session_id}/config` (returns raw DB document) and provider configuration endpoints. Define explicit Pydantic models that declare only the fields the client needs.

2. **Add `charset=utf-8`** to the SSE streaming endpoint's `media_type` parameter.

### Short-term (Sprint 2-3)
3. **Audit service layer return values** — Review `get_available_providers()`, `build_agent_chain()`, `get_agent_deployment()`, and `run_deployed_agent_logic()` to understand what fields they return and whether any contain sensitive internal data.

4. **Establish a code review checklist** requiring `response_model` on all new endpoints that return data objects, enforced via linting or PR template.

5. **Add integration tests** that verify response payloads contain ONLY expected fields — catch regressions when service functions add new internal fields.

### Long-term (Quarter)
6. **Consider a response serialization middleware** that logs warnings when an endpoint returns a dict or object without a declared response_model, making gaps visible during development.

7. **Implement OpenAPI response schema validation** in CI that compares actual response shapes against declared schemas, catching drift between implementation and contract.
