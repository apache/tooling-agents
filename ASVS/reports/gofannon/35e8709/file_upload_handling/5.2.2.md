## ASVS-5.2.2: File Extension and Content Type Validation

### Findings for 5.2.2

#### MEDIUM
**Finding ID**: ASVS-522-MED-001

**Title**: No content validation for swagger specification inputs

**File**: `webapp/packages/api/user-service/agent_factory/__init__.py`, function `generate_agent_code`, lines 34-41

**Vulnerable Code**:
```python
if request.swagger_specs:
    from .swagger_parser import parse_spec_and_generate_docs
    swagger_docs_parts = []
    for spec in request.swagger_specs:
        docs_for_spec = parse_spec_and_generate_docs(spec.name, spec.content)
        if docs_for_spec:
            swagger_docs_parts.append(docs_for_spec)
```

**Data Flow**: User input → `spec.name` (no extension validation) → `spec.content` (no format validation) → `parse_spec_and_generate_docs()`

**Analysis**: The `swagger_specs` accept content with a `name` and `content` field. There is no validation that:
1. `spec.name` has an expected extension (e.g., `.json`, `.yaml`, `.yml`)
2. `spec.content` is actually a valid OpenAPI/Swagger specification (checking magic bytes equivalent — e.g., verifying it starts with `openapi:` or `swagger:` for YAML, or contains `"openapi"` key for JSON)
3. The content type matches what `spec.name` implies

The `parse_spec_and_generate_docs` function is imported lazily and its implementation is not visible, so it's unknown whether it performs internal validation. If it doesn't validate, malicious content (e.g., YAML bombs, XXE in XML-based specs, or binary content) could be processed.

**Proof of Concept**:
```json
{
  "swagger_specs": [
    {
      "name": "exploit.xml",
      "content": "<?xml version=\"1.0\"?><!DOCTYPE foo [<!ENTITY xxe SYSTEM \"file:///etc/passwd\">]><root>&xxe;</root>"
    }
  ]
}
```

**Impact**: If the parser handles multiple formats (JSON, YAML, XML), unexpected content types could trigger parser-specific vulnerabilities. Even without exploitation, non-spec content wastes processing resources.

**Remediation**:
```python
ALLOWED_SPEC_EXTENSIONS = {'.json', '.yaml', '.yml'}

if request.swagger_specs:
    from .swagger_parser import parse_spec_and_generate_docs
    import os
    swagger_docs_parts = []
    for spec in request.swagger_specs:
        # Validate extension
        ext = os.path.splitext(spec.name)[1].lower()
        if ext not in ALLOWED_SPEC_EXTENSIONS:
            raise ValueError(f"Unsupported spec format: {ext}. Allowed: {ALLOWED_SPEC_EXTENSIONS}")
        
        # Validate content matches expected format
        content_stripped = spec.content.strip()
        if ext == '.json' and not (content_stripped.startswith('{') or content_stripped.startswith('[')):
            raise ValueError(f"Spec '{spec.name}' claims JSON but content is not JSON")
        
        docs_for_spec = parse_spec_and_generate_docs(spec.name, spec.content)
```

## Cross-cutting Architecture Observations

1. **File handling is not a primary function of this module**: The `agent_factory/__init__.py` is a code generation service, not a file upload handler. File-like content (swagger specs) is accepted as structured JSON request data rather than multipart file uploads. The traditional file upload concerns (magic bytes, MIME type, storage location) apply differently here.

2. **Defense delegation pattern**: The code delegates security-critical operations to other modules (`swagger_parser`, `RemoteMCPClient`, `call_llm`) whose implementations are not available for audit. The effectiveness of file-related controls depends on whether these downstream modules implement proper validation.

3. **Input boundary is upstream**: The actual request validation (field sizes, types, counts) is likely handled at the FastAPI route handler level via Pydantic models (`GenerateCodeRequest`). Without seeing the model definition and route handler, it's impossible to confirm whether size limits and input constraints are enforced before this function is called.

4. **No file persistence in this module**: All processing is in-memory and results are returned as API responses. This eliminates several file storage risks (5.3.1, 5.3.2) at this layer, though they may exist in other parts of the system that store the generated code.

5. **MinIO storage mentioned in domain context**: The docker-compose includes MinIO, suggesting file storage exists elsewhere. This audit cannot confirm whether file upload endpoints in other modules properly implement ASVS 5.2.x and 5.3.x controls.

## Cross-cutting Recommendations

### Immediate (High Priority)
1. **Add size limits to swagger_specs**: Implement per-spec and total content size limits before parsing, either in the Pydantic model (`max_length` on content field) or at the function entry point.
2. **Add collection count limits**: Limit the number of `swagger_specs`, `tools` URLs, `gofannon_agents`, and `invokable_models` per request.
3. **Reduce LLM timeout**: The 1800-second timeout is excessive for code generation. Consider a more reasonable limit (e.g., 120-300 seconds) with retry logic.

### Short-term (Within Sprint)
4. **Audit `swagger_parser` module**: Verify that `parse_spec_and_generate_docs` does not use `spec.name` for file operations and properly handles malformed input.
5. **Validate swagger spec content format**: Before passing to the parser, perform a basic structure check (valid JSON/YAML, contains expected OpenAPI keys).
6. **Sanitize spec.name**: Apply `os.path.basename()` and character allowlisting to spec names before passing to downstream functions.

### Long-term (Architecture)
7. **Centralized file validation middleware**: If other endpoints accept file-like content, implement a shared validation layer that enforces size limits, content type checking, and filename sanitization.
8. **Document file handling architecture**: Clarify whether MinIO-based file storage has its own upload validation layer, and ensure ASVS 5.2.x/5.3.x controls are implemented wherever files enter the system.
9. **Rate limiting on code generation**: Given the expensive nature of LLM calls (30-minute potential hold time), implement per-user rate limiting on the `generate_agent_code` endpoint to prevent resource exhaustion attacks.
