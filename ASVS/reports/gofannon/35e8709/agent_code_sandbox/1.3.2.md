# ASVS 1.3.2 Security Audit Report

## Executive Summary

**Repository:** The-AI-Alliance/gofannon  
**Audit Date:** May 05, 2026  
**Auditor:** Tooling Agents  
**ASVS Requirement:** ASVS 1.3.2  
**Description:** Verify that the application avoids the use of eval() or other dynamic code execution features such as Spring Expression Language (SpEL). Where there is no alternative, any user input being included must be sanitized before being executed.

**Files Analyzed:** 4 relevant out of 4 total, 0 skipped

**Finding Counts:**

| Severity | Count |
|----------|-------|
| Critical | 1 |
| High | 1 |
| Medium | 1 |
| Low | 0 |
| **Total** | **3** |

The application uses `exec()` for agent code execution (acknowledged as an intentional design decision). However, the **claimed mitigation** for this dynamic code execution — a sandbox with restricted builtins and controlled globals — **does not exist in the implementation**. The domain context states "The sandbox removes dangerous builtins (open, eval, exec, compile, __import__)" but the code passes the full unfiltered `__builtins__` to executed code, providing complete Python runtime access.

## Findings

### High Severity

#### ASVS-132-HIGH-002: No Execution Timeout on Agent Code

**Severity**: HIGH  
**Gap Type**: Type A — Entry point with NO control  
**File**: `webapp/packages/api/user-service/dependencies.py`  
**Function**: `_execute_agent_code`  
**Lines**: ~334-350

**Related Functions Checked:**
- `_execute_agent_code` (main execution path)
- `run_function` (agent entry point)
- `asyncio.wait_for` (not used, should be)

**Description:**

Agent code execution via `run_function()` is awaited without any timeout mechanism. This allows malicious or buggy agent code to run indefinitely, consuming CPU and memory resources, potentially causing denial of service.

**Vulnerable Code:**

```python
# With trace
if trace is not None:
    _agent_start_ms = trace.agent_start(...)
    with bind_trace(trace), capture_user_io(trace):
        try:
            result = await run_function(input_dict=input_dict, tools=tools)  # ← No timeout
            ...
# Without trace
else:
    result = await run_function(input_dict=input_dict, tools=tools)  # ← No timeout
```

**Data Flow:**

```
Source: Agent code defining run() function
    → run_function = local_scope.get("run")
    → await run_function(input_dict=input_dict, tools=tools)
    → Missing Control: No asyncio.wait_for() or timeout mechanism
    → Impact: CPU/memory exhaustion, event loop starvation
```

**Proof of Concept:**

Infinite CPU loop:

```python
async def run(input_dict, tools):
    # Infinite loop - never returns
    while True:
        _ = [i ** 2 for i in range(10000000)]  # CPU burn
    return {"outputText": "never reached"}
```

Memory exhaustion:

```python
async def run(input_dict, tools):
    data = []
    while True:
        data.append("x" * 1024 * 1024)  # 1MB per iteration
    return {"outputText": "never reached"}
```

**Impact:**

- Denial of service through CPU exhaustion
- Memory exhaustion crashing the server process
- Event loop starvation preventing other requests from processing
- Compounded by lack of builtins restriction (can spawn processes that outlive the Python task)

**Remediation:**

```python
import asyncio

AGENT_EXECUTION_TIMEOUT = int(os.getenv("AGENT_EXECUTION_TIMEOUT", "300"))  # 5 min default

try:
    result = await asyncio.wait_for(
        run_function(input_dict=input_dict, tools=tools),
        timeout=AGENT_EXECUTION_TIMEOUT,
    )
except asyncio.TimeoutError:
    if trace is not None:
        trace.error(TimeoutError(f"Agent execution exceeded {AGENT_EXECUTION_TIMEOUT}s timeout"))
    raise HTTPException(
        status_code=408,
        detail=f"Agent execution timed out after {AGENT_EXECUTION_TIMEOUT} seconds"
    )
```

### Medium Severity

#### ASVS-132-MED-003: `compile()` Used Without Source Identification

**Severity**: MEDIUM  
**File**: `webapp/packages/api/user-service/dependencies.py`  
**Function**: `_execute_agent_code`  
**Line**: ~293

**Related Functions Checked:**
- `compile()` (code compilation)
- Error handling and traceback generation

**Description:**

The `compile()` call uses the generic filename `"<string>"` which makes forensic analysis of errors and potential exploits harder. When exceptions occur in agent code, tracebacks will show `File "<string>"` without identifying which agent or run produced the error.

**Vulnerable Code:**

```python
code_obj = compile(code, "<string>", "exec")
```

**Data Flow:**

```
Source: code parameter (agent source code)
    → compile(code, "<string>", "exec")  
    → Missing Control: No agent/run identification in compiled filename
    → Impact: Reduced forensic capability, harder incident response
```

**Impact:**

- Tracebacks from malicious code are harder to attribute
- Multiple concurrent agent executions produce indistinguishable stack traces
- Incident response is slowed when investigating exploitation attempts

**Remediation:**

```python
# Include agent identifier in the compile filename for better forensics
source_label = f"<agent:{agent_name or 'unnamed'}:{trace._stack[-1] if trace else 'no-trace'}>"
code_obj = compile(code, source_label, "exec")
```

## Positive Security Patterns

1. **Trace observability system** (`agent_trace.py`): Well-designed event collection with appropriate caps (2000 events, 4KB per event), contextvar-based scoping, and proper cleanup on exit.

2. **LLM timeout auto-scaling** (`dependencies.py`, lines ~224-240): The `call_llm_with_context` wrapper intelligently scales timeouts based on max_tokens and reasoning_effort parameters.

3. **Agent chain cycle detection** (`dependencies.py`, `build_agent_chain`): Proper cycle detection with `path` set tracking, depth limiting at 8 levels, and graceful handling of missing agents.

4. **Data store user scoping** (`dependencies.py`): `AgentDataStoreProxy` is initialized with `user_id` and `agent_name`, providing per-user isolation at the data layer.

5. **Output schema validation** (`dependencies.py`, `validate_output_against_schema`): Advisory validation that checks types, missing keys, and extra keys without blocking execution.

6. **Event source tagging** (`agent_trace.py`): Events tagged with `source: "system"` vs `source: "stdout"/"log"` enabling future filtering for audit logs vs user-visible output.

7. **Operator control over capture** (`agent_trace.py`): `GOFANNON_DISABLE_USER_TRACE` allows operators to silence noisy user-origin events while keeping structural tracing.

8. **LLM call tracing** (`dependencies.py`, lines ~242-264): Every LLM call records provider, model, duration, and error status for observability regardless of outcome.

## Architecture Observations

1. **Single-process execution model**: All agent code runs in the same Python process as the API server. There is no process-level, container-level, or VM-level isolation between agents or between agents and the platform.

2. **Trust model gap**: The domain context assumes LLM-generated code is benign ("trust assumption that LLM-generated code is safe"), but the platform also supports user-edited code in the sandbox UI. Even with only LLM-generated code, prompt injection attacks can cause the LLM to generate malicious code.

3. **Shared httpx client**: Each agent execution creates its own `httpx.AsyncClient` but with no request restrictions (no allowlist, no rate limiting, no SSRF protection). This is intentional for API calls but combined with unrestricted builtins, provides a data exfiltration channel.

4. **Recursive execution without depth limit**: While `build_agent_chain` limits graph traversal to depth 8, actual recursive execution via `GofannonClient.call()` has no depth limit, allowing stack exhaustion.

## Recommendations Summary

### Immediate (P0 - CRITICAL)

1. **Implement builtin restriction** (CRIT-001): Replace `"__builtins__": __builtins__` with a filtered dict that removes `open`, `eval`, `exec`, `compile`, `__import__`, `getattr`, `setattr`, `delattr`, `globals`, `locals`, `vars`, and `type`. This is the minimum control that the entire security architecture claims to have.

2. **Add process-level isolation**: Even with builtin restriction, Python sandboxes are bypassable. Implement one of:
   - nsjail/bubblewrap per execution
   - Container per execution with no network (except explicit proxy)
   - WebAssembly runtime (e.g., wasmtime-py)

### Short-term (P1 - HIGH)

3. **Add execution timeout** (HIGH-002): Wrap `await run_function()` in `asyncio.wait_for()` with a configurable timeout (default 300s).

4. **Add memory limit**: Use `resource.setrlimit()` or container memory limits to cap per-execution memory consumption.

5. **Add recursive execution depth tracking**: Pass depth counter through `GofannonClient.call()` and reject calls beyond `MAX_AGENT_DEPTH`.

### Long-term (P2 - MEDIUM)

6. **Improve compile source identification** (MED-003): Include agent name/ID in the compile filename for forensics.

7. **Add network allowlist**: Restrict `httpx` requests to user-configured endpoints rather than allowing arbitrary outbound connections.

8. **Consider static analysis**: While the false positive patterns note this is intentional, a lightweight check for obviously dangerous patterns (`import os`, `import subprocess`, `open(`) could provide defense-in-depth warnings without blocking execution.
