#### MEDIUM

**Finding ID: ASVS-611-MED-001**
**Title: No Documented Configuration for Preventing Malicious Account Lockout via Spend Allowance**

- **File:** `webapp/packages/api/user-service/services/llm_service.py`, line 113-115

**Relevant code:**
```python
if user_service and user_id:
    user_service.require_allowance(user_id, basic_info=user_basic_info)
```

**Issue:** The spend allowance is the sole anti-abuse control, but there is no documentation defining:
1. How `require_allowance()` differentiates between legitimate high-usage and malicious behavior
2. Whether an attacker who compromises a user's session could deliberately exhaust their allowance (denial of service)
3. How allowances reset (time-based? manual? admin override?)
4. Whether there are alerts when a user approaches their limit
5. Whether there are separate rate-based limits to prevent burst consumption (as demonstrated in ASVS-231-HIGH-001)

**Impact:** Without documented configuration for these controls, operators cannot properly configure the system to prevent:
- Attackers deliberately exhausting another user's budget via stolen session tokens
- Legitimate users being locked out due to automated integrations consuming their allowance
- Cost-based denial-of-service against the platform

**Remediation:** Create documentation that covers:
```markdown
## Spend Allowance Configuration

### Allowance Structure
- Monthly allowance per user: configurable via admin panel
- Reset schedule: 1st of each month at 00:00 UTC
- Grace period: 10% over-limit allowed for in-flight requests

### Abuse Prevention
- Concurrent request limit: max 5 simultaneous LLM calls per user
- Burst detection: >20 requests in 60s triggers temporary rate limit
- Alert threshold: notification at 80% and 95% of allowance

### Recovery from Exhaustion
- Users can request emergency allowance increase via support
- Admin can reset allowance without waiting for monthly cycle
- Stolen session tokens can be revoked without affecting allowance
```

#### MEDIUM

**Finding ID: ASVS-611-MED-002**
**Title: No Circuit Breaker or Backpressure Documentation for External LLM Service Failures**

- **File:** `webapp/packages/api/user-service/services/llm_service.py`, lines 256-268

**Relevant code:**
```python
# Retry logic exists but only for timeouts, no circuit breaker
last_exception = None
for attempt in range(1 + MAX_TIMEOUT_RETRIES):
    try:
        response = await litellm.acompletion(**kwargs)
        break
    except litellm.Timeout as e:
        last_exception = e
        if attempt < MAX_TIMEOUT_RETRIES:
            wait_secs = 5 * (attempt + 1)
            await asyncio.sleep(wait_secs)
            continue
        raise
```

**Issue:** The retry mechanism handles timeouts but there is no:
1. Circuit breaker to stop calling a failed provider after repeated failures
2. Documentation of how the system behaves under sustained LLM provider outage
3. Backpressure mechanism to reject new requests when the external service is degraded
4. Documentation of how `MAX_TIMEOUT_RETRIES` and `LLM_TIMEOUT_SECONDS` should be configured for different deployment scenarios

**Impact:** During an LLM provider outage, all user requests will queue up with 600-second timeouts, consuming server resources (memory, connections, asyncio tasks) without producing results. This can cascade into a full service outage.

**Remediation:** Document circuit breaker configuration and implement:
```markdown
## External Service Resilience

### Circuit Breaker Configuration
- Failure threshold: 5 failures in 30 seconds triggers circuit open
- Circuit open duration: 60 seconds before half-open probe
- Half-open: 1 probe request allowed; success closes circuit

### Timeout Configuration
- LLM_TIMEOUT_SECONDS: 600 (default), recommended 120 for production
- LLM_TIMEOUT_RETRIES: 0 (default), recommended 1 for production
- Total max wait: timeout × (1 + retries) = 240s maximum per request

### Backpressure
- Max concurrent LLM calls per instance: 50
- Queue overflow response: HTTP 503 with Retry-After header
```

## Cross-cutting Architecture Observations

1. **Separation of concerns creates security gaps:** The `chat_service.py` acts as a caller of `llm_service.py` but deliberately opts out of security controls by passing `None` for authorization parameters. This architectural pattern means any new service that calls `call_llm()` or `stream_llm()` must remember to pass the correct user context — a fragile security model.

2. **Optional security controls are dangerous:** The `if user_service and user_id:` pattern in `llm_service.py` makes security controls optional. If any caller passes `None` (intentionally or accidentally), all protections are bypassed. Security controls should be mandatory with explicit opt-out requiring elevated privileges.

3. **Async processing disconnects authorization from execution:** The `create_chat_ticket()` method spawns a background task that executes the LLM call. Even if the HTTP handler enforces authorization, the background task has no reference to the authenticated user's context. This is a systemic pattern where async processing can outlive the authorization context.

4. **No defense-in-depth:** The spend allowance is the single point of control. There are no layered defenses (rate limiting → allowance check → concurrent request cap → circuit breaker). If the allowance is bypassed (as demonstrated), there are no fallback controls.

5. **Cost tracking gap in streaming:** The explicit comment "Cost tracking is not available for streaming responses" reveals a known gap that has not been addressed. This creates a permanent bypass path for the primary business control.

## Cross-cutting Recommendations

### Immediate (Critical — address before next release)

1. **Fix chat_service spend bypass:** Remove `user_service=None` and `user_id=None` from all `call_llm()`/`stream_llm()` invocations in `chat_service.py`. Require `user_id` as a mandatory parameter for `create_chat_ticket()` and `stream_chat()`.

2. **Make allowance check mandatory:** Refactor `call_llm()` and `stream_llm()` to require `user_id` (no default to "anonymous"). If truly anonymous access is needed, create a separate function with explicit naming (e.g., `call_llm_unmetered()`).

3. **Add concurrent request limiting:** Implement a per-user semaphore or counter that prevents more than N simultaneous LLM requests from passing the allowance check.

### Short-term (High — within 1-2 sprints)

4. **Implement rate limiting middleware:** Add per-user and per-IP rate limiting at the API gateway or middleware level. Document configuration in deployment guides.

5. **Add atomic allowance reservation:** Replace check-then-use with atomic reserve-execute-finalize pattern to eliminate TOCTOU race condition.

6. **Implement streaming cost tracking:** Use token counting or litellm's stream cost utilities to record usage for streaming responses.

7. **Create security controls documentation:** Document all rate limiting, anti-automation, and adaptive response mechanisms including their configuration parameters, defaults, and how they prevent malicious lockout.

### Long-term (Medium — within next quarter)

8. **Add circuit breakers:** Implement circuit breaker pattern for external LLM provider calls with documented configuration.

9. **Implement ticket ownership model:** Add proper authorization to ticket retrieval, ensuring users can only access their own tickets.

10. **Centralize authorization in middleware:** Move authorization checks out of individual service functions into middleware that cannot be bypassed by callers. Use dependency injection frameworks to ensure every LLM call path includes authorization.
