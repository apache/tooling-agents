#### HIGH

**Finding ID: ASVS-231-HIGH-001**
**Title: Race Condition in Spend Allowance Check Enables Concurrent Bypass (TOCTOU)**

- **File:** `webapp/packages/api/user-service/services/llm_service.py`, lines 113-115 (check) and line 328 (usage recording)

**Vulnerable code:**
```python
# Step 1: Check allowance (line 113-115)
if user_service and user_id:
    user_service.require_allowance(user_id, basic_info=user_basic_info)

# ... LLM call takes 1-600 seconds ...

# Step 2: Record usage (line 326-328) — MUCH LATER
if user_service and user_id:
    response_cost = _extract_response_cost(...)
    if response_cost is not None:
        user_service.add_usage(user_id, response_cost, basic_info=user_basic_info)
```

**Data flow:** User sends N concurrent requests → All N pass `require_allowance()` simultaneously (since none have recorded usage yet) → All N LLM calls execute → All N record usage after completion → Total spend = N × per-request cost, potentially far exceeding allowance

**Proof of Concept:**
```python
import asyncio, aiohttp

async def exploit():
    """Send 50 concurrent requests to exhaust budget in one burst"""
    async with aiohttp.ClientSession() as session:
        tasks = [
            session.post("/api/chat/stream", json={
                "messages": [{"role": "user", "content": "expensive prompt"}],
                "model": "openai/gpt-4",
                "config": {"max_tokens": 4000}
            }, headers={"Authorization": "Bearer <token>"})
            for _ in range(50)
        ]
        await asyncio.gather(*tasks)
```

**Impact:** A user with $5 remaining allowance could send 50 concurrent requests, all of which pass the check, resulting in potentially $250+ in charges before any usage is recorded. The domain context explicitly identifies this as a critical concern.

**Remediation:**
```python
# Use atomic decrement or pessimistic locking
async def call_llm(...):
    if user_service and user_id:
        # Reserve estimated cost atomically BEFORE the call
        estimated_cost = estimate_cost(model, messages, parameters)
        reservation_id = user_service.reserve_allowance(user_id, estimated_cost)
        try:
            response = await litellm.acompletion(**kwargs)
            actual_cost = _extract_response_cost(response)
            user_service.finalize_reservation(reservation_id, actual_cost)
        except Exception:
            user_service.release_reservation(reservation_id)
            raise
```

#### HIGH

**Finding ID: ASVS-231-HIGH-002**
**Title: Streaming Endpoint Skips Cost Recording Step Entirely**

- **File:** `webapp/packages/api/user-service/services/llm_service.py`, `stream_llm()` function (lines 348-410)

**Vulnerable code:**
```python
async def stream_llm(...) -> AsyncGenerator[Any, None]:
    """
    ...
    Note: Cost tracking is not available for streaming responses.
    """
    # ... allowance check happens ...
    if user_service and user_id:
        user_service.require_allowance(user_id, basic_info=user_basic_info)

    # ... LLM call happens ...
    response = await litellm.acompletion(**kwargs)
    async for chunk in response:
        yield chunk
    
    # NO add_usage() call anywhere after streaming completes
```

**Data flow:** User → `stream_llm()` → allowance check passes → LLM call executes → response streamed → **cost never recorded** → allowance never decremented

**Expected sequential flow:**
1. ✅ Check allowance
2. ✅ Execute LLM call
3. ❌ Record usage (SKIPPED)

**Impact:** Users who exclusively use streaming endpoints never have their usage recorded. Their allowance never decreases, effectively granting unlimited LLM access. This is a step-skipping vulnerability where the final step in the business logic flow is never executed.

**Remediation:**
```python
async def stream_llm(...) -> AsyncGenerator[Any, None]:
    # ... existing code ...
    try:
        response = await litellm.acompletion(**kwargs)
        async for chunk in response:
            yield chunk
    finally:
        # Track usage based on streamed content or use litellm's stream cost tracking
        if user_service and user_id:
            # Use litellm's stream_cost_tracking or estimate based on tokens
            estimated_cost = litellm.stream_chunk_builder_cost(chunks)
            if estimated_cost:
                user_service.add_usage(user_id, estimated_cost, basic_info=user_basic_info)
```

#### MEDIUM

**Finding ID: ASVS-231-MED-001**
**Title: Ticket Retrieval Lacks Ownership Verification (Authorization Step Absent)**

- **File:** `webapp/packages/api/user-service/services/chat_service.py`, `get_ticket_status()` lines 109-117

**Vulnerable code:**
```python
async def get_ticket_status(self, ticket_id: str) -> Optional[Dict[str, Any]]:
    """Get the status of a ticket"""
    ticket_path = self.storage_dir / f"{ticket_id}.json"
    
    if not ticket_path.exists():
        return None
    
    with open(ticket_path, 'r') as f:
        return json.load(f)
```

**Missing control:** No verification that the requesting user owns the ticket (no `session_id` comparison).

**Data flow:** Attacker guesses/enumerates ticket_id → calls `get_ticket_status()` → receives full ticket data including other users' messages, session_id, and LLM responses

**Impact:** Any authenticated user can retrieve any other user's chat tickets if they can guess or enumerate UUID ticket IDs. While UUIDs are hard to guess, if ticket IDs are exposed in URLs or logs, cross-user data access is possible. The ticket contains sensitive data: messages, session_id, model used, and full LLM responses.

**Remediation:**
```python
async def get_ticket_status(self, ticket_id: str, session_id: str) -> Optional[Dict[str, Any]]:
    """Get the status of a ticket, verifying ownership"""
    ticket_path = self.storage_dir / f"{ticket_id}.json"
    
    if not ticket_path.exists():
        return None
    
    with open(ticket_path, 'r') as f:
        ticket_data = json.load(f)
    
    # Verify ownership
    if ticket_data.get("session_id") != session_id:
        return None  # Or raise 403
    
    return ticket_data
```

## Cross-cutting Architecture Observations

1. **Separation of concerns creates security gaps:** The `chat_service.py` acts as a caller of `llm_service.py` but deliberately opts out of security controls by passing `None` for authorization parameters. This architectural pattern means any new service that calls `call_llm()` or `stream_llm()` must remember to pass the correct user context — a fragile security model.

2. **Optional security controls are dangerous:** The `if user_service and user_id:` pattern in `llm_service.py` makes security controls optional. If any caller passes `None` (intentionally or accidentally), all protections are bypassed. Security controls should be mandatory with explicit opt-out requiring elevated privileges.

3. **Async processing disconnects authorization from execution:** The `create_chat_ticket()` method spawns a background task that executes the LLM call. Even if the HTTP handler enforces authorization, the background task has no reference to the authenticated user's context. This is a systemic pattern where async processing can outlive the authorization context.

4. **No defense-in-depth:** The spend allowance is the single point of control. There are no layered defenses (rate limiting → allowance check → concurrent request cap → circuit breaker). If the allowance is bypassed (as demonstrated), there are no fallback controls.

5. **Cost tracking gap in streaming:** The explicit comment "Cost tracking is not available for streaming responses" reveals a known gap that has not been addressed. This creates a permanent bypass path for the primary business control.

## Cross-cutting Recommendations

### Immediate (Critical — address before next release)

1. **Fix chat_service spend bypass:** Remove `user_service=None` and `user_id=None` from all `call_llm()`/`stream_llm()` invocations in `chat_service.py`. Require `user_id` as a mandatory parameter for `create_chat_ticket()` and `stream_chat()`.

2. **Make allowance check mandatory:** Refactor `call_llm()` and `stream_llm()` to require `user_id` (no default to "anonymous"). If truly anonymous access is needed, create a separate function with explicit naming (e.g., `call_llm_unmetered()`).

3. **Add concurrent request limiting:** Implement a per-user semaphore or counter that prevents more than N simultaneous LLM requests from passing the allowance check.

### Short-term (High — within 1-2 sprints)

4. **Implement rate limiting middleware:** Add per-user and per-IP rate limiting at the API gateway or middleware level. Document configuration in deployment guides.

5. **Add atomic allowance reservation:** Replace check-then-use with atomic reserve-execute-finalize pattern to eliminate TOCTOU race condition.

6. **Implement streaming cost tracking:** Use token counting or litellm's stream cost utilities to record usage for streaming responses.

7. **Create security controls documentation:** Document all rate limiting, anti-automation, and adaptive response mechanisms including their configuration parameters, defaults, and how they prevent malicious lockout.

### Long-term (Medium — within next quarter)

8. **Add circuit breakers:** Implement circuit breaker pattern for external LLM provider calls with documented configuration.

9. **Implement ticket ownership model:** Add proper authorization to ticket retrieval, ensuring users can only access their own tickets.

10. **Centralize authorization in middleware:** Move authorization checks out of individual service functions into middleware that cannot be bypassed by callers. Use dependency injection frameworks to ensure every LLM call path includes authorization.
