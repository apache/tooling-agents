## ASVS-14.3.1: Client-side Data Protection - Clearing Authenticated Data

### Findings for 14.3.1

#### MEDIUM
**Finding ID:** ASVS-1431-MED-001  
**Title:** Operations log accumulates sensitive data previews destined for client-side UI with no cleanup mechanism  
**Location:** `webapp/packages/api/user-service/services/data_store_service.py`, `AgentDataStoreProxy._log()`, lines ~228-237 and `_preview()`, lines ~218-226

```python
def _preview(self, value: Any) -> Any:
    """Make a compact display-safe preview of a stored value."""
    if value is None:
        return None
    try:
        s = json.dumps(value)
    except (TypeError, ValueError):
        s = repr(value)
    if len(s) > self._VALUE_PREVIEW_MAX:
        return s[: self._VALUE_PREVIEW_MAX] + "…"
    return s

def _log(self, op: str, **fields) -> None:
    if self._ops_log is None:
        return
    entry = {
        "op": op,
        "namespace": self._namespace,
        "agent": self._agent_name,
        "ts": datetime.utcnow().isoformat(),
        **fields,
    }
    self._ops_log.append(entry)
```

**Data flow:** stored value → `_preview()` (up to 200 chars) → `_log()` → `ops_log` list → (presumably) transmitted to client "sandbox UI" → client DOM/memory

**Evidence from docstring (lines ~192-198):**
```python
"""
When an ``ops_log`` list is passed in, every operation appends an entry
so the sandbox UI can show a live timeline of reads and writes.
"""
```

**Analysis:** The `ops_log` accumulates `valuePreview` entries containing up to 200 characters of actual stored data. The class documentation explicitly states this data is surfaced to a "sandbox UI." If this data is transmitted to the client (via WebSocket, SSE, or HTTP response), it persists in the client's DOM or JavaScript memory. There is:
- No mechanism in this code to clear the ops_log after session termination
- No `Clear-Site-Data` header set when the session ends
- No client-side cleanup callback or signal
- The ops_log grows unbounded during a session (no eviction)

**Impact:** After session termination, value previews containing fragments of user data (potentially including sensitive information stored by agents) could remain in browser memory, DOM nodes, or JavaScript objects until the page is closed or garbage collected. If another user accesses the same browser session, this data could be exposed.

**Remediation:**

Server-side: Ensure session termination endpoints emit `Clear-Site-Data` header:
```python
@router.post("/auth/logout")
async def logout(response: Response):
    response.headers["Clear-Site-Data"] = '"cache", "cookies", "storage"'
    # ... invalidate session
    return {"status": "logged_out"}
```

Client-side: Implement cleanup in the frontend consuming the ops_log:
```javascript
// On session end (logout, token expiry, window unload)
window.addEventListener('beforeunload', () => {
    opsLog = [];
    sessionStorage.clear();
    // Clear any DOM nodes showing operation timeline
    document.querySelectorAll('.ops-timeline').forEach(el => el.innerHTML = '');
});
```

Service-side: Add an explicit clear method:
```python
def clear_ops_log(self) -> None:
    """Clear the operations log to prevent data retention."""
    if self._ops_log is not None:
        self._ops_log.clear()
```

#### LOW
**Finding ID:** ASVS-1431-LOW-001  
**Title:** No Clear-Site-Data header implementation in the codebase  
**Location:** All provided files (absence)

**Analysis:** The provided codebase contains no implementation of the `Clear-Site-Data` HTTP response header. While this is service-layer code (not route/middleware code), the absence across the entire provided scope suggests the application may not implement this control anywhere. The ASVS requirement specifically mentions this header as a mechanism for server-initiated client data cleanup.

**Note:** This could be implemented in middleware or route handlers not provided in this audit scope. This finding has low confidence without seeing the full application.

**Remediation:** Implement `Clear-Site-Data` in the logout/session-termination endpoint and in session expiry middleware:

```python
from fastapi import Response
from starlette.middleware.base import BaseHTTPMiddleware

class SessionCleanupMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        response = await call_next(request)
        # If session was invalidated during this request
        if getattr(request.state, 'session_terminated', False):
            response.headers["Clear-Site-Data"] = '"cache", "cookies", "storage"'
        return response
```

### Positive Patterns for 14.3.1

1. **Value preview truncation:** The `_VALUE_PREVIEW_MAX = 200` constant limits how much actual data appears in the ops_log entries, reducing (but not eliminating) sensitive data exposure in client-side logs.

2. **Conditional ops_log:** The `_log()` method checks `if self._ops_log is None: return`, meaning ops logging is opt-in. When the proxy is used "via the deployed-agent path where we don't surface ops to a UI," no data accumulates for client transmission.

3. **Server-side data scoping:** The `DataStoreService` keeps all raw data server-side. Only preview snippets reach the client via `ops_log`, not full records.

## Cross-cutting Architecture Observations

1. **Service-layer vs. Transport-layer separation:** The provided code is entirely service-layer (business logic and data access). ASVS 14.2.1 and 14.3.1 are primarily transport and client-side concerns. This means the most critical controls (URL parameter handling, response headers, client-side cleanup) would be implemented in FastAPI route handlers and middleware that were NOT provided for this audit.

2. **Data flows to client:** The `ops_log` mechanism is the only identified path where stored data values flow toward the client. Its design (shared mutable list, append-only, no cleanup) creates a data retention concern that spans both requirements:
   - 14.2.1: If ops_log data is transmitted via WebSocket URLs with query parameters, sensitive previews could leak into URLs
   - 14.3.1: Accumulated ops_log data on the client has no defined cleanup lifecycle

3. **User isolation model:** All database operations are scoped by `user_id`, which is a server-side parameter. This correctly prevents one user's data from reaching another user's client, but doesn't address the concern of a single user's sensitive data persisting in their browser after session end.

4. **No session awareness in service layer:** The `DataStoreService` and `AgentDataStoreProxy` have no concept of session lifecycle. They don't know when a session starts or ends, making it impossible for them to trigger data cleanup. Session lifecycle management must exist at a higher layer.

## Cross-cutting Recommendations

### Immediate (P1)
1. **Implement `Clear-Site-Data` header on logout/session-invalidation endpoints** — This is the primary server-side mechanism for ASVS 14.3.1 compliance. Add to all session termination paths.

2. **Audit FastAPI route handlers consuming DataStoreService** — Verify that `namespace`, `key`, and `user_id` values are received via POST bodies or headers, not URL query parameters or path segments that could contain sensitive data.

### Short-term (P2)
3. **Add ops_log lifecycle management** — Implement a `clear_ops_log()` method on `AgentDataStoreProxy` and ensure it's called when agent execution completes or sessions terminate.

4. **Document client-side cleanup requirements** — Since the ops_log is consumed by a frontend "sandbox UI," document the requirement for the frontend team to clear this data on session end, page navigation, and `beforeunload`.

### Long-term (P3)
5. **Implement ops_log size bounds** — Add a maximum entry count to prevent unbounded memory growth during long sessions:
   ```python
   _MAX_OPS_LOG_ENTRIES = 1000
   
   def _log(self, op: str, **fields) -> None:
       if self._ops_log is None:
           return
       if len(self._ops_log) >= self._MAX_OPS_LOG_ENTRIES:
           self._ops_log.pop(0)  # Evict oldest
       # ... append new entry
   ```

6. **Consider sensitive field marking** — Allow data store entries to be marked as sensitive, suppressing their inclusion in `_preview()` output entirely rather than just truncating.
