## ASVS-14.2.1: General Data Protection - Sensitive Data in URLs

### Findings for 14.2.1

The provided code is entirely server-side service-layer implementation. HTTP route definitions (FastAPI endpoints) that would directly demonstrate sensitive data in URLs are not in scope. However, architectural patterns in the service design reveal concerns about how this service would be consumed by route handlers.

#### LOW
**Finding ID:** ASVS-1421-LOW-001  
**Title:** Document IDs embed user identifiers suitable for URL path exposure  
**Location:** `webapp/packages/api/user-service/services/data_store_service.py`, `_make_doc_id()`, line ~72

```python
def _make_doc_id(self, user_id: str, namespace: str, key: str) -> str:
    """Generate document ID from composite key."""
    import base64
    safe_key = base64.urlsafe_b64encode(key.encode()).decode()
    return f"{user_id}:{namespace}:{safe_key}"
```

**Data flow:** user_id (from auth layer) → _make_doc_id → doc_id → potentially exposed in REST URLs or response bodies that inform client-side URL construction  
**Analysis:** The composite document ID format `{user_id}:{namespace}:{base64_key}` embeds the user identifier directly. If REST endpoints expose these IDs in responses (e.g., for direct retrieval), and if downstream route handlers construct URLs like `/api/datastore/{doc_id}`, the user_id becomes part of the URL path. While `user_id` is an identifier rather than a secret credential, the `key` parameter (base64-encoded) could contain sensitive descriptors. Base64 is encoding, not encryption—it's trivially reversible.

**Impact:** Low risk unless route handlers pass these as URL query parameters. The base64 encoding is purely for URL-safety, not confidentiality.

**Remediation:** Ensure route handlers serving this data use POST request bodies for operations that accept `key` or `namespace` values that could be sensitive, and avoid exposing composite doc_ids in URLs:

```python
# Good: POST body with sensitive identifiers
@router.post("/datastore/get")
async def get_data(request: DataStoreGetRequest, user=Depends(get_current_user)):
    return service.get(user.id, request.namespace, request.key)

# Avoid: GET with sensitive key in URL
# @router.get("/datastore/{namespace}/{key}")  # key visible in server logs, browser history
```

### Positive Patterns for 14.2.1

1. **Service-layer abstraction separates data access from HTTP concerns:** The `DataStoreService` and `AgentDataStoreProxy` accept `user_id` as a programmatic parameter rather than reading it from request URLs, allowing route handlers to extract it from authentication headers/cookies.

2. **AgentDataStoreProxy designed for injection, not direct HTTP exposure:** The proxy class is designed to be injected into agent execution contexts (`__init__` takes `user_id` from the server-side auth layer), eliminating the risk of agents putting credentials in URLs.

3. **No URL construction with sensitive data:** None of the service methods construct HTTP URLs containing API keys, session tokens, or user secrets. Database queries use server-side document IDs only.

## Cross-cutting Architecture Observations

1. **Service-layer vs. Transport-layer separation:** The provided code is entirely service-layer (business logic and data access). ASVS 14.2.1 and 14.3.1 are primarily transport and client-side concerns. This means the most critical controls (URL parameter handling, response headers, client-side cleanup) would be implemented in FastAPI route handlers and middleware that were NOT provided for this audit.

2. **Data flows to client:** The `ops_log` mechanism is the only identified path where stored data values flow toward the client. Its design (shared mutable list, append-only, no cleanup) creates a data retention concern that spans both requirements:
   - 14.2.1: If ops_log data is transmitted via WebSocket URLs with query parameters, sensitive previews could leak into URLs
   - 14.3.1: Accumulated ops_log data on the client has no defined cleanup lifecycle

3. **User isolation model:** All database operations are scoped by `user_id`, which is a server-side parameter. This correctly prevents one user's data from reaching another user's client, but doesn't address the concern of a single user's sensitive data persisting in their browser after session end.

4. **No session awareness in service layer:** The `DataStoreService` and `AgentDataStoreProxy` have no concept of session lifecycle. They don't know when a session starts or ends, making it impossible for them to trigger data cleanup. Session lifecycle management must exist at a higher layer.

## Cross-cutting Recommendations

### Immediate (P1)
1. **Implement `Clear-Site-Data` header on logout/session-invalidation endpoints** — This is the primary server-side mechanism for ASVS 14.3.1 compliance. Add to all session termination paths.

2. **Audit FastAPI route handlers consuming DataStoreService** — Verify that `namespace`, `key`, and `user_id` values are received via POST bodies or headers, not URL query parameters or path segments that could contain sensitive data.

### Short-term (P2)
3. **Add ops_log lifecycle management** — Implement a `clear_ops_log()` method on `AgentDataStoreProxy` and ensure it's called when agent execution completes or sessions terminate.

4. **Document client-side cleanup requirements** — Since the ops_log is consumed by a frontend "sandbox UI," document the requirement for the frontend team to clear this data on session end, page navigation, and `beforeunload`.

### Long-term (P3)
5. **Implement ops_log size bounds** — Add a maximum entry count to prevent unbounded memory growth during long sessions:
   ```python
   _MAX_OPS_LOG_ENTRIES = 1000
   
   def _log(self, op: str, **fields) -> None:
       if self._ops_log is None:
           return
       if len(self._ops_log) >= self._MAX_OPS_LOG_ENTRIES:
           self._ops_log.pop(0)  # Evict oldest
       # ... append new entry
   ```

6. **Consider sensitive field marking** — Allow data store entries to be marked as sensitive, suppressing their inclusion in `_preview()` output entirely rather than just truncating.
