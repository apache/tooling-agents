## ASVS-6.1.1: Authentication Documentation

### Findings for 6.1.1

#### LOW
**Finding ID:** ASVS-611-LOW-001

**Title:** Dev stub provider lacks documentation of rate limiting, anti-automation, or adaptive response controls

**File:** `webapp/packages/api/user-service/auth/providers/dev_stub.py` (entire file)

**Description:** The dev_stub provider documentation (lines 1-30) describes the provider's behavior, production usage warnings, and configuration shape. However, it does not document any rate limiting, anti-automation, or adaptive response controls — either for itself or for how the broader authentication system protects against credential stuffing or brute force attacks.

While the dev_stub is not intended for production, the `exchange_code` method (line 78) performs a simple dictionary lookup with no throttling:

```python
async def exchange_code(self, code: str, redirect_uri: str) -> UserInfo:
    """For dev_stub, ``code`` is the selected uid from the picker."""
    user = self._users_by_uid.get(code)
    if not user:
        raise ValueError(f"Unknown dev_stub user: {code}")
    return UserInfo(...)
```

**Data flow:** Attacker → `/auth/callback/dev_stub` POST → `exchange_code(uid)` → immediate success/failure with no rate limiting

**Impact:** If the dev_stub is accidentally enabled in a non-local environment, an attacker could enumerate all valid UIDs without any throttling or lockout. More broadly, no documentation in the provided codebase describes how the production OAuth-based authentication system implements rate limiting or account lockout prevention.

**Remediation:** 
1. Add documentation (either in a dedicated security design doc or in the auth base module) describing how the production authentication flow implements rate limiting on `/auth/callback/*` endpoints, anti-automation measures, and adaptive response.
2. Consider adding a runtime rate limiter in the auth router that applies regardless of provider type.

```python
# Example documentation addition to auth/README.md or base.py docstring:
"""
## Brute Force Protection (ASVS 6.1.1)

Rate limiting: All /auth/callback/* endpoints are rate-limited to 10 
requests/minute per IP via middleware (see rate_limiter.py).

Adaptive response: After 5 failed exchange_code attempts within 10 minutes 
from a single IP, the IP is temporarily blocked for 15 minutes.

Account lockout prevention: Rate limiting is per-IP, not per-account, 
preventing malicious lockout of legitimate users.
"""
```

## Cross-cutting Architecture Observations

1. **OAuth-only model eliminates password-related requirements:** The application's decision to use OAuth exclusively for production authentication means ASVS requirements 6.2.1–6.2.5 are effectively delegated to identity providers. This is a valid and secure architectural choice, but it creates a **dependency on OAuth provider security** — if the provider has weak password policies, the application inherits that weakness.

2. **Dev stub as the sole auditable auth code:** The only authentication implementation available for audit is the development stub, which by design bypasses all security controls. This makes it impossible to verify whether production authentication controls (rate limiting, adaptive response) exist in other parts of the codebase not provided for review.

3. **Gap between documentation and enforcement:** The dev_stub docstring states the provider "must never be enabled in production" and that the "session service logs a warning" — but the enforcement mechanism (environment check, runtime blocking) is not visible in this file. The warning is documented but enforcement is elsewhere (if it exists).

4. **No defense-in-depth at the auth router level:** Based on the single file provided, there's no evidence of provider-agnostic rate limiting or anti-automation controls at the auth routing layer that would protect all providers uniformly.

## Cross-cutting Recommendations

### Immediate (before next release)
1. **Verify environment guard exists:** Confirm that a hard block (not just a warning log) prevents `dev_stub` from being initialized when `APP_ENV` is `production` or `staging`. If it's only a log warning, upgrade to a `RuntimeError`.

### Short-term (next sprint)
2. **Create authentication security documentation:** Produce a document (or code-level docstring in the auth router) that explicitly addresses ASVS 6.1.1:
   - How rate limiting is applied to `/auth/callback/*` endpoints
   - What anti-automation measures exist (CAPTCHA triggers, IP blocking)
   - How adaptive response prevents account lockout
   - How the `state` parameter prevents CSRF in OAuth flows

3. **Add rate limiting middleware to auth routes:** Implement provider-agnostic rate limiting on all authentication callback endpoints to protect against abuse regardless of which provider is active.

### Long-term (roadmap)
4. **Document OAuth provider security requirements:** Since password security is delegated, document minimum requirements for acceptable OAuth providers (e.g., must support MFA, must detect compromised credentials, must enforce minimum password lengths).

5. **Consider future password support planning:** If password-based auth is ever added (as mentioned in the domain context), pre-create the architectural hooks: password validator interface, breached password service integration point, and composition rule configuration (set to "none" per ASVS 6.2.5).
