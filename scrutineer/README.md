# Scrutineer

Repository: [alpha-omega-security/scrutineer](https://github.com/alpha-omega-security/scrutineer)
License: MIT · Primary language: Go (with HTML and a little Python)
Tagline: *Security through scrutiny*

## What this is

Scrutineer is a local desktop tool for auditing open-source repositories for security vulnerabilities and managing the disclosure process end-to-end. You point it at a git URL, it runs a configurable pipeline of [claude-code skills](https://agentskills.io) against the repo, and presents the results in a web UI where you can triage findings, identify maintainers, and track disclosures from "new" through to "published."

It's the tool that produced the report set in `tooling-trusted-releases/` next to this directory — the security deep-dive, Semgrep scan, dependency listing, SBOM, maintainer summary, and the rest were all generated by Scrutineer skills running against the Apache Trusted Releases repository.

## How it works

Every scan is driven by a **skill** — a directory containing `SKILL.md` (YAML frontmatter + markdown body), an optional `schema.json`, and any helper scripts. When you add a repository, Scrutineer enqueues a single skill called `triage`, and `triage` enqueues all the others. The pipeline is fully data-driven: editing `skills/triage/SKILL.md` changes what runs, and dropping a new skill directory into `skills/` adds a scan type with no code changes.

A running skill can call back into Scrutineer over an authenticated HTTP API (per-scan bearer token in `./context.json`) to read prior scan results and enqueue further skills. This is what lets `triage` orchestrate the rest of the pipeline.

## The bundled pipeline

| Skill | What it does |
|---|---|
| `triage` | Orchestrates the default scan set via the Scrutineer API |
| `metadata` | Repo metadata from repos.ecosyste.ms |
| `packages` | Published packages from packages.ecosyste.ms |
| `advisories` | Known security advisories |
| `dependents` | Top runtime dependents per package |
| `dependencies` | `git-pkgs list` over every manifest |
| `sbom` | `git-pkgs sbom` for a CycloneDX SBOM |
| `maintainers` | Model-backed identification of maintainers + contact routes |
| `repo-overview` | `brief --json` structured project summary |
| `semgrep` | Static analysis mapped into the unified findings shape |
| `zizmor` | GitHub Actions workflow audit mapped into findings |
| `security-deep-dive` | Model-backed audit producing structured findings |

The fast skills (metadata, packages) finish in seconds. The deep-dive takes a few minutes depending on codebase size. External tools (semgrep, zizmor, git-pkgs, brief) are optional — if a tool is missing, that skill reports the failure in its scan output and the rest of the pipeline still completes.

## Output kinds

Skills declare an output kind in their frontmatter (`scrutineer.output_kind`), which picks the parser that ingests the result:

`findings` · `maintainers` · `packages` · `advisories` · `dependents` · `dependencies` · `repo_metadata` · `freeform`

Skills without these metadata keys still run; their output is captured verbatim.

## The finding workflow

Findings from `security-deep-dive` move through eight states with human gates between them:

1. **new** → Verify, skip to triage, or reject
2. **enriched** → independent confirmation ran; review and triage
3. **triaged** → confirmed real; prepare disclosure
4. **ready** → draft prepared; mark as reported
5. **reported** → sent to maintainer; await acknowledgement
6. **acknowledged** → maintainer working on fix
7. **fixed** → patch shipped; publish advisory
8. **published** → done

Each finding page has a notes section for triage reasoning and communication history.

## Quick start

Requires Go 1.26+ and an Anthropic API key.

```sh
export ANTHROPIC_API_KEY=sk-...
go run ./cmd/scrutineer -skills ./skills
open http://127.0.0.1:8080
```

Click "Add repository" in the sidebar, paste a git URL, and the `triage` pipeline runs.

### Docker

```sh
docker build -t scrutineer .
docker run -p 127.0.0.1:8080:8080 -v scrutineer-data:/data \
  -e ANTHROPIC_API_KEY=sk-... scrutineer
```

**Always bind to `127.0.0.1`.** The UI has no authentication; binding to `0.0.0.0` exposes your findings database to anyone on the network.

For per-scan container isolation, build the runner image — Scrutineer detects Docker at startup and uses it automatically:

```sh
docker build -t scrutineer-runner -f Dockerfile.runner .
```

Use `--no-docker` to disable containerised execution. Note that the current runner uses `--network none`, which blocks skills from calling the Scrutineer API or ecosyste.ms; that's a known sandbox-policy limitation.

## Flags

| Flag | Default | Purpose |
|---|---|---|
| `-config` | `./scrutineer.yaml` if present | YAML config file |
| `-addr` | `127.0.0.1:8080` | Listen address |
| `-data` | `./data` | Database and workspace directory |
| `-effort` | `high` | Claude effort level |
| `-skills` | — | Local skill directory (repeatable) |
| `-skills-repo` | — | Git HTTPS URL to clone skills from at startup |
| `--no-docker` | false | Disable containerised runner |
| `--runner-image` | `scrutineer-runner` | Docker image for per-scan containers |

Every flag can be set via the YAML config file; CLI flags win. The config can also replace the model pick list and pin a default model. See [`scrutineer.sample.yaml`](https://github.com/alpha-omega-security/scrutineer/blob/main/scrutineer.sample.yaml).

## UI navigation

The sidebar has six sections:

- **Repositories** — scanned repos with language, last scan status, finding counts. Tabs per repo: Summary, Findings, Threat Model, Packages, Dependencies, Dependents, Advisories, Maintainers, Data, Scans
- **Findings** — all findings across repos, filterable by severity. Each finding page shows the six-step analysis (trace, boundary, validation, prior art, reach, rating)
- **Packages** — registry entries with downloads, dependents, ecosystem filter
- **Maintainers** — people identified across repos with their linked repos and findings
- **Scans** — every scan run, filterable by skill and status; failed scans have a retry button
- **Skills** — installed skills from disk and from the UI; view, edit, or run any of them

The Dependencies tab supports one-click import: clicking the arrow next to a dependency resolves it via packages.ecosyste.ms and queues a full pipeline against that repo. The same applies to the Dependents tab.

## Security model

Scanning a repository is equivalent to running code from it. The containerised runner isolates each scan when available, but the default bare-metal mode runs everything as your user. **Only scan repositories you'd be willing to clone and build locally.** The full threat model is at [`threatmodel.md`](https://github.com/alpha-omega-security/scrutineer/blob/main/threatmodel.md).

## A note on attribution

The reports in the sibling `tooling-trusted-releases/` directory carry a "model: claude-opus-4-6" header in their underlying JSON — that's Scrutineer's `security-deep-dive` skill running against the target repo via the Anthropic API. The structure of those reports (boundaries → sinks → ruled-out reasoning, plus the eight-state finding workflow) is defined by the skills and the parsers in this project.
